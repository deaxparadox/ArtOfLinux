## The Problem This Solves

Before network configuration was standardized, every administrator had their own way of setting a hostname, configuring DNS, or telling the system how to resolve names. Some edited `/etc/resolv.conf` directly. Some relied on DHCP to overwrite it. Some wrote startup scripts that set everything at boot and hoped nothing changed it afterward.

The result was chaos. You'd SSH into a production server, fix the DNS configuration, and find it had reverted by morning. You'd add a hostname entry to `/etc/hosts`, then watch a service fail because it consulted DNS first and got a stale answer. You'd configure a static IP, and NetworkManager would undo it on the next DHCP renewal.

The deeper problem was that no one agreed on who was in charge. There was a file (`/etc/resolv.conf`) that everyone agreed was the canonical source of DNS configuration — but there were five different tools that all thought they owned it.

Modern Linux distributions have converged on a clearer ownership model. `systemd-resolved` or NetworkManager manages `/etc/resolv.conf`. NetworkManager manages connection profiles. `systemd-networkd` manages interface configuration on servers. But the files still exist, the old habits still persist, and the gotchas from twenty years of this mess are still alive on every production system you'll ever touch.

## Mental Model

Think of your system's network identity as a phone book with a chain of lookups.

When you call someone by name, you first check your personal address book (`/etc/hosts`). If they're not there, you ask directory assistance (`/etc/resolv.conf` tells you which directory service to call). The dispatcher who answers those calls is `systemd-resolved` — it caches answers, handles different phone books for different networks, and logs what it looked up.

The `/etc/nsswitch.conf` file is the rule book that says: "check your personal address book first, then call directory assistance." Change that order, and the same name can resolve to a completely different address.

The twist: someone else wrote your address book. NetworkManager filled it in when you connected to a network. `systemd-resolved` manages the dispatcher. If you write in the address book directly, the next time the network reconnects, someone erases what you wrote.

## Theory

### The Resolution Chain

Linux name resolution is not a single call to DNS. It is a sequence of lookups governed by a policy file, hitting different databases in a defined order. Understanding this chain is the difference between knowing where to look when DNS breaks and guessing randomly.

The chain, from top to bottom:

1. A process calls `getaddrinfo()` (or the older `gethostbyname()`)
2. The C library consults `/etc/nsswitch.conf` to determine what sources to try and in what order
3. It works through the `hosts:` line in that file — typically `files dns` or `files mdns4_minimal [NOTFOUND=return] dns`
4. `files` means `/etc/hosts` — a flat text file mapping names to IPs
5. `dns` means send a query to the nameservers listed in `/etc/resolv.conf`
6. On modern systems, that nameserver is `127.0.0.53` — the `systemd-resolved` stub listener
7. `systemd-resolved` checks its own cache, then forwards to the upstream DNS servers it learned from DHCP or static config

Every hop in this chain has its own configuration file. Misconfigure any one of them and you get silent failures or wrong answers.

### /etc/hosts

`/etc/hosts` is the oldest and most direct form of name resolution on Unix systems. It is a flat text file: each line is an IP address followed by one or more hostnames. When the C library is looking up a name and `files` appears before `dns` in `nsswitch.conf` (the universal default), `/etc/hosts` is consulted first.

Entries take precedence over DNS for that name, on that machine, for any process that goes through `getaddrinfo()`. This makes `/etc/hosts` useful for:

- Overriding DNS in development and testing (point `api.example.com` at a local container)
- Providing name resolution with no DNS server present (embedded systems, minimal containers)
- Emergency break-glass overrides on production servers when DNS is broken

The file is local-only. Every machine on your network maintains its own copy. It does not propagate anywhere, and it is not consulted by other machines resolving your hostname.

### /etc/resolv.conf

`/etc/resolv.conf` tells the C library's DNS resolver which nameservers to query and which domain suffixes to append to short names. The format is simple:

```text
nameserver 127.0.0.53
search example.com
options edns0 trust-ad
```

`nameserver` defines up to three DNS servers to query, in order. `search` defines a list of domain suffixes appended when a hostname without a dot is looked up — so a query for `api` becomes a query for `api.example.com`. `options` passes behavior flags to the resolver.

On modern systems, this file is not a file you edit. It is either a symlink or a file managed by another tool. Who manages it depends on the distribution and what stack is running:

- On Ubuntu 20.04 and later: `/etc/resolv.conf` is a symlink to `/run/systemd/resolve/stub-resolv.conf`, managed by `systemd-resolved`
- On Fedora: managed by `systemd-resolved`, same symlink arrangement
- On RHEL/CentOS with NetworkManager: managed by NetworkManager, which writes the file directly on each connection event
- On Alpine and minimal containers: a real file, often static, often written by the container runtime

The key invariant: if anything manages your network connection — NetworkManager, `systemd-networkd`, DHCP clients, or `systemd-resolved` — it considers `/etc/resolv.conf` to be its output file. Your manual edits are not persisted. They will be overwritten.

### /etc/nsswitch.conf

NSS (Name Service Switch) is the C library mechanism that allows different databases — users, groups, hostnames, services — to be looked up from different sources. `/etc/nsswitch.conf` defines the policy: for each database, which sources to consult and in what order.

The line that matters most for networking is `hosts:`:

```text
hosts:          files myhostname mdns4_minimal [NOTFOUND=return] dns resolve [!UNAVAIL=return]
```

Breaking this down:
- `files` — check `/etc/hosts` first
- `myhostname` — resolve the system's own hostname without a DNS lookup
- `mdns4_minimal [NOTFOUND=return]` — try mDNS (Avahi/zeroconf) for `.local` names; if not found, stop here
- `dns` — query the nameservers in `/etc/resolv.conf`
- `resolve` — query `systemd-resolved` directly via its D-Bus interface (Fedora-specific; bypasses the stub)

The order is enforced strictly. `files` before `dns` means a matching entry in `/etc/hosts` wins, regardless of what DNS would return. Reverse that order with `dns files` and DNS answers take priority — the `/etc/hosts` file is only consulted if DNS fails entirely.

This file is rarely touched in normal operation, but when it is misconfigured, the failures look like DNS failures even though DNS itself is fine.

### systemd-resolved

`systemd-resolved` is a caching, validating DNS stub resolver that ships as part of systemd. It runs as a local daemon and listens for DNS queries on `127.0.0.53`. When a process looks up a hostname, its query goes to `127.0.0.53`, where `systemd-resolved` handles it:

1. If the answer is in cache, return it immediately
2. If not, forward to the upstream nameservers it knows about (received from NetworkManager via DHCP or configured statically)
3. Optionally validate with DNSSEC
4. Cache the result and return it

`systemd-resolved` maintains per-interface DNS configuration. If you're on a VPN that uses `corp.example.com` as a DNS domain, `systemd-resolved` can route queries for `*.corp.example.com` to the VPN's DNS server while routing everything else to your regular resolver. This split-DNS capability is the main reason it exists — a flat `/etc/resolv.conf` with a single `nameserver` line cannot do this.

It exposes three interfaces for legacy compatibility:

- `/run/systemd/resolve/stub-resolv.conf` — lists `127.0.0.53` as the nameserver; `/etc/resolv.conf` is typically a symlink to this
- `/run/systemd/resolve/resolv.conf` — lists the actual upstream DNS servers directly; bypasses the stub and its caching
- D-Bus API — used by `resolvectl` and tools that want per-interface DNS information

### NetworkManager Connection Files

NetworkManager stores per-connection configuration in `/etc/NetworkManager/system-connections/`. Each connection — wired, wireless, VPN — has its own file named after the connection:

```text
/etc/NetworkManager/system-connections/
├── Wired connection 1.nmconnection
├── home-wifi.nmconnection
└── work-vpn.nmconnection
```

These files are in a keyfile format (INI-like). They contain the interface name, IP addressing mode (DHCP or static), DNS servers, search domains, and dozens of other parameters. The permissions on these files are `0600` — they contain secrets (WiFi passwords, VPN credentials).

NetworkManager reads these files when a connection is activated and writes DNS configuration derived from them (or from DHCP responses) into `/etc/resolv.conf`, or passes it to `systemd-resolved` via D-Bus.

### systemd-networkd Network Files

On systems without NetworkManager — servers, containers, embedded systems — `systemd-networkd` handles interface configuration. It reads `.network` files from `/etc/systemd/network/`. Each `.network` file matches one or more interfaces and specifies how to configure them:

```ini
[Match]
Name=eth0

[Network]
Address=192.168.1.100/24
Gateway=192.168.1.1
DNS=8.8.8.8
DNS=8.8.4.4
```

`systemd-networkd` and `systemd-resolved` are designed to work together. When `systemd-networkd` configures an interface, it hands the DNS information to `systemd-resolved` automatically. The two are separate services that can run independently, but on a minimal server without NetworkManager, you typically run both.

### /etc/hostname

`/etc/hostname` contains a single line: the system's hostname. This is what `hostname` returns when called without arguments, and it is the value that tools like `hostnamectl`, cloud-init, and provisioning systems use when setting up a machine.

There are three distinct hostname concepts in systemd's model:

- **Static hostname** — the value in `/etc/hostname`, persists across reboots
- **Transient hostname** — set by DHCP or other network tools, not persisted
- **Pretty hostname** — a free-form human-readable label, stored in `/etc/machine-info`

You change the static hostname with `hostnamectl set-hostname`, not by editing `/etc/hostname` directly — the former also updates the transient hostname in the running kernel immediately.

## How It Works (Mechanics)

When a process calls `getaddrinfo("api.example.com", ...)`:

```text
Process
  │
  ├─ calls getaddrinfo() in glibc
  │
  ▼
glibc reads /etc/nsswitch.conf
  │
  ├─ hosts: files dns   ← policy
  │
  ├─ Step 1: check /etc/hosts
  │     ├─ found? → return immediately
  │     └─ not found → continue
  │
  └─ Step 2: open /etc/resolv.conf → read nameserver 127.0.0.53
             │
             └─ UDP query to 127.0.0.53:53
                  │
                  systemd-resolved (listening)
                  │
                  ├─ cache hit? → return cached answer
                  │
                  └─ cache miss → forward to upstream DNS
                        (e.g. 8.8.8.8, received from DHCP)
                              │
                              └─ answer → cache it → return to process
```

The stub resolver at `127.0.0.53` is the hinge point. Everything above it is the C library's concern. Everything below it is `systemd-resolved`'s concern. The two sides only touch at a DNS query on a loopback socket.

To see `systemd-resolved`'s current upstream configuration:

```bash
resolvectl status
```

```text
Global
       Protocols: +LLMNR +mDNS -DNSOverTLS DNSSEC=no/unsupported
resolv.conf mode: stub
      DNS Servers: 8.8.8.8 8.8.4.4
       DNS Domain: ~.

Link 2 (eth0)
    Current Scopes: DNS
         Protocols: +DefaultRoute +LLMNR -mDNS -DNSOverTLS DNSSEC=no/unsupported
Current DNS Server: 8.8.8.8
       DNS Servers: 8.8.8.8 8.8.4.4
        DNS Domain: example.com
```

The `resolv.conf mode: stub` line tells you `/etc/resolv.conf` is the stub symlink — all queries go through the caching stub. If it says `foreign`, it means `/etc/resolv.conf` points elsewhere and `systemd-resolved` is running in compatibility mode.

To confirm the symlink:

```bash
ls -la /etc/resolv.conf
```

```text
lrwxrwxrwx 1 root root 39 Jan 10 12:00 /etc/resolv.conf -> ../run/systemd/resolve/stub-resolv.conf
```

If you see a real file instead of a symlink, NetworkManager or something else is writing directly.

```bash
# Check the current state of /etc/resolv.conf and who owns it
ls -la /etc/resolv.conf
```

```text
# This is /run/systemd/resolve/stub-resolv.conf managed by man:systemd-resolved(8).
# Do not edit.
#
# This file might be symlinked as /etc/resolv.conf. If you're looking at
# /etc/resolv.conf and seeing this text, you have a system configured to use
```

```text
127.0.0.1   localhost
127.0.1.1   myserver.example.com myserver
```

```bash
grep '^hosts' /etc/nsswitch.conf
```

## Real-World Example

A new Ubuntu 22.04 server has been provisioned. DNS isn't working for internal names. You need to diagnose the full resolution stack without touching anything yet.

```bash
# Step 1: Confirm the resolv.conf situation
ls -la /etc/resolv.conf
```

```text
lrwxrwxrwx 1 root root 39 Jun 10 08:32 /etc/resolv.conf -> ../run/systemd/resolve/stub-resolv.conf
```

```bash
# Step 2: Check what systemd-resolved thinks the upstream DNS is
resolvectl status
```

```text
Global
       Protocols: +LLMNR +mDNS -DNSOverTLS DNSSEC=no/unsupported
resolv.conf mode: stub
      DNS Servers: 10.0.0.1
       DNS Domain: ~.

Link 2 (eth0)
    Current Scopes: DNS
         Protocols: +DefaultRoute
Current DNS Server: 10.0.0.1
       DNS Servers: 10.0.0.1
        DNS Domain: corp.internal
```

```bash
# Step 3: Try to resolve an internal name and watch it fail
resolvectl query db.corp.internal
```

```text
db.corp.internal: resolve call failed: NXDOMAIN
```

```bash
# Step 4: Try the upstream directly (bypassing the stub) to isolate the problem
dig @10.0.0.1 db.corp.internal
```

```text
;; ANSWER SECTION:
db.corp.internal.       300     IN      A       10.0.100.50
```

DNS resolves fine when queried directly. The problem is with the local stub or its configuration. Check whether `systemd-resolved` is healthy:

```bash
systemctl status systemd-resolved
```

```text
● systemd-resolved.service - Network Name Resolution
     Loaded: loaded (/lib/systemd/system/systemd-resolved.service; enabled)
     Active: active (running) since Mon 2026-06-10 08:30:11 UTC; 2min ago
```

In this case the service is running and has the right upstream. The actual issue turns out to be that `corp.internal` was not in the `DNS Domain` for the interface when the VPN wasn't connected — `systemd-resolved`'s per-interface routing sent the query to the global resolver instead of `10.0.0.1`. The fix is to set the DNS domain for the VPN interface. This is exactly the kind of problem that only exists because `systemd-resolved` does per-interface DNS routing and a flat `/etc/resolv.conf` wouldn't have had it.

```bash
# Add a static DNS domain override for an interface using resolvectl
sudo resolvectl domain eth0 corp.internal
```

A senior sysadmin would recognize this pattern immediately: if `dig @<server>` works but regular `nslookup` doesn't, the problem is always between your process and the upstream server — either the stub, NSSwitch ordering, or a missing search domain.

## ⚠️ Gotchas

### Editing /etc/resolv.conf Directly When It's a Symlink

The most common mistake. You add `nameserver 8.8.8.8` to `/etc/resolv.conf`, test it, it works, you move on. Twenty minutes later, after a DHCP renewal or a network reconnect, it's gone.

```bash
# What looks fine
echo "nameserver 8.8.8.8" | sudo tee /etc/resolv.conf
```

```text
nameserver 8.8.8.8
```

```bash
# But resolv.conf is a symlink — you just wrote to stub-resolv.conf
ls -la /etc/resolv.conf
```

```text
lrwxrwxrwx 1 root root 39 ... /etc/resolv.conf -> ../run/systemd/resolve/stub-resolv.conf
```

```bash
# And stub-resolv.conf is managed by systemd-resolved — it will regenerate it
# Check stub-resolv.conf after a few seconds or after running resolvectl
cat /run/systemd/resolve/stub-resolv.conf
```

```text
nameserver 127.0.0.53
options edns0 trust-ad
```

Your edit is gone. The rule: on any system where `/etc/resolv.conf` is a symlink, or where NetworkManager is running, the correct way to set DNS is through the managing tool — not by editing the file.

To set DNS persistently via NetworkManager:

```bash
# Set DNS on a connection — use nmcli to find the connection name first
nmcli connection show
nmcli connection modify "Wired connection 1" ipv4.dns "8.8.8.8 8.8.4.4"
nmcli connection modify "Wired connection 1" ipv4.ignore-auto-dns yes
nmcli connection up "Wired connection 1"
```

To set DNS via systemd-resolved's static config:

```bash
# Create a drop-in config for resolved
sudo mkdir -p /etc/systemd/resolved.conf.d/
sudo tee /etc/systemd/resolved.conf.d/dns.conf > /dev/null <<EOF
[Resolve]
DNS=8.8.8.8 8.8.4.4
FallbackDNS=1.1.1.1
EOF
sudo systemctl restart systemd-resolved
```

### Bypassing the Stub Resolver Loses Caching and Split-DNS

Some administrators, frustrated by the indirection of `127.0.0.53`, replace the symlink with `/run/systemd/resolve/resolv.conf` — the file that lists the actual upstream DNS servers directly. Queries then go straight to `8.8.8.8` (or whatever) rather than through the stub.

This seems simpler. It breaks two things silently:

First, you lose the cache. Every DNS lookup now hits the upstream server. This is measurable — DNS caching eliminates round-trips for frequently resolved names, especially in container-heavy environments.

Second, and more importantly, you lose split-DNS. `systemd-resolved` does per-interface DNS routing. If your VPN is configured to use `vpn-dns.corp.com` for `*.corp.internal` and your regular interface uses `8.8.8.8` for everything else, `127.0.0.53` handles the routing. The bypass file is a flat list of servers — it doesn't know about per-interface domains. VPN hostnames fail silently.

```bash
# The wrong way — pointing directly at upstream
sudo ln -sf /run/systemd/resolve/resolv.conf /etc/resolv.conf

# The right way — keep the stub, use resolvectl to inspect routing
sudo ln -sf /run/systemd/resolve/stub-resolv.conf /etc/resolv.conf
resolvectl status    # shows per-interface routing
```

### 127.0.0.1 for the Hostname in /etc/hosts Breaks Self-Resolution

Cloud instances and VMs are frequently provisioned with entries like this in `/etc/hosts`:

```text
127.0.0.1   myserver.example.com myserver
```

The intent is good: the hostname resolves to something even without DNS. The problem: this maps the FQDN to the loopback interface, not to the machine's actual IP address. Any service that resolves its own hostname to bind to or advertise that address — databases, message brokers, clustering software, Kubernetes — gets `127.0.0.1` and either binds only to loopback or advertises an unreachable address to other nodes.

The correct pattern is:

```text
127.0.0.1   localhost
127.0.1.1   myserver        # short name on loopback is fine for some tools
10.0.1.50   myserver.example.com myserver  # FQDN maps to the real IP
```

Or better yet, leave the FQDN out of `/etc/hosts` entirely and let DNS handle it. The `myhostname` NSS plugin in `systemd-resolved` resolves the local hostname to the correct IP without any `/etc/hosts` entry.

### /etc/nsswitch.conf Order Is Globally Applied — Changing It Breaks More Than DNS

`nsswitch.conf` doesn't just control hostname resolution. It controls user lookups (`passwd:`), group lookups (`group:`), and shadow file lookups (`shadow:`) as well. The file looks like this:

```text
passwd:         files systemd
group:          files systemd
hosts:          files myhostname mdns4_minimal [NOTFOUND=return] dns
```

A common mistake is editing the `hosts:` line carelessly and accidentally replacing the whole line with just `dns files` — or worse, putting `dns` before `files`. On a server with critical hostnames in `/etc/hosts` (internal services, container networks), this silently changes which answer wins when there's a name conflict between DNS and the local file.

```bash
# Wrong — DNS checked before /etc/hosts
# In /etc/nsswitch.conf:
# hosts: dns files

# A service that's in /etc/hosts at 10.0.0.1 might now resolve to a public DNS answer first
# If the DNS record disagrees, the local override is ignored

# Right — /etc/hosts always wins (the universal default)
# hosts: files myhostname mdns4_minimal [NOTFOUND=return] dns
```

If you ever touch `nsswitch.conf`, touch only the line you mean to touch, and verify the change didn't silently alter behavior for the other databases on the same file.

## Distro Notes

| Distro Family | /etc/resolv.conf behavior | DNS Manager |
|---|---|---|
| Ubuntu 20.04+ | Symlink to `/run/systemd/resolve/stub-resolv.conf` | `systemd-resolved` (default) |
| Fedora 33+ | Symlink to `/run/systemd/resolve/stub-resolv.conf` | `systemd-resolved` (default since F33) |
| RHEL / Rocky / AlmaLinux 8–9 | Real file written by NetworkManager | NetworkManager (with optional `systemd-resolved` integration) |
| Debian 12+ | Symlink to stub-resolv.conf if `systemd-resolved` is installed | Optional — not enabled by default on Debian |
| Arch | Symlink if `systemd-resolved` is enabled; static file otherwise | Manual — you configure it |
| Alpine (containers) | Real static file | Written by the container runtime or manually |

| Distro Family | NetworkManager connection files | networkd files |
|---|---|---|
| Ubuntu / Fedora / RHEL | `/etc/NetworkManager/system-connections/` (keyfile format) | Not used unless NM is removed |
| Debian minimal / servers | May use `/etc/network/interfaces` (ifupdown) or networkd | `/etc/systemd/network/*.network` |
| Arch | Depends on what you install — NM, networkd, or dhcpcd | Your choice |
| Alpine containers | `/etc/network/interfaces` (busybox ifupdown) | Not used |

## Alternatives & Comparisons

| Tool | Role | When It's in Charge |
|---|---|---|
| `systemd-resolved` | Caching stub resolver, DNSSEC, split-DNS routing | Default on Ubuntu, Fedora; optional on Debian |
| NetworkManager | Interface config + DNS injection into resolv.conf or resolved | Desktop systems, RHEL/Fedora servers |
| `systemd-networkd` | Interface config, static/DHCP, passes DNS to resolved | Servers without NM; containers; minimal installs |
| `resolvconf` (openresolv) | Aggregator — collects DNS from multiple sources and writes resolv.conf | Older Debian-based systems, before systemd-resolved |
| Static `/etc/resolv.conf` | Direct nameserver config, no dynamic management | Containers, embedded, minimal systems |
| `dhcpcd` | DHCP client that can also write resolv.conf | Arch on desktops; historically common |

On a modern server that runs systemd: use `systemd-networkd` + `systemd-resolved` together, managed via `resolvectl` and `.network` files. Skip NetworkManager unless you're on a desktop where SSID switching and GUI tools matter. On a container: use a static `/etc/resolv.conf` written by the runtime — don't run `systemd-resolved` inside containers.

## Trade-offs

**Where this architecture shines:**
- Split-DNS across VPN and regular networks works transparently — the stub routes queries per-interface without any application changes
- DNS caching at the resolver level reduces latency for repeated lookups and reduces load on upstream servers
- DNSSEC validation happens centrally in `systemd-resolved` rather than per-application
- A single `resolvectl status` shows the complete DNS picture across all interfaces

**Where this costs you:**
- The indirection (`getaddrinfo` → glibc → stub at `127.0.0.53` → `systemd-resolved` → upstream) adds a diagnostic layer — when DNS breaks, you have more places to look
- Containers and minimal systems that don't run systemd can't use `systemd-resolved`; they need a static file or a separate resolver
- The split between "who configures DNS" (NetworkManager) and "who resolves DNS" (systemd-resolved) means you need to understand both to debug a DNS issue
- `/etc/resolv.conf` is no longer the ground truth — it points to a managed file, and the real config lives in NetworkManager profiles or `resolved.conf.d/` drop-ins

## When to Use / When to Avoid

✅ **Use /etc/hosts when:**
- You need a fast, reliable override for a single name on a single machine
- You're in development and want to point a domain at a local container without touching real DNS
- DNS is broken and you need to unblock something temporarily
- You're on a system with no DNS server (embedded, minimal container)

❌ **Avoid /etc/hosts when:**
- You need the mapping to be visible to other machines — it's local-only
- You're managing more than a handful of entries — use DNS, not a growing static file
- You're documenting infrastructure — `/etc/hosts` changes don't appear in any audit trail

✅ **Use resolvectl / systemd-resolved config when:**
- You need DNS to survive reboots and network reconnects
- You're on a system with VPN connections requiring split-DNS
- You want to set a fallback DNS server without disrupting DHCP-provided DNS

❌ **Avoid editing /etc/resolv.conf directly when:**
- It's a symlink — your changes will be overwritten
- NetworkManager is running — your changes will be overwritten
- `cloud-init` is running — your changes may be overwritten on the next boot

✅ **Use NetworkManager connection files when:**
- You need persistent static IP configuration that survives across reboots
- You want per-connection DNS settings (different DNS for work VPN vs home network)

⚠️ **Common mistakes:**
- Setting DNS by editing `/etc/resolv.conf` without checking if it's a symlink first
- Confusing `127.0.0.53` (stub resolver) with `127.0.0.1` (localhost) — they are different addresses with different purposes
- Assuming `/etc/nsswitch.conf` only affects DNS — it also governs user, group, and service lookups
- Running `systemd-resolved` inside a container expecting it to work like on a host — it usually doesn't without namespace configuration

---

## Veteran Perspective

The number of production DNS incidents that trace back to someone editing `/etc/resolv.conf` directly is remarkable. Not because the edit was wrong — the nameserver they put in was correct — but because they didn't know the file was managed, and it got overwritten six hours later during a DHCP renewal. The server went dark at 2am. By the time anyone noticed, the connection between "I edited resolv.conf yesterday" and "DNS stopped working overnight" was invisible.

The first thing I do on any unfamiliar Linux server is run `ls -la /etc/resolv.conf`. That single command tells me whether the system is using systemd-resolved (symlink to stub-resolv.conf), using resolved in a non-stub mode (symlink to resolv.conf), or has someone managing it directly (real file). The header comment in a real file often tells you what wrote it: NetworkManager leaves a comment saying so, cloud-init leaves a comment saying so, and a manually edited file has nothing.

The subtler gap that intermediate sysadmins miss is the stub resolver vs. the bypass. `127.0.0.53` is not just a proxy — it's a cache, a DNSSEC validator, and a split-DNS router. When you bypass it by pointing `/etc/resolv.conf` at the real `/run/systemd/resolve/resolv.conf` file (which lists actual upstream servers), you silently lose all three of those properties. Every query hits the network. Every VPN domain goes to the wrong server. DNSSEC is unenforced. It works, in the sense that names resolve — but it works worse.

The design principle behind all of this is that DNS configuration is not a single machine's concern anymore. An interface can be on multiple networks simultaneously (wired, wireless, VPN). Each network might have its own DNS server and its own search domains. A flat file with three `nameserver` lines can't model this. `systemd-resolved`'s per-interface DNS routing is the right architecture for that world. It's just wrapped in enough indirection that it's hard to debug until you understand the whole stack.

For technical interviews: if someone asks you how DNS resolution works on Linux, the shallow answer is "it reads `/etc/resolv.conf`." The correct answer starts with `getaddrinfo()`, mentions NSSwitch, explains that `/etc/resolv.conf` on modern systems points at a stub resolver, and covers the caching layer. That distinction between shallow and correct is exactly what separates the people who know Linux from the people who've used it.

## Try It Yourself

1. Run `ls -la /etc/resolv.conf` and determine whether your system uses a symlink or a real file. Then read the actual file content — either the symlink target or the file itself. What nameserver is listed?

2. Open `/etc/hosts` and add a test entry: `127.0.0.1 testhost.local`. Then run `ping -c 1 testhost.local` and confirm it resolves. After you've verified it, remove the entry. Notice that the change was instant — no service restart needed.

3. Run `grep '^hosts' /etc/nsswitch.conf` to see your current resolution order. Then run `getent hosts testhost.local` (with the entry from exercise 2 still in place) and `getent ahosts google.com` to see the full output. Notice that `getent` uses `getaddrinfo()` and therefore goes through NSSwitch exactly as applications do.

4. Run `resolvectl status` and identify: (a) what mode `resolv.conf` is in, (b) what the upstream DNS server is, and (c) whether any DNS domains are configured per-interface. If you have a VPN, connect to it and run `resolvectl status` again — watch the per-interface section change.

5. **Challenge (requires a VM or test server):** Set a static DNS server using NetworkManager without editing `/etc/resolv.conf` directly. Use `nmcli connection modify` to set `ipv4.dns` and `ipv4.ignore-auto-dns yes` on your primary connection, bring it down and up, then confirm the new server appears in `resolvectl status`. Then revert it and confirm it goes back to DHCP-provided DNS.

## Summary

- `/etc/resolv.conf` is managed by `systemd-resolved` or NetworkManager on modern systems — it is typically a symlink, and manual edits will be overwritten; check with `ls -la /etc/resolv.conf` before touching it
- The stub resolver at `127.0.0.53` (`systemd-resolved`) provides caching, DNSSEC validation, and per-interface split-DNS routing; bypassing it by pointing `/etc/resolv.conf` at the real upstream list silently loses all three
- `/etc/nsswitch.conf`'s `hosts:` line determines the resolution order — `files dns` (the default) means `/etc/hosts` wins over DNS; reversing it to `dns files` changes which answer is authoritative on every lookup
- `/etc/hosts` is local-only, instant, and checked before DNS by default; its main legitimate uses are development overrides and emergency break-glass configurations
- Never map a hostname's FQDN to `127.0.0.1` in `/etc/hosts` — services that resolve their own hostname to bind or advertise an IP will get the loopback address and either fail to listen on the network or advertise an unreachable address to other nodes
