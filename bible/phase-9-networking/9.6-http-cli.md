## The Problem This Solves

You've just deployed a service. It's running, the process is up, the port is open — but is it actually responding correctly? Is that API endpoint returning the right status code? Does the TLS certificate chain resolve? Is the redirect going where you think it's going?

Opening a browser tells you almost nothing. The browser hides status codes, follows redirects silently, uses cached DNS, adds its own headers, and mangles TLS errors into vague warnings. If you want to know exactly what the server is sending back — headers, body, timing, certificate details — you need a tool that speaks raw HTTP and gets out of your way.

Before `curl`, the options were grim: write a Python script with `urllib`, telnet into port 80 and type HTTP by hand, or trust your browser and hope for the best. None of those are scriptable, reproducible, or useful in an automated pipeline.

`curl` solves this by being the bare-metal HTTP client. It does exactly what you tell it, shows you exactly what the server sends back, and gets out of your way. That same call you run interactively to debug a broken endpoint can go directly into a CI pipeline, a health-check script, or an alerting rule without modification.

---

## Mental Model

Think of `curl` as a postal worker who follows instructions to the letter. You hand them a letter (the request), tell them the address (the URL), specify the envelope format (headers), and describe what to do with the response (write to a file, print to terminal, measure the timing). The postal worker doesn't read the letter for sense — they just deliver it and hand you back whatever the recipient sent. If you gave bad instructions, you get a bad result. If you gave precise instructions, you get a precise result.

This is the property that makes `curl` indispensable: it does exactly what you say, no more, no less. A browser makes hundreds of decisions for you — which TLS version to use, whether to follow a redirect, what `Accept` headers to send. `curl` makes none of those decisions unless you explicitly tell it to.

## Theory

### What curl Is

`curl` is a command-line tool and library for transferring data with URLs. It supports a large number of protocols — HTTP, HTTPS, FTP, SFTP, SCP, SMTP, and more — but in day-to-day Linux work, you're using it for HTTP and HTTPS almost exclusively. `curl` was created by Daniel Stenberg in 1998 and is maintained by him and a community of contributors. As of 2026, the current release is 8.20.0; it has been continuously developed for over 25 years and is embedded in billions of devices.

The underlying library, `libcurl`, is used by everything from web browsers to game engines. When you use `curl` the command, you're using a thin wrapper around `libcurl`. This matters because curl's behavior is well-defined, portable, and extensively documented — not platform-specific glue.

### What wget Is

`wget` is a different tool for a partially overlapping use case. Where `curl` is designed for flexibility and scriptability, `wget` is designed for downloading — particularly recursive downloading of entire website trees. `wget` can follow links, mirror directory structures, handle interrupted downloads, and resume where it left off. It does these things without requiring you to write shell logic around them.

`wget` (currently at 1.25.0, actively maintained as of 2026) is not obsolete, but its strongest use case is exactly that one: "download this thing, recursively if needed, and don't lose progress." For API calls, POST requests, header inspection, or authentication flows, `curl` is almost always the right choice.

### The Anatomy of an HTTP Request (from curl's perspective)

Every HTTP transaction has four parts:

1. **The method** — GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS. GET is the default; everything else must be specified.
2. **The URL** — where to send the request. This includes the scheme (http/https), the host, the port, and the path.
3. **The headers** — key-value metadata the client sends with the request: `Content-Type`, `Authorization`, `Accept`, `User-Agent`, etc.
4. **The body** — for POST and PUT, the data being sent. For GET and HEAD, there is no body.

`curl` maps these four parts onto flags: `-X` for method, the positional URL argument, `-H` for headers, and `-d` for the body. Once you understand this mapping, every `curl` command is readable.

### TLS: What curl Verifies by Default

When you hit an HTTPS URL, `curl` performs a full TLS handshake. By default, it verifies:
- That the certificate is issued by a trusted Certificate Authority (from the system's CA bundle)
- That the hostname in the certificate matches the hostname in the URL
- That the certificate is not expired

If any of these checks fail, `curl` refuses to proceed and prints an error. This is correct behavior. The `-k` / `--insecure` flag disables all of these checks. That flag exists for development; it has no business in any script that runs in production.

### Why --write-out Exists

HTTP latency has multiple components: DNS lookup, TCP connect, TLS handshake, time to first byte, total transfer. A plain `time curl https://example.com` gives you the wall time, but nothing else. `--write-out` lets you pull individual timing breakdowns out of curl without any external tooling — which makes it the right instrument for diagnosing where a slow request is losing time.

## How It Works (Mechanics)

When you run `curl https://api.example.com/v1/users`, this is the sequence:

1. `curl` parses the URL, extracts the hostname, scheme, and path.
2. It calls `getaddrinfo()` (or the configured resolver) to resolve the hostname to an IP address — DNS lookup.
3. It opens a TCP socket and calls `connect()` to establish a connection to the resolved IP on port 443.
4. It initiates a TLS handshake: client hello, server certificate, certificate chain verification against the system CA bundle, key exchange.
5. It sends the HTTP request over the encrypted channel: the request line (`GET /v1/users HTTP/2`), then the headers, then the body (if any).
6. It reads the response: status line, response headers, response body.
7. It writes the response body to stdout (or to a file if `-o` was given).
8. It closes the connection (or keeps it alive for subsequent requests if `--keepalive` is in effect).

The `--write-out` data is collected at step 8 — after the transfer completes — from internal curl timing counters that mirror what the kernel's socket layer reports.

A quick look at the system calls involved:

```bash
# Watch what curl actually does under the hood
strace -e trace=network curl -s -o /dev/null https://example.com 2>&1 | head -30
```

```text
socket(AF_INET6, SOCK_STREAM|SOCK_CLOEXEC, IPPROTO_TCP) = 5
connect(5, {sa_family=AF_INET6, sin6_port=htons(443), ...}, 28) = -1 EINPROGRESS
poll([{fd=5, events=POLLOUT|POLLWRNORM}], 1, 149000) = 1
...
```

The `-e trace=network` flag limits `strace` to network calls only — otherwise the output from a TLS handshake is hundreds of lines.

## Basic Example

```bash
# Fetch a URL and print the body to stdout — the most basic curl invocation
curl https://httpbin.org/get
```

```text
{
  "args": {},
  "headers": {
    "Accept": "*/*",
    "Host": "httpbin.org",
    "User-Agent": "curl/8.20.0"
  },
  "origin": "203.0.113.42",
  "url": "https://httpbin.org/get"
}
```

```bash
# See the response headers — -I sends a HEAD request (no body)
curl -I https://httpbin.org/get
```

```text
HTTP/2 200
content-type: application/json
content-length: 256
server: gunicorn/19.9.0
```

```bash
# -v shows the full transaction: TLS handshake, request headers, response headers
# Useful for debugging certificate issues, unexpected redirects, or header problems
curl -v https://httpbin.org/get 2>&1 | head -40
```

```text
*   Trying 54.208.105.16:443...
* Connected to httpbin.org (54.208.105.16) port 443
* ALPN: curl offers h2,http/1.1
* TLS handshake complete: TLSv1.3, cipher TLS_AES_256_GCM_SHA384
* Server certificate:
*   subject: CN=httpbin.org
*   expire date: Oct 15 2026
*   issuer: C=US; O=Let's Encrypt; CN=R11
> GET /get HTTP/2
> Host: httpbin.org
> User-Agent: curl/8.20.0
> Accept: */*
>
< HTTP/2 200
< content-type: application/json
...
```

Notice the `>` lines are what curl sent; the `<` lines are what the server replied. The `*` lines are curl's own connection/TLS metadata. Now try this yourself. Run the `-v` command against a site you use daily — the certificate expiry date and issuer are right there. If the output looks different, it's likely your curl version or the site's TLS configuration — both are normal variables.

## Real-World Example

Here is a complete diagnostic and API interaction sequence that a backend engineer or sysadmin would run against a real service.

```bash
# 1. Check if an API endpoint is alive and measure timing
curl -s -o /dev/null \
  --write-out "dns:%{time_namelookup}s  connect:%{time_connect}s  tls:%{time_appconnect}s  ttfb:%{time_starttransfer}s  total:%{time_total}s  http:%{http_code}\n" \
  https://api.example.com/health
```

```text
dns:0.004201s  connect:0.021843s  tls:0.087612s  ttfb:0.124508s  total:0.124781s  http:200
```

```bash
# 2. POST JSON to an API with a Bearer token
curl -s \
  -X POST \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..." \
  -d '{"username": "deploy-bot", "action": "restart", "service": "worker"}' \
  https://api.example.com/v2/actions | python3 -m json.tool
```

```text
{
    "id": "act_0193fa4e",
    "status": "queued",
    "service": "worker"
}
```

```bash
# 3. POST from a file — when the payload is too large or complex to inline
curl -s \
  -X POST \
  -H "Content-Type: application/json" \
  -d @payload.json \                   # @ prefix means "read from this file"
  https://api.example.com/v2/batch
```

```bash
# 4. Dump response headers to a file while saving the body separately
# Useful when you need to inspect Set-Cookie, Location, or X-RateLimit headers
curl -s \
  -D headers.txt \                     # -D dumps response headers to this file
  -o body.json \                       # -o saves the body here
  https://api.example.com/v2/users
cat headers.txt
```

```text
HTTP/2 200
content-type: application/json
x-ratelimit-remaining: 97
x-ratelimit-reset: 1749081600
set-cookie: session=abc123; Path=/; HttpOnly; Secure
```

```bash
# 5. Follow redirects — without -L, curl stops at the 301 and prints nothing useful
curl -sL https://bit.ly/some-shortened-url -o /dev/null -w "%{url_effective}\n"
```

```text
https://github.com/curl/curl/releases/tag/curl-8_20_0
```

```bash
# 6. Recursive download with wget — mirror a documentation site for offline reading
wget \
  --mirror \                           # enable recursive download + timestamps + infinite depth
  --convert-links \                    # rewrite links to work locally after download
  --adjust-extension \                 # add .html extension where needed
  --page-requisites \                  # fetch CSS, images, JS needed to render each page
  --no-parent \                        # don't crawl above the given URL path
  https://docs.example.com/guide/
```

**Walkthrough of the timing output in step 1:** The numbers break down the request into phases. `dns` is how long it took to resolve the hostname. `connect` is the TCP handshake completion. `tls` is when the TLS handshake finished. `ttfb` is "time to first byte" — when the server started sending data back. `total` is when the last byte arrived. If `tls` is high (over 200ms), the TLS handshake is the bottleneck — possibly a slow CA validation or certificate chain issue. If `ttfb` minus `tls` is high, the server is slow to respond after receiving the request — a backend or application-layer issue. These numbers tell you exactly where to look.

**The `-s` flag is silent mode** — it suppresses the progress meter that curl normally shows. In scripts, always use `-s` unless you want that output; it ends up in your logs as garbage otherwise.

**The `@payload.json` form in step 3** is how you send large or structured payloads without inlining them on the command line. It also means you can version-control the payload, diff it, and reuse it across environments with environment variable substitution.

## ⚠️ Gotchas

### HTTPS to HTTP Silent Downgrade on Redirect

`curl -L` follows redirects automatically. This is usually what you want. The trap is when a URL that starts as HTTPS redirects to HTTP — perhaps a misconfigured server, a legacy endpoint, or a deliberately downgraded redirect. `curl -L` will follow it silently, send your headers (including `Authorization` tokens) over the unencrypted connection, and give you no warning.

```bash
# This looks safe — but if example.com 301s to http://, your Bearer token goes in cleartext
curl -sL -H "Authorization: Bearer secret-token" https://example.com/api/data
```

```text
{"data": "..."}   # Looks fine. But the request may have gone over HTTP.
```

The fix is to restrict which protocol transitions are allowed:

```bash
# --proto-redirect restricts which protocols are acceptable after a redirect
# This allows HTTPS→HTTPS but refuses HTTPS→HTTP
curl -sL \
  --proto-redirect -all,+https \
  -H "Authorization: Bearer secret-token" \
  https://example.com/api/data
```

The underlying rule: `curl -L` is dumb about security. It follows the redirect regardless of protocol downgrade. Add `--proto-redirect -all,+https` any time you're following redirects with credentials.

### `-k` / `--insecure` Is Not a Debugging Tool

The argument for using `-k` is always "I'll remove it later." It never gets removed. `-k` disables certificate verification entirely — the hostname check, the CA trust chain, the expiry date. A server presenting a self-signed certificate for a completely different domain will succeed with `-k`. This is not debugging; it's blindfolding yourself.

```bash
# This tells you nothing about whether your certificate is correct
curl -k https://api.example.com/health
```

```text
{"status": "ok"}   # Could be a MITM proxy. You'll never know with -k.
```

The correct approaches depending on what you're actually trying to debug:

```bash
# Self-signed cert in development? Point curl at the CA cert directly.
curl --cacert /path/to/dev-ca.crt https://api.example.com/health

# Client certificate required? Provide both the cert and key.
curl --cert /path/to/client.crt --key /path/to/client.key https://api.example.com/health

# Just want to see the cert details without verifying? Use -v and read the output.
curl -v --cacert /dev/null https://api.example.com/health 2>&1 | grep -A5 "Server certificate"
```

The rule: `-k` is acceptable only when you are using a throwaway test environment and the data being sent carries zero security consequence. That narrows it to almost never.

### POST Redirects Don't Re-Send the Body

When curl follows a redirect after a POST request, it converts the POST to a GET and drops the request body. This is technically specified in RFC 9110 for 301 and 302 status codes (to preserve historical browser behavior), but it catches people off guard when they expect the full POST to be forwarded.

```bash
# You expect the POST body to arrive at the final destination — it won't
curl -sL -X POST -d '{"key": "value"}' https://api.example.com/submit
```

If `/submit` 301s to `/v2/submit`, the `-L` follow drops the body and sends a GET to `/v2/submit`. The server sees a GET with no body and likely returns a 400 or 405.

```bash
# Force curl to re-send POST data on 301 and 302 redirects
curl -sL \
  --post301 \       # re-send POST body on 301 redirect
  --post302 \       # re-send POST body on 302 redirect
  -X POST \
  -d '{"key": "value"}' \
  https://api.example.com/submit
```

For 303 redirects (which are semantically "go GET the result"), converting to GET is the correct behavior and you do not want `--post303`.

### --write-out Output Mixes With the Response Body

By default, `--write-out` output goes to stdout, alongside the response body. If you're piping the body into `jq` or another tool, the timing data lands in the middle of your JSON and breaks parsing.

```bash
# This is broken — the write-out string pollutes the JSON
curl -s --write-out "total: %{time_total}\n" https://api.example.com/data | jq .
```

```text
parse error (Invalid numeric literal at EOF on line 1, column 0)
```

The fix is to redirect the write-out output to stderr, which is separate from the body on stdout:

```bash
# Write timing to stderr, body to stdout — they never collide
curl -s \
  --write-out "total: %{time_total}\n" \
  --stderr - \                          # merge curl's own stderr to stdout... 
  2>/dev/null \                         # ...then discard it; timing goes to terminal
  https://api.example.com/data | jq .
```

The cleaner pattern for timing-only diagnostics is to suppress the body entirely:

```bash
# Discard body (-o /dev/null), print only timing to stdout
curl -s -o /dev/null \
  --write-out "dns:%{time_namelookup}  ttfb:%{time_starttransfer}  total:%{time_total}\n" \
  https://api.example.com/data
```

### Quoting JSON in -d on the Shell

Single-quoting works until your JSON contains a single quote. Double-quoting works until your JSON contains a double quote. The `$` in `$variable` inside double quotes expands before curl sees it, which means environment variable substitution happens in your JSON payload — sometimes by accident.

```bash
# This breaks if USER contains special characters, or if the JSON has a single quote
curl -d '{"user": "'"$USER"'", "note": "it'\''s fine"}' https://api.example.com/post
```

The robust approach for anything beyond a trivial payload is `@file`:

```bash
# Generate the payload into a file, then send it — no quoting issues
cat > /tmp/payload.json <<'EOF'
{"user": "nitish", "note": "it's fine, no escaping needed"}
EOF
curl -s -X POST -H "Content-Type: application/json" -d @/tmp/payload.json https://api.example.com/post
```

Or use `--data-urlencode` for form-encoded submissions where special characters must be encoded:

```bash
# --data-urlencode handles percent-encoding automatically
curl -G --data-urlencode "query=hello world & goodbye" https://api.example.com/search
```

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install curl wget` — curl comes pre-installed on most Ubuntu server images; wget does not |
| RHEL / Fedora / CentOS / Rocky | `dnf install curl wget` — curl bundled in minimal installs; system curl on RHEL may be older than upstream |
| Arch | `pacman -S curl wget` — typically ships very recent curl version |
| Alpine (containers) | `apk add curl wget` — Alpine's curl is often compiled against a minimal set of features; some TLS features may require `curl-dev`; for HTTP/2 verify with `curl --version | grep HTTP2` |

**RHEL/CentOS note:** Enterprise distributions sometimes ship curl versions that are 1-2 major versions behind upstream. On RHEL 8, you may still have curl 7.x. Check with `curl --version`. If you need a specific flag that appeared in a newer version, verify it exists before relying on it in scripts.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `curl` | API calls, scripting, debugging, any HTTP work requiring precision | Universally available; every flag imaginable; the correct default choice |
| `wget` | Downloading files, recursive site mirroring, resuming interrupted downloads | Poor at API calls; excellent at its specific job; version 2.2.1 as of 2026 |
| `httpie` (`http`) | Interactive API exploration where readable output matters more than speed | Python-based; colorized, formatted output by default; not installed by default anywhere |
| `xh` | Same use case as httpie but in scripts or on constrained systems | Rust implementation; httpie-compatible syntax; noticeably faster start time; binary with no runtime dependency |

`curl` is the right answer for anything that needs to run in production, in CI, or on a server you don't control. `httpie` and `xh` are excellent for interactive API exploration at a terminal where you want to read the response without piping through `jq`. The httpie syntax (`http POST api.example.com/users name=Alice`) is genuinely easier to type during development — but it's not installed by default, and the syntax difference means you can't copy a `curl` command from Stack Overflow and run it directly. `xh` is worth installing on a dev machine if you find yourself doing a lot of interactive API work; it costs nothing to have both.

## Trade-offs

**Where curl shines:**

- Universally available — it is on virtually every Linux system, every Docker base image, and every CI environment
- Precise control — you specify exactly what gets sent; curl makes no assumptions
- Scriptable — every flag is stable, every output format is documented, every exit code is defined
- Protocol breadth — the same tool works for HTTP, HTTPS, FTP, SFTP, and more without learning a new interface
- The `--write-out` mechanism gives you latency telemetry without installing anything extra

**Where curl costs you:**

- Verbose for complex requests — a POST with headers, auth, and a JSON body is 5-6 flags on a single line; it gets unwieldy
- Output is unformatted — response bodies come out as raw text; you need to pipe through `jq`, `python3 -m json.tool`, or similar to read JSON
- No session state — each `curl` invocation is independent; cookie jars (`-b`/`-c`) require explicit management
- Redirect semantics are subtle — the POST body behavior on redirects has tripped up experienced engineers

**Performance note:** For a single HTTP request, `curl` startup overhead is negligible (single-digit milliseconds). For high-frequency health checks or load-testing scenarios, use a dedicated tool like `wrk`, `hey`, or `k6` — `curl` is not designed for throughput measurement.

## When to Use / When to Avoid

✅ **Use curl when:**

- You need to make any HTTP request from a shell script — it's the right default tool
- You're debugging an API, TLS certificate, redirect chain, or authentication flow
- You need precise control over headers, method, body, or TLS configuration
- You're writing a health-check or readiness probe that needs to run inside a container
- You want to measure HTTP latency and its components without installing monitoring agents

✅ **Use wget when:**

- You need to download a file to disk, especially a large one that might be interrupted (`--continue` to resume)
- You need to recursively mirror a website or documentation tree
- You want timestamps preserved from the server (`--timestamping`)

❌ **Avoid curl when:**

- You need to download a large file with automatic resume — use `wget -c` instead
- You're doing interactive API exploration and want readable output — `httpie` or `xh` are less tiring to use

❌ **Avoid wget when:**

- You need to make POST requests, set custom headers, or handle authentication — use `curl`
- You need precise control over TLS — wget's TLS options are limited compared to curl

⚠️ **Common mistakes:**

- Forgetting `-s` in scripts — the progress meter output breaks log parsers and pipes
- Using `-k` and not removing it — add a comment next to every `-k` that says exactly why it's there and when it will be removed
- Piping `--write-out` output directly into a JSON parser without redirecting it separately
- Not using `-L` when hitting a URL that 301s — you get the redirect response, not the content, with no obvious error

## Veteran Perspective

The first thing a junior engineer does wrong with `curl` is treat it like a browser. They see a 301 and wonder why they got an empty body. They see a TLS error and immediately add `-k`. They pipe the response into `jq` and get a parse error because the progress meter is in the middle of the JSON. These aren't curl problems — they're gaps in the mental model. `curl` is not a browser. It does what you say. If you didn't say `-L`, it stops at the redirect. If you didn't say `-s`, it prints the progress meter. If you didn't say `-H "Content-Type: application/json"`, the server might reject your body.

The intermediate mistake is the opposite: writing `curl` scripts that are technically correct but impossible to debug. A 60-flag curl command with six subshells, piped through three tools, with `-k` and no comments — that's what happens when someone learned curl by copying examples without understanding them. The real skill is knowing which five flags you actually need for this specific request and leaving the others out.

The `--write-out` timing format is genuinely underused. Most people who need to diagnose latency either reach for a browser's Network tab (which doesn't help in production) or install something like New Relic (which is overkill for a targeted diagnostic). But the timing decomposition from `--write-out` — DNS, TCP connect, TLS handshake, time to first byte, total — is enough to localize almost any latency regression to a specific layer. The DNS time is high? Your resolver is slow or missing from cache. The TLS time is high relative to TCP? Your certificate chain has an extra hop, or OCSP stapling isn't configured. The time between `time_appconnect` and `time_starttransfer` is high? The server is slow to generate the response. That's the whole latency story, and you had it the whole time without installing anything.

The `-k` flag deserves special mention in a production context. I've reviewed infrastructure repositories where `-k` appeared in deployment scripts, health-check scripts, and CI pipelines — checked in years ago to fix a certificate problem that was never addressed. Those scripts are now dependencies. Nobody removes `-k` because nobody knows if something breaks when you do. The right rule: never commit `-k` to a repository. If you're working around a certificate problem, fix the certificate problem and document it. A `-k` in version control is a confession that the certificate was broken and nobody cared.

## Try It Yourself

1. Run `curl -I https://google.com` and read the status code and `Location` header. Now run it again with `curl -IL` and see where the redirect chain ends.

2. Make a POST request to `https://httpbin.org/post` with `-X POST -H "Content-Type: application/json" -d '{"name": "test"}'`. Read the JSON response — httpbin echoes back exactly what you sent. Verify your headers and body arrived correctly.

3. Run the timing `--write-out` command from the Real-World Example against three different domains. Compare the TLS handshake times. Which domain has the slowest TLS time? What does the certificate chain look like when you add `-v`?

4. Use `curl -D - -o /dev/null -s https://github.com` to dump just the response headers to stdout. Count how many `Set-Cookie` headers GitHub sends. Understand what `-D -` does (writing headers to stdout) vs `-D headers.txt` (writing to a file).

5. **Challenge:** Write a shell script that polls an API endpoint every 10 seconds, records the `time_starttransfer` (time to first byte) for each call with a timestamp, and writes a tab-separated log to a file. After 5 minutes, use `awk` to calculate the average TTFB and flag any requests that exceeded 200ms. Run it in a VM against `https://httpbin.org/delay/0`.

## Summary

- `curl` is the universal HTTP client for scripts and debugging: `-X` for method, `-H` for headers, `-d` for body, `-o` for output file, `-L` to follow redirects, `-v` for full transaction visibility.
- `--write-out` breaks HTTP latency into DNS, TCP connect, TLS handshake, time to first byte, and total — use `-s -o /dev/null --write-out '...'` to measure without fetching the body.
- `-k` / `--insecure` disables all TLS verification; it is not a debugging tool, and it has no place in any script that runs in production or handles real credentials.
- When `curl -L` follows a redirect, it silently drops the POST body on 301/302 by default; use `--post301 --post302` to preserve it. Add `--proto-redirect -all,+https` whenever following redirects with credentials.
- `wget` is the right tool for recursive downloads and resumable file transfers; `curl` is the right tool for everything else.
