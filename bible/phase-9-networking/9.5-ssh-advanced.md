## The Problem This Solves

You are on your laptop. Your company runs a PostgreSQL database on an internal server вҖ” `db-prod.internal` вҖ” that listens on port 5432. That port is not exposed to the internet. It never should be. The only machine with a path into that network is a bastion host вҖ” `bastion.example.com` вҖ” that accepts SSH connections on port 22.

Without tunneling, your options are: ask a sysadmin to open a firewall hole (bad), run your queries through a web proxy someone built (fragile), or copy data manually (unacceptable). With tunneling, you run one SSH command and your local PostgreSQL client connects to `localhost:5432` as if the database were sitting next to you. The traffic travels through the SSH connection вҖ” encrypted, logged, and controlled by existing access policies.

This is the real value of SSH tunneling. It is not a workaround or a hack. It is a first-class feature of the SSH protocol designed specifically for the problem of accessing services through a network boundary you cannot or should not punch a hole through.

The same mechanism works in reverse: a machine behind NAT, unreachable from the internet, can reach out through an SSH connection and publish a port on the remote server. A developer's laptop behind a coffee shop firewall can receive inbound connections. A sensor deep in a private network can be accessed by operations staff who never need to touch the NAT configuration.

Then there's connection overhead. Every SSH connection does a full TLS-equivalent handshake. If you run fifty commands in a loop вҖ” each one opening its own SSH connection вҖ” you are paying that cost fifty times. Connection multiplexing solves this: one master connection, every subsequent connection reuses the socket, latency drops to near zero.

SSH tunneling and multiplexing together are the difference between fighting your infrastructure and using it properly.

## Mental Model

Think of SSH as a pipe that already exists between your machine and a remote server. Normally, that pipe carries one thing: your shell session. Port forwarding is just putting additional things inside the same pipe.

Local forwarding is you placing a mailbox on your front porch. Anyone who drops a letter in that mailbox вҖ” your local applications, connecting to `localhost:port` вҖ” finds their mail delivered to a specific address on the other side of the pipe. Remote forwarding is the reverse: you place a mailbox on the remote server's front porch. Anyone over there who drops a letter in it gets it delivered back to your machine.

The pipe itself is unchanged. The SSH connection already runs. You are just specifying which additional channels flow through it, and in which direction.

## Theory

### What SSH Channels Are

The SSH protocol is a multiplexed channel protocol. A single SSH connection вҖ” one TCP connection between client and server вҖ” can carry multiple logical channels simultaneously. Your interactive shell is one channel. A port forward is another. An `sftp` subsystem is another. They all share the same underlying encrypted TCP connection, identified by channel IDs the protocol manages internally.

This is not an add-on feature. It is how SSH was designed from the start. The connection layer of the protocol exists specifically to multiplex these channels. Port forwarding is first-class, not a bolt-on.

### Local Port Forwarding

Local port forwarding (`-L`) instructs the SSH client to open a listening port on your local machine. When anything connects to that local port, the SSH client forwards the traffic through the encrypted tunnel to the SSH server, which then forwards it onward to a destination host and port вҖ” from the server's perspective.

The full syntax is:

```text
-L [bind_address:]localport:destination_host:destination_port
```

The destination host is resolved by the SSH server, not by your machine. This is the key insight. If the destination is `db-prod.internal`, your machine doesn't need to resolve that name вҖ” only the server does. The server is the one making the final TCP connection.

Traffic direction: your machine вҶ’ SSH server вҶ’ destination. You are pulling remote services toward yourself.

### Remote Port Forwarding

Remote port forwarding (`-R`) is the reverse. It instructs the SSH server to open a listening port on the remote machine. When anything connects to that remote port, the traffic travels back through the encrypted tunnel to the SSH client, which then connects to a destination host and port вҖ” from your machine's perspective.

The full syntax is:

```text
-R [bind_address:]remoteport:destination_host:destination_port
```

Traffic direction: remote machine вҶ’ SSH client вҶ’ destination. You are pushing a local service out to the remote server.

By default, `sshd` only binds the remote port to the loopback interface (`127.0.0.1`) on the server. To bind it to all interfaces so external machines can connect, you need `GatewayPorts yes` in `sshd_config` on the server side.

### Dynamic SOCKS Proxy

Dynamic forwarding (`-D`) is a generalization of local forwarding. Instead of forwarding to a single destination, the SSH client acts as a SOCKS5 proxy server on the specified local port. Each connection received on that port negotiates its destination through the SOCKS protocol, and the SSH server makes the outbound connection.

```text
-D [bind_address:]localport
```

This turns the SSH connection into a full proxy. Any application that supports SOCKS5 вҖ” browsers, `curl`, many command-line tools вҖ” can route its traffic through it, with all of that traffic being resolved and connected from the SSH server's network perspective.

### Connection Multiplexing

Every new SSH connection performs a full key exchange and authentication handshake. On a fast network this takes under a second; in a loop or in a deployment script that opens hundreds of connections, it becomes significant.

Connection multiplexing allows multiple SSH sessions to share one underlying TCP connection. The first connection becomes the **master** вҖ” it establishes the actual TCP connection and authenticates. Every subsequent connection to the same host reuses the master's socket without repeating the handshake.

Three `~/.ssh/config` directives control this:

- `ControlMaster` вҖ” when set to `auto`, the first connection becomes the master and all subsequent connections reuse it automatically
- `ControlPath` вҖ” the path to the Unix domain socket the master creates; subsequent connections connect through this socket
- `ControlPersist` вҖ” how long the master connection stays alive after all sessions close; `10m` keeps it alive for ten minutes after the last interactive session ends

### ProxyJump and Bastion Hosts

A bastion host is an SSH server that sits at the edge of a network and provides the only SSH access path into the interior. To reach `internal-server` you must first connect to `bastion`, then connect from `bastion` to `internal-server`.

`ProxyJump` (the `-J` flag, added in OpenSSH 7.3) handles this in one command. The SSH client connects to the bastion, establishes a tunnel through it, and completes the full SSH handshake with the destination вҖ” all end-to-end. Your traffic is encrypted all the way to the destination; the bastion can see that a connection is happening but not what is in it.

The older `ProxyCommand` directive can accomplish the same thing by specifying an arbitrary command whose stdin/stdout becomes the connection вҖ” typically `ssh -W %h:%p bastion`. ProxyJump replaced this for the common case because it is cleaner, more readable, and handles agent forwarding correctly by default.

### SSH Certificates

SSH public-key authentication scales poorly in large organizations. If a user has one key pair and accesses a hundred servers, their public key must be in `authorized_keys` on all one hundred servers. Revoking access means removing the key from all one hundred files. Adding a new server means pushing the key again.

SSH certificates solve this with a Certificate Authority. The CA is a key pair whose public key is trusted by all servers. When a user's public key is signed by the CA's private key, the resulting certificate carries identity information (username, validity period, permitted principals). Any server that trusts the CA will accept any certificate the CA has signed вҖ” without needing the individual public key pre-distributed.

This works in both directions: user certificates (the CA vouches for users to servers) and host certificates (the CA vouches for servers to users, replacing the `known_hosts` TOFU model with CA-based trust).

## How It Works (Mechanics)

### What Happens During a Port Forward

When you run `ssh -L 5432:db-prod.internal:5432 user@bastion`, the SSH client:

1. Connects to `bastion:22` and completes the normal SSH handshake
2. Sends a channel request to the server of type `direct-tcpip` (for local forwarding) or `tcpip-forward` (for remote forwarding)
3. Opens a TCP listener on `localhost:5432` on your machine
4. When any local process connects to `localhost:5432`, the client opens a new SSH channel and sends the forwarded data through it
5. The server receives the data, makes a TCP connection to `db-prod.internal:5432` from its own network interface, and proxies the data to it

The database sees the connection arriving from the bastion's IP address, not from your laptop. The bastion's logs show an SSH connection from your laptop and a TCP connection to the database. The data between your application and the database is encrypted inside the SSH tunnel.

### The ControlMaster Socket

When `ControlMaster auto` is set, the first SSH connection creates a Unix domain socket at the path specified by `ControlPath`. The socket file typically looks like:

```text
~/.ssh/control/%r@%h:%p
```

where `%r` is the remote username, `%h` is the hostname, and `%p` is the port. Subsequent SSH connections to the same host connect to this socket instead of opening a new TCP connection. The master process routes their authentication and channel requests. From the server's perspective, there is still one TCP connection.

`ssh -O check user@host` sends a probe through the socket to verify the master is alive. `ssh -O exit user@host` tells the master to close immediately, terminating all sessions using it.

## Basic Example

The simplest port forward you'll ever run вҖ” forward a remote web service to your local machine:

```bash
# Open an SSH tunnel: listen on local port 8080, forward to port 80 on the server itself
# -N: don't start a shell (we only want the tunnel)
# -L local_port:destination_host:destination_port
ssh -N -L 8080:localhost:80 user@remote-server.example.com
```

While that runs, in another terminal:

```bash
# Connect to the forwarded port вҖ” traffic goes through the SSH tunnel
curl http://localhost:8080
```

```text
<!DOCTYPE html>
<html>...
```

The `-N` flag tells SSH not to open a shell вҖ” the connection exists purely to maintain the tunnel. Without it, you'd also get a shell prompt, which works fine but leaves it ambiguous whether the tunnel is open or not.

Now try it yourself. Run the tunnel in one terminal tab and `curl localhost:8080` in another. If the remote server runs nginx or Apache, you will see the response arrive as if the web server were running on your own machine. That is the whole concept вҖ” the network boundary has been moved inside the SSH tunnel.

## Real-World Example

### Accessing a Private Database Through a Bastion

The scenario: a production PostgreSQL database runs on `db-prod.internal:5432`. It is not accessible from the internet. The only ingress point is `bastion.example.com`. You need to run a query or use a GUI client like DBeaver.

First, the `~/.ssh/config` entries that make this maintainable:

```ini
# The bastion вҖ” the only host reachable from the internet
Host bastion
    HostName bastion.example.com
    User deploy
    IdentityFile ~/.ssh/id_ed25519
    ControlMaster auto
    ControlPath ~/.ssh/control/%r@%h:%p
    ControlPersist 10m

# The internal database server вҖ” reached via the bastion
Host db-prod
    HostName db-prod.internal
    User deploy
    ProxyJump bastion
    IdentityFile ~/.ssh/id_ed25519
```

Now create the directory for control sockets:

```bash
mkdir -p ~/.ssh/control
chmod 700 ~/.ssh/control
```

Open the tunnel in the background:

```bash
# -N: no shell
# -f: fork to background after authentication
# -L: local port 5432 -> db-prod.internal:5432 through the bastion
ssh -N -f -L 5432:db-prod.internal:5432 bastion
```

Because of `ControlMaster auto` in your config, this connection also becomes the master. The next time you `ssh bastion`, it reuses the socket вҖ” no handshake.

Verify the tunnel is up:

```bash
# Check if the master connection is alive
ssh -O check bastion
```

```text
Master running (pid=38291)
```

Now connect with `psql` as if the database is local:

```bash
psql -h localhost -p 5432 -U appuser -d production
```

```text
psql (15.4)
Type "help" for help.

production=#
```

The connection travels: `psql вҶ’ localhost:5432 вҶ’ SSH tunnel вҶ’ bastion вҶ’ db-prod.internal:5432`. The database sees the connection arriving from the bastion's internal IP. Your DBA logs show what accessed the database; your SSH logs show what opened the tunnel. Full audit trail, no firewall changes.

When done, close just the tunnel without touching the master:

```bash
# Stop accepting new connections on the forwarded port, but keep the master alive
# so future ssh bastion commands still reuse the socket
ssh -O cancel -L 5432:db-prod.internal:5432 bastion
```

Or close everything:

```bash
ssh -O exit bastion
```

### Exposing a Local Dev Server via Reverse Tunnel

You are building a webhook receiver. Your local server runs on port 3000. You need GitHub to POST to it. You have a cheap VPS at `vps.example.com`.

```bash
# Expose localhost:3000 as port 80 on the VPS
# -R remoteport:destination_host:destination_port
ssh -N -f -R 80:localhost:3000 user@vps.example.com
```

Now `http://vps.example.com/webhook` reaches your laptop. GitHub can POST to it. You iterate locally with real payloads.

For this to work, `vps.example.com` must have `GatewayPorts yes` in `/etc/ssh/sshd_config` вҖ” otherwise the remote port only listens on `127.0.0.1` on the VPS, not on its public interface.

### File Transfer with rsync over SSH

`scp` works for quick copies. For anything more serious вҖ” syncing directories, resumable transfers, handling large trees efficiently вҖ” use `rsync`:

```bash
# Sync a local directory to a remote server
# -a: archive mode (recursive, preserves permissions, timestamps, symlinks, ownership)
# -v: verbose вҖ” show what's being transferred
# -z: compress in transit (useful on slow links; skip on fast LANs, it costs CPU)
# --progress: show per-file progress
rsync -avz --progress ./deploy/ user@remote-server.example.com:/var/www/app/
```


To use a non-standard SSH port or pass SSH options:


### Interactive File Transfer with sftp

When you need to explore a remote filesystem interactively вҖ” listing directories, downloading individual files, checking what's there:

```bash

```

### SSH Certificates вҖ” Creating a CA and Signing Keys

For an environment with more than a handful of servers, certificates are the right architecture:

```bash
# Step 1: Create the CA key pair (do this once, store the private key safely)
ssh-keygen -t ed25519 -f ~/.ssh/ssh_ca -C "Example Corp SSH CA"
```

```bash
# Step 2: Sign a user's public key
# -s: CA private key
# -I: certificate identity (human-readable, appears in logs)
# -n: principals (comma-separated list of usernames this cert is valid for)
# -V: validity period (+8h = 8 hours from now; use shorter for high-security environments)
ssh-keygen -s ~/.ssh/ssh_ca -I "alice@example.com" -n alice,deploy -V +8h ~/.ssh/id_ed25519.pub
```

This produces `~/.ssh/id_ed25519-cert.pub`. The user presents both the private key and the certificate; the server verifies the cert signature against the CA public key.

```bash
# Step 3: Configure the server to trust the CA
# Add to /etc/ssh/sshd_config on each server:
# TrustedUserCAKeys /etc/ssh/ssh_ca.pub
#
# Copy the CA public key to the server:
sudo cp ssh_ca.pub /etc/ssh/ssh_ca.pub
sudo chmod 644 /etc/ssh/ssh_ca.pub
```

```bash
# Step 4 (optional): Sign host keys so clients trust servers without known_hosts prompts
# -h: this is a host certificate
ssh-keygen -s ~/.ssh/ssh_ca -I "server1.example.com" -h /etc/ssh/ssh_host_ed25519_key.pub
# This produces /etc/ssh/ssh_host_ed25519_key-cert.pub
# Add to sshd_config: HostCertificate /etc/ssh/ssh_host_ed25519_key-cert.pub
```

## вҡ пёҸ Gotchas

### `-L` and `-R` Directions Are Swapped from What You Expect

This is the most common confusion with port forwarding. Here is the mental trap: "local" and "remote" describe where the *listening socket opens*, not where the traffic *comes from*.

`-L localport:host:remoteport` вҖ” the port opens **locally**. Traffic originating locally goes to the remote host. You use this to reach remote services from your machine.

`-R remoteport:host:localport` вҖ” the port opens **remotely**. Traffic arriving at the remote server gets forwarded back to you. You use this to expose local services to the remote machine.

The confusion happens because people think of "local" and "remote" in terms of where the interesting service lives. They want to "access a remote database locally" so they reach for `-R`. They should reach for `-L`.

The rule: **the flag letter tells you where the listening port opens**.

```bash
# WRONG вҖ” you want to access a remote database from your machine
ssh -R 5432:db.internal:5432 bastion  # opens port on the remote, sends traffic back to you
```

```bash
# CORRECT вҖ” open the port locally, traffic flows to the remote destination
ssh -L 5432:db.internal:5432 bastion
```

### `ssh -N -f` Exits Silently When the Network Drops

Running `ssh -N -f` forks the tunnel to the background and exits your shell. The tunnel runs. But if the network drops, the SSH process exits silently. Nothing restarts it. Your application continues trying to connect to `localhost:5432`, gets connection refused, and you spend twenty minutes wondering why the database is down.

```bash
# This looks like a persistent tunnel. It is not.
ssh -N -f -L 5432:db.internal:5432 bastion
```

For a persistent tunnel, you need either `autossh` (which monitors the tunnel and restarts it) or a `systemd` service with `Restart=always`:

```bash
# autossh: -M 0 disables autossh's own monitoring port; use SSH's built-in keepalives instead
autossh -M 0 -N -f \
  -o "ServerAliveInterval 30" \
  -o "ServerAliveCountMax 3" \
  -o "ExitOnForwardFailure yes" \
  -L 5432:db.internal:5432 bastion
```

Or as a systemd service:

```ini
[Unit]
Description=SSH tunnel to production database
After=network.target

[Service]
User=deploy
ExecStart=/usr/bin/ssh -N \
  -o "ServerAliveInterval 30" \
  -o "ServerAliveCountMax 3" \
  -o "ExitOnForwardFailure yes" \
  -L 5432:db.internal:5432 bastion
Restart=always
RestartSec=10

[Install]
WantedBy=multi-user.target
```

Note the absence of `-f` in the systemd unit. Daemonizing is systemd's job; adding `-f` would cause systemd to lose track of the process.

### ControlMaster Stale Socket After a Crash

If your machine loses power or the SSH master process is killed hard, the socket file at `ControlPath` is left behind as a dead file. The next SSH connection to that host tries to connect through the socket, finds it dead, and fails вҖ” sometimes with a confusing error:

```bash
ssh bastion
```

```text
ControlSocket ~/.ssh/control/deploy@bastion.example.com:22 already exists, disabling multiplexing
```

Or worse, it just hangs. The fix:

```bash
# First try the clean way
ssh -O exit bastion

# If that fails (master is dead), just remove the socket file
rm ~/.ssh/control/deploy@bastion.example.com:22
```

The permanent fix is to set `ControlMaster auto` instead of `ControlMaster yes`. With `auto`, if the socket is dead, SSH falls back to opening a fresh connection rather than failing.

### `scp` With Colons in Local Filenames

`scp` uses `:` as the separator between hostname and remote path. A local filename containing a colon is therefore misinterpreted as a remote host.

```bash
# WRONG вҖ” scp thinks "2024-06-15" is a hostname
scp 2024-06-15:backup.tar.gz user@remote:/backups/
```

```text
ssh: Could not resolve hostname 2024-06-15: Name or service not known
```

The fix is to prefix the local path with `./`:

```bash
# CORRECT вҖ” the ./ tells scp unambiguously that this is a local path
scp ./2024-06-15:backup.tar.gz user@remote:/backups/
```

This also applies to files with colons anywhere in the path. As a general rule, if you are transferring files with unusual characters, `rsync` handles them more gracefully than `scp`.

### Remote Port Forward Binds to Loopback by Default

You run `ssh -R 80:localhost:3000 vps.example.com` and the tunnel appears to work вҖ” `ssh -O check` confirms the master is running. But external clients cannot reach port 80 on the VPS.

The reason: by default, `sshd` only allows remote forwarded ports to bind on the loopback interface (`127.0.0.1`), not on `0.0.0.0`. The port is open on the VPS, but only processes on the VPS itself can connect to it.

To allow public binding, the server needs `GatewayPorts yes` (or the more specific `GatewayPorts clientspecified`) in `/etc/ssh/sshd_config`, followed by `sshd` reload:

```bash
# On the remote server
echo "GatewayPorts yes" | sudo tee -a /etc/ssh/sshd_config
sudo systemctl reload sshd
```

If you do not control the remote server, this approach will not work and you need an alternative like ngrok.

### rsync Trailing Slash Changes the Destination Structure

```bash
# These two commands do very different things:
rsync -av ./src/  user@host:/var/app/      # syncs the CONTENTS of src/ into /var/app/
rsync -av ./src   user@host:/var/app/      # creates /var/app/src/ and syncs into it
```

The trailing slash on the source means "the contents of this directory, not the directory itself." Without it, rsync copies the directory and its name. This has caused more than one production deployment to create a nested `src/src/` directory structure that's annoying to clean up.

The rule: if you want the files inside a directory at the destination, use a trailing slash on the source.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `scp` | Quick one-off file copies; legacy scripts | Uses SSH natively; being superseded вҖ” OpenSSH deprecated it internally in favor of the sftp protocol in 8.8 (2021); still works everywhere |
| `sftp` | Interactive file browsing; when you need to explore before transferring | Built into OpenSSH; supports tab completion in most clients |
| `rsync` | Directory synchronization; incremental updates; large transfers; production deploys | Not built-in everywhere; requires rsync on both ends; far more capable than scp for real use |
| `autossh` | Persistent tunnels that must survive network drops | Thin wrapper around SSH; use with `systemd` for production |
| `ngrok` / `bore` | Exposing local services when you don't control the remote server | Third-party tools; use them when you cannot set `GatewayPorts yes` on the server side |

For new work: use `rsync` for file transfers and `sftp` for interactive browsing. Reserve `scp` for quick copies in environments where rsync is not installed вҖ” but even then, if you're copying a directory tree, `rsync` is worth installing.

For persistent tunnels in production: always use a `systemd` service with `Restart=always`, or `autossh`. Never rely on a backgrounded `ssh -f` and expect it to survive the week.

## Trade-offs

**Where this shines:**

- Tunneling costs almost nothing over an already-open SSH connection вҖ” the overhead is one additional SSH channel, which is negligible
- Port forwarding works through firewalls and NAT without any firewall changes, as long as the SSH connection can reach the bastion
- Connection multiplexing produces dramatic latency improvements for deployment tools, config management (Ansible), and any workflow that opens many SSH connections вҖ” the difference between 50ms per connection and sub-millisecond
- SSH certificates scale cleanly to large fleets: one CA key trusted on all servers; access controlled by certificate issuance and expiry; revocation is immediate (expired certs are rejected automatically)

**Where this costs you:**

- Tunnels add one hop: your latency to the destination is now your latency to the bastion plus the bastion's latency to the destination; for real-time protocols this matters
- `ControlMaster` shares a single TCP connection between all sessions вҖ” if that connection experiences packet loss or gets dropped, all multiplexed sessions feel it simultaneously
- A forgotten background tunnel is a persistent security exposure вҖ” a backgrounded `ssh -N -f` with a local port forward is open for as long as the process runs; if your machine is compromised, the tunnel is a ready-made path into the internal network
- SSH certificate infrastructure adds operational complexity: you must protect the CA private key, manage certificate issuance (ideally automating it with something like Vault or `step-ca`), and handle the case where the CA key itself is compromised

**Performance note:** Connection multiplexing with `ControlMaster auto` and `ControlPersist 10m` typically reduces the time for repeated SSH connections to the same host from ~150вҖ“300ms (handshake cost) to under 5ms (socket connection).

## When to Use / When to Avoid

вң… **Use local port forwarding when:**
- You need to access a database, internal API, or service that is not exposed to the internet
- You want to use a local GUI tool (DBeaver, Sequel Pro, etc.) against a remote database
- You need to test against a remote service from your development machine

вң… **Use remote port forwarding when:**
- You need to receive inbound connections on a machine behind NAT (development webhooks, sharing a local dev server)
- You need to access a machine that cannot receive inbound connections from your network

вң… **Use dynamic SOCKS proxying when:**
- You need to route browser traffic through a remote server's network perspective
- You need to access multiple services behind a firewall without setting up individual local forwards

вң… **Use ProxyJump when:**
- Connecting through a bastion host (the vast majority of bastion use cases)

вң… **Use ControlMaster when:**
- Running Ansible or any tool that opens many SSH connections (the speedup is dramatic)
- Doing iterative development where you `ssh`, make a change, test, and repeat dozens of times

вң… **Use SSH certificates when:**
- Managing more than a dozen servers
- Access control needs to be auditable and revocable without touching every server's `authorized_keys`

вқҢ **Avoid backgrounded `ssh -N -f` for persistent production tunnels:**
- It dies silently when the network drops; use `autossh` or systemd instead

вқҢ **Avoid `scp` for directory trees:**
- rsync is faster (incremental), resumable, and doesn't mangle permissions on directories the way scp sometimes does

вқҢ **Avoid `GatewayPorts yes` on servers you don't fully control:**
- It allows any authenticated user to open publicly-accessible ports on your server; in a shared environment this is a significant exposure

вҡ пёҸ **Common mistakes:**
- Forgetting that remote port forwards bind to loopback only вҖ” checking with `netstat` and being confused why the port shows up but is not reachable
- Opening a local port forward and not realizing that `bind_address` defaults to loopback, so only your machine can connect to it (use `0.0.0.0:port:host:port` if other local machines need to use it)
- Using `%h` alone in `ControlPath` вҖ” if you SSH to the same host as different users, they share one socket and you get authentication errors; always include `%r` or use `%C` (the hash of all four: `%l%h%p%r`)

## Veteran Perspective

The thing that separates junior engineers from senior ones in SSH usage is not knowing the flags вҖ” it's knowing when the tool is being asked to carry more weight than it should.

Port forwarding is excellent for development access, occasional operational tasks, and prototyping. It is a poor substitute for a real VPN when dozens of engineers need persistent access to internal services. I have watched organizations run on a patchwork of personal SSH tunnels вҖ” every engineer with their own ControlMaster config pointing at their own bastion credentials вҖ” and the day someone leaves and their account is disabled, three services no one knew about stop working.

The right answer for organizational access to internal services is a proper network-layer solution: WireGuard, Tailscale, or a corporate VPN. SSH tunneling fills in the gaps and handles the one-offs. Treat it as a precision tool, not a platform.

On ControlMaster: the speedup is real and the configuration is low-effort. Every engineer who runs Ansible or does frequent deployments should have it in their `~/.ssh/config`. What almost no one configures is the monitoring. If your master connection silently dies and `ControlMaster auto` is set, subsequent connections will fall back to new connections without telling you вҖ” which means your assumption that "the database tunnel is up" may be wrong. Check with `ssh -O check` before assuming.

On certificates: every large organization I have ever audited has the same problem вҖ” hundreds of individual authorized_keys files, no central inventory, engineers who left three years ago still having key access because nobody audited the files. SSH certificates solve this completely. The operational cost of setting up a CA and a short-lived certificate issuer (something like HashiCorp Vault SSH secrets engine or Smallstep's `step-ca`) is a one-time investment. The alternative is a sprawling authorization problem that gets worse every month.

The war story version: a company discovered during a security audit that a former contractor's public key was in `authorized_keys` on 47 of their production servers. The contractor had left eighteen months earlier. Nobody had thought to remove the key because nobody had a list of where it was. With certificates set to a 24-hour validity window, this class of problem disappears вҖ” old certificates expire automatically.

On the reverse tunnel pattern for webhook development: it works, and it's clever, but if you're doing this more than occasionally, tools like `ngrok` or `bore` exist specifically for this pattern and handle the `GatewayPorts` problem, multiple domains, TLS termination, and dashboard visibility. SSH reverse tunnels are fine for "I need this working in the next five minutes" вҖ” they're not the right tool if you need it reliably for a week.

## Try It Yourself

1. Open a local port forward to a remote web server: `ssh -N -L 9090:localhost:80 user@your-server`. Then run `curl http://localhost:9090` in another terminal and verify you get the server's response. Kill the tunnel with Ctrl-C and confirm the port is gone.

2. Add a `ControlMaster auto` block to your `~/.ssh/config` for a host you connect to regularly. Time your first connection with `time ssh host echo ok` and your second connection with `time ssh host echo ok`. The difference should be dramatic (sub-100ms vs sub-5ms).

3. Use the `ssh -O check`, `ssh -O stop`, and `ssh -O exit` commands on an active master connection and observe the difference in behavior вҖ” `stop` keeps existing sessions alive while `exit` kills everything.

4. Configure a `ProxyJump` entry in your `~/.ssh/config` for a two-hop path (you вҶ’ host-A вҶ’ host-B). SSH to host-B directly and confirm the connection routes through A without manual steps. If you do not have a real bastion, you can simulate this locally with two Docker containers running `sshd`.

5. **Challenge:** Set up a systemd user service (under `~/.config/systemd/user/`) that maintains a persistent SSH tunnel to a remote host, automatically restarting if the connection drops. Use `systemctl --user enable` to make it start at login. Verify the tunnel survives a network interruption by briefly disconnecting your network and observing the service restart. Confirm it is working with `ssh -O check` after reconnection.

## Summary

- `-L localport:dest:destport` opens a port on your machine and forwards traffic to the destination via the SSH server; `-R remoteport:dest:destport` opens a port on the remote server and forwards traffic back to your machine вҖ” the letter tells you where the listening socket appears
- `ssh -N -f` backgrounds a tunnel but does not keep it alive вҖ” use `autossh` or a `systemd` service with `Restart=always` for anything that must persist
- `ControlMaster auto` with `ControlPath ~/.ssh/control/%r@%h:%p` and `ControlPersist 10m` eliminates per-connection handshake overhead and makes repeated SSH connections nearly instantaneous
- `ProxyJump` (OpenSSH 7.3+) is the clean, modern way to connect through a bastion; use it instead of the older `ProxyCommand` unless you need non-SSH proxy protocols
- SSH certificates replace the `authorized_keys` scaling problem for large fleets: a CA signs short-lived user certificates; servers trust the CA, not individual keys; revoking access is instant and does not require touching every server
