# [9.7] — Network Namespaces

> The kernel feature that lets Docker, Kubernetes, and every other container runtime give each container its own private network — without a hypervisor.

## The Problem This Solves

You're running three services on the same host. Service A needs to bind to port 8080. So does Service B. One of them loses.

You want to test a routing configuration without touching the live routing table. You can't — modifying it affects every process on the machine.

You want two containers to think they're on isolated machines, with separate interfaces, separate routing tables, separate firewall rules, and no ability to see each other's traffic unless you explicitly allow it. With a flat network stack shared by every process on the host, that's impossible.

Before network namespaces, isolation required a hypervisor. You needed a full virtual machine — full kernel, full OS image, full boot time — just to get a private network stack. The overhead was real and the operational complexity was high.

Linux has a better way. Network namespaces give each process group its own complete, isolated network stack — no hypervisor, no virtual machine, no reboot. This is how containers actually work at the kernel level.

---

## Mental Model

Think of the host's network stack as one large, open-plan office floor. Every process works on the same floor, hears every conversation, can walk up to any desk. There are no walls.

A network namespace is a set of walls. You carve out a private room with its own desks (interfaces), its own phone lines (routing table), its own security guard (firewall/iptables), and its own intercom system (loopback). Processes inside that room can only see their own room's infrastructure.

If a process in the private room needs to talk to the outside world, someone has to install a physical cable between the two rooms — that cable is a veth pair, and the switchboard that routes traffic between rooms is a bridge.

---

## Theory

A network namespace is a complete, independent copy of the Linux network stack. Each namespace gets:

- Its own set of network interfaces (initially only `lo`, the loopback)
- Its own routing table — completely separate from the host's
- Its own iptables/nftables rules
- Its own set of sockets — processes in different namespaces cannot see each other's connections
- Its own netfilter state

The word "copy" is misleading — it's not a copy of the host's config. It's a fresh, empty stack. `lo` exists but it's down by default. There are no routes. There are no rules. You build it from scratch.

Every process on a Linux system lives in exactly one network namespace at any given time. The initial namespace — the one the kernel starts in at boot — is the default and contains all the physical interfaces. Every namespace you create manually starts empty.

Namespaces are implemented as kernel data structures referenced by file descriptors. When no process is running inside a namespace and it has no other references holding it open, the kernel tears it down and reclaims the memory. The `ip netns` tool keeps namespaces alive by binding them to files under `/var/run/netns/` — those files are bind mounts that hold a reference to the namespace fd, so the namespace persists even when no process is running inside it.

The invariants that always hold:

- A fresh network namespace has exactly one interface: `lo`, and it is down
- Moving an interface into a namespace removes it from its current namespace — an interface lives in exactly one namespace at a time
- Routing, iptables rules, and sockets are all namespace-local — there is no leakage between namespaces

## How It Works (Mechanics)

### Creating and naming a namespace

When you run `ip netns add blue`, the kernel calls `unshare(CLONE_NEWNET)` in a temporary subprocess, which creates a new network namespace. The tool then bind-mounts the new namespace's file descriptor to `/var/run/netns/blue`. That bind mount is what keeps the namespace alive — it holds a reference without requiring a resident process.

```bash
# observe the bind mount that keeps the namespace alive
ls -la /var/run/netns/
```

```text
total 0
drwx--x--x 2 root root  60 Jun 15 14:02 .
drwxr-xr-x 6 root root 120 Jun 15 13:55 ..
-r--r--r-- 1 root root   0 Jun 15 14:02 blue
```

The file has size 0 — it is not a regular file. It is a bind mount of the namespace fd.

### Running a process inside a namespace

`ip netns exec blue <command>` calls `setns()` with the namespace fd bound to `/var/run/netns/blue`, then `exec()`s the command. The forked process inherits the namespace and runs entirely within the isolated network stack. When the process exits, the namespace fd reference count drops by one — but the bind mount at `/var/run/netns/blue` still holds it open.

Critically: only the network namespace is switched. The filesystem, process tree, user IDs, PID namespace, and environment variables remain the same as the calling process. `ip netns exec` is not a container — it is surgical network isolation.

### veth pairs — the virtual cable

Interfaces live in exactly one namespace. To connect two namespaces, you need a virtual cable that spans both. That cable is a `veth` (virtual ethernet) pair.

A veth pair is implemented in the kernel as two network devices that share a kernel socket buffer. Whatever you write to one end appears on the other — instantly, in kernel memory, without hitting the physical wire. The two ends start in the default namespace; you move one end into the target namespace.

```text
# ASCII diagram — veth pair connecting host to namespace

  Host network namespace              blue namespace
  ┌──────────────────────┐           ┌─────────────────────┐
  │  veth0               │◄─────────►│  veth1              │
  │  (host end)          │  kernel   │  (namespace end)    │
  │  10.0.0.1/24         │  memory   │  10.0.0.2/24        │
  └──────────────────────┘  pipe     └─────────────────────┘
```

### Software bridge — the virtual switch

When you have more than one namespace, or when you want namespaces to reach the physical network, you connect them through a bridge. A Linux bridge is a Layer 2 software switch. It learns MAC addresses, forwards frames between ports, and looks identical to a physical Ethernet switch from the perspective of connected interfaces.

```text
# ASCII diagram — bridge connecting multiple namespaces

  Host network namespace
  ┌─────────────────────────────────────────────────┐
  │                                                 │
  │   br0 (bridge / software switch)                │
  │   ┌─────────────────────────────────┐           │
  │   │  port: veth-blue   10.0.0.1/24  │           │
  │   │  port: veth-red                 │           │
  │   │  port: veth-green               │           │
  │   └─────┬─────────────┬─────────────┘           │
  └─────────┼─────────────┼─────────────────────────┘
            │             │
     blue ns│      red ns │
  ┌─────────┴──┐  ┌───────┴────┐
  │ eth0       │  │ eth0       │
  │ 10.0.0.2   │  │ 10.0.0.3   │
  └────────────┘  └────────────┘
```

Each namespace gets one end of a dedicated veth pair. The other end is added as a port on the bridge. The bridge handles routing between the namespaces at Layer 2.

### How Docker uses this

When Docker starts a container with the default bridge network, it does exactly this sequence:

1. Creates a new network namespace for the container
2. Creates a veth pair: one end named `vethXXXXXX` on the host, one end placed inside the container's namespace and renamed `eth0`
3. Connects the host end to `docker0`, Docker's built-in bridge
4. Assigns an IP address to the container end from Docker's subnet (usually 172.17.0.0/16)
5. Adds a NAT rule in iptables to masquerade outbound traffic from the container subnet

The container sees `eth0` and `lo`. It has no idea the host exists. That apparent isolation is a network namespace. You can replicate the entire thing by hand with `ip netns`, `ip link`, and `iptables` — and doing it once is the best way to understand what Docker is actually doing.

## Basic Example

This sequence creates a namespace, gives it network connectivity via a veth pair, and verifies isolation. Run all of this as root or with `sudo`.

```bash
# Create the network namespace
ip netns add blue

# Verify it was created
ip netns list
```

```text
blue
```

```bash
# Inside the new namespace: only lo exists, and it is DOWN
ip netns exec blue ip link show
```

```text
1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
```

```bash
# Create a veth pair: veth0 stays on the host, veth1 goes into the namespace
ip link add veth0 type veth peer name veth1

# Move veth1 into the blue namespace
ip link set veth1 netns blue

# Confirm: veth1 is gone from the host
ip link show type veth
```

```text
4: veth0@if3: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/ether 5e:a2:1b:44:3c:dd brd ff:ff:ff:ff:ff:ff link-netns blue
```

```bash
# Assign IPs and bring both ends up
ip addr add 10.0.0.1/24 dev veth0
ip link set veth0 up

ip netns exec blue ip addr add 10.0.0.2/24 dev veth1
ip netns exec blue ip link set veth1 up
ip netns exec blue ip link set lo up

# Verify connectivity: ping from host into namespace
ping -c 3 10.0.0.2
```

```text
PING 10.0.0.2 (10.0.0.2) 56(84) bytes of data.
64 bytes from 10.0.0.2: icmp_seq=1 ttl=64 time=0.087 ms
64 bytes from 10.0.0.2: icmp_seq=2 ttl=64 time=0.063 ms
64 bytes from 10.0.0.2: icmp_seq=3 ttl=64 time=0.071 ms
```

```bash
# Verify the namespace's view of its own network
ip netns exec blue ip addr show
```

```text
1: lo: <LOOPBACK,UP,LOWER_UP> mtu 65536 qdisc noqueue state UNKNOWN group default qlen 1000
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
    inet 127.0.0.1/8 scope host lo
3: veth1@if4: <BROADCAST,MULTICAST,UP,LOWER_UP> mtu 1500 qdisc noqueue state UP group default qlen 1000
    link/ether 2a:77:cc:89:de:11 brd ff:ff:ff:ff:ff:ff link-netnsid 0
    inet 10.0.0.2/24 scope global veth1
```

```bash
# Clean up — order matters (see Gotchas)
ip netns exec blue ip link set veth1 down
ip link set veth0 down
ip link delete veth0   # deleting one end automatically removes the pair
ip netns delete blue
```

Look at the `@if4` and `@if3` notation in the interface names. `veth0@if3` means "veth0 is paired with interface index 3, which lives in another namespace." This is the kernel telling you the other end of the cable exists somewhere — you just can't see it from this namespace.

Run this yourself. If the ping doesn't work, check that `veth0` is up on the host (`ip link show veth0`) and that you brought `lo` up inside the namespace. Those are the two most common first-try mistakes.

## Real-World Example

This example builds a two-namespace setup connected through a bridge — the same topology Docker's default bridge network uses. This is worth running in a VM so you can inspect every step.

```bash
#!/bin/bash
# Demonstrates bridge-connected namespace topology
# Run as root. Tested on kernel 5.15+ and 6.x.

# 1. Create the bridge
ip link add br0 type bridge
ip addr add 10.0.0.1/24 dev br0
ip link set br0 up

# 2. Create two namespaces
ip netns add ns-web
ip netns add ns-db

# 3. Create veth pairs — one pair per namespace
#    Convention: veth-<ns>-h is the host (bridge) end
#               veth-<ns>-n is the namespace end
ip link add veth-web-h type veth peer name veth-web-n
ip link add veth-db-h  type veth peer name veth-db-n

# 4. Move namespace ends into their namespaces
ip link set veth-web-n netns ns-web
ip link set veth-db-n  netns ns-db

# 5. Attach host ends to the bridge
ip link set veth-web-h master br0
ip link set veth-db-h  master br0
ip link set veth-web-h up
ip link set veth-db-h  up

# 6. Configure network inside ns-web
ip netns exec ns-web ip link set lo up
ip netns exec ns-web ip link set veth-web-n up
ip netns exec ns-web ip addr add 10.0.0.2/24 dev veth-web-n
ip netns exec ns-web ip route add default via 10.0.0.1   # bridge is the gateway

# 7. Configure network inside ns-db
ip netns exec ns-db ip link set lo up
ip netns exec ns-db ip link set veth-db-n up
ip netns exec ns-db ip addr add 10.0.0.3/24 dev veth-db-n
ip netns exec ns-db ip route add default via 10.0.0.1

# 8. Verify ns-web can reach ns-db
ip netns exec ns-web ping -c 2 10.0.0.3
```

```text
PING 10.0.0.3 (10.0.0.3) 56(84) bytes of data.
64 bytes from 10.0.0.3: icmp_seq=1 ttl=64 time=0.112 ms
64 bytes from 10.0.0.3: icmp_seq=2 ttl=64 time=0.098 ms
```

Step 5 is the key line: `ip link set veth-web-h master br0`. The word `master` means "add this interface as a port on this bridge." The bridge now owns that interface — it handles all Layer 2 forwarding decisions for traffic entering that port.

Step 6's `ip route add default via 10.0.0.1` gives `ns-web` a path to leave its local subnet. Without this, traffic destined for any IP outside 10.0.0.0/24 would be dropped — the namespace has no idea how to route it.

To enable outbound internet access from a namespace (the NAT step Docker does), you'd add:

```bash
# On the host — enable IP forwarding
echo 1 > /proc/sys/net/ipv4/ip_forward

# Masquerade traffic leaving the host's physical interface
iptables -t nat -A POSTROUTING -s 10.0.0.0/24 -o eth0 -j MASQUERADE
```

This is exactly Docker's model. `docker0` is `br0`. Each container's `eth0` is `veth-<ns>-n`. The iptables masquerade rule is the same one you'd see in `iptables -t nat -L -n -v` on a host running Docker.

## ⚠️ Gotchas

### A Fresh Namespace Has No Working Network — Not Even Loopback

You create a namespace and immediately try to run something that listens on localhost. It fails, or can't reach any service. The assumption is that loopback works by default.

```bash
ip netns add test
ip netns exec test ping 127.0.0.1
```

```text
connect: Network is unreachable
```

`lo` exists in every new namespace but it is DOWN. The kernel creates it in a down state. You must bring it up explicitly before anything that relies on loopback works.

```bash
ip netns exec test ip link set lo up
ip netns exec test ping -c 1 127.0.0.1
```

```text
PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.028 ms
```

This catches everyone at least once. If you're writing a setup script for namespaces, always include `ip link set lo up` as the first command you run inside a new namespace.

### Deleting a Namespace Does Not Delete Its veth Peer

You delete a namespace. You think you've cleaned up. Later you find orphaned veth interfaces on the host.

```bash
ip netns add orphan-test
ip link add veth-host type veth peer name veth-ns
ip link set veth-ns netns orphan-test
ip netns delete orphan-test

# Now check the host
ip link show type veth
```

```text
7: veth-host@if6: <BROADCAST,MULTICAST> mtu 1500 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/ether ...
```

When the namespace is deleted, any interfaces that were inside it are deleted too — including `veth-ns`. But the host end `veth-host` is in the host namespace, and it is not touched. It stays up as a half-dead orphan: the `@if6` peer reference now points to a non-existent interface.

The correct cleanup order is: delete the veth pair first, then delete the namespace.

```bash
ip link delete veth-host     # removing one end removes the pair
ip netns delete orphan-test
```

If you've already deleted the namespace and left the host end behind, just delete it directly:

```bash
ip link delete veth-host
```

### ip netns exec Is Not a Container

The mistake is using `ip netns exec` for isolation and assuming the process can't see the host filesystem, host processes, or host environment.

```bash
ip netns exec blue ls /etc/passwd
```

```text
/etc/passwd
```

The file is there. The network is isolated. The filesystem is not. PID namespace, user namespace, mount namespace — none of those are changed by `ip netns exec`. If you want full container isolation, you need all six Linux namespaces working together, plus cgroup constraints, plus a mount namespace for the filesystem. `ip netns exec` gives you exactly one of those six.

This matters operationally: a process that you put in a network namespace can still read the host's `/etc/sudoers`, still kill processes by PID, still read environment variables you've set in the calling shell. Do not confuse network isolation with security isolation.

### When One veth End Goes Down, the Peer Loses Its Carrier

You bring down one end of a veth pair. The other end stays "up" in the `ip link` sense — its admin state is UP — but it loses carrier. Any process reading the physical link state will see it as disconnected.

```bash
ip link set veth0 down
ip netns exec blue ip link show veth1
```

```text
3: veth1@if4: <NO-CARRIER,BROADCAST,MULTICAST,UP> mtu 1500 qdisc noqueue state LOWERLAYERDOWN ...
```

`LOWERLAYERDOWN` is the kernel telling you: "the admin state is up but the underlying layer has no carrier." The interface is administratively up but effectively dead. This is by design — veth pairs are a point-to-point cable and a cable with one end disconnected has no carrier. It is not a bug, but it will confuse anything that only checks `UP` and not `LOWER_UP` in the flags.

### ip -n Is a Shortcut That Skips the exec Overhead

Most tutorials use `ip netns exec <name> ip ...` for every command inside a namespace. That works, but iproute2 has a shorter form since kernel 4.x:

```bash
# Long form
ip netns exec blue ip addr show

# Equivalent short form — saves a fork/exec
ip -n blue addr show
```

The short form works for any `ip` subcommand. It does not work for arbitrary programs — those still need `ip netns exec`. For scripting namespace state queries, `ip -n` is cleaner.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `iproute2` installed by default; `/var/run/netns/` created by the tool on first use; no package needed for basic use |
| RHEL / Fedora / CentOS / Rocky | `iproute` package (same tool, different name); same behavior; `firewalld` may need zone rules added for bridge interfaces |
| Arch | `iproute2` package; same behavior |
| Alpine (containers) | `iproute2` package via `apk add iproute2`; commonly needed when testing container networking from inside an Alpine container |

On older kernels (pre-4.15), some bridge features like `bridge_nf_call_iptables` behave differently. On modern kernels (5.x+) with nftables replacing iptables, bridge traffic and nftables masquerading work cleanly. If you're using a kernel older than 4.9, test everything — bridge + namespace behavior has matured significantly since then.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `ip netns` (iproute2) | Direct namespace creation and management; testing, debugging, lab environments | The correct, modern tool; what all container runtimes use under the hood |
| `unshare --net` | Creating a namespace for a single process without naming it | Unnamed namespace — disappears when the process exits; no `ip netns list` visibility |
| Docker / Podman | Production container networking | Uses `ip netns` semantics internally; you lose direct control but gain lifecycle management |
| `nsenter` | Entering an existing namespace by PID | Useful for debugging a running container's network namespace without going through Docker; `nsenter -t <pid> -n` |
| `netns` (older approach, /proc) | Direct manipulation via `/proc/<pid>/ns/net` | Possible but unnecessary; `ip netns` wraps this cleanly |

Reach for `ip netns` directly when you're learning, debugging container networking, or building a testing environment. Reach for `nsenter` when you need to inspect a running container's namespace without the container runtime's involvement. For production, Docker and Podman abstract this away — but understanding `ip netns` means you can debug them when the abstraction leaks.

## Trade-offs

**Where this shines:**

- Complete network isolation with near-zero overhead — namespaces are a kernel data structure, not a VM; creating one takes microseconds
- Per-namespace iptables/nftables rules — you can firewall each namespace independently without affecting the host or other namespaces
- Testing complex network topologies without physical hardware — build multi-router test environments on a single machine
- Safe experimentation — a broken routing table in a namespace cannot affect the host routing table

**Where this costs you:**

- Operational complexity scales with the number of namespaces — managing dozens of namespaces by hand with `ip netns` gets unwieldy fast; this is exactly why container runtimes exist
- Debugging is harder — `ip link show` on the host doesn't show what's inside a namespace; you need `ip netns exec` or `ip -n` for every inspection command
- Network namespace isolation is not security isolation — a process in a namespace can still access the host filesystem, kill host processes, and read host environment variables; it only cannot see or touch the host network stack
- veth pairs add a hop — every veth pair is a layer of indirection; in extremely latency-sensitive paths this matters (in practice, the overhead is sub-microsecond on modern kernels)

**Performance note:** veth pair throughput on a modern kernel is in the range of tens of gigabits per second for large packets — essentially wire speed for anything inside a single machine. The overhead versus a loopback connection is negligible for virtually all workloads.

## When to Use / When to Avoid

✅ **Use this when:**
- You need to test network configurations (routing, firewall rules, packet routing) without risking the host network
- You're learning how container networking works and want to see the mechanics directly
- You're building integration test environments where services need isolated network stacks
- You need to run the same service on the same port on the same host (e.g., two copies of nginx on port 80)
- You're debugging a container's network behavior and need to inspect its namespace independently of the container runtime

❌ **Avoid this when:**
- You need full process isolation — use a container runtime (Docker, Podman) or a proper cgroup + all-namespace sandbox
- You're trying to isolate processes from reading each other's filesystem or environment — network namespaces do not help here
- You want persistent, managed network namespaces across reboots — hand-crafted `ip netns` state is not persisted by default; use a container runtime or `systemd-networkd` with proper configuration management
- You're on a kernel older than 3.8 — network namespaces became stable and complete around 3.8; older kernels have missing pieces

⚠️ **Common mistakes:**
- Forgetting `ip link set lo up` inside a new namespace — breaks anything relying on localhost
- Deleting the namespace before cleaning up veth pairs — leaves orphaned interfaces
- Assuming `ip netns exec` provides security isolation — it does not
- Trying to attach a physical interface to two namespaces simultaneously — interfaces live in exactly one namespace; moving one removes it from the other

## Veteran Perspective

Every engineer who works with containers eventually has a moment where they stop trusting Docker's abstraction and need to see what's actually underneath. That moment usually comes during an incident: a container can't reach the network, or two containers are leaking traffic to each other, or a firewall rule you applied on the host isn't affecting containers the way you expected.

When that moment arrives, knowing `ip netns` is the difference between being able to diagnose and being helpless.

The most common intermediate-level misunderstanding is treating the container runtime as a black box and the network as magic. "Docker handles networking" is true in the same way "the OS handles memory" is true — it's accurate until you need to understand why something is broken, at which point the abstraction becomes an obstacle. Every container network problem you'll ever debug eventually comes down to: does this veth pair exist? Is it in the right namespace? Does it have the right IP? Is the route table correct? Is there an iptables rule blocking this?

The intermediate-level mistake I see more often than anything else: engineers who know `ip netns exec` exists but always reach for `kubectl exec` or `docker exec` to inspect container networking. Those tools put you inside the container's mount namespace and PID namespace too — they do different things. `nsenter -t $(docker inspect --format '{{.State.Pid}}' <container>) -n` drops you into the container's network namespace with the host's tools available. That distinction matters at 3am when the container's shell doesn't have `tcpdump`.

The deeper design principle: Linux namespaces are a consequence of the Unix philosophy applied to the kernel itself. Instead of building a container primitive, the kernel built composable isolation mechanisms — network, PID, mount, IPC, user, UTS — that you can combine in whatever combination you need. A network namespace alone isn't a container. A PID namespace alone isn't a container. All six together, constrained by cgroups, mounted with an overlay filesystem — that's a container. Understanding this decomposition means you understand why containers have the security properties they have, and exactly where the boundaries are.

One war story: a team had a Kubernetes cluster where pods on the same node couldn't reach each other intermittently. The CNI plugin was Flannel. The symptom was that pings would work for a few seconds, then fail, then recover. The team spent hours looking at application logs and Kubernetes events. The actual cause was a veth pair whose host end was being periodically flapped by a misconfigured network monitoring agent that was running `ip link set <interface> down/up` on any interface it couldn't ping within its expected timeout. Because the agent was running on the host with access to all interfaces — including veth pairs belonging to pods — it was cycling the carrier state on pod interfaces without anyone noticing. The fix took thirty seconds once we ran `ip link show type veth` on the node and noticed one pair being flapped. Four hours of confusion, thirty seconds of diagnosis, ten minutes of fix. That's what it looks like when you don't know the layer underneath.

## Try It Yourself

1. Create a network namespace called `sandbox`. Run `ip netns exec sandbox ip link show` and note what interfaces exist. Bring `lo` up and confirm loopback works with a ping to `127.0.0.1`.

2. Create a veth pair, move one end into `sandbox`, assign IPs to both ends (use 192.168.100.0/24), and ping from the host into the namespace. Then ping from inside the namespace back to the host end.

3. Open two terminals. In terminal 1, run `ip netns exec sandbox bash`. In terminal 2, run `ip netns list` and `ps aux | grep bash` — confirm you can see the bash process from the host but that the namespace has its own network stack. Compare `ip addr show` inside the namespace versus on the host.

4. Create two namespaces (`ns-a` and `ns-b`) and connect them through a bridge (`br0`). Verify that `ns-a` can ping `ns-b`. Then add a third namespace `ns-c`, connect it to the same bridge, and confirm three-way reachability. *(Requires sudo and works best in a VM.)*

5. **Challenge:** Replicate Docker's default bridge network entirely by hand. Create a bridge named `mybr0` with subnet `172.20.0.0/24`. Create two namespaces with IPs in that subnet. Add iptables masquerade rules so that processes inside the namespaces can reach the internet via the host's physical interface. Verify with `ip netns exec ns-a curl -s https://example.com` (you'll need to add DNS: `ip netns exec ns-a bash -c 'echo nameserver 8.8.8.8 > /etc/resolv.conf'`). *(Requires sudo, a VM, and internet access from the host.)*

## Summary

- A network namespace is a complete, isolated Linux network stack — its own interfaces, routing table, iptables rules, and sockets. A fresh namespace has only `lo`, and it is DOWN.
- `ip netns add <name>` / `ip netns exec <name> <cmd>` / `ip netns delete <name>` are the three commands you'll use constantly; the namespace persists because `ip netns` bind-mounts it to `/var/run/netns/<name>`.
- `veth` pairs are virtual cables connecting namespaces — create with `ip link add veth0 type veth peer name veth1`, then move one end with `ip link set veth1 netns <name>`.
- `ip link add br0 type bridge` creates a software Layer 2 switch; add veth host ends as ports with `ip link set <iface> master br0` — this is exactly how Docker's `docker0` bridge works.
- `ip netns exec` is network isolation only — filesystem, PID space, and environment are unchanged; it is not a container.
