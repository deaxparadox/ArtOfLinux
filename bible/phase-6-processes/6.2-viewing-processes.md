## The Problem This Solves

Something is using 100% of a CPU. You don't know what. The disk is churning and the system is unresponsive. A service that should be running isn't. A process that should have stopped is still consuming memory. You need to find the thing that's wrong, identify it, and make a decision — in a hurry.

Without process viewing tools you are blind. You know the system is sick but you cannot see inside it. You could reboot and hope the problem doesn't recur. You could tail log files and wait. Neither of those is diagnosis — they're guessing.

The tools in this section are the eyes of a sysadmin. They exist specifically for the moment when something is wrong and you need to understand a live system without guessing.

---

## Mental Model

Think of a hospital emergency room whiteboard. Every patient has a row: their name, who brought them in, their current status (stable, critical, waiting, in surgery), and what resources they're consuming (bed, ventilator, IV drip). The board is updated constantly. You can scan it at a glance or zoom in on one row.

That's what process viewing tools do. The processes are the patients. The kernel maintains the board. Your tools — `ps`, `top`, `htop` — are different ways of reading it. Some give you a snapshot. Some give you a live feed. Some let you sort by vitals. All of them are reading the same underlying data: the kernel's internal process table, exposed through `/proc`.

---

## Theory

Every running process on a Linux system exists in the kernel's process table — a data structure the kernel maintains in memory. Each entry holds the process ID (PID), the parent process ID (PPID), the owning user, resource usage counters, and the current **process state**.

Process state is the most important field in this table. It tells you exactly what the kernel is doing with this process right now.

### Process States

The STAT column in `ps` output (and the S column in `top`) shows a state code. There are six states that matter in practice:

**R — Running or Runnable.** The process is either actively executing on a CPU right now, or it's on the run queue waiting for a CPU to become available. A healthy busy system will have a handful of R processes. Dozens of R processes with no obvious workload means something is wrong.

**S — Interruptible Sleep.** The process is waiting for something — a timer, a network packet, user input — and can be woken up immediately by a signal. This is the normal resting state for most processes. Almost everything on a quiet server is in S.

**D — Uninterruptible Sleep.** The process is waiting for a kernel I/O operation to complete and cannot be interrupted — not even by SIGKILL. This state was designed to protect the kernel from corruption that would occur if a filesystem operation were interrupted mid-flight. A few D processes are normal. Many D processes that persist is a serious symptom: it means I/O is stuck, usually a failing disk, an unresponsive NFS mount, or a storage controller problem. You cannot kill a D-state process. You have to fix the underlying I/O problem and let the kernel release it.

**Z — Zombie.** The process has finished executing but its entry remains in the process table because the parent hasn't called `wait()` to collect the exit status. Zombies consume no CPU or memory — just a PID and a table slot. One or two zombies are normal and transient. Hundreds of zombies that never clear indicate a parent process that has a bug in its child-reaping logic.

**T — Stopped.** The process has been suspended, either by `SIGSTOP`/`SIGTSTP` (Ctrl+Z in a terminal) or because a debugger attached to it. It is not running and not consuming CPU. It sits frozen, waiting to be resumed with `SIGCONT`.

**I — Idle Kernel Thread.** A kernel thread in an uninterruptible sleep that is not expected to do any work in the near future. Unlike D, an I-state kernel thread does *not* contribute to the system load average. You will only ever see `I` on kernel threads (names in square brackets in `ps` output, like `[kworker/0:0H]`). This state was added to the kernel specifically to prevent idle housekeeping threads from inflating the load average.

### The STAT Column Has Modifiers

The STAT column in `ps` output often shows more than one character. The first character is the base state. Additional characters are modifiers:

- `s` — session leader (the process created a new session, e.g., a shell)
- `+` — in the foreground process group
- `l` — multi-threaded
- `<` — high priority (negative nice value)
- `N` — low priority (positive nice value)
- `L` — has pages locked in memory (real-time processes, some I/O paths)

So `Ss` means sleeping session leader (a typical daemon), `R+` means running in the foreground, and `Z<` means a high-priority zombie.

### What `/proc` Actually Is

All process viewing tools — every single one — read their data from `/proc`, a virtual filesystem the kernel maintains in memory. There is no disk I/O when you run `ps`. The kernel synthesizes the `/proc/PID/` directory tree on-demand for each process. Running `ps aux` is essentially the kernel answering the question "tell me about every process you know about" by generating temporary virtual files.

This is important to understand because it means `ps` output is a snapshot taken at a single instant. By the time you read the output and decide to act on a PID, the process might be gone.

---

## How It Works (Mechanics)

When you run `ps aux`, the following happens:

1. `ps` opens `/proc` and reads its directory entries. Each numeric directory (e.g., `/proc/1234/`) corresponds to a live process.
2. For each PID directory, `ps` opens and reads several virtual files: `/proc/PID/status` (name, state, UID, GID), `/proc/PID/stat` (a space-separated line of 52+ fields used for CPU and memory stats), `/proc/PID/cmdline` (the full command line, null-delimited).
3. `ps` formats this data into columns and writes to stdout.

The `/proc/PID/stat` file is machine-readable and looks like this:

```text
1234 (nginx) S 1 1234 1234 0 -1 4194560 1234 0 0 0 15 3 0 0 20 0 1 0 ...
```

Fields in order: PID, command name (in parens), state letter, PPID, process group, session ID, ... then CPU jiffies (user and system), then memory pages. The `ps` tool parses this and converts jiffies to percentages, pages to kilobytes.

For `top` and `htop`, the mechanics are the same but they loop: read `/proc`, format the output, sleep briefly, read `/proc` again, diff the CPU counters, compute utilization as a rate, redraw the screen.

```bash
# You can read this directly without ps — this is what ps reads under the hood
cat /proc/$$/status
```

```text
Name:   bash
Umask:  0022
State:  S (sleeping)
Tgid:   12345
Ngid:   0
Pid:    12345
PPid:   12344
...
VmRSS:  5632 kB
...
```

The `$$` variable expands to the current shell's PID, so this reads the status of the shell you're running the command from.

## Basic Example

```bash
# List every process on the system with CPU and memory info (BSD syntax)
ps aux
```

```text
USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root           1  0.0  0.1 167580 11364 ?        Ss   Jun14   0:02 /sbin/init
root           2  0.0  0.0      0     0 ?        S    Jun14   0:00 [kthreadd]
root           3  0.0  0.0      0     0 ?        I<   Jun14   0:00 [rcu_gp]
...
www-data    1842  0.1  0.2 908024 17288 ?        Sl   Jun14   1:23 /usr/sbin/apache2
nitish      9201  0.0  0.0  10168  3412 pts/0    R+   10:43   0:00 ps aux
```

What to look for in this output:

- **USER** — who owns the process. `root` for system processes, a service user (like `www-data`) for daemons, your own username for interactive processes.
- **PID** — the process ID. You'll use this number with `kill`, `strace`, `lsof`, and `/proc/`.
- **%CPU** — average CPU usage since the process started, not instantaneous. If a process just started burning CPU, this number will look low. Use `top` or `htop` for current CPU rates.
- **%MEM** — percentage of physical RAM (RSS) relative to total RAM.
- **VSZ** — virtual memory size in kilobytes. Can look enormous; includes all mapped memory, libraries, and heap space the process has reserved but not necessarily used.
- **RSS** — resident set size: the actual physical RAM the process is using right now. This is the number that matters for memory pressure.
- **STAT** — the state column. Scan for `D` (stuck in I/O) or `Z` (zombies that won't clear).
- **TIME** — cumulative CPU time consumed. A process with `TIME` of `2d17:43` has consumed over two and a half days of CPU time since it started — that's meaningful.
- **COMMAND** — names in square brackets like `[kthreadd]` are kernel threads. Everything else is a user-space process.

Now try it yourself: `ps aux | head -20` for just the first twenty processes. Look at the STAT column. Count how many are in S, R, or D.

```bash
# List every process with parent PID visible (SysV syntax)
ps -ef
```

```text
UID          PID    PPID  C STIME TTY          TIME CMD
root           1       0  0 Jun14 ?        00:00:02 /sbin/init
root           2       0  0 Jun14 ?        00:00:00 [kthreadd]
root         623       1  0 Jun14 ?        00:00:00 /usr/sbin/sshd -D
root        9301     623  0 10:41 ?        00:00:00 sshd: nitish [priv]
nitish      9303    9301  0 10:41 ?        00:00:00 sshd: nitish@pts/0
nitish      9304    9303  0 10:41 pts/0    00:00:00 -bash
nitish      9412    9304  0 10:44 pts/0    00:00:00 ps -ef
```

The column that `ps -ef` adds which `ps aux` omits is **PPID** — the parent process ID. You can trace a process chain by following PPIDs upward. Notice in the example above: `sshd (623)` → `sshd: nitish [priv] (9301)` → `sshd: nitish@pts/0 (9303)` → `bash (9304)` → `ps (9412)`. That's an entire login session visible in four rows.

## Real-World Example

A user reports that the server "feels slow." You SSH in and need to understand the situation in under a minute.

```bash
# Step 1: get a bird's-eye view — what's consuming CPU right now?
top
```

Press `P` to sort by CPU (default), `M` to sort by memory. You're looking for any process above ~10% CPU that shouldn't be there. Press `q` to exit.

```bash
# Step 2: snapshot the process list and search for anomalies
ps aux --sort=-%cpu | head -20
```

```text
USER         PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
mysql       4821 94.3  8.1 1348224 667288 ?      Rl   Jun10  2143:17 /usr/sbin/mysqld
root        1001  0.8  0.1 113820  9284 ?        Ss   Jun10   12:34 /usr/sbin/sshd -D
...
```

MySQL is consuming 94% CPU and has accumulated over 2,000 hours of CPU time — it's been grinding continuously for days. The `TIME` field tells you this isn't a spike that just started; it's been doing this for a while.

```bash
# Step 3: check for stuck D-state processes — these indicate I/O problems
ps aux | awk '$8 ~ /^D/ { print }'
```

```text
mysql       4821 94.3  8.1 1348224 667288 ?      D    Jun10  2143:17 /usr/sbin/mysqld
```

The MySQL process is now in D state. It's waiting for I/O and it cannot be killed. The CPU usage plus D state together is a clear story: MySQL is hammering the disk with queries and the disk isn't keeping up.

```bash
# Step 4: check the parent-child relationship to understand process ownership
ps -ef --forest | grep mysql
```

```text
mysql       4821     1  94 Jun10 ?     2143:20 /usr/sbin/mysqld
mysql       4901  4821   0 Jun10 ?     00:00:00  \_ /usr/sbin/mysqld (worker)
mysql       4902  4821   0 Jun10 ?     00:00:00  \_ /usr/sbin/mysqld (worker)
```

```bash
# Step 5: find out which files or sockets the process has open
lsof -p 4821 | grep REG | tail -10
```

This tells you which database files it's hitting. Combined with a `SHOW PROCESSLIST` in MySQL, you can identify the specific query causing the load.

```bash
# Step 6: watch the process in real time to confirm it stays stuck
watch -n 1 'ps aux | grep mysql'
```

The full diagnostic flow — `ps aux` for overview, `awk` filter for D-state, `ps -ef --forest` for relationships, `lsof` for open files — is the sequence a senior sysadmin runs in the first 60 seconds of any "server is slow" investigation. Each tool narrows the question.

## ⚠️ Gotchas

### %CPU in ps aux is a Lifetime Average, Not Current Usage

The `%CPU` column in `ps aux` output is the average CPU utilization of the process since it started, not what it's using right now. A process that just started consuming 100% of a CPU one minute ago might show 2% in `ps aux` if the process has been running for 50 minutes before that.

```bash
# This can be misleading — looks calm, but might be spiking right now
ps aux | sort -k3 -rn | head -5
```

```text
USER      PID  %CPU  %MEM  ...  COMMAND
root     1234   2.1   0.0  ...  some_daemon   # was at 100% for last 60 seconds
```

The fix is to use `top` or `htop` for instantaneous CPU measurements. They read `/proc/PID/stat` twice with a short interval and compute the delta — that's a true rate.

```bash
# Use top for current CPU — press P to sort by CPU
top
```

For non-interactive scripting, `pidstat` from the `sysstat` package gives you instantaneous per-process CPU rates.

### D-State Processes Cannot Be Killed

When a sysadmin first encounters a D-state process, their instinct is to kill it with `kill -9`. This will not work. The signal is queued but the process cannot act on it until it returns from kernel space. If the I/O operation is truly stuck (unresponsive NFS server, failing disk controller), the process may stay in D state indefinitely.

```bash
# This appears to do nothing — the process stays in D
kill -9 4821
ps aux | grep 4821   # still there, still D
```

```text
mysql  4821  0.0  8.1 1348224 667288 ?  D  Jun10  2143:22 /usr/sbin/mysqld
```

The underlying rule: **D state is a kernel guarantee**, not a user-space condition. The kernel promised the process it would finish a specific I/O operation before allowing interruption. You have to fix the I/O problem to release it. For NFS: unmount the share forcefully (`umount -f -l`) or fix the NFS server. For disks: check `dmesg` for I/O errors and assess the hardware.

### pgrep Matches Substrings — Use -x for Exact Matches

`pgrep nginx` does not look for a process literally named "nginx." It looks for any process whose name *contains* the string "nginx." On a system running multiple nginx-related tools, this can return unexpected results.

```bash
# This matches "nginx", "nginx-master", "nginx-worker-pre", etc.
pgrep nginx
```

```text
1234
1235
1236
7890    # this is nginx_exporter — not nginx itself
```

If you're scripting and need to be precise, use `-x` to require an exact name match:

```bash
# Only matches processes whose name is exactly "nginx"
pgrep -x nginx
```

```text
1234
```

The same applies to `pkill` — which uses the same matching rules as `pgrep`. `pkill nginx` in a script that means to stop the web server might kill monitoring tools with "nginx" in the process name. Always use `-x` when writing scripts that act on pgrep results.

### ps Output Is a Snapshot — PIDs Recycle

The moment `ps` finishes reading `/proc` and writes its output, the world has moved on. By the time you read the output, form a plan, and run `kill PID`, the process might have exited and a completely different process might have taken that PID. Linux recycles PIDs.

```bash
# This is a classic race condition in scripts
BADPID=$(ps aux | grep suspicious_process | awk '{print $2}')
kill $BADPID   # might kill something else now
```

The safer idiom for scripted process management is `pgrep` with `pkill`, or `kill $(pgrep -x processname)` — both minimize the time window between finding and acting. Better still: use `pidfd` (Linux 5.1+) if you're writing code that needs to guarantee it's operating on the exact process it found.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install htop procps` — `htop` not installed by default; `ps` from `procps` package; `pstree` in `psmisc` |
| RHEL / Fedora / CentOS / Rocky | `dnf install htop procps-ng psmisc` — `htop` available in EPEL on older RHEL; `procps-ng` replaces older `procps` |
| Arch | `pacman -S htop procps-ng psmisc` |
| Alpine (containers) | `apk add procps` for full `ps`; Alpine's busybox `ps` has limited flags; `htop` via `apk add htop` |

Alpine containers deserve special attention: the `ps` built into BusyBox (Alpine's default) accepts a limited subset of flags and does not support `ps aux --forest` or `ps -eo`. If you're debugging inside a container and `ps aux` output looks wrong or a flag is rejected, you're hitting BusyBox `ps`. Install `procps` for the full version.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `ps aux` | Scripting, grepping, one-shot snapshots | Snapshot only; %CPU is a lifetime average |
| `ps -ef` | When PPID and full command matter | Snapshot; SysV syntax |
| `top` | Interactive, quick CPU/memory triage | Built in everywhere; limited UI |
| `htop` | Interactive deep inspection, tree view, mouse | Requires install; current version 3.4.1+ |
| `pgrep` / `pkill` | Finding/signaling by name in scripts | More precise than grep + ps pipelines |
| `pidof` | PID of a specific binary by exact path | Simpler than pgrep for scripts |
| `pstree` | Visualizing parent-child hierarchy | Great for understanding daemon trees |
| `btop` | Modern full-screen TUI with graphs | Requires install; CPU/net/disk history |

Reach for `ps aux | grep` for quick one-off inspection and for pipelines in scripts. Use `top` or `htop` any time you need real-time rates — CPU usage, memory pressure changing as you watch. Use `pgrep` in scripts instead of `ps aux | grep | awk` chains: it's cleaner, safer with PID extraction, and supports exact matching with `-x`. Save `pstree` for the moment when you need to understand which process spawned what — it's the fastest way to see a service's full family tree.

## Trade-offs

**Where this shines:**

- `ps` is zero-dependency: it's on every Linux system without exception. You can diagnose a broken system that can't install anything.
- `/proc` exposure means you can read process data with nothing but `cat` if your tools are broken — a critical property when recovering a damaged system.
- `pgrep` and `pkill` are safe for scripted process management because they avoid the grep-through-ps pipeline's pitfalls around matching "grep" itself in output.
- `htop` gives you a live, sorted, tree-capable view that compresses what would otherwise be a five-command investigation into a single interactive screen.

**Where this costs you:**

- `ps aux` %CPU is misleading for recently-changed workloads. Acting on it for performance diagnosis without corroborating with `top` leads to wrong conclusions.
- All of these tools are read-only views. They tell you what is wrong. Fixing it requires signals (covered in [6.3 — Signals](6.3-signals.md)), `nice`/`renice`, or stopping and restarting services.
- D-state processes are invisible to `kill`. Knowing a process is stuck in D is diagnostic information, not something you can resolve from the process-viewing layer.
- `htop` is not installed by default in most minimal or container environments. In a stripped-down container, you may have only BusyBox `ps`.

**Performance note:** `ps aux` on a system with thousands of processes (high-density container hosts, large databases with many worker threads) can take several hundred milliseconds and generate significant `/proc` read activity. Running `watch -n 1 'ps aux'` on such a system adds measurable overhead. Use `top` or `htop` instead — they are designed for continuous polling and do it more efficiently.

## When to Use / When to Avoid

✅ **Use `ps aux` when:**
- You need a one-shot list of all processes to grep, awk, or pipe into other tools
- You're writing a script and need to check whether a process exists
- You want CPU and memory figures for a specific process without opening an interactive tool

✅ **Use `top` or `htop` when:**
- You need real-time CPU or memory rates, not lifetime averages
- You're doing interactive triage — watching a runaway process, checking whether a service stabilizes after a config change
- You want to sort dynamically, kill from the UI, or see per-CPU bars

✅ **Use `pgrep` when:**
- You need the PID of a process by name in a script
- You need to check whether a process is running as a specific user (`pgrep -u www-data php-fpm`)
- You want to signal all matching processes at once with `pkill`

✅ **Use `pstree` when:**
- You need to understand the parent-child hierarchy of a service's worker processes
- You're debugging a process that's spawning unexpected children

✅ **Use `/proc/PID/` directly when:**
- Your tools are broken and you need raw data from the kernel
- You're writing a monitoring tool or script that needs specific fields without parsing `ps` output

❌ **Avoid `ps aux | grep processname` in scripts:**
- It matches the `grep` process itself unless you use `grep -v grep`, which is fragile
- Use `pgrep -x processname` instead — it's cleaner and exact-match safe

❌ **Avoid acting on D-state processes with `kill`:**
- It won't work, and repeatedly trying it wastes time you should spend diagnosing the I/O problem

❌ **Avoid using `ps` %CPU as the sole measure of a "hot" process:**
- Use `top` or `pidstat` for current CPU rates when making performance decisions

⚠️ **Common mistakes:**
- Running `kill $(ps aux | grep nginx | awk '{print $2}')` — this kills the grep and awk processes too if they match; use `pgrep -x nginx` instead
- Confusing VSZ (virtual size) with actual memory usage; RSS is the real number
- Assuming a process with no `ps` entry is stopped — it might have changed PID since you looked

## Veteran Perspective

The first thing I do on any sick system is `ps aux | grep -E 'D|Z'`. Not `top`, not logs — that filter first. D-state accumulation is the single most reliable signal that you have an I/O problem, not a CPU problem. Sysadmins who don't know this burn an hour staring at CPU graphs while the real problem is an NFS mount that went stale six hours ago.

The second misconception I see constantly is treating `ps aux` %CPU like it's `top`'s CPU percentage. It isn't. I've watched engineers declare a process "fine" because `ps aux` showed 0.3% CPU, while `top` was showing it at 98% at that very moment. The process had just started spiking. The lifetime average was still nearly zero. Always use `top` or `htop` when you're asking "is this process hot right now."

The grep-through-ps pipeline — `ps aux | grep myprocess | grep -v grep | awk '{print $2}'` — is a code smell. Every time I see it in a script, I know the author learned process management from a 2005 tutorial and never revisited it. Replace it with `pgrep`. It's purpose-built, handles exact matching, filters by user, and doesn't match itself.

Here's what an intermediate sysadmin misses: zombie processes are almost never the problem — they're symptoms of the problem. Seeing `Z` in your `ps` output means some parent process has a bug. Killing the zombies does nothing; they'll regenerate. The right response is to find the parent (`ps -ef | grep Z_PID` then look at PPID) and investigate whether that parent is healthy. If the parent is gone, the zombie will be reparented to init and reaped automatically.

For technical interviews, the question "explain D state and why you can't kill it" separates candidates who memorized process state letters from candidates who understand how the kernel's scheduler and I/O subsystem interact. D state is a kernel guarantee. The kernel made a promise to the process that it would not be interrupted during a specific kernel-side I/O operation. Signals are delivered when a process returns to user space. A D-state process never returns to user space — so the signal waits forever. The only exit is for the I/O to complete (or fail catastrophically). That's not just a Linux trivia fact — it's a window into how the kernel balances safety, correctness, and responsiveness.

One war story: we had a monitoring alert for "load average above 30" firing intermittently. The on-call engineer each time ran `top`, saw normal CPU usage, saw no runaway processes, and dismissed it as a false alarm. Three weeks later, the system became unresponsive for 20 minutes. `ps aux` showed 40 processes in D state — all waiting on a SAN volume whose network path was intermittently flapping. Load average counts D-state processes, not just R-state ones. The elevated load average was the correct signal. The engineer was looking at the wrong column. Know what load average actually measures: it counts processes in R *and* D. A load average spike with no CPU load means blocked I/O.

## Try It Yourself

1. Run `ps aux` on your system. Find the process with the highest RSS (physical memory). What is it? Is that what you expected?

2. Run `ps -ef --forest | head -40`. Trace the PPID chain from your shell session back to PID 1. How many steps?

3. Run `pgrep -la ssh`. Now run `pgrep -xa ssh`. Compare the results. Which processes disappear when you add `-x`? Why?

4. Find every process owned by root that is currently in interruptible sleep: `ps aux | awk '$1=="root" && $8~/^S/ {print $2, $11}' | head -20`. Pick one PID and read `/proc/PID/status`. Match the `State:` line to what `ps` showed.

5. **Challenge (requires a terminal with job control):** Open two terminals. In terminal 1, start a long-running process: `sleep 600 &`. Then suspend it with `kill -STOP $!`. In terminal 2, find it with `pgrep -la sleep` and verify its state is `T` with `ps aux`. Now resume it from terminal 2 using `kill -CONT <PID>` and watch the state change back to `S`. Then kill it cleanly.

## Summary

- `ps aux` shows all processes with CPU/memory; `ps -ef` adds PPID and uses SysV syntax — use `ps -ef` when parent-child relationships matter, `ps aux` when resource consumption matters.
- Process state D (uninterruptible sleep) cannot be killed with SIGKILL — it indicates a blocked kernel I/O operation; accumulation of D-state processes means an I/O problem, not a CPU problem.
- `%CPU` in `ps aux` is a lifetime average since process start — use `top` or `htop` for instantaneous CPU rates.
- `pgrep -x processname` is the correct way to find PIDs in scripts; avoid the `ps | grep | grep -v grep | awk` pipeline.
- All process viewing tools read `/proc` — the data is a snapshot, PIDs recycle, and the process you see may be gone before you act on it.
