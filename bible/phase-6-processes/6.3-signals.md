# [6.3] — Signals

## The Problem This Solves

You need to restart a daemon — not kill it outright, just tell it to reload its configuration file. The daemon is busy. It can't accept a TCP connection right now. There's no socket you can connect to and type "reload." How do you communicate with a running process?

Or you have a runaway process eating 100% CPU. You want to stop it. But you also want to give it a chance to clean up — close its files, flush its write buffers, remove its lock files. If you just yank its memory away, you might leave the system in an inconsistent state. How do you ask it to stop nicely before you force it?

Or you're in a shell and you type a command, and it hangs. You press Ctrl+C. The command stops. But why? You didn't close the terminal. You didn't type "exit." Something reached into that running process and interrupted it. What was that thing?

All of these problems are solved by the same mechanism: signals. Signals are asynchronous notifications the kernel delivers to processes — interruptions that a process can choose to handle gracefully, ignore, or (in two special cases) cannot refuse.

---

## Mental Model

Imagine a postal service that delivers physical knocks on a door instead of letters. Most knocks mean something standard — "there's a package" (SIGTERM: please exit), "the building is on fire" (SIGKILL: you're being terminated right now, no argument). You can choose to answer some knocks however you want, or not answer them at all. But two of the knock types — the fire marshal's knock (SIGKILL) and the emergency stop order (SIGSTOP) — bypass your door entirely. They don't wait for you to answer. The fire marshal walks in through the back, and the process is stopped whether it likes it or not.

The key word is **asynchronous**. The knock can arrive at any point — in the middle of a sentence, mid-calculation, deep inside a function. The process can deal with it, or it arrives and the process is simply gone.

## Theory

A signal is a small integer sent to a process (or process group) by the kernel, another process, or the process itself. The integer is a signal number. The number maps to a name — SIGTERM, SIGKILL, SIGHUP — but under the hood it's just a number. On Linux x86_64 there are 31 standard signals (numbers 1–31) and 32 real-time signals (numbers 34–64, discussed below).

Signals are asynchronous in the truest sense: the sender does not block waiting for the receiver to handle the signal. The kernel records that the signal is pending and returns control to the sender immediately. The receiver handles it whenever it next returns from kernel space to user space — which might be nanoseconds later or, if the process is in uninterruptible sleep (state D), possibly not for a while.

### Signal Dispositions

Every signal has a **disposition** — an instruction to the kernel about what to do when that signal is pending. There are three possible dispositions:

**SIG_DFL — Default action.** The kernel performs a built-in action for that signal. For most signals the default action is to terminate the process. For SIGSTOP and SIGTSTP it's to suspend the process. For SIGCHLD and SIGURG it's to ignore the signal silently. For SIGQUIT and SIGSEGV and several others it's to terminate *and* produce a core dump.

**SIG_IGN — Ignore.** The signal is silently discarded. The process never sees it. SIGKILL and SIGSTOP cannot be set to SIG_IGN — the kernel refuses the `sigaction()` call. Every other signal can be ignored.

**Custom handler.** The process installs a function with `sigaction(2)`. When the signal arrives, the kernel interrupts the process, runs the handler function, and (if the handler returns) resumes the original execution at the point it was interrupted. SIGKILL and SIGSTOP cannot have custom handlers — the kernel refuses those too.

The signals you'll use most often in practice are SIGTERM (15), SIGKILL (9), SIGHUP (1), SIGINT (2), SIGSTOP (19), and SIGCONT (18). The rest show up in specific contexts — SIGCHLD in process management code, SIGPIPE in anything that writes to pipes or sockets, SIGUSR1/SIGUSR2 in daemons that use them for custom actions.

### What SIGHUP Actually Means Today

SIGHUP's original meaning was "the terminal connection dropped" — back when processes always had a controlling terminal, a hangup meant the user's serial line went dead and the process had no reason to continue. Most processes had a default action of terminating.

Daemons don't have controlling terminals. So SIGHUP was repurposed by convention: send SIGHUP to a daemon and it should re-read its configuration files and restart gracefully without dropping its connections. This is not a kernel guarantee — it's a convention the daemon author chose to implement. `nginx`, `sshd`, `rsyslogd`, and most well-behaved daemons follow it. When you run `systemctl reload nginx`, systemd is sending SIGHUP to nginx.

### SIGSTOP vs. SIGTSTP

Both stop a process. The difference is in whether the process can resist.

SIGSTOP (19) is an unconditional stop from the kernel's perspective. It cannot be caught, blocked, or ignored. No call to `sigaction()` or `sigprocmask()` will intercept it. The process simply stops. It is used by debuggers (`ptrace`) and by programs that manage process groups.

SIGTSTP (20) is the "soft" version — the one generated when you press Ctrl+Z in a terminal. It has the same default action (stop the process), but it can be caught, blocked, or ignored. A program that needs to clean up before stopping — like a terminal application that has put the terminal in raw mode — can catch SIGTSTP, restore the terminal, and then raise SIGSTOP to actually stop itself. This two-step is common in programs like `vim` and `less`.

### SIGUSR1 and SIGUSR2

These two signals have no predefined meaning. They exist specifically so daemons and applications can invent their own meanings without conflicting with the kernel's signal table. Common uses:

- Apache HTTPD: SIGUSR1 triggers a graceful restart
- rsync: SIGUSR1 triggers mid-run statistics output
- Some programs: SIGUSR1 dumps internal state to a log file
- Go programs: SIGUSR1 is sometimes used to trigger garbage collection diagnostics

If you're writing a daemon and you need a "do something custom" signal, SIGUSR1 and SIGUSR2 are the right choices. Don't repurpose SIGHUP unless you actually want reload-config semantics.

### Real-Time Signals

Beyond signal 31, Linux provides real-time signals. On the kernel side, these are signals 32 through 63. At the glibc/application level, you use the constants SIGRTMIN and SIGRTMAX.

Here's the wrinkle: glibc's NPTL threading library internally uses signals 32 and 33 for thread management. So the value of `SIGRTMIN` at runtime (what you get when you call the macro or run `kill -l`) is **34**, not 32. `SIGRTMAX` is 64. That gives user programs 31 usable real-time signals.

Real-time signals differ from standard signals in two important ways. First, they queue: if you send the same real-time signal three times before the process handles it, it will be delivered three times in order. Standard signals 1–31 only set a bit — if SIGTERM is already pending and you send another SIGTERM, only one will be delivered. Second, real-time signals carry a payload via `sigqueue(3)` — a small integer or pointer value that arrives alongside the signal in the `siginfo_t` structure.

Do not hardcode the number 34 for SIGRTMIN. Use `SIGRTMIN` and `SIGRTMIN+n` and always verify `n <= (SIGRTMAX - SIGRTMIN)`.

## How It Works (Mechanics)

### How the Kernel Records a Pending Signal

Inside the kernel, each process (or more precisely, each thread) has a `task_struct`. Two fields in `task_struct` are central to signals:

```text
# Conceptual view of signal-related fields in task_struct
task_struct {
    struct sigpending   pending;       # per-thread pending signals
    sigset_t            blocked;       # blocked signal mask (set by sigprocmask)
    struct sighand_struct *sighand;    # the signal handler table
    struct signal_struct  *signal;     # process-wide state, including shared_pending
}
```

`struct sigpending` contains two things: a linked list of `sigqueue` entries (one per queued signal instance), and a `sigset_t` bitmap with one bit per signal number. The bitmap is the fast path. When you send SIGTERM, the kernel sets bit 15 in the bitmap. It does not set it again if it's already set — that's why a standard signal can only be pending once no matter how many times you send it. Real-time signals always add a new entry to the linked list, which is how they achieve queuing.

There are actually two pending queues: `task->pending` for signals targeted at a specific thread (via `tkill(2)`), and `task->signal->shared_pending` for signals sent to the process as a whole (via `kill(2)` to a PID).

### How the Kernel Notifies the Process

When a signal is sent, the kernel calls `signal_wake_up()` on the target thread. This sets the `TIF_SIGPENDING` flag — a single bit in the thread's `thread_info->flags`. If the thread is sleeping (interruptible sleep, state S), it's also woken up.

The `TIF_SIGPENDING` flag is the fast path. The kernel checks this single bit at every transition from kernel mode back to user mode — after every system call return, after every hardware interrupt, after every page fault. It does not scan the full pending signal list every time; it only does that if `TIF_SIGPENDING` is set.

### How the Signal Is Actually Delivered

When the kernel sees `TIF_SIGPENDING` is set and is about to return to user space, it calls `arch_do_signal_or_restart()`. The rough sequence:

```text
# Conceptual signal delivery path (not real code — illustrative)
arch_do_signal_or_restart(regs)
  → get_signal(&ksig)
        # dequeues the signal from pending list
        # skips signals that are in task->blocked (the signal mask)
  → handle_signal(&ksig, regs)
        # sets up user-space stack frame:
        #   - saves current CPU registers (pt_regs) onto user stack
        #   - copies siginfo_t and ucontext_t onto user stack
        #   - redirects instruction pointer (RIP) to the signal handler
        #   - sets return address to the sigreturn trampoline (in vDSO)
  → returns to user space — but now executing the signal handler

  [signal handler runs]

  → handler calls sigreturn (via vDSO __kernel_rt_sigreturn trampoline)
  → sys_rt_sigreturn() restores saved pt_regs
  → process resumes at the point it was interrupted
```

The sigreturn trampoline lives in the vDSO — a small kernel-managed shared library mapped into every process. This is why signal handlers can return cleanly even though they were invoked from an unusual point in the call stack.

### Why SIGKILL and SIGSTOP Cannot Be Caught

The kernel enforces this unconditionally in `get_signal()`. Before looking up the signal handler, the kernel checks if the signal is SIGKILL or SIGSTOP. If it is, the handler table is not consulted at all. SIGKILL causes immediate process termination. SIGSTOP forces the task state to TASK_STOPPED. No `sigaction()` call you make will ever register a handler that gets invoked, because the code path that would invoke it is bypassed entirely.

This is not a policy decision — it's a kernel invariant. Without SIGKILL, a process could intercept every termination request and run forever. Without SIGSTOP, debuggers and process supervisors couldn't reliably pause a process. The kernel needs these guarantees to function correctly.

### Blocked Signals and the Signal Mask

Before a signal is dequeued and delivered, the kernel checks whether the signal is in `task->blocked` — the signal mask. If it is, the signal stays pending and is not delivered until the mask is changed via `sigprocmask(2)`. This is how signal handlers protect themselves from being re-entered: the kernel automatically adds the current signal to the signal mask for the duration of the handler.

SIGKILL and SIGSTOP cannot be blocked. Any attempt to add them to the blocked mask is silently ignored by the kernel.

Now send a signal. Start a long-running process, find its PID, and send it SIGTERM:

```bash
# Start a process that does nothing but sleep for a very long time
sleep 600 &

# Capture the PID
SLEEP_PID=$!
echo "sleep is running as PID $SLEEP_PID"

# Send SIGTERM — the default signal for `kill`
kill $SLEEP_PID

# Verify it's gone — should print nothing or "no such process"
sleep 1
kill -0 $SLEEP_PID 2>&1 || echo "Process is gone"
```

```text
sleep is running as PID 47392
Process is gone
```

`kill -0` sends no signal — it just tests whether the process exists and you have permission to signal it. It's a clean way to check if a process is still alive.

Try this yourself. Run `sleep 600 &`, then try `kill -SIGTERM $!`, then `kill -15 $!`, then `kill -TERM $!`. They're all identical — three ways to write the same thing. If your output differs (e.g., you see "Terminated" printed by the shell), that's your shell reporting the process's exit reason — normal behavior.

## Real-World Example

This is the standard pattern for safely stopping a process on a production system: try SIGTERM, wait, and only escalate to SIGKILL if the process doesn't exit.

```bash
#!/bin/bash
# Graceful shutdown script: SIGTERM first, SIGKILL as fallback

TARGET_PID=$1

if [ -z "$TARGET_PID" ]; then
    echo "Usage: $0 <pid>" >&2
    exit 1
fi

# Verify the process exists before doing anything
if ! kill -0 "$TARGET_PID" 2>/dev/null; then
    echo "PID $TARGET_PID does not exist or no permission"
    exit 1
fi

echo "Sending SIGTERM to PID $TARGET_PID..."
kill -TERM "$TARGET_PID"

# Wait up to 10 seconds for the process to exit on its own
WAITED=0
while kill -0 "$TARGET_PID" 2>/dev/null; do
    if [ "$WAITED" -ge 10 ]; then
        echo "Process did not exit after 10 seconds. Sending SIGKILL..."
        kill -KILL "$TARGET_PID"
        break
    fi
    sleep 1
    WAITED=$((WAITED + 1))
done

# Final check
if kill -0 "$TARGET_PID" 2>/dev/null; then
    echo "WARNING: Process $TARGET_PID still exists after SIGKILL"
    exit 2
else
    echo "Process $TARGET_PID has exited"
fi
```

The walkthrough: the script first checks that the process exists and you have permission to signal it (the `kill -0` probe). Then it sends SIGTERM and enters a polling loop, checking every second whether the process is still alive. After 10 seconds it escalates to SIGKILL. After SIGKILL it does one final check — if the process is somehow still there after SIGKILL, something is very wrong (D-state, zombie, or the PID was reused).

This is a simplified version of what `systemctl stop` does. systemd's `TimeoutStopSec` setting controls how long it waits before escalating from SIGTERM to SIGKILL.

Here's a second real-world pattern: using `pkill` and `trap` together in a script that handles signals cleanly.

```bash
#!/bin/bash
# Background worker script that cleans up on exit

LOCKFILE="/tmp/worker-$$.lock"
TMPDIR="/tmp/worker-$$"

cleanup() {
    echo "Caught signal — cleaning up..."
    rm -f "$LOCKFILE"
    rm -rf "$TMPDIR"
    echo "Cleanup complete"
    exit 0
}

# Register the cleanup handler for these signals
# EXIT fires when the script exits for any reason (including normal exit)
trap cleanup SIGTERM SIGINT SIGHUP EXIT

mkdir -p "$TMPDIR"
touch "$LOCKFILE"

echo "Worker $$ started. Lock: $LOCKFILE"

# Simulate doing work
while true; do
    echo "Working..." >> "$TMPDIR/output.log"
    sleep 5
done
```

The `trap` command registers a handler for the named signals. `EXIT` is a pseudo-signal specific to bash — it fires when the script exits for any reason, including a normal `exit` call. This pattern is the right way to ensure cleanup happens whether the script is killed, interrupted, or exits on its own. Run this in one terminal, then `kill $(pgrep -f worker)` from another. You'll see the cleanup message before the script exits.

## ⚠️ Gotchas

### `kill -9` as the First Instinct

Every junior sysadmin's first instinct when a process is misbehaving is `kill -9`. It's the nuclear option. It always works. And it's wrong as a first move.

Here's what SIGKILL actually does: the kernel terminates the process immediately, without giving it any opportunity to run code. There is no cleanup. Open file descriptors are left without a proper close. Write buffers that haven't been flushed to disk are discarded. Lock files the process created are not removed. Database connections are left dangling. Temporary files accumulate. The process simply stops existing.

In a world where processes try to be well-behaved, this creates real problems. A web server killed with SIGKILL might leave a socket in TIME_WAIT that blocks a restart. A database server killed with SIGKILL will need to run crash recovery on the next start. A script killed with SIGKILL leaves its lock files behind, which means the next invocation fails immediately with "already running."

The correct pattern is always SIGTERM first:

```bash
# Wrong — SIGKILL as the first move
kill -9 1234

# Right — give the process a chance to clean up
kill -TERM 1234
sleep 5
# Check if it's still alive before escalating
kill -0 1234 2>/dev/null && kill -KILL 1234
```

SIGTERM tells the process "please exit." A well-written process catches SIGTERM, finishes what it's doing, flushes its buffers, closes its connections, removes its temp files, and exits with a proper exit code. If it doesn't exit within a reasonable time, then escalate to SIGKILL. This is the same pattern systemd uses.

The one legitimate case for jumping straight to SIGKILL: the process has already crashed, is in an unresponsive state, and you have confirmed it isn't in the middle of a write operation you care about. Even then, consider whether SIGTERM has been tried.

### SIGKILL Doesn't Work on D-State Processes

You found a zombie-like process eating resources. You've tried SIGTERM. Nothing. You escalated to SIGKILL. Still nothing. The process is still there. You check the state with `ps aux` and see `D` in the STAT column.

```bash
ps aux | grep myprocess
```

```text
root     12345  99.9  0.1 12345  1234 ?  D    14:00   5:23 myprocess
```

D-state (uninterruptible sleep) means the process is waiting for an I/O operation to complete inside the kernel. The kernel is actively protecting this operation from being interrupted — because interrupting a kernel I/O path mid-flight could corrupt filesystem data structures. SIGKILL cannot reach a D-state process. Not because SIGKILL is insufficient, but because the kernel is not delivering signals to this thread while it's inside the I/O path. The signal will be delivered, but only after the process returns from the kernel I/O wait.

This means that if the I/O never completes — a stuck NFS mount, a hardware failure, a storage controller that stopped responding — the process is stuck forever. No amount of `kill -9` will help.

```bash
# D-state processes require fixing the underlying I/O problem
# Check if it's NFS-related
ls -la /proc/12345/fd | head -20     # what files does it have open?
cat /proc/12345/wchan                 # what kernel function is it sleeping in?
```

The fix is at the hardware or filesystem layer: unmount the stuck NFS share, reset the storage controller, or (last resort) reboot. `kill -9` is not the fix here.

### SIGPIPE Is Silently Swallowed by Many Programs

This one causes endless confusion with shell pipelines.

When a process writes to a pipe and the read end of that pipe has been closed, the kernel sends SIGPIPE to the writer. The default action is termination. So in `cat bigfile | head -n 5`, when `head` reads 5 lines and exits, `cat`'s next write attempt triggers SIGPIPE and `cat` is terminated. This is correct behavior — without SIGPIPE, `cat` would write into a pipe nobody is reading, forever.

The problem is that many programs — particularly network servers, Python scripts, and anything that uses certain libraries — install `SIG_IGN` for SIGPIPE during startup. When a program ignores SIGPIPE, the write system call doesn't kill the process; instead it returns -1 with `errno = EPIPE`. The program may or may not check for EPIPE. If it doesn't, writes to the closed pipe simply return errors that are silently discarded, and the program keeps running as if nothing happened.

This is why you sometimes see "Broken pipe" printed to stderr (the program detected the EPIPE and printed a message) and sometimes you see nothing (the program died silently from SIGPIPE), and sometimes the pipeline seems to hang (the program ignored both SIGPIPE and EPIPE and is writing into the void):

```bash
# This will often show "Broken pipe" because python ignores SIGPIPE by default
python3 -c "import sys; [sys.stdout.write('x'*1000+'\n') for _ in range(10000)]" | head -n 3
```

```text
xxxxxxx...
xxxxxxx...
xxxxxxx...
Broken pipe
```

The "Broken pipe" message is Python detecting the EPIPE error from its write, not the signal itself being announced. The fix in scripts is to either set SIGPIPE back to SIG_DFL or to explicitly check write return values for EPIPE and handle it.

### Sending to a Process Group — The Double Dash Trap

You want to send SIGTERM to every process in process group 4567. You know that negative PIDs mean process groups in the `kill(2)` syscall. So you try:

```bash
# This does NOT send to process group 4567
kill -4567
```

```text
bash: kill: -4567: invalid signal specification
```

The shell's `kill` builtin (and the standalone `/bin/kill`) parses arguments left-to-right. When it sees `-4567`, it interprets this as "signal 4567" — not as a negative PID. Since there's no signal 4567, you get an error.

The fix is the `--` end-of-options marker:

```bash
# Double dash tells the parser: everything after this is a positional argument, not an option
kill -- -4567            # sends SIGTERM to process group 4567

# Alternatively, if you specify the signal explicitly, -- is not needed
# because the signal argument is already consumed and the -PID is unambiguous
kill -TERM -4567         # also works — TERM is the signal, -4567 is the target
kill -15 -4567           # also works
```

The reason `--` works: after `--`, argument parsing switches to positional-only mode. `-4567` is now treated as a positional argument to `kill`, and `kill` knows that a positional argument that starts with `-` is a negative PID (process group). The underlying kernel call in all cases above is `kill(-4567, SIGTERM)`.

### `trap` Handlers Are Not Inherited by Subshells

You've written a script that sets a trap for SIGTERM to do cleanup. You test it. The cleanup runs correctly. You deploy it. In production, the cleanup silently doesn't happen.

The difference: in production, some part of the script is running inside a subshell created with `$()` command substitution. Signal traps set in the parent shell are **not** inherited by subshells.

```bash
trap 'echo "Caught SIGTERM"' SIGTERM

# This runs in a subshell — trap is NOT inherited
result=$(
    sleep 10   # If SIGTERM is sent to the parent, this subshell does NOT
               # run your trap handler — it has its own clean signal table
)

# This runs in the current shell — trap IS active
sleep 10
```

The rule: each subshell starts with all signal traps reset to their default actions. It copies the parent's signal masks (blocked signals), but not the parent's signal handlers. If you need a subshell to handle signals, you must re-register the trap inside the subshell.

For cleanup purposes, the reliable approach is to register the trap in the main shell body and keep the logic that needs protecting out of subshells — or run cleanup logic that operates on external state (files, PIDs) that's visible to the main shell regardless of where the work happened.

## Distro Notes

Signal numbers 1–31 are standardized across all Linux architectures except MIPS, Alpha, and SPARC, which have a few differences in the upper range. On the common architectures — x86_64, ARM64, ARM32, RISC-V — the table above is accurate.

| Distro Family | Notes |
|---|---|
| All glibc-based (Debian, Ubuntu, RHEL, Fedora, Arch) | SIGRTMIN = 34 at runtime; signals 32–33 reserved by NPTL |
| Alpine (musl libc) | SIGRTMIN = 35; musl reserves one additional signal internally |
| Any (kernel only, no glibc — e.g., bare C with custom libc) | SIGRTMIN = 32; glibc reservation does not apply |

The `kill -l` output on a musl-based Alpine container will show SIGRTMIN starting at 35. This matters if you're writing shell scripts that rely on RT signals and need to run on both glibc and musl environments.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `kill` | Send signal by PID; exact control | Requires knowing the PID; works with negative PIDs for process groups |
| `pkill` | Send signal by process name pattern | Supports regex, `-u user`, `-t terminal`; sends to all matching processes |
| `killall` | Send signal by exact process name | Matches exact name only (not a pattern); on Linux, requires `psmisc` package |
| `kill -0` | Test if process exists / check permission | Sends no signal; just checks; useful in scripts for polling |

Reach for `pkill` when you know the program name and don't have the PID handy — it's faster than `ps aux | grep ... | awk '{print $2}' | xargs kill`. Use `kill` when you need exact control, are working from a PID file, or are sending to a process group. Avoid `killall` on Linux — on macOS and BSD it behaves differently (it kills all processes, period) and the name causes confusion when scripts are shared across systems.

---

## Trade-offs

**Where this shines:**

- Signals are the only mechanism for sending out-of-band notifications to a process without modifying its code or protocol — a process that has no TCP port, no socket, no API, and no IPC queue can still receive a signal
- SIGKILL is a hard guarantee: if the kernel can deliver it, the process will stop. There is no way to write a process that survives SIGKILL (barring kernel bugs)
- Signals are lightweight — sending a signal is a single `kill(2)` syscall, no data copying, no socket overhead

**Where this costs you:**

- SIGKILL leaves no cleanup: open file write buffers may not be flushed, lock files are not removed, database connections are left dangling, and temporary files accumulate; always try SIGTERM first and allow time for cleanup
- Signals are asynchronous and stateless: a signal carries no payload beyond its number (for standard signals), and if the process is in D-state or has SIGPIPE set to SIG_IGN, the signal may have no visible effect until you dig into what's happening
- Signal handlers are restricted in what they can safely do: only async-signal-safe functions (the short list in `signal-safety(7)`) can be called from a signal handler; calling `malloc`, `printf`, or locking a mutex from a signal handler can cause deadlocks
- `trap` in bash applies only to the current shell context, not subshells — cleanup logic that relies on traps is fragile in complex scripts

---

## When to Use / When to Avoid

✅ **Use this when:**

- You need to stop a process — SIGTERM first, wait, SIGKILL as last resort
- You need a daemon to reload its config without dropping connections — SIGHUP (if the daemon supports it)
- You're writing a daemon and need to expose "do something" hooks to external callers — SIGUSR1/SIGUSR2
- You're writing a shell script that creates temp files or lock files — `trap cleanup EXIT SIGTERM SIGINT`
- You need to stop an entire process group at once — `kill -- -PGID`

❌ **Avoid this when:**

- You need reliable two-way communication with a process — signals are one-way and carry no reply; use a socket, pipe, or shared memory instead
- You need to pass data along with the notification — standard signals carry only their number; use a socket or pipe
- The process might be in D-state — `kill -9` won't immediately work; investigate and fix the underlying I/O problem

⚠️ **Common mistakes:**

- Using `kill -9` as the default — always try SIGTERM first and give the process a grace period
- Sending signals to PID 1 without knowing what it is — on a container, PID 1 is your entrypoint; on the host, it's init/systemd; both treat SIGTERM differently from other processes
- Forgetting that `pkill` matches substrings by default — `pkill python` will kill every Python process on the system, including your own shell if it's using Python for something; use `-x` for exact matches or be more specific with the pattern
- Sending SIGHUP to a process assuming it will reload — not all daemons implement SIGHUP as reload; check the man page first

## Veteran Perspective

The most common signal mistake I see isn't `kill -9` as a first move — though that's wrong too. It's the absence of signal handling in scripts. I've watched cleanup scripts leave hundreds of lock files behind because nobody put `trap cleanup EXIT` at the top. The next run of the script fails with "already running" and someone spends twenty minutes figuring out why before they notice the stale lock file.

The second most common mistake: `kill -9` on a process that's in D-state, followed by the assumption that it didn't work because something is wrong with the kernel. D-state processes are not ignoring your SIGKILL — the signal is pending, waiting to be delivered when the process comes back from the kernel I/O wait. If that wait never completes, the process stays stuck. `kill -9` is not the fix. Looking at `/proc/PID/wchan` to see what kernel function it's sleeping in, and then figuring out why that I/O never returned, is the fix.

The SIGPIPE behavior in pipelines trips up people who come from Windows or higher-level languages. In a shell pipeline, the left side of the pipe getting killed by SIGPIPE is correct behavior — it means "the reader has gone away, there's no point in continuing." An exit code of 141 (128 + 13) means the process was killed by SIGPIPE. This is expected. If you're seeing lots of "Broken pipe" messages in logs from Python services, it means the service's SIGPIPE is set to SIG_IGN and the code isn't handling EPIPE returns from write calls — that's a code quality issue, not a signal issue.

In technical interviews at the senior SRE or systems level, signals come up in one of two ways: "how would you gracefully restart a service without dropping connections?" (answer: SIGHUP or a mechanism that sends SIGTERM, drains, and restarts) or "you have a process that won't die — walk me through your diagnostic approach." The correct answer to the second is: check the process state (`ps` or `cat /proc/PID/status`), check if it's D-state, check what it's waiting on (`/proc/PID/wchan`, `strace -p PID`), check if SIGKILL is actually pending (`cat /proc/PID/status | grep SigPnd`). "I'd just wait and reboot" is not a senior answer.

One thing that doesn't get enough attention: process group signals. When a shell job runs, the shell creates a process group for it. Pressing Ctrl+C in the terminal sends SIGINT to the entire process group — not just the foreground process. Every process that was forked as part of that job gets the signal. This is why killing a complex pipeline with Ctrl+C actually kills all its stages and not just the rightmost one. When you need to terminate a multi-process job programmatically, `kill -- -PGID` is the right tool — not iterating through individual PIDs, which is racy (new processes might have forked between your list and your kill loop).

## Try It Yourself

1. Run `kill -l` and count the gap between signal 31 and the real-time signals. Note where SIGRTMIN starts on your system. Try `( kill -l )` (in a subshell) and compare — the output should be identical.

2. Start `sleep 600 &`, note its PID, send it SIGSTOP, verify it's in T state with `ps -p $! -o stat`, then send SIGCONT, verify it returns to S state, then kill it with SIGTERM.

3. Open two terminals. In the first, run `cat /dev/urandom | head -c 50 | xxd`. In the second, find the PID of `cat` in that pipeline and run `kill -PIPE <pid>`. Observe the exit status with `echo $?` in the first terminal — it should be 141.

4. Write a script that creates a temp directory, starts a background `sleep` inside it, registers a `trap` to remove the temp directory on EXIT and SIGINT, then does `sleep 30`. Send SIGINT to the script from another terminal. Verify the temp directory is actually cleaned up.

5. **Challenge:** Write a script that spawns 5 background `sleep 60` processes, captures all their PIDs, puts them all in a single process group using `setsid` or by examining the shell's process group, then sends SIGTERM to the entire group at once using `kill -- -PGID`. Verify all 5 processes are gone after a 2-second wait. Note: `$BASHPID` gives the current shell's PID; `ps -o pgid= -p $BASHPID` gives its process group ID.

## Summary

- Signals are asynchronous kernel notifications delivered to processes; every signal has a disposition — SIG_DFL (default action), SIG_IGN (ignore), or a custom handler — except SIGKILL (9) and SIGSTOP (19), which cannot be caught, blocked, or ignored under any circumstances
- SIGTERM (15) is a polite termination request; a process can catch it and clean up; always send SIGTERM first, wait for the process to exit, and only escalate to SIGKILL if it doesn't
- SIGKILL kills immediately without cleanup — open buffers, lock files, and connections are left in whatever state they're in; it is not the right first move
- SIGHUP (1) by convention means "reload config" in daemons; Ctrl+C sends SIGINT (2); Ctrl+Z sends SIGTSTP (20) which can be caught, unlike SIGSTOP (19) which cannot
- To send a signal to an entire process group, use `kill -- -PGID` (double dash required so the negative number isn't parsed as a signal flag); to register cleanup in shell scripts, use `trap 'cleanup_function' EXIT SIGTERM SIGINT`
