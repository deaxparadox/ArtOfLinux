## The Problem This Solves

A process with no constraints is a liability. Leave a buggy application running unchecked and it will eventually open ten thousand file descriptors and refuse to accept new connections, fork ten thousand child processes and grind the scheduler to a halt, or allocate memory until the OOM killer starts executing random innocent processes.

Early Unix systems had none of this. An application that misbehaved ÔÇö or that was under unexpected load ÔÇö could drag the entire machine down. There was no way for the OS to say "you have had enough."

Resource limits were the answer. The kernel gained the ability to track how much of each resource a process was consuming, set a ceiling on that consumption, and enforce it ÔÇö returning an error the moment the process tried to exceed its allocation. This protected other processes, protected the system, and gave administrators a knob to tune when legitimate workloads needed more room.

The problem today is not that the mechanism doesn't work ÔÇö it works extremely well. The problem is that most people don't encounter resource limits until something breaks at 2am, and then they reach for the first `ulimit` command they can find without understanding what they're actually changing.

## Mental Model

Think of resource limits like a tab at a bar, run by an unusually careful bartender.

When you walk in, you're given a tab with a soft limit ÔÇö say, five drinks. That's your working limit. You can ask the bartender to raise it up to your pre-approved hard limit of ten, and they'll do it without calling anyone. But you can't go above ten on your own ÔÇö for that, you need the manager (root). Once you leave for the night and come back, you start fresh with the same defaults.

The bartender (the kernel) checks your tab before every pour. If you hit your soft limit, they tell you immediately. If you try to sweet-talk your way above the hard limit without the manager's say-so, you get an `EPERM`.

The key insight: **soft limit is where enforcement happens today; hard limit is the ceiling of where you're allowed to set it yourself.**

## Theory

### Two Layers of Every Limit

Every resource limit in Linux has two values: a **soft limit** and a **hard limit**.

The **soft limit** is the value that the kernel actually enforces right now. When a process tries to open a file and its open file count is at the soft limit, the `open()` syscall returns `EMFILE`. The soft limit is the real ceiling that the running process feels.

The **hard limit** is the maximum value the soft limit is allowed to reach. An unprivileged process can raise its own soft limit freely, up to but not exceeding the hard limit. It cannot raise its hard limit without root (or the `CAP_SYS_RESOURCE` capability). Once an unprivileged process lowers its own hard limit, that reduction is permanent for the lifetime of the process ÔÇö it cannot undo it.

Root and processes with `CAP_SYS_RESOURCE` can set hard limits to any value, bounded only by a separate kernel-level ceiling (`/proc/sys/fs/nr_open` for file descriptors, for example).

### What Gets Limited

The kernel tracks many types of resource. The ones you'll encounter in practice:

- **Open files** (`RLIMIT_NOFILE`) ÔÇö the maximum number of file descriptors a process may have open simultaneously. This includes sockets, pipes, and device files ÔÇö not just regular files.
- **Processes** (`RLIMIT_NPROC`) ÔÇö the maximum number of processes (and threads) the user running the process may have simultaneously. Not per-process ÔÇö per UID.
- **Stack size** (`RLIMIT_STACK`) ÔÇö the maximum size of the process stack. When a program recurses too deeply and blows the stack, this is why.
- **Virtual memory** (`RLIMIT_AS`) ÔÇö the maximum total virtual address space. Setting this unlimited means a process can `mmap()` as much as the kernel will allow.
- **Core dump size** (`RLIMIT_CORE`) ÔÇö how large a core dump file can be. Commonly set to 0 to disable core dumps entirely, or `unlimited` on development systems.
- **CPU time** (`RLIMIT_CPU`) ÔÇö total CPU time in seconds. Rarely used for routine limits; the OOM killer and cgroups are preferred for runaway processes.

### Three Places Limits Live

Understanding where limits come from is the core of diagnosing any limit problem.

**At process creation**, the child inherits a copy of the parent's resource limits. Your shell session inherits limits from PAM when you log in. Processes started by systemd inherit limits from the service unit file (or systemd's global defaults). These are independent pipelines ÔÇö changes in one do not affect the other.

**`/etc/security/limits.conf`** sets limits for PAM login sessions ÔÇö interactive logins via SSH, `su`, `sudo`, and TTYs. PAM reads this file when establishing a session and applies the limits before handing the shell to the user. It has no effect on systemd services.

**systemd unit files** set limits for service processes via `Limit*=` directives in the `[Service]` section. These override systemd's system-wide defaults and have no relationship to `limits.conf`.

**cgroups** provide a third, orthogonal layer. Where `rlimit` limits are per-process and per-user, cgroups apply to groups of processes ÔÇö an entire service and all its children ÔÇö and are managed by systemd for system services.

### The EMFILE / ENFILE Distinction

Two different errors both look like "too many open files," but they are not the same:

**`EMFILE` (errno 24)** ÔÇö the current process has hit its `RLIMIT_NOFILE` limit. Fix: raise the limit for that process or service.

**`ENFILE` (errno 23)** ÔÇö the entire kernel's global file table is exhausted. This is governed by `/proc/sys/fs/file-max`. Fix: raise `fs.file-max` via `sysctl`. This is rare on modern systems ÔÇö you would need the combined open files across every process on the machine to exceed a value typically in the millions.

In practice, almost every "too many open files" incident is `EMFILE`. `ENFILE` in production means the machine is under extraordinary load.

## How It Works (Mechanics)

### rlimit in the Kernel

Every process has a `task_struct` in the kernel. Inside that structure is an array of `rlimit` pairs ÔÇö one `{rlim_cur, rlim_max}` pair for each resource type. `rlim_cur` is the soft limit; `rlim_max` is the hard limit.

When a process calls `fork()`, the child gets a copy of the parent's entire `rlimit` array. When a process calls `execve()`, the limits are preserved across the exec ÔÇö the new program inherits the same limits the old one had.

The `getrlimit(2)` and `setrlimit(2)` syscalls let a process read and modify its own limits. The `prlimit(2)` syscall lets a process (with appropriate privilege) read or set the limits of another process by PID.

When the kernel enforces a limit ÔÇö on a file open, a fork, or a memory allocation ÔÇö it checks `rlim_cur`. If the operation would push the count above `rlim_cur`, the syscall fails with the appropriate errno and the operation never completes.

### How ulimit Talks to the Kernel

`ulimit` is a shell built-in, not an external command. You can verify this:

```bash
type ulimit
```

```text
ulimit is a shell builtin
```

When you run `ulimit -n 65536`, the shell calls `setrlimit(RLIMIT_NOFILE, ...)` on itself. Since it's a built-in, the change applies to the shell's own `task_struct` ÔÇö and therefore to every child process it subsequently forks. But it has no effect on any process already running, and no effect on the shell's parent.

### PAM and limits.conf

When you SSH into a machine or log in at a console, PAM orchestrates authentication and session setup. The `pam_limits` module reads `/etc/security/limits.conf` and calls `setrlimit()` for each matching rule before handing the session to the shell. The shell, and everything the shell starts, inherits those limits.

The PAM module configuration that wires this up is typically in `/etc/pam.d/common-session` (Debian/Ubuntu) or `/etc/pam.d/system-auth` (RHEL):

```text
session    required     pam_limits.so
```

If that line is missing or commented out, `/etc/security/limits.conf` has no effect no matter how correctly it's written.

### /proc/PID/limits

The current limits for any running process are readable without any special tools:

```bash
# View limits for a running process
cat /proc/$(pgrep nginx | head -1)/limits
```

```text
Limit                     Soft Limit           Hard Limit           Units
Max cpu time              unlimited            unlimited            seconds
Max file size             unlimited            unlimited            bytes
Max data size             unlimited            unlimited            bytes
Max stack size            8388608              unlimited            bytes
Max core file size        0                    unlimited            bytes
Max resident set          unlimited            unlimited            bytes
Max processes             15211                15211                processes
Max open files            1024                 524288               files
Max locked memory         65536                65536                bytes
Max address space         unlimited            unlimited            bytes
Max file locks            unlimited            unlimited            locks
Max pending signals       15211                15211                signals
Max msgqueue size         819200               819200               bytes
Max nice priority         0                    0
Max realtime priority     0                    0
Max realtime timeout      unlimited            unlimited            us
```

"Max open files" is `RLIMIT_NOFILE`. "Max processes" is `RLIMIT_NPROC`. The `unlimited` string (not a number) means the resource is unconstrained at the process level.

### cgroups and systemd

Where `rlimit` controls individual processes, **cgroups** (control groups) control collections of processes as a unit. The kernel exposes cgroup state as a filesystem under `/sys/fs/cgroup/`. systemd uses cgroups to group every service and its children into a named hierarchy.

cgroups v2 (the unified hierarchy, stable since Linux 4.5, default on most distros since 2021ÔÇô2022) presents a single tree under `/sys/fs/cgroup/`. Each node in the tree is a directory. Each directory contains control files the kernel reads and writes to enforce limits.

```bash
# See the cgroup tree for a systemd service
systemctl status nginx | grep CGroup

# Look at the raw cgroup files for memory
cat /sys/fs/cgroup/system.slice/nginx.service/memory.max
```

systemd's `[Service]` directives like `MemoryMax=`, `CPUQuota=`, and `TasksMax=` write to these cgroup files when the service starts. They enforce limits on the entire service cgroup ÔÇö all worker processes combined ÔÇö not on individual processes within it.

```bash
# Show all current limits for this shell session
ulimit -a
```

```text

scheduling priority                 (-e) 0

```

The number next to "open files" is your current soft limit. On most systems you'll see 1024. Now look at both soft and hard values for the same limit:

```bash
# -S shows soft limit, -H shows hard limit

```

```text

```

The gap between those two numbers is room you have to work with ÔÇö without root.

```bash
# Raise the soft limit to 65536 in this shell
ulimit -Sn 65536

# Verify
ulimit -Sn
```

```text
65536
```

Try this yourself. Notice that if you open a second terminal, it still shows 1024 ÔÇö your change only affected the shell where you ran it. That's the thing to understand: `ulimit` changes are local to the current shell and its children.

## Real-World Example

A Node.js application server starts logging this error and eventually stops accepting connections:

```text
Error: EMFILE: too many open files, open '/var/app/logs/request.log'
    at Object.openSync (node:fs:585:3)
```

Here is the diagnostic sequence a senior sysadmin runs:

```bash
# Step 1: find the process ID
pgrep -a node
```

```text
14823 node /var/app/server.js
```

```bash
# Step 2: check what limits the process is actually running under
# This is ground truth ÔÇö not what you think you set, what the kernel sees
cat /proc/14823/limits | grep -E "open files|processes"
```

```text
Max open files            1024                 524288               files
Max processes             15211                15211                processes
```

Soft limit of 1024. With a busy Node.js server holding open log files, database connections, and client sockets, hitting 1024 is entirely plausible.

```bash
# Step 3: verify by counting actual open file descriptors
# lsof -p PID lists every open file; wc -l counts them
lsof -p 14823 | wc -l
```

```text
1021
```

Three away from the limit. Confirmed.

```bash
# Step 4: understand WHAT is consuming the file descriptors
# This tells you if it's a leak or legitimate load
lsof -p 14823 | awk '{print $5}' | sort | uniq -c | sort -rn | head -10
```

```text
    847 IPv4
     89 REG
     43 FIFO
     21 DIR
      8 unix
      5 CHR
```

The vast majority are IPv4 sockets ÔÇö legitimate client connections under load, not a leak. The fix here is to raise the limit. If it had been hundreds of open regular files (REG) accumulating, that would indicate a file descriptor leak requiring a code fix, not a limit raise.

Now the fix. This is a systemd-managed service:

```bash
# Edit the service unit (creates an override, doesn't touch the original)
sudo systemctl edit node-app
```

Add to the override file:

```ini
[Service]
LimitNOFILE=65536
```

```bash
# Reload systemd and restart the service
sudo systemctl daemon-reload
sudo systemctl restart node-app

# Verify the new process got the right limit
cat /proc/$(pgrep node)/limits | grep "open files"
```

```text
Max open files            65536                65536                files
```

The walkthrough matters here: `systemctl edit` creates a drop-in override file at `/etc/systemd/system/node-app.service.d/override.conf` rather than modifying the original unit. This survives package upgrades. `daemon-reload` tells systemd to re-read the unit definitions. The restart applies the new limits to a fresh process ÔÇö there is no way to change limits on an already-running process without restarting it.

For an interactive user who needs a persistent limit increase (not a service), the right file is `/etc/security/limits.conf`:

```text
# /etc/security/limits.conf
# domain    type    item    value
deploy      soft    nofile  65536
deploy      hard    nofile  65536
```

The `deploy` user must log out and back in for this to take effect. The limits are applied by PAM at session creation ÔÇö there is no way to apply them to an already-running session.

## ÔÜá´©Å Gotchas

### ulimit Changes Only Affect the Current Shell

The mistake: running `ulimit -n 65536` in a terminal and assuming the application server on the same machine will benefit from it.

```bash
# You run this in your SSH session
ulimit -n 65536

# Then you check the running nginx process
cat /proc/$(pgrep nginx | head -1)/limits | grep "open files"
```

```text
Max open files            1024                 524288               files
```

Unchanged. `ulimit` calls `setrlimit()` on the shell process itself. It does nothing to any already-running process. The nginx master process ÔÇö started by systemd weeks ago ÔÇö has its own limits set at the time it was launched. The only way to change a running process's limits is `prlimit(1)` (requires privilege) or restarting the process with the new limits already configured.

```bash
# The right fix: change nginx's systemd unit and restart it
sudo systemctl edit nginx
# Add: [Service]\nLimitNOFILE=65536
sudo systemctl restart nginx
```

### limits.conf Has No Effect on systemd Services

This is the most common misunderstanding in production. When someone reads "how to increase open files on Linux," most guides say: edit `/etc/security/limits.conf`. That's correct for login sessions. It's irrelevant for daemons.

```text
# /etc/security/limits.conf ÔÇö you edited this
*    soft    nofile    65536
*    hard    nofile    65536
```

```bash
# Check the service anyway
cat /proc/$(pgrep postgres | head -1)/limits | grep "open files"
```

```text
Max open files            1024                 524288               files
```

Still 1024. The PostgreSQL process was started by systemd, which never ran PAM for it. `/etc/security/limits.conf` is applied by `pam_limits`, and `pam_limits` only runs when a session is established through PAM ÔÇö interactive logins, SSH, `su`. systemd starts services directly via `fork()` and `execve()`, bypassing PAM entirely.

For systemd services, the only correct places to set limits are:
- `LimitNOFILE=` in the unit file's `[Service]` section
- `DefaultLimitNOFILE=` in `/etc/systemd/system.conf` for a system-wide default for all services

### limits.conf Changes Require a New Login Session

Even when `limits.conf` is the right mechanism, editing it doesn't change anything for already-logged-in users.

```text
# /etc/security/limits.conf ÔÇö just saved
deploy    -    nofile    65536
```

```bash
# deploy user's current shell ÔÇö still the old value
ulimit -n
```

```text
1024
```

PAM applies limits at session creation. The deploy user's existing SSH session was established before the edit. They must log out and log back in. If you need to verify the change took effect, log in as that user in a fresh session and check `ulimit -n`.

### Raising ulimit -v to unlimited Then Running a Memory Leak

> ÔÜá´©Å **WARNING:** Removing the virtual memory limit means a runaway process can attempt to allocate all available RAM plus swap. On a system without other protections (cgroup `MemoryMax=` or OOM killer tuning), this can make the machine unresponsive or trigger the OOM killer against unrelated processes. Only remove `RLIMIT_AS` when you have another mechanism protecting the system. Test in a VM.

```bash
# This removes all virtual memory constraints for this shell and its children
ulimit -v unlimited
```

The `RLIMIT_AS` limit caps the total virtual address space of a process ÔÇö including all `mmap()` regions, the heap, the stack, and libraries. Setting it to `unlimited` is sometimes necessary for programs that use large memory maps (some JVMs, some databases). The danger is that it opens the door to a memory leak killing the machine rather than just killing the offending process.

The correct production approach is to use a cgroup `MemoryMax=` on the service instead ÔÇö this caps actual physical memory consumption for the entire service cgroup and triggers the OOM killer within the cgroup, leaving the rest of the system unaffected.

### ulimit -u Limits Users, Not Processes

The `-u` flag limits the number of processes, but it applies per **UID**, not per process. This means it counts all processes running as that user system-wide, not just child processes of the current shell.

```bash
ulimit -u
```

```text
15211
```

If you have a CI system where many build jobs run as the same user, they all share this pool. A deployment that fans out 200 parallel workers might bump into this limit even though no single process spawned too many children.

## Alternatives & Comparisons

| Mechanism | Scope | Persistent? | Applies to Services? | Notes |
|---|---|---|---|---|
| `ulimit` (shell built-in) | Current shell + children | No | No | Immediate; temporary; for interactive debugging |
| `/etc/security/limits.conf` | PAM login sessions | Yes | No | Requires logout/login to take effect |
| systemd `Limit*=` in unit file | One service | Yes | Yes | Right tool for daemon limits |
| `/etc/systemd/system.conf` `DefaultLimit*=` | All systemd services | Yes | Yes | System-wide default for services only |
| `prlimit(1)` | Any running process (with privilege) | No (process lifetime) | Yes | Can change limits on an already-running process |
| cgroups `MemoryMax=`, `CPUQuota=`, `TasksMax=` | Service cgroup (all processes) | Yes (via unit file) | Yes | Applies to the group, not individual processes; OOM-killer aware |

For production services, `Limit*=` in unit files and cgroup directives via `systemd.resource-control(5)` are the right tools. `limits.conf` is correct for interactive user sessions. Raw `ulimit` is for quick testing and one-off adjustments in a terminal ÔÇö never for production configuration.

## Trade-offs

**Where this shines:**

- `rlimit` enforcement is in the kernel, zero-overhead per check ÔÇö the kernel already has the count, the comparison is O(1)
- Limits are inherited on fork and preserved across exec ÔÇö a security policy set at session creation automatically applies to every subprocess a user runs
- `/proc/PID/limits` gives you ground truth about what any process is actually running under, without needing access to config files or knowing how the process was started
- cgroup memory limits trigger the OOM killer inside the cgroup, not randomly across the system ÔÇö this is vastly more controlled than an unlimited process taking down unrelated services

**Where this costs you:**

- The three-tier system (rlimit via PAM, rlimit via systemd, cgroups) is genuinely confusing ÔÇö changes in the wrong tier are silently ignored, which burns hours
- `rlimit` is per-process and inherited at fork; raising a limit on a parent after a child is forked does not update the child
- `RLIMIT_NPROC` counts by UID system-wide, which makes it unsuitable for multi-tenant systems where the same UID runs many unrelated workloads
- cgroup v2 is not available in older RHEL 8 kernels without explicit opt-in ÔÇö a production fleet with mixed kernel versions needs testing before relying on unified hierarchy features

**Performance note:** raising `RLIMIT_NOFILE` to very large values (millions) can slow down per-process `select()` calls ÔÇö `select()` iterates up to `fd_max` entries, so if your hard limit is 1,048,576, `select()` on a socket may scan a million file descriptor slots. Use `epoll` instead of `select()` on modern Linux ÔÇö `epoll` is O(1) and not affected by the file descriptor limit value.

## When to Use / When to Avoid

Ô£à **Use `ulimit` in a shell when:**
- Debugging a limit problem interactively ÔÇö raise the limit, run the command, confirm the error goes away
- Running a one-off command that needs more resources than your session default provides
- Testing what limit a script needs before committing it to a config file

Ô£à **Use `/etc/security/limits.conf` when:**
- A specific Linux user account (running application code interactively or via SSH) needs higher limits
- The change is permanent and applies to all future login sessions for that user or group

Ô£à **Use `LimitNOFILE=` in a systemd unit when:**
- A service daemon needs limits different from the systemd defaults
- This is the right answer for `nginx`, `postgres`, `node`, `redis`, and virtually every server daemon

Ô£à **Use `MemoryMax=` / `CPUQuota=` / `TasksMax=` in a systemd unit when:**
- You want to cap resource usage for an entire service (all its worker processes combined)
- You need the OOM killer to stay inside one service's cgroup rather than roaming the machine

ÔØî **Avoid setting `ulimit -v unlimited` in production when:**
- The process has any chance of a memory leak ÔÇö use `MemoryMax=` in the unit file instead; it's safer and cgroup-contained

ÔØî **Avoid editing `/etc/security/limits.conf` to fix service daemon problems:**
- It won't work; services bypass PAM. You will change the file, restart nothing, see no improvement, and waste time.

ÔØî **Avoid blindly raising `LimitNOFILE` to huge values:**
- Diagnose first with `lsof -p PID | wc -l` and the `lsof` type breakdown. A legitimate high-connection server may need 65536. A server with a file descriptor leak needs a code fix, not a bigger number.

ÔÜá´©Å **Common mistakes:**
- Running `ulimit -n 65536` in a terminal and assuming it fixed the production service
- Editing `limits.conf`, not logging out, and concluding the file "doesn't work"
- Setting `LimitNOFILE=infinity` in a unit file without understanding it resolves to `nr_open` (typically 1,048,576) ÔÇö which is fine for most services but can expose `select()` performance issues

## Veteran Perspective

The "too many open files" error has ended more production incidents that I've been called into than I can count. And in nearly every case, the root cause wasn't that the limit was wrong ÔÇö it was that someone raised the limit without understanding why the file descriptors were accumulating in the first place.

The three-step discipline is this: first, confirm the error is actually `EMFILE` (per-process limit) and not `ENFILE` (system-wide). Second, count the actual open descriptors with `lsof -p PID | wc -l`. Third, look at what types of file descriptors are open. If you see a thousand regular files open on a server that should only hold a handful, you have a leak. Raising `LimitNOFILE` papers over a leak ÔÇö it just gives you more time before the next incident.

The systemd trap catches almost everyone. I've watched experienced engineers spend forty-five minutes editing `limits.conf`, restarting the service, rechecking `/proc/PID/limits`, seeing 1024, and cycling through it twice more before accepting that `limits.conf` is irrelevant to systemd. Read the `/proc/PID/limits` file first, always ÔÇö it tells you ground truth about the running process, not what you think should be true.

On the cgroup side: `MemoryMax=` in a unit file is genuinely underused. The reflex is to add more RAM when a service OOM-kills. The better question is whether that service actually needs all the memory it's consuming, and whether a `MemoryMax=` would protect the rest of the system while you investigate. Running a service with an explicit memory ceiling means an OOM event is contained ÔÇö it kills processes within the cgroup, not random processes elsewhere on the machine.

One subtlety worth internalizing: `RLIMIT_NPROC` limits processes by UID, not by parent. A CI system that runs builds as the same user for years will eventually hit this in a confusing way ÔÇö a build that forks 50 workers will fail with "resource temporarily unavailable" on `fork()`, and it won't be obvious why until you realize there are 15,000 other processes from previous builds still running as that UID. The fix is either to clean up zombie processes, raise `nproc`, or ÔÇö better ÔÇö run builds under separate UIDs.

In interviews, resource limits come up in "scaling and reliability" discussions. The question is usually framed as a scenario: "your application is running out of file descriptors ÔÇö walk me through how you'd diagnose and fix it." The answer that impresses is: check `/proc/PID/limits` for the actual limit, count open FDs with `lsof`, categorize them by type, determine whether it's legitimate load or a leak, and then fix the right thing (limit, code, or both) in the right place (unit file for a service, `limits.conf` for a user session).

## Try It Yourself

1. Run `ulimit -a` in your shell. Identify the soft limit for open files. Then run `ulimit -Hn` to see your hard limit. How much room do you have to self-raise?

2. Raise your shell's open file soft limit to 4096 with `ulimit -Sn 4096`. Open a new terminal and run `ulimit -n` there. Confirm the change did not affect the new terminal. Then try to raise the soft limit above your hard limit in the original terminal and observe the error.

3. Pick any running process with `ps aux | grep <something>` and read its full resource limits from `/proc/PID/limits`. Compare its "Max open files" soft limit to the number of actual open file descriptors with `lsof -p PID 2>/dev/null | wc -l`. How close is it to the ceiling?

4. Edit `/etc/security/limits.conf` (or a drop-in in `/etc/security/limits.d/`) to set a higher `nofile` limit for your own username. Log out and back in, then verify the new limit took effect with `ulimit -n`. Then check the limits of a systemd service (e.g., `cat /proc/$(pgrep systemd-journald)/limits | grep "open files"`) and confirm your `limits.conf` change has no effect on it.

5. **Challenge (VM recommended):** Create a minimal systemd service unit (`/etc/systemd/system/testlimit.service`) that runs a simple shell script in a loop (`while true; do sleep 1; done`). Set `LimitNOFILE=2048` and `MemoryMax=64M` in the `[Service]` section. Start the service with `systemctl start`, then verify both limits are correctly applied by reading `/proc/$(pgrep -f "sleep 1" | head -1)/limits` and `cat /sys/fs/cgroup/system.slice/testlimit.service/memory.max`. Clean up with `systemctl stop` and `systemctl disable`.

---

## Summary

- Every process limit has a **soft limit** (currently enforced) and a **hard limit** (the ceiling the process can self-raise to); only root can raise the hard limit.
- `ulimit` is a shell built-in that calls `setrlimit()` on the current shell ÔÇö it affects the shell and its future children only, not any already-running process.
- `/etc/security/limits.conf` is applied by PAM at login session creation; it has **no effect on systemd services**, which bypass PAM entirely.
- For systemd services, set `LimitNOFILE=` (or other `Limit*=` directives) in the `[Service]` section of the unit file ÔÇö this is the only correct way to set per-service rlimits.
- When diagnosing "too many open files," read `/proc/PID/limits` for ground truth, count actual FDs with `lsof -p PID | wc -l`, and categorize them before raising any limit ÔÇö a file descriptor leak needs a code fix, not a bigger number.
