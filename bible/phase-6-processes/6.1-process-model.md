## Why This Exists

Early operating systems ran one program at a time. You loaded your program into memory, it ran, and when it finished, you loaded the next one. There was no isolation, no sharing, no notion of "running in the background." If your program crashed, the machine was gone.

The process model was the solution. It gives every running program the illusion that it owns the whole machine — its own memory space, its own view of open files, its own execution state — while the kernel secretly manages the real resources beneath. Multiple programs can run simultaneously without knowing about each other. One can crash without taking down the others. Resources can be tracked and reclaimed when a program finishes.

Without this model, every system call you make, every shell command you type, every web server serving requests would be impossible. The process model is not a Linux-specific idea — it comes from Unix — but Linux's implementation goes deeper: threads, daemons, process groups, and sessions are all built on a single, unified abstraction.

This is the foundation everything else in Phase 6 builds on.

## Mental Model

Think of the kernel as a hotel manager and processes as hotel guests.

Each guest gets their own room (memory space) with a lock on the door. The hotel manager keeps a record of every guest: which room they're in, what key card they were issued, what services they're using. Guests can check in (process creation), check out (process exit), and even call the front desk (system calls) to ask for things. When a guest checks out, the manager cleans the room so it can be used again.

When a guest has a baby (forks a child process), the baby gets its own room — but initially the baby's room looks identical to the parent's. The manager doesn't actually copy the furniture until one of them tries to rearrange something (copy-on-write). If the parent checks out before the baby does, the manager (PID 1) becomes the baby's legal guardian until it checks out too.

The key insight: **you don't run a program — the kernel creates a process to run it on your behalf.**

## Theory

### What a Process Is

A process is not a program. A program is a file on disk — a static sequence of instructions. A process is what happens when the kernel loads that program into memory and starts executing it. The same program can have dozens of processes running simultaneously, each independent.

A process has:
- A **PID** (Process ID) — a unique integer the kernel uses to identify it. PIDs are assigned sequentially and recycle when processes exit.
- A **PPID** (Parent Process ID) — the PID of the process that created it. Every process except PID 1 has a parent.
- A **memory space** — private virtual memory for code, stack, heap, and mapped files.
- An **open file table** — references to every file, socket, and pipe the process has open.
- **Credentials** — UID, GID, and supplementary groups that determine what the process is allowed to do.
- **Signal handlers** — registered functions to call when signals arrive.
- A **current working directory** and **root directory**.
- A **state** — running, sleeping, stopped, or zombie.

The kernel represents all of this in a structure called `task_struct`. There is exactly one `task_struct` per process (and per thread, which matters — more on that below).

### PID and PPID

PID 1 is special: it is the first process created by the kernel at boot. On modern systems, that is `systemd`. PID 1 has no parent — or more precisely, its PPID is 0, a kernel-internal sentinel meaning "no parent." Every other process on the system descends from PID 1.

PID 2 is typically `kthreadd`, the kernel thread manager. Kernel threads (kworkers, ksoftirqd, and similar) appear in `ps` output inside square brackets and are descended from PID 2, not PID 1.

The process tree is literally a tree. You can visualize it with `pstree`. At the root is systemd; below that are login sessions, services, shells, and everything else.

### `fork()` — Creating a Child

The only way a process comes into existence (except PID 1 at boot) is through the `fork()` system call. When a process calls `fork()`, the kernel creates an almost-exact copy of it: same memory contents, same open files, same signal handlers, same environment. The new process gets a new PID, and its PPID is set to the original process's PID.

`fork()` returns twice: in the parent, it returns the child's PID. In the child, it returns 0. This is how code knows which copy it is running in.

The memory copy is not actually a full copy at the time of `fork()` — the kernel uses copy-on-write (COW). The parent and child share the same physical memory pages initially, marked read-only. Only when one of them writes to a page does the kernel copy that specific page, give the writing process its own private copy, and mark both copies writable. This makes `fork()` fast even for processes with gigabytes of memory.

### `exec()` — Replacing the Process Image

`fork()` creates a copy. `exec()` replaces the running process's memory image with a new program. When a shell runs a command, it calls `fork()` to create a child, then the child calls `exec()` to replace itself with the command's binary. The parent (the shell) waits for the child to finish.

The `exec()` family — `execv()`, `execve()`, `execvp()`, and others — all do the same thing: load a new program and begin executing it. The process retains its PID, its PPID, its open file descriptors (unless marked close-on-exec), and its credentials. Everything in memory is replaced.

If `exec()` succeeds, it never returns — there is no "old program" left to return to. If it fails (bad path, bad permissions), control returns to the caller.

### `exit()` — Termination and the Exit Code

When a process calls `exit()` (or returns from `main()`), the kernel:
1. Closes all open file descriptors.
2. Releases allocated memory.
3. Sends `SIGCHLD` to the parent.
4. Records the **exit status** — an integer combining the exit code and how the process died (normally or by signal).
5. Moves the process into the **zombie state** — it stays in the process table until the parent retrieves the exit status.

The exit code is what `$?` shows in a shell. By convention, 0 means success, non-zero means failure. The specific non-zero value is program-defined.

### Zombie Processes

A zombie is a process that has exited but whose exit status has not yet been collected by its parent. The process is dead — it has no memory, no CPU time, no resources — but it has a slot in the process table and a PID that cannot be reused until the parent calls `wait()`.

A zombie looks like this in `ps`:

```text
  PID TTY      STAT   TIME COMMAND
12345 pts/0    Z+     0:00 [myprogram] <defunct>
```

The `Z` state and `<defunct>` tag are both markers of zombie status.

You cannot kill a zombie with `kill -9`. The zombie is already dead. Sending signals to a zombie does nothing — there is no process there to receive them. The only way to remove a zombie is for its parent to call `wait()` (or `waitpid()`). If the parent is stuck in a loop and never calls `wait()`, the zombie accumulates. A handful of zombies is normal. Thousands of zombies means the parent has a bug.

### Orphan Processes

If a parent exits before its children, those children become **orphans**. The kernel immediately reparents them to PID 1 (or the nearest registered subreaper — a process that has called `prctl(PR_SET_CHILD_SUBREAPER)`, a mechanism used by container runtimes like Docker). PID 1 (systemd) is a reliable wait()-caller, so orphaned processes are reaped when they eventually exit.

This is why orphan processes are generally harmless. They will be cleaned up. Zombies are the problem — because the parent is still alive but broken.

### Process Groups and Sessions

Every process belongs to a **process group**, identified by a PGID (Process Group ID). Process groups exist primarily to deliver signals to a collection of related processes. When you press Ctrl-C in the terminal, the kernel sends SIGINT to the entire foreground process group — not just the command you typed, but every process in that job.

When you run `cat /etc/passwd | grep root`, both `cat` and `grep` are in the same process group. Ctrl-C kills both.

A **session** is a collection of process groups, identified by a SID (Session ID). A session is typically created when you log in — your shell starts a new session, and all commands you run in that shell share the session. Sessions have a **controlling terminal**: the terminal device associated with the session. When the terminal closes, `SIGHUP` is sent to the foreground process group.

`nohup` and `disown` are tools for detaching a process from its session's controlling terminal so it survives logout.

### Threads — `clone()` with Shared Memory

Linux does not have a separate concept of "thread" at the kernel level. Threads are implemented using the `clone()` system call — a more powerful version of `fork()` that lets the caller specify exactly what to share between parent and child.

When `fork()` is called, `clone()` runs underneath with flags that say "copy everything." When a threading library like pthreads creates a thread, it calls `clone()` with flags including:
- `CLONE_VM` — share the same virtual memory space
- `CLONE_FILES` — share the same open file table
- `CLONE_SIGHAND` — share signal handlers
- `CLONE_THREAD` — place the new task in the same thread group

Because threads share memory, writing to a variable in one thread is visible in all others instantly. This is what makes threads fast for intra-process communication — and what makes threading bugs so hard to debug.

In Linux, every thread gets its own PID at the kernel level (`pid` in `task_struct`), but threads in the same process share a **Thread Group ID (TGID)**, which is the PID of the first thread. Tools like `ps` show the TGID as the "PID" by default, which is why you typically do not see individual thread PIDs in a standard `ps` listing.

### Daemons

A daemon is a process that runs in the background, detached from any terminal, typically providing a service. SSH server, web servers, cron — all daemons.

A well-written daemon uses the **double-fork** technique to fully detach from the terminal and its session:
1. `fork()` — the parent exits; the child is now an orphan, adopted by PID 1.
2. `setsid()` — the child creates a new session with no controlling terminal.
3. `fork()` again — this second fork ensures the daemon can never inadvertently reacquire a controlling terminal (only session leaders can acquire one).
4. Close stdin/stdout/stderr and redirect them to `/dev/null`.

On systemd systems, you rarely write this boilerplate manually — systemd manages daemon lifecycle. But understanding the double-fork tells you why `Type=forking` exists in systemd unit files and what it means when a service forks.

Daemons often write their PID to a file in `/run/` (e.g., `/run/nginx.pid`). This PID file lets init systems and management scripts find the daemon and send it signals without scanning the process table.

## How It Works (Mechanics)

### The fork-exec Pattern

When you type `ls` in a bash shell, this is what actually happens:

```text
# Conceptual trace — not a real command
1. bash calls fork()
   → kernel duplicates bash's task_struct, address space, file table
   → parent (bash) receives child PID; child receives 0
   → both processes are now running (on separate CPUs or time-shared)

2. child calls execve("/bin/ls", ["ls"], environ)
   → kernel loads /bin/ls ELF binary into the child's address space
   → old bash code and data are gone; child is now "ls"
   → child starts executing ls's main()

3. parent (bash) calls waitpid(child_pid, &status, 0)
   → bash is now blocked (sleeping) waiting for ls to finish

4. ls does its work, then calls exit(0)
   → kernel closes ls's file descriptors
   → releases memory
   → sends SIGCHLD to bash
   → ls enters zombie state

5. waitpid() in bash returns
   → bash reads the exit status (0 = success)
   → kernel removes ls's task_struct
   → bash prints its prompt
```

As an ASCII sequence diagram:

```text
bash (parent)                      ls (child)
─────────────                      ──────────
fork() ──────────────────────────► copy of bash created
  │                                    │
  │                                execve("/bin/ls")
  │                                    │ bash image replaced
waitpid() ──── blocks ─────────────    │
  │            (sleeping)          ls runs, prints output
  │                                    │
  │                                exit(0)
  │◄─────── SIGCHLD ───────────────────┘
waitpid() returns
  │
bash prints prompt
```

### Copy-on-Write at the Page Table Level

When `fork()` runs, the kernel does not copy physical memory pages. Instead:

1. The child's page table is populated with entries pointing to the **same physical frames** as the parent.
2. Both parent's and child's page table entries for those frames are marked **read-only** — even for pages that were writable before.
3. When either process writes to a shared page, the CPU raises a **page fault** (write to read-only page).
4. The kernel's page fault handler recognizes this as a COW fault (not a bug), allocates a new physical frame, copies the original page's content into it, updates the faulting process's page table entry to point to the new frame and mark it writable, and retries the instruction.
5. The other process keeps its page table entry pointing to the original frame.

The result: `fork()` costs proportional to the size of the page table, not the size of the address space. A process using 4 GB of memory can be forked in microseconds if neither the parent nor child writes much after the fork.

```text
Before fork():
  Parent page table:  [0x0000→Frame A] [0x1000→Frame B] [0x2000→Frame C]
  Physical memory:    Frame A (code)   Frame B (heap)   Frame C (stack)

After fork() — COW applied:
  Parent page table:  [0x0000→Frame A*] [0x1000→Frame B*] [0x2000→Frame C*]
  Child  page table:  [0x0000→Frame A*] [0x1000→Frame B*] [0x2000→Frame C*]
  (* = read-only in both page tables; same physical frames)

After child writes to Frame B (heap page fault):
  Parent page table:  [0x0000→Frame A*] [0x1000→Frame B*] [0x2000→Frame C*]
  Child  page table:  [0x0000→Frame A*] [0x1000→Frame B'] [0x2000→Frame C*]
  (Frame B' = new copy, writable; Frame B* still shared with parent)
```

### What the Kernel Stores Per Process — `task_struct`

The kernel represents each process (and thread) as a `task_struct` in kernel memory. Key fields:

```text
task_struct {
  pid_t pid;            ← kernel-level thread ID (unique per thread)
  pid_t tgid;           ← thread group ID = "PID" shown by ps (same for all threads)
  long state;           ← TASK_RUNNING, TASK_INTERRUPTIBLE, TASK_ZOMBIE, etc.
  struct task_struct *parent;   ← pointer to parent's task_struct
  struct list_head children;    ← linked list of child task_structs
  struct mm_struct *mm;         ← memory descriptor (NULL for kernel threads)
  struct files_struct *files;   ← open file table (shared across threads if CLONE_FILES)
  struct signal_struct *signal; ← signal handlers and pending signals
  struct cred *cred;            ← UID, GID, capabilities
  int exit_code;                ← exit status, read by parent's wait()
  char comm[TASK_COMM_LEN];     ← executable name (16 chars, used in ps output)
}
```

Threads in the same process share the `mm_struct` (memory) and `files_struct` (file table) — that is the definition of a thread in Linux. Processes created with plain `fork()` get their own copies of both.

## Basic Example

```bash
# Watch the fork-exec lifecycle with ps and a simple sleep command
# Run this in one terminal, observe in another
sleep 30 &         # fork-exec: bash forks, child execs sleep, bash continues
echo "sleep PID: $!"  # $! is the PID of the last backgrounded command
ps -o pid,ppid,stat,comm -p $$,$!  # show bash and sleep side by side
```

```text
sleep PID: 14823
  PID  PPID STAT COMMAND
14601 14599 Ss   bash
14823 14601 S    sleep
```

Look at the PPID column: `sleep`'s parent is `bash`. The `S` state on sleep means it is in interruptible sleep (waiting for its 30-second timer). The `Ss` on bash means it is a session leader (`s`) in interruptible sleep.

Now create a zombie deliberately:

```bash
# This small C program forks a child that exits immediately,
# but the parent sleeps without calling wait() — creating a zombie
cat > /tmp/zombie_demo.c << 'EOF'
#include <stdio.h>
#include <unistd.h>
int main() {
    pid_t child = fork();
    if (child == 0) {
        _exit(0);          /* child exits immediately */
    }
    printf("child PID: %d\n", child);
    sleep(60);             /* parent never calls wait() — zombie accumulates */
    return 0;
}
EOF
gcc -o /tmp/zombie_demo /tmp/zombie_demo.c
/tmp/zombie_demo &
sleep 1
ps -o pid,ppid,stat,comm | grep zombie_demo
```

```text
child PID: 15102
15100 14601 S    zombie_demo
15102 15100 Z    zombie_demo
```

The child (15102) shows state `Z` — zombie. Now try to kill it:

```bash
kill -9 15102   # try to kill the zombie
ps -o pid,stat,comm -p 15102  # still there
```

```text
  PID STAT COMMAND
15102 Z    zombie_demo
```

The zombie survives `kill -9`. Kill the parent and the zombie disappears:

```bash
kill 15100           # kill the parent
ps -o pid,stat,comm -p 15102  # zombie is gone
```

```text
(no output — process no longer exists)
```

Try this yourself. The zombie example is safe to run on any Linux system. If you do not have `gcc`, compile it on a system that does, or use the `/proc` approach described in the Real-World Example below.

## Real-World Example

On a production server, a runaway parent process with a `wait()` bug can accumulate thousands of zombies. This is how you diagnose it:

```bash
# Find all zombie processes on the system
ps aux | awk '$8 == "Z" { print }'
```

```text
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
www-data 31042  0.0  0.0      0     0 ?        Z    09:14   0:00 [php-fpm: pool] <defunct>
www-data 31043  0.0  0.0      0     0 ?        Z    09:14   0:00 [php-fpm: pool] <defunct>
www-data 31044  0.0  0.0      0     0 ?        Z    09:14   0:00 [php-fpm: pool] <defunct>
```

You've found zombies. Now find their parent — that's the broken process:

```bash
# Get the PPID of the first zombie
ps -o ppid= -p 31042
```

```text
31001
```

```bash
# Look at the parent
ps -o pid,comm,stat -p 31001
```

```text
  PID COMMAND         STAT
31001 php-fpm: master S
```

```bash
# How many zombies does this parent have?
ps --ppid 31001 -o stat | grep -c Z
```

```text
3
```

Three zombies under one php-fpm master. This is a known php-fpm bug in older versions where the master process fails to call `wait()` fast enough under certain signal delivery conditions. The fix: upgrade php-fpm. The short-term workaround: send SIGCHLD to the parent to prompt it to reap, or restart the service.

```bash
kill -SIGCHLD 31001    # nudge parent to call wait()
ps --ppid 31001 -o stat | grep -c Z  # check if zombies cleared
```

In production, a few zombies (under ten) are normal — they indicate a parent that calls `wait()` but perhaps not immediately. Hundreds or thousands of zombies indicate a systemic bug. Thousands of zombies do not themselves harm the system (they have no resources), but they consume PID table slots. On a system with the default PID limit of 32768, enough zombies can prevent new processes from starting.

```bash
# Check the PID limit
cat /proc/sys/kernel/pid_max
```

```text
32768
```

```bash
# Check how many processes currently exist
ls /proc | grep -c '^[0-9]'
```

## ⚠️ Gotchas

### You Cannot Kill a Zombie — Kill the Parent Instead

The single most common mistake: seeing a `<defunct>` process in `ps` and reaching for `kill -9`.

```bash
# This looks right but does nothing
kill -9 <zombie_pid>
```

```text
(no output — signal was silently ignored)
```

The zombie has no running code. It is not a process in any meaningful sense — it is a record in the kernel's process table. Signals are delivered to running code; there is no code here to receive them. The kernel discards signals sent to zombies.

The rule: **a zombie can only be removed by its parent calling `wait()`.** Your options are:

1. Wait — if the parent eventually calls `wait()`, the zombie clears itself.
2. Send `SIGCHLD` to the parent to prompt it to reap children: `kill -SIGCHLD <parent_pid>`.
3. Kill the parent — the orphaned zombie is adopted by PID 1, which will immediately reap it.
4. Reboot — this always works, but is rarely necessary.

### `fork()` Being "Expensive" on Large Processes Is a Myth

You will read advice saying "don't fork large processes, it's slow." This was true before copy-on-write. With COW, `fork()` costs proportional to the **size of the page table**, not the size of the address space.

A process with 4 GB of heap will have a large page table (~8 MB of page table entries), but `fork()` still completes in milliseconds. The cost shows up only if the child immediately writes to many of those pages — which triggers COW faults. If your child calls `exec()` immediately after `fork()` (the normal shell pattern), the entire address space is replaced before any COW faults can accumulate, making the fork cost negligible.

The pattern that is actually expensive: forking a multi-gigabyte process where the child immediately does a lot of writing before calling `exec()`. Python's `multiprocessing` module on workers that modify large data structures falls into this trap.

### Threads Share Memory — Including Bugs

When a thread corrupts a data structure, every other thread in the process is immediately affected. A memory bug in thread A is invisible until thread B reads the corrupted region and crashes with a confusing error. The corrupting thread is already gone or doing something else entirely.

```bash
# Threads appear as separate entries in /proc with their own TIDs
ls /proc/<pid>/task/   # each directory is a thread; directory name is the TID
```

```text
12345  12346  12347
```

Three threads: TID 12345 (the main thread), 12346, and 12347. The PID shown by `ps` is always the TGID — 12345 in this case. This is why `kill <pid>` sends a signal to the whole process, not to one thread.

### Orphan vs Zombie — The Direction of the Problem

These are commonly confused because both involve parent-child relationships going wrong.

- **Zombie**: child is dead, parent is alive but not calling `wait()`. The **parent** is the problem.
- **Orphan**: parent is dead, child is alive. The **parent** is gone but the child is fine — PID 1 will clean up when the child exits.

Zombies are bugs. Orphans are normal.

### PID 1 Dying Is a System Crash

If `systemd` (PID 1) exits or crashes, the kernel immediately panics. There is no recovery. This is not a graceful shutdown — it is a hard crash.

```bash
# This will immediately crash your system — do not run on a real machine
# kill -9 1   ← NEVER run this
```

On a VM, if you accidentally do this, you will see a kernel panic on the console and need to force-reboot the VM. On a real server, it is an unexpected reboot. The kernel has no concept of the system running without PID 1.

## Distro Notes

| Distro Family | PID 1 / Notes |
|---|---|
| Debian / Ubuntu | `systemd` on all modern versions (Debian 8+, Ubuntu 15.04+); `sysvinit` on older installs |
| RHEL / Fedora / CentOS / Rocky | `systemd` since RHEL 7 / Fedora 15 |
| Arch | `systemd` by default; can replace with alternatives manually |
| Alpine (containers) | No systemd; typically runs OpenRC or just the container entrypoint directly as PID 1 — container runtimes like Docker run your `ENTRYPOINT` command as PID 1, which means it must handle SIGTERM and reap zombies itself, or use a minimal init like `tini` |

The Alpine/container row matters most in practice. Many engineers run a Node.js or Python process directly as PID 1 in a Docker container, then wonder why zombie accumulation happens. The application was not written to call `wait()`. Use `tini` (or the `--init` flag in Docker) to put a minimal PID 1 in front of your application.

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| `fork()` + `exec()` | Creating a child process to run a different program | The universal pattern; what every shell does |
| `fork()` without `exec()` | True parallel work in the same program (e.g., Python multiprocessing, pre-fork servers) | Child inherits full address space; COW keeps it cheap until divergence |
| `clone()` with `CLONE_VM` | Creating threads (done by pthreads internally, not by application code directly) | Shares memory; requires careful synchronization |
| `vfork()` | Historical; child must immediately `exec()` or `_exit()` | Predates COW; parent is suspended until child execs; avoid in new code |
| `posix_spawn()` | Embedded systems or platforms where `fork()`+`exec()` is inconvenient | Implemented via `clone()` on Linux; not faster than fork-exec on modern kernels |

In practice, you never call `clone()` directly in application code — the pthreads library does it. You call `fork()` for process creation. Understanding `clone()` matters for reading kernel code, debugging with strace, and understanding containers.

## Trade-offs

**Where this shines:**

- Process isolation means a crash in one process cannot corrupt another's memory. This is why web servers pre-fork workers — each request runs in isolation.
- COW `fork()` makes pre-fork servers cheap: Apache MPM prefork, unicorn/gunicorn, and PHP-FPM all rely on this to spawn hundreds of workers from a large initialized parent cheaply.
- The session/process group model gives the terminal precise signal control over entire pipelines with a single keypress.

**Where this costs you:**

- Forking is cheap, but shared-nothing memory means inter-process communication requires explicit mechanisms (pipes, sockets, shared memory, message queues). Threads are cheaper to communicate between, but carry concurrency hazards.
- Each process has its own copy of file descriptor tables, signal handlers, and credentials. Sharing state between processes requires explicit design. Threading is tempting but replaces "hard to debug IPC" with "even harder to debug data races."
- The zombie machinery means every `fork()` requires a corresponding `wait()`. Forgetting `wait()` is a slow leak that only manifests under sustained load.

**Performance note:** On a modern Linux kernel with COW, `fork()` for a 1 GB process completes in under 1 ms on typical hardware. The page table for 1 GB of address space is approximately 2 MB; copying 2 MB is fast. The cost only accumulates as the child writes to shared pages post-fork.

## When to Use / When to Avoid

✅ **Use `fork()` + `exec()` when:**
- Launching any subprocess from your program — this is the canonical, correct pattern
- Building a pre-fork server where workers share an initialized state before diverging
- You want hard isolation: a crash in the child cannot corrupt the parent

✅ **Use threads (pthreads / `clone()` with `CLONE_VM`) when:**
- You need low-latency communication between concurrent tasks (shared memory is much faster than pipes or sockets)
- Tasks are short-lived and the overhead of `fork()` per task would dominate
- You need to share open file descriptors naturally without passing them over sockets

❌ **Avoid threads when:**
- The shared state is complex and mutable — data races are notoriously difficult to reproduce and debug
- One thread can crash the entire process (a segfault in any thread kills all threads; a crash in a child process leaves the parent alive)
- Running CPU-bound Python code — the GIL means Python threads do not run truly in parallel

❌ **Avoid leaving zombie processes:**
- Always call `wait()` or `waitpid()` in the parent, or install a `SIGCHLD` handler that reaps
- If you fork and forget (fire-and-forget child), double-fork so PID 1 adopts the child immediately

⚠️ **Common mistakes:**
- Forking inside a multithreaded process — only the calling thread is duplicated in the child, but all mutexes held by other threads are copied in their locked state. This causes deadlocks. Fork only from single-threaded code, or call `exec()` immediately after fork in a multithreaded parent.
- Assuming `kill <pid>` kills all threads — it does, but `kill <tid>` kills only one thread. Know which one you have.
- Running application code as PID 1 in a container without zombie reaping — always use `tini` or `--init`.

## Veteran Perspective

The newcomer mistake is thinking `kill -9` solves zombie processes. It doesn't, and understanding why it doesn't teaches you more about the process model than a chapter of theory.

The intermediate mistake is thinking threads are always better than processes because they're "lighter." In production, I have seen more serious incidents caused by threading bugs — data corruption, deadlocks, use-after-free — than by process overhead. The pre-fork model used by nginx, gunicorn, and PHP-FPM is not naive; it is a deliberate trade-off: give up some memory efficiency to get crash isolation and eliminate an entire class of concurrency bugs.

The design principle behind all of this is **Unix's radical simplicity of abstraction**. `fork()` and `exec()` are separate operations because they were always separate ideas: duplication and replacement. Separating them enables every clever trick in the book: redirecting file descriptors between fork and exec (how shell pipelines work), setting up environment variables, changing credentials. A language like Go or Java that starts a new process via a single "spawn" call has to rebuild all of that flexibility explicitly. Unix gave it to you for free by doing less.

Here is a war story that actually matters: a high-traffic API server was accumulating zombies at the rate of about 50 per hour. Nobody noticed for three days because the system had plenty of PIDs and the zombies consumed no CPU or memory. On day four, the PID table hit 32768. Every `fork()` started failing with `EAGAIN`. The web server could not spawn workers. The service went down completely. The root cause was a signal handler that was calling a function that was not async-signal-safe, causing the SIGCHLD handler to occasionally deadlock, which meant `wait()` never got called for some children. The fix was one line: rewrite the SIGCHLD handler to set a flag and call `wait()` in the main loop instead. The lesson: zombie accumulation is a slow-burning bug. Monitor `ps aux | awk '$8 == "Z"' | wc -l` in your dashboards.

In a technical interview, the fork-exec pattern comes up constantly in "how does a shell work" questions. Know the sequence cold: fork, exec in child, wait in parent. Know that COW makes fork cheap. Know that `clone()` is the underlying syscall for both `fork()` and thread creation, and that the difference between a process and a thread in Linux is a matter of which resources are shared.

## Try It Yourself

1. Run `pstree -p` and find your shell in the tree. Trace the path from PID 1 (systemd) down to your shell. Count the hops. On a typical desktop system, how many ancestors does your shell have?

2. Run `sleep 60 &` and then immediately run `ps -o pid,ppid,stat,comm`. Find the sleep process and confirm its PPID matches your shell's PID. Let it finish and run the same `ps` command again — it should be gone.

3. Open `/proc/<pid>/status` for your shell's PID and read the `Pid:`, `PPid:`, `Threads:`, and `VmRSS:` fields. What does `Threads: 1` tell you? Now open `/proc/<pid>/status` for a multi-threaded process (try `systemd` or a Java process if running) and compare.

4. Write a short shell script that forks a child using `(sleep 30 &)` (a subshell in background). Before the child exits, open a second terminal and run `ps -o pid,ppid,stat,comm | grep sleep`. Then kill the subshell's parent process (the subshell itself) with `kill <subshell_pid>`. Watch what happens to the PPID of the sleep process — it should reparent to 1 or your session's PID 1 equivalent.

5. **Challenge (VM recommended):** Write a C program that forks 100 children. Each child exits immediately with `_exit(0)`. The parent never calls `wait()`. Run it, then run `ps aux | awk '$8 == "Z"' | wc -l` to count zombies. Then send `SIGCHLD` to the parent with `kill -SIGCHLD <parent_pid>` — a well-written parent would reap on SIGCHLD, but yours doesn't have a handler. Kill the parent and watch all 100 zombies disappear at once as PID 1 adopts and reaps them.

## Summary

- A process is a running program: private memory, open files, credentials, and a PID. The kernel tracks it in a `task_struct`.
- `fork()` creates a child by copying the parent's page tables with COW — the copy is cheap because physical memory is shared until written. `exec()` replaces the child's memory image with a new program. These two calls together are how every command gets run.
- A zombie is a process that has exited but whose parent has not called `wait()`. You cannot kill a zombie — only the parent can reap it. Thousands of zombies means the parent has a bug.
- An orphan (parent died before child) is harmless — PID 1 adopts it and will reap it when it exits.
- Threads are processes that share memory and file tables, created by `clone()` with `CLONE_VM | CLONE_FILES`. In Linux, both threads and processes are represented by `task_struct`; the difference is which resources they share.
