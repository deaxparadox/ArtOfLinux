# [6.4] — Job Control

> You're running a long compilation, it's blocking your terminal, and your coffee is getting cold — job control is the set of tools that lets you move work between the foreground and background, pause it, resume it, and cut it free from your session entirely.

## The Problem This Solves

Before job control existed in Unix, a terminal was a one-lane road. You ran a command, it owned the terminal completely until it finished. If you wanted to do something else — check another file, run a quick lookup, tail a log — you opened a second terminal window. Or you waited. Or you killed the first process.

This was painful enough that early Unix users found workarounds. You could background a process with `&`, but once it was running, you couldn't really interact with it. If you needed to bring it back to the foreground, you couldn't. If you wanted to pause it temporarily, you couldn't. If the command took hours and you needed to log out, it died with your session.

Job control is the complete answer to all of these problems. It gives the shell a way to track multiple running and suspended processes, lets you move them between foreground and background at will, and provides the tools to detach long-running work from your terminal entirely so it survives after you log off.

---

## Mental Model

Think of your terminal as a single microphone on a stage. Only one performer can be on stage with the microphone at a time — that's the foreground job. Other performers wait in the wings — those are the background jobs. You're the director.

You can wave someone off stage (`Ctrl+Z` — pause and step aside), bring someone from the wings onto stage (`fg`), or tell a performer to keep rehearsing offstage without the microphone (`bg`). If the whole theater closes for the night and you want a performer to keep rehearsing in the parking lot after everyone leaves, that's `nohup` — they don't need the microphone anymore and they'll survive the theater shutting down.

The shell's job table is your director's clipboard: it tracks every performer and their current state.

## Theory

### What a Job Is

In the shell's model, a **job** is not the same as a process. A job is a pipeline — one or more processes connected together and managed as a unit. When you run `grep pattern bigfile | sort | uniq -c`, that entire pipeline is one job. The shell tracks it under a single job number even though three processes are involved.

Jobs exist at the session layer, above the process layer. The kernel tracks process groups; the shell tracks jobs and maps them to process groups. When the shell sends a signal to a job, it sends it to the job's entire process group using `kill(0, ...)` or `killpg()`, so every process in the pipeline receives it simultaneously.

### Foreground and Background

Every terminal session has exactly one foreground job at a time, or none. The foreground job is the only job that receives keyboard input and terminal-generated signals. When you press `Ctrl+C`, the terminal sends SIGINT to the foreground process group — not to any background jobs.

Background jobs run concurrently with the foreground but have no direct connection to the keyboard. They can still write to the terminal (and will, annoyingly, if you don't redirect their output). If a background job tries to *read* from the terminal, it receives SIGTTIN, which suspends it automatically until you bring it to the foreground.

### Job States

The shell tracks four states for each job:

**Running** — the job is actively executing. This includes background jobs that are running (`&`) and the foreground job.

**Stopped** — the job has been suspended. It holds all of its memory, open files, and open network connections, but the kernel has stopped scheduling it. It receives no CPU time. It is not gone — it is frozen in place.

**Done** — the job has exited. The shell keeps the record briefly so it can print a completion message, then removes it from the job table.

**Terminated** — the job was killed by a signal (SIGKILL, or unhandled SIGTERM, etc.).

### SIGTSTP, SIGCONT, and SIGHUP

Three signals drive job control:

**SIGTSTP** is sent to the foreground job when you press `Ctrl+Z`. The terminal driver generates it. The default disposition is to stop the process. Unlike SIGSTOP, SIGTSTP can be caught — a process can intercept it, do cleanup, and then suspend itself by sending itself SIGSTOP. Shells and interactive applications (like `vim`) use this to save state before suspending.

**SIGCONT** resumes a stopped process. When you run `fg` or `bg`, the shell sends SIGCONT to the job's process group. Even if a process has SIGCONT blocked, the kernel will wake it up — SIGCONT has special kernel-level semantics that SIGTSTP does not.

**SIGHUP** is sent when a terminal session ends. When you close an SSH connection, the kernel sends SIGHUP to the session's controlling terminal, which propagates to the shell. By default, the shell then sends SIGHUP to all jobs in its job table before exiting. If a job's default SIGHUP disposition is unmodified, it dies. This is the problem that `nohup` and `disown` solve.

### The huponexit Nuance

There is a subtlety here that trips up even experienced users. Bash's `huponexit` option (off by default) controls whether bash sends SIGHUP to background jobs when the shell exits normally — via `exit` or EOF. With `huponexit` off, if you type `exit` at a login shell, background jobs are simply orphaned; they are not killed. The new parent becomes PID 1.

But if your SSH connection drops, or if the terminal is closed forcefully, the kernel sends SIGHUP to the shell process itself. Bash then sends SIGHUP to all jobs in its job table, regardless of `huponexit`. This is the scenario that kills your background jobs — not a normal logout, but a connection drop. `nohup` and `disown` exist specifically for this.

## How It Works (Mechanics)

### Starting a Job in the Background

When the shell parses a command ending with `&`, it calls `fork()` and `exec()` as usual, but skips the call to `tcsetpgrp()` that would give the child's process group control of the terminal. The child gets a new process group ID. The shell immediately prints the job number and PID, then returns the prompt.

```bash
sleep 100 &
```

```text
[1] 4821
```

The shell adds an entry to its internal job table: job number 1, process group 4821, state Running.

### Suspending a Job

When you press `Ctrl+Z`, the terminal driver sends SIGTSTP to the foreground process group (not just the foreground process — the entire group). The kernel marks every process in that group as stopped. The shell receives notification via `waitpid()` with `WUNTRACED`, detects the stopped status, and updates its job table entry from Running to Stopped. It then prints:

```text
[1]+  Stopped                 sleep 100
```

The shell reclaims the terminal by calling `tcsetpgrp()` to reassign terminal ownership to itself.

### fg and bg

`fg %1` sends SIGCONT to job 1's process group and calls `tcsetpgrp()` to give the job's process group ownership of the terminal. The job resumes running in the foreground.

`bg %1` sends SIGCONT to job 1's process group but does *not* call `tcsetpgrp()`. The job resumes running in the background — it can still write to the terminal, but the shell retains keyboard ownership.

### nohup Mechanics

`nohup` is an external executable (not a shell builtin). When you run `nohup mycommand`, nohup calls `sigaction()` to set SIGHUP's disposition to SIG_IGN for the process before calling `exec()`. The child process inherits this ignored disposition — it cannot be overridden unless the child explicitly changes its own signal handling.

If stdout is a terminal, nohup redirects it to `./nohup.out` (falling back to `$HOME/nohup.out` if the current directory is not writable). If stderr is a terminal, nohup redirects stderr to the same destination as stdout.

nohup then calls `exec()` to replace itself with the target command. The target command runs with SIGHUP ignored and output captured.

### disown Mechanics

`disown` is a bash builtin. It operates on the shell's own job table, not on the process itself. When you run `disown %1`, bash removes job 1 from its job table. Nothing changes in the process — it keeps running with the same signal dispositions, the same open file descriptors, the same parent PID.

The protection comes from the shell's SIGHUP propagation behavior: when bash exits after receiving SIGHUP, it iterates its job table and sends SIGHUP to each entry. A disowned job is not in that table, so the shell never sends it SIGHUP.

`disown -h %1` is softer: it keeps the job in the table (so `jobs` still shows it) but marks it with a flag telling bash to skip it during SIGHUP propagation.

### wait Mechanics

`wait` is a shell builtin that wraps `waitpid()` (POSIX) or `wait4()` (Linux extension). When you call `wait $pid`, the shell calls `waitpid(pid, &status, 0)`, which blocks until the process exits. The exit status is decoded from the `status` value and returned as the builtin's exit code. The shell also removes the completed job from its job table.

`wait` with no arguments calls `waitpid(-1, &status, 0)` in a loop until all child processes have been collected.

## Basic Example

```bash
# Start a job in the background
sleep 60 &

# List jobs — shows job number, state, command
jobs

# Suspend the current foreground job (press Ctrl+Z while it runs)
# Simulated here by starting another process and immediately suspending it
sleep 120 &
kill -TSTP %2     # send SIGTSTP to job 2 directly, same effect as Ctrl+Z

# List again — shows both jobs, one Running, one Stopped
jobs

# Bring job 1 to foreground
fg %1

# After Ctrl+Z again: resume job 1 in background
bg %1

# List all jobs with PIDs
jobs -l
```

```text
[1] 5100
[1]+  Running                 sleep 60 &
[1]-  Running                 sleep 60 &
[2]+ Stopped                 sleep 120
[1]-  Running                 sleep 60 &
[2]+ Stopped                 sleep 120
[1]- 5100 Running                 sleep 60 &
[2]+ 5101 Stopped                 sleep 120
```

In the `jobs -l` output, the extra column (5100, 5101) is the process ID. The `+` next to a job marks the current job — the one `fg` and `bg` will act on by default if you give no job argument. The `-` marks the previous job.

Run this yourself. Start `sleep 300 &`, press `Ctrl+Z` on a second command, and watch the states change. Run `jobs -l` and notice how the PID appears between the current/previous marker and the state field. That PID is the process group leader.

## Real-World Example

You're on a remote server running a database dump that will take 20 minutes. Halfway through, you realize you need to run some other commands and you also need to log out soon.

```bash
# Start the dump — initially in the foreground
pg_dump mydb > /var/backups/mydb.sql 2> /var/backups/mydb.err

# You realize this will take forever. Press Ctrl+Z to suspend it.
# [1]+  Stopped   pg_dump mydb > /var/backups/mydb.sql 2> /var/backups/mydb.err

# Resume it in the background so you can use the terminal
bg %1

# Do other work... check disk space
df -h /var/backups

# Now you need to log out. The dump is still running.
# Check the job's PID first
jobs -l
# [1]+ 7832 Running   pg_dump mydb > /var/backups/mydb.sql 2> /var/backups/mydb.err

# Remove it from the job table so bash won't send SIGHUP when you disconnect
# Output is already redirected to files, so disown alone is sufficient here
disown %1

# Now log out. The pg_dump keeps running. When it finishes, output is in
# /var/backups/mydb.sql and errors (if any) in /var/backups/mydb.err.
exit
```

The key insight here: because you redirected both stdout and stderr to files *before* running the command, `disown` alone is safe. The process doesn't need the terminal for output, so losing the terminal connection won't cause EIO write errors. If you had forgotten the redirections, you'd combine `disown` with an earlier redirect, or use `nohup` from the start.

A production sysadmin would structure this more cleanly from the beginning:

```bash
# The right way to start a long job you intend to background
nohup pg_dump mydb > /var/backups/mydb.sql 2> /var/backups/mydb.err &
echo "Dump started, PID: $!"

# If you want to wait for it to finish in a script
wait $!
echo "Dump exit code: $?"
```

The `wait $!` pattern is extremely common in shell scripts. `$!` expands to the PID of the most recently backgrounded job. `wait $!` blocks until that specific job exits and returns its exit code — so your script can check whether the dump succeeded.

For scripts that fan out multiple background tasks and need to wait for all of them:

```bash
# Launch several independent tasks
process_shard.sh 1 &
process_shard.sh 2 &
process_shard.sh 3 &

# Capture their PIDs
pid1=$!
# (Note: $! only holds the last PID — capture each one immediately after launch)

# Wait for all background jobs
wait
echo "All shards finished"
```

Or, to detect individual failures:

```bash
pids=()

for shard in 1 2 3 4; do
    process_shard.sh "$shard" &
    pids+=($!)
done

failed=0
for pid in "${pids[@]}"; do
    wait "$pid"   # wait returns this process's exit code
    if [[ $? -ne 0 ]]; then
        echo "Shard $pid failed"
        failed=1
    fi
done

exit $failed
```

## ⚠️ Gotchas

### nohup Without & Blocks Your Terminal

You've heard "use nohup to keep a process running after you log out" so you run:

```bash
nohup my_long_script.sh
```

The terminal freezes. The script runs, but you can't type anything. You're stuck.

nohup makes the process immune to SIGHUP — it does not background the process. Without `&`, the command runs in the foreground, tied to your terminal, immune to HUP but still blocking your shell. The fix is always:

```bash
nohup my_long_script.sh &
```

The two are complementary, not alternatives. `nohup` handles signal immunity and output redirection. `&` handles backgrounding. You almost always want both together.

### disown Doesn't Redirect Output

You start a verbose process without redirecting output:

```bash
./noisy_script.sh &
disown %1
```

Then you log out and reconnect. The process is still running, but now every time it tries to write to stdout, it gets EIO — the terminal it was writing to is gone. Depending on the script, this may cause it to crash, hang, or silently swallow errors.

`disown` removes the job from the shell's table. It does absolutely nothing to the process's file descriptors. If the process's stdout and stderr are still pointed at a terminal (pts device), losing that terminal causes write errors.

The correct pattern when you forget to redirect from the start:

```bash
# Start with output going to a file from the beginning
./noisy_script.sh > /tmp/noisy.log 2>&1 &
disown %1
```

Or use nohup, which redirects output automatically:

```bash
nohup ./noisy_script.sh &
```

If you're in the situation where you already started a process without redirecting and need to rescue it, the right tool is `reptyr` (which reattaches a running process to a new terminal) or gdb's process attachment to close/redirect file descriptors. These are emergency measures. The lesson is: redirect before you background.

### Suspended Jobs Hold Resources

You press `Ctrl+Z` on a database client that has an active transaction open, and go do other things. Hours later you remember it's suspended.

A stopped job is not a dead job. The process is frozen, but everything it had open stays open. That means:

- File locks are held
- Database transactions remain open (potentially blocking other writers)
- Network connections are maintained (or timing out on the remote side)
- Sockets are open

Some resources survive the suspension gracefully. Others don't. TCP connections that have gone idle may be torn down by firewalls or the remote host's keepalive timeout while the process is stopped — when you `fg` the job, it will immediately see a broken connection.

The mistake is treating `Ctrl+Z` like a pause button on a video. It is more like freezing someone mid-conversation — the other party is still waiting.

If you need to pause something long-term, bring it to a known clean state first (commit the transaction, close the file), then suspend it. Or better: cancel it and restart properly.

### wait With Multiple PIDs Returns Only the Last Exit Code

You have three background processes and you want to check if any of them failed:

```bash
task_a &
task_b &
task_c &
wait $pid_a $pid_b $pid_c
echo "Exit code: $?"
```

This does *not* tell you if task_a or task_b failed. `wait` with multiple PIDs waits for all of them, but returns only the exit code of the **last** PID in the list. If task_a exited 1 and task_c exited 0, `$?` will be 0 — the failure is silently dropped.

Wait for each PID individually to capture all exit codes:

```bash
task_a & pid_a=$!
task_b & pid_b=$!
task_c & pid_c=$!

wait "$pid_a"; code_a=$?
wait "$pid_b"; code_b=$?
wait "$pid_c"; code_c=$?

if (( code_a || code_b || code_c )); then
    echo "At least one task failed"
fi
```

Alternatively, use bash 4.3+'s `wait -n` to collect results as jobs complete, rather than waiting in a fixed order.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | GNU parallel: `apt install parallel` — installs as `/usr/bin/parallel`. Note: the `moreutils` package also provides a command called `parallel` but it is a simpler, different tool. Install the `parallel` package, not `moreutils`, for GNU parallel. |
| RHEL / Fedora / CentOS / Rocky | GNU parallel: `dnf install parallel`. Available in EPEL for older RHEL versions. |
| Arch | GNU parallel: `pacman -S parallel` |
| Alpine (containers) | GNU parallel: `apk add parallel`. `nohup` is provided by busybox and is present by default; behavior is slightly simplified compared to GNU coreutils `nohup`. |

Job control behavior (`jobs`, `fg`, `bg`, `disown`, `wait`) is a bash feature and behaves identically across all distros. The shell version matters more than the distro — check `bash --version` if you need features like `wait -n` (bash 4.3+) or `wait -p` (bash 5.1+).

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `nohup cmd &` | Start a new command that should survive session end | POSIX-standard; redirects output to `nohup.out` automatically |
| `disown %job` | Detach a job that's already running in the background | Bash builtin; does not redirect output — combine with explicit redirection |
| `tmux` / `screen` | Long interactive sessions, multiple windows, re-attachment | Better for interactive work; overkill for non-interactive scripts |
| `systemd-run --user` | Background one-off tasks with proper tracking and logging | Logs go to journald; survives properly; requires systemd |
| `xargs -P N` | Parallel execution of simple commands over a list of inputs | Built-in everywhere; no output ordering; minimal features |
| GNU `parallel` | Controlled concurrency with progress, retry, ordered output | Must be installed separately; far more capable than `xargs -P` |

For running a single command that should survive logout: `nohup cmd > output.log 2>&1 &` is the portable, reliable choice. For complex long-running workflows where you want to interact with them, attach and detach, and maintain terminal state, `tmux` is the professional tool.

For parallel workloads: if you need more than "run N copies simultaneously and don't care about the details," use GNU parallel. `xargs -P` is a Swiss Army knife — useful for quick jobs, but its scrambled output and lack of error handling will bite you on anything that matters.

## Trade-offs

**Where job control shines:**

- Interactive work — suspending and resuming tasks from the same terminal session is fast and requires no external tools
- Quick background jobs in scripts — `cmd & wait $!` is the simplest way to run something and check its exit code
- Low overhead — the shell's job table is trivial; there is no daemon, no configuration, no extra process

**Where job control costs you:**

- No persistence across sessions — job table lives in the shell; if bash exits, the table is gone. `nohup`/`disown` prevent job *death* but don't let you re-inspect the job table after reconnecting
- No structured output coordination — background jobs write directly to the terminal; if you background multiple verbose jobs, their output will interleave randomly
- Limited observability — you can see job state with `jobs`, but not CPU/memory usage; for that you need `ps` or `top`
- `nohup` output is append-only to a single file — if you run `nohup cmd &` twice, both processes append to the same `nohup.out`; this is a common source of confusion

---

## When to Use / When to Avoid

✅ **Use `&` when:**
- You're starting a background task in a script and will use `wait` to collect its result
- You want to start multiple parallel tasks and coordinate completion

✅ **Use `Ctrl+Z` + `bg` when:**
- You started something in the foreground by accident and want to reclaim the terminal
- You want to resume a suspended task in the background immediately

✅ **Use `nohup cmd > file 2>&1 &` when:**
- You're starting a non-interactive process that needs to survive session end
- You want output captured to a file automatically

✅ **Use `disown` when:**
- A job is already running and you need to remove it from the shell's control before logging out
- You've already redirected output to a file (otherwise disown alone is insufficient)

✅ **Use `wait` in scripts when:**
- You need the exit code of a background process
- You want a script to block until all background tasks complete before proceeding

❌ **Avoid `Ctrl+Z` when:**
- The process holds database transactions or exclusive locks — suspend only when the process is in a clean, idle state
- You intend to log out immediately after — use `bg` then `disown` or `nohup` from the start

❌ **Avoid `nohup` alone (without `&`) when:**
- You want to run in the background — `nohup` without `&` still blocks your terminal

❌ **Avoid `xargs -P` when:**
- Output order matters — use GNU `parallel --keep-order` instead
- You need to detect and retry individual failures — `xargs -P` has no per-job error handling

⚠️ **Common mistakes:**
- Forgetting to redirect output before using `disown` — the process will eventually get EIO errors
- Running `wait` without capturing individual exit codes — only the last PID's code is returned when multiple PIDs are passed
- Using `Ctrl+C` when you meant `Ctrl+Z` — Ctrl+C sends SIGINT and kills the process; Ctrl+Z sends SIGTSTP and suspends it

## Veteran Perspective

Job control is one of those topics where beginners learn the commands and intermediate engineers learn what the commands actually do. There is a frustrating gap between them.

The single most common real-world mistake is the `disown`-without-redirect pattern. Someone starts a long-running process, backgrounds it, types `disown`, logs out — and then their process dies silently when it tries to write to a terminal that no longer exists. They come back the next morning and the process isn't running and there's no output file to inspect. The lesson is always the same: redirect *before* you detach.

The `nohup` confusion runs a close second. `nohup` is not the same as "run in the background." It is "run immune to SIGHUP." Without `&`, you've made a foreground process that can't be killed by HUP. That's rarely what anyone wanted. The correct invocation is always `nohup cmd > output.log 2>&1 &` — three separate concerns addressed in one line.

The `wait` exit-code subtlety is a real bug source. I've reviewed scripts that background four processes, call `wait $pid1 $pid2 $pid3 $pid4`, check `$?`, and assume they're checking whether everything succeeded. They're only checking the last process. The other three could have all failed silently. If you have a fanout-then-gather pattern, collect each exit code individually.

For anything beyond simple background jobs — real production background work — job control is the wrong layer. Job control lives in the shell and dies with the shell. Anything you care about enough to monitor or restart should be in systemd as a proper service or a user service (`systemctl --user`). Job control is for interactive convenience; systemd is for reliability. The professional move is knowing which one you need.

On the GNU parallel vs `xargs -P` question: `xargs -P` is already on the machine, so use it for throw-away one-liners (`find . -name "*.gz" | xargs -P4 -I{} gunzip {}`). For anything you're putting in a script, spending thirty seconds to `apt install parallel` is worth it — the output ordering and per-job error codes alone pay back the installation cost within the first real failure.

## Try It Yourself

1. Open a terminal and run `sleep 300`. While it's running, press `Ctrl+Z`. Then run `jobs`. Then run `bg %1`. Run `jobs` again and observe the state change from Stopped to Running.

2. Start two background sleeps: `sleep 200 &` and `sleep 201 &`. Run `jobs -l`. Identify the `+` and `-` markers. Run `fg` with no argument — which job comes to the foreground? Press `Ctrl+Z` again, then run `fg %1` vs `fg %2` to bring a specific job forward.

3. Write a four-line script that starts two background `sleep` commands, captures both PIDs into variables, then calls `wait "$pid1"` and `wait "$pid2"` separately, printing each exit code. Run it and verify both codes are 0. Then modify one sleep to be replaced by `false &` and verify you capture the non-zero exit code.

4. Run `nohup sleep 500 > /tmp/nohup-test.log 2>&1 &`. Verify the job is running with `jobs`. Then run `disown %1`. Run `jobs` again and confirm the job is gone from the table. Confirm the process is still alive with `ps aux | grep sleep`. Check `cat /tmp/nohup-test.log` — nohup adds a header line.

5. **Challenge:** Write a script that processes files in parallel using background jobs. Take a directory of text files, run `wc -l` on each file in the background, collect all PIDs, wait for each one individually, and report total success or failure. Cap concurrency at 4 simultaneous jobs — use a counter and `wait -n` (bash 4.3+) to throttle. Test it in a VM with a directory of 10+ files. This pattern is the foundation of every batch-processing script you'll write.

## Summary

- A **job** is a pipeline tracked by the shell; it maps to a process group. `Ctrl+Z` sends SIGTSTP to the entire foreground process group, suspending all processes in the pipeline simultaneously.
- `jobs -l` lists all jobs with their PIDs; `+` is the current job (default for `fg`/`bg`), `-` is the previous job. A Stopped job holds all its resources — file locks, connections, memory — until resumed or killed.
- `nohup cmd > file 2>&1 &` is the correct combined form: `nohup` sets SIGHUP to SIG_IGN and redirects terminal output; `&` backgrounds the process. Neither alone does the full job.
- `disown` removes a job from the shell's table so the shell won't send it SIGHUP on exit — but it does nothing to the process's stdout/stderr. Always redirect output before using `disown`.
- `wait $pid` returns the exit code of a specific background process. `wait` with multiple PIDs returns only the last one's code. In scripts, wait for each PID individually to catch all failures.
