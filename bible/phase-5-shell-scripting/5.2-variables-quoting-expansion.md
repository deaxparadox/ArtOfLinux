# [5.2] — Variables, Quoting, and Expansion

## The Problem This Solves

Early shells treated everything as text. You had no real variables, no substitution, no way to reuse a value or build a command from parts. You wrote fixed strings and hoped the world didn't change around them.

Then scripting arrived and brought variables — and suddenly the shell needed rules for how to treat the text a variable contained. Those rules are quoting and word splitting, and they are the source of a majority of real-world shell bugs.

Here is a script that looks completely correct to a reasonable person:

```bash
FILE="my report.txt"
cp $FILE /backup/
```

It doesn't copy `my report.txt` to `/backup/`. It tries to copy a file called `my` and a file called `report.txt`, then receives an argument `/backup/` — three arguments instead of two. No syntax error. No obvious failure if either partial filename exists somewhere. The bug is silent and wrong.

The fix is one pair of quotes. But the reason the fix works — and why leaving them off breaks things — requires understanding how the shell expands variables, what word splitting does, and when quoting suppresses that behavior. That is what this topic is about.

---

## Mental Model

Think of a variable in bash like a sticky note with text written on it. When you use `$VAR`, the shell peels off the sticky note and pastes the raw text directly into the command line — as if you had typed that text yourself, with no changes.

If the sticky note says `my report.txt`, pasting it raw into the command gives you three words: `my`, `report.txt` split at the space. Quoting the sticky note — `"$VAR"` — is like laminating it before you paste it: the shell pastes the entire note as one unit, preserving the space inside.

No lamination: the shell shreds the note on every space and newline. With lamination: the note arrives whole. Everything about quoting in bash flows from this one idea.

## Theory

### Variables are just named text

A bash variable stores a string. That's it. There is no integer type, no float, no bool at the language level — unless you use `declare` attributes to impose behavior. When you write `COUNT=5`, you are storing the four-character string `"5"`, not the number five. Arithmetic works because bash converts strings to numbers when it needs to, inside `$(( ))` expressions.

Variable names must start with a letter or underscore, followed by letters, numbers, or underscores. `MY_VAR`, `_count`, `file2` are valid. `2count`, `my-var` are not.

### Assignment has no spaces around `=`

This is one of the first things that catches people:

```bash
VAR=value    # correct — this is an assignment
VAR = value  # incorrect — this tries to run a command named VAR with arguments = and value
```

The space changes the meaning entirely. The shell interprets `VAR = value` as "run a command called `VAR`" because the parser sees `VAR` at the start of the line followed by whitespace. Assignment is a special form recognized only when there is no whitespace around `=`.

### Quoting modes: three levels of protection

Bash has three quoting mechanisms with distinct semantics:

**No quotes**: the raw value is subject to word splitting (split on IFS characters) and pathname expansion (globbing). Spaces in the value become argument separators. `*` becomes a list of filenames.

**Double quotes** (`"`): variables and command substitutions inside expand normally. Word splitting and globbing are suppressed. The value arrives as a single word even if it contains spaces or special characters. Single quotes, backslashes, and backticks still have meaning inside double quotes.

**Single quotes** (`'`): everything between them is completely literal. No variable expansion, no command substitution, no escape sequences. A single-quoted string cannot contain a single quote — there is no escape mechanism inside single quotes.

This three-way distinction is not an accident. Single quotes are for truly literal text (regular expressions, awk programs, any string where `$` and `\` must not be interpreted). Double quotes are for strings that should expand variables but arrive as a single word. No quotes is for cases where you deliberately want word splitting — which is almost never the case with variables.

### Word splitting: the engine under the bug

After variable expansion and command substitution, the shell looks at the result and splits it into separate words anywhere it finds a character from `IFS` (Internal Field Separator). The default value of IFS is a space, a tab, and a newline — in that order. Any sequence of those characters in the expanded value creates word boundaries.

There is a subtle detail: when IFS is the default whitespace value, leading and trailing whitespace is stripped, and multiple consecutive whitespace characters collapse to a single split. When IFS contains non-whitespace characters, each individual occurrence creates exactly one split (a field separator). This distinction matters when parsing CSV or colon-delimited strings.

Double-quoting a variable expansion suppresses word splitting entirely. The shell does not look at IFS for anything inside double quotes.

### IFS is a variable you can change

```bash
IFS=:     # now the shell splits on colons instead of whitespace
```

This is how `read` with a custom delimiter works, and how `for item in $PATH` can iterate over PATH entries — though both patterns have better alternatives. Changing IFS globally in a script is dangerous because it affects all subsequent expansions and reads. If you need a custom delimiter for one operation, scope the change:

```bash
# Change IFS only for this one read, then restore it
old_IFS="$IFS"
IFS=: read -ra parts <<< "$PATH"
IFS="$old_IFS"
```

Or use a subshell to contain the change entirely.

### The shell's expansion order

Understanding what expands before what prevents an entire category of bugs. The shell processes each command through these stages in strict order:

```text
# Conceptual flow — not a real command
1. Brace expansion          {a,b,c}  →  a b c
2. Tilde expansion          ~        →  /home/user
3. Parameter expansion      $VAR     →  the value
4. Command substitution     $(cmd)   →  the output
5. Arithmetic expansion     $((3+2)) →  5
6. Word splitting           split on IFS (only unquoted expansions)
7. Pathname expansion       * ? []   →  matching filenames (only unquoted)
```

Quoting suppresses steps 6 and 7 for the quoted region. Brace expansion and tilde expansion happen before quoting takes effect — they are not suppressible by double quotes.

## How It Works (Mechanics)

### What the shell does with `$VAR` vs `"$VAR"`

When the shell encounters `$VAR` in a command, it replaces the token with the current value of the variable. The replacement is textual — the value is inserted as raw text. Then word splitting runs over the entire command, splitting on IFS characters.

For `"$VAR"`, the shell still performs the expansion, but the double-quote context flags the result as a single unit. The word-splitting stage sees it as one token and does not split it regardless of what characters the value contains.

You can observe this directly:

```bash
FILE="my file.txt"

# Without quotes: the shell passes three arguments to ls
ls -la $FILE
# ls receives: -la, my, file.txt

# With quotes: the shell passes two arguments to ls
ls -la "$FILE"
# ls receives: -la, my file.txt
```

If `my` or `file.txt` happen to exist as separate files, the first version silently succeeds for the wrong reasons. That's the most dangerous kind of bug.

### How parameter expansion is processed

When the shell sees `${VAR:-default}`, it parses the `:-` operator and evaluates the condition before substituting. The expansion itself — the result — is then subject to word splitting and globbing just like `$VAR` would be, unless you quote it. This is a common oversight: people quote `$VAR` everywhere but forget to quote `${VAR:-default}`.

```bash
DEFAULT="some default value"
echo ${UNSET:-$DEFAULT}    # word splitting can still occur on the result
echo "${UNSET:-$DEFAULT}"  # correct — quote the entire expansion
```

### How `declare` sets variable attributes

`declare` is a shell builtin that creates variables with type constraints and attributes. The shell enforces these constraints at assignment time, not at expansion time. When you `declare -i COUNT`, any subsequent `COUNT=foo` silently sets `COUNT` to `0` (because `foo` is not a valid integer), and `COUNT=3+2` sets it to `5` (arithmetic evaluation happens automatically). The variable still expands as a string — `"$COUNT"` gives you `"5"`, not the number 5 — but assignments are interpreted arithmetically.

```bash
declare -i N
N=hello      # silently becomes 0
N=3+2        # silently becomes 5
echo $N      # prints: 5
```

## Basic Example

```bash
# Assignment — no spaces around =
NAME="Alice"
GREETING="Hello"

# Reading a variable
echo "$GREETING, $NAME"

# Curly braces required when the variable name would be ambiguous
echo "${NAME}s"      # "Alices" — braces prevent the shell from looking for NAMEs
echo "$NAMEs"        # empty — the shell looks for a variable called NAMEs

# Length of a string
echo "${#NAME}"      # 5

# Default value — use the default only if NAME is unset or empty
echo "${UNSET_VAR:-nobody}"

# Substring removal — strip file extension
FILENAME="report.tar.gz"
echo "${FILENAME%%.*}"   # report — strips longest match from the right
echo "${FILENAME%.*}"    # report.tar — strips shortest match from the right
```

```text
Hello, Alice
Alices

5
nobody
report
report.tar
```

Run this yourself. Focus on the `$NAMEs` line — the empty output is the bug. The shell silently expanded a variable that doesn't exist to empty string, no error. Now try deliberately misspelling a variable name elsewhere and watch how it silently produces empty output instead of an error. That silence is the shell's most dangerous trait.

## Real-World Example

A deployment script that processes log files with spaces in their names and passes arguments through to another command safely:

```bash
#!/bin/bash
set -euo pipefail

# $1 is the log directory, $2 is the destination
LOG_DIR="${1:?Usage: $0 <log_dir> <dest_dir>}"
DEST_DIR="${2:?Usage: $0 <log_dir> <dest_dir>}"
TIMESTAMP=$(date +%Y%m%d-%H%M%S)   # command substitution — always prefer $() over backticks
ARCHIVE_NAME="${DEST_DIR}/logs-${TIMESTAMP}.tar.gz"

# Build a count of files to process
FILE_COUNT=$(find "$LOG_DIR" -name "*.log" -type f | wc -l)
echo "Found ${FILE_COUNT} log files to archive"

# Compress — quoting variables ensures paths with spaces work correctly
tar -czf "$ARCHIVE_NAME" "$LOG_DIR"

# Report the final size using string length alternative: let the tool do it
ARCHIVE_SIZE=$(du -sh "$ARCHIVE_NAME" | cut -f1)
echo "Archive created: $ARCHIVE_NAME (${ARCHIVE_SIZE})"
```

The `${1:?...}` form is a parameter expansion that exits with an error message if `$1` is unset or empty — it makes the script self-documenting and fail-fast. `$(date +%Y%m%d-%H%M%S)` uses command substitution to capture a timestamp; the `$()` form is always preferred over backticks because it is readable, nestable without escaping, and handles multi-line output consistently. Every variable that could contain a path is double-quoted so that directory names with spaces in them work correctly. This is what defensive bash scripting looks like.

## ⚠️ Gotchas

### Unquoted Variable With Spaces — Silent Argument Splitting

You have a file path in a variable. You pass it to a command. The command fails mysteriously, or worse, operates on the wrong files silently.

```bash
FILE="my report 2024.txt"
cp $FILE /backup/
```

```text
cp: cannot stat 'my': No such file or directory
cp: cannot stat 'report': No such file or directory
cp: cannot stat '2024.txt': No such file or directory
```

After `$FILE` expands to `my report 2024.txt`, word splitting runs over the unquoted result and splits it into three separate words at every space. `cp` now receives four arguments: `my`, `report`, `2024.txt`, and `/backup/`. The command tries to copy three separate files, none of which exist.

The underlying rule: every variable expansion that may contain whitespace must be double-quoted. No exceptions. The only time you intentionally leave a variable unquoted is when you want word splitting to happen — for example, when building a list of command arguments stored in a string (and even then, arrays are the right tool).

```bash
# Correct
cp "$FILE" /backup/
```

The habit to build: double-quote every `$VAR` by default. Remove the quotes only when you have a specific, understood reason to allow word splitting.

### `$@` vs `$*` in Double Quotes — Silent Argument Merging

You write a wrapper function that passes all its arguments through to another command. You test it. It works until someone passes an argument with a space.

```bash
run_backup() {
    # Intention: pass all arguments to rsync
    rsync "$*" /backup/
}

run_backup -av "my documents" /home/user/
```

```text
# rsync receives these arguments:
# -av my documents /home/user/
# "my documents" has been split — rsync fails on "my" and "documents" separately
```

`"$*"` joins all positional parameters into a single string, with the first character of IFS (a space, by default) as the separator. In double quotes, `$*` becomes one word — all arguments glued together. Every space inside any argument disappears into the join, and word boundaries between original arguments are lost.

`"$@"` expands each positional parameter as a separately double-quoted word. Each original argument arrives at the next command exactly as it was received, spaces and all, with no merging.

```bash
run_backup() {
    # Correct: "$@" preserves each argument as a separate word
    rsync "$@" /backup/
}
```

The rule is absolute: use `"$@"` when passing arguments through. The only use for `"$*"` is building a single joined string for display — never for passing arguments to a command.

### Spaces Around `=` — Command Instead of Assignment

You type a variable assignment with spaces around the `=` sign, the way you would write it in Python or most other languages.

```bash
MY_VAR = "hello"
```

```text
bash: MY_VAR: command not found
```

The shell treats `MY_VAR` as a command name, `=` and `"hello"` as its arguments, and tries to execute it. No assignment happens. This is not a quirk — it is the language's grammar. Assignment requires `NAME=value` with no whitespace on either side of `=`. The `=` in a command context (after whitespace) is just a regular character.

```bash
# Correct
MY_VAR="hello"
```

This also means `MY_VAR =value` (space before `=`) fails the same way. There is no tolerance. The rule is zero spaces on either side of `=`.

### `${VAR:-default}` vs `${VAR-default}` — The Colon Changes the Trigger Condition

You use a parameter expansion to provide a fallback value. It works fine until someone sets the variable to an empty string, and then your fallback silently disappears.

```bash
USER_DIR=""
echo "${USER_DIR-/tmp}"   # no colon — only triggers on UNSET, not empty
```

```text
# Empty output — the variable is set (to ""), so the default is NOT used
```

Without a colon, the `:-`, `:=`, `:+`, and `:?` forms check for unset OR empty string. Without the colon, they only check for unset. If the variable is set to an empty string, the without-colon form considers it "set" and does not apply the default.

```bash
echo "${USER_DIR:-/tmp}"   # colon — triggers on unset OR empty string
```

```text
/tmp
```

The colon-forms are almost always what you want. An empty variable and an unset variable usually mean the same thing in practice — "this value was not provided." The colon form treats them identically. The without-colon form is for the rare case where an explicitly empty value is meaningful and must be preserved. If you're not deliberately using that distinction, always use the colon form.

### Backtick Nesting Requires Escaped Escapes

You try to nest one command substitution inside another using backticks. It becomes a quoting nightmare.

```bash
# Try to get the directory containing the file found by find
PARENT=`dirname \`find /etc -name "passwd" -maxdepth 1\``
```

```text
# Works, barely — but each level of nesting doubles the backslash count
# Adding a third level requires \\\ — four levels requires \\\\ — untenable
```

Backtick nesting requires manually escaping backslashes at every level. Two levels is painful. Three levels is write-only code. The `$()` form nests cleanly because it uses balanced delimiters instead of matching the same character:

```bash
# Clean nesting with $() — no escaping required
PARENT=$(dirname $(find /etc -name "passwd" -maxdepth 1))

# Or with a pipe for clarity
PARENT=$(find /etc -name "passwd" -maxdepth 1 | head -1 | xargs dirname)
```

There is no scenario where backticks are better than `$()`. The backtick form exists for historical compatibility with older POSIX `sh` implementations. In any script using `#!/bin/bash`, always use `$()`.

### Forgetting `declare -A` for Associative Arrays — Keys Silently Collapse

You write what looks like an associative array. It compiles without error. But every value overwrites the previous one.

```bash
# Looks like an associative array — it is not
declare server_info
server_info[hostname]="web01"
server_info[port]="8080"
server_info[region]="us-east"

echo "${server_info[hostname]}"
echo "${server_info[port]}"
```

```text
us-east
us-east
```

Without `declare -A`, `server_info` is an indexed array (`declare -a` is the default). Indexed arrays require integer keys. When bash sees a non-integer key like `hostname`, it evaluates it as an arithmetic expression — `hostname` is an undefined variable, so it evaluates to `0`. Every string key maps to index `0`, and each new assignment overwrites the previous one at index `0`. The last value wins silently.

```bash
# Correct — you must declare -A explicitly for associative arrays
declare -A server_info
server_info[hostname]="web01"
server_info[port]="8080"
server_info[region]="us-east"

echo "${server_info[hostname]}"  # web01
echo "${server_info[port]}"      # 8080
```

Associative arrays require bash 4.0+. The `-A` flag to `declare` is not optional — it is the only way to create an associative array in bash.

## Distro Notes

| Distro Family | Bash Version / Notes |
|---|---|
| Debian / Ubuntu (recent LTS) | Bash 5.1 or 5.2; `declare -n` (nameref, 4.3+) and `${var@Q}` (4.4+) are available |
| RHEL 8 / CentOS 8 / Rocky 8 | Bash 4.4 — all features in this topic are available; `${var@Q}` is available |
| RHEL 7 / CentOS 7 | Bash 4.2 — `declare -n` is NOT available; `${var@Q}` is NOT available; avoid both |
| Alpine (containers) | Default shell is `ash` (BusyBox) — no arrays, no `declare`, no `$((expr))` in the same form; install `bash` explicitly with `apk add bash` and use `#!/bin/bash` |
| Arch Linux | Rolling release; current bash; all features available |
| macOS | System bash is 3.2 (frozen due to GPL3 refusal) — no associative arrays, no `declare -n`, no `${var@Q}`; install bash via Homebrew if you need modern bash on a Mac |

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| `$VAR` bare | Intentional word splitting on the result (rare) | Almost always a bug when the value might contain spaces; prefer arrays when you want splitting |
| `"$VAR"` double-quoted | Default for virtually all variable use | Correct 95% of the time; the habit to build |
| `'literal'` single-quoted | Regex patterns, awk programs, strings where `$` must not expand | Nothing inside expands; cannot embed a single quote without ending the quoting |
| `$()` command substitution | Capturing command output | Always prefer over backticks; nests cleanly |
| backticks `` ` `` | Never, in new code | Historical only; nesting is unusable; use `$()` |
| `declare -A` associative array | Key-value mapping in scripts | Bash 4.0+ only; not available in POSIX sh or bash 3.x |
| `declare -n` nameref | Passing variable names to functions | Bash 4.3+ only; cleaner than `eval` for indirect variable references |

## Trade-offs

**Where bash variable expansion shines:**
- Inline string manipulation (prefix/suffix stripping, substitution) without spawning a subprocess — `${FILE%.gz}` is faster than `$(echo $FILE | sed 's/.gz//')`
- The default-value forms (`:-`, `:=`, `:?`) make scripts self-documenting and defensively fail-fast at minimal cost
- Associative arrays handle structured data without requiring Python or awk

**Where bash variable expansion costs you:**
- Quoting is manual and required everywhere — there is no safe-by-default mode; forgetting a single pair of quotes produces a bug with no error message
- There is no type system beyond `declare` attributes; `"5"` and `5` are the same until you're inside `$(( ))`, which silently treats undeclared variables as zero
- `declare -A` associative arrays do not preserve insertion order and cannot be exported to child processes
- Expansion behavior depends on bash version — scripts that use `${var@Q}` or `declare -n` silently fail or error on older bash installations (RHEL 7, macOS system bash)

**Performance note:** Shell parameter expansion like `${VAR#prefix}` and `${VAR/old/new}` runs in the current shell process with no fork or exec. Using `sed` or `awk` for the same transformation spawns a subprocess per call. In a tight loop processing thousands of strings, the difference is measurable — parameter expansion is the right choice.

## When to Use / When to Avoid

✅ **Use `"$VAR"` double-quoting as your default:**
- Any time a variable might contain spaces, tabs, newlines, or glob characters
- Any path variable: `$HOME`, `$TMPDIR`, `$1`, `"$@"`, user-supplied inputs

✅ **Use single quotes when:**
- You need a truly literal string: regular expressions, awk programs, any string where `$` and `\` must be literal characters

✅ **Use `${VAR:-default}` (colon form) when:**
- You want to substitute a default for both unset and empty — which is almost always

✅ **Use `"$@"` instead of `"$*"` when:**
- Passing arguments through to another command — always

✅ **Use `$()` instead of backticks when:**
- Any command substitution — always, in any new code

❌ **Avoid bare `$VAR` (unquoted) when:**
- The value might contain whitespace or glob characters — use `"$VAR"` instead

❌ **Avoid `declare -n` (nameref) when:**
- Targeting systems that might run bash 4.2 or older (RHEL 7, old macOS)
- The safer alternative is passing a variable name as an argument and using `eval` — but `eval` carries its own injection risks

❌ **Avoid `${var@Q}` and `${var@E}` when:**
- RHEL 7 or any bash older than 4.4 is in the picture; use `printf '%q'` for quoting instead

⚠️ **Common mistakes:**
- Using `${#array[@]}` for a normal variable when you meant `${#var}` — `${#array[@]}` gives element count, `${#var}` gives string length
- Assigning the output of `$(cmd)` without quoting: `VAR=$(cmd)` is safe — command substitution as a direct assignment does not trigger word splitting; `VAR=$(cmd); other_cmd $VAR` is not
- Using `declare -A` inside a function with `local` — use `local -A name` instead of `declare -A name` inside functions, so the array is properly scoped

## Veteran Perspective

The single biggest source of shell script bugs across every organization I've worked in is unquoted variable expansion. Not missing error handling, not logic errors, not wrong commands — unquoted variables. The pattern is always the same: the script was written and tested with clean inputs, every filename was a single word, every path had no spaces, and it worked fine for months or years. Then one day someone created a directory called "Q4 Reports" or a config file with a space in the path, and the script silently corrupted something or failed in a way that pointed nowhere near the actual cause.

The fix is a habit, not a rule to memorize: double-quote every variable expansion by default. `"$VAR"`, not `$VAR`. `"$1"`, not `$1`. `"${FOO:-default}"`, not `${FOO:-default}`. You remove the quotes only when you have a deliberate, understood reason. With that habit, you will not write this class of bug.

The `$@` vs `$*` confusion trips up people at every level. I've reviewed production code from senior engineers with ten years of shell experience who wrote `"$*"` in wrapper scripts. The bug only surfaces when someone passes an argument with a space, which in a production system is eventually certain. The mental model is simple: `$@` treats each argument as its own word; `$*` glues them all into one. For passing arguments through, `"$@"` is always correct. Burn it in.

Here's the intermediate-level trap that most tutorials miss: `${VAR:-default}` does not protect you from word splitting. The expansion produces a result, and if you don't quote the whole thing, that result is still split on IFS. People learn to use default-value forms, feel safe, and then leave the expression unquoted. The value from the default is just as susceptible to word splitting as the value from the variable. Quote the whole expression: `"${VAR:-default}"`.

The `declare -A` silent key-collision bug is one I've seen cost teams a real debugging session. The script runs, produces output, no errors, just wrong data. If you don't know that bash requires explicit `declare -A` for associative arrays and silently treats string keys as integer index `0`, you will be very confused about where all your data went. Test associative array declarations with a sanity check: after populating the array, print all its keys with `echo "${!myarray[@]}"` — if you only see `0`, you forgot `-A`.

On the advanced end: `declare -n` (namerefs, bash 4.3+) is genuinely useful for writing reusable functions that modify the caller's variables by reference. Before namerefs, you did this with `eval`, which is a shell injection vulnerability waiting to happen. Namerefs are safer and readable. But they don't work on bash 4.2 (RHEL 7 shipped with 4.2), and they have a sharp edge: if the nameref variable and the referenced variable share a name — `declare -n foo=foo` — bash detects the self-reference and errors out. Use a naming convention that makes collision unlikely, like a `_ref` suffix.

When you see a shell script that's been in production for several years and does important work, look at how its variables are quoted. That's a faster proxy for the author's shell expertise than any other single thing.

## Try It Yourself

1. Open a terminal and run these three lines:
   ```bash
   FILE="a b c"
   echo $FILE
   echo "$FILE"
   ```
   Count the arguments each `echo` receives. Now run `printf '%s\n' $FILE` and `printf '%s\n' "$FILE"` and observe how many lines each produces.

2. Try all three quoting modes on the same string and observe the differences:
   ```bash
   NAME="world"
   echo "Hello $NAME"
   echo 'Hello $NAME'
   echo Hello $NAME
   ```
   Then add a file to your current directory called `world` (use `touch world`) and re-run the unquoted version. Observe that nothing changes for `echo` — now try it with `ls` instead.

3. Write a small wrapper function that demonstrates the `"$@"` vs `"$*"` difference:
   ```bash
   show_args() { printf 'ARG: [%s]\n' "$@"; }
   show_args "hello world" foo bar
   ```
   Replace `"$@"` with `"$*"` and observe what changes in the output. Note exactly how many ARG lines you get in each case.

4. Build a small parameter expansion toolkit. For a variable `PATH_STR="/var/log/app/server.log.gz"`, use only parameter expansion (no `sed`, no `awk`) to extract: the filename (`server.log.gz`), the extension (`gz`), the name without any extension (`server`), and the directory path (`/var/log/app`). No spawning subprocesses — everything must be a `${...}` expression.

5. **Challenge:** Write a script called `safe_copy.sh` that takes two arguments (source and destination). The script must: (a) fail with a meaningful error message if either argument is missing, using `${1:?...}` and `${2:?...}`; (b) check that the source file exists before copying; (c) build the destination path as `${dest_dir}/${basename}_${timestamp}` where timestamp comes from `$(date +%s)`; (d) work correctly when the source path contains spaces. Test it by creating a file with a space in the name: `touch "test file.txt"`. Run it in a VM since it copies files.

## Summary

- Every unquoted variable expansion is subject to word splitting on IFS (default: space, tab, newline) and pathname expansion; double-quoting `"$VAR"` suppresses both and must be the default habit
- `"$@"` expands each positional parameter as a separately quoted word; `"$*"` joins all of them into one string — always use `"$@"` when passing arguments through to another command
- `${VAR:-default}` (with colon) triggers on both unset and empty string; `${VAR-default}` (without colon) triggers only on unset — the colon form is almost always what you want
- `$(cmd)` is always preferred over backtick command substitution: it nests without escaping, reads clearly, and handles edge cases more consistently
- Associative arrays require an explicit `declare -A` — without it, string keys silently map to integer index `0` and overwrite each other with no error
