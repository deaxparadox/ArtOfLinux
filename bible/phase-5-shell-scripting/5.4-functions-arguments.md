# [5.4] — Functions, Arguments, and Exit Codes

> A shell script without functions is a recipe; a shell script with functions is a program — and the difference matters the moment something goes wrong at 2am.

## The Problem This Solves

Picture a deployment script that restarts three services. Without functions, the restart logic is copy-pasted three times. When the restart procedure changes — and it always changes — you update it in one place, miss the other two, and now your script has inconsistent behavior that nobody notices until it matters.

That's the mild version of the problem. The worse version: you have 400 lines of sequential code and something fails in the middle. The script stops, or doesn't stop, and you have no idea which logical step failed because every step is just the next line in a wall of commands. There is no structure to the failure.

Functions solve the structural problem. But they introduce their own problem: every function has to report success or failure back to its caller. A function that silently fails is worse than no function at all — it gives your script the appearance of structure while hiding the actual behavior.

That reporting mechanism — exit codes — is what this topic is about. Functions and exit codes are one concept. You cannot use one correctly without the other.

---

## Mental Model

Think of a function as a specialist you call on the phone. You describe what you need, they do the work, and they call back with one of two things: a status ("done", "failed") or an answer ("the value is X").

The phone call is the function call. The status they report is the exit code. If they have an answer to give you — not just success or failure, but actual data — they don't say it as a status, they say it as a separate message, and you write it down.

The critical constraint: the phone only carries one status at a time, and the status is a number between 0 and 255. If you want them to give you a string — a filename, a computed value, an error message — that comes through a different channel. You have to ask them to write it on a shared notepad (`echo` to stdout, captured with `$(...)`), not report it as their status.

## Theory

### What a Function Is

A function is a named block of commands that executes in the current shell process. It is not a subprocess, not a separate script, not a separate binary — it is code that runs in your shell's memory, sharing your shell's environment, unless you deliberately isolate it.

That sharing is the source of both functions' power and their most common bugs.

### Syntax

Bash accepts two equivalent function definition forms:

```bash
greet() {
    echo "Hello, $1"
}
```

```bash
function greet {
    echo "Hello, $1"
}
```

Both define a function named `greet`. The first form is POSIX-compatible; the second is bash-specific. The first is preferred in any script that might need to run under `/bin/sh`.

The function body is not executed when the definition is processed. The shell reads the definition and stores it. The body runs only when you call `greet`.

### Arguments

When you call a function, the positional parameters `$1`, `$2`, `$3`, ... refer to the arguments passed to that function — not to the script's own arguments. The script's original positional parameters are temporarily replaced for the duration of the function call. When the function returns, the script's positional parameters are restored.

`$@` expands to all arguments as separate quoted words. `$*` expands to all arguments as a single string. `$#` holds the argument count. In practice, always use `$@` when you need all arguments — `$*` silently coalesces arguments that contain spaces, which breaks things in non-obvious ways.

### Return Values — Two Completely Different Things

This is where more bugs originate than anywhere else in shell scripting.

`return N` sets the function's exit code to N. That exit code is a number between 0 and 255. Zero means success. Any non-zero value means failure. When the caller tests the function in an `if` statement or captures `$?`, they see this number. That is all `return` does — it sets an integer status and exits the function.

`return N` does not return a string. It does not return a path. It does not return a computed value. If you write `return "some_value"`, bash ignores the string and returns 0.

To return actual data from a function, you `echo` it to stdout and the caller captures it with command substitution:

```bash
result=$(get_value)
```

This runs `get_value` in a subshell, collects its stdout, and assigns it to `result`. The exit code of the function is still available in `$?` immediately after the substitution.

### Variable Scope and `local`

Without the `local` keyword, every variable set inside a function is a global variable. It remains visible after the function returns and can collide with variables of the same name elsewhere in the script.

`local varname` declares a variable whose scope is the current function and any functions called from it. When the function returns, the local variable is destroyed. The global variable of the same name, if one existed, is restored.

Bash uses dynamic scope for `local`, not lexical scope. This means a function called from within the function that declared the local variable can see and modify that local variable. This is different from most programming languages and it matters.

`local` only works inside functions. Calling `local` at the top level of a script produces an error.

### Exit Codes — the Language of Success and Failure

Every command in bash terminates with an exit code. By convention:

- **0** — success
- **1** — general error (the command ran but reported failure)
- **2** — misuse of shell or command (wrong arguments, invalid syntax)
- **126** — the command was found but is not executable (permission denied)
- **127** — the command was not found (not installed, not in PATH, typo)
- **128+N** — terminated by signal N (e.g., 130 = killed by Ctrl+C, which is SIGINT = signal 2; 137 = killed by SIGKILL = signal 9)

Exit codes above 128 are how bash communicates signal terminations. If a script exits with 137, it was killed, not a normal failure. If it exits with 130, the user pressed Ctrl+C. These codes are how monitoring systems and CI pipelines tell the difference between "the script reported an error" and "the process was externally terminated."

### `$?` — Capturing Exit Codes

`$?` holds the exit code of the most recently completed foreground command. "Most recently completed" is the key phrase. Every command resets it. Every assignment resets it. Every `echo` resets it. If you do not capture `$?` on the very next line after the command you care about, you will read the exit code of whatever ran between.

### `source` — Running a Script in the Current Shell

`source script.sh` (or equivalently `. script.sh`) runs the script's commands inside the current shell process, not in a subshell. Any variables defined in the sourced file become variables in the current shell. Any functions defined in the sourced file become available in the current shell. Any `exit` in the sourced file exits the calling shell, not just the sourced file.

This is how libraries work in shell scripting: you write a file of function definitions and `source` it at the top of your main script.

Executing a script (`bash script.sh` or `./script.sh`) runs it in a child process. Changes to variables, functions, and the working directory in the child process never affect the parent.

## How It Works (Mechanics)

### What the Shell Does When a Function Is Called

When bash encounters a function call, it does not fork a new process. It does the following in the current process:

1. Saves the current values of `$1`, `$2`, ..., `$#`, `$@`, `$*`
2. Sets those variables to the arguments passed to the function
3. Sets `$FUNCNAME[0]` to the function's name (pushing the previous value to `$FUNCNAME[1]`)
4. Executes the function body
5. On `return N` (or on falling off the end of the body), sets `$?` to N
6. Restores the saved positional parameters
7. Pops `$FUNCNAME`

Because there is no fork, the function runs in the same memory space as the caller. There is no isolation unless you use `local`. A variable set without `local` is immediately visible to the caller after the function returns.

### The `FUNCNAME` Array — a Call Stack in Your Shell

`FUNCNAME` is an array bash maintains automatically whenever you're inside function calls. `${FUNCNAME[0]}` is the currently executing function. `${FUNCNAME[1]}` is the function that called it. The last element is always `main` (when the code at the top level of a script is running).

`FUNCNAME` is paired with `BASH_SOURCE` and `BASH_LINENO`. `${BASH_SOURCE[$i+1]}` is the file where `${FUNCNAME[$i]}` was called from, and `${BASH_LINENO[$i]}` is the line number. Together, these three arrays give you a full call stack trace.

```bash
stack_trace() {
    local i
    for (( i=1; i<${#FUNCNAME[@]}; i++ )); do
        echo "  $i: ${FUNCNAME[$i]} (${BASH_SOURCE[$i+1]:-main}:${BASH_LINENO[$i-1]})"
    done
}
```

This is exactly how an `ERR` trap can print a meaningful stack trace when a script fails.

### What `local` Actually Does — Dynamic Scope

When bash executes `local x=5` inside function `outer`, it pushes a new binding for `x` onto a variable stack. Functions called from `outer` see this binding. When `outer` returns, the binding is popped and whatever `x` was before is restored.

This is dynamic scope: the variable's value depends on the call chain, not the textual position of the code.

```text
# Conceptual flow — not a real command
outer() → local x=5
  inner() → reads x → sees 5 (the local from outer)
outer returns → local x is destroyed → x reverts to previous value
```

This is unintuitive if you come from Python or JavaScript where `local` (or `let`/`var`) is lexically scoped. In bash, a function called from within your function can silently see and modify your local variable if it uses the same name.

### How `getopts` Parses Options

`getopts` is a bash built-in that processes short options (`-v`, `-f filename`, `-vf`) one at a time. You call it in a loop:

```bash
while getopts "vf:" opt; do
    ...
done
```

Each call to `getopts` reads the next option from the positional parameters and stores the option letter in `opt`. `OPTARG` holds the argument for options that take one (marked with `:` after the letter in the optstring). `OPTIND` holds the index of the next unprocessed argument.

The leading `:` on the optstring — `:vf:` — switches `getopts` into silent error mode. In verbose mode (no leading `:`), `getopts` prints its own error messages to stderr when it encounters an invalid option or a missing argument. In silent mode, it suppresses those messages and instead signals the error to you by setting `opt` to `?` (unknown option) or `:` (missing argument), with `OPTARG` set to the offending option letter. Silent mode gives you control over the error messages, which matters for scripts that need consistent, localized, or structured error output.

`getopts` stops processing at the first non-option argument or at `--`. After the loop, `${@:$OPTIND}` gives you the remaining non-option arguments.

### `shift` — Consuming Positional Parameters

`shift N` removes the first N positional parameters. `shift` (no argument) removes one. After `shift 2`, what was `$3` is now `$1`.

`shift` is useful when you process options manually or when you want to consume a fixed set of arguments and then operate on the rest:

```text
# Conceptual flow
$@ = "backup" "/etc" "/var"
shift 1
$@ = "/etc" "/var"
```

## Basic Example

```bash
#!/bin/bash

# Define a function that greets a user by name
greet() {
    local name="$1"    # local: does not leak into caller
    echo "Hello, ${name}!"
}

# Call the function
greet "Alice"
greet "Bob"
```

```text
Hello, Alice!
Hello, Bob!
```

```bash
#!/bin/bash

# A function that reports success or failure via exit code
is_even() {
    local n="$1"
    (( n % 2 == 0 ))    # arithmetic expression: exits 0 if true, 1 if false
}

for num in 1 2 3 4; do
    if is_even "$num"; then
        echo "$num is even"
    else
        echo "$num is odd"
    fi
done
```

```text
1 is odd
2 is even
3 is odd
4 is even
```

```bash
#!/bin/bash

# A function that returns data via stdout, not via return
get_hostname_short() {
    hostname | cut -d. -f1    # first label of the FQDN
}

short=$(get_hostname_short)
echo "Short hostname: $short"
```

```text
Short hostname: myserver
```

Now try these yourself. In the first example, try removing `local` and adding `echo "$name"` after the calls — you'll see that without `local`, the variable leaks. In the second example, try changing `(( n % 2 == 0 ))` to `return 0` for all cases and watch the logic break. In the third example, add `echo "Exit code: $?"` after the assignment to see that the function's exit code is still captured even when its stdout is captured separately.

## Real-World Example

A script that backs up a list of directories, with a `getopts` interface for options, proper exit code handling, and a reusable `die` function for error reporting.

```bash
#!/bin/bash
set -euo pipefail

VERBOSE=0
DEST=""
LOGFILE="/var/log/backup.log"

# die: print an error and exit with a given code
# Usage: die <exit_code> <message>
die() {
    local code="$1"
    shift
    echo "ERROR: $*" >&2
    exit "$code"
}

# log: print a message with a timestamp, optionally to stdout
log() {
    local msg="[$(date '+%Y-%m-%d %H:%M:%S')] $*"
    echo "$msg" >> "$LOGFILE"
    (( VERBOSE )) && echo "$msg"
    return 0    # explicit: log() always succeeds
}

# backup_dir: back up a single directory to DEST
# Returns 0 on success, 1 on failure
backup_dir() {
    local src="$1"

    if [[ ! -d "$src" ]]; then
        log "SKIP: $src does not exist"
        return 1
    fi

    local dest_path="${DEST}/$(basename "$src")-$(date +%Y%m%d).tar.gz"

    if tar -czf "$dest_path" "$src" 2>>"$LOGFILE"; then
        log "OK: $src → $dest_path"
        return 0
    else
        log "FAIL: $src (tar exited $?)"
        return 1
    fi
}

# --- option parsing ---
while getopts ":vd:l:" opt; do
    case "$opt" in
        v) VERBOSE=1 ;;
        d) DEST="$OPTARG" ;;
        l) LOGFILE="$OPTARG" ;;
        :) die 2 "Option -$OPTARG requires an argument" ;;
        \?) die 2 "Unknown option: -$OPTARG" ;;
    esac
done
shift $(( OPTIND - 1 ))    # remove parsed options; $@ now holds source dirs

[[ -z "$DEST" ]]  && die 2 "Destination (-d) is required"
[[ $# -eq 0 ]]   && die 2 "At least one source directory is required"
[[ -d "$DEST" ]]  || die 1 "Destination $DEST does not exist"

# --- main logic ---
fail_count=0

for dir in "$@"; do
    backup_dir "$dir" || (( fail_count++ ))
done

if (( fail_count > 0 )); then
    log "Completed with $fail_count failure(s)"
    exit 1
fi

log "All backups completed successfully"
exit 0
```

The walkthrough:

`die` is a reusable error-exit function. It takes a numeric exit code as its first argument, then a message. `shift` inside `die` drops that first argument so `$*` becomes the message alone — a common idiom. Using `>&2` routes the error message to stderr, where it belongs.

`log` returns `0` explicitly. This is defensive: if the function body's last command ever changed, the function's exit code would silently change too. Making it explicit prevents surprises.

`backup_dir` captures `tar`'s exit code via the `if tar ...; then` pattern. It does not capture `$?` after the `tar` line because `tar` is directly in the `if` condition — there is no intervening command to reset `$?`.

The `getopts` loop uses a leading `:` on the optstring — `:vd:l:`. This enables silent mode. The `\?)` case handles unknown options (note the backslash to prevent glob expansion of `?`), and the `:)` case handles options that were given but are missing their required argument. Without the leading `:`, getopts would print its own generic error and you'd have no control over the message.

`shift $(( OPTIND - 1 ))` is standard `getopts` cleanup — after the loop, `OPTIND` points to the first non-option argument. Shifting by `OPTIND - 1` removes all the parsed options and leaves `$@` with just the positional arguments (the source directories).

`(( fail_count++ ))` is used inside `|| ((...))` rather than `fail_count=$((fail_count + 1))` because with `set -e`, a standalone `(( expr ))` that evaluates to 0 would exit the script. The `||` short-circuit prevents that by providing a second branch for `backup_dir`'s failure exit code. This is one of the subtle `set -e` traps that only shows up in practice.

## ⚠️ Gotchas

### Missing `local` Leaks Variables Into the Caller

This is the most common function bug in bash. The assumption is that variables inside a function stay inside the function. They do not — unless you declare them `local`.

```bash
#!/bin/bash

process() {
    count=0    # no local — this is a global
    for item in "$@"; do
        (( count++ ))
    done
}

count=99
process a b c
echo "count after process: $count"
```

```text
count after process: 3
```

The caller's `count` was overwritten. The function had no idea. This class of bug is particularly vicious because the function appears to work — it counts correctly — and the breakage is invisible inside the function. It only surfaces when the caller checks a variable it thought it owned.

```bash
#!/bin/bash

process() {
    local count=0    # local: visible only within process()
    for item in "$@"; do
        (( count++ ))
    done
    echo "$count"    # return the value via stdout
}

count=99
result=$(process a b c)
echo "result: $result"
echo "count after process: $count"
```

```text
result: 3
count after process: 99
```

The rule: declare every variable inside a function as `local` unless you specifically intend it to modify the caller's environment. The intent to modify the caller should be deliberate and documented.

### `return` Only Sets an Exit Code — It Does Not Return Data

The expectation coming from any other language: `return "some string"` returns a string to the caller. In bash, `return` accepts only an integer (or nothing, which defaults to 0). If you pass a string, bash silently uses 0.

```bash
#!/bin/bash

get_name() {
    return "Alice"    # silently discarded — bash uses 0
}

get_name
echo "Exit code: $?"
```

```text
Exit code: 0
```

"Alice" is gone. The function exited with 0 and there is no data to recover. This fails silently, which makes it worse than a clear error.

```bash
#!/bin/bash

get_name() {
    echo "Alice"    # data travels via stdout
}

name=$(get_name)    # capture stdout with command substitution
echo "Name: $name"
echo "Exit code: $?"
```

```text
Name: Alice
Exit code: 0
```

The pattern is: use `return N` to signal success or failure; use `echo` (and capture with `$(...)`) to return data. Both can be used in the same function — `echo` the value and `return 1` if something went wrong. The caller checks `$?` for the status and uses the captured string for the data.

One important side effect: command substitution `$(...)` runs the function in a subshell. Variables set inside the function will not affect the caller, even without `local`. This is safe for data return, but it also means `exit` inside the function exits the subshell, not the main script.

### `$?` Is Reset By Every Command

The assumption: you can check `$?` whenever you need to. The reality: `$?` is overwritten the moment any command runs — including `echo`, `local`, and variable assignments in some shells.

```bash
#!/bin/bash

some_command() {
    return 42
}

some_command
local saved    # this line runs between some_command and the check
# $? now holds the exit code of "local saved", not some_command
echo "Exit code: $?"
```

In this example, if this were inside a function, `local saved` would reset `$?` to 0 (since `local` succeeds). The 42 is gone.

```bash
#!/bin/bash

some_command() {
    return 42
}

some_command
exit_code=$?    # capture IMMEDIATELY — next line
echo "Exit code: $exit_code"
```

```text
Exit code: 42
```

Capture `$?` on the very next line after the command you care about. No exceptions.

### `source` Modifies the Current Shell — Including `exit`

`source script.sh` is not the same as `bash script.sh`. Executing a script runs it in a child process. Sourcing it runs it in your current shell. Any variable it sets becomes your variable. Any function it defines becomes your function. Any `cd` it runs changes your directory.

The most dangerous case is `exit`:

```bash
# In library.sh:
check_requirements() {
    if ! command -v jq &>/dev/null; then
        echo "jq is required" >&2
        exit 1    # THIS EXITS THE CALLING SHELL
    fi
}
```

If you `source library.sh` and then call `check_requirements`, and `jq` is not installed, your interactive shell exits. Your terminal window closes. If this is in a `.bashrc` or login script, you may be completely locked out of your shell.

```bash
# In library.sh — correct version:
check_requirements() {
    if ! command -v jq &>/dev/null; then
        echo "jq is required" >&2
        return 1    # exits the function only
    fi
}
```

In files intended to be sourced, always use `return` to signal failure from functions, never `exit`. Reserve `exit` for scripts that are executed directly.

### `getopts` Stops at the First Non-Option Argument

`getopts` processes options left-to-right and stops the moment it encounters an argument that doesn't start with `-` (or encounters `--`). Arguments after that point are not seen by `getopts`, even if they look like options.

```bash
#!/bin/bash

while getopts "v" opt; do
    case "$opt" in
        v) echo "verbose mode" ;;
    esac
done

# Called as: script.sh myfile -v
# getopts sees "myfile" first and stops — -v is never processed
```

This is intentional POSIX behavior. If your script needs options to appear after positional arguments, you have to handle argument parsing manually instead of using `getopts`. Alternatively, document clearly that all options must come before positional arguments — which is the standard Unix convention.

### Dynamic Scope Means Callees See Your Locals

Because bash uses dynamic scope, a function you call can see and modify your `local` variables if it happens to use the same name. This is not a theoretical concern — it happens when you write helper functions that use common names like `i`, `n`, `result`, or `tmp`.

```bash
#!/bin/bash

helper() {
    local i=999    # this shadows outer's local i
    echo "helper i: $i"
}

outer() {
    local i=0
    echo "outer i before: $i"
    helper
    echo "outer i after: $i"
}

outer
```

```text
outer i before: 0
helper i: 999
outer i after: 0
```

In this case it works correctly because `helper` also used `local i`, so it got its own binding. But if `helper` had written `i=999` without `local`, it would have modified `outer`'s `i` — and this would be a silent mutation with no error.

The protection: declare `local` in every function, even for short-lived loop variables. Use longer, more specific names in shared utility functions to reduce collision risk.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | `/bin/sh` is `dash`, which is a strict POSIX shell. `local` is supported in `dash`, but `FUNCNAME`, `BASH_LINENO`, and `BASH_SOURCE` are bash-only. Scripts using these must use `#!/bin/bash`. |
| RHEL / Fedora / CentOS / Rocky | `bash` is the default shell. `getopts` works as described. On RHEL 7 and earlier, bash 4.x is standard; RHEL 8+ ships bash 5.x. |
| Alpine (containers) | `/bin/sh` is busybox ash. `local` is available. `getopts` is available but behavior with the leading `:` in silent mode may differ from bash in edge cases — test explicitly. `FUNCNAME` is not available. |
| Arch | Default bash 5.x. All features work as described. |

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| `getopts` | Short options (`-v`, `-f file`) in bash/POSIX scripts | Built-in, portable, well-understood. Does not handle long options (`--verbose`). |
| `getopt` (external) | Long options (`--verbose`, `--file=path`) | An external command (`util-linux`), not a shell built-in. Requires careful quoting of its output. GNU `getopt` supports long options; BSD `getopt` does not. |
| Manual `case "$1"` loop with `shift` | Simple scripts, or when you need positional args mixed with options | Portable but tedious for many options. You control all behavior explicitly. |

For scripts you write and own, `getopts` is the right choice for short options. It handles edge cases (combined options like `-vf file`, missing arguments) that manual parsing gets wrong. For tools with a complex option surface that genuinely need long options, `getopt` (the external command) is the right choice — but read its man page carefully, because the quoting requirements are non-obvious.

## Trade-offs

**Where this shines:**

- Functions with `local` variables are the closest thing bash has to safe, isolated units of code — they avoid polluting the global namespace and make scripts testable in isolation
- Exit codes as a communication protocol are universally understood: every tool, CI system, and monitoring agent already speaks exit codes
- `getopts` handles combined short options, missing arguments, and unknown options correctly, which manual parsing almost never does
- `source` lets you share function libraries across scripts without subprocesses — fast and with full environment access

**Where this costs you:**

- Dynamic scope for `local` means functions are not truly isolated — a called function can silently modify your locals, which is a correctness hazard in large scripts
- Returning data via `echo` + `$(...)` runs the function in a subshell, which means any side effects (variable changes, directory changes) in the function are lost — this surprises people who think "I can return data and also modify a global"
- Exit codes are limited to 0–255 and carry no structure — you cannot distinguish between different types of success or return rich error information without out-of-band channels (writing to stderr, using a global error variable)
- `getopts` only handles short single-character options — real command-line tools with `--help`, `--output=file` need more infrastructure

**Performance note:** Functions are evaluated in the current process and have essentially no overhead compared to executing an external command. The performance cost of using functions is negligible. The cost of `$(function_call)` is a fork — there is one, because command substitution creates a subshell. For tight loops calling data-returning functions, this fork overhead is measurable. Prefer restructuring to avoid command substitution in hot paths; use global variables or output-parameter patterns instead.

## When to Use / When to Avoid

✅ **Use functions when:**
- Any logic block appears more than once — extract it immediately
- A script grows past ~50 lines — functions make it navigable
- You need to test a piece of logic independently of the rest of the script
- You're writing a library to be `source`d by other scripts

✅ **Use `local` for:**
- Every variable inside every function, always, as a default habit
- Loop counters, temporary path variables, intermediate results — anything that has no reason to escape the function

✅ **Use `return` for:**
- Communicating success or failure — the binary contract between a function and its caller
- Early exit from a function when a precondition fails

✅ **Use `echo` + `$(...)` for:**
- Returning computed strings, paths, or counts from functions

✅ **Use `getopts` when:**
- Your script accepts options from the command line and you want correct handling of `-ab` combined flags, `-f value` argument flags, and invalid option reporting

✅ **Use `source` for:**
- Loading function libraries shared by multiple scripts
- Reading configuration files that set shell variables (common pattern: `source /etc/myapp/config`)

❌ **Avoid `exit` inside sourced files:**
- Use `return` in functions in sourced files — `exit` will close the caller's shell

❌ **Avoid `getopts` when:**
- You need long options — use `getopt` (external) or manual parsing
- Your options must be freely interleaved with positional arguments — `getopts` stops at the first non-option

❌ **Avoid command substitution for functions with side effects:**
- If a function needs to set a variable in the caller's scope, command substitution will silently discard that side effect because it runs in a subshell

⚠️ **Common mistakes:**
- Using `$*` instead of `$@` when passing all arguments — `$*` coalesces arguments with spaces into one argument
- Forgetting `shift $(( OPTIND - 1 ))` after a `getopts` loop — leaves parsed options in `$@`
- Writing `if [ $? -eq 0 ]` on a line that is not immediately after the command being tested — an intervening `[` resets `$?`

## Veteran Perspective

The most persistent beginner mistake with functions is treating `return` like a return statement from Python or C. In every other language you learn first, `return value` sends the value back to the caller. In bash, `return 42` sends the number 42 to the caller as an exit status. That's it. The number means "this worked" (0) or "this failed" (non-zero). It does not carry data. When I see `return "$filename"` in a script, I know immediately that the author came to bash from another language and has not yet learned this. The code will appear to work — `return` with a string argument silently returns 0 — and then behave inexplicably when the caller tries to use the "returned" value.

The intermediate mistake is more subtle: people learn about `local`, declare local variables in their functions, and think they've got proper isolation. They haven't. Dynamic scope means any function your function calls can see your locals if the names collide. I've debugged scripts where a utility function called `cleanup()` was silently clobbering a `local path` variable in the caller because the utility function also used `path`. No error. No warning. The caller just had the wrong path variable. The fix was to rename the variable in `cleanup()` to `_cleanup_path`. The lesson: use namespaced names in utility functions and declare `local` aggressively.

The `$?` capture issue is one I still see from experienced sysadmins. They write a function, check `$?` three lines later, and wonder why it's always 0. The answer is always that something between the function call and the check reset it. The rule I give junior engineers: capture `$?` on the very next line. No exceptions, not even "it's just an echo." Especially not "it's just an echo."

On `source` vs execution: the confusion here is usually benign until it isn't. A library script that uses `exit` on error works fine when you test it by running it directly. It works fine when your main script executes it as a subprocess. It silently terminates your entire terminal session when you `source` it. This class of bug typically surfaces the first time someone runs a deploy script from a bastion host and their SSH session drops. It is fixable and preventable, but only if you understand the difference between the two execution models.

`getopts` is underused. Most shell scripts I encounter in production parse options with something like `if [[ "$1" == "-v" ]]; then VERBOSE=1; shift; fi` — repeated for each option. It handles the simple case. It breaks silently for `-vf file` (combined flags), doesn't distinguish "option missing its argument" from "unknown option," and gets unwieldy past three options. `getopts` handles all of this correctly in fifteen lines. It is worth learning.

One more thing: `FUNCNAME`, `BASH_SOURCE`, and `BASH_LINENO` together give you a real stack trace in bash. I have seen engineers write sophisticated monitoring and alerting systems without realizing that an `ERR` trap combined with these arrays gives you the full call chain on any unhandled error. If your bash scripts are long enough to need debugging, wire up an `ERR` trap that calls a `stack_trace` function. You will find bugs in minutes that would otherwise take hours.

## Try It Yourself

1. Write a function `double()` that takes a number and prints twice its value using `echo`. Call it three times with different numbers and capture the result in a variable. Print each result. Confirm the function's exit code is 0 each time.

2. Write a function `require_file()` that takes a path and returns 0 if the file exists and is readable, or 1 otherwise. Use it in an `if` statement. Test it with a file that exists (`/etc/hostname`) and one that doesn't.

3. Write a script that accepts two options: `-v` (verbose mode, no argument) and `-o output_file` (output file, required argument). Use `getopts` with a leading `:` on the optstring. Handle the case where `-o` is given without a value. If verbose mode is on, print "Verbose enabled" before doing anything else. Print the output file path at the end.

4. Write a function `factorial()` that computes the factorial of its argument recursively using `return` for the exit code and a global variable for the result (since recursive command substitution is expensive). Call `factorial 5` and print the result. Then look up what happens if you call `factorial 0`.

5. **Challenge (VM recommended):** Write a script `check-services.sh` that accepts a list of service names as arguments (e.g., `./check-services.sh nginx sshd cron`). For each service, define a function `check_service()` that uses `systemctl is-active` to determine if the service is running. The function should return 0 if running, 1 if stopped, 2 if the service is not found. The main loop should track how many services are stopped and how many are not found. At the end, print a summary and exit with code 0 if all services are running, 1 if any are stopped, 2 if any are not found. Test it with a mix of running and stopped services by temporarily stopping one with `sudo systemctl stop <name>`.

## Summary

- A function runs in the current shell process, not a subprocess — every variable it sets is global unless declared `local`.
- `return N` sets the exit code (0–255) only; data is returned by `echo`ing to stdout and capturing with `$(function_name)`.
- `$?` holds the exit code of the last command — capture it on the very next line or it will be overwritten.
- Exit codes 126 (not executable), 127 (not found), and 128+N (killed by signal N) have fixed meanings used by shells and CI systems; never use them for application-level errors.
- `source script.sh` runs the script in the current shell — `exit` inside a sourced file closes your shell; use `return` instead.
