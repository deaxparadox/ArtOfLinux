# [5.6] — Debugging Shell Scripts

## The Problem This Solves

You wrote a deployment script. It runs fine in dev. On the production server it exits partway through — no error message, no indication of which line failed, no trace of what values the variables held when it died. You're staring at a half-deployed application and a blank terminal.

This is the canonical shell scripting horror story, and it has nothing to do with your script being complex. It happens because bash, by default, tells you almost nothing about what it's doing. It runs commands, it expands variables silently, and when something goes wrong it either exits quietly or keeps going with garbage values. You have to opt in to visibility.

The debugging tools in this topic are that opt-in. They let you see every command before it runs, with all variables fully expanded. They let you catch errors as they happen, at the exact line they happen. They let you validate your script without running it. They let a static analyzer read your script like a compiler and flag the class of bugs that have taken down more production systems than any other cause in shell scripting.

Without these tools, debugging shell scripts is guessing. With them, it's a traceable, reproducible diagnostic process — the same discipline you'd apply to any other piece of code.

## Mental Model

Think of bash's debugging tools as the difference between driving a car with your eyes closed and driving it with the dashboard lit up. The car goes the same route either way. But with the dashboard on, you see every turn, every speed, every warning light, as it happens — and when something goes wrong, you see exactly where you were when it did.

`bash -x` is that lit-up dashboard. Every command appears on stderr before it runs, with all variables already expanded to their actual values. You don't have to guess what `$FILE` was — you see it. You don't have to guess which branch of an `if` ran — you see the line. `PS4` is the label above each dashboard reading. `shellcheck` is the mechanic who inspects the car before you leave the garage.

## Theory

### The Four Layers of Shell Script Debugging

Bash debugging works in four distinct layers, and understanding them separately prevents the most common confusion:

**Syntax checking** (`bash -n`) happens before any code runs. Bash reads the entire script and verifies that the structure is grammatically valid — braces match, `if` statements have `fi`, `do` has `done`. This catches typos and structural errors without executing a single command.

**Static analysis** (`shellcheck`) goes further. It reads your script without running it and catches patterns that are syntactically valid but semantically dangerous — unquoted variables, deprecated constructs, unreachable code, common logic errors. Think of it as a linter that knows the entire body of shell scripting lore about what goes wrong.

**Execution tracing** (`bash -x`, `set -x`) happens at runtime. Every command is printed to stderr — fully expanded, with all variable values filled in — immediately before it runs. This gives you a complete transcript of what the script actually did, with the real values, in the real order.

**Error trapping** (`trap ERR`, `trap EXIT`) uses bash's signal and pseudo-signal mechanism to run code at specific moments: when any command returns a non-zero exit code, or when the script exits for any reason. This lets you emit a diagnostic at the exact moment failure occurs, before the stack unwinds and context is lost.

These four layers are complementary, not redundant. Run `bash -n` and `shellcheck` before a script touches production. Use `set -x` when you need to see what's actually happening at runtime. Use traps to catch failures without having to add `|| die` to every line.

### What `bash -x` Actually Expands

When trace mode is on, bash prints each simple command after word splitting and parameter expansion, but before execution. This means you see the real values. If `$HOST` is `db-prod-01`, the trace shows `db-prod-01`, not `$HOST`. If a glob like `*.log` expands to forty filenames, the trace shows all forty. This is the information that makes trace mode so useful — it shows you what bash sees, not what you wrote.

The trace prefix is controlled by `PS4`. The default is `+` followed by a space. Each level of shell nesting adds another `+`: a subshell shows `++`, a subshell inside a subshell shows `+++`. This nesting indicator is a useful signal in complex scripts with many subshells.

### How `trap` Works

`trap` registers a command to run when the shell receives a signal or reaches a special condition. The syntax is:

```text
trap 'command' CONDITION
```

The conditions relevant to debugging are:

- `ERR` — fired when any command exits with a non-zero status, unless that command is part of a condition test (`if`, `while`, `&&`, `||`) or the script uses `set +e` at that point
- `EXIT` — fired when the script exits for any reason: normal completion, `exit`, or a fatal signal
- `DEBUG` — fired before every command (extremely verbose; useful for step-through debugging)

`ERR` is the most useful for failure detection. Combined with `$LINENO`, it gives you a line number at the moment of failure — the closest thing bash has to a stack trace.

### ShellCheck — What It Catches and How

ShellCheck (current version 0.10.x as of 2025) is a static analysis tool that knows thousands of common shell scripting failure patterns. It flags them with stable codes, documented at `shellcheck.net/wiki`. The three codes you'll encounter constantly:

**SC2086** — "Double quote to prevent globbing and word splitting." This is the single most common shell scripting bug. Writing `rm $file` instead of `rm "$file"` means bash performs word splitting and glob expansion on `$file` before passing it to `rm`. If `$file` is `my report.txt`, rm gets two arguments: `my` and `report.txt`. If it's `*.txt`, rm deletes every text file in the directory. ShellCheck flags every unquoted variable expansion that could be subject to this.

**SC2046** — "Quote this to prevent word splitting." Similar to SC2086 but applies specifically to command substitution: `$(command)`. The output of a command substitution is word-split exactly like an unquoted variable. `files=$(find . -name "*.txt"); rm $files` is broken for the same reason as the above.

**SC2006** — "Use `$(...)` notation instead of legacy backticks." Backtick command substitution is the old syntax from the 1970s Bourne shell. It has serious quoting problems (backslashes inside backticks behave differently), cannot be nested without escaping, and makes scripts harder to read. Modern bash uses `$()` exclusively. ShellCheck flags every backtick usage.

## How It Works (Mechanics)

### Where Trace Output Goes

`bash -x` and `set -x` write trace output to file descriptor 2 — stderr. This is a fundamental fact with a non-obvious consequence: if you redirect stdout to a log file, the trace is not captured in it.

```bash
bash -x script.sh > output.log    # trace goes to terminal, not output.log
bash -x script.sh 2> trace.log    # trace captured, stdout to terminal
bash -x script.sh > output.log 2>&1   # both captured together
```

You can also redirect the trace to a separate file descriptor by setting `BASH_XTRACEFD`:

```bash
exec 3>/tmp/trace.log
BASH_XTRACEFD=3
set -x
# trace now goes to /tmp/trace.log, stdout/stderr are unaffected
```

This is the right pattern for production diagnostic scripts where you need clean stdout for the application output and a separate trace file for debugging.

### How `bash -n` Parses Without Executing

When you run `bash -n script.sh`, bash reads and parses the entire script — resolving all token boundaries, checking all syntax — but never forks any processes or calls `execve`. It reads `source`/`.` files if they're referenced with literal paths, but does not evaluate variable expressions that would determine which file to source. Function bodies are parsed but not executed. `$(command)` substitutions are parsed for syntax but the commands inside are not run.

The practical consequence: `bash -n` cannot catch bugs that only manifest when variables have values or when commands produce output. It only validates structure.

### How `trap ERR` Fires

The `ERR` pseudo-signal fires when a command exits with a non-zero status in a context where `set -e` would cause an exit. The exact firing conditions:

- A simple command fails: `false` → ERR fires
- A pipeline fails (with `set -o pipefail`): `false | true` → ERR fires
- ERR does **not** fire when the failing command is in a condition: `if false; then...`, `false || true`, `false && true` — the script is testing the exit code, not treating it as an error

Inside functions, `ERR` traps do not automatically inherit unless the function was invoked under `set -e` conditions and the function's exit code is checked. To propagate ERR behavior into functions reliably, use `set -E` (capital E), which causes ERR traps to be inherited by shell functions and subshells.

```bash
set -eE   # -e exits on error, -E inherits ERR trap in functions
trap 'echo "Error at line $LINENO"' ERR
```

## Basic Example

The simplest demonstration: a failing script with no debugging, then with tracing enabled.

```bash
# Create a minimal script to debug
cat > /tmp/demo.sh << 'EOF'
#!/usr/bin/env bash
GREETING="hello"
NAME="world"
echo $GREETING $NAME
echo "Result: $((1 + 1))"
false    # deliberately fail
echo "This line never runs"
EOF
chmod +x /tmp/demo.sh
```

Run it without debugging — you get no trace, no indication of where it failed:

```bash
bash /tmp/demo.sh
```

```text
hello world
Result: 2
```

Now run with `bash -x` and watch every step:

```bash
bash -x /tmp/demo.sh
```

```text
+ GREETING=hello
+ NAME=world
+ echo hello world
hello world
+ echo 'Result: 2'
Result: 2
+ false
```

Each line prefixed with `+` is a trace line — bash printing the command with variables already expanded, immediately before running it. The script stopped at `+ false` because `false` returns exit code 1. Notice that `$GREETING` and `$NAME` are fully expanded in the trace — you see `echo hello world`, not `echo $GREETING $NAME`. That expansion is the core value of trace mode.

Now try it yourself. Add a variable with a space in its value: `NAME="John Doe"`. Run with `-x` again and notice how the trace still shows the quotes. Then remove the quotes from the `echo` line inside the script and watch what happens.

## Real-World Example

A deployment script with proper debugging infrastructure — error trapping, targeted trace, and a cleanup trap:

```bash
#!/usr/bin/env bash
set -euo pipefail

# Customize the trace prefix to show file, line, and function
export PS4='+(${BASH_SOURCE[0]##*/}:${LINENO}): ${FUNCNAME[0]:+${FUNCNAME[0]}(): }'

# Trap errors with context
trap 'echo "FATAL: command failed at line $LINENO (exit code $?)" >&2' ERR

# Trap cleanup on any exit
WORK_DIR=""
cleanup() {
    if [[ -n "$WORK_DIR" && -d "$WORK_DIR" ]]; then
        rm -rf "$WORK_DIR"
        echo "Cleaned up $WORK_DIR" >&2
    fi
}
trap cleanup EXIT

deploy() {
    local service="$1"
    local version="$2"

    WORK_DIR=$(mktemp -d)

    echo "Deploying $service version $version..."

    # Only trace the critical section
    set -x
    cp -r "/opt/releases/$service-$version" "$WORK_DIR/"
    systemctl stop "$service"
    rsync -a "$WORK_DIR/$service-$version/" "/opt/live/$service/"
    systemctl start "$service"
    set +x

    echo "Deploy complete."
}

deploy "myapp" "2.4.1"
```

Walk through what each piece does.

The `PS4` customization changes the trace prefix from bare `+` to `+(filename:linenumber): funcname(): `. The `${BASH_SOURCE[0]##*/}` strips the path from the filename, leaving just the base name. The `${FUNCNAME[0]:+${FUNCNAME[0]}(): }` only shows the function name when you're inside a function — it's empty at the top level. With this PS4, a trace line looks like `+(deploy.sh:32): deploy(): rsync -a ...` instead of just `+ rsync -a ...`, which is the difference between immediately knowing where you are and having to count lines.

The `trap ERR` fires the moment any command fails. Because `$LINENO` is expanded at trap execution time, it shows the line number of the failing command — not the line where `trap` was set. This is your closest approximation to a stack trace in bash.

The `trap cleanup EXIT` fires on any exit: normal completion, a fatal error, even `Ctrl-C`. This ensures `$WORK_DIR` is always cleaned up. Note that `WORK_DIR` is initialized to an empty string before `trap` is set, and the cleanup function checks it before acting — otherwise the function would try to delete an empty path if the script exited before `mktemp` ran.

The `set -x` / `set +x` wrapping enables the trace only around the critical rsync and systemctl calls, not the entire script. In a long deployment script, tracing everything is overwhelming. Targeting the trace to the section where failures actually occur is the mature pattern.

## ⚠️ Gotchas

### Trace Goes to Stderr — Redirecting Stdout Loses It

The most common mistake when adding debugging to a script is doing this:

```bash
bash -x deploy.sh > /var/log/deploy.log
```

You look at the log file and see no trace — only the stdout output. The trace went to the terminal, not the file, because `-x` writes to stderr (fd 2) and `>` only redirects stdout (fd 1).

```text
# In the log file — no trace lines at all
Deploy complete.

# On the terminal — trace appears here instead
+ cp -r /opt/releases/myapp-2.4.1 /tmp/tmp.xK2pl9/
+ systemctl stop myapp
...
```

The fix depends on what you want. To capture both in the same file:

```bash
bash -x deploy.sh > /var/log/deploy.log 2>&1
```

To keep a separate trace file:

```bash
exec 3>/var/log/trace.log; BASH_XTRACEFD=3; set -x
```

### `bash -n` Passes Scripts With Runtime Errors

This trips up everyone. `bash -n` is a syntax check, not a logic check.

```bash
bash -n script.sh
echo "Exit code: $?"
```

```text
Exit code: 0
```

The script "passes" — but it might still fail at runtime because `bash -n` cannot catch:

- A variable that's unset when the script runs
- A file that doesn't exist when a command tries to open it
- A command that doesn't exist on the target system
- Logic that silently produces wrong output

The implication: passing `bash -n` means the script is grammatically valid. It says absolutely nothing about whether the script will work. Use `bash -n` as the first gate, then run `shellcheck`, then test in a staging environment that mirrors production. Never treat `bash -n` exit 0 as "the script works."

### `PS4` with `$LINENO` Inside a Function Shows the Wrong Line

This is subtle and will mislead you during debugging if you don't know about it.

```bash
PS4='+(line $LINENO): '

my_function() {
    echo "inside function"   # line 4 of the function
}

my_function   # called from line 7 of the script
```

When you see `+(line 4): echo inside function` in the trace, that's the line number within the function definition in the source file — specifically, it's the physical line in the script where the command lives. That's usually what you want.

But if you want the line in the *caller* — where the function was called from — you need `${BASH_LINENO[0]}`:

```bash
PS4='+(${BASH_SOURCE[0]##*/}:${LINENO}): '
# $LINENO = line of the current command being executed
# ${BASH_LINENO[0]} = line of the most recent function call (caller context)
```

`BASH_LINENO` is an array. `${BASH_LINENO[0]}` is the line number in the calling script where the current function was invoked. This is the context you want in an ERR trap — it tells you where in the *calling* code the failure originated:

```bash
trap 'echo "Called from line ${BASH_LINENO[0]}, failed at $LINENO" >&2' ERR
```

### `trap ERR` Does Not Fire Inside Functions by Default

This is the most dangerous `trap ERR` misconception, and it has caused people to think their error handling was working when it silently wasn't.

```bash
#!/usr/bin/env bash
set -e
trap 'echo "ERROR at $LINENO"' ERR

my_function() {
    false    # this fails
    echo "This should not run"
}

my_function   # trap does NOT fire for the false inside my_function
echo "Script continues here"   # this WILL run
```

```text
Script continues here
```

The `false` inside `my_function` caused the function to exit with status 1 due to `set -e`, but the ERR trap never fired. The calling scope saw `my_function` return non-zero, which also did not fire the trap because the calling code didn't have a direct failed command in trap-eligible context.

The fix is to add `-E` to your `set` options:

```bash
set -eE    # capital E causes ERR traps to be inherited by functions and subshells
trap 'echo "ERROR at $LINENO" >&2' ERR
```

With `-E`, ERR traps are inherited. The `false` inside `my_function` now fires the trap.

### Debugging `set -e` Surprises — Commands That Fail Silently

`set -e` exits on error, but "error" means "a command returned non-zero in a context where bash treats that as fatal." Several common patterns return non-zero without being actual failures:

```bash
set -e
grep "pattern" file.txt    # exits 1 if pattern not found — script dies here
count=$(grep -c "pattern" file.txt)   # also exits 1 if not found — script dies
```

```text
# script exits with no message — this is the set -e "silent kill"
```

This is one of the most common sources of mysterious script failures. The fix is to either handle the expected non-zero case explicitly or use the `||` operator to swallow expected non-zero exits:

```bash
# Explicitly handle not-found
if grep -q "pattern" file.txt; then
    echo "found"
fi

# Or suppress the exit for a known non-error case
count=$(grep -c "pattern" file.txt) || count=0
```

When debugging `set -e` surprises, `set -x` is your first tool — run the failing script with `bash -x` and watch which command the script dies on. The trace stops at the failing line. Then ask: is this command expected to sometimes return non-zero, and did I account for that?

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `bash -n` | Quick syntax check before running | Syntax only — not logic |
| `shellcheck` | Static analysis before every commit | The standard; catches classes of bugs `bash -n` misses entirely |
| `set -x` / `bash -x` | Runtime trace for known-failing scripts | Goes to stderr; use `PS4` to add file/line context |
| `bashdb` | Step-through interactive debugging | Largely unmaintained as of 2020; rarely needed; available on SourceForge |
| `set -v` | Raw token trace before expansion | Shows input as read, not as executed — useful for debugging quoting issues |
| `trap DEBUG` | Pre-command hook for custom step tracing | Extremely verbose; useful for controlled step-through, not general use |

`bashdb` exists and follows gdb syntax, but the project has seen minimal activity since 2019 and is rarely used in practice. For the vast majority of shell script debugging, `bash -x` with a custom `PS4` plus `shellcheck` plus `trap ERR` covers everything you need. The only scenario where a full interactive debugger adds value is a large, complex script with many conditional branches — and at that complexity level, you should question whether bash is the right tool.

## Trade-offs

**Where this shines:**

- `bash -x` trace is zero-cost to add and zero-cost to remove — it requires no instrumentation inside the script and produces a complete execution record
- `shellcheck` catches entire classes of bugs (unquoted variables, word splitting, deprecated syntax) that can't be caught at syntax-check time but have predictable failure patterns across millions of scripts
- `trap EXIT` guarantees cleanup runs regardless of how the script exits — it's more reliable than manual `rm` calls scattered through the script
- Targeted `set -x` / `set +x` sections let you trace just the dangerous part of a long script without drowning in output

**Where this costs you:**

- `bash -x` trace output is proportional to script complexity — a script that processes 10,000 files will produce 10,000+ trace lines, which can fill a log or overwhelm a terminal
- `trap ERR` has subtle scoping rules (function inheritance, `set -E` required) that make it easy to think you have error trapping when you don't
- `shellcheck` has occasional false positives — it flags some patterns that are intentionally correct, requiring `# shellcheck disable=SC2086` inline suppressions; these suppressions, overused, become noise
- Over-relying on `bash -n` as a "the script works" check is actively dangerous — it creates false confidence without providing real validation

## When to Use / When to Avoid

✅ **Use `bash -x` when:**
- A script is failing and you have no idea which command is the problem
- You need to verify that variable expansions produce the values you expect
- You're debugging a script on a system you can't modify (you run it externally)

✅ **Use `set -x` / `set +x` inline when:**
- You already know which section of the script is misbehaving and want a targeted trace without drowning in output for the whole script

✅ **Use `shellcheck` when:**
- Always — before any script goes anywhere near production; many teams run it in CI as a required gate

✅ **Use `trap ERR` when:**
- You want to know the exact line of failure in a long script without adding `|| die` to every command
- Remember to add `set -E` so it fires inside functions

✅ **Use `trap EXIT` when:**
- Your script creates temporary files, locks, or other resources that must be cleaned up regardless of how the script ends

❌ **Avoid `trap DEBUG` in production scripts:**
- It fires before every single command — this is a severe performance impact and produces output so voluminous it defeats the purpose; use it only for controlled, local debugging sessions

❌ **Avoid treating `bash -n` exit 0 as "script works":**
- It doesn't mean that. It means the script has valid syntax. Run `shellcheck` and test in staging.

❌ **Avoid `bashdb` for new work:**
- Largely unmaintained. The time it takes to install and learn the interface exceeds the time to debug most shell scripts with `bash -x` and a good `PS4`.

⚠️ **Common mistakes:**
- Setting `PS4` without `export` — child shells won't inherit it and subshell traces will revert to `+`
- Adding `set -x` at the top of a long script and then being surprised by thousands of trace lines — trace the section you care about
- Forgetting that `trap ERR` output goes to stderr — if your monitoring only looks at stdout, silent failures stay silent

## Veteran Perspective

The most common debugging mistake I see from engineers who are comfortable in other languages is adding `echo` statements everywhere — the classic printf debugging pattern. It works, but it's the slowest possible approach in bash. `bash -x` gives you every command, with every variable fully expanded, for free. You don't need to add a single line to the script. Run `bash -x script.sh 2>&1 | less` and search for the last `+` line before things went wrong. That's where the script failed. That's your starting point. Add `echo` statements only when the trace isn't giving you enough context about data flow — which is rare.

The second most common mistake: people check the exit code of `shellcheck` in CI and then ignore the specific warnings it produces, fixing only the ones that block the build. SC2086 in particular — the unquoted variable warning — gets dismissed as "I know what this variable contains, it's fine." I've heard that exact sentence before a script deleted a production directory because a filename had a space in it. ShellCheck isn't paranoid. It's seen more bugs than you have. Take the warnings seriously.

The `set -e` / `trap ERR` combination has a subtlety that trips up even experienced engineers: `grep` returning 1 for "not found" is not an error. `ssh` returning 1 for "host unreachable" is not a bash error. `diff` returning 1 for "files differ" is not an error. These are all legitimate non-zero exit codes that carry information. When `set -e` is active, every one of them kills your script unless you handle them explicitly. The discipline is: for every command you add to a script with `set -e`, ask yourself "what are the non-zero exit codes this command can return, and are any of them expected?" Answer that question upfront, not during a 3am incident.

The `BASH_XTRACEFD` variable is underused. Redirecting trace output to a separate file descriptor means you can have a completely clean stdout for your application logic and a separate trace file for debugging, without the two streams ever mixing. In any serious production script that needs diagnosability, this is the right pattern:

```bash
exec 9>/var/log/scripts/trace-$(date +%Y%m%d-%H%M%S).log
export BASH_XTRACEFD=9
export PS4='+(${BASH_SOURCE[0]##*/}:${LINENO}): '
set -x
```

Set this at the top of every important script. When something goes wrong six months from now, you'll have a timestamped trace log. When nothing goes wrong, the overhead is negligible.

Finally: `trap EXIT` is one of the most valuable patterns in bash and one of the most underused. Every time your script creates a `mktemp` file or directory, every time it acquires a lock file, every time it stops a service — register cleanup in a trap. The trap fires on normal exit, on error exit with `set -e`, on SIGTERM, on SIGINT. Write cleanup once. Run it always.

## Try It Yourself

1. Create a script with `#!/usr/bin/env bash` and two variables — one with a space in the value. Run it with `bash -x` and watch how the space is preserved in the trace output.

2. Take any shell script you have (or create a five-line one) and run `shellcheck` against it. Fix any SC2086 warnings by double-quoting the flagged variables. Re-run `shellcheck` until it reports no issues.

3. Write a script that creates a temporary directory with `mktemp -d`, stores the path in a variable, and registers a `trap cleanup EXIT` function that removes it. Verify the cleanup runs by printing a message from inside the cleanup function and watching it appear even when you add `exit 1` mid-script.

4. Customize `PS4` in a script to show the filename and line number. Add a function that calls another function. Run with `bash -x` and observe how the trace prefix changes to include the function name inside each function.

5. **Challenge:** Write a script (safe to run in a VM) that deploys a fake application: creates a working directory, copies some files, and then intentionally fails. Use `set -eE`, `trap ERR` to log the exact line of failure, `trap EXIT` for cleanup, `set -x` / `set +x` around only the "deployment" section, and a custom `PS4`. Then run it and verify: the trace shows only the targeted section, the ERR trap fires at the right line, and the EXIT trap runs cleanup. Introduce different failure points and confirm the line number reported by `trap ERR` is always correct.

## Summary

- `bash -x` and `set -x` write a full execution trace to **stderr** — every command with variables already expanded, immediately before it runs; redirecting stdout does not capture it
- `bash -n` checks syntax only — a clean `bash -n` does not mean a script will work; use `shellcheck` for real static analysis that catches unquoted variables (SC2086), word splitting in command substitution (SC2046), and deprecated backtick syntax (SC2006)
- `PS4='+(${BASH_SOURCE[0]##*/}:${LINENO}): ${FUNCNAME[0]:+${FUNCNAME[0]}(): }'` transforms a trace from a stream of `+` lines into a precise, file-and-line-annotated record; set `BASH_XTRACEFD` to redirect trace output to a dedicated file without mixing it with stdout
- `trap 'handler' ERR` fires at the exact failing command, but only if `set -E` is also active — without `-E`, the trap silently does not fire inside functions
- `trap 'cleanup' EXIT` is the correct pattern for resource cleanup — it fires on normal exit, error exit, and interruption, making it more reliable than cleanup code scattered through the script body
