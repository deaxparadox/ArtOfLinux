# [5.5] — Writing Safe Scripts

## The Problem This Solves

You write a deployment script. It runs perfectly in testing. You push it to production and it runs unattended as a cron job. Halfway through, a directory it depends on doesn't exist. The `mkdir` call fails silently. The script keeps running. It copies files into paths that don't exist. It reports success. Your deployment is broken and no one knows it — not until a customer hits the missing service hours later.

This is not a contrived scenario. It is the normal failure mode of Bash scripts written without safety flags. Bash, by default, ignores errors. A command that exits non-zero is noted and forgotten. The next command runs. The script completes. Exit status: 0. Everything is fine. Nothing is fine.

The interactive shell behaves this way deliberately. When you type commands at a prompt, you want each command to run regardless of what the last one did. You `ls` a nonexistent path, get an error, and then type the next command. That's correct behavior for a human at a keyboard. It is catastrophic behavior for an unattended script making real changes to a real system.

The five practices in this file — the shebang, `set -euo pipefail`, quoted variables, safe temp files, and shellcheck — close the gap between "works in testing" and "safe to run in production." Every production script uses them. Scripts that don't are accidents waiting to happen.

## Mental Model

Think of a pilot's pre-flight checklist. Before takeoff, the pilot methodically verifies every critical system: engines, controls, fuel, pressurization. Not because pilots are forgetful. Because when you're at 35,000 feet and something goes wrong, you can't pull over. The consequences of skipping a check are asymmetric — the time cost of the checklist is small and predictable; the cost of missing a failure is potentially catastrophic.

`set -euo pipefail` at the top of your script is that checklist. It says: if anything is wrong — a command failed, a variable wasn't set, a pipeline quietly dropped an error — stop now, while I can still see what happened, rather than continuing into an increasingly broken state and failing mysteriously five steps later.

## Theory

### The Shebang

The shebang (`#!`) on the first line of a script tells the kernel which interpreter to use. When you make a script executable and run it, the kernel reads the first two bytes. If they're `#!`, it reads the rest of the line as the path to the interpreter and passes the script as an argument to that program.

Two common forms:

```bash
#!/bin/bash
```

```bash
#!/usr/bin/env bash
```

`#!/bin/bash` hardcodes the path. This works reliably on every Linux system — `/bin/bash` is where bash lives on Linux, always. It is slightly faster because the kernel can exec directly without searching `PATH`.

`#!/usr/bin/env bash` searches `PATH` for `bash` and uses whatever it finds. This is the portable form used when you need a script to run on macOS, BSD, or systems where bash might be installed elsewhere (e.g., `/usr/local/bin/bash` on macOS after a Homebrew install). It also means the script uses the user's `bash` — which could be a different version than the system bash, for better or worse.

For Linux-only scripts, `#!/bin/bash` is the right choice. It's explicit about exactly what will run the script. For scripts that need to be portable across operating systems, `#!/usr/bin/env bash` is the convention.

One thing the shebang cannot do: mix the interpreter path with arguments. `#!/usr/bin/env bash -euo pipefail` does not work portably — behavior of arguments after the interpreter path in an `env` call varies across systems. Set your options with `set` on the next line.

### set -e — Exit on Error

`set -e` causes the shell to exit when any command exits with a non-zero status, with critical exceptions. The POSIX specification lists exactly which command positions are exempt:

- A command that is part of the test in an `if` or `elif` statement
- A command that is part of the condition in a `while` or `until` loop
- Any command in a `&&` or `||` list, except the last command in the list
- A command whose return status is being inverted by `!`
- A command in a subshell invoked as part of a compound command, if the compound command itself does not fail

These exceptions exist because error checking in those positions is intentional — `if grep -q pattern file; then` is supposed to handle grep's non-zero exit (no match) as a control-flow signal, not as a fatal error.

`set -e` applies to the last command in a pipeline. Combined with `set -o pipefail`, it covers the full pipeline.

### set -u — Unset Variables Are Errors

`set -u` causes the shell to treat references to unset variables as an error. Without it, referencing an unset variable produces an empty string silently:

```bash
echo $UNSET_VAR   # prints nothing, exits 0
```

With `set -u`, the same reference exits the script with an error. This catches a large class of bugs: typos in variable names, logic paths that fail to set a variable, copy-paste errors where a variable from one script block is used in another context where it doesn't exist.

The special variables `$@` (all positional arguments) and `$*` are exempt — referencing them when no arguments were passed does not trigger a `set -u` error. This is intentional; scripts that accept optional arguments must be able to check `$@` safely.

### set -o pipefail — Pipeline Exit Status

By default, a pipeline's exit status is the exit status of the last command in the pipeline. Commands earlier in the pipeline can fail silently:

```bash
cat /nonexistent/file | grep pattern   # grep exits 1 (no match), but the pipeline
                                        # exits 1 for grep, not for cat's failure
```

With `set -o pipefail`, the pipeline's exit status is the exit status of the rightmost command that failed. If any command in the pipeline fails, the pipeline fails.

`pipefail` was added to bash in version 3.1 (released 2005). It is not part of POSIX sh — it was added to POSIX.1-2024 only recently. If you need strict POSIX sh portability, you cannot rely on it. In bash scripts, it's been universally available for nearly two decades.

### Quoting Variables

When the shell expands an unquoted variable, it performs two additional operations: word splitting and pathname expansion (globbing). Word splitting splits the expanded value on `IFS` characters (spaces, tabs, newlines by default). Pathname expansion then treats any glob characters (`*`, `?`, `[`) in the resulting words as patterns and expands them to matching filenames.

```bash
file="my report.txt"
rm $file             # runs: rm my report.txt — two arguments, not one
rm "$file"           # runs: rm "my report.txt" — one argument, correct
```

The rule is absolute: quote every variable expansion. `"$var"`, `"${array[@]}"`, `"$(command)"`. The only exception is deliberate word splitting, which is rare, intentional, and should be commented when you do it.

### Temporary Files Safely — mktemp and trap

Scripts that need temporary files have two requirements: the filename must be unique and unpredictable (to prevent symlink attacks), and the file must be cleaned up when the script exits — even when it exits due to an error.

`mktemp` creates a temporary file with a unique name in a secure directory (`/tmp` by default) and prints the filename. `trap ... EXIT` registers a command to run when the shell exits for any reason, including signals and `exit N` calls.

### flock — Preventing Concurrent Execution

When a script modifies shared state — databases, configuration files, deployment directories — running two instances simultaneously can corrupt that state. `flock` acquires an advisory lock on a file and either waits for it or fails immediately if it's already held.

### shellcheck — Static Analysis

`shellcheck` reads a shell script and reports bugs: unquoted variables, missing quoting in command substitutions, deprecated syntax, portability issues, common logical errors. It understands bash, sh, dash, and ksh. It is the single highest-value tool in shell scripting. Running it and fixing all warnings before deploying a script is not optional in serious environments.

## How It Works (Mechanics)

### How set -e Propagates Through the Script

When `set -e` is active, after every simple command completes, bash checks whether `errexit` mode is active and whether that command's context is one of the exempt positions. If the command exited non-zero and is not exempt, bash calls its own exit handler — the same path as an explicit `exit` call. Any `trap EXIT` handler fires at this point.

The exemption for `&&`/`||` lists applies to every command in the list except the last. This means `command1 && command2 || fallback` — if `command1` fails, bash runs `fallback` and does not exit. If `command2` fails, bash runs `fallback` and does not exit. Only if `fallback` fails does `set -e` fire, because `fallback` is the last command in the list.

Function calls are treated like simple commands: if a function exits non-zero and the call is not in an exempt position, the script exits. This means `set -e` does propagate into function bodies. The function's internal commands are checked individually; a failing command inside a function that is called in a non-exempt position will exit the script.

### How set -u Checks Variable State

Bash tracks whether each variable has been assigned a value (the "set" attribute). The "set" attribute is separate from the value — a variable can be set to an empty string (set, but empty). `set -u` checks the "set" attribute, not the value. `var=""` is set; `unset var` is not set.

When the shell is about to expand a variable, if `set -u` is active and the variable is not set, bash immediately treats this as an error — exit status 127 and an error message to stderr — without even completing the expansion.

```text
bash: UNSET_VAR: unbound variable
```

### How pipefail Works

Bash tracks the exit status of every command in a pipeline as a list called `PIPESTATUS`. After a pipeline completes, `PIPESTATUS` holds one exit status per command in the pipeline order.

With `pipefail`, after a pipeline completes, bash scans `PIPESTATUS` from right to left and returns the first non-zero status it finds. The overall pipeline exit status — what `set -e` sees, what `$?` contains — is that first non-zero status.

Without `pipefail`, `$?` is always `${PIPESTATUS[-1]}` — only the last command's exit status.

### How trap EXIT Works

`trap 'command' EXIT` registers a command to run when the shell is about to exit. "About to exit" means: reaching the end of the script, executing an `exit` call, or being killed by a signal that would cause exit (if a `trap SIGNAL` handler re-exits).

When `set -e` triggers an exit due to a command failure, that path goes through the same exit mechanism as an explicit `exit` call. The EXIT trap fires.

The EXIT trap does not fire when the process is killed with `SIGKILL` — nothing does. `SIGKILL` is handled entirely by the kernel; the process gets no chance to run any code.

## Basic Example

```bash
#!/bin/bash
# Demonstrates set -euo pipefail catching errors immediately
set -euo pipefail

echo "Step 1: Creating a directory"
mkdir /tmp/safe_script_demo

echo "Step 2: This runs because mkdir succeeded"
ls /tmp/safe_script_demo

echo "Step 3: Attempting to list a nonexistent path"
ls /tmp/this_does_not_exist    # exits non-zero — script exits here

echo "Step 4: This line never runs"
```

```text
Step 1: Creating a directory
Step 2: This runs because mkdir succeeded
Step 3: Attempting to list a nonexistent path
ls: cannot access '/tmp/this_does_not_exist': No such file or directory
```

The script stops after Step 3. Without `set -e`, Step 4 would print. Step 4 printing in a real script might mean "copy files to a destination that doesn't exist" or "assume a previous step succeeded when it didn't."

Run this yourself. Watch where it stops. Then remove `set -e` and run it again — Step 4 prints, exit status is 0, and you've just simulated the class of bug that causes silent deployment failures.

## Real-World Example

This is a realistic backup script that uses all five practices together:

```bash
#!/bin/bash
set -euo pipefail

# --- Configuration ---
BACKUP_SOURCE="${1}"                        # first argument — require it to be passed
BACKUP_DEST="/var/backups/$(date +%Y-%m-%d)"
LOCKFILE="/var/run/backup.lock"
TMPFILE=""                                  # initialize so the trap can check it safely

# --- Cleanup on any exit ---
cleanup() {
    # idempotent: safe to call even if TMPFILE was never created
    [[ -n "$TMPFILE" && -f "$TMPFILE" ]] && rm -f "$TMPFILE"
    # release the lock — flock releases automatically when the fd closes,
    # but being explicit is better than relying on that
    echo "Backup script exiting"
}
trap cleanup EXIT

# --- Prevent concurrent runs ---
# Open the lockfile on fd 9, then try to acquire an exclusive lock
# (-n = non-blocking: fail immediately if the lock is already held)
exec 9>"$LOCKFILE"
if ! flock -n 9; then
    echo "ERROR: Another backup is already running. Exiting." >&2
    exit 1
fi

# --- Validate input ---
if [[ -z "$BACKUP_SOURCE" ]]; then
    echo "Usage: $0 <source_directory>" >&2
    exit 1
fi

if [[ ! -d "$BACKUP_SOURCE" ]]; then
    echo "ERROR: Source directory does not exist: $BACKUP_SOURCE" >&2
    exit 1
fi

# --- Do the work ---
mkdir -p "$BACKUP_DEST"

# mktemp creates a unique temporary file and returns its path
TMPFILE="$(mktemp)"

echo "Collecting file list..."
find "$BACKUP_SOURCE" -type f > "$TMPFILE"   # if find fails, set -e exits

echo "Files to back up: $(wc -l < "$TMPFILE")"

echo "Copying to $BACKUP_DEST..."
rsync -a "$BACKUP_SOURCE/" "$BACKUP_DEST/"   # trailing slash on source is intentional

echo "Backup complete."
```

The structure here is deliberate. The `cleanup` function is registered with `trap EXIT` before any work happens — that way, even if `mkdir -p` fails on line 35, cleanup still fires. The `TMPFILE` variable is initialized to an empty string before the trap is registered, so the trap's `[[ -n "$TMPFILE" ]]` check handles the case where the script exits before `mktemp` runs.

The `flock` block uses a file descriptor (fd 9) opened with `exec`. `flock -n 9` tries to acquire an exclusive lock on whatever fd 9 points to. If the lock is already held — because another instance of this script is running — `flock -n` returns non-zero immediately. The `if ! flock -n 9` construct is deliberately in an `if` condition, which is an exempt position for `set -e`. This is intentional: we want to handle the "already locked" case ourselves, not let `set -e` exit with no message.

Notice that every variable is quoted — `"$BACKUP_SOURCE"`, `"$BACKUP_DEST"`, `"$TMPFILE"`. If any of these contained spaces (a path with a space, a date format that somehow included whitespace), unquoted expansion would break the commands silently and in hard-to-diagnose ways.

## ⚠️ Gotchas

### set -e Does Not Apply Inside if/while Conditions

The assumption: `set -e` means "exit if any command fails." The subtler truth: it only exits if the failing command is in a position where the shell is not already checking the return value.

```bash
#!/bin/bash
set -e

# This does NOT exit the script when grep fails (no match)
if grep -q "pattern" /nonexistent/file; then
    echo "found"
fi
echo "script continues"   # this runs even though grep exited non-zero
```

```text
grep: /nonexistent/file: No such file or directory
script continues
```

The shell is already handling the return value of `grep` — it's the condition of the `if`. The shell needs to know whether it succeeded or failed to decide which branch to take. `set -e` does not fire here because the failure is not unhandled; it is the signal that controls the branch.

The same exemption applies to `while` and `until` conditions, and to every command in a `&&`/`||` chain except the last. This is usually what you want. The trap is assuming `set -e` protects you in these positions — it doesn't. If you have an `if` condition that does something destructive (writes to a file, calls a function), a failure there continues execution.

```bash
#!/bin/bash
set -e

# Safe: grep failure is handled by the if
if grep -q "RUNNING" /var/run/app.status; then
    echo "app is running"
else
    echo "app is not running"
fi
```

### set -e and Subshells — The Compound Command Exception

`set -e` propagates into subshells. But there is a specific exception: if a subshell is part of a compound command, and the compound command itself is in an exempt position, `set -e` does not trigger for failures inside the subshell that are part of that compound command.

```bash
#!/bin/bash
set -e

# val gets set to "" if the subshell fails — the assignment itself does not fail
val=$(some_command_that_fails) || true    # || true exempts the whole thing
```

The practical consequence: wrapping a subshell with `|| true` silences all errors inside it from `set -e`'s perspective, not just the subshell's exit status. Use this deliberately.

### set -u and Empty Arrays in Bash Before 4.3

In bash versions before 4.3, expanding an empty array with `${arr[@]}` under `set -u` throws "unbound variable." The array exists and was declared as `arr=()`, but bash treated expanding an empty array the same as expanding an unset variable.

```bash
#!/bin/bash
set -u

arr=()
echo "${arr[@]}"   # bash < 4.3: error: arr: unbound variable
                   # bash >= 4.3: prints nothing, exits 0
```

```text
bash: arr[@]: unbound variable
```

The fix for scripts that must run on older bash (RHEL 7 ships bash 4.2, for example):

```bash
# The ${var+expansion} form expands to expansion only if var is set.
# Applied to an array: expands to "${arr[@]}" only if arr is set and non-empty.
echo "${arr[@]+"${arr[@]}"}"
```

This idiom means: if `arr` is set (even if it's set to nothing), expand `"${arr[@]}"`. If `arr` is unset, expand to nothing. It's ugly, but it works across bash versions. On bash 4.3 and later, just write `"${arr[@]}"` directly.

If you're writing scripts for modern systems only (RHEL 8+, Ubuntu 18.04+, Debian 10+), bash 4.3 is a safe minimum assumption and this workaround is not needed.

### set -o pipefail and grep|head — The SIGPIPE False Positive

This one causes real confusion. With `set -o pipefail` active, this common pattern fails:

```bash
#!/bin/bash
set -euo pipefail

# Find the first matching line in a log file
first_match=$(grep "ERROR" /var/log/app.log | head -1)
```

```text
Terminated
```

The script exits non-zero even though `grep` found matches and `head` printed one. Here's what happens: `head -1` reads one line and exits. When `head` exits, the pipe is closed. `grep` tries to write its next match to the closed pipe and receives `SIGPIPE` — signal 13. By default, `grep` terminates with signal 13, which translates to exit status 141 (128 + 13). `pipefail` sees that `grep` exited non-zero and reports the pipeline as failed.

The workaround depends on what you need:

```bash
# Option 1: tolerate SIGPIPE on grep specifically
first_match=$(grep "ERROR" /var/log/app.log | head -1 || true)

# Option 2: use grep -m1 to stop after the first match — grep exits 0
first_match=$(grep -m1 "ERROR" /var/log/app.log)

# Option 3: check PIPESTATUS manually
grep "ERROR" /var/log/app.log | head -1
# PIPESTATUS[0] is grep's exit, PIPESTATUS[1] is head's exit
# 141 = SIGPIPE = acceptable termination, not an error
[[ ${PIPESTATUS[0]} -eq 0 || ${PIPESTATUS[0]} -eq 141 ]] || exit 1
```

Option 2 is the cleanest: if `grep -m1` finds a match, it exits 0 after writing one line. If it finds no matches, it exits 1 and the assignment fails, which is correct — `set -e` should catch that.

### trap EXIT Fires on Every Exit — Make Cleanup Idempotent

The trap fires on every exit: end of script, `exit 0`, `exit 1`, a `set -e` triggered exit, a caught signal that terminates. This is the feature — you don't have to remember to clean up in every exit path.

The trap is not double-called. It fires once. But your cleanup function must be safe to call at any point during the script's execution — including before any of the resources it cleans up have been created.

```bash
#!/bin/bash
set -euo pipefail

TMPDIR_CREATED=""

cleanup() {
    # Wrong: will fail with error if script exits before mktemp ran
    rm -rf "$WORK_DIR"
}

# Right: check before removing
cleanup() {
    [[ -n "${WORK_DIR:-}" && -d "$WORK_DIR" ]] && rm -rf "$WORK_DIR"
}

trap cleanup EXIT

WORK_DIR="$(mktemp -d)"
```

Note the `${WORK_DIR:-}` — the `:-` default value expansion prevents a `set -u` error if the script exits before `WORK_DIR` is set.

### Unquoted Variables With Glob Characters

`set -u` catches unset variables. It does not catch the word-splitting and glob-expansion problems from unquoted variables that are set. This is shellcheck's territory.

```bash
#!/bin/bash
set -euo pipefail

logdir="/var/log/app"
# If any file in logdir contains a space, this iterates incorrectly
for f in $(ls "$logdir"); do      # Wrong: word splits on spaces in filenames
    process "$f"
done

# Correct: glob expansion, each result is handled as a unit
for f in "$logdir"/*; do
    process "$f"
done
```

`shellcheck` reports the first form as SC2045 ("Iterating over ls output is fragile"). Fix it before deployment.

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| `set -euo pipefail` (bash) | Any bash script that runs unattended | The standard. Use it. |
| `#!/bin/sh` with `set -eu` | Portability to POSIX sh required (Alpine, minimal containers) | `pipefail` is not POSIX until very recently; avoid it in sh scripts |
| `#!/usr/bin/env bash` | Scripts that must run on macOS, BSD, or developer machines alongside Linux | Slightly less predictable; relies on PATH |
| shellcheck in CI | Every project with shell scripts | Run `shellcheck -e SC2236 *.sh` in your pipeline; treat warnings as errors |
| `bats` (Bash Automated Testing System) | Scripts complex enough to unit-test | Testing framework for shell scripts; out of scope here but worth knowing |
| Python / Ruby / Go instead | Scripts over ~200 lines or with complex data structures | At some complexity level, shell is the wrong tool; the safety mechanisms here paper over a language that was not designed for that complexity |

The honest recommendation: `set -euo pipefail` is not a complete solution to shell script safety — it's a baseline. Past a certain complexity threshold, shell scripts become error-prone regardless of which flags are set. The flags buy you safety in simple scripts; they don't transform shell into a safe language for complex programs.

## Trade-offs

**Where this shines:**
- Catches the single largest class of shell script bugs: commands that fail silently and let the script continue into a broken state
- `set -u` catches variable name typos at runtime — errors that are otherwise invisible
- `set -o pipefail` catches failures in complex pipelines that `set -e` alone misses
- `trap EXIT` cleanup is more reliable than cleanup at every explicit exit point — it fires even on `set -e` exits that you didn't write a handler for
- Zero runtime overhead — these are shell-internal flags

**Where this costs you:**
- `set -e` makes certain patterns verbose: any command that is allowed to fail (e.g., `grep` returning 1 for "no match") must be handled explicitly with `|| true` or an `if` condition, or it will exit the script
- `set -o pipefail` breaks the `grep | head` pattern that nearly every sysadmin has written — requires rethinking pipelines that intentionally read partial output
- `set -u` breaks the common pattern of using an unset variable as a default-empty value (e.g., `${OPTIONAL_FLAG}` where the intent is "empty if not set"); you must use `${OPTIONAL_FLAG:-}` explicitly
- `trap EXIT` cleanup complicates scripts that intentionally exit with different states from different paths — cleanup must be idempotent and state-aware
- Scripts with these flags are marginally harder to write correctly — the flags catch real bugs but also require you to be explicit about every place you intend for a command to possibly fail

---

## When to Use / When to Avoid

✅ **Use this when:**
- Any script that will run unattended (cron, CI/CD, deployment automation, system init)
- Any script that modifies state: creates files, writes to databases, changes configuration
- Any script longer than about 20 lines — at that length, silent failures become genuinely hard to debug
- Scripts that will be run by other people or maintained long-term
- Any script that already bit you with a silent failure and you're rewriting it

❌ **Avoid this when:**
- Quick throwaway scripts you type once and never run again — the overhead isn't worth it for 5-line interactive hacks
- Scripts that must be strictly POSIX sh portable (drop `pipefail` at minimum, and test carefully)
- Scripts that intentionally probe for the presence of tools or paths, where many commands are expected to fail — consider restructuring with explicit `if` checks instead

⚠️ **Common mistakes:**
- Putting `set -euo pipefail` after the first real command — it only applies to commands after the line where it's set; always put it on line 2, immediately after the shebang
- Writing `set -e; set -u; set -o pipefail` on three separate lines — this works, but `set -euo pipefail` on one line is the established convention; readers recognize it instantly
- Using `|| true` everywhere to suppress errors — if you're suppressing every error, the safety flags are doing nothing; restructure with explicit error handling instead
- Forgetting that `set -u` in the parent shell does not carry into subshells unless they also have `set -u` — a subshell invoked without it can still use unset variables silently

## Veteran Perspective

The first thing I check when I open an unfamiliar shell script is line 1 and line 2. If line 2 isn't `set -euo pipefail`, I immediately lower my confidence in everything else the script does. Not because the person who wrote it was incompetent — but because writing robust shell without these flags requires actively thinking about every command's failure mode. Most people don't. The flags are a forcing function.

The intermediate mistake — the one I see in scripts written by people who've been burned by silent failures — is using `set -e` without `pipefail`. They know enough to stop on errors, but a pipeline like `generate_data | transform | write_output` can have `generate_data` fail and produce no output, `transform` succeed on empty input, and `write_output` write an empty file — all exits 0, pipeline exit 0, `set -e` doesn't fire. You've silently written nothing to your output. This has caused data loss in real systems.

The `set -u` failure mode is different but equally common. You have a script with a variable named `$OUTPUT_DIR`. Somewhere above the usage, you have `OUTPUT_DIR="/var/data/output"`. Six months later someone adds a new code path and forgets to set `OUTPUT_DIR` in that path. Without `set -u`, the `rm -rf $OUTPUT_DIR/old_data` in the cleanup function expands to `rm -rf /old_data`. If the script runs as root — and cron jobs often do — that's a real problem. With `set -u`, the script exits immediately at the unset reference with a clear error message.

The `trap EXIT` pattern is the one that consistently separates junior scripts from senior scripts. Junior: cleanup code duplicated at every `exit` call, or absent entirely. Senior: one `trap cleanup EXIT` at the top, and the cleanup function is idempotent. The test is simple: read the cleanup function and ask whether it's safe to call it at any point during the script's execution, including before any resource has been created. If the answer is no, the cleanup is wrong.

`shellcheck` is not optional. Running it is the cheapest code review you'll ever get. It knows about quoting, it knows about glob expansion, it knows about the `[` vs `[[` distinction, it knows about `local` variable scoping in functions. SC2086 ("Double quote to prevent globbing and word splitting") is the single most common warning and is almost always a real bug. If your team has CI and your shell scripts aren't being run through `shellcheck` as part of it, that's a gap worth closing.

One war story worth telling: a deployment script at a previous company ran daily via cron to rotate log files. It had been in production for two years. The script included a line like `cd $LOG_DIR && rm -f *.log`. `LOG_DIR` was set from a config file. One day the config file had a formatting error — the variable got set to an empty string. The script ran `cd && rm -f *.log`. `cd` with no argument goes to the home directory. The home directory was `/root`. The `*.log` glob found no `.log` files in `/root`, so `rm` exited cleanly. The script reported success. The logs weren't rotated, but no one noticed for three days. Now: imagine there had been `.log` files in `/root`. Or imagine the command was `rm -rf $DEPLOY_DIR` instead. `set -u` would have caught the empty variable. `"$LOG_DIR"` quoting would have turned the `cd` into `cd ""` which fails and would have triggered `set -e`. Two characters and three flags would have saved the incident.

## Try It Yourself

1. Create a file called `test_safe.sh` with `#!/bin/bash`, `set -euo pipefail`, and three commands: `echo "start"`, `ls /tmp/definitely_missing_$$`, and `echo "end"`. Make it executable and run it. Verify that "end" does not print.

2. Add `|| true` after the `ls` command from prompt 1. Run it again. Observe that "end" now prints. This is the explicit "I know this might fail and I don't care" pattern. Remove `|| true` when you're done.

3. Write a script that creates a temp file with `mktemp`, writes the date into it, then deliberately exits with `exit 1`. Add a `trap` that prints "cleaning up" and removes the temp file. Verify that the trap fires and the temp file is removed even though the script fails.

4. Write a script that takes one argument and stores it in a variable named `INPUT`. Add `set -u`. Run the script with no arguments — observe the error. Then add proper argument validation using `if [[ $# -eq 0 ]]` before any variable usage, so the script gives a helpful error message instead of a cryptic "unbound variable."

5. **Challenge:** Write a script that uses `flock` to ensure only one instance runs at a time. The script should: acquire a lock on `/tmp/myapp.lock`, sleep for 10 seconds (simulating work), then exit. Open two terminals and start the script in both simultaneously. The second should either wait or fail with a "already running" message, depending on whether you use blocking or non-blocking `flock`. Add a `trap EXIT` that ensures the lockfile is cleaned up. Test that after the first instance exits, the second can acquire the lock. (Requires a VM or a throw-away environment is not needed — this is safe to run anywhere.)

## Summary

- `set -euo pipefail` belongs on line 2 of every script that runs unattended; `set -e` exits on error, `set -u` treats unset variables as errors, `pipefail` makes pipeline exit status reflect the first failing command in the pipeline
- `set -e` does not apply inside `if`/`elif`/`while` conditions, `&&`/`||` chains, or negated commands with `!` — failures in those positions are intentional control flow, not unhandled errors
- Quote every variable expansion with double quotes; unquoted variables undergo word splitting and glob expansion, which silently corrupts arguments containing spaces or glob characters
- Use `mktemp` to create temporary files and `trap cleanup EXIT` to ensure they're removed on any exit, including unexpected failures; make the cleanup function idempotent
- `shellcheck` finds quoting bugs, common logic errors, and portability issues that the safety flags cannot catch; run it and treat its warnings as errors
