## Why This Exists

Early Unix systems had no interactive command interface built in. You booted the machine and got a login prompt connected directly to the kernel. Users needed a way to run programs, chain them together, and automate repetitive work — without recompiling the kernel every time they wanted to add a new command.

The solution was a user-space program called a shell: a process that reads your input, interprets it, and translates your intentions into system calls. The shell is not part of the kernel. It is an ordinary program that happens to know how to launch other programs.

That design decision — shell as a separate, replaceable program — is why you have `bash` on your laptop, `dash` running your system init scripts, and `sh` in a Docker container, all doing roughly the same job with different trade-offs. It is also why a badly written script can silently fail in ways that a compiled program would not.

---

## Mental Model

Think of the shell as a smart translator at a press conference.

You speak in human-readable commands. The translator (shell) doesn't just pass your words straight through to the kernel — it first unfolds abbreviations, substitutes variables, expands wildcards into file lists, and resolves aliases. Only after all that preparation does the translated instruction reach the official (the kernel).

When the translator gets something wrong — misreads your abbreviation, expands a wildcard you didn't intend, or uses a different dialect than you expected — the official hears something different from what you said. No error. No warning. Just a different outcome.

That's the shell. Understanding how it translates is the difference between someone who uses Linux and someone who controls it.

## Theory

### What a shell is, precisely

A shell is a user-space process that provides a command language interpreter. It reads input — either from an interactive terminal or from a script file — and uses a defined set of rules to expand, parse, and execute it. The kernel never sees your raw command. The shell sees it first, transforms it, and then asks the kernel to run something specific.

The shell is also responsible for:
- Managing your environment (environment variables that child processes inherit)
- Setting up I/O redirection before a child process starts
- Handling job control — stopping, resuming, and backgrounding processes
- Providing a programming language: variables, conditionals, loops, functions

### POSIX sh and the shell family

POSIX defines a standard called `sh` — a portable shell language specification. Any shell claiming POSIX compliance must support this baseline. The important invariant: a script written for POSIX `sh` will run on any compliant shell. A script that uses bash-specific features — called "bashisms" — will only run correctly on bash.

Bash (the Bourne Again Shell) is a superset of POSIX sh. It adds arrays, `[[` compound conditions, `$'...'` quoting, `{1..10}` brace expansion, process substitution `<(...)`, `shopt` options, and much more. These are powerful features. They are also features that will silently fail or behave differently when the script is run by `dash`, `ash`, or `sh`.

Bash 5.3 is the current stable release (released July 2025). It added a new form of in-context command substitution, a faster command lookup table, and associative array improvements. Most production servers still run bash 5.1 or 5.2, so treat 5.3 features as not yet universally available.

### sh vs bash vs dash vs zsh

**bash** is the default interactive login shell on the vast majority of Linux distributions. When you open a terminal and start typing, you are almost certainly talking to bash.

**dash** (the Debian Almquist Shell) is a minimal, fast, strictly POSIX-compliant shell. It does not support most bashisms. Its startup time is roughly three to ten times faster than bash, which is why Debian and Ubuntu made it the default `/bin/sh` starting with Ubuntu 6.10 in 2006. Dash is used to run system scripts during boot — not as your interactive shell.

**sh** is not a shell — it is a name. On your system, `/bin/sh` is a symlink to something. What it points to is a distro decision. Check: `ls -la /bin/sh`.

**zsh** (Z Shell) is a modern alternative to bash with superior interactive features: smarter tab completion, shared history across sessions, theme support, powerful glob qualifiers, and a plugin ecosystem (Oh My Zsh). macOS switched its default shell to zsh in Catalina (2019). Many developers prefer zsh for interactive use. For scripting, the same portability rules apply — use `#!/bin/zsh` explicitly if you want zsh features, and never assume zsh on a server.

For scripting, the rule is: if you need arrays, `[[`, or process substitution, use `#!/bin/bash`. If you're writing a script that must be portable to any POSIX system, use `#!/bin/sh` and restrict yourself to POSIX features.

### Startup files: which file loads in which mode

The shell has four distinct startup modes, and each loads a different set of files. Getting these wrong is the number one reason why aliases and environment variables disappear in cron jobs.

A **login shell** is invoked when you first authenticate: SSH sessions, virtual console logins, `su -`. It reads:
1. `/etc/profile` (system-wide)
2. The first of these that exists: `~/.bash_profile`, `~/.bash_login`, `~/.profile`

A **non-login interactive shell** is what you get when you open a terminal emulator inside a running desktop session. It reads:
1. `/etc/bash.bashrc` (system-wide, Debian/Ubuntu) or `/etc/bashrc` (RHEL/Fedora)
2. `~/.bashrc`

A **non-interactive shell** — a shell running a script, a cron job, a subshell — reads neither. The only environment it has is what was exported from its parent process, plus anything set by the `BASH_ENV` variable if that's defined.

A **login shell exiting** reads `~/.bash_logout` before closing.

The most common practical consequence: `~/.bashrc` is not read in a cron job. Your custom PATH, your aliases, your functions — gone. You must export what you need, or source files explicitly inside the script.

### PATH: how the shell finds executables

When you type `ls`, the shell doesn't know where `ls` lives. It searches each directory in the `PATH` environment variable from left to right and runs the first match it finds.

`PATH` is just a colon-separated list of directories:

```text
/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin
```

The shell caches the result of each PATH lookup in a hash table. This is an optimization — subsequent calls to `ls` don't need to walk `PATH` again.

### Builtins vs external commands

Some commands are implemented inside the shell itself. `cd`, `echo`, `type`, `alias`, `export`, `source`, `read`, `exit` — these are builtins. They run directly in the current shell process. They have no corresponding file on disk (try `which cd` — you'll get nothing or a wrapper). They cannot be strace'd as a separate process because they are not separate processes.

External commands — `ls`, `cat`, `grep`, `awk` — are files on disk. The shell forks a child process, sets up I/O, and calls `execve()` to run the binary. You can find them with `which ls`. You can strace them.

The distinction matters because only external commands can be replaced by installing a different binary. Builtins can only be overridden by shell functions or aliases defined before them in lookup order. And only external commands respect PATH — builtins don't need it.

## How It Works (Mechanics)

### The shell's processing pipeline

When you press Enter, the shell does not immediately execute what you typed. It runs the input through a series of transformation stages, in order. Understanding the order is essential — a mistake at any stage produces a different command than you intended.

```text
# Conceptual flow — not a real command
1. Tokenisation       → split input into words and operators
2. Alias expansion    → replace alias names with their definitions
3. Parsing            → build a command tree (pipelines, redirects, compound commands)
4. Brace expansion    → expand {a,b,c} and {1..5} into word lists
5. Tilde expansion    → replace ~ with $HOME
6. Parameter/variable expansion   → replace $VAR with its value
7. Command substitution           → replace $(cmd) with the output of cmd
8. Arithmetic expansion           → evaluate $((expr))
9. Word splitting     → split unquoted expansions on IFS characters
10. Pathname expansion (globbing) → expand *, ?, [...] into matching filenames
11. Command lookup    → search builtins, functions, then PATH hash table, then PATH
12. Execution         → fork + execve for external commands; direct call for builtins
```

The order matters in ways that catch people off guard. Aliases are expanded early, before the command is parsed — which is why an alias can rewrite a command entirely. Globbing happens after word splitting — which is why unquoted variables containing spaces cause unexpected glob patterns. Quotes suppress most expansions from step 4 onward within the quoted region.

### Command lookup order

When the shell resolves a command name, it checks in this order:
1. Aliases
2. Shell functions
3. Shell builtins
4. The PATH hash table (cached lookups)
5. A fresh search through PATH directories

This means a function named `ls` will shadow the real `ls`. An alias named `grep` will shadow a function named `grep`. The `type` builtin (not `which`) shows you exactly what the shell will actually run for a given name.

### The hash table

After the first time the shell finds an external command via PATH, it stores the full path in a hash table indexed by the command name. On subsequent calls, it skips the PATH walk and uses the cached path directly.

```bash
# Show the current hash table contents
hash
```

```text
hits	command
   1	/usr/bin/git
   3	/usr/bin/grep
   7	/usr/bin/ls
```

The number under "hits" is how many times that cached entry has been used. The table is per-session — it resets when you open a new shell. It also resets when you reassign `PATH`.

## Basic Example

```bash
# Show what type of thing a command name resolves to
type ls
type cd
type type
```

```text
ls is /usr/bin/ls
cd is a shell builtin
type is a shell builtin
```

```bash
# Show all interpretations of a command name (function, alias, builtin, and file)
type -a echo
```

```text
echo is a shell builtin
echo is /usr/bin/echo
```

```bash
# See current aliases
alias
```

```text
alias ll='ls -alF'
alias la='ls -A'
alias ls='ls --color=auto'
```

```bash
# Show what /bin/sh actually is
ls -la /bin/sh
```

```text
lrwlink rwxrwxrwx 1 root root 4 Jul 18  2023 /bin/sh -> dash
```

Run `type ls` and `type cd` on your own system. The difference between a path and "shell builtin" in the output is the entire distinction between builtins and external commands. Now try `type -a echo` — you'll see that `echo` exists as both a builtin and a file, and the shell always uses the builtin first.

## Real-World Example

A new team member installs a custom version of `node` into `/usr/local/bin/` and then complains that the system is still running the old version from `/usr/bin/`. Here's the diagnostic and fix:

```bash
# Step 1: See what the shell is actually going to run
type node
```

```text
node is hashed (/usr/bin/node)
```

The word "hashed" tells you the shell is using a cached lookup and the cache points to the old location. The new binary at `/usr/local/bin/node` is earlier in PATH but the cache hasn't been updated.

```bash
# Step 2: Confirm PATH order — /usr/local/bin should come first
echo $PATH
```

```text
/usr/local/bin:/usr/bin:/bin:/usr/sbin:/sbin
```

`/usr/local/bin` is first. The new binary is there. The cache is just stale.

```bash
# Step 3: Clear the hash table and re-check
hash -r
type node
```

```text
node is /usr/local/bin/node
```

```bash
# Or: remove just the one stale entry without clearing everything
hash -d node
```

A senior sysadmin sees "hashed" in `type` output and immediately suspects a stale cache. A junior sysadmin runs `which node`, sees `/usr/local/bin/node`, concludes the PATH is correct, and can't explain why the wrong version runs. `type` tells you what the shell will run. `which` tells you what it would find if it searched PATH fresh — those can be different things.

## ⚠️ Gotchas

### `#!/bin/bash` vs `#!/bin/sh` — the silent failure on Ubuntu servers

You write a script with `#!/bin/sh` at the top. You test it on your machine. It works. You deploy it to an Ubuntu server. It fails silently — no error, just wrong behavior.

```bash
#!/bin/sh
# This looks fine
items=("apple" "banana" "cherry")   # bash array syntax
echo "${items[0]}"
```

```text
/path/to/script.sh: 3: Syntax error: "(" unexpected
```

On your development machine, `/bin/sh` might point to bash. On Debian and Ubuntu, `/bin/sh` is `dash`. Dash doesn't support arrays, `[[ ]]`, `$'...'` string quoting, process substitution, or many other bash features. The script fails with a syntax error — or worse, silently does something different.

The rule is absolute: if your script uses any feature you can't find in the POSIX specification, the first line must be `#!/bin/bash`, not `#!/bin/sh`. Check a script for bashisms with `checkbashisms` (from the `devscripts` package on Debian/Ubuntu). Never rely on what `/bin/sh` points to on your local machine to tell you what it points to on a target server.

### `which` lies about builtins — use `type` instead

A common pattern for checking whether a command exists or finding its path:

```bash
which cd
```

```text
# (no output, or on some systems:)
/usr/bin/cd
```

`which` is an external program. It knows nothing about the shell's builtin table, function table, or alias table. It searches PATH and reports what it finds there. For `cd`, it finds nothing (or finds a rarely-used external `cd` wrapper that's not what runs when you type `cd`). For an alias like `ll`, it finds nothing. For a shell function, it finds nothing.

`type` is the correct tool:

```bash
type cd
type ll
type grep   # may show an alias if you have alias grep='grep --color=auto'
```

```text
cd is a shell builtin
ll is aliased to `ls -alF'
grep is aliased to `grep --color=auto'
```

Use `type -a name` to see every interpretation of a name. Use `command -v name` in scripts for POSIX-portable "does this exist" checks. Use `which` only when you specifically want to know the filesystem path of an external binary — and even then, understand its blind spots.

### Aliases don't work in non-interactive scripts

You define a useful alias in `~/.bashrc`:

```bash
alias deploy='git pull && make install && systemctl restart myapp'
```

You write a script that calls `deploy`. You run it. It fails:

```bash
#!/bin/bash
deploy   # call the alias
```

```text
./deploy.sh: line 2: deploy: command not found
```

Aliases are an interactive-shell feature. They are loaded from `~/.bashrc`. Non-interactive shells don't read `~/.bashrc`. Even if you force-source it, aliases are disabled in non-interactive shells unless you explicitly enable them with `shopt -s expand_aliases`.

The correct answer for scripts is shell functions, not aliases. Functions are defined in files you explicitly source, or inline in the script. They work in all shell modes. If you absolutely must use an alias in a script, add `shopt -s expand_aliases` before the alias definition and before any calls to it.

### A newly installed binary isn't found until the hash cache updates

You install a new tool:

```bash
sudo apt install ripgrep
rg --version
```

```text
bash: rg: command not found
```

But `which rg` shows `/usr/bin/rg` exists. The problem: the shell doesn't automatically detect new binaries. The hash table was built before `rg` was installed. The shell tried to find `rg` before installation, got "not found", and cached that negative result — or simply hasn't searched for it yet in a session that was open during the install.

Fix:

```bash
hash -r      # clear the entire hash table; forces fresh PATH walk on next command
# or:
hash rg      # explicitly add rg to the hash table right now
```

Opening a new shell also solves it. Assigning `PATH=$PATH` also clears the hash table (because any PATH modification flushes it). In automated scripts that install software and then immediately use it, the safest approach is to use the full path or call `hash -r` explicitly.

### Login vs non-login: why your PATH and aliases vanish in cron jobs

A cron job runs as a non-interactive, non-login shell. It does not read `~/.bash_profile`, `~/.profile`, or `~/.bashrc`. The `PATH` it gets by default is a minimal system path: usually something like `/usr/bin:/bin`. Your custom `PATH` additions, your `nvm` or `rbenv` or `pyenv` shims, your aliases — none of them exist in cron.

```bash
# In a crontab — this will fail if python3 lives in /usr/local/bin
* * * * * python3 /home/user/script.py
```

The fix is to either:
1. Use absolute paths in cron jobs: `/usr/local/bin/python3 /home/user/script.py`
2. Set `PATH` explicitly at the top of the crontab: `PATH=/usr/local/bin:/usr/bin:/bin`
3. Source your profile inside the script: `. /home/user/.bash_profile` — but understand this can have side effects

This same "missing environment" problem hits sudo sessions, systemd service units, and any other context where a shell is started without going through the full login process.

## Distro Notes

| Distro Family | `/bin/sh` target | Default login shell | Notes |
|---|---|---|---|
| Debian / Ubuntu | `dash` | `bash` | Changed from bash to dash in Ubuntu 6.10 (2006); system scripts use dash, interactive sessions use bash |
| RHEL / Fedora / CentOS / Rocky | `bash` | `bash` | `/bin/sh` symlinks to bash; `checkbashisms` not installed by default |
| Arch Linux | `bash` | `bash` | `/usr/bin/sh` is owned by the bash package |
| Alpine (containers) | `busybox ash` | `ash` | Neither bash nor dash; ash is even more minimal; install bash explicitly with `apk add bash` |
| macOS | `bash` (old) / `zsh` (Catalina+) | `zsh` | `/bin/sh` uses Apple's vendored bash 3.2 (GPL2) on Intel; moved default login shell to zsh in 2019 |

## Alternatives & Comparisons

| Shell | When to Use | Notes |
|---|---|---|
| `bash` | Default for scripting and interactive use on Linux servers | Ubiquitous; safe default; avoid 5.3-specific features on older servers |
| `dash` | System init scripts; anything where startup speed matters | POSIX only; no arrays, no `[[`, much faster than bash |
| `zsh` | Interactive developer workstations; personal dotfiles | Excellent completion and UX; macOS default since 2019; not universally installed on servers |
| `sh` (POSIX) | Maximum portability scripts; Docker entrypoints | Not a real shell — it's whatever `/bin/sh` points to; test on your target |
| `ash` / `busybox sh` | Container images built on Alpine | Very minimal; only BusyBox builtins plus basic POSIX; no bashisms |
| `fish` | Personal interactive use | Not POSIX; incompatible syntax; excellent for learning and daily use; never for scripts others will run |

For scripting on Linux servers: use `bash` with `#!/bin/bash`. For maximum portability (Docker base images, embedded systems, CI containers): use `sh` with POSIX-only syntax and test on `dash`. For interactive developer experience: `zsh` is the honest recommendation in 2026.

## Trade-offs

**Where bash shines:**
- Arrays, associative arrays, and the full bash feature set make complex scripting genuinely feasible
- Ubiquitous — it's on almost every Linux system; your bash script will run without installing anything
- Excellent debugging tools: `set -x`, `set -e`, `set -u`, `set -o pipefail`
- Extensive documentation and decades of answered questions online

**Where bash costs you:**
- Startup overhead — bash initializes more state than dash; matters when a script spawns thousands of subshells
- Portability — bash scripts tied to bashisms can't run on Alpine, BusyBox, or strict POSIX environments
- Shellcheck hygiene — bash's permissiveness means errors that other languages would catch at parse time silently succeed at runtime
- The `set -e` trap — bash's "exit on error" behavior has edge cases subtle enough to have caused real production incidents

**Performance note:** Bash script startup takes roughly 5–15ms per invocation on a modern system. A script that forks 1,000 bash subshells pays ~5–15 seconds in startup overhead alone. For batch processing at that scale, Python or a compiled language is the right tool.

---

## When to Use / When to Avoid

✅ **Use bash when:**
- Automating sequences of shell commands that you'd otherwise type interactively
- Writing system administration scripts that will run on known Linux targets
- Gluing together existing Unix tools (grep, awk, cut, find) in pipelines
- Quick one-off scripts for your own use

✅ **Use POSIX sh (`#!/bin/sh`) when:**
- Writing scripts for Docker base images (especially Alpine-based)
- Writing scripts that ship with packages and must run on any Linux system
- Writing system init scripts or scripts that run early in boot

❌ **Avoid bash scripting when:**
- The logic is complex enough to require data structures beyond arrays — use Python
- The script will be long-lived and maintained by multiple people — bash is hard to review and test
- You need robust error handling — bash's error model is full of exceptions and surprises
- Performance requires tight loops over large data — bash is slow; use Python, Rust, or Go

⚠️ **Common mistakes:**
- Writing `#!/bin/sh` but using bash arrays or `[[ ]]` — works on your machine, fails on the server
- Putting customizations in `~/.bash_profile` instead of `~/.bashrc` and wondering why they don't appear in new terminal windows
- Calling `which` to check if a command exists in a script — use `command -v name` for POSIX portability, or `type name` in bash

## Veteran Perspective

The single most important thing about the shell is that it transforms your input before executing it. Beginners think of the shell as a thin wrapper around the kernel. It isn't. It is a full interpreter that rewrites what you type in multiple stages. Once you internalize that the shell runs before the command, every quoting rule, every globbing surprise, every "why did that expand that way" makes sense.

The `#!/bin/bash` vs `#!/bin/sh` mistake has taken down production deployments more times than I want to count. The pattern is always the same: developer writes script on a Mac or Fedora box where `/bin/sh` is bash, deploys to Ubuntu, script runs during a deployment or boot sequence, and something silently does the wrong thing — or errors with a cryptic syntax message during a production outage window. The fix costs thirty seconds: type `#!/bin/bash`. The investigation costs thirty minutes. Make the right choice once at the top of the file.

The `type` vs `which` distinction separates people who understand the shell from people who use it. `which` is a relic that many senior engineers still reach for out of habit. `type -a` gives you the complete picture — alias, function, builtin, and file — which is what you actually need when debugging "why is the wrong command running."

The hash table gotcha catches intermediate users more than beginners. Beginners see "command not found" and install the package again. Intermediate users have already learned enough to use `which` and `type`, check that the binary exists at the expected path, confirm PATH is correct — and then spend twenty minutes confused about why the shell still can't find it. `type` showing "hashed" with the old path is the diagnostic. `hash -r` is the fix. Keep it in your toolkit.

The startup file confusion — login vs non-login vs interactive vs non-interactive — is one of the most persistent sources of "it works on my machine" failures in shell scripting. Draw yourself the matrix: four modes, three file groups. Burn it into memory. Every time something works interactively but fails in a cron job or service, check which mode the failing shell is in and which files it does and doesn't read.

Aliases feel powerful when you're learning the shell. In practice, they are interactive conveniences. Shell functions are their scriptable equivalent. If you're putting complex behavior in an alias, you're about to be surprised by quoting, you can't pass arguments naturally, and you can't call it from a script. Write a function instead.

## Try It Yourself

1. Run `type ls`, `type cd`, `type type`, and `type alias`. Note which are builtins and which have filesystem paths. Then run `type -a echo` — observe that echo exists as both a builtin and an external binary.

2. Check what `/bin/sh` points to on your system: `ls -la /bin/sh`. Then run `dash --version` (or install dash with `apt install dash`). Start a dash session with `dash` and try running `echo ${BASH_VERSION}` — notice the difference from bash.

3. Open two terminal windows. In the first, define an alias: `alias greet='echo hello'`. Try calling `greet` in the second terminal. Observe that it doesn't exist there. Now close both and put the alias in `~/.bashrc`. Open new terminals and confirm it works in each.

4. Create a test script named `test-mode.sh` with `#!/bin/bash` and the single line `echo "BASH_ENV=$BASH_ENV; interactive=$-"`. Run it as `bash test-mode.sh` and compare the output to running `bash -i test-mode.sh`. Look up what the letters in `$-` mean (hint: `man bash`, search for "special parameters").

5. **Challenge:** Install a package that provides a command (`ripgrep`, `fd-find`, `bat`, or similar — whichever isn't already on your system). Before installing, run `type rg` (or your chosen command). After installing, run `type rg` again without opening a new shell. Observe the result. Then use `hash -r` to fix it. Write a three-line shell snippet that installs a package and immediately makes the new binary available in the same shell session, without opening a new shell.

## Summary

- The shell is a user-space interpreter that transforms your input through multiple expansion stages before the kernel ever sees a command; the order is: brace expansion → tilde → parameter/variable/command/arithmetic expansion → word splitting → glob → command lookup → execution.
- `/bin/sh` is a symlink, not a shell — it points to `dash` on Debian/Ubuntu, `bash` on RHEL/Arch, and `busybox ash` on Alpine; using bash features in a `#!/bin/sh` script will silently fail wherever sh is not bash.
- Bash startup files depend on shell mode: login shells read `/etc/profile` then `~/.bash_profile`; non-login interactive shells read `~/.bashrc`; non-interactive shells (scripts, cron jobs) read neither — your custom PATH and aliases do not exist there.
- `type` shows you what the shell will actually run (alias, function, builtin, or file); `which` only searches PATH and lies about builtins — always use `type` for debugging command resolution.
- The shell caches external command paths in a hash table after the first lookup; installing a new binary doesn't update the cache — run `hash -r` or `hash <name>` to force a fresh lookup in the current session.
