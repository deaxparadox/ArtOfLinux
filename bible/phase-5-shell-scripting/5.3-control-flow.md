# [5.3] — Control Flow

## The Problem This Solves

Imagine writing a backup script that has to do two different things depending on whether a directory exists, and a third thing if the directory exists but is empty. Without branching, you'd need separate scripts for each case, or you'd have to accept that your script blindly runs the same steps regardless of what it finds — hoping the world matches its assumptions.

The same problem appears in loops: if you need to process every log file in a directory, you could write twenty `cp` commands if there happened to be twenty files right now. But tomorrow there are fifteen. The day after, thirty-two. A script that hardcodes the count breaks the moment reality changes.

Control flow — `if`, `case`, `for`, `while`, `until` — is how a script becomes a program instead of a recipe. It lets the script ask questions about the state of the system and change its behavior based on the answers.

The subtlety that trips people up: in bash, every condition is just an exit code. There is no boolean type. There is no `true` or `false` keyword in the C sense. When an `if` statement asks "is this condition true?", it is asking "did this command exit zero?" Understanding that single rule unlocks every form of branching in bash.

---

## Mental Model

Picture a train on a track. A normal list of commands is a straight track — the train passes every station in order, no matter what. Control flow is the set of switches and junctions on that track.

An `if` statement is a junction: depending on the signal (the exit code of the test), the train takes the left track or the right track. A `for` loop is a loop of track that the train circles a set number of times before breaking out. A `while` loop is a loop where a switchman checks a signal before each lap — when the signal changes, the train exits the loop. `break` and `continue` are emergency switches that let the train skip ahead or bail out early.

The signal is always an exit code — 0 means "go", anything else means "stop". Every form of branching in bash reads that signal.

## Theory

### Exit Codes Are the Foundation

Every command in bash exits with a numeric status between 0 and 255. By universal Unix convention: 0 means success, non-zero means failure (with different non-zero values meaning different kinds of failure). The `if` keyword, `while`, and `until` all evaluate conditions by running a command and reading its exit code. Zero is true. Non-zero is false.

This is why you can write `if grep -q "error" /var/log/syslog` — `grep` exits 0 if it found the pattern and 1 if it did not. The `if` is just reading that exit code. There is no separate boolean step.

### `test`, `[`, and `[[` — Three Ways to Ask a Question

Bash provides several built-in tools for constructing conditions:

`test` is the original POSIX utility. It evaluates file tests, string comparisons, and integer comparisons, and exits 0 or 1 accordingly. `[` is exactly the same thing under a different name — it exists so you can write `if [ -f file ]` and have it read like natural syntax. Both are commands, which means they participate in the normal word-splitting and filename-expansion rules.

`[[` is different in kind, not just in syntax. It is not a command — it is a shell keyword. The shell handles `[[ ... ]]` as a special grammatical construct, not as a function call with arguments. Because of that, word splitting and filename expansion do not happen inside `[[ ]]`. You can write `[[ $filename == *.log ]]` and the `*` is treated as a glob pattern matching, not as a glob expansion against the filesystem. You can write `[[ $VAR == "some value" ]]` without quoting `$VAR` and the shell won't break it on whitespace. You can use `&&` and `||` inside `[[ ]]` instead of `-a` and `-o`. You can use `=~` for regex matching. `[[` is bash-only (also in ksh and zsh), not POSIX `sh`.

### `(( ))` — Arithmetic Conditions

`(( expression ))` evaluates an arithmetic expression and returns 0 if the result is non-zero, or 1 if the result is zero. This is the C convention: non-zero is true, zero is false. The implication is the opposite of what you might expect: `(( 1 ))` succeeds (exits 0), `(( 0 ))` fails (exits 1). Inside `(( ))` you can use `+`, `-`, `*`, `/`, `%`, `**`, `++`, `--`, `==`, `!=`, `<`, `>`, `<=`, `>=`, and bitwise operators. No `$` required for variable names — `(( count > 0 ))` works without `$count`.

### `if` / `elif` / `else` — Branching on Exit Codes

The `if` statement runs a command and checks its exit code. If the exit code is 0, the `then` block runs. Otherwise the next `elif` is tested, or if there are no more `elif`s, the `else` block runs. `fi` closes the construct. There is no limit on the number of `elif` branches.

Every `if` condition is a command list, not a value. `if grep -q "pattern" file; then` is just as valid as `if [[ -f "$file" ]]; then`. The semicolon before `then` is required when the command and `then` are on the same line; if `then` is on its own line, the semicolon can be omitted.

### `case` / `esac` — Pattern-Based Branching

`case` matches a value against a series of patterns and runs the code for the first matching pattern. Patterns are shell glob patterns — `*`, `?`, `[abc]`, `foo|bar` (pipe means "or"). A `case` branch ends with `;;`, which means "stop here and skip the rest". Two additional terminators were added in bash 4.0: `;&` (fall through unconditionally to the next clause's command list) and `;;&` (test the next pattern and execute its command list if it matches, continuing case execution as if that pattern hadn't matched yet).

### `for` Loops — Two Forms

The list-style `for` loop iterates over a whitespace-separated list of words: `for var in item1 item2 item3`. The C-style `for` loop uses arithmetic: `for (( i=0; i<10; i++ ))`. The C-style form is bash-only — it does not exist in POSIX `sh`. Both forms run a body delimited by `do` and `done`.

### `while` and `until` Loops

`while` runs its body as long as the condition command exits 0. `until` runs its body as long as the condition command exits non-zero — it is the logical inverse of `while`. Both test their condition at the top of the loop before running the body. Both are delimited by `do` and `done`.

### `break` and `continue`

`break` exits the innermost loop immediately. `continue` skips the rest of the current iteration and jumps to the next. Both accept an optional numeric argument: `break 2` breaks out of two levels of nested loops. This is one of the few places bash has explicit loop nesting control.

### `select` — Menu Generation

`select var in list` prints a numbered menu, reads the user's choice, sets `$var` to the selected item, and sets `$REPLY` to the raw input. It loops indefinitely until `break` is called or the user presses Ctrl-D. The prompt is controlled by `$PS3`. `select` is a convenience feature for simple interactive menus — it is rarely used in production scripts.

### `&&` and `||` — Short-Circuit Evaluation

`cmd1 && cmd2` runs `cmd2` only if `cmd1` exits 0. `cmd1 || cmd2` runs `cmd2` only if `cmd1` exits non-zero. These are not boolean operators — they are control flow using exit codes, exactly like `if`. The idiom `mkdir -p /tmp/mydir && cd /tmp/mydir` is idiomatic bash: if `mkdir` fails, the `cd` never runs. The idiom `[[ -n "$var" ]] || die "var is required"` is a compact guard clause. Used carefully, `&&` and `||` chains are readable and production-safe. Used carelessly in long chains, they become hard to reason about.

## How It Works (Mechanics)

### How `if` Evaluates a Condition

When bash encounters `if cmd; then`, it forks a child process to run `cmd` (or simply runs the builtin directly without forking, for builtins like `[` and `[[`). The parent waits for the command to finish and reads its exit status via `waitpid()`. If the exit status is 0, the `then` block is executed. Otherwise the next `elif` or `else` branch is selected.

Because the condition is just a command, the shell applies all the normal command-processing rules: parameter expansion, command substitution, word splitting, quote removal. This is the root of most `[ ]` gotchas — the words passed to `[` after expansion can be different from what you wrote.

`[[` short-circuits this by being a keyword. The shell's parser recognizes `[[ ... ]]` as a syntactic unit and handles the contents with special rules — no word splitting, glob expansion happens only in specific positions (pattern matching with `==` or `!=`), and `&&`/`||` inside are logical operators rather than list separators.

### How `[` Differs from `[[` Internally

`[` is a regular builtin command. When bash evaluates `[ $name == "hello" ]`, it performs full word splitting on `$name` first, then calls the `[` builtin with whatever words resulted. If `$name` is `foo bar`, it expands to two words and `[` receives `foo`, `bar`, `==`, `hello`, `]` — five arguments where it expected four. It errors. This is not a bug in `[`; it's the consequence of `[` being a command that receives arguments after expansion, not a keyword that sees the expression before expansion.

`[[` sees `$name` before word splitting because the shell recognizes `[[ ]]` grammatically and suppresses word splitting inside it. The variable's value is passed as a unit regardless of whitespace.

### How `(( ))` Returns an Exit Code

`(( expr ))` evaluates the arithmetic expression using C-language integer math. The resulting integer is then mapped to an exit code: if the integer is non-zero, the exit code is 0 (success/true). If the integer is zero, the exit code is 1 (failure/false). This is the standard C boolean convention flipped into shell exit code convention.

This means `(( x++ ))` has a side effect (it increments `x`) and also returns an exit code based on the value of `x` *before* the increment. If `x` was 0, the expression evaluates to 0, and the exit code is 1. Under `set -e`, that causes the script to exit immediately — while the increment still happened.

### How `case` Pattern Matching Works

`case` evaluates each pattern against the word being matched using the shell's pattern matching rules (same as filename globbing: `*`, `?`, `[...]`). Patterns are tested in order from top to bottom. When a match is found, the associated command list runs. With `;;`, the case construct is done. With `;&`, the next clause's command list runs without testing its pattern. With `;;&`, the next pattern is tested; if it also matches, its command list runs and the scan continues; if it doesn't match, the scan continues to the one after that.

## Basic Example

```bash
#!/bin/bash

# Demonstrate if/elif/else with [[ ]]
file="/etc/hostname"

if [[ -f "$file" ]]; then
    echo "File exists: $file"
    echo "Contents: $(cat "$file")"
elif [[ -d "$file" ]]; then
    echo "That path is a directory, not a file"
else
    echo "Path does not exist: $file"
fi
```

```text
File exists: /etc/hostname
Contents: myserver
```

The condition `[[ -f "$file" ]]` tests whether the path exists and is a regular file. `-d` tests for a directory. If neither matches, the `else` branch runs. Try changing `file` to a directory like `/etc` to see the `elif` branch fire, and to `/nonexistent` to see `else`.

```bash
#!/bin/bash

# Demonstrate a for loop over a list
for fruit in apple banana cherry; do
    echo "Fruit: $fruit"
done
```

```text
Fruit: apple
Fruit: banana
Fruit: cherry
```

```bash
#!/bin/bash

# Demonstrate a while loop with a counter
count=0
while (( count < 3 )); do
    echo "count is $count"
    (( count++ ))
done
```

```text
count is 0
count is 1
count is 2
```

Notice that the condition `(( count < 3 ))` uses arithmetic — no `$` needed in front of `count`. The loop runs while the arithmetic result is non-zero (true), and exits when `count < 3` evaluates to 0 (false).

*Try running each of these yourself. Modify the list in the `for` loop, change the threshold in the `while` loop, and observe the exact point where each loop exits.*

## Real-World Example

The following script is a deployment pre-flight check. It validates several conditions before proceeding, handles different types of input, and loops over a list of required services.

```bash
#!/bin/bash
set -euo pipefail

REQUIRED_SERVICES=("nginx" "postgresql" "redis")
DEPLOY_DIR="/srv/app"
LOG_FILE="/var/log/deploy.log"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $*" | tee -a "$LOG_FILE"
}

# Check required argument
if [[ $# -lt 1 ]]; then
    echo "Usage: $0 <environment>" >&2
    exit 1
fi

ENV="$1"

# Validate environment name against known values
case "$ENV" in
    production)
        DEPLOY_USER="deploy-prod"
        ;;
    staging)
        DEPLOY_USER="deploy-staging"
        ;;
    *)
        echo "Unknown environment: $ENV" >&2
        echo "Valid options: production, staging" >&2
        exit 1
        ;;
esac

log "Starting pre-flight check for environment: $ENV"

# Verify the deploy directory exists and is writable
if [[ ! -d "$DEPLOY_DIR" ]]; then
    log "ERROR: Deploy directory does not exist: $DEPLOY_DIR"
    exit 1
elif [[ ! -w "$DEPLOY_DIR" ]]; then
    log "ERROR: Deploy directory is not writable by $(whoami)"
    exit 1
fi

# Check each required service is running
all_ok=true
for service in "${REQUIRED_SERVICES[@]}"; do
    if systemctl is-active --quiet "$service"; then
        log "OK: $service is running"
    else
        log "FAIL: $service is not running"
        all_ok=false
    fi
done

if [[ "$all_ok" == false ]]; then
    log "Pre-flight failed: one or more services are not running"
    exit 1
fi

log "Pre-flight complete. Proceeding with deployment as $DEPLOY_USER."
```

Walk through what this script does:

The `case "$ENV"` block handles environment routing. The `*)` pattern is the catch-all — it matches anything not already matched. If someone passes `development` or a typo, they get a clear error and a list of valid options.

The `if [[ ! -d "$DEPLOY_DIR" ]]` block uses compound conditions: `-d` for directory existence, `!` for negation. Notice the `elif` — the checks are ordered so you don't attempt a write test on a path that doesn't exist.

The `for service in "${REQUIRED_SERVICES[@]}"` loop iterates over a bash array with proper quoting. `"${REQUIRED_SERVICES[@]}"` expands to each element as a separate word, preserving any spaces in service names (rare, but correct). A plain `$REQUIRED_SERVICES` without `[@]` gives you only the first element.

The `all_ok` flag is a string, not an integer, because bash has no native boolean type. The pattern `var=true`/`var=false` is idiomatic for tracking state across a loop when you need to report after the loop finishes rather than exiting immediately.

The `systemctl is-active --quiet` command exits 0 if the service is active and non-zero otherwise — the `if` reads that exit code directly, no `[[ ]]` needed.

## ⚠️ Gotchas

### Unquoted Variable in `[ ]` Breaks on Spaces

You expect `[ $name == "Alice" ]` to work. It does — until `$name` contains a space.

```bash
name="Alice Smith"
if [ $name == "Alice Smith" ]; then
    echo "match"
fi
```

```text
bash: [: too many arguments
```

What the shell actually runs after expansion is `[ Alice Smith == Alice Smith ]`. The `[` command receives five arguments where it expected three. It fails with an error — or worse, in some forms it silently produces wrong results.

The underlying rule: `[` is a command. Every word you write inside it is an argument. Word splitting happens before the arguments are passed. If `$name` splits into two words, `[` gets an unexpected number of arguments.

The fix is either to use `[[`, which suppresses word splitting entirely:

```bash
name="Alice Smith"
if [[ $name == "Alice Smith" ]]; then
    echo "match"
fi
```

Or to double-quote the variable inside `[`:

```bash
if [ "$name" == "Alice Smith" ]; then
    echo "match"
fi
```

Use `[[` in bash scripts. Reserve `[` for scripts that must be POSIX portable.

---

### `[ -z ]` With No Argument Is Silently True

You expect `[ -z ]` with a missing argument to fail or produce an error. It produces the opposite of what you probably intended.

```bash
# Imagine VAR is unset or empty — this is a guard to check for it
VAR=""
if [ -z $VAR ]; then
    echo "VAR is empty"
fi
```

```text
VAR is empty
```

That looks correct. Now consider what happens when `$VAR` is unset entirely:

```bash
unset VAR
if [ -z $VAR ]; then
    echo "guarded"
fi
```

```text
guarded
```

Still works. But this is for the wrong reason — `$VAR` expands to nothing, so `[` sees only the single argument `-z`. In many implementations, a single non-empty argument to `[` is treated as a string truth test: `-z` is a non-empty string, so it evaluates to true. The `-z` flag is never even evaluated as a flag. You got lucky.

The correct form protects against this by quoting:

```bash
if [[ -z "$VAR" ]]; then
    echo "VAR is empty or unset"
fi
```

Inside `[[`, the empty string is passed as a single zero-length argument and `-z` is evaluated correctly. The behavior is unambiguous.

### `(( 0 ))` Returns Exit Code 1 — Breaking `set -e`

You write a loop counter using arithmetic and wonder why your script exits unexpectedly.

```bash
#!/bin/bash
set -e

count=0
(( count++ ))   # increments count to 1 — but also exits with code 1 because pre-increment value is 0
echo "count is $count"
```

```text
(script exits with no output)
```

The script never reaches `echo`. Here's why: `count++` is the post-increment operator. It evaluates to the value of `count` *before* the increment — which is 0. In arithmetic context, 0 is false. So `(( count++ ))` evaluates to 0, which means the exit code is 1 (non-zero). Under `set -e`, any command that exits non-zero causes the script to abort. The increment happened — `count` is now 1 — but the script is already dead.

This is not a bug you can predict without understanding the C convention inside `(( ))`. The fix is either to avoid using post-increment as a standalone command when starting from zero, or to suppress the exit-code check:

```bash
# Option 1: pre-increment (evaluates to 1 after the increment, so exit code 0)
(( ++count ))

# Option 2: suppress the non-zero exit code explicitly
(( count++ )) || true

# Option 3: use assignment form, which always exits 0
count=$(( count + 1 ))
```

The `|| true` form is idiomatic when you genuinely want to increment and don't care about the arithmetic result. The assignment form `$(( ))` is safe with `set -e` because it's a substitution, not a command — it cannot fail in the exit-code sense.

### `for f in $(ls)` Mangles Filenames with Spaces and Globs

You want to loop over files in a directory. You reach for `$(ls)` because it looks natural.

```bash
for f in $(ls /tmp/my-files); do
    echo "Processing: $f"
done
```

If any filename contains a space, the word splitting that happens on the output of `$(ls)` splits that filename into multiple words. If any filename contains a glob character like `*` or `?`, it may get expanded against the current directory. Both are silent corruption — the loop iterates over the wrong values, and no error is produced.

The correct idiom is to let the shell expand the glob directly:

```bash
for f in /tmp/my-files/*; do
    echo "Processing: $f"
done
```

The shell performs this glob expansion after word splitting, producing one word per path — regardless of spaces in the names. If you need only certain file types, use a more specific glob: `/tmp/my-files/*.log`. If the directory might be empty and you don't want the literal `*` passed when there are no matches, use `shopt -s nullglob` before the loop.

---

### `=~` Regex Pattern Must Not Be Quoted

You write a regex match in `[[` and quote the pattern, expecting it to work as a regex.

```bash
email="user@example.com"
if [[ "$email" =~ "^[a-z]+@[a-z]+\.[a-z]+" ]]; then
    echo "valid email"
fi
```

```text
(no output — the test fails even for a valid email)
```

When the right-hand side of `=~` is quoted, bash treats it as a literal string, not a regex pattern. The `^`, `[`, `.`, and `+` characters lose their regex meaning and are matched literally. The fix: do not quote the regex pattern.

```bash
email="user@example.com"
pattern='^[a-z]+@[a-z]+\.[a-z]+'
if [[ "$email" =~ $pattern ]]; then
    echo "valid email"
fi
```

The best practice is to store the pattern in a variable and use the variable unquoted. This avoids ambiguity, survives complex patterns with backslashes, and makes the intent readable. The captured groups are available in `${BASH_REMATCH[@]}` after a successful match.

---

### `case` Fall-Through: `;&` vs `;;&` Are Different Things

Both `;&` and `;;&` cause execution to continue past a matched clause, but they do different things. Confusing them produces silent logic errors.

```bash
val="b"
case "$val" in
    a)
        echo "matched a"
        ;&          # fall through unconditionally to next clause
        ;;
    b)
        echo "matched b"
        ;;&         # test the next pattern
        ;;
    b|c)
        echo "matched b or c"
        ;;
    *)
        echo "catch-all"
        ;;
esac
```

```text
matched b
matched b or c
```

`;&` runs the next clause's command list without testing its pattern. If the `a` clause used `;&`, `b`'s commands would run even when `val` is `a`.

`;;&` tests the next pattern. `b` matches `b|c`, so that clause runs. If the next pattern had been `x|y`, it would not match and the scan would continue further.

The distinction: `;&` is "execute the next block", `;;&` is "keep testing patterns". Both were added in bash 4.0 and are absent in POSIX `sh`. Use them deliberately — their absence from older shells makes scripts non-portable.

## Distro Notes

| Distro Family | Behavior |
|---|---|
| Debian / Ubuntu | Default `sh` is `dash`, not bash. Scripts with `#!/bin/sh` get no `[[`, no `(( ))`, no C-style for loop, no `;&` or `;;&`. Always use `#!/bin/bash` if you need bash features. |
| RHEL / Fedora / CentOS / Rocky | Default `sh` is bash in compatibility mode on most versions, but do not rely on that — use an explicit `#!/bin/bash` shebang. |
| Alpine (containers) | Default shell is `ash` (BusyBox). `[[` is not available. `[` and `test` work. If your container script breaks on Alpine, this is why. |
| macOS (reference) | Ships with bash 3.2 due to GPL licensing. `;&` and `;;&` are bash 4.0 features and are not available on the stock macOS shell. Install bash 5.x via Homebrew if needed. |

## Alternatives & Comparisons

| Construct | Portability | When to Reach For It |
|---|---|---|
| `[` / `test` | POSIX — works in `sh`, `dash`, `ash` | Scripts that must run anywhere; `#!/bin/sh` scripts |
| `[[` | Bash, ksh, zsh only | All bash scripts; preferred over `[` when portability isn't required |
| `(( ))` | Bash, ksh only | Arithmetic conditions and counters; be aware of the zero-is-false exit code |
| `case` with `;;` | POSIX | Pattern-based dispatch in any shell |
| `case` with `;&`/`;;&` | Bash 4.0+ only | Intentional fall-through; never in POSIX scripts |
| C-style `for (( ))` | Bash only | Numeric iteration with index; unavailable in `sh` or `dash` |

If you are writing a script for a system where bash is guaranteed (a server you control, a CI environment that pins bash), prefer `[[` over `[` in all cases. The safety improvements — no word splitting, proper handling of empty strings, regex support — are meaningful enough that there is no reason to use `[` in a bash script. The only exception is deliberate portability: if your shebang is `#!/bin/sh`, you cannot use `[[`.

## Trade-offs

**Where this shines:**

- `[[` removes an entire class of word-splitting bugs that have silently corrupted production scripts for decades — the cognitive overhead of remembering to quote disappears
- `case` pattern matching is more readable and faster than a long chain of `elif [[ $var == ... ]]` — the intent is immediately scannable
- `(( ))` makes arithmetic conditions look like arithmetic instead of string comparisons — `(( i < 10 ))` is clearer than `[ "$i" -lt 10 ]`
- `break N` and `continue N` make nested loop control explicit, where other languages would require a flag variable and extra logic

**Where this costs you:**

- `[[`, `(( ))`, and C-style `for` are bash-only — scripts written with these features silently break on `dash`, `ash`, `busybox sh`, and stock macOS `/bin/sh`
- The exit-code-as-boolean model means `set -e` interacts unpredictably with arithmetic — `(( 0 ))` aborts a `set -e` script; this is non-obvious and has caused real production failures
- `case` with `;&` and `;;&` fall-through is bash 4.0+ only and is unfamiliar to most readers; code that uses it requires a comment explaining the intent
- Deeply nested loops with `break 2` or `break 3` are hard to reason about — when the level count is wrong, the break exits the wrong loop and produces silent logic errors

**Performance note:** `[[` avoids the overhead of forking a child process that `[` (as an external command) would require in older shells. In modern bash, `[` is a builtin and the fork overhead is gone — but `[[` is still faster in tight loops because it skips the word-splitting pass entirely.

## When to Use / When to Avoid

✅ **Use `[[` when:**
- Writing any bash script where portability to POSIX `sh` is not required
- Comparing strings that might contain spaces — no quoting anxiety
- Using regex matching with `=~`
- Using `&&` or `||` inside a condition (more readable than `-a` / `-o`)

✅ **Use `[` / `test` when:**
- Writing `#!/bin/sh` scripts that must be POSIX-portable
- Writing scripts for Alpine containers or other minimal environments
- You explicitly want to signal to readers that the script is portable

✅ **Use `(( ))` when:**
- Performing arithmetic comparisons in loops or conditions
- Incrementing and decrementing counters
- Doing bitwise operations

❌ **Avoid `(( var++ ))` as a standalone command under `set -e` when:**
- The variable starts at 0 — the first post-increment exits with code 1 and aborts the script; use `(( ++var ))` or `var=$(( var + 1 ))` instead

❌ **Avoid `for f in $(ls ...)` when:**
- Iterating over files — always — use `for f in /path/*` or `while IFS= read -r f` with a `find` pipeline instead

❌ **Avoid `;&` and `;;&` in `case` when:**
- Writing scripts that may run on systems with bash older than 4.0 or any non-bash shell
- The fall-through logic is not clearly commented — it is a maintainability hazard

⚠️ **Common mistakes:**
- Writing `if [[ $? -eq 0 ]]` after a command — just use `if cmd` directly; checking `$?` is redundant and fragile
- Forgetting `do` and `done` around loop bodies — bash's error message for this is not always obvious
- Using `-eq`, `-lt`, `-gt` inside `[[ ]]` for numeric comparisons — this works, but `(( a < b ))` is clearer; inside `[[`, `<` and `>` are string comparisons, not numeric
- Writing `for i in {1..10}` and expecting it to work with variables like `{1..$n}` — brace expansion happens before variable expansion; `{1..$n}` does not expand as expected; use a C-style `for (( ))` loop instead

## Veteran Perspective

The single thing that will improve your bash scripts more than anything else is internalizing one rule: every condition is an exit code. Not a value. Not a boolean. An exit code. Once that settles in, the whole language makes sense. `if grep ...` is not magic — `grep` exits 0 on match, non-zero otherwise, and `if` reads that. `&&` and `||` are not boolean operators — they're short-circuit control flow on exit codes. `(( 0 ))` fails because zero maps to exit code 1. Every apparent quirk collapses into that one rule.

The `[[ ]]` vs `[ ]` distinction has tripped up more production scripts than I can count. The pattern is always the same: someone writes a script, tests it with filenames that happen not to have spaces, deploys to production, and the first filename with a space in it causes a mysterious failure. The root cause is always unquoted variable expansion inside `[ ]`. The permanent fix is `[[` for any script that doesn't need to be POSIX-portable. If you see `[ ]` in a bash script with unquoted variables, that's a bug waiting to happen.

The `(( count++ ))` trap under `set -e` is in a special category because it is entirely non-obvious and the failure mode is silent: the script exits cleanly with code 1 and nothing in the logs explains why. I have seen this burn a senior engineer for two hours. The fix is mechanical — use `(( ++count ))` or `var=$(( var + 1 ))` — but you have to know to look for it. The general lesson is that `set -e` is far less safe than it appears, and arithmetic is one of its most dangerous interactions.

The `for f in $(ls)` antipattern deserves its own paragraph. It is in every tutorial from 2005, it is in countless Stack Overflow answers, and it is wrong. Not "wrong in edge cases" — wrong as a matter of correctness. Any filename with a space, a newline, a glob character, or certain leading characters will be processed incorrectly. The correct form, `for f in /path/*`, is not harder to type. It is never acceptable to use `$(ls)` in a loop over files. If you see it in a code review, block it.

`select` is worth knowing exists. You will use it once every few years when you need a quick interactive menu in a maintenance script. Don't reach for it in production-critical paths — it is not meant for that.

On the question of `&&`/`||` chains vs. `if` statements: chains are fine for simple two-command guards — `[[ -f "$file" ]] || { echo "missing"; exit 1; }`. When the logic becomes a three-step conditional with different error messages at each step, switch to an `if` block. The readability cliff happens faster than people expect, and nobody wants to debug a five-operator `&&`/`||` chain at 3am.

One last point on `case` with `;;&`: it was added to bash 4.0 in 2009, and in fifteen years I have seen it used correctly perhaps twice. Most uses of `;;&` I've encountered were written by someone who read about it and wanted to use it but actually wanted `;&`, or wanted plain `;;` with a combined pattern. Think carefully before reaching for it. If you can rewrite the logic as `pattern1|pattern2)`, do that instead.

## Try It Yourself

1. Write a two-line script that checks whether `/etc/passwd` exists and prints either "found" or "not found". Use `[[ ]]`.

2. Write a `for` loop that prints the numbers 1 through 5 using the C-style `for (( ))` syntax. Then rewrite it using `while (( ))` instead.

3. Write a `case` statement that takes a single argument (`$1`) and prints "low" for values `1` or `2`, "medium" for `3` or `4`, and "high" for anything else. Test it by calling the script with different arguments.

4. Write a script that uses `for f in /etc/*.conf` to iterate over config files in `/etc`, prints the name of each file, and uses `break` to stop after the first five files. Observe what happens if you change `break` to `continue` — which makes no difference here and why?

5. **Challenge:** Write a script that accepts a directory as `$1`, loops over every file in that directory using a glob (not `$(ls)`), and prints the filename along with its size in bytes (hint: `stat -c %s`). Guard against the case where the argument is not a directory or is not provided. Use `set -euo pipefail` at the top. Test it on a directory that contains a file with a space in its name to verify it handles that correctly.

## Summary

- Every `if` condition in bash is a command exit code — 0 is true, non-zero is false; this is the single rule that governs all control flow
- `[[` is a bash keyword that suppresses word splitting and filename expansion; prefer it over `[` in all bash scripts unless you need POSIX portability
- `(( expr ))` returns exit code 1 when the arithmetic result is 0 — this interacts dangerously with `set -e`; never use post-increment on a zero-initialized variable as a standalone `(( ))` command under `set -e`
- Never use `for f in $(ls)` — glob expansion directly in the `for` header (`for f in /path/*`) is always correct and never mangles filenames
- `=~` regex in `[[` requires the pattern to be unquoted (or stored in a variable); quoting the pattern makes it a literal string match
