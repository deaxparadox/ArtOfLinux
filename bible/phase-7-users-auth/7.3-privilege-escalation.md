## The Problem This Solves

Early Unix had two privilege levels: root and everyone else. If a sysadmin needed to let a junior operator restart a web server, the only option was to give them the root password. If they needed to let a developer check system logs, same answer: root password. If you needed ten different people to do ten different administrative tasks, you had exactly one credential to hand out — and once someone had it, they could do anything.

This caused real pain in three directions. First, accountability: when something broke, you couldn't tell which operator did it because everyone shared the same root identity. Second, blast radius: an operator making an honest mistake while restarting nginx could accidentally `rm -rf /etc` with the same credentials. Third, hygiene: people share passwords. Root passwords especially. An ex-employee, a compromised workstation, a shoulder-surf — and now someone outside your organization has full control of your system.

The principle behind the fix is called **least privilege**: every person and process should have exactly the access they need to do their job, and nothing more. Give the operator the ability to restart nginx. Not to read `/etc/shadow`. Not to delete system files. Just restart nginx.

That's what `sudo` solves. And `sudo`'s power entirely lives inside one file: `/etc/sudoers`.

## Mental Model

Think of a hospital where the pharmacy is locked. Nurses need access to specific medications, but not every nurse needs every drug, and no nurse should be able to walk into the pharmacy and take anything. The pharmacy maintains a policy binder: "Nurse Ana may dispense ibuprofen and acetaminophen without authorization. Nurse Ben may dispense anything on ward 3. Dr. Chen may dispense anything, but must log it."

When a nurse needs something, they go to the pharmacy window, state who they are and what they need, the pharmacist checks the binder, and dispenses exactly the authorized item — no more. Every transaction is logged. The nurse never gets a key to the pharmacy itself.

`sudo` is the pharmacy window. `/etc/sudoers` is the binder. `visudo` is the only safe way to edit the binder, because a misprinted page can accidentally grant Dr. Chen access to nothing — and he'll need medication urgently at 3am.

## Theory

### The Two Tools: `su` and `sudo`

Linux has had two mechanisms for privilege escalation since the early days. They solve different problems and come from different design philosophies.

`su` — *substitute user* — is the older mechanism. It authenticates you as the target user directly. To become root with `su`, you type `su -` and supply **root's password**. To become any other user, you type `su - username` and supply **that user's password**. Once authenticated, you are running as that user for the duration of the session. When you're done, you `exit`.

The critical distinction is `su` versus `su -`. Plain `su` switches your user identity but inherits your current environment — your `$PATH`, your `$HOME`, your shell variables, all of it. `su -` simulates a fresh login as the target user, loading that user's full login environment: their `$HOME`, their shell configuration, their `$PATH`. In practice, `su -` is almost always what you want, because running root commands with a non-root `$PATH` leads to confusing failures.

`sudo` — *superuser do* — is the modern mechanism. It authenticates you as **yourself**, then checks a policy file to decide whether you're allowed to run the requested command. You never supply root's password. Root can have no password at all on many modern systems, because nobody ever logs in as root directly. All privilege escalation goes through the policy.

The design difference matters: `su` requires sharing the target user's password. `sudo` never shares any password. This is why `sudo` dominates modern systems and many distributions disable direct root login entirely.

### What `/etc/sudoers` Actually Is

`/etc/sudoers` is a policy file. It lists, for every user or group on the system, what commands they may run, on which hosts, as which user, and whether they need to authenticate first.

The file also controls security properties of `sudo` itself: whether the authentication token caches (so you don't re-type your password within a few minutes), what environment variables are preserved when escalating, and whether `sudo` logs to syslog.

`/etc/sudoers.d/` is a drop-in directory. Any file placed here gets parsed as part of sudoers policy. The convention is to use `visudo -f /etc/sudoers.d/myfile` to create per-application or per-team policy files. This lets you manage sudo grants in discrete pieces — the nginx team's access in one file, the database team's in another — rather than a single monolithic config.

### Linux Capabilities: A Finer Scalpel

There is a third mechanism that most people encounter late: Linux capabilities. Rather than binary root-vs-nonroot, Linux has broken the root privilege into about 40 discrete capabilities that can be granted independently. A process can have `cap_net_bind_service` (the ability to bind ports below 1024) without having `cap_sys_admin` (the ability to mount filesystems or manipulate namespaces). These capabilities can be assigned to specific binaries using `setcap`, allowing them to run as non-root users while retaining exactly the one privilege they need.

Capabilities are the most precise least-privilege mechanism Linux has. They are also the most frequently misunderstood.

### PolicyKit and `pkexec`

`pkexec` is PolicyKit's command-line tool for running commands as another user — conceptually similar to `sudo`, but designed for graphical desktop environments where a GUI authentication dialog pops up instead of a password prompt. It reads policy files in `/usr/share/polkit-1/actions/` and `/etc/polkit-1/rules.d/`.

PolicyKit matters in modern Linux desktops and in container environments, but it has a darker recent history: CVE-2021-4034 (PwnKit) — a memory corruption vulnerability that existed in `pkexec` for over a decade, present in virtually every major distribution since 2009, that allowed any local unprivileged user to escalate to root. The connection to capabilities is direct: the exploit worked by manipulating `pkexec`'s argument handling to force execution of attacker-controlled environment variables as commands. The patch that fixed it also changed how the binary handles its capabilities.

## How It Works (Mechanics)

### The `sudo` Execution Flow

When you type `sudo nginx -t`, the following happens:

```text
# Conceptual execution trace — not a real command
1. sudo starts (owned root, setuid bit set — same mechanism as passwd)
2. sudo reads /etc/sudoers and all files in /etc/sudoers.d/
3. sudo looks up the calling user in the policy
4. If no matching rule: denied, logged, exit
5. If matching rule found: check NOPASSWD tag
6. If no NOPASSWD: prompt for the calling user's own password
7. Authenticate against PAM
8. If authenticated: fork, set effective UID to the runas user (default: root)
9. exec() the requested command
10. Log the invocation to syslog (auth.info by default)
```

Every invocation — whether it succeeds or fails — is logged. This is not optional and cannot be disabled without modifying the sudo build. The log includes the username, the terminal, the working directory, and the full command.

### The `visudo` Safety Mechanism

`/etc/sudoers` is not a daemon that hot-reloads. It is read fresh on every `sudo` invocation. This means a syntax error in the file will block **all** `sudo` access immediately and permanently until the file is corrected — and if you can't use `sudo` to fix it, you're looking at either a root login (if one is available) or recovery mode.

`visudo` is a wrapper around your editor that adds one critical step: after you save, it parses the file for syntax errors before replacing the live copy. If the parse fails, it refuses to save and shows you the line number. The live file is never touched if the new version is invalid.

`visudo` picks its editor from the environment variables `SUDO_EDITOR`, then `VISUAL`, then `EDITOR`, in that order. If none are set, it falls back to a compile-time default — `/usr/bin/editor` on Debian/Ubuntu (which is an alternatives-managed symlink), and `vi` on RHEL/Fedora and most other systems.

### Capabilities in the Kernel

Every process has five capability sets: Permitted, Effective, Inheritable, Bounding, and Ambient. The important ones are:

- **Effective**: what the kernel checks for privilege decisions right now
- **Permitted**: the ceiling — what a process is allowed to have in its Effective set
- **Bounding**: the hard ceiling — no exec can raise capabilities beyond this set

When a binary with file capabilities is executed, the kernel computes the new capability sets for the resulting process using the binary's file capabilities and the calling process's sets. This is separate from, and in some ways more powerful than, setuid.

The key constraint: capabilities granted via `setcap` on a binary interact poorly with the setuid bit. The kernel's rules for capability propagation through `exec()` make setuid-root binaries and file capabilities deeply incompatible. If you set capabilities on a binary and someone also makes it setuid-root, the capability sets behave differently than you'd expect. The right answer is: use `setcap` on non-setuid binaries. Pick one mechanism.

## Basic Example

```bash
# Check who you are, then elevate and check again
whoami
```

```text
alice
```

```bash
sudo whoami
```

```text
root
```

```bash
# List what sudo rules apply to the current user
sudo -l
```

```text
Matching Defaults entries for alice on myserver:
    env_reset, mail_badpass, secure_path=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

User alice may run the following commands on myserver:
    (ALL : ALL) ALL
```

```bash
# Switch to a different user entirely (su requires that user's password)
su - bob
```

```bash
# Switch to root with a full login environment
su -
```

The `sudo -l` output is your diagnostic starting point on any unfamiliar system. The `(ALL : ALL) ALL` line means alice can run any command as any user — effectively unrestricted. A more restricted output would look like `(root) /usr/bin/systemctl restart nginx`.

Try this on your own system. If `sudo -l` returns nothing, your user has no sudo access. If it returns a list, you're seeing the exact rules that govern your escalation privileges.

## Real-World Example

### Configuring sudo for a Web Operations Team

Scenario: you have a team of three web operators who need to restart web services and read logs, but must not be able to install packages, modify system files, or drop to a root shell.

> ⚠️ **WARNING:** The following uses `visudo` — never edit `/etc/sudoers` directly with any other editor. A syntax error in sudoers prevents all future sudo access. Always use `visudo`, which validates syntax before saving.

```bash
# Always use visudo — never vi /etc/sudoers, nano /etc/sudoers, etc.
sudo visudo -f /etc/sudoers.d/webops
```

This opens a new drop-in file in your configured editor. Write the following policy:

```text
# WebOps team sudo policy
# Managed by: sysadmin@company.com — last updated 2024-01-15

# Define the group of web operators
User_Alias WEBOPS = alice, bob, carol

# Define the commands they're allowed to run
Cmnd_Alias WEB_SERVICES = /usr/bin/systemctl restart nginx, \
                          /usr/bin/systemctl reload nginx, \
                          /usr/bin/systemctl status nginx, \
                          /usr/bin/systemctl restart php8.2-fpm, \
                          /usr/bin/systemctl reload php8.2-fpm

Cmnd_Alias WEB_LOGS = /usr/bin/tail -f /var/log/nginx/access.log, \
                      /usr/bin/tail -f /var/log/nginx/error.log, \
                      /usr/bin/journalctl -u nginx

# Grant the webops team access to web commands without a password
# (they are already authenticated via their own login)
WEBOPS ALL = (root) NOPASSWD: WEB_SERVICES, WEB_LOGS
```

Now verify the syntax parses cleanly:

```bash
sudo visudo -c -f /etc/sudoers.d/webops
```

```text
/etc/sudoers.d/webops: parsed OK
```

Test the grant as alice:

```bash
sudo -u alice sudo -l
```

```text
User alice may run the following commands on this host:
    (root) NOPASSWD: /usr/bin/systemctl restart nginx, ...
```

A few things worth noticing about this configuration:

The `Cmnd_Alias` lists use full absolute paths. Sudo always requires absolute paths in command lists — relative paths are rejected. This also prevents an attacker from substituting a malicious binary named `nginx` earlier in `$PATH`.

The `NOPASSWD` tag is used here because these operators are running automated deployment scripts that can't interactively supply a password. In interactive contexts, omitting `NOPASSWD` is safer — the password prompt is a confirmation step that prevents accidents.

The commands in `WEB_LOGS` include specific flags. `sudo` matches commands including their arguments by default. If you write `/usr/bin/tail` without arguments, the operator can run `tail` with any arguments — including `tail /etc/shadow`. By specifying `/usr/bin/tail -f /var/log/nginx/access.log`, you constrain them to exactly that invocation.

### Setting File Capabilities for a Web Server

Nginx needs to bind port 80 and 443. The traditional approach runs the master process as root. The capability approach grants only the specific privilege needed:

```bash
# Grant nginx the ability to bind privileged ports, nothing else
sudo setcap 'cap_net_bind_service=+ep' /usr/sbin/nginx
```

```bash
# Verify the capability was set correctly
getcap /usr/sbin/nginx
```

```text
/usr/sbin/nginx cap_net_bind_service=ep
```

The string `cap_net_bind_service=+ep` sets the capability in both the Effective (`e`) and Permitted (`p`) sets. This means when nginx starts, it immediately has the privilege active in its effective set — it does not need to raise it manually.

```bash
# Now nginx can start as a non-root user and still bind port 80
# (assuming nginx.conf has worker_processes running as www-data)
sudo -u www-data nginx -c /etc/nginx/nginx.conf
```

To remove capabilities if you no longer need them:

```bash
sudo setcap -r /usr/sbin/nginx
```

```bash
getcap /usr/sbin/nginx
```

```text
# No output — capabilities removed
```

## ⚠️ Gotchas

### Editing `/etc/sudoers` Directly Locks You Out Permanently

This is the highest-consequence gotcha in day-to-day Linux administration. The mistake looks like this:

> ⚠️ **WARNING:** Do not run the following command. It is shown here as an example of exactly what you must never do. A syntax error in sudoers removes all sudo access from the system until you can log in as root or boot into recovery mode.

```bash
# WRONG — never do this
nano /etc/sudoers
vi /etc/sudoers
```

You make a change, save the file, log out, and now `sudo` fails with a parse error. Your user account can no longer escalate. If root login is disabled (as it is on Ubuntu by default), you cannot `su -`. Your only recovery paths are:

1. If you have another root session open in another terminal: use it to run `visudo` and fix the file
2. If the system is a VM: boot into recovery mode, mount the root filesystem read-write, and fix it manually
3. If it's a cloud instance: use the provider's out-of-band console

The correct approach is always `visudo`:

```bash
# CORRECT — always use visudo
sudo visudo
```

`visudo` will not save a file with syntax errors. That's its entire reason for existing.

### `sudo -i` vs `sudo -s` — They Are Not the Same Shell

Both drop you to a root shell, but they differ in a way that causes real bugs in scripts and deployment tools.

```bash
sudo -s
echo $HOME
```

```text
/home/alice    # ← still alice's home directory
```

```bash
sudo -i
echo $HOME
```

```text
/root          # ← root's home directory
```

`sudo -s` runs a shell as root but inherits your current environment. `$HOME`, `$PATH`, and your shell configuration stay as your user. This means tools that look for configuration in `$HOME` (like `~/.npmrc`, `~/.aws/credentials`, `~/.gitconfig`) will find *your* files, not root's.

`sudo -i` simulates a full root login. It resets the environment to root's, reads `/etc/profile` and root's `~/.bashrc`, and sets `$HOME=/root`. It is effectively equivalent to `su -` but using your own password and going through the sudo policy.

The rule: if you need to reliably run commands exactly as root would in a clean login environment, use `sudo -i`. If you need to stay in your current directory and environment while just running a command as root, use `sudo -s`. For scripts and automation, spell out what you need — never assume.

### `NOPASSWD: ALL` on a User Account Is Effectively No Security

```text
# This looks convenient. It is dangerous.
alice ALL=(ALL) NOPASSWD: ALL
```

Anyone who gets shell access as `alice` — through a stolen SSH key, a web application exploit, a compromised laptop — immediately has full root access with no additional authentication. There is no speed bump. No alert. No second factor. They just run `sudo bash` and they own the machine.

`NOPASSWD: ALL` is appropriate for service accounts running automated processes in controlled environments. It is not appropriate for human operator accounts. If someone argues for it on the basis of convenience, the answer is: configure SSH agent forwarding and a short sudo timeout instead.

### Capabilities Set with `setcap` Are Lost on Setuid Binaries

```bash
# This combination produces undefined behavior
sudo setcap 'cap_net_bind_service=+ep' /usr/sbin/myserver
sudo chmod u+s /usr/sbin/myserver   # DO NOT do this
```

The Linux kernel's capability propagation rules treat setuid-root binaries and file capabilities differently. When a binary is setuid-root, the kernel uses a different path to determine what capabilities the resulting process gets — the file capability sets on a setuid-root binary are ignored. The two mechanisms are not additive; they conflict.

The correct pattern: use `setcap` on a binary that is **not** setuid. The entire point of `setcap` is to replace setuid for scenarios where you need one specific privilege. If you find yourself wanting both, you've likely misunderstood what the binary actually needs.

### `Cmnd_Alias` Cannot Carry `NOPASSWD`

This is a common syntax error that `visudo` will catch, but the error message is not always obvious.

```text
# WRONG — NOPASSWD cannot appear inside a Cmnd_Alias
Cmnd_Alias WEBOPS = NOPASSWD: /usr/bin/systemctl restart nginx
```

```text
# CORRECT — NOPASSWD is a tag on the user specification, not the alias
Cmnd_Alias WEBOPS = /usr/bin/systemctl restart nginx
alice ALL = (root) NOPASSWD: WEBOPS
```

`NOPASSWD` is a *tag* that modifies how the grant is applied. It belongs in the user specification line, to the left of the command list. Tags cannot appear inside `Cmnd_Alias`, `User_Alias`, or any other alias definition.

## Distro Notes

| Distro Family | Behavior / Notes |
|---|---|
| Debian / Ubuntu | `visudo` defaults to `/usr/bin/editor` (the alternatives system — typically nano). Root login disabled by default; first user gets unrestricted sudo via `(ALL:ALL) ALL`. Drop-ins in `/etc/sudoers.d/`. |
| RHEL / Fedora / CentOS / Rocky | `visudo` defaults to `vi`. The `wheel` group is granted sudo access; add users to `wheel` with `usermod -aG wheel username`. Drop-ins in `/etc/sudoers.d/`. |
| Arch | `visudo` defaults to `vi`. The `wheel` group is present but not enabled by default — uncomment the `%wheel` line in sudoers. |
| Alpine (containers) | `sudo` may not be installed. Install with `apk add sudo`. Often `doas` is preferred in Alpine environments. |

**Changing `visudo`'s editor temporarily:**

```bash
# Use nano instead of vi for this one visudo invocation
EDITOR=nano sudo visudo
```

**Or set it permanently per-user in your shell configuration:**

```bash
# In ~/.bashrc or ~/.zshrc
export VISUAL=nano
```

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `sudo` | Day-to-day privilege escalation on any server or workstation | The standard. Every sysadmin must know it. |
| `su` | When you need a full interactive session as another user, or on systems without `sudo` | Requires knowing the target user's password. Older mechanism. |
| `doas` | Minimal sudo replacement; common in BSDs and some Alpine environments | Simpler config format, fewer features, smaller attack surface |
| `pkexec` / PolicyKit | GUI desktop privilege escalation; service-to-service authorization on desktops | Not for servers. After CVE-2021-4034, audit carefully before relying on it. |
| File capabilities (`setcap`) | Granting a specific kernel privilege to a single binary without setuid | Most precise mechanism; pairs poorly with setuid; not inherited across forks by default |

`sudo` is the correct default for any multi-user server. `doas` is worth considering if you want a simpler configuration model and a smaller binary with fewer historical CVEs. `su` still exists and is still useful, but in any environment where you can use `sudo`, prefer it — the audit trail alone justifies it.

## Trade-offs

**Where this shines:**

- `sudo` provides per-command, per-user, per-host access control that is impossible to express with traditional Unix permissions alone
- Every invocation is logged — you have an audit trail that `su` cannot give you
- `NOPASSWD` grants let automated pipelines escalate without interactive input, without distributing root passwords
- File capabilities allow fine-grained privilege assignment to specific binaries without running processes as root at all
- Drop-in files in `/etc/sudoers.d/` make privilege management composable and machine-readable

**Where this costs you:**

- A misconfigured sudoers file can grant unintended root access silently — there is no runtime warning when a policy is too broad
- `NOPASSWD` removes the authentication speed bump — any attacker who gains shell access as that user has root without any additional friction
- File capabilities on a binary survive on disk even if the binary is updated — `setcap` metadata persists through many package upgrades, and you must audit for stale capability grants
- The sudoers syntax is terse and unforgiving; a misplaced comma or missing colon causes a parse failure that locks you out
- Capabilities do not survive `exec()` in all scenarios — the propagation rules are subtle and require reading `capabilities(7)` carefully before relying on them in production

## When to Use / When to Avoid

✅ **Use `sudo` when:**
- Any human user needs to run specific administrative commands without becoming root permanently
- You need an audit trail of who ran what and when
- You want to restrict a service account to exactly the commands it needs
- You're onboarding a new team member and want to grant them graduated access

✅ **Use file capabilities (`setcap`) when:**
- A long-running service needs exactly one kernel privilege (e.g., binding a low port) and nothing else
- You want to run that service as a non-root user with no sudo access at all
- You're hardening a specific binary and want to remove its setuid root status

❌ **Avoid `NOPASSWD: ALL` on human accounts:**
- It converts any account compromise into an immediate full root compromise — use scoped `NOPASSWD` with explicit command lists instead

❌ **Avoid `su` on modern multi-user servers:**
- It requires sharing passwords, produces no per-user audit trail, and offers no command-level restrictions

❌ **Avoid editing `/etc/sudoers` directly:**
- A single syntax error locks out all sudo access — always use `visudo`

❌ **Avoid `setcap` on setuid binaries:**
- The capability propagation rules change in unexpected ways — pick one mechanism and use it exclusively

⚠️ **Common mistakes:**
- Granting `sudo ALL` to a user "temporarily" and forgetting to remove it
- Using relative paths in `Cmnd_Alias` (sudo rejects them — always use absolute paths)
- Assuming capabilities persist after a binary is replaced by a package upgrade — verify with `getcap` after updates
- Running `pkexec` in production server environments where it has no business being

## Veteran Perspective

The sudoers file is one of those configurations that accumulates technical debt faster than almost anything else on a server. Teams start with tight restrictions, someone needs to do something urgently at 2am, they add `NOPASSWD: ALL` as a temporary fix, and three years later it's still there. I've audited systems where the "junior developer" account had unrestricted sudo because somebody needed to restart a process once in 2019. The original developer left the company in 2020. The account was still active.

Run `sudo -l -U <username>` on every user account during security audits. Run `grep -r NOPASSWD /etc/sudoers /etc/sudoers.d/` and justify every single entry. Anything you can't justify in one sentence should be removed.

The intermediate trap is believing that `Cmnd_Alias` restrictions are airtight. They're not, if you're not careful about arguments. Granting `/usr/bin/vim` seems harmless until you remember that vim has a shell escape (`:!bash`). Granting `/usr/bin/find` seems fine until you realize `find -exec` can run arbitrary commands. There's a well-maintained list of sudo escapes at GTFOBins. Before you grant any command in sudoers, check whether it appears there.

Linux capabilities are the right tool more often than people reach for them. Every time you see a service running as root because it binds port 443, ask whether `setcap 'cap_net_bind_service=+ep'` and a dedicated non-root user account would work instead. It usually would. The result is a process that can't read `/etc/shadow`, can't write to system directories, and can't spawn setuid children — it can only do the one thing it was given permission to do.

CVE-2021-4034 is worth understanding not just as a historical CVE, but as a case study in what happens when privilege escalation code is complex, rarely audited, and present by default everywhere. `pkexec` had been shipping with a memory corruption vulnerability since 2009. Qualys found it in 2022. It allowed any local user on any major Linux distribution to become root in seconds. The complexity of the PolicyKit architecture — policy files, D-Bus, helper binaries — was itself the attack surface. The lesson: every privilege escalation mechanism you run is attack surface. Run only what you need. Audit what you run.

The thing that doesn't get said enough: `sudo -i` and `sudo su -` are both ways to become root interactively, but they don't mean the same thing in audit logs. With `sudo -i`, the log shows `alice : TTY=pts/1 ; PWD=/home/alice ; USER=root ; COMMAND=/bin/bash`. With `sudo su -`, the log shows alice ran `su`, but the subsequent root session is not logged through sudo — it's a separate root login. If your organization cares about audit trails, `sudo -i` is the correct way to get an interactive root shell, because every command in that session is still traceable to alice's original sudo invocation in the system logs.

## Try It Yourself

1. Run `sudo -l` on your current user. Read every line. Find the `Defaults` entries and look up what two of them mean in `man sudoers`.

2. Open a second terminal. In terminal 1, run `sudo visudo` and intentionally introduce a syntax error (add a random character on a line). Try to save. Observe what `visudo` does. Then fix the error and save correctly.

3. Create a drop-in file with `sudo visudo -f /etc/sudoers.d/test-policy`. Grant your user `NOPASSWD` access to run exactly one command (try `/usr/bin/id`). Test it, then remove the file.

4. Check whether any binaries on your system have file capabilities set: `sudo getcap -r / 2>/dev/null`. You'll likely find `ping`, `python`, or similar. Research why each one has the capability it has.

5. **Challenge:** Create a new user with `sudo useradd -m testoperator`. Write a sudoers drop-in that grants this user the ability to restart exactly one service of your choice, without a password, but prevents them from accessing any shell through that command. Test it with `sudo -u testoperator sudo -l`. Then verify you cannot escalate further by checking whether the service binary appears in GTFOBins. Clean up by removing the drop-in and deleting the user. Run this in a VM.

## Summary

- `sudo` is the right default for privilege escalation on modern Linux — it authenticates you as yourself, applies a policy, and logs every invocation; `su` requires sharing passwords and produces no per-command audit trail
- **Never edit `/etc/sudoers` directly** — always use `visudo`, which validates syntax before saving; a syntax error in sudoers removes all sudo access until you can recover via root or recovery mode
- Sudoers rule format is `who host = (runas) commands`; `NOPASSWD` is a tag on the user specification line, not inside a `Cmnd_Alias`; all paths must be absolute
- Linux file capabilities (`setcap`/`getcap`) allow granting a single kernel privilege to a specific binary without setuid or root ownership — the right tool when a service needs exactly one privilege; do not combine with setuid
- CVE-2021-4034 (PwnKit) demonstrated that `pkexec` had a root escalation bug present since 2009 — every privilege escalation mechanism is attack surface; run only what your environment requires
