## The Problem This Solves

Before SSH, remote access to a Unix machine meant `telnet` or `rlogin`. Both protocols sent everything over the network in plain text — your username, your password, every command you typed, every line of output you received. Anyone with access to the network between you and the server could read the entire session with a packet sniffer. This wasn't a theoretical risk. In the mid-1990s, packet sniffers were common tools on university networks, and password harvesting from telnet sessions was routine.

The problem wasn't just authentication. Once you were logged in, every command you typed — `su root`, the root password, the commands you ran — traveled across the wire in the clear. A network tap anywhere along the route meant full compromise.

`rsh` (remote shell) was worse: it trusted the source IP address of the client. If you were connecting from a "trusted" host, you didn't need a password at all. Spoofing an IP address is not hard.

Tatu Ylönen designed SSH in 1995 after a password-sniffing attack on his university network. The design principle was simple: encrypt everything, authenticate the server cryptographically so you know you're talking to the right machine, and authenticate the user with a key pair instead of a password that travels over the wire.

SSH replaced telnet, rlogin, and rsh entirely on any system administered by people who understood what those protocols actually did. Today, SSH is how you manage every Linux server in existence. Understanding it thoroughly is not optional.

## Mental Model

Think of SSH like a sealed diplomatic pouch between two embassies. Before any messages travel, both embassies verify each other's official seal (host key verification). Once identity is confirmed, they establish a private cipher that only they share — even if a spy intercepts every pouch from this point forward, they can't read what's inside. Your key pair is the proof of identity you show before the embassy accepts your messages. The passphrase on your key is the combination to the safe where you keep that proof.

The crucial detail: the embassy's seal (host key) is checked *before* you present your own identity. If someone swaps out the embassy's building overnight and puts up a fake seal, you should refuse to enter — no matter how authentic-looking the interior is. That's what `known_hosts` enforces.

## Theory

### What SSH Is

SSH (Secure Shell) is a cryptographic network protocol for operating network services securely over an unsecured network. Its primary use is providing a secure command-line shell to a remote machine, but the protocol is general enough to tunnel arbitrary TCP connections, forward X11 display sessions, transfer files (via `scp` and `sftp`), and mount remote filesystems (via `sshfs`).

The protocol operates in two layers. First, a transport layer negotiates encryption and authenticates the server to the client. Second, an authentication layer authenticates the client to the server. After both layers complete, a connection layer multiplexes the encrypted tunnel into logical channels — your interactive shell is one channel, a port forward is another.

### Key Types — Why They Matter

SSH supports several key types for public-key authentication. The choice matters because they represent different cryptographic assumptions.

**DSA** was the original SSH key type. It is inherently weak — limited to 1024-bit keys, which are computationally breakable today. OpenSSH removed DSA support entirely in 2025. If you have DSA keys anywhere, replace them now.

**RSA** is the workhorse that ran the internet for decades. With a 4096-bit key it is still considered secure, but it is slow to generate, produces large key files, and the security margin depends entirely on the difficulty of factoring large integers — a problem that quantum computers will eventually solve. The minimum safe RSA key size in 2025 is 3072 bits; 4096 is the conservative choice. RSA is still widely supported for compatibility with legacy systems.

**ECDSA** (Elliptic Curve Digital Signature Algorithm) is faster and produces smaller keys than RSA for equivalent security. The catch: the NIST P-256 curve it typically uses has a controversial history with the NSA, and ECDSA is vulnerable to PRNG failures — if the random number generator is weak at signing time, the private key can be recovered from signatures. This has happened in practice.

**Ed25519** uses the Edwards-curve digital signature algorithm on Curve25519 — a curve designed entirely in the open with no involvement from NIST or the NSA, with explicit security proofs. Ed25519 is faster than ECDSA, produces smaller keys, is immune to PRNG failures (it uses deterministic signing), and is resistant to timing side-channel attacks. It requires OpenSSH 6.5+ (released in 2014), which is installed everywhere that matters. **Ed25519 is the correct choice for new keys in 2025 and beyond.**

The only reason to use RSA-4096 in 2025 is compatibility with a legacy system that predates Ed25519 support. In all other cases, use Ed25519.

### Public-Key Authentication — The Core Idea

SSH public-key authentication works on asymmetric cryptography. You generate a key pair: a private key that never leaves your machine, and a public key that you distribute to servers. When you connect, the server sends a challenge encrypted with your public key. Only your private key can decrypt it. The response proves you hold the private key without ever transmitting the private key itself.

The public key lives on the remote server in `~/.ssh/authorized_keys`. The private key lives on your local machine, typically in `~/.ssh/id_ed25519`. If someone steals the authorized_keys file, they cannot use it to authenticate — it contains no secret. If someone steals your private key, they can authenticate to any server that has your public key. The private key is the secret.

Passphrases add a second layer: the private key file on disk is itself encrypted with a symmetric cipher using the passphrase as the key. This means stealing the private key file is not enough — the attacker also needs the passphrase.

### Host Key Verification — Trust on First Use

When you connect to a server for the first time, SSH presents you with the server's host key fingerprint and asks you to verify it. This is the Trust On First Use (TOFU) model. Once you accept it, SSH stores the fingerprint in `~/.ssh/known_hosts`. On every subsequent connection, SSH checks the fingerprint against the stored value. If they don't match, SSH refuses the connection and warns you loudly — because a changed host key is either a server reinstall (expected) or a man-in-the-middle attack (catastrophic).

The TOFU model has a known weakness: the first connection is unverified. If an attacker positions themselves between you and the server on your first connection, you'll store their host key, not the server's. This is why for high-security environments, host keys should be verified out-of-band — through the cloud provider's console, or by distributing them via a configuration management system.

### The sshd Configuration Philosophy

The SSH daemon (`sshd`) runs on the server and listens for incoming connections. Its configuration file, `/etc/ssh/sshd_config`, controls what authentication methods are permitted, which users are allowed, and how connections are handled. The defaults on modern Linux distributions are reasonable but not production-hardened. Hardening means explicitly disabling things that should never be allowed — password authentication, root login, weak ciphers — rather than relying on the fact that they're not being used today.

## How It Works (Mechanics)

### Connection Establishment

When you run `ssh user@host`, the following happens in sequence:

```text
# Conceptual flow — not a real command
1. TCP connection established to port 22
2. Both sides exchange protocol version strings
3. Key exchange negotiation: both sides advertise supported algorithms
4. Server sends its host public key
5. Client checks known_hosts — does this fingerprint match?
6. Diffie-Hellman key exchange: shared session key derived
   (Since OpenSSH 10.0: mlkem768x25519-sha256, a hybrid post-quantum exchange by default)
7. All subsequent traffic is encrypted with the session key
8. Client authenticates: tries publickey, then password (if allowed)
9. Session opened: shell, command, or subsystem
```

Step 5 is where `known_hosts` matters. Steps 6 onward are where encryption begins. Step 8 is where your key pair or password is used. Everything before step 6 is in the clear — but the only information exchanged there is algorithm negotiation and the host key, neither of which is secret.

### Key Exchange and Post-Quantum Cryptography

OpenSSH 10.0 (April 2025) enabled `mlkem768x25519-sha256` as the default key exchange algorithm. This is a hybrid exchange that combines classical X25519 Diffie-Hellman with ML-KEM-768 (a NIST-standardized post-quantum key encapsulation mechanism). The hybrid approach means the session key is secure even if either the classical or the post-quantum algorithm is broken. If you connect to an older server that doesn't support the hybrid algorithm, OpenSSH falls back to the classical exchange and emits a warning.

### What ssh-agent Actually Does

`ssh-agent` is a process that holds your decrypted private keys in memory. When SSH needs to sign a challenge during authentication, it asks the agent to perform the signing operation. The agent does the signing internally and returns only the signature — your private key never leaves the agent's memory. This means:

1. You type your passphrase once to add the key to the agent
2. Subsequent SSH operations authenticate without re-entering the passphrase
3. The private key material is never written to disk in decrypted form during the session

The agent communicates via a Unix domain socket, whose path is stored in `$SSH_AUTH_SOCK`. Any SSH client that has access to this socket can request signing operations.

## Basic Example

```bash
# Generate an Ed25519 key pair
# -t: key type
# -C: comment (use email or hostname for identification)
# -f: output file path (default is ~/.ssh/id_ed25519)
ssh-keygen -t ed25519 -C "your@email.com" -f ~/.ssh/id_ed25519
```

```text
Generating public/private ed25519 key pair.
Enter passphrase (empty for no passphrase): 
Enter same passphrase again: 
Your identification has been saved in /home/user/.ssh/id_ed25519
Your public key has been saved in /home/user/.ssh/id_ed25519.pub
The key fingerprint is:
SHA256:abc123XYZexamplefingerprint your@email.com
The key's randomart image is:
+--[ED25519 256]--+
|        .o+..    |
...
+----[SHA256]-----+
```

```bash
# Copy your public key to the remote server
# This appends your public key to ~/.ssh/authorized_keys on the remote
ssh-copy-id -i ~/.ssh/id_ed25519.pub user@192.168.1.100
```

```text
/usr/bin/ssh-copy-id: INFO: attempting to log in with the new key(s)
Number of key(s) added: 1

Now try logging into the machine, with:   "ssh 'user@192.168.1.100'"
```

```bash
# Connect to the remote server
ssh user@192.168.1.100
```

Two files were created: `~/.ssh/id_ed25519` (your private key — guard this like a password) and `~/.ssh/id_ed25519.pub` (your public key — safe to share). The `.pub` file is what `ssh-copy-id` placed on the remote server.

Now try this yourself. Generate a key pair and connect to a local VM. On the first connection you'll see the host key fingerprint prompt — accept it, then confirm the fingerprint was saved to `~/.ssh/known_hosts` with `cat ~/.ssh/known_hosts`.

## Real-World Example

### Full Production SSH Setup

Here's how a senior sysadmin configures SSH for a production environment with a bastion host, multiple servers, and agent forwarding handled carefully.

**Step 1: Generate a dedicated key for this environment**

```bash
# Separate key per environment — compromise of one key doesn't expose all servers
ssh-keygen -t ed25519 -C "prod-access-$(date +%Y%m%d)" -f ~/.ssh/id_ed25519_prod
```

**Step 2: Configure ~/.ssh/config**

```ini
# ~/.ssh/config
# Global defaults
Host *
    ServerAliveInterval 60
    ServerAliveCountMax 3
    IdentitiesOnly yes          # Only use keys explicitly specified — prevents agent enumeration

# The bastion host
Host bastion
    HostName 203.0.113.10
    User deploy
    IdentityFile ~/.ssh/id_ed25519_prod
    ControlMaster auto
    ControlPath ~/.ssh/sockets/%r@%h-%p
    ControlPersist 10m

# Internal app servers — jump through bastion automatically
Host app-*
    User deploy
    IdentityFile ~/.ssh/id_ed25519_prod
    ProxyJump bastion

# A specific internal server
Host app-01
    HostName 10.0.1.11

Host app-02
    HostName 10.0.1.12

# Database servers — separate key, no agent forwarding
Host db-*
    User dbadmin
    IdentityFile ~/.ssh/id_ed25519_db
    ProxyJump bastion
    ForwardAgent no             # Never forward agent to database hosts
```

**Step 3: Set correct permissions**

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_ed25519_prod
chmod 644 ~/.ssh/id_ed25519_prod.pub
chmod 600 ~/.ssh/config
```

**Step 4: Load keys into agent for the session**

```bash
# Start the agent if not already running
eval "$(ssh-agent -s)"

# Add the production key — passphrase prompted once
ssh-add ~/.ssh/id_ed25519_prod
```

**Step 5: Connect to an internal server through the bastion**

```bash
# This single command automatically uses the bastion as a jump host
ssh app-01
```

```text
Welcome to Ubuntu 24.04.2 LTS (GNU/Linux 6.8.0-48-generic x86_64)
deploy@app-01:~$
```

**Step 6: Harden the server's sshd_config**

```ini
# /etc/ssh/sshd_config — production hardening
Port 22
Protocol 2

# Authentication
PermitRootLogin no
PasswordAuthentication no
PubkeyAuthentication yes
AuthorizedKeysFile .ssh/authorized_keys
PermitEmptyPasswords no

# Restrict access to specific users
AllowUsers deploy dbadmin

# Brute-force mitigation
MaxAuthTries 3
MaxSessions 5

# Connection keepalive — disconnect idle sessions after 30 min
ClientAliveInterval 300
ClientAliveCountMax 6

# Disable legacy/weak features
X11Forwarding no
AllowAgentForwarding no        # Only enable on bastions that need it
AllowTcpForwarding no          # Restrict unless port forwarding is required
PrintMotd no
```

```bash
# After editing sshd_config, always test before reloading
# Running sshd in test mode parses config and exits — catches syntax errors
sshd -t

# Reload (not restart) — keeps existing sessions alive
systemctl reload sshd
```

The `ProxyJump` directive in `~/.ssh/config` tells the SSH client to first connect to `bastion`, then from there open a connection to the target host. The entire connection to the target host — including the authentication handshake — is tunneled through the encrypted bastion connection. No credentials touch the bastion in cleartext. This is cleaner than the old `ProxyCommand ssh bastion -W %h:%p` syntax and handles multi-hop scenarios naturally.

`ControlMaster auto` + `ControlPersist 10m` means the first connection to the bastion creates a persistent master socket in `~/.ssh/sockets/`. Subsequent connections to any `app-*` host reuse that socket instead of negotiating a new TCP connection and key exchange. For a production morning where you run 20 ad-hoc commands across multiple servers, this is a meaningful latency improvement.

`IdentitiesOnly yes` in the global `Host *` block is a subtle but important security setting. Without it, SSH will offer all keys loaded into your agent to every server it connects to — even keys meant for a different environment. This leaks information about your key infrastructure to every server you connect to. `IdentitiesOnly yes` restricts each connection to only the key specified in `IdentityFile`.

## ⚠️ Gotchas

### authorized_keys Permissions — The Silent Failure

The mistake: you copy your public key to the server, permissions look fine, but SSH keeps asking for a password anyway.

```bash
# This looks like it should work
ls -la ~/.ssh/
```

```text
drwxrwxr-x  2 alice alice 4096 Jan 15 10:00 .ssh/
-rw-rw-r--  1 alice alice  400 Jan 15 10:00 .ssh/authorized_keys
```

```text
# What you see in /var/log/auth.log on the server
Authentication refused: bad ownership or modes for directory /home/alice/.ssh
```

`sshd` silently ignores `authorized_keys` if the `~/.ssh` directory or the `authorized_keys` file is world-writable, or if the home directory itself is writable by others. The daemon won't tell you this in the connection output — it falls through to the next authentication method (usually password) without explanation. You'll sit there wondering why key auth isn't working when the server is actively refusing to look at your key.

The rule: `sshd` refuses to read `authorized_keys` if it believes the file could have been tampered with by an unauthorized party. World-writable means anyone on the server could have added a key. `sshd` treats that as a security violation.

```bash
# The fix
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys

# Also verify the home directory itself isn't group/world writable
chmod 755 ~
```

The correct permission hierarchy: home directory `755`, `.ssh` directory `700`, `authorized_keys` file `600`. All three must be correct. Fixing one and leaving the others wrong still breaks authentication.

### ssh-add Without a Running Agent

The mistake: you try to add a key and get a confusing error.

```bash
ssh-add ~/.ssh/id_ed25519
```

```text
Could not open a connection to your authentication agent.
```

This means `$SSH_AUTH_SOCK` is unset or points to a socket that no longer exists. The ssh-agent process is not running, or you started it in a different shell session and the environment variable didn't propagate.

`ssh-agent` communicates via a Unix socket. When you start it with `ssh-agent -s`, it prints the commands to set `SSH_AUTH_SOCK` and `SSH_AGENT_PID` in your shell. If you ran `ssh-agent -s` but didn't `eval` the output, those variables were never set in your current shell.

```bash
# Wrong: starts agent but variables don't reach your shell
ssh-agent -s

# Right: starts agent AND sets the environment variables in your current shell
eval "$(ssh-agent -s)"

# Then add your key
ssh-add ~/.ssh/id_ed25519
```

Many desktop environments (GNOME, KDE) and terminal multiplexers (tmux, screen) start an agent automatically. If you open a new terminal and `$SSH_AUTH_SOCK` is already set, you're using the desktop's agent — `ssh-add` will work immediately. In headless or non-desktop environments, you manage the agent lifecycle yourself.

### ForwardAgent — Your Keys Are Now on the Remote Host

The mistake: you set `ForwardAgent yes` globally and wonder why a compromise of one server grants access to all your servers.

```ini
# This seems convenient — don't do this globally
Host *
    ForwardAgent yes
```

`ForwardAgent yes` means the remote SSH daemon forwards connections back to `$SSH_AUTH_SOCK` on the remote host into your local agent. Any process running as your user on the remote host — or any process running as root — can make signing requests to your local agent. If the remote server is compromised, the attacker doesn't need your private key files. They can use your agent to authenticate to every server in `authorized_keys` while your connection is active.

This isn't theoretical. Agent forwarding attacks are a well-documented lateral movement technique.

The correct approach: use `ProxyJump` instead of agent forwarding for jumping through bastion hosts. `ProxyJump` establishes the end-to-end SSH connection on your local machine — the bastion never sees your authentication. No agent socket is forwarded to the bastion. You get the same connectivity without the exposure.

```ini
# Right: ProxyJump for bastions, ForwardAgent only where truly needed
Host bastion
    ForwardAgent no

Host internal-*
    ProxyJump bastion
    ForwardAgent no

# Only enable ForwardAgent for hosts where you actually need it
# (e.g., a build server that needs to push to git)
Host build-server
    ForwardAgent yes
```

### ControlMaster Stale Socket

The mistake: your master connection died (network drop, timeout, server restart) and new connections hang indefinitely.

```bash
ssh app-01
```

```text
# Hangs with no output — waiting for a master socket that's dead
```

When `ControlMaster` is enabled, new connections try to attach to the existing master socket before establishing their own TCP connection. If the socket file exists but the master process is dead, the connection attempt hangs.

```bash
# Check if a socket exists
ls ~/.ssh/sockets/

# Tell the master to exit cleanly
ssh -O exit app-01

# If that hangs too, kill the socket file directly
rm ~/.ssh/sockets/deploy@app-01-22

# Then reconnect normally
ssh app-01
```

Use a `%C` hash in your `ControlPath` to prevent socket name collisions between different users or environments:

```ini
ControlPath ~/.ssh/sockets/%C
```

`%C` is a hash of the connection parameters — guaranteed unique per connection target, short enough to not exceed Unix socket path limits.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install openssh-server openssh-client`; service: `ssh.service`; config: `/etc/ssh/sshd_config` |
| RHEL / Fedora / CentOS / Rocky | `dnf install openssh-server openssh-clients`; service: `sshd.service`; config: `/etc/ssh/sshd_config`; SELinux may block custom ports — use `semanage port -a -t ssh_port_t -p tcp 2222` |
| Arch | `pacman -S openssh`; service not enabled by default — `systemctl enable --now sshd` |
| Alpine (containers) | `apk add openssh`; service: `rc-service sshd start`; often used in containers without a running sshd — use host-based key injection instead |

On RHEL/Rocky systems, `firewalld` is enabled by default. If you change the SSH port, you must update both `sshd_config` and firewalld: `firewall-cmd --permanent --add-port=2222/tcp && firewall-cmd --reload`.

## Alternatives & Comparisons

| Tool / Method | When to Use | Notes |
|---|---|---|
| Ed25519 keys | New deployments, anything after OpenSSH 6.5 (2014) | Best choice: fast, small, secure, PRNG-resistant |
| RSA-4096 keys | Legacy systems requiring RSA, some old cloud providers | Still secure but slower; minimum 3072 bits |
| ECDSA keys | Almost never for new keys | PRNG vulnerability, NIST curve controversy |
| Password auth | Temporary recovery only | Disable immediately after key setup |
| SSH certificates (CA) | Fleets of 10+ servers or teams of 5+ users | Eliminates authorized_keys management; see below |
| Mosh | Roaming connections, high-latency networks | UDP-based, resumes sessions on IP change; not a replacement for SSH, requires SSH for initial setup |

**Honest recommendation:** Use Ed25519 keys for everything. Disable password auth immediately. If you manage more than a handful of servers or have a team of users, invest the time to set up SSH certificate authentication — the operational savings at scale are enormous.

## Trade-offs

**Where this shines:**

- End-to-end encryption makes SSH safe over any network, including hostile ones
- Public-key authentication eliminates password-based brute force attacks entirely
- The protocol is mature, audited, and has a strong track record — vulnerabilities are found and fixed quickly
- Multiplexing (`ControlMaster`) amortizes connection setup cost across multiple commands
- Port forwarding turns SSH into a general-purpose secure tunnel for any TCP application
- Certificate authentication scales to fleets of thousands of servers without per-server authorized_keys management

**Where this costs you:**

- Key management is a real operational burden at scale — who has keys to which servers, when were they rotated, are any orphaned keys still active?
- `ForwardAgent` creates a security exposure that is easy to enable and easy to forget
- `ControlMaster` introduces a stale socket failure mode that is confusing to diagnose the first time
- Certificate revocation requires a Key Revocation List that every server must check — it's not automatic
- SSH is a direct-access protocol with no audit trail by default; you know someone connected, not what they did

**Performance note:** Ed25519 key exchange and signing is roughly 2–3x faster than RSA-4096 for the cryptographic operations alone. On a connection to a server 100ms away, this is imperceptible. On a script that opens 500 SSH connections sequentially, it matters. `ControlMaster` matters more — reusing an existing connection reduces per-connection overhead from ~100ms to ~1ms for the socket handshake.

## When to Use / When to Avoid

✅ **Use this when:**

- You need a secure interactive shell on any remote machine
- You need to transfer files securely (`scp`, `sftp`, `rsync -e ssh`)
- You need a secure tunnel for an application that has no native encryption (database connections over SSH, for example)
- You are accessing internal services through a bastion host
- You are deploying code from a CI/CD system that needs to reach a server

❌ **Avoid this when:**

- You need a long-lived, resumable session over an unreliable network — reach for `mosh` or run a terminal multiplexer (`tmux`, `screen`) inside the SSH session
- You need to give a third party temporary access to a specific command and nothing else — `authorized_keys` supports `command=` restrictions, but evaluate whether a purpose-built API or webhook is cleaner

⚠️ **Common mistakes:**

- Leaving password authentication enabled "temporarily" — it stays enabled until something bad happens
- Using the same key pair for every server and environment — one compromise exposes everything
- Not setting `ClientAliveInterval` on the server — idle connections accumulate and waste resources
- Checking `sshd_config` into version control with default settings and never hardening it
- Changing the SSH port as a "security measure" — port 22 vs port 2222 provides zero real security benefit, only noise reduction in logs; spend that effort on key-only auth instead

## Veteran Perspective

The newcomer mistake with SSH is treating it as a login protocol. You learn `ssh user@host`, it works, you move on. The subtler mistake — the one that bites intermediate sysadmins — is not understanding that SSH is a trust system and every configuration decision is a trust decision.

`PermitRootLogin no` is not just a checkbox. It removes the possibility of an attacker using root-level credentials over the network, which means the blast radius of a compromised password is reduced to a normal user account. Combined with `PasswordAuthentication no`, you've eliminated the entire class of brute-force attacks, credential stuffing attacks, and password sniffing attacks. At that point, the only way in is to compromise a private key. These two settings together are the single most impactful thing you can do to an SSH configuration.

On `known_hosts`: most people blindly type `yes` at the first connection prompt without reading the fingerprint. This is the original sin of SSH security. In a production environment, the correct workflow is to get the server's host key fingerprint out-of-band — from the cloud provider's console, from your configuration management tool, from the person who built the server — and verify it before accepting. Not everyone does this. Do it anyway for any server that matters.

`ForwardAgent` is a trap that catches experienced people, not just beginners. It's seductive because it solves a real problem (getting from a bastion to an internal server without a key on the bastion) in an apparently convenient way. The mental model of "my keys are forwarded" sounds safe. The reality is that your agent socket is accessible to any process running as your user on the remote host — and if root is compromised, to any process at all. Use `ProxyJump`. It solves the same connectivity problem without exposing the agent socket to the remote host.

Here's a scenario I've seen play out: a developer enables `ForwardAgent yes` globally because a senior engineer told them to "for the bastion." Six months later, a web application on one of the internal servers has an RCE vulnerability. The attacker gets a shell as `www-data`, escalates to the developer's user (common via misconfigurations), and finds a live agent socket. From there they authenticate to every other server that trusts that key, pull source code, access databases, and exfiltrate credentials. The original vulnerability was a web app bug. The blast radius became the entire infrastructure because of one `ForwardAgent yes` in `~/.ssh/config`.

SSH certificates deserve more attention than they get in most Linux education. The model is elegant: instead of managing `authorized_keys` on every server, you run a Certificate Authority. You sign user keys with the CA. Every server trusts the CA. When a user leaves, you revoke their certificate. When a new server comes online, it trusts the CA automatically. No `authorized_keys` to sync, no orphaned keys to audit. At scale — anything beyond five servers or three users — the operational difference is significant. The tooling is built into `ssh-keygen`; you don't need additional software to start.

In technical interviews, SSH comes up as infrastructure glue more often than as a standalone topic. Interviewers ask about bastion patterns, key management at scale, and "how would you audit who has access to what?" The certificate model is the right answer to all three. If you can explain the difference between distributing authorized_keys and running a CA, and explain what short-lived certificates buy you, you're ahead of most candidates.

## Try It Yourself

1. Generate a new Ed25519 key pair with `ssh-keygen -t ed25519 -C "learning-$(hostname)"`. Inspect both the private and public key files. Notice the difference in size between an Ed25519 public key and an RSA-4096 public key if you generate both.

2. Add the key to your local `authorized_keys` file (`cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys`) and connect to `localhost`: `ssh -i ~/.ssh/id_ed25519 -o IdentitiesOnly=yes localhost`. Fix any permission issues that arise.

3. Set up an `~/.ssh/config` entry for `localhost` with a custom alias: `Host localtest`. Add `HostName localhost`, `User yourusername`, and `IdentityFile ~/.ssh/id_ed25519`. Then connect with just `ssh localtest`.

4. Set up local port forwarding to tunnel traffic through SSH: `ssh -L 8080:localhost:80 localtest` (if you have a web server running) or `ssh -L 2222:localhost:22 localtest`. Verify the tunnel with `ss -tlnp | grep 8080`.

5. **Challenge:** Set up a minimal SSH Certificate Authority on a VM. Generate a CA key with `ssh-keygen -t ed25519 -f ~/ssh-ca`. Sign a user's public key with `ssh-keygen -s ~/ssh-ca -I "test-cert" -n yourusername -V +1h ~/.ssh/id_ed25519.pub`. Inspect the resulting certificate with `ssh-keygen -L -f ~/.ssh/id_ed25519-cert.pub`. Configure `sshd_config` to trust the CA with `TrustedUserCAKeys /path/to/ssh-ca.pub` and authenticate using the certificate. Revoke the certificate before it expires using `ssh-keygen -k -f ~/krl -z 1 ~/.ssh/id_ed25519-cert.pub`.

---

## Summary

- Ed25519 is the correct key type for all new keys in 2025+; RSA-4096 is acceptable only for legacy compatibility; DSA is dead and must not be used
- `~/.ssh/authorized_keys` must be mode `600`, `~/.ssh` must be mode `700`, and the home directory must not be world-writable — `sshd` silently refuses to read authorized_keys if these conditions are not met
- `ForwardAgent yes` exposes your agent socket to the remote host; use `ProxyJump` instead for bastion connectivity — it provides the same access without the risk
- Disabling `PasswordAuthentication` and `PermitRootLogin` in `/etc/ssh/sshd_config` eliminates the entire class of brute-force and credential-stuffing attacks; do this on every server
- SSH certificates (signed with `ssh-keygen -s`) eliminate per-server `authorized_keys` management at scale — the right architecture for any fleet beyond a handful of machines
