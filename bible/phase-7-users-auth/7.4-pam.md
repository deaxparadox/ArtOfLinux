# [7.4] — PAM (Pluggable Authentication Modules)

> PAM is the layer between every login prompt on your system and every way of proving who you are — swap out the authentication mechanism without touching a single application.

## The Problem This Solves

In the early 1990s, every program that needed to authenticate a user — `login`, `su`, `ftpd`, `rshd` — contained its own authentication code. If you wanted to switch from standard Unix passwords to Kerberos, you recompiled every single one of those programs. If you discovered a vulnerability in how passwords were checked, you patched every single one of those programs. If your organization decided to require hardware tokens, you audited every single one of those programs and determined which supported tokens and which didn't.

This was not just inconvenient. It was a security disaster waiting to happen. Applications disagreed on how to hash passwords, how many failed attempts to allow, how to handle expired credentials, and what session-related setup to do after login. One program would enforce account lockouts; another would not. The authentication policy lived inside the binaries, scattered and inconsistent.

Sun Microsystems published the PAM specification in 1995. The insight was a clean separation: applications should ask "is this user authenticated?" and the system should decide how to answer — completely independently of the application. An application calls the PAM API. PAM reads a per-service configuration file. That file describes a stack of modules — each a shared library — to run in sequence. The application never needs to know whether authentication used `/etc/shadow`, a Kerberos ticket, a hardware token, or TOTP. It just gets a yes or a no.

## Mental Model

Think of a nightclub with a velvet rope. The bouncer doesn't decide the entry policy — management does. Management posts a laminated checklist on the door: check ID (required), check for name on the VIP list (sufficient — if you're on the list, you're in), verify dress code (required). The bouncer runs down the checklist in order. Different nights, different checklists — but the bouncer's job stays the same: run the list, report the result.

PAM is that checklist system. Every login service (SSH, `sudo`, the display manager, the screensaver) is a bouncer. The checklist for each service lives in `/etc/pam.d/`. Each line on the checklist is a module. You can swap modules, reorder them, add new ones, and no bouncer needs retraining.

## Theory

### The Four Module Types

Every PAM module serves one of four roles. A full PAM interaction runs all four in sequence:

- **auth**: Verifies that the user is who they claim to be. This is the "prove your identity" step — checking a password, validating an OTP, verifying a certificate. A single login attempt may run multiple `auth` modules in sequence (password first, then TOTP, for 2FA).

- **account**: Checks whether the authenticated account is *permitted* to log in right now. The identity check succeeded, but is the account expired? Is the password past its maximum age? Is this IP address allowed? Is it past curfew for this user? This module type answers those questions.

- **password**: Handles password changes. When you run `passwd`, the `password` stack runs — it enforces minimum length, checks against previous passwords, ensures complexity requirements, and writes the new hash to the right place.

- **session**: Manages what happens after authentication succeeds and before the user gets a shell, and what happens when they log out. Mounting home directories, setting resource limits, setting environment variables, logging the session — all session modules.

### The Per-Service Configuration

PAM configuration is not a single global file. It is per-service. Every application that links against PAM by its service name. `sshd` uses the `sshd` config. `sudo` uses the `sudo` config. The login program uses `login`. Each service has its own file in `/etc/pam.d/`.

The file format is simple. Each line specifies one module in one of four module types:

```text
# module-type   control-flag   module-path   [module-arguments]
auth            required       pam_unix.so   nullok
account         required       pam_unix.so
password        required       pam_unix.so   sha512
session         required       pam_unix.so
```

Lines are evaluated top to bottom, within each module type. When a service authenticates a user, PAM runs the full `auth` stack first, then the full `account` stack, then (on password change) the `password` stack, then the `session` stack.

### The Control Flags

The control flag on each line determines what happens when a module succeeds or fails. This is the most misunderstood part of PAM, and getting it wrong can either lock everyone out or silently bypass authentication.

**`required`**: The module must succeed for the overall result to be success. If it fails, PAM marks the result as failure — but continues running the remaining modules in the stack. The user doesn't find out which module failed until the end. This behavior is deliberate: an attacker timing responses cannot determine which specific check failed. All `required` modules run regardless of earlier failures.

**`requisite`**: Also must succeed for the overall result to be success. But if it fails, PAM immediately returns failure to the application — it does not continue evaluating the stack. The failure is visible immediately. Use this when there is no point in continuing (e.g., a module that checks whether a user even exists — if the user doesn't exist, running the password module is pointless and potentially leaks information).

**`sufficient`**: If this module succeeds *and all preceding modules in the stack have also succeeded*, PAM immediately returns success to the application without running any remaining modules. If this module fails, its failure is ignored and the stack continues. Use this for "any one of these paths grants access" logic — for example, if a user is in the `wheel` group, grant access without a password check.

**`optional`**: The module's success or failure does not affect the overall result (unless it is the only module for that type). Use this for modules that do side-effect work — setting environment variables, writing to an audit log — where you don't want a failure to block login.

The bracket notation `[success=ok new_authtok_reqd=ok ignore=ignore default=bad]` is the extended form and gives finer-grained control, but the four named flags cover the vast majority of real-world configurations.

### The Stack Runs to Completion

With `required` modules, this is the hardest thing to internalize: even after a `required` module fails, the stack keeps running. The user sees a single "authentication failure" at the end. This makes brute-force timing attacks harder — an attacker who fails at module 1 still has to wait for modules 2 through N to complete before getting a response.

With `requisite`, the stack stops on failure. This is faster but reveals that the check at that position failed — which can help an attacker determine which condition was not met.

Neither behavior is universally right. The choice depends on what you're protecting and whether the speed/information trade-off matters.

## How It Works (Mechanics)

When a PAM-aware application — say, `sshd` — needs to authenticate a user, it calls `pam_start()` with its service name and the username, then calls `pam_authenticate()`. Here is what happens:

```text
# Conceptual flow — not a real command
sshd calls pam_start("sshd", "alice", ...)
  → libpam reads /etc/pam.d/sshd
  → finds @include common-auth (Debian) or include system-auth (RHEL)
  → resolves the full auth stack

sshd calls pam_authenticate()
  → libpam loads pam_unix.so, calls its auth function
    → pam_unix.so reads /etc/shadow, checks the hash
    → returns PAM_SUCCESS or PAM_AUTH_ERR
  → libpam loads next module in stack, calls its auth function
  → ... (all modules run, subject to control flag logic)
  → libpam aggregates results, returns final PAM_SUCCESS or PAM_AUTH_ERR

sshd calls pam_acct_mgmt()   ← account stack
sshd calls pam_open_session() ← session stack
  → pam_limits.so enforces /etc/security/limits.conf
  → pam_env.so sets environment variables
  → pam_unix.so logs the session to wtmp/utmp
sshd gives alice a shell
...
sshd calls pam_close_session() on logout
```

Each module is a shared library (`.so` file) loaded at runtime. Modules live in `/lib/security/` or `/lib/x86_64-linux-gnu/security/` depending on the distro. They export standard function names (`pam_sm_authenticate`, `pam_sm_acct_mgmt`, `pam_sm_open_session`, etc.) that libpam calls directly.

The application and the modules communicate through the PAM conversation function — a callback registered by the application that handles the actual I/O (prompting for passwords, displaying messages). This is why PAM modules can prompt for a second factor: they call back into the application's conversation function to display a new prompt, and the application handles however that prompt is shown (terminal, X window, SSH challenge-response).

You can watch the module loading and return values with PAM debug output:

```bash
# Enable debug output in /etc/pam.d/common-auth — add debug to a module
# Then check syslog for the detailed trace
grep pam /var/log/auth.log | tail -20
```

```text
Jun 16 11:30:01 host sshd[12345]: pam_unix(sshd:auth): authentication failure; logname= uid=0 euid=0 tty=ssh ruser= rhost=10.0.0.1  user=alice
Jun 16 11:30:04 host sshd[12345]: pam_faillock(sshd:auth): Consecutive login failures for user alice account temporarily locked
```

## Basic Example

```bash
# Look at the PAM config for sudo — a simple, readable stack
cat /etc/pam.d/sudo
```

```text
#%PAM-1.0

@include common-auth
@include common-account
@include common-session-noninteractive
```

```bash
# Look at what common-auth actually contains (Debian/Ubuntu)
cat /etc/pam.d/common-auth
```

```text
# here are the per-package modules (the "Primary" block)
auth    [success=1 default=ignore]  pam_unix.so nullok
# here's the fallback if no module succeeds
auth    requisite                   pam_deny.so
# prime the stack with a suitable default
auth    required                    pam_permit.so
# and here are more per-package modules (the "Additional" block)
auth    optional                    pam_cap.so
```

The `[success=1 default=ignore]` on `pam_unix.so` means: if it succeeds, skip 1 line (jumping over `pam_deny.so` to `pam_permit.so`). If it fails, ignore the result and continue (falling through to `pam_deny.so`, which always fails). This is the Debian `pam-auth-update` machine-generated pattern — it looks dense but it's designed to let packages insert modules without hand-editing the file.

Now try this yourself. On your own system, `cat /etc/pam.d/sudo` and trace through what happens when you type the right password versus a wrong one. If you're on RHEL/Rocky, the file looks different — `include system-auth` instead of `@include common-auth` — but the logic is the same.

## Real-World Example

This example adds account lockout after failed login attempts using `pam_faillock`, then adds TOTP 2FA using `pam_google_authenticator`. Both are realistic production changes on a server that needs stronger authentication than a plain password.

> ⚠️ **WARNING:** Editing PAM configuration files incorrectly can lock every user — including root — out of SSH and the console simultaneously. Before editing any PAM file, open a second terminal session and keep it open. If your edited session fails, use the second session to fix the configuration. Never close your working session until you have confirmed the new configuration works.

### Step 1: Configure pam_faillock for account lockout

On RHEL/Rocky 8+ and modern Debian/Ubuntu, `pam_faillock` is the current module for failed-attempt lockout. (`pam_tally2` is removed — do not use it.)

```bash
# Check that pam_faillock is available on your system
ls /lib/security/pam_faillock.so 2>/dev/null || ls /lib/x86_64-linux-gnu/security/pam_faillock.so
```

```text
/lib/x86_64-linux-gnu/security/pam_faillock.so
```

```bash
# Configure faillock global settings
# These go in /etc/security/faillock.conf (preferred over inline module arguments)
cat /etc/security/faillock.conf
```

```text
# Lock account after 5 failures within 10 minutes
deny = 5
# Unlock automatically after 15 minutes (900 seconds)
unlock_time = 900
# Count failures even for root
even_deny_root
# Log failures to syslog
audit
```

On RHEL 8+, `authselect` manages the PAM stack. Add faillock via:

```bash
# RHEL 8+ — use authselect to enable faillock (do not edit pam.d files directly)
sudo authselect enable-feature with-faillock
sudo authselect apply-changes
```

On Debian/Ubuntu, edit `/etc/pam.d/common-auth` to add `pam_faillock` calls. Place these lines around the main auth block:

```text
# At the TOP of /etc/pam.d/common-auth — before existing auth lines
auth    required                    pam_faillock.so preauth silent audit

# ... existing auth lines ...

# At the BOTTOM of /etc/pam.d/common-auth — after existing auth lines
auth    [default=die]               pam_faillock.so authfail audit
auth    sufficient                  pam_faillock.so authsucc audit
```

```bash
# Verify the lockout is working — check fail counts
sudo faillock --user alice
```

```text
alice:
When                Type  Source                                           Valid
2024-06-16 11:30:01 RHOST 10.0.0.1                                             V
```

```bash
# Manually unlock an account that was locked
sudo faillock --user alice --reset
```

### Step 2: Add TOTP 2FA with pam_google_authenticator

```bash
# Install the PAM module
# Debian/Ubuntu:
sudo apt install libpam-google-authenticator

# RHEL/Rocky (requires EPEL):
sudo dnf install epel-release && sudo dnf install google-authenticator
```

```bash
# Each user runs this to set up their TOTP key
# Run as the user themselves, not as root
google-authenticator
```

```text
Do you want authentication tokens to be time-based (y/n) y
<QR code appears here — scan with your authenticator app>
Your new secret key is: JBSWY3DPEHPK3PXP
Your verification code is 123456
Your emergency scratch codes are:
  12345678
  87654321
  ...

Do you want me to update your "/home/alice/.google_authenticator" file? (y/n) y
Do you want to disallow multiple uses of the same authentication token? (y/n) y
Do you want to increase the time skew window to 4 minutes? (y/n) n
Do you want to enable rate-limiting? (y/n) y
```

```bash
# Add the module to /etc/pam.d/sshd
# Add this line BEFORE the @include common-auth line:
# auth required pam_google_authenticator.so nullok
# The nullok flag allows login for users who haven't set up TOTP yet
```

```bash
# Enable challenge-response authentication in sshd
sudo grep -n 'KbdInteractiveAuthentication\|ChallengeResponseAuthentication' /etc/ssh/sshd_config
```

```text
KbdInteractiveAuthentication yes
```

```bash
# If using older sshd (pre-8.7), the option name is ChallengeResponseAuthentication
# Set it to yes, then restart
sudo systemctl restart sshd
```

The walkthrough: `pam_faillock` runs before the password check (`preauth`) to see if the account is already locked. If not, `pam_unix.so` checks the password. On failure, `pam_faillock` records the failure (`authfail`). On success, it clears the counter (`authsucc`). The TOTP check in `pam_google_authenticator` runs as a second `auth` step after the password. SSH's challenge-response mode triggers a second prompt for the TOTP code after the password is accepted. Both must pass for login to succeed.

## ⚠️ Gotchas

### Breaking PAM Locks You Out of Everything — Simultaneously

The mistake is editing a PAM file, saving it, and then closing your terminal to "test a fresh login." You mistyped one argument. Now every login — SSH, console, `su`, `sudo` — fails. You have no way in.

> ⚠️ **WARNING:** Always keep an existing authenticated root session open while testing PAM changes. Closing your session before verifying the new config works is the most reliable way to lock yourself out of a server permanently. On a remote server, this means a call to the hosting provider for console access.

```bash
# The safe workflow for any PAM change:
# 1. Keep this terminal open (root or sudo-capable)
# 2. Open a SECOND terminal
# 3. Make your PAM change
# 4. In the second terminal, test: ssh localhost (or su - testuser)
# 5. Only close the first terminal when step 4 succeeds
```

If you do get locked out and have physical or console access, boot into single-user mode (RHEL: append `rd.break` or `init=/bin/bash` to the kernel line in GRUB; Debian: select recovery mode in GRUB). Single-user mode typically bypasses PAM entirely and drops you to a root shell. Fix the PAM file and reboot normally.

### `requisite` vs `required` — They Fail Differently

Both mean "this module must succeed." The difference is in what happens next.

```text
# A stack with required:
auth  required   pam_unix.so        ← fails (wrong password)
auth  required   pam_faillock.so    ← still runs
auth  required   pam_deny.so        ← still runs
# User sees "authentication failure" only after all three complete
```

```text
# A stack with requisite:
auth  requisite  pam_unix.so        ← fails (wrong password)
auth  required   pam_faillock.so    ← DOES NOT RUN
auth  required   pam_deny.so        ← DOES NOT RUN
# User sees "authentication failure" immediately after pam_unix.so fails
```

The practical difference: if `pam_faillock.so` comes after a `requisite` module that fails, the failure counter is never incremented — because `pam_faillock.so` never runs. This is why the standard `pam_faillock` configuration uses `required` for the preauth check (so the account-locked check happens even if the stack later fails) and places the `authfail` recorder after the main auth module.

### `sufficient` Stops on Success, Not on Failure

The common misreading: "if `sufficient` fails, the stack stops." It doesn't. `sufficient` is asymmetric:

- **Success:** Stack stops, overall result is success (assuming no prior `required` module already failed).
- **Failure:** Module failure is silently ignored, stack continues to the next module.

```text
auth  sufficient  pam_unix.so      ← if password is wrong, continues — does NOT stop
auth  required    pam_deny.so      ← this runs if pam_unix.so failed
```

If you want a module to stop the stack on failure, use `requisite`. `sufficient` only short-circuits on success.

### `pam_tally2` Is Gone — Stop Using It

You'll find guides online that configure account lockout with `pam_tally2`. This module was deprecated and then physically removed from upstream Linux-PAM. On RHEL 8+, `pam_tally2.so` does not exist. On recent Debian/Ubuntu, it has also been removed.

```bash
# Checking if pam_tally2 even exists on your system:
ls /lib/security/pam_tally2.so 2>/dev/null || echo "not found — use pam_faillock"
```

```text
not found — use pam_faillock
```

If you copy a PAM config from an old guide that includes `pam_tally2.so` and that file doesn't exist, PAM will fail to load the module. Depending on the control flag, this can either silently fail (if `optional`) or lock everyone out (if `required`). The replacement is `pam_faillock`. The management command is `faillock` (not `pam_tally`).

### The `nullok` Option on `pam_unix.so`

By default, `pam_unix.so` will deny login for accounts with an empty password — which is the right behavior for most services. But some configurations remove `nullok` from the wrong service and break password-less local accounts that legitimately have no password set.

The more common mistake is the opposite: leaving `nullok` on a service that should require passwords. `nullok` means "an account with no password in `/etc/shadow` can authenticate without providing one." On SSH with password authentication enabled, this means any account with a blank password is freely accessible. Check for accounts with no password:

```bash
# Find accounts with empty passwords — the second field in /etc/shadow is empty
sudo awk -F: '($2 == "" || $2 == "!!" || $2 == "!") { print $1 }' /etc/shadow
```

Accounts with `!` or `!!` in the password field are locked — they can't authenticate at all. Accounts with an empty string can authenticate without a password if `nullok` is present.

### Editing PAM Files That Are Managed by `pam-auth-update` or `authselect`

On Debian/Ubuntu, the files `/etc/pam.d/common-auth`, `/etc/pam.d/common-password`, `/etc/pam.d/common-account`, and `/etc/pam.d/common-session` are machine-generated by `pam-auth-update`. If you hand-edit them and then a package installs a new PAM module and triggers `pam-auth-update`, your edits may be overwritten.

The correct approach on Debian/Ubuntu: add custom modules in the service-specific files (e.g., `/etc/pam.d/sshd`) and let `pam-auth-update` manage the common files.

On RHEL 8+, `authselect` manages the PAM stack. Directly editing `/etc/pam.d/system-auth` or `/etc/pam.d/password-auth` is not supported — `authselect apply-changes` overwrites them. Use `authselect` commands or create a custom profile.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | Shared config files: `/etc/pam.d/common-auth`, `common-account`, `common-password`, `common-session`. Managed by `pam-auth-update`. Services use `@include common-auth`. Install: `libpam-google-authenticator` for TOTP, `libpam-pwquality` for password quality. |
| RHEL / Fedora / CentOS / Rocky 8+ | Shared config files: `/etc/pam.d/system-auth` and `/etc/pam.d/password-auth` (symlinks to `-ac` files managed by `authselect`). Do not hand-edit — use `authselect`. `pam_faillock` built-in; configure via `/etc/security/faillock.conf`. `pam_tally2` removed. Install EPEL for `google-authenticator`. |
| RHEL 7 and older | `authconfig` instead of `authselect`. `pam_tally2` still present but deprecated. Prefer `pam_faillock` even here. |
| Arch | No shared helper files — each service has its own full stack. PAM config directly in `/etc/pam.d/`. `libpam-google-authenticator` available in AUR. |
| Alpine (containers) | PAM support is optional and often not installed — Alpine uses `busybox` auth by default. Install `linux-pam` explicitly if needed. Common in containers where PAM is not used. |

## Alternatives & Comparisons

| Tool / Module | When to Use | Notes |
|---|---|---|
| `pam_unix.so` | Standard Unix password auth against `/etc/shadow` | Present everywhere; the baseline for almost all stacks |
| `pam_faillock.so` | Account lockout after failed attempts | Current standard; replaced `pam_tally2` |
| `pam_tally2.so` | Legacy lockout — do not use | Removed from upstream; not present on RHEL 8+, recent Debian |
| `pam_google_authenticator.so` | TOTP 2FA without a dedicated auth server | Good for individual servers; requires per-user setup |
| `pam_krb5.so` | Kerberos/AD authentication | Enterprise environments with a domain controller; more complex setup |
| `pam_ldap.so` / `pam_sss.so` (SSSD) | LDAP / Active Directory | `pam_sss.so` via SSSD is the modern approach; handles caching and offline auth |
| `pam_limits.so` | Enforce resource limits at login | Reads `/etc/security/limits.conf`; affects every process spawned from the session |
| `pam_env.so` | Set environment variables at login | Reads `/etc/security/pam_env.conf`; runs before the user's shell profile |

For most servers that need to stay standalone, `pam_unix.so` + `pam_faillock.so` + `pam_google_authenticator.so` is a complete, manageable stack. For environments with multiple servers and centralized user management, replace `pam_unix.so` with `pam_sss.so` backed by SSSD — single credential source, offline caching, group-based access control from LDAP/AD.

## Trade-offs

**Where this shines:**

- True separation of policy from mechanism: changing authentication requirements (add 2FA, change lockout policy, switch to LDAP) requires only config changes, not application changes.
- Per-service granularity: `sshd` can require 2FA while `sudo` requires only a password, and the local console can require something else entirely — all managed from one place.
- Composable: you can layer modules — password check, then lockout counter, then TOTP — and they compose cleanly as long as you understand the control flags.
- Universally adopted: every modern Linux authentication path goes through PAM. Understanding it gives you leverage over the entire authentication stack.

**Where this costs you:**

- Misconfiguration is catastrophic: a syntax error or wrong module path in a `required` module can lock every user out of every login method simultaneously. There is no safety net in the config files themselves.
- The control flag semantics are subtle: `requisite` vs `required`, and the asymmetric behavior of `sufficient`, are genuinely confusing. Most people who "understand PAM" still mis-predict behavior under some combinations.
- Debugging is opaque: when authentication fails, the application gets "denied" — it doesn't get told which module failed or why. You're reading `/var/log/auth.log` with PAM debug options enabled, which is noisy and requires knowing what you're looking for.
- Module quality varies: third-party modules (including `pam_google_authenticator`) are not part of the kernel or core distribution. Their behavior can differ subtly across versions and distros.

## When to Use / When to Avoid

✅ **Use `pam_faillock` when:**
- Any service faces the public internet and authenticates with passwords — account lockout is a baseline defense against brute-force attacks.
- You need to comply with standards that mandate login attempt limits (PCI DSS, CIS benchmarks).

✅ **Use `pam_google_authenticator` when:**
- You want 2FA on a server without deploying a RADIUS server or a full identity provider.
- You have a small number of users and can manage per-user `.google_authenticator` files.

✅ **Use `pam_limits` / `pam_env` when:**
- You need resource limits (`ulimit`) to apply to interactive sessions and to any process they spawn — not just the shell.
- You need environment variables set before a user's `.profile` runs.

❌ **Avoid editing `common-auth` or `system-auth` directly when:**
- Your distro uses `pam-auth-update` (Debian/Ubuntu) or `authselect` (RHEL 8+) — use those tools or service-specific files instead. Your changes will be silently overwritten on the next package update.

❌ **Avoid `pam_tally2` entirely** — it is removed from current distros. Use `pam_faillock`.

❌ **Avoid adding PAM modules to `system-auth` or `common-auth` for one service's requirements** — if you only want 2FA for SSH, add it to `/etc/pam.d/sshd`, not to the shared stack that affects every login service.

⚠️ **Common mistakes:**
- Adding `pam_faillock` configuration but not including the `preauth` call — the lockout check only fires after all auth modules run, letting attackers attempt unlimited failures before the lock ever triggers.
- Setting `ChallengeResponseAuthentication no` (or `KbdInteractiveAuthentication no`) in `sshd_config` and wondering why TOTP prompts never appear — `pam_google_authenticator` needs challenge-response mode to prompt for the OTP.
- Using `requisite` on the `pam_faillock.so authfail` recorder — if the recorder never runs (because a preceding `requisite` short-circuited), failures don't get counted.

## Veteran Perspective

Here is what I would tell a junior after their first PAM-related outage: the reason PAM configuration is so dangerous is that there is no lint step, no dry run, no "test mode." You write a line, save the file, and the next login either works or it doesn't. The config file is evaluated live. If you typo a module path, PAM tries to `dlopen()` a file that doesn't exist. If the control flag is `required`, the module's return is treated as failure. The login fails. For everyone. Including root.

The session-before-editing rule is not optional and it is not paranoia. I've seen a $40,000-per-hour production system become unreachable because an engineer edited `common-auth` on a remote server, saved the file, closed the terminal, and tried to SSH back in. The file was fine syntactically — they'd used `required` on a module that needed `optional`, and every login attempt was taking 30 seconds and then failing because the module was trying to reach an LDAP server that wasn't reachable from that host. The fix was trivial. Getting to the server to apply the fix required a call to the data center at 2am for console access.

The intermediate gap is the control flag semantics. Most people learn that `required` means "must succeed" and `sufficient` means "enough to pass" and stop there. They don't internalize the timing: `required` marks failure but continues; `requisite` stops immediately; `sufficient` only short-circuits on success. The consequence is that `pam_faillock` placement is not arbitrary — the `preauth` call must come before the password module, and the `authfail` recorder must come after, using `[default=die]` so that a failed password attempt is always recorded even if the stack has already decided to fail.

The deeper principle: PAM is a pipeline. Modules don't directly communicate with each other; they communicate through the PAM context object — a set of key-value items that any module can read or write. `pam_unix.so` writes `PAM_AUTHTOK` (the entered password) to the context so that a subsequent `pam_cracklib.so` or `pam_pwquality.so` in the `password` stack can read it without prompting again. This is why some module orders matter beyond just the control flag logic.

In a security audit, the PAM configuration is always on my list. The things I check: Is `pam_faillock` (or equivalent) present on any service facing the network? Are there `required` lines that reference modules that don't exist (guaranteed lockout waiting to happen)? Is `nullok` present on services where it shouldn't be? Are the `common-auth` or `system-auth` files being hand-edited on a distro that manages them automatically? And — the one that catches people — is `pam_google_authenticator` configured with `nullok` and `sshd` configured with `AuthenticationMethods publickey` such that key-based logins bypass PAM entirely? Because if someone adds a public key to `~/.ssh/authorized_keys`, they can log in without the TOTP check unless `sshd_config` explicitly requires `AuthenticationMethods publickey,keyboard-interactive`.

## Try It Yourself

1. On your system, run `cat /etc/pam.d/sudo` and `cat /etc/pam.d/login`. Compare the two stacks — which modules appear in both? Which are login-specific? Trace through what happens when you type a correct versus incorrect password.

2. Look at the `auth` stack for `common-auth` (Debian) or `system-auth` (RHEL). Find the module responsible for checking the Unix password. What control flag does it use? What would happen if you changed it from `required` to `requisite`?

3. Check whether `pam_faillock` is configured on your system: run `sudo faillock --user $USER`. If it shows no output, examine your PAM stack to understand why. On RHEL, check `sudo authselect current`.

4. Read through the `pam_limits.so` session module: `cat /etc/security/limits.conf`. Find the default limits for your user with `ulimit -a`. Add a line to `/etc/security/limits.conf` that sets a hard limit of 100 open files for your user, then log out and back in, and verify the limit changed with `ulimit -Hn`.

5. **Challenge (use a VM):** Set up SSH TOTP 2FA on a VM using `pam_google_authenticator`. Requirements: (1) users without TOTP configured can still log in with just a password (`nullok`), (2) users with TOTP configured must provide both password and OTP, (3) accounts with 5 failed logins are locked for 10 minutes via `pam_faillock`. Keep a root session open throughout. Test that a failed TOTP attempt increments the faillock counter. Verify that `ssh -o PreferredAuthentications=publickey` bypasses PAM auth entirely, and decide whether that is acceptable for your threat model.

## Summary

- PAM decouples authentication policy (how to verify identity) from applications (what needs verification). Services read per-service config files in `/etc/pam.d/`; modules are shared libraries loaded at runtime.
- The four module types run in fixed order: `auth` (prove identity) → `account` (is the account permitted?) → `password` (handle changes) → `session` (setup and teardown). Each type has its own independent stack.
- Control flag semantics: `required` marks failure but continues the stack; `requisite` stops immediately on failure; `sufficient` stops immediately on success (ignoring subsequent modules); `optional` has no effect on the overall result.
- `pam_tally2` is removed from modern distros — `pam_faillock` is the current replacement for account lockout. Manage it via `faillock.conf` and the `faillock` command.
- Always keep an existing authenticated session open while editing any PAM file. A configuration error locks out all login methods — SSH and console — simultaneously.
