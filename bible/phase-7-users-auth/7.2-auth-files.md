# [7.2] — Authentication Files: passwd, shadow, and group

## The Problem This Solves

Early Unix systems stored password hashes directly in `/etc/passwd`. The file was world-readable — it had to be, because many system utilities needed to map UIDs to usernames and home directories. Every process on the system, running as any user, could read the file. And that meant every process on the system could read the password hashes.

With the hardware of the 1970s and 1980s, this was slow to exploit. With the hardware of today, MD5 and even SHA-512 hashes from `/etc/passwd` would be cracked in minutes. A single breach of any low-privileged account would immediately expose every user's password for offline brute-forcing.

The shadow password system was Julie Haugh's solution, written in 1987 after a break-in on a SCO Xenix system. The idea: split the user database into two files. Keep the public information — usernames, UIDs, home directories, shells — in `/etc/passwd` where any process can read it. Move the actual hashes to `/etc/shadow`, readable only by root. Everything programs needed for UID-to-name mapping stayed readable. Everything attackers wanted became inaccessible without root.

That split is still the foundation of Linux authentication today. But the files contain more than most people realize — and several of the details have real security consequences that are not obvious from the surface.

---

## Mental Model

Think of a hotel. The front desk keeps a public directory: guest names, room numbers, check-in and check-out dates. Anyone in the lobby can look at the directory and find out that Alice is in room 412. What the directory does not contain is Alice's safe combination or payment card details — those are stored separately, in a locked filing cabinet behind the desk that only staff can access.

`/etc/passwd` is the public directory. It has to be world-readable because dozens of programs need to answer questions like "what's the home directory for UID 1001?" or "what shell does bob use?" `/etc/shadow` is the locked cabinet — same guests, same room numbers, but now the sensitive credential data stored separately with much tighter access controls.

The room number (UID) is what the kernel actually cares about. The name is for humans. This matters more than it sounds.

## Theory

### What the kernel actually uses: UIDs and GIDs

Linux does not authenticate users by username. It authenticates them by UID. Once authentication is complete and a session is established, the kernel tracks everything by UID and GID — integer numbers, not strings. Every file ownership check, every capability check, every privilege decision operates on these numbers.

This means `/etc/passwd` is, from the kernel's perspective, a lookup table for userspace programs. The kernel itself never reads it. When `login` or `sshd` authenticates a user, they do the file lookups in userspace, establish the correct UID/GID, and hand a process to the kernel with those numbers already set.

The invariant that follows from this: **UID 0 grants root privileges regardless of the username field.** If an entry in `/etc/passwd` has UID 0 in the third field, the kernel treats any process running as that user as root. The username — whether it's "root", "toor", "system", or "backdoor" — is irrelevant to the kernel. Only the number matters. This is an attack vector that experienced auditors check for immediately.

### /etc/passwd

`/etc/passwd` has mode 644 — world-readable by design. Every user and every process can read it. It contains seven colon-separated fields per line:

```text
username:password:UID:GID:GECOS:home_directory:shell
```

| Field | What it contains |
|---|---|
| username | The login name. Case-sensitive. Should not contain capital letters by convention. |
| password | Always `x` on modern systems. The literal character `x` is a sentinel meaning "look in `/etc/shadow`." |
| UID | Numerical user ID. UID 0 = root. UIDs 1–99 are statically assigned to system users. UIDs 100–999 are dynamically assigned service accounts (`useradd -r`). UIDs 1000+ are regular human users. |
| GID | The user's **primary group** ID. This GID is assigned to new files the user creates, unless overridden. It does not reflect all groups the user belongs to — supplementary groups are in `/etc/group`. |
| GECOS | The "comment" field. Historically stood for "General Electric Comprehensive Operating System" — it stored batch-job parameters for a GE mainframe. Today it holds a human-readable display name, phone number, or other metadata. The `finger` command parsed subfields separated by commas. Most systems just put the user's full name here. |
| home_directory | The user's home directory. Assigned to `$HOME`. |
| shell | The login shell. If empty, `/bin/sh` is used. Service accounts use `/usr/sbin/nologin` or `/bin/false` here. |

The password field being `x` is not just a placeholder — it's a protocol. When PAM's `pam_unix` module or the `login` binary reads `/etc/passwd` and sees `x`, it calls `getspnam()` to fetch the corresponding shadow entry. If the field contains anything other than `x` — an actual hash, for instance — the system will attempt to authenticate against that hash directly, without consulting shadow at all. This backward compatibility is how pre-shadow Unix systems worked and is why you should never edit `/etc/passwd` directly.

### /etc/shadow

`/etc/shadow` has mode 640 and is owned by root:shadow. Regular users cannot read it. On some distributions it is mode 000 and only readable via `sudo` or through the shadow group. It has nine colon-separated fields:

```text
username:hash:lastchg:min:max:warn:inact:expire:flag
```

| Field | Name | Meaning |
|---|---|---|
| 1 | username | Must exactly match an entry in `/etc/passwd` |
| 2 | hash | The password hash, or a special marker (see below) |
| 3 | lastchg | Days since 1970-01-01 UTC when password was last changed. If 0, user must change password at next login. If empty, password aging is disabled. |
| 4 | min | Minimum number of days before a user can change their password again. Empty or 0 means no minimum. |
| 5 | max | Maximum number of days before a password expires. Empty disables expiry entirely. |
| 6 | warn | Number of days before expiry to start warning the user. Empty or 0 means no warning. |
| 7 | inact | Grace period in days after expiry before the account is locked. Empty means no grace period. |
| 8 | expire | Absolute account expiry date as days since 1970-01-01. The account is disabled on this date, regardless of password status. Empty means never expires. |
| 9 | flag | Reserved. Currently unused. |

Notice that fields 3 and 8 are day counts, not Unix timestamps (seconds). Password aging operates at daily granularity by design. To convert a shadow date to a human-readable date: `date -d "1970-01-01 +$DAYS days"`.

The hash field uses a structured format called crypt(3). The format is:

```text
$id$[options$]salt$hash
```

The `$id$` prefix tells `crypt()` which algorithm produced the hash:

| Prefix | Algorithm | Status |
|---|---|---|
| `$1$` | MD5crypt | Insecure; obsolete. Never use. |
| `$5$` | SHA-256 | Acceptable; found on some older systems |
| `$6$` | SHA-512 | Current RHEL/CentOS/Rocky/AlmaLinux default |
| `$y$` | yescrypt | Current Ubuntu/Debian/Fedora default; memory-hard |
| `$2b$` | bcrypt | Supported by some systems, not default on any major distro |

The hash field can also contain one of these special markers instead of a hash:

| Value | Meaning |
|---|---|
| `*` | Password login explicitly disabled. No hash stored. Set by `useradd --system` or `adduser --disabled-password`. |
| `!` or `!$6$salt$hash` | Account locked. A `!` prefix means `passwd -l` or `usermod -L` was used. If the original hash is preserved after the `!`, it can be unlocked with `passwd -u` to restore it. |
| `!!` | Password was never set. The account was created without a password. Cannot be unlocked since there is nothing to restore. |
| empty | No password required. Anyone can log in without a password. This is not intentional on production systems. |

### /etc/group

`/etc/group` has four colon-separated fields:

```text
groupname:password:GID:members
```

| Field | Meaning |
|---|---|
| groupname | The group name |
| password | Almost always `x` or empty. Group passwords (for `newgrp`) are stored in `/etc/gshadow`. Rarely used in practice. |
| GID | Numerical group ID |
| members | Comma-separated list of usernames who have this as a **supplementary** group |

The distinction between primary and supplementary groups matters. A user's primary group is stored in `/etc/passwd` field 4. It's what gets assigned to newly created files. Supplementary groups are listed in `/etc/group` and grant additional permissions — being in the `docker` group, `sudo` group, or `disk` group gives access to those resources, but it's the supplementary group mechanism doing that work.

A user does not need to appear in `/etc/group` for their primary group — the GID in `/etc/passwd` is sufficient. But a user must be listed in `/etc/group` to have a supplementary group. Run `id username` to see both at once.

### NSS: the Name Service Switch

`/etc/passwd`, `/etc/shadow`, and `/etc/group` are the default source of user information — but not the only one. The Name Service Switch (NSS) framework, controlled by `/etc/nsswitch.conf`, tells glibc where to look up user information when a program calls `getpwnam()`, `getpwuid()`, or `getgrent()`.

```text
# /etc/nsswitch.conf (typical entry)
passwd:   files systemd sss
group:    files systemd sss
shadow:   files sss
```

Each service token corresponds to an NSS module:

| Token | Library | Queries |
|---|---|---|
| `files` | `libnss_files.so` | `/etc/passwd`, `/etc/shadow`, `/etc/group` directly |
| `sss` | `libnss_sss.so` | SSSD daemon (LDAP, Active Directory, Kerberos) via Unix socket |
| `systemd` | `libnss_systemd.so` | systemd dynamic users (defined in unit files) |
| `ldap` | `libnss_ldap.so` | LDAP server directly (legacy, pre-SSSD) |
| `winbind` | `libnss_winbind.so` | Samba Winbind (Active Directory) |

The `getent` command is a direct exerciser of the NSS API. `getent passwd alice` calls `getpwnam("alice")` — it will find Alice whether she's in `/etc/passwd`, an LDAP directory via SSSD, or a systemd dynamic user. This is what makes `getent` more reliable than `grep` for user lookups in enterprise environments.

## How It Works (Mechanics)

### How login authentication actually flows

When you type a username and password into a login prompt, the following happens in sequence:

```text
# Conceptual flow — not a real command
1. login/sshd receives username + credential
2. Calls getpwnam(username) via NSS
   → NSS queries sources in nsswitch.conf order (files, sss, etc.)
   → Returns struct passwd: {username, "x", uid, gid, gecos, home, shell}
3. Calls getspnam(username) to get the shadow entry
   → Returns struct spwd: {username, "$6$salt$hash", lastchg, min, max, ...}
4. Calls crypt(typed_password, "$6$salt$hash")
   → Extracts $6$ prefix → selects SHA-512 algorithm
   → Extracts salt from the hash string
   → Runs SHA-512-crypt with that salt
   → Returns new hash string
5. Compares returned hash with stored hash
   → Match → authentication success
   → No match → failure
6. On success: setuid(uid), setgid(gid), initgroups(username, gid)
   → kernel now tracks session by UID, not username
7. exec() the login shell
```

Step 4 is the key: `crypt()` is deterministic. Given the same password and the same salt, it always produces the same output. The salt is stored as part of the hash — it's the middle section between the second and third `$`. Comparing hashes means computing the hash of the provided password using the stored salt, then comparing the output to the stored value character by character.

### How the kernel resolves privileges from UIDs

Once a process has a UID assigned, the kernel tracks permissions through three UID values per process: real UID, effective UID, and saved UID. For most processes these are identical. For privilege escalation (setuid binaries, `sudo`), the effective UID differs from the real UID — but that's a topic for [7.3 — Privilege Escalation](7.3-privilege-escalation.md).

The kernel's privilege check is a single comparison: `if (current->euid == 0)`. Everything that matters about root privilege is encoded in that UID. The kernel does not read `/etc/passwd` at runtime to make decisions. The UID is set during `exec()` of a setuid binary or explicitly by a privileged process calling `setuid()`, and it stays with the process until explicitly changed.

### Why `/etc/passwd` must be world-readable

The `ls -l` command needs to display file ownership as usernames, not UIDs. To do that, it calls `getpwuid(uid)` for each file's owner. If `/etc/passwd` were readable only by root, unprivileged `ls -l` would display raw UIDs for every file — which would make the system nearly unusable.

The same applies to `ps aux`, `stat`, `find -user`, and dozens of other tools. Making `/etc/passwd` world-readable is not a security flaw — it's a deliberate design choice that moves the secret (the hash) out of the readable file and into the restricted one.

### How `getent` differs from `grep`

```bash
# These look equivalent. They are not.
grep "alice" /etc/passwd
getent passwd alice
```

`grep /etc/passwd` only searches the local file. `getent passwd alice` calls the full NSS stack: it will find Alice in LDAP, SSSD, Active Directory, or systemd-managed users, not just the local file. In an enterprise environment with SSSD-backed Active Directory, the majority of users will not appear in `/etc/passwd` at all — they exist only in the directory. `grep` will find nothing. `getent` will find them.

## Basic Example

```bash
# Look at your own entry in /etc/passwd
getent passwd $(whoami)
```

```text
nitish:x:1000:1000:Nitish Kushwaha,,,:/home/nitish:/bin/bash
```

The seven fields separated by colons: username, `x` (password in shadow), UID 1000, GID 1000, GECOS (full name), home directory, shell.

```bash
# See what's actually in /etc/shadow for your account
# (requires sudo — shadow is not readable by regular users)
sudo getent shadow $(whoami)
```

```text
nitish:$y$j9T$someBase64Salt$someBase64Hash:19900:0:99999:7:::
```

The hash prefix `$y$` identifies yescrypt (Ubuntu 22.04+ and Debian 11+). On RHEL 9 or Rocky Linux you'd see `$6$` for SHA-512. The aging fields: last change was day 19900 since epoch, minimum age 0 (can change immediately), maximum age 99999 days (effectively never), warning period 7 days. The last three fields (inact, expire, flag) are empty.

```bash
# Look at group memberships — both primary and supplementary
id $(whoami)
```

```text
uid=1000(nitish) gid=1000(nitish) groups=1000(nitish),4(adm),24(cdrom),27(sudo),30(dip),46(plugdev)
```

Your primary group (gid=1000) is from `/etc/passwd` field 4. Every group after that is a supplementary group, listed in `/etc/group`. The `sudo` group (27) is what grants this user elevated privileges.

Try this yourself. Run `getent passwd $(whoami)` and count the fields. Then run `sudo cat /etc/shadow | grep $(whoami)` and identify the hash algorithm prefix, the salt, and the aging fields. If your hash starts with `$y$`, you're on a distribution that has adopted yescrypt. If it starts with `$6$`, you're on RHEL or an older system.

## Real-World Example

This is a common task on production systems: auditing service accounts to verify they're locked down correctly.

```bash
#!/bin/bash
# Audit service accounts: verify no shell, locked password, and correct UID range

echo "=== Service accounts with interactive shells ==="
# Service accounts should have nologin or false as their shell
# UID range 1-999 on most distros; field 3 is UID, field 7 is shell
awk -F: '$3 >= 1 && $3 <= 999 && $7 != "/usr/sbin/nologin" && $7 != "/bin/false" && $7 != "/usr/bin/false" && $7 != "" {
    printf "%-20s UID=%-6s Shell=%s\n", $1, $3, $7
}' /etc/passwd

echo ""
echo "=== Accounts with UID 0 (all should be root) ==="
# Any account with UID 0 that is NOT named "root" is a serious red flag
awk -F: '$3 == 0 { print $1 }' /etc/passwd

echo ""
echo "=== Shadow status of service accounts ==="
# Check whether service accounts have locked (*) or never-set (!!) passwords
# This requires sudo because /etc/shadow is root-readable only
sudo awk -F: '$1 ~ /^(daemon|bin|sys|www-data|nobody|systemd)/ {
    if ($2 == "*")     status = "DISABLED (*)";
    else if ($2 == "!!") status = "NEVER SET (!!)";
    else if (substr($2,1,1) == "!") status = "LOCKED (!)";
    else if ($2 == "")  status = "EMPTY (NO PASSWORD)";
    else               status = "HAS HASH - INVESTIGATE";
    printf "%-20s %s\n", $1, status
}' /etc/shadow
```

```text
=== Service accounts with interactive shells ===
sync                 UID=4     Shell=/bin/sync

=== Accounts with UID 0 (all should be root) ===
root

=== Shadow status of service accounts ===
daemon               DISABLED (*)
bin                  DISABLED (*)
sys                  DISABLED (*)
www-data             DISABLED (*)
nobody               DISABLED (*)
systemd-network      NEVER SET (!!)
```

Walkthrough: The first block finds any service account (UID 1–999) that has an interactive shell. `sync` legitimately exists on many Debian systems — it runs `/bin/sync` as its shell, which flushes disk buffers. That's intentional. Any unfamiliar name here warrants investigation.

The second block is non-negotiable on a production system: any UID 0 entry that isn't `root` is either a misconfiguration or a compromise. Pull the thread immediately.

The third block shows the expected state for service accounts. `*` (created with `--system` or `--disabled-password`) is slightly more locked down than `!!` (never set a password). Both prevent password login. As you'll see in the gotchas, neither prevents SSH key authentication — something most people don't realize.

## ⚠️ Gotchas

### Editing `/etc/passwd` Directly Can Corrupt It

The most dangerous thing you can do to a running system is open `/etc/passwd` in a text editor, save halfway through an edit, and have another process attempt a lookup during the write. The file has no locking mechanism that editors respect. A partial write leaves a syntactically broken file. A broken `/etc/passwd` means every `getpwnam()` call fails. Which means `sudo` fails. `ssh` fails. `login` fails. You are now locked out of your own system.

```bash
# This is how people have destroyed production servers
vim /etc/passwd    # DO NOT DO THIS
nano /etc/passwd   # ALSO DO NOT DO THIS
```

The correct tool is `vipw`. It uses an advisory lock, opens the file in your `$EDITOR`, validates the syntax before saving, and only commits the change if the file is valid.

```bash
# The correct way — always
sudo vipw
```

`vipw` also handles `/etc/shadow` via `vipw -s`. For groups, use `vigr` and `vigr -s`. These tools exist precisely because `/etc/passwd` is not a database — it has no transactional protection — and the locking these tools provide is the only safety mechanism available.

### Locked Accounts Can Still Log In with SSH Keys

This is the most operationally dangerous misunderstanding about account locking on modern Linux.

When you run `passwd -l username` or `usermod -L username`, you prepend a `!` to the shadow hash. The intention is clear: this account is locked and should not be able to authenticate. The problem: on most enterprise distributions, `sshd` runs with `UsePAM yes` — and when `UsePAM yes` is set, OpenSSH's built-in check for the `!` prefix is completely bypassed.

```bash
# Lock an account
sudo passwd -l alice

# Verify it looks locked
sudo getent shadow alice | cut -d: -f2 | cut -c1-5
```

```text
!$6$s
```

The hash is locked. Now try SSH:

```bash
# If alice has an authorized SSH key, this still works on most distros
ssh alice@localhost
```

On Debian, Ubuntu, and RHEL with default `UsePAM yes`, this login succeeds. Alice's account is "locked" from password authentication but fully open to key-based login.

The reason: `UsePAM yes` hands all authentication decisions to PAM. PAM's account phase (which runs after authentication succeeds) checks expiry dates and password age, but does not check whether the password hash is prefixed with `!`. That check only exists in OpenSSH's pre-PAM code path.

If you need to actually prevent access, use `chage -E 0 username` to set an account expiry date of the Unix epoch — PAM's account phase checks `sp_expire` and will block any authentication method, including SSH keys:

```bash
# This actually blocks all login, including SSH keys
sudo chage -E 0 alice

# Or set a past expiry with a real date
sudo chage -E 1970-01-02 alice
```

Alternatively, use `DenyUsers alice` in `/etc/ssh/sshd_config`, or remove the authorized_keys file.

### UID 0 Belongs to Root — Regardless of Username

The username in `/etc/passwd` field 1 is a label. The kernel never reads it. An account named "backdoor" with UID 0 is root in every meaningful sense. It can read any file. It can kill any process. It can load kernel modules.

```bash
# Look for any UID 0 accounts beyond root
awk -F: '$3 == 0' /etc/passwd
```

```text
root:x:0:0:root:/root:/bin/bash
```

One line is correct. Two lines should trigger an immediate incident response. Some systems legitimately have a second UID 0 account (`toor` on some BSDs is a historical example with no shell, used to maintain a known-good shell binary), but on Linux it's almost always a sign of compromise or a catastrophic misconfiguration. Check this on any system you're auditing — it's in every security checklist for a reason.

### `*` Is Not The Same as `!` For SSH Authentication

Accounts with `*` in the shadow password field are handled differently by OpenSSH than accounts with `!`. The `*` prefix does not trigger OpenSSH's locked-account check — either with `UsePAM yes` or `UsePAM no`. Accounts with `*` can authenticate with SSH keys on all major distros regardless of PAM configuration.

This is intentional. `*` means "this account has no password and no password will ever be set — it authenticates by other means (SSH keys, Kerberos, etc.)." Service accounts like `www-data` that need to run web servers but not accept logins use `*`. System users that exist for SSH tunneling or automated deployment use `*`.

The distinction to remember:

| Marker | Intended meaning | Password login | SSH key login (UsePAM yes) |
|---|---|---|---|
| `*` | No password, may use other auth | Blocked | Allowed |
| `!$hash` | Locked by admin | Blocked | Allowed (bypasses check) |
| `!!` | Never had password set | Blocked | Allowed (bypasses check) |

To block everything, set account expiry with `chage -E 0`.

### The `!!` After Account Creation Is Not An Error

On RHEL-family distributions, creating a user with `useradd username` (without `-p`) produces a shadow entry with `!!` in the password field. This looks alarming — like two exclamation marks denoting some kind of error state. It is not. It simply means "this account was created but a password was never set."

```bash
# Create a user without setting a password (RHEL/Fedora behavior)
sudo useradd newstaffer

sudo getent shadow newstaffer | cut -d: -f2
```

```text
!!
```

On Debian/Ubuntu, `adduser` prompts for a password during creation, so `!!` is rare in practice. On RHEL/Fedora, `useradd` without `-p` is standard workflow — you create the account, then the user sets their own password via `passwd` or through a web portal.

The operational implication: `!!` means the account cannot be used for password login until a password is set. If the account is purely for SSH key access, leaving it as `!!` is fine — but see the above gotcha about SSH keys and locking.

## Distro Notes

| Distro Family | Default hash | Config file | Notes |
|---|---|---|---|
| Ubuntu 22.04+ | yescrypt (`$y$`) | `/etc/pam.d/common-password` | `pam_unix.so yescrypt`; switched from SHA-512 in 22.04 |
| Ubuntu 20.04 | SHA-512 (`$6$`) | `/etc/pam.d/common-password` | `pam_unix.so sha512` |
| Debian 11+ | yescrypt (`$y$`) | `/etc/pam.d/common-password` | Bookworm and Bullseye both default to yescrypt |
| RHEL / Rocky / AlmaLinux 9 | SHA-512 (`$6$`) | `/etc/pam.d/system-auth` | Red Hat declined yescrypt in RHEL 9; may change in a future release |
| Fedora 35+ | yescrypt (`$y$`) | `/etc/pam.d/system-auth` | Fedora adopted yescrypt ahead of RHEL |
| Arch Linux | SHA-512 (`$6$`) | `/etc/pam.d/system-auth` | Default; can be changed via `/etc/login.defs` |

The hash algorithm configuration lives in two places and both must agree: `/etc/pam.d/` (controls what `passwd` uses when changing passwords) and `/etc/login.defs` (controls what `useradd` and `adduser` use when creating accounts). On most modern systems these are kept in sync by the package manager. If you change one manually, change the other.

On RHEL 8+, use `authselect` to manage PAM configuration rather than editing `/etc/pam.d/` files directly. `authselect` maintains consistency across the PAM stack.

## Alternatives & Comparisons

| Approach | When it's used | Notes |
|---|---|---|
| Local files (`/etc/passwd`, `/etc/shadow`) | Small teams, single servers, embedded systems | No external dependency; everything is local; no HA concern |
| SSSD + LDAP | Enterprise Linux; centralized user management | Users exist in directory; `getent` finds them; local files hold only system accounts |
| SSSD + Active Directory / Kerberos | Windows-integrated environments | Kerberos tickets for auth; group memberships from AD; nologin/password in shadow for local service accounts only |
| NIS/YP | Legacy enterprise, pre-LDAP | Deprecated; no encryption in transit; avoid in new deployments |

The local files never go away entirely, even in SSSD environments. Root, system accounts, and emergency access accounts always live in local files — SSSD is for regular users. If your SSSD connection to the directory dies at 3am and you need to log in, the local accounts in `/etc/passwd` are what saves you. Always keep a root password set and accessible out-of-band.

## Trade-offs

**Where this shines:**

- The split between world-readable passwd and root-only shadow cleanly solves the information exposure problem without breaking any of the userspace tools that need UID-to-name mapping
- Password aging fields give system administrators fine-grained control over credential lifecycle without requiring any external tools — it's built into the OS
- The NSS abstraction (`getent`, `nsswitch.conf`) lets the same authentication logic work for local files, LDAP, Active Directory, and Kerberos without the application caring which backend is in use

**Where this costs you:**

- The `!`-locking semantics are subtly broken on distros with `UsePAM yes` as default — an account that looks locked to `passwd -l` is not fully locked, and this surprises almost everyone who discovers it the hard way
- There is no atomic locking for `/etc/passwd` — concurrent writes without `vipw` can corrupt the file, and a corrupted `/etc/passwd` is a lockout event
- The files are plain text; any tool that has read access to `/etc/shadow` (i.e., root) gets every hash at once, making a root compromise immediately a full credential database exfiltration
- Shadow dates are day-count integers — converting them to human-readable dates requires a calculation, which makes `chage -l username` the only sane way to read aging state

## When to Use / When to Avoid

✅ **Use local auth files when:**

- Managing a single server or a small fleet where centralized directory overhead is not worth the complexity
- Creating service accounts for daemons — always local, always should be
- You need emergency access accounts that survive SSSD/LDAP outages
- Running containers — container images use local files; they have no directory service

❌ **Avoid managing users purely via local files when:**

- You have more than a handful of servers and need consistent user access across all of them — that's what SSSD and LDAP are for; don't sync `/etc/passwd` manually across servers
- You need audit trails for user authentication — `/etc/shadow` tells you when a password was last changed, not when the account was last used; for audit trails you need PAM accounting or `pam_tally2` / `pam_faillock`

⚠️ **Common mistakes:**

- Editing `/etc/passwd` or `/etc/shadow` directly with a text editor instead of `vipw` / `vipw -s`
- Assuming `passwd -l` fully locks an account — it only locks password authentication; SSH keys bypass it on default configurations
- Checking for locked accounts with `grep '!:' /etc/shadow` and concluding they can't log in — SSH key auth is not blocked
- Creating service accounts without explicitly disabling login — `useradd myservice` on RHEL leaves the account with `!!` in shadow, which is not the same as `*`; use `useradd -r -s /usr/sbin/nologin myservice` to be explicit

## Veteran Perspective

The most dangerous thing I see junior admins do with user accounts is treating `passwd -l` as a security control. They lock an account because a contractor left, they feel like they've done the right thing, and then six months later that contractor uses their old SSH key to log right in. On any enterprise Linux system with `UsePAM yes` — which is default on Debian, Ubuntu, and RHEL — the `!` prefix in shadow does nothing for SSH key authentication. Nothing. The only reliable way to disable all access to an account is `chage -E 0`, which sets an expiration date that PAM's account stack actually checks.

The second thing I've watched cause real trouble: the UID 0 check. When auditing a compromised system, one of the first things you do is `awk -F: '$3 == 0' /etc/passwd`. I've seen systems where an attacker added a backdoor user with UID 0 and a nonstandard name. The system had no intrusion detection, no file integrity monitoring. The account sat there for months. Root in everything but name. If you're doing any kind of security review — on your own systems or someone else's — this check takes five seconds and has caught real compromises.

The yescrypt vs SHA-512 split is worth understanding even if your organization has standardized on one distro. If you're building automation that installs from a mix of Ubuntu and RHEL, your hashes in `/etc/shadow` will look different. If you're migrating users between systems, you cannot just copy shadow entries across distros when the algorithm differs — the old hash won't work because the algorithm isn't available or isn't enabled. You'll need to force a password reset. I've seen migrations go wrong for exactly this reason.

The GECOS field has bitten teams in surprising ways. Many internal tools parse it naively — they split on the first comma to get the display name, or they use `getpwnam()` and display `pw_gecos` directly. If a user has a comma in their name, or if someone ever put structured data in the GECOS field for automation purposes, tools that parse it can break in subtle ways. The `finger` command expected comma-separated subfields (full name, office, phone, etc.). Most modern systems ignore all but the first subfield. Don't put anything load-bearing in GECOS.

In interviews for senior Linux roles, the auth files question usually comes in one of two forms: "walk me through exactly what happens when a user logs in via SSH" (they want you to trace from key verification through PAM to session establishment, hitting `nsswitch.conf`, `getspnam()`, and `setuid()` in the right order), or "how would you audit user accounts on a compromised system?" For the second, the correct answer starts with: UID 0 check, shadow markers for locked accounts, `lastlog` for recent access, and `/etc/sudoers` for privilege grants. Not just "grep the passwd file."

One more thing that's worth saying clearly: `/etc/shadow` is not encryption. It's hashing. A hash with a weak password is cracked in seconds with modern GPUs and wordlists. The reason you use yescrypt or SHA-512 with high rounds is to make cracking expensive per hash. If an attacker gets root on your system and reads `/etc/shadow`, your users' passwords are effectively compromised unless those passwords are long and random. Shadow protection stops credential theft by unprivileged users. Once root is compromised, the hashes are just arithmetic problems.

## Try It Yourself

1. Run `getent passwd $(whoami)` and identify all seven fields. Then run `sudo getent shadow $(whoami)` and identify the hash algorithm prefix, the salt, and whether any aging fields are set. What's the days-since-epoch value in field 3? Convert it to a date with `date -d "1970-01-01 +N days"`.

2. Run `awk -F: '{print $3, $1}' /etc/passwd | sort -n | head -20` to see the lowest-UID accounts on your system. Identify which are static system accounts (UID 1–99), which are dynamic service accounts (100–999), and which are human users (1000+).

3. Look at `/etc/group` for your own user: `grep $(whoami) /etc/group`. Compare what you see there with the output of `id $(whoami)`. Are all your supplementary groups listed? Now check your primary GID — it's in `/etc/passwd` field 4. Does your primary group appear in `/etc/group` with you as a member, or is the GID just inherited from `/etc/passwd`?

4. Run `sudo awk -F: '$3 == 0' /etc/passwd` to check for UID 0 accounts. Then run `sudo awk -F: '$3 >= 1 && $3 <= 999 && $7 != "/usr/sbin/nologin" && $7 != "/bin/false" && $7 != ""' /etc/passwd` to find any service accounts with unexpected shells.

5. **Challenge (VM only):** Create a test user: `sudo useradd -m testuser && sudo passwd testuser`. Set a password. Then lock it: `sudo passwd -l testuser`. Check the shadow entry to confirm the `!`. Add an SSH key to `/home/testuser/.ssh/authorized_keys`. Verify that password login fails (ssh should reject it) but key login succeeds. Then use `sudo chage -E 0 testuser` and verify that key login now also fails. Run `sudo chage -E -1 testuser` to re-enable the account when done. Clean up with `sudo userdel -r testuser`.

## Summary

- `/etc/passwd` is world-readable by design — it stores usernames, UIDs, home directories, and shells, but the password field always contains the literal `x`, directing the system to look in `/etc/shadow` for the actual hash
- `/etc/shadow` (mode 640, root-readable only) stores the password hash with an algorithm prefix (`$6$` = SHA-512 on RHEL/Rocky, `$y$` = yescrypt on Ubuntu 22.04+/Debian 11+/Fedora 35+), a salt, and six password aging fields measured in days since 1970-01-01
- UID 0 grants root privilege regardless of username — any account with UID 0 in `/etc/passwd` is root from the kernel's perspective; always audit with `awk -F: '$3 == 0' /etc/passwd`
- A `!` prefix in `/etc/shadow` locks password authentication but does NOT block SSH key authentication on distros with `UsePAM yes` (default on Ubuntu, Debian, RHEL); to fully disable an account use `chage -E 0 username`
- Always edit `/etc/passwd` and `/etc/shadow` with `vipw` and `vipw -s`, never with a text editor — these files have no transactional protection and a partial write locks you out of the system
