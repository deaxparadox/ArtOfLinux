## The Problem This Solves

On a single-user machine, every file belongs to one person and one person runs everything. There's no need for boundaries. But the moment a second person sits down at the keyboard — or the moment you run a web server, a database, and a mail daemon on the same machine — you have a problem: how do you prevent the web server from reading the database files? How do you prevent one person's processes from killing another person's processes? How do you let two developers share a project directory without giving each other access to their home directories?

The naive answer is "just use root for everything." The result of that answer is why you have heard the phrase "the server got owned." Root can read anything, write anything, and execute anything. A web server running as root that has one exploitable vulnerability doesn't compromise the web server — it compromises the entire machine.

The Unix answer is identity. Every process runs with a numeric user identity (UID) and a numeric group identity (GID). Every file is owned by a UID and a GID. The kernel's permission checks are nothing more than comparing these numbers. A process with UID 1001 trying to read a file owned by UID 1002 that has mode `600` is simply denied — no policy file, no rule engine, just arithmetic. The power of this model is its simplicity and universality.

User and group management is the work of creating those identities, assigning them, and maintaining them. Get it right and you have a machine where the database daemon can't read the web server's keys, where developers can collaborate without trampling each other's files, and where a compromised service is contained to its own identity. Get it wrong and you have a machine that looks secure until it isn't.

## Mental Model

Think of UIDs and GIDs as employee badge numbers in a building with keycards. The badge number is what the door reader sees — not your name, not your face, just the number. Two people can have the same name, but they cannot have the same badge number without the system treating them as the same person. The badge number is assigned when you join the company (account created), can be transferred to a new person (bad practice, but possible), and when you leave (account deleted) the badge number should stop working — but your old possessions sitting on your desk don't disappear; they still have your number on them.

Groups are like team assignments: your badge number gets you into your office, but you also have a team membership that gets you into the shared lab. You can be on multiple teams simultaneously, but your primary affiliation is the one printed on your badge. The kernel checks these numbers at every door, and the numbers are what matter — not the names you give them.

## Theory

### The Three Identity Files

Linux stores user and group information in three plain-text files in `/etc`. Understanding their formats before touching any management commands is essential — the commands are just structured ways to edit these files, and when things go wrong, you'll be editing them by hand.

**`/etc/passwd`** is the master list of user accounts. It's world-readable — any process on the system can read it. This seems alarming until you realize it contains no passwords (the `x` placeholder is where passwords used to live, before they were moved to a file that isn't world-readable). Every line has seven colon-separated fields:

```text
username:x:UID:GID:GECOS:home_dir:shell
```

- **username** — the human-readable login name, up to 32 characters
- **x** — a literal `x`, meaning "look at `/etc/shadow` for the password hash"
- **UID** — the numeric user identity the kernel actually uses
- **GID** — the primary group GID; this becomes the group of new files the user creates (unless overridden by `setgid` on a directory)
- **GECOS** — the "General Electric Comprehensive Operating System" comment field; holds the user's full name and other contact info; mostly historical baggage
- **home_dir** — the user's home directory path; set as `$HOME` at login
- **shell** — the login shell; if set to `/usr/sbin/nologin` or `/bin/false`, the account exists but interactive login is denied

**`/etc/shadow`** is where the actual password hashes live. It's readable only by root (mode `000` or `640` depending on distro), and it contains aging information alongside the hash:

```text
username:$algorithm$salt$hash:last_change:min:max:warn:inactive:expire:reserved
```

The hash field uses the format `$id$salt$hash` where `$id` identifies the algorithm: `$6$` is SHA-512, `$y$` is yescrypt (modern Debian default), `$2b$` is bcrypt. A `!` or `*` at the start of the hash field means the password is locked — authentication via password will fail, but key-based SSH or `sudo` may still work.

**`/etc/group`** lists groups and their members:

```text
group_name:x:GID:member_list
```

The `x` placeholder follows the same pattern as passwd — group passwords exist but are essentially never used. The member list is comma-separated usernames. Note that a user's primary group (defined in `/etc/passwd`) does not appear in `/etc/group`'s member list for that group — the primary group membership is implicit, not explicit.

### UID Ranges

The kernel doesn't care what number a UID is — 0 is exactly as valid as 1000. The conventions exist because humans need them.

- **UID 0** — root. The only truly special UID in the kernel. A process with UID 0 bypasses most permission checks. There is nothing magic about the name "root" — a user named "toor" with UID 0 has identical privileges. The UID is what matters.
- **UIDs 1–999** — system accounts. These exist to give daemons and services their own identities without requiring root. `www-data` (UID 33 on Debian), `nobody` (UID 65534), `sshd`, `postgres` — these accounts don't correspond to humans and have no login shells or home directories that matter. The systemd convention formalized this range; pre-systemd systems used 1–499 (some older Red Hat systems) or 1–99.
- **UIDs 1000+** — regular human users. The first human account created on a fresh Ubuntu or Fedora install gets UID 1000.
- **UID 65534** — `nobody`. The conventional "no privilege" identity for processes that should have minimal access. Often used by NFS for anonymous mounts.

These conventions are enforced by `/etc/login.defs` (the `UID_MIN`, `UID_MAX`, `SYS_UID_MIN`, `SYS_UID_MAX` values), not by the kernel. The kernel doesn't know or care about "system" vs "regular" — that distinction lives entirely in userspace tooling.

### Supplementary Groups

A user has one primary group (set in `/etc/passwd`) but can belong to many supplementary groups (listed in `/etc/group`). When a user logs in, the kernel builds a credentials structure that includes the primary GID plus all supplementary GIDs. A file with group ownership `docker` and mode `660` is accessible to any user whose credentials include the `docker` GID — regardless of whether `docker` is their primary or supplementary group.

This is how the pattern "add the user to the `sudo` group" works: the user's UID doesn't change, but their credentials now include the `sudo` GID, which the `sudo` binary checks before proceeding.

## How It Works (Mechanics)

### What `useradd` Actually Does

When you run `useradd -m -s /bin/bash alice`, the kernel is barely involved. This is a userspace operation:

1. `useradd` reads `/etc/login.defs` for defaults (UID range, password aging policy, home directory skeleton)
2. It finds the next available UID in the configured range
3. It appends a new line to `/etc/passwd` (using file locking via `lckpwdf()`)
4. It appends a new line to `/etc/shadow` with a locked password entry
5. If `-m` is specified (or the distro default is to create home), it copies `/etc/skel/` into the new home directory and `chown`s the result to the new UID/GID
6. It writes an unlock record to the system log

The file-locking step matters: without `lckpwdf()`, two simultaneous `useradd` calls could corrupt `/etc/passwd`. This is why you should never edit these files directly with a text editor in production — use `vipw` and `vigr` instead, which handle the locking and run a syntax check before saving.

### How the Kernel Resolves a Username

The kernel never looks up `/etc/passwd` directly. When a process calls `getpwnam("alice")`, that call goes through the Name Service Switch (NSS), configured in `/etc/nsswitch.conf`. NSS decides where to look: files (i.e., `/etc/passwd`), LDAP, NIS, systemd-homed, or any combination. In a standalone system with no directory service, `files` is the only source and the lookup is trivial. On enterprise systems, NSS is where `/etc/passwd` entries and Active Directory identities get merged into a single view.

```text
# /etc/nsswitch.conf — typical standalone config
passwd:   files systemd
group:    files systemd
shadow:   files
```

This also explains why `useradd` creating a line in `/etc/passwd` is sufficient on a standalone machine but not enough on a machine enrolled in LDAP — there, users exist in the directory service, and `/etc/passwd` may only hold local system accounts.

### Session Credentials Are Set at Login

When a user logs in, `login`, `sshd`, or `pam` calls `initgroups()`, which reads all groups from `/etc/group` that contain the user, plus the primary GID from `/etc/passwd`, and builds the complete supplementary group list. This list is set once on the process and inherited by all child processes for the duration of the session.

This is the root cause of one of the most common support tickets in Linux administration: you add a user to a group, the user tries to use the new permission, and it doesn't work. Nothing is wrong — the group was added correctly. The user's running session just predates the change. They need to log out and back in so `initgroups()` runs again with the updated `/etc/group`.

## Basic Example

```bash
# Create a user with a home directory and bash shell
sudo useradd -m -s /bin/bash -c "Alice Example" alice

# Set a password
sudo passwd alice

# Verify the account was created
id alice
```

```text
uid=1001(alice) gid=1001(alice) groups=1001(alice)
```

```bash
# View the passwd and shadow entries
grep alice /etc/passwd
```

```text
alice:x:1001:1001:Alice Example:/home/alice:/bin/bash
```

```bash
grep alice /etc/shadow
```

```text
alice:$6$rounds=656000$...:19800:0:99999:7:::
```

The `id` output shows three things: the UID, the primary GID (same number as UID by default — `useradd` creates a matching group), and the list of all groups. Right now Alice is only in her own group.

Run this on a test VM and compare the shadow hash format on your distro — Debian 12 and Ubuntu 22.04+ use `$y$` (yescrypt) instead of `$6$` (SHA-512). Both are secure; the format tells you which algorithm was used.

## Real-World Example

This is how you'd provision a shared developer account on a team server: a service account that can run a deploy script, belongs to the `www-data` and `docker` groups, cannot log in interactively, and has its password aged out.

```bash
# Create a non-login service account for CI/CD deploys
# -r creates it as a system account (UID in system range, no aging defaults)
# -s sets the shell to nologin — SSH or su will be rejected
# -d sets a specific home directory for deploy keys and scripts
sudo useradd -r -s /usr/sbin/nologin -d /opt/deploy -m deployer

# Add to the groups it needs — www-data for web files, docker for container deploys
# -a is mandatory here — without it we'd wipe any existing group memberships
sudo usermod -aG www-data,docker deployer

# Lock the password entirely — key-based auth only, no password fallback
sudo passwd -l deployer

# Confirm the full identity
id deployer
```

```text
uid=998(deployer) gid=998(deployer) groups=998(deployer),33(www-data),999(docker)
```

```bash
# Verify the shell is nologin
grep deployer /etc/passwd
```

```text
deployer:x:998:998::/opt/deploy:/usr/sbin/nologin
```

```bash
# Check the shadow entry — the ! prefix confirms the password is locked
sudo grep deployer /etc/shadow
```

```text
deployer:!:19800::::::
```

Walk through what each decision means. The `-r` flag keeps the UID in the system range (below 1000), which signals to both humans and tooling that this is a service account, not a human user. The `-s /usr/sbin/nologin` shell means any attempt to open an interactive session — `ssh deployer@host`, `su - deployer` — will print "This account is currently not available" and exit. The `passwd -l` step sets the `!` prefix on the shadow hash, disabling password authentication entirely, so even if someone knows the account name, they cannot brute-force a password.

The deploy pipeline authenticates with an SSH key stored in `/opt/deploy/.ssh/authorized_keys`, which gets placed there separately (managed by your provisioning system, not by hand).

Now here is a realistic day-two task: the deployer needs to write to `/var/log/app/`. The directory is owned by `app:applog`. You want to add deployer to the `applog` group without disturbing its existing memberships.

```bash
# Add ONE more group without touching the existing ones
sudo usermod -aG applog deployer

id deployer
```

```text
uid=998(deployer) gid=998(deployer) groups=998(deployer),33(www-data),999(docker),1005(applog)
```

Notice how all three previous groups are still there. That's what `-a` does. Forget it and you'd have just `groups=998(deployer),1005(applog)` — the web and Docker access silently gone.

## ⚠️ Gotchas

### `usermod -G` Without `-a` Silently Wipes All Supplementary Groups

This is the most dangerous command in this entire topic. The mistake is easy because `-G` alone looks right — you're specifying a group — and it succeeds with no error or warning.

> ⚠️ **WARNING:** The following command silently removes the user from every supplementary group they currently belong to. If that user is an admin, you may be locking yourself out. Test the fix (with `-a`) in a VM before running on production.

```bash
# What you meant to do: add alice to docker
# What this actually does: replace alice's entire supplementary group list with just docker
sudo usermod -G docker alice
```

```bash
# Before: alice was in sudo, www-data, docker, developers
# After this command:
id alice
```

```text
uid=1001(alice) gid=1001(alice) groups=1001(alice),999(docker)
```

Every other group — including `sudo` — is gone. There's no warning, no prompt, no undo. The access revocation is immediate for new processes; existing sessions keep their credentials until logout.

The underlying rule: `-G` is a setter, not an appender. It says "the user's supplementary groups ARE these" not "add these to the user's supplementary groups." The `-a` flag changes the semantic to append.

```bash
# The correct version — always use -aG, never bare -G
sudo usermod -aG docker alice
```

If you've already run the bare `-G` and need to recover: get the full list of intended groups from a backup of `/etc/group`, then run `usermod -aG group1,group2,group3 alice` to restore them.

### `userdel` Without `-r` Leaves Orphaned Files Everywhere

```bash
# Delete the user account but forget to remove their home directory
sudo userdel alice
```

The account is gone. The UID is gone. But `/home/alice` still exists, still contains all of alice's files, and is now owned by a UID that no longer has a name. `ls -la /home/` will show the numeric UID instead of a username:

```text
drwxr-xr-x  3 1001  1001  4096 Jun 10 14:23 alice
```

That orphaned home directory is a security risk — if UID 1001 is reassigned to a new user later (which `useradd` will happily do), that new user suddenly owns alice's files. It's also a disk leak on systems where users come and go frequently.

```bash
# The correct version — remove home directory and mail spool with the account
sudo userdel -r alice
```

> ⚠️ **WARNING:** `userdel -r` permanently deletes the home directory. Make sure the user's data has been backed up or transferred before running this. There is no undo.

If alice owned files outside her home directory (in `/var/www/`, `/opt/`, etc.), those are not touched by `userdel -r`. Find and deal with them separately:

```bash
# Find all files owned by the now-deleted UID (use the numeric UID, not username)
find / -uid 1001 -ls 2>/dev/null
```

### New Group Memberships Don't Apply to Running Sessions

You add a user to the `docker` group. You tell them to try it. They get "permission denied." You check the group membership — it's correct. You both scratch your heads.

```bash
sudo usermod -aG docker bob

# Bob immediately tries:
docker ps
```

```text
permission denied while trying to connect to the Docker daemon socket
```

The group was added correctly. Bob's *current* shell session was started before the change, and `initgroups()` ran before `docker` was in the list. His running session doesn't know about the new group.

There's no command that patches a running session's credentials (short of replacing the session's process). The user must log out completely and log back in.

```bash
# Bob logs out, logs back in, then:
id bob
```

```text
uid=1001(bob) gid=1001(bob) groups=1001(bob),999(docker)
```

```bash
docker ps
```

```text
CONTAINER ID   IMAGE   COMMAND   CREATED   STATUS    PORTS   NAMES
```

In a pinch — when a full logout/login isn't possible — `newgrp docker` opens a new shell with the `docker` group active, but only for that shell's child processes. It's a workaround, not a fix.

### `useradd` on Debian/Ubuntu Does Not Create a Home Directory by Default

On RHEL-family systems, `/etc/login.defs` sets `CREATE_HOME yes`, so `useradd alice` creates `/home/alice`. On Debian and Ubuntu, the default is no home directory — `useradd alice` creates the account but leaves the home field pointing at a directory that doesn't exist.

```bash
# On Ubuntu — the home directory does NOT get created:
sudo useradd alice
ls /home/
```

```text
# alice is not here
```

```bash
# The account exists but the home doesn't:
grep alice /etc/passwd
```

```text
alice:x:1001:1001::/home/alice:/bin/sh
```

When alice logs in, her shell will start in `/` (or fail to start) and dotfiles like `.bashrc` won't exist.

```bash
# Always use -m on Debian/Ubuntu to create the home directory:
sudo useradd -m alice
```

Or use `adduser` on Debian/Ubuntu, which does this automatically. The distro difference here has caused more "why doesn't my home directory exist" tickets than I can count — especially when people copy `useradd` commands from RHEL documentation and run them on Ubuntu.

## Distro Notes

| Distro Family | Behavior |
|---|---|
| Debian / Ubuntu | `useradd` does **not** create home directory by default; use `-m`. `adduser` is a Perl wrapper that does everything interactively and creates home by default. `adduser` reads `/etc/adduser.conf`. Default password hash: yescrypt (`$y$`) on Debian 12+/Ubuntu 22.04+. |
| RHEL / Fedora / CentOS / Rocky / AlmaLinux | `useradd` creates home directory by default (`CREATE_HOME yes` in `/etc/login.defs`). `adduser` is a symlink to `useradd` — there is no interactive wrapper. Default hash: SHA-512 (`$6$`). |
| Arch | `useradd` does not create home by default. Uses shadow-utils. Default hash is configurable. `adduser` not installed by default. |
| Alpine (containers) | `adduser` is BusyBox's implementation — entirely different from the Debian Perl script. Very limited flags. Often used in Dockerfiles: `adduser -D -s /bin/sh appuser`. |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `useradd` | Scripts, automation, Ansible, Dockerfiles, any non-interactive context | Low-level; requires explicit flags; consistent across distros |
| `adduser` | Interactive provisioning on Debian/Ubuntu servers | Friendlier defaults; not portable; symlinks to `useradd` on RHEL |
| `usermod` | Modifying an existing account (groups, shell, lock/unlock) | The standard tool; no alternative |
| `vipw` / `vigr` | Directly editing `/etc/passwd` or `/etc/group` | Handles file locking; runs a syntax check before saving; use instead of a bare editor |
| `chpasswd` | Batch password resets from a script | Reads `user:password` pairs from stdin; avoids interactive `passwd` prompts |

Use `useradd` in any automated context — scripts, Ansible, CI/CD, Dockerfiles. It's predictable, portable, and produces no stdin. Use `adduser` only when you're sitting at a terminal on a Debian/Ubuntu machine and want the interactive walkthrough. Never use `adduser` in a script intended to run on both distro families.

## Trade-offs

**Where this model shines:**

- The UID/GID permission model is enforced in the kernel, not in a policy daemon. There's no daemon to crash, no policy file to misconfigure. The kernel checks the numbers on every syscall.
- Service isolation is cheap — one `useradd -r` command and a daemon runs as its own identity, contained. The cost is near zero.
- The flat-file format of `/etc/passwd` and `/etc/group` means you can inspect and repair a broken user database from a rescue shell with nothing but `vi`. There's no service to restart, no schema to query.
- NSS allows transparent layering — local files + LDAP + systemd-homed can coexist without changing a single command or application.

**Where this costs you:**

- UID/GID integers are global to the machine. If UID 1001 is "alice" on one machine and "bob" on another, shared NFS mounts will give the wrong person access to the wrong files. Container orchestration multiplies this problem significantly.
- There's no built-in concept of "temporary access" or "time-bounded group membership." You add someone to a group, you have to remember to remove them. `chage -E` can expire the entire account but not individual group memberships.
- The shadow file format's aging fields are machine-local. Enterprise environments need LDAP or a directory service to centralize and audit access.
- Supplementary group limits: Linux allows up to 65536 supplementary groups per process, but many older tools and configurations assume the NGROUPS_MAX of 32. This matters in organizations with fine-grained group structures.

**Performance note:** NSS lookups for large `/etc/passwd` files are linear scans — if you have 10,000 local accounts in a flat file, every `getpwnam()` call reads up to 10,000 lines. At that scale you need a database-backed name service (LDAP, SSSD) rather than flat files.

## When to Use / When to Avoid

✅ **Use `useradd -r` when:**
- Creating service accounts for daemons (web servers, databases, CI agents)
- You want the UID in the system range (1–999) to signal "not a human account"
- You need the account to have no login shell and no password

✅ **Use `usermod -aG` when:**
- Granting a user access to a device group (`dialout`, `audio`, `video`)
- Adding a user to `sudo` or `wheel`
- Any time you're modifying group membership on an account that already belongs to other groups

✅ **Use `chage` when:**
- Setting password expiry policies for compliance requirements
- Creating a temporary account that must expire on a specific date
- Forcing a password change at next login after a security incident (`chage -d 0 user`)

❌ **Avoid editing `/etc/passwd` directly with a text editor** — use `vipw`, which locks the file and validates it before saving. A syntax error in `/etc/passwd` can make `su`, `login`, and `sshd` fail for all users.

❌ **Avoid reusing UIDs from deleted accounts** — `useradd` will pick the next available UID, which may be a previously deleted account's UID. Files left behind from the old account will immediately be "owned" by the new user. Clean up orphaned files before the UID gets recycled.

❌ **Avoid `adduser` in scripts that need to run on multiple distro families** — it behaves completely differently on Debian/Ubuntu vs RHEL. Use `useradd` with explicit flags for portability.

⚠️ **Common mistakes:**
- Setting `-e` (expiry date) on a service account and forgetting it — the service starts failing on a specific date and nobody can explain why
- Using `passwd -d` (delete password, effectively empty password) instead of `passwd -l` (lock) — the former allows login without a password; the latter prevents it
- Confusing `/etc/group`'s member list with primary group membership — a user's primary group doesn't appear in the member list, which confuses many tools that parse the file naively

## Veteran Perspective

The `usermod -G` without `-a` trap has hit every sysadmin at least once. The insidious part isn't that it fails — it's that it succeeds perfectly, and the damage surfaces later when someone can't access something they've always been able to access. I've seen a developer locked out of `sudo` because someone was "just adding them to the docker group." The fix is easy; the discovery that you need the fix is not.

Here's the deeper lesson: Linux user management commands are mostly wrappers around three text files. The commands abstract the file manipulation, but they don't add safety rails. `usermod -G` doesn't ask "are you sure you want to remove all other group memberships?" because it was designed by people who assumed you knew what you were doing. The kernel-level equivalent — a file permission check — is extremely safe by design. The management tooling around it is not.

The `useradd` vs `adduser` distinction trips up every junior sysadmin who learned on Ubuntu and then gets handed a RHEL machine, or vice versa. The mistake usually shows up in Ansible playbooks or Dockerfiles that were written on one distro and deployed to another. The symptom is a missing home directory or an unexpected interactive prompt that blocks automation. The rule is simple: always use `useradd` in automation, always use explicit flags, never rely on distro defaults.

Password aging via `chage` sounds bureaucratic until you've cleaned up after a breach where a former employee's account was still active because nobody remembered to disable it. The pattern I recommend for service accounts is: `passwd -l` to lock the password, no shell, and `chage -E $(date -d "+90 days" +%Y-%m-%d)` with a calendar reminder to extend or audit. The account will stop working if you forget — and that's better than the alternative.

One thing the man pages genuinely underplay: the UID number is the identity, not the username. You can rename a user with `usermod -l`, and every file they own will still show the old name until you re-run `chown` — but the access permissions haven't changed, because the kernel tracks the number. Conversely, if you delete a user and create a new one who gets the same UID (easy to do if you're not careful), the new user silently inherits all of the old user's files. This UID recycling problem is why cloud and container environments often use very large UID ranges or random UIDs to prevent accidental reuse.

The last thing: `last` and `lastb` are underutilized. Running `last -20` at the start of an incident review tells you who logged in, from where, and when. Running `lastb` tells you who *tried* to log in and failed — which is where you find the brute-force attempts against service accounts that shouldn't have been accessible in the first place. If `lastb` shows hundreds of failed login attempts for `root` or `admin` from external IPs, that's not a user management problem — that's a firewall problem. But you find it here.

## Try It Yourself

1. Run `id` with no arguments. Read every field — UID, GID, groups. Then run `id root`. Notice what's different about root's groups on your system.

2. Create a user named `testuser` with a home directory and a bash shell. Set a password. Log in as that user with `su - testuser`. Verify you land in `/home/testuser` and your prompt changes. Exit and delete the account with `userdel -r testuser`.

3. Create two users (`alice` and `bob`) and a group called `project`. Add both to the group. Create a directory owned by `project` with group-write permission. Log in as each user and verify they can both write to it. (Requires `sudo`.)

4. Use `chage -l` to view the password aging settings for your own account. Then use `chage -l root`. Compare the `Maximum number of days between password change` field — what does `99999` mean and why does root often have it?

5. **Challenge:** Write a shell script that accepts a username as an argument, creates the account as a system service account (no home, no shell, no password, system UID range), adds it to a group named after itself, and prints a summary using `id`. Make it idempotent: running it twice for the same user should not error — it should detect that the account already exists and skip creation. Test it in a VM. (Hint: `getent passwd username` returns non-zero if the user doesn't exist.)

## Summary

- The kernel identifies users by UID and groups by GID — numbers, not names. UIDs 0 is root, 1–999 are system accounts by convention, 1000+ are human users.
- `/etc/passwd` is world-readable and maps UIDs to names and metadata; `/etc/shadow` is root-only and holds the actual password hashes and aging policy.
- Always use `usermod -aG groupname user`, never `usermod -G groupname user` — the bare `-G` replaces all supplementary groups rather than appending; the mistake produces no error.
- New group memberships don't take effect in existing sessions — the user must log out and back in (or use `newgrp` as a workaround) because credentials are set at login time by `initgroups()`.
- On Debian/Ubuntu, `useradd` does not create a home directory by default — use `-m`; on RHEL-family systems it does. Use `useradd` with explicit flags in any script; `adduser` is not portable.
