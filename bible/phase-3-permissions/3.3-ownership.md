# [3.3] — Ownership

> Every file on Linux belongs to exactly one user and one group — and the kernel enforces that ownership at every access check, every time.

## The Problem This Solves

Imagine a server with fifty users. Every file on the system is just bytes on disk. Without some way to associate a file with its creator, any user could read anyone else's private data, overwrite system binaries, and delete configuration files they've never touched. A shell that any user can run has no concept of "yours" or "mine" — it only knows bytes.

The early Unix designers faced this problem in 1969. They needed a multi-user system where process isolation and data protection were enforced by the OS, not by hope or convention. Their solution was to attach two numeric identifiers to every file: the UID (user ID) of the owner, and the GID (group ID) of the owning group. The kernel checks these numbers at every file operation. No exceptions. No overrides — unless you're root.

This sounds simple. The complexity lives in the edges: what group does a new file get? What happens when you change ownership on a directory tree of ten thousand files? What does it mean for a file to change groups? Those are the questions this topic answers.

---

## Mental Model

Think of a filing cabinet in an office. Each drawer has a nameplate on the front: "Alice — Marketing". That nameplate has two parts: a person's name (the owner) and a department (the group). Alice can always open her own drawer. Anyone in the Marketing department can open it if the drawer's lock is set to allow that. Everyone else has to go through the front desk.

The filing cabinet doesn't ask Alice to prove she's Alice every time she opens it — it just reads the nameplate and checks whether the person standing in front of it matches. Reassigning a drawer to Bob means changing the nameplate. Changing the department means calling Facilities. Root is the master key that bypasses the nameplate entirely.

The nameplate is the ownership. The lock settings are the permissions (covered in 3.2). They work together, but they're separate mechanisms.

## Theory

### UIDs and GIDs — numbers, not names

The kernel never works with usernames or group names. It works with numbers. When you log in, the kernel assigns your process a UID (User ID) and a GID (primary Group ID). Every file on disk stores these numbers directly in its inode — not the string "alice", but the number `1001`. The strings you see in `ls -l` are translations done by the C library, which looks them up in `/etc/passwd` and `/etc/group` at display time.

This has a practical consequence: if you delete a user account but don't delete their files, the files still exist with the old UID in their inode. `ls -l` will display the raw UID number because there's no name to translate it to. The files are still owned by that UID — they just have no corresponding username anymore.

### One owner, one group — always

Every file has exactly one owner UID and exactly one group GID. Not two owners. Not a list of groups. One of each. This is one of Unix's foundational constraints.

This constraint is why the group system exists in the first place. A researcher who belongs to both the `lab` group and the `sysadmin` group needs a way to share files with each group without making every file world-readable. The group field on the file is how that works.

### Primary group vs. supplementary groups

Every user has one primary group. It's the GID stored in `/etc/passwd`. When a user creates a file, the new file's group is set to the creator's current primary group — the effective GID of the creating process.

Users can also belong to supplementary groups, listed in `/etc/group`. These give the user read/write/execute access to files owned by those groups, but they do not affect what group gets stamped onto newly created files. The newly created file always gets the effective GID of the creating process — which is normally the primary group.

There is one exception: if a directory has the **setgid bit** set, new files created inside that directory inherit the directory's group instead of the creator's primary group. That behavior is covered in [3.4 — Special Permissions](3.4-special-permissions.md).

### Why regular users can't `chown` to arbitrary users

On Linux, only root can give away file ownership to a different user. This is intentional and non-obvious: it prevents privilege escalation. If regular users could `chown` files to root, they could create a malicious executable, assign it to root, and then engineer a situation where root runs it. The restriction is enforced by the kernel — there's no configuration knob to allow it for non-root users.

For groups, the rule is slightly less strict: a non-root user can change the group of a file they own, but only to a group they're currently a member of. You can't grant access to a group you don't belong to.

### `chown` vs `chgrp`

`chown` can change the owner, the group, or both in a single command. `chgrp` changes only the group and is older than the combined syntax of `chown`. In practice, `chown user:group` is what you'll see in scripts and on servers — `chgrp` survives for cases where you're changing only the group and want to make that intent explicit in a script.

## How It Works (Mechanics)

When you run `chown alice:developers report.pdf`, the following happens:

1. The shell calls the `chown(2)` or `lchown(2)` system call, passing the target path, the new UID, and the new GID.
2. The kernel resolves the path to an inode.
3. The kernel performs a capability check: is the calling process running as root (UID 0), or does it have the `CAP_CHOWN` capability? If not, and the target UID is different from the process's effective UID, the system call returns `EPERM` (permission denied).
4. For group changes without `CAP_CHOWN`: the kernel verifies that the new GID is either the process's effective GID or one of its supplementary GIDs.
5. If all checks pass, the kernel writes the new UID and GID directly into the inode. This updates `ctime` because inode metadata changed.

You can observe the system call with `strace`:

```bash
strace -e trace=chown chown alice report.txt 2>&1 | grep chown
```

```text
chown("report.txt", 1001, -1)           = 0
```

The `-1` for GID means "don't change the group" — which is exactly what `chown user` (no colon) does.

When the `-R` flag is used for recursive operation, `chown` traverses the directory tree with `openat(2)`, `fstatat(2)`, and `fchownat(2)` system calls for each file. There is no atomic "change this whole tree" kernel operation — it's a file-by-file loop in userspace, which means it can be interrupted partway through, leaving a tree with mixed ownership.

## Basic Example

```bash
# Create a test file and examine its default ownership
touch testfile.txt
ls -l testfile.txt
```

```text
-rw-rw-r-- 1 alice alice 0 Jun 15 10:00 testfile.txt
```

```bash
# Change owner to bob, leave group unchanged
sudo chown bob testfile.txt
ls -l testfile.txt
```

```text
-rw-rw-r-- 1 bob alice 0 Jun 15 10:00 testfile.txt
```

```bash
# Change owner and group together
sudo chown bob:developers testfile.txt
ls -l testfile.txt
```

```text
-rw-rw-r-- 1 bob developers 0 Jun 15 10:00 testfile.txt
```

```bash
# Change only the group (two equivalent ways)
sudo chown :marketing testfile.txt
# or
sudo chgrp marketing testfile.txt
ls -l testfile.txt
```

```text
-rw-rw-r-- 1 bob marketing 0 Jun 15 10:00 testfile.txt
```

Look at the third and fourth columns in the `ls -l` output — that's owner and group respectively. Notice that `chown bob` alone left the group as `alice`; only the owner column changed. Now try this yourself. If your output shows numeric IDs instead of names (like `1001` instead of `alice`), the user or group doesn't exist on your system — create it first with `useradd` or substitute usernames that do exist.

## Real-World Example

A web application running as the `www-data` user needs to write to an upload directory, while a deploy script running as the `deploy` user needs to update the application code. A developer named `alice` is doing the initial server setup. Here's how she'd configure ownership correctly:

```bash
# Application code: owned by deploy user, readable by www-data via group
sudo chown -R deploy:www-data /var/www/myapp
# Directories need execute to traverse; files don't need execute
sudo find /var/www/myapp -type d -exec chmod 750 {} \;
sudo find /var/www/myapp -type f -exec chmod 640 {} \;

# Upload directory: www-data needs to write here, deploy needs to read uploads
sudo chown www-data:deploy /var/www/myapp/uploads
sudo chmod 750 /var/www/myapp/uploads

# Verify the result
ls -la /var/www/myapp/
```

```text
total 24
drwxr-x--- 5 deploy   www-data 4096 Jun 15 10:00 .
drwxr-xr-x 3 root     root     4096 Jun 15 09:55 ..
drwxr-x--- 2 deploy   www-data 4096 Jun 15 10:00 config
drwxr-x--- 2 deploy   www-data 4096 Jun 15 10:00 public
drwxr-x--- 2 www-data deploy   4096 Jun 15 10:00 uploads
```

The `-R` flag on `chown` changes ownership recursively through the entire directory tree in a single pass. The `find` commands handle permissions separately — because a blanket `chmod -R 750` would apply execute bits to regular files too, which is wrong. A senior sysadmin would see this pattern and immediately understand the intention: `deploy` owns the code, `www-data` can read it, neither can access the other's primary domain without permission on the specific directory that bridges them.

Notice that the `uploads` directory has the ownership flipped: `www-data` owns it (can write into it), and `deploy` is the group (can read what was uploaded). This is a deliberate design decision, not an accident.

## ⚠️ Gotchas

### `chown user:` (trailing colon) also changes the group

Most people know `chown user:group` changes both. Most people know `chown user` changes only the owner. Almost nobody remembers what `chown user:` (trailing colon, no group name) does — and it's different from both.

```bash
# alice's primary group is "developers"
# This file is currently owned by alice:marketing
ls -l project.conf
```

```text
-rw-r--r-- 1 alice marketing 512 Jun 15 10:00 project.conf
```

```bash
# Looks like you're just reassigning the owner...
chown alice: project.conf
ls -l project.conf
```

```text
-rw-r--r-- 1 alice developers 512 Jun 15 10:00 project.conf
```

The group changed too — from `marketing` to `developers`. The trailing colon instructs `chown` to set the group to the owner's primary login group. This is documented in the man page but routinely overlooked. If you want to change only the owner and preserve the group, use `chown alice` (no colon at all).

### `chown -R` on the wrong path — the catastrophic version

> ⚠️ **WARNING:** Running `chown -R root:root /` will attempt to recursively change the ownership of every single file on the system. This will immediately break sudo, SSH keys, cron jobs, setuid binaries, home directory files, and most running services. Recovery usually requires booting from external media. Do not run this command on a live system.

The mistake is always a typo or a copy-paste error. Someone means to run `chown -R root:root /var/www/app` and accidentally runs `chown -R root:root /` by leaving a space before the path or having the path variable empty.

```bash
# This is the mistake pattern — NEVER run this
TARGET_DIR=""  # variable is empty due to a bug
sudo chown -R root:root $TARGET_DIR/  # expands to: chown -R root:root /
```

The correct approach: always verify the variable is non-empty before running recursive ownership changes.

```bash
# Safe pattern for scripts that use variables in paths
TARGET_DIR="/var/www/app"
if [[ -z "$TARGET_DIR" ]]; then
    echo "ERROR: TARGET_DIR is empty. Aborting." >&2
    exit 1
fi
sudo chown -R www-data:www-data "$TARGET_DIR"
```

The same class of mistake applies to `/etc`, `/home`, and `/usr` — any system directory where wrong ownership causes subtle breakage that may not manifest until hours later (the next login, the next cron run, the next service restart).

### Recursive `chown` on a symlink — `-h` changes the link, not the target

By default, `chown` on a symlink changes the ownership of the **target** that the link points to, not the symlink file itself. On most Linux filesystems, symbolic links have their own inode with their own UID/GID, but this ownership is largely irrelevant — what matters for access is the target's ownership. Still, if you're auditing file ownership and you're not getting what you expect:

```bash
ls -la /var/www/app/config
```

```text
lrwxrwxrwx 1 root root 22 Jun 15 10:00 config -> /etc/myapp/config.d
```

```bash
# This changes /etc/myapp/config.d ownership, not the symlink
sudo chown www-data:www-data /var/www/app/config

# This changes the symlink itself
sudo chown -h www-data:www-data /var/www/app/config
```

In practice, you almost never need `-h`. If you're doing `chown -R` on a directory tree and you want symlinks inside it to be updated rather than their targets, add `-h`. The combination of `-R` and `-h` is the only common production scenario where this flag matters.

### A user can only give away a file to their own groups

If you're logged in as `alice` and `alice` is a member of `developers` and `qa`, you can change the group of your own file to `developers` or `qa`. You cannot change it to `marketing` or `ops` or any other group you don't belong to.

```bash
# alice is NOT a member of "finance"
chown :finance report.pdf
```

```text
chown: changing group of 'report.pdf': Operation not permitted
```

This trips people up when they're trying to share a file with a team. The solution is not to fight the permission system — it's to add yourself to the relevant group via `usermod -aG groupname username` (requires root), then log out and back in, or use `newgrp` to activate the new group membership in the current session.

### `newgrp` changes your active group in the current session only

`newgrp` lets you switch your effective primary group without logging out. It works by spawning a new child shell with the specified group as your effective GID. Files you create in that shell will be owned by the new group.

```bash
id
```

```text
uid=1001(alice) gid=1001(alice) groups=1001(alice),1002(developers),1003(qa)
```

```bash
newgrp developers
id
```

```text
uid=1001(alice) gid=1002(developers) groups=1001(alice),1002(developers),1003(qa)
```

```bash
touch newfile.txt
ls -l newfile.txt
```

```text
-rw-rw-r-- 1 alice developers 0 Jun 15 10:00 newfile.txt
```

When you exit the `newgrp` shell (`exit` or Ctrl-D), you return to your original shell with your original primary group. The group change does not persist. It does not affect other open terminals. It is strictly scoped to the child shell spawned by `newgrp`.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `chown`, `chgrp` are from GNU coreutils, installed by default. `/etc/passwd` and `/etc/group` for local users. |
| RHEL / Fedora / CentOS / Rocky | Same GNU coreutils. In environments using SSSD or LDAP, UIDs/GIDs may be resolved from a remote directory — `id username` may return different results than `/etc/passwd` alone. |
| Arch | Same GNU coreutils. Same behavior. |
| Alpine (containers) | Uses BusyBox `chown`, which supports most of the same flags. `-h` for symlinks works. Some long options (`--no-dereference`) may not be available. |
| macOS (reference) | BSD `chown` — syntax is identical for basic use. Some GNU long flags differ. |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `chown user:group` | Changing owner and group together | The go-to for any ownership change |
| `chown :group` | Changing group only, using `chown` syntax | Equivalent to `chgrp` — use whichever is clearer in context |
| `chgrp group` | Changing group only | Explicit — makes intent clear in a script; slightly older idiom |
| `newgrp group` | Temporarily switching active group in current shell | Session-scoped only; spawns a child shell |
| `sg group -c "command"` | Running a single command with a different effective group | Doesn't spawn an interactive shell; useful in scripts |

`chgrp` and `chown :group` are interchangeable — pick one and be consistent. In scripts, `chgrp` reads clearly as "this is a group change." In one-liners at the terminal, `chown :group` saves a syllable. `sg` is underused and worth knowing: when you need a single command to run with a different group but don't want to spawn an interactive `newgrp` shell, `sg group -c "command"` does it cleanly.

## Trade-offs

**Where this shines:**

- The UID/GID model is enforced at the kernel level — it's not a suggestion or a convention. There is no way for userspace code to bypass it without root.
- Ownership is cheap to check. The kernel reads two numbers from the inode and compares them to two numbers in the process credential structure. This is O(1) and happens billions of times per day on a busy server.
- The model is universal. Every Unix-like system from 1970 to today has UIDs and GIDs. Scripts written for Linux run on macOS, BSD, and AIX with no changes to ownership logic.
- Recursive ownership changes are scriptable and auditable — `chown -R` is predictable and produces verifiable results with `find -uid` or `ls -lR`.

**Where this costs you:**

- The one-owner, one-group model is genuinely restrictive. A file can only have one owning group. If a file needs to be writable by members of both `developers` and `ops`, you have to create a third group that contains members of both, or use ACLs (which live outside this model entirely, covered in a later topic).
- Recursive `chown -R` on a large directory tree is not atomic. A process can be killed partway through, leaving inconsistent ownership. There is no "transaction" concept.
- UIDs and GIDs are per-system numbers. In distributed environments (NFS, shared storage, containers), a UID of `1001` on one host may mean `alice` and on another host may mean `charlie`. NFS has partially solved this with NFSv4 name-to-ID mapping, but container environments with shared volumes still hit this problem regularly.

## When to Use / When to Avoid

✅ **Use `chown user:group` when:**
- Deploying application files that need to be run or read by a service account
- Handing off a file from your account to another user or service after creating it as root during setup
- Setting up a shared project directory where multiple users need to collaborate via a shared group

✅ **Use `chown -R` when:**
- Deploying a fresh application to a server and setting ownership in bulk
- Restoring files from a backup that came from a different system with different UIDs
- Fixing a directory tree that ended up with mixed ownership after an interrupted deploy

❌ **Avoid `chown -R` when:**
- The path is derived from a variable or user input without validation — always guard against empty or unexpected values
- The target contains symlinks pointing outside the tree — you'll change ownership of files you didn't intend to touch
- You're in a container or shared-volume environment with UID mismatches — fix the root cause (consistent UIDs) rather than patching with recursive chown

❌ **Avoid `newgrp` when:**
- You need the group change to persist — log out and back in, or have an admin add you to the group properly
- You're writing a script — the child shell created by `newgrp` has its own environment; use `sg group -c "command"` instead

⚠️ **Common mistakes:**
- Running `chown` to "fix permissions" when the real problem is the permission bits — ownership and permissions are separate; check both with `ls -l`
- Forgetting that `chown user:` (trailing colon) changes the group too — if you want only the owner changed, use `chown user` with no colon
- Changing ownership of files in `/etc` when you mean to change ownership of application config — always double-check your path

## Veteran Perspective

The most common ownership mistake I see in production isn't the catastrophic `chown -R root:root /` — that's rare and immediately obvious. The common one is more subtle: developers who deploy their application files as root and never change ownership to the service account. The app starts. It works. For six months, it works. Then they add a log rotation script that runs as `www-data`, and the log directory is owned by root and not group-writable, and now logs stop rotating, disk fills up, and the app crashes at 3am. The root cause was a `chown` that never got run during the initial deploy.

The intermediate mistake is misunderstanding what `newgrp` actually does. I've watched engineers spend an hour debugging why their `newgrp developers` session isn't affecting their original terminal, or why files created after `newgrp` still show their old primary group in a different shell. `newgrp` spawns a child shell. Full stop. When you `exit` that shell, you're back where you started. If you need the group change to survive a logout, you need `usermod -aG groupname username` and a fresh login.

The deeper principle is one of the oldest in Unix: trust is never implicit. The kernel does not trust usernames — it trusts numbers. It does not trust that a UID in a tarball corresponds to a valid user on the current system. It does not trust that the UID you had last week is still the right one. When you extract a tarball as root and the extracted files have UID 1003, those files are now owned by whatever user currently has UID 1003 on this system — which might not be who you intended. `tar --no-same-owner` exists for this reason.

In interviews, ownership questions tend to be practical: "Why can't a non-root user `chown` a file to another user?" Know the privilege escalation answer cold. The deeper follow-up is: "What capability would you need?" The answer is `CAP_CHOWN`. If you can name the specific Linux capability rather than just saying "you need to be root," that's the answer that shows you understand the architecture rather than just the surface behavior.

A pattern I still use today on any server I set up: after deploying application files, I always run `find /path/to/app -not -user appuser -o -not -group appgroup | head -20` as a quick sanity check before the app starts. If that find returns anything, something is wrong. It takes two seconds and it's caught more deployment errors than any checklist.

## Try It Yourself

1. Create a file in your home directory and run `ls -l` on it. Identify the owner and group columns. Confirm both match your current username. Run `id` and find your primary group — is it the same as the file's group?

2. Create a file, then run `sudo chown root yourfile`. Run `ls -l` again. Now try to run `chown $USER yourfile` (without sudo). What error do you get, and why?

3. Run `id` to see your supplementary groups. Try to `chown :groupname yourfile` where `groupname` is a group you belong to. Then try it with a group you don't belong to. Compare the results.

4. Run `newgrp` with a group you belong to (use `id` to find one). Create a file. Exit the `newgrp` shell and create another file with the same name in the same location. Run `ls -l` on both. Do they have the same group?

5. **Challenge (run in a VM):** Create a test directory structure — `/tmp/ownership-lab/` with three subdirectories: `shared/`, `web/`, and `uploads/`. Create two test users (`testuser1` and `testuser2`) and a group `testteam`. Make both users members of `testteam`. Set ownership such that: `shared/` is writable by any member of `testteam`, `web/` is owned by `testuser1` and readable (but not writable) by `testteam`, and `uploads/` is writable by `testuser2` but readable by `testteam`. Verify by switching to each user with `su` and attempting to create/read files in each directory.

## Summary

- Every file has exactly one owner UID and one group GID, stored directly in the inode; the kernel compares these numbers to process credentials at every access check.
- `chown user:group` changes both; `chown user` changes only the owner (group unchanged); `chown :group` or `chgrp` changes only the group; `chown user:` (trailing colon, no group name) changes the owner AND sets the group to the owner's primary login group.
- New files get the creator's current primary group by default — supplementary groups do not affect this unless the directory has the setgid bit set (covered in [3.4 — Special Permissions](3.4-special-permissions.md)).
- Only root can change file ownership to a different user; a regular user can only change a file's group to a group they belong to.
- `chown -R` is not atomic and operates file-by-file; always validate the target path — especially when it comes from a variable — before running it.
