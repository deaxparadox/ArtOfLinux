## The Problem This Solves

The standard Unix permission model gives every file exactly three permission slots: one for the owner, one for a single group, and one for everyone else. For decades, that was enough.

Then real organizations started using shared servers. Consider a web project with three teams: the developers who need read-write access, the designers who need read-write access on the assets directory but only read on the code, and an external auditor who needs read-only access to specific directories without belonging to any of the internal groups. The basic model breaks immediately.

```bash
# The file is owned by user "deploy", group "webteam"
ls -l /srv/webapp/config.yml
```

```text
-rw-r----- 1 deploy webteam 4210 Jan 15 09:23 /srv/webapp/config.yml
```

With standard permissions you have exactly three choices: give the auditor full group access (which means they get write access too), add them as a secondary group member with no way to limit which files in the group they can access, or use a world-readable scheme that exposes the file to everyone. None of these are right.

What you actually need is the ability to say: "alice gets rwx, bob gets r-x, the auditor group gets r--, and the owning group still has its own permissions." That's precisely what POSIX ACLs provide — an arbitrary number of per-user and per-group permission entries on a single file.

---

## Mental Model

Think of a building's key card access system. Every employee swipes their card, and the system checks a specific list for that person: Alice is authorized for floors 2, 3, and 4. The accounting team is authorized for floor 5. Visitors are authorized for the lobby only. The building owner can always enter everywhere.

The key card reader doesn't care that there are only three slots in the old-fashioned permission model. It checks the access list for each person specifically. When you update one person's access, you change only their entry — everyone else's access is unaffected.

ACLs are exactly that. Instead of three fixed permission slots, the kernel maintains an ordered list of (who, permissions) entries on each file. The check walks the list in a defined order and grants or denies based on the first matching entry.

## Theory

### ACLs Are a POSIX Standard, Not a Linux Invention

POSIX ACLs (formally defined in POSIX.1e, later withdrawn but widely implemented) extend the basic DAC model without replacing it. Every file with an ACL still has the traditional `rwxrwxrwx` bits — the ACL adds entries on top. Tools that don't understand ACLs still see and use the traditional bits; tools that do understand ACLs use the full list.

The three traditional entries — owner, owning group, and others — are also present in the ACL as mandatory base entries. The ACL cannot exist without them.

### The ACL Entry Types

Every file's ACL consists of entries in this format: `type:qualifier:permissions`. There are exactly four entry types:

**`user::`** — The file owner (no qualifier). This is always the same as the `rwx` bits for the owner in `ls -l`.

**`user:name:`** — A named user. This is the power of ACLs: any user on the system, by name or UID, can be given specific permissions on this file regardless of group membership.

**`group::`** — The owning group (no qualifier). This entry's effective permissions are limited by the mask (explained below).

**`group:name:`** — A named group. Any group on the system can be given specific permissions.

**`mask::`** — The effective rights mask. This single entry governs the maximum effective permissions for all named users, the owning group, and all named groups. The file owner and others are never affected by the mask.

**`other::`** — Everyone not matched by any other entry.

The check order is: owner first, then named users, then owning group and named groups, then others. The first match wins. Root bypasses the ACL check entirely (DAC does not apply to root).

### The Mask Entry — The Part Everyone Gets Wrong

The mask is the most misunderstood piece of the POSIX ACL design. It is not optional and it is not cosmetic.

The effective permission for a named user, the owning group, or a named group is computed as:

```text
effective = entry_permissions AND mask_permissions
```

If alice's ACL entry says `rwx` but the mask says `r--`, alice's effective permission is `r--`. Her entry still shows `rwx`, but `getfacl` will annotate it with `#effective:r--` to tell you that the mask is limiting what she can actually do.

The mask exists because `chmod` must remain able to reason about file permissions without knowing anything about ACLs. When you run `chmod g-w` on a file, the kernel applies the change to the mask — not to the owning group entry — because the group permission bits in `ls -l` display the mask value, not the owning group's entry. This design preserved backward compatibility: every chmod, every script that sets permissions, every tool that uses the traditional three slots continues to work exactly as before, and the mask faithfully tracks what the group bits mean.

The mask is auto-recalculated every time you add or modify a named user or named group entry. The new mask is the bitwise union (OR) of all permissions held by the owning group, all named users, and all named groups. This means adding `alice:rwx` to a file that previously had a `r--` mask will widen the mask to `rwx`.

### How the `+` Flag Signals an ACL

When a file has extended ACL entries (entries beyond the three base entries), `ls -l` appends a `+` character to the permission string:

```text
-rw-r-x---+ 1 deploy webteam 4210 Jan 15 09:23 /srv/webapp/config.yml
```

The group bits (`r-x` above) display the current mask value. The actual owning group entry may be different. Use `getfacl` to see the truth.

### Default ACLs — Inheritance on Directories

Directories can carry a second ACL called the default ACL. When a new file is created inside a directory that has a default ACL, the new file's access ACL is automatically set from the directory's default ACL. New subdirectories inherit both the default ACL as their access ACL and a copy of the default ACL as their own default ACL — propagating the behavior down the tree.

Default ACLs do nothing to files that already exist. They are future-looking only.

## How It Works (Mechanics)

### Where ACLs Are Stored

POSIX ACLs are stored as extended attributes in the `system.` namespace. The access ACL lives in `system.posix_acl_access` and the default ACL in `system.posix_acl_default`. These are binary-encoded on disk and managed through the kernel's VFS ACL layer, not directly via `getfattr`/`setfattr`.

When a process opens a file, the kernel calls `get_acl()` in the VFS layer, which retrieves the encoded ACL from the inode's xattr storage. The permission check then walks the ACL entries in defined order until a match is found.

### The Permission Check Walk

The kernel's POSIX ACL check in `fs/posix_acl.c` follows this exact sequence when a process requests access to a file:

```text
# Conceptual trace — not a real command
1. Is process EUID == file owner UID?   → apply user:: entry, STOP
2. Is process EUID in any ACL user:name: entry?  → apply that entry AND mask, STOP
3. Does process EGID or any supplementary GID match owning group or any group:name: entry?
   → apply the best-matching group entry AND mask, STOP
4. Fall through to other:: entry, apply it directly (mask does NOT apply), STOP
```

Steps 2 and 3 are masked. Steps 1 and 4 are not. The file owner always gets exactly what the `user::` entry says. Everyone in the "others" bucket always gets exactly what `other::` says. Everything in between is subject to the mask.

### Filesystem Support

ACLs require both kernel support (compiled in or as a module) and filesystem support:

- **ext4**: ACL support is on by default since kernel 2.6.39. No mount option needed. `tune2fs -l /dev/sdX | grep "Default mount options"` will show `acl user_xattr`.
- **xfs**: ACL support is hardcoded on; XFS does not accept an `acl` mount option (specifying it causes a mount error on RHEL 7.4+).
- **btrfs**: ACL support enabled by default at mount time.
- **tmpfs**: Requires `CONFIG_TMPFS_POSIX_ACL=y` at kernel build time; most mainstream distros enable this (ALSA requires it for `/dev/snd/`).
- **NFSv4**: Uses NFSv4 ACLs (a different model — ALLOW/DENY ACEs with inheritance flags). Managed via `nfs4_getfacl`/`nfs4_setfacl` from the `nfs4-acl-tools` package; POSIX `getfacl`/`setfacl` do not interoperate correctly with NFSv4 ACLs.

## Basic Example

```bash
# Create a test file and demonstrate the full ACL lifecycle
mkdir /tmp/acltest
touch /tmp/acltest/report.txt
chmod 640 /tmp/acltest/report.txt

# View the default ACL (no extended entries yet)
getfacl /tmp/acltest/report.txt
```

```text
# file: tmp/acltest/report.txt
# owner: you
# group: yourgroup
user::rw-
group::r--
other::---
```

```bash
# Grant a specific user read-write access
setfacl -m u:nobody:rw- /tmp/acltest/report.txt

# View the ACL again — note the new entry and the mask
getfacl /tmp/acltest/report.txt
```

```text
# file: tmp/acltest/report.txt
# owner: you
# group: yourgroup
user::rw-
user:nobody:rw-
group::r--
mask::rw-
other::---
```

```bash
# ls -l now shows the + indicator
ls -l /tmp/acltest/report.txt
```

```text
-rw-rw----+ 1 you yourgroup 0 Jan 15 09:23 /tmp/acltest/report.txt
```

```bash
# Remove nobody's specific entry
setfacl -x u:nobody /tmp/acltest/report.txt

# Strip all extended ACL entries at once
setfacl -b /tmp/acltest/report.txt

# Confirm ACL is back to base entries only
getfacl /tmp/acltest/report.txt
```

```text
# file: tmp/acltest/report.txt
# owner: you
# group: yourgroup
user::rw-
group::r--
other::---
```

Look at the group permission column in `ls -l` after you add `nobody:rw-` — it changed from `r--` to `rw-`. That's the mask, not the owning group's entry. Now try this yourself. If you get `Operation not permitted` when running `setfacl`, your filesystem may not have ACL support enabled — try it in `/tmp` first, which is typically on tmpfs with ACL support on modern distros.

## Real-World Example

A shared project directory where the `devteam` group needs full access, an external contractor `casey` needs read-only access, and all new files created in the directory should inherit this policy automatically.

```bash
# Create the shared directory, set ownership to the project group
sudo mkdir /srv/project
sudo chown deploy:devteam /srv/project
sudo chmod 2770 /srv/project   # setgid bit so new files inherit group

# Grant the devteam group full access (they already have it via group::)
# but also grant casey specific read-execute access
sudo setfacl -m g:devteam:rwx /srv/project
sudo setfacl -m u:casey:r-x /srv/project

# Set default ACLs so every new file/directory inherits this policy
sudo setfacl -d -m u::rwx /srv/project       # owner gets rwx by default
sudo setfacl -d -m g:devteam:rwx /srv/project
sudo setfacl -d -m u:casey:r-x /srv/project
sudo setfacl -d -m o::--- /srv/project       # others get nothing

# Verify the full ACL including defaults
getfacl /srv/project
```

```text
# file: srv/project
# owner: deploy
# group: devteam
# flags: -s-
user::rwx
user:casey:r-x
group::rwx
group:devteam:rwx
mask::rwx
other::---
default:user::rwx
default:user:casey:r-x
default:group::rwx
default:group:devteam:rwx
default:mask::rwx
default:other::---
```

```bash
# Create a new file inside — watch what ACL it gets
sudo -u deploy touch /srv/project/new_report.txt
getfacl /srv/project/new_report.txt
```

```text
# file: srv/project/new_report.txt
# owner: deploy
# group: devteam
user::rw-
user:casey:r-x          #effective:r--
group::rwx              #effective:rw-
group:devteam:rwx       #effective:rw-
mask::rw-
other::---
```

Notice that the new file inherited the default ACL, but new files typically get created with `0666` before umask — the execute bits from the default ACL are stripped because the creating process's umask applied. The `#effective:` annotations on casey and the groups are the mask limiting execute access on this regular file. That's correct and expected behavior. Casey can still read the file; she just can't execute it (which is correct — it's a text file, not a script).

Now walk through what an auditor or on-call engineer should know about this setup: `getfacl` shows the full truth. The `#effective:` annotations tell you immediately if the mask is silently suppressing permissions. If you see an entry like `user:casey:rwx #effective:r--`, something has tightened the mask — almost always a `chmod` call somewhere in a deployment script.

To back up and restore this ACL policy:

```bash
# Save ACLs for the entire tree
getfacl -R /srv/project > /backup/project_acls.txt

# Restore ACLs from the backup
setfacl --restore=/backup/project_acls.txt
```

## Extended Attributes and `chattr` / `lsattr`

ACLs are one specific use of the kernel's extended attribute (xattr) infrastructure. Since these tools are closely related and often confused with each other, they belong in the same topic.

### Extended Attributes (`getfattr` / `setfattr`)

Extended attributes attach arbitrary key-value metadata to files at the filesystem level. The key is a namespaced string; the value is a byte sequence up to 64 KiB. There are four namespaces, each with different access rules:

**`user.`** — The only namespace available to normal users. Requires read permission to read, write permission to write. Limited to regular files and directories (not symlinks, devices, or FIFOs). Used for application metadata: MIME types, download URLs, checksums, comments.

```bash
# Set a user namespace attribute
setfattr -n user.comment -v "reviewed by ops 2025-01-15" /srv/project/config.yml

# Read it back
getfattr -n user.comment /srv/project/config.yml
```

```text
# file: srv/project/config.yml
user.comment="reviewed by ops 2025-01-15"
```

```bash
# Dump ALL user.* attributes on a file (default behavior of -d)
getfattr -d /srv/project/config.yml

# Dump attributes from ALL namespaces (not just user.*)
getfattr --match=- -d /srv/project/config.yml

# Remove an attribute
setfattr -x user.comment /srv/project/config.yml
```

**`security.`** — Used by kernel security modules. `security.selinux` holds the SELinux label; `security.capability` holds VFS file capabilities. You can read these with `getfattr -n security.selinux filename` if you have access, but you manage them through `chcon`, `restorecon`, `setcap`, and `getcap` — not `setfattr` directly.

**`trusted.`** — Requires `CAP_SYS_ADMIN` to read or write. Used by privileged kernel mechanisms. Invisible to normal users even with `getfattr -d`.

**`system.`** — Managed entirely by the kernel. This is where ACLs live: `system.posix_acl_access` and `system.posix_acl_default`. You do not touch these directly; `setfacl` manages them.

### `chattr` / `lsattr` — Filesystem-Level File Flags

`chattr` sets per-inode flags at the filesystem level — not permission bits, not ACL entries, but flags in the inode itself that tell the kernel how to handle the file's data. `lsattr` reads them.

```bash
# View flags on all files in the current directory
lsattr /etc/passwd /etc/shadow
```

```text
----i--------e-- /etc/passwd
----i--------e-- /etc/shadow
```

The flag letters appear in fixed column positions. `e` means extent format (normal on ext4); `i` means immutable.

**`+i` — Immutable**

The immutable flag is the nuclear option for protecting a file. When set, the file cannot be modified, renamed, deleted, or hard-linked — by anyone, including root. Even `rm -f` fails with `Operation not permitted`.

> ⚠️ **WARNING:** `chattr +i` prevents deletion even by root. If you set this on the wrong file, the only way out is `chattr -i` first. If you set it on a file inside a running system directory, you may need to boot from rescue media to clear it.

```bash
# Protect a critical config file
sudo chattr +i /etc/resolv.conf

# Verify
lsattr /etc/resolv.conf
```

```text
----i--------e-- /etc/resolv.conf
```

```bash
# Try to overwrite it — fails even as root
sudo bash -c "echo 'nameserver 1.1.1.1' > /etc/resolv.conf"
```

```text
bash: /etc/resolv.conf: Operation not permitted
```

```bash
# To remove the protection
sudo chattr -i /etc/resolv.conf
```

The check happens in the VFS layer before DAC permission checks. Root's DAC bypass does not apply here. You need `CAP_LINUX_IMMUTABLE` to set or clear `+i`.

**`+a` — Append-Only**

The file can only be opened with `O_APPEND`. Direct writes, overwrites, and truncation are blocked. Deletion and renaming are also blocked. The intended use is log files: a compromised process that writes to the log can keep appending, but it cannot overwrite or erase existing log content. Requires `CAP_LINUX_IMMUTABLE` to set or clear.

```bash
sudo chattr +a /var/log/app/audit.log
```

**`+e` — Extent Format**

Informational only on ext4. Means the file uses the modern extent-based block mapping rather than the legacy indirect block tree. All new files on ext4 get this flag automatically; you cannot clear it. If you see it in `lsattr` output, it means nothing is wrong — it's normal.

**`+C` — No Copy-on-Write (btrfs only)**

Disables btrfs's copy-on-write semantics for this file. Use for databases and VM disk images where COW causes write amplification and fragmentation. Side effect: disables per-file checksums, so btrfs silent corruption detection is lost for this file. Can only be set on empty files. Ignored by other filesystems.

**A note on `+s` (Secure Delete)**

`chattr +s` appears to mark a file for secure deletion (zero-on-delete). It does not. This flag is **not implemented** in ext2, ext3, ext4, or any other mainline Linux filesystem. The kernel stores the bit but does nothing with it at deletion time. For actual secure deletion, use `shred`, full-disk encryption (LUKS), or hardware-level sanitize commands. Relying on `+s` for security is a false sense of protection.

## ⚠️ Gotchas

### The Mask Silently Limits Permissions You Thought You Granted

This is the trap that bites everyone on first contact with ACLs.

You add alice to a file with full permissions:

```bash
setfacl -m u:alice:rwx /srv/data/report.txt
```

Alice reports she can't write to the file. You check:

```bash
getfacl /srv/data/report.txt
```

```text
# file: srv/data/report.txt
# owner: deploy
# group: webteam
user::rw-
user:alice:rwx          #effective:r--
group::r--
mask::r--
other::---
```

Alice's entry says `rwx`. The mask says `r--`. Her effective permission is `r--`. The `#effective:r--` annotation tells you this in the `getfacl` output, but it's easy to miss.

The mask got set to `r--` from the owning group's `r--` entry when the file was first given ACLs. Every time you run `setfacl -m` without `-n`, setfacl recalculates the mask as the union of all masked entries. In this case, adding alice's `rwx` should have widened the mask to `rwx` — but something narrowed it back down afterward. The usual culprit is a `chmod` call.

When `chmod` modifies the group bits on a file with ACLs, it modifies the mask, not the owning group entry:

```bash
chmod g-w /srv/data/report.txt
# This sets mask::r-- regardless of what named user entries say
```

The fix is to set the mask explicitly, or re-add alice's entry (which triggers mask recalculation):

```bash
setfacl -m u:alice:rwx /srv/data/report.txt
# setfacl auto-recalculates mask to rwx (union of group::r-- and alice::rwx)
```

Or set the mask explicitly:

```bash
setfacl -m m::rwx /srv/data/report.txt
# Sets mask to rwx; alice's effective permission becomes rwx
```

The underlying rule: `chmod` and ACLs share the group permission bits. `chmod g-w` reads as "set the mask write bit to 0." If you mix `chmod` and `setfacl` on the same files, always check `getfacl` afterward.

### `cp` Does Not Copy ACLs by Default

You back up a directory, restore it, and the permissions look right — `ls -l` shows the same bits. But the ACLs are gone.

```bash
# ACLs are silently dropped here
cp /srv/project/config.yml /backup/config.yml.bak
```

`cp` without flags preserves nothing beyond basic mode and timestamps in a best-effort way. Specifically, `cp -p` preserves mode, ownership, and timestamps, and makes a best-effort attempt at POSIX ACLs via `libacl` — but this is system-dependent and not guaranteed.

`cp -a` is the reliable option. It is equivalent to `cp -dR --preserve=all`, which includes all extended attributes and ACLs:

```bash
# Reliable: -a copies ACLs, xattrs, ownership, symlinks, and recursion
cp -a /srv/project/ /backup/project/
```

Note: `cp --preserve=acl` is not a valid flag. The valid `--preserve=` keywords are `mode`, `ownership`, `timestamps`, `links`, `context`, `xattr`, and `all`. Using `acl` as a keyword will error out.

### `rsync` Does Not Preserve ACLs by Default Either

The `-a` flag in rsync is archive mode — it expands to `-rlptgoD`. There is no `-A` in that list.

```bash
# ACLs silently dropped
rsync -a /srv/project/ remote:/backup/project/
```

```bash
# Correct: add -A for ACLs, -X for extended attributes
rsync -aAX /srv/project/ remote:/backup/project/
```

The `-A` / `--acls` flag is the explicit opt-in. Combine it with `-X` / `--xattrs` to also preserve SELinux labels and other xattrs. If the destination filesystem doesn't support ACLs, rsync will log errors but continue — check its output.

### Default ACLs Do Not Affect Existing Files

You set a default ACL on a shared directory to give `casey` read access to everything inside it:

```bash
setfacl -d -m u:casey:r-x /srv/shared/
```

Casey still can't read the files that were already there before you ran that command. Default ACLs are inherited only by files and directories created after the default ACL was set. Existing files are not retroactively updated.

To apply the policy to existing files, iterate explicitly:

```bash
# Apply to existing files in the directory (not recursively)
setfacl -m u:casey:r-x /srv/shared/*

# Or recursively to all files and subdirectories
find /srv/shared/ -type f -exec setfacl -m u:casey:r-- {} \;
find /srv/shared/ -type d -exec setfacl -m u:casey:r-x {} \;
```

### `chattr +i` Prevents Deletion Even by Root — Including by Yourself

This is not a gotcha for most situations — it's the point. But it becomes a trap when you set it on a config file, then need to rotate it, and forget the flag is there.

```bash
sudo chattr +i /etc/hosts
# ... later ...
sudo vim /etc/hosts   # opens fine for reading, save fails silently or with an error
sudo bash -c "echo '10.0.0.2 db' >> /etc/hosts"
```

```text
bash: /etc/hosts: Operation not permitted
```

```bash
# The fix — always check lsattr first when a write fails unexpectedly
lsattr /etc/hosts
```

```text
----i--------e-- /etc/hosts
```

```bash
sudo chattr -i /etc/hosts
```

The `i` flag survives reboots, survives `sudo`, survives `-f` flags on `rm`. If you lock down a file in a container without the ability to clear `CAP_LINUX_IMMUTABLE`, you may have to rebuild the image.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| POSIX ACLs (`setfacl`/`getfacl`) | Fine-grained per-user/per-group access on local filesystems | Standard; works on ext4, xfs, btrfs |
| NFSv4 ACLs (`nfs4_setfacl`) | Access control on NFSv4 network shares | Different model (ALLOW/DENY ACEs); not interchangeable with POSIX ACLs |
| Supplementary groups + directory permissions | Simple shared access where all members get the same access level | The right first choice before reaching for ACLs |
| `sudo` rules | Controlling which commands a user can run as root | A completely different layer — process-level, not file-level |
| SELinux labels | Mandatory access control, policy-based, system-wide | Operates alongside ACLs; SELinux can deny access even when ACLs allow it |

Reach for supplementary groups first. If the group model is genuinely too coarse — you need different permissions for different users within the same directory — then ACLs are the right tool. ACLs are not a substitute for good group design; they're a precision instrument for cases where group design alone isn't enough.

## Trade-offs

**Where this shines:**

- Shared project directories where different team members need different permission levels — no group proliferation required
- Audit scenarios where a read-only third party needs access to specific paths without being added to operational groups
- Inherited policies via default ACLs: set once on a directory, every new file automatically gets the right permissions
- Backup and restore fidelity: `getfacl -R | setfacl --restore` gives you a portable, human-readable ACL snapshot

**Where this costs you:**

- Complexity: the mask interaction with `chmod` is non-obvious and causes silent permission changes that `ls -l` alone will not reveal
- Tooling gaps: `cp`, `rsync`, and `tar` all require explicit flags to preserve ACLs; a misconfigured backup silently discards them
- Portability: ACLs are lost when files move to filesystems that don't support them (FAT, old NFS, some network shares); there is no warning
- Debugging difficulty: `ls -l` shows `+` but not what the ACL says; diagnosing permission failures requires `getfacl`, not just `ls`
- NFSv4 impedance mismatch: POSIX ACLs and NFSv4 ACLs are structurally different; cross-version NFS is a source of ACL loss

**Performance note:** Reading and checking ACLs requires an additional `getxattr` syscall compared to basic DAC permission checks. On filesystem-intensive workloads with many small files, this is measurable but typically not significant on modern hardware.

---

## When to Use / When to Avoid

✅ **Use this when:**

- You need per-user permissions on a shared file and adding a group for the purpose is disproportionate
- Multiple groups need different permission levels on the same directory tree
- You need a consistent inherited policy on a shared directory (use default ACLs)
- You're building a collaborative environment where new files should automatically have the right permissions

❌ **Avoid this when:**

- The problem is solvable with groups — adding someone to a group with appropriate permissions is simpler, more visible, and harder to misconfigure
- The filesystem doesn't support ACLs (FAT32, some network shares) — your ACLs will be silently dropped on the next copy
- You're on NFSv4 — use `nfs4_setfacl` and the NFSv4 ACL model instead
- You're templating configurations across many systems without tooling that preserves xattrs — the ACLs will not survive the deployment

⚠️ **Common mistakes:**

- Setting `chattr +i` on a config file managed by a deployment tool, then wondering why the deployment fails silently
- Running `chmod` after `setfacl` and unknowingly narrowing the mask
- Forgetting `-A` in rsync and losing ACLs on every backup run
- Using `setfacl -x u:alice` and expecting the mask to shrink — it does not auto-shrink; run `setfacl -m m::value` to explicitly adjust it

## Veteran Perspective

The most common ACL mistake I've seen isn't a typo in a command — it's a deployment pipeline that runs `chmod 644` on everything after a file copy. Every `chmod` call on a file with ACLs silently modifies the mask. A well-intentioned security hardening script that normalizes permissions will quietly destroy ACL configurations it didn't even know existed. I've watched two hours of an incident review turn into a conversation about why alice could write to a file yesterday but not today — the answer was a Jenkins job that ran `chmod` as part of a deploy step.

The intermediate mistake is using `ls -l` to verify ACL state. The `+` tells you ACLs exist; it tells you nothing about what they say. Get into the habit: when a permission problem involves a file with `+` in its listing, your first command is `getfacl`, not `ls -l`. The `#effective:` annotations in `getfacl` output will show you exactly what the mask is doing in thirty seconds flat.

The deeper design principle: ACLs are not a replacement for good group design. I've seen production systems with ACLs on hundreds of files because someone decided it was easier than maintaining groups. It wasn't. Three months later, no one knew what the ACLs were for, the original engineer had left, and removing them was too risky because auditing them would take days. ACLs work best when used surgically — one entry here, a default ACL there. If you're writing a shell loop to set ACLs on fifty files, you probably need a group redesign, not more ACLs.

The `chattr +i` flag is worth knowing cold. It's the right tool for protecting files that should never change — `/etc/resolv.conf` on a container, a pinned hosts file, a certificate that must not be rotated automatically. But it will absolutely bite you during incident response at 2am when you're trying to roll back a config file and getting `Operation not permitted` without any obvious reason. The first thing you check when a root write fails inexplicably: `lsattr`.

In technical interviews, ACLs come up in two contexts: storage/NFS questions ("how do you handle permissions across NFS?") and Linux administration questions ("how would you give a contractor read access to an application log without adding them to the group?"). The answer to the second question starts with `setfacl -m u:contractor:r-- /var/log/app/*.log`. Know the mask gotcha cold before you walk into that room.

## Try It Yourself

1. Create a file in `/tmp`, use `setfacl -m u:nobody:rw-` to add the `nobody` user, then use `getfacl` to see the mask entry that appeared. Run `ls -l` on the same file and confirm the `+` is present.

2. Add the `nobody` user with `rwx` on a file, then run `chmod g-w` on the same file. Use `getfacl` to observe how the mask changed and what `nobody`'s effective permissions became after the `chmod`.

3. Create a directory in `/tmp`, set a default ACL with `setfacl -d -m u:nobody:r-x`, then create a new file inside that directory and use `getfacl` on the new file. Confirm it inherited the default ACL. Then create a second file in the same directory and check whether it also inherits.

4. Create two files in `/tmp`. Copy the first to the second using `cp` (no flags) and check whether the ACL was preserved with `getfacl`. Then copy again using `cp -a` and compare the result.

5. **Challenge:** Set up a shared directory where the group `staff` gets `rwx`, a specific user `alice` (create with `useradd -M alice`) gets `r-x`, and all new files in the directory inherit these permissions automatically. Then verify: create a file as yourself inside the directory, confirm it got the right ACL, then simulate what happens when a deployment script runs `chmod 644` on the new file — use `getfacl` to see the mask change. Use `sudo` and run in a VM.

## Summary

- ACLs extend the basic permission model by adding per-user and per-group entries beyond the single owner/group slot — `setfacl -m u:alice:rwx` and `setfacl -m g:team:r-x` are the core operations
- The mask entry is the maximum effective permission for all named users, the owning group, and all named groups — it is auto-recalculated by `setfacl` and is modified by `chmod` on the group bits; `getfacl` shows `#effective:` annotations when the mask limits an entry
- `ls -l` shows `+` when a file has extended ACL entries, and the group bits display the mask value — use `getfacl` to see the full truth
- `cp -a` preserves ACLs; `rsync` requires the `-A` flag; forgetting either silently drops all ACL configuration in backups and copies
- `chattr +i` makes a file immutable even to root; it survives reboots and `sudo rm -f`; `lsattr` reads the flags; `chattr -i` removes the flag
