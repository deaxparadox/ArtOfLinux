# [3.4] — Special Permissions

> The setuid bit is the reason `/usr/bin/passwd` can change your password even though `/etc/shadow` is owned by root — and it's also one of the most common privilege escalation vectors on misconfigured Linux systems.

## The Problem This Solves

Consider what happens when a normal user wants to change their own password.

The password database on a modern Linux system lives in `/etc/shadow`. That file is owned by root and readable only by root — intentionally, because it stores hashed passwords. If any user could read it directly, the hashes would be trivially available to crack offline.

```bash
ls -l /etc/shadow
```

```text
-rw-r----- 1 root shadow 1423 Jun 10 08:14 /etc/shadow
```

So the problem is: a regular user needs to *write* to a file that they have no permission to touch. One approach would be to give users write access to `/etc/shadow` — which would be a catastrophic security hole. Another approach would be to require a sysadmin to change every password for every user — which doesn't scale to thousands of users.

The Unix designers needed a way to say: "this specific program, when run by any user, should execute with the identity of the file's owner." That's exactly what the setuid bit does. The `passwd` binary is owned by root and has the setuid bit set. When you run it, the kernel temporarily grants the running process root's identity — just long enough to write to `/etc/shadow`. Your own identity is never elevated. Only the `passwd` process's identity is changed, for the duration of that execution.

The setgid bit applies the same logic to groups. The sticky bit solves a different but related problem: in a shared directory like `/tmp`, you want everyone to be able to write files, but you don't want users deleting each other's files. Linux has a solution for that too.

## Mental Model

Think of a vending machine that dispenses medicine. You don't have a key to the medicine cabinet — only the machine does. When you put in your money and make a selection, the machine opens its internal cabinet, retrieves exactly one item, and hands it to you. The machine's key never leaves the machine. You don't gain permanent access to the cabinet. You just get the one thing the machine was designed to give you.

Setuid works the same way. The `passwd` binary holds the "key" — root ownership and the setuid bit. When you run it, the kernel lets it open the cabinet (`/etc/shadow`) on your behalf. When the program exits, the key goes away. You never had root access. The program had root access, briefly, in order to do one specific thing.

## Theory

### The Three UIDs Every Process Carries

Every process on Linux carries three user IDs at all times. Understanding all three is non-negotiable for understanding how setuid works.

The **real UID (RUID)** identifies who originally launched the process — the actual logged-in user. This is what gets set when you log in, and it doesn't change just because you run a setuid program.

The **effective UID (EUID)** is what the kernel actually checks when making permission decisions. When you access a file, the kernel looks at the process's EUID, not its RUID. This is the UID that setuid changes.

The **saved set-UID (SUID)** stores a copy of the EUID before a privileged process drops its privileges temporarily. A process can lower its effective UID to do some work as an unprivileged user, then later restore it from the saved set-UID — without needing to call back to root to get the privilege back. This is what allows programs like `passwd` to operate safely: they drop root privileges after opening the files they need, do the work at lower privilege, and only re-elevate if they must.

When the kernel makes a permission check, it uses the EUID (with one major exception: filesystem access sometimes also checks the filesystem UID, `fsuid`, which for most purposes tracks the EUID).

### What the Setuid Bit Does

The setuid bit is stored in the file's permission metadata alongside the regular read/write/execute bits. It occupies the fourth digit in octal notation — the digit before the familiar three-digit permission pattern.

When a user executes a setuid binary, the kernel's `execve()` call inspects the file's owner and its setuid bit. If the bit is set, the kernel sets the new process's EUID to the file owner's UID before handing control to the program. The RUID remains the original user's UID. The SUID is also set to the file owner's UID, so the process can drop and restore privileges as needed.

This happens at the kernel level, inside the `execve()` system call, before the first instruction of the new program runs. The program cannot opt out of receiving this elevated EUID — it either has the bit set or it doesn't.

### What the Setgid Bit Does

The setgid bit mirrors the setuid bit, but for groups. On executable files, it sets the process's effective GID (EGID) to the file's group at exec time, regardless of what groups the running user belongs to.

On directories, the setgid bit does something more useful and distinct from file behavior: any file or subdirectory created inside a setgid directory inherits the directory's group ownership, instead of inheriting the creator's primary group. Subdirectories also inherit the setgid bit itself, so the behavior propagates down the tree.

This is the standard mechanism for shared project directories: create a group for the project, set that group on the directory, set the setgid bit, and every file created in that directory will belong to the project group regardless of which team member created it.

### What the Sticky Bit Does

The sticky bit on a directory means: any user can create and write files in the directory, but only the file's owner, the directory's owner, or root can delete or rename that file. Write permission on the directory alone is not enough to delete someone else's file.

This is enforced by the kernel's VFS layer during `unlink()` and `rename()` system calls. Before the operation proceeds, the kernel checks whether the directory has the sticky bit set. If it does, the calling process's EUID must match either the file's owner or the directory's owner.

`/tmp` always has the sticky bit set for exactly this reason. On a multiuser system, every user needs to write temporary files. Without the sticky bit, any user could delete any other user's temp files — or, more nefariously, delete a file and replace it with a symlink pointing somewhere harmful before another process reads from it (a classic TOCTOU race condition).

On files (not directories), the sticky bit is historical and has no effect on modern Linux. It was originally used on early Unix to keep executable code in swap for faster repeated execution. No version of Linux has ever implemented this behavior. If you set the sticky bit on a file today, nothing happens.

### Octal Notation for Special Bits

The special bits occupy the first of the four octal digits in a full permission specification:

| Octal | Meaning |
|---|---|
| `4xxx` | setuid bit set |
| `2xxx` | setgid bit set |
| `1xxx` | sticky bit set |
| `7xxx` | all three set |

The three `x` digits are the usual owner/group/other permission octals. In practice:

- `4755` — setuid + owner rwx, group rx, other rx (e.g., `/usr/bin/passwd`)
- `2755` — setgid + owner rwx, group rx, other rx (e.g., shared project directories)
- `1777` — sticky + owner rwx, group rwx, other rwx (e.g., `/tmp`)

## How It Works (Mechanics)

### The execve() Path for Setuid Binaries

When your shell runs `/usr/bin/passwd`, the following sequence happens inside the kernel:

```text
# Conceptual trace — not a real command
1. shell calls fork() → child process inherits your RUID and EUID
2. child calls execve("/usr/bin/passwd", ...)
3. kernel reads the inode for /usr/bin/passwd
4. kernel checks: is the setuid bit set? → yes
5. kernel reads the file owner's UID → 0 (root)
6. kernel sets new process EUID = 0
7. kernel sets new process saved-set-UID = 0
8. kernel sets new process RUID = your UID (unchanged)
9. execve() hands control to passwd's main()
10. passwd opens /etc/shadow (EUID=0 → permitted)
11. passwd writes the new hash
12. passwd exits → process gone, elevated EUID gone with it
```

You can observe this directly with `id` run through a setuid binary. The RUID and EUID will differ.

### Confirming All Three UIDs

```bash
# id shows all three UID values for the current process
id
```

```text
uid=1000(alice) gid=1000(alice) groups=1000(alice),4(adm),27(sudo)
```

When running normally, RUID=EUID. A setuid program during execution would show them diverging. You can use `strace` on `passwd` to watch the credential changes:

```bash
# strace passwd to see getresuid() calls — shows ruid, euid, suid
strace -e trace=getresuid,setresuid passwd 2>&1 | head -20
```

### Why Setuid Root on a Shell Is Instant Root Escalation

If `/bin/bash` had the setuid bit set and was owned by root, any user who ran it would get a shell with EUID=0. That's full root access. The shell inherits the elevated EUID and passes it to every command it spawns.

This is not theoretical. Attackers specifically look for this pattern. If you ever find a shell binary with the setuid bit and root ownership, your system is compromised.

```bash
# How an attacker checks — this is what they run first
find / -user root -perm -4000 -type f 2>/dev/null
```

### How the Kernel Enforces the Sticky Bit

When a process calls `unlink()` to delete a file in a sticky directory:

```text
# Conceptual trace — not a real command
1. kernel checks: does directory have sticky bit set?
2. if yes: is caller's EUID == file owner UID? → allow
3.         is caller's EUID == directory owner UID? → allow
4.         is caller's EUID == 0? → allow
5.         none of the above → EPERM (operation not permitted)
```

The kernel does this check *before* checking whether the caller has write access to the directory. Sticky takes priority.

## Basic Example

```bash
# Examine the setuid bit on passwd
ls -l /usr/bin/passwd
```

```text
-rwsr-xr-x 1 root root 59976 Nov 24  2023 /usr/bin/passwd
```

The `s` in the owner execute position is the setuid bit. Because the underlying execute bit (`x`) is also set, the `s` is lowercase. This means: when any user runs this binary, it executes with root's EUID.

```bash
# Check /tmp to see the sticky bit
ls -ld /tmp
```

```text
drwxrwxrwt 22 root root 4096 Jun 15 14:30 /tmp
```

The `t` at the end of the permissions is the sticky bit. Everyone can write (`rwxrwxrw`), but only file owners can delete their own files.

```bash
# Create a shared group directory with setgid
sudo groupadd projectteam
sudo mkdir /srv/project
sudo chown root:projectteam /srv/project
sudo chmod 2775 /srv/project    # 2 = setgid, 775 = owner rwx, group rwx, other rx
ls -ld /srv/project
```

```text
drwxrwsr-x 2 root projectteam 4096 Jun 15 14:32 /srv/project
```

The `s` in the group execute position is the setgid bit. Any file created here will belong to `projectteam`, not the creator's primary group.

Try this yourself. Run `ls -l /usr/bin/passwd` on your system. The `s` should be there — if it isn't, your system has either hardened passwd differently or something is wrong. Now run `ls -ld /tmp`. The `t` will be there. Those two bits are load-bearing on every Linux system.

## Real-World Example

A common real-world scenario: a development team needs a shared directory where multiple users can collaborate on files, and all files should automatically belong to the shared group — without developers having to remember to `chgrp` after every file creation.

```bash
# Set up a shared project directory — realistic team scenario

# Create the group and add team members
sudo groupadd --system devteam
sudo usermod -aG devteam alice
sudo usermod -aG devteam bob

# Create the directory, set ownership and setgid
sudo mkdir -p /srv/shared/project-x
sudo chown root:devteam /srv/shared/project-x

# 2775: setgid=2, owner rwx=7, group rwx=7, other rx=5
sudo chmod 2775 /srv/shared/project-x

# Verify
ls -ld /srv/shared/project-x
```

```text
drwxrwsr-x 2 root devteam 4096 Jun 15 14:35 /srv/shared/project-x
```

```bash
# Now test as alice — she creates a file
sudo -u alice touch /srv/shared/project-x/report.txt
ls -l /srv/shared/project-x/
```

```text
-rw-rw-r-- 1 alice devteam 0 Jun 15 14:36 report.txt
```

The file belongs to `devteam`, not `alice`'s primary group. Bob can now write to it because he's also in `devteam`. Neither developer had to remember a single `chgrp`. The setgid bit on the directory handled the inheritance automatically.

Now audit the setuid and setgid binaries on the system — something every sysadmin should do when taking over a server:

```bash
# Find all setuid root binaries
# -perm -4000 means: setuid bit is set (regardless of other bits)
# 2>/dev/null suppresses "Permission denied" from inaccessible directories
find / -perm -4000 -type f 2>/dev/null

# Find all setgid binaries
find / -perm -2000 -type f 2>/dev/null

# Find both in one pass
find / \( -perm -4000 -o -perm -2000 \) -type f 2>/dev/null | sort
```

```text
/usr/bin/chfn
/usr/bin/chsh
/usr/bin/gpasswd
/usr/bin/mount
/usr/bin/newgrp
/usr/bin/passwd
/usr/bin/su
/usr/bin/sudo
/usr/bin/umount
/usr/lib/dbus-1.0/dbus-daemon-launch-helper
/usr/lib/openssh/ssh-keysign
/usr/sbin/unix_chkpwd
```

Every binary on this list should be there for a reason. If you see something unexpected — a compiler, an editor, a scripting tool — that is a security problem. Go through the list and ask: "Why does this need elevated privilege to run?" If you can't answer, remove the bit.

## ⚠️ Gotchas

### The `S` (capital S) Means the Bit Is Set But Broken

When you look at permissions with `ls -l`, the position of the `s` tells you which setuid/setgid bit is set. But the case of the `s` tells you something critical: whether the underlying execute bit is also set.

Lowercase `s` (`rws`) means the execute bit *and* the special bit are both set. The program can run, and the bit is active.

Uppercase `S` (`rwS`) means the special bit is set but the execute bit is *not* set. The program cannot be executed — the setuid bit is present but completely ineffective.

```bash
# Create a file and add setuid WITHOUT the execute bit
touch /tmp/test_setuid
chmod u+s /tmp/test_setuid   # sets setuid without first setting execute
ls -l /tmp/test_setuid
```

```text
-rwSr--r-- 1 user user 0 Jun 15 14:40 /tmp/test_setuid
```

The capital `S` tells you something is wrong. The setuid bit is set but it does nothing because the file isn't executable. This happens most often when someone runs `chmod 4644` instead of `4755` — they added the setuid bit but forgot to give execute permission to group or other, or stripped the owner execute bit.

```bash
# The correct approach: set execute first, then setuid
chmod 4755 /tmp/test_setuid
ls -l /tmp/test_setuid
```

```text
-rwsr-xr-x 1 user user 0 Jun 15 14:41 /tmp/test_setuid
```

The same logic applies to setgid on directories: `drwxrwSr-x` means the setgid bit is set, but the group execute bit is off — which means users can't even enter the directory, making the setgid bit academic.

### Setuid on Shell Scripts Does Nothing

This one surprises almost everyone the first time. You have a bash script you want to run with elevated privileges. You set the setuid bit on it. You test it. It still runs as your own user. Nothing changed.

```bash
# This looks like it should work — it doesn't
sudo chown root /usr/local/bin/my-script.sh
sudo chmod 4755 /usr/local/bin/my-script.sh
ls -l /usr/local/bin/my-script.sh
```

```text
-rwsr-xr-x 1 root root 128 Jun 15 14:45 /usr/local/bin/my-script.sh
```

```bash
# Run it and check who we are inside it
cat /usr/local/bin/my-script.sh
```

```text
#!/bin/bash
id
```

```bash
./my-script.sh
```

```text
uid=1000(alice) gid=1000(alice) groups=1000(alice)   # Still alice. Still no root.
```

The kernel deliberately ignores the setuid bit on interpreted scripts — any file that begins with a `#!` shebang line. The reason is a race condition that cannot be safely closed: between the moment the kernel reads the script's setuid bit and the moment the interpreter opens the script file to read its contents, an attacker can swap the file. The kernel has already committed to "this script runs as root," but it hasn't read what the script contains yet. An attacker who can win that race can inject arbitrary commands that run as root.

Compiled binaries don't have this problem because the kernel maps them directly into memory before executing them — there's no second open-and-read step.

If you genuinely need a script to run with elevated privileges, write a small C wrapper program that does the setuid and then exec()s your script. Or, better: use `sudo` with a targeted sudoers entry that grants exactly the permissions needed.

### Setgid on a Directory Doesn't Retroactively Fix Existing Files

You set up a shared directory. You add users to a group. You set the setgid bit on the directory. Then you notice that all the files that were in the directory *before* you set the bit still have wrong group ownership. You expected setgid to fix them automatically.

```bash
# Files already in the directory are NOT affected by adding setgid
ls -l /srv/shared/
```

```text
-rw-rw-r-- 1 alice alice   1024 Jun 10 10:00 old-report.txt   # alice's group, not devteam
-rw-rw-r-- 1 bob   alice   2048 Jun 08 09:15 old-data.csv    # wrong group
-rw-rw-r-- 1 alice devteam  512 Jun 15 14:36 new-report.txt  # correct, created after setgid
```

The setgid bit on a directory only affects files created *after* the bit is set. Existing files keep whatever group they had when they were created. To fix existing files, you need `chgrp` and probably `find`:

```bash
# Fix existing files in the directory
sudo find /srv/shared -type f -exec chgrp devteam {} \;
# Fix existing subdirectories too
sudo find /srv/shared -type d -exec chgrp devteam {} \;
```

### Running `find` for Setuid Binaries Without Redirecting Errors

You search for setuid binaries across the whole filesystem. Your terminal fills with hundreds of "Permission denied" lines from `/proc`, `/sys`, and inaccessible directories. The actual results you care about are buried in the noise, and you have to scroll backward to find them.

```bash
# This works but produces an unreadable mess
find / -perm -4000 -type f
```

```text
find: '/proc/tty/driver': Permission denied
find: '/proc/1234/fd': Permission denied
find: '/root': Permission denied
/usr/bin/passwd
find: '/run/sudo/ts': Permission denied
...hundreds more lines...
```

The fix is two characters:

```bash
# Redirect stderr to /dev/null — errors vanish, results stay
find / -perm -4000 -type f 2>/dev/null
```

```text
/usr/bin/chfn
/usr/bin/chsh
/usr/bin/gpasswd
/usr/bin/mount
/usr/bin/newgrp
/usr/bin/passwd
/usr/bin/su
/usr/bin/sudo
/usr/bin/umount
```

The `2>/dev/null` redirects file descriptor 2 (stderr) to the null device. Error messages go to stderr; real output goes to stdout. They're separate streams. Redirecting one doesn't affect the other.

### The Sticky Bit on a File Is Silently Ignored

If you set the sticky bit on a regular file (not a directory), the kernel accepts it without error, `ls` shows the `t` in the permissions, and absolutely nothing happens.

```bash
# chmod accepts this and shows it — but it does nothing
chmod +t /tmp/myfile.txt
ls -l /tmp/myfile.txt
```

```text
-rw-rw-r-T 1 alice alice 512 Jun 15 14:50 /tmp/myfile.txt
```

Note the capital `T` — same convention as the capital `S` for setgid/setuid: the bit is set, but the execute bit below it is not, so the display uses uppercase. For files, the sticky bit has been meaningless since before Linux existed. No version of Linux has ever respected it on files. If you see the sticky bit on a file in the wild, it means someone applied `chmod +t` to the wrong target, or copied a script from ancient documentation.

The sticky bit is only meaningful on directories.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `/usr/bin/passwd` is setuid root (`-rwsr-xr-x`). `newgrp`, `gpasswd`, `chsh`, `chfn`, `mount`, `umount`, `su` are all setuid root by default. `sudo` is setuid root. |
| RHEL / Fedora / CentOS / Rocky | Same setuid binaries as above. Fedora additionally applies SELinux domain transitions — `passwd` runs in the `passwd_t` domain, providing a second layer of access control beyond the EUID. The EUID grants kernel-level access; SELinux policy further constrains what the process can do with that access. |
| Arch | Similar default setuid binaries. Arch's philosophy means the base install is minimal — fewer setuid binaries than Debian by default. |
| Alpine (containers) | Container environments typically strip setuid bits from most or all binaries. `passwd` in Alpine containers is often not setuid because root is the default user inside containers. Verify with `find / -perm -4000 2>/dev/null` inside a container — a clean Alpine container often shows nothing. |

## Alternatives & Comparisons

| Mechanism | When to Use | Notes |
|---|---|---|
| setuid bit | When a specific binary must always run with its owner's privileges | Coarse — all-or-nothing; the whole program gets elevated EUID |
| `sudo` with sudoers entry | When a user needs to run specific commands with elevated privileges on demand | More granular; logs every invocation; requires explicit permission grant |
| Linux capabilities | When a binary needs *one specific root privilege* (e.g., bind to port 80) but not full root | Finer-grained than setuid root; modern preferred approach for new software |
| setgid bit on directory | When a shared team directory should enforce consistent group ownership on all new files | Purpose-built for collaboration; no security elevation involved |
| ACLs | When group/user permissions need more specificity than the 3-bit UGO model allows | See 3.5 — ACLs; more flexible but adds complexity |

The honest recommendation: setuid is a legacy mechanism that works, but every new software project that previously would have used setuid root should instead use Linux capabilities (`cap_net_bind_service`, `cap_dac_override`, etc.). The setgid bit on directories remains the right tool for shared workspaces — there's no better alternative. `sudo` is almost always the right answer when an admin wants to grant elevated access to a user for specific commands.

## Trade-offs

**Where this shines:**

- setuid makes privileged operations accessible to unprivileged users without requiring a persistent privileged daemon or a session-level privilege grant
- setgid on directories enforces consistent group ownership automatically — no human memory required, no `chgrp` after every file creation
- The sticky bit on `/tmp` prevents a class of race-condition attacks (TOCTOU) in shared directories without restricting write access
- All three bits require no configuration files — they're encoded directly in the inode's permission bits and enforced universally by the kernel

**Where this costs you:**

- Every setuid root binary is a potential privilege escalation vector. A vulnerability in any setuid root program — a buffer overflow, a path traversal, a format string bug — can hand an attacker a root shell. The attack surface is proportional to the number and complexity of setuid binaries on the system.
- Setuid binaries are opaque to standard audit trails. When `passwd` runs, your shell's history shows you ran `passwd`. What it doesn't show is everything the setuid process did with its elevated EUID.
- setgid on directories only fixes group inheritance for newly created files. Teams that add the bit to existing directories still need to manually fix existing files.
- The setuid bit on scripts is silently ineffective — the kernel ignores it without error. This creates a false sense of security: the bit is set, `ls` shows the `s`, but the script runs without any privilege elevation.

**Performance note:** There is no meaningful performance difference between running a setuid binary and a non-setuid binary. The privilege change happens entirely within `execve()`, before the program's first instruction, as part of the normal process creation path.

## When to Use / When to Avoid

✅ **Use setuid when:**
- A specific binary needs to perform privileged operations (write to restricted files, bind to low ports, access kernel interfaces) on behalf of unprivileged users, and you fully control the source code and can audit it
- The binary is already a well-known system utility (`passwd`, `su`, `mount`) — these have been hardened specifically for this use

✅ **Use setgid on a directory when:**
- Multiple users in a shared group need to collaborate on files in a directory, and you want new files to automatically belong to the shared group
- You're creating a shared data directory for a service (e.g., `/var/lib/myapp`) where the application user and operator user need matching group ownership

✅ **Use the sticky bit when:**
- Creating a shared writable directory where multiple users write files (`/tmp`, `/var/tmp`, shared upload directories)
- You need write access to be universal but deletion to be restricted to file owners

❌ **Avoid setuid when:**
- The binary is a shell, interpreter, or script — it won't work on Linux and it's unsafe on systems where it does
- You want to grant a user elevated access interactively — use `sudo` instead, which is audited, revocable, and per-command
- The binary has a history of CVEs or complex attack surface — the privilege it carries becomes the attacker's prize
- You need fine-grained privilege (just bind to port 80) — use Linux capabilities instead of setuid root

❌ **Avoid setgid on files when:**
- You're setting it on a compiled executable to make it run under a specific group — this is obscure, rarely needed, and almost always replaceable with proper group membership

⚠️ **Common mistakes:**
- Setting `chmod 4644` instead of `4755` — the setuid bit is set but execute is missing (capital `S`), so the bit does nothing
- Expecting setgid to fix files that existed before the bit was set — only new files inherit the group
- Setting the sticky bit on a file expecting it to protect that file from deletion — it has no effect on files; it only works on directories

## Veteran Perspective

The most dangerous thing about setuid is how invisible it is.

A junior sysadmin sets up a server, installs some software, follows a five-year-old Stack Overflow answer that says "just `chmod u+s` it," and moves on. The setuid bit is tiny. It doesn't show up in monitoring dashboards. It doesn't generate log entries. It doesn't trigger alerts. And it sits there, quietly, until someone finds it with `find / -perm -4000 2>/dev/null`.

The first thing a competent attacker does after gaining any foothold on a Linux system is run that `find` command. They're looking for setuid root binaries they can exploit — either directly (a known CVE in the binary) or indirectly (a binary that calls other programs using relative paths, which can be hijacked via `$PATH` manipulation; a binary that reads a config file the attacker can write to). GTFOBins at `gtfobins.github.io` is a public database of exactly these techniques. It lists common binaries and the specific commands that, when the binary is setuid root, give you a root shell. `find`, `vim`, `python`, `perl`, `awk` — if any of these have the setuid bit set on a system, the game is over.

The intermediate mistake is treating setuid as a permissions bypass when `sudo` would have been the right answer. "I need this user to run this script as root." The instinct is to `chmod u+s`. The right answer is a targeted `sudo` entry in `/etc/sudoers.d/`. `sudo` logs every invocation to syslog. It requires the user to be explicitly listed. It can restrict to specific commands and arguments. Setuid does none of those things.

The deeper design principle is that setuid was designed as a surgical tool for a very specific problem: a trusted, audited binary needs to temporarily acquire privileges to perform one task. The operative words are *trusted*, *audited*, *temporary*, and *one task*. The `passwd` binary does exactly this — it's been audited for decades by thousands of security researchers, it acquires root only to write to `/etc/shadow`, and it gives it back immediately. When setuid is applied to custom code that has never been audited, the entire model breaks.

I've seen this exact scenario in an incident review: a team deployed a backup script with setuid root because the backup needed to read files in root-owned directories. The script had a command injection vulnerability — it incorporated a filename from user input without sanitization. Any user on the system could trigger a root shell by creating a file with a carefully chosen name and running the backup script on it. The setuid bit turned a minor web application vulnerability into full system compromise.

The right habit is to treat setuid binaries like open ports: document them, justify each one, and audit the list when anything changes. Run `find / -perm -4000 2>/dev/null > /tmp/setuid-audit.txt` when you take over a system. If you can't explain why every binary on that list has the bit, you have a problem.

## Try It Yourself

1. Run `ls -l /usr/bin/passwd` and `ls -ld /tmp` on your system. Confirm the `s` on passwd and the `t` on `/tmp`. Note the difference between lowercase `s`/`t` (execute bit also set) and uppercase `S`/`T` (execute bit missing).

2. Create a test file with `touch /tmp/testfile && chmod u+s /tmp/testfile`. Check the output of `ls -l /tmp/testfile`. Note the capital `S`. Now add the execute bit with `chmod u+x /tmp/testfile` and check again. Observe how the `S` becomes `s`.

3. Create a directory, set the setgid bit on it (`chmod g+s`), create a file inside it as your own user, and verify that the file's group is the directory's group rather than your primary group. Use `ls -l` to check.

4. Look at the full list of setuid binaries on your system with `find / -perm -4000 -type f 2>/dev/null`. For each one, look up what it does and try to explain in one sentence why it needs setuid. If you find anything unexpected, research it.

5. **Challenge** (requires a VM with two test users): Create a shared directory `/srv/collab` with a `collab` group, add both test users to the group, set `chmod 2770` on the directory, and verify that files created by either user automatically belong to the `collab` group. Then test the sticky bit: create `/srv/sticky` with `chmod 1777`, create a file as user A, and verify that user B cannot delete it. Document your findings.

## Summary

- The setuid bit (`chmod u+s`) causes a binary to execute with its file owner's EUID instead of the calling user's EUID — this is how `/usr/bin/passwd` can write to `/etc/shadow` as any user.
- Every process carries three UIDs: real UID (who launched it), effective UID (what the kernel checks for permissions), and saved set-UID (a copy of the pre-drop EUID for privilege restoration).
- The setgid bit on a directory causes all newly created files inside it to inherit the directory's group — it does not affect already-existing files, and does not retroactively fix them.
- The sticky bit on a directory (`chmod +t`) restricts deletion so only a file's owner, the directory's owner, or root can delete it — the mechanism behind `/tmp`'s safety on multiuser systems.
- Setuid on shell scripts is silently ignored by the Linux kernel for security reasons; a lowercase `s` means the bit and execute are both set, an uppercase `S` means execute is missing and the bit is ineffective.
- Every setuid root binary is a potential privilege escalation vector — audit them with `find / -perm -4000 -type f 2>/dev/null` and question any binary that doesn't need to be there.
