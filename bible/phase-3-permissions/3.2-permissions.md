# [3.2] — File Permissions

## The Problem This Solves

Imagine a machine where any program run by any user can read, modify, or delete any file on disk. The web server's configuration file is readable by the database process. The user's private key is readable by every other logged-in user. The system binary that boots the machine can be overwritten by the intern's script. There's no boundary. Every process trusts every other process. One bug, one compromised account, one careless script — and the entire system is open.

This was the actual state of personal computing on DOS and early Windows. One running process could read or write any memory address and any file. The consequences were predictable.

Unix was designed from the beginning as a multi-user, multi-process system. Bell Labs engineers in the late 1960s needed a machine where dozens of researchers could work simultaneously without one user being able to destroy another's work. The permission model — read, write, execute for three classes of users — was the mechanism they chose. It's stayed essentially unchanged for over fifty years because it turns out to be the right level of complexity for most access control decisions.

The alternative — ACLs, capability systems, mandatory access control frameworks — exists, but those are built on top of the Unix permission model, not instead of it. Understanding the base model is required before any of the extensions make sense.

## Mental Model

Think of a filing cabinet in an office. Each drawer has a lock, and the key ring has three separate keys: one for the person who owns the drawer, one for people in the same department, and one for everyone else in the building. For each key, the cabinet administrator has three separate switches: one that controls whether the key holder can read the contents, one that controls whether they can change the contents, and one that controls whether they can open the drawer at all to reach inside.

You can set any combination of those switches independently for each key. The owner might have all three switches on. The department might have read-only. Everyone else might have nothing at all. The filing cabinet doesn't care why — it just checks the key, checks the switches, and either opens or doesn't.

That filing cabinet is every file on your Linux system. The switches are the permission bits.

## Theory

### The Three Classes and Three Permissions

Every file and directory in Linux has exactly nine permission bits, organized as three groups of three. The three groups correspond to three classes of users:

- **User (u):** The owner of the file. One specific UID.
- **Group (g):** Members of the file's owning group. One specific GID.
- **Other (o):** Everyone else — any process whose UID doesn't match the file owner and whose GID (including supplementary groups) doesn't match the file's group.

Within each class, there are three permission bits:

- **Read (r):** Value 4. On a file: permission to open and read the file's contents. On a directory: permission to list the directory's contents — to see the names of files inside it.
- **Write (w):** Value 2. On a file: permission to modify the file's contents. On a directory: permission to create, delete, or rename files inside that directory.
- **Execute (x):** Value 1. On a file: permission to run the file as a program (via `execve()`). On a directory: permission to traverse it — to use it as a component of a path.

These three bits, across three classes, give nine bits total. The kernel stores them in the inode's mode field as a 12-bit value: the upper three bits are the special bits (setuid, setgid, sticky), and the lower nine are the permission bits.

### Execute on Files vs. Execute on Directories — The Critical Distinction

Execute means completely different things depending on what it's set on, and conflating the two is one of the most common points of confusion in all of Linux.

On a **file**, execute means "this file can be launched as a process." Without it, `execve()` refuses with `EACCES` — the kernel won't even look at the file's contents to see if it's a valid binary or script.

On a **directory**, execute means "this directory can be traversed — used as a step in a path lookup." When the kernel resolves a path like `/home/alice/projects/app.conf`, it needs execute permission on `/`, `/home/`, `/home/alice/`, and `/home/alice/projects/` before it can even reach `app.conf`. Without execute on one of those intermediate directories, the path resolution fails immediately with `EACCES`, regardless of what permissions `app.conf` itself has.

This means read and execute on a directory are fully independent:

- **Read without execute (`r--`):** You can run `ls` and see filenames inside the directory, but you cannot `cd` into it, and you cannot access any file inside it by path. The names are visible; the contents are locked.
- **Execute without read (`--x`):** You can `cd` into the directory and access files inside it — but only if you already know their exact names. You cannot list the contents. This is a real access control pattern: give someone access to a specific file in a directory without revealing what else lives there.
- **Both (`r-x`):** Normal access — list and traverse.
- **Neither (`---`):** The directory is completely inaccessible.

### The Octal Notation

Because each permission bit is either on or off, and each class has three bits, each class maps cleanly to a single octal digit (0–7):

```text
r w x   → 4 + 2 + 1 = 7
r w -   → 4 + 2 + 0 = 6
r - x   → 4 + 0 + 1 = 5
r - -   → 4 + 0 + 0 = 4
- w x   → 0 + 2 + 1 = 3
- w -   → 0 + 2 + 0 = 2
- - x   → 0 + 0 + 1 = 1
- - -   → 0 + 0 + 0 = 0
```

Three classes × one digit each = a three-digit octal number. `644` means owner has read+write (6), group has read-only (4), other has read-only (4). `755` means owner has read+write+execute (7), group has read+execute (5), other has read+execute (5).

Common values you'll encounter constantly:

| Mode | String | Meaning | Typical use |
|---|---|---|---|
| 755 | `rwxr-xr-x` | Owner can do everything; others can read and execute | Executables, directories |
| 644 | `rw-r--r--` | Owner can read/write; others read-only | Config files, static content |
| 700 | `rwx------` | Owner only, full access | Private scripts, SSH directories |
| 600 | `rw-------` | Owner read/write only | SSH private keys, secrets |
| 777 | `rwxrwxrwx` | Everyone can do everything | Almost never correct |

### umask — Default Permissions for New Files

When a process creates a new file or directory, it requests a set of permissions via the `open()` or `mkdir()` system call. Conventionally, programs request 0666 for files (read/write for everyone, no execute) and 0777 for directories. But the kernel doesn't grant those permissions directly. It filters them through the process's umask.

The formula is:

```text
# Conceptual flow — not a real command
actual_permissions = requested_mode & (~umask)
```

The umask is a bitmask of permissions to *remove*. A umask of `022` removes write permission from group and other. So a file requested at `0666` becomes `0644`:

```text
# Conceptual flow — not a real command
0666 & ~022
= 0666 & 0755
= 0644
```

And a directory requested at `0777` becomes `0755`:

```text
# Conceptual flow — not a real command
0777 & ~022
= 0777 & 0755
= 0755
```

The umask is not an assignment. It's a subtraction. A umask of `077` removes group and other entirely, giving `600` for new files and `700` for new directories — a tight default appropriate for private data.

## How It Works (Mechanics)

### The Kernel DAC Check

When a process tries to access a file — open it, execute it, stat it, create a file inside a directory — the kernel performs a Discretionary Access Control (DAC) check. The check sequence is strictly first-match:

```text
# Conceptual flow — not a real command
if process.effective_uid == file.uid:
    apply user permission bits → grant or deny → stop
else if process.effective_gid in {file.gid} ∪ process.supplementary_groups:
    apply group permission bits → grant or deny → stop
else:
    apply other permission bits → grant or deny → stop
```

This is non-additive. If your UID matches the file owner, only the owner bits apply. The group and other bits are completely ignored — even if they would grant broader access. This catches people out: a file owned by `alice` with mode `040` (group read, nothing for owner) gives `alice` no access at all, because the UID check fires first and finds owner bits empty.

### Why Root Bypasses Permissions

A process running as UID 0 (root) has the `CAP_DAC_OVERRIDE` capability. The kernel's access check short-circuits: before even consulting the permission bits, it checks whether the process is root (or has `CAP_DAC_OVERRIDE`), and if so, grants read and write access unconditionally.

Execute is different. Root still needs at least one execute bit set somewhere in the mode — in any of the three classes — to execute a file. A file with mode `000` or `666` cannot be executed even by root. The kernel checks: if any of the nine permission bits includes an execute bit (bits `0100`, `0010`, or `0001` in the mode), root can execute. If none of the three triplets has an execute bit, even root gets `EACCES` on exec.

### The Inode Mode Field

The inode stores a 12-bit mode field. The full layout in octal:

```text
# Conceptual layout — not a real command
Bit position (octal)  Meaning
─────────────────────────────────────────
4000                  setuid
2000                  setgid
1000                  sticky bit
 400                  owner read
 200                  owner write
 100                  owner execute
  40                  group read
  20                  group write
  10                  group execute
   4                  other read
   2                  other write
   1                  other execute
```

When you run `chmod 755 file`, the kernel calls the `chmod(2)` syscall which writes `0755` into this field, replacing whatever was there. When you run `ls -l`, the shell reads this field via `stat(2)` and translates it to the `rwxr-xr-x` string you see.

### What chmod Does at the System Call Level

`chmod 755 file` ultimately calls:

```bash
# Illustrative — strace shows the actual syscall
strace chmod 755 /tmp/testfile 2>&1 | grep fchmodat
```

```text
fchmodat(AT_FDCWD, "/tmp/testfile", 0755) = 0
```

The kernel writes `0755` directly to the inode's mode field. There's no logging, no history, no undo. The previous permission value is gone.

For symbolic mode (`chmod u+x file`), the kernel first does a `stat()` to read the current mode, the C library constructs the new mode by OR-ing or AND-ing the relevant bits, and then calls `fchmodat()` with the result.

## Basic Example

```bash
# Create a file and examine its default permissions
touch /tmp/permtest.txt
ls -l /tmp/permtest.txt
```

```text
-rw-r--r-- 1 alice alice 0 Jun 16 10:00 /tmp/permtest.txt
```

Reading the output left to right: `-` (regular file) `rw-` (owner: read+write) `r--` (group: read only) `r--` (other: read only) `1` (link count) `alice` (owner) `alice` (group) `0` (size) `Jun 16 10:00` (mtime) `permtest.txt` (name).

```bash
# Set the execute bit for the owner only
chmod u+x /tmp/permtest.txt
ls -l /tmp/permtest.txt
```

```text
-rwxr--r-- 1 alice alice 0 Jun 16 10:00 /tmp/permtest.txt
```

```bash
# Set permissions using octal — all three groups at once
chmod 640 /tmp/permtest.txt
ls -l /tmp/permtest.txt
```

```text
-rw-r----- 1 alice alice 0 Jun 16 10:00 /tmp/permtest.txt
```

```bash
# Check your current umask
umask
```

```text
0022
```

```bash
# See what umask gives new files vs directories
mkdir /tmp/umasktest && touch /tmp/umasktest/newfile.txt
ls -ld /tmp/umasktest /tmp/umasktest/newfile.txt
```

```text
drwxr-xr-x 2 alice alice 4096 Jun 16 10:01 /tmp/umasktest
-rw-r--r-- 1 alice alice    0 Jun 16 10:01 /tmp/umasktest/newfile.txt
```

The directory got 755 (0777 minus 022), the file got 644 (0666 minus 022). Notice the leading `d` on the directory entry. Now try this yourself — your umask might differ if you're on Ubuntu/Debian, where logged-in users with a private primary group often get `002` instead of `022`, giving group-writable defaults of 664 for files and 775 for directories.

## Real-World Example

You're deploying a web application. The app runs as the `www-data` user. Configuration files contain database passwords. Log files need to be writable by the app and readable by your monitoring user. The deploy script, which runs as `deploy`, must be executable but not modifiable by the app itself.

```bash
# Deploy script: executable by owner, readable by group, nothing for others
chmod 750 /opt/myapp/deploy.sh
ls -l /opt/myapp/deploy.sh
```

```text
-rwxr-x--- 1 deploy deploy 4096 Jun 16 10:00 /opt/myapp/deploy.sh
```

```bash
# Config file with database password: owner read/write only
chmod 600 /opt/myapp/config/database.yml
ls -l /opt/myapp/config/database.yml
```

```text
-rw------- 1 deploy deploy 512 Jun 16 10:00 /opt/myapp/config/database.yml
```

```bash
# Log directory: app user (www-data) needs to write; monitor user needs to read
# Set group ownership to a shared group and give group read/write/execute on the dir
chown deploy:applog /opt/myapp/logs
chmod 775 /opt/myapp/logs
ls -ld /opt/myapp/logs
```

```text
drwxrwxr-x 2 deploy applog 4096 Jun 16 10:00 /opt/myapp/logs
```

```bash
# Verify actual access — check what www-data (a member of applog) can do
# by looking at the effective permission set with namei
namei -l /opt/myapp/logs
```

```text
f: /opt/myapp/logs
 drwxr-xr-x root   root   /
 drwxr-xr-x root   root   opt
 drwxr-xr-x deploy deploy myapp
 drwxrwxr-x deploy applog logs
```

`namei -l` walks each component of a path and shows its permissions — this is what the kernel actually checks during path resolution. If any directory in the chain lacks execute for your process's UID/GID, access fails at that component, not at the target. This command is the first thing to reach for when you get an unexpected `Permission denied` and can't figure out where the access is being blocked.

The config file is 600 — only `deploy` can read it. If `www-data` tries to open it, the UID doesn't match (so owner bits apply to `deploy` only), the GID doesn't match, and other has no permissions. The kernel denies the access before the application code even starts to read the file. No amount of application logic can override that.

## ⚠️ Gotchas

### Execute on a Directory Means Traverse, Not Run

You know you need execute to run a program. So when you can't `cd` into a directory, you assume you need write permission, or maybe read permission, and you start adding those. The directory remains inaccessible.

```bash
# Create a directory and remove execute
mkdir /tmp/noexec_dir
touch /tmp/noexec_dir/secret.txt
chmod 644 /tmp/noexec_dir   # read and write, no execute
ls -ld /tmp/noexec_dir
```

```text
drw-r--r-- 2 alice alice 4096 Jun 16 10:00 /tmp/noexec_dir
```

```bash
# Try to enter it
cd /tmp/noexec_dir
```

```text
bash: cd: /tmp/noexec_dir: Permission denied
```

```bash
# You can list the filenames (read permission), but cannot access the files
ls /tmp/noexec_dir
```

```text
secret.txt
```

```bash
# But you cannot read the file itself — path traversal fails
cat /tmp/noexec_dir/secret.txt
```

```text
cat: /tmp/noexec_dir/secret.txt: Permission denied
```

The execute bit on a directory controls whether the kernel can traverse that directory during path resolution. Read permission controls whether you can list contents. They're orthogonal. You need execute to use the directory as a step in any path — `cd`, `open()`, `stat()`, anything. The fix is to add execute back:

```bash
# Give owner the execute bit — now traversal works
chmod u+x /tmp/noexec_dir
cd /tmp/noexec_dir && echo "now it works"
```

```text
now it works
```

### chmod -R 777 Is a Fire Sale on Security

When something doesn't work and you've run out of ideas, `chmod -R 777 /some/path` feels like a solution. It makes the problem go away. It also throws away every security boundary protecting that directory.

> ⚠️ **WARNING:** `chmod -R 777` on any directory accessible by a web server, a shared user, or any untrusted process means that process can now modify every file in that tree. On a web-writable directory, this enables arbitrary file upload, script injection, and complete server compromise. Never run this on a production system without understanding exactly what you're opening up.

```bash
# The nuclear option — removes all access control from an entire tree
chmod -R 777 /var/www/html   # DO NOT DO THIS
```

What actually broke almost always comes down to one of two things: the process doesn't have execute on one of the *parent* directories, or the file's ownership is wrong (a different user owns it, so no amount of permission changes you make as your own user will help — you need to fix the ownership with `chown`).

The right diagnostic sequence:

```bash
# 1. Find where path resolution actually fails
namei -l /var/www/html/app/config.php

# 2. Check what user the process is running as
ps aux | grep nginx     # or apache2, php-fpm, etc.

# 3. Fix only the specific component that needs fixing
# If www-data can't traverse /var/www/html:
chmod o+x /var/www/html
# If www-data can't read a specific config file:
chown www-data:www-data /var/www/html/app/config.php
chmod 640 /var/www/html/app/config.php
```

Surgical fixes, not blanket 777.

### umask Is a Subtraction, Not an Assignment

People often read `umask 022` as "set default permissions to 022." That's backwards. The umask specifies what to *remove* from the base permissions.

```bash
# What umask 022 actually produces
umask 022
touch /tmp/umask_test.txt
ls -l /tmp/umask_test.txt
```

```text
-rw-r--r-- 1 alice alice 0 Jun 16 10:00 /tmp/umask_test.txt
```

The file got 644, not 022. The kernel started from 0666 (what `open()` requests for a normal file) and removed the bits specified by 022 — which is write for group (020) and write for other (002). The remaining bits are 644.

If you want files to come out as 660 (owner and group read/write, nothing for other), set `umask 006` (removes read+write from other). Not `umask 660`.

```bash
# To verify: umask 007 → files get 660, dirs get 770
umask 007
touch /tmp/umask_007_test.txt && mkdir /tmp/umask_007_dir
ls -ld /tmp/umask_007_test.txt /tmp/umask_007_dir
```

```text
-rw-rw---- 1 alice alice    0 Jun 16 10:00 /tmp/umask_007_test.txt
drwxrwx--- 2 alice alice 4096 Jun 16 10:00 /tmp/umask_007_dir
```

The umask change is per-shell-session. Setting it in the terminal doesn't persist; it must be set in `~/.bashrc`, `~/.profile`, or `/etc/profile` to survive login.

### A Script With a Shebang but No Execute Bit

You write a shell script, add `#!/bin/bash` at the top, and try to run it. Linux refuses. The file is clearly readable — you can `cat` it fine. The shebang is correct. The error message is confusing: "Permission denied."

```bash
# Create a script without execute permission
printf '#!/bin/bash\necho hello\n' > /tmp/hello.sh
ls -l /tmp/hello.sh
```

```text
-rw-r--r-- 1 alice alice 20 Jun 16 10:00 /tmp/hello.sh
```

```bash
/tmp/hello.sh
```

```text
bash: /tmp/hello.sh: Permission denied
```

The kernel's `execve()` syscall checks the execute bit before it reads a single byte of the file. The shebang line (`#!/bin/bash`) is only read after `execve()` decides the file is allowed to run. Without the execute bit, `execve()` returns `EACCES` immediately — the interpreter is never consulted.

```bash
# Fix: add the execute bit
chmod u+x /tmp/hello.sh
/tmp/hello.sh
```

```text
hello
```

Read permission is not execute permission. They answer different questions. Read says "can you see the contents?" Execute says "can you launch this as a process?" The kernel treats them as completely independent.

Note: if you run `bash /tmp/hello.sh` (explicitly calling the interpreter), this works even without the execute bit — because you're not using `execve()` on the script; you're using `execve()` on `/bin/bash`, which does have execute permission, and passing your script as an argument. The execute bit check is bypassed entirely. This is a common workaround during development but not how deployed scripts should run.

### chmod Octal Mode Replaces All Bits Simultaneously

With symbolic mode, `chmod u+x` adds execute for the owner and leaves everything else alone. People assume octal mode works the same way — that `chmod 755` sets the three digits and leaves the others unchanged. Octal mode replaces all nine permission bits at once.

```bash
# A file with setuid set (the 4 in 4755)
touch /tmp/setuid_test
chmod 4755 /tmp/setuid_test
ls -l /tmp/setuid_test
```

```text
-rwsr-xr-x 1 alice alice 0 Jun 16 10:00 /tmp/setuid_test
```

```bash
# Now "fix the permissions" with a 3-digit octal
chmod 755 /tmp/setuid_test
ls -l /tmp/setuid_test
```

```text
-rwxr-xr-x 1 alice alice 0 Jun 16 10:00 /tmp/setuid_test
```

The setuid bit (the `s` in `rws`) is gone. `chmod 755` silently zeros the special bits (setuid, setgid, sticky) because it implicitly treats the leading digit as 0. If you need to preserve special bits while changing permissions, use symbolic mode or a four-digit octal (`chmod 4755`).

## Distro Notes

| Distro Family | Notable Behavior |
|---|---|
| Debian / Ubuntu | Default umask for users with a private primary group is `002` (via `pam_umask` with `usergroups` option), giving `664` for new files and `775` for directories. This is intentional: when your primary group is private to you, group-write is no less safe than owner-write. System-wide default in `/etc/login.defs` is still `022`. |
| RHEL / Fedora / CentOS / Rocky | Default umask is `022` universally. `/etc/profile.d/` scripts set this. SELinux adds a separate mandatory access control layer on top of DAC — a file can have permissive Unix permissions but still be blocked by an SELinux context. |
| Arch | Default umask `022`. Minimal additions beyond base POSIX behavior. |
| Alpine (containers) | Default umask `022`. Common in containers where there's typically one user (root or a service account), making the group/other distinction less meaningful. |

## Alternatives & Comparisons

| Mechanism | When to Use | Notes |
|---|---|---|
| Unix DAC (this topic) | Standard file access control on most Linux systems | Fast, simple, universally supported; limited to one owner, one group per file |
| POSIX ACLs (`setfacl`/`getfacl`) | When a single file needs permissions for multiple specific users or groups | Extends DAC; stored in extended attributes; not all filesystems support them |
| SELinux / AppArmor | Mandatory access control — policy overrides user decisions | Adds a second independent check after DAC; used heavily on RHEL (SELinux) and Ubuntu (AppArmor) |
| Capabilities (`setcap`) | Grant a binary one specific privileged operation without full root | Replaces the blunt instrument of setuid-root for specific cases |

For most day-to-day tasks — who can read a config file, who can write to a log directory, who can execute a script — Unix DAC is the right and sufficient tool. POSIX ACLs solve the "I need three different users to have different permissions on one file" problem cleanly. SELinux and AppArmor are for environments where you don't trust the users themselves to set correct permissions.

## Trade-offs

**Where this shines:**

- Simple and fast: the kernel checks three integers (UID, GID, mode bits) — essentially free at runtime.
- Universal: every Linux filesystem supports it, every tool understands it, every language's standard library can read and set it.
- Auditable: `ls -l` shows the full state at a glance. There's no hidden policy file; the permissions are on the inode itself.
- Composable with groups: the group bit lets you express "these N users can access this file" without listing each user individually — just put them in a group.

**Where this costs you:**

- One group per file: a file can have exactly one owning group. If you need user A, user B, and user C to have different access levels on the same file, Unix DAC alone can't express that — you need POSIX ACLs.
- No deny rules: you can only grant permissions. You can't say "everyone except bob." If bob is in the owning group, he gets group permissions, full stop.
- World-readable is default enough that secrets leak: with a umask of `022`, files are readable by everyone on the system unless you deliberately restrict them. Private keys, config files with passwords, anything sensitive must be explicitly locked down to `600` or `640`.
- `chmod -R` is blunt: there's no built-in way to say "apply 755 to directories but 644 to files in one command" without using `find`.

---

## When to Use / When to Avoid

✅ **Use this when:**
- You need to control who can read a file, write to it, or execute it — this is the primary tool.
- You're creating a file that will contain secrets (passwords, API keys, private keys): set it to `600` immediately on creation, before writing the secret.
- You're setting up a shared directory where a group of users all need write access: set the directory to group-writable (`g+w`) and add users to the owning group.
- You're deploying a script that should run as a service: ensure the script has execute permission for the user the service runs as.

❌ **Avoid this when:**
- Multiple distinct users need different permission levels on one file: reach for POSIX ACLs (`setfacl`) instead.
- You're running a web-accessible directory and want to restrict which application can write there: consider DAC plus AppArmor or SELinux to prevent a compromised process from writing even if the Unix permissions would allow it.
- You think `chmod 777` is a valid fix: it isn't. It's a symptom that the ownership or the parent directory's permissions haven't been diagnosed correctly.

⚠️ **Common mistakes:**
- Setting a directory to `644` and wondering why `cd` fails — directories need the execute bit to be traversable.
- Changing a file's permissions but forgetting that the parent directory's permissions control whether you can even reach the file.
- Confusing `umask 022` with "the default permissions are 022" — the default permissions are `0666 & ~022 = 0644`.
- Running `chmod -R 755 /var/www` without realizing this makes every config file in that tree world-readable.

## Veteran Perspective

The newcomer mistake is reaching for `chmod 777` the moment something doesn't work. It fixes the symptom — the access error goes away — and completely obscures the actual problem. Nine times out of ten, the real issue is one of three things: the process is running as the wrong user, the parent directory doesn't have execute permission, or the file's ownership is wrong (and `chmod` by a different user on a file you don't own fails silently or with a confusing error). `chmod 777` papers over all of these and teaches nothing. The next time something breaks, you don't know any more than you did before.

The intermediate mistake is not understanding that the DAC check is first-match. I've watched a senior developer spend an hour debugging why a file with mode `040` was inaccessible to the user who owned it. The file was in a shared directory and the developer had set group-read to let teammates access it — but the owner had no owner-bits at all. The kernel matched on UID first, found zero owner permissions, and denied. Group bits never got checked. Adding `u+r` fixed it in one second, but only after understanding why.

The deeper principle is that Unix permissions are about process identity, not user intent. The kernel doesn't care that you "meant" to give read access. It checks the effective UID of the calling process against the inode's UID. If your web server runs as `www-data` and a file is owned by `root` with mode `600`, the web server cannot read that file — and no amount of wanting it to work will change the math. You have to change the fact: either change the file's ownership, change the permissions to allow other-read, or use a group.

In security reviews, I look for three specific patterns. First: `600` or `640` on any file containing a credential — if I find a `.env` file or a `database.yml` with world-readable permissions, that's an immediate finding. Second: directories under `/var/www` or any web root that have `757` or `777` — this means a web-accessible process can write arbitrary files, which is a shell injection waiting to happen. Third: setuid or setgid binaries in unexpected places — `find / -perm /6000 -type f` reveals all of them, and anything unexpected is suspicious.

The interview question version of all this: "What's the difference between read and execute on a directory?" Anyone who can answer that with confidence — including the non-obvious cases like `--x` without `r` — has moved past the surface level.

One scenario that sticks with me: a junior admin was asked to "give the monitoring user access to the log files." They ran `chmod -R 777 /var/log`. The monitoring user could now read the logs — and so could every other process on the system, including an exploited PHP application that was writing to the webserver's access log. Within an hour, the attacker had read `/var/log/auth.log` and found a leaked temporary password. `chmod 640` with the monitoring user in the owning group would have worked. `777` was catastrophic. The difference was understanding what the second and third permission triplets actually mean.

1. Create a file, check its permissions with `ls -l`, and read the three permission triplets out loud — owner, group, other.

2. Set your test file to `600`, then try to read it as your own user, then check what `cat` returns. Change it to `000` and try again. Restore it to `644` when done.

## Summary

- Permission bits are nine flags in the inode (3 per class: user, group, other), each class controlling read (4), write (2), and execute (1). The kernel checks them in strict order — UID match → group match → other — and the first match wins.
- Execute on a **file** means the kernel will run it via `execve()`. Execute on a **directory** means the kernel can traverse it during path resolution. You need execute on every directory component of a path, not just the final target.
- `chmod` with an octal mode (e.g., `chmod 644`) replaces all nine permission bits at once. Symbolic mode (e.g., `chmod u+x`) modifies only the specified bits and leaves others alone.
- `umask` is a subtraction mask applied to every new file and directory creation. `umask 022` produces 644 for files (from base 0666) and 755 for directories (from base 0777).
- Root bypasses read and write DAC checks entirely, but still needs at least one execute bit set somewhere in the mode to run a file.
