## The Problem This Solves

When a disk arrives in a server, it is a single, undivided block of storage. You cannot put a filesystem on the whole thing and call it done — not in any real system. Here's why.

Your operating system needs to live somewhere that is completely separate from your user data. If `/` and `/home` share one big filesystem, a user who fills up their home directory will starve the operating system of space and the server will stop accepting new log entries, stop being able to write temp files, and eventually stop running. The OS and the data need walls between them.

You also need a swap region. Swap is not a filesystem — it has its own internal layout and is never mounted in the traditional sense. It has to live on a dedicated region of the disk.

If you ever use LVM — and on a production server, you should — LVM needs a dedicated Physical Volume region to work from.

And if you're running UEFI, the bootloader lives on a specially-typed partition that the firmware knows how to read before the OS is loaded.

None of this is possible without partitioning: the process of dividing a raw block device into named, typed regions, and writing a partition table at the front of the disk that tells the kernel where each region starts and ends.

Partitioning is also the dividing line between the tools that only help you look at storage and the tools that can destroy it. Everything in this topic up to the point you write a partition table is reversible. The moment you write it, whatever was there before is gone.

---

## Mental Model

Think of a brand-new disk as a completely empty warehouse with no interior walls. A partition table is the architect's floor plan: it draws lines on the floor and labels each section — "server room here, office space there, loading dock in the back." The labels don't put anything in those sections. They just define the boundaries and what each section is meant for.

Formatting (which comes in the next topic) is the act of actually setting up each section: putting in shelving, labeling boxes, building the filing system. You can't do that until the floor plan exists.

The partition table sits at the very front of the disk — just like a floor plan is posted at the entrance of a building. The kernel reads it first thing when the disk is discovered, before anything else happens.

## Theory

### What a Block Device Is

A block device is a storage device that the kernel can read from and write to in fixed-size chunks called blocks. This is in contrast to character devices, which stream bytes one at a time. Disks — HDDs, SSDs, NVMe drives — are all block devices. The kernel gives them a special file in `/dev/` that represents the entire device as a flat sequence of addressable blocks.

When you look at `/dev/sda`, you are looking at the entire first SATA disk. There is no filesystem there yet. It is just raw blocks. The kernel doesn't know what's on it until it reads the partition table.

### Device Naming

Linux names block devices based on the driver that handles them:

**SATA and USB drives** use the SCSI driver layer and are named with the `sd` prefix. The first disk is `/dev/sda`, the second is `/dev/sdb`, and so on. Partitions on those disks are numbered: `/dev/sda1`, `/dev/sda2`, `/dev/sda3`.

**NVMe drives** use the NVMe driver and have a three-level name. The first controller is `/dev/nvme0`. The first namespace on that controller (almost always the only one) is `/dev/nvme0n1`. The first partition on that namespace is `/dev/nvme0n1p1`. The `n` in the middle is for namespace — a feature of the NVMe protocol that allows one physical drive to present multiple isolated storage regions. In practice, consumer and datacenter NVMe drives almost always have exactly one namespace, so you will mostly see `nvme0n1`.

**Virtual disks** in KVM/QEMU guests use the `vd` prefix: `/dev/vda`, `/dev/vdb`. Some hypervisors expose virtual disks with the SCSI layer, giving `/dev/sda`.

The critical rule: device names are not stable across reboots. `/dev/sda` this boot may be `/dev/sdb` next boot if the kernel detects the drives in a different order. This is why you never put `/dev/sda1` in `/etc/fstab` — you use the UUID. More on this in the Gotchas section.

### MBR vs. GPT

There are two partition table formats in wide use. You will encounter both.

**MBR (Master Boot Record)**, also called DOS partitioning, dates to 1983. The partition table lives in the first 512 bytes of the disk alongside the bootloader. It supports a maximum of four primary partitions. To get around this limit, one primary partition can be designated an "extended partition" which contains logical partitions — but this is a layered hack, not a real design. MBR can address a maximum of 2TB of disk space. On a disk larger than 2TB, any space beyond 2TB is invisible to an MBR partition table.

**GPT (GUID Partition Table)** was introduced with UEFI and is the right choice for every new disk today. GPT supports up to 128 partitions by default (the limit is configurable). It can address disks up to 9.4 zettabytes. Every partition gets a unique identifier (a GUID/UUID) that is embedded in the table itself. GPT stores the partition table at both the front and the back of the disk as a redundant backup. It uses CRC32 checksums to detect corruption.

If you are partitioning a disk on a modern system — UEFI firmware, disks larger than a few hundred gigabytes — use GPT. The only reason to use MBR today is compatibility with very old bootloaders or very old firmware that cannot read GPT.

### Partition Types

Partitions carry a type code that tells the system what the partition is intended to hold. This is metadata — the type code does not enforce anything. You could create a partition with the "swap" type and format it with ext4, and the kernel would let you. The type code is a hint for tools and humans.

In MBR, type codes are single-byte hex values. The commonly used ones:

- `83` — Linux filesystem (the default for any regular partition)
- `82` — Linux swap
- `8e` — Linux LVM Physical Volume
- `ef` — EFI System Partition (for UEFI boot)

In GPT, type codes are full GUIDs. `gdisk` uses short four-hex-digit codes as aliases for them:

- `8300` — Linux filesystem data (GUID: `0FC63DAF-8483-4772-8E79-3D69D8477DE4`)
- `8200` — Linux swap (GUID: `0657FD6D-A4AB-43C4-84E5-0933C84B4F4F`)
- `8e00` — Linux LVM (GUID: `E6D6D379-F507-44C2-A23C-238F2A3DF928`)
- `EF00` — EFI System Partition (GUID: `C12A7328-F81F-11D2-BA4B-00A0C93EC93B`)

The EFI System Partition type code matters more than the others — UEFI firmware actively looks for a partition with that type code when it needs to find the bootloader. Getting this wrong will produce a system that doesn't boot.

## How It Works (Mechanics)

### How the Kernel Discovers a Disk

When a disk is connected — at boot or via hotplug — the kernel's driver detects it and creates the block device file in `/dev/`. For a SATA disk, this is the SCSI layer creating `/dev/sda`. At the same time, the kernel reads the first sector of the disk looking for a partition table. If it finds a valid MBR signature (bytes 0x1FE–0x1FF equal `0x55AA`), it parses the MBR partition entries. If it finds a valid GPT header at LBA 1 (the second 512-byte sector), it parses the GPT. For each partition found, the kernel creates a child block device: `/dev/sda1`, `/dev/sda2`, etc.

These child devices appear in `/dev/` and in `sysfs` at `/sys/block/sda/sda1/`. Tools like `lsblk` read `sysfs` to build the device tree you see.

### How fdisk Works

`fdisk` is an interactive partition editor. It reads the current partition table from the disk into memory and presents you with a menu. All changes — adding, deleting, resizing partitions — happen only in memory until you type `w` to write. Typing `q` discards all changes and exits without touching the disk. This is the most important thing to understand about `fdisk`: nothing is committed until `w`.

When you type `w`, `fdisk` calls the `write()` system call to overwrite the partition table sectors on the disk. The kernel is then notified via an `ioctl(BLKRRPART)` call to re-read the partition table for that device. If any partition on the device is currently mounted, the kernel may refuse to re-read the table — in that case, the writes succeed but the in-kernel view of the device is stale until you reboot or run `partprobe`.

### What `partprobe` Does

After writing a new partition table to a disk that the kernel already knows about, the new partitions won't automatically appear as device files. `partprobe /dev/sda` sends a signal to the kernel to re-read the partition table and update its in-memory representation — creating or removing the `/dev/sdaN` child devices accordingly. If any partition on the disk is mounted, the kernel will refuse to update, and you'll need to reboot.

## Basic Example

The safest way to start is to just look — before touching anything.

```bash
# List all block devices in a tree view
lsblk
```

```text
NAME        MAJ:MIN RM   SIZE RO TYPE MOUNTPOINTS
sda           8:0    0   500G  0 disk
├─sda1        8:1    0     1G  0 part /boot
├─sda2        8:2    0     8G  0 part [SWAP]
└─sda3        8:3    0   491G  0 part /
nvme0n1     259:0    0   1.8T  0 disk
└─nvme0n1p1 259:1    0   1.8T  0 part /data
```

The tree structure shows the parent-child relationship between disks (type `disk`) and their partitions (type `part`). The MOUNTPOINTS column tells you where each partition is currently attached to the filesystem tree.

```bash
# Show the partition table on a specific disk (read-only — safe to run)
sudo fdisk -l /dev/sda
```

```text
Disk /dev/sda: 500 GiB, 536870912000 bytes, 1048576000 sectors
Disk model: VBOX HARDDISK
Units: sectors of 1 * 512 = 512 bytes
Sector size (logical/physical): 512 bytes / 512 bytes
I/O Size (minimum/optimal): 512 bytes / 512 bytes
Disklabel type: gpt
Disk identifier: 4B2A8C3E-1F2D-4E5A-8B9C-0D1E2F3A4B5C

Device       Start        End    Sectors  Size Type
/dev/sda1     2048    2099199    2097152    1G Linux filesystem
/dev/sda2  2099200   18874367   16775168    8G Linux swap
```

`fdisk -l` never touches the disk. Run it on any device you want to inspect. Notice the "Disklabel type: gpt" line — this tells you whether the disk uses GPT or MBR before you do anything else.

Now try this yourself. Run `lsblk` on your system and find at least one disk with multiple partitions. Then run `sudo fdisk -l /dev/<device>` on that disk and identify the partition type (GPT or MBR) and what each partition is used for. On most modern Linux installs, you'll see at least a boot partition and a root partition.

## Real-World Example

This is the workflow for adding a second disk to a Linux server — a common task when you need dedicated storage for databases, logs, or backups.

The scenario: a new 500 GB disk has been attached. The server is running. Nothing is mounted on the new disk yet.

```bash
# Step 1: Confirm the kernel sees the new disk and identify its device name
lsblk
```

```text
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sda      8:0    0   50G  0 disk
├─sda1   8:1    0    1G  0 part /boot/efi
├─sda2   8:2    0    8G  0 part [SWAP]
└─sda3   8:3    0   41G  0 part /
sdb      8:16   0  500G  0 disk
```

`sdb` is the new disk — no partitions yet, nothing mounted. Confirm it is the right device before proceeding.

```bash
# Step 2: Verify the disk is empty and check its current state
sudo fdisk -l /dev/sdb
```

```text
Disk /dev/sdb: 500 GiB, 536870912000 bytes, 1048576000 sectors
Disk model: DataDisk-500
Units: sectors of 1 * 512 = 512 bytes
Sector size (logical/physical): 512 bytes / 4096 bytes
I/O Size (minimum/optimal): 4096 bytes / 4096 bytes
```

No "Disklabel type" line means no partition table exists. The disk is completely raw.

```bash
# Step 3: Check the UUID of the existing disks to confirm sdb is new and unknown
blkid /dev/sda1 /dev/sda2 /dev/sda3
```

```text
/dev/sda1: UUID="A1B2-C3D4" TYPE="vfat" PARTLABEL="EFI System Partition" PARTUUID="..."
/dev/sda2: UUID="12345678-abcd-ef01-2345-6789abcdef01" TYPE="swap" PARTUUID="..."
/dev/sda3: UUID="abcdef01-2345-6789-abcd-ef0123456789" TYPE="ext4" PARTUUID="..."
```

`blkid` returns nothing for `/dev/sdb` — it has no recognizable structure, confirming it is truly empty.

> ⚠️ **WARNING:** The next command opens an interactive session that can destroy the partition table on any disk you target. Double-check that the device path (`/dev/sdb` here) is the blank new disk, not an existing disk with data. There is no undo after writing the partition table. Verify with `lsblk` and `fdisk -l` before proceeding.

```bash
# Step 4: Open fdisk on the new disk
sudo fdisk /dev/sdb
```

Inside fdisk, you will see a prompt. Type these commands in sequence:

```text
Welcome to fdisk (util-linux 2.39.3).
Changes will remain in memory only, until you decide to write them.
Be careful before using the write command.

Device does not contain a recognized partition table.
Created a new DOS (MBR) disklabel with disk identifier 0x...

Command (m for help): g
Created a new GPT disklabel (GUID: ...).

Command (m for help): n
Partition number (1-128, default 1):
First sector (2048-1048575966, default 2048):
Last sector, +/-sectors or +/-size{K,M,G,T,P} (2048-1048575966, default 1048575966): +500G

Created a new partition 1 of type 'Linux filesystem' and of size 500 GiB.

Command (m for help): p
Disk /dev/sdb: 500 GiB, 536870912000 bytes, 1048576000 sectors
...
Device     Start        End    Sectors  Size Type
/dev/sdb1   2048 1048574975 1048572928  500G Linux filesystem

Command (m for help): w
The partition table has been altered.
Calling ioctl() to re-read partition table.
Syncing disks.
```

The fdisk commands used here:
- `g` — create a new GPT partition table (always use GPT for new disks)
- `n` — new partition; accept defaults for number and start sector; specify `+500G` for size
- `p` — print the proposed layout before committing; verify it looks right
- `w` — write the partition table to disk; this is the point of no return

```bash
# Step 5: Verify the kernel picked up the new partition
lsblk /dev/sdb
```

```text
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sdb      8:16   0  500G  0 disk
└─sdb1   8:17   0  500G  0 part
```

```bash
# If lsblk doesn't show sdb1 immediately, force the kernel to re-read
sudo partprobe /dev/sdb
```

```bash
# Step 6: Get the UUID for this partition — you'll need it for fstab
blkid /dev/sdb1
```

```text
/dev/sdb1: PARTUUID="9f4a2c1b-3d5e-4f6a-b7c8-d9e0f1a2b3c4"
```

Note that `blkid` returns only a `PARTUUID` here, not a `UUID`. That's because the partition exists but has no filesystem on it yet. Once you run `mkfs` (covered in 10.2), a filesystem UUID will appear. Use the filesystem UUID — not the PARTUUID — in `/etc/fstab`.

A senior sysadmin reviewing this workflow would check two things before signing off: that `g` was used (GPT, not the default MBR) and that `p` was run before `w` so the layout was visually confirmed. They would also write down the UUID immediately after `blkid`, because finding UUIDs later when the disk is mounted and in use is slightly more work.

## ⚠️ Gotchas

### Writing the Partition Table Is Irreversible

The biggest mistake beginners make is typing `w` in fdisk without carefully confirming they have the right device. You are looking at a prompt that looks like any other shell prompt. It does not shout at you. It does not ask "are you sure?" by default.

```bash
# This opens fdisk — the device name is the only thing that matters here
sudo fdisk /dev/sda   # ← is this really the empty new disk?
```

```text
Command (m for help): w
The partition table has been altered.
```

That's it. Whatever partition table was on `/dev/sda` is now gone. If `/dev/sda` was your existing system disk with data on it, the partition table that described where your root filesystem, swap, and boot partition live is now overwritten. The data itself is not immediately destroyed (the blocks are still there), but the map to the data is gone, and recovery requires specialized tools and expertise.

The correct sequence before any fdisk session: run `lsblk` to see all devices, run `sudo fdisk -l /dev/<target>` to see the current state of the disk you intend to edit, and confirm that it is what you think it is. If you are in a VM, the disk model name in `fdisk -l` output will tell you. If you are on bare metal with multiple disks, double-check the size. Never rely only on device name ordering.

### Device Names Can Swap Between Reboots

The kernel assigns `/dev/sda`, `/dev/sdb`, etc. based on the order in which it detects drives during boot. This order is not guaranteed to be stable. A firmware update, a BIOS setting change, a different PCIe slot — any of these can cause the OS drive to become `/dev/sdb` on next boot when it was `/dev/sda` before.

```bash
# This is wrong — never put a device path in fstab
/dev/sda3   /       ext4   defaults   0 1
```

If `/dev/sda3` becomes `/dev/sdb3` after a reboot, that `fstab` line matches nothing. The system will fail to mount root and drop you into emergency mode — if you're lucky. On some configurations it will just hang.

```bash
# This is correct — always use UUID in fstab
UUID=abcdef01-2345-6789-abcd-ef0123456789   /   ext4   defaults   0 1
```

UUIDs are written into the filesystem metadata at format time. They don't change when the disk is moved to a different port, a different server, or a different slot. They are the stable identifier. Always use them in `/etc/fstab`.

### Partitioning a Mounted Disk — fdisk Lets You Try

`fdisk` will not stop you from opening a session on a disk that has mounted partitions. You can open `/dev/sda` while `/dev/sda3` is mounted as `/`. You can type `w`. The write will succeed. But when `fdisk` tries to notify the kernel to re-read the partition table, the kernel will see that partitions on this device are in use and refuse. Your in-kernel partition table is now stale — the on-disk table says one thing, the running kernel's view says another. You need to reboot to reconcile them.

```bash
# Check if any partitions on a disk are mounted before editing
lsblk /dev/sda
```

```text
NAME   MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
sda      8:0    0   50G  0 disk
├─sda1   8:1    0    1G  0 part /boot
└─sda2   8:2    0   49G  0 part /
```

If `MOUNTPOINTS` is non-empty for any partition on the disk you intend to edit, you cannot do the edit on a live system without a reboot. For data disks (not the OS disk), you can unmount them first, edit the partition table, and `partprobe` to reload. For the OS disk itself, you have to boot from a live USB or recovery environment to repartition.

### `fdisk` on a GPT Disk Shows a "GPT PMBR Size Mismatch" Warning

If a disk's physical size has changed after a GPT partition table was written — for example, if a virtual disk was expanded — `fdisk` will show this:

```text
GPT PMBR size mismatch (41943039 != 83886079) will be corrected by write.
```

This happens because GPT stores the disk size in two places: the primary header at LBA 1 and the backup header at the last LBA. When the disk grows, these recorded sizes no longer match the actual disk size. `fdisk` offers to fix it, but the more reliable path is `gdisk`, which has a dedicated recovery menu (option `r`, then `e` to move the backup GPT to the end of the disk).

```bash
# Fix a mismatched GPT on an expanded disk using gdisk
sudo gdisk /dev/sdb
```

```text
GPT fdisk (gdisk) version 1.0.9

Partition table scan:
  MBR: protective
  BSD: not present
  APM: not present
  GPT: present

Found valid GPT with protective MBR; using GPT.

Command (? for help): r
Recovery/transformation command (? for help): e
Moving backup data structures to the end of the disk

Recovery/transformation command (? for help): w

Final checks complete. About to write GPT data.  THIS WILL OVERWRITE EXISTING
PARTITIONS!!

Do you want to proceed? (Y/N): Y
OK; writing new GUID partition table (GPT) to /dev/sdb.
The operation has completed successfully.
```

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `fdisk`, `gdisk`, `parted` all available; `gdisk` may need `apt install gdisk`; `smartmontools` needs `apt install smartmontools` |

`smartmontools` (for `smartctl`) is a separate package on all distros — it is not installed by default. `hdparm` is also separate on most distros.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `fdisk` | Interactive partitioning, both MBR and GPT; everyday use | Ships with `util-linux`; always present; full GPT support since util-linux 2.29 |
| `gdisk` | GPT-only workflows; fixing GPT corruption; GUID-specific operations | Separate package (`gptfdisk`); more GPT recovery options than `fdisk` |
| `parted` | Scripted/automated partitioning; used by installers | Non-interactive mode (`parted /dev/sdb mkpart ...`); changes take effect immediately — no `w` to commit |
| `cfdisk` | Interactive TUI with a more visual layout than `fdisk` | Same underlying library as `fdisk`; good for humans, not scripts |
| `sfdisk` | Scripted partitioning; partition table backup and restore | Reads/writes partition table in a scriptable format; used by automation tools |

The honest recommendation: use `fdisk` for hands-on work on modern systems. It handles both GPT and MBR, it's always present without installing anything, and its interactive menu is well-documented. Use `gdisk` when you specifically need to repair a corrupt GPT header or work with GPT's UUID features directly. Use `parted` in scripts and automation because it can be invoked non-interactively in a single command. Never use `parted` interactively if you're new to it — changes to the partition table apply immediately without a `w` confirmation step.

## Trade-offs

**Where this shines:**

- GPT partitioning gives every partition a UUID baked into the partition table itself — you can identify a partition even before it has a filesystem
- Separating the OS disk into dedicated partitions for `/`, `/boot`, `/home`, and swap provides hard isolation: a full `/home` doesn't kill the OS, and a corruption in `/boot` doesn't reach `/home`
- Dedicated partition types tell automated tools (systemd, udev) what to do with a partition on discovery — an EFI partition with type `EF00` is automatically handled by the bootloader without configuration

**Where this costs you:**

- Partition boundaries are fixed at creation time; resizing a partition later requires unmounting, resizing the partition table, and resizing the filesystem separately — all of which carry risk
- Repartitioning a live OS disk is not possible without a live environment; you can extend the last partition on some setups, but any other structural change requires downtime
- Getting partition alignment wrong (not aligning to 4K physical sector boundaries or 1MiB boundaries) causes permanent performance degradation on SSDs and NVMe drives that no amount of later tuning can fix

**Performance note:** On SSDs and NVMe drives, partition alignment matters. Modern tools (`fdisk`, `gdisk`, `parted`) default to 1MiB alignment, which is correct for all current hardware. If you are working with older tools or manually specifying start sectors, align to a multiple of 2048 sectors (1MiB at 512-byte sectors) or you will cause unaligned writes that halve write performance on the affected partition.

## When to Use / When to Avoid

✅ **Use GPT when:**
- The disk is larger than 2TB (MBR physically cannot address it)
- The system uses UEFI firmware (GPT is required for UEFI boot)
- You need more than four partitions
- The disk will be used in any modern system — this is essentially always

✅ **Use MBR only when:**
- The target system has legacy BIOS that cannot boot from GPT
- You are creating a disk image for compatibility with very old hypervisors or embedded systems
- You explicitly know the entire toolchain downstream only speaks MBR

❌ **Avoid repartitioning live mounted disks:**
- Editing the partition table of a mounted OS disk without a recovery environment is how you get a server that doesn't come back up after a reboot
- Use a live USB, netboot environment, or cloud provider console for OS disk repartitioning

❌ **Avoid using device paths (`/dev/sda`) in persistent configuration:**
- Never put `/dev/sda1` in `/etc/fstab`, systemd `.mount` units, or any config that persists across reboots
- Always use UUID (filesystem UUID from `blkid`) or LABEL

⚠️ **Common mistakes:**
- Creating a partition without the EFI type code on a UEFI system and wondering why the bootloader install fails
- Forgetting to run `partprobe` after writing a new partition table, then wondering why `/dev/sdb1` doesn't exist yet
- Using `+100%FREE` or equivalent in `parted` and accidentally leaving no room for the GPT backup header at the end of the disk

## Veteran Perspective

The newcomer mistake is not knowing the commands. The intermediate mistake is knowing the commands but not the blast radius.

I have seen people run `fdisk /dev/sda` on a production database server while trying to work on `/dev/sdb`, realize their mistake at the `w` prompt, quit out, and think they were fine — because they didn't write. That's actually fine. But I've also seen someone type `w` by reflex before they read the screen, and then spend the next six hours recovering a database from backup because the partition table on the OS disk was gone. The commands are not hard. The attention required to not destroy something is the hard part.

The intermediate gap I see most often: engineers who know how to partition a disk but put device names in `/etc/fstab` because it worked fine during the build. It works fine until the third reboot after a firmware update, at 3am, when the system doesn't come back up because `/dev/sda` is now `/dev/sdb`. This is not a theoretical risk. It happens. Always use UUID.

The deeper design principle behind all of this is that Linux separates the idea of "where data lives" from "what device currently holds it." UUIDs, labels, and `/dev/disk/by-*` symlinks are all expressions of this principle: stable identifiers that are independent of hardware detection order. The partition table is hardware-level metadata. The UUID is filesystem-level metadata. The mount point is OS-level configuration. They are three separate layers that compose together — don't conflate them.

On disk health: `smartctl -a /dev/sda` is not optional on bare metal. A disk that is about to fail will almost always tell you before it fails, if you read the SMART data. Reallocated sector counts, uncorrectable errors, and pending sector counts are the numbers to watch. If you're setting up a new bare-metal server and you don't check SMART on every disk before you write data to it, you are operating on faith. Production systems deserve better than faith.

One more thing: `parted`'s non-interactive mode (`parted /dev/sdb mkpart primary ext4 1MiB 100%`) is extremely useful in automation, but it has a trap for the unwary. Changes take effect immediately — there is no buffered-in-memory staging like `fdisk`. If you call `parted` in a script with the wrong device path, the damage is done before the next line executes. Always add a confirmation check in scripts before invoking `parted`.

## Try It Yourself

1. Run `lsblk` on your current system. Identify every disk and every partition. For each partition, note its type (`disk`, `part`) and whether it is mounted. What is the device name of your root partition?

2. Run `sudo fdisk -l` with no device argument. This lists every disk `fdisk` can find. How many disks do you have? What partition table type (GPT or MBR) does each one use?

3. Run `blkid` on your root partition (e.g., `sudo blkid /dev/sda3`). Find the UUID. Now open `/etc/fstab` and confirm that the same UUID appears there. If your system uses device names instead of UUIDs, you have found a real issue worth fixing.

4. Install `smartmontools` (`apt install smartmontools` or `dnf install smartmontools`) and run `sudo smartctl -a /dev/sda`. Read the SMART attributes section. Look for the "Reallocated_Sector_Ct" and "Pending_Sector_Count" values. What do yours show?

5. **Challenge:** In a VM, attach a second blank disk (1–10 GB). Without rebooting, use `fdisk` to create a GPT partition table with two partitions: an 8 GB Linux filesystem partition and a remaining swap partition with the correct type code (`8200`). Use `lsblk` and `blkid` to confirm both partitions are visible to the kernel. Then use `sudo mkswap /dev/vdb2` (or `sdb2`) and `sudo swapon /dev/vdb2` to activate the swap. Verify with `free -h` that the new swap is active.

## Summary

- Block devices are named by driver: SATA disks use `/dev/sda`, NVMe disks use `/dev/nvme0n1`, partitions add a number suffix; device names are not stable across reboots — always use UUID for persistent references
- GPT is the correct partition table format for all new disks; MBR is a 1983 design with a 4-partition limit and a 2TB ceiling that has no place on modern hardware
- `fdisk` is the go-to interactive tool — changes are staged in memory and only committed when you type `w`; `parted` is for scripting and automation; `gdisk` is for GPT-specific recovery
- After writing a new partition table, run `partprobe /dev/<device>` to notify the kernel; if any partition on the disk is mounted, you'll need to reboot instead
- Writing a partition table is irreversible — confirm the target device with `lsblk` and `fdisk -l` before every `fdisk` session, and always run `p` to verify the layout before `w`
