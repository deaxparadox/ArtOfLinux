# [10.5] — Disk Usage

## The Problem This Solves

Your monitoring system fires a disk full alert. You SSH in. You run `df -h`. The disk is at 98%. You start looking for the culprit — `du -sh /var/log` comes back clean, `du -sh /home` looks fine, `du -sh /tmp` is nothing. You've looked everywhere and you've accounted for maybe 60% of the disk. The other 38% has vanished. The system is throwing "No space left on device" errors, but `du` says there's space.

This happens on real servers. Regularly.

Or the inverse: `df -h` shows the disk is fine — 40% free. But every `touch newfile` fails with "No space left on device." The disk is not full. Something else is full.

Without knowing how disk space accounting actually works — what `df` measures, what `du` measures, why they can disagree, and what a third, invisible category of usage exists — you will chase your tail during every disk-related incident. The tools exist. The knowledge of when each one lies to you is what matters.

---

## Mental Model

Think of a library. `df` asks the librarian: "How many shelves do you have, and how many are occupied?" The librarian counts the shelves — including shelves with books that have been "checked out for disposal" but haven't physically left yet. `du` walks the stacks itself, measuring only the books it can actually see on the shelves. If a book was removed from the catalog (deleted) but is still sitting on a shelf because a reading group has it checked out (a process has it open), `du` won't find it — it's not in the catalog. But `df` sees the shelf is occupied.

That gap — between what `du` finds by walking directories and what `df` sees as consumed at the filesystem level — is where disk space disappears.

---

## Theory

Linux has two separate accounting systems for disk space, and they measure different things. Conflating them is the root cause of most disk-investigation failures.

**`df` — filesystem-level accounting.** `df` queries the filesystem's own bookkeeping structures. It asks the kernel: how many blocks does this filesystem have total, and how many are marked as in-use? A block is marked in-use if any inode references it — whether that inode is reachable from the directory tree or not. When you delete a file, the directory entry is removed (the filename disappears), but the inode and its blocks are only released when every open file descriptor referencing that inode is closed. Until then, the blocks remain in-use. `df` counts them.

**`du` — directory tree traversal.** `du` works by walking the directory tree recursively. It finds every file it can reach through directory entries, calls `stat()` on each one, and sums the `st_blocks` values. Files with no directory entry — deleted files held open by processes — are invisible to it. `du` cannot find them because there is no path to follow.

**The gap between them** is the deleted-but-open category. It is the single most common cause of `df` and `du` disagreeing significantly. On a long-running server with rotating logs or a crashed process holding a large file open, this gap can be tens of gigabytes.

**Inodes — the second dimension.** Disk space is not just blocks. Every file also consumes one inode, and the number of inodes on a filesystem is fixed at format time (on ext4; btrfs is more flexible). A filesystem can exhaust its inodes while still having plenty of block space. At that point, creating any new file fails — not because the disk is full, but because the inode table is full. This affects servers with huge numbers of small files: mail spools, session caches, package manager caches, build artifact trees, and PHP session directories are the classic culprits.

**Root-reserved blocks.** ext4 reserves 5% of total blocks for the root user by default. This is a deliberate safety feature: if a runaway process fills the disk, root can still log in and clean up. A filesystem may show 100% full to regular users while `df` shows a few percent still available. The reserved percentage is tunable.

## How It Works (Mechanics)

When you run `df`, it calls the `statfs()` system call on the mounted filesystem. The kernel returns a `struct statfs` containing `f_blocks` (total blocks), `f_bfree` (blocks free), `f_bavail` (blocks available to non-root users), `f_files` (total inodes), and `f_ffree` (free inodes). This is kernel-reported state — it reflects the filesystem's internal data structures, not the directory tree.

When you run `du`, it calls `opendir()` and `readdir()` to walk the directory tree, then `stat()` on each entry. It sums the `st_blocks` field from each `stat` result. It cannot discover files that are not reachable through the directory hierarchy.

The deleted-but-open file mechanism works through the kernel's VFS layer. When a process opens a file, the kernel increments the reference count on that file's inode. When you `unlink()` a file (what `rm` calls under the hood), the directory entry is removed and the inode's link count drops by one — but the inode itself is not freed until its reference count also reaches zero. Processes holding open file descriptors hold a reference. The inode, and all blocks it points to, remain allocated until the last file descriptor is closed or the process exits.

You can observe this with `lsof`. `lsof` reads `/proc/<pid>/fd/` for every process and resolves each file descriptor's path. When the kernel reports a file whose path has been unlinked, `lsof` marks it `(deleted)` in its output. The file descriptor is real; the size shown is still the size of the held-open file.

```bash
# strace shows what df actually calls
strace -e statfs df /
```

```text
statfs("/", {f_type=EXT2_SUPER_MAGIC, f_bsize=4096, f_blocks=10484668,
f_bfree=4193024, f_bavail=3672960, f_files=2621440, f_ffree=2587321, ...}) = 0
...
```

The gap between `f_bfree` (blocks free including root reserve) and `f_bavail` (blocks free to non-root) is the root reservation in action.

## Basic Example

```bash
# Show filesystem disk usage, human-readable
df -h
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        20G   14G  5.3G  73% /
tmpfs           2.0G  1.2M  2.0G   1% /dev/shm
/dev/sda2       100G   42G   53G  44% /home
```

```bash
# Show with filesystem type
df -hT
```

```text
Filesystem     Type   Size  Used Avail Use% Mounted on
/dev/sda1      ext4    20G   14G  5.3G  73% /
tmpfs          tmpfs  2.0G  1.2M  2.0G   1% /dev/shm
/dev/sda2      ext4   100G   42G   53G  44% /home
```

```bash
# Show inode usage
df -i
```

```text
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda1      1310720 312419  998301   24% /
/dev/sda2      6553600 187234 6366366    3% /home
```

```bash
# Show disk usage for the current directory, human-readable summary
du -sh .
```

```text
4.7G    .
```

```bash
# Show disk usage one level deep — the most useful quick form
du -sh /var/*
```

```text
8.0K    /var/backups
156M    /var/cache
4.0K    /var/games
1.1G    /var/lib
8.0K    /var/local
0       /var/lock
381M    /var/log
4.0K    /var/mail
4.0K    /var/opt
0       /var/run
20K     /var/spool
4.0K    /var/tmp
```

The `df -h` output shows per-filesystem usage — not per-directory. Each line is a separate filesystem. `tmpfs` filesystems are in-memory and reset on reboot. The `du -sh /var/*` form is the fastest way to find which subdirectory is consuming space — the `-s` flag summarizes each argument rather than recursing into it, and `-h` gives human-readable output.

Now try this yourself. Run `df -h` on your system and note which filesystem `/var/log` lives on. Run `df -i` and compare the inode usage percentages to the block usage percentages. On most desktop machines they will be far apart — inode exhaustion is a server problem.

## Real-World Example

A production web server returns "No space left on device" errors for new log writes and temp file creation. `df -h` shows the root filesystem at 97%. You start investigating.

```bash
# Step 1: Confirm which filesystem is the problem
df -h
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/xvda1       30G   29G  488M  97% /
/dev/xvdb1      200G   12G  176G   7% /data
```

```bash
# Step 2: Find the top consumers under /
# --max-depth=1 avoids traversing the whole tree at once
du -sh --max-depth=1 /* 2>/dev/null | sort -h
```

```text
0       /proc
0       /sys
4.0K    /root
8.0K    /lost+found
12K     /boot
1.2M    /etc
5.6M    /bin
...
2.1G    /var
6.7G    /usr
18G     /home
```

```bash
# Step 3: Drill into /var
du -sh /var/* 2>/dev/null | sort -h
```

```text
4.0K    /var/backups
128M    /var/cache
44M     /var/lib
0       /var/lock
312M    /var/log
0       /var/run
8.0K    /var/spool
```

```bash
# Step 4: All the obvious directories add up to about 21G.
# But df says 29G is used. There's an 8G gap.
# Check for deleted files held open by running processes.
lsof -nP +L1 2>/dev/null | head -30
```

```text
COMMAND     PID   USER   FD   TYPE DEVICE    SIZE/OFF NLINK NODE NAME
nginx      1842   root    4u   REG  202,1  8589934592     0 1048 /var/log/nginx/access.log (deleted)
```

There it is. `nginx` is holding an 8 GB log file open. Someone rotated the logs by deleting the file directly instead of truncating it or using `logrotate` properly. The file descriptor is still open in the nginx process, so the kernel has not freed the 8 GB of blocks.

```bash
# Step 5: The cleanest fix is to reload nginx so it re-opens its log file descriptors
# This is safe — nginx handles SIGHUP by re-opening log files gracefully
systemctl reload nginx
```

After the reload, nginx closes its old file descriptors and opens fresh log files. The kernel releases the inode and all 8 GB of blocks. `df -h` immediately shows the space recovered.

```bash
# Alternative if you cannot restart the process:
# Truncate the file through the /proc file descriptor without closing it
# This zeroes the file content while keeping the fd valid
> /proc/1842/fd/4
```

Now check inode exhaustion — a separate failure mode on the same server:

```bash
# Inode check
df -i
```

```text
Filesystem      Inodes    IUsed    IFree IUse% Mounted on
/dev/xvda1     1966080  1966080        0  100% /
```

100% inode usage with block space still available. The system cannot create any new files.

```bash
# Find which directory has the most files
find / -xdev -printf '%h\n' 2>/dev/null | sort | uniq -c | sort -rn | head -20
```

```text
 942847 /var/spool/postfix/deferred
 821033 /var/spool/postfix/active
  42817 /var/cache/apt/archives
  ...
```

The mail spool has accumulated nearly two million small files — one per queued email. Each consumes an inode. The block usage is modest (small emails), but the inode table is full.

```bash
# Clean the postfix mail queue (review what's there first)
postsuper -d ALL deferred
```

The walkthrough: `du` found the big directories in steps 2–3, but the 8 GB gap between `du` and `df` pointed to deleted-but-open files. `lsof -nP +L1` specifically lists files with zero link count (the `+L1` flag means "link count less than 1") — that is the precise filter for deleted files held open. The inode check is a separate `df -i` call and should always be part of a disk-full investigation.

## ⚠️ Gotchas

### `df` and `du` Disagree — the Gap Is Real, Not a Bug

You run `du -sh /` and get 18 GB. `df -h` says 27 GB is used. You assume one of the tools is wrong. Neither is wrong — they measure different things.

```bash
# du walks the directory tree — only sees files reachable by path
du -sh /
```

```text
18G     /
```

```bash
# df queries filesystem-level block accounting
df -h /
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        30G   27G  1.7G  94% /
```

The 9 GB difference is almost certainly deleted-but-open files. `du` cannot see them because they have no directory entry. `df` counts them because their blocks are still marked in-use.

```bash
# Find the deleted files filling that gap
lsof -nP +L1
```

The underlying rule: the directory tree is not the same as the filesystem block map. Files exist in both dimensions. `rm` removes them from the directory tree immediately, but the blocks are held until all file descriptors referencing that inode are closed.

### Inode Exhaustion Looks Identical to Disk Full

Your application throws "No space left on device." You check `df -h`. The filesystem is at 52% — plenty of space. You are confused.

```bash
df -h /
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   26G   22G  55% /
```

Looks fine. But:

```bash
df -i /
```

```text
Filesystem      Inodes  IUsed   IFree IUse% Mounted on
/dev/sda1      3276800 3276800       0  100% /
```

Zero free inodes. Every attempt to create a new file — regardless of its size — fails with `ENOSPC`. The error is identical to a block-full condition from the application's perspective.

```bash
# Find the directory with the most files
find / -xdev -printf '%h\n' 2>/dev/null | sort | uniq -c | sort -rn | head -10
```

Common causes: PHP session files in `/var/lib/php/sessions`, Postfix mail queue in `/var/spool/postfix`, pip or npm cache files, build artifact directories, and container image layers. The fix is to delete the files, not to try to free block space.

### `du -sh /` on a Live Server Is Slow and Will Miss Files

Running `du -sh /` across a full production filesystem is slow — potentially minutes — and can generate substantial I/O load. More critically, it will not find the deleted-but-open files you may be hunting.

```bash
# This walks the ENTIRE filesystem. On a busy server this can take minutes
# and will spike iowait. It also misses deleted-but-open files entirely.
du -sh /
```

The right approach during a disk investigation is to work top-down with `--max-depth=1` or `du -sh /path/*` to narrow the search, and to run `lsof -nP +L1` in parallel to check for the deleted-but-open category. Never start with `du -sh /` on a production system.

```bash
# This is faster and immediately useful
du -sh --max-depth=1 /* 2>/dev/null | sort -h
```

### Root-Reserved Blocks — the "Full" Disk That Isn't

ext4 reserves 5% of blocks for the root user by default. A regular user filling a disk will see "No space left on device" when `df -h` shows 5% available.

```bash
df -h /dev/sda1
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        100G   95G  5.0G  95% /
```

The `df` output shows 5G available. But that 5G is the reserved portion — a normal user process cannot use it. From the process's perspective, the disk is full.

```bash
# Check reserved blocks with tune2fs (run as root, read-only query)
tune2fs -l /dev/sda1 | grep -i reserved
```

```text
Reserved block count:     524288
Reserved GDT blocks:      100
```

On a data volume where root-only reservation makes no sense, you can reduce it:

> ⚠️ **WARNING:** Only reduce root-reserved blocks on non-root, data-only filesystems. The reservation exists to keep root functional when the disk fills. Removing it from the root filesystem means a full disk can lock out root.

```bash
# Reduce reservation to 1% on a data volume — not the root filesystem
tune2fs -m 1 /dev/sdb1
```

### `ncdu` Shows a Different Number Than `df`

`ncdu` is a directory tree browser — it runs `du` logic internally. It will not show deleted-but-open files. If `ncdu` shows the filesystem is 60% used but `df` says 90%, the 30% gap is in the deleted-but-open category that no directory browser can see.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `df` | Check filesystem-level usage; check inodes; check reserved space | Always available; no install needed; reflects true kernel state |
| `du` | Find which directory is consuming space; build a usage tree | Always available; misses deleted-but-open files |
| `ncdu` | Interactive drill-down on a server when `du` output is hard to navigate | Requires install; wraps `du` logic; also misses deleted-but-open files |
| `lsof -nP +L1` | Find deleted-but-open files consuming space invisibly | Always available; essential for the df/du gap |
| `lsblk` | See raw device sizes and partition layout | Shows physical sizes, not filesystem usage |
| `find -xdev -printf '%h\n' \| sort \| uniq -c \| sort -rn` | Find which directory is exhausting inodes | Slow but accurate; use when `df -i` shows 100% |

The right diagnostic sequence is `df -h` (blocks) → `df -i` (inodes) → `du -sh --max-depth=1 /* | sort -h` (find block consumers) → `lsof -nP +L1` (find deleted-but-open). `ncdu` is a quality-of-life improvement over raw `du` output when you're doing an interactive investigation and want to navigate the tree without repeatedly re-running `du`.

## Trade-offs

**Where this shines:**

- `df` is instantaneous — it calls `statfs()` once per filesystem and returns. No traversal, no I/O cost.
- `du` is accurate for the directory tree and naturally reveals the structure of consumption, not just the total.
- `ncdu` presents large filesystems in a navigable form that raw `du` output doesn't.
- `lsof +L1` catches the category that nothing else sees — there is no substitute for it in the deleted-but-open case.

**Where this costs you:**

- `du` on a large filesystem is slow and I/O-intensive — it must `stat()` every file. On a filesystem with millions of files, this can take minutes and noticeably impact server performance.
- `du` gives a misleading picture when deleted-but-open files are large — it will consistently undercount actual usage.
- Inode exhaustion is invisible until it hits 100% — there is no gradual warning from typical monitoring unless you explicitly monitor `df -i`.
- The root-reserved block behavior surprises nearly everyone the first time they encounter it.

**Performance note:** `du -sh /` on a filesystem with 10M+ files can take 5–15 minutes and push sustained iowait above 30% on spinning disk. On SSDs the time is much shorter, but the I/O still competes with production workloads.

## When to Use / When to Avoid

✅ **Use `df -h` when:**
- You get a disk full alert and need to confirm which filesystem is actually full
- You want to see all filesystem usages at once
- You need the true picture including deleted-but-open files

✅ **Use `df -i` when:**
- `df -h` shows space available but writes are still failing
- You're running a server with large numbers of small files (mail, web sessions, caches)
- Add it to your standard disk-full investigation checklist — always check inodes

✅ **Use `du -sh /path/* | sort -h` when:**
- `df` has confirmed a filesystem is full and you need to find what's consuming it
- You want to identify the top-level directory to drill into

✅ **Use `lsof -nP +L1` when:**
- `du` cannot account for all the space `df` says is used
- You suspect a log rotation or crash has left files held open

✅ **Use `ncdu` when:**
- You're doing an interactive investigation and want to navigate the tree
- You have permission to install tools on the server

❌ **Avoid running `du -sh /` on a production system:**
- Use `--max-depth=1` and drill down; never traverse the whole tree at once
- On busy servers this causes measurable I/O impact

❌ **Avoid trusting only `du` when `df` disagrees significantly:**
- The gap is real; `du` is not giving you the full picture

⚠️ **Common mistakes:**
- Checking block usage but forgetting to check inode usage — always run both `df -h` and `df -i` during a disk investigation
- Running `logrotate` by deleting log files directly instead of truncating them — this leaves the process holding a deleted inode open
- Treating `ncdu` output as the ground truth for filesystem usage — it has the same blind spot as `du`

## Veteran Perspective

Disk full incidents are one of the most common pages a sysadmin receives, and they are almost always solved within five minutes by someone who knows the two-minute checklist: `df -h`, `df -i`, `du -sh --max-depth=1 /* | sort -h`, `lsof -nP +L1`. The same incidents take 45 minutes for someone who only knows `du` and keeps rerunning it expecting a different answer.

The deleted-but-open file case specifically trips up people who learned about disk usage from documentation. The documentation teaches `du`. The documentation is not wrong — `du` does what it says. But the documentation rarely mentions that `du` has a structural blind spot that only `df` can reveal. You do not discover this gap from reading; you discover it at 2am when `du` swears there is space and the kernel swears there is not.

The inode exhaustion case is worse, because it produces the exact same error message as a block-full condition. "No space left on device" means either your blocks are full or your inodes are full. The error code is `ENOSPC` for both. Your application cannot tell the difference. Only `df -i` tells the difference. Make it a reflex.

The root-reserved blocks gotcha has a history. In the early days of Linux servers, a runaway process filling the root filesystem would lock out root itself — no logs, no cleanup, no recovery without a reboot. The 5% reservation exists so root always has room to maneuver. On the root filesystem, leave it. On a dedicated data volume at `/data` or `/srv`, it is reasonable to reduce it to 1%. Never eliminate it from a root filesystem — you will eventually regret it.

If you are ever in an incident review and someone asks "why did the disk fill up and the application not catch it," the answer is almost always one of three things: a process held a deleted log open, inode exhaustion wasn't being monitored, or a log rotation script deleted files directly instead of using `logrotate` with a `copytruncate` or `postrotate` reload. These are not exotic failure modes. They are the normal failure modes.

One more thing: when you find the problem and fix it, always explain why the space came back. "I reloaded nginx" is not a complete incident write-up. "nginx was holding a deleted 8 GB log file open because the log rotation script deleted the file directly without sending SIGHUP to reload file descriptors — space was recovered by reloading the process, and the rotation script was updated to use `logrotate`'s `postrotate` directive" is. The root cause matters more than the fix, because the fix without the root cause means it will happen again.

## Try It Yourself

1. Run `df -h` and `df -i` on your system. Find the filesystem that hosts `/var`. What is the inode usage percentage vs. the block usage percentage?

2. Run `du -sh /var/* 2>/dev/null | sort -h`. Which subdirectory of `/var` is using the most space? Drill one level deeper into it with the same command pattern.

3. Install `ncdu` and run `ncdu /var`. Navigate with arrow keys, press `d` to see deleted file space (it won't show any, which confirms ncdu shares du's blind spot). Press `q` to quit. Then run `lsof -nP +L1 2>/dev/null` and compare — are there any deleted-but-open files?

4. In a terminal, run `tail -f /tmp/bigfile &` to start a background process holding a file open. Then `rm /tmp/bigfile`. Run `du -sh /tmp/bigfile` — it's gone. Run `lsof -nP +L1 | grep bigfile` — it's still there. Kill the tail process with `kill %1` and check `lsof` again. Watch the space disappear.

5. **Challenge:** Write a script that produces a disk health report for a system. It should show: block usage per filesystem, inode usage per filesystem, the top 10 directories by disk usage under `/var`, and a list of any deleted-but-open files with their sizes. The script should flag any filesystem above 80% block usage or 70% inode usage as WARNING, and above 90% as CRITICAL. Run it as root and save the output to `/tmp/disk-report.txt`. (Requires `sudo` for `lsof`.)

## Summary

- `df` measures filesystem-level block and inode accounting — it includes deleted-but-open files. `du` traverses the directory tree — it cannot see files with no directory entry. The gap between them is always deleted-but-open files.
- "No space left on device" means either blocks are full (`df -h`) or inodes are full (`df -i`) — always check both.
- `lsof -nP +L1` is the only tool that reveals deleted-but-open files; it is essential when `df` and `du` disagree significantly.
- ext4 reserves 5% of blocks for root by default — reduce this only on dedicated data volumes, never on root filesystems.
- The disk-full investigation checklist is: `df -h` → `df -i` → `du -sh --max-depth=1 /* | sort -h` → `lsof -nP +L1`.
