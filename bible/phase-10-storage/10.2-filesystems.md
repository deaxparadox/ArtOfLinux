## The Problem This Solves

A raw partition is a range of sectors on a block device. It has no concept of a file. It doesn't know where one file ends and the next begins. It has no idea who owns what, whether a file was partially written, or whether it's even safe to read. Without a filesystem, all you have is a flat array of numbered blocks.

Early systems used their own ad-hoc schemes for organizing those blocks. Programs wrote to specific sector offsets they had hard-coded. There was no consistent way to list what was stored, no protection against partial writes corrupting data, and no concept of a directory hierarchy.

The filesystem is the layer that imposes structure on those raw blocks: it defines where inodes live, how directory entries are indexed, which blocks belong to which file, and how the on-disk state is kept consistent. Every time you run `ls`, read a log file, or write a database row, the filesystem is doing the translation between the path you see and the blocks on disk.

The question is which filesystem you choose — and that choice has real consequences. Different filesystems make different trade-offs around performance, reliability, feature set, and administrative flexibility. A filesystem optimized for large sequential writes will behave differently on a database full of random 4KB reads. A filesystem with copy-on-write semantics gives you free snapshots but adds overhead to every write. A filesystem that cannot shrink will strand you if you underprovisioned a partition.

You make this choice once, at format time. You can't change it without reformatting and losing everything.

## Mental Model

Think of a raw partition as a freshly poured concrete floor with nothing on it. The filesystem is the blueprint for what gets built on top: where the walls go (directory structure), where items are catalogued (inode table), how to find anything quickly (extent tree or block bitmap), and what to do if the building is interrupted mid-construction (journal).

Different blueprints produce different buildings. ext4 builds a solid, conventional structure that everyone knows how to maintain. XFS builds a high-performance warehouse optimized for throughput and large storage. Btrfs builds a smart building with a camera system that can take a snapshot of every room at any instant — but the camera hardware adds a little overhead to every move you make. You don't change the blueprint after the concrete sets. Pick the right one before you pour.

## Theory

### What a Filesystem Actually Does

A filesystem solves four problems simultaneously:

**Space management:** It tracks which blocks are free and which are in use, so writes go to free space and freed space is reclaimed correctly.

**File metadata:** For each file, it stores the owner, permissions, timestamps, and the mapping from file offset to disk block. This metadata lives in structures called inodes.

**Directory indexing:** It organizes filenames into a hierarchy and provides fast lookup of "given this name in this directory, which inode does it point to?"

**Consistency:** It ensures that partial writes — interrupted by a crash, power failure, or kernel panic — leave the filesystem in a recoverable state rather than a corrupt one.

### Journaling

Modern filesystems use a journal: a small, dedicated region on disk where changes are written as log entries before they are applied to the main filesystem structures. If a crash interrupts a write, the journal gives the filesystem checker enough information to replay or roll back the incomplete operation at the next mount, rather than discovering a half-written inode table.

Without a journal, recovering from an unclean shutdown meant running `fsck` over the entire filesystem — a process that could take hours on large disks. With a journal, recovery is seconds: replay the journal, done.

### The Three Filesystems You'll Actually Use

**ext4** is the direct descendant of ext2 (Linux's original filesystem) with journaling added in ext3 and performance improvements in ext4. It is the most widely deployed Linux filesystem. It uses extents (contiguous block ranges described by a start and length) rather than individual block pointers, which makes large files efficient. Delayed allocation lets the kernel batch writes and choose optimal placement. ext4 is conservative, well-understood, and supported by every Linux tool that touches filesystems. It is the default on Ubuntu and Debian.

**XFS** is a high-performance journaling filesystem from SGI, merged into the mainline kernel in 2001. It was designed for large files and high-throughput workloads, and it delivers — sequential write performance on XFS can exceed ext4 substantially on large storage. XFS scales well to very large filesystems (up to 8 exabytes, practically unlimited). It has been the default filesystem on RHEL since RHEL 7, and on Fedora before Fedora 33. The critical constraint: **XFS cannot be shrunk.** You can grow an XFS filesystem online with `xfs_growfs`, but you cannot shrink it. Plan your partition size in advance, or use LVM so you can manage the underlying volume instead.

**Btrfs** (B-tree filesystem) is a copy-on-write filesystem that became the default on Fedora (since Fedora 33) and has been the default on openSUSE for years. It provides native features that other filesystems lack: snapshots at near-zero cost, transparent compression, subvolumes (filesystem namespaces within a single filesystem), online defragmentation, and per-block checksums that detect silent data corruption. As of kernel 6.12, Btrfs core features are stable and production-ready. The caveats: RAID 5/6 support in Btrfs has had long-standing correctness issues and is officially not recommended for production data. Copy-on-write write patterns can cause fragmentation on databases and VMs with large write-heavy files. Red Hat specifically removed Btrfs from RHEL in RHEL 8 and does not support it in enterprise contexts — not because it's broken, but because XFS fits their target workloads better.

### The Others Worth Knowing

**ZFS (OpenZFS)** is a combined volume manager and filesystem originally from Sun Microsystems. It combines what LVM and a filesystem do into a single layer, adding enterprise-grade features: checksums on every block, copy-on-write snapshots, pooled storage, built-in RAID (raidz), transparent compression, and data deduplication. It is the gold standard for data integrity in storage-heavy environments. On Linux it runs as an out-of-tree module (OpenZFS) due to license incompatibility with the kernel — which means it requires a separate installation and won't be in the initramfs by default on most distros. It is the right tool for NAS systems, backup servers, and environments where silent data corruption is unacceptable.

**tmpfs** is a RAM-backed filesystem. It lives entirely in kernel memory and has no on-disk representation. Data written to tmpfs is lost on unmount (and on reboot). It is used for `/tmp`, `/run`, and `/dev/shm` on every modern Linux system. The performance is exactly what you'd expect from RAM.

**OverlayFS** is what container runtimes use. It stacks a read-only lower layer (the container image) with a writable upper layer (the container's runtime state). Reads come from the lower layer unless the upper layer has a newer copy. Writes always go to the upper layer. When a container is deleted, the upper layer is discarded — the image underneath is unchanged. Docker and Podman use OverlayFS as their default storage driver.

## How It Works (Mechanics)

### Creating a Filesystem: What mkfs Does

When you run `mkfs.ext4 /dev/sdb1`, the tool writes several distinct on-disk structures to the partition:

```text
# Conceptual layout — not a real command
Partition /dev/sdb1:

[ Superblock ]           ← filesystem metadata: size, block size, inode count, feature flags
[ Group Descriptors ]    ← table of block groups
[ Block Group 0 ]
  [ Block Bitmap ]       ← one bit per block: 0=free, 1=used
  [ Inode Bitmap ]       ← one bit per inode slot: 0=free, 1=used
  [ Inode Table ]        ← array of inode structures (256 bytes each in ext4)
  [ Data Blocks ]        ← file contents and directory entries
[ Block Group 1 ]
  ...
[ Block Group N ]
```

The superblock is the most important structure — it's the filesystem's birth certificate. It records the filesystem UUID, block size (typically 4096 bytes), total inode count, features enabled (journaling, extent trees, large files, 64-bit mode), and a clean/dirty flag that tells the kernel whether the last unmount was clean. If the dirty flag is set at mount time, the kernel knows a crash occurred and replays the journal before making the filesystem available.

### The Journal: How Consistency Works

When you write to a file on an ext4 filesystem, the kernel doesn't write directly to the data blocks. It first writes a journal transaction: a record of what is about to change. The sequence is:

```text
# Conceptual flow — not a real command
write(fd, buf, len):
  1. Kernel buffers the write in page cache
  2. Inode and block metadata changes are logged to journal
  3. Journal commit record written (fsync flushes this to disk)
  4. Later: actual data and metadata written from page cache to disk
  5. After data+metadata are on disk: journal entry can be freed
```

If a crash occurs between steps 3 and 4, the filesystem is still in a consistent state: the journal has the commit record, and at next mount the metadata changes are replayed. If a crash occurs between steps 1 and 3, the write never committed and is simply absent after recovery — the file appears as it was before the write. No corruption.

### How Btrfs CoW Works

Btrfs never overwrites a block in place. When you modify a file, Btrfs writes the new version to a free block, then updates the tree to point to the new block, then releases the old block. This is copy-on-write (CoW).

```text
# Conceptual flow — not a real command
Before write:
  inode → extent tree → Block A (contains file data)

write(fd, buf):
  Btrfs writes buf to free Block B
  Updates extent tree to point to Block B
  Marks Block A as free

After write:
  inode → extent tree → Block B (new data)
  Block A → free
```

The snapshot feature is a direct consequence of CoW: a snapshot is just a copy of the tree root pointing to the same blocks. Since writes never modify existing blocks, both the snapshot and the live filesystem can point to the same blocks safely. The snapshot only diverges as new writes come in — you only pay for blocks that actually change, not for blocks that stay the same.

### What xfs_repair Does vs. What fsck Does

`fsck` is a frontend dispatcher — it calls the appropriate filesystem-specific checker based on filesystem type: `e2fsck` for ext2/3/4, `xfs_repair` for XFS, `btrfsck` for Btrfs. For XFS specifically, `fsck.xfs` is a stub that does nothing and exits zero — XFS performs journal replay automatically at mount time. To actually check and repair an XFS filesystem, you must call `xfs_repair` directly.

Both tools require the filesystem to be unmounted. On a root filesystem, this means running from a live USB, the recovery target, or single-user mode with the root remounted read-only.

```bash
# Create an ext4 filesystem with a label
# -L sets a human-readable label (visible in blkid and findmnt)
# -m 1 reserves 1% of space for root (default is 5%; on large disks 5% wastes gigabytes)
sudo mkfs.ext4 -L mydata -m 1 /dev/sdb1
```


Now try it yourself. After mounting, run `df -h /mnt/mydata` — you'll see the 10GB partition with roughly 9.9GB available (the 1% reserved space is already accounted for). Run `tune2fs -l /dev/sdb1 | grep -E "Reserved|Volume name"` to see the reserved block count and the label you set. If your output shows a different filesystem size, the partition may have been formatted differently — `lsblk -f` will show the current filesystem type on each partition.

## Real-World Example

You're setting up a new application server. The root disk uses ext4 (handled at OS install). You have a second 500GB disk (`/dev/sdb`) that will hold application data. You need to decide on a filesystem, format it correctly for production, and add it to fstab.

The workload is a PostgreSQL database: random 8KB reads and writes, many small files in the data directory. This rules out Btrfs (CoW adds overhead to random overwrites in database files; the standard fix is to disable CoW per-directory with `chattr +C`, which negates most of Btrfs's benefits). XFS would perform well, but since you're on Ubuntu and may need to shrink later if you switch to LVM, ext4 is the safe default.

```bash
# Confirm the disk is the right one and is unformatted
sudo lsblk -f /dev/sdb
```

```text
NAME   FSTYPE FSVER LABEL UUID FSAVAIL FSUSE% MOUNTPOINTS
sdb
└─sdb1
```

```bash
# Create a partition (assumes 10.1 was completed — partition already exists)
# Format with ext4, production-grade options:
# -L pgdata    — label for identification
# -m 1         — reserve 1% for root (not 5%; 5% of 500GB = 25GB wasted)
# -E lazy_itable_init=0,lazy_journal_init=0  — fully initialize now, not lazily
#    lazy init defers inode table zeroing to a background thread after mount;
#    on production systems you want initialization complete before first use
#    so there's no background I/O competing with your first writes
```

> ⚠️ **WARNING:** The `mkfs.ext4` command below will erase all data on `/dev/sdb1`. Confirm with `lsblk` that this is the correct, empty partition before continuing.

```bash
sudo mkfs.ext4 \
  -L pgdata \
  -m 1 \
  -E lazy_itable_init=0,lazy_journal_init=0 \
  /dev/sdb1
```

```text
mke2fs 1.47.0 (5-Feb-2023)
Creating filesystem with 122094592 4k blocks and 30531584 inodes
Filesystem UUID: b2c3d4e5-f6a7-8b9c-0d1e-2f3a4b5c6d7e
Superblock backups stored on blocks:
        32768, 98304, 163840, ...

Allocating group tables: done
Writing inode tables: done
Creating journal (262144 blocks): done
Writing superblocks and filesystem accounting information: done
```

```bash
# Get the UUID — always use UUID in fstab, never /dev/sdb1
sudo blkid /dev/sdb1
```

```text
/dev/sdb1: LABEL="pgdata" UUID="b2c3d4e5-f6a7-8b9c-0d1e-2f3a4b5c6d7e" BLOCK_SIZE="4096" TYPE="ext4" PARTUUID="..."
```

```bash
# Create mount point
sudo mkdir -p /var/lib/postgresql/data

# Test mount manually first — before touching fstab
sudo mount UUID=b2c3d4e5-f6a7-8b9c-0d1e-2f3a4b5c6d7e /var/lib/postgresql/data

# Confirm it's mounted, with the right filesystem and device
findmnt /var/lib/postgresql/data
```

```text
TARGET                       SOURCE     FSTYPE  OPTIONS
/var/lib/postgresql/data     /dev/sdb1  ext4    rw,relatime
```

```bash
# Test write access
sudo touch /var/lib/postgresql/data/.write_test && echo "write: OK" \
  && sudo rm /var/lib/postgresql/data/.write_test

# Check reserved space setting
sudo tune2fs -l /dev/sdb1 | grep "Reserved block count"
```

```text
write: OK
Reserved block count:      1220945
```

```bash
# Unmount before adding to fstab — then mount -a will mount it fresh from fstab
sudo umount /var/lib/postgresql/data
```

Now add the entry to `/etc/fstab`:

```ini
# PostgreSQL data disk — formatted ext4, 1% reserved
UUID=b2c3d4e5-f6a7-8b9c-0d1e-2f3a4b5c6d7e  /var/lib/postgresql/data  ext4  rw,nosuid,nodev,relatime,nofail  0  2
```

```bash
# Test fstab entry without rebooting
sudo mount -a

# Verify the mount is active
findmnt /var/lib/postgresql/data
```

```text
TARGET                       SOURCE     FSTYPE  OPTIONS
/var/lib/postgresql/data     /dev/sdb1  ext4    rw,nosuid,nodev,relatime
```

The walkthrough: `lazy_itable_init=0` and `lazy_journal_init=0` tell `mkfs.ext4` to fully initialize the inode tables and journal before returning. By default, ext4 defers this work to a kernel background thread that runs after first mount. For cloud instances and CI environments where you immediately start writing data, the background thread competes with your application I/O. Paying the cost upfront at format time eliminates that competition. `-m 1` saves 25GB of space over the default 5% reserved-for-root on a 500GB disk — on large data disks, the default reservation is a silent waste. `nosuid` and `nodev` are defensive: a database data directory has no business hosting setuid binaries or device files.

## ⚠️ Gotchas

### mkfs on the Wrong Device Destroys Data Instantly and Silently

This is the mistake that has ended more than one sysadmin's evening. `mkfs` doesn't ask for confirmation. It doesn't check whether the device has a filesystem already. It overwrites the partition's metadata structures immediately and with no undo path.

> ⚠️ **WARNING:** Running `mkfs` on the wrong device — even for one second — will destroy all data on that device. The kernel may still have old data in its page cache, but the on-disk structures are gone. There is no recovery without a backup.

```bash
# Dangerous: this formats /dev/sdb — if /dev/sdb is not what you think it is,
# your data is gone before you finish reading this line
sudo mkfs.ext4 /dev/sdb
```

The correct procedure, every time, without exception:

```bash
# Step 1: list devices and confirm the target
lsblk -f

# Step 2: confirm the specific device is not mounted
findmnt /dev/sdb1

# Step 3: confirm the device is what you think it is (size, label, current FS)
sudo blkid /dev/sdb1

# Step 4: only then run mkfs
sudo mkfs.ext4 -L mydata /dev/sdb1
```

The most common mistake is transposing `sdb` and `sdc` on a server with multiple disks. Device naming (`sda`, `sdb`, `sdc`) is assigned by the kernel in enumeration order and can shift between reboots or when hardware is added. Always verify with `lsblk` first. Always.

### fsck on a Mounted Filesystem Can Corrupt It

`fsck` and `e2fsck` assume exclusive access to the filesystem. If you run them while the filesystem is mounted and being written to, the checker's view of the on-disk structures will be inconsistent with what the kernel is writing. The result is not just a false-clean result — `fsck` can act on stale metadata and write changes that corrupt data that was actually fine.

```bash
# This looks like it might be safe — it's not
sudo fsck /dev/sda1    # sda1 is currently mounted as /
```

```text
fsck from util-linux 2.38.1
e2fsck 1.47.0 (5-Feb-2023)
/dev/sda1 is mounted.
e2fsck: Cannot continue, aborting.
```

Modern `e2fsck` will refuse to run on a mounted filesystem if it detects the mount. But it can't always detect it, and not every checker has this protection.

The correct approach for emergency filesystem repair:

```bash
# For a non-root filesystem: unmount first
sudo umount /dev/sdb1
sudo fsck /dev/sdb1

# For the root filesystem: boot from a live USB or recovery target,
# or single-user mode with root remounted read-only
sudo mount -o remount,ro /
sudo fsck /dev/sda1
```

Systemd handles this correctly automatically: at boot, filesystems with `pass=1` or `pass=2` in `/etc/fstab` are checked by the `systemd-fsck@.service` before the corresponding mount unit activates. You do not need to manually run `fsck` at boot on a healthy system — the journal handles recovery. `fsck` is for when things have gone wrong.

### XFS Cannot Be Shrunk — Ever

This is not a temporary limitation. It is a design constraint of XFS's allocation architecture, and it will not be fixed. XFS uses a highly parallel block allocation system (allocation groups) that assumes the filesystem boundaries are fixed. Shrinking would require rewriting the allocation group metadata in ways the XFS developers have explicitly declined to support.

```bash
# This will fail with an error
sudo xfs_growfs -D smaller_size /mnt/xfs_volume
```

```text
xfs_growfs: cannot shrink filesystem
```

If you provision an XFS filesystem on a raw partition and later need to reclaim space, your only options are: back up the data, delete the partition, re-create it larger, format it, restore the data. Or use LVM — with LVM you can shrink the logical volume, but you must first back up the data, wipe the XFS filesystem, shrink the LV, reformat, and restore. XFS itself cannot be shrunk in place.

The rule: provision XFS partitions with room to grow, or use LVM from the start if you anticipate storage changes.

### resize2fs Must Run AFTER lvextend — Reverse Order Corrupts the Filesystem

When you extend an ext4 filesystem on an LVM logical volume, the sequence matters absolutely. You extend the underlying block device first, then tell the filesystem about the extra space.

> ⚠️ **WARNING:** Running `resize2fs` before extending the underlying logical volume will cause `resize2fs` to write filesystem structures beyond the current end of the block device. This corrupts the filesystem.

```bash
# WRONG ORDER — do not do this
sudo resize2fs /dev/vg0/data     # ← runs first, but LV hasn't grown yet
sudo lvextend -L +50G /dev/vg0/data   # ← too late; filesystem already corrupted
```

```text
# What resize2fs does when the LV is too small:
resize2fs: New size too large to be expressed in 32 bits
# or worse: silent corruption at the edge of the old filesystem boundary
```

The correct order, always:

```bash
# Step 1: extend the logical volume first
sudo lvextend -L +50G /dev/vg0/data

# Step 2: now extend the filesystem to fill the new space
# -f forces if resize2fs is confused about whether the FS is consistent
sudo resize2fs /dev/vg0/data
```

```text
resize2fs 1.47.0 (5-Feb-2023)
Filesystem at /dev/vg0/data is mounted on /data; on-line resizing required
old_desc_blocks = 12, new_desc_blocks = 19
The filesystem on /dev/vg0/data is now 39321600 (4k) blocks long.
```

ext4 resize is online — the filesystem stays mounted throughout. This is intentional: on a production system, taking the filesystem offline to resize it is often unacceptable. The `resize2fs` tool uses ext4's online resize feature, which the kernel supports natively.

For XFS, the equivalent is `xfs_growfs` (not `resize2fs` — that only works on ext filesystems):

```bash
# Extend LV first
sudo lvextend -L +50G /dev/vg0/xfsdata

# Then grow the XFS filesystem (always online — xfs_growfs requires a mount point, not a device)
sudo xfs_growfs /mnt/xfsdata
```

### tune2fs Tuning Is Permanent Until You Change It Back

`tune2fs` modifies the ext4 superblock. Changes take effect immediately and persist across reboots. There's no "preview" mode.

```bash
# This disables reserved blocks entirely — on a busy root filesystem,
# this means root itself can't write when the FS is 100% full
sudo tune2fs -m 0 /dev/sda1
```

```text
tune2fs 1.47.0 (5-Feb-2023)
Setting reserved blocks percentage to 0% (0 blocks)
```

The reserved block percentage exists specifically to give root a buffer when the filesystem fills up — so `rsyslog` can still write logs, and you can delete files to recover. Setting it to 0 on the root filesystem is asking for an unbootable system the moment `/` fills up. Keep it at 1–5% on root, 1% on large data disks.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | Default filesystem is ext4. `mkfs.ext4`, `resize2fs`, `tune2fs`, `e2fsck` are in `e2fsprogs`. XFS tools: `apt install xfsprogs`. Btrfs tools: `apt install btrfs-progs`. |
| RHEL / Rocky / AlmaLinux | Default filesystem is XFS since RHEL 7. Btrfs was removed in RHEL 8 and is not supported. `xfsprogs` and `e2fsprogs` are installed by default. ZFS requires the `epel-release` repo or manual install of OpenZFS RPMs. |
| Fedora | Default filesystem is Btrfs since Fedora 33. `btrfs-progs` is installed. XFS and ext4 tools available via `dnf`. The Btrfs default includes subvolumes for `/` and `/home` by default. |
| openSUSE / SUSE | Btrfs default with snapper integration for automatic snapshots. The `/` filesystem uses Btrfs with rollback capability via `snapper`; `/home` is typically XFS. |
| Arch | No default; chosen at install. All tools available in the `core` and `extra` repos. `btrfs-progs`, `xfsprogs`, `e2fsprogs`, `zfs-dkms` (AUR). |
| Alpine (containers) | ext4 is standard in Alpine-based container base images. Container overlays use OverlayFS provided by the host kernel — Alpine itself doesn't configure this. |

## Alternatives & Comparisons

| Filesystem | When to Use | Notes |
|---|---|---|
| ext4 | General-purpose default; any workload where you're unsure; systems needing shrink capability | The safe choice. Widest tool support, most documentation, online resize in both directions (with LVM for shrink). Default on Debian/Ubuntu. |
| XFS | Large files, high sequential throughput, large filesystems (>16TB), RHEL-based systems | Default on RHEL/Rocky/Alma. Cannot shrink. `xfs_growfs` for online growth. Use `xfs_repair` not `fsck`. |
| Btrfs | Desktops, laptops, systems needing snapshots or compression; Fedora/openSUSE | Stable for most use cases as of kernel 6.12. Avoid RAID 5/6. Avoid raw CoW on database files — disable with `chattr +C` on DB directories. |
| ZFS (OpenZFS) | NAS/storage servers, backup systems, environments needing checksums, pools of multiple disks | Out-of-tree module; not in kernel. Superior data integrity. `zpool` for volume management, `zfs` for filesystem. |
| tmpfs | `/tmp`, `/run`, session caches, any ephemeral data | RAM-backed; no persistence. Competes with application memory. Size it consciously. |
| OverlayFS | Container image layers, read-only base + writable layer | Used by Docker/Podman. Not for general use as a standalone filesystem. |

The honest recommendation: start with ext4 unless you have a specific reason not to. It is not glamorous, but it is battle-tested, universally supported, and has fewer sharp edges than its alternatives. Move to XFS when you're on RHEL or when you have large sequential I/O workloads and benchmarks justify it. Reach for Btrfs when you need its specific features — snapshots, compression, subvolumes — and you're on a distro that ships it by default. Reserve ZFS for dedicated storage systems where its pooled storage and checksum features are the main reason the system exists.

## Trade-offs

**Where ext4 shines:**
- Stable, proven codebase with over 15 years of production deployment at scale
- Online resize in both directions when combined with LVM (shrink: backup, reformat, restore; grow: `lvextend` then `resize2fs`)
- Compatible with every Linux recovery tool and most forensics tools
- Predictable performance across diverse workloads — no CoW overhead on writes

**Where ext4 costs you:**
- No native snapshots; you need LVM snapshots or external tools like `rsync`
- No checksums on data blocks — silent data corruption is possible and undetectable
- Journal overhead on metadata-heavy workloads (millions of small files)

**Where XFS shines:**
- Sequential throughput on large files and large storage — consistently faster than ext4 in benchmarks on multi-terabyte workloads
- Parallel I/O architecture (allocation groups) scales with core count and drive bandwidth
- `xfs_growfs` grows online with no downtime

**Where XFS costs you:**
- Cannot shrink under any circumstances — this is permanent and by design
- `fsck.xfs` is a no-op; you must use `xfs_repair` for manual repair
- Worse than ext4 for small random I/O on HDDs (allocation group locking)

**Where Btrfs shines:**
- Snapshots are free (CoW means unchanged blocks are shared, not duplicated)
- Per-block checksums detect bit rot before it spreads — critical for long-term storage
- Subvolumes give you filesystem namespaces inside a single filesystem, useful for backup and rollback workflows
- Transparent compression (`zstd`) can reduce I/O and extend SSD life on compressible data

**Where Btrfs costs you:**
- Write amplification from CoW on overwrite-heavy workloads (databases, VM images) — raw writes create and discard blocks rather than updating in place
- RAID 5/6 support is officially unsafe — do not use for redundant production data
- Fragmentation accumulates over time on long-running systems; `btrfs fi defrag` mitigates it
- More complex operational model — `btrfs check` is slower than `e2fsck`; snapper or similar tools needed to manage snapshot proliferation

**Performance note:** On NVMe SSDs with large files, XFS consistently outperforms ext4 by 10–30% in sequential throughput benchmarks. For random 4KB I/O (database patterns), the difference between ext4 and XFS is typically under 5% — not worth changing filesystem type. Btrfs with `zstd` compression can achieve higher effective throughput than either on compressible data because the I/O volume is reduced; on incompressible data (already-compressed media, encrypted data), Btrfs adds a small overhead.

## When to Use / When to Avoid

✅ **Use ext4 when:**
- You're setting up a general-purpose server, development machine, or any system where you don't have a specific performance requirement driving a different choice
- You need the ability to shrink the filesystem in the future (with LVM: backup, reformat, restore)
- You're on a Debian/Ubuntu system and want to match what the installer chose
- You need maximum recovery tool compatibility after a disaster

✅ **Use XFS when:**
- You're on a RHEL, Rocky Linux, or AlmaLinux system — match the distro default
- You have large files (video, database WAL archives, VM images) and sequential throughput is the priority
- The filesystem is large (multi-terabyte) and you want allocation group parallelism
- You've used LVM to manage the volume and you never plan to shrink below current size

✅ **Use Btrfs when:**
- You're on Fedora or openSUSE where it's the default and the tooling is integrated
- You want cheap, frequent snapshots — for rollbacks, backups, or system state preservation
- You're storing data where silent bit rot is a concern (photo archives, backups, long-term cold storage)
- Transparent `zstd` compression would meaningfully reduce your I/O (log files, code repositories, text-heavy workloads)

✅ **Use ZFS when:**
- You're building a NAS, backup server, or any system where the primary job is reliable storage at scale
- You need pooled storage across multiple disks with built-in redundancy
- Silent data corruption is unacceptable and you want checksums on every block

✅ **Use tmpfs when:**
- Storing build artifacts, compilation temporaries, or process sockets that don't need to survive a reboot
- `/tmp` is a write bottleneck and you have spare RAM

❌ **Avoid Btrfs when:**
- You're running a database (PostgreSQL, MySQL, MongoDB) directly on Btrfs without disabling CoW with `chattr +C` on the data directory — the overhead is real and measurable
- You need RAID 5/6 and data integrity — use ZFS, or mdraid with ext4/XFS
- You're on RHEL/Rocky/Alma — Btrfs is not supported in that ecosystem

❌ **Avoid XFS when:**
- You think you might need to shrink the filesystem later and don't have LVM underneath it
- Your workload is millions of tiny files with random access patterns on spinning disk

❌ **Avoid ext4 when:**
- You're on RHEL-family systems managing terabytes — XFS is better supported and tested in that ecosystem for large-scale storage

⚠️ **Common mistakes:**
- Choosing a filesystem based on features you'll never use — if you don't need snapshots, Btrfs complexity costs you something for nothing
- Formatting with default `-m 5` on large disks — 5% of 4TB is 200GB reserved for root; set `-m 1` on data disks
- Running `tune2fs -l` expecting to see the label — the label is shown as "Volume name"; UUID is shown as "Filesystem UUID"

## Veteran Perspective

The filesystem choice conversation in interviews and team reviews tends to devolve into religious debate. Here's the practical reality: for 90% of workloads, the difference between ext4 and XFS is benchmarks, not incidents. The thing that actually causes incidents is running `mkfs` on the wrong device. Or running `fsck` on a mounted filesystem. Or not having a backup when either of those happens.

I've seen experienced engineers spend 45 minutes debating ext4 vs XFS for a 200GB application server that does 500 IOPS at peak. Pick ext4, move on. Spend those 45 minutes making sure the fstab entry uses the UUID and has `nofail` set correctly.

The XFS-cannot-shrink constraint bites people in a specific way: they provision a 4TB XFS filesystem in a hurry, fill half of it, and then realize they need to shrink it to 2TB to reprovision the hardware. There is no path. You need a full backup, a new filesystem, and a restore. If you're using raw partitions and you think there's any chance you'll want to resize down later, use ext4 or put LVM underneath everything from day one. XFS on LVM solves the storage management problem — you shrink the LV by creating a new XFS FS on a smaller LV and migrating data.

The Btrfs situation is more nuanced than the ext4-vs-XFS debate. Btrfs on a desktop or laptop running Fedora with snapper is genuinely excellent — free system snapshots before every package update, rollback if something breaks, and you barely notice it's there. Btrfs running a database without `chattr +C` on the data directory is a performance problem waiting to turn into a 3am call. The PostgreSQL documentation specifically calls this out: CoW write patterns fragment the on-disk layout of database pages over time, causing I/O patterns to degrade. This is not a hypothesis — it's been measured. Disable CoW on database directories. Or use XFS. Or use ext4. All of them are the right answer for databases over Btrfs-without-CoW-disabled.

The `lazy_itable_init` issue is subtle and almost never discussed outside of performance-focused documentation. When you format a large ext4 partition with default settings, `mkfs` completes quickly and hands you a filesystem that looks ready. Behind the scenes, a kernel thread is still zeroing the inode tables in the background. This background initialization competes with your application's first writes. On a 4TB disk, initialization can take 30–45 minutes. In a cloud environment where you're formatting a new volume and immediately starting a service, that background I/O can degrade your application's early performance in a way that's invisible to monitoring — it looks like the application is just slow at startup. Use `-E lazy_itable_init=0,lazy_journal_init=0` on production builds and pay the cost upfront at format time.

The `resize2fs` / `lvextend` order issue is a classic example of how doing the right things in the wrong order is as bad as doing the wrong things. The command sequence is correct. The order is the failure mode. I've watched this exact sequence run backwards when someone copy-pasted steps from a tutorial that had them in the wrong order. The filesystem appeared to extend successfully. The corruption showed up three days later when PostgreSQL tried to write to a page beyond what the block device actually had. Backups saved the data. The rule: LV first, filesystem second. Write it on a sticky note if you have to.

## Try It Yourself

1. Run `lsblk -f` on your system. Find the root partition — note its filesystem type, UUID, and label (if any). Run `sudo blkid /dev/sda1` (or whatever your root partition is) and compare the output to what `lsblk -f` shows.

2. If you have a spare partition or a VM with an unformatted disk: create an ext4 filesystem with `sudo mkfs.ext4 -L test-fs -m 1 /dev/sdb1`, mount it at `/mnt/test`, and run `sudo tune2fs -l /dev/sdb1`. Read through the output — find the UUID, the volume name, the inode count, the journal size, and the reserved block count. Unmount and clean up.

3. Run `findmnt -t tmpfs` to see all current tmpfs mounts on your system. Note which of `/tmp`, `/run`, `/dev/shm` are tmpfs-backed. Run `df -h` and cross-reference — note how the tmpfs entries appear alongside real filesystems.

4. On a system with Btrfs (Fedora, openSUSE, or a VM formatted with Btrfs): run `sudo btrfs subvolume list /` to see the subvolume tree. Run `sudo btrfs filesystem show /` to see the device and space usage. Run `sudo btrfs filesystem df /` — notice how Btrfs reports data vs. metadata separately. *(Read-only commands, safe to run on any Btrfs system.)*

5. **Challenge:** In a VM with LVM already configured, extend a logical volume and the ext4 filesystem on it online without unmounting. Use `sudo lvextend -L +2G /dev/vg0/lvdata` to grow the LV, then `sudo resize2fs /dev/vg0/lvdata` to grow the filesystem. Verify with `df -h` before and after that the mounted filesystem grew while staying mounted. Then attempt the same with an XFS filesystem using `xfs_growfs` instead. *(Requires a VM with LVM. Start with creating a VG and LV from an unformatted disk.)*

## Summary

- The filesystem choice is made at format time with `mkfs` and cannot be changed without destroying data and reformatting — get it right before you write anything
- ext4 is the safe default: universally supported, online resize (grow with `resize2fs` after `lvextend`; shrink requires backup and reformat), 15+ years of production deployment
- XFS is the RHEL default, optimized for large files and sequential throughput, but **cannot be shrunk under any circumstances** — plan partition size in advance or use LVM
- Btrfs is stable for general use (default on Fedora since 2020), offers free snapshots and checksums, but RAID 5/6 is unsafe and CoW should be disabled on database directories with `chattr +C`
- Always run `lsblk` to verify the target before `mkfs` — there is no undo; always run `fsck` and `xfs_repair` only on unmounted filesystems; always extend the LVM logical volume before running `resize2fs`
