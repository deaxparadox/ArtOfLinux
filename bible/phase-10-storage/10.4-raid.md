## The Problem This Solves

A single disk is a single point of failure. It doesn't matter how redundant your power supplies are, how many network cards you have, or how carefully you've configured your services — if the one disk running your database fails at 2am on a Saturday, you're down until a replacement arrives and you restore from backup. And the restore is going to take hours.

Before RAID, the options were: accept that risk, or pay for expensive hardware solutions. The expensive hardware solutions worked, but they locked you into specific vendors, specific controllers, and specific failure modes you couldn't debug yourself.

Linux software RAID, managed by `mdadm`, solves the single-disk failure problem in software — entirely at the OS level, using commodity disks. The kernel presents what looks like a single block device (`/dev/md0`) that actually spans multiple physical disks, with data written in a way that lets the array survive one or more disk failures and continue running without interruption.

This is the difference between a degraded array that keeps serving requests at 3am while you order a replacement, and a full outage that wakes up your whole team.

But RAID does exactly one thing: it protects against hardware disk failure. It does not protect you from deleting a file. It does not protect you from a bug that corrupts data. It does not protect you from ransomware that encrypts everything in place. If any of those things happen, every redundant copy of your data is equally affected. **RAID is not a backup.** This deserves saying twice before we go any further.

## Mental Model

Think of RAID like a construction crew that works in parallel on multiple copies of a blueprint.

RAID 0 splits the work across the crew — everyone works simultaneously, so the job gets done faster, but there's only one copy of each piece. Lose one worker and you've lost that piece of the blueprint forever.

RAID 1 has two workers copy the exact same section. Lose one — the other still has it.

RAID 5 is smarter: three workers, but they don't just copy. Two workers take real data sections, and the third calculates a "checksum" piece (parity) that can reconstruct either of the others if one goes missing. Efficient — you only lose one worker's worth of capacity, not half. But if a second worker drops their material while you're still recovering from losing the first, the reconstruction is impossible.

RAID 10 is a hybrid: pair workers into copy-pairs first (like RAID 1), then split work across the pairs (like RAID 0). You get speed and redundancy, at the cost of needing at least four workers.

The reconstruction process when you replace a failed disk is called a **rebuild**. During a rebuild, the surviving disks have to do extra work, which puts them under more stress at exactly the moment they are most exposed.

## Theory

### What RAID Actually Does

RAID — Redundant Array of Independent Disks — is a technique for combining multiple physical disks into a single logical unit, with behavior that varies by **RAID level**. The OS sees one block device. The kernel's `md` (multiple devices) driver handles the translation to actual physical disk I/O behind the scenes.

There are two implementation strategies: **hardware RAID** and **software RAID**. Hardware RAID uses a dedicated controller card with its own processor and memory that handles everything before the OS even sees it. Software RAID is done entirely by the kernel. Linux `mdadm` is software RAID.

### RAID 0 — Striping

Data is split into chunks (stripes) and written across all disks alternately. If you have two 1TB disks in RAID 0, you get 2TB of usable space. Both disks write simultaneously, which is fast — throughput scales roughly linearly with the number of disks.

The cost: zero redundancy. If either disk fails, the array is destroyed. You lose everything. RAID 0 is not actually "redundant" at all — the name is a historical artifact. Use it only when speed matters and data loss is acceptable (scratch space, caches, rendering scratch disks).

### RAID 1 — Mirroring

Every write goes to every disk simultaneously. Two 1TB disks in RAID 1 give you 1TB of usable space — exactly one disk's worth, because everything is duplicated. Any single disk can fail and the array keeps running perfectly.

Read performance can actually improve — the controller can service reads from whichever disk has the head in the right position. Write performance is bounded by the slowest disk.

### RAID 5 — Distributed Parity

The minimum configuration is three disks. Data is striped across all disks, but for every set of stripes, one disk holds **parity** — a calculated value (XOR of the data blocks) that can reconstruct any single missing block. Critically, parity is distributed across all disks rather than dedicated to one, so there's no single "parity disk" bottleneck.

With three 1TB disks, you get 2TB usable (one disk's worth of capacity is consumed by parity). With four, you get 3TB.

RAID 5 can survive one disk failure. The array operates in a **degraded** state — it's still running, but slowly, because every read now requires calculating missing data from parity. Performance tanks during degraded operation. Crucially, the array is now vulnerable: a second failure before the rebuild completes will destroy the array entirely.

The serious risk with RAID 5 and modern large disks is **URE (Unrecoverable Read Error)**. Consumer SATA drives have a URE rate of approximately 1 in 10^14 bits, which corresponds to about 12.5TB. A rebuild of a degraded four-disk RAID 5 array using 4TB drives requires reading roughly 12TB of data from the surviving disks. At those numbers, hitting a URE during the rebuild is not unlikely — it's statistically probable. When a URE occurs during rebuild, the entire array is lost, not just the failing disk. On drives of 4TB or larger, RAID 5 is genuinely dangerous for anything you care about keeping.

### RAID 6 — Dual Parity

RAID 6 extends RAID 5 with two independent parity blocks per stripe. Minimum four disks. With four 1TB disks, you get 2TB usable. The array can survive **two simultaneous disk failures**.

The dual parity makes RAID 6 the appropriate choice for any array with large-capacity disks where URE risk during rebuild is a real concern. The write penalty is higher than RAID 5 because two parity calculations must happen per write.

### RAID 10 — Mirrored Stripes

RAID 10 (also written RAID 1+0) creates mirrors first, then stripes across them. Minimum four disks. With four 1TB disks, you get 2TB usable.

Performance is excellent — you get the parallel read/write speed of striping with the redundancy of mirroring. Any single disk can fail. In the right configuration, multiple disks can fail as long as they're from different mirror pairs. This is the choice when you need both high throughput and high availability.

The cost is capacity efficiency: exactly half your raw disk space is usable, regardless of how many disks you add.

### Hardware RAID vs. Software RAID

Hardware RAID controllers intercept disk I/O before the OS. The array appears as a single device to the OS — the kernel has no visibility into the individual members. This means:

- If the RAID controller fails, you may need the exact same model to recover the array
- The controller's battery-backed write cache can survive power loss where software RAID cannot (by default)
- The controller handles all parity computation, freeing CPU

Linux software RAID (`mdadm`) runs in the kernel. The md driver presents `/dev/md0` to userspace while managing the member disks itself. Advantages:

- Member disks can be moved to any machine running Linux and assembled
- No vendor lock-in on the controller
- The metadata format is open and documented
- Modern CPUs make parity computation cheap — the CPU overhead is not significant
- Full transparency: you can always inspect exactly what the kernel is doing

For most use cases today, software RAID is preferred over hardware RAID unless you specifically need the write-cache battery protection or are in an environment with existing hardware RAID infrastructure.

### The mdadm Metadata Versions

When `mdadm` creates an array, it writes **superblock** metadata to each member disk describing the array configuration. The current default is version **1.2**, which stores the superblock at the beginning of the device (sector 8). Version 1.0 stores it at the end. Version 0.90 is the legacy format. Version 1.2 is what you want for new arrays — it supports booting from the array and is universally understood by modern Linux systems.

## How It Works (Mechanics)

### The md Driver

When you create a RAID array with `mdadm`, the kernel's `md` (multiple devices) driver is involved. The md driver exports a block device (`/dev/md0`, `/dev/md1`, etc.) that maps logical block addresses to physical block addresses across the member disks according to the RAID level's rules.

For RAID 5, a write to a single logical block triggers:

1. Determine which stripe this block belongs to
2. Read the old data from the data disk (if needed for partial stripe writes)
3. Read the old parity
4. Calculate new parity: `new_parity = new_data XOR old_data XOR old_parity`
5. Write new data to the data disk
6. Write new parity to the parity disk

This read-modify-write cycle for parity is why RAID 5 has a write penalty — it's unavoidable. RAID 6 has a larger write penalty because two parity calculations must happen.

### /proc/mdstat

The kernel exposes real-time RAID state through `/proc/mdstat`. Every running md device has an entry here, updated continuously. During a rebuild, the progress line updates as the rebuild proceeds. Polling `/proc/mdstat` is the fastest way to monitor rebuild progress without tools.

```text
Personalities : [raid1] [raid6] [raid5] [raid4] [raid10]
md0 : active raid5 sdd[3] sdc[2] sdb[1] sda[0]
      3145728 blocks super 1.2 level 5, 512k chunk, algorithm 2 [4/4] [UUUU]

md1 : active raid1 sdf[1] sde[0]
      1048576 blocks super 1.2 [2/2] [UU]

unused devices: <none>
```

The `[UUUU]` notation shows device status: `U` means Up, `_` means failed/missing. `[4/4]` means four out of four devices are active. A degraded array shows something like `[3/4] [UUU_]`.

During a rebuild:

```text
md0 : active raid5 sdd[3] sdc[2] sdb[1] sda[0]
      3145728 blocks super 1.2 level 5, 512k chunk, algorithm 2 [4/3] [UUU_]
      [==>..................] recovery = 12.8% (134217728/1048576000) finish=47.3min speed=321234K/sec
```

The progress bar, percentage, and estimated finish time all live in that one line.

### Array Assembly at Boot

Each member disk carries a superblock with a UUID that identifies which array it belongs to. At boot, the kernel scans block devices for md superblocks and auto-assembles any arrays it recognizes, but only if the configuration is registered in `/etc/mdadm/mdadm.conf`. On systems without that entry, the array may come up with a different device name (`/dev/md127` instead of `/dev/md0`) or not at all.

## Basic Example

The following creates a RAID 1 array from two disks. Run this in a VM — it writes to actual block devices.

> ⚠️ **WARNING:** The `--create` command will destroy any existing data on `/dev/sdb` and `/dev/sdc`. Run `lsblk` first to confirm the correct device names. These examples use `/dev/sdb` and `/dev/sdc` — substitute your actual unused devices.

```bash
# Confirm target devices — look for empty disks with no partitions
lsblk -o NAME,SIZE,TYPE,MOUNTPOINT
```

```text
NAME   SIZE TYPE MOUNTPOINT
sda     20G disk
└─sda1  20G part /
sdb     10G disk
sdc     10G disk
```

```bash
# Create a RAID 1 array across two disks
# /dev/md0 is the name of the new array device
# --level=1 is RAID 1 (mirroring)
# --raid-devices=2 tells mdadm how many member disks to expect
sudo mdadm --create --verbose /dev/md0 \
    --level=1 \
    --raid-devices=2 \
    /dev/sdb /dev/sdc
```

```text
mdadm: Note: this array has metadata at the start and
    may not be suitable as a boot device.  If you plan to
    store '/boot' on this device please ensure that
    your boot-loader understands md/v1.x metadata, or use
    --metadata=0.90
mdadm: size set to 10476544K
Continue creating array? (y/n) y
mdadm: Largest drive (/dev/sdb) is larger than smallest (/dev/sdb).
mdadm: Defaulting to minimum device size - loss of space.
mdadm: array /dev/md0 started.
```

```bash
# Confirm the array is running and check initial sync status
cat /proc/mdstat
```

```text
Personalities : [raid1]
md0 : active raid1 sdc[1] sdb[0]
      10476544 blocks super 1.2 [2/2] [UU]
      [===================>.]  resync = 99.7% (10448640/10476544) finish=0.0min speed=128156K/sec
```

Watch the `[UU]` — both devices are Up. The initial `resync` operation syncs both disks to a consistent state; it completes on its own. If you create the array with `--assume-clean`, the initial resync is skipped — only do this if you know both disks are identical.

Now try this yourself. The `[UU]` status and the resync progress bar are what you're looking for. On a small VM disk, the initial resync typically finishes within a minute.

## Real-World Example

This is the full sequence for deploying a production RAID 5 array across three disks, making it persistent across reboots, and verifying the configuration. This is closer to what you'd run on a dedicated storage server.

> ⚠️ **WARNING:** These commands write to real block devices and are irreversible. Every command below that targets `/dev/sdX` devices will destroy existing data on those devices. Run `lsblk` and `fdisk -l` before proceeding to confirm you have the right devices. Test in a VM first.

```bash
# Confirm three spare disks are available and unpartitioned
lsblk -o NAME,SIZE,TYPE,FSTYPE,MOUNTPOINT
```

```text
NAME   SIZE TYPE FSTYPE MOUNTPOINT
sda     20G disk
└─sda1  20G part ext4   /
sdb      4T disk
sdc      4T disk
sdd      4T disk
```

```bash
# Create a RAID 5 array across three 4TB disks
# chunk size of 512k is a reasonable default for most workloads
sudo mdadm --create --verbose /dev/md0 \
    --level=5 \
    --raid-devices=3 \
    --chunk=512 \
    /dev/sdb /dev/sdc /dev/sdd
```

```bash
# Watch the initial parity calculation — this takes a while on large disks
watch -n 5 cat /proc/mdstat
```

```text
Personalities : [raid5]
md0 : active raid5 sdd[2] sdc[1] sdb[0]
      8378269696 blocks super 1.2 level 5, 512k chunk, algorithm 2 [3/3] [UUU]

```

```bash
# Register the array in mdadm.conf so it persists across reboots
# Without this, the array may not assemble automatically, or may come up
# as /dev/md127 instead of /dev/md0
sudo mdadm --detail --scan | sudo tee -a /etc/mdadm/mdadm.conf

sudo update-initramfs -u

# On RHEL/Fedora/Rocky: regenerate the initrd
sudo dracut --force
```

```bash
# Create a filesystem and mount the array
sudo mkfs.ext4 -F /dev/md0
sudo mkdir -p /data
sudo mount /dev/md0 /data
```

```bash
# Add to fstab for automatic mounting — use UUID to avoid device name shifts
sudo blkid /dev/md0
```

```text

```

```bash
# Add the fstab entry (substitute your actual UUID)
echo 'UUID=f1e2d3c4-b5a6-7890-1234-567890abcdef /data ext4 defaults,nofail 0 0' | sudo tee -a /etc/fstab
```

```bash

```

## ⚠️ Gotchas

### RAID Is Not a Backup — Say It Again

The biggest misunderstanding in storage: RAID and backups are not interchangeable.

RAID protects against hardware disk failure. The entire array is online and serving requests; one disk dies; the array continues running without data loss. That's it. That's the entire protection model.

RAID does not protect against:
- Accidental deletion (`rm -rf /data/` destroys every mirror simultaneously)
- Silent data corruption (bit rot on one disk propagates to parity and gets written everywhere)
- Ransomware (encrypts every file across all disks before you notice)
- Catastrophic events (fire, flood, theft take the whole machine)
- Software bugs (a bug that corrupts a database writes corrupt data to all mirrors)

If you have a RAID 1 array and you delete a critical file, both mirrors have the deletion. Your RAID saved you nothing.

Backups are separate copies of your data, stored separately from the primary. RAID is for uptime. Backups are for recovery. You need both.

### RAID 5 + Large Disks = URE Roulette During Rebuild

This is the one that has cost real people real data in production.

Consumer hard drives have a rated Unrecoverable Read Error (URE) rate of 1 in 10^14 bits — roughly one unreadable sector per 12.5TB of data read. Enterprise drives are 10^15 (one per 125TB).

A rebuild of a three-disk RAID 5 array using 4TB drives reads approximately 8TB of data from the surviving two disks. At consumer URE rates, the probability of hitting at least one bad sector during that read is roughly 50%. Hit a URE during a RAID 5 rebuild on a degraded array, and the rebuild fails. The array does not degrade gracefully — it's gone. Both the failed disk and the URE on the surviving disk mean the parity calculation cannot complete. Total data loss.

On 8TB drives, the math is worse. On 12TB drives, it's nearly certain you'll hit a URE on a consumer-grade array.

The rule: for any RAID 5 array using drives of 4TB or larger, use RAID 6 instead. RAID 6 can survive the URE on the surviving disk during rebuild because the second parity allows reconstruction even with one read error. It's not optional caution — it's basic arithmetic.

### Forgetting mdadm.conf — Array Doesn't Come Back After Reboot

You create your RAID array, put a filesystem on it, mount it, everything works. You reboot the server. The array comes up as `/dev/md127` instead of `/dev/md0`, or it doesn't come up at all, and now your fstab entry for `/dev/md0` fails to mount and the system drops into emergency mode.

```bash
# What you thought was enough — it isn't
sudo mdadm --create /dev/md0 --level=5 --raid-devices=3 /dev/sdb /dev/sdc /dev/sdd
# (no mdadm.conf entry written)
```

The array works fine until the next reboot, at which point the kernel may auto-assemble it under any available name.

```bash
# The correct way — always run this immediately after creating an array
sudo mdadm --detail --scan | sudo tee -a /etc/mdadm/mdadm.conf

# Then update initramfs (Debian/Ubuntu) or initrd (RHEL/Fedora) so early boot knows about it
sudo update-initramfs -u        # Debian/Ubuntu
sudo dracut --force              # RHEL/Fedora/Rocky
```

The `--detail --scan` output contains the UUID of the array. The kernel uses this UUID to identify and correctly name the array on reassembly. Without this entry, you're relying on the kernel's auto-detection logic, which doesn't guarantee consistent naming.

### mdadm --stop Without --zero-superblock — Ghost Arrays

You stop an array because you're reconfiguring or decommissioning it.

```bash
sudo mdadm --stop /dev/md0
```

This stops the array in the running kernel, but the superblock on each member disk is still there. On the next boot, the kernel will scan block devices, find those superblocks, and reassemble the array automatically — even if you didn't want it to.

If you're done with the array and want to repurpose the disks:

> ⚠️ **WARNING:** The following command destroys the RAID superblock on the disk, making the disk unrecognizable as an array member. This is irreversible. Only run this if you are certain you want to permanently disassemble the array and do not need the data on it.

```bash
# Stop the array first
sudo mdadm --stop /dev/md0

# Then wipe the superblock from every member disk
sudo mdadm --zero-superblock /dev/sdb
sudo mdadm --zero-superblock /dev/sdc
sudo mdadm --zero-superblock /dev/sdd
```

If you skip `--zero-superblock`, those disks carry ghost array membership forever and will cause confusing auto-assembly behavior on any Linux system you plug them into.

### Slow Rebuild Throttling Your I/O

By default, Linux limits RAID rebuild speed to protect foreground I/O. The default minimum rebuild speed is around 1000 KB/s — so slow that a 4TB rebuild could theoretically take days. Meanwhile your array is running degraded, exposed to a second failure, the entire time.

```bash
# Check current rebuild speed limits
cat /proc/sys/dev/raid/speed_limit_min
cat /proc/sys/dev/raid/speed_limit_max
```

```text
1000
200000
```

The real rebuild speed you see in `/proc/mdstat` is somewhere between these bounds, adjusted dynamically based on system load. On an idle system, increase the minimum to reduce rebuild time:

```bash
# Raise the minimum rebuild speed during active maintenance (not permanent)
echo 50000 | sudo tee /proc/sys/dev/raid/speed_limit_min
echo 500000 | sudo tee /proc/sys/dev/raid/speed_limit_max
```

For RAID 5 and 6, `stripe_cache_size` has a dramatic effect on rebuild speed. The default is 256 pages (~1MB of cache). Raising it to 16384 (64MB) can increase rebuild speed by 5x on typical hardware:

```bash
# Increase stripe cache size for /dev/md0 (RAID 5/6 only — not RAID 1 or 10)
echo 16384 | sudo tee /sys/block/md0/md/stripe_cache_size
```

```text
16384
```

This consumes roughly 200MB of RAM per array. On modern systems that's negligible. On a memory-constrained machine, don't push it past available RAM or you'll cause more problems than you solve. These settings are not persistent — add them to a startup script or `/etc/rc.local` if you want them permanent.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install mdadm`; conf at `/etc/mdadm/mdadm.conf`; update initramfs with `update-initramfs -u` after changes |
| RHEL / Fedora / CentOS / Rocky | `dnf install mdadm`; conf at `/etc/mdadm.conf` (no subdirectory); update initrd with `dracut --force` |
| Arch | `pacman -S mdadm`; conf at `/etc/mdadm.conf`; update initramfs with `mkinitcpio -P` |
| Alpine (containers) | `apk add mdadm`; primarily useful for persistent storage on bare metal Alpine installs; containers rarely use RAID directly |

On Debian/Ubuntu, the `mdadm` package includes a monitoring daemon that sends email on array events — configure it in `/etc/mdadm/mdadm.conf` with `MAILADDR root`. On RHEL-family systems you typically set this up with a separate monitoring script or systemd timer.

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| `mdadm` software RAID | New deployments on commodity hardware; any situation where portability and transparency matter | Default choice; member disks are portable across any Linux system |
| Hardware RAID controller | Environments where battery-backed write cache is critical; legacy infrastructure already using hardware RAID | Controller-specific metadata; recovery requires same controller model |
| ZFS (raidz1/raidz2/mirror) | When you want RAID + filesystem + checksumming + snapshots as one integrated system | Available via `zfsutils-linux` on Debian/Ubuntu; more memory hungry but detects and repairs silent corruption; preferred for NAS builds |
| Btrfs RAID | Development environments; single-machine personal NAS | RAID 5/6 in Btrfs is still considered experimental and has had data loss bugs; do not use for production |
| LVM striping/mirroring | When you're already using LVM and want basic redundancy without a separate layer | Less flexible than dedicated mdadm; mixed LVM+mdadm setups are common in production (LVM on top of mdadm) |

For new storage builds in 2025 and beyond: if you want the simplest reliable solution, use `mdadm`. If you want integrated data integrity checking (protection against silent bit rot), use ZFS. Don't use Btrfs RAID 5/6 on data you care about — the feature has a long history of instability.

## Trade-offs

**Where this shines:**

- Protects against hardware disk failure transparently — the OS and applications see no interruption when a disk fails
- Software RAID with `mdadm` has no vendor lock-in; member disks are reassembleable on any Linux machine
- RAID 10 delivers both high read/write throughput and redundancy — hard to beat for transactional workloads
- RAID 6 gives you "rebuild insurance" — the second parity allows the array to survive a URE during rebuild

**Where this costs you:**

- RAID of any level adds latency for parity writes (RAID 5/6); the write penalty is real
- During rebuild, the array operates degraded — read performance drops significantly on RAID 5/6 as surviving disks calculate missing data on every read
- A degraded RAID 5 array is more dangerous than no RAID at all — it creates false confidence while the exposure window is open
- RAID requires identical-size drives for full capacity; mixing drive sizes wastes the extra capacity of larger disks (they all behave like the smallest)
- Storage complexity increases: one more layer to diagnose when things go wrong

**Performance note:** RAID 5 write throughput is bounded by parity calculation. On modern CPUs with hardware XOR acceleration, this is rarely the bottleneck. Sequential read throughput on RAID 5 scales nearly linearly with drive count. Random I/O performance on a spinning disk RAID 5 degrades dramatically during rebuild.

## When to Use / When to Avoid

✅ **Use this when:**

- You need continuous availability when a single disk fails — any production server that can't have unplanned downtime
- You're building a NAS or file server and need more capacity than a single disk provides with fault tolerance
- You need to protect against hardware failure as one layer of a defense-in-depth storage strategy (RAID + backups)
- You have four or more disks and want both performance and redundancy: RAID 10

❌ **Avoid this when:**

- You think RAID replaces backups — it doesn't; use proper backup tools (`restic`, `borgbackup`, snapshot-based solutions)
- You're deploying RAID 5 on drives of 4TB or larger without a backup and verified restore path — use RAID 6 instead
- You want data integrity protection against silent corruption — use ZFS instead; RAID alone does not detect bit rot
- You're running stateless containers with ephemeral storage — RAID adds complexity for no benefit here

⚠️ **Common mistakes:**

- Using RAID 0 for "performance" on production data — the risk is not worth any speed gain; use NVMe instead
- Forgetting to add the array to `mdadm.conf` and regenerate initramfs/initrd before the first reboot
- Building a RAID array then never testing that it actually works — simulate a disk failure in a VM before trusting it in production
- Using the same model and batch of disks from the same order — if one fails from a manufacturing defect, the others from the same batch are at elevated risk of failing during the rebuild

## Veteran Perspective

The mistake I see most often from junior sysadmins is treating RAID as a solved problem once the array is created. They create the array, see the `[UUU]` in `/proc/mdstat`, and move on. Six months later a disk fails, the array goes degraded, nobody gets paged because nobody set up monitoring, and the second disk fails before anyone notices. Total loss.

RAID is not a one-time setup. It's an ongoing operational responsibility. You need monitoring. A degraded array must be treated as an emergency. You should know the answers to these questions for every RAID array you manage: What is the current array state? How old are the member disks? What is the rebuild time if a disk fails today? Do I have a replacement disk on the shelf?

The intermediate mistake is mixing up "the array survived" with "the data survived." I've seen people replace a failed RAID 1 disk, watch the rebuild complete, and then discover the data was corrupted six months before the disk failed — and the corruption was faithfully mirrored to both disks the whole time. RAID 1 guarantees that both disks are identical. It does not guarantee the data is correct.

The RAID 5 URE problem is not a theoretical concern. It's been killing real arrays on real servers since drives passed the 1TB mark. The storage industry has known about this since around 2007. The fix — RAID 6 or RAID 10 — has been available for just as long. Every year, someone loses a production array because they didn't make the switch. Don't be that person.

On the hardware vs. software RAID debate: hardware RAID's main real advantage is the battery-backed write cache, which lets the controller acknowledge writes to the OS before they're committed to disk — safe because the cache survives a power failure. Software RAID without a UPS has no equivalent. If you're running software RAID on a machine without a UPS, a power cut during a write can leave the array inconsistent. It usually recovers after a resync, but "usually" is not a good operational story.

In technical interviews, RAID questions are a common filter for storage competence. They'll ask you to explain the write penalty in RAID 5. They'll ask why RAID is not a backup. They'll ask what happens during a degraded rebuild. Know these cold. The URE discussion — being able to articulate why RAID 5 on modern large disks is a meaningful risk — immediately signals operational experience.

The hardest thing to explain to a non-technical stakeholder is why you need RAID *and* backups. "But doesn't RAID protect against data loss?" The answer is: RAID protects against one specific kind of data loss, hardware disk failure. Backups protect against everything else. You cannot choose one. Both are required for any data you actually care about.

## Try It Yourself

1. On a VM with two spare disks, create a RAID 1 array with `mdadm --create`. Watch `/proc/mdstat` until the initial sync completes. Check `mdadm --detail /dev/md0` and find the `State : clean` line.

2. After creating a RAID 1 array, mark one disk as failed and watch what happens: `sudo mdadm /dev/md0 --fail /dev/sdb`. Check `/proc/mdstat` and `mdadm --detail`. Then add it back: `sudo mdadm /dev/md0 --remove /dev/sdb` followed by `sudo mdadm /dev/md0 --add /dev/sdb`. Watch the rebuild.

3. Create a RAID 1 array, put an ext4 filesystem on it, mount it, write a file. Then stop the array (`mdadm --stop /dev/md0`) and reassemble it by hand from the member disks: `sudo mdadm --assemble /dev/md0 /dev/sdb /dev/sdc`. Mount it and verify the file is still there. This proves the data lives on the disks, not in the array abstraction.

4. After creating a RAID 5 array on three disks, check the default stripe cache size: `cat /sys/block/md0/md/stripe_cache_size`. While the initial resync is running, compare the rebuild speed at the default size vs. after doubling it. Watch `/proc/mdstat` for the speed line.

5. **Challenge:** Create a RAID 5 array on three virtual disks in a VM. Register it in `mdadm.conf`, put a filesystem on it, create some files, and then reboot. Confirm the array auto-assembles with the correct name. Then simulate a disk failure by hot-unplugging one virtual disk (or using `--fail`), confirm the array is degraded, add a new disk as a hot spare and watch the rebuild complete. Write up the full sequence of commands and outputs so you could hand it to a colleague as a runbook.

---

## Summary

- RAID protects against hardware disk failure and nothing else — it is not a backup, and deletes, corruption, and ransomware affect every mirror equally
- RAID 5 on drives of 4TB or larger carries a statistically significant probability of losing the entire array to a URE during rebuild; use RAID 6 or RAID 10 for large disks
- After creating any array, immediately run `mdadm --detail --scan >> /etc/mdadm/mdadm.conf` and regenerate initramfs/initrd — without this, the array may not assemble correctly after reboot
- `/proc/mdstat` is the fastest way to check array state and rebuild progress; `mdadm --detail /dev/md0` gives the full picture
- `mdadm --stop` stops the array in the running kernel but leaves superblocks on member disks intact; use `mdadm --zero-superblock` on each member if you want to permanently decommission the array
