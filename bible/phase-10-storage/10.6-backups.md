## The Problem This Solves

You've spent six months building a production server. Databases, configs, TLS certificates, custom scripts, years of accumulated `/etc`. One morning you get a call: a botched `rm -rf`, a failed `apt upgrade`, a ransomware infection, a disk that finally gave out after the SMART warnings you meant to look at last week. Now you need everything back — fast, clean, and to a known-good state.

Without backups, the conversation ends there. With backups, the conversation is: "How long will the restore take?"

But here's the part that kills systems admins who thought they were prepared: they had backups. They never tested the restore. The backup job had been silently failing for three weeks. The tape was good but nobody had tested the restore procedure on a clean machine. The backup was there but it was of the wrong partition. The snapshot existed but the application was mid-transaction when it ran, and the database came back corrupt.

A backup you haven't restored from is a promise you haven't verified. This file is about making backups you can actually trust.

## Mental Model

Think of backups like a photograph album of your server. A full backup is a single photograph of everything at one moment. An incremental backup is a sheet of sticky notes on top of that photo, marking what changed since the last shot. A deduplicated backup (like restic or borg) is a shared negative library — every unique piece of content is stored once, and every backup is just a reference list pointing to the pieces it needs.

The 3-2-1 rule tells you how many albums to keep and where: three copies, on two different types of media, with one offsite. If your house burns down, the photo album on your bookshelf and the one on your coffee table are both gone. The one at your parents' house is what you get to keep.

## Theory

### The 3-2-1 Rule

Every serious backup strategy follows the 3-2-1 rule:

- **3 copies** of the data — the live copy plus two backups
- **2 different media types** — e.g., local disk and cloud, or internal drive and external USB
- **1 offsite copy** — geographically separated from the others

Why three? Because two copies can fail simultaneously. A disk failure during a backup run can corrupt both the source and the destination. A fire, a flood, a power surge — localized disasters don't care that you have two drives in the same rack. The offsite copy is the one that saves you when everything else is gone.

### Full, Incremental, and Differential

A **full backup** copies everything. It is slow and large, but self-contained and simple to restore.

An **incremental backup** copies only what changed since the last backup of any kind. Each backup is small and fast, but a restore chains together: you need the last full backup plus every incremental since. If any link in that chain is missing or corrupt, the restore fails.

A **differential backup** copies everything changed since the last full backup. Restores are simpler — you only need the last full plus the last differential — but each differential grows over time.

Modern deduplicating backup tools like `restic` and `borgbackup` sidestep this entire taxonomy. They split data into content-defined chunks and store each unique chunk exactly once. Every backup appears to be a full snapshot (you can browse and restore it independently), but under the hood only the new or changed chunks take up space. This is the best of all worlds: simple restores, minimal storage.

### RAID Is Not a Backup

RAID keeps you running when a disk fails. It does not protect you from:

- Accidental deletion (the deletion is mirrored to all disks instantly)
- File corruption (the corruption is mirrored to all disks instantly)
- A bad `rm -rf` (mirrored instantly)
- Ransomware (mirrored instantly)
- A misconfigured `rsync --delete` that wipes a directory (mirrored instantly)
- A complete server loss — house fire, theft, catastrophic power event

RAID is a high-availability mechanism. Backups are a data-recovery mechanism. They solve different problems and you need both. This is covered in [10.4 — RAID](10.4-raid.md) as well, because different audiences need to hear it and some only read one file.

### Consistency — the Problem Nobody Thinks About Until It Bites Them

A filesystem backup taken while applications are writing is not guaranteed to be consistent. If PostgreSQL is in the middle of a write when your backup snapshot runs, the database files captured may be internally inconsistent — and will fail to start on restore.

There are three ways to handle this:

1. **Stop the application** before backing up (simple, always works, requires downtime)
2. **Use the application's own dump tools** (`pg_dump`, `mysqldump`) before backing up the dump file
3. **Use filesystem snapshots** (LVM or btrfs) — a snapshot is atomic at the filesystem level; the application's crash-recovery mechanisms handle the rest on restore, just as they would after a power loss

Approach 3 is what production systems use when they can't afford downtime. The snapshot is taken in milliseconds, the backup is run against the snapshot, and the application never pauses.

## How It Works (Mechanics)

### How rsync Copies Files

`rsync` uses a rolling checksum algorithm to compare files on source and destination. For each file, it divides the file into fixed-size blocks and computes checksums for each block. It then compares those checksums to what exists at the destination. Only the blocks that differ are transferred. This is why `rsync` is efficient even over slow networks — after the first full sync, each subsequent run transfers only the changed portions of changed files.

### How `--link-dest` Creates Apparent Full Backups from Incremental Data

`--link-dest=DIR` tells rsync: "When you're deciding whether to copy a file to the destination, first check if an identical file exists at DIR. If it does, create a hard link to that inode instead of copying the data."

Hard links are multiple directory entries pointing to the same inode — the same physical data on disk. From the outside, every backup directory looks like a full copy of the source. From the disk's perspective, unchanged files exist only once; only new and modified files consume additional space.

```text
# Conceptual layout — not real commands

Snapshot 2026-06-01/
  ├── etc/passwd      → inode 12345 (actual data on disk)
  ├── etc/hosts       → inode 12346 (actual data on disk)
  └── var/log/app.log → inode 12347 (actual data on disk)

Snapshot 2026-06-02/   (only app.log changed)
  ├── etc/passwd      → inode 12345 (same inode — no extra disk space)
  ├── etc/hosts       → inode 12346 (same inode — no extra disk space)
  └── var/log/app.log → inode 12399 (new inode — only this was copied)
```

The `--link-dest` directory must be on the same filesystem as the destination, because hard links cannot cross filesystem boundaries.

### How LVM Snapshots Work

When you create an LVM snapshot of a logical volume, LVM sets up a copy-on-write (CoW) mechanism. The original volume continues to be used normally by applications. The snapshot volume starts empty — it contains only a change log. When a write is made to the original volume, LVM copies the original block to the snapshot's change log *before* the write goes through. The snapshot always shows the data as it existed at the moment of creation.

This is how you get a consistent point-in-time view of a live filesystem without stopping the applications writing to it.

The critical constraint: the snapshot's change log has a finite size. Once it fills up — meaning more blocks were written to the original volume than the snapshot can track — the snapshot becomes invalid and is automatically unmounted by the kernel. Any backup in progress against that snapshot immediately fails.

### How restic Deduplicates

`restic` divides file contents into variable-size chunks using content-defined chunking (CDC). The boundaries between chunks are determined by the file's content rather than fixed offsets. This means that inserting a byte at the beginning of a large file doesn't invalidate all subsequent chunks — only the chunks around the insertion point change. Each chunk is hashed (SHA-256), and that hash is the chunk's ID. If an identical chunk was already stored in the repository, restic doesn't store it again. All chunks and their metadata are encrypted with AES-256 before being written to storage.

## Basic Example

```bash
# Create a simple rsync mirror of /home/alice to an external drive
# -a = archive mode (preserves permissions, timestamps, symlinks, ownership)
# -v = verbose — shows each file being transferred
# -h = human-readable sizes in the progress output
rsync -avh /home/alice/ /mnt/external/backup-alice/
```

```text
sending incremental file list
./
.bashrc
.ssh/
.ssh/authorized_keys
Documents/
Documents/notes.txt
Documents/project/
Documents/project/main.py

sent 142.3K bytes  received 1.2K bytes  71.75K bytes/sec
total size is 1.23G  speedup is 8,567.42
```

Notice the trailing slash on `/home/alice/` — it means "copy the contents of this directory." Without the trailing slash, rsync copies the directory itself, creating `/mnt/external/backup-alice/alice/`. This is one of the most common rsync surprises and it's covered in the Gotchas section below.

Try this yourself with any directory you own. If your output shows a longer file list, that's your actual data — that's normal. Run it a second time and watch the output shrink to almost nothing — that's the incremental behavior in action.

## Real-World Example

A production backup strategy using `rsync --link-dest` to create space-efficient daily snapshots, and `restic` for offsite encrypted backups. This is close to what you'd actually deploy on a small production server.

### Part 1: Daily Snapshots with rsync --link-dest

```bash
#!/bin/bash
# /usr/local/bin/daily-snapshot.sh
# Creates a new dated snapshot, hard-linking unchanged files to the previous snapshot.
# Run via cron: 0 2 * * * /usr/local/bin/daily-snapshot.sh

BACKUP_ROOT="/mnt/backup"
SOURCE="/srv/data"
TODAY=$(date +%Y-%m-%d)
YESTERDAY=$(date -d "yesterday" +%Y-%m-%d)

DEST="${BACKUP_ROOT}/${TODAY}"
LINK="${BACKUP_ROOT}/${YESTERDAY}"

# --link-dest creates hard links to unchanged files from yesterday's snapshot
# Only files that changed since yesterday will actually be copied
rsync -aH \
  --delete \
  --link-dest="${LINK}" \
  "${SOURCE}/" \
  "${DEST}/"

echo "Snapshot complete: ${DEST}"

# Remove snapshots older than 30 days
find "${BACKUP_ROOT}" -maxdepth 1 -type d -name "????-??-??" \
  -mtime +30 -exec rm -rf {} \;
```

Each directory under `/mnt/backup/` looks like a complete snapshot of `/srv/data`. Restoring from any day is a single `rsync` or `cp -a` — no chain to reconstruct. But disk usage grows only by the amount that actually changed each day.

The `--delete` flag is what makes this a mirror rather than an accumulation. Files deleted from the source will be removed from the new snapshot. The previous day's snapshot still has them, which is your protection against accidental deletions.

### Part 2: Offsite Encrypted Backup with restic

```bash
# Initialize a restic repository on an S3-compatible backend
# RESTIC_PASSWORD is read from the environment — never pass it as a flag
export RESTIC_PASSWORD="$(cat /etc/backup/restic-password)"
export AWS_ACCESS_KEY_ID="$(cat /etc/backup/s3-key-id)"
export AWS_SECRET_ACCESS_KEY="$(cat /etc/backup/s3-secret)"

# One-time initialization
restic -r s3:s3.amazonaws.com/my-backup-bucket init

# Run a backup — restic handles deduplication and encryption automatically
restic -r s3:s3.amazonaws.com/my-backup-bucket \
  backup /srv/data /etc /home \
  --exclude /home/*/.cache \
  --exclude /srv/data/tmp

# Check what snapshots exist
restic -r s3:s3.amazonaws.com/my-backup-bucket snapshots
```

```text
repository abc12345 opened (version 2, compression level auto)
ID        Time                 Host        Tags        Paths
---------------------------------------------------------------------
d3f1a2b1  2026-06-14 02:00:01  prod-web                /srv/data
                                                        /etc
                                                        /home
e5c9b3a2  2026-06-15 02:00:01  prod-web                /srv/data
                                                        /etc
                                                        /home
---------------------------------------------------------------------
2 snapshots
```

```bash
# Restore a specific file from a snapshot
restic -r s3:s3.amazonaws.com/my-backup-bucket \
  restore e5c9b3a2 \
  --target /tmp/restore-test \
  --include /srv/data/important-file.db

# Verify repository integrity — run this monthly
restic -r s3:s3.amazonaws.com/my-backup-bucket check

# Remove old snapshots — keep last 7 daily, 4 weekly, 12 monthly
restic -r s3:s3.amazonaws.com/my-backup-bucket forget \
  --keep-daily 7 \
  --keep-weekly 4 \
  --keep-monthly 12 \
  --prune
```

The `forget --prune` command is two operations in one: `forget` marks old snapshots for deletion according to your retention policy, and `prune` physically removes the data chunks that are no longer referenced by any snapshot. Run them together.

### Part 3: The Most Important Step — Testing the Restore

Testing a restore is not optional. Schedule it. Do it on a fresh VM or a different machine.

```bash
# Restore the most recent snapshot to a test directory
restic -r s3:s3.amazonaws.com/my-backup-bucket \
  restore latest \
  --target /tmp/restore-test

# Verify specific critical files came back correctly
diff /srv/data/important-file.db /tmp/restore-test/srv/data/important-file.db

# For databases: restore and then verify the application can start
# (run this in a VM, not on the production server)
pg_restore -d test_db /tmp/restore-test/srv/data/db.dump
psql -d test_db -c "SELECT count(*) FROM users;"
```

If the restore fails in a test environment, it will fail in production — except in production you'll be under pressure at 3am and there will be no fallback.

## ⚠️ Gotchas

### dd with Wrong if= and of= — Imaging the Wrong Direction Destroys the Source

This is the most dangerous mistake in this entire file. `dd` is not smart about directions. You tell it where to read (`if=`) and where to write (`of=`). If you swap them, it will silently and immediately overwrite your source with the (empty) destination. There is no confirmation prompt. There is no undo.

> ⚠️ **WARNING:** Before every `dd` command, run `lsblk` to confirm your device names. On different systems, the same physical drive can appear as `/dev/sda`, `/dev/sdb`, `/dev/nvme0n1`, or `/dev/vda`. The device that was `/dev/sdb` last boot may be `/dev/sdc` today. A single transposed argument destroys data instantly and silently. Always test `dd` usage in a VM before running it on real hardware.

```bash
# WRONG — this writes the (blank) destination ONTO the source disk
# Do NOT run this
dd if=/dev/sdb of=/dev/sda bs=4M status=progress
```

```text
# It runs. It shows progress. It silently destroys /dev/sda.
# No error. No warning. No confirmation.
```

The rule behind this: `dd` reads from `if` (input file) and writes to `of` (output file). The convention is "source first, destination second" — identical to how `cp source dest` works. But there is no guard rail. Always confirm with `lsblk` before you run it.

```bash
# CORRECT workflow
lsblk                              # confirm which device is which before touching anything

# Create an image of /dev/sdb to a file on /mnt/external
dd if=/dev/sdb of=/mnt/external/disk-image.img bs=4M status=progress

# Restore an image to /dev/sdb (destructive — this is intentional here)
dd if=/mnt/external/disk-image.img of=/dev/sdb bs=4M status=progress
```

`status=progress` requires GNU coreutils 8.24 or later, which ships with every modern distro. On older systems, use `pv` instead: `dd if=/dev/sdb bs=4M | pv | dd of=/mnt/external/disk-image.img bs=4M`.

### rsync --delete Without --dry-run First — Silent Destruction

`rsync --delete` removes files at the destination that no longer exist at the source. This is what makes rsync a mirror instead of an accumulation. It is also how one mistyped source path wipes a production backup.

The trap looks like this:

```bash
# You meant to sync /srv/data/ but you typed /srv/dat/ (typo)
# --delete will now remove everything in /mnt/backup/ that isn't in the (empty) /srv/dat/
rsync -av --delete /srv/dat/ /mnt/backup/
```

```text
sending incremental file list
deleting app/logs/2026-06-01.log
deleting app/logs/2026-06-02.log
deleting app/
deleting etc/nginx/nginx.conf
...
```

The deletion is quiet, fast, and complete. By the time you notice the typo, the backup is gone.

The fix is to always dry-run first with `-n` (or `--dry-run`). A dry run shows you exactly what would happen without actually doing it:

```bash
# Always dry-run first when using --delete
rsync -avn --delete /srv/data/ /mnt/backup/

# Read the output carefully. When you're satisfied it's correct:
rsync -av --delete /srv/data/ /mnt/backup/
```

Make `-n` a habit before any `rsync --delete` run, even when you're confident. The few seconds it takes to read the dry-run output is nothing compared to the hours it takes to explain a lost backup.

### rsync Trailing Slash Changes What Gets Copied

This is not a safety issue but it causes endless confusion. The trailing slash on the source path means something specific to rsync, and its absence means something different.

```bash
# WITH trailing slash: copies the CONTENTS of /home/alice into /backup/
rsync -av /home/alice/ /backup/
# Result: /backup/.bashrc, /backup/Documents/, etc.

# WITHOUT trailing slash: copies the DIRECTORY ITSELF into /backup/
rsync -av /home/alice /backup/
# Result: /backup/alice/.bashrc, /backup/alice/Documents/, etc.
```

Neither is wrong — they do different things. Confusion arises when you run both forms at different times and wonder why the backup structure changed. Pick one and be consistent.

### LVM Snapshot Fills Up and Becomes Invalid

LVM snapshots have a fixed size for their change log. If the original volume receives more writes than the snapshot can track before the backup finishes, the kernel marks the snapshot invalid and unmounts it. Your backup job fails mid-run, often silently, often leaving a partial backup that looks complete.

```bash
# Creating a snapshot that's too small for a busy volume
lvcreate -L 1G -s -n data-snap /dev/vg0/data
# If /dev/vg0/data gets more than 1G of writes before the backup finishes,
# the snapshot silently becomes invalid.
```

```bash
# Check snapshot health — look for "allocated" percentage
lvs -o lv_name,lv_size,snap_percent /dev/vg0/data-snap
```

```text
  LV        LSize  Snap%  
  data-snap 1.00g  94.23
```

When `Snap%` approaches 100, the snapshot is about to become invalid. If it hits 100, it's already gone.

The correct approach: size snapshots at 10-20% of the original volume for a quiet system. For a busy database server, size them at 30-50% or use **thin LVM provisioning**, where snapshots don't have a fixed change-log size.

```bash
# Better: create a larger snapshot
lvcreate -L 20G -s -n data-snap /dev/vg0/data  # 20% of a 100G volume

# Better still: extend a snapshot if it starts filling up
lvextend -L +10G /dev/vg0/data-snap
```

### tar Without -p Doesn't Preserve Permissions

When you use `tar` to create a backup archive, the default behavior drops SUID/SGID bits and may not preserve numeric user IDs — only symbolic names. If the restored system doesn't have the same user names, files end up owned by UID 0 or the wrong user.

```bash
# Without -p — creates the archive but silently loses special permission bits
tar -czf backup.tar.gz /etc /home
```

For any backup archive that will be restored to a different system (or after a reinstall where UIDs might differ), always use `-p` (preserve permissions) and `--numeric-owner`:

```bash
# Correct for backup archives
tar -czpf backup.tar.gz --numeric-owner /etc /home
```

`--numeric-owner` stores UID and GID numbers rather than names. On restore, the numeric IDs are preserved exactly, even if the user account names don't match. This matters when restoring onto a fresh system.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `rsync --link-dest` | Local/SSH incremental snapshots; no extra software; full control | Hard links only work within a filesystem; no built-in encryption |
| `tar` | Simple archives; moving backups between systems; quick point-in-time captures | No deduplication; no built-in incrementals; good for single-file transport |
| `dd` | Full disk images; cloning drives; bare-metal restores | Block-level copy; captures everything including partition table and boot sector; slow and large |
| `restic` | Cloud and offsite backups; dedup + encryption required; cross-platform (Linux/macOS/Windows) | Best-in-class for cloud backends (S3, B2, SFTP, Azure, GCS); simpler than borg; faster restores; currently at 0.19.0 |
| `borgbackup` | Local/SSH backups where compression matters most; existing borg installations | Excellent compression (especially text-heavy data); FUSE mount for browsing snapshots; Linux/macOS only |
| LVM snapshots | Consistent live backups of databases and active filesystems | Requires LVM; snapshot must be used with another tool to get data offsite |
| `btrfs send`/`receive` | Streaming incremental snapshots between btrfs filesystems | Only works btrfs-to-btrfs; efficient; built into the filesystem driver |

For new setups in 2026: use `restic` for anything going offsite or to cloud storage. Use `rsync --link-dest` for local time-machine-style snapshots when you want transparency and no extra dependencies. Use LVM or btrfs snapshots for consistent database backups. Use `borgbackup` if you're already running it or you need FUSE mounting of backup archives.

## Trade-offs

**Where this shines:**

- `rsync --link-dest` snapshots are transparent — every backup directory is browsable with `ls`; no special tool needed to read them
- `restic` repositories are verifiable with `restic check` — you can confirm data integrity without performing a full restore
- LVM snapshots provide point-in-time consistency without application downtime
- `dd` images are complete — they capture everything including the partition table, boot sector, and filesystem metadata; a bare-metal restore from a `dd` image requires no OS reinstall

**Where this costs you:**

- `dd` images are the full size of the device regardless of how much data is on it — a 500GB drive with 10GB of data produces a 500GB image (unless you pipe through `gzip`)
- `rsync --link-dest` provides no encryption — backups on a stolen drive are fully readable
- LVM snapshots become invalid if the change log fills up during a long backup of a busy volume
- `restic` and `borgbackup` repositories are opaque — you can't browse them with `ls`; you always need the tool installed to access the data
- `tar` archives without compression can be very large; with compression, a single corrupt byte in the stream can make the entire archive unextractable (use `gzip` with `-z`, which at least has better error recovery than `bzip2`)

**Performance note:** `restic` deduplication operates in userspace and is CPU-bound during the chunking phase. On a server doing its first backup of terabytes of data, expect to saturate a CPU core. Subsequent incremental backups are fast — only new chunks are processed.

## When to Use / When to Avoid

✅ **Use this when:**

- You need offsite or cloud backups with encryption: reach for `restic`
- You need local time-machine-style browsable snapshots without extra software: `rsync --link-dest`
- You're backing up a live database and can't take downtime: LVM snapshot first, then backup the snapshot
- You need a complete disk clone for bare-metal restore: `dd`
- You need a quick, portable archive of a directory tree: `tar -czpf --numeric-owner`
- You're on btrfs and want efficient incremental backups between two btrfs hosts: `btrfs send`/`receive`

❌ **Avoid this when:**

- You think RAID replaces backups — RAID keeps you running; backups get you back; you need both (see [10.4 — RAID](10.4-raid.md))
- You're using `dd` on a mounted, live filesystem for "backups" — the image will be inconsistent; unmount the filesystem first or use LVM/btrfs snapshots
- You're using `rsync` without `--link-dest` and expecting space-efficient incremental snapshots — without `--link-dest`, each backup is a full copy
- You're using LVM snapshots without monitoring snapshot fill percentage — a full snapshot silently invalidates mid-backup

⚠️ **Common mistakes:**

- Running `restic forget` without `--prune` — the snapshot metadata is removed but the data chunks remain on disk; storage doesn't shrink until you prune
- Using `borgbackup` from EPEL on RHEL 7 — old borg versions have known bugs; upgrade or use restic
- Scheduling backups but never scheduling restore tests — backups rot; restore tests are the only way to know if they still work
- Backing up `/proc`, `/sys`, `/dev`, `/run` with `tar` or `rsync` — these are virtual filesystems; including them fills the archive with garbage or causes the backup to hang; always exclude them explicitly

## Veteran Perspective

The biggest mistake I see from junior sysadmins isn't a wrong command — it's a false belief. They set up a cron job to run `rsync` nightly, watch it succeed for a week, and decide the backup problem is solved. Then six months later they need the restore and the cron job has been failing silently because the external drive went offline, or the destination ran out of space, or the `rsync` started terminating early due to a permission error. The success from week one was cached in their head. The failure was never noticed.

Monitoring your backup jobs is not optional. At minimum: every backup job should write a timestamped log and that log should be checked. In production: your backup job should send a heartbeat to a monitoring system (something like healthchecks.io or a Prometheus gauge), and you alert when the heartbeat is missing. A backup that runs but doesn't notify you is only slightly better than no backup.

The second thing I see constantly: people backup the data but not the metadata needed to restore it. You backed up `/var/lib/postgresql/` but not the PostgreSQL configuration in `/etc/postgresql/`. You backed up the web root but not `/etc/nginx/`. You backed up the database files but not the `pg_hba.conf` that controls who can connect. When you go to restore, you have the data and no idea how the service was configured. Always backup `/etc` alongside application data.

The third failure mode — and the most expensive one — is testing the restore in the wrong environment. You test the restore on the same server the backup came from. Everything looks fine. But the disaster scenario is that the server is *gone*. The restore test needs to happen on a clean machine with nothing pre-installed. The database has to start from scratch against the restored files. The application has to come up and serve traffic. That is the test. Anything less is rehearsing without the costume.

What does this look like in a technical interview? "Walk me through your backup strategy for a production database server." The answer they're looking for isn't `pg_dump`. It's: application-consistent backup (LVM snapshot or `pg_dump` first), encrypted offsite copy, retention policy, and — critically — a documented and periodically executed restore procedure. If you don't have the last part, you don't have a backup strategy; you have a backup job.

## Try It Yourself

1. Run `rsync -avh /etc/ /tmp/etc-backup/` and look at the output. Notice the number of files transferred. Run it again immediately — see how much faster the second run is and how few files it copies.

2. Modify one file in `/tmp/etc-backup/`, then run rsync again. Watch rsync identify and re-transfer only that one file.

3. Create three dated snapshot directories using `rsync --link-dest`. After the third snapshot, run `du -sh /tmp/snapshots/*/` and compare the reported sizes to the actual inode counts with `ls -li /tmp/snapshots/2026-*/etc/hostname` — confirm that the identical files share the same inode number.

4. Create a `tar` archive of `/etc` with `tar -czpf --numeric-owner /tmp/etc-backup.tar.gz /etc`, then extract it to `/tmp/restore-test/` and compare file permissions: `stat /etc/sudoers` vs `stat /tmp/restore-test/etc/sudoers`. Confirm UID, GID, and mode are identical.

5. **Challenge** (requires a VM with LVM): Take a running service (even a simple `nc -l 8080` writing to a log file), create an LVM snapshot of its volume, and back up the snapshot with `rsync` while the service continues writing. Restore the snapshot contents to a second directory and verify you got a consistent state. Then remove the snapshot and confirm the service never paused.

---

## Summary

- The 3-2-1 rule: three copies, two media types, one offsite — every backup strategy is measured against this
- RAID is not a backup: it mirrors live data, including deletions and corruption; backups are separate
- `rsync --link-dest` creates space-efficient snapshots where every directory looks like a full backup but unchanged files are hard links sharing a single inode
- A backup you haven't restored from is a hypothesis — schedule restore tests on clean machines, not just backup verification checks
- `dd` requires `lsblk` verification before every run — transposing `if=` and `of=` destroys the source instantly and silently; always double-check device names
