# [10.3] — LVM: Logical Volume Manager

> LVM gives you the power to resize, move, and snapshot storage while the system is running — the feature that turns "we're out of disk space at 2am" from a crisis into a five-minute fix.

## The Problem This Solves

Picture a server running out of disk space on `/var/log`. It's 11pm. The disk is a fixed-size partition — `/dev/sda2`, 20GB, and it's 97% full. Logs are failing to write. Applications are crashing. You have one option: attach a new disk, copy everything over, update `/etc/fstab`, reboot, and pray. That's at minimum an hour of downtime, during which the filesystem is offline.

Or maybe the opposite: you provisioned 200GB for a database volume that turned out to only ever use 30GB. That space is locked up forever. You can't hand it back to the OS. The partition boundaries are carved in stone.

Traditional partitioning treats disk space like carved stone — boundaries are fixed at creation time. Adding a disk means copying data. Running out of space means an outage. Growing a partition requires the partition after it to be deleted first, which requires its data to go somewhere, which is a cascade that can take hours.

This is the problem LVM solves. LVM introduces a layer of indirection between physical storage and the filesystems that use it. You stop thinking in terms of fixed partitions and start thinking in terms of pools of storage that can grow, shrink, and move while the system is running.

---

## Mental Model

Think of physical hard drives as water tanks. A Volume Group is a reservoir made by connecting multiple tanks together. Logical Volumes are pipes you run out of that reservoir — each pipe can be any length you want, and if you add another tank to the reservoir, you can extend any pipe to be longer without stopping the water flow. If a tank develops a crack, you can pump its water to a new tank while the pipes stay active and the water keeps flowing.

The reservoir doesn't care which physical tank the water came from. The pipes don't care that a new tank was added. You manage the reservoir, and the reservoir handles the plumbing.

---

## Theory

LVM has four core concepts. Every command you'll ever run maps back to one of these.

**Physical Volume (PV).** A raw block device — a whole disk (`/dev/sdb`) or a partition (`/dev/sdb1`) — that has been initialized for use by LVM. `pvcreate` stamps an LVM label on it and records how many Physical Extents (PEs) it contains. A PE is the smallest unit of allocation in LVM, typically 4MB. The PV is the raw material.

**Volume Group (VG).** One or more PVs pooled together. When you run `vgcreate`, LVM merges the extents from all member PVs into a single addressable pool of storage. To the rest of the system, a VG looks like a reservoir of N total extents. The VG is the pool.

**Logical Volume (LV).** A slice of a VG, presented to the OS as a block device at `/dev/VG_NAME/LV_NAME` (also accessible via the device mapper path `/dev/mapper/VG_NAME-LV_NAME`). You put a filesystem on an LV, mount it, and use it like you would a partition. The LV is the pipe. Its size is measured in extents taken from the VG's pool.

**Physical Extent (PE).** The atomic unit of allocation. When an LV is created, LVM assigns it a set of PEs — possibly from different PVs, possibly not contiguous. The kernel's device mapper (`dm`) layer handles the translation of reads and writes on the LV to the correct physical locations. The PE is the unit of accounting.

The invariants that always hold:

- A PV belongs to at most one VG at a time
- An LV can only draw extents from its own VG
- The total extents allocated to all LVs in a VG cannot exceed the VG's total extents
- The device mapper handles the PV→PE→LV mapping transparently; applications see only the LV

LVM also supports two advanced features built on this foundation:

**Snapshots.** A snapshot LV records the original state of a source LV at a point in time using copy-on-write (COW). When a write lands on the origin LV, the original block is first copied to the snapshot's COW area, then the write proceeds. The snapshot LV reflects the origin as it existed at snapshot-creation time. This is how you take consistent backups of a live filesystem without unmounting.

**Thin Provisioning.** A thin pool is a special LV that provides "overcommitted" storage — you can create thin LVs whose combined size exceeds the pool's actual physical size, betting that not all of it will be used simultaneously. This is production-ready (been in the mainline kernel since 3.2) and is how cloud providers create disk images without actually allocating all declared space upfront. The `dm_thin_pool` kernel module handles it.

## How It Works (Mechanics)

LVM is implemented as a set of device mapper targets in the kernel. When you create a VG and an LV, LVM writes configuration metadata to the beginning of each PV (the LVM label area). The `lvm2` userspace tools read this metadata, construct the mapping tables, and call `ioctl()` on `/dev/mapper/control` to load those tables into the kernel's device mapper.

A simplified trace of `lvcreate -L 10G -n data myvg`:

```text
# Conceptual flow — not a real command
1. lvcreate reads VG metadata from PV headers
2. Selects 2560 free PEs (at 4MB each = 10GB) from the VG pool
3. Constructs a device mapper "linear" mapping table:
      0 10485760 linear /dev/sdb 2048
      (LV block 0, length 10M sectors, maps to /dev/sdb starting at sector 2048)
4. Calls dm ioctl to load the table as /dev/mapper/myvg-data
5. Creates the symlink /dev/myvg/data → /dev/mapper/myvg-data
6. Writes updated metadata (new LV, used PEs) back to PV headers
```

The kernel then handles all I/O through the device mapper translation table. When an application writes to `/dev/myvg/data`, the block layer intercepts it, the dm-linear target translates the LV sector address to the physical PV sector address, and the I/O proceeds to the underlying device.

Snapshots use the `dm-snapshot` target instead of `dm-linear`. The snapshot target maintains a COW exception table. On first write to any block of the origin, the exception table triggers a read-then-write: original block → snapshot's COW area, then the write → origin. Subsequent writes to the same block skip the copy (the original is already preserved).

`pvmove` works by temporarily converting the LV's mapping to a `dm-mirror` target — mirroring each PE being moved to its destination simultaneously — then, once sync is complete, switching the mapping to point exclusively at the destination.

## Basic Example

Setting up a simple LVM stack from scratch. This builds a VG from one disk, creates an LV, formats it, and mounts it.

```bash
# Step 1: Initialize a disk as a Physical Volume
# Replace /dev/sdb with your actual unused disk — verify with lsblk first
sudo pvcreate /dev/sdb
```

```text
  Physical volume "/dev/sdb" successfully created.
```

```bash
# Step 2: Create a Volume Group named "datavg" using that PV
sudo vgcreate datavg /dev/sdb
```

```text
  Volume group "datavg" successfully created
```

```bash
# Step 3: Create a Logical Volume — 10GB, named "data"
sudo lvcreate -L 10G -n data datavg
```

```text
  Logical volume "data" created.
```

```bash
# Step 4: Put a filesystem on the LV
sudo mkfs.ext4 /dev/datavg/data
```

```bash
# Step 5: Mount it
sudo mkdir /mnt/data
sudo mount /dev/datavg/data /mnt/data

# Verify
df -h /mnt/data
```

```text
Filesystem               Size  Used Avail Use% Mounted on
/dev/mapper/datavg-data  9.8G   24K  9.3G   1% /mnt/data
```

The device appears as `/dev/mapper/datavg-data` — that is the device mapper name. `/dev/datavg/data` is a symlink to it. Both refer to the same LV.

Now try this yourself. Pay attention to the `df` output: the filesystem is 9.8G despite the LV being 10G — ext4 reserves some space for its own metadata. That's normal. If your output looks slightly different, it's likely a difference in the ext4 block reservation percentage — run `tune2fs -l /dev/datavg/data` to inspect the filesystem parameters.

## Real-World Example

A server running low on `/var` — a very common production scenario. The disk `/dev/sdc` has been attached as a new physical disk. The goal is to extend the existing VG, extend the LV, and resize the filesystem, all without an outage.

```bash
# Confirm what we have before touching anything
sudo vgs
sudo lvs -o lv_name,lv_size,vg_name
```

```text
  VG      #PV #LV #SN Attr   VSize   VFree
  sysvg     1   2   0 wz--n- <50.00g    0

  LV      LSize   VG
  root    40.00g  sysvg
  var     10.00g  sysvg
```

The VG has no free space — VFree is 0. The `/var` LV is 10GB. We need to add `/dev/sdc`.

```bash
# Initialize the new disk as a PV
sudo pvcreate /dev/sdc
```

```text
  Physical volume "/dev/sdc" successfully created.
```

```bash
# Add it to the existing VG — the VG now has more free extents
sudo vgextend sysvg /dev/sdc
```

```text
  Volume group "sysvg" successfully extended
```

```bash
# Confirm free space is now available
sudo vgs
```

```text
  VG      #PV #LV #SN Attr   VSize    VFree
  sysvg     2   2   0 wz--n- <100.00g <50.00g
```

```bash
# Extend the LV and resize the filesystem in one step
# -r / --resizefs tells lvextend to also resize the filesystem automatically
# -L +20G adds 20GB to the current size
sudo lvextend -r -L +20G /dev/sysvg/var
```

```text
  Size of logical volume sysvg/var changed from 10.00 GiB (2560 extents) to 30.00 GiB (7680 extents).
  Logical volume sysvg/var successfully resized.
resize2fs 1.46.5 (30-Dec-2021)
Filesystem at /dev/mapper/sysvg-var is mounted on /var; on-line resizing required
old_desc_blocks = 2, new_desc_blocks = 4
The filesystem on /dev/mapper/sysvg-var is now 7864320 (4k) blocks long.
```

```bash
# Verify — the filesystem is now larger, mounted live
df -h /var
```

```text
Filesystem               Size  Used Avail Use% Mounted on
/dev/mapper/sysvg-var    30G   8.4G   20G  30% /var
```

Walk through what just happened: `vgextend` added `/dev/sdc`'s physical extents to `sysvg`'s pool. `lvextend -r` both allocated 20GB of new extents to the `var` LV and called `resize2fs` to expand the ext4 filesystem to fill the new LV size. The filesystem was live and mounted throughout. Applications writing to `/var` saw no interruption.

The `-r` flag is the right default. It removes the most common mistake in LVM — extending the LV but forgetting to extend the filesystem. If you ever need to do it manually (for example, `-r` fails on an unusual filesystem), the sequence is:

```bash
# Manual two-step: extend the LV first, then the filesystem
sudo lvextend -L +20G /dev/sysvg/var
# ext4: pass the device path
sudo resize2fs /dev/sysvg/var
# XFS: pass the MOUNT POINT, not the device path
# sudo xfs_growfs /var
```

For adding to `/etc/fstab` so the mount survives a reboot, use the device mapper path or a UUID:

```bash
# Get the UUID of the LV's filesystem
sudo blkid /dev/sysvg/var
```

```ini
# /etc/fstab entry
UUID=a1b2c3d4-...  /var  ext4  defaults  0  2
```

## ⚠️ Gotchas

### `lvextend` Without Resizing the Filesystem

You extend the LV. `lvextend` reports success. You check with `df` and the free space hasn't changed. The LV is larger — but the filesystem sitting on it still has the old size stamped into its superblock. It doesn't know the LV grew.

```bash
# Looks right — LV is now bigger
sudo lvextend -L +10G /dev/datavg/data
```

```text
  Logical volume datavg/data successfully resized.
```

```bash
# But the filesystem hasn't heard the news
df -h /mnt/data
```

```text
Filesystem               Size  Used Avail Use% Mounted on
/dev/mapper/datavg-data  9.8G  9.1G  700M  93% /mnt/data
```

The LV is 20GB. The filesystem still thinks it's 10GB. The extra 10GB of the LV exists but is unreachable — the filesystem won't write there because it doesn't know it exists.

The fix is always to follow `lvextend` with a filesystem resize. Use `-r` to do both in one step, or run `resize2fs` for ext4 or `xfs_growfs` for XFS manually. After the filesystem resize, `df` will reflect the full new size.

---

### `lvreduce` Before Shrinking the Filesystem

> ⚠️ **WARNING:** Shrinking an LV before shrinking the filesystem guarantees data corruption and an unbootable volume. There is no recovery without a backup. The order is non-negotiable: filesystem first, LV second.

This is the most dangerous sequence in LVM. The filesystem's superblock records how many blocks exist. If you shrink the LV first, the filesystem's tail blocks — where live data may live — are now physically gone. The filesystem has no idea. It will try to read and write those blocks. Reads return I/O errors or garbage. Writes to "free" space in the now-truncated region silently corrupt data.

```bash
# DO NOT DO THIS FIRST
# Shrinking the LV before the filesystem:
sudo lvreduce -L -5G /dev/datavg/data    # LV shrinks — filesystem now hangs off the end
```

The ext4 filesystem now thinks it has 15GB of blocks but only 10GB of LV exists. Every block beyond sector ~20971520 is inaccessible. If the journal or inode table lives there, the filesystem won't mount. If user data lives there, it's gone.

The correct order, every time:

```bash
# Step 1: Unmount — you cannot shrink an ext4 filesystem online
sudo umount /mnt/data

# Step 2: Check the filesystem for errors before touching it
sudo e2fsck -f /dev/datavg/data

# Step 3: Shrink the FILESYSTEM to the new target size first
sudo resize2fs /dev/datavg/data 10G

# Step 4: Now shrink the LV — it's safe because the filesystem no longer uses the space
sudo lvreduce -L 10G /dev/datavg/data

# Step 5: Remount
sudo mount /dev/datavg/data /mnt/data
```

The mnemonic: **shrink the filesystem before you shrink the container it lives in.** Always.

XFS cannot be shrunk at all — it only grows. If you need a smaller XFS volume, you must back up the data, destroy the LV, create a new smaller LV, format it, and restore.

---

### Snapshot Fills Up and Becomes Invalid

A snapshot doesn't store the full content of the origin LV — it stores only the blocks that have changed since the snapshot was taken (the COW exception list). If the origin LV is busy and the snapshot's COW area fills up, the snapshot becomes invalid. Reads from an invalid snapshot return I/O errors. The snapshot doesn't affect the origin — the origin continues working — but your backup or rollback target is gone.

```bash
# Creating a snapshot that's too small for a busy volume
sudo lvcreate -s -L 1G -n data_snap /dev/datavg/data
# If data_snap's 1G COW area fills before you finish using it...
```

```text
# Reading from the snapshot after it's exceeded its COW area:
Input/output error
```

Size snapshots at 10–20% of the origin for a quiet volume, or use thin provisioning snapshots (which draw from the shared pool and don't have a fixed COW area). Monitor with:

```bash
sudo lvs -o lv_name,data_percent,snap_percent
```

When `snap_percent` approaches 100, extend the snapshot immediately or it will invalidate:

```bash
sudo lvextend -L +5G /dev/datavg/data_snap
```

---

### `pvremove` on a PV That's Still in a VG

You're decommissioning an old disk. You run `pvremove`. It fails — which is the correct behavior, not a bug.

```bash
sudo pvremove /dev/sdb
```

```text
  PV /dev/sdb is used by VG datavg so please use vgreduce first.
  (If you are certain you need pvremove, then confirm by using --force twice.)
```

LVM refuses because the VG's metadata still references this PV, and removing it without migrating data off it would orphan any extents allocated there. The safe path is to migrate data off the PV with `pvmove`, remove it from the VG with `vgreduce`, and then `pvremove` it:

```bash
# Move all extents off /dev/sdb to remaining PVs in the VG
sudo pvmove /dev/sdb

# Remove the empty PV from the VG
sudo vgreduce datavg /dev/sdb

# Now pvremove is safe — the PV has no VG claiming it
sudo pvremove /dev/sdb
```

---

### `pvmove` Is Interrupted — Data Is Safe, But Migration Must Be Resumed

`pvmove` does not resume automatically after a crash or reboot. If you interrupt a `pvmove` (power loss, Ctrl-C, reboot), the migration is left mid-flight in a mirrored state. The data is safe — both source and destination hold valid copies — but neither is authoritative until the move finishes.

After a reboot, check status:

```bash
sudo lvs -a -o lv_name,copy_percent
```

If `copy_percent` shows an incomplete value, resume the move:

```bash
sudo pvmove    # no arguments — resumes all in-progress moves
```

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install lvm2`; Ubuntu 24.04 ships 2.03.16; config at `/etc/lvm/lvm.conf` |
| RHEL / Fedora / Rocky / AlmaLinux | `dnf install lvm2`; RHEL 9 ships ~2.03.33; thin provisioning tools in `device-mapper-persistent-data` (`thin_check`, `thin_repair`) |
| Arch | `pacman -S lvm2`; typically at upstream version (2.03.41 as of 2026) |
| Alpine (containers) | `apk add lvm2`; rarely used in containers directly; relevant for VM-backed container hosts |

LVM is a kernel feature — the version shipped by the distro is the userspace tools (`lvm2`). The underlying `dm-*` kernel modules are universal across distros on the same kernel version.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| LVM2 | General-purpose flexible storage on physical or VM hosts | Current standard; version 2.03.x; supports thin, snapshots, pvmove |
| Btrfs | When you want built-in snapshotting and pooling at the filesystem level without a separate volume manager | Subvolume snapshots are faster and more space-efficient than LVM snapshots for many workloads; no separate LV management needed |
| ZFS (via OpenZFS) | NAS/SAN workloads, maximum data integrity, deduplication | Not in mainline kernel; requires DKMS or native module; significantly different mental model |
| Plain partitions | Single-disk systems where flexibility is never needed | Simpler, zero overhead, but no resize/move/snapshot without downtime |
| Stratis | RHEL 9+ as a higher-level abstraction over LVM/XFS | Manages LVM and XFS together via a daemon; hides LVM complexity but removes some control |

For a production Linux server where you need flexibility — especially online resize and disk replacement — LVM is the right default. Btrfs is a reasonable choice if you're already familiar with it and want integrated snapshotting. ZFS is compelling for storage-heavy workloads if you can commit to managing the DKMS module. Plain partitions belong on single-purpose systems where nothing will ever need to change.

## Trade-offs

**Where this shines:**
- Online resize without downtime — extend a live filesystem in minutes
- Disk replacement while running — `pvmove` migrates data live, then swap the disk
- Consistent backups on a live system — create a snapshot, back up the snapshot, delete it
- Thin provisioning — overcommit storage in environments where you know not everything will be used
- Aggregating multiple small disks into one large pool without hardware RAID

**Where this costs you:**
- Additional complexity — one more abstraction layer to understand, debug, and maintain; when things go wrong, the LVM layer is one more place to check
- Recovery is harder — a corrupted VG metadata header can make all LVs inaccessible simultaneously; regular `vgcfgbackup` is not optional
- Snapshot COW overhead — heavy write workloads on snapshotted volumes add latency because every modified block is copied to the COW area before write; thin snapshots reduce this but add different complexity
- Not portable without the VG — an LV on a disk moved to another machine requires the VG metadata to be imported with `vgscan` and `vgchange -ay`

**Performance note:** A linear LVM LV on a single PV has near-zero overhead — the device mapper translation is a simple arithmetic operation in the kernel hot path. The overhead becomes measurable only with snapshots (COW copies on write) or with heavily fragmented LVs spread across many PVs with many small extents.

## When to Use / When to Avoid

✅ **Use this when:**
- Any server-grade installation where disk space requirements may change over time (which is nearly all of them)
- You need online filesystem growth without a reboot or service interruption
- You're replacing a physical disk on a running system and need to migrate data first
- You need point-in-time snapshots for consistent backups without unmounting
- You're pooling multiple disks into a single logical volume for a database or data directory

❌ **Avoid this when:**
- A single-use container or VM with a fixed, known-forever storage requirement — the overhead of LVM setup adds nothing
- You're already using ZFS or Btrfs — both handle pooling and snapshots natively; layering LVM underneath creates redundancy without benefit
- An embedded system or minimal appliance where the `lvm2` daemon is a significant overhead
- You need RAID redundancy — LVM mirroring exists but software RAID (`mdadm`) is better for that specific job; use LVM on top of RAID, not instead of it

⚠️ **Common mistakes:**
- Using `lvextend` without `-r` and wondering why the filesystem is still the old size
- Creating snapshots without monitoring `snap_percent` — a full snapshot silently invalidates
- Skipping `vgcfgbackup` — the VG configuration is stored on disk and should be backed up regularly; if all PV headers corrupt, the LVM metadata is gone
- Mixing LVM mirroring and hardware RAID without understanding which layer owns what

## Veteran Perspective

The most common mistake beginners make with LVM is treating it as a partitioning tool. It's not. It's a storage abstraction layer. The difference matters the moment something goes wrong: when a partition breaks, you fix a partition. When an LV breaks, you need to understand the full chain — is it the PV, the VG metadata, the LV mapping, or the filesystem on top?

The intermediate mistake is skipping `vgcfgbackup`. LVM automatically backs up VG configuration to `/etc/lvm/backup/` on every change — most people don't know this file exists until they're recovering from a disaster. Make sure you know where it is and that it's being backed up off-machine. If every PV in a VG simultaneously loses its header (it happens — a bad firmware update, a rogue `dd`), that file is how you recover.

The subtle thing most sysadmins miss: `pvmove` is not instantaneous. On a 2TB PV with active I/O, a `pvmove` can take hours. Running it in a `screen` or `tmux` session isn't optional — it's required. And because it doesn't resume automatically after a reboot, you need to track it: don't reboot the system during a `pvmove` unless you're prepared to re-run it.

The order-of-operations for shrinking is the trap that has caused more filesystem corruption than any other LVM operation. I've seen the wrong order happen twice in production. Both times, the sysadmin remembered the command syntax correctly but got the order backwards. The filesystem went read-only. The system needed to be booted to a live image, and recovery was possible only because there was a recent backup. The mnemonic is: **filesystem first, always**. Shrink the filesystem, then shrink the container.

For snapshots in production: use them for backups, not for rollbacks. The COW overhead on a busy write-heavy LV is not zero. Taking a snapshot of a database LV and then hammering writes on the origin will make the database noticeably slower until the snapshot is removed. Thin provisioning snapshots are better here — they use the shared pool rather than a dedicated COW area — but they add the complexity of managing pool space.

In a technical interview, LVM comes up in two forms: "explain how you'd extend a production filesystem with no downtime" (walk them through `vgextend` → `lvextend -r`) and "what's the difference between an LV and a partition?" (it's the abstraction layer, the device mapper, the PE accounting — not just that you can resize it). The second question is the one that separates people who've used LVM from people who understand it.

## Try It Yourself

These exercises assume access to a VM or a system with a spare disk you can safely use. Do not run any of these on a disk that holds data you want to keep.

1. Create a PV on a spare disk, build a VG, and create a 5GB LV. Format it with ext4, mount it, and confirm the mount with `df -h`. Run `pvs`, `vgs`, and `lvs` after each step and notice what changes.

2. Extend the LV from 5GB to 8GB using `lvextend -r`. Confirm that `df -h` shows the larger size before and after. Then run `lvextend` *without* `-r` to add another 2GB — notice that `df` doesn't change. Fix it with `resize2fs` manually.

3. Create a snapshot of your LV while it's mounted. Write a large file to the origin LV. Then compare `df -h` on the origin and snapshot mounts to see that the snapshot still reflects the original state. Check `snap_percent` with `lvs` to see how much of the COW area was consumed.

4. Add a second disk to your VG with `vgextend`. Use `pvmove` to migrate all data off the first disk. Use `vgreduce` to remove the first PV from the VG. Confirm with `pvs` and `vgdisplay` that the first disk is no longer part of the VG.

5. **Challenge:** Set up a thin pool LV, create three thin LVs whose combined declared size exceeds the pool size (deliberately overcommit). Format and mount all three. Monitor pool usage with `lvs -o lv_name,data_percent`. Write data to the thin LVs until you approach pool exhaustion — observe what happens. Then extend the thin pool with `lvextend`. Do all of this in a VM.

## Summary

- LVM's three layers are Physical Volume (raw block device), Volume Group (the pool), and Logical Volume (the usable slice) — understanding all three is required to use LVM safely
- Always use `lvextend -r` to extend both the LV and filesystem in one step; if done manually, the filesystem resize (`resize2fs` for ext4, `xfs_growfs` for XFS) must come *after* `lvextend`
- When shrinking, the filesystem must be shrunk *before* the LV — reversing this order causes immediate data corruption; XFS cannot be shrunk at all
- Snapshot COW areas must be monitored — a snapshot that fills up becomes invalid; size them at 10–20% of the origin or use thin provisioning snapshots
- `pvmove` migrates data live but does not auto-resume after a reboot; always run it in a persistent session and re-run it manually if interrupted
