# [12.4] — Load Averages

> A high load average on its own tells you nothing — it is the combination of load average, CPU utilization, and I/O statistics together that tells you what your system is actually doing.

## The Problem This Solves

Before you had a number like load average, answering "is my system under pressure?" required looking at a half-dozen things simultaneously — how many processes were competing for the CPU, whether disk I/O was backed up, whether the run queue was growing. System administrators needed a single at-a-glance number that said "yes, this system is being pushed hard" or "this system is idle."

Load average gives you that number. It tells you the average number of processes that were either running or waiting for a resource over the last 1, 5, and 15 minutes. A single glance at the three values lets you immediately see whether your system is under load right now, was under load earlier, or has been consistently saturated for a while.

The problem is that most people stop there — they see a high load average and start debugging CPU. That leads them in circles because load average is not a CPU metric. It is a demand metric. The system could be pegged at load 20 with CPU usage at 5%, and every bit of that pressure could be coming from processes waiting on a slow disk. Load average only asks the question. It takes other tools to answer it.

---

## Mental Model

Think of load average as the average number of cars trying to use a road at any given moment — not just cars actively moving, but also cars stuck at a red light or waiting to merge. If you have a 4-lane highway (4 CPU cores) and 4 cars on it, traffic flows fine. If you have 8 cars for 4 lanes, some cars are waiting. If 2 of those 8 cars are stuck waiting because a truck is blocking the merge lane (a slow disk), they still count as part of your "load" — they are still trying to get somewhere and the road is still congested because of them.

That last point is where Linux differs from other Unix systems. The original Unix load average only counted cars that were actively moving or ready to move. Linux, since 1993, also counts cars stuck behind the blocking truck. You cannot tell from the car count alone whether the congestion is because the road is too narrow or because something is blocking the merge lane. That requires looking at the road itself.

## Theory

Load average in Linux measures the average number of processes in one of two states:

- **Runnable (R):** The process is either actively running on a CPU or ready to run and waiting for a CPU to become free.
- **Uninterruptible sleep (D):** The process is waiting for a hardware resource — almost always I/O — and cannot be interrupted or killed while it waits.

Every five seconds, the Linux kernel samples how many processes are currently in either of those two states and folds that sample into three running averages — the 1-minute, 5-minute, and 15-minute load averages. These are not rolling windows that compute exact averages over 60, 300, and 900 seconds. They are **exponentially weighted moving averages (EWMA)**. Each new sample has the most weight, and the influence of older samples decays exponentially over time.

The practical consequence of EWMA: the "1-minute" average is not the average of the last 12 samples. It is the average where the most recent sample counts the most, and a sample from 5 minutes ago has barely any weight at all. The 15-minute average is the slowest to change — it carries memory of system state going back much further. This is why the three numbers together tell a story about the direction load is moving, not just its current level.

**Load average is not CPU utilization.** This is the most important thing to understand about this metric. CPU utilization measures what fraction of available CPU time is actually being used — a percentage. Load average measures how many processes are demanding system resources — an absolute count. You can have 100% CPU utilization with a load average of 1.0 (one process, fully occupying one core), or you can have 5% CPU utilization with a load average of 20.0 (20 processes all blocked waiting for a slow disk, each briefly waking up to do a tiny bit of computation before sleeping again). The numbers measure completely different things.

**Normalizing load average by CPU count.** Because load average is an absolute count of processes, you must compare it to the number of CPUs (or CPU cores) on the system to decide whether it is high or low. On a 1-core system, a load average of 1.0 means exactly one process was demanding the CPU at any moment — that core was fully saturated. On a 4-core system, a load average of 4.0 means all four cores were exactly saturated. On a 32-core machine, a load average of 4.0 means the system was barely busy — only 12.5% of capacity was in demand.

The rule of thumb: if load average divided by CPU count is consistently above 1.0, you have more demand than capacity. Below 1.0 means demand fits within available resources. At exactly 1.0 per core, the system is at full utilization with no slack.

**The D-state contribution and why it was added.** Classic Unix systems — BSD, Solaris — only counted runnable processes in the load average. They did not include processes in uninterruptible sleep. Linux diverged from this in 1993, when a kernel developer named Matthias Urlichs submitted a patch to include TASK_UNINTERRUPTIBLE in the load average calculation. His reasoning, recorded in the patch itself: load averages should reflect system load, not just CPU load. If the system is loaded with I/O-bound processes all waiting for disk, that is real demand on the system — the system is not idle, even if the CPU is. Excluding those processes would make the load average report "low" precisely when the system was under the most stress from a slow storage subsystem.

The consequence of this design decision is that on Linux, unlike on other Unix systems, a disk I/O saturation event will show up as high load average even if the CPU is almost entirely idle. This is correct behavior by design. It also means you must look at more than load average to know whether your problem is CPU or I/O.

## How It Works (Mechanics)

The kernel calculates load average in `kernel/sched/loadavg.c`. Every tick of a timer (approximately every 5 seconds via `LOAD_FREQ`), the scheduler counts the number of processes currently on the global run queue (state R) plus the number in uninterruptible sleep (state D) across all CPUs, and feeds that sample into the EWMA calculation.

The EWMA formula for each of the three averages uses a decay constant `exp(-5/60)`, `exp(-5/300)`, and `exp(-5/900)` for the 1-, 5-, and 15-minute averages respectively. In practice, the kernel approximates these with fixed-point arithmetic to avoid floating-point operations in interrupt context. The calculation looks like this:

```text
# Conceptual flow — not real code
load_1min  = load_1min  * exp(-5/60)   + n * (1 - exp(-5/60))
load_5min  = load_5min  * exp(-5/300)  + n * (1 - exp(-5/300))
load_15min = load_15min * exp(-5/900)  + n * (1 - exp(-5/900))
```

Where `n` is the current sample count of R + D processes. Each new value is a weighted blend of the old average and the new sample, with the old average carrying more weight for the slower averages.

The three resulting values are stored in the kernel and exposed to userspace through `/proc/loadavg`:

```bash
cat /proc/loadavg
```

```text
2.14 1.87 1.63 3/412 28741
```

The five fields are: 1-minute average, 5-minute average, 15-minute average, the ratio of currently runnable processes to total processes (3/412 here means 3 processes are runnable, 412 exist total), and the PID of the most recently created process.

Every tool that shows load average — `uptime`, `top`, `w` — reads from this file. They are all reading the same three numbers.

`uptime` is the simplest way to see both load averages and how long the system has been running:

```bash
uptime
```

```text
 14:32:05 up 42 days, 3:17,  2 users,  load average: 2.14, 1.87, 1.63
```

The output tells you: current time, how long the system has been up, how many users are logged in, and the three load averages. This is the standard first glance — if the load numbers look elevated, you then go to `vmstat` or `iostat` to understand why.

## Basic Example

```bash
# The simplest way to see load averages — uptime
uptime
```

```text
 14:32:05 up 42 days,  3:17,  2 users,  load average: 0.45, 0.61, 0.72
```

```bash
# Check how many CPUs you have, to put the load average in context
nproc
```

```text
4
```

```bash
# Read load average directly from the kernel
cat /proc/loadavg
```

```text
0.45 0.61 0.72 1/412 28741
```

On this 4-core system, a load average of 0.45 (1-minute) means the system is lightly loaded — only about 11% of CPU capacity is in demand. Notice the direction: the 15-minute average (0.72) is higher than the 1-minute average (0.45), which means load has been decreasing over the last quarter-hour. Whatever was pushing the load up has calmed down.

Try it yourself. Run `uptime` on your system, then run `nproc` to get your CPU count, and calculate load / nproc for each of the three averages. Are any of them above 1.0? If so, which direction is load trending? Try it now — look at whether the 1-minute is higher or lower than the 15-minute.

## Real-World Example

A monitoring alert fires: load average is 18.0 on an 8-core production database server. Here is the diagnostic sequence.

```bash
# First: confirm the load and check direction
uptime
```

```text
 16:44:11 up 128 days,  7:02,  1 user,  load average: 18.32, 15.47, 8.21
```

Load of 18.32 on an 8-core machine means 2.3x the cores are in demand. The 1-minute is higher than the 15-minute, so load is increasing — this is happening now and getting worse.

```bash
# Second: check CPU utilization — is the CPU actually busy, or is this I/O?
# vmstat 1: run every 1 second; the first line is since boot (ignore it), subsequent lines are real-time
vmstat 1 5
```

```text
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
 2 16      0 142368  81204 3847920    0    0  4821 38291 1247 2341  3  2 14 81  0
 1 17      0 141872  81204 3848016    0    0  4904 37812 1189 2298  2  1 15 82  0
 2 15      0 142240  81204 3848016    0    0  5012 39104 1302 2401  3  2 14 81  0
 1 16      0 141568  81204 3848016    0    0  4788 38560 1191 2319  2  1 16 81  0
 3 15      0 142112  81204 3848016    0    0  4923 37998 1244 2374  3  2 14 81  0
```

The `wa` column under `cpu` is 81%. I/O wait. The CPU is idle 14% of the time, doing real work only 5% of the time — but 81% of the time it is waiting for I/O to complete. The `b` column under `procs` shows 15-17 processes in uninterruptible sleep (D state) at any moment. These are the processes driving the load average up.

The `bi` and `bo` columns show block I/O: 4,800-5,000 blocks/second being read, 37,000-39,000 blocks/second being written. Something is hammering writes.

```bash
# Third: drill into disk performance with iostat
# -x: extended stats; -d: devices only (no CPU section); 1 3: every second, 3 times
iostat -xd 1 3
```

```text
Device            r/s     w/s     rkB/s     wkB/s   await  %util
sda              12.0   192.0     96.0  19456.0   312.4   98.2
sdb               0.0     0.1      0.0      0.4     1.2    0.1
```

`sda` is at 98.2% utilization and has an `await` of 312 milliseconds — every I/O request is taking over 300ms to complete. A healthy disk serves requests in under 1ms. This disk is saturated. The 19,456 kB/s of writes is a database flush — likely a checkpoint or compaction job.

```bash
# Fourth: find which processes are stuck in D state right now
# ps with axo to get state and command; grep for D state; ignore the grep process itself
ps axo stat,pid,comm | grep '^D'
```

```text
D  8842 postgres
D  8843 postgres
D  8844 postgres
D  8845 postgres
D  8846 postgres
D  8847 postgres
D  8848 postgres
D  8849 postgres
D  8850 postgres
D  8851 postgres
D  9012 postgres
D  9013 postgres
D  9014 postgres
D  9015 postgres
D  9016 postgres
```

Fifteen postgres worker processes all in uninterruptible sleep, waiting for the saturated disk. That is exactly the number the `b` column in `vmstat` showed. The load average is not a CPU problem — it is 15 postgres workers waiting on a single saturated disk. The fix is storage-side, not CPU-side.

This is the complete diagnostic story that load average alone cannot tell. High load was the alarm. CPU vs. I/O split via `vmstat wa` was the direction. `iostat -x` confirmed the specific device and its saturation. `ps` named the processes. Load average asked the question — every other tool answered it.

## ⚠️ Gotchas

### High Load Does Not Mean Something Is Wrong

Load average above 1.0 per CPU core is not inherently a problem. It is a measure of demand. Batch processing systems — video encoding farms, CI/CD workers, scientific computing clusters, database servers doing bulk ingest — routinely run at 200-400% load by design. They are intentionally saturating the machine with work.

The mistake is treating any elevated load average as an alert threshold without context. A load of 8.0 on a 4-core interactive web server is a problem because user-facing requests are queuing. A load of 8.0 on a 4-core batch worker that is crunching through a nightly job queue is the system working exactly as intended.

Before paging anyone, ask: is this system expected to be loaded right now? What is its baseline load at this time of day? Many monitoring systems trigger on absolute load average values without accounting for CPU count or expected workload. A machine that has run at load 12.0 every Tuesday at 2am because of a scheduled backup job does not need a 3am page.

---

### The 15-Minute Average Lags — Load May Have Already Passed

The three numbers in load average track different windows, and the 15-minute average carries memory from 15 minutes ago. If you see something like this:

```text
load average: 0.21, 2.14, 6.87
```

The 1-minute average is low (0.21). The 15-minute average is high (6.87). Whatever caused that spike has already resolved. The system is fine right now. Spending time diagnosing a problem that no longer exists — and finding nothing — is a common result of misreading these three numbers.

Read the direction: if 1-minute > 15-minute, load is climbing and the problem is happening now. If 1-minute < 15-minute, load is falling and the worst is behind you. If all three are roughly equal, the system is in a steady state.

---

### Load Average Does Not Distinguish CPU-Bound From I/O-Bound

Load average counts R + D processes. It does not tell you the ratio. A load of 10.0 could be:

- 10 CPU-bound processes competing for cores (fix: more CPU, or reduce parallelism)
- 10 I/O-bound processes all blocked on disk (fix: faster storage, or reduce I/O concurrency)
- 7 processes waiting on disk and 3 competing for CPU (mixed)

Load alone cannot tell you which. You must check `vmstat`'s `wa` column to see if I/O wait is significant, and then `iostat` to identify the device.

```bash
# Do this whenever you see elevated load — before doing anything else
vmstat 1 3
```

```text
procs -----------memory---------- ---swap-- -----io---- -system-- ------cpu-----
 r  b   swpd   free   buff  cache   si   so    bi    bo   in   cs us sy id wa st
```

If the `wa` column is high (above 10-15% is worth investigating), the load is I/O-driven. If `wa` is near zero and `us + sy` is high, the load is CPU-driven. If both are low and load is still high, look at swap (`si`/`so`) and check for memory pressure.

---

### `uptime` Load Average on Containers Is Often Misleading

Inside a container (Docker, LXC), `cat /proc/loadavg` and `uptime` show the load average of the **host kernel**, not the container. The container has no separate kernel — it shares the host's. This means a container that appears to have load average 0.1 might be running on a host that is under severe load from other containers. And a container that contributes heavily to host load will see a high load average that includes the load from completely unrelated workloads.

If you are doing performance analysis inside a container, load average is the wrong tool. Use CPU quota metrics from the container runtime, or cgroup CPU accounting instead.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | `uptime`, `vmstat`, `iostat` all available by default; `iostat` is in the `sysstat` package if missing: `apt install sysstat` |
| RHEL / Fedora / Rocky / AlmaLinux | `iostat` is in `sysstat`: `dnf install sysstat`; available but not always installed by default on minimal installs |
| Arch | `sysstat` package available via `pacman -S sysstat` for `iostat`; `uptime` and `vmstat` are in core |
| Alpine (containers) | `uptime` available; `vmstat` in `procps` package (`apk add procps`); load average reflects host kernel, not container — see gotcha above |

## Alternatives & Comparisons

| Tool | What It Adds Over `uptime` | When to Reach For It |
|---|---|---|
| `uptime` | Shows load averages and system uptime | First glance — is the system loaded? |
| `w` | Load averages + who is logged in + what they are running | Same data as `uptime` with user session context |
| `top` / `htop` | Load averages + per-process CPU/memory breakdown | When you need to identify which process is driving load |
| `vmstat 1` | CPU breakdown (us/sy/id/wa) + process queue depths (r/b) | First tool after seeing high load — determines CPU vs. I/O |
| `iostat -x` | Per-device I/O stats, utilization, await times | Confirms which disk is saturated; required for I/O-driven load |
| `sar -q` | Historical load average from `sysstat` | When you need to reconstruct what load looked like an hour ago |

Load average is a summary metric. Every tool below it in this table provides more specific evidence. The diagnostic path is always: `uptime` to see load → `vmstat` to see CPU vs. I/O split → `iostat` or `top` to identify the specific resource → `ps` to identify the processes.

## Trade-offs

**Where this shines:**
- Single at-a-glance number that captures both CPU demand and I/O demand in one metric — immediately surfaces problems that CPU utilization alone would miss
- Three time windows together reveal trends without requiring you to keep a history yourself — you can tell at a glance whether load is climbing, falling, or steady
- Exposed by virtually every monitoring system, uptime checker, and terminal prompt — universal availability across all Linux systems, no installation required
- D-state inclusion means I/O saturation events surface immediately as elevated load, rather than requiring an explicit I/O monitor to catch them

**Where this costs you:**
- Conflates CPU pressure and I/O pressure into one number — high load with unknown source requires additional tools to diagnose
- EWMA means spikes are smoothed — a 30-second burst of extreme load barely moves the 1-minute average, and moves the 15-minute average almost not at all
- Does not account for CPU count without manual normalization — a load of 8.0 is very different on a 2-core machine versus a 32-core machine
- Misleading inside containers — shows host kernel load, not container-scoped load

---

## When to Use / When to Avoid

✅ **Use this when:**
- You need a quick first-pass answer to "is this system healthy?" — `uptime` takes one second
- You want to see whether a system is trending toward or away from saturation — read the three numbers as a trend line
- You are writing a monitoring alert and want a coarse-grained demand signal — load average normalized by CPU count is a reasonable alert threshold when combined with other signals

❌ **Avoid this when:**
- You need to diagnose what specifically is causing system pressure — load average alone cannot answer this; use `vmstat` and `iostat`
- You are inside a container and trying to understand container-specific load — use cgroup metrics instead
- You are trying to measure CPU utilization for capacity planning — use `%usr + %sys` from `vmstat` or `sar -u`, not load average

⚠️ **Common mistakes:**
- Alerting on absolute load average values without normalizing by CPU count — a threshold of 5.0 is dangerously low for a 32-core machine and dangerously high for a 2-core machine
- Starting to debug CPU when `wa` is 80% — always check the `wa` column in `vmstat` before assuming load is CPU-driven
- Assuming the problem is still happening because the 15-minute average is high — check the 1-minute average; if it is low, the problem has already passed

## Veteran Perspective

The number of times I have watched engineers SSH into a server, see load average at 12.0, immediately run `top`, find nothing using CPU, and then declare "I can't reproduce it" — I have stopped counting. Load average is not a CPU metric. It has never been a CPU metric on Linux. The first thing you run after seeing high load is `vmstat 1 3`, not `top`. `vmstat` tells you in two seconds whether this is a CPU problem or a storage problem. Top cannot tell you that.

The intermediate mistake — the one I see from people who know about D-state — is jumping to `iostat` before `vmstat`. `vmstat` is the triage step. It gives you the CPU vs. I/O split and the count of blocked processes in one table. If `wa` is near zero and `b` is near zero, then you go to `top` to look at CPU. If `wa` is high or `b` is high, then you go to `iostat`. Skipping `vmstat` and going straight to `iostat` means you might spend ten minutes looking at I/O stats on a system where the real problem is a runaway process.

The 1993 design decision to include D-state is, in my opinion, exactly right. Before that patch, a system with 20 processes all waiting on a failing disk would report a load average near zero — because no one was competing for the CPU. That is a lie. The system is not idle. The system is in crisis. Matthias Urlichs fixed this, and the Linux load average has been honest about I/O pressure ever since.

The batch system context is genuinely misunderstood in monitoring. I have seen teams receive pages every night because a scheduled job runs their servers at load 300% for an hour. After two weeks of investigating and finding nothing wrong, they add a suppression. The right answer is to understand your baseline and set thresholds relative to it — not to suppress alerts blindly. Know what your system looks like when it is healthy. A load average spike only means something when it deviates from normal.

The one question this topic consistently generates in senior sysadmin interviews is: "You see load average at 20.0 on an 8-core machine, but CPU usage is 5%. What do you do?" The answer they are looking for is: run `vmstat 1` to confirm I/O wait is high, identify the blocked process count in the `b` column, then use `iostat -x` to find which device is saturated. Anyone who jumps straight to "add more CPU" without checking I/O has not actually debugged a production I/O saturation before.

## Try It Yourself

1. Run `uptime` on your system. Write down the three load average numbers. Then run `nproc` to get your CPU count and calculate the load-per-core ratio for each average. Is your system lightly loaded, at capacity, or over capacity?

2. Run `cat /proc/loadavg` and compare the output to `uptime`. The first three numbers should match. Then note the fourth field (the `running/total` process ratio). Run `ps aux | wc -l` and compare the total to what `/proc/loadavg` reports — they should be within a few processes of each other.

3. Run `vmstat 1 5` and watch the `r`, `b`, `wa`, `us`, and `sy` columns for 5 seconds. On a typical idle system, `wa` should be near zero and `r` should be 0 or 1. Note the relationship between the `b` column and what `uptime` shows for load average — they will not match exactly (different timing), but they should be in the same ballpark.

4. Simulate I/O load and watch load average respond. Run this in a terminal (requires `sudo`):
   ```bash
   # Write 1GB to /tmp continuously for 30 seconds — safe on any system with space
   dd if=/dev/zero of=/tmp/loadtest bs=1M count=1024 & sleep 15 && uptime && wait && uptime
   ```
   The first `uptime` runs while the write is in progress. The second runs after it completes. Watch the 1-minute average — does it reflect the write load? Then clean up: `rm /tmp/loadtest`

5. **Challenge:** Write a one-liner that watches load average every 2 seconds and prints it with a timestamp, a normalized load-per-core value, and a plain-English verdict ("ok", "elevated", "saturated") based on whether load/nproc is below 0.7, between 0.7 and 1.5, or above 1.5. Use only tools available on a base Linux install (`bash`, `awk`, `nproc`, `/proc/loadavg`). Run it in a terminal while you also run the `dd` command from prompt 4 in another terminal and watch the verdict change in real time.

---

## Summary

- Load average measures the average number of processes in R state (runnable) or D state (uninterruptible sleep) — it is a demand metric, not a CPU utilization metric
- Linux has included D-state (I/O-waiting) processes in load average since 1993 by design — a high load average with low CPU usage almost always means I/O saturation
- Normalize by CPU count: load / nproc above 1.0 means more demand than capacity; below 1.0 means slack remains
- Read all three numbers as a trend: 1-minute higher than 15-minute means load is climbing now; 1-minute lower means the problem has passed
- Load average alone cannot diagnose the cause — always follow up with `vmstat 1` to see the CPU vs. I/O split, then `iostat -x` if I/O wait is high
