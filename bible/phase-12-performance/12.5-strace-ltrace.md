# [12.5] — strace and ltrace

> When a program misbehaves and logs tell you nothing, strace shows you exactly which system calls it's making — the raw conversation between your process and the kernel.

## The Problem This Solves

A service is failing. It starts, runs for a moment, and exits with code 1. The logs say nothing useful. You check the config files — they look fine. You restart it — same result. You add more logging to the application — but you don't have the source, or it's a third-party binary, or the logging happens before the failure does.

You're staring at a black box.

The exact same problem happens when a program is slow. It's doing *something* — CPU is busy, or I/O is churning — but you can't tell what. Is it making thousands of redundant `stat()` calls? Is it reading the same config file in a loop? Is it hammering DNS? Without visibility into what the process is asking the kernel to do, you're guessing.

Before tracing tools existed, the answer was: attach a debugger, set breakpoints, and hope. That requires debug symbols, source code, and time you don't have during an incident.

`strace` gives you visibility without any of that. It intercepts every conversation the process has with the kernel and shows it to you, live, in human-readable form.

---

## Mental Model

Think of the kernel as a government bureaucracy. Your process is a citizen who needs things done — open a file, allocate memory, send a packet. The citizen can't do any of these things directly. They must submit a formal request — a system call — and wait for the bureaucracy to respond.

`strace` is someone standing behind the citizen with a notepad, writing down every request submitted and every response received: the form number, the arguments, the timestamp, and what came back. The citizen doesn't know they're being watched. The bureaucracy doesn't know either. You just read the notepad.

That's all `strace` does. It watches the request/response pairs and writes them down.

---

## Theory

Every program running on Linux operates in user space — a restricted environment where code cannot directly touch hardware, access arbitrary memory, or perform I/O. To do any of those things, the program must ask the kernel via a **system call** (syscall). System calls are the formal interface between user space and kernel space.

When a process calls `open("/etc/passwd", O_RDONLY)`, the C library translates that into a `sys_openat` syscall instruction. The CPU switches from user mode to kernel mode, the kernel handles the request, switches back, and returns a result — either a file descriptor or an error code. Every file open, every memory allocation, every network connection, every process fork — all of them go through this interface.

There are roughly 400 system calls on a modern Linux x86-64 kernel. Most programs use a small fraction of them repeatedly: `read`, `write`, `openat`, `close`, `mmap`, `brk`, `stat`, `clone`, `futex`. Understanding what each does is how you read strace output fluently.

`strace` works by using the kernel's `ptrace()` facility — the same mechanism used by debuggers like `gdb`. When you attach `strace` to a process, the kernel notifies the strace tracer on every syscall entry and exit. The tracer wakes up, reads the syscall arguments and return value from the traced process's registers and memory, formats them, and writes them to output. Then the traced process is allowed to continue.

This stop-and-report cycle on every single syscall is why strace has real overhead. It is not passive observation — it is active interruption.

`ltrace` works the same way, but at the library call level rather than the syscall level. Instead of intercepting calls to the kernel, it intercepts calls to shared libraries — `malloc`, `fopen`, `strcmp`, `printf`. If you're debugging why a program is calling the wrong library function or passing wrong arguments, ltrace is the right tool. If you're debugging I/O, file access, or anything that ultimately reaches the kernel, strace is what you want.

## How It Works (Mechanics)

When you run `strace ls`, here is what actually happens:

1. `strace` calls `fork()` to create a child process.
2. The child calls `ptrace(PTRACE_TRACEME, ...)` to declare itself traceable, then calls `execve("ls", ...)`.
3. The kernel, seeing the `PTRACE_TRACEME` flag, stops the child at the entry of every syscall and sends `SIGTRAP` to the strace parent.
4. strace wakes up, calls `ptrace(PTRACE_GETREGS, ...)` to read the syscall number and arguments from the child's registers.
5. strace formats the call and arguments into human-readable output.
6. strace calls `ptrace(PTRACE_SYSCALL, ...)` to resume the child until the syscall returns.
7. The kernel stops the child again on syscall exit. strace reads the return value from the registers.
8. strace formats the return value and prints the completed call.
9. Repeat for every syscall, until the child exits.

For attaching to a running process with `-p PID`, step 1-2 are replaced by `ptrace(PTRACE_ATTACH, PID)`, which sends `SIGSTOP` to the target process and forces it under tracer control.

The important structural detail: the traced process is **stopped** while strace is doing its work. Every syscall produces two context switches — one to wake up strace, one to wake up the traced process again. On a process making thousands of syscalls per second, this is where the overhead comes from. It's not monitoring in the background. It is physically pausing the process on every kernel call.

Here is what a simple `openat` call looks like decoded:

```text
# Conceptual trace of a single syscall — not a real command
Process calls open("/etc/resolv.conf", O_RDONLY)
  → CPU switches to kernel mode
  → kernel stops process, signals strace
  → strace reads: syscall=openat, args=AT_FDCWD, "/etc/resolv.conf", O_RDONLY
  → strace prints entry (when using -e raw or verbose mode)
  → strace resumes process
  → kernel completes the open, prepares to return
  → kernel stops process again at syscall exit
  → strace reads: return value = 3 (file descriptor)
  → strace prints: openat(AT_FDCWD, "/etc/resolv.conf", O_RDONLY) = 3
  → process continues with fd=3
```

The output format is always: `syscall_name(arg1, arg2, ...) = return_value`.

## Basic Example

```bash
# Trace all syscalls made by the 'ls' command
strace ls /tmp
```

```text
execve("/usr/bin/ls", ["ls", "/tmp"], 0x7ffd... /* 24 vars */) = 0
brk(NULL)                               = 0x5606e4e42000
arch_prctl(0x3001 /* ARCH_??? */, 0x7ffd...) = -1 EINVAL (Invalid argument)
mmap(NULL, 8192, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0) = 0x7f3a...
access("/etc/ld.so.preload", R_OK)      = -1 ENOENT (No such file or directory)
openat(AT_FDCWD, "/etc/ld.so.cache", O_RDONLY|O_CLOEXEC) = 3
fstat(3, {st_mode=S_IFREG|0644, st_size=86684, ...}) = 0
mmap(NULL, 86684, PROT_READ, MAP_PRIVATE, 3, 0) = 0x7f3a...
close(3)                                = 0
...
openat(AT_FDCWD, "/tmp", O_RDONLY|O_NONBLOCK|O_CLOEXEC|O_DIRECTORY) = 3
getdents64(3, /* 14 entries */, 32768)  = 440
...
write(1, "file1.txt  file2.log\n", 21) = 21
close(1)                                = 0
exit_group(0)                           = ?
+++ exited with 0 +++
```

Look at what this tells you before a single line of `/tmp` is printed: the dynamic linker opens `/etc/ld.so.cache`, maps it into memory, loads shared libraries. Then `ls` opens `/tmp` as a directory, calls `getdents64` to read directory entries, and writes the formatted result to stdout (fd 1). The very last line confirms a clean exit.

The `= -1 ENOENT` on the `access("/etc/ld.so.preload", ...)` call is not an error — it's the linker checking for a preload file that doesn't exist. This is normal. You'll see this on almost every traced command.

Try this yourself. Run `strace ls /etc 2>&1 | head -30` to see the first 30 lines. Notice that strace output goes to stderr by default — the `2>&1` merges it with stdout so you can pipe it. Count how many syscalls happen before any directory is even opened.

## Real-World Example

The scenario: a Python web application is starting up slower than expected — about 8 seconds from launch to first request. There's nothing obviously wrong. You want to know where the time is being spent.

**Step 1 — Get a syscall summary first, not a full trace**

```bash
# -c counts and times all syscalls; much lower overhead than full trace
# -S time sorts the summary table by time spent descending
strace -c -S time python3 app.py &
sleep 10
kill %1
```

```text
% time     seconds  usecs/call     calls    errors syscall
------ ----------- ----------- --------- --------- ----------------
 61.23    3.742108        1247      3001             stat
 18.44    1.126219         421      2673         42 openat
  9.11    0.556892         185      3012             read
  4.32    0.263991          98      2694             close
  3.21    0.196243         654       300             getdents64
  ...
------ ----------- ----------- --------- --------- ----------------
100.00    6.109432                 14823       187 total
```

Three thousand `stat()` calls accounting for 61% of startup time. That's the signal. Something is stat-ing a lot of paths — probably the module import machinery scanning `sys.path` entries looking for packages.

**Step 2 — Filter to just the file-related calls to see what paths are being stat'd**

```bash
# -e trace=%file traces all file-related syscalls as a group
# -tt adds microsecond timestamps to each line
# -s 256 shows up to 256 chars of string arguments (default is 32, which truncates paths)
# -o saves strace output to a file so it doesn't mix with the app's output
strace -e trace=%file -tt -s 256 -o /tmp/app-strace.txt python3 app.py
```

```text
# /tmp/app-strace.txt (excerpt)
09:14:22.103442 stat("/usr/lib/python3/dist-packages/django/__init__.py", ...) = 0
09:14:22.104891 stat("/usr/lib/python3/dist-packages/django/core/__init__.py", ...) = 0
09:14:22.105203 openat(AT_FDCWD, "/usr/lib/python3/dist-packages/django/core/handlers/__init__.py", O_RDONLY) = 7
09:14:22.112445 stat("/home/deploy/app/venv/lib/python3.11/site-packages/requests/__init__.py", ...) = 0
09:14:22.113001 stat("/home/deploy/app/venv/lib/python3.11/site-packages/requests/adapters.py", ...) = 0
...
09:14:25.881002 stat("/home/deploy/app/models/user.py", ...) = 0
09:14:25.881449 stat("/home/deploy/app/models/orders.py", ...) = 0
```

The timestamps expose the story: the first few seconds are Django's import chain. Then there's a gap around 09:14:25 where something is stat-ing the app's own models directory. By grepping for the largest timestamp gaps:

```bash
# Find the calls that took more than 100ms (lines where timestamp jumps)
awk -F'[: ]' '/stat|openat/{print $0}' /tmp/app-strace.txt | head -60
```

You've gone from "slow startup" to "here is the exact directory being scanned and when." That's diagnosable. In this case, the app had a misconfigured watchdog that was re-importing models on every startup. Without strace, this would have taken an hour of log archaeology. With it, ten minutes.

**Step 3 — Attach to a running process that's hung**

```bash
# Find the PID first
pgrep -a python3

# -p attaches to existing process without restarting it
# -f follows forks — attaches to all threads of a multithreaded process
# -tt for timestamps so you can see where it's spending time
strace -p 14823 -f -tt -e trace=%network,read,write 2>&1 | head -40
```

```text
[pid 14823] 09:31:17.001442 recvfrom(8, "", 4096, 0, NULL, NULL) = 0
[pid 14823] 09:31:17.001891 close(8)   = 0
[pid 14824] 09:31:17.002100 futex(0x7f..., FUTEX_WAIT_PRIVATE, 0, NULL) = 0
[pid 14824] 09:31:17.002200 futex(0x7f..., FUTEX_WAIT_PRIVATE, 0, NULL) = 0
[pid 14823] 09:31:22.118000 connect(9, {sa_family=AF_INET, sin_port=htons(5432), sin_addr=inet_addr("10.0.0.5")}, 16) = -1 EINPROGRESS (Operation now in progress)
```

The five-second gap between 09:31:17 and 09:31:22 before the `connect()` to the database host at port 5432 — that's your hang. The process received an empty response on one connection (`recvfrom` returning 0 = EOF), closed it, and then spent five seconds before attempting to reconnect. The `futex` calls in between are threads waiting on a mutex — probably a connection pool lock. The connection pool was exhausted, threads were waiting, and the reconnect logic had a 5-second retry backoff baked in.

None of that is in any log. All of it is visible in strace output.

## ⚠️ Gotchas

### strace Overhead Can Cascade in Production

The 2-10x slowdown figure is a floor, not a ceiling. On a process making heavy syscall volume — a database, an application server handling hundreds of requests per second — strace can slow it to the point where it stops responding to health checks, its upstream load balancer marks it dead, and traffic redistributes to the remaining servers, which then get overloaded, and the cascade begins.

```bash
# This looks reasonable — you're just attaching to one process
strace -p $(pgrep nginx | head -1)
```

If that nginx worker handles 5,000 req/s and drops to 500 req/s under strace, the upstream will consider it unhealthy. The fix is not to trace production workers under load without preparation:

```bash
# Use -c for a summary instead — much lower overhead (14% vs 95%)
strace -c -p <PID>

# Or isolate one backend instance from the load balancer first,
# then trace it before re-adding it to the pool
```

If you need production tracing without this risk, use `perf trace` instead — it uses kernel ring buffers and has roughly 10-15% overhead vs strace's 2-100x.

### Multithreaded Processes: -p Only Attaches to One Thread

If you attach to a multithreaded process by PID, strace only traces the main thread. The other threads keep running untraced. If your bug is in a worker thread, you'll see nothing useful.

```bash
# Attaches to PID 14823 only — other threads invisible
strace -p 14823
```

```text
# Output shows minimal syscalls — the main thread is just waiting
futex(0x... FUTEX_WAIT_PRIVATE, 0, NULL) = 0
futex(0x... FUTEX_WAIT_PRIVATE, 0, NULL) = 0
```

The real work is happening in worker threads you're not tracing. Use `-f` to follow all threads:

```bash
# -f follows forks and threads — all thread syscalls appear, prefixed with [pid N]
strace -f -p 14823
```

The `-f` flag was originally for following `fork()` into child processes. It also works for threads on Linux because threads are implemented as processes sharing memory (via `clone()` with `CLONE_THREAD`).

### strace Output Goes to stderr — Redirects Bite You

This is the most common first-day mistake.

```bash
# You want to capture the app's output and strace separately
strace myapp > output.txt
```

```text
# output.txt contains: the app's stdout only
# strace output: appears on your terminal, not in the file
```

strace deliberately writes to stderr to avoid corrupting the traced program's stdout. So `> output.txt` captures the program's output, not strace's output. To capture strace's output, use `-o`:

```bash
# -o writes strace output to a file; app stdout/stderr go to terminal normally
strace -o /tmp/strace.log myapp > output.txt
```

Or if you want everything in one place:

```bash
# 2>&1 merges strace's stderr with app's stdout, then both go to the file
strace myapp > output.txt 2>&1
```

The problem with `2>&1` is that strace output and app output are interleaved. Use `-o` in all production scenarios.

### Use Syscall Groups, Not Individual Syscall Lists

Beginners write things like `-e trace=open,openat,read,write,stat,lstat,fstat`. This breaks in two ways: you miss related calls (there are multiple `stat` variants — `fstatat`, `newfstatat`), and it's fragile across kernel versions.

```bash
# Fragile — misses fstatat, newfstatat, statx, and future variants
strace -e trace=stat,lstat,fstat,open,openat myapp
```

Use predefined groups instead:

```bash
# %file covers all file-related syscalls: open, openat, stat, lstat, fstat,
# statx, access, mkdir, rename, unlink, and more — as a maintained set
strace -e trace=%file myapp

# %network covers connect, bind, send, recv, socket, etc.
strace -e trace=%network myapp

# %process covers fork, clone, exec, wait, exit
strace -e trace=%process myapp
```

The groups are maintained with the strace binary and stay correct across kernel updates. Individual syscall lists are maintenance debt.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install strace ltrace` — both available by default in repos |
| RHEL / Fedora / CentOS / Rocky | `dnf install strace ltrace` — strace usually pre-installed; ltrace may need explicit install |
| Arch | `pacman -S strace ltrace` |
| Alpine (containers) | `apk add strace` — ltrace not in main repos; strace available but may need `--privileged` in Docker |

**Note on `-k` (stack traces):** The `-k` flag requires strace to be compiled with `libunwind` support. Debian/Ubuntu and Fedora packages include it; some minimal RHEL builds do not. Check with `strace --help | grep -i unwind` — if `-k` is listed, it's available. If not, install `strace` from a full package or build from source.

**Container environments:** `strace` requires `CAP_SYS_PTRACE`. In Docker, pass `--cap-add=SYS_PTRACE` or `--privileged`. In Kubernetes, this typically means a privileged debug pod or a node-level tool like `kubectl debug`.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `strace` | Debugging a specific process; understanding syscall behavior; interactive investigation | 2-100x overhead; best for targeted debugging, not production monitoring |
| `perf trace` | Production tracing where overhead matters; needs lower impact | ~10-15% overhead; uses kernel ring buffers; output similar to strace |
| `ltrace` | Debugging library call behavior (C stdlib, third-party libs) | Higher overhead than strace; less useful for I/O and kernel bugs |
| `bpftrace` | Production monitoring; custom tracing probes; fleet-wide observability | Near-zero overhead; requires kernel 4.9+ with BPF enabled; steeper learning curve |
| `sysdig` | Container-aware syscall tracing; historical recording | Production-capable; commercial and open-source variants exist |

The honest recommendation: reach for `strace` first in a debugging session — it's fast to start, output is immediately readable, and flags like `-e` and `-c` let you narrow focus quickly. If you're in a production environment where you can't afford the overhead, or if you need to trace without attaching, `perf trace` is the right tool. If you're building monitoring or need to trace across a fleet, learn `bpftrace`.

## Trade-offs

**Where this shines:**

- Diagnosing application startup failures with no logs — the failing `openat()` call and its `ENOENT` return value point directly at the missing file
- Identifying unexpected file access patterns — finding the config file being read from the wrong path, or the directory being scanned 3,000 times
- Understanding what a binary does without source code — black-box analysis using only observable kernel interface
- Debugging hung processes — the blocked syscall tells you exactly what the process is waiting for
- Teaching — strace output is one of the best ways to understand what libc functions actually do under the hood

**Where this costs you:**

- **Overhead:** The ptrace-based stop-and-inspect model adds 2-100x slowdown on syscall-intensive workloads. This is not configurable — it's the architectural cost of the approach
- **Noise volume:** A non-filtered trace of almost any program produces hundreds to thousands of lines before any application-specific behavior appears; filtering discipline is required
- **Multithreaded complexity:** A heavily threaded process with many threads all making syscalls produces interleaved output that's hard to read, even with `-f` and per-thread filtering
- **Missing context:** strace shows you the syscall arguments and return value. It does not show you the call stack that led there, or the application-level state. For that, combine with `-k` (stack traces) or a debugger

**Performance note:** The `-c` summary mode (counting and timing syscalls) reduces overhead to roughly 10-15% because strace only reads the syscall return value, not the full argument list. Full call logging (`-tt -s 256`) maximizes overhead because strace reads all string arguments from the traced process's memory on every call.

## When to Use / When to Avoid

✅ **Use this when:**

- A process exits unexpectedly and you need to find the failing syscall and its error code
- A program is reading from the wrong config file and you need to confirm the actual path being opened
- A process is slow and you want to identify which syscall category dominates execution time (use `-c` first)
- You need to understand what a third-party binary does without source code
- A process is hung and you need to identify what syscall it's blocking on

❌ **Avoid this when:**

- You're on a production server under load and cannot afford 2-10x slowdown — use `perf trace` instead
- You need fleet-wide or continuous tracing — use `bpftrace` or `sysdig`
- The process is in a container without `CAP_SYS_PTRACE` and you can't modify the runtime — use `kubectl debug` or a privileged pod
- You're debugging logic bugs or wrong calculations — strace shows kernel calls, not application logic; use a debugger

⚠️ **Common mistakes:**

- Running `strace -f` on a heavily forked process like Apache in prefork mode — you'll attach to hundreds of workers and drown in output; filter with `-e` first
- Using `-s 32` (the default) when tracing file paths — paths longer than 32 chars get truncated to `"/very/long/path/"...` and you lose the critical suffix; use `-s 256` or `-s 4096`
- Forgetting that `ltrace` has even higher overhead than `strace` — it intercepts every dynamic library call, not just kernel calls; never leave it running on production

## Veteran Perspective

The first thing you learn with strace is to run it. The second thing you learn is to filter it. A raw `strace myapp` on anything but the most trivial command produces output faster than you can read it, mostly linker initialization, memory mapping, and locale setup you don't care about. Most of strace's power is in knowing what to filter.

The pattern I use in almost every investigation: start with `strace -c` to see the breakdown, identify the dominant syscall category, then re-run with `-e trace=%that_category -tt -s 256 -o /tmp/trace.txt` to get a readable file. Never scroll raw strace output in a terminal — pipe it to a file and `grep` it.

The second lesson that takes longer to learn: when you attach to a live process with `-p`, you are not an invisible observer. You are physically stopping that process on every kernel call. I've watched a junior engineer attach strace to a PostgreSQL backend process during a slow query investigation, and within thirty seconds the client application timed out, the connection was dropped, and the backend process exited. The strace session ended with `+++ exited with 0 +++`. Now the slow query was gone and the trace was gone with it. Plan your escape before you attach: know how you'll detach (`Ctrl+C`), know what will happen to the traced process when you do (it continues normally), and know whether you can afford the overhead.

On the deeper principle: system calls are the only true contract between a program and the operating system. Everything else — library versions, language runtimes, application frameworks — is an implementation detail that sits on top of this interface. When you read strace output fluently, you can debug any program on any Linux system regardless of what language it's written in, what libraries it uses, or whether you have the source. That's a skill that doesn't get stale and doesn't depend on framework knowledge. It's worth developing.

One pattern that comes up constantly in interviews: interviewers ask "how would you debug a process that's running but not doing anything useful?" The answer they want is: attach strace, look for the blocked syscall, and read what it's waiting on. A process blocked on `futex()` is waiting for a lock. A process blocked on `read()` with a socket fd is waiting for data. A process blocked on `select()` or `epoll_wait()` with a timeout is doing scheduled sleeping. These four patterns cover 90% of "my process is hung" questions.

The one thing strace cannot tell you is *why* a syscall is being made — the application logic that led to it. When strace shows you 10,000 `stat()` calls on startup, it tells you there's a problem. Finding which part of the application is responsible for those calls requires either `ltrace` (if it's coming through a library), `-k` stack traces (if your strace has libunwind), or a profiler. strace is the detector; it is not always the complete diagnosis.

## Try It Yourself

1. Run `strace ls /etc 2>&1 | tail -20` and find the `write()` call that prints the directory listing. What is the file descriptor number? What does that number mean?

2. Run `strace -c find /usr -name "*.conf" -type f 2>&1 | tail -20`. Which syscall dominates the time? Does the answer change your intuition about how `find` works?

3. Run `strace -e trace=%file -s 256 -o /tmp/ps-trace.txt ps aux`, then run `grep "ENOENT" /tmp/ps-trace.txt | head -10`. Every `ENOENT` is a file that was looked for and not found. Are there any surprising paths being probed?

4. Find the PID of your shell (`echo $$`), then run `strace -p $$ -e trace=%process -f` in a second terminal, then in the first terminal run `ls`. You should see the `clone()` and `execve()` calls that your shell makes to run `ls`. Detach with `Ctrl+C` and confirm the shell is still alive.

5. **Challenge:** Write a small shell script that creates 1,000 empty files in `/tmp/test-strace/`, then use `strace -c` to profile the script. Compare the syscall profile of the script using a loop vs. using `touch /tmp/test-strace/file{1..1000}`. Where does the time go in each version? Use `strace -e trace=%process -f` to see how many `clone()`/`fork()` calls each approach generates. (Requires creating the directory first; safe to run in a VM or your home directory.)

## Summary

- `strace` intercepts system calls using `ptrace()` — it physically stops the traced process on every syscall entry and exit, which is why overhead is 2-100x
- The output format is `syscall_name(arg1, arg2, ...) = return_value`; a negative return with `ENOENT`, `EACCES`, or similar is the direct cause of most "why won't this open?" bugs
- Use `-c` first (syscall count summary) to identify the dominant category, then narrow with `-e trace=%file`, `%network`, or `%process` — never list individual syscalls by name
- Use `-o file` to capture strace output separately from the traced program's output; use `-s 256` to avoid truncated path arguments; use `-f` to trace all threads of a multithreaded process
- For production tracing where overhead matters, use `perf trace` instead — it buffers in-kernel and adds roughly 10-15% overhead vs strace's 2-100x
