# [12.6] — lsof: List Open Files

> On a Linux system, everything is a file — and `lsof` is the tool that shows you exactly which files every process has open right now, including sockets, pipes, devices, and the deleted files that are quietly consuming disk space nobody can account for.

## The Problem This Solves

You've deleted a large log file to free up disk space. `du` says the directory is empty. `df` still shows the filesystem as 95% full. The space didn't come back. Where did it go?

Or: a deployment fails because port 8080 is already in use. `ps aux | grep java` shows nothing. `netstat` has been deprecated and removed from this server. The port is occupied by something, but you have no idea what.

Or: you're about to unmount `/mnt/backup` and the kernel refuses with "device is busy." Something has a file open in there. You have no idea which process, which user, or which file.

All three of these are the same problem: you need to know what processes have open right now, and the standard filesystem and process tools don't give you a cross-reference between the two. `ps` shows processes but not their open files. `ls` shows the filesystem but not which processes are using it. There's no visible connection between the two.

`lsof` exists to make that connection visible. It reads the kernel's internal tables — the file descriptor table for every process — and produces a unified view of what every process on the system currently has open. Every file, every socket, every pipe, every device node.

---

## Mental Model

Think of the kernel as a hotel manager who issues keycards. Every time a process opens a file, the kernel hands it a numbered keycard — a file descriptor. The kernel keeps a master ledger: keycard number 7 issued to room 2341 (PID 2341), pointing to suite 4417 (inode 4417) on the 3rd floor (device 8,1). `lsof` is a printout of that master ledger — every keycard currently outstanding, who has it, and what it opens.

The critical insight from this analogy: a guest can check out of the hotel (a file can be deleted from the directory), but as long as they're still holding a keycard, their room isn't freed and reassigned (the inode and its data blocks remain occupied). The disk space is not returned until the last keycard is handed back — until the last file descriptor is closed.

## Theory

`lsof` stands for "list open files," but on Linux, "file" means far more than the regular files in a directory. Because of the [everything-is-a-file](../phase-2-filesystem/2.2-everything-is-a-file.md) design of Unix, a process's file descriptor table can hold references to:

- Regular files on disk
- Directories
- Symbolic links
- Character and block device nodes (`/dev/sda`, `/dev/null`, `/dev/tty`)
- Named pipes (FIFOs)
- Anonymous pipes (the `|` in a shell pipeline)
- Unix domain sockets (IPC between processes on the same machine)
- IPv4 and IPv6 TCP and UDP sockets (network connections)
- Netlink sockets (kernel-userspace communication)
- `inotify` instances
- `epoll` instances
- `timerfd` and `eventfd` objects
- Memory-mapped files (shown as type `MEM`)

Every one of these is represented as a file descriptor in the kernel's per-process file descriptor table, and `lsof` can show all of them.

The key invariant: **a file's data blocks on disk are not freed until its link count reaches zero AND all open file descriptors pointing to it are closed.** Deleting a file with `rm` decrements the link count. If a process still holds an open descriptor, the link count is zero but the data persists. `lsof` is the tool that reveals this state because standard filesystem tools — `ls`, `find`, `du` — navigate the directory tree. A file with a zero link count has no directory entry. It's invisible to those tools. It's only visible through the process's file descriptor table.

`lsof` ships as part of the `lsof` package (maintained at [lsof-org/lsof](https://github.com/lsof-org/lsof)), currently at version 4.99.5 as of mid-2025. It runs in userspace and reads kernel tables via `/proc` — specifically `/proc/<pid>/fd/`, `/proc/<pid>/fdinfo/`, `/proc/<pid>/maps`, and `/proc/net/` for socket details.

## How It Works (Mechanics)

When you run `lsof`, it does roughly the following:

```text
# Conceptual flow — not a real command
1. Open /proc and iterate over all numeric directories (one per PID)
2. For each PID:
   a. Read /proc/<pid>/status → get command name, UID
   b. Read /proc/<pid>/fd/ → list all open file descriptors (symlinks to their targets)
   c. Readlink each fd symlink → get the file path or socket/pipe annotation
   d. Read /proc/<pid>/fdinfo/<fd> → get flags, position
   e. Cross-reference socket inodes against /proc/net/tcp, /proc/net/tcp6,
      /proc/net/udp, /proc/net/unix for socket state and address info
   f. Read /proc/<pid>/maps → identify memory-mapped files
3. Format and output the combined result
```

This has important implications for performance. Reading `/proc` for every process on a busy system means hundreds or thousands of directory reads, symlink resolutions, and file reads. On a system with 500 running processes, `lsof` with no filters touches thousands of files in `/proc`. With network connections, it additionally parses `/proc/net/tcp` which can be large. Without the `-n` flag (skip DNS reverse lookups) and `-P` flag (skip port-to-service name lookups), `lsof` makes network calls for every socket it finds. On a server with 200 established connections, that's 200 DNS lookups. That's why unfiltered `lsof` can take 10–30 seconds and why `-nP` is non-negotiable in scripts.

The output columns for a regular file look like this:

```text
COMMAND  PID   USER  FD   TYPE DEVICE SIZE/OFF      NODE NAME
nginx    1234  root  4r   REG  8,1    204800        4417 /var/log/nginx/access.log
```

- **COMMAND** — first 9 characters of the process name
- **PID** — process ID
- **USER** — username of the process owner
- **FD** — file descriptor number, followed by the access mode: `r` (read), `w` (write), `u` (read+write), `mem` (memory-mapped), `cwd` (current working directory), `txt` (program text / executable), `DEL` (deleted mapping)
- **TYPE** — file type: `REG` (regular file), `DIR`, `FIFO`, `SOCK`, `CHR`, `BLK`, `IPv4`, `IPv6`, `unix`
- **DEVICE** — major,minor device numbers
- **SIZE/OFF** — file size in bytes, or current file offset for character devices
- **NODE** — inode number
- **NAME** — file path, or for sockets: the connection info

For network sockets, the NAME column shows the connection in the format `local-address:port->remote-address:port (STATE)`:

```text
COMMAND  PID   USER  FD  TYPE DEVICE SIZE/OFF NODE NAME
nginx    1234  root  6u  IPv4 28741  0t0      TCP  *:80 (LISTEN)
nginx    1234  root  7u  IPv4 29102  0t0      TCP  10.0.0.5:80->10.0.0.1:52341 (ESTABLISHED)
```

## Basic Example

```bash
# List all open files for the current shell session
lsof -p $$
```

```text
COMMAND  PID USER   FD   TYPE DEVICE SIZE/OFF   NODE NAME
bash    4891 user  cwd    DIR    8,1     4096 131073 /home/user
bash    4891 user  rtd    DIR    8,1     4096      2 /
bash    4891 user  txt    REG    8,1  1183448 524847 /usr/bin/bash
bash    4891 user  mem    REG    8,1  2220400 527319 /usr/lib/x86_64-linux-gnu/libc.so.6
bash    4891 user    0u   CHR  136,0      0t0      3 /dev/pts/0
bash    4891 user    1u   CHR  136,0      0t0      3 /dev/pts/0
bash    4891 user    2u   CHR  136,0      0t0      3 /dev/pts/0
bash    4891 user  255u   CHR  136,0      0t0      3 /dev/pts/0
```

Look at the FD column. `cwd` is the current working directory — not a file descriptor number, but a special label for the process's working directory reference. `rtd` is the root directory (what `chroot` changes). `txt` is the program binary itself. `mem` entries are shared libraries loaded into memory. Then you see the actual numbered file descriptors: 0, 1, and 2 are stdin, stdout, and stderr — all pointing to the terminal (`/dev/pts/0`). Descriptor 255 is bash's internal copy of stdin kept open for script reading.

Try this yourself. Run `lsof -p $$` in your terminal. Then open a file in another terminal on the same system and run `lsof -p <that-pid>` — you'll see the file appear as an `r` or `w` descriptor.

## Real-World Example

Here are four production scenarios that cover most of what `lsof` gets used for on real servers.

### Scenario 1: Disk full despite empty directory

```bash
# Step 1: Confirm the disk is full
df -h /var/log
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   50G     0 100% /var/log
```

```bash
# Step 2: Find deleted-but-open files consuming the space
# +L1 means: show open files with a link count below 1 (i.e., deleted from directory)
# -nP: skip DNS lookups and port name resolution — much faster
lsof +L1 -nP
```

```text
COMMAND    PID USER   FD   TYPE DEVICE  SIZE/OFF NLINK   NODE NAME
nginx     1234 root   10w   REG    8,1  49283072     0 131081 /var/log/nginx/access.log (deleted)
nginx     1234 root   11w   REG    8,1   1073741     0 131082 /var/log/nginx/error.log (deleted)
```

```bash
# Step 3: Recover the content if needed (before restarting the process)
# The file still exists at the file descriptor path — the kernel keeps it alive
cat /proc/1234/fd/10 > /tmp/recovered-access.log

# Step 4: Free the space without restarting nginx
# Truncating through the fd — nginx keeps writing; the old data is gone, the space is freed
truncate -s 0 /proc/1234/fd/10
```

The reason `df` showed the disk full while the directory appeared empty: `rm` removed the directory entry (decremented the link count to zero) but nginx still had the file open for writing. The kernel keeps the inode and data blocks allocated as long as any process holds an open descriptor. `lsof +L1` finds exactly this situation — it filters for files with a link count below 1 but still referenced by a file descriptor.

The recovery step via `/proc/PID/fd/` works because every open file descriptor is exposed as a symlink in that directory. Even with no directory entry on the filesystem, you can read or write the file directly through `/proc`.

### Scenario 2: Port 8080 is in use — find what's using it

```bash
# -i :8080 means: show open files that are network sockets on port 8080
# -nP: skip name resolution, so output appears immediately
lsof -i :8080 -nP
```

```text
COMMAND   PID     USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
java     2847     app   52u  IPv6  48291      0t0  TCP  *:8080 (LISTEN)
```

```bash
# If you only want the PID (useful in scripts):
lsof -t -i :8080
```

```text
2847
```

The `-t` flag produces terse output — PID only, no header. This feeds cleanly into `kill` or `kill -9` in automation.

### Scenario 3: What does a specific process have open?

```bash
# Show everything PID 2847 has open, without name resolution
lsof -p 2847 -nP
```

```text
COMMAND  PID USER  FD   TYPE DEVICE SIZE/OFF    NODE NAME
java    2847  app  cwd   DIR    8,1     4096  524288 /opt/app
java    2847  app  txt   REG    8,1 12483521  393217 /usr/bin/java
java    2847  app    0r  CHR    1,3      0t0       5 /dev/null
java    2847  app    1w   REG    8,1        0  657922 /opt/app/logs/app.log
java    2847  app    2w   REG    8,1        0  657922 /opt/app/logs/app.log
java    2847  app   52u IPv6  48291      0t0    TCP   *:8080 (LISTEN)
java    2847  app   53u IPv6  48309      0t0    TCP   10.0.0.5:8080->10.0.0.1:43201 (ESTABLISHED)
java    2847  app   54u IPv6  48310      0t0    TCP   10.0.0.5:8080->10.0.0.2:51088 (ESTABLISHED)
```

This output tells a story. The process is running from `/opt/app`. Its stdin is `/dev/null` — standard for a daemonized process. Both stdout and stderr (FDs 1 and 2) point to the same log file (same inode 657922). It's listening on port 8080 on all interfaces and has two established connections. This is what a normal production JVM looks like.

### Scenario 4: Block an unmount — find what's holding a path open

```bash
# +D scans all open files in the directory tree recursively
# Without -nP this is already slow; -nP at least skips name resolution
lsof +D /mnt/backup -nP
```

```text
COMMAND  PID USER   FD   TYPE DEVICE SIZE/OFF    NODE NAME
rsync   3901 root  cwd    DIR    8,2     4096       2 /mnt/backup
rsync   3901 root    4r   REG    8,2 10485760  131073 /mnt/backup/db-2025-06-14.tar.gz
```

Now you know exactly which process and which file to deal with before retrying the unmount.

## ⚠️ Gotchas

### Running Without Root Hides Other Users' Processes

You run `lsof -i :8080` and get no output. The port shows as in use by `ss -tlnp`. You assume nothing is there, try to bind your application to port 8080, and it fails.

```bash
# As a regular user — only shows your own processes
lsof -i :8080 -nP
```

```text
(no output)
```

```bash
# As root — shows everything
sudo lsof -i :8080 -nP
```

```text
COMMAND   PID   USER   FD   TYPE DEVICE SIZE/OFF NODE NAME
node     8823   app   18u  IPv6  71023      0t0  TCP  *:8080 (LISTEN)
```

`lsof` shows only the processes the current user can read from `/proc`. Each `/proc/<pid>/fd/` directory is owned by the process's user and not world-readable. Without root (or `CAP_DAC_READ_SEARCH`), you see only your own processes. The output is not empty because nothing is there — it's empty because you don't have permission to see it. Always run `lsof` with `sudo` when diagnosing production issues, or you'll chase ghosts.

### Running `lsof` Without `-nP` in Scripts Is a Time Bomb

On a server with many network connections, running `lsof -i` without `-nP` will do a reverse DNS lookup for every remote IP address and a `/etc/services` lookup for every port number. On a server with 300 established connections, this takes 30–60 seconds or more if DNS is slow. Worse, if it's inside a loop or a monitoring script that runs every minute, it creates a thundering herd of DNS queries.

```bash
# Dangerously slow — never use this in a script
lsof -i | grep LISTEN
```

```bash
# Correct — -n skips DNS, -P skips port-to-name resolution
lsof -i -nP | grep LISTEN
```

The rule: `-nP` is always correct. There is no scenario where you need `lsof` to resolve names in a script. If you want human-readable names, pipe through a separate lookup step where you control the timeout.

### `lsof +D` on a Busy Filesystem Is Extremely Slow

`+D /some/path` works by reading the entire open file list for every process on the system and filtering for anything under that path. There is no shortcut. On a system with 1000 processes where many have hundreds of open file descriptors under the target directory, this scan can take minutes. There is no faster alternative — `lsof +D` is the only way to get this information from userspace.

```bash
# This will block for a long time on a busy NFS mount or container overlay filesystem
lsof +D /var/lib/docker/overlay2
```

If you're in an emergency, consider whether `fuser` (from the `psmisc` package) gives you enough:

```bash
# fuser is faster — it shows PIDs using a specific mount point
fuser -m /mnt/backup
```

`fuser -m` is faster than `lsof +D` but shows only PIDs, not the specific files. Use `fuser` when you need speed and only care about which processes to kill; use `lsof +D` when you need to know exactly which files are open.

### The `(deleted)` Label in `grep deleted` Is Not Reliable Alone

People commonly run `lsof | grep deleted` to find deleted-but-open files. This works, but the `(deleted)` annotation only appears when `lsof` can resolve the path. For anonymous inodes or files on certain virtual filesystems, the annotation may not appear even though the file is no longer linked.

```bash
# This works for most cases but may miss some
lsof | grep deleted
```

```bash
# This is more reliable — +L1 filters by link count, not by annotation
# It catches all files with link count < 1, regardless of how they're labeled
lsof +L1 -nP
```

The NLINK column shows the actual link count. A value of 0 means the file has been unlinked from all directories. Use `+L1` when the goal is to find disk space held by deleted files — it's the right tool for that specific question.

### `lsof` Output Shows Stale Data Between the Read and Your Action

`lsof` produces a snapshot. The moment it finishes reading `/proc`, processes can open and close files. If you `grep` the output for a PID and then act on it, that process may have exited or the file may have been closed in the milliseconds between. This is especially relevant when scripting `lsof -t -i :8080 | xargs kill` — the PID returned may belong to a different process by the time `kill` runs.

```bash
# Slightly safer: check lsof, store PID, verify the process exists before killing
PORT_PID=$(lsof -t -i :8080 -nP 2>/dev/null)
if [ -n "$PORT_PID" ] && kill -0 "$PORT_PID" 2>/dev/null; then
    kill "$PORT_PID"
fi
```

`kill -0` checks whether the process exists without sending a signal. It's a safety check, not a guarantee — but it catches the most common race.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install lsof` — usually pre-installed; version in Ubuntu 24.04 LTS is 4.95.0 |
| RHEL / Fedora / CentOS / Rocky | `dnf install lsof` — pre-installed on most RHEL/Rocky installs |
| Arch | `pacman -S lsof` |
| Alpine (containers) | `apk add lsof` — not installed by default; minimal Alpine images often omit it entirely |

On minimal container images (Alpine, distroless, scratch-based), `lsof` is frequently absent. In those environments, reading `/proc/<pid>/fd/` directly and cross-referencing `/proc/net/tcp` is the fallback. It's verbose but it works without any additional packages.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `lsof -i` | Find which process owns a port; see all network connections with process context | Needs root for full visibility; slower without `-nP` |
| `ss -tlnp` | Quick snapshot of listening sockets; faster than `lsof` for network-only questions | No file system context; doesn't show deleted-file state |
| `fuser` | Quickly find PIDs using a file or mount point; faster than `lsof +D` | Less detail than `lsof`; output is PID-only |
| `netstat -tlnp` | Legacy — avoid on modern systems | Deprecated; absent from many modern distros; use `ss` instead |
| `/proc/<pid>/fd/` | Reading open files in a container or minimal environment without `lsof` | Requires shell navigation; no filtering; readable by the process owner only |

`ss` and `lsof` are not competitors for network questions — they're complementary. `ss` is faster and more featureful for pure socket inspection. `lsof -i` is the right choice when you need to correlate a socket with a specific process, user, or file descriptor number. The most common pattern on a production server is to reach for `ss -tlnp` first (it's instant), then follow up with `lsof -i :PORT -nP` when you need to know exactly which process and why.

## Trade-offs

**Where this shines:**

- Cross-referencing open resources across the entire system in a single command — no tool gives a more complete picture of what a process is doing with files and sockets simultaneously
- Finding deleted-but-open files — this is the only practical way to detect phantom disk consumption without rebooting
- Diagnosing "device busy" errors before unmounting — `+D` gives you the specific file and process, not just a yes/no
- Tying a listening port to a binary and a user when you have no other context

**Where this costs you:**

- Speed — unfiltered `lsof` on a busy system is slow; always filter by PID, user, or port before running blind
- Completeness without root — without `CAP_DAC_READ_SEARCH`, output is incomplete and silently so; there's no warning that you're missing other users' processes
- Point-in-time snapshot — the output is stale the moment it's printed; for continuously changing state (high-traffic network server), it's a photograph, not a live view
- `+D` on large directory trees can be extremely slow with no progress indicator; there's no way to know if it will finish in 5 seconds or 5 minutes

**Performance note:** `lsof` without `-nP` makes one DNS reverse-lookup per unique remote IP address found. On a server with 100 distinct remote clients and a slow DNS resolver, this adds 30–90 seconds to runtime. Use `-nP` in all automated contexts and most manual ones.

## When to Use / When to Avoid

✅ **Use this when:**

- You need to find which process is using a port and `ss -tlnp` doesn't give enough detail
- Disk usage reported by `df` exceeds what `du` can account for — find deleted-but-open files with `lsof +L1`
- A `umount` fails with "device or resource busy" — use `lsof +D /mount/point` to find the culprit
- You need to recover content from a file that was deleted while still open — read it via `/proc/PID/fd/N`
- You want to audit what files a specific process is touching — `lsof -p PID`
- You want to see all resources open by a specific user on a shared system — `lsof -u username`

❌ **Avoid this when:**

- You only need a network socket overview — `ss -tlnp` is faster and cleaner
- You're inside a tight monitoring loop — use more targeted kernel interfaces or `ss` for network state
- You're in a container with a minimal image — `lsof` may not be installed; read `/proc/<pid>/fd/` directly or install it first
- You need continuous file-access monitoring — use `inotifywait` (from `inotify-tools`) instead; `lsof` produces a snapshot, not a stream

⚠️ **Common mistakes:**

- Forgetting that `lsof` without `sudo` on a multi-user system gives an incomplete picture — missing entries are silent, not flagged
- Piping the full unfiltered `lsof` output through `grep` — always filter at the `lsof` level with `-p`, `-u`, `-i`, or `+D` to avoid the performance cost of reading all of `/proc` first
- Using `lsof | grep deleted` instead of `lsof +L1` — the annotation-based grep can miss files on some filesystems; `+L1` filters by actual link count

## Veteran Perspective

The deleted-file disk leak is the one I've seen most often catch junior sysadmins off guard. They `rm` a 30GB log file, `ls -la` shows nothing in the directory, but `df` still shows 95% full. They assume `df` is caching stale data and wait. Or they restart the service, which finally closes the file descriptor and frees the space — but they don't understand why the restart fixed it, so next time they make the same mistake. The moment you understand that `rm` only removes a directory entry and that the kernel doesn't free the data blocks until all open descriptors are closed, `lsof +L1` becomes a reflex.

The second pattern: "nothing is on port 8080" when `lsof -i :8080` returns empty output. Nine times out of ten this is a permissions issue. The person runs `lsof -i :8080` as their own user, sees nothing, and concludes the port is free. They try to start their application and it immediately fails to bind. Running `sudo lsof -i :8080 -nP` would have shown the answer in one second. The rule is simple: if you're investigating another user's processes or system-level services, you need root.

The `+D` trap comes up during unmount operations. Someone runs `lsof +D /mnt/data` against a busy NFS mount and then walks away when it doesn't return immediately. Five minutes later they're back wondering if it hung. It didn't hang — it's just doing thousands of file reads against `/proc`. The best practice when using `+D` is to combine it with `-nP` to eliminate the network overhead, and to filter with `-u username` or `-p pid` when you have any hint about who might be holding the mount.

The deeper principle behind `lsof` is that the Unix model of "everything is a file" creates a unified API for resource tracking. The same kernel mechanism that opens `/etc/passwd` also opens a TCP socket, an anonymous pipe, or an `inotify` instance. They all live in the same per-process file descriptor table. `lsof` is the tool that reads that unified table. Understanding that design — not just memorizing the flags — is what lets you reason about what `lsof` can and can't tell you, and why certain things appear in its output that look surprising until you understand what a file descriptor really is.

In technical interviews for SRE and sysadmin roles, "the disk is full but I can't find the files" scenario comes up regularly. The expected answer sequence is: `df -h` to confirm, `du -sh /*` to narrow, and then `lsof +L1` when `du` doesn't explain the discrepancy. If you can walk through that sequence and explain why the deleted-but-open file is invisible to `du`, you've demonstrated that you understand the inode lifecycle — which is one of the most fundamental things about how Linux filesystems actually work.

## Try It Yourself

1. Run `lsof -p $$` in your terminal. Identify which file descriptors correspond to stdin, stdout, and stderr. Notice that all three point to the same device — your terminal.

2. Open a file for reading in Python: `python3 -c "f = open('/etc/hostname'); input('press enter')"` — while it waits for input, run `lsof -p $(pgrep python3) -nP` in another terminal. Find the `/etc/hostname` entry in the output.

3. Start a simple HTTP server: `python3 -m http.server 9999 &` — then run `lsof -i :9999 -nP` to confirm it's listening. Kill the server with `kill %1` and run `lsof -i :9999 -nP` again to confirm the port is released.

4. Create a file, open it in Python without closing it, then delete it from the shell: `python3 -c "f = open('/tmp/test_lsof.txt', 'w'); input('press enter')"` — in another terminal run `rm /tmp/test_lsof.txt`, then run `lsof +L1 -nP` to find the deleted-but-open file. Notice the NLINK column shows 0.

5. **Challenge (requires sudo in a VM):** On a system with a running web server (nginx or apache), run `sudo lsof -p $(pgrep -f nginx | head -1) -nP` and identify: (a) which log files are open for writing, (b) which TCP sockets are in LISTEN state, (c) whether any files are `mem`-mapped (shared libraries). Then delete one of the log files (`sudo rm /var/log/nginx/access.log`) and re-run `sudo lsof +L1 -nP` to confirm it appears as a deleted-but-open file. Finally, truncate it through the file descriptor (`sudo truncate -s 0 /proc/<pid>/fd/<fd_number>`) and verify the space is freed with `df -h`.

## Summary

- `lsof` reads the kernel's per-process file descriptor tables and presents a unified view of every open file, socket, pipe, and device across the entire system — filtered however you need by process, user, path, or port.
- A file deleted with `rm` is not freed from disk until all open file descriptors pointing to it are closed; `lsof +L1` finds these invisible files by filtering for link count below 1 — this is the standard diagnostic for "disk full but directory empty."
- Always run `lsof` with `sudo` when investigating system services or other users' processes — without it, output is silently incomplete.
- Always pass `-nP` in scripts and automation to skip DNS and port-name lookups; without it, `lsof` can take 30–90 seconds on servers with many network connections.
- Content from a deleted-but-still-open file is recoverable via `/proc/<pid>/fd/<fd_number>` for as long as the process holds the descriptor open.
