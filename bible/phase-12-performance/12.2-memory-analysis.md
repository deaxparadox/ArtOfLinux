## The Problem This Solves

You're on a production server. Response times are climbing. A process died 20 minutes ago with no error in the application logs. You run `free -h` and see that "free" memory is 12 MB on a 32 GB box.

You panic.

You're wrong to. That 12 MB number is almost always misleading — Linux deliberately fills free RAM with cache to speed up disk reads. The system may be completely healthy. Or it may be 30 seconds from an OOM kill. The two situations look identical at a glance, and reacting to the wrong one causes its own damage.

Before Linux gave us proper memory accounting tools, debugging memory pressure was guesswork. Administrators killed processes that weren't the problem, disabled caches that were keeping the system fast, and rebooted boxes that didn't need rebooting. The tools in this topic exist to replace that guesswork with facts.

## Mental Model

Think of your Linux system's RAM like a restaurant kitchen with a prep area.

The stove and cutting boards are your actual application memory — work that's in progress and can't be moved. The prep area holds chopped vegetables, sauces that are ready to go, and ingredients staged for the next order. That prep work speeds things up enormously. But if a big order comes in and you need the counter space, you sweep the staged ingredients aside without ceremony — they can be re-prepped from the walk-in if needed.

Linux does exactly this. "Free" memory is counter space that's sitting completely empty — valuable, but wasteful. The kernel fills it with cached disk reads (the staged prep work). That cache is instantly evictable at no cost if a real application needs the space. So "free memory = 0" doesn't mean you're in trouble. It means Linux is doing its job. What you actually care about is whether there's enough evictable cache and reclaimable memory to satisfy the next allocation request — and that's what `MemAvailable` tells you.

## Theory

### The Memory Hierarchy

Linux manages memory in pages, typically 4 KB each. Every page of RAM is in one of a small number of states: it's free, it's holding application data (anonymous memory), it's holding file-backed content (page cache), or it's been swapped out to disk.

Anonymous memory is memory allocated by applications — heap, stack, memory-mapped anonymous regions. It has no backing file on disk. The only way to reclaim it is to write it to swap.

File-backed memory is the page cache: every time you read a file, the kernel keeps those bytes in RAM. If you read the same file again, it comes from cache and the disk never spins. This cache is the most important performance optimization in the kernel, and it's free to evict — the kernel can always re-read from disk.

### MemFree vs MemAvailable

`MemFree` is the count of completely unused pages. This number looks alarming on a healthy, loaded system because Linux makes it a point to use free RAM for cache.

`MemAvailable` (introduced in kernel 3.14) is what you actually need. It estimates how much memory could be freed without swapping — by combining `MemFree`, reclaimable page cache, reclaimable slab memory, and adjusting downward by kernel low-watermark reserves. This is the single number that tells you whether new allocations can be satisfied without pain.

The rule: watch `MemAvailable`, not `MemFree`.

### Swap

Swap is disk space that the kernel can use as overflow for anonymous memory pages that haven't been accessed recently. When physical RAM fills up, the kernel writes cold pages to swap and frees the RAM for something else.

Swap comes as either a dedicated partition (`/dev/sda2`, typed as `82` in fdisk) or a swap file on an existing filesystem. Both work identically at the kernel level. Modern systems frequently use a swap file because it's easier to resize.

Swap gets a bad reputation. That reputation is mostly earned on underpowered systems where the kernel is constantly swapping under pressure, which causes a vicious cycle: reading from swap takes hundreds of milliseconds, during which the process is blocked, memory pressure stays high, and more swapping happens. But on a well-resourced system, a small swap partition is valuable insurance — it lets the kernel quietly page out anonymous memory that hasn't been touched in hours, without disrupting anything, and prevents an OOM kill when a one-time spike overflows RAM.

### vm.swappiness

`vm.swappiness` is a kernel tunable (0–200 on kernel 5.8+, 0–100 on older kernels) that controls the relative cost the kernel assigns to swapping anonymous pages versus evicting file cache pages. It is not a percentage. It is not a threshold. It does not tell the kernel when to start swapping.

What it controls is the balance between two reclaim strategies when the kernel needs to free memory:
- Evict file cache pages (cheap — can always re-read from disk)
- Swap out anonymous pages (expensive — requires disk I/O, and the memory might be needed again)

At `swappiness=60` (the default), the kernel moderately prefers to evict cache over swapping. At `swappiness=0`, it will evict essentially all reclaimable cache before touching anonymous pages — but it will still swap if it has to. At `swappiness=200`, it treats swap as aggressively as cache eviction.

The implication: `vm.swappiness=0` does not disable swap. It just makes the kernel avoid it as long as any other option exists.

### The OOM Killer

When the kernel cannot satisfy an allocation request by any means — no free pages, no reclaimable cache, no swap space — it invokes the OOM (Out Of Memory) killer. This is the last resort before a kernel panic.

The OOM killer selects a victim using a score (`oom_score`, visible in `/proc/<pid>/oom_score`). The score is computed from:
- How much memory the process is using (the dominant factor)
- How long the process has been running (longer-running processes get some protection)
- Whether the process is root-owned (a small discount)

Every process also has an `oom_score_adj` (in `/proc/<pid>/oom_score_adj`), ranging from -1000 to +1000. This is an operator-controllable offset added to the raw score. Setting it to -1000 makes the process immune to OOM kills. Setting it to +1000 makes it the first target.

The OOM killer does not kill the process that caused the OOM. It kills the process with the highest score — which is usually the process using the most memory at that moment. That process could be completely unrelated to what triggered the pressure.

## How It Works (Mechanics)

### Memory Reclaim Under Pressure

When an allocation cannot be satisfied from free pages, the kernel enters memory reclaim. It works through two lists per memory zone: the active list (recently accessed pages) and the inactive list (cold pages). Pages age from active to inactive over time. The reclaim path scans the inactive list and either:
- Drops clean file-backed pages (instantly free)
- Writes dirty file-backed pages to disk, then frees them
- Writes anonymous pages to swap, then frees them

The `kswapd` kernel thread runs in the background doing this proactively before pressure builds. When allocation pressure is so high that `kswapd` can't keep up, the allocating process itself enters direct reclaim — it stalls mid-allocation, waiting for pages to be freed. This is the first symptom of true memory pressure: processes start slowing down.

### How the OOM Killer Fires

```text
# Conceptual flow — not a real command
process calls malloc() or mmap()
  → kernel tries free pages              → success → done
  → kernel tries kswapd reclaim          → success → done
  → kernel tries direct reclaim          → success → done
  → kernel checks for OOM condition      → yes
    → select victim by oom_score
    → send SIGKILL to victim
    → log event to kernel ring buffer
    → retry allocation
```

The killed process's memory is freed immediately. The kernel logs the full event to the ring buffer (readable via `dmesg` or `journalctl -k`).

### smem and PSS

Most per-process memory tools report RSS (Resident Set Size) — the total pages physically in RAM attributed to a process. But shared libraries are shared. A process that loaded `libc` doesn't own all of `libc`'s pages in RAM; it shares them with every other process that loaded the same library.

PSS (Proportional Set Size) splits shared pages proportionally across the processes that share them. If three processes share a 3 MB library, each gets 1 MB of PSS. This gives a much more honest accounting of how much RAM each process truly consumes, and it makes PSS values addable — the sum of all processes' PSS is approximately total RAM used.

`smem` is the tool that surfaces PSS.

### /proc/meminfo Field Map

The kernel writes its internal memory counters to `/proc/meminfo` at read time (no stale data — it's always current). Key fields:

```text
# Conceptual map of /proc/meminfo fields
MemTotal      = total physical RAM (minus kernel reserved)
MemFree       = completely unused pages
MemAvailable  = estimate of memory available for new allocations without swapping
                (= MemFree + reclaimable cache + reclaimable slab - lowwatermark reserves)
Buffers       = raw block device cache (small on modern systems)
Cached        = file cache (page cache) — this is the "staged prep work"
SwapCached    = pages that were swapped out and then re-read back; still in swap too
Active        = pages in the active LRU list (recently accessed)
Inactive      = pages in the inactive LRU list (candidates for reclaim)
SwapTotal     = total swap space
SwapFree      = unused swap space
Dirty         = pages modified in cache, not yet written to disk
Writeback     = pages currently being written to disk
Slab          = kernel slab allocator memory (kernel data structures)
SReclaimable  = slab memory that can be reclaimed under pressure (e.g., inode/dentry cache)
SUnreclaim    = slab memory that cannot be reclaimed
```

## Basic Example

```bash
# Memory overview — start here every time
free -h
```

```text
               total        used        free      shared  buff/cache   available
Mem:            15Gi        3.2Gi       312Mi       412Mi        12Gi        11Gi
Swap:          2.0Gi          0B        2.0Gi
```

The columns to read:
- `total` — total physical RAM
- `used` — memory in active use by processes
- `free` — completely empty pages (this will always look small)
- `buff/cache` — page cache and buffer cache (evictable on demand)
- `available` — the real number: what's actually available for new applications

In this output, "free" is 312 MB and "available" is 11 GB. The system has 11 GB it can hand out before hitting trouble. The 312 MB "free" column is meaningless for diagnosing pressure.

Now look at the full accounting in `/proc/meminfo`:

```bash
# See every memory counter the kernel tracks
cat /proc/meminfo
```

```text
MemTotal:       16268040 kB
MemFree:          319572 kB
MemAvailable:   11543616 kB
Buffers:          126504 kB
Cached:         11485448 kB
SwapCached:            0 kB
Active:          3842160 kB
Inactive:        9126832 kB
...
SwapTotal:       2097148 kB
SwapFree:        2097148 kB
Dirty:             18624 kB
Slab:             643512 kB
SReclaimable:     489280 kB
SUnreclaim:       154232 kB
```

Try this on your own system. Notice how `Cached` is large (that's the page cache working), `MemFree` is small, and `MemAvailable` is close to `Cached + MemFree`. That relationship holds on any healthy loaded system.

## Real-World Example

This is a real production memory diagnostic workflow — the exact sequence you'd run after getting paged.

### Step 1: Is there actual pressure?

```bash
# Quick health check: available memory and swap usage
free -h

# Watch it live for 5 seconds to see if it's moving
watch -n 1 free -h
```

If `available` is dropping fast and `SwapFree` is dropping with it, you have a real problem. If both are stable, you don't — move on.

### Step 2: Who is using memory?

```bash
# Sort processes by RSS descending — find the top consumers
ps aux --sort=-%mem | head -20
```

```text
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
postgres  1421  2.1 18.3 4524816 2986312 ?     Ss   08:12   4:21 postgres: autovacuum
app       3089  0.4 12.1 2145432 1974480 ?     Sl   09:44   1:02 /usr/bin/node server.js
app       3102  0.3 11.8 2098000 1924560 ?     Sl   09:44   0:58 /usr/bin/node server.js
```

RSS is the crude tool. For accurate accounting, use `smem`:

```bash
# PSS-based report — more honest than RSS
# -k: show in KB; -s pss: sort by PSS; -r: descending
smem -k -s pss -r | head -20
```

```text
  PID User     Command                         Swap      USS      PSS      RSS
 3089 app      /usr/bin/node server.js            0  1890244  1923882  1974480
 3102 app      /usr/bin/node server.js            0  1885120  1918700  1969120
 1421 postgres postgres: autovacuum               0  2801440  2834560  2986312
```

PSS is the number to trust. USS (Unique Set Size) is purely private — not shared with anything. PSS adds back each process's fair share of shared mappings.

### Step 3: Did the OOM killer fire?

```bash
# Search kernel log for OOM events
journalctl -k --since "1 hour ago" | grep -i "oom\|killed process\|out of memory"
```

```text
Jun 15 02:47:13 prod-web-01 kernel: java invoked oom-killer: gfp_mask=0x140dca(GFP_HIGHUSER_MOVABLE|__GFP_COMP|...), order=0, oom_score_adj=0
Jun 15 02:47:13 prod-web-01 kernel: Out of memory: Killed process 8823 (java) total-vm:8192000kB, anon-rss:7340032kB, file-rss:20480kB, shmem-rss:0kB, UID:1001 pgtables:15360kB oom_score_adj:0
```

The log tells you exactly what happened: `java` triggered the OOM condition (it called `gfp_mask`, meaning it was trying to allocate), and it was also the process killed (highest OOM score at that moment). Note: the process that *invokes* the OOM killer and the process that *gets killed* are not always the same.

### Step 4: Inspect a specific process's memory map

```bash
# Show virtual memory regions for a process
# Replace 3089 with the actual PID
pmap -x 3089 | head -30
```

```text
3089:   /usr/bin/node server.js
Address           Kbytes     RSS   Dirty Mode  Mapping
0000556b2f400000   71824   62144     544 r-x-- node
0000556b34830000    1020    1020    1020 r---- node
0000556b34930000     256     256     256 rw--- node
00007f8b60000000  524288  490240  490240 rw---   [ anon ]
00007f8b80000000  262144  245760  245760 rw---   [ anon ]
...
00007f8bf0000000    2048    1792       0 r-x-- libz.so.1.2.11
...
----------------  ------  ------  ------
total kB         2145432 1974480 1892352
```

The `[ anon ]` lines are heap and stack. The named libraries show shared mappings. Large dirty anonymous regions with no backing file are memory the kernel can only reclaim by swapping.

### Step 5: Check and tune OOM protection for a critical process

```bash
# Check current OOM score for a process
cat /proc/$(pgrep -x postgres)/oom_score
```

```text
247
```

```bash
# Check current adjustment
cat /proc/$(pgrep -x postgres)/oom_score_adj
```

```text
0
```

```bash
# Protect postgres from OOM kills — set adjustment to minimum
# This requires root
echo -1000 > /proc/$(pgrep -x postgres)/oom_score_adj

# Verify
cat /proc/$(pgrep -x postgres)/oom_score_adj
```

```text
-1000
```

To make this survive reboots, set `OOMScoreAdjust=-1000` in the service's systemd unit file:

```ini
[Service]
OOMScoreAdjust=-1000
```

### Step 6: Inspect swap and manage it

```bash
# Current swap status
swapon --show
```

```text
NAME      TYPE SIZE USED PRIO
/dev/sda2 partition   2G   0B   -2
```

```bash
# Create a 4 GB swap file if you need more headroom
# WARNING: this creates and formats a new swap file at /swapfile
fallocate -l 4G /swapfile
chmod 600 /swapfile          # swap files must not be world-readable
mkswap /swapfile
swapon /swapfile

# Make permanent — add to /etc/fstab
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

```bash
# Check vm.swappiness
sysctl vm.swappiness
```

```text
vm.swappiness = 60
```

```bash
# Tune it for a database server (prefer cache, avoid swapping application memory)
# Apply temporarily
sysctl -w vm.swappiness=10

# Apply permanently
echo 'vm.swappiness=10' >> /etc/sysctl.d/99-memory.conf
sysctl -p /etc/sysctl.d/99-memory.conf
```

### Step 7: Check huge pages

```bash
# Current THP setting
cat /sys/kernel/mm/transparent_hugepage/enabled
```

```text
always [madvise] never
```

The bracketed value is the active setting. `madvise` means THP is only used by processes that explicitly request it — the safest default for mixed workloads.

```bash
# See if THP is allocating pages
grep -i hugepages /proc/meminfo
```

```text
AnonHugePages:    614400 kB
ShmemHugePages:        0 kB
FileHugePages:         0 kB
HugePages_Total:       0
HugePages_Free:        0
HugePages_Rpriv:       0
HugePages_Surp:        0
Hugepagesize:       2048 kB
```

`AnonHugePages` is memory that THP has backed with 2 MB pages. `HugePages_Total` is static huge pages (explicitly pre-allocated, separate from THP).

A senior engineer reading this output notices two things: the system is using `madvise` for THP (explicit opt-in only), and there are no static huge pages configured (`HugePages_Total: 0`). For a database like PostgreSQL, you'd want to evaluate whether static huge pages would reduce TLB pressure. For most applications, `madvise` or `never` is correct.

## ⚠️ Gotchas

### "Free" Memory Being Near Zero Is Normal

The expectation: if `free -h` shows 20 MB of free memory on a 16 GB box, the system is almost out of RAM and about to crash.

```bash
free -h
```

```text
               total        used        free      shared  buff/cache   available
Mem:            15Gi        3.8Gi        18Mi       312Mi        11Gi        11Gi
```

The `free` column is 18 MB. The `available` column is 11 GB.

The underlying rule: Linux actively avoids leaving RAM completely idle. Every free page is a wasted opportunity to cache a file that might be needed soon. So the kernel fills free memory with page cache. That cache is evicted instantly — at zero I/O cost — the moment a real allocation needs the space.

The number to watch is `available`, not `free`. This confusion has caused more unnecessary panic and unnecessary reboots than almost any other Linux misconception.

### The OOM Killer Kills the Wrong Process

The expectation: the OOM killer kills whichever process caused the out-of-memory condition.

What actually happens: the OOM killer kills the process with the highest `oom_score` at the moment the kill fires. The scoring algorithm weights primarily by current memory consumption. The process that triggered the OOM — the one that tried to allocate the last few bytes — could be a tiny helper process with a low score. It survives. The largest process in memory — your database, your application server — dies.

This has taken down production systems in a very specific pattern: a background job (log rotator, a one-off script) triggers OOM pressure. MySQL gets killed. Nobody touched MySQL. The background job is still running.

```bash
# Check scores for all processes to understand who's at risk
ps aux | awk '{print $2}' | while read pid; do
  score=$(cat /proc/$pid/oom_score 2>/dev/null)
  name=$(cat /proc/$pid/comm 2>/dev/null)
  [ -n "$score" ] && echo "$score $pid $name"
done | sort -rn | head -20
```

Protect critical services before you need to:

```bash
# For a running process (temporary — lost on restart)
echo -1000 > /proc/$(pgrep -x mysqld)/oom_score_adj

# For a systemd service (permanent)
# Add to /etc/systemd/system/mysql.service.d/override.conf:
# [Service]
# OOMScoreAdjust=-1000
```

### vm.swappiness=0 Does Not Disable Swap

The expectation: setting `vm.swappiness=0` prevents the kernel from using swap.

```bash
sysctl -w vm.swappiness=0
# assumption: kernel will never swap now
```

This is wrong. `swappiness=0` tells the kernel to exhaust all reclaimable cache before touching anonymous pages, but if the system runs out of reclaimable cache, it will still swap. The kernel will not OOM kill a process when swap space is available, regardless of swappiness.

To actually prevent swap usage:

```bash
# Disable all active swap devices
swapoff -a

# Verify
swapon --show
# (no output — swap is disabled)
```

> ⚠️ **WARNING:** Disabling swap on a production server under memory pressure can immediately trigger OOM kills. Do this only when you have enough available RAM to absorb any spike.

### THP Causes Latency Spikes for Databases

The expectation: transparent huge pages are an optimization — larger pages mean fewer TLB misses, which means better performance.

For general-purpose workloads, this is broadly true. For databases (PostgreSQL, MySQL, Redis, MongoDB), it causes a known pathology: when `khugepaged` collapses small pages into a huge page, it holds the memory lock for the duration. For databases that manage their own memory in large contiguous regions, this causes periodic multi-millisecond stalls — noticeable latency spikes at seemingly random intervals.

The fix is to set THP to `madvise` (only processes that opt in get huge pages) or `never`:

```bash
# Temporary change
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled

# Permanent — add to /etc/rc.local or a systemd oneshot unit
# echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
```

Every major database vendor (PostgreSQL, MySQL, MongoDB, Redis) documents this issue and recommends disabling THP or setting it to `madvise`. If you're running a database and seeing unexplained latency spikes with no CPU or I/O explanation, check THP first.

### swap partition vs. swap file — the btrfs exception

Creating a swap file works on ext4 and xfs. On btrfs, a swap file has specific requirements: it cannot be on a btrfs filesystem with compression enabled, and it requires special handling (`btrfs filesystem mkswapfile`). Using the standard `mkswap` + `swapon` sequence on a btrfs-compressed filesystem will fail silently or corrupt things.

| Filesystem | Swap file support |
|---|---|
| ext4 | Full support, standard procedure |
| xfs | Full support, standard procedure |
| btrfs | Requires `btrfs filesystem mkswapfile` (kernel 6.1+), no compression |
| tmpfs / overlayfs | Not supported |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `free -h` | First look — always | Output in human-readable units; `-w` flag separates buffers from cache |
| `vmstat -s` | Snapshot of counters including page faults, swap in/out | Good for scripting; less human-readable than `free` |
| `/proc/meminfo` | Full accounting, scripting, monitoring | The authoritative source; `free` reads this |
| `smem` | Per-process PSS — who actually owns what | Needs installation; slower than `ps` but more accurate |
| `pmap -x PID` | Per-mapping view of a single process | Find heap fragmentation, leaked mappings, shared library overhead |
| `ps aux --sort=-%mem` | Quick-and-dirty ranking | Reports RSS (not PSS) — misleads on shared-library-heavy processes |
| `valgrind --tool=massif` | Heap profiling over time for a specific application | Development tool, not a production diagnostic |

For production memory triage, start with `free -h` → `/proc/meminfo` → `smem` → `pmap`. For ongoing monitoring, put `MemAvailable` and `SwapFree` into your metrics pipeline and alert on thresholds, not on `MemFree`.

## Trade-offs

**Where page cache shines:**
- Repeated reads of the same file data hit RAM instead of disk — often 100x faster
- File-backed cache is free to evict: no I/O cost to reclaim it
- The kernel manages it automatically; no application changes needed

**Where it costs you:**
- `MemFree` looks alarming and confuses operators who don't know the model
- Under heavy write workloads, dirty cache pages accumulate until `pdflush`/`writeback` kicks in — this can cause write latency spikes

**Where swap shines:**
- Provides a safety valve against one-off memory spikes without OOM kills
- Allows the kernel to quietly page out anonymous memory that hasn't been touched in hours, freeing RAM for more useful work
- On modern NVMe SSDs, the I/O cost of swapping is dramatically lower than on spinning disks

**Where swap costs you:**
- Swap under real pressure (when the system is genuinely short on RAM) causes thrashing — a positive feedback loop where paging in data to handle a request causes other data to be paged out, which is then immediately needed again
- Any swap I/O adds latency; even low-level swapping adds unpredictable stalls for latency-sensitive applications

**Where THP shines:**
- Reduces TLB misses for processes with large contiguous working sets (scientific computing, in-memory analytics, some JVM workloads)
- Transparent to applications — no code changes needed

**Where THP costs you:**
- `khugepaged` compaction stalls cause latency spikes in latency-sensitive workloads
- Memory overhead: 2 MB pages reserved for a region that only uses 4 KB of it is 511 KB of waste per allocation
- Databases that manage their own memory (PostgreSQL shared_buffers, for example) get no benefit and pay the compaction cost

**Performance note:** On systems running a database with `vm.swappiness=60` and THP `always`, it's common to see unexplained p99 latency increases of 10–50 ms that disappear entirely after setting `vm.swappiness=10` and THP to `madvise`. These are the two highest-leverage memory tuning knobs on database servers.

## When to Use / When to Avoid

✅ **Use `MemAvailable` as your primary memory health signal when:**
- Building monitoring alerts for production systems
- Triaging a system under pressure
- Any time you want to know "is this system about to run out of memory"

✅ **Use swap when:**
- Running on a system that may see occasional memory spikes (web application servers, batch jobs)
- You want a safety net against OOM kills for predictable but rare overflows
- The workload can tolerate occasional sub-second latency (swap on NVMe is fast)

✅ **Set `oom_score_adj = -1000` when:**
- A critical daemon (database, message broker, monitoring agent) must not be killed even under memory pressure
- Your service is worth more than the memory it uses

✅ **Disable THP or set it to `madvise` when:**
- Running any database (PostgreSQL, MySQL, Redis, MongoDB, Elasticsearch)
- You observe unexplained periodic latency spikes with no I/O or CPU explanation

❌ **Avoid disabling swap entirely when:**
- The system is under active memory pressure — doing so can immediately trigger OOM kills; use `swapoff -a` only on well-provisioned systems
- Running without enough RAM headroom to absorb spikes

❌ **Avoid setting `vm.swappiness=0` and calling it "swap disabled" when:**
- You actually want no swap activity — use `swapoff -a` instead

❌ **Avoid trusting `ps aux` RSS for memory accounting when:**
- Your system runs many processes that share libraries (JVMs, Node.js clusters, Python services) — use `smem` for PSS

⚠️ **Common mistakes:**
- Rebooting a server because `free -h` showed almost no "free" memory — check `MemAvailable` first
- Setting `OOMScoreAdjust` on one critical process but forgetting others — audit all critical daemons
- Creating a swap file on btrfs with compression enabled — this silently causes problems

## Veteran Perspective

The junior engineer sees `MemFree: 12 MB` in `free -h` and raises a P1 incident. The senior engineer glances at `MemAvailable: 11 GB` and closes the ticket. This single misunderstanding generates more noise in operations channels than almost anything else in Linux. Burn the available/free distinction into your muscle memory.

The intermediate engineer knows to look at `MemAvailable` but still doesn't understand the OOM killer. They see that a process got OOM-killed and assume the process was misbehaving. The actual pattern is almost always: something unrelated consumed an unusual amount of memory (a one-off report job, a log processor, a deploy script), tipped the balance, and the kernel's scoring algorithm killed the most memory-hungry process — which happened to be the production database. No amount of fixing the database helps. You fix the other thing, or you add swap, or you protect the database with `oom_score_adj`.

The deeper principle: Linux memory management is designed around the assumption that unused RAM is wasted RAM. Everything the kernel does — page cache, slab cache, THP, readahead — follows from this. Understanding this design intent makes every confusing memory metric readable.

In technical interviews, memory analysis is asked like this: "A server is running slowly. `free -h` shows almost no free memory. What do you do?" The wrong answer is "add more RAM" or "kill processes." The right answer is "check MemAvailable in free -h or /proc/meminfo. If MemAvailable is large, this is normal caching behavior, not a problem. If MemAvailable is small and dropping, check dmesg for OOM kills and smem for the top consumers."

A brief war story: a team I worked with had an unexplained nightly performance degradation on a PostgreSQL database. p99 query time would spike from 5 ms to 80 ms between 2 and 4 AM, then recover. CPU was flat. IOPS were flat. Memory usage was flat. After two weeks of adding instrumentation, someone finally checked the THP setting: it was `always`. The nightly `khugepaged` scan — triggered by the OS doing its memory compaction — was holding page locks on PostgreSQL's shared buffer pool for tens of milliseconds at a time. Setting THP to `madvise` stopped the spikes the same night. Two weeks of investigation. One line fix.

## Try It Yourself

1. Run `free -h` on your system. Note the `free` column and the `available` column. Now read `MemAvailable` directly from `/proc/meminfo` and verify they match the `available` column in `free`.

2. Read `/proc/meminfo` and calculate how much memory is being used as page cache: it's `Cached + Buffers`. Compare this to `MemAvailable`. Notice that on a healthy system, a large fraction of `MemAvailable` comes from reclaimable cache.

3. Find the five processes with the highest `oom_score` on your system using the loop from the Real-World Example. Then check `/proc/<pid>/oom_score_adj` for each of them. Are any of your critical daemons at risk?

4. Check the current THP setting on your system (`/sys/kernel/mm/transparent_hugepage/enabled`). Check `AnonHugePages` in `/proc/meminfo`. If THP is `always` and you have a database running, consider what would happen if you changed it to `madvise`.

5. **Challenge (VM recommended):** Create a 1 GB swap file on a test VM using `fallocate`, `chmod`, `mkswap`, and `swapon`. Verify it appears in `swapon --show` and `free -h`. Then write a small script that allocates memory in a loop (using `dd if=/dev/zero of=/dev/null` won't work — use Python: `python3 -c "x = bytearray(500 * 1024 * 1024)"`) and watch `/proc/meminfo` and `vmstat 1` as the system responds. Observe when and whether it starts using swap.

## Summary

- `MemFree` being near zero is normal on a loaded Linux system — the kernel fills free RAM with page cache. The metric that tells you whether you're actually running out of memory is `MemAvailable`.
- `vm.swappiness` controls the balance between evicting file cache and swapping anonymous memory — it is not a threshold and `swappiness=0` does not disable swap; `swapoff -a` does.
- The OOM killer selects its victim by score, not by who triggered the OOM — protect critical processes with `OOMScoreAdjust=-1000` in their systemd unit files before you need it.
- For per-process memory accounting, PSS (from `smem`) is more accurate than RSS (from `ps`) because it splits shared library pages proportionally across consumers.
- Transparent huge pages should be set to `madvise` or `never` on any system running a database — `always` causes periodic `khugepaged` compaction stalls that appear as unexplained latency spikes.
