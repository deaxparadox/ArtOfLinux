# [12.3] — I/O Analysis

> A server that feels slow but passes CPU and memory checks is almost certainly choking on disk I/O — and the number that will tell you that is not `%util`.

## The Problem This Solves

Imagine you're looking at a production database server that has been responding slowly for two hours. CPU sits at 12%. RAM is 70% used with plenty of free cache. The network is quiet. The on-call engineer runs `top` and shrugs — nothing looks wrong. The application team insists the queries are fine. But response times are four seconds for operations that should take 40 milliseconds.

The culprit is usually sitting one layer deeper than `top` can show: the storage subsystem. Processes are blocked waiting for reads or writes to complete. The CPU isn't busy because it is *waiting* — and waiting doesn't register as CPU usage. Without I/O-specific tooling, you're diagnosing a chest pain patient by only checking their blood pressure.

The tools in this topic let you see what `top` cannot: which devices are saturated, how long individual operations take, which processes are responsible, and whether your storage is actually working as fast as it should.

---

## Mental Model

Think of your block device as a bank teller window. Each I/O request is a customer in line. The teller can only process one customer at a time.

`%util` tells you how often the teller was busy — but a teller can be "busy" 100% of the time and still have a short line because they're fast. That's an NVMe SSD doing well. The metric you actually care about is `aqu-sz` — the average queue length. A long queue means customers are stacking up faster than the teller can serve them. That's saturation.

`await` is the total time each customer spent: time waiting in line plus time at the window. High `await` with a short queue means the teller is slow per transaction — typical of a spinning disk doing random reads. High `await` with a long queue means the disk is both slow and overwhelmed.

---

## Theory

Linux I/O is not a direct wire between your application and the physical disk. Writes from user processes first land in the **page cache** — kernel-managed memory that buffers disk content. The kernel periodically flushes dirty pages (modified cache pages not yet written to disk) back to the device through a process called **writeback**. Reads first check the cache before going to disk. This layered architecture means that what you observe at the device level may not directly reflect what your application is doing at any given instant.

**Block layer and queues.** Between the page cache and the physical device sits the Linux block layer. This subsystem manages request queues, reorders I/O for efficiency, and dispatches to device drivers. The component responsible for reordering is the **I/O scheduler**. Different schedulers make different trade-offs between throughput, latency, and fairness.

**Four schedulers ship in modern kernels:**

| Scheduler | Design intent | Best for |
|---|---|---|
| `none` | No reordering — requests go straight to the driver | NVMe SSDs; the device's own internal queue handles ordering better than any software scheduler |
| `mq-deadline` | Multiqueue deadline; prevents request starvation; reads prioritized over writes | HDDs, SATA SSDs, VMs on shared storage |
| `bfq` | Budget Fair Queueing; allocates bandwidth fairly across processes | Desktop systems, Bluetooth/USB storage, interactive workloads where fairness matters |
| `kyber` | Token-based latency targeting; limits in-flight requests to hit a latency target | High-speed NVMe where you want latency control rather than raw throughput |

The kernel selects a default scheduler per device based on whether the device exposes itself as rotational or non-rotational. NVMe devices default to `none`. Rotational devices (HDDs) default to `mq-deadline`. You can override this per device.

**Dirty page writeback.** Two sysctl values govern when the kernel flushes dirty pages to disk:

- `vm.dirty_background_ratio` — the percentage of total RAM that can hold dirty pages before background flush threads (the `kworker` processes) begin writing quietly in the background. The default is 10%. Your application keeps running; writeback happens alongside it.
- `vm.dirty_ratio` — the hard ceiling. When dirty pages hit this percentage of RAM, the kernel forces any process that creates more dirty data to block and wait for writeback before it can continue. Default is 20%. When an application hits this limit, you will see it as a spike in `await` and a sudden drop in write throughput.

These two values create a two-stage response: soft background flushing followed by hard application-blocking flushing. The gap between them is the burst absorption zone.

## How It Works (Mechanics)

**How `iostat` measures `%util`.** The kernel timestamps every moment the device's request queue has at least one in-flight request. `%util` is the fraction of the measurement interval during which the device was not completely idle — the device had at least one request being processed. For a single-queue spinning disk, this correlates well with saturation because the disk can only do one thing at a time.

For NVMe — which has 65,535 hardware queues and can process thousands of concurrent requests — `%util` hits 100% while the device still has room to absorb vastly more load. The device is "busy" in the sense that it was never sitting completely idle, but it is nowhere near saturated.

**`aqu-sz` is the real saturation signal.** `aqu-sz` is the average number of requests outstanding — in the queue waiting plus actively being serviced — averaged over the measurement interval. This is computed from Little's Law. When `aqu-sz` is consistently above 1 on a single-spindle HDD, the disk has more work than it can complete in real time. On an NVMe that can handle thousands of concurrent requests, an `aqu-sz` of 10 is nothing. Context matters.

**How `iotop` works.** `iotop` reads per-process I/O accounting from the kernel via the `taskstats` netlink interface. The kernel tracks bytes read and written per task. `iotop` samples these counters, computes deltas, and displays a sorted live view. This requires `CONFIG_TASK_IO_ACCOUNTING` and `CONFIG_TASK_DELAY_ACCT` to be compiled into the kernel — both are enabled in all major distributions.

**Scheduler path.** When a process calls `write()`, the data lands in page cache (a dirty page). When the kernel decides to flush that page, it submits a `bio` (block I/O) structure to the block layer. The I/O scheduler holds this in a staging queue, potentially merges it with adjacent requests, reorders based on its policy, then dispatches to the multi-queue submission path. The driver places the request into one of the device's hardware queues. The device signals completion via an interrupt; the block layer marks the request done and wakes any waiting processes.

```text
# Conceptual trace — not a real command
write() syscall → page cache (dirty page)
                     ↓ writeback trigger
              bio submitted to block layer
                     ↓
              I/O scheduler (mq-deadline / bfq / none)
                     ↓
              device hardware queue (NVMe: up to 65535)
                     ↓
              physical media
                     ↓
              interrupt → request complete → wake waiters
```

## Basic Example

```bash
# Install sysstat if not present (provides iostat)
# Debian/Ubuntu: apt install sysstat
# RHEL/Fedora:   dnf install sysstat

# -x: extended statistics (the useful columns)
# 1:  refresh every 1 second
# 3:  show 3 iterations then stop
iostat -x 1 3
```

```text
Linux 6.8.0-45-generic (hostname)   06/15/2026   _x86_64_   (8 CPU)

avg-cpu:  %user   %nice %system %iowait  %steal   %idle
           3.20    0.00    1.40    0.80    0.00   94.60

Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  aqu-sz  %util
sda              1.20   18.40     48.00    736.00     4.20     3.80     0.07    5.20
nvme0n1          0.00  420.00      0.00  105000.00    0.12     0.09     0.04   38.00
```

The first iteration after `iostat` starts is a cumulative average since boot — ignore it. The second and third iterations are true per-interval samples. Look at those.

For `sda` (a spinning disk): `r_await` 4.2ms and `w_await` 3.8ms are healthy for sequential I/O on an HDD. `aqu-sz` 0.07 — essentially no queuing. The disk is barely loaded.

For `nvme0n1`: writing 105 MB/s with sub-millisecond `await`. `%util` at 38% looks alarming if you're used to HDDs, but `aqu-sz` 0.04 tells you the queue is nearly empty. This NVMe has room for ten times this load.

Run this yourself on any Linux system. If `%iowait` in the CPU summary is above 5% and `await` on any device is above 20ms, you've found a bottleneck worth investigating.

## Real-World Example

A database server shows P99 query latency spiking from 40ms to 4 seconds every few minutes, but only during the nightly ETL job. Here's the full diagnostic sequence:

```bash
# Step 1: confirm I/O is the suspect — look at %iowait
iostat -x 1 5
```

```text
Device            r/s     w/s     rkB/s     wkB/s   r_await  w_await  aqu-sz  %util
sda              82.00  240.00   1640.00   9600.00    48.20   122.40    37.80   98.60
```

`aqu-sz` at 37.8 on an HDD is catastrophic. `w_await` at 122ms means every write is waiting over 100ms just to get serviced. `%util` at 98.6% — confirmed saturated. The disk is drowning.

```bash
# Step 2: find which process is causing the writes
sudo iotop -o -b -n 3 -d 2
# -o: only show processes with active I/O (hides idle processes)
# -b: batch mode (non-interactive, good for logging)
# -n 3: three iterations
# -d 2: 2-second intervals
```

```text
Total DISK READ:         0.00 B/s | Total DISK WRITE:       94.32 M/s
Current DISK READ:       0.00 B/s | Current DISK WRITE:     94.32 M/s
  TID  PRIO  USER     DISK READ  DISK WRITE  SWAPIN     IO>    COMMAND
 8821 be/4 postgres    0.00 B/s  91.14 M/s  0.00 % 98.12 %  postgres: etl_worker COPY etl_staging
  342 be/4 root        0.00 B/s   3.18 M/s  0.00 %  1.88 %  jbd2/sda1-8
```

The postgres ETL worker is consuming 91 MB/s of write bandwidth. `jbd2/sda1-8` is the ext4 journal daemon — normal background overhead. The ETL job is the culprit.

```bash
# Step 3: check the scheduler for this device
cat /sys/block/sda/queue/scheduler
```

```text
[mq-deadline] bfq none
```

`mq-deadline` is active (shown in brackets). Good — it is the right choice for an HDD. But what if the ETL job is competing with the database for the same disk? BFQ would give each process a fair share of bandwidth. Let's check whether switching to BFQ reduces the impact on live queries:

```bash
# Change the scheduler at runtime (no reboot required)
# This does NOT persist across reboots — see Distro Notes for persistence
echo bfq | sudo tee /sys/block/sda/queue/scheduler
```

```bash
# Step 4: check dirty page pressure — is writeback backing up?
cat /proc/meminfo | grep -i dirty
```

```text
Dirty:            819200 kB
Writeback:        204800 kB
```

800 MB of dirty pages with 200 MB actively being written back. The system is at the edge of its writeback capacity. Check the current thresholds:

```bash
sysctl vm.dirty_ratio vm.dirty_background_ratio
```

```text
vm.dirty_ratio = 20
vm.dirty_background_ratio = 10
```

On a server with 32 GB RAM, `dirty_ratio = 20` means up to 6.4 GB of writes can accumulate before processes block. That's a lot of data to lose on a crash. For a write-heavy ETL workload that doesn't need crash guarantees on intermediate data, you might raise `dirty_background_ratio` to 15 to start flushing sooner and avoid the hard limit:

```bash
# Apply at runtime — lost on reboot
sudo sysctl vm.dirty_background_ratio=15

# To persist across reboots, add to /etc/sysctl.conf:
# vm.dirty_background_ratio = 15
```

```bash
# Step 5: baseline the device's actual capability with fio
# Run this in a test environment — it writes to a file
fio --name=seq-write \
    --ioengine=libaio \
    --rw=write \
    --bs=1M \
    --size=4G \
    --numjobs=1 \
    --iodepth=32 \
    --direct=1 \
    --runtime=30s \
    --time_based \
    --filename=/tmp/fio-test.tmp
# --direct=1: bypasses page cache so we measure device speed, not RAM speed
# --iodepth=32: 32 concurrent in-flight requests — realistic for database workloads
```

```text
seq-write: (g=0): rw=write, bs=1M-1M, ioengine=libaio, iodepth=32
...
  write: IOPS=180, BW=180MiB/s (189MB/s)(5400MiB/30001msec)
    lat (msec): min=4, max=312, avg=177.54, stdev=28.61
```

The disk peaks at 180 MB/s sequential. The ETL job was trying to push 91 MB/s of application writes on top of whatever the database itself was writing — easily enough to saturate this device. This confirms the fix is not scheduler-tuning alone; the ETL job needs to be rate-limited or moved to a separate device.

## ⚠️ Gotchas

### `%util` at 100% Does Not Mean the Disk Is the Bottleneck

This has caused more incorrect capacity calls than any other iostat misreading. An NVMe SSD will show `%util` at 100% while delivering 3 GB/s with sub-millisecond latency and a near-empty queue. The device is "always busy" in the sense that it was never completely idle during the sample interval — but it is nowhere near saturated.

```bash
iostat -x 1
```

```text
Device       r/s    w/s   rkB/s    wkB/s  r_await  w_await  aqu-sz  %util
nvme0n1    0.00  8200.00   0.00  820000.00   0.08     0.09    0.73  100.00
```

`%util` 100%. Looks alarming. But `aqu-sz` is 0.73 and `await` is under 0.1ms. This device is fine — it is delivering 820 MB/s of writes with no queue buildup. The application is not blocked; it is going as fast as the NVMe can handle.

The rule: always check `aqu-sz` alongside `%util`. A saturated device has both high `%util` AND a growing `aqu-sz` (above 1 for HDDs, significantly higher for NVMe before concern).

### High `await` with Low `%util` Means the Disk Is Slow, Not Saturated

A spinning disk doing random I/O will show high `await` because every read requires a seek — the disk head physically moves. But `%util` may be only 30% because most of that time is spent seeking, not serving requests. The disk is slow per operation, not overwhelmed with volume.

```text
Device    r/s    w/s   rkB/s   wkB/s  r_await  w_await  aqu-sz  %util
sda      80.00   5.00  320.00  200.00   28.40    6.20    0.22   31.00
```

28ms `r_await` with 31% `%util` and `aqu-sz` under 1. The disk is not saturated — it is slow because random reads require seeks. You cannot fix this with a scheduler change. The fix is caching (more RAM for page cache), or moving to an SSD.

### `iotop` Without Root Hides Process Names

Running `iotop` as a non-root user or without `CAP_NET_ADMIN` will show kernel threads and some processes but hide process command details for others. You'll see entries that look like `[unknown]` or missing command lines.

```bash
# Run as normal user — process names are often missing
iotop -o
```

```text
  TID  PRIO  USER     DISK READ  DISK WRITE  SWAPIN     IO>    COMMAND
 8821 be/4 ?           0.00 B/s  91.14 M/s  0.00 % 98.12 %  [unknown]
```

```bash
# Run with sudo for full process visibility
sudo iotop -o
```

```text
  TID  PRIO  USER     DISK READ  DISK WRITE  SWAPIN     IO>    COMMAND
 8821 be/4 postgres    0.00 B/s  91.14 M/s  0.00 % 98.12 %  postgres: etl_worker
```

Always run `iotop` with `sudo` in production diagnostics. Missing process names mean missing root cause.

### `vm.dirty_ratio` Too Low Causes Write Stalls; Too High Risks Data Loss on Crash

Setting `vm.dirty_ratio` to a very low value — say 5% — means the kernel forces writeback aggressively. On a write-heavy workload, processes will block frequently waiting for dirty pages to flush. Throughput drops and latency spikes.

```bash
# Overly aggressive — processes will stall on writes under load
sudo sysctl vm.dirty_ratio=5
```

The reverse problem: a very high `vm.dirty_ratio` (say 80%) means potentially gigabytes of writes sitting in RAM, unflushed. A power outage or kernel panic loses all of it. For financial data or databases that depend on `fsync` guarantees, the application's own fsync calls bypass this — but general file writes are at risk.

The right value depends on your workload:
- Write-heavy ETL on a server with crash-safe application logging: higher values (20–40%) are acceptable to maximize throughput.
- Database server where the application manages its own durability via fsync: defaults are fine; the database's WAL mechanism is the real durability guarantee.
- Any workload where you cannot afford to lose buffered writes: keep defaults or lower them, and rely on UPS hardware.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install sysstat iotop fio`; sysstat service disabled by default — enable with `systemctl enable --now sysstat` to collect periodic stats |
| RHEL / Fedora / CentOS / Rocky | `dnf install sysstat iotop fio`; sysstat service available as `sysstat.service` |
| Arch | `pacman -S sysstat iotop fio` |
| Alpine (containers) | `apk add sysstat iotop fio`; `iostat` available; container environments often lack real block devices — you may see no output from iostat in a container |

**Scheduler persistence across reboots.** Writing to `/sys/block/*/queue/scheduler` takes effect immediately but does not survive a reboot. To make it permanent:

On RHEL/Fedora: use a udev rule in `/etc/udev/rules.d/`:

```text
# /etc/udev/rules.d/60-io-scheduler.rules
ACTION=="add|change", KERNEL=="sda", ATTR{queue/scheduler}="bfq"
```

On Debian/Ubuntu: same udev approach, or use tuned profiles (`tuned-adm` with the `throughput-performance` profile sets `mq-deadline` for non-NVMe devices).

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `iostat -x` | First-pass device-level I/O analysis — always reach for this first | Part of `sysstat`; the `-x` flag is non-negotiable for real diagnostics |
| `iotop` | Once you have a device-level problem, find which process is responsible | Requires kernel `CONFIG_TASK_IO_ACCOUNTING`; always run with `sudo` |
| `fio` | Baseline a device's raw capability before blaming the application | Does real I/O — do not run on production data directories without a dedicated test file |
| `blktrace` / `blkparse` | Deep per-request tracing when you need to see individual I/O events | Significant overhead; use only for targeted investigations, not continuous monitoring |
| `dstat` | Combined CPU/memory/disk/network view in one terminal | Useful for quick triage but less depth than `iostat -x`; some distros ship `dstat` deprecated in favor of `pcp-dstat` |
| `nmon` | Comprehensive interactive monitor with historical charting | Excellent for long-running monitoring sessions; less common than `iostat` |

`iostat` and `iotop` cover 90% of real-world I/O investigations. `fio` is essential for any storage capacity planning. Reach for `blktrace` only when you've exhausted what `iostat` can tell you and need to debug the I/O scheduler behavior itself.

## Trade-offs

**Where this shines:**

- `iostat -x` gives you device-level I/O metrics with almost zero overhead — sampling `/proc/diskstats` is cheap.
- The page cache writeback design lets Linux absorb write bursts in RAM and smooth them out to disk, dramatically improving throughput for bursty workloads.
- I/O schedulers can be swapped at runtime without a reboot — you can test a scheduler change and roll it back in seconds.

**Where this costs you:**

- The page cache writeback model means Linux lies to applications about write completion: `write()` returns success when data is in RAM, not on disk. If you need true durability, you must call `fsync()` explicitly — or use `O_SYNC`. The page cache will not protect you from power loss.
- `iotop` requires root and kernel task accounting. In restricted environments (containers, locked-down cloud instances), it may simply not work.
- `fio` can destroy data if pointed at the wrong device or file path. It writes real data to whatever target you give it.
- I/O scheduler tuning only affects the Linux block layer. If your storage sits behind a RAID controller with its own write-back cache, or behind a SAN, the scheduler you see in `/sys/block/` may have little real effect on observed latency.

**Performance note:** `iostat` sampling at 1-second intervals is essentially free — it reads counters from `/proc/diskstats`. `iotop` is slightly heavier because it uses the netlink `taskstats` interface, but still negligible in practice. `fio` with `--direct=1` bypasses the page cache and generates real device load — do not run it against production paths.

## When to Use / When to Avoid

✅ **Use this when:**

- A server has high latency or is slow, but CPU and RAM look normal — always check `iostat -x` next.
- You are buying or provisioning storage and need to know what the device can actually do — run `fio` first.
- A database is showing slow queries with no obvious query plan change — high `await` on the database device is a common root cause.
- You are tuning a write-heavy workload and want to understand whether writeback is causing stalls — check `Dirty` in `/proc/meminfo` under load.

❌ **Avoid this when:**

- You suspect a network storage issue (NFS, iSCSI) — `iostat` shows the block device as the kernel sees it, but the actual bottleneck is network latency. Use `nfsiostat` or `iscsiadm` for those.
- You want to measure application-level I/O semantics (whether `fsync` is being called, which files a process opens) — use `strace -e trace=write,fsync` or `inotifywait` for that level of detail.

⚠️ **Common mistakes:**

- Looking only at `%util` without `aqu-sz` — the single most common way to miss both saturation and non-saturation.
- Running `fio` without `--direct=1` — you end up benchmarking the page cache (RAM speed) rather than the actual device.
- Forgetting that `iostat`'s first line of output after boot is a cumulative average, not a real-time sample — always pass an interval and discard the first iteration.
- Changing `vm.dirty_ratio` without knowing how much RAM the system has — 20% of 1 GB is very different from 20% of 256 GB.

## Veteran Perspective

The `%util` trap gets people every time. I've watched engineers declare an NVMe SSD a bottleneck because `%util` was at 100%, then add more servers to "fix" the I/O problem — when the actual fix was to stop worrying about a number that meant nothing for their storage type. Check `aqu-sz`. Check `await`. Those two numbers tell the real story.

Here's the misunderstanding most intermediate sysadmins carry: they think `%util = 100%` maps directly to "disk full" the same way CPU 100% maps to "CPU full." It doesn't. `%util` was meaningful in the era of single-queue spinning disks where one request meant one operation and the disk was the serialization point. On modern NVMe with hardware-parallel queues, `%util` is nearly meaningless. The kernel still computes it the old way because changing the metric would break every monitoring tool ever written, but the semantics no longer match the hardware.

The dirty page writeback settings are one of those things that everyone copies from a blog post without understanding. The typical advice is "raise `vm.dirty_ratio` for better write performance" — which is true on write-heavy batch workloads but dangerous on anything where write ordering matters. A MongoDB or PostgreSQL instance that manages its own WAL doesn't care about `vm.dirty_ratio` for data durability, because it calls `fsync()` at appropriate points. But a service doing unbuffered file writes absolutely will lose data between background flushes if you've raised the dirty ratio high and then lose power.

Know your fsync story before you tune the dirty values. If your application syncs explicitly, tuning the dirty ratios is safe and can meaningfully improve throughput. If it doesn't — fix the application first.

The I/O scheduler conversation looks more complicated than it is. For anything NVMe: use `none`. For HDD: use `mq-deadline`. Only consider `bfq` on desktop systems or if you have a specific multi-process fairness requirement. `kyber` is interesting in theory but rarely changes outcomes in practice for most workloads.

The last thing I'll say: disk I/O problems are infrastructure problems, not tuning problems. A disk that can't keep up with the workload will not be fixed by choosing the right scheduler. The scheduler affects latency distribution; it does not increase IOPS capacity. Run `fio` to know what the device can actually do. If the workload requires more than the device provides, no amount of `/sys` tuning will help.

## Try It Yourself

1. Run `iostat -x 1 5` on your system and find the device where your root filesystem lives. What is the `await` for that device? What is the `aqu-sz`?

2. Check what I/O scheduler is currently active for each of your block devices: `cat /sys/block/*/queue/scheduler`. If you have an NVMe, confirm it is using `none`. If you have an HDD, confirm it is using `mq-deadline`.

3. Create some disk activity and observe it. Run `dd if=/dev/urandom of=/tmp/test.img bs=1M count=512` in one terminal. In a second terminal, watch `iostat -x 1` and observe how `%util`, `await`, and `aqu-sz` change while the write is in progress. When `dd` finishes, watch the metrics settle back to baseline.

4. Check your system's current dirty page accounting under load: while the `dd` from prompt 3 is running, in a third terminal run `cat /proc/meminfo | grep -E "Dirty|Writeback"`. Watch the `Dirty` value grow. After `dd` finishes, watch how fast the kernel flushes it to zero.

5. **Challenge (VM recommended):** Install `fio` and benchmark your storage device. Run a sequential write test with `--direct=1 --ioengine=libaio --rw=write --bs=1M --iodepth=32 --size=2G --runtime=30s` to get the device's maximum sequential write speed. Then run the same test with `--rw=randwrite --bs=4k` to get random 4K write IOPS. Compare the two results. If your sequential throughput is 500 MB/s but your random 4K IOPS are only 500, that's a spinning disk. If they're both high, that's an SSD.

## Summary

- `%util` measures whether the device was idle — not whether it is saturated. For NVMe SSDs, always check `aqu-sz` (average queue depth) and `await` (request service time) to assess actual saturation.
- A device is genuinely saturated when `aqu-sz` is consistently above 1 (for HDDs) or when `await` climbs significantly above the device's rated latency — not simply when `%util` reaches 100%.
- The default I/O scheduler for NVMe is `none` (let the device manage its own queues); for HDDs it is `mq-deadline`. Schedulers can be changed per-device at runtime via `/sys/block/<dev>/queue/scheduler`.
- `vm.dirty_background_ratio` triggers silent background writeback; `vm.dirty_ratio` is the hard limit where application writes block. The `write()` syscall returning success does not mean data is on disk — only `fsync()` guarantees that.
- `iotop -o` shows which processes are causing I/O; always run it with `sudo` or process names will be hidden.
