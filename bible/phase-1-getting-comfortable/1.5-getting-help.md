## The Problem This Solves

You've just found a command in a script you didn't write. It does something — you can tell from context — but there are four flags you don't recognize and you have no internet connection. Or you're on a server at 2am and something is broken and you can't remember if the flag is `-r` or `-R` or `--recursive`. Or you know what you need to do but you have no idea which command does it.

Every one of these situations has a built-in solution that requires no network access, no browser, and no Stack Overflow. The documentation is already on your system. It's been there since Linux was installed. Most beginners never learn to use it effectively, so they remain dependent on internet searches forever — which is a real liability when you're working on an air-gapped machine, a cloud server with no outbound access, or in the middle of an incident where speed matters.

The Linux help system is a layered set of tools, from the quick-reference `--help` flag to the comprehensive manual pages to the searchable `apropos` index. Learn the whole stack, and you'll rarely need to leave the terminal.

---

## Mental Model

Think of the Linux help system as a library in your building, not a library across town.

The `--help` flag is the summary card taped to the front of a reference book — it tells you the most common options at a glance. `man` is the full reference manual on the shelf — dense, authoritative, complete. `apropos` is the card catalog — you don't know which book you need, so you search by topic. `whatis` is reading only the spine of each book to get the one-line title. `info` is the hyperlinked extended edition sitting in the reference section — more complete than man for some GNU tools, but harder to navigate.

The key insight: all of this is local. No network required. You just have to know how to use the catalog.

## Theory

### Why documentation is built into the system

Unix was designed at Bell Labs in the late 1960s and early 1970s by a small group of engineers. One of the first things they built — even before many of the tools themselves were finished — was the manual page system. The reasoning was practical: there was no internet. If you wanted to know how something worked, the documentation had to ship with the software. This convention has never been abandoned. Every serious Linux command still ships with a man page, and that page lives on your local filesystem at the moment the package is installed.

### Man page sections

The manual is divided into numbered sections, each covering a different category of documentation. This is not just organizational housekeeping — it has a direct practical consequence: the same name can appear in multiple sections.

The eight standard sections are:

| Section | Contents |
|---|---|
| 1 | User commands — programs you run in the shell |
| 2 | System calls — functions the kernel exposes to programs |
| 3 | Library functions — the C standard library and other libraries |
| 4 | Special files — devices in `/dev`, kernel interfaces |
| 5 | File formats and conventions — config files, `/etc/passwd`, `/etc/fstab` |
| 6 | Games and screensavers |
| 7 | Miscellaneous — conventions, protocols, character sets, standards |
| 8 | System administration commands — tools that require root or manage the system |

The section number appears in parentheses after the command name in documentation: `printf(1)` is the shell command, `printf(3)` is the C library function. They are completely different pages.

### The `apropos` index

`apropos` does not search man page files directly. It reads from a pre-built database called the whatis database, which contains the name and short description of every installed man page. This database is built by `mandb` (on most modern distros) and is updated when packages are installed — but on a fresh install or a stripped-down container image, the database may not yet exist. When it doesn't, `apropos` returns nothing, which looks like the search failed when actually the catalog hasn't been built yet.

### What `type`, `which`, and `whereis` actually answer

These three commands answer subtly different questions about a command name:

- `which` answers: "What file path would the shell run if I typed this name?" It searches only `PATH` for executable files.
- `type` answers: "What is this name, from the shell's perspective?" It knows about shell builtins, functions, aliases, and keywords — all things that `which` cannot see because they aren't files on disk.
- `whereis` answers: "Where are the binary, the man page, and the source for this name?" It searches standard locations beyond just `PATH`.

The distinction between `which` and `type` is significant. `echo`, `cd`, `alias`, and `type` itself are shell builtins — they have no file path. `which echo` may return nothing or `/bin/echo` (the external binary version), but `type echo` correctly tells you it's a builtin. If you're debugging why a command behaves differently than expected, `type` is almost always the right first question.

## How It Works (Mechanics)

### How `man` works

When you run `man ls`, the shell locates the `man` binary, which then:

1. Searches a list of directories (`MANPATH`) for a compressed file matching the requested page and section. On most systems these files live under `/usr/share/man/` in section-specific subdirectories like `man1/`, `man5/`, `man8/`.
2. Decompresses the file (man pages are typically stored gzipped as `.gz`) and passes it through `groff`, which formats the troff-source content into readable text with proper width.
3. Pipes the formatted output through `less`, the standard pager.

This is why navigating a man page uses `less` keybindings — because it literally is `less`. Press `q` to quit, `Space` to page forward, `b` to page back, `/` to search, `n` to jump to the next match.

### How the `apropos`/`whatis` database works

The `mandb` command scans all man page directories, reads the `NAME` section of every page (which contains the one-line description), and stores the result in a Berkeley DB database — typically at `/var/cache/man/`. Both `apropos` and `whatis` query this database directly, which is why they are fast even when thousands of pages are installed.

The database is automatically updated when packages are installed on Debian/Ubuntu systems via a `dpkg` trigger. On other systems or in containers, you may need to run `mandb` manually.

### How `type` works

`type` is a shell builtin, not an external program. The shell already knows everything that is built in to itself, every function and alias defined in the current session, and every directory in `PATH`. When you run `type ls`, the shell walks that knowledge in priority order — builtins first, then functions, then aliases, then PATH — and reports the first match it finds.

## Basic Example

```bash
# Read the man page for ls
man ls
```

You'll drop into `less`. Navigate with `Space` (forward), `b` (back), `/` (search). Press `q` to quit.

```bash
# Get the quick-reference help for ls
ls --help
```

```text
Usage: ls [OPTION]... [FILE]...
List information about the FILEs (the current directory by default).
Sort entries alphabetically if none of -cftuvSUX nor --sort is specified.

Mandatory arguments to long options are mandatory for short options too.
  -a, --all                  do not ignore entries starting with .
  -A, --almost-all           do not list implied . and ..
...
```

```bash
# Ask what section 1 and section 5 of "passwd" are
man 1 passwd   # the passwd command for changing passwords
man 5 passwd   # the format of the /etc/passwd file
```

```bash
# Search for commands related to a topic
apropos partition
```

```text
addpart (8)          - tell the kernel about the existence of a partition
cfdisk (8)           - display or manipulate a disk partition table
delpart (8)          - tell the kernel to forget about a partition
fdisk (8)            - manipulate disk partition table
parted (8)           - a partition manipulation program
...
```

```bash
# Find out what a command is, not just where it lives
type cd
type ls
type ll    # try this — result depends on whether ll is aliased in your shell
```

```text
cd is a shell builtin
ls is /usr/bin/ls
ll is aliased to `ls -alF'
```

Try each of these yourself. When you look at the man page for `ls`, scroll down past the OPTIONS section until you find the EXIT STATUS section — it's there on most pages and is rarely mentioned in tutorials, but it matters when you're writing scripts. Your output may look slightly different depending on your distribution, but the section layout is consistent across all of them.

## Real-World Example

You've logged into a server after a report that disk space is critically low. You remember there's a command for finding large files, but you can't remember the name. Here's the full workflow using only the built-in help system:

```bash
# Search the man database for anything to do with disk usage
apropos "disk usage"
```

```text
df (1)               - report file system disk space usage
du (1)               - estimate file space usage
```

```bash
# You know du is the right tool — check how to sort by size
man du
```

Inside `man du`, press `/` and type `sort` to jump to anything mentioning sort. You find there's no built-in sort — you'll need to pipe to `sort`. Now check what flags `sort` takes:

```bash
man sort
# or faster:
sort --help | grep -i human
```

```text
  -h, --human-numeric-sort    compare human readable numbers (e.g., 2K 1G)
```

```bash
# Confirm what du's -h flag does before using it
whatis du
du --help | head -5
```

```text
du (1)               - estimate file space usage
Usage: du [OPTION]... [FILE]...
      or:  du [OPTION]...
Summarize device usage of each FILE, depth-first.
```

```bash
# Now you have everything you need
du -sh /var/* | sort -rh | head -20
```

```text
4.2G    /var/log
1.1G    /var/cache
512M    /var/lib
...
```

Notice what happened: you went from "I don't know the command" to a working diagnostic command using only local tools. No browser, no Stack Overflow. The `apropos` → `man` → `--help` pipeline is the real workflow. Burn it in.

## ⚠️ Gotchas

### `apropos` returns nothing on a fresh install

You run `apropos network` on a newly provisioned server and get:

```bash
apropos network
```

```text
network: nothing appropriate.
```

This doesn't mean no pages are installed. It means the whatis database hasn't been built yet. The database is separate from the man pages themselves, and some minimal installs or container images skip building it.

The fix is to build the database manually:

```bash
sudo mandb
```

```text
Processing manual pages under /usr/share/man...
Updating index cache for path `/usr/share/man/man1'. Wait...done.
...
```

After `mandb` completes, `apropos` works. On Debian/Ubuntu, this is usually triggered automatically when packages are installed. In containers and minimal installs, you'll need to run it yourself.

### `man` uses `less` — and `less` navigation is not obvious

New users open `man ls` and immediately try to scroll with the arrow keys, get confused when nothing happens the way they expect, and press `Ctrl+C` — which exits without reading anything.

`man` is not a document viewer. It is `less`. The keybindings are:

| Key | Action |
|---|---|
| `Space` or `f` | Page forward |
| `b` | Page back |
| `j` / `k` | One line down / up |
| `/pattern` | Search forward |
| `n` | Jump to next search match |
| `N` | Jump to previous search match |
| `q` | Quit |
| `g` | Go to top |
| `G` | Go to bottom |

If you already know `less` from reading log files, you already know how to use `man`. They are the same program.

### `--help` output goes to stderr on some programs, stdout on others

This trips people up when they try to pipe or redirect help output. Most programs send `--help` to stdout, but some send it to stderr — particularly older Unix tools and poorly behaved programs.

```bash
# This may work or may produce nothing in the file
somecommand --help > help.txt

# This always captures it, regardless of which stream it goes to
somecommand --help > help.txt 2>&1
```

The `2>&1` redirects stderr (file descriptor 2) to wherever stdout (file descriptor 1) is currently pointing — in this case, the file. This is a safe habit to build for any `--help` redirection.

### `which` is blind to shell builtins

This is one of the most common confusions for people coming from Windows or macOS.

```bash
which cd
```

```text
# Returns nothing, or /usr/bin/cd on some systems
```

`cd` is a shell builtin. There is no `/bin/cd` that your shell runs when you type `cd`. The shell handles it internally. `which` searches PATH for executables and never finds builtins. If `which` returns nothing, that doesn't mean the command doesn't exist — it may simply be built into the shell.

```bash
# Use type instead — it knows about builtins, functions, and aliases
type cd
```

```text
cd is a shell builtin
```

Use `type` when you want to know what a command actually is. Use `which` only when you specifically need the filesystem path of an external binary.

### `man 3 printf` and `man 1 printf` are entirely different pages

This catches people who run `man printf` and land on the shell command documentation, then wonder why there's no mention of format strings, `%d`, or `%s`. Those belong to section 3 — the C library function.

```bash
man 1 printf    # the shell command for printing formatted text
man 3 printf    # the C library function for formatted output
```

When you run `man printf` without a section number, man searches sections in order and returns the first match. On most systems section 1 comes first, so you get the command. If you're reading C documentation and looking for `printf(3)`, always specify the section explicitly.

## Distro Notes

| Distro Family | `apropos` database | `tldr`/`tealdeer` install |
|---|---|---|
| Debian / Ubuntu | `sudo mandb` — usually auto-updated via dpkg trigger | `sudo apt install tealdeer` |
| RHEL / Fedora / Rocky | `sudo mandb` (modern) or `sudo makewhatis` (older RHEL 6-era) | `sudo dnf install tealdeer` |
| Arch | `sudo mandb` | `sudo pacman -S tealdeer` |
| Alpine (containers) | `mandb` may not be installed; install `man-db` first | `apk add` not in main repo; use `cargo install tealdeer` |

**A note on `makewhatis`:** You'll see `makewhatis` referenced in older documentation and on older RHEL/CentOS 6 systems. Modern distributions (RHEL 7+, Fedora, all current Debian/Ubuntu/Arch) use `mandb`. They do the same job — build the whatis database — but `mandb` is the current standard.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `man` | Full, authoritative documentation | Ships with the software; always current for installed version |
| `--help` | Quick flag reference; faster than man | Not standardized; depth varies widely by program |
| `info` | Extended GNU documentation for GNU tools | More complete than man for bash, gcc, grep, coreutils; harder to navigate |
| `pinfo` | Same as `info` but with better navigation | Not installed by default; `apt/dnf/pacman install pinfo`; recommended over raw `info` |
| `tldr` / `tealdeer` | Practical examples for common use cases | Community-written; not authoritative; great for quickly recalling syntax |
| `apropos` | Finding which command does something | Requires built database; returns all matches, not just the best one |

The honest recommendation: use `--help` first for flag lookups, `man` for anything you need to understand deeply, and `tldr` for a fast memory jog on command syntax. `info` is worth learning for specifically GNU-heavy documentation — the bash manual inside `info bash` is more complete than `man bash`. `pinfo` makes `info` tolerable if you need to navigate it frequently.

## When to Use / When to Avoid

✅ **Use `man` when:**
- You need authoritative, complete documentation for the installed version
- You're reading file format documentation (`man 5 fstab`, `man 5 crontab`)
- You need to know system call semantics (`man 2 open`, `man 2 fork`)
- You're on an air-gapped or network-isolated machine

✅ **Use `--help` when:**
- You need a fast reminder of flags while in the middle of a task
- You're already in the terminal and don't want to context-switch to a full man page
- The command is one you roughly know but need a specific flag for

✅ **Use `apropos` when:**
- You don't know which command does what you need
- You're exploring a new system or a new area of the OS

✅ **Use `type` when:**
- Something behaves unexpectedly and you want to know what the shell actually resolves the name to
- You're debugging aliases, functions, or PATH conflicts

❌ **Avoid relying on `tldr` exclusively when:**
- You need to understand a flag's full behavior, not just the common case — tldr pages intentionally simplify
- The command is obscure or newly installed — community pages may not exist yet

❌ **Avoid `which` for builtins:**
- Use `type` instead; `which` will silently return nothing for builtins and give you a false negative

⚠️ **Common mistakes:**
- Running `apropos` on a fresh install before `mandb` has been run and concluding the tool is broken
- Using `man` section 1 to look up C library function signatures — always specify `man 3 <name>`
- Piping `--help` output without `2>&1` and getting an empty file

## Veteran Perspective

The first mistake beginners make is treating the help system as a last resort. They try to remember the flag, fail, then Google it, then copy the answer from Stack Overflow, then use it without fully understanding it. That workflow compounds ignorance. The right habit is the opposite: when in doubt, check `man` or `--help` first. Not because Google is wrong, but because the man page documents the exact version installed on your system, and Google doesn't.

The second mistake is never learning the `less` keybindings. I've watched people run `man tar`, scroll for two seconds, and close it — then ask a colleague what the flag was. The man page for `tar` is dense, but it's searchable. Press `/`, type `extract`, hit `Enter`. You're there in three keystrokes. Once you know `/` to search in `less`, man pages become navigable. Before that, they're a wall of text.

The third mistake — and this one bites more experienced people — is not knowing about man page sections. Every time I've seen someone confused about why `man printf` doesn't show format strings, the answer is section 3. When you're working near the C boundary — writing scripts that call compiled tools, reading code that uses system calls, debugging programs that interact with the kernel — you need sections 2, 3, and 5 in your mental model. A sysadmin who only knows section 1 is missing half the library.

`apropos` is underused in general. Most people don't know the command they need when they're working in an unfamiliar area of the OS. The answer isn't always Google — on a fresh install, `sudo mandb && apropos filesystem` or `apropos partition` will surface the right tool without leaving the terminal. This matters especially in environments where internet access is restricted.

One more thing about `tldr`: it is genuinely useful and I reach for `tealdeer` myself when I need a fast syntax reminder. But use it as a complement to `man`, not a replacement. The tldr page for `find` shows you five common invocations. The man page for `find` has 47 primaries and 8 operators, any combination of which might matter in a production scenario. Know both exist; know when to reach for each.

## Try It Yourself

1. Run `man ls` and use `/` to search for the word `color`. Find the flag that controls colorized output. Then quit with `q`.

2. Run `man 5 passwd` and `man 1 passwd`. Note the difference in what each page describes. These are two completely different things that happen to share a name.

3. Run `apropos copy` and look at the results. Then pick one command you don't recognize and read its man page. Find its description and one flag you didn't know existed.

4. Run `type echo`, `type ls`, `type cd`, and `type if`. Notice which are builtins, which are external commands, and which are shell keywords. Then run `which echo` and compare the result to `type echo` — they may not match.

5. **Challenge:** You're on a system and need to find every installed command that deals with network interfaces. Use `apropos` to build a list of candidates. For each interesting result, check `whatis` to get its one-line description, and read the man page for at least one you've never used before. Do this entirely without a browser.

## Summary

- `man` is the authoritative local documentation system; learn `less` navigation or man pages will always feel unusable
- Man pages are divided into 8 sections; the same name in section 1 (command) and section 5 (file format) or section 3 (C function) can be completely different pages — always specify the section when there's ambiguity
- `apropos` searches the whatis database, not the man pages directly — if it returns nothing, run `sudo mandb` to build the database first
- `type` finds builtins, aliases, and functions that `which` cannot see — use `type` when debugging why a command behaves unexpectedly
- `tldr` is a useful complement for syntax reminders, not a replacement for `man` — it deliberately omits the edge cases and full flag coverage that man pages contain
