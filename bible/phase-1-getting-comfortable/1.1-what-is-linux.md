## Why This Exists

In 1969, computing was expensive and exclusive. Computers filled entire rooms, cost millions of dollars, and each machine ran its own incompatible software. If you wrote a program for one computer, it ran on that computer and nothing else. Porting software meant rewriting it from scratch. Sharing tools between teams meant duplicating months of effort. There was no common ground.

The engineers at Bell Labs who built Unix had a different idea. What if an operating system was designed around portability — written in a high-level language instead of raw assembly, with an interface that programs could rely on regardless of the underlying hardware? What if the tools themselves were designed to be composable, so a small set of sharp utilities could be combined in ways their authors never anticipated?

That idea worked. Unix spread through universities and research labs through the 1970s and 1980s, but it remained proprietary. You needed a license to use it, a license to study it, a license to modify it. The source code was locked away. AT&T could — and did — change the terms.

Richard Stallman left MIT's AI Lab in 1983 to write a free Unix from scratch. Not free as in "it doesn't cost money" — free as in you could read it, modify it, and share it without asking anyone's permission. He called this project GNU (GNU's Not Unix) and founded the Free Software Foundation to fund it. By the early 1990s, GNU had nearly all the pieces: a C compiler, a shell, text editors, utilities. What it was missing was a kernel — the core of the operating system that actually talks to the hardware.

In August 1991, a Finnish computer science student named Linus Torvalds posted a message to the comp.os.minix newsgroup:

> *"I'm doing a (free) operating system (just a hobby, won't be big and professional like gnu) for 386(486) AT clones."*

That kernel became Linux. Paired with the GNU tools, it produced a complete, free operating system. What started as a hobby project is now the most widely deployed operating system kernel in history — running the servers that power the internet, the phones in two billion pockets, the world's fastest supercomputers, and satellites in orbit.

Linux exists because the engineers who built it were frustrated with the alternative: software you couldn't study, couldn't fix, and couldn't share. That frustration is baked into its design.

## Mental Model

Think of an operating system like a restaurant.

The **kernel** is the kitchen. It does the real work — managing hardware, allocating memory, running processes, handling storage. Customers (your programs) never walk into the kitchen directly. They place orders through a well-defined interface.

The **userland** — the shell, the utilities, the desktop, the applications — is everything customers actually see and interact with: the dining room, the menu, the waitstaff. When you run `ls` or open a browser, you're interacting with userland. Userland passes requests to the kitchen when it needs something the hardware controls.

A **distribution** is the entire restaurant: a particular kitchen (kernel version), a particular menu (pre-installed software), a particular style of service (package manager, default configuration). Different restaurants use the same kitchen design, but the dining room, menu, and pricing vary.

Linux is technically the name of the kitchen. When most people say "Linux," they mean the whole restaurant.

## Theory

### The Unix Lineage

Everything in this curriculum descends from a design philosophy established at Bell Labs between 1969 and 1973. Ken Thompson wrote the first version of Unix in assembly language in a matter of weeks. Dennis Ritchie then invented the C programming language specifically to rewrite Unix in something portable. This was radical: an operating system written in a language that could be compiled for different hardware. When you understand why Unix was designed the way it was, almost everything else in Linux makes sense.

The fundamental design decisions made in those years still govern every system you'll ever administer:

- Files are the universal abstraction. Hardware devices, network sockets, inter-process communication channels — all of it is accessed through the same read/write interface as a regular file.
- Processes are cheap and composable. Rather than building large monolithic programs that do everything, Unix encourages small tools that do one thing well and pass their results to the next tool.
- Text is the universal data format. Tools communicate through plain text streams, which means any tool can send its output to any other tool without special adapters.

This is the Unix philosophy, and it is not merely aesthetic. It has concrete consequences. A system built this way is auditable — you can understand what it's doing by reading text. It's composable — you can build new capabilities by combining existing tools. It's debuggable — problems surface as readable error messages or inspectable file states.

### The GNU Project and the GPL

Richard Stallman's contribution was not just technical — it was legal. He invented the GNU General Public License, a software license with a specific and unusual property: any software distributed under the GPL must carry the same license. If you take GPL code, modify it, and distribute the result, your modifications must also be GPL. The license propagates.

This is called "copyleft." It uses copyright law to guarantee the opposite of what copyright usually does — instead of restricting use, it ensures that software can never be locked away by a downstream actor. A company can use Linux in their product, but they cannot take the kernel proprietary. They can sell support, they can add extensions, but the kernel source stays open.

The distinction between "free software" (Stallman's framing) and "open source" (a term coined in 1998 by a different faction) is real but mostly philosophical for working sysadmins. Both mean you can read, modify, and share the source. The difference is in the *why*: Stallman sees it as a matter of ethics and freedom; the open source movement framed the same idea in terms of engineering quality and business pragmatism. Linux operates under both framings simultaneously.

### Linux: The Kernel

When Torvalds says "Linux is the kernel, the rest is GNU," he's making a precise technical distinction. The kernel is the program that runs at the lowest level of the system — it boots first, it has direct access to hardware, and every other program runs at its permission. Everything else — the shell you type commands into, the programs you run, the libraries they use — runs on top of the kernel in userspace.

The Linux kernel is a monolithic kernel. All the core subsystems — memory management, process scheduling, filesystems, device drivers, networking — live in a single kernel address space. This is a deliberate engineering choice: kernel components can call each other directly without the overhead of inter-process communication. Loadable kernel modules allow drivers to be added and removed at runtime without rebooting, but they still run in kernel space.

### Kernel Versioning

Linux uses a `MAJOR.MINOR.PATCH` version scheme. After version 2.6, Torvalds moved to even/odd minors having no special significance — both even and odd minor versions are production releases.

Each release cycle runs roughly 9–10 weeks. At the end of each cycle, a new version ships. Some versions are designated **LTS (Long-Term Support)** — they receive backported security and bug fixes for several years rather than the usual 2–3 months of a standard stable release.

As of June 2026, the kernel landscape looks like this:

- **7.0.x** — current mainline stable, released April 2026
- **6.12.x** — active LTS
- **6.6.x** — active LTS
- **6.1.x** — active LTS
- **5.15.x** — active LTS (extended, scheduled for EOL December 2026)
- **5.10.x** — active LTS (extended)
- **5.4.x** — active LTS (extended)

You can always check the current state at [kernel.org/releases.html](https://www.kernel.org/releases.html).

To see what kernel version a running system has, use:

```bash
uname -r
```

```text
6.12.25-generic
```

The number before the first dot is the major version. The number after it is the minor. The number after the second dot is the patch level. Your distribution may append extra characters after a dash — these are distribution-specific packaging identifiers, not upstream kernel version numbers.

### What a Distribution Is

Nobody ships just the kernel. A **Linux distribution** bundles the kernel with a package manager, a userland (usually GNU tools), an init system, default configuration, a software repository, and installation tooling. The result is a complete, installable operating system.

Distributions make choices. They choose which kernel version to ship (and how aggressively to update it). They choose a package format and a default set of installed software. They choose when and how to apply security patches. They choose defaults for network configuration, user management, and service startup. These choices have real consequences in production.

The distribution landscape breaks into families — groups of distributions that share a common ancestor, a common package format, or both.

### The Debian Family

Debian is the oldest and most influential downstream of the three major package families. It was started in 1993 by Ian Murdock (who named it after himself and his then-girlfriend, Debra). Debian uses the `.deb` package format and the `apt` package manager. Its release cycle is slow by design — Debian Stable is genuinely stable, often shipping kernel and library versions that are one or two years old by the time they appear in a release. This conservatism is a feature, not a bug, for anyone running infrastructure.

**Ubuntu** is the most widely used desktop Linux distribution and a major server platform. Built by Canonical on top of Debian, it ships LTS releases every two years (in April of even-numbered years) with five years of support, and standard releases every six months. Ubuntu 26.04 LTS "Resolute Raccoon" is the current LTS as of mid-2026.

**Linux Mint** is built on Ubuntu and targets users migrating from Windows. It ships with a more conventional desktop layout and all proprietary codecs included. Common on personal machines; not a server choice.

**Kali Linux** is a Debian derivative maintained by Offensive Security and pre-loaded with security testing and penetration testing tools. It is designed for security professionals, not general use. Running Kali as your daily desktop because it "looks cool" is how people get compromised — the defaults are set for attack tooling, not defense.

**Raspberry Pi OS** (formerly Raspbian) is the official distribution for Raspberry Pi hardware. Debian-based, ARM-optimized.

### The Red Hat Family

Red Hat was founded in 1993 and became the first company to build a viable commercial business around Linux support rather than software licensing. **Red Hat Enterprise Linux (RHEL)** is the commercial distribution — you pay for a subscription that includes support and access to patches. RHEL ships infrequently (major versions years apart), stays on each release for a decade, and backports security fixes aggressively rather than updating to newer upstream versions. If you administer enterprise infrastructure, you will encounter RHEL.

**Fedora** is Red Hat's community distribution — the upstream proving ground where new technologies land first before being considered for RHEL. Fedora ships every six months and tracks closer to bleeding-edge packages. Fedora 44 is the current release as of mid-2026. Fedora is excellent for developers who want a current, well-supported system; it is less appropriate for anything requiring years of stability.

**CentOS** was for twenty years the free, community-rebuilt version of RHEL — binary-compatible, no subscription required. In December 2020, Red Hat announced that CentOS Linux would be discontinued and replaced with **CentOS Stream**, a rolling-release distribution positioned *upstream* of RHEL rather than downstream. This change was controversial. CentOS 7 reached end-of-life in June 2024. CentOS 8 had already reached EOL in December 2021. If you still have CentOS 7 or 8 machines, they are running without security patches.

**Rocky Linux** emerged directly from community frustration with the CentOS change. Founded by Gregory Kurtzer (one of CentOS's original founders), Rocky Linux aims to be what CentOS was: a free, binary-compatible downstream rebuild of RHEL. Rocky Linux 10 was released in June 2025, tracking RHEL 10.

**AlmaLinux** is the other major CentOS successor, backed by the CloudLinux company. It released its first build within two months of the CentOS announcement, and AlmaLinux 10 shipped in May 2025. AlmaLinux made a notable decision to move from strict binary compatibility with RHEL toward Application Binary Interface (ABI) compatibility — meaning it tracks RHEL's behavior without necessarily matching every binary exactly. In practice, this is not a meaningful difference for most workloads.

Both Rocky Linux and AlmaLinux are mature, production-ready, and actively maintained. Either is a sound choice as a CentOS replacement.

### The Arch Family

**Arch Linux** is a rolling-release distribution. There are no "versions" of Arch — the system is continuously updated. A fresh Arch install today gets the same current packages as a system installed three years ago and kept up to date. Arch ships nothing by default. The installation process is entirely manual, which is its point: you build exactly the system you want, and you understand every piece of it because you put it there yourself.

Arch's package manager is `pacman`. Its community-maintained software repository, the AUR (Arch User Repository), contains an enormous breadth of packages that are not in the official repos.

**Manjaro** is a user-friendly Arch derivative with a graphical installer and more conservative update cadence. It targets users who want Arch's rolling model without the manual installation.

**EndeavourOS** is a lighter Arch derivative with a graphical installer but a philosophy closer to vanilla Arch. More popular than Manjaro among users who want Arch-adjacent without full DIY.

### The SUSE Family

**openSUSE Leap** is a community distribution built from binaries shared with **SUSE Linux Enterprise Server (SLES)**, the commercial offering from the German company SUSE. Leap follows a predictable release cycle and is stable in the Debian/RHEL sense.

**openSUSE Tumbleweed** is SUSE's rolling-release offering — continuously updated, similar in philosophy to Arch but with more curated packaging.

SUSE's package manager is `zypper`, and the package format is `.rpm` — the same format used by the Red Hat family.

**SLES** is a commercial enterprise Linux aimed at corporate data centers. It competes directly with RHEL and is particularly common in environments running SAP workloads.

### Independent Distributions

Not every distribution belongs to a family:

**Gentoo** is a source-based distribution: instead of installing pre-compiled binary packages, you compile software from source on your own hardware with your own optimization flags. This is slow. It is also deeply educational — the Portage package manager and the process of configuring a Gentoo system teaches you more about how Linux software builds than almost any other approach. Gentoo is not a production server choice; it is a learning environment and a toolkit for embedded systems where precise optimization matters.

**Slackware** is the oldest surviving Linux distribution, first released in 1993 by Patrick Volkerding. It has no automatic dependency resolution in its package manager — you install what you need, manually. Slackware is minimalist, stable, and opinionated. It has a small but loyal following, particularly among people who learned Linux in the 1990s.

**NixOS** takes a radically different approach to the entire problem of package management and system configuration. Everything — packages, system configuration, service definitions — is described in a declarative functional language called Nix. The system can be reproduced exactly from its configuration file. Upgrades are atomic and rollbacks are trivial. NixOS is gaining traction in development environments and for reproducible infrastructure. It requires a mental model shift that most sysadmins find steep.

**Alpine Linux** is a minimal, security-hardened distribution built on musl libc and BusyBox rather than glibc and GNU coreutils. The base image is under 10MB. Alpine is the dominant base image for Docker containers — its tiny footprint and security posture make it ideal for containerized workloads where you want nothing installed that you don't explicitly need.

**Void Linux** is an independent rolling-release distribution using runit as its init system rather than systemd. It supports both glibc and musl. Popular with users who want a modern rolling distribution without systemd.

### POSIX: The Portable Contract

When Unix fragmented in the 1980s into competing proprietary variants — BSD, System V, SunOS, HP-UX, AIX — the resulting incompatibility became genuinely painful. A program written for one Unix worked differently (or not at all) on another. The IEEE and The Open Group responded by creating POSIX: a standard that defines the interface an operating system must provide to be considered Unix-compatible.

POSIX specifies:
- Which system calls exist and what they do (file I/O, process management, signals)
- The behavior of standard utilities (`ls`, `find`, `grep`, `awk`, `sed`, and dozens more)
- The standard C library interface
- Thread and networking behavior

The current version is **POSIX.1-2024** (IEEE Std 1003.1, 2024 Edition, also called The Open Group Base Specifications Issue 8), published June 2024.

Linux is not officially POSIX-certified — official certification costs money and requires ongoing compliance testing. But Linux closely follows POSIX in practice, and the vast majority of POSIX-compliant programs run on Linux without modification. When you write a shell script using standard tools, you are writing against the POSIX interface. When that script works on Linux and macOS and the BSDs, it's because all of them implement the same underlying contract.

POSIX is why the command line feels the same everywhere. It is not an accident.

### Linux in the World

Linux is not a niche operating system. It is the operating system.

- **Servers:** Over 90% of internet servers run Linux. Every major cloud provider — AWS, Google Cloud, Azure — runs its infrastructure on Linux. When you load a web page, make an API call, or stream a video, the server handling that request is almost certainly running Linux.
- **Android:** Android's kernel is Linux. Approximately 72% of mobile devices worldwide run on a Linux kernel.
- **Supercomputers:** As of the most recent Top500 survey, 100% of the world's 500 fastest supercomputers run Linux.
- **Embedded systems:** Routers, smart TVs, car infotainment systems, industrial controllers, network switches, digital cameras. If a device runs complex software, there is a high probability it runs Linux.
- **Cloud infrastructure:** Containers (Docker, Kubernetes), virtual machines, serverless functions — all run on Linux hosts. The container revolution of the 2010s was built entirely on Linux kernel features.
- **Satellites and spacecraft:** Linux runs on board systems including the Mars helicopter Ingenuity.

This matters to you as someone learning Linux administration: the skills you are developing are not niche. They apply to the infrastructure that runs almost everything.

## How It Works (Mechanics)

This is a conceptual topic, so "mechanics" here means understanding the layered structure of a running Linux system — how the pieces relate to each other at runtime.

### The Ring Model

Modern CPUs enforce privilege levels in hardware, commonly called "rings." Linux uses two:

```text
Ring 0 — Kernel space
  ┌───────────────────────────────────────────────┐
  │  Kernel (process scheduler, memory manager,   │
  │  VFS, network stack, device drivers)          │
  └───────────────────────────────────────────────┘
        ↕ system calls (the only bridge)
Ring 3 — User space
  ┌───────────────────────────────────────────────┐
  │  Your programs: bash, nginx, python, ls ...   │
  │  GNU C Library (glibc) — wraps syscalls       │
  └───────────────────────────────────────────────┘
```

Code running in kernel space has unrestricted access to hardware and memory. Code in user space can only do what the kernel permits. The only way for user-space code to cross into kernel space is through a **system call** (syscall) — a formal request to the kernel to perform a privileged operation. Reading a file, creating a process, opening a network connection: all syscalls.

This boundary is why a crashing user-space program cannot bring down the system (usually). The kernel is isolated from the crash.

### What Happens When You Type a Command

When you type `ls` and press Enter, a substantial amount of machinery runs:

```text
# Conceptual flow — not a real command
1. Shell reads your input
2. Shell calls fork() — creates a child process
3. Child process calls execve("/usr/bin/ls", ...) — replaces itself with ls
4. ls calls opendir(), readdir() — each is a syscall into the kernel
5. Kernel checks permissions: do you have read+execute on this directory?
6. Kernel returns directory entries through the VFS (virtual filesystem layer)
7. ls formats the output and writes to stdout (file descriptor 1)
8. Output appears on your terminal
9. ls exits; kernel cleans up the process; shell resumes
```

Every step that touches the filesystem, creates a process, or reads from hardware goes through the kernel. User space programs never directly address hardware — that is the kernel's job.

### `uname -r` — Reading the Running Kernel Version

```bash
uname -r
```

```text
6.12.25-generic
```

`uname` makes a `uname()` syscall, which returns information the kernel stored about itself at compile time. The `-r` flag requests just the kernel release string. The format is `MAJOR.MINOR.PATCH-DISTRO_SUFFIX`. The distro suffix is added by your distribution's packaging — it is not part of the upstream kernel version.

## Basic Example

The simplest way to see the kernel underneath a running system:

```bash
# Show the running kernel version
uname -r
```

```text
6.12.25-generic
```

```bash
# Show full system information: kernel name, hostname, version, machine arch
uname -a
```

```text
Linux myhostname 6.12.25-generic #25-Ubuntu SMP PREEMPT_DYNAMIC Wed Mar 12 14:22:05 UTC 2026 x86_64 x86_64 x86_64 GNU/Linux
```

The fields in `uname -a` output, left to right: kernel name (`Linux`), hostname, kernel release, kernel version (build timestamp and compiler info), machine hardware, processor type, platform, and operating system.

```bash
# Show which distribution you're running
cat /etc/os-release
```

```text
NAME="Ubuntu"
VERSION="26.04 LTS (Resolute Raccoon)"
ID=ubuntu
ID_LIKE=debian
PRETTY_NAME="Ubuntu 26.04 LTS"
VERSION_ID="26.04"
HOME_URL="https://www.ubuntu.com/"
SUPPORT_URL="https://help.ubuntu.com/"
BUG_REPORT_URL="https://bugs.launchpad.net/ubuntu/"
```

`/etc/os-release` is the standard file for distribution identification — it exists on every modern Linux distribution and uses a consistent format you can parse in scripts.

Try these three commands on your own system. Notice that `uname -r` tells you the kernel version, while `/etc/os-release` tells you the distribution. These are separate pieces of information. The distribution chooses which kernel version to ship, which is why they can differ from what you'd see if you compiled the kernel yourself.

## Real-World Example

On a production server, the first thing any experienced sysadmin does after getting access to an unfamiliar system is orient themselves: what distribution, what kernel, what architecture, is this a VM or bare metal?

```bash
# Identify the distribution
cat /etc/os-release

# Identify the kernel version and architecture
uname -r
uname -m

# Check if this is a virtual machine (common in cloud deployments)
systemd-detect-virt

# Check how long the system has been running (uptime since last reboot)
uptime -p
```

```text
up 47 days, 3 hours, 12 minutes
```

```bash
# On a RHEL/Rocky/AlmaLinux system, check subscription and support status
cat /etc/redhat-release
```

```text
Rocky Linux release 10.1 (Red Quartz)
```

The walkthrough: you use `cat /etc/os-release` rather than relying on `lsb_release` because `lsb_release` requires the `lsb-release` package to be installed, which it often isn't on minimal server installs. `/etc/os-release` is part of the systemd project's standard and exists on every systemd-based distribution. `uname -m` tells you the machine architecture — `x86_64` is the standard 64-bit Intel/AMD architecture; `aarch64` is 64-bit ARM (common in cloud instances and Raspberry Pi). `systemd-detect-virt` returns the virtualization technology in use (`kvm`, `vmware`, `xen`, `docker`, etc.) or `none` if you're on bare metal. Knowing you're in a VM matters because some kernel behaviors differ in virtualized environments and some diagnostic tools don't work correctly.

## ⚠️ Gotchas

### "Linux" Usually Means the Whole Distribution, Not Just the Kernel

When someone says "we run Linux," they almost never mean they've installed just the kernel. They mean a full Linux distribution. When someone says "upgrade Linux," they might mean upgrade the distribution, upgrade the kernel package within the distribution, or upgrade from one major distribution version to another — these are three very different operations with very different procedures and risks.

When you're in a conversation about a system and someone says "it runs Linux," your next question should be: which distribution, which version?

### The Kernel Version and the Distribution Version Are Not the Same Thing

A common beginner assumption is that "Ubuntu 26.04" ships "Linux 26." It doesn't. The version number of a distribution has nothing to do with the kernel version it ships. Ubuntu 26.04 ships kernel 7.0. Ubuntu 24.04 shipped kernel 6.8. Debian 13 ships kernel 6.1.

The distribution version describes the userland, the packages, the release date, and the support timeline. The kernel version describes the specific kernel build. Both matter, they track separately, and confusing them will make you look confused in front of experienced sysadmins.

### CentOS Is Dead — Don't Plan Around It

If you're reading documentation, tutorials, or forum posts about CentOS, check the date. CentOS 7 reached end-of-life in June 2024. CentOS 8 reached end-of-life in December 2021. There are no more security patches for either. Any CentOS machine still running in production is running unpatched software.

CentOS Stream is not a replacement in the same sense — it's a rolling development preview of what will become the next RHEL, not a stable downstream rebuild. If you need a free RHEL-compatible system, use Rocky Linux or AlmaLinux.

### "Free" in Linux Means Freedom, Not Price

Linux distributions can cost money. RHEL requires a subscription. SLES requires a subscription. Support contracts cost money. Paying for Linux is normal and common in enterprise environments.

"Free software" in the GNU/Linux context means freedom: the freedom to study, modify, and redistribute. It does not mean "no one charges money for it." This distinction matters when people say "but why would I pay for Linux when it's free?" — because what they're paying for is support, stability guarantees, and compliance certification, not the right to use the software.

### Kernel Modules Are Still Kernel Code

You'll eventually encounter the concept of loadable kernel modules — pieces of kernel code (usually device drivers) that can be loaded and unloaded without rebooting. A common beginner assumption is that because modules load dynamically, they're somehow sandboxed or safer than the main kernel. They're not. Kernel modules run in kernel space with the same privileges as the rest of the kernel. A buggy module can crash your entire system. A malicious module has complete control over the machine. This is why production systems often restrict which modules can load.

## Alternatives & Comparisons

| System | Relationship to Linux | When You'd Encounter It |
|---|---|---|
| **macOS** | Not Linux — Darwin kernel (XNU), BSD userland. POSIX-compliant. | Developer laptops; some tools behave differently than on Linux |
| **FreeBSD / OpenBSD / NetBSD** | Not Linux — separate kernel lineage from Unix. POSIX-compliant. | High-performance networking (FreeBSD powers parts of Netflix CDN), security-focused servers |
| **Windows Server** | Not Unix-derived. WSL2 runs a full Linux kernel inside a VM. | Enterprise environments; PowerShell is the shell analog |
| **Android** | Linux kernel, but radically different userland (Bionic libc, no GNU tools, no systemd) | Mobile devices; embedded |
| **Chrome OS** | Linux kernel, Gentoo-derived userland, runs a full Linux container (Crostini) | Consumer Chromebooks |

macOS is worth a specific note. It is POSIX-compliant, uses a bash/zsh terminal, and many command-line tools behave similarly to Linux. This leads to the common misconception that macOS skills transfer directly. They do — partially. The filesystem layout is different, package management is entirely different (Homebrew vs. apt/dnf), system paths differ, and kernel behavior differs in ways that matter for low-level work. macOS is Unix-adjacent, not Linux. Production server administration on Linux will expose differences that catch macOS-familiar engineers off guard.

## When to Use / When to Avoid

This section applies to choosing a distribution for a specific purpose, not to "using Linux" (you're already committed to learning it).

✅ **Use Debian/Ubuntu for:**
- General-purpose servers where you want broad package availability and long LTS support windows
- Development environments where apt's package breadth saves time
- Raspberry Pi and ARM embedded projects

✅ **Use RHEL/Rocky Linux/AlmaLinux for:**
- Enterprise workloads where formal support, compliance certification (PCI, HIPAA), and multi-year stability are required
- Environments that already standardize on Red Hat tooling
- Anywhere you need a distribution your vendor explicitly supports

✅ **Use Fedora for:**
- Development machines where you want current tooling
- Testing software against a close-to-RHEL environment before it reaches RHEL

✅ **Use Alpine for:**
- Container base images where image size and attack surface matter
- Embedded or resource-constrained environments

✅ **Use Arch for:**
- Personal machines where you want to learn deeply and control everything
- Rolling packages where you want the newest version of something immediately

❌ **Avoid using Kali Linux as a daily driver.** It's not hardened for that use case and its default configuration assumes an attacker's posture.

❌ **Avoid using CentOS 7 or CentOS 8 for anything new.** They are end-of-life and unpatched.

❌ **Avoid making Arch a production server** unless you have a strong reason and a rigorous update testing process. Rolling releases break things occasionally.

⚠️ **Common mistakes:**
- Choosing a distribution based on aesthetics (desktop themes, logos) rather than support lifecycle and package availability
- Picking an obscure distribution because it's "more advanced" — the advanced skill is administering any Linux system well, not picking the hardest installer
- Assuming your desktop Linux knowledge translates directly to server administration — it does, but the priorities are different

## Veteran Perspective

The first mistake beginners make about Linux is thinking the distribution choice is the most important decision. It isn't. The concepts you're learning — the filesystem, permissions, processes, networking, the shell — are essentially identical across every distribution. The differences between Ubuntu and Rocky Linux are smaller than the differences between understanding how the system works and not understanding it.

The second mistake is confusing the kernel with the operating system. When Linus Torvalds makes statements about Linux, he means the kernel. When your company says "we run Linux," they mean a full distribution. The distinction matters in context: kernel bugs affect all distributions; distribution packaging bugs affect only users of that distribution.

Here's what most documentation won't say directly about the CentOS situation: Red Hat made a business decision that was rational for Red Hat and damaging to a large part of their community. Many organizations had built their entire infrastructure standardization around "CentOS is free RHEL." When CentOS Stream was announced, they needed a new answer. Rocky Linux and AlmaLinux exist because of that trust gap. The situation is a good example of a recurring pattern in open source: commercial interests and community interests are aligned most of the time, but when they diverge, forks happen. The Linux ecosystem is healthy partly because forks are cheap — the GPL guarantees the source is always available.

The Unix philosophy — do one thing well, compose small tools into larger workflows — is not nostalgia. It is the design principle that makes Linux administration fundamentally different from Windows administration. On Windows, a GUI application manages most system state. On Linux, system state is files and text. That means you can inspect it, script it, version-control it, and replicate it. The operations teams that are best at their jobs are the ones who've internalized this: everything is a file, everything is a process, and text is the protocol.

I've talked to engineers who spent years on Linux servers without ever understanding where the kernel ends and the distribution begins. It created real confusion when they needed to, say, decide whether a kernel bug patched upstream was present in their distribution's kernel package. Understanding the layering isn't pedantic — it's practical.

POSIX matters more than people realize until the moment it matters. A shell script written to POSIX sh standards runs on Linux, macOS, and every BSDs. A script written using bash-isms runs on bash. If you get into the habit of knowing which features are POSIX and which are bash extensions, you will write scripts that don't surprise you on systems where bash isn't the default shell (Alpine uses ash; some minimal server images use dash as /bin/sh). This is the kind of thing that causes 3am incidents when someone deploys a script to a new environment.

## Try It Yourself

1. Run `uname -r` on your system. What kernel version are you running? Now look up your distribution's current kernel version on its website. Are they the same? If not, why might they differ?

2. Run `cat /etc/os-release`. Identify the `ID` and `ID_LIKE` fields. The `ID_LIKE` field tells you which distribution family yours derives from. What does yours say?

3. Run `uname -a`. Write down what each field means. If you're not sure about one of them, look it up in `man uname`.

4. Find out whether your system is a virtual machine by running `systemd-detect-virt`. If you're on a desktop or laptop, it might say `none` (bare metal). If you're in a cloud VM or a VM on your own machine, it should name the hypervisor. Why would a sysadmin want to know this?

5. **Challenge:** The kernel version string from `uname -r` includes distribution-specific suffixes. Research what the suffix on your specific system's kernel version means — who added it, what it encodes (version, architecture, date, etc.), and where it is set during the kernel build process. This will require reading your distribution's kernel packaging documentation. Post your findings as a note to yourself.

## Summary

- Linux is a kernel, not a complete operating system. A full Linux system is the kernel paired with GNU tools and a distribution's packaging — often called a "Linux distribution" or "distro."
- The Linux kernel uses a `MAJOR.MINOR.PATCH` version scheme. LTS kernels receive security backports for several years; stable non-LTS kernels are maintained for a few months. As of mid-2026, the current mainline stable is 7.0.x; active LTS versions include 6.12.x, 6.6.x, and 6.1.x.
- Linux distributions fall into families that share package formats and tooling: Debian family (apt, .deb), Red Hat family (dnf, .rpm), Arch (pacman), and SUSE (zypper, .rpm). The choice of distribution affects support lifecycle, package availability, and default tooling — not your fundamental Linux skills.
- POSIX.1-2024 is the current standard defining the interface that Unix-compatible systems must provide. Linux follows POSIX closely in practice, which is why shell skills transfer across Linux, macOS, and BSD systems.
- The Unix philosophy — small tools that do one thing well, communicating through text streams — is not aesthetic preference. It is the design principle that makes Linux systems auditable, composable, and scriptable in ways that other operating systems are not.
