## The Problem This Solves

Imagine you've SSH'd into a server for the first time — or opened a terminal on your own machine — and you're staring at a prompt. No icons. No sidebar. No file explorer window. Just a blinking cursor.

You don't know where you are. You don't know what files are here. You don't know how to get to the files you need. And you don't know how to get back if you take a wrong turn.

On a GUI desktop, the operating system holds your hand. There are folders on the screen, you double-click to enter them, the title bar shows you where you are, and clicking the back arrow takes you back. The navigation is embedded in the visuals.

In a terminal, none of that exists. The filesystem is still there — the exact same directory tree — but you navigate it entirely through text commands. You type where you want to go. You type to see what's there. If you make a wrong turn, you type your way back.

This sounds more difficult. In practice, once you've learned five commands, it's faster and more precise than any GUI file manager ever built. You can jump to any directory on the system in two keystrokes. You can list files matching a pattern across the entire filesystem in one command. You can search ten thousand files by name in under a second.

But first, you need to learn those five commands — and more importantly, you need to understand the mental model that ties them together.

## Mental Model

Think of the Linux filesystem as a city, and your terminal session as a person standing somewhere in that city.

At any moment, you are standing on exactly one street corner — your **current working directory**. Every command you run happens from that corner. When you list files, you see what's on your corner. When you open a file without specifying a path, the system looks for it on your corner.

Streets in this city have addresses. An address that starts from the very center of the city (the root, `/`) is a **full address** — an absolute path. An address given as directions from where you're currently standing ("three blocks north, then left") is a **relative path**. Both get you to the same destination, but you calculate relative paths from wherever you currently are.

The commands in this topic are:
- `pwd` — "Where am I right now?"
- `ls` — "What's on this corner?"
- `cd` — "Walk to a different corner."
- `tree` — "Show me the whole neighborhood in one view."
- `find` — "Search the entire city for something matching a description."
- `locate` — "Ask the city's directory service — it's faster, but the listing might be a day old."

Hold this model. Everything else in this topic is just filling in the details.

## Theory

### The Filesystem Is a Tree

Every Linux system has a single unified directory tree. It starts at the root, `/`, and branches out from there. There is no "C: drive" or "D: drive" — there is one tree. Additional storage devices, when mounted, appear as subdirectories within that tree, not as separate roots. (Mounting is covered in depth in **2.5 — Mounting**.)

Directories are nodes in this tree. Files are leaves. A directory can contain files and other directories. Nothing in the tree is floating free — everything is attached at exactly one location.

### Absolute vs. Relative Paths

This is the concept that trips up more beginners than any other in this topic.

An **absolute path** always starts with `/`. It is the full address of a file or directory, measured from the root of the filesystem. It means exactly the same thing no matter where you are when you type it.

```text
/home/alice/documents/report.txt
```

A **relative path** does not start with `/`. It is directions from your current working directory. Its meaning changes depending on where you are.

```text
documents/report.txt
```

If you're standing in `/home/alice`, then `documents/report.txt` resolves to `/home/alice/documents/report.txt`. If you're standing in `/tmp`, then `documents/report.txt` looks for `/tmp/documents/report.txt` — which probably doesn't exist, and you'll get an error.

Two special relative path components come up constantly:

- `.` (a single dot) means "the current directory." `./script.sh` means "the file called `script.sh` in the current directory."
- `..` (two dots) means "the parent directory." `../other-project/` means "go up one level, then into `other-project`."

You can chain them: `../../etc/hosts` means "go up two levels, then look for `etc/hosts`."

### The Home Directory

Every user on a Linux system has a **home directory** — a personal directory where their files, configuration, and settings live. For a user named `alice`, the home directory is typically `/home/alice`. For the `root` user, it's `/root`.

The shell represents your home directory with the `~` (tilde) shorthand. `~/documents` is always equivalent to `/home/youruser/documents`, no matter where you are in the filesystem.

When you open a terminal or log in, your current working directory is set to your home directory automatically.

### Hidden Files

In Linux, any file or directory whose name begins with a dot (`.`) is considered hidden. "Hidden" is a weak word for this — it just means `ls` skips it by default. The file is there, fully accessible, fully functional. Tools that respect this convention simply don't show it unless asked.

Most configuration files and directories follow this convention: `.bashrc`, `.ssh/`, `.config/`. They're hidden because they're not files you work with directly every day — the noise reduction is intentional.

## How It Works (Mechanics)

### `pwd` — Where Are You?

The shell maintains a variable called `PWD` that tracks your current working directory. When you run `pwd`, it reads that variable and prints it. The `-P` flag resolves any symbolic links in the path and prints the physical path on disk — useful when your current directory was reached through a symlink.

```bash
# shows current directory, following symlinks if PWD contains them
pwd

# resolves all symlinks and shows the real physical path
pwd -P
```

### `ls` — What's Here?

`ls` calls `opendir()` and `readdir()` system calls to read directory entries from the kernel, then formats and prints the results. Without flags, it lists only the names of non-hidden files. The output order defaults to alphabetical by filename.

The flags you'll use constantly:

- `-l` — long format: permissions, link count, owner, group, size in bytes, modification timestamp, name
- `-a` — all: include hidden files (names starting with `.`)
- `-h` — human-readable sizes: shows `4.0K`, `1.2M`, `3.4G` instead of raw byte counts — only meaningful with `-l`
- `-R` — recursive: list the current directory and all subdirectories
- `-t` — sort by modification time, newest first
- `-S` — sort by file size, largest first
- `--color` — colorize output by file type (most distros alias `ls` to include this by default)

Flags combine: `ls -lah` is `ls -l -a -h` compressed into one argument.

### `cd` — Moving Through the Tree

`cd` changes the shell's current working directory by updating the `PWD` environment variable and calling the `chdir()` system call. It takes a path — absolute or relative — as its argument.

Three shortcuts you'll use constantly:

- `cd` with no argument: goes to your home directory (`~`)
- `cd -`: goes to the previous directory (the shell stores it in `OLDPWD`)
- `cd ..`: moves up one level to the parent directory

`pushd` and `popd` extend this: `pushd /some/path` changes to that directory while saving your current location on a stack. `popd` returns you to the saved location. You can push multiple directories and pop them in LIFO order — useful when you're jumping between several directories in a workflow.

### `tree` — Visualizing the Structure

`tree` recursively reads a directory tree and renders it as an ASCII diagram. It is not part of GNU coreutils — it is a separate package that may not be installed by default. See the Distro Notes section for installation commands.

Useful flags:

- `-L <n>` — limit depth to `n` levels (without this, `tree` on a large directory can produce thousands of lines)
- `-a` — include hidden files (names starting with `.`)
- `-d` — show only directories, not files

### `find` — Searching the Filesystem in Real Time

`find` walks the directory tree starting from a given path and evaluates expressions against every entry it encounters. It works in real time — no index, no cache, it actually reads the filesystem as it goes.

Core options for navigation:

- `-name "pattern"` — match by filename (case-sensitive; use `-iname` for case-insensitive)
- `-type f` — match only regular files; `-type d` for directories; `-type l` for symbolic links
- `-maxdepth n` — do not descend more than `n` levels below the starting point

### `locate` / `plocate` — Searching a Pre-built Index

`locate` searches a database of filenames that was built by `updatedb` (which typically runs nightly via cron). It's dramatically faster than `find` for filename searches because it reads an index instead of walking the filesystem. The trade-off: the index is only as fresh as the last `updatedb` run, so recently created files won't appear.

Modern Linux distributions have largely replaced `mlocate` (the older implementation) with `plocate`, which uses a compressed posting-list index and is significantly faster, especially on large filesystems. The command name (`locate`) stays the same — `plocate` installs a compatible `locate` binary.

## Basic Example

```bash
# See where you are right now
pwd
```

```text
/home/alice
```

```bash
# List files in the current directory — names only
ls
```

```text
Desktop  Documents  Downloads  Music  Pictures  Videos
```

```bash
# Long format, including hidden files, with human-readable sizes
ls -lah
```

```text
total 64K
drwxr-x--- 10 alice alice 4.0K Jun 10 14:22 .
drwxr-xr-x  4 root  root  4.0K May  3 09:11 ..
-rw-------  1 alice alice 3.4K Jun 10 14:22 .bash_history
-rw-r--r--  1 alice alice  220 May  3 09:11 .bash_logout
-rw-r--r--  1 alice alice 3.5K May  3 09:11 .bashrc
drwx------  2 alice alice 4.0K May  3 09:15 .ssh
drwxr-xr-x  2 alice alice 4.0K Jun  1 10:00 Desktop
drwxr-xr-x  3 alice alice 4.0K Jun  8 16:44 Documents
drwxr-xr-x  2 alice alice 4.0K May 20 11:30 Downloads
drwxr-xr-x  2 alice alice 4.0K May  3 09:11 Music
drwxr-xr-x  2 alice alice 4.0K May  3 09:11 Pictures
drwxr-xr-x  2 alice alice 4.0K May  3 09:11 Videos
```

Reading the `ls -lah` output column by column:

| Column | Example | Meaning |
|---|---|---|
| Type + permissions | `drwxr-xr-x` | `d` = directory; next 9 chars = owner/group/other permissions |
| Hard link count | `3` | number of hard links pointing to this inode |
| Owner | `alice` | the user who owns the file |
| Group | `alice` | the group associated with the file |
| Size | `4.0K` | file size (human-readable with `-h`) |
| Modification time | `Jun 8 16:44` | last time the file's contents were changed |
| Name | `Documents` | filename |

Notice `.` and `..` appear when you use `-a` — those are the current directory and its parent. They are real directory entries, not decorations.

```bash
# Move into Documents
cd Documents

# See that you moved
pwd
```

```text
/home/alice/Documents
```

```bash
# Go back to where you just came from
cd -
```

```text
/home/alice
```

Now try this yourself. Run `ls -lah` in your home directory and find the hidden files. If your output looks different — different usernames, different directories — that's completely normal. The structure is the same; only the names differ.

## Real-World Example

You've just SSH'd into a server you've never touched before. The first thing a real sysadmin does is orient themselves: where are they, what's running, what's the disk situation. Here's a navigation sequence that gives you that picture in under a minute.

```bash
# Step 1: Where am I? (always start here on an unfamiliar system)
pwd
```

```text
/root
```

```bash
# Step 2: What's in the root home directory? Any interesting scripts or notes?
ls -lah
```

```text
total 44K
drwx------  6 root root 4.0K Jun 12 03:14 .
drwxr-xr-x 20 root root 4.0K Jun  1 09:00 ..
-rw-------  1 root root 3.1K Jun 12 03:14 .bash_history
-rw-r--r--  1 root root 3.2K Jan 10 2023  .bashrc
drwx------  2 root root 4.0K Jun  1 09:00 .ssh
-rw-r--r--  1 root root  287 Jun  8 14:22 server-notes.txt
drwxr-xr-x  3 root root 4.0K Jun  1 09:00 scripts
```

```bash
# Step 3: Look at the structure of a key system directory without descending into everything
ls -lah /etc/ | head -30
```

```text
total 1.1M
drwxr-xr-x 112 root root  4.0K Jun 12 00:01 .
drwxr-xr-x  20 root root  4.0K Jun  1 09:00 ..
-rw-r--r--   1 root root  3.0K Jun  1 09:00 adduser.conf
drwxr-xr-x   2 root root  4.0K Jun  8 14:30 apt
...
```

```bash
# Step 4: Quickly see directory layout of /etc without all the file noise
tree -L 1 -d /etc
```

```text
/etc
├── apt
├── cron.d
├── cron.daily
├── default
├── init.d
├── logrotate.d
├── nginx
├── ssh
├── ssl
└── systemd
...
```

```bash
# Step 5: Find all config files for nginx specifically
find /etc/nginx -type f -name "*.conf"
```

```text
/etc/nginx/nginx.conf
/etc/nginx/sites-available/default
/etc/nginx/conf.d/gzip.conf
```

```bash
# Step 6: Find a specific script you know should be somewhere on the system
locate deploy.sh
```

```text
/root/scripts/deploy.sh
/opt/automation/deploy.sh
```

The walkthrough: `pwd` first, always — never assume you know where you are, especially after a complex series of `cd` commands or after a script changes directory. `ls -lah` on the home directory reveals scripts, notes, and config files left by previous admins. `tree -L 1 -d` on a directory like `/etc` gives you the shape of the configuration tree at a glance without flooding the screen. `find /etc/nginx -type f` limits the search to a specific subtree — far more targeted than `find /`. `locate` is the right tool when you know a filename but not its location and you want the answer immediately.

## ⚠️ Gotchas

### Absolute vs. Relative: The Path That Works Somewhere Else but Not Here

Every beginner does this. You copy a command from a tutorial — or from your own notes — and it fails with "No such file or directory." The command was correct. The problem is that it was correct in a different directory.

```bash
# You're in /home/alice
# You want to list files in your Documents folder
ls Documents
```

```text
Desktop  notes.txt  project
```

```bash
# You change directory for something else
cd /tmp

# Now you try the same command
ls Documents
```

```text
ls: cannot access 'Documents': No such file or directory
```

The command didn't change. Your location did. `Documents` is a relative path — it means "the `Documents` directory inside wherever I am right now." When you're in `/tmp`, that resolves to `/tmp/Documents`, which doesn't exist.

The fix is either to use an absolute path when you need the path to work regardless of location:

```bash
ls /home/alice/Documents
```

Or to use `~` to anchor it to your home directory:

```bash
ls ~/Documents
```

The rule: if a path starts with `/` or `~`, it works everywhere. If it doesn't, it depends on where you are.

### Hidden Files That Aren't Actually Hidden

```bash
# You copy your entire home directory somewhere, thinking you're getting everything
cp -r /home/alice /backup/alice
```

This silently skips `.ssh/`, `.bashrc`, `.bash_history`, and every other dotfile. Your "complete backup" is missing the SSH keys and shell configuration — the things that actually matter.

```bash
# The fix: use -a (archive) with cp, which preserves everything including dotfiles
# But you still need to list them first — ls -a shows what's really there
ls -a /home/alice
```

```text
.  ..  .bash_history  .bashrc  .ssh  Desktop  Documents  Downloads
```

The rule: dotfiles are not hidden from the system — they're hidden from casual viewing. Any tool that doesn't explicitly include hidden files will skip them. `cp` without flags, `tar` without the right options, `ls` without `-a` — all of them silently exclude dotfiles. Always check what you're actually working with.

### `find /` Without `-maxdepth` Will Scan Everything

```bash
# This looks fine. It isn't.
find / -name "nginx.conf"
```

This command will walk the entire filesystem tree, starting from root. On a typical server with tens of thousands of files, this takes 30 seconds to several minutes. On a system with network mounts, it can hang indefinitely trying to traverse a slow or unreachable NFS share. It also generates significant I/O load.

```bash
# Narrow it to where config files actually live
find /etc -name "nginx.conf"

# Or add maxdepth if you know roughly how deep it should be
find /etc -maxdepth 3 -name "nginx.conf"
```

The rule: always give `find` the narrowest starting path that still covers your search. `/etc` for config files. `/var/log` for logs. `/home` for user files. Reserve `find /` for when you genuinely don't know which part of the filesystem to search — and even then, add `-maxdepth 5` to avoid runaway traversal.

### `cd` With No Argument Is Not an Error

Beginners sometimes see `cd` typed alone and assume it's incomplete — a typo, a mistake. It isn't. `cd` with no argument is a well-defined shortcut that takes you home.

```bash
# No matter where you are...
cd /var/log/nginx

# ...this always takes you to your home directory
cd

pwd
```

```text
/home/alice
```

It's the equivalent of `cd ~`. You'll use it dozens of times a day once it becomes muscle memory.

### `tree` May Not Be Installed

On a fresh Ubuntu server, RHEL install, or Alpine container, `tree` is often absent.

```bash
tree
```

```text
-bash: tree: command not found
```

This is not a system error — it's a missing package. See the Distro Notes section for the install command for your system. On a system where you can't install packages, `find` with `-maxdepth` and `ls -R` can substitute.

### `locate` Returns Stale Results for New Files

```bash
# You just created a file
touch /tmp/new-important-script.sh

# And immediately search for it
locate new-important-script.sh
```

```text
# (no output)
```

The file exists. `locate` doesn't know about it because the index hasn't been rebuilt. The index is rebuilt nightly by `updatedb`.

```bash
# Force an immediate rebuild (requires root)
sudo updatedb

# Now it appears
locate new-important-script.sh
```

```text
/tmp/new-important-script.sh
```

The rule: `locate` is fast because it reads a pre-built index, not the live filesystem. For files created in the last 24 hours, use `find`.

## Distro Notes

### `tree` Installation

| Distro Family | Install Command |
|---|---|
| Debian / Ubuntu | `sudo apt install tree` |
| RHEL / Fedora / CentOS / Rocky | `sudo dnf install tree` |
| Arch | `sudo pacman -S tree` |
| Alpine (containers) | `apk add tree` |

### `locate` Implementation

| Distro Family | Default Implementation | Notes |
|---|---|---|
| Debian 11+ / Ubuntu 21.04+ | `plocate` | `plocate` package; `locate` command works as before |
| RHEL / Fedora 34+ | `plocate` | Fedora adopted plocate as default; older RHEL may ship `mlocate` |
| Arch | `plocate` | `plocate` package in `extra` repo |
| Alpine (containers) | neither installed | `apk add mlocate` or `apk add plocate` if available; `find` is more reliable in containers |
| Older systems | `mlocate` | `mlocate` package; same `locate` command, slower on large filesystems |

On any system, `locate --version` will tell you which implementation is running. `plocate` and `mlocate` are drop-in compatible — the `locate` command works identically from the user's perspective.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `find` | Real-time filesystem search; fresh files; matching by type, size, or date | Slower on large trees; extremely powerful with expressions |
| `locate` / `plocate` | Fast filename search; the file is not brand new; any system with `updatedb` running | Index may be up to 24h stale; `plocate` is the modern implementation |
| `ls -R` | Quick recursive listing when `tree` is unavailable | Output is harder to read than `tree` |
| `tree` | Visualizing directory structure at a glance | Not installed by default; not available everywhere |

The honest recommendation: `find` and `locate` are complementary, not competing. Use `locate` when you're searching for a file you know exists and has been there a while — it will answer in milliseconds. Use `find` when the file is new, when you need to filter by type or permissions or date, or when you're scripting something that needs to be reliable regardless of index freshness.

## When to Use / When to Avoid

✅ **Use absolute paths when:**
- Writing scripts — the script might be run from any working directory
- Specifying paths in config files — the service reading it may not have your working directory
- Sharing commands with others — they may be in a different location

✅ **Use relative paths when:**
- Working interactively in a directory you know you're in
- Referencing files adjacent to each other (e.g., `../config/settings.yaml` in a project)

✅ **Use `find` when:**
- The file was created recently (within the last `updatedb` cycle)
- You need to filter by type, permissions, size, or modification date — not just name
- You're writing a script that must work reliably without a pre-built index

✅ **Use `locate` when:**
- You want a fast, immediate answer for a filename that's been on the system for more than a day
- You're working interactively and don't need filtering beyond filename matching

❌ **Avoid `find /` without `-maxdepth` when:**
- You're on a production system — the I/O load can be visible
- The system has NFS or other network mounts — traversal can hang
- You're in a hurry — use a narrower path or `locate` instead

❌ **Avoid `locate` when:**
- The file was just created — the index won't have it
- You need to filter by anything other than filename (use `find`)
- You're in a container without `updatedb` running — `locate` will always return nothing or stale data

⚠️ **Common mistakes:**
- Running `ls` and concluding a file doesn't exist — it might be a hidden dotfile; run `ls -a`
- Using a relative path in a cron job — cron sets the working directory to `/`, so relative paths almost always fail; use absolute paths in cron
- Using `find` in a script without quoting the `-name` pattern — shell glob expansion can mangle it before `find` sees it; always quote: `find . -name "*.log"` not `find . -name *.log`

## Veteran Perspective

The thing that separates a beginner's command line session from a professional's is path discipline. Beginners navigate by feel — they `cd` somewhere, run a command, `cd` somewhere else. After ten minutes they've lost track of where they are, and they fix it by opening a new terminal. That's not a character flaw; it's what happens when you haven't internalized the model yet.

The shift happens when you stop thinking about the filesystem as a place you walk through and start thinking about it as a coordinate system. You're always at a specific point. Every path is either absolute (measured from the origin) or relative (measured from where you are). Once that clicks, you stop getting lost.

The `cd -` trick is genuinely useful and almost nobody teaches it. If you've just jumped from `/var/log/nginx` to `/etc/nginx` to check a config, `cd -` takes you back to `/var/log/nginx` in one keystroke. No path to remember. No history-scrolling. Build this into your muscle memory.

`pushd` and `popd` are the power-user version of `cd -`. If you're coordinating a deployment that involves bouncing between `/opt/app`, `/etc/app`, and `/var/log/app`, push each location onto the stack and pop your way back. It sounds like a niche tool; it becomes indispensable once you try it.

On the `ls` output: the number that most beginners ignore is the hard link count — the second column. For directories, this number is `2 + number of subdirectories`. That's because every directory has a `.` entry (pointing to itself) and a `..` entry in each subdirectory pointing back to the parent. If you see a directory with a link count of 5, it has 3 subdirectories. This is a minor curiosity now, but it becomes meaningful when you study **inodes** (covered in 2.4).

The `locate` / `find` choice matters in production. I've seen shell scripts that do `find / -name config.yaml` run in cron jobs — and those scripts would spike disk I/O every night, causing slowdowns during backup windows. The fix was a two-second change to `find /opt /etc -name config.yaml`. Know your search space.

One thing nobody tells you early enough: `find` without an action defaults to `-print`, which just lists matching paths. That's useful. But `find` can also `-delete`, `-exec`, `-move`, and do dozens of other things to every file it matches. The filesystem navigation skills you're building now are the foundation for those more powerful uses later.

## Try It Yourself

1. Open a terminal. Run `pwd` to see where you are. Then run `ls -lah` and count how many hidden files are in your home directory.

2. Navigate to `/etc` using `cd /etc`. Run `ls -lt` to see files sorted by modification time. What was the most recently changed file? Then run `cd -` to go back.

3. Use `tree -L 2 /etc` to see the top two levels of `/etc`. If `tree` isn't installed, install it first using the command from the Distro Notes section. How many subdirectories are there at depth 1?

4. Use `find /etc -maxdepth 2 -name "*.conf" -type f` to list all `.conf` files within two levels of `/etc`. How many did you find? Now remove `-maxdepth 2` and run it again — did more appear?

5. **Challenge:** Use `pushd` and `popd` to build a workflow: push to `/var/log`, push to `/etc`, push to `/tmp`. Check your directory stack with `dirs`. Now pop back to each location in order. Finally, combine what you know: use `find` from `/var/log` to find all files larger than 1MB using `find /var/log -type f -size +1M`. (This is safe to run on any system.)

## Summary

- Every path is either absolute (starts with `/`, works from anywhere) or relative (measured from your current directory, changes meaning when you move)
- `ls -la` is the standard way to see everything in a directory, including hidden dotfiles; `ls` alone skips them
- `cd` with no arguments takes you home; `cd -` takes you to the previous directory
- `tree` is not installed by default on most systems; `find -maxdepth` is a reliable fallback
- `find` searches the live filesystem in real time; `locate` searches a pre-built index that's typically updated nightly — use `find` for new files, `locate` for speed
