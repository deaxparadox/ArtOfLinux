## The Problem This Solves

Before package managers existed, installing software on Linux meant hunting down a tarball, extracting it, reading a README, running `./configure && make && make install`, and hoping none of the dependencies were missing. When a dependency was missing, you went and found that tarball too. And the one it depended on. This was called dependency hell, and it was a real thing that cost real hours.

Package managers solved the dependency problem: a central database knows what every package needs, and when you ask to install something, the tool resolves the whole chain automatically. That part is solved and has been for decades.

But most package managers create a new problem: everything in the official repositories has been reviewed, built, and signed by the distribution maintainers. That process takes time. A lot of time. On Debian, a package can sit in the queue for months before it lands in stable. If you want software that isn't in the official repo — say, a niche audio tool, an obscure CLI utility, or the latest version of something the distribution ships an older version of — you're on your own.

Arch Linux made a different bet. The official `core`, `extra`, and `multilib` repos are curated and maintained. But alongside them sits the Arch User Repository — the AUR — where any community member can publish a build recipe. No review gate. No maintainer approval. Just a shell script and the expectation that you'll read it before you run it.

pacman handles the official repos. The AUR requires either your own hands or an AUR helper. Understanding both, and where each one trusts you to know what you're doing, is what this topic covers.

---

## Mental Model

Think of the official Arch repos like a vetted grocery store: the food has been inspected, labeled accurately, and the expiration dates are tracked. You can walk in and trust the stock.

The AUR is a farmers market in the same parking lot. The produce is often fresher and more varied — things the grocery store doesn't carry. But there's no inspector. The person running the stall might be a local farmer who's been doing this for twenty years, or it might be someone who set up last week. Before you eat anything, you ask: where does this come from, and does this person seem trustworthy? The asking is your job, not the market's.

pacman is your cart. It works in the grocery store. The farmers market has its own checkout process — makepkg — and a set of helpers who'll carry your bags (yay, paru), but those helpers are built on top of your own judgment, not a replacement for it.

## Theory

### Arch is a Rolling Release

Most distributions have versions: Ubuntu 24.04, Fedora 41, Debian 12. You install a version and it's stable until the next major release. Packages are pinned at known-good versions. Upgrades between major versions are a defined event.

Arch has no versions. It is a rolling release: packages are updated continuously as upstream projects publish new releases. There is no "Arch 2026." There is just Arch, and either your system is up to date or it isn't.

This design has a critical consequence for how you must use pacman. Because every package is built against the current set of libraries on a fully updated system, there is no supported path for a partial upgrade. You cannot update one package while leaving others behind. If the new version of `firefox` links against `libssl 3.5` and your system still has `libssl 3.4`, the binary won't load. Arch's package model assumes your system is always fully synchronized. If it isn't, things break in subtle and difficult-to-diagnose ways.

This is not a bug. It's the deal you made when you installed Arch. The entire system must be upgraded as a unit, every time.

### The Package Database

pacman maintains a local database at `/var/lib/pacman/`. This database tracks every installed package: its name, version, list of files, dependencies, and the reason it was installed (explicitly by the user, or pulled in automatically as a dependency). When you run `pacman -Syu`, pacman syncs its local copy of the remote repository databases, computes what needs to change, downloads the required package archives (`.pkg.tar.zst` files), and installs them in dependency order.

The remote repository databases are a set of compressed files that pacman downloads to `/var/lib/pacman/sync/`. Each repo (core, extra, multilib) has its own database file. These files tell pacman what packages are available and what versions they're at. Running `pacman -Sy` syncs these databases. Running `pacman -Su` upgrades installed packages against the synced databases. The shorthand `pacman -Syu` does both in one command.

### What the AUR Is and Isn't

The AUR is not a package repository in the same sense as `core` or `extra`. It does not host compiled packages. It hosts PKGBUILDs.

A PKGBUILD is a bash script that describes how to build a package: where to download the source, what checksums to verify against, how to build it, and how to package the result into a `.pkg.tar.zst` file that pacman can then install. When you use the AUR, you are downloading a build recipe, running it on your own machine to compile the software, and then installing the resulting local package.

This means every AUR package is built from source on your machine. It also means that the PKGBUILD — a script executed by your shell during the build — runs with your privileges. There is no sandbox. If the PKGBUILD contains malicious code, that code runs.

The AUR trust model is explicit about this: the AUR is community-maintained, unreviewed, and the user is responsible for auditing what they run. This is not fine print. It is the central design decision, and ignoring it has real consequences — including a high-profile 2026 supply chain attack ("Atomic Arch") where over 400 orphaned AUR packages were backdoored with infostealers and an eBPF rootkit, delivered through modified PKGBUILDs.

## How It Works (Mechanics)

### pacman's Install Flow

When you run `pacman -S firefox`, pacman:

1. Reads the synced database at `/var/lib/pacman/sync/` to find the `firefox` package entry.
2. Resolves the full dependency tree — every package `firefox` requires, including transitive dependencies.
3. Checks which of those are already installed and at the right version.
4. Downloads the `.pkg.tar.zst` archives for everything that needs to be added or upgraded, verifying each file's PGP signature against the Arch Linux keyring.
5. Extracts each archive and places files into the filesystem, registering every installed file in the local database at `/var/lib/pacman/local/`.
6. Runs any post-install hooks defined in the packages.

The key operation is atomic at the level of a single package: either the package installs completely or pacman rolls it back. But a full upgrade is not atomic across all packages — if the system loses power mid-upgrade, the database can be left in a partially updated state.

### The makepkg Build Flow

When you build an AUR package manually:

```bash
# Clone the AUR package's git repository
git clone https://aur.archlinux.org/some-package.git
cd some-package

# Read the PKGBUILD before doing anything else
less PKGBUILD

# Build the package (and install dependencies from official repos)
makepkg -si
```

`makepkg` reads the PKGBUILD and executes it in stages:

1. **prepare()** — applies patches, runs pre-build setup.
2. **build()** — compiles the source (this is where `./configure && make` happens).
3. **check()** — runs the upstream test suite if present.
4. **package()** — copies build artifacts into a staging directory that mirrors the root filesystem.

After `package()` finishes, makepkg wraps the staging directory into a `.pkg.tar.zst` archive and optionally calls `pacman -U` to install it (`-i` flag). The `-s` flag tells makepkg to install any missing build dependencies from the official repos using pacman before starting the build.

The entire PKGBUILD executes as your user. The `sudo` calls inside it — if any — happen only when pacman is invoked for the final install step.

# Sync the repository databases and upgrade all installed packages
# This is the only correct upgrade command on Arch -- never -Su alone
pacman -Syu

## Real-World Example

This is the standard maintenance workflow on an Arch system — the commands you'd run during a weekly or monthly update session.

```bash
# Step 1: Full system upgrade -- always before installing anything new
pacman -Syu

# Step 2: Install a package you need
pacman -S neovim ripgrep fd

# Step 3: Query what package owns a specific file on disk
pacman -Qo /usr/bin/rg
```

```text
/usr/bin/rg is owned by ripgrep 14.1.1-1
```

```bash
# Step 4: List all files installed by a package
pacman -Ql ripgrep
```

```text
ripgrep /usr/
ripgrep /usr/bin/
ripgrep /usr/bin/rg
ripgrep /usr/share/
ripgrep /usr/share/bash-completion/
ripgrep /usr/share/bash-completion/completions/
ripgrep /usr/share/bash-completion/completions/rg
...
```

```bash
# Step 5: Find orphaned dependencies -- packages installed as deps
# that are no longer required by anything
pacman -Qdt
```

```text
python-charset-normalizer 3.4.1-1
python-urllib3 2.3.0-1
```

```bash
# Step 6: Remove orphans along with their config files and
# any further dependencies they uniquely introduced
pacman -Rns python-charset-normalizer python-urllib3
```

```bash
# Step 7: Inspect a package in the repo before installing it
pacman -Si neovim
```

```text
Repository      : extra
Name            : neovim
Version         : 0.10.4-1
Description     : Fork of Vim aiming to improve user experience, plugins, and GUIs
Architecture    : x86_64
URL             : https://neovim.io
Licenses        : Apache-2.0  custom:neovim
Groups          : None
Provides        : vim-plugin-runtime
Depends On      : libluv  libtermkey  libuv  libvterm  lpeg  luajit  msgpack-c  tree-sitter  unibilium

Conflicts With  : None
Replaces        : None

Build Date      : Fri 14 Mar 2026 08:12:44 AM UTC
Validated By    : MD5 Sum  SHA-256 Sum  Signature
```

The walkthrough: `pacman -Syu` is always first — you never install a new package onto a system that hasn't been upgraded, because the package was built against current library versions. `-Qo` is your answer to "where did this binary come from?" and you'll use it constantly in debugging. `-Qdt` followed by `-Rns` is routine housekeeping — the `n` flag removes leftover `.pacsave` configuration files, and `s` recursively removes anything that was only installed to support these packages.

## ⚠️ Gotchas

### Partial Upgrades Will Break Your System

The mistake most people make on their first month with Arch is installing a new package without upgrading first:

```bash
# What looks reasonable
pacman -S some-new-tool
```

```text
warning: some-new-tool: ignoring package upgrade (1.2.0-1 => 1.3.0-1)
```

Or worse — it installs fine, then something unrelated breaks two days later. You updated `some-new-tool`, which pulled in a newer version of `libfoo`. Now `other-program` links against an older `libfoo` that's no longer on your system. It crashes with a cryptic `error while loading shared libraries` message that gives you no indication that a package manager operation three days ago caused it.

The rule is absolute: **always run `pacman -Syu` before installing anything new.** Not sometimes. Every time. Arch's rolling release model has no mechanism to support a system where some packages are at current versions and others aren't.

```bash
# The correct form -- always
pacman -Syu some-new-tool
```

This syncs databases, upgrades everything, and installs the new package all in one transaction.

### `-Rns` vs `-R` — Know What You're Removing

Most guides just tell you to use `pacman -R`. That's incomplete.

```bash
# What most people use
pacman -R somepackage
```

This removes the package but leaves behind any configuration files it created (renamed with `.pacsave` extension) and leaves behind any packages that were only installed to satisfy this package's dependencies.

```bash
# What you should always use
pacman -Rns somepackage
```

The flags break down: `-R` removes the package. `-n` removes `.pacsave` files — leftover config backups that accumulate and confuse future reinstalls. `-s` recursively removes dependencies that are now unneeded. Without `-s`, your system gradually fills up with orphaned dependencies that `-Qdt` will later flag anyway.

The only time you'd skip `-n` is if you intentionally want to preserve configuration for a reinstall. That's rare enough that `-Rns` should be your default muscle memory.

### `pacman -Qdt` Misses Explicitly Installed Orphans

This one is subtle and causes confusion.

```bash
pacman -Qdt
```

This shows packages that were installed as dependencies and are no longer required by anything — the classic definition of an orphan. But it will **not** show packages that you installed explicitly (packages you manually asked for with `-S`) even if you no longer want them and nothing depends on them.

If you installed `nodejs` months ago to test something, forgot about it, and nothing depends on it, `pacman -Qdt` returns nothing for it. It thinks you want it because you asked for it directly.

```bash
# Shows all unrequired packages -- including explicitly installed ones
# that nothing depends on (the double t is intentional)
pacman -Qdtt
```

This shows both categories. Review the list carefully before removing — some packages you installed explicitly are things you still want even though nothing formally depends on them (standalone tools like `git`, `htop`, `vim`). Use judgment. Don't pipe the output directly to a removal command.

### PKGBUILDs Execute Arbitrary Code As Your User

This isn't just a theoretical risk.

```bash
# This looks like a normal install, but read what it actually does:
git clone https://aur.archlinux.org/some-aur-package.git
makepkg -si  # ← this runs the PKGBUILD as YOU
```

The PKGBUILD is a bash script. It can do anything your user account can do — read your SSH keys, exfiltrate files, install a backdoor. The 2026 "Atomic Arch" supply chain attack compromised over 400 AUR packages by taking over abandoned-but-trusted package maintainer accounts and modifying their PKGBUILDs to install malware. The attack worked because users trusted the package name without reading the current PKGBUILD.

The mitigation is not "don't use the AUR." The mitigation is: **read the PKGBUILD, every time, before running makepkg.** Read the `.install` file too if one exists. Check the `source` array — does it fetch from the expected upstream URL? Check the `package()` function — does it do anything unexpected?

```bash
# The correct AUR workflow
git clone https://aur.archlinux.org/some-package.git
cd some-package
less PKGBUILD          # read this before anything else
less some-package.install  # if it exists, read this too
makepkg -si            # only after you understand what will run
```

With AUR helpers like paru and yay, you get a diff view opportunity before building — don't skip it. Every helper offers a "view PKGBUILD" prompt. The one time you mash enter without reading is the one time you get compromised.

## Distro Notes

pacman and the AUR are specific to Arch Linux and Arch-based distributions. The behavior described in this topic does not apply to other distribution families.

| Distro Family | Package Manager | Notes |
|---|---|---|
| Arch Linux | `pacman` + AUR | Rolling release; pacman 7.1.0 as of 2026; AUR community-maintained |
| Manjaro | `pacman` (modified) | Arch-based but with a staged release model; holds packages back by ~2 weeks for stability testing; same pacman syntax, different update philosophy |
| EndeavourOS | `pacman` + AUR | Arch-based; closer to vanilla Arch than Manjaro; full rolling release |
| Debian / Ubuntu | `apt` + `dpkg` | Versioned releases; official repos only by default; PPAs are roughly analogous to AUR but with a different trust model |
| RHEL / Fedora | `dnf` + `rpm` | Versioned or semi-rolling (Fedora); no community-maintained user repository equivalent to the AUR |

## Alternatives & Comparisons

### AUR Helpers

| Tool | Language | Status | Notes |
|---|---|---|---|
| `paru` | Rust | Actively maintained | Feature-rich; shows PKGBUILD diffs by default; more robust dependency resolution for complex `provides`/`conflicts` arrays; recommended for new setups |
| `yay` | Go | Actively maintained | Longer history; widely documented; slightly faster startup due to Go compilation; still a solid choice; some community perception of slower development pace |
| `aurutils` | bash | Actively maintained | Lower-level; doesn't auto-install — gives you control over each step; preferred by users who want explicit control over what the helper does |
| Manual (`git` + `makepkg`) | — | Always works | Maximum transparency; no helper to trust; the fallback when helpers fail or for one-off installs |

Both paru and yay wrap pacman and provide AUR support with a pacman-like interface. The honest recommendation: use paru for new installs — it defaults to showing you PKGBUILD diffs before building, which is the security-correct behavior and means the safe path is also the default path. yay is equally functional if you already know it.

Neither helper changes the fundamental security model. An AUR helper is a convenience layer, not a trust layer. The responsibility to read the PKGBUILD is yours regardless of which helper you use.

## Trade-offs

**Where this shines:**

- The AUR gives Arch access to a staggeringly large set of packages — over 90,000 as of 2026 — covering niche tools, cutting-edge software, and proprietary packages that no official repository would carry.
- pacman's sync model is fast. Package queries against the local database are instantaneous. Full system upgrades on Arch are typically faster than equivalent operations on apt or dnf because there's no version-pinning overhead.
- The rolling release model means you always have current software. Security patches land quickly. You're never on a version that upstream has stopped supporting.

**Where this costs you:**

- Rolling releases break things. Not often, and usually not catastrophically, but occasionally an upgrade will require manual intervention — a changed configuration format, a kernel module that needs a rebuild, a package conflict that pacman can't auto-resolve. You need to know how to recover from this. Arch is not a set-and-forget server OS.
- The AUR's community-trust model is a genuine security surface. Every AUR package is a bet that the current PKGBUILD maintainer hasn't been compromised, made a mistake, or been replaced by an attacker. No automated review catches this.
- There is no LTS (Long Term Support) version of Arch. There's no "stable" branch to fall back to. If an upstream project makes a breaking change, it lands in Arch quickly.
- Partial upgrades are unsupported — you can't "pin" a package at an old version without breaking the rolling model. Holding back packages is possible but unsupported and will eventually require manual intervention.

## When to Use / When to Avoid

✅ **Use pacman and Arch when:**

- You want current software and are willing to do occasional manual maintenance to keep the system healthy.
- You need access to the AUR's breadth of packages.
- You understand rolling releases and can recover from a broken upgrade (knowing how to use a chroot from a live ISO, for example).
- You're building a personal workstation or development machine where control and recency matter more than stability guarantees.

❌ **Avoid Arch (and by extension pacman/AUR) when:**

- You're managing a production server that needs a defined maintenance window and reproducible system state — use RHEL, Rocky, or Debian stable instead.
- You need an OS that works unattended for months without upgrades — Arch will drift into partial-upgrade territory and things will break.
- You're responsible for a system other people depend on and you can't respond quickly when an upgrade needs manual intervention.

⚠️ **Common mistakes:**

- Installing AUR helpers system-wide via `sudo pip install` or similar — AUR helpers should be AUR packages themselves, installed through the AUR.
- Trusting an AUR package just because it has many votes or comments. Votes are not security review. The "Atomic Arch" attack targeted popular, high-vote packages specifically because users trusted them.
- Running `pacman -Sy` (sync only, no upgrade) before installing a package — this partially syncs the database but doesn't upgrade the installed packages, creating exactly the partial-upgrade condition you're trying to avoid.
- Editing `/etc/pacman.conf` to add unofficial repositories without understanding what you're trusting — third-party repos like `[archlinuxcn]` or `[chaotic-aur]` have their own trust models that are separate from the AUR.

## Veteran Perspective

The most common mistake I see from newcomers to Arch is treating it like a stable distribution that happens to have newer packages. It isn't. The rolling release model is a contract: you keep the system current, and in exchange you get the latest of everything. Break the contract — let the system go three months without `pacman -Syu` — and you will eventually have a bad time. Not because Arch is unreliable, but because you're trying to use it in a way it wasn't designed for.

The intermediate mistake is subtler: using Arch for servers. I've seen this multiple times. The reasoning is usually "I know Arch well and I want current software on my server." The problem surfaces six months later when an upgrade requires manual intervention and the server is unattended at 2am. Arch's rolling model requires an engaged operator. If you want "latest software on a server," use Fedora — it has a real release cycle and a support window.

The AUR deserves its own honest assessment. The breadth of packages is genuinely remarkable. I use it constantly. But the 2026 Atomic Arch attack wasn't a surprise to anyone who understood the trust model — it was the predicted failure mode of "community-maintained scripts that run as your user, unreviewed." The attack worked because users got comfortable. AUR helpers made the install flow smooth enough that the PKGBUILD review step started feeling like an optional nag, not a critical gate. That smoothness is the danger.

Read the PKGBUILD. Every time. Not just the first time you install a package — after updates too, because the PKGBUILD can change. `paru` shows you a diff of what changed since your last build. Use that feature. It takes thirty seconds and it's the only real protection you have.

The `/etc/pacman.conf` configuration is worth understanding once and then leaving alone. The options that actually matter in practice are `Color` (turn it on — the default is unreadable), `ParallelDownloads` (set to 5 or so for faster syncs), and the `[multilib]` repo section (uncomment it if you need 32-bit software, which you will if you run Steam or Wine). Everything else is advanced territory — unofficial repos, custom signing keys, `IgnorePkg` — that you should only touch when you know exactly why.

One thing the documentation undersells: `pacman -Qo /path/to/file` is one of the most useful diagnostic commands in Linux. When you find an unexpected binary on a production server and need to know what installed it, that's your first move. When a config file is being overwritten by an upgrade and you need to know which package owns it, that's your move. It works on any file under pacman's control, instantly, from the local database.

## Try It Yourself

1. Run `pacman -Ss neovim` to search for neovim in the official repos. Notice how the output shows the repository name, package name, version, and description. Then run `pacman -Si neovim` to see the full package info including its dependencies.

2. Run `pacman -Qdt` to list orphaned dependencies on your system. If you get output, look up each package to understand what originally needed it before removing anything.

3. Pick any binary in `/usr/bin/` — say `/usr/bin/ls` — and run `pacman -Qo /usr/bin/ls` to find out which package owns it. Then run `pacman -Ql coreutils` (or whatever package was returned) to see every file that package installed.

4. Open `/etc/pacman.conf` and find the `[multilib]` section. Read the comment above it explaining what it's for. Don't enable it unless you need it — this is just to understand what lives in the config and why.

5. **Challenge (requires a VM):** Find an AUR package you'd actually want (check `aur.archlinux.org`). Clone the git repo manually with `git clone https://aur.archlinux.org/<pkgname>.git`. Read the full PKGBUILD: understand the `source=()` array (where is it downloading from?), the `sha256sums=()` array (what is it verifying?), and the `package()` function (what files does it install and where?). Then build it with `makepkg -si`. Compare the experience to `pacman -S` — you now understand what AUR helpers abstract away.

---

## Summary

- Arch is a rolling release: there is no supported partial-upgrade path. Always run `pacman -Syu` before installing any new package — never `-S package` alone on an un-upgraded system.
- `pacman -Rns` is the correct removal command: `-n` removes leftover config files, `-s` removes unneeded dependencies. `-R` alone is incomplete.
- The AUR provides PKGBUILDs — shell scripts that build packages on your machine and run as your user. Read the PKGBUILD before every build; it executes arbitrary code with your privileges and can change between updates.
- `pacman -Qdt` shows dependency orphans; `pacman -Qdtt` also shows explicitly installed packages that nothing depends on. `pacman -Qo /path` tells you which package owns any file.
- paru and yay are AUR helpers that wrap the manual `git clone` + `makepkg -si` workflow. They are convenience tools, not security tools — the responsibility to audit PKGBUILDs remains yours.
