## The Problem This Solves

In the early days of Unix and Linux, software was distributed as source tarballs. Installing anything meant downloading a `.tar.gz`, reading a `README`, hoping all the required libraries were already installed, running `./configure && make && make install`, and then discovering that it installed files into fifteen different directories with no record of what went where. Uninstalling meant manually hunting down every file the build system had scattered across your filesystem — if you could even remember which version you'd installed or when.

The dependency problem was worse. A program might need library A, which needs library B, which needs a specific version of library C. Install them in the wrong order and nothing works. Install the wrong version of one and everything else breaks. This was called "dependency hell," and it was not a metaphor. It was a genuine engineering crisis that made Linux administration miserable.

Debian invented the `.deb` package format in 1993 to solve the installation side: every package contained the files to install, a list of what it depended on, and scripts to run before and after installation. That was `dpkg`. It solved the bookkeeping problem — you now had a database of every installed file — but it still couldn't automatically fetch and install dependencies. If you tried to install a package whose dependencies weren't present, `dpkg` just told you what was missing and stopped.

APT (Advanced Package Tool) was introduced in 1998 to solve the dependency-resolution problem. APT reads a list of repositories, builds a complete picture of what's available and what depends on what, and then fetches and installs everything a package needs — automatically, in the right order, from the internet. That combination — `dpkg` for low-level package operations and `apt` for high-level dependency resolution — is the foundation of every Debian-based system running today.

## Mental Model

Think of it as two layers at a hardware store.

The stockroom worker (`dpkg`) knows exactly what's on every shelf. When you hand them a box, they'll put it on the right shelf, record its location, and hand you the receipt. When you ask them to remove something, they'll find it and take it down. But if the item you brought in requires mounting hardware that isn't in stock, the stockroom worker will set it down and tell you: "you need these three other things first." They will not go get them for you.

The purchasing manager (`apt`) handles that part. You tell them what you want. They check the catalog, figure out everything that needs to come with it, place all the orders at once, and hand the stockroom worker a complete pile of boxes in the right installation order. You never talk to the stockroom directly — the purchasing manager handles it all.

You will occasionally need to talk to the stockroom worker directly. When something goes wrong at the low level — a package is half-installed, the database is corrupt, you have a `.deb` file you downloaded manually — that's when you reach past the purchasing manager and work with `dpkg` yourself.

## Theory

### The Two Layers

Every Debian-based system has two distinct package management layers, and conflating them is the source of most confusion.

**dpkg** is the low-level package manager. It maintains a database of installed packages at `/var/lib/dpkg/`. It knows what version of every package is installed, what files each package owns, and what each package's state is (installed, half-installed, unpacked, etc.). When you install a `.deb` file, `dpkg` extracts it, copies the files to the correct locations, runs the package's pre- and post-installation scripts, and updates the database. It does all of this without any network access and without any knowledge of where packages come from. It cannot fetch anything. It cannot resolve dependencies. It only operates on what you hand it directly.

**apt** is the high-level package manager. It knows about repositories — remote (or local) collections of packages described by an index. It reads your repository configuration, downloads the package index, builds a dependency graph of everything available, and uses a dependency solver to figure out exactly which packages need to be installed, upgraded, or removed to satisfy your request. Once it has a plan, it downloads the `.deb` files and hands them to `dpkg` to do the actual installation.

This separation is not an accident. It reflects Unix philosophy: one tool does one thing well. `dpkg`'s job is local package database management. `apt`'s job is remote repository management and dependency solving. The two talk to each other through well-defined interfaces, and you can replace or bypass either layer when you need to.

### The dpkg Database

The dpkg database lives in `/var/lib/dpkg/`. The two directories you'll interact with most are:

- `/var/lib/dpkg/info/` — one file per installed package, containing its list of installed files (`.list`), checksums (`.md5sums`), and maintenance scripts (`.preinst`, `.postinst`, `.prerm`, `.postrm`)
- `/var/lib/dpkg/status` — a flat-file database listing the name, version, dependencies, and state of every package dpkg knows about

The status file is plain text. If it gets corrupted, your package management system breaks. This is the file you back up before a risky operation.

### Repository Configuration

`apt` learns about available packages from repositories defined in two places:

- `/etc/apt/sources.list` — the legacy single-file configuration, still supported everywhere
- `/etc/apt/sources.list.d/` — a directory of `.list` files (legacy one-line format) or `.sources` files (modern DEB822 format); each third-party repository typically drops a file here

The traditional one-line format looks like this:

```text
deb [signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu noble stable
```

The modern DEB822 format (Ubuntu 24.04 defaults to this for new repositories added via `add-apt-repository`) uses a structured multi-line layout in `.sources` files:

```text
Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: noble
Components: stable
Signed-By: /etc/apt/keyrings/docker.gpg
```

Both formats remain fully supported. Ubuntu 24.04 ships its own system repositories in DEB822 format at `/etc/apt/sources.list.d/ubuntu.sources`. As of apt 2.9.26+, you can run `apt modernize-sources` to automatically convert `.list` files to `.sources` files.

### Repository Signing and Trust

Every repository in a modern APT configuration must be cryptographically signed. The package index is signed with a GPG key, and apt verifies that signature before trusting any package from that repository.

The correct modern approach is to download the repository's public key as a `.gpg` file and reference it in your repository definition with `Signed-By=`. Keys live under `/etc/apt/keyrings/` by convention. The `apt-key` command — which stored keys in a shared, global keyring at `/etc/apt/trusted.gpg` — was deprecated in Debian and Ubuntu because a key added with `apt-key add` was trusted for *all* repositories, not just the one it was intended for. A malicious repository key could theoretically sign packages from a different repository. The `signed-by=` approach scopes each key to a single repository.

### Package States and Selections

`dpkg` tracks each package's state. The states you'll encounter most often are:

- **install ok installed** — normal, fully installed package
- **install ok half-configured** — dpkg was interrupted during configuration (post-install scripts didn't finish)
- **install ok half-installed** — dpkg was interrupted during file extraction
- **deinstall ok config-files** — package was removed but configuration files remain (this is what `apt remove` produces vs. `apt purge`)

The "hold" selection prevents a package from being upgraded. A held package shows as `hold` in `dpkg --get-selections`. `apt-mark hold` sets this. Note the distinction between a held package and a pinned package: holds prevent upgrades via `apt upgrade`; pinning (via `/etc/apt/preferences`) controls which version apt will prefer and can enforce stronger constraints.

### unattended-upgrades

`unattended-upgrades` is the daemon that applies security updates automatically without human intervention. It is the right answer for production servers where you cannot afford to go weeks between security patches but also cannot afford an on-call engineer to run updates manually every night.

It is controlled by two files. `/etc/apt/apt.conf.d/20auto-upgrades` controls the schedule — how often apt updates its package lists and how often unattended-upgrades actually runs. `/etc/apt/apt.conf.d/50unattended-upgrades` controls what gets upgraded — which repository origins are allowed, whether the system reboots automatically after a kernel update, what email address receives notifications, and which packages are excluded.

## How It Works (Mechanics)

### What Happens When You Run `apt install`

When you run `apt install nginx`, the following sequence occurs:

1. **Dependency solver** — apt reads the cached package index (updated by `apt update`) and builds a dependency graph. It finds all packages `nginx` depends on that are not yet installed at the required version.
2. **Conflict check** — apt verifies that installing the new packages won't break any existing packages. If there's a conflict, apt will tell you and stop.
3. **Download** — apt downloads `.deb` files from the repository to `/var/cache/apt/archives/`.
4. **dpkg invocation** — apt calls `dpkg --install` on each `.deb` file in dependency order (dependencies before dependents).
5. **Pre-install scripts** — dpkg runs the package's `preinst` script if it has one (creates users, stops services, etc.).
6. **File extraction** — dpkg extracts the package's files to the filesystem.
7. **Post-install scripts** — dpkg runs the package's `postinst` script (enables services, runs `ldconfig`, triggers `debconf` configuration, etc.).
8. **Database update** — dpkg records the package as installed in `/var/lib/dpkg/status`.

`apt update` does *not* install anything. It downloads the package index from every configured repository and caches it locally at `/var/lib/apt/lists/`. This is the step that tells apt what versions are currently available. Running `apt install` without a recent `apt update` means apt may offer you outdated package versions.

### dpkg's Two-Phase Installation

dpkg uses a two-phase approach to protect against interrupted installations. First it unpacks the package to a temporary location and then atomically moves files into place. This is why you see both "unpacked" and "configured" states. If the system crashes after unpacking but before configuration, `dpkg --configure -a` picks up where it left off.

### APT Cache

`/var/cache/apt/archives/` is apt's download cache. Downloaded `.deb` files accumulate there and are reused if you reinstall a package. `apt clean` removes all cached `.deb` files. `apt autoclean` removes only `.deb` files for packages that are no longer in any repository (i.e., old versions). If you're low on disk space, `apt clean` is safe — the files will be re-downloaded from the repository if you need them again.

### debconf and dpkg-reconfigure

Many Debian packages use `debconf` to prompt for configuration during installation — things like "which timezone?" or "what should the root MySQL password be?" `debconf` stores these answers so that if you reinstall the package, it uses the previously entered values. `dpkg-reconfigure <package>` re-runs these prompts for an already-installed package, letting you change the configuration without uninstalling and reinstalling.

## Real-World Example

This is the sequence you'd run when adding a third-party repository the modern way — here, Docker's official repository on Ubuntu 24.04.

```bash
# Step 1: Install the tools needed to fetch and store the GPG key
sudo apt install -y ca-certificates curl

# Step 2: Create the keyrings directory if it doesn't exist
sudo install -m 0755 -d /etc/apt/keyrings

# Step 3: Download Docker's public signing key and store it in dearmored binary format
# curl -fsSL: fail silently on errors, follow redirects, suppress progress meter
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
    -o /etc/apt/keyrings/docker.asc

# Step 4: Restrict permissions — apt reads this file, but only root should write it
sudo chmod a+r /etc/apt/keyrings/docker.asc

# Step 5: Add the repository in DEB822 format, scoped to Docker's key
echo \
  "Types: deb
URIs: https://download.docker.com/linux/ubuntu
Suites: $(. /etc/os-release && echo "$VERSION_CODENAME")
Components: stable
Signed-By: /etc/apt/keyrings/docker.asc" | \
  sudo tee /etc/apt/sources.list.d/docker.sources > /dev/null

# Step 6: Update the package index to include the new repository
sudo apt update

# Step 7: Install Docker
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-compose-plugin
```

Walk through what happened here. The `Signed-By:` line in the `.sources` file tells apt to trust packages from this repository *only* if they're signed by the key at `/etc/apt/keyrings/docker.asc`. If Docker's signing key is ever compromised and they revoke it, you update that one file and apt immediately stops trusting anything signed with the old key. No other repository is affected.

The `$(. /etc/os-release && echo "$VERSION_CODENAME")` substitution inserts the Ubuntu release codename — `noble` for 24.04, `jammy` for 22.04 — so the same command works across Ubuntu versions without modification.

Now see how to pin Docker to its current version to prevent an unattended automatic upgrade from breaking production:

```bash
# Hold Docker at its currently installed version
sudo apt-mark hold docker-ce docker-ce-cli containerd.io

# Verify the holds are in place
apt-mark showhold
```

```text
containerd.io
docker-ce
docker-ce-cli
```

```bash
# When you're ready to upgrade Docker manually:
sudo apt-mark unhold docker-ce docker-ce-cli containerd.io
sudo apt update
sudo apt upgrade docker-ce docker-ce-cli containerd.io
sudo apt-mark hold docker-ce docker-ce-cli containerd.io
```

This is the pattern for any critical service: hold it, upgrade deliberately during a maintenance window, re-hold it. The hold prevents `apt upgrade` from touching it silently. Unattended-upgrades will also respect holds.

## ⚠️ Gotchas

### `apt upgrade` Silently Skips Held-Back Packages

You run `apt upgrade` and see a line like "3 packages were kept back." You assume they'll be upgraded next time. They won't. "Kept back" means apt wants to install a new version but can't because doing so would require removing or installing additional packages — and `upgrade` is forbidden from doing that. It's not the same as `apt-mark hold`. It's just `upgrade`'s self-imposed constraint that it will never touch the package count.

```bash
sudo apt upgrade
```

```text
Reading package lists... Done
Building dependency tree... Done
Reading state information... Done
Calculating upgrade... Done
The following packages have been kept back:
  linux-generic linux-headers-generic linux-image-generic
3 upgraded, 0 newly installed, 0 to remove and 3 not upgraded.
```

Those kernel packages are "kept back" because upgrading the kernel requires installing new packages alongside the old ones (the new kernel image and headers). `apt upgrade` will not do this.

```bash
# full-upgrade allows package removal and installation to complete the upgrade
sudo apt full-upgrade
```

`full-upgrade` (formerly `dist-upgrade`) is not dangerous by default — it just has broader permission to add and remove packages to satisfy the dependency graph. Run it with review: check what it says it will do before confirming. On a production server, run `apt full-upgrade --dry-run` first to see the plan without executing it.

### `dpkg -r` Leaves Config Files Behind

You remove a package to clean up the system. The config files stay. Six months later, you reinstall the package and it behaves oddly because it found old config files you forgot were there.

```bash
# -r removes the package but keeps config files in /etc/
sudo dpkg -r nginx
dpkg -l nginx
```

```text
rc  nginx  1.24.0-2ubuntu7  amd64  small, powerful, scalable web/proxy server
```

The `rc` prefix means: `r` = marked for removal, `c` = config files still present. The package is gone, but `/etc/nginx/` is still there.

```bash
# -P purges: removes the package AND its config files
sudo dpkg -P nginx
# Or use the apt alias, which is what you should reach for day-to-day:
sudo apt purge nginx
```

The distinction matters when you're trying to get a clean slate. If you `remove` and then reinstall, the package's postinstall scripts see existing config files and may skip generating fresh defaults. `purge` is the correct tool when you want a clean reinstall or when you're permanently removing a service from a system.

### `apt-key add` Is Deprecated — Keys Added This Way Are Too Permissive

Many installation guides still tell you to run `curl ... | sudo apt-key add -`. This stores the key in `/etc/apt/trusted.gpg`, the global trusted keyring. Any package signed by a key in that keyring is trusted regardless of which repository it comes from. A malicious actor who owns a legitimate repository key could potentially sign packages from another repository and have apt accept them.

```bash
# This pattern is deprecated and should never appear in new configurations:
curl -fsSL https://example.com/key.gpg | sudo apt-key add -
```

```text
Warning: apt-key is deprecated. Manage keyring files in trusted.gpg.d instead (see apt-key(8)).
OK
```

apt literally warns you. The correct pattern stores the key in a scoped location and references it with `signed-by=`:

```bash
# Download the key to a scoped location
sudo curl -fsSL https://example.com/key.gpg \
    -o /etc/apt/keyrings/example.gpg

# Reference it in the repository definition
echo "deb [signed-by=/etc/apt/keyrings/example.gpg] https://example.com/repo stable main" \
    | sudo tee /etc/apt/sources.list.d/example.list
```

If you have old keys from `apt-key add`, migrate them:

```bash
# List keys in the old global keyring
apt-key list

# Export a specific key (use the last 8 characters of the fingerprint)
sudo apt-key export AABBCCDD | sudo gpg --dearmor \
    -o /etc/apt/keyrings/example.gpg
```

Then update the repository's `.list` file to add `[signed-by=/etc/apt/keyrings/example.gpg]` and remove the key from the global keyring.

### `apt-mark hold` Does Not Survive `full-upgrade` Removing the Package

`apt-mark hold` prevents `apt upgrade` from upgrading a package. It does *not* prevent `apt full-upgrade` from removing a package that conflicts with something else being upgraded. If another package's new version declares a conflict with your held package, `full-upgrade` will offer to remove it.

```bash
sudo apt-mark hold critical-service
sudo apt full-upgrade
```

```text
The following packages will be REMOVED:
  critical-service
```

If you need a harder guarantee — "this package must never be removed or upgraded" — use APT pinning:

```bash
# /etc/apt/preferences or /etc/apt/preferences.d/critical-service.pref
sudo tee /etc/apt/preferences.d/critical-service.pref << 'EOF'
Package: critical-service
Pin: version 2.1.*
Pin-Priority: 1001
EOF
```

A priority of 1001 tells apt to prefer the installed version even over higher-version candidates from any repository. Combined with a version glob, this prevents apt from upgrading beyond the 2.1.x series or removing the package to satisfy a conflict. Pinning is the production-grade tool; `apt-mark hold` is a convenience shorthand for lighter situations.

### Forgetting `apt update` After Adding a Repository

You add a repository, try to install the package, and get "package not found." The repository was added, but apt's cached index hasn't been updated yet.

```bash
# Wrong — apt still doesn't know the new repository's packages exist
sudo tee /etc/apt/sources.list.d/example.list > /dev/null << 'EOF'
deb [signed-by=/etc/apt/keyrings/example.gpg] https://example.com/repo stable main
EOF
sudo apt install example-package
```

```text
E: Unable to locate package example-package
```

```bash
# Correct — always update after adding a repository
sudo apt update && sudo apt install example-package
```

The `apt update` step downloads the new repository's package index. Without it, apt has no idea the package exists.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian 12 (Bookworm) | Ships apt 2.6.x; uses traditional `/etc/apt/sources.list` format by default; DEB822 `.sources` format supported but not the default |

| Arch | Uses `pacman`, not apt; no `.deb` packages — covered in **8.3** |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `apt` | Interactive package management on Debian/Ubuntu | Designed for terminal use; progress bars, color output; output format may change between versions |
| `apt-get` | Scripts and automation | Stable, predictable output since 1998; officially recommended for scripts and `Dockerfile` instructions |
| `aptitude` | Complex dependency resolution, interactive TUI | More sophisticated conflict resolver than apt; useful when apt refuses to install something due to dependency conflicts; not installed by default |
| `dpkg` | Direct `.deb` manipulation, database queries | Reach for this when apt-level operations aren't what you need: installing a downloaded `.deb`, listing package contents, finding what package owns a file |
| `snap` | Application isolation, upstream-controlled updates | Entirely separate package system; packages run in a confinement sandbox; maintainer pushes updates directly; no apt involvement |
| `flatpak` | Desktop application isolation | Similar to snap; repository-based; common on Fedora/desktop systems; not a replacement for system packages |

For everything that involves managing system packages on a Debian/Ubuntu server, `apt` is the right tool interactively and `apt-get` is the right tool in scripts. `aptitude` is worth installing on any system where you'll do complex package troubleshooting — its conflict resolver can find solutions that `apt` gives up on. `dpkg` you'll use directly for specific diagnostic and low-level tasks, not as a replacement for apt.

## Trade-offs

**Where this shines:**

- Dependency resolution is genuinely solved. Installing any package in the Debian universe, no matter how many transitive dependencies it pulls in, happens in one command without manual intervention.
- The package database gives you a complete audit trail. `dpkg -l` tells you exactly what's installed and at what version. `dpkg -S /usr/bin/nginx` tells you exactly which package owns any file on the system.
- Repository signing means you get cryptographic assurance that every package you install came from a source you trusted when you configured the repository. You're not downloading random binaries; you're downloading signed artifacts from a chain of trust that goes back to the Debian or Ubuntu security team.
- `unattended-upgrades` closes the gap between "patches available" and "patches applied" on production systems where no one is checking daily.

**Where this costs you:**

- Debian stable deliberately ships older package versions. Stable repositories trade currency for stability — the version of PostgreSQL in Debian 12 stable is not the latest release. Adding third-party repositories solves this but introduces trust decisions.
- `apt full-upgrade` can remove packages. If you're not paying attention to the summary, it will silently remove something you care about. On a headless server with no monitoring of what's installed, this is a real operational risk.
- The APT configuration system has layered complexity: `sources.list`, `sources.list.d/`, `preferences`, `preferences.d/`, `apt.conf.d/` — there are a lot of files that affect behavior, and when something goes wrong it requires methodical checking to isolate which layer is misbehaving.
- `dpkg` state corruption — from a power loss during an install, or a filesystem error — requires careful manual repair. `dpkg --configure -a` handles the common case, but deep corruption requires editing `/var/lib/dpkg/status` by hand, which is not a comfortable place to be.

## When to Use / When to Avoid

✅ **Use `apt` when:**
- Installing, removing, or upgrading packages interactively on a Debian/Ubuntu system
- You want to see a human-readable summary of what will happen before confirming
- Doing a system upgrade (`apt upgrade` or `apt full-upgrade`)

✅ **Use `apt-get` when:**
- Writing shell scripts, `Dockerfile` instructions, or any automation that runs unattended
- You need predictable, parseable output that won't change format between apt versions
- Running in CI/CD pipelines — `apt-get install -y` with `-qq` for quiet output is the standard pattern

✅ **Use `dpkg` directly when:**
- Installing a locally downloaded `.deb` file (`dpkg -i`)
- Querying what files a package installed (`dpkg -L`)
- Finding which package owns a specific file (`dpkg -S /path/to/file`)
- Repairing interrupted installations (`dpkg --configure -a`)
- Inspecting installed package state and version (`dpkg -l`)

❌ **Avoid `dpkg -i` for packages from the internet:**
- `dpkg` doesn't fetch dependencies. If you `dpkg -i` a package that has unmet dependencies, you'll put your package system into a broken half-installed state. Use `apt install ./package.deb` instead — it resolves dependencies and then calls dpkg.

❌ **Avoid `apt-key add`:**
- It's deprecated and adds keys to the global trusted keyring. Use `signed-by=` with a scoped key file in `/etc/apt/keyrings/`.

❌ **Avoid running `apt upgrade` in automated scripts without review:**
- A kernel upgrade or a major package update can cause unexpected service restarts or compatibility breaks. For automation, scope the upgrade to specific packages or use `unattended-upgrades` with appropriate origin filters instead.

⚠️ **Common mistakes:**
- Running `dpkg -r` when you meant to purge — leaves config files that will confuse a future reinstall
- Adding a PPA and forgetting to `apt update` before installing
- Using `apt install -y` in scripts without pinning versions — the "latest" version of a dependency changes over time and can break your script months later
- Running `apt autoremove` without reviewing the list — it will remove packages that nothing explicitly depends on anymore, which can include tools you installed for one-off use but still want

## Veteran Perspective

The mistake I see most often from people new to Debian-based systems is treating `apt update` as optional. They add a repository, they try to install a package, it fails, they're confused. Every time. `apt update` is not an upgrade — it's a refresh of apt's knowledge of what exists. Your package manager is working from a cached snapshot of the world. If that cache is stale, its decisions are based on outdated information. Always update before you install.

The second thing: the difference between `apt upgrade` and `apt full-upgrade` trips up people who've used systems where there's only one upgrade command. On a system with held-back packages — and most production systems have them — `apt upgrade` will silently skip them and you will think your system is up to date when it isn't. `apt full-upgrade --dry-run` before a maintenance window is the habit that catches this. It shows you exactly what will happen, including any packages that will be removed, before you commit to anything.

Here's what the man page won't tell you about `apt-mark hold`: it's a suggestion, not a hard constraint. In my experience it works reliably for `apt upgrade`. It does not work reliably against `apt full-upgrade`. If you're holding a package because removing it would be catastrophic — a database server, a kernel module for custom hardware — use APT pinning with a priority of 1001. That's the mechanism that survives `full-upgrade`. I've watched engineers spend a full hour trying to figure out why a held package got removed during a routine maintenance window. It was `full-upgrade`. It always is.

The repository key migration away from `apt-key` is real and ongoing, and it matters more than it looks. In 2022, Debian and Ubuntu officially deprecated `apt-key`. If you're following a setup guide and it tells you to run `curl ... | apt-key add -`, that guide is outdated. The `signed-by=` approach isn't just newer — it's architecturally better. Scoped keys mean a compromised key at one vendor can't be used to poison packages from a completely different repository. That's the kind of trust boundary that prevents supply-chain attacks. The five minutes it takes to set up correctly is not optional on a production system.

On `dpkg --configure -a`: if you see packages in a "half-configured" or "half-installed" state, run this immediately. Don't ignore it. Don't work around it. Unresolved dpkg state propagates — the next time you try to install anything, apt will refuse because dpkg is in an inconsistent state. `dpkg --configure -a` runs the configuration scripts for any packages that were interrupted mid-install. It is almost always safe to run and it fixes the common case. The uncommon case — where it fails — means the package's `postinst` script is itself failing, and you need to look at the specific error message.

I've watched `unattended-upgrades` save servers and I've watched it break them. The difference was whether someone had configured it with actual thought or just left it at the defaults. The default configuration applies all security updates automatically. That is correct behavior for most systems. Where it bites you is when a security update for a library causes a behavioral change that breaks an application depending on that library — and no one finds out until a user hits an error at 2am. The mitigation is not to disable unattended-upgrades. The mitigation is to configure the email notifications, monitor the logs at `/var/log/unattended-upgrades/`, and test your applications against security updates before they hit production. `unattended-upgrades` is doing the right thing. The gap is lack of observability.

## Try It Yourself

1. Run `dpkg -l | head -30` to see the first thirty installed packages and their states. Look at the two-character state prefix on each line. Most should say `ii` (installed). If you see anything other than `ii`, look it up.

2. Pick any binary on your system — try `/usr/bin/find` or `/bin/bash`. Run `dpkg -S /usr/bin/find` to find which package owns it. Then run `dpkg -L <package>` to see every file that package owns.

3. Run `apt-cache policy nginx` (or any package you know is available). Read the output: it shows the installed version, the candidate version available in your repositories, and the priority of each configured repository. This is the command to run when you need to understand where apt will install a package from.

4. Add a hold on a package of your choice with `sudo apt-mark hold <package>`, verify it with `apt-mark showhold`, then remove the hold with `sudo apt-mark unhold <package>`. Practice the workflow before you need it in production.

5. **Challenge:** Configure `unattended-upgrades` on a VM to send email notifications. Install `mailutils` or `postfix`, edit `/etc/apt/apt.conf.d/50unattended-upgrades` to set the `Unattended-Upgrade::Mail` option to `root`, and run `sudo unattended-upgrade --dry-run --debug` to see what it would do without actually upgrading anything. Check `/var/log/unattended-upgrades/unattended-upgrades.log` after it runs.

## Summary

- `dpkg` operates on `.deb` files and the local package database — it handles installation, removal, and database queries but has no network access and no dependency resolution. `apt` wraps `dpkg` and adds repository management, dependency solving, and package downloading.
- Use `apt` interactively and `apt-get` in scripts. Their underlying behavior is the same; `apt-get` has a stable output format that won't break your automation when apt adds a new UI feature.
- `apt upgrade` never removes packages and silently skips packages it can't upgrade without changing the package count. `apt full-upgrade` can remove and install packages to complete upgrades — always run it with `--dry-run` first on production systems.
- `dpkg -r` removes a package but leaves config files; `dpkg -P` (and `apt purge`) removes everything. The difference matters when you want a clean reinstall.
- `apt-key add` is deprecated. Modern repository configuration uses `signed-by=/etc/apt/keyrings/<key>.gpg` to scope trust to a single repository. `apt-mark hold` is convenient but weak — use APT pinning (`/etc/apt/preferences`) when you need a hard constraint that survives `full-upgrade`.
