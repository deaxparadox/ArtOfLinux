# [8.5] — Containers and Software Distribution

## Why This Exists

You've written an application. It works on your laptop. You hand it to a colleague on a different machine — same Ubuntu version, even — and it crashes on startup. After an hour of debugging, you find it: their system has `libssl 3.0`, yours has `libssl 3.1`. A minor version difference. The API changed. Your binary was linked against the newer one.

This is "it works on my machine" at the operating system level, and it has been the central distribution problem in software for decades. Package managers help — they track dependencies and resolve conflicts — but they operate at the system level. Every application installed on the same machine shares the same libraries. If application A needs `libfoo >= 2.0` and application B needs `libfoo == 1.8`, you have a conflict with no clean resolution. You can have one or the other, or you start manually managing parallel installations, or you spin up a second machine.

Shipping to production made this worse. A developer's laptop ran Fedora 38. The CI server ran Ubuntu 22.04. The production machines ran RHEL 8 — an older glibc, different default Python, different OpenSSL. Every deployment was a compatibility negotiation. Teams wrote fragile shell scripts that tried to probe the host environment and install the right version of the right thing. Builds that passed CI still failed in production because CI used a different base image than production. On-call engineers spent weekend nights debugging environment differences rather than actual bugs.

The solution the industry settled on — after trying application bundles, virtualenvs, chroot jails, and a dozen other partial answers — was this: don't make the application adapt to the environment. Ship the environment with the application.

A container image bundles your code, your runtime, your libraries, and your OS filesystem in a single artifact. The container runtime on the host executes it in an isolated namespace. It doesn't matter if the host runs Ubuntu, RHEL, or Alpine — as long as it has a Linux kernel, your container runs identically. The "it works on my machine" problem goes away because the machine travels with the code.

## Mental Model

Think of a container image as a shipping container in international trade.

Before the standardized shipping container, cargo moved in whatever form it came in — barrels, crates, sacks. Every port had to know how to handle every type of cargo. Loading and unloading took days. Damage happened constantly because the cargo wasn't protected for its journey.

The standardized shipping container changed everything. The cargo — whatever it is — gets packed inside a sealed metal box with standard dimensions. Every port, every ship, every crane handles the same box. The contents are irrelevant. The interface is standardized.

A container image is the software equivalent. Your application, its runtime, its libraries, its configuration — all packed into a layered archive with a standardized format. Every container runtime — Docker, containerd, Podman — handles the same format. The host doesn't need to know your application uses Python 3.12 and a specific version of libpq. It just runs the box.

The analogy holds further: just like shipping containers can be stacked and reused, container image *layers* are reused. If two images both start from `ubuntu:22.04`, the runtime downloads that base layer once and shares it.

## Theory

### The OCI Standard

Before containers had a standard, Docker had a format and everyone else had to reverse-engineer it. In 2015, Docker, CoreOS, and others formed the Open Container Initiative (OCI) under the Linux Foundation to define portable, vendor-neutral specifications. Today OCI defines both an image format and a runtime specification. The current image spec version is **v1.1.1** (released March 2025).

An OCI image layout is a directory on disk with a specific structure:

- `oci-layout` — a JSON file at the root declaring the layout version (`{"imageLayoutVersion": "1.0.0"}`)
- `index.json` — the top-level Image Index; entry point that references all image manifests by cryptographic digest
- `blobs/<algorithm>/<digest>` — a content-addressable blob store where every component (manifests, configs, and layer tarballs) lives as a file named by its SHA-256 hash

This content-addressable design is deliberate. Because every blob is named by its hash, any modification changes the name. There's no such thing as a tampered image that still has the same digest — the math prevents it. You can verify an image you pulled from the internet against a known digest and trust it completely.

### The Image Manifest

An OCI image manifest ties the pieces together. It is itself a blob stored in `blobs/sha256/`, referenced from `index.json`. It has three required fields:

- `schemaVersion: 2` — always
- `config` — a descriptor (digest + size + mediaType) pointing to the container config blob
- `layers` — an ordered array of layer descriptors

The **container config** is the JSON object that describes everything the runtime needs to run the container: the command to execute, environment variables, the user to run as, exposed ports, and the filesystem changeset history.

The **layers** are the actual filesystem content — gzipped tarballs (or zstd-compressed, a newer option) containing the files that make up the container's root filesystem. They are applied in order, each building on top of the last.

### The Layer Model

Container images are not flat archives. They are stacks of deltas, applied in order to produce a complete filesystem.

Start with a base image — say, `ubuntu:22.04`. That image might have three layers: a core filesystem layer, a layer that adds package metadata, and a layer that applies security patches. Each layer is a tar archive containing only the files that changed from the previous state.

When you build your application on top, your `Dockerfile` adds more layers:

```text
# Conceptual layer stack — not a real command
Layer 0: ubuntu:22.04 base filesystem     ← shared across many images
Layer 1: ubuntu:22.04 package updates     ← shared across many images
Layer 2: RUN apt-get install python3      ← shared by anything based on python3
Layer 3: COPY requirements.txt .          ← specific to your app
Layer 4: RUN pip install -r requirements.txt
Layer 5: COPY . .                         ← your actual application code
```

The runtime assembles these layers into a unified view using a **union filesystem** (OverlayFS on modern Linux kernels). OverlayFS presents multiple directories as a single merged view, with upper layers overriding lower ones when files conflict. The merged view is what the container sees as `/`. The original layers are never modified — writes from inside a running container go to a thin writable layer on top, discarded when the container exits unless you use volumes.

This layer model has a direct implication for image size and build speed that every developer eventually learns the hard way.

### The "latest" Tag Problem

Every container image in a registry has a tag. Tags are mutable human-readable names that point to a specific image digest. The tag `ubuntu:22.04` and the tag `ubuntu:latest` are just pointers. The pointer can be moved to a different image at any time — and `latest` almost always is moved, on every new release.

When you write `docker run ubuntu:latest`, you're saying "give me whatever the maintainer currently calls latest." Today that means Ubuntu 24.04. Six months from now it might mean Ubuntu 26.04. The actual image digest changes. If you're deploying software, pinning to `latest` means your deployment is non-deterministic. You have no guarantee that what runs in production tomorrow is the same image that passed your CI tests today.

The correct practice is to pin to either a specific tag (`ubuntu:22.04`) or, for maximum reproducibility, a specific digest (`ubuntu@sha256:...`). A digest is immutable — it cannot be reassigned. The image it names is the exact image, forever.

### Static Binaries — the Furthest End of Self-Contained

Container images solve the distribution problem at the environment level. Static binaries solve it at the binary level — they eliminate external dependencies entirely by embedding everything needed at link time.

A **static binary** is an ELF executable that carries all its library code inside itself. No `.so` files needed. `ldd` returns "not a dynamic executable." The binary runs on any compatible kernel, regardless of what shared libraries are installed on the host.

**Go** produces static binaries with a single environment variable:

```bash
CGO_ENABLED=0 go build -o myapp .
```

When `CGO_ENABLED=0`, Go disables its C foreign-function interface entirely and uses pure-Go fallback implementations for the parts of the standard library (`net`, `os/user`) that would otherwise call libc. Go's own linker compiles and links the entire runtime — including the goroutine scheduler and garbage collector — directly into the output ELF. A minimal Go binary is roughly 1.2 MB.

**Rust** can also produce static binaries, but the path is different. By default, Rust links against glibc, which has a problematic quirk: even if you specify static linking (`-C target-feature=+crt-static`), glibc performs DNS lookups and character set conversions via `dlopen()` at runtime — loading shared libraries even in a "statically linked" binary. This makes truly static glibc binaries unreliable in non-trivial configurations.

The solution is to compile against **musl libc** instead. musl is a C standard library for Linux (MIT license) that was designed from the start to support complete static linking. Unlike glibc, musl contains its own implementations of DNS resolution and character set conversion — no `dlopen()` at runtime, ever.

```bash
rustup target add x86_64-unknown-linux-musl
sudo apt install musl-tools          # provides the musl-gcc wrapper (Debian/Ubuntu)
cargo build --target x86_64-unknown-linux-musl --release
```

The resulting binary has no glibc version dependency. It runs on any x86_64 Linux, including containers built `FROM scratch` — a completely empty base image with nothing but your binary.

Static binaries are the extreme end of self-contained distribution. They're common in Go and Rust tooling ecosystems, in container base images, and anywhere where you want zero runtime dependencies. They do come with costs: larger binary size (everything is included), no sharing of library code between processes, and security patches to bundled libraries require a full rebuild and redeploy.

### Container Registries

Container images need somewhere to live between being built and being run. That's a **container registry** — an HTTP server that stores and serves OCI image manifests and blobs, indexed by repository name and tag.

The three dominant public registries are:

**Docker Hub** (`hub.docker.com`) is the oldest and most widely referenced. It's the default registry for `docker pull` and most tooling — when you write `FROM ubuntu:22.04` in a Dockerfile with no explicit registry, Docker Hub is implied. Docker Hub rate-limits unauthenticated pulls (100 per 6 hours per IP as of mid-2026) and requires a paid plan for unlimited pulls. Official images (the ones with names like `ubuntu`, `nginx`, `python`) are curated and maintained by Docker or the upstream project.

**GitHub Container Registry** (`ghcr.io`) is tightly integrated with GitHub Actions and GitHub repositories. Public images are free with no rate limits on anonymous pulls. It launched in 2020 and has become the default for open-source projects hosted on GitHub. The older `docker.pkg.github.com` registry was shut down in February 2025 and is no longer accessible.

**Quay.io** (`quay.io`) is operated by Red Hat. It runs the open-source Project Quay registry software and includes a built-in Clair vulnerability scanner that automatically scans images on push. It's the default for Red Hat and CentOS Stream container images and is common in enterprise environments.

For production, you'll typically run a private registry (either self-hosted Quay, Harbor, or a managed service like AWS ECR, GCP Artifact Registry, or Azure Container Registry) rather than pushing your production images to a public registry.

### Image Signing and Supply Chain Security

Pulling an image from a registry gives you a collection of bytes. Without signing, you have no cryptographic proof that those bytes are what the publisher intended — or that they haven't been tampered with in transit or storage.

**Sigstore** is an open-source software supply chain security project under the Open Source Security Foundation (Linux Foundation). Its flagship tool for containers is **cosign**, which signs and verifies OCI images without modifying the original image digest. Signatures are stored as separate OCI artifacts in the same registry, attached to the original image via the OCI 1.1 Referrers API (or a deterministic tag on older registries).

Sigstore's most significant innovation is **keyless signing**, which eliminates the problem of long-lived private keys:

1. An ephemeral keypair is generated in memory at signing time
2. The signer authenticates via OIDC (Google, GitHub, or Microsoft identity)
3. **Fulcio** (Sigstore's code-signing CA) verifies the OIDC token and issues a short-lived certificate that binds the signer's identity to the ephemeral public key — the certificate lives for minutes
4. The signing event is logged to **Rekor**, Sigstore's public, immutable transparency log
5. The ephemeral private key is discarded

Verification then works on identity, not key material: "this image was signed by the GitHub Actions workflow in this repository." This is a fundamentally different trust model — it ties provenance to the CI/CD identity rather than to a private key that can be leaked, rotated, or revoked.

Container images are software supply chain artifacts. An unverified image you pulled from a public registry is an unverified artifact. Image signing is the mechanism that turns "I got this image from this registry" into "I can prove this image was produced by this pipeline."

## How It Works (Mechanics)

When you run `docker run nginx:1.27`, several distinct operations happen in sequence:

```text
# Container startup flow — conceptual trace
1. docker CLI parses "nginx:1.27", defaults registry to Docker Hub
2. dockerd checks local image store for nginx@sha256:<manifest digest>
3. If not cached:
   a. Resolve tag "1.27" → specific manifest digest via registry HTTP API
   b. Fetch manifest blob (small JSON)
   c. Read layer list from manifest
   d. Fetch each layer blob (gzipped tarballs) in parallel
   e. Store blobs content-addressed in /var/lib/docker/overlay2/
4. dockerd instructs containerd to create a container from the image
5. containerd tells runc to:
   a. Set up OverlayFS — merge all image layers into one unified view
   b. Create Linux namespaces: pid, net, mnt, uts, ipc
   c. Apply cgroup limits if specified
   d. exec() the entry point (nginx -g "daemon off;")
6. Container process runs in its isolated namespace environment
```

The key pieces are:

**OverlayFS** — the union filesystem that presents the stacked image layers as a single directory tree. It has two components: `lowerdir` (the read-only image layers) and `upperdir` (a writable scratch layer). The container process sees a merged view. Any file it writes goes to the upperdir. Any file it reads that it hasn't written comes transparently from the lowerdir. When the container exits, the upperdir is discarded unless you have a volume mounted at that path.

**Linux namespaces** — the isolation mechanism. Containers are not VMs. There is no separate kernel. The container process runs on the host kernel. Namespaces partition kernel resources: `pid` namespace gives the container its own process tree (PID 1 is the container's init), `net` namespace gives it its own network stack, `mnt` namespace gives it its own filesystem view. From inside, the container sees a complete Linux system. From outside, the host sees a regular process tree.

**runc** — the OCI-compliant low-level runtime that actually calls the kernel. It reads an OCI bundle (a rootfs directory and a `config.json`) and invokes the Linux syscalls to create namespaces, configure cgroups, set up OverlayFS, and exec the entry point. Docker and containerd both delegate to runc for the final container creation step.

The Docker daemon layer (`dockerd`) sits above containerd and provides the higher-level Docker-specific features: image builds, networking management, volumes, and the Docker CLI API. containerd handles image lifecycle (pull, store, snapshot) and container lifecycle (create, run, stop). runc handles the actual Linux primitives.

Try this yourself. Once it's running, open `http://localhost:8080` in a browser. If you see the nginx welcome page, everything is working — a web server started and served content without touching anything on your host system except a port mapping. When you stop the container, the web server is gone. No trace left behind. No `apt remove nginx` needed.

## Real-World Example

This scenario is how container images appear in actual production: a Node.js application packaged for deployment, demonstrating the layer ordering principle and multi-stage builds to control image size.

```bash
# Bad Dockerfile — do not copy this; it's shown to illustrate the anti-pattern
# Every code change rebuilds the npm install layer (expensive, slow)
```

```text
# anti-pattern: COPY . . before RUN npm install
FROM node:20-alpine
WORKDIR /app
COPY . .                        ← copies all source files first
RUN npm install                 ← installs dependencies AFTER source copy
CMD ["node", "server.js"]
```

Every time a developer changes a single line of `server.js`, Docker invalidates the cache at the `COPY . .` step and re-runs `npm install` from scratch — downloading all dependencies every build. On a project with hundreds of npm packages, that's minutes of build time for a one-line change.

The correct approach separates the dependency layer (which rarely changes) from the source layer (which changes constantly):

```text
# Correct layer ordering — dependencies cached independently of source changes
FROM node:20-alpine
WORKDIR /app
COPY package.json package-lock.json ./   ← only the manifest files
RUN npm ci --only=production             ← install deps; this layer caches until
                                         #  package-lock.json changes
COPY . .                                 ← source code; cache invalidated here
CMD ["node", "server.js"]
```

Now `npm ci` only re-runs when `package-lock.json` actually changes — that is, when you add or update a dependency. A code-only change hits the cache through the `npm ci` layer and only re-executes `COPY . .` and later, taking seconds instead of minutes.

For production, you'd typically add a multi-stage build to exclude development tooling from the final image:

```text
# Multi-stage build: build stage has full toolchain; final image is minimal
FROM node:20-alpine AS builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
COPY . .
RUN npm run build                        ← compile TypeScript, bundle assets, etc.

FROM node:20-alpine AS final             ← fresh base; only production artifacts
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package.json .
CMD ["node", "dist/server.js"]
```

The `builder` stage contains the TypeScript compiler, dev dependencies, and build tooling. None of that ends up in the `final` image — only the compiled output and production dependencies. The final image is significantly smaller and has a smaller attack surface. Build tooling is a common vector in compromised container images; excluding it from the image that runs in production is a basic hardening step.

To build and inspect the image:

```bash
docker build -t myapp:1.0.0 .
docker image ls myapp
```

```text
REPOSITORY   TAG     IMAGE ID       CREATED          SIZE
myapp        1.0.0   d4f2a8c1e9b3   12 seconds ago   98MB
```

```bash
# Inspect the image layers — shows exactly what each layer added
docker history myapp:1.0.0
```

```text
IMAGE          CREATED          CREATED BY                                      SIZE
d4f2a8c1e9b3   12 seconds ago   CMD ["node" "dist/server.js"]                   0B
<missing>      12 seconds ago   COPY /app/node_modules ./node_modules            45MB
<missing>      12 seconds ago   COPY /app/dist ./dist                            2MB
...
```

`docker history` is the right tool when debugging why an image is larger than expected. Each row is a layer; the SIZE column tells you which step is contributing bulk.

## ⚠️ Gotchas

### `latest` Is Not a Version Pin

The assumption: `FROM ubuntu:latest` means "the latest stable Ubuntu, always reproducible."

```bash
# This Dockerfile produces a different image on different build dates
docker build -t myapp .
```

```text
# Today: pulls ubuntu:24.04 (Noble Numbat)
# After Ubuntu 26.04 releases and maintainers update the tag: pulls ubuntu:26.04
```

The `latest` tag is a mutable pointer. Docker Hub maintainers move it to whatever their current release is. When you build against `latest`, you may get a different base image than what passed your tests six months ago. The problem is silent — the build succeeds, the image exists, and you find out something changed when behavior is different in production.

Use explicit tags:

```bash
# Reproducible: this exact tag resolves to the same Ubuntu release
FROM ubuntu:24.04

# Maximum reproducibility: pin to a digest — cannot be reassigned
FROM ubuntu@sha256:a6d2b38300ce0316684c9379e066e4f8b24e60e4acbbf8bb453d1fefd3eef79f
```

The digest form is the only guarantee. Tags can be updated by maintainers. Digests are content-addressed and immutable.

### A Container Is Not a VM — Root Inside Is Root

The assumption: "I'm inside a container, so it's sandboxed. I can run as root safely."

Containers are not virtual machines. There is no separate kernel. Container processes run on the host kernel, isolated by namespaces. If a container is running as root and a vulnerability in the containerized software allows escape from the namespace isolation — or if the container is configured to mount the host filesystem — root inside the container is root on the host.

```bash
# This is dangerous — mounting the host Docker socket gives the container
# full control over the Docker daemon (and therefore the host)
docker run -v /var/run/docker.sock:/var/run/docker.sock myimage
```

The correct posture is to run container processes as non-root users:

```text
# In a Dockerfile
RUN addgroup --system --gid 1001 appgroup && \
    adduser --system --uid 1001 --gid 1001 appuser
USER appuser
```

```bash
# At runtime — override the image's default user
docker run --user 1001:1001 myimage
```

An image running as UID 0 inside a container is a security risk even with namespace isolation. Namespace escapes are rare but not theoretical — CVE-2019-5736 (runc), CVE-2020-15257 (containerd shim), CVE-2022-0847 (Dirty Pipe) all involved kernel-level vulnerabilities that had container escape implications. Defense in depth means: non-root user inside the container, minimal capabilities, and no unnecessary host mounts.

### Layer Ordering Determines Rebuild Cost

The assumption: "It doesn't matter what order I put the `COPY` and `RUN` commands — Docker will figure it out."

```text
# This re-runs npm install on every source code change:
COPY . .                  ← source files copied first
RUN npm install           ← runs after COPY; any source change invalidates this
```

Docker's build cache is invalidated when any layer changes. A layer change means the content of the files being copied changed, or a `RUN` command's input changed. Everything after the changed layer is re-built. If you copy your source code before installing dependencies, then any code change forces a full dependency reinstall.

```text
# Correct: dependencies installed before source is copied
COPY package.json package-lock.json ./
RUN npm install           ← only re-runs when package-lock.json changes
COPY . .                  ← cache miss here doesn't trigger npm install
```

The rule: put the things that change least at the top of the Dockerfile, the things that change most at the bottom.

### Large Base Images Carry Hidden Vulnerabilities

The assumption: "I'm using the official image from Docker Hub, so it's secure."

Official images are not audited for vulnerabilities on an ongoing basis — they're built at a point in time. A `node:20` image contains Ubuntu packages from whenever it was last rebuilt. Those packages may have known CVEs by the time you use the image.

```bash
# Scan an image for vulnerabilities before using it
docker scout cves node:20

# Or use trivy (common in CI pipelines)
trivy image node:20
```

```text
Total: 47 (HIGH: 12, MEDIUM: 23, LOW: 12)
┌──────────────────┬───────────┬──────────┬──────────────────┬───────────────────┐
│     Library      │  Vuln ID  │ Severity │ Installed Version│   Fixed Version   │
├──────────────────┼───────────┼──────────┼──────────────────┼───────────────────┤
│ libssl3          │ CVE-2024- │ HIGH     │ 3.0.2-0ubuntu1   │ 3.0.2-0ubuntu1.14 │
...
```

The mitigation has two parts. First, use minimal base images: `node:20-alpine` (based on Alpine Linux) or Google's **distroless** images (which contain only the runtime — no shell, no package manager, no utilities). Smaller images have fewer packages and therefore fewer potential vulnerabilities. Second, run a vulnerability scanner (Trivy, Grype, Docker Scout) in your CI pipeline so you're notified when a new CVE affects your base image, even if you didn't change your Dockerfile.

## Distro Notes

Container tooling installation varies by distro family:

| Distro Family | Docker Install | Podman Install | Notes |
|---|---|---|---|
| Debian / Ubuntu | `apt install docker.io` (distro) or Docker's own repo (`docker-ce`) | `apt install podman` | Docker's official `docker-ce` package is newer than `docker.io` from distro repos; prefer it for production |
| RHEL / Fedora / CentOS / Rocky | Not included by default; use Docker's RHEL/Fedora repo | `dnf install podman` (pre-installed on RHEL 8+) | Red Hat ships Podman as the default container tool; Docker requires a separate repo |
| Arch | `pacman -S docker` | `pacman -S podman` | Both in community repo; Docker daemon must be started: `systemctl enable --now docker` |
| Alpine (containers) | Alpine is commonly used as a container *base image*; Docker CLI: `apk add docker` | `apk add podman` | Alpine's musl libc makes it ideal for hosting statically-linked binaries |

The Docker socket (`/var/run/docker.sock`) is owned by the `docker` group. To run Docker commands without `sudo`, add your user to that group: `sudo usermod -aG docker $USER`, then log out and back in.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| Docker (`docker`) | Local development, developer workstations, teams already using Docker Compose | The most widely documented runtime; `docker compose` is the standard for local multi-container development; Docker daemon runs as root (security concern for servers) |
| Podman (`podman`) | RHEL/Fedora servers, rootless container environments, security-conscious deployments | Daemonless — each invocation is a direct process, no long-running daemon; rootless by default; drop-in CLI replacement for `docker` (`alias docker=podman` works for most use cases) |
| containerd (`nerdctl`) | Kubernetes infrastructure, environments where you want Docker-compatible CLI without the Docker daemon | containerd is the container runtime underneath Kubernetes (~95% of production clusters); `nerdctl` provides Docker-compatible CLI; use when you want to match production Kubernetes behavior locally |
| Static binary (no container) | Single-binary CLI tools, minimal Lambda/serverless deployments, `FROM scratch` base images | The extreme of self-contained; works when your application has no external runtime dependencies and you don't need OS-level isolation; Go and Rust are the primary ecosystems |

For local development, Docker remains the practical default — the documentation, community resources, and `docker compose` tooling are unmatched. For production Kubernetes, you won't interact with the container runtime directly; the kubelet handles it. For RHEL-based production servers running containers outside Kubernetes, Podman's rootless default is the right security posture.

## Trade-offs

**Where this shines:**

- **Dependency isolation**: your application runs with exactly the libraries it was tested against, regardless of what's installed on the host
- **Reproducibility**: the same image, built once, behaves identically across development, staging, and production environments — "works on my machine" becomes "works in my image"
- **Fast deployment**: pulling a pre-built image and starting a container is faster than running a configuration management tool against a bare server
- **Clean teardown**: stopping a container leaves no trace on the host filesystem (unless you used volumes)
- **Parallel isolation**: run multiple versions of the same software on the same machine without conflicts — two different `nginx` versions, each in its own container

**Where this costs you:**

- **Size**: container images are not small. A `node:20` image is 1.1GB. Even `node:20-alpine` is 170MB. You're shipping an entire OS filesystem with every deployment artifact
- **Complexity**: debugging requires knowing which container a process is in, where logs go, how networking is wired between containers, and what the container runtime is doing
- **Kernel sharing**: containers share the host kernel — kernel vulnerabilities affect all containers; kernel features not present on the host are unavailable to containers
- **Not VMs**: the isolation is namespace-based, not hardware-virtualized; stronger than a process, weaker than a VM; security models built on the assumption of full isolation are wrong
- **Stateful applications are hard**: databases and anything with persistent state require volumes, which reintroduce the "what machine is this data on?" problem that containers otherwise eliminate

**Performance note:** Container overhead versus running a process directly is measurable but small — typically 1-3% CPU overhead from the namespace and cgroup machinery. OverlayFS read performance is close to native. Network performance through the default bridge network has more overhead; for latency-sensitive applications, host networking (`--network=host`) removes the virtual network layer.

## When to Use / When to Avoid

✅ **Use containers when:**

- You're deploying an application with complex or specific library dependencies that conflict with what the host provides
- You need identical behavior across multiple environments (dev, CI, staging, production)
- You're running third-party software (databases, message queues, proxies) on a development machine without wanting to manage their installation and configuration natively
- You need multiple isolated instances of the same software on one machine
- You're building for Kubernetes — containers are the deployment unit and there's no alternative

❌ **Avoid containers when:**

- The application has hard requirements on kernel version or kernel modules that may not match your host (e.g., eBPF programs, custom kernel modules — use a VM instead)
- You need the full isolation guarantees of a hypervisor for a multi-tenant security boundary — use VMs, not containers
- The application is a simple static binary with no runtime dependencies — there's no problem to solve; just copy the binary
- You're on an embedded system or constrained environment where the container runtime overhead (disk space, memory for containerd/dockerd) is unacceptable

⚠️ **Common mistakes:**

- Running the container entrypoint process as root inside the container — almost always unnecessary and increases attack surface
- Storing application state (database files, uploaded files) inside the container filesystem instead of in a mounted volume — all that data is lost when the container is removed
- Building images in CI without a vulnerability scanner step — you won't know when a base image CVE affects your deployments
- Using `docker run` for production instead of an orchestrator — `docker run` has no restart policy, health checking, or deployment rollback; use Docker Compose at minimum, Kubernetes for anything serious

## Veteran Perspective

The newcomer mistake with containers is treating them like lightweight VMs. They're not. People write Dockerfiles that install SSH, add a cron daemon, configure a syslog service, and set up a process supervisor. They end up with a container that looks like a tiny server. That's fighting the model. Containers are designed around one process per container. The init system, the syslog daemon, the SSH server — none of that belongs there. When the container's PID 1 exits, the container exits. That's the contract.

The intermediate mistake is ignoring what's in the base image. I've watched teams spend weeks hardening their application code, doing careful input validation, running static analysis — and then deploying it on top of `ubuntu:20.04` with 40 unpatched CVEs in the base system packages. The application layer is a small fraction of the attack surface. The base image is the larger fraction. Pin your base images to specific digests, run Trivy in CI, and use distroless or Alpine where you can.

The subtler mistake, which takes longer to hit, is using containers to avoid thinking about operational complexity rather than to manage it. Containers make it easy to ship a working application. They do not automatically solve log aggregation, secret management, configuration management, health checking, graceful shutdown, or data persistence. Those problems exist whether you're in a container or not. What I've seen in practice is that teams ship to containers, the application breaks in a confusing way in production, and they can't debug it because they didn't think about how to get logs out, or the container is running as root and they can't exec into it in a hardened environment.

On the static binary side: the Go and Rust ecosystems have largely solved the distribution problem without containers for CLI tooling. Every major Go tool — `kubectl`, `helm`, `terraform`, `caddy` — ships as a static binary. Download it, `chmod +x`, run it. No container needed. This is often the right answer for CLI tools and simple servers with no external runtime dependencies. Use the simplest model that solves the problem.

The OCI standard matters more than most people realize. Before it existed, the ecosystem was fragmented. Today, an image built with Docker runs on containerd, runs on Podman, runs in Kubernetes, runs on any OCI-compliant runtime. The format is a genuine open standard, not a vendor lock-in play. When you sign an OCI image with cosign and publish it to ghcr.io, you're participating in a supply chain provenance system that didn't exist five years ago. Image signing is moving from "best practice that large enterprises do" toward "table stakes that every CI pipeline does." Start doing it now.

For a final framing: containers are a distribution and isolation mechanism built on Linux kernel primitives — namespaces, cgroups, and OverlayFS. You will understand them far more deeply after Phase 14, which covers those primitives directly. What you need to understand now is the *what* and *why*, not the kernel internals. The kernel internals are Phase 14's job.

## Try It Yourself

1. Run `docker pull alpine:3.20` and then `docker run --rm alpine:3.20 cat /etc/os-release`. You're running a complete (minimal) Linux distribution in a container. Notice how fast it starts compared to booting a VM.

2. Run `docker image inspect alpine:3.20` and look at the `RootFS.Layers` field. How many layers does it have? Pull `ubuntu:24.04` and compare. What does the difference in layer count suggest about how the image was built?

3. Write a two-line Dockerfile (`FROM alpine:3.20` then `RUN apk add --no-cache curl`) and build it with `docker build -t myalpine .`. Run it twice. Notice the second build uses the cached layer. Then change `curl` to `wget` and rebuild — which layers are rebuilt and which are cached?

4. Build the same Dockerfile twice with different tag strategies: once with `FROM alpine:latest` and once with `FROM alpine:3.20`. Use `docker image inspect` on both to check the actual image ID. Are they the same image? Then do the same comparison a month later (or pull the latest alpine and see if it changed). This is the `latest` tag problem made concrete.

5. **Challenge:** Write a Go program (`main.go`) that prints "Hello from a static binary" and exits. Build it two ways: first normally with `go build`, then with `CGO_ENABLED=0 go build`. Run `ldd` on both binaries. Then write a two-line `Dockerfile` that uses `FROM scratch` as the base image and `COPY` to add only your static binary. Build and run it — you're running a container with no OS at all, just your binary and the kernel.

## Summary

- A container image is a layered, content-addressed archive in the OCI format containing everything an application needs to run — the runtime, libraries, and OS filesystem — so the host environment is irrelevant
- Image layers are stacked in build order and merged by OverlayFS at runtime; the order matters for build cache efficiency: put the things that change least (dependencies) before the things that change most (source code)
- `latest` is a mutable tag, not a version pin; it will point to a different image over time; pin to explicit version tags or digests for reproducible builds
- Containers are not VMs; they share the host kernel via Linux namespaces; a process running as root inside a container is a security risk and should be avoided
- Static binaries (Go with `CGO_ENABLED=0`, Rust with `x86_64-unknown-linux-musl`) are the extreme end of self-contained distribution — zero runtime dependencies, runnable anywhere including `FROM scratch` containers
