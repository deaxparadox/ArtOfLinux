## The Problem This Solves

Your distro ships version 3.1 of a library. The project you need to build requires 3.4. You file a bug. The maintainer says it'll be in the next release, eighteen months away. You need it now.

Or you're benchmarking a database server. The distro package was compiled with debug symbols and without AVX2 optimization, because the maintainer had to build a binary that runs on every machine from a 2012 laptop to a 2024 server. Your hardware supports AVX2 and you want to use it. You can't get that binary from `apt`.

Or you're writing a patch and need to test it against a running binary. Or the project is not packaged anywhere — it lives on GitHub, it has a README, and it says `./configure && make && make install`.

All of these situations lead to the same place: compiling from source. This isn't exotic. It's a core Linux skill that every sysadmin eventually needs, and it's more approachable than it looks — once you understand what the three steps are actually doing.

---

## Mental Model

Think of building software the same way you'd think of cooking from a recipe in a foreign kitchen.

The recipe calls for ingredients (libraries and header files). Before you start cooking, you walk through the kitchen and check that everything is there — this is `./configure`. If something is missing, configure tells you and stops. Once you have everything, you follow the recipe steps in order — this is `make`. At the end, you plate the food and put it on the table so guests can eat it — this is `make install`.

The "table" is `/usr/local`. The "kitchen staff" is your compiler toolchain. The `Makefile` is the recipe card. The analogy survives contact with reality: just like a recipe, a `Makefile` can have errors, steps can depend on earlier steps being done first, and running things in the wrong order produces something inedible.

## Theory

### Why `/usr/local`?

The Filesystem Hierarchy Standard divides software into three tiers. `/bin`, `/lib`, `/usr/bin` — these are managed by the distro's package manager. You do not touch these by hand. `/usr/local/bin`, `/usr/local/lib` — these are for software you install yourself, outside the package manager. The name "local" means "local to this machine," as opposed to software that came from the distribution. When you install to `/usr/local`, you don't conflict with the package manager, and you know exactly which files you put there.

This separation matters. If you installed something to `/usr/bin` by hand and then the package manager tried to install the same program, you'd have a conflict. `/usr/local` is your sandbox. Its `bin` is already in your `PATH` on virtually every Linux system.

### The Build System Landscape

Most open-source C and C++ software uses one of three build systems:

**Autotools** (`./configure && make && make install`) is the traditional Unix approach, dating back decades. The project ships a `configure` shell script (usually generated from `configure.ac`) that probes your system — checks for libraries, checks compiler features, checks paths — and then writes a `Makefile` customized to your system. It's verbose, sometimes slow, and occasionally arcane, but it's universal. If a project has been around since the 90s, it almost certainly uses Autotools.

**CMake** (`cmake -B build && cmake --build build`) is the modern cross-platform approach. Instead of a shell script, it uses a `CMakeLists.txt` file that CMake reads. CMake generates native build files — Makefiles, Ninja files, or Visual Studio projects depending on the platform and your choice. CMake 3.13 introduced the `-B` flag for out-of-source builds and `--build` for the platform-neutral build invocation. As of 2026, CMake 4.x is current; the `-B`/`--build` workflow has been the recommended approach since 3.14.

**Meson** (`meson setup build && ninja -C build`) is the newest of the three, written in Python, with Ninja as its backend. It's significantly faster than Autotools, has cleaner syntax than CMake, and has become the standard for GNOME, Mesa, systemd, and most of the freedesktop.org ecosystem. If you're building anything from that world — GLib, GStreamer, Wayland compositors — you'll encounter Meson.

### What `make` Actually Is

`make` is a dependency-resolution tool. It reads a `Makefile` — a file that describes targets, the prerequisites each target depends on, and the commands to build each target. When you run `make`, it walks the dependency graph, checks whether targets are up to date (by comparing file modification timestamps), and runs only the commands needed to bring the target up to date.

A simplified `Makefile` looks like this:

```text
# Makefile anatomy — conceptual, not a real project
CC = gcc                          # variable: the C compiler to use
CFLAGS = -O2 -Wall                # variable: compiler flags

# target: prerequisites
#     command (must be indented with a real tab, not spaces)

myprogram: main.o utils.o
	$(CC) $(CFLAGS) -o myprogram main.o utils.o

main.o: main.c utils.h
	$(CC) $(CFLAGS) -c main.c

utils.o: utils.c utils.h
	$(CC) $(CFLAGS) -c utils.c

.PHONY: clean install
clean:
	rm -f *.o myprogram

install:
	install -m 755 myprogram /usr/local/bin/
```

The `.PHONY` declaration tells make that `clean` and `install` are not real files — they're action targets. Without `.PHONY`, if a file called `clean` ever appeared in the directory, `make clean` would say "nothing to do" because the target file is already up to date.

### What `pkg-config` and `pkgconf` Are

When your code needs a library — say, `libpng` — the compiler needs to know two things: where the header files are (for compilation), and where the library itself is (for linking). `pkg-config` is the standard tool for answering both questions. Libraries install a `.pc` file (in `/usr/lib/pkgconfig/` or `/usr/share/pkgconfig/`) that describes their compile flags and link flags. You query it:

```bash
pkg-config --cflags libpng   # outputs: -I/usr/include/libpng16
pkg-config --libs libpng     # outputs: -lpng16
```

`pkgconf` is a modern, actively maintained drop-in replacement for the original `pkg-config`. It's already the default on Fedora and is compatible when called as `pkg-config`. New systems will increasingly have `pkgconf` providing the `pkg-config` command. From a user's perspective, the interface is identical — but `pkgconf` handles edge cases in `.pc` files (particularly `Provides:` and `Cflags.private:` rules) that the original implementation gets wrong. If you're on a fresh Fedora or Arch system and `which pkg-config` points to `pkgconf`, that's correct and intentional.

### Shared Libraries and `ldconfig`

When you compile a program that links against a shared library (a `.so` file), the program records the library's name in its binary. At runtime, the dynamic linker — `/lib/x86_64-linux-gnu/ld-linux-x86-64.so.2` — needs to find that library by name. It doesn't search every directory on your filesystem at startup; that would be too slow. Instead, it reads a cache: `/etc/ld.so.cache`.

`ldconfig` builds that cache. It reads `/etc/ld.so.conf` (which includes files in `/etc/ld.so.conf.d/`) to find which directories to scan, scans them for `.so` files, and writes the cache. The cache maps library names to filesystem paths so the dynamic linker can find them in microseconds.

You need to run `ldconfig` when you install a shared library into a non-standard path — one that isn't already in the cache. After `make install` puts a `.so` file into `/usr/local/lib`, the dynamic linker doesn't know it's there until you run `ldconfig` as root.

`ldd` is the tool for examining which shared libraries a binary needs and whether the linker can find them:

```bash
ldd /usr/bin/curl
```

If a library is listed as `not found`, the binary will fail to run. Either the library isn't installed, or it's installed somewhere `ldconfig` doesn't know about.

## How It Works (Mechanics)

### The Autotools Configure Step

When you run `./configure`, you're executing a large shell script — sometimes 10,000+ lines. It probes your system by attempting to compile small test programs, then checking whether the compilation succeeded. For example, to check for `zlib`:

```text
# Conceptual view of what ./configure does for a library check
configure: checking for zlib...
  → writes a test C file: #include <zlib.h> ... compress2(...)
  → runs: gcc -o conftest conftest.c -lz
  → if exit code 0: zlib is present, sets HAVE_ZLIB=yes in config.h
  → if exit code non-zero: zlib not found; configure either aborts or disables the feature
```

It's not checking for the library by looking for a file — it's checking whether the compiler can actually use the library. This catches version mismatches and ABI issues that a simple `ls /usr/lib` check would miss.

The `--prefix` flag tells configure where to install. `--enable-foo` and `--disable-foo` turn optional features on and off. The complete list of available options is always in `./configure --help`.

When configure finishes successfully, it has written:
- A `Makefile` (or several, for subdirectories) customized to your system
- A `config.h` file with `#define` macros the source code uses to conditionally compile features

### Parallel Builds with `-j`

By default, `make` runs one job at a time. On a 16-core machine, 15 cores sit idle while one core compiles a single file. `make -j$(nproc)` tells make to run as many parallel jobs as you have processor cores. On a modern machine, this can reduce a 20-minute build to 3 minutes.

```text
# What make -j$(nproc) does
$(nproc) → returns the number of processing units available (e.g. 16)
make -j16 → starts up to 16 compilation jobs simultaneously
             each job takes a target whose prerequisites are already satisfied
             when a job finishes, make starts the next available job
```

The catch: parallel builds expose Makefile dependency bugs. If a `Makefile` has a missing or incorrect dependency declaration between two targets, a sequential `make -j1` might work by accident (because targets happen to be built in the right order), but `make -j$(nproc)` will sometimes try to build a target before its prerequisite is ready, producing a cryptic compilation error. This isn't your fault — it's a bug in the project's `Makefile`. The fix is to run `make -j1` to get a clean sequential error message that shows the real problem.

### The CMake Two-Step

CMake's modern workflow separates the configuration and build phases cleanly:

```text
cmake -B build        # configure: read CMakeLists.txt, generate build files in ./build/
cmake --build build   # build: invoke the native build tool inside ./build/
```

The `-B build` flag creates an out-of-source build directory. All generated files — Makefiles, object files, intermediate outputs — go into `./build/`. Your source tree stays clean. `cmake --build build` then invokes whatever build tool CMake chose (usually `make` or `ninja`), so you don't need to know or care which one it picked.

To pass parallel jobs through CMake: `cmake --build build -- -j$(nproc)` (the `--` separates CMake flags from flags passed to the underlying build tool).

To install after building: `sudo cmake --install build` (this replaces `make install` in the CMake world).

### The Meson + Ninja Two-Step

```text
meson setup build     # configure: read meson.build, generate Ninja files in ./build/
ninja -C build        # build: run Ninja using the files in ./build/
```

Meson generates only Ninja build files — there is no Makefile generator in Meson. Ninja is designed specifically to be fast: it does as little work as possible, starts jobs in parallel by default, and has almost no overhead. `ninja -C build` is equivalent to `cd build && ninja`, but without changing your working directory.

## Basic Example

```bash
# Install build tools if they're not already present (Debian/Ubuntu)
sudo apt install build-essential

# Download a simple project — GNU hello is the canonical example
wget https://ftp.gnu.org/gnu/hello/hello-2.12.tar.gz

# Extract it
tar -xzf hello-2.12.tar.gz
cd hello-2.12

# Step 1: configure — check dependencies, generate Makefile
./configure

# Step 2: compile
make

# Step 3: install (to /usr/local by default)
sudo make install
```

```text
checking for a BSD-compatible install... /usr/bin/install -c
checking whether build environment is sane... yes
checking for a thread-safe mkdir -p... /bin/mkdir -p
...
checking for gcc... gcc
checking whether the C compiler works... yes
...
config.status: creating Makefile
config.status: creating src/Makefile
...
make[1]: Entering directory '/home/user/hello-2.12/src'
  CC       hello.o
  CCLD     hello
make[1]: Leaving directory '/home/user/hello-2.12/src'
...
```

After `make install`, verify it worked:

```bash
which hello
hello
```

```text
/usr/local/bin/hello
Hello, world!
```

Look at the configure output. It says things like "checking for gcc... gcc" and "checking whether the C compiler works... yes." Each of those lines is configure running a probe. If one fails, it prints a one-line error and stops. The error line is almost always preceded by "no" — scan for the last "no" to find where it died.

Try this yourself. If your `./configure` fails immediately with "no C compiler found," you need `build-essential` (Debian/Ubuntu) or `gcc make` (RHEL/Fedora) or `base-devel` (Arch). Once you have those, the configure step for a simple project like GNU hello should run cleanly.

## Real-World Example

This is a realistic workflow for building a project that uses CMake, requires a custom install prefix, and installs a shared library that other programs need to find.

```bash
# Install build dependencies
sudo apt install build-essential cmake libssl-dev pkg-config

# Clone a project (using a hypothetical project as an example)
git clone https://github.com/example/myproject.git
cd myproject

# Configure: out-of-source build, install to /usr/local, Release optimizations
cmake -B build \
    -DCMAKE_BUILD_TYPE=Release \
    -DCMAKE_INSTALL_PREFIX=/usr/local \
    -DBUILD_SHARED_LIBS=ON

# Build using all available cores
cmake --build build -- -j$(nproc)

# Install the compiled files
sudo cmake --install build

# If the project installed shared libraries to /usr/local/lib,
# the dynamic linker doesn't know about them yet — update the cache
sudo ldconfig

# Verify: check what shared libraries the installed binary needs
ldd /usr/local/bin/myproject
```

```text
	linux-vdso.so.1 (0x00007ffd8a3c4000)
	libmyproject.so.1 => /usr/local/lib/libmyproject.so.1 (0x00007f3a12345000)
	libssl.so.3 => /usr/lib/x86_64-linux-gnu/libssl.so.3 (0x00007f3a11234000)
	libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f3a10000000)
	/lib64/ld-linux-x86-64.so.2 (0x00007f3a12600000)
```

The walkthrough: `-DCMAKE_BUILD_TYPE=Release` enables optimizations (`-O3`, no debug symbols) — the equivalent of a distro release build. `-DBUILD_SHARED_LIBS=ON` tells the project to build `.so` files rather than static `.a` files; you want shared libraries when multiple programs will use the same code and you want to be able to update the library without rebuilding every consumer. `-j$(nproc)` uses all cores.

After `cmake --install`, the `ldconfig` call is mandatory if the project put anything in `/usr/local/lib`. Without it, running `myproject` will fail with "error while loading shared libraries: libmyproject.so.1: cannot open shared object file: No such file or directory" — even though the file is right there at `/usr/local/lib/libmyproject.so.1`.

The `ldd` output confirms everything is resolved. If you see `not found` next to any library, the program will not run. Either that library isn't installed, or it's in a directory `ldconfig` hasn't been told about.

Now here's the same workflow using `checkinstall` instead of `cmake --install`, so the system package manager can track and remove what you installed:

```bash
# Instead of: sudo cmake --install build
# Use: sudo checkinstall cmake --install build

sudo checkinstall cmake --install build
```

`checkinstall` wraps the install command, watches every file it touches, and creates a `.deb` (or `.rpm` on RHEL systems) that packages all of those files. It then installs that package via `dpkg` instead of copying files directly. The result: `dpkg -l | grep myproject` shows the package, and `sudo dpkg -r myproject` removes every file it installed. Without `checkinstall`, your only options for uninstalling a `make install` are to run `make uninstall` if the project supports it (many don't), or to manually track and delete every file by hand.

And here's a Meson build for a GNOME-adjacent project:

```bash
sudo apt install meson ninja-build libglib2.0-dev

git clone https://gitlab.gnome.org/someproject.git
cd someproject

meson setup build --prefix=/usr/local --buildtype=release
ninja -C build
sudo ninja -C build install
sudo ldconfig
```

`--buildtype=release` in Meson is the equivalent of `-DCMAKE_BUILD_TYPE=Release`. The prefix is passed at configure time, not install time.

## ⚠️ Gotchas

### `./configure` Succeeds but `make` Fails with a Missing Header

This is the most common trap when building from source, and it's confusing because you'd expect configure to catch it.

```bash
./configure
```

```text
...
checking for libcurl... yes
...
configure: creating ./config.h
configure: creating ./Makefile
```

```bash
make
```

```text
In file included from src/download.c:3:
src/download.c:3:10: fatal error: curl/curl.h: No such file or directory
   3 | #include <curl/curl.h>
     |          ^~~~~~~~~~~~~
compilation terminated.
make[1]: *** [Makefile:234: src/download.o] Error 1
```

Configure checked that libcurl exists and is linkable. It did not check that the header file `curl/curl.h` is where the compiler can find it. On Debian and Ubuntu, libraries are split into two packages: `libcurl4` (the runtime library — the `.so` file) and `libcurl4-openssl-dev` (the development package — the `.h` header files and the `.pc` file). If you only installed the runtime package, configure finds the library but make can't find the headers.

The fix is almost always "install the `-dev` package":

```bash
# Debian / Ubuntu
sudo apt install libcurl4-openssl-dev

# RHEL / Fedora
sudo dnf install libcurl-devel

# Arch
sudo pacman -S curl    # Arch ships both runtime and headers in one package
```

After installing the dev package, run `./configure` again — don't just re-run `make`. Configure needs to re-probe and update its flags to point at the newly installed headers.

### `make install` Without `checkinstall` — No Way to Uninstall

This is the silent footgun of building from source. `make install` copies files to dozens of locations: binaries to `/usr/local/bin`, libraries to `/usr/local/lib`, headers to `/usr/local/include`, man pages to `/usr/local/share/man`, and whatever else the project decided to install.

Six months later you want to remove this program. You deleted the source directory. There is no record of what was installed where. `make uninstall` doesn't exist for this project. You're left running `find /usr/local -name "myproject*"` and hoping you found everything.

```bash
# Wrong: installs files with no tracking
sudo make install

# Right: installs via a trackable package
sudo checkinstall make install
# or for CMake projects:
sudo checkinstall cmake --install build
```

`checkinstall` asks you a few questions (package name, version, description) and then builds a `.deb` or `.rpm` from whatever the install command writes. Now you have a real package: `dpkg -l | grep myproject` shows it, and `sudo dpkg -r myproject` removes everything cleanly.

The caveat: `checkinstall` is not actively developed — the last real release was 1.6.2 in 2009, though distributions carry patched versions. It works for the common case. For a more robust alternative, look into building proper distro packages (a separate skill), or at minimum maintain a plain text list of what was installed.

### `make -j$(nproc)` Hides the Real Error

You run a parallel build and get a wall of errors. The errors seem contradictory. You can't tell what actually went wrong.

```bash
make -j$(nproc)
```

```text
...
cc1: error: unrecognized command-line option '-mfoo'
In file included from src/bar.c:1: utils.h: No such file or directory
make[2]: *** [src/CMakeFiles/bar.dir/bar.c.o] Error 1
make[2]: *** [src/CMakeFiles/utils.dir/utils.c.o] Error 1
make[1]: *** [src/CMakeFiles/bar.dir/all] Error 2
...
(40 more lines of cascade errors)
```

With 16 parallel jobs, when one job fails, 15 other jobs are still running. They may themselves fail — either because of the original error or because they depended on the output of the failed job. The result is an avalanche of errors where the first one is buried in the middle and the cascade errors are red herrings.

The fix: if a parallel build fails, re-run with one job:

```bash
make -j1
```

```text
cc1: error: unrecognized command-line option '-mfoo'
make[2]: *** [src/CMakeFiles/bar.dir/bar.c.o] Error 1
```

With one job, the error is the first thing printed, it's clear, and the build stops at the first failure. This is your diagnostic pass. Once you understand the actual error, fix it, then go back to parallel builds.

### `LD_LIBRARY_PATH` as a Permanent Fix

You install a shared library to `/usr/local/lib`. The binary you just built fails to run. Someone tells you to add `export LD_LIBRARY_PATH=/usr/local/lib` to your `.bashrc`. It works — the binary runs. You move on.

Don't do this.

`LD_LIBRARY_PATH` is processed before every other library search path. It applies to every process you launch from that shell — including processes that weren't involved in the original problem. It affects `sudo` (depending on the system), `cron` jobs, and any script that sources your `.bashrc`. It can override system libraries with your custom versions, accidentally causing other programs to load the wrong library. It's a global variable with global consequences.

The right fix is two commands:

```bash
# Tell ldconfig about /usr/local/lib (if it isn't already known)
echo "/usr/local/lib" | sudo tee /etc/ld.so.conf.d/usr-local.conf

# Rebuild the cache
sudo ldconfig
```

Now the dynamic linker knows about `/usr/local/lib` permanently, for all users, without environment variable pollution. `/usr/local/lib` is actually already included in most distros' default `ldconfig` configuration — run `ldconfig -v 2>/dev/null | grep "usr/local"` to check. If it's already listed, you just need `sudo ldconfig` after installing the library.

## Distro Notes

| Distro Family | Build Tools | Dev Packages |
|---|---|---|
| Debian / Ubuntu | `sudo apt install build-essential` — provides `gcc`, `g++`, `make` | Library headers: `lib<name>-dev` (e.g., `libssl-dev`, `libsqlite3-dev`) |
| RHEL / Fedora / Rocky | `sudo dnf groupinstall "Development Tools"` | Library headers: `<name>-devel` (e.g., `openssl-devel`, `sqlite-devel`) |
| Arch | `sudo pacman -S base-devel` | Arch ships dev headers with the library package — no separate `-dev` package |
| Alpine (containers) | `apk add build-base` | Header packages: `<name>-dev` — same naming as Debian |

CMake is available as `cmake` on all distros. Meson is `meson` on all distros; Ninja is `ninja-build` on Debian/Ubuntu and `ninja-build` on RHEL/Fedora, and `ninja` on Arch.

`checkinstall` is available in Debian/Ubuntu as `checkinstall`. It is **not** packaged in RHEL/Fedora or Arch. On RHEL-family systems, `checkinstall` can build `.rpm` packages but is not in the default repositories — look in EPEL. On Arch, use the AUR or manually track installed files.

`pkgconf` provides the `pkg-config` command on Fedora (since Fedora 38), Arch, and increasingly on Debian. On Ubuntu 22.04+, `pkg-config` is a wrapper that calls `pkgconf`. The CLI is identical — use `pkg-config` in all scripts regardless of which binary backs it.

## Alternatives & Comparisons

| Build System | Language / Projects | When You'll See It |
|---|---|---|
| Autotools (`./configure && make`) | C, C++ — GNU projects, most software predating 2010 | Anything with a `configure.ac` or `configure` script in the repo |
| CMake | C, C++, Fortran — cross-platform projects, modern libraries | `CMakeLists.txt` in the repo root; Qt, KDE, LLVM, many game engines |
| Meson + Ninja | C, C++ — GNOME, Mesa, systemd, Wayland | `meson.build` in the repo root; GLib, GStreamer, GDK |
| plain `Makefile` | Anything — small projects, kernel, Redis, SQLite | Just a `Makefile`, no configure step |
| `cargo` | Rust | `Cargo.toml` — handled entirely by cargo, no separate `make install` step |
| `pip install .` | Python | `pyproject.toml` — handled by pip, installs to the active Python environment |

For projects using Autotools: use the classic three-step and always run `./configure --prefix=/usr/local`. For CMake projects: use `cmake -B build && cmake --build build` — it's cleaner than the old `mkdir build && cd build && cmake ..` approach. For Meson projects: `meson setup build && ninja -C build`. Never mix build systems within a single project — follow what the project ships.

## Trade-offs

**Where this shines:**

- You get software at any version, with any compile-time options, before it's packaged for your distro
- Custom `CFLAGS` and feature flags let you build software tuned to your hardware or stripped of features you don't need
- Debugging and profiling become easier when you build with `--enable-debug` or `-DCMAKE_BUILD_TYPE=Debug` — the binary has symbols, source paths, and no optimization that scrambles stack traces
- You understand the dependency graph of a piece of software far better after building it than you ever would from installing a prebuilt package

**Where this costs you:**

- Security updates become your problem: when a CVE is patched upstream, you need to rebuild manually; the package manager handles this automatically for managed packages
- Dependency management is entirely manual: you need to track what you installed, in what order, and whether any of it conflicts with packages the package manager manages
- `make install` without `checkinstall` leaves files scattered across `/usr/local` with no inventory and no clean uninstall path
- Build failures are often cryptic and distro-specific — a configure check that passes on Ubuntu may fail on Alpine because musl libc lacks a glibc extension the test relies on

**Performance note:** `make -j$(nproc)` genuinely makes a measurable difference. On an 8-core machine, a build that takes 18 minutes with `make -j1` typically takes 3-4 minutes with `make -j8`. The savings are real and you should always use it on builds you'll do more than once.

## When to Use / When to Avoid

✅ **Use this when:**

- The version in the distro repository is too old for your requirements and there is no backport or PPA
- You need to pass custom compile-time flags — optimization flags, feature flags, target architecture
- The project is not packaged at all in your distro
- You're developing or patching the software itself and need to test changes
- You need debug symbols for profiling or debugging a production issue

❌ **Avoid this when:**

- A distro package or a well-maintained PPA/COPR/AUR package exists — the packaging work is done for you, security updates are automatic, and uninstall is clean; use it
- You're on a production system where reproducibility matters — a manual source build is hard to reproduce exactly on another machine; use containers or configuration management with pinned packages instead
- You don't have time to maintain it — manually built software doesn't receive security updates unless you rebuild it; this is a commitment, not a one-time action

⚠️ **Common mistakes:**

- Running `make install` to `/usr/` (not `/usr/local`) — this puts files in the package manager's territory; a future `apt upgrade` or `dnf update` may overwrite your files or leave the system in an inconsistent state; always use `--prefix=/usr/local`
- Not running `sudo ldconfig` after installing a shared library — the binary will fail at runtime with a library-not-found error even though the `.so` is sitting right there
- Deleting the source directory before running `make uninstall` — if the project supports `make uninstall`, you need the source tree to do it; if you might need to uninstall, use `checkinstall`
- Running `./configure` once and re-running `make` after installing a missing library — you need to re-run `./configure` too, because configure cached the failed probe result

## Veteran Perspective

Every sysadmin I know has a story about a server that worked for two years and then stopped working because someone had installed something from source, without documentation, and the next person had no idea what was there or why `/usr/local/lib` had a bunch of `.so` files that weren't in any package database. The machine was stable but opaque. Then it needed a major OS upgrade, and nobody knew which manually built things were safe to carry forward and which ones needed to be rebuilt.

The real discipline in building from source isn't the build itself — that's a one-time thing you can look up. The real discipline is tracking what you installed, why you installed it, and what the upgrade path looks like. Use `checkinstall`. Write a comment in your runbook or Ansible playbook. Even a simple text file in `/usr/local/share/doc/` with "installed myproject 1.4.2 from https://github.com/... on YYYY-MM-DD for reason X" is infinitely more useful than nothing.

The `./configure` fails with "missing library" error is a rite of passage. The first time you see it, you Google the library name, find out there's a `-dev` package you need, install it, re-run configure, and feel like you've unlocked a secret level. After the fifth time, you internalize the pattern: the `-dev` or `-devel` package is always the answer when configure or make says it can't find a header file.

`checkinstall` has a reputation for being old and unmaintained, and that reputation is deserved — the last major release was 2009. But it works for the common case and it's the right habit to build. The alternative — building proper distro packages with `debhelper` and `dh_make` or `rpmbuild` — is the right answer for software you maintain, but it's a skill in its own right and overkill for "I need to install this one thing from source right now."

The CMake ecosystem has improved significantly in the last several years. The old idiom of `mkdir build && cd build && cmake .. && make` is obsolete — `cmake -B build && cmake --build build` does the same thing cleanly from the source root, doesn't require you to change directory, and doesn't require you to know whether the native build tool is `make` or `ninja` or something else. Use the new form. If you're reading a project's build instructions and they're still showing the old form, it's fine — the old form still works, but the new form is cleaner and is what all modern CMake documentation recommends.

The distinction between `ldd` and `ldconfig` trips people up: `ldd` tells you what libraries a specific binary needs and whether they can be found right now. `ldconfig` rebuilds the cache that tells the dynamic linker where to look. They're different operations. `ldd` is a diagnostic tool. `ldconfig` is a maintenance operation you run after installing libraries.

## Try It Yourself

1. Install `build-essential` (or its equivalent for your distro), download `https://ftp.gnu.org/gnu/hello/hello-2.12.tar.gz`, and walk through `./configure && make && sudo make install`. Verify with `hello` and `which hello`. Then read `./configure --help` and notice how many feature flags exist.

2. Run `./configure --prefix=$HOME/.local` in the same hello source tree (without `sudo`). Run `make && make install`. Check `~/.local/bin/hello`. Notice you didn't need root for this — `/usr/local` requires root because it's system-wide; your home directory doesn't.

3. Build GNU hello with parallel jobs: `time make -j1` vs `time make -j$(nproc)`. The project is too small to see a dramatic difference, but the pattern is correct. Try a larger project if you want the real effect.

4. After installing GNU hello, run `ldd $(which hello)`. Identify which libraries it depends on. Then run `pkg-config --libs --cflags zlib` — notice the format. This is what `./configure` is using internally to find library flags when it calls pkg-config in its probes.

5. **Challenge:** Clone a small CMake project from GitHub — something with a `CMakeLists.txt` and actual library dependencies (not just hello world). Build it using `cmake -B build -DCMAKE_BUILD_TYPE=Release && cmake --build build -- -j$(nproc)`. If it fails, use `make -j1` to isolate the error. Install it with `sudo checkinstall cmake --install build`. Verify with `dpkg -l | grep <packagename>` that it's tracked. Then remove it cleanly with `sudo dpkg -r <packagename>`. Do this in a VM.

## Summary

- The Autotools three-step is `./configure` (probe dependencies, generate Makefile) → `make` (compile) → `sudo make install` (copy to `/usr/local`); install to `/usr/local`, never to `/usr/` by hand
- Missing header errors during `make` mean you need the `-dev` or `-devel` package for that library; re-run `./configure` after installing it, not just `make`
- `make -j$(nproc)` uses all CPU cores for parallel compilation and is almost always the right choice; if it fails with confusing errors, re-run `make -j1` to see the real first error
- After installing a shared library to `/usr/local/lib`, run `sudo ldconfig` — without it, the dynamic linker's cache is stale and binaries that need the library will fail at runtime
- Wrap `make install` or `cmake --install` with `sudo checkinstall` to create a trackable `.deb` or `.rpm` that the package manager can cleanly uninstall later
