## The Problem This Solves

Traditional Unix permissions have a fatal design assumption: if you are root, you can do anything. Full stop. There is no policy above root. This model worked fine when servers ran one or two services managed by a trusted administrator, but it fails catastrophically in the modern world for a simple reason: services run as root, or as users with near-root privileges, and they get compromised.

Picture a web server running as the `apache` user. Standard DAC — Discretionary Access Control, the classic `rwxrwxrwx` model — says that `apache` can read its config files, write to its log directory, and serve files from `/var/www`. That sounds fine. But when an attacker exploits a vulnerability in Apache and gets code execution as the `apache` user, DAC has nothing more to say. The attacker can now read `/etc/shadow` if the permissions are loose anywhere in the chain. They can write to directories `apache` owns. They can open network connections to exfiltrate data. DAC's only question is: "Is this user allowed?" It never asks: "Should an Apache process be doing this?"

The deeper problem is that DAC is discretionary — file owners control access policy. A developer who creates an application can, intentionally or accidentally, grant permissions that violate the system's intended security posture. There is no central authority enforcing a consistent policy.

MAC — Mandatory Access Control — solves this by moving access decisions out of the hands of individual users and into a system-wide policy enforced by the kernel. The policy says: "An Apache process is only allowed to read files labeled as web content, write to files labeled as Apache log files, and listen on port 80. Even if a process is running as root. Even if the file permissions say otherwise." The policy is mandatory. Nobody overrides it.

Linux has two main implementations of MAC: SELinux, developed by the NSA and now the default on RHEL-family systems, and AppArmor, simpler and path-based, the default on Debian/Ubuntu systems.

---

## Mental Model

Think of the traditional Unix permission system as a building where every door has a lock and everyone who has a key can open any door that key fits. The building owner (root) has a master key that opens everything. If someone steals the master key, every room is compromised.

MAC is a different kind of security system layered on top: a 24-hour security guard at every door who checks not just whether you have the right key, but whether your *job role* is supposed to be in this room *right now*. Even if you have the master key — even if you are the building owner — the security guard checks the policy binder. The policy says the night janitor is not authorized to enter the server room, period, regardless of what keys they carry. The guard doesn't care who you are. They care what you're allowed to do.

## Theory

### DAC vs. MAC — The Fundamental Distinction

In DAC, the *owner* of a resource decides who can access it. You create a file, you decide the permissions. Root can always override. The operating system enforces whatever the owner chose. Access is discretionary.

In MAC, a *central policy authority* — the kernel enforced policy — decides who can access what, regardless of ownership, regardless of root status. Access is mandatory. No user, not even root, can override the MAC policy at runtime without changing the policy itself.

This distinction has a critical consequence: under MAC, a compromised service can only do damage within the permissions its policy grants. A compromised Apache process cannot read `/etc/passwd` even if the file permissions would otherwise allow it, because the SELinux or AppArmor policy for the Apache process type does not include that access. The blast radius of a compromise is contained to the policy envelope.

### SELinux — Labels on Everything

SELinux implements MAC using a labeling system. Every single object on the system — every file, directory, process, socket, and port — has a security label called a *context*. Every access decision the kernel makes is checked against a policy that says which contexts are allowed to interact.

A SELinux context has four components:

```text
user:role:type:level

system_u:system_r:httpd_t:s0
```

In practice, the *type* component is what matters most for daily administration. A file with type `httpd_sys_content_t` is web content. A process with type `httpd_t` is an Apache process. The policy says: `httpd_t` can read `httpd_sys_content_t` files. It does not say `httpd_t` can read `shadow_t` files. That rule simply doesn't exist in the policy, so the access is denied, regardless of DAC permissions.

The three modes SELinux operates in:

- **Enforcing** — the policy is active and enforced. Denials are logged and the action is blocked.
- **Permissive** — the policy is active but denials are only logged, not blocked. The system behaves as if SELinux is off, but the audit trail builds up. This is a diagnostic mode, not a permanent state.
- **Disabled** — SELinux is completely off. The kernel does not label objects or enforce policy. Getting back to enforcing from disabled requires a full filesystem relabel on the next boot, which takes time proportional to the number of files on the system.

The current RHEL 9 default ships with the `targeted` policy, which confines network-facing services and high-risk processes while leaving user-space processes mostly unconfined. The policy RPM as of RHEL 9.5 is `selinux-policy-targeted-38.1.45` with a max kernel policy version of 33.

### AppArmor — Profiles by Path

AppArmor takes a simpler approach: instead of labeling every object on the filesystem, it uses path-based rules. A profile for an application says: "This binary is allowed to read files matching `/etc/nginx/**`, write to `/var/log/nginx/*.log`, and listen on TCP port 80." The rules are about filesystem paths, not kernel-level labels.

This makes AppArmor easier to understand and easier to write profiles for, at the cost of some flexibility. Path-based rules can be fooled by symlinks in ways that label-based rules cannot. But for many real-world applications, AppArmor profiles are straightforward and effective.

AppArmor profiles live in `/etc/apparmor.d/`. Each profile corresponds to the full path of the binary it confines.

AppArmor modes per profile:

- **Enforce** — rules are active; violations are blocked and logged.
- **Complain** — violations are logged but not blocked. The application runs freely while you observe what it tries to do. This is used to build and refine profiles without breaking production.
- **Unconfined** — the binary has no profile loaded; AppArmor has nothing to say about it.

## How It Works (Mechanics)

### SELinux — The Access Vector Cache

Every time a process tries to access an object — open a file, bind a socket, connect to a port — the kernel's LSM (Linux Security Module) hook intercepts the syscall before it proceeds. The SELinux module checks the requesting process's type against the target object's type in the policy's Access Vector Cache (AVC).

```text
# Conceptual flow — not a real command
process syscall (e.g., open("/etc/shadow"))
  → LSM hook fires before the VFS operation
  → SELinux looks up: httpd_t reading shadow_t
  → AVC cache miss → policy lookup
  → No allow rule found → DENY
  → Denial written to /var/log/audit/audit.log
  → syscall returns -EACCES to the process
```

The AVC is a cache of recent decisions. On a cache hit, the decision is instant. On a miss, the full policy database is consulted. This is why SELinux has a measurable but small performance overhead — the overwhelming majority of access checks hit the cache.

### AppArmor — The Profile Mediator

AppArmor hooks into the same LSM framework. When a binary executes, the kernel loads its AppArmor profile (if one exists) into the process's security blob. On every subsequent syscall involving a path, AppArmor checks the path against the profile's rules.

```text
# Conceptual flow — not a real command
process exec("/usr/sbin/nginx")
  → AppArmor matches binary path to /etc/apparmor.d/usr.sbin.nginx
  → Profile loaded into process security context
  → process syscall (open("/etc/passwd"))
  → AppArmor checks: /etc/passwd against profile rules
  → No matching allow rule → DENY (in enforce mode) or LOG (in complain)
```

## Real-World Example

### Diagnosing and Fixing a SELinux Denial

You've deployed a new application that serves files from a custom directory `/srv/myapp/data`. The application runs as the `nginx` user, and you've configured nginx to serve from that path. nginx starts, but returns 403 errors and you see nothing wrong in the standard nginx error log.

```bash
# Step 1: Check the audit log for AVC denials
sudo grep "avc: denied" /var/log/audit/audit.log | tail -20
```

```text
type=AVC msg=audit(1718000000.123:456): avc:  denied  { read } for  pid=12345 comm="nginx" name="data" dev="sda1" ino=987654 scontext=system_u:system_r:httpd_t:s0 tcontext=system_u:object_r:var_t:s0 tclass=dir permissive=0
```

The denial tells you everything: the `httpd_t` process (nginx) tried to `read` a directory with type `var_t`. The problem is that `/srv/myapp/data` has the wrong SELinux type — it inherited `var_t` from `/srv`, but nginx expects `httpd_sys_content_t`.

```bash
# Step 2: Translate the denial into plain English
sudo audit2why < /var/log/audit/audit.log
```

```text
type=AVC msg=audit(1718000000.123:456): avc:  denied  { read } for pid=12345 comm="nginx"...

        Was caused by:
                Missing type enforcement (TE) allow rule.

                You can use audit2allow to generate a loadable module to allow this access.
```

```bash
# Step 3: Try the boolean fix first — always check booleans before writing custom policy
getsebool -a | grep httpd
```

```text
httpd_anon_write --> off
httpd_builtin_scripting --> on
httpd_can_network_connect --> off
httpd_can_network_connect_db --> off
httpd_enable_homedirs --> off
httpd_read_user_content --> off
httpd_use_nfs --> off
...
```

No boolean directly covers custom data directories. The right fix here is to label the directory correctly.

```bash
# Step 4: Add a persistent file context rule for the custom path
sudo semanage fcontext -a -t httpd_sys_content_t "/srv/myapp/data(/.*)?"
```

```bash
# Step 5: Apply the label to existing files on disk
sudo restorecon -Rv /srv/myapp/data
```

```text
Relabeled /srv/myapp/data from system_u:object_r:var_t:s0 to system_u:object_r:httpd_sys_content_t:s0
Relabeled /srv/myapp/data/index.html from system_u:object_r:var_t:s0 to system_u:object_r:httpd_sys_content_t:s0
```

```bash
# Step 6: Verify the new label
ls -Z /srv/myapp/data/
```

```text
system_u:object_r:httpd_sys_content_t:s0 index.html
```

Now nginx can read those files. The fix is permanent: `semanage fcontext` writes to the policy database, and `restorecon` applies it. Even if you recreate those files, `restorecon` will always relabel them correctly.

The key insight here: the `semanage fcontext` + `restorecon` pattern is the correct, permanent fix. Using `chcon` directly changes the label on disk but does not update the policy — the next time SELinux relabels the filesystem (which happens at boot if triggered, or after `restorecon -R /`), your `chcon` changes are wiped out.

### Building an AppArmor Profile from Scratch

You have a new Go binary at `/usr/local/bin/myservice` that has no AppArmor profile. You want to confine it.

```bash
# Step 1: Generate a new profile interactively
# aa-genprof creates an empty profile, puts it in complain mode,
# then invokes aa-logprof to parse observed behavior
sudo aa-genprof /usr/local/bin/myservice
```

```text
Writing updated profile for /usr/local/bin/myservice.
Setting /usr/local/bin/myservice to complain mode.

Before you begin, you may wish to check if a
profile already exists for the application you
wish to confine. See the following wiki page for
more information:
https://gitlab.com/apparmor/apparmor/wikis/Profiling_with_tools

Profiling: /usr/local/bin/myservice

Please start the application to be profiled in
another window and exercise its functionality now.

Once completed, select the (S)can option below in
order to scan the system logs for AppArmor events,
...
[(S)can system log for AppArmor events] / (F)inish
```

In a second terminal, run `myservice` and exercise its normal functionality. Then come back and press `S`. `aa-genprof` reads the complain-mode log entries and interactively asks you to allow or deny each access pattern.

```bash
# Step 2: After the initial profile is built and the service runs in production,
# refine the profile based on accumulated complain log entries
# aa-logprof is what you use for ongoing refinement, without starting over
sudo aa-logprof
```

```bash
# Step 3: Move the profile to enforce mode
sudo aa-enforce /usr/local/bin/myservice
```

```bash
# Step 4: Confirm
sudo aa-status | grep myservice
```

```text
   /usr/local/bin/myservice
```

The entry appearing under "enforce mode" confirms it's now fully active.

## ⚠️ Gotchas

### Permissive Mode Is a Diagnostic Tool, Not a Solution

The most common MAC mistake: a service breaks, `setenforce 0` makes it work, so permissive mode stays in place permanently. This is not a fix. This is the equivalent of removing the circuit breaker because the breaker kept tripping.

```bash
# What looks like a fix — do not do this permanently
sudo setenforce 0
systemctl restart nginx
# nginx works now... but you've disabled the entire MAC enforcement layer
```

When a service works in permissive mode and fails in enforcing mode, that tells you one thing: SELinux is blocking a specific access, and you need to find and fix the policy, not disable the policy. The system is working correctly — it found an access violation and reported it. Your job is to determine whether the access is legitimate (fix the label or add a boolean) or illegitimate (your service is doing something it shouldn't).

If you put a RHEL 9 system in permissive mode, every service on that box now runs without MAC containment. The containment of httpd, sshd, named, and everything else is gone. You've widened the blast radius of every potential compromise from "this service only" to "the entire system."

The correct diagnostic flow:
1. Set permissive mode temporarily: `sudo setenforce 0`
2. Reproduce the failure to confirm SELinux is the cause
3. Check `audit.log` for the specific denial
4. Fix the label, boolean, or port context
5. Set enforcing again: `sudo setenforce 1`
6. Test
7. Update `/etc/selinux/config` only after you've confirmed enforcing works

```bash
# Temporary diagnostic — note that this does not survive a reboot
sudo setenforce 0

# Permanent mode change requires editing the config file AND rebooting, or setenforce
# Never set SELINUX=disabled unless you are certain and have a plan to relabel
```

### `audit2allow` Generates Broad Policies if You Feed It Too Much

`audit2allow` reads denial log entries and generates SELinux policy modules that would allow those denials. The problem is that if you feed it an entire audit log, you get a policy that allows everything that was ever denied — including things that were denied for good reason.

```bash
# This is dangerous — do not run this way
sudo audit2allow -a -M mypolicy
sudo semodule -i mypolicy.pp
# You've now allowed every SELinux denial on the entire system
```

```bash
# The correct approach: isolate the specific denial you need to fix
# Use grep to extract only the relevant denial
sudo grep "httpd_t" /var/log/audit/audit.log | grep "avc: denied" | sudo audit2allow -M httpd-myapp
sudo cat httpd-myapp.te  # Always review the .te file before loading
sudo semodule -i httpd-myapp.pp
```

Before running `semodule -i`, always read the generated `.te` file. If you see rules like `allow httpd_t file_type:file { read write }` — that's a wildcard grant across all file types. That rule was generated because nginx touched many different file types during a busy period. That is not the rule you want. Narrow it down. The specific rule you want for the scenario above is `allow httpd_t httpd_sys_content_t:file { read open getattr }`.

Treat `audit2allow` output as a first draft from a tool that is correct about what was denied but knows nothing about what should be allowed.

### `chcon` Labels Don't Survive Relabeling

`chcon` changes the SELinux label on a file directly. It looks like a permanent fix but it is not.

```bash
# This appears to work
sudo chcon -t httpd_sys_content_t /srv/myapp/data/index.html
ls -Z /srv/myapp/data/index.html
```

```text
system_u:object_r:httpd_sys_content_t:s0 /srv/myapp/data/index.html
```

This label lives in the file's extended attributes. It is correct right now. But the moment anyone runs `restorecon` on that path, or the system performs a full filesystem relabeling (which happens when you switch from disabled to enforcing, or when the `/.autorelabel` file exists), the label is reset to whatever the policy database says — which is still `var_t` for `/srv/myapp/data` because you never updated the policy.

```bash
# The permanent fix: update the policy database first, then apply it
sudo semanage fcontext -a -t httpd_sys_content_t "/srv/myapp/data(/.*)?"
sudo restorecon -Rv /srv/myapp/data
```

The rule: `semanage fcontext` is the permanent record. `restorecon` is the application of that record to disk. Always do both.

### rsync and cp Don't Preserve SELinux Contexts

When you copy files using `cp` or `rsync`, the new files inherit the SELinux context of the *destination directory*, not the source file. Or in some cases, they inherit a generic context from the copy tool's process type. Either way, the context is almost certainly wrong.

```bash
# Copying data to a new location — this silently resets SELinux labels
rsync -av /var/www/html/ /srv/myapp/html/
ls -Z /srv/myapp/html/
```

```text
system_u:object_r:var_t:s0 index.html
# Wrong — should be httpd_sys_content_t
```

```bash
# After any bulk file copy or rsync, always relabel the destination
sudo restorecon -Rv /srv/myapp/html/
```

`rsync` has a `--xattrs` flag (`-X`) that copies extended attributes, which includes SELinux labels. But this only works if the destination path's policy context matches the source. If you're moving files from one path to a different path with a different policy context, `--xattrs` will copy the wrong label. `restorecon` is always the right final step.

### AppArmor Complain Mode Logs but Doesn't Block — Check the Logs Before Thinking It Works

When you put an AppArmor profile into complain mode, the application runs freely. A common mistake is to assume complain mode means "the profile is fine" and promote it to enforce mode without checking the logs first.

```bash
# Profile is in complain mode — application appears to work fine
sudo aa-complain /usr/local/bin/myservice
systemctl restart myservice
# Service starts, no errors — looks good
```

Behind the scenes, the kernel is logging every access that would have been denied, but letting them through. The service works because AppArmor is not blocking anything. If you enforce the profile without reviewing those logs, the service will break.

```bash
# Check what AppArmor would have blocked during complain mode
sudo journalctl -xe | grep apparmor | grep ALLOWED
# Or on systems with auditd:
sudo grep "apparmor=\"ALLOWED\"" /var/log/audit/audit.log
```

```bash
# Review the complain-mode events and update the profile before enforcing
sudo aa-logprof
# Then enforce
sudo aa-enforce /usr/local/bin/myservice
```

The correct workflow: deploy in complain mode, run realistic traffic or test scenarios, review the log, refine the profile with `aa-logprof`, then enforce. Never enforce without reviewing complain-mode logs first.

## Distro Notes

| Distro Family | MAC System | Default State | Key Packages |
|---|---|---|---|
| RHEL 9 / Rocky 9 / AlmaLinux 9 | SELinux | Enforcing, targeted policy | `selinux-policy-targeted`, `policycoreutils`, `policycoreutils-python-utils` |
| Fedora | SELinux | Enforcing, targeted policy | Same as RHEL |
| CentOS Stream 9 | SELinux | Enforcing, targeted policy | Same as RHEL |
| Ubuntu 24.04 LTS | AppArmor | Enabled, baked into kernel | `apparmor`, `apparmor-utils`, `apparmor-profiles` |
| Debian 12 | AppArmor | Installed, profiles vary by package | `apparmor`, `apparmor-utils` |
| Arch Linux | Neither by default | Disabled | Install `apparmor` from extra, enable manually |
| Alpine (containers) | Neither by default | Host kernel determines AppArmor availability | Relies on host |

**A note on disabling:** On RHEL/Rocky/Alma, setting `SELINUX=disabled` in `/etc/selinux/config` requires a reboot and triggers a full filesystem relabeling when you re-enable. On Ubuntu, disabling AppArmor is `systemctl disable apparmor` — but individual profiles can be put in complain mode without touching the service.

## Alternatives & Comparisons

| System | Model | Complexity | Primary Platform | Approach |
|---|---|---|---|---|
| SELinux | Label-based (type enforcement) | High | RHEL, Fedora, Android | Labels on every object; fine-grained, powerful, steep learning curve |
| AppArmor | Path-based profiles | Medium | Ubuntu, Debian, SUSE | Rules on filesystem paths; easier to write, less granular |
| Smack | Label-based, simpler | Low | Embedded Linux (Tizen) | Simplified label model, not common on servers |
| TOMOYO | Path-based, learning mode | Medium | Japan-origin, mainline kernel | Similar to AppArmor, less ecosystem tooling |

For server administration, your choice is determined by your distribution. SELinux on RHEL-family; AppArmor on Debian/Ubuntu-family. You don't choose between them — you learn whichever is on your system. The concepts transfer between the two even though the tools differ.

## Trade-offs

**Where this shines:**

- Containing a compromised service to a known-small blast radius — if httpd is exploited, the attacker is locked inside the `httpd_t` policy envelope
- Compliance requirements — PCI-DSS, CIS benchmarks, DISA STIGs all mandate MAC enabled and enforcing
- Defense in depth — MAC is the last line of defense after DAC, capabilities, and seccomp; it operates at the kernel level and cannot be circumvented by userspace tools
- Auditability — every denied access is logged with full context (process type, target type, syscall) — better forensic detail than anything DAC provides

**Where this costs you:**

- Complexity and operational overhead — every new service, custom path, or non-standard port requires policy work; this cost is real and should not be minimized
- Debugging difficulty — a service that silently fails because of a MAC denial looks identical to a service that silently fails for any other reason; knowing to check `audit.log` is non-obvious
- Risk of misconfiguration in the other direction — an overly permissive policy (generated by `audit2allow -a` with no review) may be worse than no MAC at all because it creates false confidence
- Context loss on file operations — cp, rsync, tar, and many backup tools don't preserve SELinux labels without extra flags; restoring from backup can strip all labels

**Performance note:** SELinux adds a measurable but small overhead on syscall-heavy workloads. Benchmarks consistently show 1–3% throughput reduction on I/O-intensive services. The AVC cache keeps this low for steady-state workloads. This is not a reason to disable SELinux in production.

## When to Use / When to Avoid

✅ **Use MAC (keep it enforcing) when:**

- You are running any internet-facing service — the entire value proposition of MAC is containing compromise, which only applies to services exposed to untrusted input
- Your compliance requirements mandate it — and they almost always do at this level
- You have or are building policy for your services — even a basic correct policy is far better than no policy
- You are in a shared hosting or multi-tenant environment — MAC is one of the few mechanisms that can constrain what one tenant's process can do to another's files

❌ **Avoid disabling MAC when:**

- A service is misbehaving — this is exactly the situation where MAC is protecting you. Disable the protection and the service might work, but now you've also removed the protection from every other service on the box. Diagnose the denial and fix the policy.
- You think it's "just a dev box" — habits formed on dev boxes become production habits. Run enforcing everywhere so the habit sticks.

⚠️ **Common mistakes:**

- Running `setenforce 0` and leaving it — this is the single most common SELinux mistake in the field; the system logs it, auditors catch it, and it invalidates your security posture
- Applying `chcon` without `semanage fcontext` — the label is temporary and will be reset
- Enforcing an AppArmor profile without reviewing complain-mode logs first — the service will break in unpredictable ways
- Using `audit2allow -a` as a one-shot fix — you are likely granting far more access than the service needs; always review the generated `.te` file
- Assuming AppArmor's default Ubuntu profiles cover your custom configuration — default profiles cover default paths; if you change nginx's document root, you need to update the AppArmor profile

## Veteran Perspective

The number of production RHEL systems I've encountered running `SELINUX=permissive` is genuinely depressing. The comment in the `/etc/selinux/config` file almost always says "TODO: fix this properly." That TODO is years old. The system is effectively unconfined and nobody remembers who set it to permissive or why. This is not a theoretical risk — it is the default state of the majority of "SELinux enabled" systems in the wild.

Here's the thing about SELinux that takes a while to internalize: it is not adversarial. It is not trying to make your life difficult. When SELinux blocks something, it is doing exactly what you asked it to do when you installed and enabled it. You told the system: "Only allow what the policy explicitly permits." The policy doesn't explicitly permit your new service's access to a custom directory, so SELinux blocks it. That is the correct behavior. Your job is to express the policy correctly, not to turn off the system that's doing its job.

The second thing that takes time to internalize: the audit log is your friend, not your enemy. Every `avc: denied` line is a gift. It tells you exactly what process, what action, what target, and what type mismatch caused the denial. Almost no other subsystem gives you this level of detail automatically. A segfault gives you a core dump. A permission error gives you `EACCES`. SELinux gives you a full narrative: "The process running as `httpd_t` attempted to `read` the file `/srv/myapp/secret.key` which has type `admin_home_t`. This access was not in the policy." That is an incredibly precise description of what happened. Use it.

The distinction between `semanage fcontext` + `restorecon` and plain `chcon` is the thing that separates someone who has read the man page from someone who has managed SELinux in production. `chcon` is fast and seems to work. It fails silently the first time someone runs `restorecon` on that path, which might be three months later during an unrelated backup restore, which is when you discover that your "SELinux fix" was never real.

On the AppArmor side: the tool ecosystem on Ubuntu is mature and the defaults are solid, but default profiles only cover default configurations. If you install nginx from the package manager and use the default config, the AppArmor profile is correct. The moment you change the document root, add a custom log path, or compile a module that opens a new type of connection, you're outside the profile. The service will appear to work because the relevant access either wasn't caught in complain mode or was allowed by an overly broad glob rule. Check your denials after every significant configuration change.

For interviews: if someone asks you how to troubleshoot a service that's failing on RHEL, "check SELinux" is always on the list. The full answer is: check `ausearch -m avc -ts recent` or `grep avc /var/log/audit/audit.log`, identify the denial, check if a boolean fixes it, fix the file context with `semanage fcontext` + `restorecon`, and only fall back to a custom policy module as a last resort. Stating that flow clearly, with the correct tools in the correct order, demonstrates that you've actually debugged SELinux on a real system.

## Try It Yourself

1. Run `sestatus` on a RHEL/Rocky/Alma system (or `sudo aa-status` on Ubuntu) and identify: what mode is it in, what policy is loaded, and how many profiles or booleans are active. This is your baseline read of the MAC state.

2. Run `ps -eZ | grep sshd` on a SELinux system to see the type label on the SSH daemon. Then run `ls -Z /etc/ssh/sshd_config` to see the file's label. Look up what `sshd_t` is allowed to read using `sesearch --allow -s sshd_t -c file` — you'll see the full policy for that type. (Requires `setools-console` package.)

3. On a SELinux system, create a file at `/tmp/test-selinux.txt` and check its label with `ls -Z`. Now move it to `/var/www/html/test.txt` and check the label again — does it match what nginx expects? Then run `restorecon -v /var/www/html/test.txt` and check the label one more time. You just observed the difference between inherited context and policy-defined context.

4. List all SELinux booleans related to your web server with `getsebool -a | grep httpd`. Toggle one — `setsebool httpd_can_network_connect on` — observe that it's now `on`, then turn it back off. Note that without `-P`, the boolean reverts at reboot. Run it with `-P` to make it persistent. Check `/etc/selinux/targeted/active/booleans.local` or `semanage boolean -l` to see the persistent state.

5. **Challenge (requires a VM):** Install `nginx` on a RHEL 9 VM. Change nginx's document root to `/srv/mysite/html`. Create an `index.html` there. nginx will return 403. Without using permissive mode, diagnose the denial in `/var/log/audit/audit.log`, apply the correct `semanage fcontext` rule, run `restorecon`, and verify nginx returns 200. Then reboot the VM and verify the label is still correct — this confirms your fix is permanent and survived the reboot relabeling pass.

## Summary

- MAC enforces access policy at the kernel level and applies even to root — a compromised service is contained within its policy type, not free to roam the system
- SELinux uses type labels on every process and file; access is only allowed where the policy has an explicit allow rule — the default is deny
- When SELinux blocks something, the correct fix is `semanage fcontext` + `restorecon` for path labels, or `setsebool -P` for booleans — `chcon` is temporary and `setenforce 0` is not a fix, it's a diagnostic step
- `audit2allow` output is a draft, not a policy — always review the generated `.te` file and narrow overly broad rules before loading a custom module with `semodule -i`
- AppArmor uses path-based profiles in enforce or complain mode; always run complain mode before enforce mode and review `aa-logprof` output before promoting a profile to enforce
