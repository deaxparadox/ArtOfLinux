# 13.4 — Kernel Hardening with sysctl

## The Problem This Solves

A stock Linux installation is configured for compatibility, not security. The kernel ships with network features enabled that most servers don't need — IP forwarding, ICMP redirect acceptance, source routing — because someone might need them. It exposes kernel addresses in `/proc` because debugging tools need them. It allows core dumps from setuid binaries because developers find that useful.

Every one of those defaults is a gift to an attacker. IP forwarding lets a compromised machine become a pivot point inside your network. Readable kernel addresses in `/proc/kallsyms` give exploit authors the symbol offsets they need to bypass ASLR. A core dump from a setuid process can contain password hashes and private keys written to a world-readable file.

The kernel has a runtime configuration interface — `sysctl` — that lets you close most of these gaps without recompiling anything. The problem is that the defaults exist for a reason, and changing them carelessly has caused more production outages than I can count. `ip_forward=0` will silently break Docker networking. `modules_disabled=1` locks the module subsystem permanently until reboot. The parameters interact in non-obvious ways, and the documentation is scattered across kernel source trees and security guides that contradict each other.

This topic gives you the set of parameters that matters, explains exactly what each one does and why, and tells you the traps before you walk into them.

## Mental Model

Think of a factory building with doors. When the factory opens, the facilities team unlocks every door because they don't know yet which ones workers will need. Over time, the security team walks through and locks every door that no one is actually using — storerooms, side exits, old maintenance hatches. The factory still works exactly the same for everyone doing legitimate work. But now an intruder can't use the unlocked storeroom to move quietly between wings.

`sysctl` hardening is that walkthrough. You're not rebuilding the factory. You're systematically locking doors that were left open by default because they might be useful, but on your specific machine, they aren't — and they represent attack surface.

## Theory

The Linux kernel exposes a virtual filesystem at `/proc/sys/` that contains tunable kernel parameters as pseudo-files. Reading a file reads the current value of that parameter. Writing to a file changes the kernel's behavior immediately, at runtime, without a restart.

`sysctl` is the command-line interface to this filesystem. Under the hood, it is doing nothing more than reading and writing files under `/proc/sys/`. The parameter name `net.ipv4.tcp_syncookies` maps to the file `/proc/sys/net/ipv4/tcp_syncookies`. Dots become path separators.

Parameters are organized into namespaces:

- `kernel.*` — core kernel behavior: ASLR, core dumps, module loading, pointer visibility
- `net.ipv4.*` — IPv4 networking behavior: forwarding, TCP hardening, ICMP handling
- `net.ipv6.*` — IPv6 equivalents
- `fs.*` — filesystem behavior: core dump permissions, file handle limits
- `vm.*` — virtual memory subsystem

Runtime changes made with `sysctl -w` are lost on reboot. To persist settings across reboots, write them to configuration files:

- `/etc/sysctl.conf` — the traditional single file; still works on all distributions
- `/etc/sysctl.d/*.conf` — the modern drop-in directory; each file is applied in alphabetical order

The drop-in directory is the right approach on any modern system. It lets you keep distro-provided defaults in files like `10-console-messages.conf` and add your hardening in `99-hardening.conf`, applied last, overriding whatever came before. The naming convention matters — files are sorted lexicographically, so `99-` reliably comes after `10-`.

The critical invariant: kernel parameters set via sysctl are not persistent across reboots unless written to a config file and loaded at boot. An attacker who can reboot your machine effectively resets your runtime-only hardening.

## How It Works (Mechanics)

When the kernel boots, `systemd-sysctl.service` (or the equivalent init script on older systems) reads all configuration files from these locations in order:

```text
/etc/sysctl.d/*.conf
/run/sysctl.d/*.conf
/usr/lib/sysctl.d/*.conf
/etc/sysctl.conf
```

Files are processed in lexicographic order within each directory. Later files override earlier ones. This is why naming your hardening file `99-hardening.conf` is correct — it applies after the distro's `10-*.conf` defaults.

When you call `sysctl -w net.ipv4.tcp_syncookies=1`, the kernel executes this:

```text
# Conceptual flow — not a real command
sysctl -w → open /proc/sys/net/ipv4/tcp_syncookies → write "1" → kernel handler invoked → parameter updated in kernel memory
```

The kernel handler for each parameter validates the value (rejecting invalid inputs) and applies the change immediately. Some parameters — notably `kernel.modules_disabled` — are one-way switches: once set to 1, they cannot be changed back in the same boot session, even by root.

The `rp_filter` (reverse path filter) parameter works by checking every incoming packet: the kernel looks up the source address in the routing table and verifies that the interface the packet arrived on is also the interface the kernel would use to send a reply back to that source. If not, it's likely a spoofed packet arriving from an unexpected direction, and it's dropped.

SYN cookies work by encoding session state into the TCP sequence number of the SYN-ACK response. When the SYN backlog queue is full, instead of dropping connections, the server encodes the session parameters (timestamps, window size, MSS) into the sequence number. When the legitimate client sends an ACK, the server reconstructs the state from the sequence number without having stored anything. A flooder sending SYN packets with fake source addresses never sends that final ACK, so no state is ever allocated for those connections.

## Basic Example

Read and temporarily set a kernel parameter, then verify it takes effect immediately:

```bash
# Read the current ASLR level
sysctl kernel.randomize_va_space
```

```text
kernel.randomize_va_space = 2
```

```bash
# Read the current value directly from the proc filesystem — same result
cat /proc/sys/kernel/randomize_va_space
```

```text
2
```

```bash
# Set a parameter at runtime (not persistent — lost on reboot)
sudo sysctl -w net.ipv4.tcp_syncookies=1
```

```text
net.ipv4.tcp_syncookies = 1
```

```bash
# Verify it took effect immediately
sysctl net.ipv4.tcp_syncookies
```

```text
net.ipv4.tcp_syncookies = 1
```

The output confirms the change is live. You're reading directly from the kernel's memory. If you reboot now, the value returns to whatever the config files specify. Run this yourself and confirm that dots in the parameter name correspond exactly to slashes in the `/proc/sys/` path — they're the same thing.

## Real-World Example

This is a production sysctl hardening configuration, the kind you'd deploy via Ansible or drop into a freshly provisioned server. Every parameter is chosen for a reason.

```bash
# Create the hardening config file
sudo tee /etc/sysctl.d/99-hardening.conf > /dev/null << 'EOF'
# ============================================================
# Network Hardening
# ============================================================

# Disable IP forwarding — this machine is not a router.
# Enable this ONLY if you run Docker, WireGuard, OpenVPN, or act as a gateway.
net.ipv4.ip_forward = 0
net.ipv6.conf.all.forwarding = 0

# Do not send ICMP redirects — we are not a router
net.ipv4.conf.all.send_redirects = 0
net.ipv4.conf.default.send_redirects = 0

# Do not accept ICMP redirects — prevents route hijacking
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.default.accept_redirects = 0
net.ipv6.conf.all.accept_redirects = 0
net.ipv6.conf.default.accept_redirects = 0

# Do not accept secure ICMP redirects (from "trusted" gateways)
# They're not as secure as they sound
net.ipv4.conf.all.secure_redirects = 0
net.ipv4.conf.default.secure_redirects = 0

# Reject source-routed packets — the sender should not control the path
net.ipv4.conf.all.accept_source_route = 0
net.ipv4.conf.default.accept_source_route = 0
net.ipv6.conf.all.accept_source_route = 0

# Enable reverse path filtering — drops spoofed packets
# 1 = strict mode; use 2 (loose) only on asymmetric routing setups
net.ipv4.conf.all.rp_filter = 1
net.ipv4.conf.default.rp_filter = 1

# Enable SYN cookies — the right response to a SYN flood
net.ipv4.tcp_syncookies = 1

# Log packets from impossible addresses (martians) — useful for detecting spoofing
net.ipv4.conf.all.log_martians = 1
net.ipv4.conf.default.log_martians = 1

# Ignore ICMP broadcast pings — prevents smurf amplification attacks
net.ipv4.icmp_echo_ignore_broadcasts = 1

# Ignore bogus ICMP error responses
net.ipv4.icmp_ignore_bogus_error_responses = 1

# Enable RFC 1337 fix — prevents TIME_WAIT assassination
net.ipv4.tcp_rfc1337 = 1

# ============================================================
# Kernel Information Exposure
# ============================================================

# Hide kernel symbol addresses from unprivileged users and root.
# 0 = expose to all; 1 = hide from non-root; 2 = hide from everyone including root
# Use 2 in production. Debugging tools can use /proc/kallsyms as root with kptr_restrict=1 if needed.
kernel.kptr_restrict = 2

# Restrict dmesg to root only — kernel logs reveal hardware layout and driver details
# that help attackers fingerprint your system
kernel.dmesg_restrict = 1

# ============================================================
# Memory Hardening
# ============================================================

# Full ASLR — randomizes stack, VDSO, shared memory, AND heap.
# Value 1 is wrong: it skips heap randomization. Always use 2.
kernel.randomize_va_space = 2

# ============================================================
# Core Dumps
# ============================================================

# Disable core dumps from setuid programs — they can contain secrets
# (passwords, private keys, session tokens) and are written as the process owner
fs.suid_dumpable = 0

# Redirect all core dumps to /dev/null on this production system.
# In development, change this to: |/usr/lib/systemd/systemd-coredump %P %u %g %s %t %c %h
kernel.core_pattern = /dev/null

# ============================================================
# Module Loading
# ============================================================

# Prevent loading of kernel modules after boot is complete.
# WARNING: This is a one-way switch — cannot be reversed without reboot.
# Ensure all required modules are already loaded before enabling this.
# Verify with: lsmod | grep -E 'required_module' before enabling.
# kernel.modules_disabled = 1
EOF
```

```bash
# Apply all settings from /etc/sysctl.d/ and /etc/sysctl.conf in one pass
# This is NOT the same as sysctl -p — see Gotchas
sudo sysctl --system
```

```text
* Applying /usr/lib/sysctl.d/50-default.conf ...
* Applying /usr/lib/sysctl.d/50-pid-max.conf ...
* Applying /etc/sysctl.d/99-hardening.conf ...
net.ipv4.ip_forward = 0
net.ipv6.conf.all.forwarding = 0
net.ipv4.conf.all.send_redirects = 0
...
kernel.randomize_va_space = 2
fs.suid_dumpable = 0
kernel.core_pattern = /dev/null
```

```bash
# Verify a selection of critical parameters are actually set
sysctl kernel.randomize_va_space kernel.kptr_restrict net.ipv4.tcp_syncookies net.ipv4.ip_forward
```

```text
kernel.randomize_va_space = 2
kernel.kptr_restrict = 2
net.ipv4.tcp_syncookies = 1
net.ipv4.ip_forward = 0
```

Notice the walkthrough of what each group accomplishes:

The **network hardening** block closes the most commonly exploited default-open features. `send_redirects` and `accept_redirects` deal with ICMP redirect messages — an ICMP redirect tells a host "use a different gateway for this destination." That's a legitimate optimization, but it's also a textbook route hijacking vector. On a server that isn't a router, there is no reason to send or accept them. `rp_filter=1` (strict mode) kills spoofed-source-address attacks dead — if the reply path doesn't match the arrival interface, the packet is silently dropped.

The **information exposure** block is about denying attackers the reconnaissance they need. `kptr_restrict=2` zeroes out all kernel pointer values visible through `/proc` — tools like `cat /proc/kallsyms` return addresses of `0000000000000000` for every symbol. An exploit author needs real addresses to bypass ASLR and build reliable shellcode; hiding them makes their job significantly harder. `dmesg_restrict=1` keeps kernel boot messages and hardware information out of reach of unprivileged processes.

The **core dump** block is often skipped and shouldn't be. A process crash can happen to any daemon, including ones handling authentication tokens or private keys. If that crash produces a core dump that lands in a world-accessible location, you've handed an attacker a snapshot of process memory that may contain every secret the process was holding. `fs.suid_dumpable=0` ensures setuid processes (like `sudo`, `su`, `passwd`) cannot produce core dumps at all. `kernel.core_pattern=/dev/null` is belt-and-suspenders: even for non-setuid crashes, the dump goes nowhere.

`kernel.modules_disabled=1` is left commented out in this config because it requires human verification first. Once set, you cannot load any new kernel module — no `modprobe`, no inserting a network driver, nothing — until the next reboot. Run `lsmod` and verify your required modules are loaded, then enable it. It's a strong lockdown for systems where you know exactly what the kernel needs.

## ⚠️ Gotchas

### `sysctl -p` Only Loads `/etc/sysctl.conf` — Not Your Drop-in Files

This is the most common mistake, and it's invisible — you run `sysctl -p`, see output, and assume your config in `/etc/sysctl.d/99-hardening.conf` was applied. It wasn't.

```bash
# This looks like it's applying all your config
sudo sysctl -p
```

```text
net.ipv4.ip_forward = 0
kernel.randomize_va_space = 2
```

If you see output, it loaded `/etc/sysctl.conf`. Your drop-in file in `/etc/sysctl.d/` was silently ignored. There's no error — `sysctl -p` without a filename argument only reads `/etc/sysctl.conf` by design. It accepts an optional filename argument, so `sysctl -p /etc/sysctl.d/99-hardening.conf` loads just that file, but even that misses other drop-in files.

The correct command to load everything — all drop-in files, in the correct order, exactly as the init system does — is:

```bash
# Load ALL sysctl configuration from all standard paths, in correct order
sudo sysctl --system
```

```text
* Applying /usr/lib/sysctl.d/50-default.conf ...
...
* Applying /etc/sysctl.d/99-hardening.conf ...
net.ipv4.ip_forward = 0
...
```

Use `sysctl --system` whenever you modify any file under `/etc/sysctl.d/`. Write this on the inside of your eyelids if you work with hardened servers.

---

### Drop-in Files Are Applied in Alphabetical Order — Naming Is Not Optional

Two configuration files defining the same parameter: the last one wins. Distros ship their own defaults in files with names like `10-console-messages.conf`, `50-default.conf`, `50-pid-max.conf`. If your hardening file is named `00-hardening.conf`, it applies before those — and the distro defaults that come later will silently undo your settings.

```bash
# This naming is wrong — a distro file named 50-default.conf will override it
sudo tee /etc/sysctl.d/00-hardening.conf > /dev/null << 'EOF'
kernel.randomize_va_space = 2
EOF

# Apply and check
sudo sysctl --system
sysctl kernel.randomize_va_space
```

```text
kernel.randomize_va_space = 0   # Overridden by 50-default.conf
```

The fix is naming your hardening file with a high prefix:

```bash
# This naming is correct — 99 guarantees it applies after all distro defaults
sudo mv /etc/sysctl.d/00-hardening.conf /etc/sysctl.d/99-hardening.conf
sudo sysctl --system
sysctl kernel.randomize_va_space
```

```text
kernel.randomize_va_space = 2
```

The convention is `99-` for anything you want to be the final word. If you need something to apply before distro defaults, use `10-`. If you need it to override distro defaults but leave room for site-level overrides, use `60-`.

---

### `ip_forward=0` Silently Breaks Docker, WireGuard, and OpenVPN

If you apply the hardening config above on a machine running Docker and then wonder why your containers can no longer reach the internet, it's `ip_forward=0`. Docker uses the Linux kernel's routing subsystem to connect containers to the outside world. Containers have their own network interfaces; traffic from a container destined for the internet must be forwarded through the host interface. With forwarding disabled, the kernel drops those packets.

```bash
# After applying hardening config with ip_forward=0, Docker containers lose internet
docker run --rm alpine ping -c2 8.8.8.8
```

```text
PING 8.8.8.8 (8.8.8.8): 56 data bytes
--- 8.8.8.8 ping statistics ---
2 packets transmitted, 0 packets received, 100% packet loss
```

The same breakage affects WireGuard (when used as a gateway/router, not just as a client tunnel), OpenVPN server mode, KVM/QEMU NAT networking, and any application that uses network namespaces to bridge traffic.

The fix is conditional: only disable `ip_forward` when the machine is not doing routing. If it is:

```bash
# Override specifically for the machine's role
# In /etc/sysctl.d/99-hardening.conf:
# Uncomment the line below if this machine runs Docker, WireGuard, OpenVPN, or KVM
# net.ipv4.ip_forward = 1
sudo sysctl -w net.ipv4.ip_forward=1
```

Check before you deploy: `docker ps`, `wg show`, `ip rule show` — if any routing is happening, you need forwarding on.

---

### `randomize_va_space=1` Is Wrong — Heap Is Not Randomized

Many guides, blog posts, and even some vendor documentation show `kernel.randomize_va_space = 1`. This is incorrect for a hardened production system.

Value 1 randomizes the stack, VDSO, and shared memory regions. Value 2 additionally randomizes the heap. The heap is where most dynamic memory allocations happen — it's where secrets live, where buffers are allocated, where long-running daemons keep their working data. An attacker who knows the heap address can leverage a heap overflow exploit with much greater reliability.

```bash
# Wrong — heap is not randomized, this is weaker than it looks
echo "kernel.randomize_va_space = 1" | sudo tee /etc/sysctl.d/99-hardening.conf

# Right — stack, VDSO, shared memory, AND heap are all randomized
echo "kernel.randomize_va_space = 2" | sudo tee /etc/sysctl.d/99-hardening.conf
```

Value 2 is the Linux default on most modern distributions. If you're auditing a system and see `=1`, it was deliberately weakened — find out why.

---

### `kernel.modules_disabled=1` Is a One-Way Door in This Boot Session

Once you set `kernel.modules_disabled=1`, the kernel refuses all subsequent module load requests — permanently, until reboot. You cannot reverse it with `sysctl -w kernel.modules_disabled=0`. The kernel ignores the write.

```bash
sudo sysctl -w kernel.modules_disabled=1

# Try to load a module — fails silently or with an error
sudo modprobe nf_conntrack
```

```text
modprobe: ERROR: could not insert 'nf_conntrack': Operation not permitted
```

```bash
# Try to undo it — the kernel ignores this write
sudo sysctl -w kernel.modules_disabled=0
sysctl kernel.modules_disabled
```

```text
kernel.modules_disabled = 1   # Still 1 — unchanged
```

The correct workflow is: boot the system, ensure all required modules are loaded (networking drivers, filesystem modules, security modules), verify with `lsmod`, then set `modules_disabled=1`. Add it to your sysctl config so it applies at every subsequent boot after the early-boot module loading is complete.

> ⚠️ **WARNING:** Setting `kernel.modules_disabled=1` in a sysctl config that loads too early in the boot process — before required drivers are loaded — can make the system unbootable or non-functional. Test in a VM before applying to production. Confirm required modules with `lsmod` on a running system first.

---

### `kptr_restrict=2` Breaks Some Profiling and Debugging Tools

With `kptr_restrict=2`, kernel symbol addresses in `/proc/kallsyms` and similar files are replaced with zeros even for root. Tools that rely on kernel symbol resolution — `perf`, `bpftrace`, some eBPF programs, certain audit frameworks — may fail or produce meaningless output.

```bash
# With kptr_restrict=2, symbol addresses are hidden even from root
sudo cat /proc/kallsyms | head -3
```

```text
0000000000000000 T startup_64
0000000000000000 T secondary_startup_64
0000000000000000 T startup_64_array
```

If you need to run a profiling session or debug the kernel, temporarily lower it:

```bash
# Temporarily allow root to see kernel pointers — restore afterward
sudo sysctl -w kernel.kptr_restrict=1
# ... do your profiling ...
sudo sysctl -w kernel.kptr_restrict=2
```

Do not leave `kptr_restrict=1` in your persistent config. Root on a compromised system is a real threat, and you want even root to face this barrier.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | Drop-in files in `/etc/sysctl.d/`; distro ships defaults in `/usr/lib/sysctl.d/`; `sysctl --system` loads everything correctly |
| RHEL / Fedora / Rocky / AlmaLinux | Same drop-in structure; CIS-hardened images from vendors often ship with many of these already set; verify before duplicating |
| Arch Linux | Minimal distro defaults; `/etc/sysctl.d/` is the correct location; very few pre-existing sysctl files to worry about ordering against |
| Alpine (containers) | Often run with a read-only root; many sysctl parameters can be set from the container host's config instead; `ip_forward` is typically managed by the container runtime |

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| `sysctl` via `/etc/sysctl.d/` | All production servers — the standard approach | Flexible, auditable, applied at every boot |
| Security-focused kernel (grsecurity, linux-hardened) | Extremely high-security environments where sysctl alone is insufficient | Not available in mainline; requires custom kernel builds; grsecurity is commercial for production use |
| Container-level sysctl namespacing | Containers that need different settings from the host | Kubernetes `securityContext.sysctls`; Docker `--sysctl`; only namespaced parameters can be overridden per container |
| CIS-CAT / OpenSCAP automated hardening | Compliance-driven environments (PCI-DSS, SOC 2) | Generates and applies a full baseline; good for large fleets, but understand what it changes before applying |

Manual `sysctl` configuration via drop-in files is the right approach for the overwhelming majority of cases. It's transparent, version-controllable, and human-readable. The alternatives add complexity that isn't justified unless you have a specific compliance mandate or are operating in a threat model where the mainline kernel itself isn't trusted.

## Trade-offs

**Where this shines:**

- Runtime changes — no reboot needed to apply or test settings
- Low overhead — kernel parameters are not performance-sensitive (except VM tuning)
- Auditable — every setting lives in a text file under `/etc/sysctl.d/`; `git diff` tells you what changed
- Defense-in-depth — each parameter closes a specific attack vector without affecting unrelated functionality

**Where this costs you:**

- `ip_forward=0` breaks all routing — Docker, WireGuard, KVM, OpenVPN will lose connectivity without exception
- `modules_disabled=1` prevents emergency driver loading — if a network card needs a driver that wasn't loaded at boot, you're rebooting
- `kptr_restrict=2` and `dmesg_restrict=1` impede kernel debugging, profiling, and some eBPF tooling for everyone including root
- Settings that are wrong for your specific hardware can cause unexpected behavior — `rp_filter=1` (strict mode) breaks asymmetric routing, which is common in multi-homed servers and BGP setups

**Performance note:** `rp_filter=1` adds a routing lookup per incoming packet. On extremely high-throughput systems (10Gbit+, millions of packets per second), this is measurable. The mitigation is `rp_filter=2` (loose mode) rather than disabling it entirely, but loose mode provides less spoofing protection.

---

## When to Use / When to Avoid

✅ **Use this when:**

- Hardening any server that will face the internet — apply all network hardening parameters as baseline
- Building a new server image or AMI — bake the sysctl config in before the first deployment
- Conducting a security audit — `sysctl -a | grep -E 'ip_forward|randomize_va|kptr|dmesg'` gives you a quick read on where a system stands
- Responding to a vulnerability advisory that recommends a specific parameter change — `sysctl -w` lets you apply it in seconds without a maintenance window

❌ **Avoid this when:**

- `ip_forward=0` — avoid on Docker hosts, VPN gateways, Kubernetes nodes, KVM hypervisors, or any machine acting as a NAT router; leave it at 1 and document why
- `rp_filter=1` (strict mode) — avoid on multi-homed servers with asymmetric routing or BGP peers; use `rp_filter=2` (loose mode) instead
- `kernel.modules_disabled=1` — avoid applying this before thoroughly auditing which modules the system requires; incorrect application can require a rescue boot

⚠️ **Common mistakes:**

- Running `sysctl -p` and assuming all drop-in files were applied — use `sysctl --system`
- Copying a hardening blog post verbatim without verifying each parameter against the machine's role
- Setting `randomize_va_space=1` instead of `2` — the difference matters and `1` is frequently cited incorrectly
- Not testing in a VM before applying to a production fleet — one wrong parameter can take down a service silently

## Veteran Perspective

Here's what I actually say when reviewing a server hardening PR:

The most dangerous moment is when someone applies a sysctl hardening baseline from a security guide and declares victory. The guide was written for a generic server. Your server is not generic. `ip_forward=0` on a Kubernetes node will quietly break pod-to-pod communication, and the symptom is intermittent timeouts that take hours to trace back to a sysctl parameter that was set three weeks ago. I have watched this exact chain of events play out twice.

The second most dangerous moment is `kernel.modules_disabled=1` applied from an Ansible playbook that runs before the storage drivers are confirmed loaded. If the iSCSI or NFS driver that mounts `/var` wasn't loaded before that parameter was set, your server will boot fine and then hang trying to mount a filesystem. It's recoverable — single-user mode, fix the config — but it's also 2am and you're explaining to your manager why the deployment window went wrong.

The right way to think about sysctl hardening is incremental, not all-or-nothing. Start with the network parameters — they're safe on almost any server that isn't acting as a router, and the security benefit is immediate. Apply `randomize_va_space=2`, `kptr_restrict=2`, `dmesg_restrict=1` — these have no functional impact on normal server operation. Add `fs.suid_dumpable=0` and `kernel.core_pattern=/dev/null`. Now audit the machine's role before touching `ip_forward` or `modules_disabled`.

In a technical interview, you'll often get a question like: "How would you harden a new Linux server?" The answer that separates a senior from a junior is not the list of parameters — anyone can memorize those. It's understanding which parameters are safe universally, which depend on the machine's role, and which are irreversible at runtime. If you can articulate that `ip_forward=0` is the right default for a web server but wrong for a Docker host, and that `modules_disabled=1` requires pre-flight verification, you understand what you're doing. If you just recite the CIS Benchmark list, you're a configuration script with legs.

The war story worth telling: a team applied a comprehensive sysctl hardening config across their fleet using Ansible, including `rp_filter=1` (strict mode). Everything passed testing. Two weeks later, after a network infrastructure change moved one of the servers to a dual-homed configuration, inbound traffic from a secondary upstream started getting silently dropped. The server was receiving packets fine — `tcpdump` showed them arriving — but they never made it to the application. Six hours of debugging before someone thought to check sysctl. Strict `rp_filter` was rejecting the packets because the return path for that source address used the other interface. The fix was two characters: change `1` to `2`. The lesson is: document every non-default setting and why it was chosen, because the person debugging the problem at 2am is often not the person who applied it.

## Try It Yourself

1. Read the current values of five hardening-relevant parameters: `kernel.randomize_va_space`, `net.ipv4.tcp_syncookies`, `net.ipv4.ip_forward`, `kernel.kptr_restrict`, `kernel.dmesg_restrict`. Compare them to the recommended values in this topic — note which ones are already correct on your system and which aren't.

2. Check whether `/proc/sys/` and the sysctl parameter namespace map correctly: read `cat /proc/sys/net/ipv4/tcp_syncookies` and compare to `sysctl net.ipv4.tcp_syncookies`. They should return the same value.

3. List all existing `.conf` files under `/usr/lib/sysctl.d/` and `/etc/sysctl.d/` with `ls -la`. Identify which files are distro-provided and which (if any) are custom. Check the ordering — will a `99-` file you create override all of them?

4. Create `/etc/sysctl.d/99-hardening.conf` with only the network hardening parameters from this topic (skip `ip_forward` and `modules_disabled`). Apply with `sudo sysctl --system` and verify each setting with `sysctl -a | grep -E 'syncookies|rp_filter|accept_redirects|send_redirects'`. (Requires `sudo`; safe to do on any Linux system that is not acting as a router.)

5. **Challenge:** Write a small shell script that audits a system's sysctl configuration against the hardening baseline in this topic, reports which parameters are compliant and which are not, and flags `ip_forward=1` as informational rather than a failure (since it may be intentional). Run it against your own system and interpret the output. (Safe to run without `sudo`; requires only `sysctl -a` access.)

---

## Summary

- `sysctl` reads and writes kernel parameters via `/proc/sys/`; changes are immediate but lost on reboot unless written to `/etc/sysctl.d/*.conf` and applied with `sudo sysctl --system`
- `sysctl -p` only loads `/etc/sysctl.conf` — use `sysctl --system` to load all drop-in files in the correct order; name your file `99-hardening.conf` so it applies after distro defaults
- `net.ipv4.ip_forward=0` is correct for servers but must be `1` for Docker hosts, WireGuard gateways, OpenVPN servers, KVM hypervisors, and Kubernetes nodes — setting it wrong silently breaks all container and VM networking
- `kernel.randomize_va_space` must be `2`, not `1` — value `1` skips heap randomization; value `2` adds it; many guides show `1` incorrectly
- `kernel.modules_disabled=1` is a one-way switch for the current boot session — it cannot be reversed without a reboot, so verify all required modules are loaded before enabling it
