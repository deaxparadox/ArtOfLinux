# [13.5] — Audit Logging and Intrusion Detection

> Without audit logging, you cannot prove what happened — only guess. When an incident occurs, the difference between a two-hour recovery and a two-week investigation is whether the right events were recorded.

## The Problem This Solves

Something bad happened on your server. Maybe a file was changed that shouldn't have been. Maybe someone ran a command as root that escalated a breach. Maybe an account you thought was locked was used to authenticate at 3am. You don't know who, you don't know when, you don't know exactly what.

Without audit logging, the operating system gives you almost nothing to work with after the fact. `/var/log/syslog` and `/var/log/messages` tell you that services started and stopped. The shell history in `~/.bash_history` — if the attacker didn't wipe it — tells you what someone typed interactively. That's it.

There's no record of which files were read or written. There's no record of which system calls were made. There's no record of who ran what with elevated privileges. A skilled attacker who knows this can act with relative impunity and leave you with almost nothing to reconstruct the timeline.

Compliance frameworks — PCI-DSS, HIPAA, SOC 2, NIST 800-53 — mandate audit logging precisely because of this. But the operational argument is stronger than the compliance argument: a server without audit logging is a server where you're flying blind during an incident.

Linux has a better way. The kernel's audit subsystem sits at the syscall layer — below shell history, below application logging, below anything an attacker running in user space can suppress. It records what actually happened, not what someone reported happened.

---

## Mental Model

Think of a high-security building with physical access control. Every door has a reader. Every badge swipe is logged — who, which door, what time. But there's a second layer: motion cameras that record not just badge swipes but actual movement in the hallways. Even if someone propped a door open and walked in without swiping, the camera caught it.

`auditd` is the camera layer. Application logs are the badge swipe logs — they only record what the application chooses to record. `auditd` hooks into the kernel at the level of actual system calls, so it sees everything that physically happens: every file opened, every privilege used, every process spawned. An attacker can delete their shell history. They cannot delete what the kernel already handed to `auditd`.

`fail2ban` sits at the building entrance. When someone tries the wrong keycard combination too many times, the system automatically locks them out. It doesn't detect what happened inside the building — it prevents repeated failed attempts from becoming a successful one.

## Theory

Linux audit logging has two distinct layers. Understanding which layer does what prevents a lot of confusion.

**The kernel audit subsystem** is built into the kernel itself. It intercepts system calls and generates structured event records for any call that matches a configured rule. These records include the timestamp, the user (real UID, effective UID, login UID), the process, the syscall arguments, and the result. The kernel hands these records to `auditd` over a netlink socket.

**`auditd`** (the Linux Audit Daemon) is the user-space daemon that receives these kernel event records and writes them to disk — by default at `/var/log/audit/audit.log`. It also handles log rotation, rate limiting, and what to do when the disk fills up (configurable via `/etc/audit/auditd.conf`). Without `auditd` running, kernel audit events are discarded. With `auditd` running, they are reliably written to disk.

The rules that control which events the kernel generates are loaded with `auditctl`. There are three types:

**Control rules** modify the behavior of the audit system itself — the queue buffer size, the failure mode, whether the configuration can be locked.

**File watch rules** (`-w`) tell the kernel to generate an event whenever a specific file or directory is accessed. The `-p` flag specifies which access types to watch: `r` (read), `w` (write), `x` (execute), `a` (attribute change — permissions, ownership, timestamps).

**Syscall rules** (`-a`) tell the kernel to generate an event when a specific system call is made, optionally filtered by architecture, user ID, group ID, process ID, or other fields. This is more powerful but also more verbose.

Every rule should end with `-k keyname` — a string label that `ausearch` can use to retrieve only the events matching that rule. Without keys, you're trawling through thousands of unrelated events by hand.

Rules loaded with `auditctl` at the command line are **runtime rules**. They are active immediately but exist only until the next reboot. To survive a reboot, rules must be written to files under `/etc/audit/rules.d/` and loaded by `augenrules --load` (modern systems) or written directly to `/etc/audit/audit.rules` (older approach). The `rules.d/` directory is processed in alphabetical order, and the combined result is assembled into the active ruleset.

The **login UID** (`auid`) deserves special mention. When a user logs in, the kernel assigns them a login UID for that session. This UID is inherited by every process spawned from that session — including processes that switch users with `su` or `sudo`. This means audit logs can track "root did X" back to "this root session was started by the login of user alice." It's one of the most valuable features of the audit subsystem and the one most often overlooked.

## How It Works (Mechanics)

When you add a rule with `auditctl -w /etc/passwd -p wa -k passwd-change`, this is what happens:

1. `auditctl` makes a `AUDIT_ADD_RULE` netlink message to the kernel's audit subsystem.
2. The kernel registers the watch on `/etc/passwd` in its inode watch table.
3. When any process calls `openat(2)` or `write(2)` on `/etc/passwd`, the kernel audit hook fires.
4. The kernel constructs an audit event record and queues it on the kernel audit buffer.
5. `auditd` reads the event off the netlink socket and writes it to `/var/log/audit/audit.log`.

The resulting log entry looks like this (simplified):

```text
type=SYSCALL msg=audit(1718400000.123:4521): arch=c000003e syscall=257 success=yes exit=3 a0=ffffff9c a1=7f... a2=0 a3=0 items=1 ppid=1234 pid=5678 auid=1000 uid=0 gid=0 euid=0 ... comm="passwd" exe="/usr/bin/passwd" key="passwd-change"
type=PATH msg=audit(1718400000.123:4521): item=0 name="/etc/passwd" inode=655364 dev=fd:00 mode=0100644 ouid=0 ogid=0 rdev=00:00 objtype=NORMAL ...
```

Each audit event can span multiple records linked by the same timestamp and serial number. The `SYSCALL` record has the process-level context. The `PATH` record has the file-level context. `ausearch` and `aureport` understand these linked records and reassemble them.

For syscall rules, the architecture flag matters. Modern 64-bit systems run both 64-bit (`b64`) and 32-bit (`b32`) syscalls. A rule with `-F arch=b64 -S execve` only catches 64-bit `execve` calls. An attacker using a 32-bit binary will slip through. Production audit rules cover both:

```text
# Conceptual — not a runnable command, just the rule pair
-a always,exit -F arch=b64 -S execve -k exec-log
-a always,exit -F arch=b32 -S execve -k exec-log
```

`fail2ban` operates at a completely different layer. It watches log files — `/var/log/auth.log`, `/var/log/secure`, or application logs — for patterns that indicate failed authentication. When a configured threshold is crossed (e.g., 5 failed SSH logins within 10 minutes), it fires an action, typically inserting a firewall rule to block that IP. On RHEL/Fedora, the default action uses `firewalld`. On Debian/Ubuntu, it defaults to `iptables` (or nftables via the compat layer on modern systems).

The ban is temporary by default (600 seconds). The unban is handled by `fail2ban-server` itself via a scheduled action, not by the firewall's built-in timeout.

```bash
# Check if auditd is running
sudo systemctl status auditd
```

```text

     Loaded: loaded (/usr/lib/systemd/system/auditd.service; enabled; preset: enabled)
     Active: active (running) since Mon 2026-06-16 09:00:00 UTC; 2h 5min ago
```

```bash
# Add a temporary watch on /etc/passwd — tracks writes and attribute changes
sudo auditctl -w /etc/passwd -p wa -k passwd-change

# Verify it was loaded
sudo auditctl -l
```

```text
-w /etc/passwd -p wa -k passwd-change
```

```bash
# Trigger the watch: touch the file to change its timestamp (an attribute change)
sudo touch /etc/passwd

# Search the audit log for events tagged with our key
sudo ausearch -k passwd-change
```

```text
----
time->Mon Jun 16 09:05:23 2026
type=PROCTITLE msg=audit(1718528723.441:301): proctitle=746F756368002F6574632F706173737764
type=PATH msg=audit(1718528723.441:301): item=0 name="/etc/passwd" inode=655364 dev=fd:00 mode=0100644 ouid=0 ogid=0 rdev=00:00 objtype=NORMAL cap_fp=0 cap_fi=0 cap_fe=0 cap_fver=0 cap_frootid=0
type=CWD msg=audit(1718528723.441:301): cwd="/root"
type=SYSCALL msg=audit(1718528723.441:301): arch=c000003e syscall=257 success=yes exit=3 a0=ffffff9c a1=7f4c3b2a1000 a2=941 a3=1b6 items=1 ppid=4312 pid=5891 auid=1000 uid=0 gid=0 euid=0 egid=0 suid=0 sgid=0 fsuid=0 fsgid=0 tty=pts1 ses=3 comm="touch" exe="/usr/bin/touch" key="passwd-change"
```

The `auid=1000` field is the login UID — the real user who started this session, even if they later ran `sudo`. The `uid=0` tells you the command ran as root. Together, they answer "who actually did this?"

Try this yourself. After running `sudo touch /etc/passwd`, the event appears within seconds. Notice that `auid` and `uid` can differ — that's the whole point of the login UID.

## Real-World Example

This is a realistic incident investigation. Suppose you get an alert that `/etc/sudoers` was modified. Your job is to find out who did it, when, and what else they touched.

First, make sure the right audit rules are in place. These would live in `/etc/audit/rules.d/50-privileged.rules`:

```ini
# /etc/audit/rules.d/50-privileged.rules
# Watch privilege escalation configuration files
-w /etc/sudoers -p wa -k sudoers-change
-w /etc/sudoers.d/ -p wa -k sudoers-change

# Watch sensitive authentication files
-w /etc/passwd -p wa -k identity
-w /etc/shadow -p wa -k identity
-w /etc/gshadow -p wa -k identity
-w /etc/group -p wa -k identity

# Log all sudo usage (execve on sudo binary)
-a always,exit -F arch=b64 -S execve -F path=/usr/bin/sudo -k sudo-use
-a always,exit -F arch=b32 -S execve -F path=/usr/bin/sudo -k sudo-use
```

Load the rules (required after editing files in `rules.d/`):

```bash
sudo augenrules --load
```

Now walk through the investigation. A modification to `/etc/sudoers` already happened. Search for it:

```bash
# Find all events tagged sudoers-change, interpret numeric IDs into names
sudo ausearch -k sudoers-change -i
```

```text
----

type=PROCTITLE msg=audit(06/16/2026 02:17:44.331:889): proctitle=visudo
```

The `auid=deploy` field reveals that the `deploy` service account ran `visudo` at 02:17. Now look at what else that session did in the hour around that time:

This returns every audited event for that login session, ordered by time. You can see the full sequence: what commands were run, what files were touched, what processes were spawned.

## ⚠️ Gotchas

### `auditctl` Rules Vanish on Reboot

This is the most common audit logging mistake. You run `auditctl -w /etc/passwd -p wa -k passwd-change` on a production server. It works. You verify it. You close the ticket. Then the server reboots during a maintenance window and the rule is gone.

`auditctl` is for runtime rule management — immediate effect, no persistence. Rules loaded this way exist only in the kernel's memory.

```bash
# This looks right but it doesn't survive a reboot
sudo auditctl -w /etc/sudoers -p wa -k sudoers-change

# Verify it's loaded
sudo auditctl -l
```

```text
-w /etc/sudoers -p wa -k sudoers-change
```

After reboot:

```bash
sudo auditctl -l
```

```text
No rules
```

The correct approach: write the rule to a file under `/etc/audit/rules.d/` and load it with `augenrules`:

```bash
# Write the rule to a persistent file — use auditctl syntax without the 'auditctl' prefix
sudo tee /etc/audit/rules.d/50-privileged.rules <<'EOF'
-w /etc/sudoers -p wa -k sudoers-change
EOF

# Compile and load all rules.d/ files into the kernel
sudo augenrules --load
```

```bash
# Verify the persistent rule is now active
sudo auditctl -l
```

```text
-w /etc/sudoers -p wa -k sudoers-change
```

The `rules.d/` files are read by `augenrules` at boot (via the auditd service) and whenever you call `augenrules --load`. Files are processed alphabetically — use numeric prefixes like `10-`, `30-`, `50-`, `99-` to control ordering. The conventionally final file is `99-finalize.rules`, which typically locks the audit configuration with `-e 2` to prevent runtime modification.

### High-Volume `execve` Logging Will Fill Your Disk

Watching every `execve` call sounds like a great idea — you'd know every command ever run. In practice, on a moderately active server, this can generate 100MB or more of audit log per day. On a busy application server, gigabytes.

```bash
# This will generate an event for EVERY process launch — dangerous on busy systems
sudo auditctl -a always,exit -F arch=b64 -S execve -k exec-log
```

Within minutes on a production system:

```bash
sudo du -sh /var/log/audit/
```

```text
4.2G	/var/log/audit/
```

Two mitigation strategies work in practice. The first is to filter by UID — only audit `execve` calls from privileged accounts or from UIDs above the system user threshold:

```bash
# Only audit execve for login users (UID >= 1000) — skip daemons and system processes
sudo auditctl -a always,exit -F arch=b64 -S execve -F auid>=1000 -F auid!=unset -k exec-log
```

The second is to enable rate limiting in `/etc/audit/auditd.conf`:

```ini
# /etc/audit/auditd.conf
# Drop audit events if more than 100 are generated per second
rate_limit = 100
```

Use both. On production systems, also use `max_log_file_action = rotate` and `num_logs = 5` to cap how much disk audit logging can consume.

### `/var/log/auth.log` Does Not Exist on RHEL

Scripts and runbooks written on Ubuntu that hardcode `/var/log/auth.log` will silently produce no output on RHEL, Fedora, Rocky, or AlmaLinux. There is no error — the file simply doesn't exist, so `grep` or `cat` against it returns nothing.

```bash
# Works on Debian/Ubuntu — silently finds nothing on RHEL
grep "Failed password" /var/log/auth.log
```

On RHEL-family systems, SSH and PAM authentication events go to `/var/log/secure`:

```bash
# Works on RHEL/Fedora/Rocky/AlmaLinux
grep "Failed password" /var/log/secure
```

Write scripts that handle both:

```bash
# Handle both distro families
AUTH_LOG=""
if [ -f /var/log/auth.log ]; then
    AUTH_LOG="/var/log/auth.log"
elif [ -f /var/log/secure ]; then
    AUTH_LOG="/var/log/secure"
else
    echo "No auth log found" >&2
    exit 1
fi
grep "Failed password" "$AUTH_LOG"
```

Or use `journalctl`, which is universal and covers both:

```bash
# Works on any systemd-based system regardless of distro
journalctl -u sshd --since "1 hour ago" | grep "Failed password"
```

### `fail2ban` Uses Different Firewall Backends by Distro

After configuring a `fail2ban` jail on Ubuntu and testing it successfully, the same jail.local file deployed to a RHEL server may silently fail to ban IPs — or produce errors — because the default firewall backend differs.

On Debian/Ubuntu, `fail2ban` defaults to `iptables-multiport` as its `banaction`. On RHEL/Fedora/Rocky, `firewalld` is the default firewall manager, and `fail2ban` is packaged to use `firewallcmd-ipset` or `firewallcmd-rich-rules` as its `banaction`.

Check what's actually active after any config change:

```bash
sudo fail2ban-client status sshd
```

If the jail shows `Currently banned: 0` when you expected bans, or if `fail2ban-client` reports action errors in `/var/log/fail2ban.log`, the backend mismatch is the first place to look.

Explicitly set the backend in `/etc/fail2ban/jail.local` to remove ambiguity:

```ini
# /etc/fail2ban/jail.local

[DEFAULT]
# On RHEL/Fedora/Rocky — use firewalld
banaction = firewallcmd-ipset
banaction_allports = firewallcmd-allports

# On Debian/Ubuntu — use iptables (or nftables on modern installs)
# banaction = iptables-multiport
# banaction_allports = iptables-allports
```

Never assume the default is correct. Always check `fail2ban-client status <jailname>` after deployment.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `apt install auditd audispd-plugins fail2ban`; auth log at `/var/log/auth.log`; `fail2ban` defaults to `iptables` backend; rules in `/etc/audit/rules.d/` |
| RHEL / Fedora / CentOS / Rocky / Alma | `dnf install audit fail2ban`; auth log at `/var/log/secure`; `fail2ban` defaults to `firewalld` backend; `augenrules` available to compile `rules.d/`; SELinux AVC events also appear in audit log |
| Arch | `pacman -S audit fail2ban`; auth events via `journald` (`journalctl -u sshd`); no `/var/log/auth.log` by default |
| Alpine (containers) | `apk add audit`; minimal auditd, often disabled in containers; prefer container-level security policies over auditd inside containers |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `auditd` | Comprehensive kernel-level audit trail; compliance requirements; incident forensics | Built into the kernel; lowest layer available; high fidelity but verbose |
| `syslog` / `journald` | General application and service logging | Does not capture syscall-level events; can be suppressed by applications |
| `osquery` | Endpoint visibility with SQL queries; fleet-wide monitoring | Excellent for security analytics; requires agent deployment; higher resource cost |
| `fail2ban` | Automatic IP banning on repeated failed authentication | Reactive, not preventive; does not help against distributed/low-rate attacks |
| `sshguard` | Lighter-weight alternative to `fail2ban` for SSH specifically | Less configurable but simpler to set up; good for single-service cases |
| `crowdsec` | Modern, community-driven threat intelligence + IP banning | Shares threat intelligence across instances; heavier than fail2ban |

For compliance-driven environments, `auditd` is non-negotiable — nothing else runs at the same layer or produces the same level of evidence. For brute-force SSH protection, `fail2ban` is the industry standard, though increasingly `crowdsec` is being adopted for its shared threat intelligence.

## Trade-offs

**Where this shines:**

- `auditd` operates below the application layer — it cannot be suppressed by a compromised application or a deleted shell history
- Kernel audit events include login UIDs that survive `su`/`sudo` — you can trace root actions back to the originating human account
- `aureport` and `ausearch` make structured analysis of millions of events practical without writing custom log parsers
- `fail2ban` stops the vast majority of automated brute-force attacks without requiring any manual intervention

**Where this costs you:**

- High-volume syscall rules (unrestricted `execve` logging) generate enormous log volume — misconfigured audit rules have filled disks and crashed servers
- `auditd` rate limiting, when triggered, silently drops events — you may have gaps in your audit trail without knowing it
- `fail2ban` with aggressive thresholds can accidentally ban legitimate users (or even monitoring tools) that retry connections
- `/var/log/audit/audit.log` is a high-value target for attackers who want to cover their tracks; protect it with tight permissions and consider shipping logs to a remote syslog server immediately

**Performance note:** Unrestricted `execve` syscall auditing on a production application server can add measurable overhead — the stop-and-record on every process launch is not free. Filter by UID or limit to specific paths. File watch rules on `/etc/` directories are very low overhead in practice.

## When to Use / When to Avoid

✅ **Use this when:**

- You operate under any compliance framework (PCI-DSS, HIPAA, SOC 2, NIST) — most mandate kernel-level audit logging
- You need forensic evidence after an incident — auditd gives you a timeline that shell history and application logs cannot
- You are hardening a server that runs privileged operations — track every write to `/etc/`, every `sudo` use, every `cron` modification
- You expose SSH to the internet — `fail2ban` should be running on every public-facing server, unconditionally

❌ **Avoid this when:**

- Running `auditd` inside containers — it requires elevated privileges and the audit events go to the host kernel's audit subsystem anyway; configure auditd at the host level, not in the container
- Setting broad `execve` logging without UID filtering on high-throughput application servers — profile the event rate first in a test environment
- Relying on `fail2ban` alone as your SSH security strategy — it stops brute force, not credential stuffing or attacks using previously stolen keys; combine with key-only authentication and `AllowUsers`/`AllowGroups` in `sshd_config`

⚠️ **Common mistakes:**

- Forgetting `-F auid!=unset` in syscall rules — events from kernel threads have `auid=4294967295` (unset) and will flood your logs without this filter
- Writing rules with `auditctl` and not persisting them to `rules.d/` — then wondering after a reboot why events stopped being captured
- Not testing `fail2ban` bans actually work after deploy — check `fail2ban-client status <jailname>` and verify the firewall rule actually appears

## Veteran Perspective

The first thing a junior sysadmin does with auditd is add `execve` logging for everything. The first thing a senior sysadmin does is roll that back after the disk fills up. There is no audit policy that works right the first time. You tune it iteratively, watching what gets generated, cutting the noise, keeping the signal.

The hard-won rule: **audit what you can act on**. Logging every file read in `/proc` is noise. Logging every write to `/etc/`, every `cron` modification, every `sudoers` change, every authentication binary execution — that's signal. When you're designing audit rules, ask yourself: "If this event fires, would I want to know? Is there an action I'd take?" If the answer is no, the rule is overhead.

The login UID (`auid`) is the most important field in the audit record and the one least understood by people new to auditd. Here is the situation it was built for: an attacker compromises the `deploy` service account. They escalate to root using a vulnerability in a setuid binary. They do damage as root. Without the login UID, your audit trail shows "root did X" — and you have 12 root-capable accounts to investigate. With the login UID, it shows "root did X in a session that originated from the `deploy` login" — and your investigation is immediately focused.

On `/var/log/auth.log` vs `/var/log/secure`: I've seen incident response engineers spend 20 minutes convinced a server had no SSH logs before someone asked which distro they were on. Put both paths in your scripts. Better yet, switch to `journalctl -u sshd` which works everywhere systemd does, is queryable by time, and is harder to truncate accidentally.

`fail2ban` protects you from the automated bots that are hitting every public IP on the internet every day. It does not protect you from a targeted attack by a human using a single slow attempt from a rotating list of IPs. Don't let it give you false confidence on SSH security. The correct posture is: key-only auth, `fail2ban` for bots, `AllowUsers` to limit who can even attempt authentication, and a non-standard port if you want quieter logs (not real security, but it reduces noise). In that order. `fail2ban` is layer two, not layer one.

The war story that sticks with me: a client's staging server showed up in a threat intelligence feed as a source of SSH scanning. Nobody knew anything had happened — the attacker had gotten in, used the server as a pivot for scanning, and moved on. The audit log told the whole story in retrospect: an `execve` rule had captured the attacker running `masscan` and `nmap`. But the audit log had only been looked at after the tip from the threat feed. The data was there. Nobody was watching it. Audit logging without a log review process is filing a report that nobody reads.

---

## Try It Yourself

1. Verify `auditd` is running with `systemctl status auditd`. If it's not running, start it and check the logs for why it stopped.

2. Add a temporary watch on `/tmp/testfile` — create the file first with `touch /tmp/testfile`, add the rule with `auditctl`, write something to the file, then search for the event with `ausearch -k test-watch`. Remove the rule with `auditctl -W /tmp/testfile -p wa -k test-watch` when done.

3. Write a persistent rule for `/etc/cron.d/` to track any writes. Save it to `/etc/audit/rules.d/40-cron.rules`, load it with `sudo augenrules --load`, verify it appears in `auditctl -l`, then reboot and verify it survived.

4. Run `sudo aureport --summary --start today` on a system that's been running for a while. Find the count of failed logins. Then use `sudo ausearch -m USER_AUTH --start today -i` to see the actual failed authentication events. Match the count.

5. **Challenge (requires a VM):** Install `fail2ban`, configure a jail for SSH with `maxretry = 3` and `bantime = 120`, deliberately trigger 3 failed SSH logins from a second machine (or loopback), confirm the IP is banned with `fail2ban-client status sshd`, verify the firewall rule exists with `iptables -L -n` (or `firewall-cmd --list-rich-rules` on RHEL), then wait for the ban to expire and confirm the rule is removed.

---

## Summary

- `auditd` operates at the kernel syscall layer — below application logging, below shell history; a compromised application cannot suppress it
- Rules loaded with `auditctl` are runtime-only and vanish on reboot; persist them in `/etc/audit/rules.d/` and load with `augenrules --load`
- The login UID (`auid`) in audit records survives `su`/`sudo` escalation — it always traces root actions back to the originating user session
- `/var/log/auth.log` exists only on Debian/Ubuntu; on RHEL-family systems it's `/var/log/secure`; use `journalctl -u sshd` for distro-agnostic queries
- `fail2ban` stops automated brute-force attacks but uses different firewall backends by distro — always verify with `fail2ban-client status <jailname>` after deployment
