# [13.1] — Principle of Least Privilege

> Every process, user, and service should operate with the minimum access it needs to do its job — and the moment you grant more than that, you've made a bet that nothing in that process will ever go wrong.

## Why This Exists

In the earliest Unix systems, most daemons ran as root. It was simpler. Root could open any port, read any file, write anywhere. You didn't have to think about what the process needed — you just gave it everything.

The problem showed up when something went wrong. A buffer overflow in a network daemon. A remote code execution in a web server. A misconfigured database that accepted unauthenticated connections. When those things happened — and they happened constantly — the attacker didn't just own the vulnerable service. They owned the entire machine. Root is root. There's no containment.

The principle of least privilege is the structural answer to that problem. The idea is simple: a process should hold only the exact capabilities it needs to function. If a web server only needs to read files in `/var/www` and listen on port 80, then that web server should not be able to read `/etc/shadow`, write to `/usr/bin`, or mount filesystems. If an attacker compromises that web server, they get what the web server had — which is as little as possible.

This isn't just a rule for services. It applies to user accounts, sudo rules, file permissions, network access, and every other axis along which access can be scoped. The question at every step is the same: does this entity need this access to do its job? If the answer is no, remove it.

The hardest part of least privilege isn't the concept — it's applying it consistently over time as systems grow and change. The default in most organizations is to add access when something breaks, and almost never to audit and remove access that's no longer needed. Over months, a system drifts from least privilege to a state where every service account can read almost everything, because it was easier to grant access than to figure out the minimum required. That drift is exactly what attackers rely on.

## Mental Model

Think of a hospital. Every staff member carries a keycard. The keycard for a ward nurse opens the supply closet, the patient rooms, and the nurses' station. It does not open the pharmacy, the operating theatre, or the records room. The pharmacy keycard opens the pharmacy and nothing else. The night security keycard opens every exterior door but nothing clinical.

Nobody decided this by asking "what could each person possibly need in an emergency?" They asked "what does each person need to do their job today?" Emergencies are handled by escalating through proper channels — not by giving every nurse a master key in case they ever need one.

The master key is root. And in a hospital where every nurse, janitor, and visitor has the master key, a stolen keycard is a catastrophe. In a hospital where everyone has only what they need, a stolen keycard is an inconvenience.

## Theory

Least privilege is a security principle, not a single tool or command. You implement it through the combination of several Linux mechanisms, each controlling a different axis of access.

**User accounts and process ownership** are the first axis. A service running as root has unrestricted access to the entire system. A service running as a dedicated unprivileged user — say, `nginx`, `postgres`, or `redis` — operates within the constraints of that account's permissions. If the service is compromised, the attacker is constrained to what that account can do.

**File and directory permissions** are the second axis. Even a service running as an unprivileged user can read world-readable files. If `/etc/shadow` were world-readable, least privilege on the service account would offer no protection for password hashes. Both the account and the file permissions have to be right. These two axes compound: a weak account running against strong file permissions is better than nothing, but the strongest system requires both.

**Linux capabilities** are the third axis. Traditional Unix had a binary choice: root (UID 0), who could do anything, or non-root, who could not do privileged operations. This forced developers to run services as root whenever they needed a single privileged operation — like binding to a port below 1024 or setting network interface parameters. Linux capabilities break root's privileges into about 40 discrete units that can be granted individually to processes and binaries, without granting all of root.

**sudo rules** are the fourth axis. When a human operator needs to run a privileged command, the temptation is to give them full sudo access — `username ALL=(ALL) ALL`. Least privilege means writing rules that allow only the specific commands needed: `username myhost=(root) NOPASSWD: /usr/bin/systemctl restart nginx`. If that account is compromised, the attacker can restart nginx — not run arbitrary commands as root.

**Network access** is the fifth axis. A service that only needs to accept connections on one port should not be listening on ten. A service that only needs to talk to a database on the same host should not have outbound internet access. Firewall rules, network namespaces, and service binding addresses all apply least privilege to the network layer.

The key invariant: **access never granted cannot be exploited.** Least privilege is fundamentally about reducing the blast radius of any single failure — whether that failure is a vulnerability, a misconfiguration, a compromised credential, or a supply chain attack.

### Linux Capabilities in Detail

When the kernel evaluates whether a process can perform a privileged operation, it checks the process's capability sets. There are three relevant sets:

- **Permitted** — the maximum set of capabilities the process may hold
- **Effective** — the capabilities currently active for permission checks
- **Inheritable** — capabilities that can be inherited across an `exec()` call

For file-based capabilities (set with `setcap`), there are analogous sets on the file itself. When a binary with file capabilities is executed, the kernel computes the new process capability sets from a formula that combines the process's inherited capabilities with the file's sets.

The capabilities you'll encounter most often in production:

| Capability | What it permits |
|---|---|
| `CAP_NET_BIND_SERVICE` | Bind to ports below 1024 |
| `CAP_NET_ADMIN` | Configure network interfaces, routing, firewalls |
| `CAP_NET_RAW` | Use raw and packet sockets (needed by `ping`, `tcpdump`) |
| `CAP_SYS_ADMIN` | A wide range of system administration operations (see below) |
| `CAP_SYS_PTRACE` | Trace arbitrary processes with `ptrace` |
| `CAP_DAC_OVERRIDE` | Bypass file read/write/execute permission checks entirely |
| `CAP_DAC_READ_SEARCH` | Bypass file read and directory read/execute checks |
| `CAP_SETUID` | Switch to any UID — effectively as powerful as root for identity |
| `CAP_SETGID` | Switch to any GID |
| `CAP_SYS_MODULE` | Load and unload kernel modules |
| `CAP_SYS_CHROOT` | Use `chroot()` |
| `CAP_KILL` | Send signals to any process regardless of ownership |
| `CAP_AUDIT_WRITE` | Write to the kernel audit log |
| `CAP_PERFMON` | Access performance monitoring interfaces |
| `CAP_BPF` | Use privileged BPF operations |
| `CAP_CHECKPOINT_RESTORE` | Checkpoint/restore process state (CRIU) |

`CAP_SYS_ADMIN` deserves special attention. It is not one capability — it is a catch-all that permits mounting and unmounting filesystems, performing `ioctl` operations on many devices, setting hostname and domainname, configuring loop devices, managing namespaces, and dozens of other operations. The kernel developers themselves have acknowledged it's too broad. Granting `CAP_SYS_ADMIN` to a service is almost as dangerous as running the service as root. The only legitimate use case is when you genuinely need most of what it provides — container runtimes, for instance. For anything else, stop and ask whether a more specific capability covers the actual need.

## How It Works (Mechanics)

### The `setcap` / `getcap` pair

File capabilities are stored in a file's extended attributes, specifically in the `security.capability` xattr. The kernel reads them during `execve()`.

```bash
# Grant a binary the ability to bind to ports below 1024
# =+ep means: add to Effective and Permitted sets
sudo setcap 'cap_net_bind_service=+ep' /usr/bin/node
```

```bash
# Verify what capabilities a binary has
getcap /usr/bin/node
```

```text
/usr/bin/node cap_net_bind_service=ep
```

```bash
# Scan a directory for binaries with capabilities
# -r recurses; combine with sudo to see capabilities on root-owned binaries
getcap -r /usr/bin/ 2>/dev/null
```

```text
/usr/bin/ping cap_net_raw=ep
/usr/bin/mtr-packet cap_net_raw=ep
/usr/bin/node cap_net_bind_service=ep
```

```bash
# Remove all capabilities from a binary
sudo setcap -r /usr/bin/node
```

The kernel mechanics during `execve()`:

```text
# Conceptual flow — not a real command
1. Kernel reads the file's security.capability xattr
2. Computes new capability sets:
   - new_permitted = (inheritable & file_inheritable) | (file_permitted & bounding_set)
   - new_effective = file_effective ? new_permitted : empty
   - new_inheritable = inheritable (unchanged)
3. Process starts with those computed sets
4. When the process performs a privileged syscall,
   kernel checks whether the required capability is in new_effective
```

### Dedicated service accounts

When `systemd` starts a service unit, it can create and use a dedicated system account via the `User=` and `Group=` directives. System accounts have UID below 1000 (conventionally), no login shell, no home directory in `/home`, and are not listed in `/etc/passwd` with an interactive shell.

```bash
# Create a dedicated system account for a service
# -r: system account (UID < 1000, no expiry)
# -s /sbin/nologin: no interactive login
# -M: no home directory created
sudo useradd -r -s /sbin/nologin -M myservice
```

### systemd sandboxing

systemd service units can declare a set of sandboxing directives in the `[Service]` section. These are enforced by the kernel — they're not advisory:

```text
# Conceptual sandboxing constraints — not a real command
PrivateTmp=yes      → service gets its own /tmp, invisible to other services
ProtectSystem=strict → /usr, /boot, /etc mounted read-only for the service
ProtectHome=yes      → /home, /root, /run/user not visible at all
NoNewPrivileges=yes  → process cannot gain more privileges via setuid or capabilities
PrivateDevices=yes   → only a minimal set of device nodes accessible
CapabilityBoundingSet=CAP_NET_BIND_SERVICE → cap out at exactly what's needed
RestrictAddressFamilies=AF_INET AF_INET6 → no Unix sockets, no raw sockets
```

The `systemd-analyze security <unit>` command produces a numeric security score for a unit by checking which of these directives are present.

## Basic Example

```bash
# Step 1: See which capabilities are set on common binaries
getcap -r /usr/bin/ /usr/sbin/ 2>/dev/null
```

```text
/usr/bin/ping cap_net_raw=ep
/usr/bin/mtr-packet cap_net_raw=ep
```

```bash
# Step 2: Understand why ping needs a capability
# ping creates a raw ICMP socket — normally requires root
# CAP_NET_RAW lets the binary do this without running as root

# Confirm ping works without sudo
ping -c 1 127.0.0.1
```

```text
PING 127.0.0.1 (127.0.0.1) 56(84) bytes of data.
64 bytes from 127.0.0.1: icmp_seq=1 ttl=64 time=0.040 ms

--- 127.0.0.1 ping statistics ---
1 transmitted, 1 received, 0% packet loss, time 0ms
rtt min/avg/max/mdev = 0.040/0.040/0.040/0.000 ms
```

```bash
# Step 3: Check the process running as non-root (run this while ping runs in another terminal)

```

```text
user     4821  0.0  0.0   8524  1152 pts/1  S+  14:23   0:00 ping -c 10 127.0.0.1
```

This is least privilege in action. `ping` needs raw socket access — a privilege historically requiring root. `CAP_NET_RAW` grants exactly that one capability to the binary, so it runs as your user, not as root. If `ping` were compromised, the attacker would have raw socket access, not a full root shell.

Try this yourself: run `getcap /usr/bin/ping`. If your system uses setuid instead of capabilities (some distros still do), you'll see the setuid bit with `ls -la /usr/bin/ping`. That's the older approach — capabilities are the modern replacement.

## Real-World Example

Here is a complete setup for a Node.js API server that needs to bind to port 443 (HTTPS) on a production system, configured with least privilege throughout.

```bash
# Step 1: Create a dedicated unprivileged service account
sudo useradd -r -s /sbin/nologin -M -d /opt/myapp myapp
```

```bash
# Step 2: Set ownership and permissions on the application directory
sudo mkdir -p /opt/myapp
sudo chown -R myapp:myapp /opt/myapp
# The app can write its logs; it cannot write to anything outside /opt/myapp
sudo chmod 750 /opt/myapp
```

```bash
# Step 3: Grant only cap_net_bind_service to the Node binary
# This allows Node to bind to port 443 without running as root
# =+ep: add to Effective and Permitted sets
sudo setcap 'cap_net_bind_service=+ep' /usr/bin/node
```

```bash
# Step 4: Verify the capability was set
getcap /usr/bin/node
```

```text
/usr/bin/node cap_net_bind_service=ep
```

```bash
# Step 5: Create the systemd unit file
sudo tee /etc/systemd/system/myapp.service > /dev/null << 'EOF'
[Unit]
Description=My Node.js API
After=network.target

[Service]
Type=simple
User=myapp
Group=myapp
WorkingDirectory=/opt/myapp
ExecStart=/usr/bin/node /opt/myapp/server.js
Restart=on-failure

# Filesystem sandboxing
PrivateTmp=yes
ProtectSystem=strict
ProtectHome=yes
ReadWritePaths=/opt/myapp/logs

# Privilege restrictions
NoNewPrivileges=yes
PrivateDevices=yes

# Network restrictions — only IPv4/IPv6, no raw sockets
RestrictAddressFamilies=AF_INET AF_INET6

# Cap the maximum capabilities the process can ever hold
# Even if the binary somehow gains more, the bounding set is the ceiling
CapabilityBoundingSet=CAP_NET_BIND_SERVICE

[Install]
WantedBy=multi-user.target
EOF
```

```bash
# Step 6: Reload systemd and start the service
sudo systemctl daemon-reload
sudo systemctl start myapp
sudo systemctl status myapp
```

```text
● myapp.service - My Node.js API
     Loaded: loaded (/etc/systemd/system/myapp.service; enabled)
     Active: active (running) since Mon 2026-01-13 14:30:00 UTC; 3s ago
   Main PID: 9821 (node)
      Tasks: 11
     CGroup: /system.slice/myapp.service
             └─9821 /usr/bin/node /opt/myapp/server.js
```

```bash
# Step 7: Measure the sandboxing score
sudo systemd-analyze security myapp.service
```

```text
  NAME                                                        DESCRIPTION                                                             EXPOSURE
✓ PrivateNetwork=                                            Service has no access to the host network                                    0.5
✓ User=/DynamicUser=                                        Service runs under a static non-root user identity                            0
✓ NoNewPrivileges=                                          Service processes cannot acquire new privileges                               0
✓ PrivateTmp=                                               Service has access to only its own temporary directory                        0
✓ ProtectSystem=                                            Service has very limited write access to the OS file hierarchy                 0
✓ ProtectHome=                                              Service has no access to home directories                                     0
✓ CapabilityBoundingSet=~CAP_SYS_ADMIN                     Service has very limited capabilities                                         0
...
→ Overall exposure level for myapp.service: 3.2 MEDIUM
```

Walk through what each layer is doing. `User=myapp` means the process starts with the privileges of an unprivileged account — no sudo, no special file access. `CapabilityBoundingSet=CAP_NET_BIND_SERVICE` means even if someone found a way to escalate the process's capabilities, the ceiling is that one capability. `NoNewPrivileges=yes` means `execve()` inside the service cannot result in a more privileged process — setuid binaries won't gain their setuid bit. `ProtectSystem=strict` means the service sees `/usr`, `/boot`, and `/etc` as read-only; `ReadWritePaths=/opt/myapp/logs` carves out the one directory it actually needs to write. `PrivateTmp=yes` means its `/tmp` is isolated — no other service can read files it writes there.

This is what least privilege looks like in a real production deployment: multiple overlapping layers, each restricting a different axis. No single layer is sufficient on its own.

## ⚠️ Gotchas

### `cap_sys_admin` Is Not "Just System Admin" — It's Almost Root

The name suggests a moderate administrative capability. The reality is that `CAP_SYS_ADMIN` covers mounting and unmounting filesystems, `clone()` with new namespaces, `ioctl` on many devices, `setdomainname`, `sethostname`, configuring loop devices, loading keys into the kernel keyring, overriding disk quota limits, and more. It is the largest capability in the kernel. Security researchers sometimes call it "the new root."

```bash
# This looks like a minimal grant — it is not
sudo setcap 'cap_sys_admin=+ep' /usr/bin/myservice
```

If `myservice` is compromised with `CAP_SYS_ADMIN`, an attacker can mount a filesystem, use user namespace tricks to escape a container, or perform any number of operations that lead to full root. Granting `CAP_SYS_ADMIN` to a service because it "needs a couple of admin operations" defeats the entire purpose of capabilities. Before granting it, audit exactly which syscall is failing and look up whether a more specific capability covers it — often it does.

### `setcap` Is Wiped on Binary Replacement

File capabilities are stored as an extended attribute on the binary. When a package manager upgrades the binary — replacing the inode — the extended attribute is gone. Your service restarts after the upgrade and suddenly can't bind to its port because the capability that was there last week no longer exists.

```bash
# After a package upgrade, check whether your capability survived
getcap /usr/bin/node
```

```text
(no output)
```

```bash
# It's gone. Re-apply it.
sudo setcap 'cap_net_bind_service=+ep' /usr/bin/node
```

The fix is documentation and automation. Every capability assignment should be recorded — in a runbook, an Ansible playbook, or a post-install hook. The canonical place to handle this reliably is in the systemd unit file using `ExecStartPre` to re-apply the capability, or by wrapping the start sequence in a script that checks and restores it. Another option is to use `AmbientCapabilities=` in the systemd unit directly, which avoids file capabilities entirely and lets systemd manage the capability grant at process start.

```ini
# Alternative: manage the capability in the systemd unit instead of on the binary
# This survives binary replacement because it's applied at launch, not stored on the file
[Service]
User=myapp
AmbientCapabilities=CAP_NET_BIND_SERVICE
CapabilityBoundingSet=CAP_NET_BIND_SERVICE
```

This approach is more robust for production — the capability is declared in configuration, not stored invisibly in an xattr that gets silently wiped.

### Running as `nobody` Is Not a Security Guarantee

"Run as `nobody`" is a common reflex for making a service less privileged. The problem is that `nobody` is a shared account. Every service you configure to run as `nobody` shares the same identity. If one service is compromised, it runs as `nobody`. If another service also runs as `nobody` and has files it owns that are only restricted to its own user, the first compromised service can read or modify them.

The correct approach is a dedicated service account per service. Each service has its own UID. File ownership and permissions are set to that specific UID. The blast radius of one compromise is bounded by one service's files.

```bash
# Wrong: multiple services all running as nobody
# A compromise in service A can read files owned by nobody used by service B
User=nobody

# Right: each service has its own dedicated account
# myapp_svc can only read/write what it owns
sudo useradd -r -s /sbin/nologin -M myapp_svc
User=myapp_svc
```

Beyond shared identity, `nobody` running as a shared account can still read any world-readable file. If `/etc/app-secrets.conf` has permissions `644` (world-readable), a service running as `nobody` can read it — and so can every other service running as `nobody`, and every regular user on the system. Least privilege requires restricting both the account identity and the file permissions it operates against.

### A Service Running as Non-Root Can Still Read World-Readable Sensitive Files

This is the mistake that makes people think they've applied least privilege when they haven't. You create a dedicated `mydb` user, run your database as that user, and feel secure. But `/etc/mydb/credentials.conf` has permissions `644` — world-readable. Anyone on the system can read it. Every service running as any user can read it.

```bash
# This file is readable by every user and process on the system
ls -la /etc/mydb/credentials.conf
```

```text
-rw-r--r-- 1 root root 512 Jan 13 14:30 /etc/mydb/credentials.conf
```

```bash
# Correct: the file should be readable only by the service account that needs it
sudo chown root:mydb /etc/mydb/credentials.conf
sudo chmod 640 /etc/mydb/credentials.conf
```

```text
-rw-r----- 1 root mydb 512 Jan 13 14:30 /etc/mydb/credentials.conf
```

The group ownership `mydb` with permissions `640` means only root and members of the `mydb` group can read it. The service process, running as user `mydb` in group `mydb`, can read it. Nothing else can. This is what least privilege on files looks like — not just running the service as non-root, but ensuring the files it touches are also appropriately restricted.

### `ProtectSystem=strict` Will Break Services That Write Outside Their Declared Paths

You add `ProtectSystem=strict` to a service unit, reload, and the service fails to start. The logs show permission denied errors for paths that definitely exist. The cause is that `ProtectSystem=strict` makes the entire OS hierarchy read-only to the service. If the service writes to `/var/log/myapp/`, `/run/myapp/`, or anywhere outside `ReadWritePaths=`, it fails.

```bash
# Check journal for the specific path that's being denied
journalctl -u myapp.service -n 50 --no-pager | grep "Permission denied"
```

```text
Jan 13 14:31 myapp[9821]: Error: EACCES: permission denied, open '/var/log/myapp/app.log'
```

```ini
# The fix: declare every path the service writes to
[Service]
ProtectSystem=strict
ReadWritePaths=/var/log/myapp /run/myapp /opt/myapp/data
```

The right approach is to add `ProtectSystem=strict` first, then watch what breaks in the journal, and add each denied path to `ReadWritePaths=`. This process produces a precise map of every location the service writes to — which is valuable information in itself.

## Distro Notes

| Distro Family | Package / Path / Behavior |
|---|---|
| Debian / Ubuntu | `setcap` and `getcap` are in the `libcap2-bin` package — `apt install libcap2-bin`; `systemd-analyze security` available from systemd 240+ (Ubuntu 20.04+) |
| RHEL / Fedora / CentOS / Rocky | `setcap`/`getcap` are in `libcap`; available by default on most installs; `systemd-analyze security` available from RHEL 8+ |
| Arch | `libcap` package provides both tools; `pacman -S libcap` |
| Alpine (containers) | `apk add libcap`; note that Alpine uses `musl` libc and some capability-related behaviors differ in edge cases; container environments should prefer `AmbientCapabilities=` in unit files over file capabilities when possible |

The `AmbientCapabilities=` systemd directive requires systemd 229+ (Linux 4.3+ kernel). It is the preferred approach for container-friendly deployments because it does not depend on extended attributes persisting through binary replacement.

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| File capabilities (`setcap`) | Binary needs a privilege permanently; unmanaged by systemd | Fragile — wiped on binary replacement; requires documentation |
| `AmbientCapabilities=` in systemd | Service is managed by systemd; privilege needed at runtime | Preferred for services — survives upgrades; declared in configuration |
| `PrivilegedPort` via systemd socket activation | Service needs to accept on a privileged port | Alternative to `cap_net_bind_service` for socket-activated services |
| Reverse proxy (nginx/caddy) in front of app | App serves HTTP/HTTPS; port privilege is the only concern | Cleanest solution — the privileged port listener is a hardened proxy; app runs fully unprivileged on a high port |
| Full AppArmor/SELinux profile | Fine-grained mandatory access control beyond capabilities | Stronger than capabilities alone; steeper learning curve; both MAC systems can complement what capabilities provide |

For the specific problem of "my app needs to bind to port 443": the reverse proxy pattern is the cleanest answer. Nginx or Caddy runs as root briefly, drops privileges, handles TLS termination, and forwards to your application running as an unprivileged user on port 8443. Your application never needs `cap_net_bind_service` at all. The capability approach is the right answer when a reverse proxy isn't appropriate — for databases, custom protocols, or binary-level constraints.

## Trade-offs

**Where this shines:**

- Blast radius reduction — a compromised service with least privilege applied cannot escalate to root, cannot read files outside its scope, cannot affect other services' state
- Auditability — a service with explicit capability grants, a dedicated account, and `ProtectSystem=strict` has a precisely documented access footprint; you know exactly what it can do
- Defense in depth — capability restrictions, file permissions, and systemd sandboxing are independent layers; an attacker has to defeat all of them, not just one

**Where this costs you:**

- Initial configuration complexity — getting least privilege right for a new service requires understanding exactly what it needs: which files, which ports, which kernel interfaces; shortcuts produce incorrect security, not working security
- Maintenance overhead — capability grants on binaries are wiped by package upgrades; service accounts and their file permissions must be managed as infrastructure; new service requirements mean updating the security configuration
- Debugging friction — `ProtectSystem=strict` and `PrivateTmp=yes` make certain debugging approaches fail (paths that exist on the host aren't visible to the service; writes outside declared paths fail silently in some configurations)
- Risk of over-restriction — over-applied sandboxing can break services in ways that take time to diagnose; the path to correct configuration is iterative, not immediate

**Performance note:** Linux capabilities checks happen at the syscall layer with negligible overhead — they are constant-time bitmask operations. The cost of least privilege is configuration complexity and maintenance, not runtime performance.

## When to Use / When to Avoid

✅ **Use this when:**

- Deploying any network-facing service — the default should be a dedicated service account, `NoNewPrivileges=yes`, and `ProtectSystem=strict`; opt out of sandboxing only for what fails, not as the default
- A service needs to bind to a port below 1024 — use `AmbientCapabilities=CAP_NET_BIND_SERVICE` in the systemd unit rather than running the service as root
- Granting a user or operator elevated access — write specific sudo rules for specific commands, not `ALL=(ALL) ALL`
- Auditing an existing system for privilege creep — `getcap -r /usr/bin/ /usr/sbin/` and reviewing service unit `User=` directives finds most drift in minutes

❌ **Avoid this when:**

- Granting `CAP_SYS_ADMIN` because it's "easier" than finding the specific capability needed — this is not least privilege; investigate which specific syscall requires it and look for the narrower capability
- Running multiple services as `nobody` as a shortcut — create dedicated accounts; shared identity is not isolation
- Using `cap_net_bind_service` on a binary when a reverse proxy would let the application run fully unprivileged — the reverse proxy pattern is usually the right answer for HTTP/HTTPS services

⚠️ **Common mistakes:**

- Applying `User=someuser` in the systemd unit but not auditing which files are world-readable on the system — the account restriction alone is half the job
- Setting capabilities on a binary and assuming they'll persist through package upgrades — use `AmbientCapabilities=` in the unit file instead
- Treating "not root" as equivalent to "least privilege" — a non-root service can still do enormous damage if it has unnecessary capabilities, writes to shared locations, or has access to sensitive world-readable files

## Veteran Perspective

The pattern I've watched repeat most often: a developer reports that their service needs to run as root because it has to bind to port 80. The operations team says okay and adds `User=root` or just doesn't set `User=` at all. That's it. No one asks why — they just grant it and move on. Six months later, the service has a remote code execution vulnerability and the attacker owns the machine. `cap_net_bind_service` would have prevented the "run as root" step. A reverse proxy in front of the service would have prevented even that. The fix was a five-minute conversation about what the service actually needed — and it never happened.

The more experienced version of this mistake: someone who knows about capabilities but doesn't understand the difference between `cap_sys_admin` and the specific capability they need. They see "access denied" in the logs, don't know which capability covers it, and grant `cap_sys_admin` because it fixes the error. Now they have a service with near-root privileges that they've convinced themselves is sandboxed because it's not actually running as UID 0. In an audit or an incident review, this is the kind of thing that makes the security team's heads explode. Before you grant `cap_sys_admin`, run `man capabilities` and read every operation it covers. If the service genuinely needs all of that, fine. If it needs one specific thing, find the right capability.

The subtler gap at the intermediate level: people who apply `User=serviceaccount` and `ProtectSystem=strict` and call it done, without checking the file permissions on the service's configuration files, secrets, or data directories. A service running as a dedicated non-root account with strict system protection can still exfiltrate a world-readable `/etc/app/api-keys.conf`. Least privilege requires auditing both the account's capabilities and the permissions on everything it can reach. These are two separate questions and you need to answer both.

The most underused tool in this space is `systemd-analyze security`. Run it against every service unit you manage. The output is a scored list of sandboxing directives the service lacks, ranked by their impact on the exposure score. It doesn't tell you whether adding each directive will break your service — you have to test that — but it gives you a prioritized list of what to try. On a default install, most service units score `UNSAFE` or `EXPOSED`. Getting a service to `MEDIUM` or `LOW` in that score, while keeping it functional, is the concrete work of applying least privilege to a running service.

The war story: a PostgreSQL instance on a cloud server had been configured by someone who gave the postgres user full sudo access because "it was easier." Nobody reviewed it. Two years later, a misconfigured connection string exposed the database to the internet. The attacker connected, ran `COPY TO PROGRAM`, and had a shell as the `postgres` user. With full sudo access, that became root in one command. With proper least privilege — no sudo, `ProtectSystem=strict`, dedicated account with no write access outside `/var/lib/postgresql` — the attacker would have had access to the database contents (bad enough, but contained) and nothing else on the machine.

In technical interviews for security engineering and senior SRE roles, the question often comes up as a scenario: "your web server is compromised — what did the attacker get?" The answer reveals whether the candidate understands blast radius thinking. If the service ran as root, the answer is "the machine." If least privilege was applied correctly, the answer is specific: "the files in `/var/www`, the ability to write logs, network connections the service had established." The difference between those two answers is the entire point of this topic.

## Try It Yourself

1. Run `getcap -r /usr/bin/ /usr/sbin/ 2>/dev/null` on your system. Identify at least two binaries with capabilities and look up what each capability permits. Notice which capabilities appear most often and why those specific binaries need them.

2. Check how `ping` gets its raw socket access on your system: run `ls -la $(which ping)` to check for the setuid bit, and `getcap $(which ping)` to check for file capabilities. Only one will show something — different distros use different approaches. Understand why the result is what it is.

3. Create a dedicated system account for a practice service: `sudo useradd -r -s /sbin/nologin -M practicesvc`. Create a directory `/opt/practicesvc`, set ownership to that account, and set permissions to `750`. Confirm that your regular user cannot `ls` the directory contents.

4. If you have a service managed by systemd, run `sudo systemd-analyze security <servicename>.service`. Read the output. Identify the three highest-impact directives that are missing and look up what each one does. Do not add them yet — just understand what they would restrict.

5. **Challenge (requires a VM with a test Node.js or Python app):** Configure a service that runs on port 443 with full least privilege: dedicated service account, `AmbientCapabilities=CAP_NET_BIND_SERVICE` in the unit file, `ProtectSystem=strict`, `NoNewPrivileges=yes`, `PrivateTmp=yes`. Start with a bare unit file and add sandboxing directives one at a time, watching the journal after each restart to catch restrictions that break the service. When the service is running correctly, run `sudo systemd-analyze security` against it and report your exposure score.

## Summary

- Least privilege means every process, user, and service operates with the minimum access needed — applied across user accounts, file permissions, Linux capabilities, sudo rules, and network access simultaneously; restricting only one axis while leaving others open is not least privilege.
- Linux capabilities split root's privileges into about 40 discrete units; `setcap`/`getcap` manage file-based capabilities, but `AmbientCapabilities=` in systemd unit files is the preferred production approach because it survives binary replacement by package upgrades.
- `CAP_SYS_ADMIN` covers dozens of privileged operations and is nearly as powerful as full root — granting it to a service that needs one specific privilege defeats the purpose; audit `man capabilities` to find the specific capability that covers the actual need.
- Running a service as a non-root dedicated account is necessary but not sufficient — world-readable configuration files, secrets, and data directories undermine account-level restrictions; file permissions and account restrictions must both be right.
- `systemd` sandboxing directives (`PrivateTmp`, `ProtectSystem`, `NoNewPrivileges`, `CapabilityBoundingSet`) provide filesystem and privilege isolation beyond what user accounts offer; `systemd-analyze security <unit>` produces a scored audit of which directives are missing.
