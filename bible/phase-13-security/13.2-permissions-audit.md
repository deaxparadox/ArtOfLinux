# [13.2] — Permissions Audit

> A misconfigured SUID binary or a world-writable directory without a sticky bit is a standing invitation for privilege escalation — the audit commands in this file are how you find those invitations before an attacker does.

## The Problem This Solves

Permission drift is silent. A developer sets a script world-writable to debug a cron job, fixes the bug, and forgets to restore the permissions. A package install creates a setuid binary you didn't ask for. A long-running server accumulates unowned files from users that were deleted years ago. None of this is visible in normal use. Everything keeps working. The system logs nothing. From the outside, it looks healthy.

Then someone with local access finds the world-writable cron script. They edit it. Root runs it. Game over.

The problem isn't any single misconfiguration — it's that misconfiguration is invisible unless you actively look. A permissions audit is the act of looking. It's a systematic scan of the filesystem for the categories of permission error that are most commonly exploited: elevated-privilege binaries, files that anyone can write to, and files that no valid user owns.

On a fresh OS install, the audit is a baseline. On a server that's been running for a year, it's a confession. The gap between those two states is where attackers live.

---

## Mental Model

Think of a large office building. Most doors require a keycard — you can only enter rooms you're authorized for. But over the years, some doors were propped open for maintenance, a few master keys were handed out and never collected, and some rooms have no record of who they belong to anymore.

A permissions audit is walking every corridor with a checklist: which doors are propped open (world-writable), which keys give access beyond their stated purpose (SUID/SGID), and which rooms have no owner listed on the directory (unowned files). The building works fine for everyone using it legitimately. The audit is about finding what someone with bad intent could exploit.

---

## Theory

Linux permissions are a three-layer system: owner, group, and others. Every file has a mode — a bitmask that encodes what each layer can do (read, write, execute). Beyond the standard nine bits, three special bits — setuid, setgid, and sticky — modify how the kernel handles execution and directory access.

Under normal operations, you never see the full picture. `ls -l` on a specific path shows you one file's permissions. To audit the entire system, you need to sweep the filesystem with `find`, which can filter by permission bitmask, ownership, and file type in a single pass.

Four categories of permission anomaly are worth auditing regularly:

**Setuid binaries** run with the file owner's identity, not the caller's. If the owner is root, the binary executes as root regardless of who invokes it. This is necessary for programs like `passwd`, `ping`, and `su`. It becomes a vulnerability when an unexpected binary acquires the setuid bit — either because a package misconfigured it, because a developer temporarily set it and forgot to remove it, or because an attacker planted a setuid shell.

**Setgid binaries** run with the file's group identity. The same logic applies — some are legitimate (like `crontab` and `ssh-agent` on certain distros), and unexpected ones are a red flag. Setgid on directories has a different effect: files created inside inherit the directory's group rather than the creator's primary group.

**World-writable files and directories** are accessible by every user on the system. For files, this means any user can overwrite the content. For directories, any user can create or delete files. World-writable without a sticky bit on a directory is particularly dangerous — any user can delete any other user's files. The sticky bit (mode bit 01000 on a directory) restricts deletion: only the file's owner, the directory's owner, or root can remove files. `/tmp` is world-writable and has the sticky bit — that combination is intentional and safe. A world-writable directory *without* the sticky bit is almost always wrong.

**Unowned files** have a numeric UID or GID that doesn't correspond to any current user or group in `/etc/passwd` and `/etc/group`. This typically happens when a user account is deleted without cleaning up that user's files first. These files represent a subtle risk: if a new user is later created and happens to receive the same UID, they silently inherit ownership of those orphaned files.

The filesystem attribute layer sits below the permission system. `chattr` sets extended attributes on files that are enforced by the filesystem driver, not by the kernel's permission check. The immutable flag (`+i`) prevents a file from being modified, renamed, or deleted — by anyone, including root — until it is explicitly cleared. This is a useful hardening technique for configuration files that should never change at runtime.

## How It Works (Mechanics)

When `find` encounters a file, it evaluates each expression against the file's metadata read from the inode. The `-perm` flag takes a permission mask in octal. The meaning of the prefix matters:

- `-perm 4000` — mode is exactly 4000 (rarely what you want)
- `-perm -4000` — all bits in 4000 are set (mode includes setuid; other bits may be anything)
- `-perm /4000` — at least one bit in 4000 is set (same effect as `-4000` for a single bit)

For auditing, `-perm -4000` (setuid set) and `-perm -2000` (setgid set) are the right forms. `-perm -0002` means "the write bit for others is set" — world-writable.

The `-nouser` and `-nogroup` tests work differently. They don't compare against a permission bitmask. They look up the file's numeric UID in `/etc/passwd` and GID in `/etc/group`. If no entry exists for that number, the test matches. These are the files orphaned by deleted accounts.

`find /` scans the entire filesystem tree, which includes `/proc` and `/sys` — pseudo-filesystems that expose kernel data structures as virtual files. Scanning them is not just slow; it generates a flood of spurious permission errors and can trigger unexpected kernel behavior. The practical approach is to redirect stderr with `2>/dev/null` to suppress permission-denied messages and, for thorough audits, explicitly exclude pseudo-filesystems.

The `chattr +i` command reaches below the VFS permission layer. Even when root calls `unlink()` or `write()` on an immutable file, the ext2/ext4/xfs filesystem driver checks the `FS_IMMUTABLE_FL` flag on the inode and returns `EPERM`. Root's effective UID is 0, but the filesystem attribute check is not gated on UID — it requires the `CAP_LINUX_IMMUTABLE` capability to set or clear the flag, and that same capability is needed to honor the removal.

```bash
# strace confirms the kernel returns EPERM even for root
strace rm /etc/resolv.conf 2>&1 | grep -E 'unlink|EPERM'
```

```text
unlinkat(AT_FDCWD, "/etc/resolv.conf", 0) = -1 EPERM (Operation not permitted)
```

`lsattr` reads the same inode flags that the filesystem driver enforces. The output uses a fixed-width column of flag characters — a `-` for each unset flag, the letter for each set flag.

## Basic Example

```bash
# Find all setuid binaries on the system
# -perm -4000: setuid bit is set (run as file owner)
# -type f: only regular files, not directories
# 2>/dev/null: suppress "Permission denied" from directories we can't read
find / -perm -4000 -type f 2>/dev/null
```

```text
/usr/bin/chsh
/usr/bin/mount
/usr/bin/newgrp
/usr/bin/passwd
/usr/bin/su
/usr/bin/sudo
/usr/bin/umount
/usr/bin/gpasswd
/usr/bin/chfn
/usr/bin/pkexec
```

The key thing to notice is that every binary in the output has the setuid bit set, meaning it runs as its owner when executed. All of these are owned by root. Each one is a deliberate design decision — `passwd` needs to write `/etc/shadow`, `mount` needs to manipulate the kernel's mount table, `sudo` needs to check policy and change effective UID. The audit question isn't "why are these here?" — it's "is everything in this list something you recognize and can justify?"

Now check one to confirm the setuid bit in its listing:

```bash
ls -l /usr/bin/passwd
```

```text
-rwsr-xr-x 1 root root 59976 Mar 22 08:14 /usr/bin/passwd
```

The `s` in the owner execute position is the setuid bit. If the execute bit were not set, it would appear as `S` (capital) — indicating the setuid bit is set on a non-executable file, which is meaningless and almost always a mistake.

Try this yourself on your system. The list will vary by distro and installed packages. If you see anything in `/home`, `/tmp`, `/opt`, or a custom application path, stop and investigate it.

## Real-World Example

This is the full audit sequence you'd run on a new server before handing it to a team, or on a production system during a security review. Each command targets a different class of permission problem.

```bash
#!/bin/bash
# Full permissions audit — run as root for complete coverage
# Exclude /proc and /sys to avoid kernel pseudo-filesystem noise

echo "=== SETUID BINARIES ==="
find / \
  -not \( -path /proc -prune \) \
  -not \( -path /sys -prune \) \
  -perm -4000 -type f \
  -ls \
  2>/dev/null

echo ""
echo "=== SETGID BINARIES ==="
find / \
  -not \( -path /proc -prune \) \
  -not \( -path /sys -prune \) \
  -perm -2000 -type f \
  -ls \
  2>/dev/null

echo ""
echo "=== WORLD-WRITABLE FILES (non-symlink) ==="
find / \
  -not \( -path /proc -prune \) \
  -not \( -path /sys -prune \) \
  -perm -0002 -type f \
  -ls \
  2>/dev/null

echo ""
echo "=== WORLD-WRITABLE DIRECTORIES WITHOUT STICKY BIT ==="
find / \
  -not \( -path /proc -prune \) \
  -not \( -path /sys -prune \) \
  -perm -0002 -type d \
  ! -perm -1000 \
  -ls \
  2>/dev/null

echo ""
echo "=== UNOWNED FILES AND DIRECTORIES ==="
find / \
  -not \( -path /proc -prune \) \
  -not \( -path /sys -prune \) \
  \( -nouser -o -nogroup \) \
  -ls \
  2>/dev/null
```

The `-ls` flag at the end of each `find` invocation prints full metadata — permissions, link count, owner, group, size, date, and path — in a single line per file. This is more useful than the default path-only output because you can see at a glance who owns each file and what its exact mode is.

The `! -perm -1000` clause in the world-writable directory scan deserves attention. The octal `1000` is the sticky bit. The `!` negates the match — so this line says "and does NOT have the sticky bit set." Without this clause, you'd flag `/tmp` and `/var/tmp` as problems when they're intentionally world-writable by design, and safely so because of the sticky bit.

For the SUID/SGID results, your job is to compare against a known-good baseline. On a Debian/Ubuntu system, `dpkg --verify` will flag files whose permissions deviate from what the package manager installed. On RHEL/Fedora/Rocky:

```bash
# Verify installed package file attributes against RPM database
rpm -Va --noscripts 2>/dev/null | grep '^.M'
```

```text
.M.......  g /var/log/cups
SM5....T.  c /etc/ssh/sshd_config
```

The `.M` prefix means a mode mismatch — the file's current permissions differ from what the RPM database says they should be. Any SUID binary flagged here that isn't in the RPM database warrants immediate investigation.

For locking down a critical configuration file that must not change at runtime:

```bash
# Set the immutable flag on resolv.conf to prevent NetworkManager from overwriting it
sudo chattr +i /etc/resolv.conf

# Verify the flag is set — look for 'i' in the attribute column
lsattr /etc/resolv.conf
```

```text
----i---------e------- /etc/resolv.conf
```

The `i` character in the attribute string confirms the immutable flag. To remove it:

```bash
sudo chattr -i /etc/resolv.conf
```

To audit multiple files at once:

```bash
# Check all files in /etc/ssh
lsattr /etc/ssh/
```

```text
--------------e------- /etc/ssh/moduli
--------------e------- /etc/ssh/ssh_config
--------------e------- /etc/ssh/sshd_config
--------------e------- /etc/ssh/ssh_host_ecdsa_key
--------------e------- /etc/ssh/ssh_host_ed25519_key
```

The `e` flag (extents) is set automatically by the filesystem and is harmless — it means the file uses extents for block allocation on ext4. The `i` flag would appear here alongside it if you'd set it.

## ⚠️ Gotchas

### Running `find /` Without `2>/dev/null` Floods the Output

The most common mistake when running a permissions audit is omitting the stderr redirect. `/proc`, `/sys`, and many root-owned directories under `/run` will generate "Permission denied" messages even when running as root, because some pseudo-filesystem entries explicitly deny access regardless of privilege.

```bash
# What most people try first
find / -perm -4000 -type f
```

```text
find: '/proc/tty/driver': Permission denied
find: '/proc/1/fd': Permission denied
find: '/proc/1/fdinfo': Permission denied
find: '/proc/1/ns': Permission denied
... (thousands of lines)
/usr/bin/passwd
find: '/proc/2/fd': Permission denied
... (thousands more lines)
```

The actual results are buried inside hundreds or thousands of error lines. You'll either miss the output entirely or spend minutes scrolling.

```bash
# Redirect stderr to /dev/null — suppress the noise, keep the results
find / -perm -4000 -type f 2>/dev/null
```

```text
/usr/bin/chsh
/usr/bin/mount
/usr/bin/passwd
/usr/bin/su
/usr/bin/sudo
```

Clean, readable, actionable. The stderr redirect is not optional on a real system — treat it as mandatory syntax for any `find /` invocation.

### A Setuid Binary Is Only Dangerous If You Can't Justify It

The audit output will include `/usr/bin/passwd`, `/usr/bin/sudo`, `/usr/bin/ping`, and a dozen other binaries on a fresh install. Beginners sometimes panic at the length of the list. The length isn't the problem — the unknown entries are.

Every SUID binary is a deliberate trust decision by the package maintainer. The audit's job is to verify that the list matches what you deliberately installed and that nothing unexpected has appeared. Cross-reference against GTFOBins: that site catalogs exactly how common binaries can be abused when they have the SUID bit and an attacker has local access. If `find`, `vim`, `python3`, `nmap`, or `base64` appears in your SUID scan results, an attacker with a local shell can use them to read any file on the system or spawn a root shell.

```bash
# find with SUID set — trivial root shell
find . -exec /bin/sh \; -quit
```

The above one-liner would give any local user a root shell if `find` had the SUID bit set. This is why the audit list matters: the same binary that's safe to have installed is dangerous if it accidentally acquires the SUID bit.

### `chattr +i /etc/hosts` Will Break NetworkManager and cloud-init

The immutable flag is powerful precisely because it's unconditional. If you set it on a file that a system process legitimately needs to modify, that process will fail — silently or with a cryptic permission error — even though root is running it.

```bash
sudo chattr +i /etc/hosts
```

On a cloud instance or a desktop using NetworkManager, this immediately breaks:
- NetworkManager's ability to write DHCP-assigned hostnames
- cloud-init's hostname injection at boot (on AWS, GCP, Azure VMs)
- Some VPN clients that inject hostnames into `/etc/hosts` for DNS split-tunneling
- Docker's automatic container hostname insertion

The failure mode isn't obvious. NetworkManager doesn't log "immutable flag set." It logs "failed to write /etc/hosts" — which looks like a generic write error, not a chattr issue. On a cloud VM, cloud-init failing silently can leave the instance with a stale or wrong hostname, which breaks distributed service discovery.

Reserve `chattr +i` for files that genuinely must never be modified at runtime:
- `/etc/resolv.conf` on servers where DNS is static and manually managed
- `/etc/motd` where you want a fixed message of the day
- Critical audit logs where you want tamper evidence

Before setting the flag, confirm that nothing in your init system, NetworkManager, or cloud tooling writes to that file:

```bash
# Check what processes have the file open or recently touched it
sudo lsof /etc/resolv.conf
sudo inotifywait -m /etc/resolv.conf  # watch for writes before locking
```

### World-Writable `/tmp` with Sticky Bit Is Intentional — Don't Remove the Write Bit

When you first run the world-writable audit, `/tmp` and `/var/tmp` will appear in results that don't filter for the sticky bit. Beginners sometimes remove the write permission from `/tmp` to "fix" this.

```bash
# Do NOT do this
sudo chmod o-w /tmp
```

This breaks virtually every program that creates temporary files — which is almost every program. Package managers, compilers, browsers, editors, and countless daemons use `/tmp`. The right behavior is `/tmp` being world-writable with the sticky bit set (`drwxrwxrwt`) — the `t` at the end is the sticky bit. That combination means anyone can create files but only the file's owner can delete them.

The flag to watch for in your audit is the *absence* of the sticky bit on a world-writable directory, not the presence of world-write permission:

```bash
# The safe state — world-writable, sticky bit set
ls -ld /tmp
```

```text
drwxrwxrwt 20 root root 4096 Jun 14 18:45 /tmp
```

The `t` at the end is what makes this safe. If you see `drwxrwxrwx` instead — world-writable with no sticky bit — that's the actual problem.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | `dpkg --verify` checks file attributes against the package database; `debsums` provides checksum verification. `chattr` and `lsattr` are in the `e2fsprogs` package, installed by default on ext4 systems. |
| RHEL / Fedora / CentOS / Rocky | `rpm -Va` verifies all installed package files. The `.M` prefix flags mode mismatches. CIS Benchmark scans for RHEL include world-writable directory checks as scored findings. |
| Arch | AUR packages are not tracked by `pacman -Qkk` (the package verification command) — only official repo packages. Manual review of SUID binaries in AUR-installed packages is required. |
| Alpine (containers) | `apk audit` checks file permissions against the package database. In container images, world-writable files are a significant risk because container breakout attacks often exploit filesystem permissions. |

`chattr` and `lsattr` only work on filesystems that support extended attributes: ext2, ext3, ext4, XFS, Btrfs. They have no effect on tmpfs (where `/tmp` often lives in RAM), FAT/exFAT, NFS mounts, or most FUSE filesystems. On tmpfs, the `chattr` call will return `EOPNOTSUPP`.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `find` (manual) | Ad-hoc audits, scripted checks, cron jobs | Built into every Linux system; no dependencies; precise control over what you scan |
| `lynis` | Comprehensive automated security audit | Open-source hardening tool; runs hundreds of checks including permissions; generates scored report with remediation advice |
| `aide` | File integrity monitoring over time | Creates a cryptographic database of file attributes and detects drift; better for ongoing monitoring than one-shot audits |
| `tripwire` | Enterprise file integrity monitoring | Commercial and open-source editions; more configuration overhead than AIDE but more features |
| `rpm -Va` / `dpkg --verify` | Verifying package-installed files match database | Only covers files installed by the package manager — custom scripts, manual changes, and third-party software are invisible to it |

For a one-shot security review, the `find` commands in this file are sufficient and have no dependencies. For ongoing production monitoring, pair them with `aide` or `lynis` running as a cron job — point-in-time snapshots miss drift that happens between audits.

## Trade-offs

**Where this shines:**

- `find`-based audits are fast, dependency-free, and scriptable — the commands in this file run on any Linux system without installing anything
- `chattr +i` provides genuine tamper protection that survives root-level mistakes and automated tooling that doesn't know to check filesystem attributes
- `lsattr` exposes a layer of file protection that `ls -l` is completely blind to — in incident response, missing this layer has cost real sysadmins real time

**Where this costs you:**

- `find /` on a large filesystem (multi-terabyte, millions of files) is slow — a full audit of a production NFS server can take hours; scope with `-path` exclusions or run off-hours
- `chattr +i` is a double-edged tool: the same property that prevents attackers from modifying a file prevents legitimate tooling from doing so too; the cost is operational friction and subtle breakage
- Manual `find` audits are point-in-time — they find the state right now, not the state five minutes ago; a determined attacker who knows the audit schedule can plant and remove a SUID binary between runs
- Unowned file audits produce false positives in container environments where UID mapping between the host and container namespaces creates apparent orphans

**Performance note:** On an NVMe system with a few hundred thousand files, `find / 2>/dev/null` completes in 5–15 seconds. On a spinning-disk server with millions of files, the same command can take 30–90 minutes. For those environments, schedule audits during low-I/O windows or run them against a filesystem snapshot rather than the live volume.

## When to Use / When to Avoid

✅ **Use this when:**

- Hardening a new server before it goes into production — establish a baseline, then use it for comparison
- After a suspected intrusion — SUID binaries and world-writable files are a common persistence mechanism
- Before and after a major package upgrade — package updates occasionally alter SUID bits
- As part of a CIS Benchmark compliance pass — most CIS profiles include SUID and world-writable directory checks as scored findings
- When `/etc/resolv.conf` keeps getting overwritten by NetworkManager or DHCP on a server that should have static DNS

❌ **Avoid this when:**

- You want ongoing continuous monitoring — use `aide` or a dedicated file integrity monitor instead of running `find` on a tight cron schedule; it creates unnecessary I/O load
- You want to lock down `/tmp` itself — don't remove the world-write bit from `/tmp`; that breaks too much; focus on world-writable directories that should not be writable
- Setting `chattr +i` on files managed by cloud-init, NetworkManager, or any init-time tooling without first confirming those tools don't need write access

⚠️ **Common mistakes:**

- Running the SUID audit as a non-root user — you'll get incomplete results because `find` can't read into directories you don't have access to
- Comparing SUID lists across distros — `/usr/bin/ping` requires SUID on older kernels but uses capabilities on modern ones; `/usr/bin/newgrp` may be present on Debian but absent on Alpine; a list that's clean on Ubuntu isn't a baseline for Rocky Linux
- Using `find / -perm 4000` (exact match) instead of `find / -perm -4000` (bit set) — the exact match requires the mode to be *only* 4000, which almost no real binary is; you'll find almost nothing

## Veteran Perspective

The surface-level mistake is treating this audit as a one-time task. A junior comes in, runs the `find` commands, confirms the SUID list looks reasonable, and marks the ticket done. Three months later a developer installs a debugging tool with SUID set for convenience and forgets to remove it. The audit never ran again. You find out during a penetration test.

The intermediate gap is not knowing GTFOBins. Every sysadmin knows that a setuid shell is dangerous. Far fewer know that `env`, `awk`, `python3`, `perl`, `vim`, `less`, `more`, `man`, `tee`, `cp`, `mv`, `find`, `nmap`, and `base64` are all on GTFOBins — meaning if any of them have the SUID bit, an attacker with local access can escalate to root in seconds. When you audit your SUID list, you're not just looking for custom shells. You're looking for anything that appears on that list.

The deeper principle behind this is that Linux's permission model assumes the SUID bit is used sparingly and deliberately. The designers never intended dozens of SUID binaries to accumulate on a system. Every one of them is a surface area reduction failure. Modern Linux mitigates this with capabilities: instead of granting a binary full root identity via SUID, you can grant it just the specific capability it needs. `/usr/bin/ping` on modern kernels uses `CAP_NET_RAW` instead of SUID. The trend is toward capabilities, not SUID — when you audit, that's the direction your hardening should push toward too.

The `chattr +i` story has a specific war story behind it. A sysadmin hardened a production server by setting `chattr +i` on `/etc/resolv.conf` to prevent a rogue update process from clobbering the DNS config. It worked perfectly — until the cloud provider rotated the DNS server addresses, the DHCP-assigned resolver changed, and the system couldn't resolve anything. The immutable flag held. NetworkManager couldn't update the file. DNS broke silently. The symptom was "the application can't connect to the database." It took 45 minutes to find the real cause. The lesson isn't "don't use `chattr +i`" — it's "know exactly what writes to every file you lock, and test your assumptions."

In a technical interview, if someone asks you how you'd find privilege escalation vectors on a Linux system, the answer is not just "run `find` for SUID binaries." The complete answer includes: SUID/SGID scan cross-referenced with GTFOBins, world-writable directories without sticky bits, unowned files, writable cron directories, writable files in root's PATH, and files with `chattr` attributes that might be hiding something. Interviewers who ask this question are checking whether you know the full surface area or just the surface.

## Try It Yourself

1. Run `find / -perm -4000 -type f 2>/dev/null` on your system and count the results. Compare the list to the binaries described in [3.4 — Special Permissions](../phase-3-permissions/3.4-special-permissions.md). Can you explain why each one needs the SUID bit?

2. Run `lsattr /etc/passwd /etc/shadow /etc/hosts /etc/resolv.conf`. Note the current attributes on each. Is any of them already immutable? Now set and then immediately clear the immutable flag on `/etc/resolv.conf` using `sudo chattr +i` followed by `sudo chattr -i`. Confirm the state with `lsattr` at each step.

3. Run the world-writable directory scan: `find / -perm -0002 -type d ! -perm -1000 2>/dev/null`. If your system is clean, you'll get no output. If anything appears, look at each result and ask whether the write permission is intentional.

4. Create a test file, set it immutable, then try to delete it as root: `touch /tmp/test-immutable && sudo chattr +i /tmp/test-immutable && sudo rm /tmp/test-immutable`. Watch the error. Then clear the flag and remove it cleanly.

5. **Challenge** (requires a VM): Deliberately create a permission misconfiguration — copy `/bin/bash` to `/tmp/testshell`, set the SUID bit on it with `chmod u+s /tmp/testshell`, then write a script that automates the full audit described in the Real-World Example section. Have it run as a cron job every 5 minutes and write results to a log file. Confirm it detects the planted SUID binary. Then use the RPM or dpkg verification command for your distro to check what *else* might be misconfigured on the system.

## Summary

- `find / -perm -4000 -type f 2>/dev/null` finds all setuid binaries; the `2>/dev/null` redirect is mandatory or error output buries the results
- A SUID binary is dangerous when unexpected — `/usr/bin/passwd` being setuid is correct; anything on GTFOBins (like `find`, `python3`, `vim`, `env`) having the SUID bit is an immediate escalation risk
- World-writable `/tmp` with a sticky bit (`drwxrwxrwt`) is intentional and safe; the problem is world-writable *without* the sticky bit — use `! -perm -1000` in the audit query to filter correctly
- `chattr +i` enforces immutability below the VFS permission layer — even root gets `EPERM`; remove with `chattr -i`; never set it on files written by NetworkManager, cloud-init, or DHCP tooling without testing first
- Unowned files (`-nouser -o -nogroup`) are a UID reuse risk — if a new user receives the same UID as a deleted user, they silently inherit those files
