# [4.3] — sed — The Stream Editor

> `sed` is the tool you reach for when you need to transform text programmatically — substituting, deleting, inserting, or reordering lines — and you want it to run in a pipeline, in a script, or across a thousand files without opening a single one by hand.

## The Problem This Solves

You have a config file with 400 lines. One variable name has changed everywhere. Open it in a text editor and do a find-and-replace? Fine — once. But now the same change needs to happen across 60 config files on a server, inside a deployment script, triggered by a CI pipeline, at 2am without a terminal. Or you have a 10-million-line log file and you need to strip the timestamp prefix from every line before piping it to another tool. Or you need to delete every blank line from a file, or insert a comment above every line matching a pattern, or swap two fields.

These are the tasks that text editors are wrong for. Editors are interactive; they wait for a human. They don't compose into pipelines. They don't operate on stdin. They can't be invoked programmatically without significant ceremony.

The Unix answer is `sed`. It reads text line by line, applies a script of editing commands to each line, and writes the result to stdout. No interaction. No waiting. No state across invocations except what you explicitly carry in the hold space. It's built to be automated, piped, and trusted.

---

## Mental Model

Think of `sed` as a factory assembly line for text. Each line of your input is a part placed on the conveyor belt. As it moves down the line, a series of workers — your sed commands — each get a chance to inspect it and do something: stamp it, cut a piece off, swap it with a part from a holding shelf, or pull it off the line entirely. At the end, whatever's left on the belt gets shipped out (printed to stdout). The holding shelf is the hold space — one slot where you can stash something while you work with something else.

The factory doesn't care about the whole order — it only sees one part at a time. That's the key to how sed thinks: one line, one pass, no look-ahead, no look-back, unless you explicitly use the hold space to carry state forward.

## Theory

### The Two-Buffer Model

`sed` maintains two text buffers as it processes input:

- **Pattern space:** The working area. Each line is read from input into the pattern space. Commands operate on the pattern space. At the end of each cycle (barring explicit control), the pattern space is printed to stdout and cleared. This is the conveyor belt.
- **Hold space:** Persistent storage across cycles. It starts empty and holds nothing unless you explicitly put something there. It survives from one line to the next. This is the holding shelf.

Every `sed` command operates on one or both of these buffers. Most commands only touch the pattern space.

### The Processing Cycle

For each line of input, `sed` runs through what's called the main cycle:

1. Read the next line into the pattern space (stripping the trailing newline).
2. Apply every command in the script, in order, to the current pattern space.
3. If `-n` (silent mode) is not active, print the pattern space to stdout.
4. Clear the pattern space and begin the next cycle.

The cycle is the fundamental invariant. If you understand it, sed's behavior becomes predictable — even the parts that initially seem strange.

### Addresses

Not every command needs to operate on every line. `sed` lets you restrict a command to specific lines using an **address** before the command:

- **Line number:** `5` — applies only to line 5.
- **Last line:** `$` — applies only to the last line of input.
- **Regex:** `/pattern/` — applies to every line where `pattern` matches.
- **Range:** `addr1,addr2` — applies to all lines from addr1 through addr2 (inclusive).
- **Negation:** `addr!` — applies to every line that does NOT match the address.

Without an address, a command applies to every line. Addresses don't have to be line numbers — two regex addresses can form a range: `/start/,/end/` applies to the block of lines between the first match of `start` and the next match of `end`.

### The Script

A sed script is a series of commands. You can provide it as an argument to `-e` (one expression), as a file with `-f`, or as a single quoted string. Multiple `-e` flags chain commands in order. Commands are separated by semicolons or newlines.

The most common commands are:

| Command | What it does |
|---|---|
| `s/pattern/replacement/flags` | Substitute — replace pattern with replacement |
| `d` | Delete — remove the pattern space and start the next cycle |
| `p` | Print — explicitly print the pattern space |
| `q` | Quit — stop after the current line |
| `i\text` | Insert — print `text` before the current line |
| `a\text` | Append — print `text` after the current line |
| `c\text` | Change — replace the entire matched line with `text` |
| `y/chars1/chars2/` | Transliterate — replace each char in chars1 with the corresponding char in chars2 |
| `h` / `H` | Copy/append pattern space into hold space |
| `g` / `G` | Copy/append hold space into pattern space |
| `x` | Exchange pattern and hold spaces |
| `n` | Read the next line into pattern space (printing current if not `-n`) |
| `N` | Append the next line to pattern space (with a literal `\n` between) |
| `:label` | Define a label for branching |
| `b label` | Branch unconditionally to label (or end of script if no label) |
| `t label` | Branch to label if a substitution has been made since the last input line or `t` test |

### Substitution Flags

The `s` command is the workhorse of sed. Its flags change behavior:

- **`g`** — global; replace all occurrences, not just the first
- **`i`** — case-insensitive match (GNU sed extension; not POSIX)
- **`2`, `3`, `N`** — replace only the Nth occurrence
- **`p`** — print the pattern space if a substitution was made (useful with `-n`)
- **`w file`** — write the pattern space to a file if a substitution was made

Flags combine: `s/foo/bar/2g` replaces the 2nd and all subsequent occurrences.

### The Substitution Delimiter

The delimiter between the three parts of `s/pattern/replacement/` does not have to be `/`. Any character that doesn't appear in the pattern or replacement works. This matters when you're substituting paths — `sed 's|/usr/local|/opt|g'` is cleaner than `sed 's/\/usr\/local/\/opt/g'`. The character immediately after `s` becomes the delimiter.

### The `&` Replacement Metacharacter

In the replacement part of a substitution, `&` refers to the entire matched string. This lets you wrap a match without capturing groups:

```bash
# Add square brackets around any word starting with capital letter
sed 's/[A-Z][a-z]*/[&]/g'
```

`&` is not a regex — it's a replacement-side feature. It gets substituted with whatever the pattern matched.

## How It Works (Mechanics)

### One Pass, Left to Right

When `sed` reads a line, it passes the pattern space through the script from top to bottom. Commands are applied in sequence. The pattern space can be modified by one command before the next command sees it. This matters for substitutions: if you have two `s` commands in a row, the second operates on the output of the first.

```bash
# This replaces 'cat' with 'dog', then 'dog' with 'hog'
echo "the cat sat" | sed -e 's/cat/dog/' -e 's/dog/hog/'
```

```text
the hog sat
```

The first substitution changed `cat` to `dog`. The second then matched `dog` — which now exists — and changed it to `hog`. This chaining behavior is intentional and powerful, but it means you must think about the order of operations.

### In-Place Editing

`sed` normally reads from stdin or a file and writes to stdout. With `-i` (in-place), it opens a temporary file, writes output there, and then replaces the original — atomically on most Linux filesystems. The original file's inode is unchanged; only its content is replaced.

The temporary file approach means that if `sed` is interrupted mid-run, you might end up with a partially written file. This is why the backup suffix exists: `sed -i.bak` creates `file.bak` before replacing `file`. If the run was wrong, you have an undo.

### The Hold Space — State Across Lines

The hold space starts empty. It stays empty until you put something in it. The commands `h` (copy pattern to hold), `H` (append pattern to hold), `g` (copy hold to pattern), `G` (append hold to pattern), and `x` (swap) are the only ways to move data in and out.

This is how you do multi-line processing in sed. A classic example: joining two consecutive lines into one. The `N` command appends the next line to the pattern space with a `\n` separator, letting you work on both lines at once with a substitution.

```bash
# Conceptual trace — what happens inside sed for each cycle
# Line 1 read → pattern space: "foo"
# N command → read line 2, pattern space: "foo\nbar"
# s/\n/ / → pattern space: "foo bar"
# Print: "foo bar"
```

### Labels and Branching — Loops

`sed` can loop. The `:label` / `b` / `t` trio implements control flow. The `t` command branches only if a substitution was made since the last input line was read or the last `t` was executed. This lets you apply a substitution repeatedly until it no longer matches — a while loop over the pattern space.

## Basic Example

```bash
# Create a sample file to work with
printf 'The quick brown fox\nThe lazy dog\nThe quick cat\n' > /tmp/animals.txt

# Basic substitution: replace 'quick' with 'slow'
sed 's/quick/slow/' /tmp/animals.txt
```

```text
The slow brown fox
The lazy dog
The slow cat
```

```bash
# Global substitution: replace ALL occurrences on each line (not just the first)
echo "aabbaa" | sed 's/a/X/g'
```

```text
XXbbXX
```

```bash
# Address: only apply to line 2
sed '2s/lazy/energetic/' /tmp/animals.txt
```

```text
The quick brown fox
The energetic dog
The quick cat
```

```bash
# Delete lines matching a pattern
sed '/fox/d' /tmp/animals.txt
```

```text
The lazy dog
The quick cat
```

```bash
# Print only matching lines (-n suppresses default output; p prints explicitly)
sed -n '/quick/p' /tmp/animals.txt
```

```text
The quick brown fox
The quick cat
```

Run these yourself against your own file. Notice that without `-n`, `p` prints matching lines twice — once from the explicit `p` and once from the normal end-of-cycle print. That double-print behavior is one of the most common sources of confusion for new sed users. `-n` + `p` is a pattern that appears constantly in real scripts.

## Real-World Example

Here are the kinds of sed one-liners and short scripts that appear regularly in production systems.

### Editing a config file in place during deployment

```bash
# In a deployment script: update the database host in an application config
# Always test without -i first — remove it to see the output before committing
sed -i.bak \
    -e 's/^DB_HOST=.*/DB_HOST=prod-db-01.internal/' \
    -e 's/^DB_PORT=.*/DB_PORT=5432/' \
    /etc/myapp/database.conf
```

The `^` anchor ensures we only match actual config lines, not commented ones or values containing `DB_HOST` mid-line. The `.*` at the end of the pattern matches the entire existing value — whatever it was — so the replacement doesn't need to know the old value.

The `-i.bak` creates `database.conf.bak` before modifying. If the regex was wrong, `database.conf.bak` is your undo.

### Stripping comments and blank lines from a config

```bash
# Remove lines that are blank or start with # (comments)
sed -e '/^[[:space:]]*#/d' -e '/^[[:space:]]*$/d' /etc/nginx/nginx.conf
```

Two `-e` expressions chained. The first deletes comment lines (zero or more whitespace chars, then `#`). The second deletes blank lines (zero or more whitespace chars, nothing else). The result is a compact view of the active config — useful for auditing.

### Extracting a block between two markers

```bash
# Print only lines between BEGIN and END markers (inclusive)
sed -n '/^BEGIN$/,/^END$/p' /var/log/app.log
```

The range address `/^BEGIN$/,/^END$/` activates when `BEGIN` appears and deactivates after `END` appears. Combined with `-n` and `p`, this extracts only the matching block. This is the sed equivalent of a grep that understands sections.

### Using `&` to wrap matched text

```bash
# Quote every IP address in a log file
sed 's/[0-9]\{1,3\}\.[0-9]\{1,3\}\.[0-9]\{1,3\}\.[0-9]\{1,3\}/"&"/g' access.log
```

The `&` inserts the full matched IP address inside the quotes without needing a capture group. In BRE (basic regex — the default), `\{1,3\}` is the repetition quantifier. If you use `sed -E` (extended regex), it becomes `[0-9]{1,3}`.

### Avoiding the slash trap when editing paths

```bash
# Change /usr/local/bin to /opt/bin in a shell script
# Wrong: escaping is ugly and error-prone
sed 's/\/usr\/local\/bin/\/opt\/bin/g' deploy.sh

# Right: use a different delimiter
sed 's|/usr/local/bin|/opt/bin|g' deploy.sh
```

Any character that doesn't appear in the pattern or replacement works as a delimiter. `|`, `#`, `,`, `@` are all common choices when paths are involved.

## ⚠️ Gotchas

### `sed -i` on macOS Requires a Suffix Argument — Always

On Linux with GNU sed, in-place editing works like this:

```bash
sed -i 's/foo/bar/' file.txt
```

Take that exact command to a Mac running BSD sed:

```bash
sed -i 's/foo/bar/' file.txt
```

```text
sed: 1: "file.txt": undefined label 'ile.txt'
```

BSD sed treats the string immediately after `-i` as the backup suffix — it's not optional. When you write `sed -i 's/foo/bar/'`, BSD sees `-i` followed by `'s/foo/bar/'` as the suffix, and then `file.txt` as the script — which fails.

The BSD-compatible form is `sed -i '' 's/foo/bar/' file.txt`. The empty string is the suffix (meaning "no backup"). But if you write the GNU form in a script that runs on macOS, it breaks silently or with a confusing error.

The portable cross-platform pattern is:

```bash
# Works on both GNU sed (Linux) and BSD sed (macOS)
sed -i.bak 's/foo/bar/' file.txt
```

Providing an actual backup suffix is the one invocation that works everywhere — and it gives you an undo, which is always the right choice anyway.

### No Backup with `-i` Means No Undo

```bash
# This modifies the file immediately, with no recovery path
sed -i 's/production_db/dev_db/g' /etc/myapp/config.conf
```

If the regex was wrong — maybe it matched something it shouldn't have — the original is gone. `sed` does not keep a trash can. The file is overwritten.

Always test your sed command without `-i` first. Point it at the file, see the output on stdout, confirm it's correct, then add `-i`.

```bash
# Step 1: verify the output
sed 's/production_db/dev_db/g' /etc/myapp/config.conf

# Step 2: only once you've confirmed it's correct
sed -i.bak 's/production_db/dev_db/g' /etc/myapp/config.conf
```

This order matters. The five seconds it takes to verify has prevented more incidents than any other single habit in text processing.

### `p` Without `-n` Prints Every Line Twice

```bash
# Intent: print only lines containing ERROR
sed '/ERROR/p' /var/log/app.log
```

```text
INFO  starting up
ERROR disk full
ERROR disk full
WARNING low memory
```

The `ERROR` line appears twice: once from the explicit `p`, and once from the normal end-of-cycle automatic print. To print only matching lines, you must suppress the automatic output:

```bash
# Correct: -n suppresses automatic printing; p prints only matches
sed -n '/ERROR/p' /var/log/app.log
```

```text
ERROR disk full
```

The pairing `-n` + `p` is the correct idiom for "grep-like" behavior in sed. Without `-n`, `p` is an amplifier, not a filter.

### Address `1,/pattern/` vs `0,/pattern/` — First-Line Match Trap

This is a GNU sed extension, but it matters when it trips you up.

```bash
# File contents:
# BEGIN
# data
# END
# BEGIN
# more data

# Intent: print everything from the first BEGIN to the first END
sed -n '1,/END/p' file.txt
```

```text
BEGIN
data
END
```

That works. But what happens when `BEGIN` is on line 1 and you're matching from a regex start?

```bash
# Same intent, using regex for both addresses
sed -n '/BEGIN/,/END/p' file.txt
```

This also works — but there's a subtlety. The form `1,/pattern/` starts matching at line 1 and ends at the first line after line 1 that matches `pattern`. The form `0,/pattern/` (GNU extension) is needed when you want the end pattern to be able to match on line 1. If you use `1,/END/` and `END` is on line 1, the range doesn't close — it extends to the next `END` in the file. With `0,/END/`, the first line is eligible to close the range.

In practice: use `0,/pattern/` when you want to match "up to and including the first occurrence, even if it's on line 1."

### `-r` vs `-E` — Portability of Extended Regex Flag

GNU sed historically used `-r` for extended regular expressions. POSIX later standardized `-E`. Both flags do the same thing in GNU sed, but only `-E` is portable to BSD sed and other POSIX-conforming implementations.

```bash
# GNU sed only — -r is not POSIX
sed -r 's/[0-9]+/NUM/g' file.txt

# Portable — -E works on GNU sed, BSD sed, and POSIX sed
sed -E 's/[0-9]+/NUM/g' file.txt
```

If you're writing scripts that might run on macOS CI runners, Alpine containers, or any non-GNU system, use `-E`. There is no reason to use `-r` in new scripts.

### Delimiter Collisions — Silent Wrong Results

```bash
# Attempting to replace a URL containing slashes
sed 's/https://example.com/https://newsite.com/' file.txt
```

```text
sed: -e expression #1, char 22: unterminated `s' command
```

The shell sees the second `/` in `https://` as closing the delimiter and the regex becomes malformed. This doesn't always error out — sometimes it silently produces wrong output depending on what the trailing characters happen to match.

```bash
# Fix: use a delimiter that doesn't appear in the values
sed 's|https://example.com|https://newsite.com|' file.txt
```

Anytime your pattern or replacement contains `/`, switch the delimiter. This is not optional — it's a basic correctness requirement.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu / RHEL / Fedora / Arch | GNU sed (version 4.8+). `sed -i` without suffix works. `-r` and `-E` both accepted. |
| Alpine (containers) | GNU sed is available via `apk add sed`. The default BusyBox `sed` has a limited feature set — no hold space branching, limited regex. Install GNU sed explicitly in Dockerfiles when scripts depend on it. |
| macOS (BSD sed) | `-i` always requires a suffix argument. `-E` is supported; `-r` is not. No `\+`, `\?` BRE extensions. Scripts that run on Linux will not always run on macOS without modification. |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `sed` | Line-oriented transformations, substitution, deletion, in-place file editing | The right default for stream transformation |
| `awk` | Structured field manipulation, arithmetic, multi-pass logic over columns | Better than sed when lines have columns or when you need variables and math |
| `perl -pi -e` | Complex multi-line transforms, when sed's hold space is too limited | Full Perl regex (including lookaheads), but requires Perl — not always available in minimal containers |
| `tr` | Single-character transliteration, case conversion, deleting individual characters | Faster and simpler than sed `y///` for character-level work |
| Python/Ruby inline | Complex transforms that require real data structures, JSON, XML | When sed starts feeling like the wrong abstraction |

The honest recommendation: `sed` is the right tool for the vast majority of line-oriented text transformations. When you find yourself fighting the hold space to do something that should be simple, switch to `awk` or `perl`. The boundary is roughly: if you need more than two lines of sed involving the hold space, a different tool will be cleaner.

## Trade-offs

**Where this shines:**
- In-place batch editing of files: sed is the industry-standard way to modify config files during automated deployment — fast, reliable, no external dependencies
- Pipeline composition: sed is a first-class pipeline citizen; it reads stdin and writes stdout with zero overhead
- Speed on large files: sed processes line by line with minimal memory — it can process a 10 GB log file in the time it takes an editor to open it
- Availability: GNU sed is on every Linux system. You will never be on a server where sed is absent.

**Where this costs you:**
- Multi-line processing is painful: the hold space is powerful but cryptic; sed scripts that need to correlate multiple lines often become unreadable maintenance liabilities
- No real data structures: sed cannot handle JSON, CSV with quoted fields, XML, or any format where the logical unit is larger than one line
- POSIX portability is a moving target: the difference between GNU sed and BSD sed means scripts written on Linux often need adjustment for macOS or CI environments
- Regex-only matching: sed has no awareness of file format, quoting rules, or encoding; it sees bytes, not structure

**Performance note:** For files under a few hundred megabytes, sed is effectively instantaneous. For multi-gigabyte files, sed's line-at-a-time model with constant memory use is genuinely faster than tools that must load the file.

## When to Use / When to Avoid

✅ **Use this when:**
- You need to substitute, delete, or insert text in a file as part of an automated script or deployment
- You're working in a pipeline and need to transform one or more fields in a stream of lines
- You need to strip comments, blank lines, or prefixes from a text file
- You need to do the same transformation across many files at once (`sed -i` in a loop or with `find -exec`)
- You need a quick one-liner and `grep` can find the lines but can't transform them

❌ **Avoid this when:**
- The data has structure (columns, fields, JSON, XML) — use `awk`, `jq`, or `xmllint` instead
- You need to match or transform content that spans multiple lines — `awk` or `perl` handle this more clearly
- You're on a script that must run portably on both Linux and macOS without modification — test on both, or use `perl -pi -e` for the `-i` cases
- The file is binary — sed operates on text; binary data will produce garbage or errors

⚠️ **Common mistakes:**
- Forgetting `-n` when using `p` — results in doubled output
- Using `-i` without a backup suffix during development or testing
- Using `-r` instead of `-E` in scripts meant to run cross-platform
- Using `/` as delimiter when the pattern contains paths
- Writing a multi-step hold-space script when `awk` would do it in five readable lines

## Veteran Perspective

The newcomer mistake with sed is thinking it's just a "fancier grep." They reach for it only for substitution and only on single obvious patterns. That's fine for one-off interactive use, but it means they write deployment scripts that use Python subprocess calls to invoke sed one at a time in a loop, rather than writing a clean sed script that does it in one pass.

The intermediate mistake is overusing the hold space. I've seen sed scripts with four or five hold-space operations chained together, doing things that `awk` would express in three lines of clear code. The hold space is powerful but it's genuinely cryptic — scripts that use it heavily become write-only. If you find yourself writing `:loop`, `h`, `n`, `G`, `s/\n.*//`, step back and ask if `awk '{print $0}' RS=` would do it better.

The subtlety that separates competent sed use from expert sed use is understanding the `-i` portability trap at a systemic level. Most organizations that run Linux servers also have engineers developing on macOS. Scripts that do `sed -i 's/foo/bar/' file` work on the Linux servers but silently fail or error on macOS laptops during testing. This isn't a minor inconvenience — it causes genuine CI failures, and it's entirely avoidable by either using `sed -i.bak` (which works everywhere) or by using a wrapper function in your shell profile.

The deeper design principle: sed is the Unix philosophy taken seriously. It does one thing — stream edit — and does it well. It composes cleanly with everything because it only ever reads stdin and writes stdout. The `-i` flag is arguably a violation of this principle (it has side effects on the filesystem), which is why the POSIX committee resisted standardizing it for years. They were right to be cautious — `-i` is where portability breaks.

If someone on your team runs `sed -i` on a production config file without first testing the output, stop and have a conversation. It's not that the command is always wrong — it's that the habit of testing first before committing is what separates reliable infrastructure work from careless infrastructure work. The two-step pattern (test on stdout first, then `-i`) takes five seconds and has saved data I care about more than once.

## Try It Yourself

1. Run `sed 's/the/THE/' /etc/os-release` — watch that only the first occurrence on each line is replaced. Then add `g` flag and see the difference.

2. Use `sed -n '1,5p'` on any log file in `/var/log/` to print its first five lines. Compare the output to `head -5`. They should be identical.

3. Combine sed with a pipeline: run `ps aux | sed -n '/sshd/p'` to filter process list lines containing `sshd`. Notice how sed replaces grep here — and then try adding a substitution in the same command to also transform the output.

4. Create a file with 10 lines where some are blank and some start with `#`. Write a single sed command using two `-e` expressions to strip both blank lines and comments, printing only the active content. The output should have no blank lines and no comments.

5. **Challenge:** Write a sed script that reads a file where paragraphs are separated by blank lines, and numbers each paragraph. The output should add `Paragraph N:` before the first line of each paragraph. You'll need the hold space or the `=` command to track line context. Test in a VM. Hint: look at the `=` command for printing line numbers, and consider whether `awk` might be cleaner once you've solved it in sed.

## Summary

- `sed` processes text one line at a time: read into pattern space, apply script, print, clear — that's the main cycle, and everything else is a variation on it.
- The substitution command `s/pattern/replacement/flags` is the workhorse; `g` makes it global, `i` makes it case-insensitive (GNU extension), and `N` targets the Nth occurrence.
- `sed -i` does in-place editing; always use `sed -i.bak` in production scripts — no backup means no undo, and the cost of getting it wrong is a corrupted file.
- `sed -i` on macOS (BSD sed) requires a suffix argument: `sed -i '' ...` or `sed -i.bak ...`; the GNU form `sed -i 's/.../.../' file` silently misparses on BSD.
- Use `-E` instead of `-r` for extended regex — `-E` is POSIX-standard and works on both GNU and BSD sed; `-r` is GNU-only.
