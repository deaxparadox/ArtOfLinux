## The Problem This Solves

Before Unix, programs were written to talk directly to specific devices. A program that printed output was wired to a particular terminal or printer. A program that read input was wired to a keyboard or a specific file. If you wanted a program to read from a different source or write to a different place, you modified the program's source code. Every time.

Consider what that meant in practice: if you had a program that generated a report and you wanted to save that report to a file instead of watching it scroll past on screen, you had to either write a new version of the program with a different output destination, or write a wrapper around it. You could not compose tools. You could not chain them. Every pipeline you wanted to build had to be compiled in.

The real problem was even deeper than convenience. Programs tangled two different concerns: what they computed and where they sent the result. These are not the same problem and should not be solved by the same code.

Unix separated them. Programs would read from a standard input channel and write to a standard output channel, with no knowledge of what was on the other end. The shell — or any parent process — would wire those channels to files, other programs, or the terminal, at runtime. The programs themselves stayed clean.

That design decision is why you can run `grep error /var/log/syslog | sort | uniq -c | sort -rn | head -20` and get a sorted frequency table of unique error messages, without grep, sort, uniq, or head knowing anything about each other.

## Mental Model

Picture a water system in a building. Each apartment has a cold water inlet (stdin) and a drain (stdout). There's also a separate greywater drain for overflow and errors (stderr).

Normally the inlet comes from the municipal supply and the drains go to the sewer. But a plumber can reroute them. They can connect the drain of apartment 3A to the inlet of apartment 4B so that 3A's wastewater becomes 4B's supply. They can tee a drain so the water flows both to the sewer and into a collection tank. They can cap an inlet to receive nothing, or cap a drain so output is discarded.

The plumber is the shell. The apartments are programs. The water is data. The programs don't care what's on either end of their pipes — they just read from their inlet and push to their drains. The shell decides the topology.

## Theory

### The Three Standard Streams

Every process starts with three open file descriptors, inherited from its parent. By convention — and by OS design — these are:

- **fd 0 (stdin)** — the process reads its input from here
- **fd 1 (stdout)** — the process writes its normal output here
- **fd 2 (stderr)** — the process writes error messages and diagnostics here

These are not magic. They are small integers that index into the process's file descriptor table, which the kernel maintains. A file descriptor is a reference to an open file description — it can point to a regular file, a terminal device, a socket, a pipe, or any other file-like object the kernel knows about.

The reason there are two output streams — stdout and stderr — is intentional and important. A program's useful output (the data it was asked to produce) should travel separately from its error messages and diagnostics. This separation means you can process the data stream without interference from error text, redirect errors to a log file without losing the data, and silence errors completely when you don't care about them.

### Redirection Changes Where a Descriptor Points

Output redirection (`>`) tells the shell: before running this program, change what fd 1 points to. The program itself never knows. It still writes to fd 1 using the exact same write syscall. The kernel has simply re-wired where that fd goes.

The same is true for input redirection (`<`), error redirection (`2>`), and combined redirection (`&>`). They all work at the fd level, not at the program level. Programs don't need a "save to file" feature because redirection makes it a shell feature.

### Pipes Connect Two Processes Directly

A pipe (`|`) creates an in-kernel buffer. The shell wires the stdout of the left-hand process to the write end of that buffer, and the stdin of the right-hand process to the read end. Data flows through without ever touching the filesystem.

Critically: only stdout flows through a pipe. Stderr does not. When a command writes to stderr, that text goes wherever stderr was already pointed — usually the terminal — regardless of any pipes in the command. This surprises new users constantly when they pipe a command and see error messages appearing on the terminal. That's not a bug; it's the design.

### Pipelines Have an Exit Code

Without configuration, a pipeline's exit code is the exit code of the last command in the pipeline. Every command before the last can fail silently. With `set -o pipefail` enabled, a pipeline's exit code becomes the exit code of the rightmost command that exited non-zero, or zero if all commands succeeded. This was introduced in bash 3.0.

## How It Works (Mechanics)

### How `>` Works — Open, Dup2, Exec

When the shell processes `cmd > file`, it does the following before calling `exec()` to run the command:

```c
// Conceptual C — not a real command
int fd = open("file", O_WRONLY | O_CREAT | O_TRUNC, 0666);
dup2(fd, 1);   // make fd 1 point to the same file
close(fd);     // close the extra reference
```

`dup2(fd, 1)` duplicates the file descriptor `fd` onto slot number 1 in the file descriptor table, closing whatever was previously at slot 1. After this, fd 1 points to the file. The command is then `exec()`'d and it inherits that modified table. The program writes to fd 1 and its output goes to the file.

For `>>` (append), the only difference is `O_APPEND` instead of `O_TRUNC`. The file is not truncated on open; each write is positioned at the end.

### How `|` Works — Pipe, Fork, Dup2

When the shell processes `cmd1 | cmd2`:

```text
# Conceptual flow — not a real command
1. shell calls pipe()  → creates two fds: pipe[0] (read end), pipe[1] (write end)
2. shell forks child1  → child1 runs cmd1
   - dup2(pipe[1], 1)  → cmd1's stdout → write end of pipe
   - close both pipe fds
   - exec cmd1
3. shell forks child2  → child2 runs cmd2
   - dup2(pipe[0], 0)  → cmd2's stdin ← read end of pipe
   - close both pipe fds
   - exec cmd2
4. shell waits for both children
```

The kernel's pipe buffer is typically 64 KB. When `cmd1` writes faster than `cmd2` reads, `cmd1` blocks. When `cmd2` reads faster than `cmd1` writes, `cmd2` blocks. The kernel handles all the synchronization. Neither process needs to know about the other.

Stderr (fd 2) is untouched by this operation. Both child processes still have their fd 2 pointing to whatever the shell's fd 2 pointed to — the terminal.

### How `2>&1` Works — Duplicating File Descriptors

The syntax `2>&1` means: duplicate fd 1 onto fd 2, at this point in the redirection chain. After it executes, fd 2 and fd 1 point to the same open file description.

This is a snapshot, not a link. If you later redirect fd 1 to something else, fd 2 does not follow. That's the root cause of the most common redirection mistake (covered in Gotchas).

### Process Substitution — `<(cmd)` and `>(cmd)`

Process substitution is a bash extension that presents a command's output as if it were a file. The shell runs the command in a background process, connects a named pipe (or `/dev/fd/N` on Linux), and substitutes the path to that pipe in place of `<(cmd)`.

```text
# Conceptual flow — not a real command
diff <(sort file1) <(sort file2)
↓
1. bash runs "sort file1" → writes to /dev/fd/63
2. bash runs "sort file2" → writes to /dev/fd/62
3. bash runs: diff /dev/fd/63 /dev/fd/62
4. diff reads both fds as if they were regular files
```

This lets you pass command output to programs that expect filename arguments rather than stdin. It is not POSIX and not available in `/bin/sh`.

## Basic Example

```bash
# Redirect stdout to a file (overwrites)
echo "hello world" > /tmp/test.txt

# Show the file was written
cat /tmp/test.txt
```

```text
hello world
```

```bash
# Append a second line
echo "second line" >> /tmp/test.txt
cat /tmp/test.txt
```

```text
hello world
second line
```

```bash
# Redirect stderr to a file — try to list a file that does not exist
ls /nonexistent 2> /tmp/errors.txt
cat /tmp/errors.txt
```

```text
ls: cannot access '/nonexistent': No such file or directory
```

```bash
# Pipe: count the number of files in /etc
ls /etc | wc -l
```

```text
187
```

```bash
# tee: write to a file AND see output on screen at the same time
echo "logged line" | tee /tmp/tee-test.txt
```

```text
logged line
```

```bash
# The file also contains it
cat /tmp/tee-test.txt
```

```text
logged line
```

```bash
# tee -a: append instead of overwrite
echo "second entry" | tee -a /tmp/tee-test.txt
cat /tmp/tee-test.txt
```

```text
logged line
second entry
```

Now try this yourself. The `wc -l` count you see from `ls /etc | wc -l` will vary by distro and what you have installed — that's normal. Try `ls /usr/bin | wc -l` and see how it compares.

## Real-World Example

A common production task: run a long-running build script, capture both stdout and stderr to a log file, watch the output in real time, and check whether the build succeeded.

```bash
# Run a build, capture everything, and watch it live
# 2>&1 merges stderr into stdout first, then tee splits to file and terminal
make build 2>&1 | tee /var/log/build-$(date +%Y%m%d-%H%M%S).log
echo "Build exited with: $?"
```

The date-stamped log name means each build gets its own file — no single log.txt that constantly overwrites.

Now a diagnostic pipeline — finding which services are generating the most log traffic in the last hour:

```bash
# journalctl outputs to stdout, everything flows through the pipeline
journalctl --since "1 hour ago" \
  | awk '{print $5}' \       # extract the service field
  | sort \
  | uniq -c \                # count occurrences
  | sort -rn \               # sort by count, highest first
  | head -10                 # show top 10
```

```text
    842 sshd[1234]:
    317 systemd[1]:
    201 kernel:
     89 NetworkManager[847]:
     43 sudo[9821]:
```

Notice that if any command in this pipeline fails — say `awk` has a syntax error — you won't know it unless you have `set -o pipefail` enabled, because the pipeline exit code will be `head`'s exit code (0, since it successfully printed nothing).

A here document passes multiline input to a command that normally reads from stdin:

```bash
# Create a config file using a here document
cat > /tmp/app.conf <<EOF
[database]
host=localhost
port=5432
name=myapp

[cache]
host=127.0.0.1
port=6379
EOF

cat /tmp/app.conf
```

```text
[database]
host=localhost
port=5432
name=myapp

[cache]
host=127.0.0.1
port=6379
```

A here string is the compact form when you only need a single string:

```bash
# Feed a string directly to a command's stdin
# Useful when the string is already in a variable
log_line="2024-01-15 ERROR: disk full"
grep -o "ERROR.*" <<< "$log_line"
```

```text
ERROR: disk full
```

Process substitution lets you compare two sorted lists without temporary files:

```bash
# diff normally needs two filenames — process substitution provides them
diff <(ls /etc | sort) <(ls /usr/share | sort) | head -5
```

The walkthrough: both `ls` commands run in background processes. Their output goes to `/dev/fd/63` and `/dev/fd/62` respectively. `diff` sees two file paths and reads them. The result is a standard diff output showing what's in one directory but not the other — with no temporary files created, and no cleanup required.

## ⚠️ Gotchas

### The Redirection Order Trap — `2>&1 > file` vs `> file 2>&1`

This has caused more confusion than almost any other shell behavior. The expectation is that both forms send everything to the file. Only one of them does.

```bash
# What most people write when they want to capture everything
ls /etc /nonexistent 2>&1 > /tmp/output.txt
```

```text
ls: cannot access '/nonexistent': No such file or directory
```

The error message appears on the terminal. The file contains only the valid directory listing. This is the wrong order.

Redirections are evaluated left to right. At the time `2>&1` is evaluated, fd 1 still points to the terminal. So `2>&1` makes fd 2 point to the terminal. Then `> /tmp/output.txt` redirects fd 1 to the file. But fd 2 already got its value — it still points to the terminal. The two streams end up separated.

```bash
# The correct form — redirect stdout to the file FIRST, then merge stderr into it
ls /etc /nonexistent > /tmp/output.txt 2>&1
cat /tmp/output.txt
```

```text
ls: cannot access '/nonexistent': No such file or directory
/etc:
adjtime
bash.bashrc
...
```

The rule: `> file 2>&1` — the file redirect must come before `2>&1`. You can also use the shorthand `&>` which always does the right thing:

```bash
ls /etc /nonexistent &> /tmp/output.txt
```

`&>` is equivalent to `> file 2>&1` but there is no order ambiguity. It is bash-specific (not POSIX).

### Stderr Doesn't Go Through Pipes

A pipeline only connects stdout of the left command to stdin of the right command. Stderr is not in the pipeline.

```bash
# This looks like it should count total output lines, including errors
ls /etc /nonexistent 2>/dev/null | wc -l
```

```text
187
```

The error message for `/nonexistent` goes to the terminal, not through the pipe to `wc`. If you want to count error lines, you must explicitly merge stderr before the pipe:

```bash
# Merge stderr into stdout before the pipe
ls /etc /nonexistent 2>&1 | wc -l
```

```text
188
```

Now both the directory listing and the error line are counted. The order matters here too: `2>&1` must appear before the `|`.

### Pipefail — Silent Failures in Pipelines

Without `set -o pipefail`, a failing command in the middle of a pipeline is invisible.

```bash
# grep exits 1 if it finds no matches — but in a pipeline, that's hidden
cat /var/log/syslog | grep "NOSUCHMESSAGE" | wc -l
echo "Exit code: $?"
```

```text
0
Exit code: 0
```

The pipeline "succeeded" because `wc -l` exited 0. The fact that `grep` found nothing and exited 1 was discarded. In a script with `set -e` (exit on error), this pipeline would not stop the script because the overall exit code is 0.

```bash
set -o pipefail

cat /var/log/syslog | grep "NOSUCHMESSAGE" | wc -l
echo "Exit code: $?"
```

```text
0
Exit code: 1
```

With `pipefail`, the exit code of the pipeline is the exit code of the rightmost non-zero command. Now `grep`'s exit 1 propagates.

Important nuance: `grep` returning 1 does not mean it failed — it means no matches were found. In production scripts, you often need to distinguish between `grep` finding nothing (exit 1, expected) and `grep` crashing (exit 2, unexpected). Handle this explicitly rather than relying on the exit code alone.

### Here Document Indentation — Tabs Only

`<<-EOF` looks like it strips indentation. It does — but only tabs. Not spaces.

```bash
# This looks like it should work, but uses spaces for indentation
if true; then
    cat <<-EOF
        This line appears with extra spaces
    EOF
fi
```

```text
        This line appears with extra spaces
```

The spaces are preserved. The `<<-` stripped the leading tabs on the `EOF` delimiter (otherwise it would not have been recognized as a closing delimiter), but the body lines use spaces, which are not stripped.

```bash
# Correct form — use real tab characters for indentation
if true; then
	cat <<-EOF
	This line has no leading whitespace
	because the real tab was stripped
	EOF
fi
```

```text
This line has no leading whitespace
because the real tab was stripped
```

The practical problem is that most editors are configured to silently convert tabs to spaces. You may write a here document with apparent tab indentation, your editor converts it, and `<<-` stops working silently — the heredoc body has spaces that are preserved, and if the closing `EOF` has spaces instead of a tab, the shell never finds the end of the heredoc.

If you're writing here documents that will be indented in scripts, test that your editor is actually inserting tabs with `:set list` in vim or equivalent. The alternative is to not use `<<-` and accept that the heredoc body is flush-left.

### `echo -e` Is Not Portable

```bash
# This works in bash but may not work in sh or other environments
echo -e "line one\nline two"
```

```text
line one
line two
```

The `-e` flag tells `echo` to interpret backslash escape sequences. But in some shells and some distros, `/bin/echo` ignores `-e` and prints it literally. The POSIX spec for `echo` does not require `-e`.

```bash
# printf is always reliable — use it when you need escape sequences
printf "line one\nline two\n"
```

```text
line one
line two
```

`printf` handles `\n`, `\t`, and all escape sequences portably across every POSIX shell and every distro. Use `printf` when your output format matters. Reserve `echo` for simple strings with no escape sequences.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | `/bin/sh` is `dash`, which is a strict POSIX shell. Process substitution and `&>` are bash-only features — scripts using them must use `#!/bin/bash`, not `#!/bin/sh`. |
| RHEL / Fedora / CentOS / Rocky | `/bin/sh` is typically bash in compatibility mode on older RHEL; on RHEL 8+ it is bash. `set -o pipefail` works as expected. |
| Alpine (containers) | `/bin/sh` is busybox ash — no process substitution, no `&>`, very limited here-string support. Write container scripts for sh, not bash. |
| Arch | `/bin/sh` is bash by default. All bash features work, but portability discipline still matters for scripts shared across systems. |

## Alternatives & Comparisons

| Form | What It Does | When to Use |
|---|---|---|
| `&>` | Redirects both stdout and stderr to a file | Bash scripts only; cleaner than `> file 2>&1` when portability doesn't matter |
| `> file 2>&1` | Same result as `&>` | Use when the script must run under POSIX sh |
| `2>/dev/null` | Silences stderr only | Suppressing expected errors without losing stdout |
| `\|&` | Pipe both stdout and stderr — bash shorthand for `2>&1 \|` | Bash only; use `2>&1 \|` for portability |
| `tee` | Split output to file and stdout | When you need both logging and real-time output |
| `tee -a` | Append to file and stdout | When multiple runs should accumulate in one log |

The honest recommendation: for interactive use and bash-only scripts, `&>` and `|&` are cleaner. For any script that might run under `/bin/sh` — including scripts in Docker containers on Alpine or scripts you hand to a colleague — write `> file 2>&1` and `2>&1 |`. The POSIX forms are four characters longer and save you a debugging session.

## Trade-offs

**Where this shines:**

- Composing specialized tools without modifying any of them — the Unix model
- Zero-copy data transfer through kernel pipe buffers — faster than writing to disk and reading back
- `tee` lets you log and process simultaneously without buffering the entire stream in memory
- Process substitution eliminates temporary files in complex pipelines

**Where this costs you:**

- Pipe buffering creates backpressure — if a downstream command is slow, the entire pipeline stalls. A slow `gzip` at the end of a pipeline stalls the data producer at the start.
- Error handling in pipelines is genuinely hard. Without `pipefail`, middle-command failures are silent. With `pipefail`, legitimate "no results" exits from `grep` become false errors.
- Redirection order errors are syntactically valid and produce no warning — the shell silently does something different from what you intended.
- Process substitution creates background processes. If the background process crashes, the foreground command may get a broken pipe and fail in confusing ways.

**Performance note:** Kernel pipe buffers are 64 KB per pipe on Linux (configurable via `/proc/sys/fs/pipe-max-size`). For pipelines passing large data volumes, the producer and consumer must be matched in throughput or you'll see one process spinning at 100% CPU while the other is blocked waiting for buffer space. Profile with `pv` if pipeline throughput matters.

## When to Use / When to Avoid

✅ **Use pipes when:**
- You're composing standard text-processing tools (grep, sort, awk, sed, uniq, cut)
- You want to avoid temporary files — pipes are faster and require no cleanup
- You need real-time streaming — output flows as it's produced, not after the first command completes

✅ **Use `> file` when:**
- The output needs to persist beyond the current session
- A downstream tool needs random access to the data (pipes are sequential only)
- You're logging the output of a long-running job

✅ **Use `tee` when:**
- You need both a log and real-time visibility — a deployment script that logs everything but also shows output on the terminal

✅ **Use `set -o pipefail` when:**
- Writing bash scripts where correctness matters — any production or CI script
- You need the pipeline's exit code to reflect whether anything in the chain failed

✅ **Use `&>/dev/null` when:**
- Running cron jobs where you don't want email on expected stderr output
- Suppressing "file not found" errors from commands that are probing for optional resources

❌ **Avoid pipes when:**
- The data is binary and the tools in the pipeline are not binary-safe — `grep`, `sed`, and `awk` operate on text; piping binary data through them corrupts it
- You need the exit code of a middle command and you're not using `pipefail` — you'll silently lose it
- The pipeline is so complex that debugging it requires inserting `tee` at every stage anyway — at that point, temporary files with clear names are more debuggable

❌ **Avoid process substitution when:**
- The script must be POSIX-portable or run in Alpine containers
- The background command can fail silently — use explicit temporary files if you need the failure to be visible

⚠️ **Common mistakes:**
- Using `>>` in a loop to append, then forgetting to empty the file before the loop starts — old runs accumulate
- Running `cmd > file` and then immediately trying to read `file` in the same line — the redirection happens before cmd runs, but if cmd also reads `file`, it will read an empty file (truncation happens at open, before reading begins)
- Forgetting that `/dev/null` is a device, not a regular file — you cannot append to it, you cannot get data back from it, and `cat /dev/null` produces no output

## Veteran Perspective

The most common mistake I see from people who have been using Linux for a year or two is treating `2>&1` as magic. They copy it from Stack Overflow, it works, and they never look at why. Then they write `2>&1 > file` instead of `> file 2>&1` and spend an hour debugging why errors are still appearing on the terminal. The fix is to understand one rule: redirections are applied left to right, and `2>&1` is a snapshot, not a link. Write it on a card and put it next to your keyboard until it's automatic.

The second thing people miss is that stderr doesn't flow through pipes. This surprises people in specific ways: they pipe a command through `less` to page through output and the error messages fly past without paging. They pipe through `grep` to filter output and the errors still appear. They write a log parser that reads a pipeline and the errors never appear in the log. In all cases the fix is the same: `2>&1 |` to merge before the pipe. But the understanding — that pipes only carry stdout — is what prevents the confusion in the first place.

On `set -o pipefail`: every bash script that runs in production should have this. I've seen monitoring scripts that silently failed their `grep` step and reported "no issues found" because the pipeline exit code was the `wc -l` exit code (always 0), not the `grep` exit code (1 — no match found). The fix was one line at the top of the script. The outage it would have caught was twelve hours of unnoticed disk degradation.

Here documents look deceptively simple and hide two traps. The first is the tab-vs-space problem with `<<-EOF`. The second is variable expansion: by default, variables inside a heredoc are expanded. If you want a heredoc with literal dollar signs — say, you're generating a shell script that will itself use variables — quote the delimiter: `<<'EOF'`. The quotes suppress all expansion inside the heredoc.

Process substitution is genuinely elegant when you need it. `diff <(sorted_list_one) <(sorted_list_two)` is cleaner than writing two temp files, running diff, and cleaning up. But it is bash-only and it creates background processes. For scripts that need to run in containers or be portable, stick to temp files with explicit cleanup in a `trap`.

The deeper point: every one of these features — `>`, `>>`, `|`, `tee`, `<<`, `<<<`, `<()` — is built on two syscalls: `open()` and `dup2()`, and one more: `pipe()`. Everything else is the shell setting up file descriptors before `exec()`. Once you see it at that level, the behavior stops being mysterious. Weird redirection behavior almost always comes down to "which fd was pointing where at the moment of that operation."

## Try It Yourself

1. Run `ls /etc > /tmp/etc-list.txt` then `wc -l /tmp/etc-list.txt`. Now run `ls /etc | wc -l`. The counts should match. If they don't, figure out why — is there a trailing newline difference?

2. Run `ls /etc /nonexistent_dir > /tmp/out.txt 2>&1` and then `cat /tmp/out.txt`. Confirm you see both the directory listing and the error message in the file. Now try it with `2>&1 >` reversed. See what happens.

3. Write a pipeline that reads `/etc/passwd`, extracts the shell field (the last field, colon-delimited), sorts the result, removes duplicates, and shows how many users have each shell. Use `cut`, `sort`, and `uniq -c`. Then add `| sort -rn` to sort by frequency.

4. Write a one-liner that runs `find /tmp -name "*.log"` and splits the output: append to `/tmp/found.log` using `tee -a` while also feeding it through `wc -l` to count the results, showing the count on the terminal.

5. **Challenge:** Write a bash script (with `#!/bin/bash` and `set -o pipefail`) that tails the last 100 lines of `/var/log/syslog` (or `/var/log/messages` on RHEL), filters for lines containing "error" or "Error" case-insensitively, counts the occurrences, and writes a timestamped report to `/tmp/error-report-$(date +%Y%m%d).txt`. If no errors are found, the script should explicitly report "No errors found" rather than producing an empty file. Run it from cron (or at a set time) and confirm the file appears. Use a VM so you can generate test log entries with `logger "test Error message"`.

## Summary

- Every process has three standard file descriptors — stdin (0), stdout (1), stderr (2) — and redirects work by replacing what those integers point to using `dup2()` before `exec()`.
- Redirections are evaluated left to right: `> file 2>&1` sends both streams to the file; `2>&1 > file` sends stderr to the terminal and stdout to the file.
- A pipe (`|`) only connects stdout — stderr from the left command bypasses the pipe entirely and goes to the terminal unless you explicitly write `2>&1 |`.
- `set -o pipefail` makes a pipeline's exit code reflect the rightmost failing command; without it, a failing `grep` or `awk` mid-pipeline is silently ignored.
- `<<-EOF` strips only leading tab characters, not spaces — and process substitution `<(cmd)` is bash-only, not available in POSIX sh.
