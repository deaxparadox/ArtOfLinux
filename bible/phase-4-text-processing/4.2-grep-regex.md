## The Problem This Solves

You have a 400,000-line application log. Something went wrong at 03:17 this morning. You need every line that mentions an error — but only errors from a specific service, only from that time window, and only where the HTTP status code is in the 5xx range. You're not writing a script. You don't have time.

Without pattern matching, your options are: open the file in an editor and manually scan it (impossible), write a Python script (five minutes you don't have), or pray that the monitoring system already caught it (it didn't). Any of those would be slower, more fragile, and harder to adapt on the fly.

The fundamental operation missing from early computing was *filtered reading* — read a stream, emit only the lines that match a description. Once you can describe what you're looking for with precision, the rest of the problem collapses. That description language is regular expressions. The tool that applies them to text streams is `grep`.

`grep` was written by Ken Thompson in 1973, extracted from the `ed` editor's global search command (`g/re/p` — global, regular expression, print). The name isn't an acronym you need to memorize; it's a fossil from the editor. What matters is that in fifty years, nothing has replaced it for interactive search.

---

## Mental Model

Think of `grep` as a bouncer at the door of a text stream. Every line of input walks up to the door. The bouncer has one job: check the line against the pattern on the list. If the line matches — it gets in (printed to stdout). If it doesn't match — it's turned away (discarded). The bouncer doesn't rewrite lines, doesn't reorder them, doesn't combine them. It only decides: pass or discard.

The pattern on the list is a regular expression. A regular expression is not a search string — it's a *description of a set of strings*. The string `error` describes exactly one string. The pattern `err[oa]r` describes both `error` and `errar`. The pattern `[0-9]{3}` describes any three consecutive digits. The more expressive your description language, the more precisely you can describe what you want, without enumerating every possibility.

## Theory

### What grep Is

`grep` reads lines from one or more files (or from standard input if no files are given), tests each line against a pattern, and prints the lines that match. That's the whole contract. It does not modify input. It does not accumulate state across lines (by default). It processes one line at a time, in order.

This simplicity is the point. Because `grep` is stateless and composable, it pipes cleanly with every other Unix tool. `grep` produces lines in; another tool processes them. The Unix philosophy in one command.

### Three Flavors of Regular Expressions in grep

`grep` supports three distinct regex dialects, and the dialect you use changes what metacharacters mean:

**Basic Regular Expressions (BRE)** — the default mode when you just type `grep`. BRE has the smallest set of metacharacters. Grouping and repetition operators like `\(\)` and `\{n,m\}` require backslash-escaping to be treated as operators; without the backslash, they're literal characters. This is the historical default and the most portable across systems.

**Extended Regular Expressions (ERE)** — activated with `grep -E` or the `egrep` command (which is just `grep -E` under the hood on modern systems). ERE inverts the escaping logic: `()`, `{}`, `+`, `?`, and `|` are all operators by default, without backslashes. This is what most people think of as "normal" regex syntax. ERE is almost always what you want for interactive use.

**Perl-Compatible Regular Expressions (PCRE)** — activated with `grep -P`. PCRE adds a large set of features not in POSIX: `\d`, `\w`, `\s` shorthand classes, lookahead and lookbehind assertions, non-greedy quantifiers, named capture groups, and more. This is the most powerful mode — and the least portable. `-P` is not available on every system or every build of `grep`. More on this in Distro Notes.

### The Core Building Blocks

A regular expression is assembled from these pieces:

**Literal characters** — any character that isn't a metacharacter matches itself. `cat` matches the sequence `c`, then `a`, then `t`.

**The dot `.`** — matches any single character except a newline. `c.t` matches `cat`, `cut`, `c1t`, `c@t`. It does not match `ct` (too short) or `coat` (too long, because `.` is exactly one character).

**Character classes `[]`** — matches exactly one character from the set inside the brackets. `[aeiou]` matches any lowercase vowel. `[a-z]` matches any lowercase letter. `[^aeiou]` matches any character that is *not* a lowercase vowel — the `^` inside brackets negates the class.

**POSIX character classes** — portable named classes that work correctly regardless of locale and character encoding. The most important ones: `[[:alpha:]]` (letters), `[[:digit:]]` (digits 0-9), `[[:alnum:]]` (letters and digits), `[[:space:]]` (spaces, tabs, newlines), `[[:upper:]]` (uppercase letters), `[[:lower:]]` (lowercase letters), `[[:punct:]]` (punctuation). Always prefer `[[:digit:]]` over `[0-9]` in scripts — see the Gotchas section for why.

**Anchors** — `^` matches the start of a line; `$` matches the end. `^error` only matches lines that begin with `error`. `error$` only matches lines that end with `error`. `^$` matches empty lines.

**Word boundaries** — `\b` matches the position between a word character and a non-word character. `\bcat\b` matches the word `cat` but not `concatenate` or `scatter`. `\B` is the opposite — matches a position that is *not* a word boundary.

**Repetition** — how many times the preceding element must appear:
- `*` — zero or more times (BRE and ERE)
- `+` — one or more times (ERE only; BRE requires `\+` or doesn't support it)
- `?` — zero or one time — makes the preceding element optional (ERE; BRE: `\?`)
- `{n}` — exactly n times (ERE: `{n}`; BRE: `\{n\}`)
- `{n,m}` — between n and m times (ERE: `{n,m}`; BRE: `\{n,m\}`)

**Alternation `|`** — matches either the left or right side. `cat|dog` matches `cat` or `dog`. (ERE only; BRE requires `\|` and it's not even standard in strict BRE.)

**Grouping `()`** — groups a sub-expression so repetition or alternation applies to the whole group. `(ha)+` matches `ha`, `haha`, `hahaha`. (ERE: `()`; BRE: `\(\)`.)

### How Greedy Matching Works

All repetition operators in BRE and ERE are **greedy** by default — they match as many characters as possible while still allowing the overall expression to match. `.*` doesn't match "as few characters as possible"; it matches "as many as possible." This matters enormously when the pattern has characters after the greedy portion.

Given the string `<b>bold</b> and <i>italic</i>`, the pattern `<.*>` does not match `<b>` — it matches `<b>bold</b> and <i>italic</i>` (everything from the first `<` to the last `>`). The `.*` consumed as much as it could. This catches people constantly. PCRE mode (`-P`) supports non-greedy `.*?`, but BRE and ERE do not.

### grep Flags You Actually Use

| Flag | What It Does |
|---|---|
| `-i` | Case-insensitive matching — `Error`, `ERROR`, `error` all match `error` |
| `-v` | Invert match — print lines that do NOT match the pattern |
| `-r` / `-R` | Recursive — search all files under a directory; `-R` follows symlinks |
| `-l` | Print only filenames of files that contain a match, not the matching lines |
| `-n` | Prefix each matching line with its line number in the file |
| `-c` | Print only a count of matching lines per file, not the lines themselves |
| `-A n` | Print n lines of context After each match |
| `-B n` | Print n lines of context Before each match |
| `-C n` | Print n lines of context Before and After each match (combines -A and -B) |
| `-o` | Print only the matched portion of the line, not the whole line |
| `-w` | Match whole words only — the pattern must match a complete word |
| `-x` | Match whole lines only — the pattern must match the entire line |
| `-q` | Quiet — produce no output; exit with 0 (match found) or 1 (no match) |
| `-E` | Use Extended Regular Expressions (ERE) |
| `-P` | Use Perl-Compatible Regular Expressions (PCRE) — not universal |
| `-F` | Treat the pattern as a fixed string, not a regex — faster for literal searches |
| `--color` | Highlight matching text (usually on by default via alias) |
| `--include=GLOB` | When used with `-r`, only search files matching the glob pattern |
| `--exclude=GLOB` | When used with `-r`, skip files matching the glob pattern |

## How It Works (Mechanics)

When you run `grep pattern file`, the sequence is:

1. `grep` opens `file` (or reads from stdin if no file is given) and reads it line by line.
2. For each line, it applies the compiled regular expression against the line's bytes.
3. The regex engine (a finite automaton for BRE/ERE, a backtracking engine for PCRE) reports whether the pattern matches anywhere in the line.
4. If `-v` is not set and the pattern matches, the line is written to stdout. If `-v` is set, the line is written only if there is no match.
5. At the end, `grep` exits with code 0 if at least one line matched, 1 if no lines matched, and 2 if an error occurred.

That exit code distinction is important and widely used. In shell scripts, `grep -q pattern file` is a standard pattern test — no output, just the exit code. `if grep -q "error" logfile; then ...` is idiomatic shell.

For BRE and ERE, GNU grep uses a deterministic finite automaton (DFA) internally. A DFA converts the regex into a state machine where each state transition depends only on the next character — no backtracking, linear time in the length of the input. This is why `grep -E 'something'` on a gigabyte file is fast: it never goes back.

PCRE (`-P`) uses a backtracking engine. Backtracking is required for features like lookahead and non-greedy quantifiers, but it also means certain patterns can cause exponential worst-case behavior (catastrophic backtracking). On untrusted input, this matters.

When you use `grep -r`, `grep` walks the directory tree using `opendir`/`readdir` and opens each file it finds. It applies the same line-by-line logic to each file. With `--include=*.log`, it uses `fnmatch()` to filter filenames before opening them.

You can observe the system calls `grep` makes:

```bash
# watch what grep opens and reads — illustrative only
strace -e trace=openat,read,write grep "root" /etc/passwd 2>&1 | head -20
```

```text
openat(AT_FDCWD, "/etc/passwd", O_RDONLY) = 3
read(3, "root:x:0:0:root:/root:/bin/bash\n"..., 32768) = 2847
write(1, "root:x:0:0:root:/root:/bin/bash\n", 32) = 32
...
```

`grep` reads in large chunks (32KB by default), not one byte at a time, for efficiency.

## Basic Example

```bash
# create a sample file to search against
cat > /tmp/sample.log << 'EOF'
2024-01-15 03:17:01 INFO  service=auth user=alice action=login status=ok
2024-01-15 03:17:05 ERROR service=db   user=alice action=query status=500 msg="connection refused"
2024-01-15 03:17:09 INFO  service=auth user=bob   action=login status=ok
2024-01-15 03:17:12 ERROR service=api  user=bob   action=request status=503 msg="upstream timeout"
2024-01-15 03:18:00 INFO  service=db   user=root  action=reconnect status=ok
EOF

# basic match — lines containing "ERROR"
grep "ERROR" /tmp/sample.log
```

```text
2024-01-15 03:17:05 ERROR service=db   user=alice action=query status=500 msg="connection refused"
2024-01-15 03:17:12 ERROR service=api  user=bob   action=request status=503 msg="upstream timeout"
```

```bash
# case-insensitive
grep -i "error" /tmp/sample.log
```

```text
2024-01-15 03:17:05 ERROR service=db   user=alice action=query status=500 msg="connection refused"
2024-01-15 03:17:12 ERROR service=api  user=bob   action=request status=503 msg="upstream timeout"
```

```bash
# invert — lines that are NOT errors
grep -v "ERROR" /tmp/sample.log
```

```text
2024-01-15 03:17:01 INFO  service=auth user=alice action=login status=ok
2024-01-15 03:17:09 INFO  service=auth user=bob   action=login status=ok
2024-01-15 03:18:00 INFO  service=db   user=root  action=reconnect status=ok
```

```bash
# with line numbers
grep -n "ERROR" /tmp/sample.log
```

```text
2:2024-01-15 03:17:05 ERROR service=db   user=alice action=query status=500 msg="connection refused"
4:2024-01-15 03:17:12 ERROR service=api  user=bob   action=request status=503 msg="upstream timeout"
```

```bash
# only count how many lines match
grep -c "ERROR" /tmp/sample.log
```

```text
2
```

```bash
# show context — 1 line before and after each match
grep -C 1 "ERROR" /tmp/sample.log
```

```text
2024-01-15 03:17:01 INFO  service=auth user=alice action=login status=ok
2024-01-15 03:17:05 ERROR service=db   user=alice action=query status=500 msg="connection refused"
2024-01-15 03:17:09 INFO  service=auth user=bob   action=login status=ok
--
2024-01-15 03:17:09 INFO  service=auth user=bob   action=login status=ok
2024-01-15 03:17:12 ERROR service=api  user=bob   action=request status=503 msg="upstream timeout"
2024-01-15 03:18:00 INFO  service=db   user=root  action=reconnect status=ok
```

```bash
# extract only the matched portion — status codes
grep -oE "status=[0-9]+" /tmp/sample.log
```

```text
status=ok
status=500
status=ok
status=503
status=ok
```

Notice the `--` separator between context groups: that's `grep` telling you the context blocks are from different parts of the file. The `-o` flag with `-E` is a powerful combination — it turns `grep` into a field extractor without needing `awk` or `cut`.

Run these yourself. If you're on a system where `/tmp` is a tmpfs, the file won't survive a reboot — that's fine for practice. Try replacing `"ERROR"` with a pattern of your own.

## Real-World Example

You're debugging a production system. The application logs are at `/var/log/app/`. You need to:

1. Find all 5xx errors in today's logs
2. See what services are producing them
3. Check if any single user is hitting errors repeatedly
4. Verify no errors occurred in the last 10 minutes

```bash
# search all .log files under /var/log/app/ for 5xx status codes
# -r: recursive  -n: show line numbers  -E: extended regex  --include: only .log files
grep -rn -E "status=5[0-9]{2}" --include="*.log" /var/log/app/
```

```bash
# extract just the service names from error lines
grep -E "ERROR.*status=5[0-9]{2}" /var/log/app/app.log | grep -oE "service=[^ ]+" | sort | uniq -c | sort -rn
```

```text
      5 service=payments
      3 service=auth
      1 service=api
```

```bash
# find users with more than one error — extract usernames, count, filter
grep -E "ERROR.*status=5[0-9]{2}" /var/log/app/app.log \
  | grep -oE "user=[^ ]+" \
  | sort | uniq -c | sort -rn \
  | grep -v "^ *1 "
```

```text
      4 user=alice
      2 user=bob
```

```bash
# check if there are any errors in the last 10 minutes
# -q: quiet — we only care about the exit code, not the output
# date command generates the timestamp prefix for the last 10 minutes
SINCE=$(date -d "10 minutes ago" "+%Y-%m-%d %H:%M")
if grep -q "ERROR.*status=5" /var/log/app/app.log; then
    echo "Active errors in log — investigate"
else
    echo "No errors found"
fi
```

```bash
# during an incident, quickly find the first and last occurrence of an error
grep -n "ERROR.*payments" /var/log/app/app.log | head -1  # first occurrence
grep -n "ERROR.*payments" /var/log/app/app.log | tail -1  # last occurrence
```

The pattern in the second command is the Unix way: `grep` finds the lines, `sort | uniq -c | sort -rn` counts and ranks them, and a second `grep -v` filters the result. Each tool does one thing. No single command needs to understand the whole problem.

The `-q` flag in the fourth command is widely used in production scripts. A monitoring check, a pre-flight validation, a deploy gate — all of them want a yes/no answer, not a dump of matching lines. `grep -q` gives you the answer in the exit code and keeps stdout clean.

## ⚠️ Gotchas

### `grep "pattern" *` Silently Skips Directories

You're searching for a string across a directory of configs. You run:

```bash
grep "PermitRootLogin" /etc/ssh/*
```

```text
grep: /etc/ssh/ssh_config.d: Is a directory
/etc/ssh/sshd_config:PermitRootLogin no
```

`grep` printed an error for the directory and moved on. On a system where the interesting file is inside a subdirectory, you get no match and no indication that you missed it. The error message goes to stderr — if you're capturing output with `2>/dev/null` or piping it, you won't see it at all.

The rule: when `grep` is given a directory as an argument (including via glob expansion that picks up a directory), it refuses to read it and errors unless you tell it to recurse. It is not silently skipping — it is erroring — but in a pipeline, that distinction disappears.

```bash
# correct: recurse explicitly; restrict to the files you care about
grep -r "PermitRootLogin" /etc/ssh/

# or: restrict to specific file patterns
grep -r "PermitRootLogin" --include="*.conf" /etc/ssh/
```

### Greedy `.+` or `.*` Matches More Than You Expect

You want to extract the value inside HTML tags. You try:

```bash
echo "<b>bold</b> and <i>italic</i>" | grep -oE "<.+>"
```

```text
<b>bold</b> and <i>italic</i>
```

The entire string matched — from the first `<` to the last `>`. You expected `<b>` and `<i>` separately.

The rule: `.*` and `.+` are greedy. They consume as many characters as possible while still allowing the rest of the pattern to match. `<.+>` means "a `<`, then one or more of anything, then a `>` — but make the middle part as long as possible." The engine starts from the first `<`, expands `.+` until it hits the last `>` in the string, and reports that as one match.

BRE and ERE have no non-greedy quantifier. The fix, when you need non-greedy behavior, is either to use `-P` mode (if available) for `<.+?>`, or to rewrite the pattern to exclude the character you're trying not to match: `<[^>]+>` — "a `<`, then one or more characters that are not `>`, then a `>`."

```bash
# correct: exclude the delimiter character from the match
echo "<b>bold</b> and <i>italic</i>" | grep -oE "<[^>]+>"
```

```text
<b>
</b>
<i>
</i>
```

This pattern — `[^delimiter]+` instead of `.*` — is cleaner than non-greedy quantifiers in most cases anyway.

### Forgetting to Quote the Pattern — Shell Eats It First

You want to search for lines starting with `#` followed by a space:

```bash
grep ^# /etc/passwd
```

The shell sees `#` as the start of a comment. The `#` and everything after it on that command line is discarded before `grep` ever runs. `grep` receives only `^` as the pattern and `/etc/passwd` as the argument.

Or you search for an IP address range:

```bash
grep 192.168.1.* /var/log/syslog
```

The shell tries to expand `192.168.1.*` as a glob. If there happen to be no matching filenames in the current directory, the shell passes the literal string to `grep`. If there are files matching that pattern (unlikely but possible), `grep` receives filenames as arguments instead of a pattern.

The rule: always quote your regex pattern in single quotes. Single quotes pass the string to `grep` with zero interpretation by the shell. Use double quotes only when you need variable expansion inside the pattern.

```bash
# correct: single-quoted pattern
grep '^#' /etc/passwd

# correct: single-quoted IP pattern
grep '192\.168\.1\.' /var/log/syslog

# double quotes when you need a variable
SERVICE="auth"
grep "service=${SERVICE}" /var/log/app/app.log
```

### `[0-9]` vs `[[:digit:]]` — Locale Can Break Range Expressions

You're on a system with a non-C locale (French, German, Turkish, Unicode-aware locales are all common on modern Linux). You write:

```bash
grep '[0-9]' somefile
```

In the `C` locale, `[0-9]` reliably means the ASCII digits 0 through 9. In some other locales under some libc implementations, a range expression like `[0-9]` is interpreted according to the locale's collation order — and that collation order might include non-ASCII characters that fall "between" 0 and 9 in the locale's sort order. The behavior is implementation-defined.

More concretely: `[a-z]` in a locale that doesn't separate uppercase and lowercase characters in its collation might inadvertently match uppercase letters. `[A-Z]` might match some lowercase letters. The range syntax is fragile outside of the `C` locale.

`[[:digit:]]` is defined by POSIX to always mean the decimal digits — exactly `0123456789` — regardless of locale. It's not relying on character ordering; it's a named category.

```bash
# locale-dependent: potentially unreliable in non-C locales
grep '[0-9]' /etc/passwd

# correct: POSIX class, always means exactly 0-9
grep '[[:digit:]]' /etc/passwd

# or: force C locale for the command
LC_ALL=C grep '[0-9]' /etc/passwd
```

In scripts that run on mixed-locale systems — which is most production servers with internationalized application users — always use POSIX character classes or force `LC_ALL=C`.

### `grep -r` Follows Symlinks — Potential Loops

```bash
# this command could run forever on a filesystem with circular symlinks
grep -r "pattern" /some/directory/
```

`grep -r` (lowercase) does not follow symlinks to directories by default on GNU grep. `grep -R` (uppercase) does follow symlinks to directories, and if the directory tree contains a symlink that points to a parent directory, `grep -R` will recurse infinitely until it hits a path length limit or runs out of memory.

The distinction: `-r` — recurse into directories, do not follow symlinks to directories (but does follow symlinks to files). `-R` — recurse and follow all symlinks including directory symlinks.

```bash
# safer: lowercase -r avoids following directory symlinks
grep -r "pattern" /var/log/

# if you must follow symlinks and are worried about loops:
# ripgrep handles this safely by default
rg "pattern" /var/log/
```

On most real systems you won't hit this in `/var/log`, but on custom deployments with symlinked log directories pointing back to parent directories, `-R` has caused infinite loops and filled disks with grep's own output.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | `grep` is installed by default. PCRE support (`-P`) is compiled in via `libpcre2`. `grep` package: `grep`. The dedicated PCRE2 tool `pcre2grep` is in the `pcre2-utils` package. |
| RHEL / Fedora / CentOS / Rocky | `grep` is installed by default with PCRE2 support. `pcre2grep` is available in the `pcre2-tools` package. |
| Arch | `grep` is in `base` with PCRE2 support. `ripgrep` is in `extra`: `pacman -S ripgrep`. |
| Alpine (containers) | `grep` from BusyBox by default — significantly reduced functionality; no `-P`, no `-A/-B/-C` context flags. Install GNU grep: `apk add grep`. PCRE2 support with GNU grep on Alpine depends on the build. When in doubt, use `pcre2grep` or `ripgrep`. |

Alpine is the one that bites people most often. When your Dockerfile is based on `alpine` and your scripts use `grep -P` or `grep -A`, the BusyBox `grep` silently produces different results or errors. Always test grep behavior in the same environment the script will run in.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `grep` (GNU) | Interactive search, scripts, any POSIX system | The universal baseline. Available everywhere. `-E` covers 90% of real patterns. |
| `grep -F` / `fgrep` | Searching for a literal string with no regex special characters | Faster than regex matching; no risk of metacharacter misinterpretation. |
| `grep -P` | Lookahead, lookbehind, `\d`/`\w`/`\s`, non-greedy matching | Powerful, but not portable. Not available on BusyBox or builds without PCRE2. |
| `ripgrep` (`rg`) | Searching source code directories, large trees, daily development use | 5–13× faster than `grep -r` on large trees. Respects `.gitignore`. Better defaults. Written in Rust. |
| `pcre2grep` | When you need PCRE2 and `grep -P` is not available | Dedicated PCRE2 tool; same interface as grep but always PCRE2. |
| `ack` | Programmer-specific search with language-aware file-type filtering | Slower than `ripgrep`; less reason to reach for it today. |
| `ag` (the silver searcher) | Faster recursive search than grep; pre-ripgrep era | Largely superseded by `ripgrep` in performance and features. |

For interactive development work and searching code, use `ripgrep`. It respects `.gitignore`, handles binary files gracefully, and has sane defaults for colored output. For scripts, system administration, and anything you need to run portably on machines you don't control, use `grep`.

`fgrep` / `grep -F` is underused and worth knowing. When you're searching for a literal string — especially one that contains regex metacharacters like `.`, `*`, `[`, `(` — `grep -F` avoids having to escape the metacharacters and is faster because no regex compilation occurs.

```bash
# searching for a literal IP address: . is a metacharacter in regex
# without -F, you'd need: grep '192\.168\.1\.1'
grep -F "192.168.1.1" /var/log/syslog
```

## Trade-offs

**Where this shines:**

- Line-oriented filtering of streams and files — `grep` is unbeatable at "show me the lines matching X"
- Interactive log investigation — low syntax overhead, immediate results
- As a filter in pipelines — composes cleanly with `sort`, `uniq`, `awk`, `cut`, `wc`
- Portability — `grep` with BRE or ERE works on every Unix-like system
- Scripted conditions — the exit code makes `if grep -q ...` idiomatic and correct

**Where this costs you:**

- Multi-line patterns — `grep` is fundamentally line-oriented. Matching a pattern that spans two lines requires awkward workarounds (`pcregrep -M` or `grep -P` with `\n`, but only under specific conditions). For multi-line matching, `sed` or `awk` is the right tool.
- Extracting structured fields — `-o` can extract a matched portion, but if you need the third field or a named group, `awk` is more direct.
- Large directory trees — `grep -r` on `/var/log` with thousands of files is slower than `ripgrep` by a significant margin. Not a problem for one-off commands; relevant for scripts that run repeatedly.
- PCRE portability — patterns using `\d`, lookahead, or non-greedy matching written with `-P` will break on Alpine containers, BusyBox environments, and any system where GNU grep wasn't compiled with PCRE2.

**Performance note:** On GNU grep with BRE/ERE, the DFA engine provides O(n) matching time — linear in the input size. On large files (several gigabytes), `grep` is memory-efficient and fast. PCRE mode (`-P`) uses a backtracking engine; carefully crafted patterns or untrusted input can cause exponential backtracking. `ripgrep` (14.1.0 as of 2025) benchmarks at 5–13× faster than `grep -r` on large source trees due to multi-threaded search and SIMD optimizations.

## When to Use / When to Avoid

✅ **Use `grep` when:**

- Searching for a pattern in one or a few files interactively
- Writing portable shell scripts that need to run on any Linux system
- Filtering a stream in a pipeline — `command | grep pattern | next_command`
- Testing for a condition in a script — `if grep -q pattern file; then ...`
- Quick log searches where you need results immediately

✅ **Use `ripgrep` (`rg`) when:**

- Searching recursively through a source code directory
- Searching a tree that has a `.gitignore` — `rg` respects it automatically
- Speed matters — large trees, frequent searches
- You want better defaults without flags — colored output, binary file skipping

✅ **Use `grep -F` when:**

- Searching for a literal string containing regex metacharacters (`.`, `*`, `[`, `(`)
- The performance difference matters for very high-frequency literal searches

❌ **Avoid `grep -P` when:**

- Your script runs on Alpine, minimal containers, or systems you don't control
- Use `grep -E` first; only reach for `-P` when ERE genuinely can't express the pattern

❌ **Avoid `grep -r` (or especially `-R`) when:**

- The directory tree is large and speed matters — use `ripgrep` instead
- There's any possibility of symlink loops — prefer `rg` or audit the tree first

⚠️ **Common mistakes:**

- Using `grep pattern *` instead of `grep -r pattern .` — glob expansion picks up directories and fails
- Writing `.+` or `.*` and expecting non-greedy behavior — use `[^delimiter]+` or `-P` with `.*?`
- Forgetting that `grep` exit code 1 means "no match" (not an error) — don't treat it as a failure unconditionally in scripts with `set -e`; use `|| true` or check the exit code intentionally

## Veteran Perspective

The mistake most people make with `grep` is treating it as a string searcher with fancy syntax bolted on. It isn't. It's a filter that applies a formal language — regular expressions — to a stream of lines. Once that distinction lands, all the behavior makes sense. Greedy matching isn't a quirk; it's how finite automata naturally work. Anchors aren't shorthand; they're assertions about position in the string. Once you internalize that a regex is a *description*, the confusion drops dramatically.

The intermediate mistake is over-engineering patterns with `-P` for things that don't need it. I see scripts with `grep -P '\d{3}-\d{2}-\d{4}'` on Alpine images, and they break in CI. `grep -E '[0-9]{3}-[0-9]{2}-[0-9]{4}'` works everywhere and is just as readable. Reach for `-P` only when ERE genuinely can't express the pattern — lookaheads, lookbehinds, named captures, `\K` (keep the left part out of the match). That's rare.

The exit code behavior of `grep` is something beginners routinely hit in scripts with `set -e` (exit on error). `grep` returns exit code 1 when no lines match — this is not an error, it is a successful execution with a negative result. But `set -e` treats any nonzero exit as a fatal error. Patterns like `grep pattern file | wc -l` hide the exit code. Bare `grep pattern file` in a `set -e` script will kill the script silently if there are no matches. The right idiom is `grep -c pattern file || true` when you're counting, or test the exit code explicitly.

On the `ripgrep` vs `grep` question: in production scripts, use `grep`. It's always there. In your day-to-day development workflow, use `ripgrep`. The speed difference on a 200K-file source tree is not subtle — it's the difference between waiting and not waiting. The `.gitignore` awareness alone is worth it; you stop getting matches from `node_modules`, `.git`, compiled artifacts, and vendored code without doing anything.

Here's a real scenario: an engineer was investigating a slowdown in a service. They ran `grep -r "SELECT \*" /app/src/` to find unbounded database queries. The command ran for 45 seconds on a large Node project — most of the time was spent opening files in `node_modules`. They didn't know about `--exclude-dir`. A colleague showed them `rg "SELECT \*" /app/src/`, which returned in under a second because ripgrep skipped `node_modules` via `.gitignore`. The engineer had been doing this search manually every few weeks for months. The total time lost to that one oversight was hours.

## Try It Yourself

1. Run `grep -n "root" /etc/passwd` and observe the line numbers. Then run `grep -c "root" /etc/passwd`. The count should match the number of line-numbered results.

2. Search for all lines in `/etc/passwd` that contain a username starting with a lowercase letter followed by exactly two more lowercase letters (three-letter usernames): `grep -E '^[[:lower:]]{3}:' /etc/passwd`. If nothing matches, try `grep -E '^[[:lower:]]{2,5}:' /etc/passwd` to see what's there.

3. Run `grep -v "^#" /etc/ssh/sshd_config | grep -v "^$"` to strip comments and blank lines from the SSH config file, leaving only the active directives. This is a common pattern for reading config files.

4. Use `grep -oE "([0-9]{1,3}\.){3}[0-9]{1,3}" /var/log/syslog 2>/dev/null | sort -u | head -20` to extract unique IP addresses from the system log. If syslog is empty, try `/var/log/auth.log` or `/var/log/messages`.

5. **Challenge:** Write a pipeline that counts the top 10 most frequent words in `/etc/passwd` — not fields, individual words. A word here means a sequence of alphabetic characters. The pipeline should use `grep -oE`, then `sort`, then `uniq -c`, then `sort -rn`, then `head`. No `awk`, no `cut`, no `tr` — just `grep`, `sort`, `uniq`, and `head`.

## Summary

- `grep` filters lines by pattern; it never modifies input. Exit code 0 means match found, 1 means no match, 2 means error — the exit code is part of its interface.
- Use `grep -E` (ERE) for `+`, `?`, `|`, `()` without backslashes. Use `grep -F` for literal strings. Use `grep -P` for lookahead/lookbehind and `\d`/`\w` — but only when you control the environment.
- All repetition operators (`*`, `+`, `{n,m}`) are greedy by default in BRE and ERE. When a `.*` or `.+` matches more than you expect, rewrite it as `[^delimiter]+` — don't reach for `-P` first.
- Always single-quote your regex patterns to prevent shell glob expansion and comment interpretation from mangling the pattern before `grep` sees it.
- Use `[[:digit:]]`, `[[:alpha:]]`, and other POSIX character classes in scripts instead of range expressions like `[0-9]` or `[a-z]` — they're locale-independent and always mean exactly what they say.
