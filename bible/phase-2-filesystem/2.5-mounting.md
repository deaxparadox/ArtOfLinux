# [2.5] — Mounting

> The reason `/home` can live on a different disk than `/`, why a USB drive appears at `/media/usb` the moment you plug it in, and why a bad line in `/etc/fstab` can prevent your server from booting — all of it comes down to one operation: mounting.

## The Problem This Solves

Early systems treated storage as a flat, hardware-specific resource. If you had two disks, you had two completely separate storage spaces. Programs had to know which disk a file lived on — something like "disk A, sector 14." The application was coupled to the hardware.

Unix solved this by introducing the unified directory tree. There is exactly one root (`/`). Every file on every storage device lives somewhere under that root, reached by a single path like `/home/alice/notes.txt`. The application doesn't know or care whether that file is on a local NVMe drive, an external USB disk, an NFS share across the network, or a RAM-backed virtual filesystem. From the application's perspective, it's just a path.

The mechanism that makes this possible is mounting: the kernel's ability to attach a filesystem — whether it lives on physical hardware, a network server, a RAM region, or even a regular file — to a directory in the tree.

Without mounting, you couldn't separate your operating system disk from your data disk without rewriting every application to understand storage topology. You couldn't boot from one disk and store `/home` on another. You couldn't plug in a USB drive and access its files through a normal path. You couldn't give containers an isolated view of the filesystem. You couldn't mount a disk image without special hardware.

Mounting is what turns "a collection of separate storage devices" into "one tree."

---

## Mental Model

Think of the Linux filesystem tree as a peg board. Each peg is a directory. A filesystem is a cardboard box full of files. Mounting is the act of hanging a box on a peg: the box's contents become visible through that peg's location, and everything above it in the tree still belongs to the original board.

When you're done with the box, you unmount it — you take the box off the peg. The peg is still there. The directory it was hanging on is still there, empty as it was before. The box goes back to wherever it came from.

The key insight: the box doesn't become part of the board permanently. You can swap one box for another on the same peg, hang the same box on multiple pegs at once, hang a very small box on a peg so only a subset of the box is visible, or hang a box that generates its contents from RAM rather than from anything written on disk. The tree is the structure. Filesystems are the boxes you hang on it.

## Theory

### What a Filesystem Is

A filesystem is a structured region of storage — whether that's a partition on a disk, a file, a region of RAM, or a network export — that contains an organized set of files and directories. Every filesystem has its own inode table, its own directory tree, and its own root.

When you format a partition with `mkfs.ext4`, you are creating a filesystem: writing the superblock, the inode table, and the initial directory structure. That filesystem has its own root inode. On its own, that root is unreachable by any user. The filesystem exists, but it's floating — not attached to anything.

Mounting takes that floating root and grafts it onto a directory in the system's main tree, called the **mount point**. After the mount, the filesystem's root becomes accessible through that directory path.

### The Mount Table

The kernel maintains a mount table — an in-kernel data structure that records every active mount. Each entry maps a filesystem (identified by its source device or pseudo-device name) to a directory in the tree (the mount point). When the kernel resolves a path like `/home/alice/notes.txt`, it walks the directory tree and checks the mount table at each step. If a directory is a mount point, the kernel switches to the root of the mounted filesystem and continues path resolution there.

The mount table is visible at `/proc/self/mountinfo`. This is the authoritative, kernel-maintained record of all active mounts on the system. The older `/proc/mounts` is a simplified view of the same data; `findmnt` reads `mountinfo` directly and gives you the full picture.

### Mount Points

A mount point is just a directory. Any directory can be a mount point. When you mount a filesystem onto a directory, the directory's previous contents become invisible — hidden behind the newly mounted filesystem's root. They're not deleted. They come back when you unmount.

The classic example: `/` is always the first filesystem mounted (by the kernel itself, using information from the bootloader). Then early boot scripts mount `/proc`, `/sys`, and `/dev` onto their directories. Then `systemd` processes `/etc/fstab` and mounts everything else — `/boot`, `/home`, `/var`, swap — each onto its designated directory in the tree.

### What Gets Mounted

The `mount` syscall can attach:

- **Block device filesystems**: ext4, xfs, btrfs, vfat — data lives on a physical or virtual block device
- **Pseudo-filesystems**: proc, sysfs, cgroup2, devtmpfs — data is generated by the kernel, nothing stored on disk
- **Network filesystems**: NFS, CIFS/SMB — data lives on another machine
- **RAM-backed filesystems**: tmpfs — data lives in kernel page cache (RAM or swap)
- **Loop-backed filesystems**: a regular file presented to the kernel as if it were a block device
- **Bind mounts**: a directory made accessible at a second path in the tree

Each of these uses the same `mount()` syscall under the hood. The kernel's Virtual File System (VFS) layer dispatches to the right driver based on the filesystem type.

### Persistent Mounts: /etc/fstab

A manual `mount` command lasts until the next reboot. For mounts that should survive reboots, the traditional mechanism is `/etc/fstab`. This file tells the boot system — and `systemd`, via `systemd-fstab-generator` — which filesystems to mount, where, and with what options.

On modern systemd-based systems, `/etc/fstab` entries are read at early boot and converted into transient mount units. The fstab is not consulted again during normal operation — the systemd units it generated are what actually govern the mounts.

## How It Works (Mechanics)

### The mount() Syscall

When you run `mount /dev/sda2 /home`, the `mount` binary calls the `mount(2)` syscall:

```text
# Conceptual flow — not a real command
mount(source, target, filesystemtype, mountflags, data)
   │
   ▼
kernel VFS layer:
  1. Looks up the source device (or pseudo-device)
  2. Finds the registered filesystem driver for 'filesystemtype'
  3. Calls driver's mount() function → creates struct super_block
  4. Gets the root dentry of the new filesystem
  5. Creates a new struct mount entry in the mount table
  6. Attaches it to the target directory's dentry in the tree
  7. Returns 0 (success) or an error code
```

After step 6, any path resolution that reaches the target directory triggers a mount table lookup and switches to the new filesystem's root. The old directory contents are still there in memory — they're just shadowed.

### How Path Resolution Crosses Mount Points

```text
Path: /home/alice/notes.txt

Kernel resolves:
  /           → root filesystem (ext4 on /dev/sda1)
  /home       → check mount table: YES, ext4 on /dev/sda2 is mounted here
                → switch to /dev/sda2's root
  /alice      → continue on /dev/sda2
  /notes.txt  → inode lookup on /dev/sda2 → found
```

This crossing is transparent to the application. `open("/home/alice/notes.txt", ...)` works regardless of how many mount points are crossed on the path.

### What /proc/self/mountinfo Contains

Each line in `/proc/self/mountinfo` describes one mount entry. The fields are:

```text
36 35 8:1 / / rw,relatime shared:1 - ext4 /dev/sda1 rw,errors=remount-ro
│  │  │   │ │ │           │         │ │    │          │
│  │  │   │ │ │           │         │ │    │          └─ per-superblock options
│  │  │   │ │ │           │         │ │    └── source device
│  │  │   │ │ │           │         │ └─ filesystem type
│  │  │   │ │ │           │         └── separator
│  │  │   │ │ │           └── optional fields (propagation)
│  │  │   │ │ └── per-mount options
│  │  │   │ └─ mount point
│  │  │   └─ root within the source filesystem
│  │  └── major:minor device numbers
│  └── parent mount ID
└── mount ID
```

This is richer than `/proc/mounts`. The "root within the source filesystem" field is what distinguishes a bind mount (root is a subdirectory) from a full mount (root is `/`). The propagation fields record whether mount events propagate between mount namespaces — critical in container environments.

### How systemd-fstab-generator Works

At early boot (both in the initrd and in the main system), `systemd-fstab-generator` reads `/etc/fstab` and creates transient `.mount` unit files in `/run/systemd/generator/`. Each fstab entry becomes one unit file. Dependencies between units (`After=`, `Requires=`, `Wants=`) are added based on fstab options (`_netdev`, `noauto`, `nofail`).

```bash
# You can read the generated units at runtime
ls /run/systemd/generator/*.mount
systemctl cat home.mount    # view the generated unit content
```

These generated units are ephemeral — they live in `/run/` and are recreated on every boot. If you want a persistent, hand-crafted mount unit that systemd controls directly, you write it to `/etc/systemd/system/` instead.

## Basic Example

```bash
# Create a mount point (directory must exist before mounting)
sudo mkdir -p /mnt/test

# Mount a tmpfs (RAM-backed filesystem) — no physical device needed
sudo mount -t tmpfs -o size=64m tmpfs /mnt/test

# Verify it's mounted
findmnt /mnt/test
```

```text
TARGET     SOURCE FSTYPE OPTIONS
/mnt/test  tmpfs  tmpfs  rw,relatime,size=65536k
```

```bash
# Write a file to it
echo "this is in RAM" | sudo tee /mnt/test/hello.txt

# See the file
ls -lh /mnt/test/

# Unmount it — hello.txt is gone forever
sudo umount /mnt/test

# The mount point is still there, empty
ls /mnt/test/
```

```text
(empty — the directory exists but the filesystem is gone)
```

What to look for: `findmnt` shows the mount in the tree hierarchy, the source (`tmpfs`), the type, and the options including the size limit you set. After `umount`, the directory exists but is empty — the mount point itself is never affected by mounting or unmounting.

Now try it yourself. After running `sudo mount -t tmpfs -o size=64m tmpfs /mnt/test`, run `df -h /mnt/test`. You'll see a 64MB filesystem listed with zero bytes used. Write a few files, run `df -h /mnt/test` again — you'll see the usage climb. If your output shows a different size, verify the `size=64m` option was passed correctly.

## Real-World Example

This is a production-grade scenario: you've added a second disk to a server for storing database backups. The disk is `/dev/sdb`, formatted as ext4. You need to mount it persistently at `/mnt/backups` with appropriate security options, and verify everything is correct before rebooting.

```bash
# Step 1: Identify the disk and its UUID (never use /dev/sdb in fstab)
sudo blkid /dev/sdb1
```

```text
/dev/sdb1: UUID="a8f3c21d-4b7e-4e9f-9f1b-2d3e4f5a6b7c" BLOCK_SIZE="4096" TYPE="ext4" PARTUUID="d7e8f9a0-01"
```

```bash
# Step 2: Create the mount point
sudo mkdir -p /mnt/backups

# Step 3: Test the mount manually first — before touching fstab
sudo mount -o rw,nosuid,nodev,relatime UUID=a8f3c21d-4b7e-4e9f-9f1b-2d3e4f5a6b7c /mnt/backups

# Verify it's mounted correctly
findmnt /mnt/backups
```

```text
TARGET        SOURCE    FSTYPE  OPTIONS
/mnt/backups  /dev/sdb1 ext4    rw,nosuid,nodev,relatime
```

```bash
# Step 4: Check write access and available space
df -h /mnt/backups
touch /mnt/backups/testfile && echo "write: OK" && rm /mnt/backups/testfile
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sdb1       1.8T   28K  1.7T   1% /mnt/backups
write: OK
```

```bash
# Step 5: Unmount the test mount before adding to fstab
sudo umount /mnt/backups
```

```bash
# Step 6: Add the entry to /etc/fstab
# Use sudoedit or your editor with sudo
sudo nano /etc/fstab
```

Add this line (using the UUID from blkid, not /dev/sdb1):

```ini
# Data backup disk — mounted at /mnt/backups
UUID=a8f3c21d-4b7e-4e9f-9f1b-2d3e4f5a6b7c  /mnt/backups  ext4  rw,nosuid,nodev,relatime,nofail  0  2
```

```bash
# Step 7: Test the fstab entry without rebooting
# mount -a processes all fstab entries not currently mounted
sudo mount -a

# Verify it worked
findmnt /mnt/backups
```

```text
TARGET        SOURCE    FSTYPE  OPTIONS
/mnt/backups  /dev/sdb1 ext4    rw,nosuid,nodev,relatime
```

```bash
# Step 8: Verify systemd sees it correctly (on systemd systems)
systemctl daemon-reload
systemctl status mnt-backups.mount
```

```text
● mnt-backups.mount - /mnt/backups
     Loaded: loaded (/etc/fstab; generated)
     Active: active (mounted) since ...
      Where: /mnt/backups
       What: /dev/sdb1
      Tasks: 0 (limit: ...)
```

The walkthrough: `blkid` gives you the UUID — copy it exactly, including the hyphens. The `UUID=` form in fstab is stable regardless of how many disks you add or remove later. The `nosuid` and `nodev` options are standard defensive practice for non-root data disks: they prevent anyone from placing a device file or a setuid binary in your backup directory and using it to escalate privileges. `nofail` means if this disk is ever absent (disk failure, replaced hardware), the system boots anyway rather than dropping to an emergency shell. The `pass=2` tells fsck to check this filesystem at boot, but after the root (`pass=1`) is clean. Always run `mount -a` before rebooting — this is the test that catches typos before they become boot failures.

## ⚠️ Gotchas

### /dev/sdb in fstab Will Eventually Mount the Wrong Disk

This is the mistake that destroys production data. Device names like `/dev/sda`, `/dev/sdb` are assigned by the kernel in the order devices are detected during boot. That order is not stable.

```bash
# This looks fine on Tuesday
/dev/sdb1   /mnt/backups   ext4   defaults   0   2

# On Wednesday, after adding a USB drive or a SAN LUN,
# what was /dev/sdb may now be /dev/sdc.
# /dev/sdb1 might now be your OS disk, a temporary volume,
# or something that doesn't exist at all.
```

```text
# Possible outcomes: wrong disk mounted, data corruption,
# failed boot, or data overwritten if the disk holds a different filesystem
```

The kernel assigns `sda`, `sdb`, `sdc` in enumeration order. That order depends on PCI bus scan order, driver load order, and which physical ports have devices. None of these are guaranteed across hardware changes or kernel updates.

The correct form is always UUID:

```bash
# Find the UUID
sudo blkid /dev/sdb1

# Use it in fstab — this identifies the filesystem itself, not its hardware location
UUID=a8f3c21d-4b7e-4e9f-9f1b-2d3e4f5a6b7c  /mnt/backups  ext4  defaults  0  2
```

The UUID is written into the filesystem's superblock by `mkfs` and doesn't change until you reformat. It doesn't matter if the disk moves from `/dev/sdb` to `/dev/sdf` — the UUID is the same, and `udev` creates `/dev/disk/by-uuid/a8f3c21d-...` pointing to wherever the disk currently lives.

### A Bad fstab Entry Will Drop the System to an Emergency Shell

This has ended careers. If you add a fstab entry with a typo in the UUID, a wrong filesystem type, or a mount point directory that doesn't exist, the mount unit systemd generates from it will fail at boot. If that mount is a dependency of `local-fs.target` — which it is by default for any local filesystem — the boot sequence stops.

```ini
# The typo: UUID has one wrong character
UUID=a8f3c21d-4b7e-4e9f-9f1b-2d3e4f5a6XXXX  /mnt/backups  ext4  defaults  0  2
```

```text
# What you see at boot:
[FAILED] Failed to mount /mnt/backups.
[DEPEND] Dependency failed for Local File Systems.
Welcome to emergency mode! ...
Give root password for maintenance (or press Control-D to continue):
```

The fix from the emergency shell:

```bash
# Root filesystem is read-only in emergency mode — remount it writable
mount -o remount,rw /

# Fix the bad entry
nano /etc/fstab

# Use blkid to find the correct UUID if you need it
blkid | grep ext4

# Sync and reboot
sync
reboot -f
```

> ⚠️ **WARNING:** The `nofail` option prevents this exact failure for non-critical mounts. Add `nofail` to any fstab entry for a disk that the system can function without (external drives, secondary data disks, backup volumes). Never add `nofail` to mounts that system services depend on — `/var`, `/usr`, `/tmp`.

**The rule:** Always test with `sudo mount -a` immediately after editing `/etc/fstab`. This runs the same mount logic that will run at boot. If it fails, you find out now, with a working system, not at 3am when the server reboots after a kernel update.

### umount Fails with "target is busy"

You try to unmount a filesystem and the kernel refuses:

```bash
sudo umount /mnt/data
```

```text
umount: /mnt/data: target is busy.
```

The filesystem has references the kernel cannot release: a process has a file open on it, a process's current working directory is under it, or a running program's binary was loaded from it.

```bash
# Find what's holding the filesystem open
sudo lsof +D /mnt/data
```

```text
COMMAND  PID   USER   FD   TYPE DEVICE SIZE/NODE NAME
bash     4521  alice  cwd  DIR   8,17   4096      /mnt/data
vim      6789  bob    4    REG   8,17  28012      /mnt/data/config.yaml
```

The `cwd` entry for bash is the most common cause — someone `cd`'d into the mounted filesystem and forgot about it. The fix is to have that shell `cd` elsewhere, or to identify and stop the process holding a file open.

```bash
# See which processes have their cwd on the mount
sudo fuser -mv /mnt/data
```

```text
                     USER        PID ACCESS COMMAND
/mnt/data:           root     kernel mount  /mnt/data
                     alice      4521 ..c..  bash
                     bob        6789 ..f..  vim
```

The access code `c` means current working directory. Have alice run `cd /tmp`. The code `f` means open file — tell bob to save and quit vim. Then retry `umount`.

If you cannot wait for the processes to finish (maintenance window, stuck process), lazy unmount detaches the filesystem from the tree immediately and lets existing file handles drain naturally:

```bash
# Detach now — new processes cannot access the path, old ones finish gracefully
sudo umount --lazy /mnt/data
```

The lazy unmount makes `/mnt/data` unreachable to new processes immediately. Existing open file handles continue to work until closed. Once the last reference drops, the kernel finishes the unmount.

### tmpfs Data Is Lost on Unmount, Not Just Reboot

Everyone knows tmpfs is wiped on reboot. The part people miss: the data is also wiped the instant the filesystem is unmounted, regardless of whether the system is rebooting.

```bash
sudo mount -t tmpfs -o size=512m tmpfs /mnt/cache
echo "important data" > /mnt/cache/data.txt

# This destroys data.txt immediately
sudo umount /mnt/cache

# Even though the system is still running, the file is gone
ls /mnt/cache/    # empty
```

```text
(empty)
```

This matters in production when tmpfs mounts are used for application caches, session data, or inter-process communication. Unmounting during maintenance — even with the intent to remount immediately — wipes all data. Design your applications to treat tmpfs data as ephemeral at the granularity of each umount, not just each reboot.

### The pass Field Being Wrong Causes fsck Ordering Issues

The `pass` field in fstab (the sixth column) controls when `fsck` checks the filesystem at boot:

- `0` — do not check (swap, tmpfs, proc, network mounts)
- `1` — check first; only the root filesystem (`/`) should be `1`
- `2` — check after all `pass=1` filesystems; may run in parallel with other `pass=2` filesystems on different physical devices

The mistake is setting the root filesystem to `2`, or setting a non-root filesystem to `1`.

```ini
# Wrong: root filesystem set to pass=2
UUID=abc123  /      ext4  defaults  0  2   ← BAD: root must be 1

# Wrong: data disk set to pass=1
UUID=def456  /data  ext4  defaults  0  1   ← BAD: only root gets 1
```

If root is `0`, it is never checked — filesystem errors accumulate silently. If root is `2`, fsck tries to run the root check in parallel with other filesystems, which fails because the root needs to be clean before others can be checked. If a data disk is `1`, it delays the root check or runs before the root is verified.

The correct template:

```ini
UUID=...   /       ext4   defaults   0   1   ← root: always 1
UUID=...   /home   ext4   defaults   0   2   ← other local: 2
UUID=...   /boot   ext4   defaults   0   2   ← other local: 2
tmpfs      /tmp    tmpfs  defaults   0   0   ← pseudo-fs: always 0
UUID=...   none    swap   sw         0   0   ← swap: always 0
```

### Bind Mounts Are Invisible to df

You create a bind mount to expose a directory at a second path:

```bash
sudo mount --bind /srv/data /mnt/data-view
df -h
```

```text
Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        50G   12G   36G  25% /
/dev/sdb1        2.0T 500G  1.4T  27% /srv
```

`/mnt/data-view` doesn't appear in `df` output at all — it shares the same major:minor device numbers as `/srv`, so `df` suppresses the duplicate to avoid counting the same storage twice.

```bash
# findmnt shows all mount entries, including bind mounts
findmnt | grep data-view
```

```text
└─/mnt/data-view  /dev/sdb1[/data]  ext4  rw,relatime
```

The `[/data]` notation shows the root within the source filesystem — the bind mount starts at `/data` on the source, not at `/`. This is how you distinguish a bind mount from a full mount in `findmnt` output.

## Alternatives & Comparisons

| Mechanism | When to Use | Notes |
|---|---|---|
| `/etc/fstab` | Most persistent mounts on single-system servers | Universal, well-understood, required reading for every sysadmin |
| systemd `.mount` unit | When you need systemd dependency ordering, ordering relative to services, or conditional mounts | More verbose than fstab but integrates with `systemctl` and the journal |
| systemd `.automount` unit | Single known mount point that should be mounted on-demand | Pairs with a `.mount` unit; simpler than autofs for static paths |
| `autofs` | NFS home directories (wildcard key `*`); many mount points activated on-demand | The right choice when you need dynamic mounts (e.g., `*/export/home/&` matching any username) |
| `mount --bind` / bind mount | Exposing a subdirectory at another path; container bind mounts | Lightweight, no extra filesystem; same inode table, two access paths |
| Loop mount | Mounting disk images, ISOs, container base images | Needs a loop device from `losetup`; use `--partscan` for partitioned images |

For most production servers, fstab is still the right tool. systemd mount units add overhead without benefit unless you genuinely need the dependency integration. `autofs` is specifically designed for the NFS home directory problem — using it for anything else is usually overkill. If you find yourself writing many static `.automount` units, you probably want `autofs` with a wildcard key instead.

## Trade-offs

**Where this shines:**

- The unified mount namespace gives every process a consistent view of the filesystem, regardless of where data physically lives — applications don't need to know storage topology
- Mount options (`nosuid`, `nodev`, `noexec`) let you apply security policies at the filesystem level rather than per-file, reducing the attack surface on world-writable or untrusted mounts
- tmpfs provides zero-latency storage for ephemeral data — no disk I/O at all — which is measurably faster than `/tmp` on a spinning disk for write-heavy workloads like compilation artifact caches
- Mount namespaces (used by containers) let the kernel give each container a completely independent mount table from the same physical hardware

**Where this costs you:**

- A misconfigured `/etc/fstab` can make a system unbootable; the cost of a typo is not a failed command but a failed boot
- tmpfs memory is not free — it competes with application memory and the page cache; an oversized tmpfs can cause the kernel to swap application pages to make room
- Deep mount hierarchies (many filesystems stacked under `/`) increase path resolution overhead slightly; noticeable only at extreme syscall rates
- Bind mounts and mount namespaces are invisible to tools that read `/proc/mounts` rather than `/proc/self/mountinfo` — some older monitoring agents misreport disk usage when bind mounts are present

**Performance note:** On SSDs, the `relatime` option (default since kernel 2.6.30) eliminates the atime write-on-every-read overhead of `strictatime` while remaining compatible with programs that check access times. For SSD-heavy workloads that write many small temporary files (build systems, CI runners), mounting `/tmp` as tmpfs typically cuts build times by 10–30% by moving all temporary I/O to RAM.

## When to Use / When to Avoid

✅ **Use UUID= in fstab when:**
- Identifying any local filesystem on a production system — this is the correct default, always

✅ **Use tmpfs when:**
- Storing build artifacts, session caches, or inter-process sockets that don't need to survive a reboot
- You want to guarantee no sensitive data is written to disk (e.g., secret key material during a cryptographic operation)
- `/tmp` is a performance bottleneck and you have RAM to spare

✅ **Use bind mounts when:**
- You need a directory accessible at two paths (exposing a subdirectory to a container, making a config directory available at a canonical path)
- You want to apply different mount options to the same data (e.g., a read-only bind of a writable directory for a restricted process)

✅ **Use `nofail` when:**
- The fstab entry is for a non-critical mount — external drives, secondary data disks, NFS shares — that the system can boot without

✅ **Use systemd `.mount` units when:**
- You need the mount to have an explicit ordering dependency on a network service, database, or other systemd unit
- You need `systemctl start/stop/status` on the mount as part of operational runbooks

❌ **Avoid `/dev/sdX` in fstab** — device names change; a typo in hardware configuration can mount the wrong filesystem

❌ **Avoid mounting network filesystems without `_netdev`** — without this option, systemd will try to mount the NFS share before the network is up, causing a boot hang or failure

❌ **Avoid `umount -f` on local filesystems** — force unmount is designed for unreachable NFS servers; using it on a local ext4 in use can corrupt the filesystem

❌ **Avoid setting `pass=0` for the root filesystem** — it will never be checked by fsck; silent filesystem errors accumulate

⚠️ **Common mistakes:**
- Editing fstab and rebooting without running `mount -a` first — always test before rebooting
- Forgetting that `umount` destroys tmpfs data immediately, not just at reboot
- Using `noexec` and expecting it to prevent `bash script.sh` — `noexec` blocks `execve()` on files in that filesystem; it does not prevent an interpreter loaded from elsewhere from reading and executing a script file
- Setting a `size=` on tmpfs and thinking the memory is pre-allocated — tmpfs only consumes RAM for data actually written to it; the size is a ceiling, not a reservation

## Veteran Perspective

The most dangerous misconception about `/etc/fstab` is that it's forgiving. It isn't. It's one of the few config files on the system where a single typo can render the machine unbootable. I've watched engineers add a backup disk at 4pm, add the fstab entry without testing it, reboot at 4:15pm for "a quick kernel update," and spend the rest of the evening on the console recovering from an emergency shell. The fix was always the same: `mount -a` first. Always. No exceptions.

The `/dev/sdX` problem is the one that trips up people who learn Linux on a laptop and then go to work on a server. On a laptop with one SSD, `/dev/sda1` is always your SSD. On a server with four drives, a RAID controller, and a USB console cable, the enumeration order is unpredictable and changes when hardware changes. I've seen database servers mount the wrong volume after a storage array firmware update because someone had used `/dev/sdb1` in fstab for three years without incident — right up until the incident. UUIDs are the answer. Always UUIDs.

The intermediate mistake with tmpfs is sizing it and then forgetting that the size is a ceiling, not a reservation. An engineer sets `/tmp` to `size=8g` thinking it's "safe" because the server has 64GB of RAM. Six months later, a runaway build process fills `/tmp` with 8GB of artifacts, the kernel starts evicting page cache to accommodate, database query latency spikes, and the incident report says "unexplained performance degradation." The root cause is that tmpfs competes with everything else for physical RAM. Size it conservatively for production systems, and monitor it.

Bind mounts are underused outside of container work. They solve a class of problems that people usually reach for symlinks to fix — but a bind mount has real filesystem semantics that a symlink doesn't. `chroot`'d processes follow bind mounts correctly. You can set mount options on a bind mount (read-only) that differ from the source. You can unmount a bind mount without affecting the source. Symlinks work 90% of the time, but when they don't work (inside chroot jails, with certain syscalls, across namespace boundaries), bind mounts are always the right answer.

On the `pass` field: I've seen new hires set everything to `pass=1` thinking "more checking is better." It isn't. The `1` is specifically reserved for the root filesystem. It means "check this first, before anything else, because nothing else can be checked until this is clean." Setting a data disk to `1` breaks the parallelism that makes boot fast on multi-disk systems. Set root to `1`, everything else that needs checking to `2`, and everything that can't be fsck'd to `0`. This is not a preference — it's the documented contract.

The `nofail` option is a professional habit worth developing. Every secondary mount — backup volumes, secondary data disks, removable media — should have `nofail`. The scenario where this saves you: you replace a failed disk, forget to `mkfs` with the same UUID, and reboot the server. Without `nofail`, the server doesn't come back. With `nofail`, it comes back without `/mnt/backups` — and you have a chance to fix it rather than an emergency call about a server that won't boot.

## Try It Yourself

1. Run `findmnt` with no arguments and read the tree output. Find at least three different filesystem types. Find the entry for `/tmp` — is it a tmpfs or is it on your root disk? Note how the indentation shows which mounts are nested under others.

2. Run `sudo mount -t tmpfs -o size=64m tmpfs /mnt/test` (create `/mnt/test` first with `sudo mkdir /mnt/test`). Write a file to it. Check `df -h /mnt/test` and `findmnt /mnt/test`. Then unmount it and verify the file is gone. *(Safe, creates no permanent changes.)*

3. Run `sudo blkid` to list all block devices and their UUIDs on your system. Cross-reference the UUIDs against your `/etc/fstab` — find the entry for your root filesystem. Confirm the UUID in fstab matches what `blkid` reports for your root partition.

4. Create a bind mount: `sudo mkdir /mnt/bind-test && sudo mount --bind /etc /mnt/bind-test`. Run `ls /mnt/bind-test` — you should see the contents of `/etc`. Then run `df -h` and note that `/mnt/bind-test` does NOT appear. Run `findmnt /mnt/bind-test` — it appears here, with the `[/]` notation showing it's a bind of the whole source. Clean up with `sudo umount /mnt/bind-test`.

5. **Challenge:** Create a loop-mounted disk image. Use `truncate -s 256m /tmp/test.img` to create a sparse image file. Format it with `mkfs.ext4 /tmp/test.img`. Mount it with `sudo mount -o loop /tmp/test.img /mnt/test`. Check `losetup -a` to see the loop device that was created automatically. Write files to it, unmount it, and verify with `losetup -a` that the loop device was released. Then try: `losetup -f --show /tmp/test.img` to attach the loop device explicitly, inspect it with `lsblk`, mount it manually, and detach it with `losetup -d`. *(Requires `sudo`. Safe to run in a VM or on your own system — creates no permanent changes.)*

## Summary

- Mounting attaches a filesystem's root inode to a directory in the tree; after mounting, all paths under that directory resolve into the mounted filesystem — the kernel crosses mount points transparently during path resolution
- Always use `UUID=` in `/etc/fstab`, never `/dev/sdX` — device names are assigned at boot in enumeration order and change when hardware changes; UUIDs are stable for the lifetime of the filesystem
- After every edit to `/etc/fstab`, run `sudo mount -a` before rebooting — this tests the new entries with a running system and catches typos before they become boot failures
- tmpfs data is lost the instant the filesystem is unmounted, not just on reboot; size it conservatively because it competes with application memory
- The `pass` field controls `fsck` order: `0` for pseudo-filesystems and swap, `1` for root only, `2` for all other local filesystems that need checking
