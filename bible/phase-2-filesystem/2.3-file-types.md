## Why This Exists

The "everything is a file" philosophy only works if the kernel can tell different *kinds* of files apart. A process reading from a terminal behaves differently from a process reading from a disk. A directory entry behaves differently from a data file. Sockets and named pipes have completely different lifecycle rules.

If every entry in a filesystem looked identical, the kernel would have no way to route operations correctly. Reading a directory entry the same way you read a text file would produce garbage — or worse, silent corruption.

Linux solves this with seven distinct file types, each encoded in the first character of the ten-character mode field you see in `ls -l`. Seven types. One character. Every file on the system falls into exactly one of them, and the kernel uses that single character to decide which operations are valid and how to execute them.

This is not trivia. The first time you try to `cat` a block device by accident, or create a symlink that points nowhere, or hardlink a file across filesystems and get an error you don't understand — knowing this taxonomy is what turns confusion into clarity.

## Mental Model

Think of the filesystem as a city's postal system, and every entry in a directory as an address placard nailed to a door.

Most placards say "Ordinary Building" — you knock, someone answers, normal building contents are inside. That's a regular file.

Some placards say "Directory Office" — inside, there's nothing but a list of other addresses. That's a directory.

Some placards say "Forwarding Address" — you go there and find a piece of paper that says "the thing you want is actually at 42 Oak Street." That's a symlink. The placard itself is the symlink; 42 Oak Street is the target. If 42 Oak Street gets demolished, your forwarding note still hangs there — it's just pointing somewhere that no longer exists.

Some placards lead directly to hardware. "Block 3, Sector 7 — this is the disk." "Terminal Line 2 — this is the keyboard stream." Those are device files. The kernel wired them in.

Some placards are mailboxes that two different delivery workers share — data goes in one side, comes out the other, gone forever. Those are pipes.

And some are switchboards — processes call in, processes call out, real-time connection. Those are sockets.

The placard type doesn't tell you what's inside. It tells you how to interact with the address. That's the whole model.

## Theory

### The Seven File Types

Every directory entry on a Linux filesystem has a type. The kernel exposes this type via the first character in `ls -l` output:

| Character | Type | Description |
|---|---|---|
| `-` | Regular file | Data stored on disk: text, binaries, images, scripts |
| `d` | Directory | A file containing name-to-inode mappings |
| `l` | Symbolic link | A pointer to another path |
| `b` | Block device | Disk, partition, LVM volume — accessed in fixed-size blocks |
| `c` | Character device | Serial port, terminal, `/dev/null` — accessed one character at a time |
| `s` | Socket | IPC endpoint for inter-process communication |
| `p` | Named pipe (FIFO) | Persistent pipe that survives in the filesystem |

This character comes directly from the inode's `st_mode` field, which `ls` reads via the `stat()` system call. It is not stored as a separate metadata attribute — the type bits and the permission bits are packed together in the same 16-bit field. The kernel uses the high 4 bits for the type and the low 12 bits for the permissions.

### Regular Files

A regular file (`-`) is what most people think of when they hear "file" — a sequence of bytes stored on disk. Text files, compiled binaries, shell scripts, images, compressed archives — all regular files. The kernel treats them uniformly: you open them, read or write a stream of bytes, and close them.

Linux does not use file extensions to determine what a file *is*. A file named `script.py` is only a Python script if the first line says `#!/usr/bin/env python3` or if you explicitly run it with `python3`. A file named `photo.jpg` might be a JPEG or it might be renamed executable — the extension is a hint for humans, not a fact the kernel acts on. The `file` command reads the actual content (specifically, "magic bytes" — known byte sequences at the start of a file that identify its format) to determine the real type.

### Directories

A directory (`d`) is not a "folder" in the GUI sense. That word is a visual metaphor invented for graphical desktops. On disk, a directory is a file — a special one — whose contents are a table of name-to-inode mappings. Each row in that table says: "the name `passwd` maps to inode 123456." Nothing more.

When you `cd` into a directory, the kernel looks up the inode for that directory, reads its table, and gives you a handle into that region of the tree. When you create a file, the kernel creates a new inode and adds a row to the containing directory's table. When you delete a file with `rm`, the kernel removes that row from the table and, if no other rows point to the same inode, frees the inode.

This design means that file names live in directories, not in inodes. A file doesn't know its own name. That's why you can give the same file multiple names — multiple directory entries pointing to the same inode. This is the mechanism behind hard links.

### Symbolic Links

A symbolic link (`l`) stores a path as its data, nothing more. When the kernel encounters a symlink during path resolution, it reads the stored path and substitutes it for the symlink's location in the path. Then it continues resolving from there.

The critical difference between a symlink and a hard link: a symlink stores a *path*, not an inode number. If the target path changes, the symlink becomes invalid. The symlink doesn't know or care about the inode behind the target — it just knows a name.

This means symlinks can point across filesystem boundaries, can point to directories, and can point to targets that don't exist yet. Hard links cannot do any of these things.

### Hard Links

A hard link (`-`, same character as a regular file) is a second directory entry pointing to the same inode. There is no "original" and no "copy" — both entries are equally valid names for the same underlying data. When you check the link count in `ls -l` output (second column), you're seeing how many directory entries currently reference that inode.

Deleting one hard link does not delete the data. The kernel only frees the inode when the link count drops to zero and no process has the file open. This is why `rm` is technically called "unlink" at the system call level — it removes one directory entry, which may or may not delete the underlying data.

Hard links have two fundamental restrictions:

1. **Hard links cannot span filesystems.** Each inode number is only meaningful within its own filesystem. A hard link is a directory entry containing an inode number — that number has no meaning on a different filesystem. Attempting this fails with `EXDEV` (cross-device link).

2. **Hard links to directories are not permitted** (with the exception of `.` and `..`, which the kernel manages internally). If this were allowed, a directory could contain a hard link to one of its ancestors, creating a cycle. Filesystem traversal tools (`find`, backup utilities, filesystem checkers) would loop forever. The `..` entry already creates a controlled relationship between a directory and its parent — uncontrolled hard links to directories would make `..` ambiguous and the filesystem tree structure meaningless.

### Block and Character Device Files

Device files (`b` and `c`) are the "everything is a file" principle applied to hardware. They don't store data — they're entry points into kernel drivers.

A block device (`b`) represents hardware that the kernel communicates with in fixed-size chunks (typically 512 bytes or 4096 bytes). Disks and their partitions are block devices. LVM logical volumes are block devices. The kernel buffers block device I/O and can satisfy reads from cache without touching the hardware.

A character device (`c`) represents hardware that streams bytes without buffering. Terminals (`/dev/tty`), serial ports (`/dev/ttyS0`), the kernel random number generator (`/dev/urandom`), the null device (`/dev/null`), and the zero device (`/dev/zero`) are all character devices. There are no blocks, no seeking, no caching at the device level — just a stream.

Both device types are identified by a major and minor number pair stored in the inode. The major number identifies the driver; the minor number identifies the specific instance managed by that driver. You can see these in `ls -l` output in place of the file size field.

### Sockets

A socket (`s`) is an IPC endpoint created by a running process. Unix domain sockets appear in the filesystem as a file. When the process exits, it is responsible for removing the socket file — if it crashes, the socket file usually remains. Unlike named pipes, sockets are bidirectional and support multiple simultaneous connections.

You'll see them in `/tmp`, `/run`, and `/var/run`. `mysqld.sock`, `docker.sock`, `systemd's` private sockets — these are all Unix domain socket files. A process connects to a socket by opening it with the `connect()` system call, not the `open()` system call.

### Named Pipes (FIFOs)

A named pipe (`p`) is a pipe that exists as a filesystem entry. An anonymous pipe (created with `|` in the shell) lives only in memory and disappears when both ends close. A named pipe persists on disk until explicitly removed with `rm`.

A writer opens the named pipe and writes. A reader opens the named pipe and reads. Bytes go in one end, come out the other, in order, exactly once — there's no seeking, no random access. If the reader hasn't opened the pipe yet, the writer blocks. If the writer closes its end, the reader gets EOF.

Named pipes are mostly used in controlled production environments where two separate processes need to exchange a stream of data without using a file as intermediary, and where the processes might start at different times.

## How It Works (Mechanics)

### How a Directory Entry Maps a Name to an Inode

When the kernel reads a directory, it reads a sequence of directory entries. On ext4, each entry contains:

- A 4-byte inode number
- A 2-byte record length (to allow variable-length names)
- A 1-byte name length
- A 1-byte file type hint (a redundant copy of the type, so `readdir()` doesn't need to `stat()` every entry)
- The filename bytes (variable length, null-terminated)

```text
# Conceptual layout of a directory's data block
[ inode=2 | reclen=12 | namlen=1 | type=d | "." ]
[ inode=1 | reclen=12 | namlen=2 | type=d | ".." ]
[ inode=7  | reclen=16 | namlen=6 | type=r | "passwd" ]
[ inode=8  | reclen=16 | namlen=5 | type=r | "group" ]
[ inode=12 | reclen=20 | namlen=9 | type=d | "ssl" ]
```

When you run `ls`, the kernel calls `getdents64()` to return these entries. When you run `stat()` or `ls -l`, the kernel takes each inode number and reads the full inode structure to get permissions, size, timestamps, and type.

The directory file itself has an inode, permissions, and a size — it is a real file. Its contents are the table above. This is why you need read permission on a directory to list its contents, and execute permission on a directory to traverse into it.

### How Symlinks Store Their Target

Short symlinks — under about 60 bytes on ext4 — are stored as "fast symlinks": the target path is written directly into the inode's data area, where block pointers would normally live. No separate data block is allocated. This is purely a performance optimization: reading a short symlink doesn't require an extra disk read.

Longer symlinks allocate a full data block and store the path there, like any other file.

You can see this target path with `readlink`:

```bash
readlink /etc/mtab
```

```text
../proc/self/mounts
```

That stored string `../proc/self/mounts` is the entire content of the symlink's data. Nothing else.

### How the Kernel Follows Symlinks During Path Resolution

Path resolution happens in the VFS (Virtual Filesystem) layer, in the kernel function that walks a pathname component by component. Here is the step-by-step trace for resolving `/etc/ssl/cert.pem` on a system where `cert.pem` is a symlink to `../certs/ca-certificates.crt`:

```text
# Conceptual trace — not a real command
1. Start at the root inode for "/"
2. Look up "etc" in root's directory table → inode 5243
3. Stat inode 5243: type=directory, proceed
4. Look up "ssl" in inode 5243's directory table → inode 8891
5. Stat inode 8891: type=directory, proceed
6. Look up "cert.pem" in inode 8891's directory table → inode 9012
7. Stat inode 9012: type=symlink
8. Read symlink data: "../certs/ca-certificates.crt"
9. Restart resolution from the symlink's directory (/etc/ssl)
   with the new component: "../certs/ca-certificates.crt"
10. Resolve ".." → /etc
11. Look up "certs" in /etc → inode 8901
12. Look up "ca-certificates.crt" in inode 8901 → inode 9105
13. Stat inode 9105: type=regular file, done
```

The kernel maintains a counter of symlink resolutions for the entire path walk. In current kernels, this limit is **40** (defined as `MAXSYMLINKS` in `include/linux/namei.h`). If the counter reaches 40 before the path resolves — whether from genuine loops or deeply nested chains — the kernel returns `ELOOP`.

This counter is per path resolution, not per symlink. Resolving a path with 5 symlinks in the chain costs 5 increments. A symlink loop costs 40 increments (because the kernel tries 40 times before giving up).

Note that the kernel does not detect loops by tracking visited inodes. It detects them by counting. After 40 symlink resolutions, it gives up and returns `ELOOP` regardless of whether there's actually a loop. This means a legitimately deep chain of 41 symlinks also fails, even with no loop present.

```bash
# strace reveals the ELOOP error at the system call level
strace ls /path/to/looping/symlink 2>&1 | grep ELOOP
```

```text
lstat("/path/to/looping/symlink", 0x...) = -1 ELOOP (Too many levels of symbolic links)
```

## Basic Example

```bash
# Create one of each type and observe the first character in ls -l
mkdir /tmp/file-types-demo
cd /tmp/file-types-demo

# Regular file
echo "hello" > regular.txt

# Directory
mkdir mydir

# Symbolic link (pointing to regular.txt)
ln -s regular.txt mylink

# Named pipe
mkfifo mypipe

# View them all
ls -la /tmp/file-types-demo
```

```text
total 24
drwxrwxr-x  3 alice alice 4096 Jun 10 14:00 .
drwxrwxrwt 22 root  root  4096 Jun 10 14:00 ..
-rw-rw-r--  1 alice alice    6 Jun 10 14:00 regular.txt
drwxrwxr-x  2 alice alice 4096 Jun 10 14:00 mydir
lrwxrwxrwx  1 alice alice   11 Jun 10 14:00 mylink -> regular.txt
prw-rw-r--  1 alice alice    0 Jun 10 14:00 mypipe
```

Look at the first character on each line. `d` for the directories, `-` for the regular file, `l` for the symlink, `p` for the named pipe. The symlink line also shows `-> regular.txt` — `ls` appends the target path automatically.

Notice the symlink's size: `11`. That's the byte count of the string `regular.txt`, which is exactly what's stored in the symlink's data area.

Device files and sockets won't appear here without root privileges or a running process. Clean up when done:

```bash
rm -rf /tmp/file-types-demo
```

Now try this yourself. If you run `ls -la /dev | head -20`, you'll see block devices (`b`) and character devices (`c`) alongside their major and minor numbers where the size would normally be. Try it and look for `/dev/null` — it should show `c` with `1, 3` as its major and minor numbers.

## Real-World Example

You're debugging why a Python application can't find its database — the connection string points to a socket path, and the app is getting "no such file or directory." Here's how you'd diagnose it.

```bash
# Check what type the path is (or isn't)
ls -la /var/run/postgresql/
```

```text
total 12
drwxrwsr-x  2 postgres postgres  80 Jun 10 09:14 .
drwxr-xr-x 50 root     root     1600 Jun 10 09:14 ..
srwxrwxrwx  1 postgres postgres    0 Jun 10 09:14 .s.PGSQL.5432
-rw-------  1 postgres postgres   54 Jun 10 09:14 .s.PGSQL.5432.lock
```

```bash
# Confirm the socket file type
file /var/run/postgresql/.s.PGSQL.5432
```

```text
/var/run/postgresql/.s.PGSQL.5432: socket
```

```bash
# Check whether PostgreSQL is actually running
# (a socket file leftover from a crashed process won't work)
ss -lx | grep PGSQL
```

```text
Listening  0  5  /var/run/postgresql/.s.PGSQL.5432  users:(("postgres",pid=1234,fd=5))
```

Now check whether the application is looking at the right path, and whether `cert.pem` for the TLS connection is a symlink or an actual file:

```bash
# ls -l gives you type at a glance
ls -la /etc/ssl/certs/ca-certificates.crt
```

```text
lrwxrwxrwx 1 root root 37 Jan 15 09:00 /etc/ssl/certs/ca-certificates.crt -> /etc/ssl/certs/ca-bundle.crt
```

```bash
# Trace the full chain — useful when symlinks point to symlinks
readlink -f /etc/ssl/certs/ca-certificates.crt
```

```text
/etc/ssl/certs/ca-bundle.crt
```

The walkthrough: `ls -la` is the fastest first check — the type character tells you immediately whether the path is what you expect. `file` confirms it independently if the type looks suspicious. `ss -lx` verifies that a socket file is actually backed by a listening process — a leftover socket from a crashed service looks identical in `ls` output but will refuse connections. `readlink -f` resolves the entire symlink chain, which is essential when packages create symlinks to symlinks (a pattern common in SSL certificate management and alternatives systems).

## ⚠️ Gotchas

### "Folder" Is Not a Linux Concept — and the Distinction Matters

A directory is not a folder. "Folder" is a GUI metaphor — it implies a container you open to see things inside. A Linux directory is a *file* that contains name-to-inode mappings. This distinction matters the moment you start interacting with directories as files.

```bash
# A directory has permissions, an owner, a size, a modification time
ls -ld /etc
```

```text
drwxr-xr-x 112 root root 4096 Jun 12 00:01 /etc
```

```bash
# You can stat it like any file
stat /etc
```

```text
  File: /etc
  Size: 4096      	Blocks: 8          IO Block: 4096   directory
Device: 8,1	Inode: 655361      Links: 112
Access: (0755/drwxr-xr-x)  Uid: (    0/    root)   Gid: (    0/    root)
```

The size `4096` is the size of the directory file on disk — the table of name-to-inode mappings. It is not the sum of the files inside it. Permission `755` means owner can read/write/execute the directory (execute on a directory means traverse — you need it to `cd` in), and everyone else can read and traverse but not modify the contents. You can read the table without being able to add to it.

When you understand a directory as a file, operations like `chmod 000 /some/dir` become intuitive rather than mysterious. You're removing read/write/execute access to the directory file itself — nobody can list it, traverse it, or modify it.

### Dangling Symlinks Exist — and ls Shows Them

A dangling symlink is a symlink whose target path no longer exists. The symlink itself is intact. The path it stores is still there. The object it points to is gone.

```bash
# Create a symlink, then delete the target
ln -s /tmp/target-that-will-vanish /tmp/mylink
# (don't create the target — it never existed)

# ls -l shows it anyway
ls -la /tmp/mylink
```

```text
lrwxrwxrwx 1 alice alice 26 Jun 10 14:05 /tmp/mylink -> /tmp/target-that-will-vanish
```

```bash
# But trying to use it fails
cat /tmp/mylink
```

```text
cat: /tmp/mylink: No such file or directory
```

The error says "no such file or directory" about `mylink` — even though `ls` just showed it. The confusion: the error refers to the *target* (`/tmp/target-that-will-vanish`), not the symlink itself. The symlink exists. The target doesn't.

On a terminal with color output, dangling symlinks appear in red or blinking. `ls -l` always shows them; `stat` on the symlink itself (with `-L` to not follow it) succeeds; `stat` following the link fails.

To check programmatically whether a symlink is dangling:

```bash
# test -L checks that the symlink exists (does not follow it)
# test -e checks that the target exists (follows it)
[ -L /tmp/mylink ] && ! [ -e /tmp/mylink ] && echo "dangling"
```

```text
dangling
```

### Hard Links Cannot Cross Filesystem Boundaries

This error is not a permission problem. It's a fundamental constraint of how hard links work.

```bash
# Attempting to hard link across two different filesystems
ln /mnt/external-disk/data.csv /home/alice/data.csv
```

```text
ln: failed to create hard link '/home/alice/data.csv' => '/mnt/external-disk/data.csv': Invalid cross-device link
```

An inode number is only meaningful within its own filesystem. A hard link stores an inode number in a directory entry. That number has no meaning on a different filesystem — there might be a completely different file with the same inode number there. The kernel returns `EXDEV` (cross-device link), which the shell's `ln` reports as "Invalid cross-device link."

If you need to reference the same data from two different filesystems, use a symlink — it stores a path, not an inode number, so it works across filesystem boundaries.

```bash
# Symlinks work across filesystems
ln -s /mnt/external-disk/data.csv /home/alice/data.csv
```

### Hard Links to Directories Are Refused

```bash
# Attempting to hard link a directory
ln /home/alice /home/alice-hardlink
```

```text
ln: /home/alice: hard link not allowed for directory
```

The kernel refuses this explicitly. If directory hard links were permitted, a subdirectory could hard link back to one of its ancestors, creating a cycle. The filesystem tree would become a graph with loops. Tools like `find`, `du`, and backup utilities traverse the tree recursively — they would loop forever. The `..` entry in every directory would become ambiguous (which parent?). The entire concept of a tree-structured filesystem depends on directories having at most one parent.

The only directory "hard links" that exist are `.` (every directory links to itself) and `..` (every directory links to its parent). The kernel creates and manages these. You cannot create them manually.

### The ELOOP Error: Symlink Loops and Deep Chains

```bash
# Create a symlink loop
ln -s /tmp/link-b /tmp/link-a
ln -s /tmp/link-a /tmp/link-b

# Try to follow it
ls /tmp/link-a
```

```text
ls: cannot access '/tmp/link-a': Too many levels of symbolic links
```

```bash
# The same error from the system call level
stat /tmp/link-a
```

```text
stat: cannot statx '/tmp/link-a': Too many levels of symbolic links
```

The kernel hit the 40-resolution limit (`MAXSYMLINKS`) and returned `ELOOP`. It did not detect the loop directly — it just stopped trying after 40 attempts.

The same error occurs for chains that are legitimately deep but not looping. A chain of 41 symlinks, each pointing to the next, will also fail with `ELOOP` even though there is no cycle. If you ever build a system that chains symlinks programmatically, keep the chain under 40.

To inspect a symlink without following it (and without triggering `ELOOP`):

```bash
# readlink reads the stored path without following it
readlink /tmp/link-a
```

```text
/tmp/link-b
```

```bash
# ls -l also shows symlink targets without following them
ls -la /tmp/link-a /tmp/link-b
```

```text
lrwxrwxrwx 1 alice alice 10 Jun 10 14:10 /tmp/link-a -> /tmp/link-b
lrwxrwxrwx 1 alice alice 10 Jun 10 14:10 /tmp/link-b -> /tmp/link-a
```

### File Extensions Mean Nothing to the Kernel

```bash
# Rename a shell script to .jpg
cp /usr/bin/echo /tmp/echo.jpg
chmod +x /tmp/echo.jpg

# The kernel still executes it as a binary
/tmp/echo.jpg "I am an executable"
```

```text
I am an executable
```

```bash
# The file command reads magic bytes to determine real type
file /tmp/echo.jpg
```

```text
/tmp/echo.jpg: ELF 64-bit LSB pie executable, x86-64, version 1 (SYSV), ...
```

Linux ignores the extension entirely. The kernel looks at the first few bytes when deciding how to execute something (ELF magic bytes: `7f 45 4c 46`; shebang line: `#!`). The extension is metadata for humans. `file /path` is the correct tool to determine what a file actually is.

## Distro Notes

The seven file types and their `ls -l` characters are universal — POSIX mandates them, and every Linux distribution, BSD, and macOS uses the same encoding. There are no distro differences for this topic.

The `mkfifo` command (to create named pipes) and `mknod` command (to create device nodes) are part of GNU coreutils and present on every distribution.

The `file` command is installed by default on virtually every distribution but may be absent in minimal containers:

| Distro Family | Install Command |
|---|---|
| Debian / Ubuntu | `sudo apt install file` |
| RHEL / Fedora / CentOS / Rocky | `sudo dnf install file` |
| Arch | `sudo pacman -S file` |
| Alpine (containers) | `apk add file` |

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| Symlink (`ln -s`) | Cross-filesystem references; pointing to directories; targets that may move | Target is a path; becomes dangling if target moves or is deleted |
| Hard link (`ln`) | Multiple names for the same data on the same filesystem; rename-safe references | Target is an inode; follows the data even if original name is deleted |
| Bind mount | Mounting a directory at a second location in the tree | Heavier mechanism; requires root; survives across directories on the same or different filesystems |

For most use cases where you want a file to appear in two places, a symlink is the right choice — it's simple, cross-filesystem capable, and clearly visible in `ls -l`. Hard links are useful when you explicitly want the data to survive deletion of the "original" name, or when you're building tools that depend on link count (like certain backup systems that use hardlink trees to represent snapshots efficiently).

## When to Use / When to Avoid

✅ **Use symlinks when:**
- You need a canonical path to live in one place but be accessible from another (`/usr/local/bin/python3 -> /usr/bin/python3.12`)
- The target might change, and you want to update the symlink rather than every reference to it
- The target is on a different filesystem

✅ **Use hard links when:**
- You need a file to be accessible at two different names on the same filesystem
- You want deletion of one name to leave the data accessible via the other
- You're building snapshot tools that use hard links to represent unchanged files efficiently (like `rsync --link-dest`)

✅ **Use named pipes when:**
- Two processes on the same machine need to exchange a stream of data without using a file as intermediary
- The processes may start at different times (one can wait for the other to open the pipe)

❌ **Avoid hard links when:**
- The files are on different filesystems — use a symlink instead
- You're working with directories — it's not allowed

❌ **Avoid symlinks when:**
- The target path is volatile and you need the reference to survive target deletion — use a hard link instead
- You're doing something where the symlink transparency causes confusion (e.g., `tar` following symlinks and duplicating data)

⚠️ **Common mistakes:**
- Checking `test -e /path/to/symlink` to see if something exists — this follows the symlink and returns false for dangling symlinks even though the symlink file itself exists; use `test -L` to check the symlink itself
- Using `cp` without `-L` or `-H` flags when you want to copy the target's content rather than reproduce the symlink
- Counting on file extensions to identify file types in scripts — always use `file` or check the inode type bits

## Veteran Perspective

The mistake that trips up the most beginners is confusing "the file doesn't exist" with "the thing the file points to doesn't exist." When you see "No such file or directory" on a path you can see with `ls -l`, the first question is: is this a symlink? If the type character is `l`, you have a dangling symlink. The file exists. Its target doesn't. The error message is technically accurate — it's reporting on the resolution result, not the symlink — but it sends people down the wrong diagnostic path for ten minutes.

The `file` command is underused. I've watched engineers spend thirty minutes trying to figure out why a config file isn't being parsed, when the real answer was that `file` would have immediately revealed that the "config file" was a binary accidentally placed in the wrong directory. Two seconds. Learn the reflex: when something behaves wrong and you don't know what type it is, run `file` on it.

The hard link link-count trick is genuinely useful. Every directory has a link count that equals `2 + number of immediate subdirectories`. The `2` comes from `.` (the directory links to itself) and the parent's entry for this directory. If you see a directory with link count `7`, it has 5 subdirectories. `du -s` and `find` use this to optimize traversal. You don't need to know this to use the system, but the first time you look at backup systems that use hardlink trees to store snapshots (like `rsnapshot` or `Time Machine`-style tools), it will click immediately.

The most expensive misconception I've seen in production: treating symlinks as if they're copies. A system had a `/etc/ssl/certs` directory full of symlinks to a shared certificate store. An automated deployment script ran `chmod -R 644 /etc/ssl/certs` — and because it followed the symlinks, it changed permissions on the actual certificate files in the shared store, which cascaded to break TLS for every application reading from that store. `chmod -h` changes permissions on the symlink itself (which is mostly cosmetic — the kernel uses target permissions). `chmod` without `-h` follows the symlink. Know which one you're running before touching SSL cert directories.

Sockets are invisible to tools that don't know about them. `cat /run/mysqld/mysqld.sock` fails — you cannot read from a socket with normal file I/O. You connect to a socket with `connect()`. If a service is supposed to be listening on a socket path and the socket file exists but the service won't respond, your first check is: is the service actually running? The socket file from a crashed process looks identical to one from a healthy process. `ss -lx` shows which socket files are backed by live processes.

## Try It Yourself

1. Run `ls -la /dev | head -30`. Find one block device (`b`) and one character device (`c`). Note the two numbers in the size column — that's the major and minor device number. Look up `/dev/null` specifically.

2. Run `ls -la /run` (or `/var/run`). Find at least two socket files (`s`). Check whether each socket has a live process behind it using `ss -lx`.

3. Create a symlink to a file that doesn't exist yet: `ln -s /tmp/does-not-exist /tmp/mylink`. Run `ls -la /tmp/mylink`. Run `cat /tmp/mylink`. Now create the target: `touch /tmp/does-not-exist`. Run `cat /tmp/mylink` again. Watch the behavior change without touching the symlink itself. Clean up with `rm /tmp/mylink /tmp/does-not-exist`.

4. Create two hard links to the same file and observe the link count. Run `echo "shared data" > /tmp/original`. Then `ln /tmp/original /tmp/hardlink`. Run `ls -li /tmp/original /tmp/hardlink` — the `-i` flag shows inode numbers. Confirm both entries share the same inode number and that the link count shows `2`. Delete `original` and confirm `hardlink` still contains the data.

5. **Challenge:** Use `find /etc -maxdepth 3 -type l` to find all symlinks within three levels of `/etc`. For each result, run `readlink` on it to see what it points to. Find at least one that forms a chain (a symlink pointing to another symlink). Use `readlink -f` to resolve the full chain to its final target. (Safe to run as a regular user; no writes required.)

## Summary

- The first character in `ls -l` output encodes the file type: `-` regular, `d` directory, `l` symlink, `b` block device, `c` character device, `s` socket, `p` named pipe
- A directory is a file containing name-to-inode mappings — not a "folder"; it has permissions, ownership, and size like any other file
- Symlinks store a target path, not an inode number — they can span filesystems and point to directories, but become dangling if the target is deleted or moved
- Hard links are multiple directory entries pointing to the same inode — they cannot cross filesystem boundaries and cannot link directories, because either would break the filesystem tree invariants
- The kernel limits symlink resolution to 40 steps per path walk; exceeding this limit returns `ELOOP`, whether from a genuine loop or a legitimately long chain
