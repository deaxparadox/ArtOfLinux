## Why This Exists

Before Unix, every operating system had a separate interface for every kind of resource. Want to read from a tape drive? Use the tape API. Want to send data to a printer? Use the printer API. Want to communicate between two processes? Use the IPC API. Each API had its own function names, its own error codes, its own quirks. Writing a program that combined a network connection with a log file with a terminal meant learning three completely different interfaces and keeping them all straight.

The engineers at Bell Labs who designed Unix asked a different question: what do all these things actually have in common? Every one of them is a source of bytes or a sink of bytes. You read from them, you write to them, you open them, you close them. The operations are always the same. Only the thing on the other end changes.

So they made a decision: expose everything through the same interface. Reading from a file, reading from a keyboard, reading from a hardware sensor, reading from another process — all of it would be `open()`, `read()`, `write()`, `close()`. The abstraction that holds all of this together is the **file descriptor**.

This decision has consequences that the original designers didn't fully anticipate. It means that programs don't need to know what they're talking to. A program that writes to stdout doesn't know or care whether stdout is a terminal, a file, a network socket, or a pipe to another program. The shell wires them up. The program just writes. This is why Unix tools compose so naturally: they're all speaking the same language.

## Mental Model

Think of a file descriptor like a telephone handset.

The handset doesn't care whether the voice on the other end is your colleague sitting next door, someone calling from across the country, or a recorded message. From your end, you pick it up, you listen, you speak, you hang up. The handset's interface doesn't change. What changes is what the switchboard connects it to.

In Unix, every open resource gets a handset — a file descriptor, which is just a small integer. Your program uses that number. The kernel operates the switchboard, routing your `read()` and `write()` calls to whatever is actually behind the connection: a file on disk, a keyboard, a USB device, another process. Your program never changes. The switchboard decides where the call goes.

## Theory

### The File Descriptor as Universal Interface

Every process on a Linux system has a table of open file descriptors. Each entry in that table is a small non-negative integer — 0, 1, 2, 3, and so on — that the process uses as a handle to an open resource. When a process calls `open()`, the kernel returns the next available integer in that table.

Three file descriptors exist by convention when any process starts:

- **0** — standard input (stdin)
- **1** — standard output (stdout)
- **2** — standard error (stderr)

Every other file descriptor is opened explicitly. They're just numbers. The number itself carries no information about what it represents — that information is in the kernel.

### The Special Files in /dev

The abstraction works because the kernel implements device files — special files in `/dev` that look like regular files to programs but map to kernel drivers behind the scenes.

**`/dev/null`** is the bit bucket. Every byte you write to it disappears. Every byte you read from it is EOF. Nothing is stored. Nothing is buffered. It's useful precisely because it has no behavior.

**`/dev/zero`** produces an endless stream of null bytes (byte value `0x00`). Reads never return EOF — the kernel generates the bytes on demand without any backing storage. It's used to zero-fill memory regions or create blank disk images of a precise size.

**`/dev/urandom`** produces cryptographic-quality random bytes. Every read returns a different sequence of bytes that is computationally indistinguishable from true randomness. It is safe to use for generating cryptographic keys, passwords, tokens, and salts.

**`/dev/tty`** is the controlling terminal of the current process — whichever terminal the process is attached to. It's how programs that need to interact with a human directly do so, even when their stdout has been redirected elsewhere. The `sudo` password prompt uses `/dev/tty` precisely because stdout may be a pipe.

### Block Device Files vs. Character Device Files

Device files come in two varieties, distinguished by how data moves through them.

**Character devices** transfer data byte by byte with no internal buffering at the block level. `/dev/null`, `/dev/tty`, `/dev/urandom` are all character devices. The driver handles each read and write directly. These show up in `ls -l` output with a `c` as the first character of the permission string.

**Block devices** transfer data in fixed-size blocks, and the kernel maintains a buffer cache for them. `/dev/sda` (a SATA disk), `/dev/nvme0n1` (an NVMe SSD), and `/dev/mmcblk0` (an SD card) are block devices. The block device interface is what filesystems are mounted on. These show up with a `b` as the first character.

Both types are identified by two numbers: a **major number** that identifies the driver, and a **minor number** that identifies the specific device instance within that driver. Every `/dev/sda` on every Linux system has the same major number (8 for SCSI/SATA disks), but `/dev/sda` has minor 0, `/dev/sdb` has minor 16, and so on.

### Unix Domain Sockets

A Unix domain socket is a bidirectional communication channel between processes — like a TCP socket, but it lives in the filesystem instead of on a network. You can see it, you can set permissions on it, and you can connect to it using the same `connect()` and `accept()` syscalls you'd use for a network socket.

They appear in `ls -l` output with `s` as the first character. Common examples:

- `/var/run/docker.sock` — the Docker daemon listens here; the `docker` CLI connects to it to send commands
- `/run/systemd/notify` — processes use this to tell systemd they've finished starting up
- `/var/run/postgresql/.s.PGSQL.5432` — PostgreSQL's default local connection socket
- `/run/user/1000/gnupg/S.gpg-agent` — GPG agent socket, per-user

Unix domain sockets are dramatically faster than loopback TCP connections because the kernel handles the data transfer entirely in memory, without the TCP/IP stack overhead. For services that only need local communication, always prefer a Unix socket over binding to `127.0.0.1`.

### Named Pipes (FIFOs)

A named pipe — created with `mkfifo` — is a pipe that persists in the filesystem. Unlike an anonymous pipe (`|` in the shell), a named pipe has a path. Two unrelated processes — ones that don't share a parent — can communicate through it.

Named pipes are strictly FIFO: first byte written is the first byte read. They appear in `ls -l` with `p` as the first character. The key behavior that distinguishes them from regular files: a read on an empty named pipe blocks until another process writes to it, and a write blocks until another process reads. There is no storage — the kernel passes the bytes directly from writer to reader.

This blocking contract makes them useful for:
- Coordinating stages of a pipeline where both ends aren't started simultaneously
- Decoupling a log producer from a log consumer across terminals
- Streaming data into commands that can't normally accept piped input

### Why This Abstraction Matters

The file abstraction is the reason Unix tools compose. When you write:

```bash
cat /etc/passwd | grep root | cut -d: -f1
```

`cat` doesn't know it's writing to `grep`. `grep` doesn't know it's reading from `cat` or writing to `cut`. Each program just reads from file descriptor 0 and writes to file descriptor 1. The shell connects the file descriptors. The programs don't change. This composability is not accidental — it is the deliberate consequence of the universal file interface.

## How It Works (Mechanics)

### Three Tables the Kernel Maintains

When you open a file, the kernel maintains three distinct data structures to track what's happening. Understanding them once makes everything else — `dup()`, `fork()`, pipes, redirection — obvious rather than mysterious.

```text
Process A                   Kernel                      Storage
┌──────────────┐      ┌──────────────────┐      ┌──────────────────┐
│ fd table     │      │ Open File Table  │      │ Inode Table      │
│              │      │ (per open() call)│      │ (per file)       │
│  fd 0  ──────┼─────▶│ offset: 0        │      │                  │
│  fd 1  ──────┼──┐  │ flags: O_RDONLY  │      │ /etc/passwd      │
│  fd 2  ──────┼─┐│  │ inode ptr ───────┼─────▶│ size, perms,     │
│  fd 3  ──────┼─┼┼─▶│ ─────────────────│      │ block pointers   │
└──────────────┘ ││  │ offset: 4096     │      │                  │
                 ││  │ flags: O_WRONLY  │      │ /var/log/app.log │
Process B        ││  │ inode ptr ───────┼──┐  │                  │
┌──────────────┐ ││  │─────────────────│  └─▶│ inode for        │
│ fd table     │ ││  │ offset: 4096     │     │ /var/log/app.log │
│              │ │└─▶│ flags: O_WRONLY  │─────┘                  │
│  fd 0  ──────┼─┘   │ inode ptr ──────┘                         │
│  fd 1  ──────┼────▶│ (same open file entry as Process A fd 2) │
└──────────────┘     └──────────────────┘
```

**The fd table** (per process) maps small integers to entries in the open file table. When you call `open()`, the kernel finds the lowest unused integer in your process's fd table and returns it. When you call `close(fd)`, that slot is freed.

**The open file table** (system-wide, one entry per `open()` call) holds the current file offset and access flags. This is why two processes opening the same file get independent read positions — each `open()` creates a new entry. But when a process `fork()`s, the child inherits copies of the parent's fd table entries pointing to the *same* open file table entries — so parent and child share the same file offset, and writes from one advance the position seen by the other.

**The inode table** (per file, not per open) holds the metadata and data block locations. Multiple open file table entries can point to the same inode — which is exactly what happens when two processes open the same file independently, or when you create a hard link.

### How open(), read(), write(), close() Work the Same

The beauty of the abstraction is that the system call interface is identical regardless of what's behind the file descriptor.

```c
/* These four calls work identically whether fd refers to: */
/* - a regular file                                        */
/* - /dev/null                                             */
/* - /dev/urandom                                          */
/* - a Unix domain socket                                  */
/* - a named pipe                                          */
/* - a network socket                                      */
int fd = open(path, flags);   /* ask the kernel to open a resource      */
read(fd, buf, count);         /* ask the kernel to fill buf from fd      */
write(fd, buf, count);        /* ask the kernel to drain buf into fd     */
close(fd);                    /* release the fd table slot               */
```

### How the Kernel Dispatches to Drivers

When the kernel opens a file in `/dev`, it reads the file's major and minor numbers from the inode. It looks up the major number in a kernel-internal table of registered drivers — a table populated when device drivers are loaded. The driver provides a set of function pointers: one for `open`, one for `read`, one for `write`, one for `ioctl`, and so on.

```text
# Conceptual flow — not a real command
user process calls read(fd, buf, 4096)
  │
  ▼
kernel looks up fd → open file table entry → inode
  │
  ▼
inode type = character device, major = 1, minor = 9
  │
  ▼
kernel looks up major 1 in driver table → mem driver (handles /dev/null, /dev/zero, /dev/urandom)
  │
  ▼
kernel calls mem_driver.read(inode, minor=9, buf, 4096)  ← /dev/urandom is minor 9
  │
  ▼
driver fills buf with CSPRNG output, returns 4096
  │
  ▼
kernel copies buf to user space, read() returns 4096
```

This dispatch layer is the Virtual File System (VFS). Every filesystem — ext4, XFS, tmpfs, procfs, sysfs — registers a set of VFS operations. The kernel calls VFS operations; VFS calls the appropriate driver. Your program never sees any of this. It just calls `read()`.

You can watch the actual system calls using `strace`:

```bash
strace -e trace=open,read,close cat /dev/urandom 2>&1 | head -20
```

```text
open("/dev/urandom", O_RDONLY)          = 3
read(3, "\x7f\x45\x4c\x46...", 131072) = 131072
write(1, "\x7f\x45\x4c\x46...", 131072) = 131072
...
```

The `open()` returns `3` (file descriptors 0, 1, 2 are already taken by stdin/stdout/stderr). The `read()` asks for 131072 bytes and gets exactly that — the driver generates them on demand.

## Basic Example

```bash
# Write to /dev/null — the output of this command disappears
echo "this goes nowhere" > /dev/null

# /dev/null always succeeds; check the exit code
echo $?
```

```text
0
```

```bash
# Read from /dev/urandom — pipe through xxd to see the raw bytes as hex
# We use 'head -c 16' to stop after 16 bytes; without a limit, this runs forever
head -c 16 /dev/urandom | xxd
```

```text
00000000: a3f2 1c8b 4d7e 99b1 0245 dc3f 88a6 12e9  ....M~...E.?....
```

```bash
# Read from /dev/zero — sixteen null bytes, shown as hex
head -c 16 /dev/zero | xxd
```

```text
00000000: 0000 0000 0000 0000 0000 0000 0000 0000  ................
```

```bash
# List the device files — notice the 'c' (character) in the first column
# and the major,minor numbers where file size normally appears
ls -l /dev/null /dev/zero /dev/urandom /dev/tty
```

```text
crw-rw-rw- 1 root root 1, 3 Jun  1 08:00 /dev/null
crw-rw-rw- 1 root root 1, 5 Jun  1 08:00 /dev/zero
crw-rw-rw- 1 root root 1, 9 Jun  1 08:00 /dev/urandom
crw-rw-rw- 1 root tty  5, 0 Jun 16 11:42 /dev/tty
```

Notice that where a regular file would show a size in bytes, device files show two numbers separated by a comma. The first is the **major number** (which driver handles this device), the second is the **minor number** (which instance of that driver). Major 1 is the `mem` driver, which handles `/dev/null` (minor 3), `/dev/zero` (minor 5), and `/dev/urandom` (minor 9).

Now try it yourself. Run `ls -l /dev/sda` (or `ls -l /dev/nvme0n1` if you're on an NVMe drive). The character in the first column will be `b` for block device, and you'll see a different major number. On most systems, SATA disks have major 8 and NVMe disks have major 259.

## Real-World Example

This example shows three patterns you'll encounter regularly in production: using `/dev/null` to silence output, using `/dev/urandom` to generate a secret, and creating a named pipe to connect two processes.

```bash
# Pattern 1: suppress stdout but capture stderr to a log
# Useful in cron jobs where you want to catch errors but not spam the cron log
some_backup_script > /dev/null 2> /var/log/backup-errors.log
```

```bash
# Pattern 2: generate a 32-byte URL-safe random token for a session secret
# base64 encodes the binary output; tr removes chars that aren't URL-safe
# head -c 32 reads exactly 32 bytes from /dev/urandom (no infinite loop risk)
head -c 32 /dev/urandom | base64 | tr -d '+/=' | head -c 32
```

```text
K7mX2nRvQpLsYcHtJwBeFdAuZgNiMoVk
```

```bash
# Pattern 3: connect a log producer to a log consumer via a named pipe
# Both processes are independent — they're not related by fork()

# Create the named pipe (this persists in the filesystem)
mkfifo /tmp/app-log-pipe

# In terminal 1: the "consumer" — reads from the pipe and timestamps each line
# This blocks until something writes to the pipe
while IFS= read -r line; do
    echo "$(date '+%Y-%m-%dT%H:%M:%S') $line"
done < /tmp/app-log-pipe > /var/log/app-timestamped.log &

# In terminal 2: the "producer" — any process writing to the pipe
echo "Application started" > /tmp/app-log-pipe
echo "Request received: GET /api/status" > /tmp/app-log-pipe

# Clean up the pipe when done
rm /tmp/app-log-pipe
```

```bash
# Pattern 4: check if a Unix domain socket exists and is reachable
# The docker daemon is unreachable without the socket
ls -la /var/run/docker.sock
```

```text
srw-rw---- 1 root docker 0 Jun 16 08:00 /var/run/docker.sock
```

The `s` in the first column is the socket type. The file has zero bytes — it's not storage, it's a rendezvous point. The permissions matter: members of the `docker` group can connect to the daemon; everyone else gets `EACCES`. This is how you control access to a service without managing a separate authentication layer — just use filesystem permissions.

The `docker` CLI talks to this socket using the same `connect()`, `read()`, `write()` calls that a TCP client would use. The only difference is the address: a file path instead of an IP and port. From the application developer's perspective, switching from a Unix socket to a TCP socket is often a one-line config change.

## ⚠️ Gotchas

### Reading /dev/urandom Without a Byte Limit Runs Forever

`/dev/urandom` never returns EOF. A bare `cat /dev/urandom` will fill your terminal with garbage bytes until you kill it. If you redirect it to a file, it will fill your disk.

```bash
# This looks like it might terminate. It won't.
cat /dev/urandom > /tmp/random-data
```

```text
(your terminal fills with garbage, disk usage climbs, Ctrl+C to stop)
```

The correct form always specifies how many bytes you want:

```bash
# Read exactly 32 bytes and stop
head -c 32 /dev/urandom > /tmp/random-data

# Or use dd with a count — here, 1 block of 512 bytes
dd if=/dev/urandom of=/tmp/random-data bs=512 count=1
```

The same applies to `/dev/zero`. Both devices generate data on demand with no termination condition.

### /dev/random vs /dev/urandom — The Old Mental Model Is Wrong

If you learned Linux before 2020, you probably learned that `/dev/random` blocks when the entropy pool is exhausted and `/dev/urandom` doesn't. You may have been told that `/dev/urandom` is "less secure" because of this. **That mental model is wrong on modern systems.**

Since Linux 5.6 (released March 2020), `/dev/random` no longer blocks except during early boot before the CSPRNG is initialized. Once the kernel's CSPRNG (ChaCha20-based since 4.8) has been seeded — which happens within seconds of boot on any system with hardware RNG support — `/dev/random` and `/dev/urandom` produce identical, cryptographically strong output. The blocking pool was removed entirely.

```bash
# Check your kernel version
uname -r
```

```text
6.12.25-generic
```

```bash
# On kernel 5.6+, these produce equivalent cryptographic-quality output
# The old advice to "prefer /dev/random for keys" is obsolete
head -c 32 /dev/random | xxd   # won't block on modern kernels
head -c 32 /dev/urandom | xxd  # same CSPRNG, same quality
```

The correct advice for any modern system: use `/dev/urandom` (or the `getrandom()` syscall) for all cryptographic purposes. `/dev/random` exists; it works; it no longer offers any additional security guarantee over `/dev/urandom` on Linux 5.6+.

### Writing to /dev/null Always Succeeds — Including Error Cases

Because `/dev/null` always returns success, redirecting both stdout and stderr to it can mask genuine errors that your scripts should be detecting.

```bash
# This looks like a "run silently" wrapper. It is. But it also hides all errors.
./my_script > /dev/null 2>&1

# If my_script fails catastrophically, you'll never know.
echo $?  # might be non-zero, but you saw nothing
```

When silencing output in scripts, redirect stdout but not stderr unless you have a specific reason, or check the exit code explicitly:

```bash
# Silent stdout, still shows errors
./my_script > /dev/null

# Or capture errors to a log file instead of discarding them
./my_script > /dev/null 2>> /var/log/my_script.log
```

### Named Pipes Deadlock If Both Ends Aren't Ready

A named pipe write blocks until a reader is attached. A named pipe read blocks until a writer is attached. If you try to open both ends from the same process without using background jobs or non-blocking I/O, you will deadlock.

```bash
# This hangs — the shell tries to open the pipe for writing, which blocks
# waiting for a reader that will never come because the shell is stuck
mkfifo /tmp/test-pipe
echo "hello" > /tmp/test-pipe   # blocks here forever
cat /tmp/test-pipe               # never reached
```

The correct approach is to ensure the reader and writer run concurrently:

```bash
mkfifo /tmp/test-pipe
cat /tmp/test-pipe &          # start the reader in the background first
echo "hello" > /tmp/test-pipe # now the write can complete
rm /tmp/test-pipe
```

### Unix Domain Sockets Are Not Files — Copying Them Does Nothing

A Unix domain socket in the filesystem is a rendezvous point, not storage. You cannot `cp` a socket to move a service. You cannot `scp` it to another host. The socket only works while the daemon that created it is running and listening.

```bash
# This will "succeed" but accomplish nothing useful
cp /var/run/docker.sock /tmp/docker-copy.sock

# The copy looks right...
ls -la /tmp/docker-copy.sock
```

```text
srw-r--r-- 1 user user 0 Jun 16 11:42 /tmp/docker-copy.sock
```

```bash
# ...but nothing is listening on it
docker -H unix:///tmp/docker-copy.sock ps
```

```text
Cannot connect to the Docker daemon at unix:///tmp/docker-copy.sock.
Is the docker daemon running?
```

If you need to proxy or forward a Unix socket, use `socat` or configure the daemon to listen on a different path.

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | `/dev/urandom` and `/dev/random` behave identically on any kernel shipped since Ubuntu 20.10 (kernel 5.8+). `udevd` manages `/dev` — device nodes are created at boot, not stored on disk. |
| RHEL / Fedora / CentOS / Rocky | Same kernel behavior. RHEL 8+ ships kernel 4.18 with backports; RHEL 9+ ships 5.14+. `/dev/urandom` is safe for all cryptographic use on any supported RHEL version. |
| Arch | Rolling release, always current kernel. `/dev/hwrng` is available on physical hardware and seeded into the entropy pool by `rngd` or the kernel's built-in hardware RNG support. |
| Alpine (containers) | Containers typically don't have `/dev/random` or `/dev/urandom` unless the container runtime mounts them. Docker and podman mount them by default. If you see `open /dev/urandom: no such file`, check your container's device mounts. |

## Alternatives & Comparisons

| Mechanism | When to Use | Notes |
|---|---|---|
| Anonymous pipe (`\|`) | Two processes with a parent-child relationship, short-lived | Created by the shell, no filesystem entry, cleaned up automatically |
| Named pipe (FIFO) | Two unrelated processes need to communicate via byte stream | Persists in filesystem, requires both ends to be opened; good for log forwarding |
| Unix domain socket | Bidirectional, high-performance local IPC; service APIs | Faster than TCP loopback; supports sending file descriptors between processes (`SCM_RIGHTS`) |
| TCP loopback (`127.0.0.1`) | When the client might be on a different host, or you need network-level tools | Higher overhead; works across containers and VMs when Unix sockets don't reach |
| Shared memory (`/dev/shm`) | Very high throughput IPC where the overhead of kernel-mediated I/O matters | Requires separate synchronization mechanism; not a file descriptor interface |

For local service communication, the right answer is almost always a Unix domain socket. It's faster than TCP loopback, the permissions model is familiar, and it integrates naturally with systemd socket activation. Named pipes are the right choice when you need to connect two programs that communicate via stdin/stdout and aren't being launched together.

## When to Use / When to Avoid

✅ **Use `/dev/null` when:**
- You want to discard a command's stdout or stderr entirely in a script
- You want to benchmark a command without I/O overhead affecting the result (`command > /dev/null`)
- You need to test that a command exits cleanly without caring about its output

✅ **Use `/dev/urandom` when:**
- Generating passwords, tokens, salts, or cryptographic keys
- Seeding a test suite with non-deterministic data
- Creating unique identifiers that don't need to be sequential

✅ **Use named pipes when:**
- Two unrelated processes need to stream data between them without creating a temporary file
- You want to decompress data on the fly and feed it into a tool that reads from stdin
- You're building a data pipeline where stages run at different times

✅ **Use Unix domain sockets when:**
- You're building a daemon that local clients will connect to
- Performance matters and all clients are on the same host
- You want to use filesystem permissions to control access

❌ **Avoid reading `/dev/urandom` without a byte limit** — it never terminates.

❌ **Avoid `/dev/random` in new code** — the old blocking behavior is gone on modern kernels; there's no benefit over `/dev/urandom` and the behavior difference across kernel versions creates maintenance confusion.

❌ **Avoid named pipes for bidirectional communication** — they are strictly one-directional. For two-way communication between processes, use Unix domain sockets.

❌ **Avoid Unix domain sockets when the client might be on a different host** — the socket only exists on the local filesystem. Use TCP with proper TLS if network access is possible.

⚠️ **Common mistakes:**
- Running `cat /dev/urandom` in a terminal to "see what random data looks like" — your terminal will receive binary data it interprets as control sequences and may behave oddly
- Assuming a service is running because its socket file exists — the socket file can outlive the process that created it
- Using `/dev/zero` to wipe a drive securely — `/dev/zero` produces predictable data; modern SSDs require vendor-specific secure erase; a single overwrite with zeros is not a security guarantee on any modern drive

## Veteran Perspective

The newcomer's mistake is thinking "everything is a file" is a metaphor. It isn't. It's a precise engineering contract. When you open `/dev/urandom`, the kernel goes through exactly the same VFS machinery as when you open `/etc/passwd`. The same system calls. The same fd table entry. The same permission check. The only difference is what the VFS dispatches to at the bottom of the stack.

The intermediate sysadmin's mistake is knowing about this but not using it. When you see code that generates a random token using a language library's random function instead of `/dev/urandom`, or code that shells out to a custom IPC mechanism when a Unix socket would do — that's a sign the developer knows about these abstractions but doesn't think of them first. Reach for these primitives early. They're already there, they're already fast, and they're already secure.

The deeper principle is composability by convention. When every resource looks like a file, tools built to work with files work with everything. `cat`, `head`, `tail`, `dd`, `tee`, `nc` — all of these work on devices, pipes, and sockets because they all use the same four syscalls. You can use standard file permission tooling to control access to a local API (`chmod 660 /var/run/myservice.sock`). You can use `strace` to see exactly what bytes are flowing through a socket. You can use `lsof` to find every process that has a given socket or pipe open. The toolchain applies everywhere.

Here's what you learn after you've debugged enough Docker incidents: the permissions on `/var/run/docker.sock` are a security boundary. If a container has that socket mounted inside it, the process inside the container can send arbitrary commands to the Docker daemon — including running new containers with `--privileged`. You've given the container root on the host. This is not theoretical. This is a documented container escape vector. "Everything is a file" means "everything inherits file-based security assumptions" — including the bad ones.

The `/dev/urandom` vs `/dev/random` debate is mostly resolved now, but if you work in security you'll still see code from 2015 that calls `open("/dev/random", ...)` and blocks waiting for entropy on a headless server. The code is correct but unnecessary — the blocking served a purpose when the CSPRNG wasn't ChaCha20 and entropy estimation was conservative. On any kernel you'll actually deploy to in 2026, that code is just a latency hazard waiting to manifest during the first boot after a VM clone.

## Try It Yourself

1. Run `ls -la /dev/null /dev/zero /dev/urandom /dev/tty` and read the output character by character. What does the leading `c` mean? What are the two numbers where a file size would normally appear?

2. Use `head -c 32 /dev/urandom | base64` to generate a random base64 string. Run it three times. Every output should be different. Now run `head -c 0 /dev/urandom | base64` — why is the output empty?

3. Run `echo "test" > /dev/null && echo "write succeeded"`. Confirm the write returns exit code 0. Now run `cat /dev/null` — what does it return, and why?

4. Create a named pipe with `mkfifo /tmp/mypipe`. In one terminal, run `cat /tmp/mypipe`. In another terminal, run `echo "hello from terminal 2" > /tmp/mypipe`. Watch what happens in the first terminal. Then run `rm /tmp/mypipe` to clean up.

5. **Challenge:** Write a script that uses `/dev/urandom` to generate a random 16-character password containing only alphanumeric characters, prints it, and also writes it (with a timestamp) to a named pipe `/tmp/password-log`. Have a second process read from the pipe and append each entry to `/tmp/password-history.txt`. Test that passwords are being recorded. Clean up the pipe when done. *(Safe to run on your own machine — generates no privileged operations.)*

## Summary

- Every resource in Linux — regular files, hardware devices, sockets, pipes — is accessed through the same four syscalls: `open()`, `read()`, `write()`, `close()`. The thing behind the file descriptor changes; the interface doesn't.
- `/dev/urandom` produces cryptographic-quality random bytes and never blocks on any kernel 5.6+; `/dev/random` on modern systems is identical. Always use `/dev/urandom` for secrets.
- Block devices (`b` in `ls -l`) transfer data in buffered blocks and back filesystems. Character devices (`c`) transfer byte-by-byte. Both are identified by major and minor numbers, not sizes.
- Unix domain sockets (`s` in `ls -l`) are faster than TCP loopback for local IPC, accept filesystem permissions for access control, and are the standard interface for local daemons (Docker, systemd, PostgreSQL).
- Named pipes (`p` in `ls -l`) block on reads until a writer arrives and block on writes until a reader arrives — there is no storage. Always start the reader before the writer, and always clean up the pipe file afterward.
