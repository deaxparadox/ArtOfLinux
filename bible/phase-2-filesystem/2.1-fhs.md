# [2.1] — Filesystem Hierarchy Standard (FHS)

## Why This Exists

In the early days of Unix, every vendor organized their filesystem differently. `/bin` might be on one machine but `/usr/bin` on another. A shell script that worked on SunOS broke on HP-UX. A sysadmin who knew one system had to relearn the next one from scratch. Software vendors couldn't ship packages that reliably found their own dependencies. Each Unix was, in the words of one early developer, "a snowflake" — unique and incompatible.

Linux inherited the worst of this problem in its early years. By the early 1990s, different Linux distributions were putting the same files in different places. A program that ran on Slackware might refuse to start on Red Hat because the libraries it needed were under `/lib` on one and `/usr/lib` on another. Packaging systems — the early ancestors of `apt` and `rpm` — had no way to write install scripts that worked everywhere.

The Filesystem Hierarchy Standard was the industry's answer. First formalized in 1994 (as the Linux Filesystem Standard) and eventually adopted as the FHS under the Linux Foundation, it defined a common contract: if you are a Linux system, `/etc` contains configuration, `/bin` contains essential binaries, `/var/log` contains logs, and so on. Software could be written against the standard instead of against each individual distribution. A sysadmin who knew the FHS could find their way around any compliant system.

The current version is **FHS 3.0**, originally released in June 2015 by the Linux Foundation. In late 2025, maintenance was transferred to FreeDesktop, which republished and assumed stewardship of the standard. No version 4.0 exists yet, though a slow-moving effort to revise it has been underway.

The FHS is not enforced by the kernel. The kernel doesn't care whether your logs are in `/var/log` or `/home/logs` or `/chaos`. The FHS is a convention — but a convention so widely followed that scripts, packages, and administrators treat it as law.

## Mental Model

Think of the Linux filesystem as a large, well-organized government building.

Each floor and wing has a defined purpose: the archive room holds old records, the operations center is where current work happens, the mechanical room is for infrastructure the public never touches, and the mailroom handles things that come and go. Nobody decided these rooms were laid out this way arbitrarily — over decades of experience, people learned what goes where so that any employee, on their first day, can find what they need without a guided tour.

The FHS is the building's floor plan. The kernel is the building itself. Your programs are the employees. When a program needs to find a config file, it goes to the fourth floor, wing C, room 47 — which every employee knows as `/etc`. When it needs a shared library, it goes to the supply room — `/usr/lib`. The layout is the same in every government building of this type, in every city, on every continent. That's the whole point.

## Theory

### One tree, no exceptions

Linux uses a single-rooted filesystem tree. Everything — every file, every directory, every device, every running kernel interface — hangs off `/`. There are no drive letters (no `C:\`, no `D:\`). If you have a second hard drive, it doesn't appear as a separate letter; it gets *mounted* at some point in the tree, and from that point forward its contents appear as a directory under `/`. This is one of the most disorienting things for people coming from Windows, and one of the most elegant design decisions in Unix.

The root is always `/`. The FHS defines what lives directly under root and why.

### The primary subdivision: essential vs. non-essential

The most important conceptual split in the FHS is between *essential* and *non-essential* binaries. This distinction dates back to the early days of Unix, when hard drives were small and expensive, and `/usr` might be on a separate, possibly network-mounted filesystem.

- **Essential** binaries — the tools you need to boot the system and repair it even if `/usr` isn't mounted yet — live in `/bin` and `/sbin`.
- **Non-essential** binaries — everything else — live in `/usr/bin` and `/usr/sbin`.

This split made sense when it was designed. Whether it still makes sense is a different conversation (see Gotchas). But understanding *why* the split exists is essential to understanding why `/bin` and `/usr/bin` are both there.

### The directory taxonomy

The FHS organizes directories along two axes:

| | Shareable | Non-shareable |
|---|---|---|
| **Static** (doesn't change without admin action) | `/usr`, `/opt` | `/etc`, `/boot` |
| **Variable** (changes during normal operation) | `/var/mail`, `/var/spool` | `/var/run`, `/var/lock` |

*Shareable* means the directory can be NFS-mounted and shared across multiple machines. *Static* means it doesn't need write access during normal operation. This explains, for example, why `/etc` is not shareable — each machine's configuration is unique. And why `/usr` can be mounted read-only across a network — it's static and shareable.

### What each directory is for

**`/` — the root**
The apex of the entire tree. Every path starts here. It must be available before anything else can run. Only the files and directories absolutely required to boot and recover the system belong here. Everything else belongs under `/usr` or `/var`.

**`/bin` and `/sbin` — essential binaries**
`/bin` contains essential user commands: `ls`, `cp`, `mv`, `rm`, `cat`, `bash`, `sh`. These must be present to boot the system and repair a broken installation. `/sbin` contains essential system binaries: `mount`, `fsck`, `init`, `ip`. The traditional distinction between `/bin` and `/sbin` is that `/bin` is for all users and `/sbin` is for the superuser (root). On modern distributions, both are now typically symlinks to `/usr/bin` and `/usr/sbin` — see the Gotchas section.

**`/usr` — the secondary hierarchy**
The largest directory on most Linux systems. It contains the full set of installed user utilities and applications. Think of `/usr` as a read-only, shareable, static software layer. The key subdirectories:

- `/usr/bin` — the majority of user executables on the system
- `/usr/sbin` — system administration commands that don't belong in `/sbin`
- `/usr/lib` — shared libraries for programs in `/usr/bin` and `/usr/sbin`
- `/usr/lib64` — 64-bit shared libraries (on 64-bit systems)
- `/usr/include` — C/C++ header files for development
- `/usr/share` — architecture-independent data: man pages, locale files, icons, documentation
- `/usr/local` — software installed locally by the system administrator, outside the package manager; its own mini-hierarchy (`/usr/local/bin`, `/usr/local/lib`, etc.)

**`/etc` — system configuration**
All system-wide configuration files live here. No binaries. No data. Configuration only. If you want to change how a program behaves system-wide, the answer is almost always somewhere in `/etc`. Examples: `/etc/passwd`, `/etc/fstab`, `/etc/nginx/nginx.conf`, `/etc/ssh/sshd_config`. User-specific configuration overrides live in the user's home directory (`~/.config`, `~/.bashrc`, etc.), not in `/etc`.

**`/var` — variable data**
Data that changes during normal system operation and must persist across reboots: logs, mail spools, caches, package databases, print queues. The key subdirectory:

- `/var/log` — system and service logs: `/var/log/syslog`, `/var/log/auth.log`, `/var/log/nginx/`
- `/var/lib` — persistent state data for applications: package databases (`/var/lib/dpkg/`, `/var/lib/rpm/`), databases, service state
- `/var/spool` — data waiting to be processed: print jobs, mail queues, cron jobs
- `/var/cache` — application cache data; safe to delete (though software may regenerate it slowly)
- `/var/tmp` — temporary files that must survive reboots (unlike `/tmp`)

**`/run` — runtime data**
A tmpfs filesystem (lives entirely in RAM) that holds data a running system needs from boot until shutdown: PID files, Unix domain sockets, lock files. It is cleared on every reboot because it doesn't live on disk. This is where systemd writes its socket files, where the SSH daemon writes its PID, where NetworkManager stores its current state. `/var/run` is now a symlink to `/run` on all modern distributions; the canonical path is `/run`.

**`/tmp` — temporary files**
Scratch space. Any process can create files here. On most modern distributions, `/tmp` is also a tmpfs, meaning it lives in RAM and is cleared on reboot. On older or minimal systems, `/tmp` may be on disk, in which case the kernel does not clear it on reboot. Applications that need scratch space that survives reboots should use `/var/tmp`, not `/tmp`.

**`/home` — user home directories**
Every regular user gets a subdirectory here: `/home/alice`, `/home/bob`. This is where user-specific configuration files, documents, and data live. The system does not own this space — the users do. In enterprise environments, `/home` is often a separate partition or an NFS mount. The FHS requires this to be optional (a machine can be configured without user accounts), but it's present on virtually every desktop and interactive server.

**`/root` — root's home directory**
The superuser's home directory. It is deliberately *not* `/home/root`. The reason: `/home` might be a separate partition that isn't mounted early in the boot process. If something breaks during boot and you need to log in as root to fix it, root's home directory must be available on the root filesystem, not on a potentially-unmounted partition. `/root` is always on `/`.

**`/dev` — device files**
Not a real directory in the traditional sense. On modern Linux, `/dev` is managed by `udev` and is populated dynamically as hardware is detected. Every device on the system has a file here: `/dev/sda` (first SATA disk), `/dev/tty1` (first virtual terminal), `/dev/null` (the null device), `/dev/random` (random number generator), `/dev/zero` (endless zeros). Reading and writing these files talks directly to devices. This is "everything is a file" in its most concrete form — covered in depth in **2.2 — Everything Is a File**.

**`/proc` — process and kernel information**
A virtual filesystem — nothing in `/proc` exists on disk. The kernel generates its contents on-the-fly whenever you read it. `/proc/1/` contains information about PID 1 (systemd or init). `/proc/cpuinfo` describes your CPU. `/proc/meminfo` describes memory. `/proc/sys/` exposes tunable kernel parameters that can be written to change kernel behavior at runtime. Reading `/proc/cpuinfo` involves no disk I/O — the kernel generates that data in memory in response to your read call.

**`/sys` — kernel device and driver information (sysfs)**
Another virtual filesystem, introduced in kernel 2.6 to replace the parts of `/proc` that weren't process-related. `/sys` exposes the kernel's view of all hardware, drivers, and kernel subsystems in a structured hierarchy. `/sys/class/net/` shows network interfaces. `/sys/block/` shows block devices. `/sys/bus/` shows hardware buses. Like `/proc`, nothing in `/sys` lives on disk.

**`/boot` — bootloader and kernel**
The files required to boot: the kernel image itself (`vmlinuz`), the initial RAM filesystem (`initrd.img` or `initramfs.img`), and the GRUB configuration (`/boot/grub/grub.cfg`). This directory is often a separate partition formatted as FAT32 or ext2/ext4 — bootloaders like GRUB need to read these files before the main filesystem is even mounted.

**`/lib` and `/lib64` — essential shared libraries**
Shared libraries required by the binaries in `/bin` and `/sbin` — the C library (`libc.so`), the dynamic linker (`ld-linux.so`), kernel modules. Like `/bin`, these are now typically symlinks to `/usr/lib` and `/usr/lib64` on modern distributions.

**`/mnt` — temporary mount points**
A convention, not a requirement. Sysadmins use `/mnt` as a scratch mount point when manually mounting a filesystem: `mount /dev/sdb1 /mnt`. There's nothing special about `/mnt` — it's just a well-known, agreed-upon scratch pad so that temporary mounts don't create clutter in `/`.

**`/media` — automounted removable media**
USB drives, optical discs, and SD cards that are automatically mounted by the desktop environment or `udisks` appear here: `/media/username/USB_DRIVE_NAME`. This is a desktop-oriented convention; servers typically use `/mnt` instead.

**`/opt` — optional third-party software**
Software that installs itself as a self-contained package outside the normal package manager lives here. Google Chrome installs to `/opt/google/chrome/`. Oracle installs to `/opt/oracle/`. The convention is that each package gets its own subdirectory under `/opt` and manages its own layout within it, completely isolated from the standard FHS tree.

**`/srv` — service data**
Data served by the system to the outside world. Web server document roots, FTP roots, version control repositories. The FHS defines it, but in practice it's rarely used consistently. Many distributions and admins use `/var/www` for web roots instead, which is technically a misuse of `/var` but common enough to be the de facto standard.

**`/lost+found` — filesystem recovery**
Present on ext2/ext3/ext4 filesystems only. When `fsck` (filesystem check) runs after an unclean shutdown and finds blocks that belong to a file but the file's directory entry is gone, it places those blocks here as numbered files. There is one `/lost+found` directory per filesystem — it lives at the root of each filesystem, not just at `/lost+found`.

## How It Works (Mechanics)

### The virtual filesystems: /proc and /sys

The most important mechanics to understand here are `/proc` and `/sys`, because they look like directories but behave like nothing else on the system.

When the kernel boots, it registers two pseudo-filesystems: `procfs` (mounted at `/proc`) and `sysfs` (mounted at `/sys`). These are not backed by a block device. They have no entries in `/etc/fstab` on most systems because systemd mounts them automatically. The kernel handles all read and write operations against these paths through special virtual file operations defined in the kernel source.

When you run `cat /proc/cpuinfo`, here is what actually happens:

```text
# Conceptual flow — not a real command
cat /proc/cpuinfo
  → read() syscall on /proc/cpuinfo
  → VFS (Virtual File System layer) routes to procfs handler
  → procfs handler calls kernel function: show_cpuinfo()
  → show_cpuinfo() reads cpu_data[] from kernel memory
  → formats it as text and returns it to the read() call
  → cat prints it to stdout
  → no disk I/O occurred at any point
```

You can verify this with `strace`:

```bash
strace -e trace=read,open,openat cat /proc/cpuinfo 2>&1 | head -20
```

```text
openat(AT_FDCWD, "/proc/cpuinfo", O_RDONLY) = 3
read(3, "processor\t: 0\nvendor_id\t: Genuin"..., 131072) = 4096
read(3, "", 131072)                     = 0
...
```

No `lseek`, no block reads — just a single `openat` and `read` that the kernel answers entirely from memory.

Writing to `/proc/sys/` works the same way in reverse: `echo 1 > /proc/sys/net/ipv4/ip_forward` triggers a write handler in the kernel that changes a live kernel variable without requiring a reboot.

### How /run is mounted as tmpfs

Early in the boot sequence — before systemd has started any services — the kernel mounts a small set of essential filesystems. `/run` is one of them:

```bash
# Show all tmpfs mounts
mount | grep tmpfs
```

```text
tmpfs on /run type tmpfs (rw,nosuid,nodev,noexec,relatime,size=1638400k,mode=755)
tmpfs on /dev/shm type tmpfs (rw,nosuid,nodev)
tmpfs on /run/lock type tmpfs (rw,nosuid,nodev,noexec,relatime,size=5120k)
tmpfs on /tmp type tmpfs (rw,nosuid,nodev,relatime)
```

Because `/run` lives in RAM, a PID file written to `/run/nginx.pid` disappears when the machine reboots. This is intentional — stale PID files from a previous boot can confuse init systems. Systemd leverages this by creating service-specific directories under `/run/systemd/` for live state that should never persist.

### How the usr merge works at the filesystem level

On a merged system, `/bin` is a symlink:

```bash
ls -la / | grep -E '^l'
```

```text
lrwxrwxrwx   1 root root    7 Jan  1  2023 bin -> usr/bin
lrwxrwxrwx   1 root root    7 Jan  1  2023 lib -> usr/lib
lrwxrwxrwx   1 root root    7 Jan  1  2023 lib64 -> usr/lib64
lrwxrwxrwx   1 root root    7 Jan  1  2023 sbin -> usr/sbin
```

The kernel doesn't know or care. When a process calls `execve("/bin/bash", ...)`, the VFS resolves the symlink `/bin` → `/usr/bin`, finds `bash` at `/usr/bin/bash`, and executes it. The process sees `/bin/bash`; the kernel sees `/usr/bin/bash`. They're the same inode.

## Basic Example

The simplest way to internalize the FHS is to walk the top-level directories yourself and see what's actually in them.

```bash
# List the top-level entries under / with their types
ls -la /
```

```text
total 72
drwxr-xr-x  20 root root  4096 Jun 10 08:12 .
drwxr-xr-x  20 root root  4096 Jun 10 08:12 ..
lrwxrwxrwx   1 root root     7 Jan  1  2023 bin -> usr/bin
drwxr-xr-x   4 root root  4096 Jun 10 08:12 boot
drwxr-xr-x  20 root root  4780 Jun 15 14:22 dev
drwxr-xr-x 132 root root 12288 Jun 15 14:23 etc
drwxr-xr-x   3 root root  4096 May 20 09:01 home
lrwxrwxrwx   1 root root     7 Jan  1  2023 lib -> usr/lib
lrwxrwxrwx   1 root root     7 Jan  1  2023 lib64 -> usr/lib64
drwx------   5 root root  4096 Jun 14 22:10 root
drwxr-xr-x  37 root root   980 Jun 15 14:23 run
lrwxrwxrwx   1 root root     7 Jan  1  2023 sbin -> usr/sbin
drwxr-xr-x  14 root root  4096 Jun  1 10:00 srv
drwxr-xr-x  12 root root     0 Jun 15 14:22 sys
drwxrwxrwt  18 root root  4096 Jun 15 15:01 tmp
drwxr-xr-x  14 root root  4096 Jan  1  2023 usr
drwxr-xr-x  13 root root  4096 Jun  1 10:00 var
```

Look for the `l` at the start of the permissions column — those are symlinks. On this system (Debian 12 or Ubuntu 22.04+), `bin`, `lib`, `lib64`, and `sbin` are all symlinks pointing to their `/usr/` counterparts. That's the usr merge in effect. If your output shows `drwxr-xr-x` for `/bin` instead of `lrwxrwxrwx`, your distribution hasn't completed the merge yet.

```bash
# Confirm what /bin points to
readlink /bin
```

```text
usr/bin
```

```bash
# Verify /proc and /sys are virtual (zero disk usage)
du -sh /proc /sys
```

```text
0	/proc
0	/sys
```

That zero is not a mistake. There is no disk data there.

Now try this yourself. Run `ls -la /` on your system and look at which entries are symlinks versus real directories. If you see `/bin -> usr/bin`, you're on a usr-merged system. Then run `ls /proc` and pick one of the numbered directories — those are running processes. Pick your current shell's PID (`echo $$`) and look at `/proc/$$/cmdline`.

## Real-World Example

A junior sysadmin is diagnosing a web server that stopped serving requests. Here is how knowledge of the FHS turns a confusing situation into a systematic investigation:

```bash
# Step 1: Check if the web server process is running
# PID files in /run tell us if the service thinks it's alive
ls /run/nginx.pid 2>/dev/null && echo "PID file exists" || echo "No PID file — service may not be running"
```

```text
No PID file — service may not be running
```

```bash
# Step 2: Check the logs — /var/log is where service logs live
# nginx writes to /var/log/nginx/ by convention
tail -50 /var/log/nginx/error.log
```

```text
2024/06/15 14:45:12 [emerg] 12345#12345: bind() to 0.0.0.0:80 failed (98: Address already in use)
2024/06/15 14:45:12 [emerg] 12345#12345: still could not bind
```

```bash
# Step 3: Something is already on port 80 — find what
# /proc exposes live network state
ss -tlnp | grep ':80'
```

```text
LISTEN    0    128    0.0.0.0:80    0.0.0.0:*    users:(("apache2",pid=9871,ident=33))
```

```bash
# Step 4: Verify by checking that process's config
# /proc/<pid>/exe is a symlink to the running binary
ls -la /proc/9871/exe
```

```text
lrwxrwxrwx 1 root root 0 Jun 15 14:22 /proc/9871/exe -> /usr/sbin/apache2
```

```bash
# Step 5: Find the apache2 config that enabled port 80
# All system config lives in /etc
grep -r 'Listen 80' /etc/apache2/
```

```text
/etc/apache2/ports.conf:Listen 80
```

Five steps. Every single one relied on knowing where to look. Logs in `/var/log`, runtime state in `/run`, live process data in `/proc`, config in `/etc`, binaries in `/usr/sbin`. If you knew the FHS, this took two minutes. If you didn't, you'd be searching blindly.

The walkthrough: The missing PID file told us nginx wasn't running. The log told us why — port 80 was taken. `ss` showed us the process holding port 80. `/proc` confirmed which binary was running. `/etc` showed us its config. This is the FHS working as designed: each directory has a specific purpose, and that makes the investigation systematic instead of random.

## ⚠️ Gotchas

### Hardcoded /bin paths break on merged systems

A script written ten years ago — or by someone who learned on an older system — might start with:

```bash
#!/bin/sh
```

That's fine on any system. But you might also see:

```bash
#!/bin/bash
# or worse, embedded paths:
PYTHON=/usr/bin/python
SENDMAIL=/usr/sbin/sendmail
```

These hardcoded paths work fine on most systems because even on merged distributions, `/bin/bash` still resolves correctly via the symlink. The real problem arises in a different direction: scripts that test for the *separate* existence of `/bin` vs `/usr/bin` to determine what system they're on, or Docker containers and chroot environments built from minimal base images where the symlinks may not be present. Always use `env` in shebangs for portability:

```bash
#!/usr/bin/env bash
```

And never rely on the existence of a specific path to detect the distribution — use `/etc/os-release` instead.

### /tmp is not guaranteed to be cleared on reboot

The FHS says `/tmp` "must be made available to programs, and users may delete files in this directory." It does not guarantee clearance on reboot. On systemd-based distributions with `TMPFS` enabled for `/tmp`, the directory is RAM-backed and cleared automatically. On older or minimal systems, `/tmp` might be a real directory on disk:

```bash
# Check whether /tmp is a tmpfs (RAM-backed) or disk-backed
findmnt /tmp
```

```text
TARGET SOURCE FSTYPE OPTIONS
/tmp   tmpfs  tmpfs  rw,nosuid,nodev,relatime
```

If the `FSTYPE` column shows `tmpfs`, it's RAM-backed and will be cleared. If it shows nothing (the command returns no output) or shows your root filesystem's device and type, `/tmp` is on disk and may not be cleared on reboot. For data that genuinely must not survive a reboot, use `/run`. For temporary data that *can* survive a reboot, use `/var/tmp`. Never bet your application's correctness on `/tmp` behavior — check, or use the right directory.

### /var/run still exists but is not the canonical path

Pre-systemd documentation, older scripts, and some third-party software write PID files and sockets to `/var/run`. On modern systems:

```bash
ls -la /var/run
```

```text
lrwxrwxrwx 1 root root 4 Jan  1  2023 /var/run -> /run
```

`/var/run` is a symlink to `/run`. Writing to `/var/run/myapp.pid` and writing to `/run/myapp.pid` are the same operation. The canonical path is `/run`. Use `/run` in any new code or configuration you write. Scripts that use `/var/run` will work — but they're carrying technical debt that may eventually trip someone up.

### /proc and /sys are not real directories — don't treat them like storage

The mistake looks like this: a monitoring script that does `ls -la /proc` to check "disk usage" on the root filesystem, or a backup script that includes `/proc` in its source path.

```bash
# This will run forever or produce garbage output
tar -czf backup.tar.gz /proc
```

The content of `/proc` is generated on-demand by the kernel. There is infinite "data" there in the sense that every read generates a fresh response. A backup of `/proc` captures nothing meaningful — the data you care about (a process's open files, its memory maps) is live and changes every millisecond. Always explicitly exclude `/proc` and `/sys` from backup operations, and explicitly exclude them from `du` calculations when measuring disk usage.

```bash
# Correct: exclude virtual filesystems from du
du -sh --exclude=/proc --exclude=/sys --exclude=/dev /
```

### The /bin vs /usr/bin distinction in scripts targeting minimal environments

Container images — particularly Alpine Linux and scratch-based Docker images — do not always have the usr-merge symlinks. A script that runs fine on your Ubuntu workstation may fail in a container:

```bash
# Works on Ubuntu (has /bin -> usr/bin symlink):
/bin/python3 script.py

# Fails on Alpine (no symlink, and Python may be at /usr/bin/python3):
# /bin/python3: not found
```

The defensive pattern: always use `/usr/bin/env python3` in shebangs, and in Dockerfiles, confirm the actual binary path with `which` or `command -v` during image build rather than assuming.

### /lost+found is per-filesystem, not global

A common misunderstanding: there is exactly one `/lost+found` directory per ext2/ext3/ext4 filesystem, located at the root of that filesystem. If you mount a second disk at `/data`, its recovered files appear in `/data/lost+found`, not `/lost+found`. If `/home` is a separate partition, recovered files for home-directory corruption appear in `/home/lost+found`. Always look in the right place after running `fsck`.

## Distro Notes

### The usr-merge: which distros have merged and when

| Distro | /bin status | /sbin status | Merged since |
|---|---|---|---|
| Fedora | symlink → `/usr/bin` | symlink → `/usr/sbin` | Fedora 17 (2012) |
| Arch Linux | symlink → `/usr/bin` | symlink → `/usr/bin` | 2012 |
| openSUSE | symlink → `/usr/bin` | symlink → `/usr/sbin` | openSUSE 12.3 (2013) |
| Ubuntu | symlink → `/usr/bin` | symlink → `/usr/sbin` | Ubuntu 20.04+ (new installs) |
| Debian | symlink → `/usr/bin` | symlink → `/usr/sbin` | Debian 12 Bookworm (new installs, 2023) |
| RHEL / Rocky / AlmaLinux | symlink → `/usr/bin` | symlink → `/usr/sbin` | RHEL 7+ |
| Alpine Linux | **real directory** | **real directory** | Not merged (intentional) |
| Gentoo | Optional (`merge-usr`) | Optional | User-configurable |

**Fedora 42 (2025) went further:** it merged `/usr/sbin` *into* `/usr/bin` — `sbin` is now a symlink to `bin` within `/usr` itself. This means `/usr/sbin`, `/sbin`, and `/usr/bin` and `/bin` all resolve to the same physical directory. No other major distribution has done this yet.

Note that existing Debian/Ubuntu systems upgraded from older versions may still have real `/bin` directories. The merge was applied only to fresh installs or systems that ran the `usrmerge` package. Check with `ls -la /bin` on your specific machine.

### /tmp behavior

| Distro | /tmp is tmpfs? | Configurable? |
|---|---|---|
| Fedora | Yes (default) | `systemctl mask tmp.mount` to disable |
| Ubuntu | Yes (Ubuntu 22.04+) | `/etc/systemd/system/tmp.mount` |
| Debian | No (default on-disk) | Install `systemd-tmpfs` or enable `tmp.mount` |
| Arch | Yes | `systemd-tmpfs` active by default |
| RHEL 8/9 | Yes | `systemctl mask tmp.mount` to disable |
| Alpine | No (on-disk) | Manual; add tmpfs entry to `/etc/fstab` |

### Package manager paths

| Distro Family | Config path convention | Package database |
|---|---|---|
| Debian / Ubuntu | `/etc/default/` for daemons | `/var/lib/dpkg/` |
| RHEL / Fedora / Rocky | `/etc/sysconfig/` for daemons | `/var/lib/rpm/` |
| Arch | `/etc/conf.d/` (optional) | `/var/lib/pacman/` |
| Alpine (containers) | `/etc/conf.d/` | `/var/lib/apk/` |

## Alternatives & Comparisons

The FHS is not the only filesystem organization standard that has existed.

| Standard | Scope | Status |
|---|---|---|
| FHS 3.0 | Linux distributions | Current; maintained by FreeDesktop since late 2025 |
| LSB (Linux Standard Base) | Binary compatibility, also references FHS | Largely inactive; last meaningful update was LSB 5.0 (2015) |
| BSD filesystem layout | BSD-derived systems (FreeBSD, macOS) | Active on BSDs; differs significantly (`/usr/local` is primary, no `/opt`) |
| GoboLinux | Experimental per-application layout | Niche; each program in `/Programs/<Name>/<Version>/` |

The FHS has no real competitor on Linux. Every major distribution follows it to varying degrees. The practical question is not "FHS or something else" but "how strictly does this distro follow it" — and the answer for most modern distributions is: closely enough that the FHS map will get you oriented on any of them.

## When to Use / When to Avoid

This section is about applying FHS knowledge, not about the FHS itself.

✅ **Look in /etc when:**
- A service is behaving unexpectedly — its configuration lives there
- You need to change a system-wide setting
- You need to find what hostname, DNS server, or timezone the system uses

✅ **Look in /var/log when:**
- Something broke and you need to know why
- You're auditing what happened at a specific time
- A service died and left no other trace

✅ **Use /run (not /var/run) when:**
- Writing PID files, socket files, or lock files in new code or scripts
- The data must not survive a reboot

✅ **Use /var/tmp (not /tmp) when:**
- You need scratch space that must survive a reboot (e.g., a long-running job's intermediate results)

✅ **Use /usr/local/bin for:**
- Scripts or programs installed manually by the administrator, not by the package manager
- Keeps admin-installed software separate from packager-installed software; `/usr/local/bin` takes precedence over `/usr/bin` in the default `$PATH` on most distributions

❌ **Never put binaries in /etc:**
- `/etc` is for configuration. Programs that find a binary in `/etc` have a bug.

❌ **Never put runtime data in /tmp if it must survive a reboot:**
- Use `/var/tmp` for that.

❌ **Never include /proc or /sys in backups or disk-usage calculations:**
- They are virtual. Including them either produces garbage output or hangs the operation.

⚠️ **Common mistakes:**
- Writing to `/var/run` instead of `/run` in new code — works, but is technically wrong and adds confusion
- Putting application data in `/opt` without a self-contained layout — `/opt/myapp/` should be a full mini-hierarchy, not a single file dropped into `/opt/`
- Confusing `/srv` and `/var/www` — historically muddled; pick one convention and be consistent within a project

## Trade-offs

This section applies to the *design* of a single-rooted, FHS-organized filesystem — the costs and benefits of this layout compared to alternatives.

**Where this shines:**

- **Predictability across machines.** Any sysadmin who knows the FHS can find their way around any compliant system. This has real value in large organizations where many people administer many servers.
- **NFS sharing.** The shareable/non-shareable distinction means `/usr` can be NFS-mounted read-only across an entire datacenter of identical machines. One copy of `/usr`, thousands of machines.
- **Package management.** The dpkg/rpm/pacman model depends on knowing exactly which files belong where. Without the FHS, package scripts would have to be distro-specific at every level.
- **Separation of concerns.** Config in `/etc`, data in `/var`, binaries in `/usr/bin`. When things go wrong, you know where to look.

**Where this costs you:**

- **The essential/non-essential split is now mostly historical debt.** The reason `/bin` is separate from `/usr/bin` — so you can repair the system when `/usr` isn't mounted — stopped being practical once SSDs eliminated boot time as a concern and initramfs could mount any filesystem before the system came up. The usr-merge is the industry's acknowledgment of this debt.
- **FHS 3.0 is a decade old.** It was not designed around containers, immutable OS images, or modern application delivery (OCI images, Flatpak, Snap). Tools like Flatpak (`/var/lib/flatpak/`) and Snap (`/snap/`) extend or bypass FHS conventions entirely.
- **`/srv` is effectively undefined in practice.** The FHS defines it, but there is no enforced convention. Different distributions use it differently or not at all, making it the least reliable directory in the standard.

## Veteran Perspective

Here's the thing about the FHS that most tutorials miss: it is a *contract*, not a truth. The Linux kernel doesn't know what `/etc` is. It doesn't care. The kernel will execute a binary in `/tmp/malware` just as happily as one in `/usr/bin`. The FHS is only meaningful because every distro, every package manager, every init system, and every sysadmin agrees to honor it. The moment someone deviates, they pay a compatibility tax.

This has a practical consequence. When you're debugging a production incident at 3am and something is broken, the FHS is the map that lets you move fast. You don't think — you go to `/var/log` for logs, `/etc` for config, `/run` for runtime state. If a developer decided to put their application's logs in `/opt/myapp/logs/` or their config in `/home/appuser/` (and I've seen both), that map breaks. You stop moving fast. You start searching. And at 3am, searching costs you.

The deeper lesson: filesystem layout is not a religious debate about correctness. It's an operational SLA. Put things where people expect them. Future-you, your colleagues at 3am, and the next person who inherits the system will all thank you.

The intermediate trap is believing the FHS is universally obeyed. It isn't. `/srv` is a dead letter on most systems. `/opt` is used inconsistently. Several commercial applications dump files into `/var/lib/` without any pretense of following subdirectory conventions. `/etc` contains both config files and, on some systems, executable helper scripts (which technically violates the "no binaries in /etc" rule). The FHS is a *tendency*, not a guarantee. Verify rather than assume.

The one thing I would tell any junior who works with containers: the FHS assumptions that hold universally on bare metal do not hold in minimal container images. Alpine doesn't have the usr-merge. A scratch Docker image has nothing. The very concept of "the system's `/etc`" becomes ambiguous when you have twelve containers each with their own rootfs. The FHS was designed for a single machine with a single administrator. Containers require you to hold the FHS loosely and verify explicitly.

The final thing worth saying: `/proc` and `/sys` are two of the most useful debugging tools on a Linux system, and most people treat them like black boxes. The fact that you can read a running process's open file descriptors from `/proc/<pid>/fd/`, see its exact memory maps from `/proc/<pid>/maps`, check its current working directory from `/proc/<pid>/cwd`, and change live kernel behavior by writing to `/proc/sys/` — without any special tools, using only `cat` and `echo` — is genuinely powerful. The FHS chapter is where that story starts.

## Try It Yourself

1. Run `ls -la /` on your system. Count how many entries are symlinks (`l` at the start of permissions). Which directories have been merged? Which are still real directories? What does this tell you about your distribution?

2. Run `cat /proc/cpuinfo` and find the `model name` line. Then run `cat /proc/meminfo` and find `MemTotal`. These are your CPU and RAM reported by the running kernel — no tools required, just reads from virtual files.

3. Find the PID of your current bash shell with `echo $$`. Then explore `/proc/$$` — look at `cmdline`, `status`, and `fd/`. The `fd/` directory shows every file your shell currently has open. What do you see?

4. Pick any running service on your system (try `systemctl list-units --type=service --state=running` to find candidates). Find its log in `/var/log`, its config in `/etc`, and its PID or socket file in `/run`. Trace all three and write down the exact paths.

5. **Challenge (run in a VM):** Create a minimal directory tree that satisfies the FHS for a single-purpose web server. You don't need real files — use `mkdir -p` to create the hierarchy. Then explain in comments (in a shell script) why you put each component where you did. Compare your tree against a real running system with `ls /`.

## Summary

- The FHS is a convention, not a kernel feature — it is an agreement between distributions, package managers, and administrators so that any Linux system can be navigated predictably.
- Everything hangs off a single root `/`. There are no drive letters. Other filesystems are *mounted* into the tree.
- `/proc` and `/sys` are virtual filesystems backed by the running kernel — they have no disk presence, generate content on-demand, and must never be included in backups or disk usage calculations.
- `/run` is the canonical location for runtime data (PID files, sockets); it is a tmpfs cleared on reboot. `/var/run` is a legacy symlink to `/run` on all modern distributions.
- On Debian 12+, Ubuntu 20.04+, Fedora 17+, Arch, and most modern distributions, `/bin`, `/sbin`, `/lib`, and `/lib64` are symlinks into their `/usr/` counterparts — this is the usr-merge; hardcoded paths still work, but new code should use `/usr/bin/env` in shebangs and `/usr/bin/` for explicit paths.
