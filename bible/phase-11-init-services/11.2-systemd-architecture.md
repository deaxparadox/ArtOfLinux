## Why This Exists

Before systemd, Linux systems used SysV init — a shell-script-based init system inherited from Unix. When the kernel finished booting, it ran `/sbin/init`, which read `/etc/inittab` and launched a series of numbered shell scripts one after another: `S01logging`, `S10network`, `S20sshd`, and so on. Each script ran to completion before the next started. The system booted serially.

This had real costs. On modern hardware with SSDs and multiple CPUs, there was no reason your logging daemon had to finish starting before your network stack began. But SysV had no way to express "these two services can start at the same time." Boot times on a modest server could easily run 30–60 seconds for no good reason.

Dependency tracking was also a fiction. Nothing enforced that the database actually started before the application that needed it. If `S40postgresql` happened to be slow one day, `S50webapp` would start anyway — and crash in a confusing way. Restart behavior was entirely custom logic per-script, and monitoring whether a service was actually running meant parsing `ps` output, writing your own PID files, and hoping the PID file was current.

The fragmentation was real: different distributions had different init scripts, different conventions, different restart mechanisms. A service that worked on Red Hat might need a rewritten script on Debian.

systemd replaced all of this with a single, unified system that speaks in declarative unit files, tracks dependencies explicitly, can start services in parallel when there's no dependency between them, restarts failed services automatically, and gives you a consistent interface across every major distribution.

## Mental Model

Think of systemd as an air traffic controller at a busy airport.

Each flight is a unit — a service, a mount point, a timer, a socket. The controller knows the dependencies: the fuel truck needs to finish before the plane can push back; gate B7 needs to be clear before the inbound flight can land. Some of these are hard constraints ("the runway *must* be clear"), some are soft ("we'd *like* the gate agent to be there, but we can board without them"). The controller also knows ordering: departure happens after boarding, even if boarding isn't technically a hard prerequisite for the engines to spin up.

When you boot, systemd is the controller coming on shift. It looks at everything that needs to happen, figures out what can happen in parallel, what has hard dependencies, what can fail without cascading, and orchestrates it all — tracking what succeeded, what failed, and why, in a log you can always go back and read.

The airport doesn't shut down because one catering truck is late. Some things fail softly. Others are so critical that if they fail, everything must stop. systemd lets you express exactly which is which, declaratively, in a config file.

## Theory

### What systemd Is

systemd is both an init system and a service manager — and it's more than both of those things. As an init system, it is PID 1: the first process the kernel starts after it finishes setting up memory and scheduling. As a service manager, it starts, stops, restarts, and tracks the state of every service on the system. Beyond that, it handles:

- Mount points (replacing `/etc/fstab` execution)
- Device events (via udev integration)
- Scheduled tasks (replacing cron for many use cases)
- Logging (via journald)
- Resource management (via cgroups)
- Network management (via networkd, on distros that use it)

Every one of these capabilities is expressed through the same abstraction: the **unit**.

### Units

A unit is a named configuration object that systemd knows how to manage. The unit's name includes its type as a suffix: `sshd.service`, `home.mount`, `systemd-tmpfiles-clean.timer`. The suffix tells systemd which set of rules apply.

The eleven unit types you'll encounter:

- **`.service`** — A daemon or one-shot process. This is what most people mean when they say "a systemd service."
- **`.socket`** — A network socket or IPC socket. systemd listens on the socket and only starts the associated service when a connection arrives (socket activation).
- **`.timer`** — A scheduled job. Think cron, but with proper dependency tracking and journald logging.
- **`.mount`** — A filesystem mount point. systemd generates these automatically from `/etc/fstab` entries, but you can write them by hand too.
- **`.automount`** — Like `.mount`, but mounts on first access rather than at boot.
- **`.target`** — A grouping unit. Has no executable action of its own — it exists purely to pull in other units and represent a system state.
- **`.device`** — A kernel device that appears in `/sys`. Lets you declare dependencies on hardware being present.
- **`.path`** — Watches a file or directory and activates a service when the path condition is met.
- **`.slice`** — A node in the cgroup hierarchy for resource management. Organizes other units into groups for CPU, memory, and I/O limits.
- **`.scope`** — Like a service, but for externally started processes that systemd adopts. Used by session managers and containers.
- **`.swap`** — A swap partition or swap file.

### The Unit File

Every unit is described by a plain text configuration file. Unit files are divided into sections, each marked with a `[SectionName]` header. The three most important sections are:

**`[Unit]`** — Metadata and dependency declarations. This section is the same regardless of unit type. Key directives:

- `Description=` — Human-readable name that appears in `systemctl status` output and journal entries.
- `After=` — Ordering: start this unit after the listed units, if those units are also starting. This is **ordering only**, not a dependency.
- `Before=` — Ordering: start this unit before the listed units.
- `Requires=` — Hard dependency: if a listed unit fails to start, this unit fails too. If a listed unit is stopped while this unit is running, this unit is stopped.
- `Wants=` — Soft dependency: attempt to start listed units alongside this one, but if they fail or are absent, this unit continues anyway.
- `BindsTo=` — Like `Requires=`, but also propagates stop: if the bound unit stops (for any reason, including clean exit), this unit stops too. Used when one unit is meaningless without another.
- `Conflicts=` — Negative dependency: starting this unit stops the listed units, and vice versa.
- `ConditionPathExists=` — A condition check: only activate this unit if the named path exists. The unit is silently skipped (not failed) if the condition is false.

**`[Service]`** — For `.service` units specifically. Defines `ExecStart=`, `ExecStop=`, `Restart=`, `Type=`, environment variables, working directories, and more.

**`[Install]`** — Controls `systemctl enable/disable`. The `WantedBy=` directive says which target should pull in this unit when it's enabled. This section is read by `systemctl enable`, not by systemd at runtime.

### Targets

Targets are systemd's replacement for SysV runlevels — but they are not a direct mapping and should not be thought of that way.

A target is a named synchronization point. It has no `ExecStart=` and runs no process. Its job is to group units together and represent a desired system state. When you ask systemd to reach `graphical.target`, it pulls in everything that `graphical.target` requires or wants, and everything those units require or want, and so on recursively.

The targets you'll encounter most:

| Target | What it represents |
|---|---|
| `sysinit.target` | Early initialization complete (mounts, hostname, kernel modules) |
| `basic.target` | Basic system functionality ready (timers, sockets, paths) |
| `multi-user.target` | Multi-user text-mode system, all network services up |
| `graphical.target` | Full graphical session on top of multi-user.target |
| `rescue.target` | Single-user mode, minimal services, root shell only |
| `emergency.target` | Bare minimum — root filesystem read-only, just a shell |
| `shutdown.target` | System is shutting down |
| `reboot.target` | System is rebooting |

Targets are not exclusive. `graphical.target` depends on `multi-user.target`, which depends on `basic.target`. When you boot to `graphical.target`, all three are active simultaneously. This layering is why "targets are runlevels" is a misleading shorthand — SysV runlevels were mutually exclusive states; targets are accumulated layers.

### Unit File Locations and Precedence

systemd searches for unit files in three locations, in order of decreasing permanence:

| Location | Purpose | Priority |
|---|---|---|
| `/etc/systemd/system/` | Administrator customizations | Highest |
| `/run/systemd/system/` | Runtime-generated units (lost on reboot) | Middle |
| `/lib/systemd/system/` | Package-installed units | Lowest |

On some distributions (notably Fedora and Arch), packages install to `/usr/lib/systemd/system/` rather than `/lib/systemd/system/` — but `/lib` is often a symlink to `/usr/lib` on modern systems, so the behavior is identical.

The rule is: a file in `/etc/systemd/system/` with the same name as one in `/lib/systemd/system/` completely overrides it. When you want to modify a vendor-supplied unit, you do not edit the file in `/lib/`. You either create an override in `/etc/systemd/system/` or use a **drop-in file** — a `.conf` file in a directory named `/etc/systemd/system/unitname.service.d/` that merges with the original rather than replacing it.

## How It Works (Mechanics)

### Startup: From PID 1 to a Running System

When the kernel hands off control to systemd (PID 1), systemd does not run a shell script. It reads its own configuration from `/etc/systemd/system.conf`, then does the following:

```text
# Conceptual flow — not a real command
1. systemd activates default.target (a symlink to the configured default target)
2. systemd resolves the full dependency graph for default.target recursively
3. systemd calculates which units can start in parallel (no ordering edge between them)
4. systemd starts units in waves, respecting After=/Before= constraints
5. Each unit reports activation state: active, failed, or inactive
6. systemd continues until all reachable units have settled
```

The key insight: systemd builds a directed acyclic graph (DAG) of units and traverses it. Units with no ordering relationship between them start concurrently. This is why modern Linux systems boot in seconds rather than minutes — dozens of services start simultaneously.

### Dependency Resolution

When systemd starts a unit, it:

1. Checks all `Requires=` and `Wants=` entries — these determine which other units get scheduled to start alongside this one.
2. Checks all `After=` entries — these determine which units this one must wait for before beginning activation.
3. Checks all `Conflicts=` entries — these determine which units must be stopped before this one can start.
4. Checks all `Condition*=` entries — these are evaluated locally; a false condition causes the unit to be skipped (not failed).

The separation between dependency (`Requires=`/`Wants=`) and ordering (`After=`/`Before=`) is the most important architectural distinction in systemd. You can have a dependency without ordering (two units must both start, but order doesn't matter) and you can have ordering without a dependency (if both happen to be starting, one goes first — but neither requires the other).

### Socket Activation

Socket activation is one of systemd's most elegant features. Instead of a service binding a socket at startup, systemd binds the socket at boot — which is nearly instantaneous. The service doesn't start until the first connection arrives.

```text
# Conceptual flow
1. systemd activates cups.socket at boot
2. systemd is now listening on the CUPS socket — cups.service is NOT running
3. A print job arrives on the socket
4. systemd starts cups.service and hands it the already-open socket file descriptor
5. cups.service handles the connection
```

This means you can have hundreds of "ready" services with near-zero memory cost. Services only consume memory when actually used.

### The Default Target

systemd's default target is stored as a symlink:

```bash
# Show where the symlink points
ls -la /lib/systemd/system/default.target
```

```text
lrwxrwxrwx 1 root root 16 Jan 10 08:00 /lib/systemd/system/default.target -> graphical.target
```

On servers without a graphical environment, this typically points to `multi-user.target`. `systemctl set-default` simply rewrites this symlink.

## Basic Example

```bash
# Show the full content of a real unit file
systemctl cat sshd.service
```

```text
# /lib/systemd/system/ssh.service
[Unit]
Description=OpenBSD Secure Shell server
Documentation=man:sshd(8) man:sshd_config(5)
After=network.target auditd.service
ConditionPathExists=!/etc/ssh/sshd_not_to_be_run

[Service]
EnvironmentFile=-/etc/default/ssh
ExecStartPre=/usr/sbin/sshd -t
ExecStart=/usr/sbin/sshd -D $SSHD_OPTS
ExecReload=/bin/kill -HUP $MAINPID
KillMode=process
Restart=on-failure
RestartPreventExitStatus=255
Type=notify
RuntimeDirectory=sshd
RuntimeDirectoryMode=0755

[Install]
WantedBy=multi-user.target
Alias=sshd.service
```

Now look at what this file actually says:

- `After=network.target auditd.service` — sshd will start after both of these, but it does NOT require them. If `auditd.service` isn't enabled, sshd starts fine.
- `ConditionPathExists=!/etc/ssh/sshd_not_to_be_run` — the leading `!` inverts the check: "only start if this file does NOT exist." This is how some distros provide a disable mechanism outside of systemctl.
- `Restart=on-failure` — if sshd crashes, systemd restarts it. A clean exit (exit code 0) does not trigger restart.
- `WantedBy=multi-user.target` — when you run `systemctl enable sshd`, systemd creates a symlink in `multi-user.target.wants/`. That's all "enable" does.

Now try this yourself on your machine. If the service is named `ssh.service` instead of `sshd.service`, that's a Debian/Ubuntu convention — the name differs but the content is essentially the same.

```bash
# See all units currently loaded and their states
systemctl list-units --type=service
```

```text
  UNIT                        LOAD   ACTIVE SUB     DESCRIPTION
  cron.service                loaded active running Regular background program...
  dbus.service                loaded active running D-Bus System Message Bus
  getty@tty1.service          loaded active running Getty on tty1
  NetworkManager.service      loaded active running Network Manager
  ssh.service                 loaded active running OpenBSD Secure Shell server
  ...
```

Look at the LOAD/ACTIVE/SUB columns — these are the three states every unit has. LOAD is whether the unit file was parsed successfully. ACTIVE is the high-level state. SUB is the low-level state specific to the unit type (a service can be `running` or `exited`; a timer can be `waiting` or `elapsed`).

## Real-World Example

Here's a complete, production-realistic unit file for a web application. Walk through it carefully — every directive is here for a reason.

```ini
[Unit]
Description=Acme Web Application
Documentation=https://internal.example.com/runbooks/acme
After=network-online.target postgresql.service
Wants=network-online.target
Requires=postgresql.service

[Service]
Type=notify
User=acme
Group=acme
WorkingDirectory=/opt/acme

# Load secrets from an environment file not checked into version control
EnvironmentFile=/etc/acme/environment

ExecStartPre=/opt/acme/bin/check-db-connection
ExecStart=/opt/acme/bin/server --config /etc/acme/config.yml
ExecReload=/bin/kill -USR1 $MAINPID

# Restart on failure, but not on clean exit or SIGTERM
Restart=on-failure
RestartSec=5s

# Hard limits — the OOM killer won't save you if you don't set these
LimitNOFILE=65536
MemoryMax=2G

# Sandboxing — service can't write outside its directories
ProtectSystem=strict
ReadWritePaths=/var/lib/acme /var/log/acme
PrivateTmp=true
NoNewPrivileges=true

[Install]
WantedBy=multi-user.target
```

Walk through the dependency section carefully:

```text
After=network-online.target postgresql.service
Wants=network-online.target
Requires=postgresql.service
```

This says three distinct things:

1. **`After=network-online.target postgresql.service`** — Don't start until both of these have activated, if they're starting. Ordering only.
2. **`Wants=network-online.target`** — Also try to start `network-online.target` when this unit starts. If it fails, keep going anyway. Combined with `After=`, this means: "start the network, wait for it, then proceed — but if the network service itself is broken, start anyway."
3. **`Requires=postgresql.service`** — Hard dependency. If PostgreSQL fails to start, this unit fails. If PostgreSQL stops while this unit is running, this unit stops. Combined with `After=`, this means: "wait for PostgreSQL, and if PostgreSQL isn't available, don't even try."

The `Wants=` on `network-online.target` and `Requires=` on `postgresql.service` is a deliberate asymmetry. A degraded network might be acceptable (the app might serve cached responses). No database is fatal.

The `ExecStartPre=` line runs a check before the main process starts. If the database connection check fails, the unit fails before `ExecStart=` ever runs — giving you a clear error in `journalctl -u acme` instead of an ambiguous application crash.

## ⚠️ Gotchas

### `After=network.target` Does Not Mean "Wait for the Network to Be Up"

This is the single most common systemd misunderstanding and it has silently broken thousands of services.

`network.target` does NOT mean "the network is configured and interfaces are up." It means "NetworkManager or networkd has started." That's a process being running, not the network being ready.

```ini
# This looks correct. It is not.
[Unit]
After=network.target
```

A service with `After=network.target` can and will start before your interfaces have IP addresses, before DNS is working, before remote hosts are reachable. If your service tries to connect to a remote host at startup, it will fail.

The correct target is `network-online.target` — which is explicitly designed to wait until at least one network interface is configured and online:

```ini
# Correct for services that need actual network connectivity
[Unit]
After=network-online.target
Wants=network-online.target
```

You need both `After=` (ordering) and `Wants=` (to ensure `network-online.target` is actually activated). `After=` alone doesn't cause `network-online.target` to start — it only controls order if it happens to be starting.

### `After=` and `Requires=` Are Independent — You Almost Always Need Both

Ordering and dependency are separate concepts, and forgetting that causes subtle failures.

```ini
# Wrong: requires postgresql but no ordering
# Both units start in parallel — your app may try to connect before postgres is ready
[Unit]
Requires=postgresql.service
```

```ini
# Wrong: ordered after postgresql but no dependency
# If postgresql isn't enabled, your app starts anyway without waiting
[Unit]
After=postgresql.service
```

```ini
# Correct: hard dependency AND ordering
[Unit]
After=postgresql.service
Requires=postgresql.service
```

If you only write `Requires=`, both units start in parallel (undefined order). If you only write `After=`, and PostgreSQL isn't enabled, systemd won't start it. When you need "both are running and mine comes second," you need both directives.

### `/etc/systemd/system/` Wins — Never Edit Files in `/lib/systemd/system/`

Editing `/lib/systemd/system/sshd.service` directly will work until the next package update, which silently overwrites your changes. You will spend an hour debugging why your custom configuration disappeared.

The right workflow:

```bash
# Wrong: editing the vendor file directly
sudo vim /lib/systemd/system/sshd.service

# Right option 1: copy to /etc and edit the copy
sudo cp /lib/systemd/system/sshd.service /etc/systemd/system/
sudo vim /etc/systemd/system/sshd.service

# Right option 2: create a drop-in that only overrides what you need
sudo mkdir -p /etc/systemd/system/sshd.service.d/
sudo vim /etc/systemd/system/sshd.service.d/override.conf
```

Drop-ins are usually the better choice. A drop-in file only needs to contain the section and directives you're changing. The rest of the unit file is inherited from `/lib/systemd/system/`. This means package updates to the base unit don't break your customization, and your diff is minimal and clear.

After any change to unit files, reload the daemon:

```bash
# Required after any unit file change before systemctl commands will see it
sudo systemctl daemon-reload
```

### Targets Are Not Runlevels — They Overlap

`multi-user.target` is sometimes described as "like runlevel 3" and `graphical.target` as "like runlevel 5." This is wrong in a way that matters.

In SysV, runlevels were mutually exclusive. You were either in runlevel 3 or runlevel 5 — never both. Switching runlevels shut down all services from the old level and started services for the new level.

In systemd, `graphical.target` has `Requires=multi-user.target`. When you're in `graphical.target`, you are also simultaneously in `multi-user.target` — all services that belong to multi-user are running. Targets are additive layers, not exclusive states.

```bash
# You can see all currently active targets simultaneously
systemctl list-units --type=target --state=active
```

```text
  UNIT                   LOAD   ACTIVE SUB    DESCRIPTION
  basic.target           loaded active active Basic System
  cryptsetup.target      loaded active active Local Encrypted Volumes
  getty.target           loaded active active Login Prompts
  graphical.target       loaded active active Graphical Interface
  local-fs.target        loaded active active Local File Systems
  multi-user.target      loaded active active Multi-User System
  network-online.target  loaded active active Network is Online
  network.target         loaded active active Network
  paths.target           loaded active active Paths
  remote-fs.target       loaded active active Remote File Systems
  slices.target          loaded active active Slices
  sockets.target         loaded active active Sockets
  sysinit.target         loaded active active System Initialization
  timers.target          loaded active active Timers
```

Multiple targets are active at the same time. This is normal and expected.

### Unit Names Are Case-Sensitive

`sshd.service` and `SSHD.service` are different units. systemd matches unit names byte-for-byte.

```bash
# This fails silently — systemd looks for SSHD.service and finds nothing
systemctl status SSHD.service
```

```text
Unit SSHD.service could not be found.
```

Always use lowercase and check the exact filename in `/lib/systemd/system/` if you're unsure.

## Alternatives & Comparisons

| Init System | When You'll Encounter It | Status |
|---|---|---|
| systemd | All major distros since ~2015 | Current standard |
| SysV init | Legacy RHEL ≤ 6, very old Debian, embedded | Obsolete on mainstream distros |
| OpenRC | Gentoo, Alpine, Devuan | Active; popular in containers and embedded |
| runit | Void Linux, some containers | Active; extremely minimal |
| Upstart | Ubuntu 14.04 and earlier | Abandoned |

SysV init is still worth understanding for legacy system work and because systemd provides a compatibility layer (`systemctl` maps to old-style runlevel changes). OpenRC is a legitimate choice if you need a minimal init for containers and don't want the full systemd stack. For any standard server or desktop Linux today, you're using systemd — learning OpenRC or runit is specialist territory.

## Trade-offs

**Where this shines:**

- Parallel startup: services with no dependency relationship start simultaneously, making boot times on modern hardware dramatically faster than serial init
- Explicit, auditable dependency graphs: you can read exactly what depends on what, and `systemd-analyze dot` will render the full graph
- Consistent interface: the same `systemctl` commands work identically on Ubuntu, Fedora, RHEL, and Arch
- Built-in supervision: failed services restart automatically; no external process supervisor needed
- Socket activation: services consume zero memory until first used, letting you have hundreds of "ready" services with trivial overhead
- Integrated logging: every unit's output goes to the journal with timestamps, ordering, and filtering without additional log infrastructure

**Where this costs you:**

- Complexity: systemd is not a small, auditable codebase. Debugging subtle startup failures requires understanding the full dependency resolver, condition checks, and journald simultaneously
- Binary logs: `journald` stores logs in a binary format; if the journal is corrupted or systemd isn't running, you need `strings` to extract anything
- Scope creep: systemd has absorbed timers, network management, DNS resolution, login session management, and more — on a minimal system, this is more than you may want from PID 1
- Portability: unit files are not portable to non-systemd systems; a service written with OpenRC in mind needs a complete rewrite for systemd and vice versa

**Performance note:** `systemd-analyze blame` will show you actual activation times per unit. On a typical system with an SSD, total boot time from kernel handoff to `graphical.target` is 5–15 seconds. The same services with SysV init in serial would take 30–60 seconds.

## When to Use / When to Avoid

✅ **Use systemd unit files when:**

- You're managing a long-running service on any modern Linux system — this is the right tool, full stop
- You need automatic restart on failure with backoff
- You need to express ordering and dependency between services cleanly
- You're replacing a cron job that needs proper logging and dependency tracking — use a `.timer` unit instead
- You need socket activation to reduce memory footprint of rarely-used services

❌ **Avoid or be cautious when:**

- Writing unit files by hand for one-off testing — use `systemd-run` for transient units instead of creating and cleaning up files
- Editing `/lib/systemd/system/` files directly — use `/etc/systemd/system/` or drop-ins; the package manager will overwrite your changes
- Assuming `systemctl enable` starts a service — enable only creates the symlink that causes it to start on next boot; use `systemctl enable --now` or follow up with `systemctl start`
- Treating `Requires=` as sufficient without `After=` — parallel activation with a hard dependency is almost always wrong

⚠️ **Common mistakes:**

- Writing `After=network.target` when you need actual network connectivity — use `network-online.target`
- Forgetting `daemon-reload` after editing a unit file — systemd will keep using the old version in memory
- Using `BindsTo=` when you meant `Requires=` — BindsTo stops your unit when the dependency stops even cleanly; this is usually too aggressive
- Writing a full unit file in `/etc/systemd/system/` when a drop-in would have been cleaner and safer

## Veteran Perspective

The thing newcomers get wrong first is conflating ordering with dependency. They write `After=postgresql.service` and assume their service won't start unless PostgreSQL is running. It won't start *before* PostgreSQL — but if PostgreSQL isn't enabled, systemd won't start it either, and `After=` won't block anything. I've seen this exact assumption burn a team when they moved a service to a new host, forgot to enable the database, and the app came up clean — connecting to nothing.

The thing intermediate sysadmins get wrong is using `network.target` instead of `network-online.target`. It's the most-documented systemd gotcha and it still catches people constantly because `network.target` is what you see in most examples online. If your service hits a remote host at startup, use `network-online.target` and add the `Wants=` too.

The design principle behind all of this is that systemd separates mechanism from policy. "What order do things start?" is a separate question from "what must exist for this to start?" These are genuinely different things, and conflating them in SysV init was a source of constant confusion. systemd makes you be explicit about both — which feels verbose at first and saves you every time you're debugging a production incident.

On the unit file location hierarchy: I cannot count how many times I've seen a junior sysadmin edit `/lib/systemd/system/sshd.service`, have it work fine for six months, then lose their changes in an `apt upgrade` and spend three hours diagnosing a "mysteriously changed" service config. The rule is simple and absolute: customizations go in `/etc/`. The only time you touch `/lib/` is to read it for reference.

On targets: the "targets = runlevels" shorthand is technically wrong but practically harmless for daily operations. It becomes harmful when you're debugging a boot failure and trying to understand why `rescue.target` and `multi-user.target` appear to both be active. They can be. That's by design. Once you understand that targets are additive layers rather than exclusive states, the entire boot sequence becomes readable.

One thing the official documentation undersells: `systemd-analyze` is your friend. `systemd-analyze critical-chain graphical.target` will show you the longest path through your boot dependency graph — which unit is the bottleneck. I've used it to cut 8 seconds off a boot by discovering a single network timeout was on the critical path.

## Try It Yourself

1. Run `systemctl cat sshd.service` (or `ssh.service` on Debian/Ubuntu). Find the `After=` and `WantedBy=` lines. What does each one mean in plain English?

2. Run `systemctl list-units --type=target --state=active`. Count how many targets are simultaneously active on your running system. Is `multi-user.target` active even though you're in a graphical session?

3. Run `systemctl get-default` to see your current default target. Then run `systemd-analyze critical-chain` to see the dependency chain that leads to it. Which unit takes the longest?

4. Create a simple drop-in for an existing service without modifying the vendor file. For `sshd.service`, add a `[Service]` drop-in at `/etc/systemd/system/sshd.service.d/limits.conf` that sets `LimitNOFILE=65536`. Run `systemctl daemon-reload` and then `systemctl cat sshd.service` to confirm the drop-in appears merged at the bottom.

5. **Challenge:** Write a complete `.service` unit file for a custom script of your choice (a Python HTTP server, a bash monitoring loop, anything). Include: `After=network-online.target`, `Wants=network-online.target`, `Restart=on-failure`, `RestartSec=5s`, a non-root `User=`, and a `WantedBy=multi-user.target`. Place it in `/etc/systemd/system/`, run `daemon-reload`, `start` it, check `status`, and verify it appears in `journalctl`. Then deliberately break it (wrong path in `ExecStart=`) and watch `systemctl status` show the restart loop. (Use a VM for the last part.)

## Summary

- systemd is PID 1 and the ancestor of every process; it manages units — declarative config objects representing services, mounts, timers, sockets, and more — building a dependency graph at boot and starting units in parallel where possible.
- `After=` and `Before=` control ordering only; `Requires=` and `Wants=` control dependency; you almost always need both an ordering and a dependency directive together — one without the other causes subtle bugs.
- `Wants=` is a soft dependency (unit continues if the wanted unit fails); `Requires=` is a hard dependency (unit fails if the requirement fails or stops); `BindsTo=` is a hard dependency that also propagates stop.
- Unit files in `/etc/systemd/system/` override those in `/lib/systemd/system/`; never edit files in `/lib/` — use drop-ins in `/etc/systemd/system/unitname.service.d/` for partial overrides.
- Targets are not exclusive runlevels — they are additive layers; multiple targets are simultaneously active in a running system, and `graphical.target` includes `multi-user.target` by dependency.
