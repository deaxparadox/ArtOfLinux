## Why This Exists

A computer is dead hardware until something breathes life into it. The CPU powers on knowing one thing: start executing code at a fixed address in ROM. From that single instruction, the entire chain of events that produces a running Linux system must be constructed — firmware, bootloader, kernel, and init system, each stage handing control to the next.

This chain exists because no single piece of software can do all of it. The firmware understands the hardware but knows nothing about filesystems. The bootloader understands filesystems but can't run a full OS. The kernel can run a full OS but needs a temporary environment just to find and mount the real disk. And `systemd` can manage services but only after the kernel has set up the userspace prerequisites.

Every stage of the boot process is the answer to a specific problem its predecessor couldn't solve.

---

## Mental Model

Think of booting a Linux system as a relay race with four runners.

The first runner (firmware) sprints out of the blocks the instant the gun fires. Their only job is to find the second runner and pass the baton. They don't care about the destination — just the handoff.

The second runner (GRUB) knows the track. They pick up the baton, look at the map, and figure out which kernel to run and with which parameters. Then they load the kernel into memory and hand it the baton.

The third runner (the kernel) is the heavy lifter. They set up memory management, detect hardware, but immediately hit a problem: the real finish line (the root filesystem on disk) requires drivers that are stored on the root filesystem. Chicken-and-egg. So the kernel carries a small backpack (initramfs) with just enough tools to solve that problem, mount the real disk, and then hand off.

The fourth runner (systemd, PID 1) takes the baton after the real root is mounted and never stops running. They start every service, mount every filesystem, and keep the system alive for its entire lifetime.

The race only completes when the fourth runner reaches the login prompt. Every dropped baton is a kernel panic or a boot failure.

## Theory

### The Four Stages

The Linux boot process has four distinct stages. Each stage has exactly one responsibility: complete its own setup and hand control to the next stage.

**Stage 1 — Firmware (BIOS or UEFI)**

When the CPU receives power, it begins executing code stored in read-only firmware on the motherboard. On older hardware this is BIOS (Basic Input/Output System); on hardware made after roughly 2012, it is UEFI (Unified Extensible Firmware Interface).

Both do the same core job: POST (Power-On Self-Test), which validates that memory, CPU, and essential hardware are functional, then locates a bootloader and transfers control to it.

**Stage 2 — Bootloader (GRUB2)**

The bootloader's job is narrow but critical: locate the kernel and initramfs images on disk, load them into memory, and execute the kernel with the right command-line parameters. On modern Linux systems, that bootloader is GRUB2 (Grand Unified Bootloader version 2).

GRUB2 understands enough filesystem formats to read the kernel and initramfs from `/boot` — but it runs before the kernel, so it is not the kernel, and it has no kernel modules. Once the kernel is in memory and handed the parameters, GRUB2 is finished and never runs again during that boot.

**Stage 3 — Kernel Initialization**

The kernel decompresses itself, initializes memory management, detects and initializes hardware drivers, mounts the initramfs as a temporary root filesystem, and runs the first userspace process (`/init` inside the initramfs).

The initramfs script (or `systemd` inside initramfs on modern systems) mounts the real root filesystem and calls `pivot_root` or `switch_root` to make the real root the actual `/`. The initramfs is then discarded from memory.

**Stage 4 — Init (systemd)**

With the real root mounted, the kernel executes `/sbin/init`, which on every major modern distro is `systemd`. Systemd becomes PID 1 and takes over all further bootstrapping: mounting filesystems from `/etc/fstab`, starting services, setting up networking, and eventually launching the login manager or getty processes. The boot is complete when the default target (typically `multi-user.target` or `graphical.target`) has been reached.

### BIOS vs UEFI

BIOS and UEFI differ in fundamental ways that affect GRUB configuration, partition layout, and Secure Boot.

**BIOS** reads the first 512 bytes of the boot disk (the Master Boot Record, or MBR). The first stage of GRUB is embedded there — all 446 bytes of it. That tiny stub knows how to load the rest of GRUB from a fixed disk location called the "BIOS boot partition" (a special 1–2 MiB partition with no filesystem, just GRUB code). BIOS uses MBR partition tables, which limit disks to 2 TiB and four primary partitions.

**UEFI** replaces all of this with a proper firmware application environment. Instead of reading a raw 512-byte stub, UEFI reads a FAT32 filesystem on a dedicated partition called the EFI System Partition (ESP), typically mounted at `/boot/efi`. UEFI can load `.efi` executable files directly — GRUB2 ships as `grubx64.efi` on x86_64 systems. UEFI uses GPT (GUID Partition Table), which supports disks over 2 TiB and up to 128 partitions by default.

UEFI also maintains a list of boot entries in NVRAM — persistent firmware memory that survives power cycles. Each entry points to an `.efi` binary. You can read and manipulate these entries with `efibootmgr`.

**Secure Boot** is a UEFI feature (BIOS has no equivalent). It requires every piece of boot code to be cryptographically signed by a key trusted by the firmware. The firmware ships with Microsoft's key enrolled by default on most x86 hardware. Linux distributions ship a small signed binary called `shim` that then loads GRUB (signed with the distro's own key), which loads the kernel (also signed). The chain of trust runs: firmware → shim → GRUB → kernel → kernel modules.

As of 2026, Secure Boot is in active transition: Microsoft's 2011 signing certificate used by the original shim is expiring on June 27, 2026. Updated shim packages signed with both the 2011 and 2023 Microsoft CA certificates have been released by Red Hat, Debian, and Ubuntu. On any production system with Secure Boot enabled, you must verify that your shim and GRUB packages are up to date.

### GRUB2

GRUB2 is the standard bootloader across Debian, Ubuntu, RHEL, Fedora, Arch, and most other major distributions. The current release is GRUB 2.14 (released January 2026).

The file that GRUB actually reads at boot is `grub.cfg`. You must never hand-edit `grub.cfg`. It is a generated file, and any change you make will be overwritten the next time the generator runs — or worse, you'll introduce a syntax error that makes the system unbootable.

The correct workflow is:
1. Edit `/etc/default/grub` for global settings (timeout, default kernel, kernel parameters)
2. Add custom entries in `/etc/grub.d/` scripts
3. Run the generator to regenerate `grub.cfg`

`/etc/default/grub` controls the most important settings:

```ini
# /etc/default/grub — the file you actually edit
GRUB_DEFAULT=0                        # boot the first menu entry by default
GRUB_TIMEOUT=5                        # show the menu for 5 seconds
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"  # params appended to every normal boot
GRUB_CMDLINE_LINUX=""                 # params appended to ALL entries (including recovery)
```

The kernel parameters in `GRUB_CMDLINE_LINUX_DEFAULT` are passed to the kernel at boot. Common parameters:

- `quiet` — suppress most kernel messages during boot
- `splash` — show a graphical boot splash instead of text
- `ro` — mount the root filesystem read-only initially (standard; fsck runs, then remounts rw)
- `nomodeset` — disable Kernel Mode Setting; the kernel won't try to control the GPU at boot
- `systemd.unit=rescue.target` — drop to a rescue shell instead of full boot
- `init=/bin/bash` — bypass init entirely and go straight to a bash shell (recovery)

### Kernel Parameters: Editing at Boot Time

When the GRUB menu appears, pressing `e` opens the entry for editing. You can modify the `linux` line in place. The most important line looks like:

```text
linux   /boot/vmlinuz-6.8.0-47-generic root=UUID=... ro quiet splash
```

Changes made here are one-time — they apply only to this boot and are not saved. This is your escape hatch when a bad kernel parameter in `/etc/default/grub` is preventing normal boot.

### vmlinuz and initrd

`/boot/vmlinuz-<version>` is the compressed Linux kernel image. `vm` stands for "virtual memory" (a historical holdover from when the kernel added virtual memory support), `z` indicates it is compressed (with gzip, lzma, or zstd depending on build config). GRUB decompresses it into RAM during stage 2.

`/boot/initrd.img-<version>` (Debian/Ubuntu) or `/boot/initramfs-<version>.img` (RHEL/Fedora) is the initial RAM disk image. It is a compressed `cpio` archive containing a minimal temporary filesystem: the tools and drivers needed to mount the real root. After the kernel loads it, it is mounted as the initial root at `/`.

### initramfs: The Chicken-and-Egg Solver

The classic problem: the kernel needs to load filesystem and disk drivers to mount the root filesystem, but those drivers are stored on the root filesystem. For simple systems with compiled-in drivers this is fine. For real systems with modular drivers, RAID, LVM, encrypted volumes, or network filesystems, it is not.

The initramfs is the solution. It is a tiny self-contained root filesystem in RAM, built specifically for a given kernel version, containing exactly the drivers and tools needed to mount the real root. Once the real root is mounted, `switch_root` tears down the initramfs, frees its memory, and moves all processes over to the real root. The kernel never sees the transition — from its perspective, `/` was always there.

The initramfs is built at kernel install time by the distribution's packaging scripts. When you install a new kernel, the initramfs is generated automatically. When you add new drivers or change configuration that affects early boot, you need to rebuild it manually.

### The PID 1 Handoff

After `switch_root`, the kernel executes `/sbin/init`. On every major modern distribution, this is `systemd`. It becomes PID 1 — a position it holds for the entire lifetime of the system. PID 1 is not just the first process; it is the reaper of all orphaned processes. When any process's parent dies, the orphan is reparented to PID 1. If PID 1 dies, the kernel panics immediately — there is no recovery path.

## How It Works (Mechanics)

### The Full Sequence, Step by Step

```text
# Conceptual trace — not a real command
1.  Power on
2.  CPU jumps to firmware entry point (ROM)
3.  Firmware runs POST (memory, CPU, devices)
4.  BIOS: reads MBR → executes GRUB stage 1
    UEFI: reads ESP → executes grubx64.efi
5.  GRUB reads grub.cfg from /boot/grub/ or /boot/grub2/
6.  GRUB displays menu (or auto-selects default after timeout)
7.  GRUB loads vmlinuz into RAM
8.  GRUB loads initrd/initramfs into RAM
9.  GRUB executes the kernel, passing command-line parameters
10. Kernel decompresses itself
11. Kernel initializes memory management (paging, buddy allocator)
12. Kernel detects and initializes hardware (PCI, USB, block devices)
13. Kernel mounts initramfs as root (/)
14. Kernel runs /init inside initramfs
15. initramfs /init loads drivers, assembles LVM/RAID/crypto if needed
16. initramfs /init mounts the real root filesystem
17. switch_root: real root becomes /, initramfs freed from RAM
18. Kernel executes /sbin/init → systemd, PID 1
19. systemd reads /etc/systemd/system/ and default.target
20. systemd starts units in dependency order
21. Login manager or getty appears
```

### What GRUB Reads

GRUB's generated config file lives at different paths depending on distro and firmware:

```bash
# Debian/Ubuntu — BIOS
ls -la /boot/grub/grub.cfg

# RHEL/Fedora — BIOS
ls -la /boot/grub2/grub.cfg

# UEFI systems (any distro) — may also be here
ls -la /boot/efi/EFI/*/grub.cfg
```

```text
-rw-r--r-- 1 root root 7842 Jun 10 14:23 /boot/grub/grub.cfg
```

### What the Kernel Ring Buffer Captures

From the moment the kernel starts until userspace tools can read it, the kernel writes diagnostic messages into a circular buffer. This is the kernel ring buffer, readable via `dmesg`.

```bash
# Show the full ring buffer with human-readable timestamps
dmesg --human

# Show only the first messages (firmware handoff to kernel)
dmesg --human | head -30

# Filter for errors and warnings only
dmesg --level=err,warn
```

```text
[    0.000000] Linux version 6.8.0-47-generic (buildd@lcy02-amd64-050)
[    0.000000] Command line: BOOT_IMAGE=/boot/vmlinuz-6.8.0-47-generic root=UUID=... ro quiet splash
[    0.000000] BIOS-provided physical RAM map:
...
[    3.142857] EXT4-fs (sda1): mounted filesystem with ordered data mode
```

The timestamps are seconds since kernel start, sub-millisecond precision. The kernel ring buffer is one of the most valuable diagnostic tools during boot failures — if the system gets far enough to start the kernel, `dmesg` has a record.

### Rebuilding initramfs

After a kernel update, after adding a module that must be present at early boot, or after recovering from a corrupted initramfs, you rebuild it:

```bash
# Debian / Ubuntu — rebuild for the running kernel
sudo update-initramfs -u

# Debian / Ubuntu — rebuild for a specific kernel version
sudo update-initramfs -u -k 6.8.0-47-generic

# Debian / Ubuntu — rebuild for ALL installed kernels
sudo update-initramfs -u -k all

# RHEL / Fedora / Rocky — rebuild for the running kernel
sudo dracut --force

# RHEL / Fedora / Rocky — rebuild for a specific kernel
sudo dracut --force /boot/initramfs-$(uname -r).img $(uname -r)

# RHEL / Fedora / Rocky — rebuild for all installed kernels
sudo dracut --regenerate-all --force
```

### Reading Boot Timing with systemd-analyze

Once systemd has started, `systemd-analyze` can report how long each stage took:

```bash
# Total boot time broken into firmware, loader, kernel, userspace
systemd-analyze

# All units ranked slowest-first
systemd-analyze blame

# The critical path — the chain of units that determined total boot time
systemd-analyze critical-chain

# SVG graphic of the full boot timeline
systemd-analyze plot > /tmp/boot.svg
```

## Basic Example

```bash
# See the full boot sequence that just happened
dmesg --human | head -50
```

```text
[  +0.000000] Linux version 6.8.0-47-generic ...
[  +0.000001] Command line: BOOT_IMAGE=/boot/vmlinuz-6.8.0-47-generic root=UUID=abc123 ro quiet splash
[  +0.000005] BIOS-provided physical RAM map:
[  +0.000006] BIOS-e820: [mem 0x0000000000000000-0x000000000009fbff] usable
...
[  +3.102441] EXT4-fs (sda1): mounted filesystem
[  +3.891200] systemd[1]: Reached target Swap. (2ms 40us)
```

Look at the timestamps on the left — those are seconds since kernel start. The first few lines show the kernel version, the exact command line GRUB passed, and the memory map the firmware reported. The entry point for the root filesystem (`root=UUID=...`) shows exactly what the kernel was told to mount.

Now check how systemd describes the same event from its perspective:

```bash
systemd-analyze
```

```text
Startup finished in 3.291s (firmware) + 2.104s (loader) + 1.847s (kernel) + 4.632s (userspace) = 11.874s
graphical.target reached after 4.551s in userspace
```

Try it yourself. Your firmware time will likely be different — UEFI systems typically have shorter firmware times than legacy BIOS machines. If your firmware time is more than 10 seconds, the firmware is doing something unusual: slow POST, scanning for USB devices, or a stuck boot order entry.

## Real-World Example

A server won't boot after a kernel update. Someone reports it drops to a `(initramfs)` prompt. Here's the diagnostic and recovery workflow.

```bash
# At the (initramfs) prompt, check what's available
ls /dev/sd*
ls /dev/nvme*

# Try to mount the root filesystem manually
# (substitute your actual root device — check /proc/cmdline for the UUID)
mount /dev/sda1 /root

# If the mount fails with "wrong fs type", the filesystem may need fsck
fsck /dev/sda1
```

If `mount` succeeds, the problem is not the disk — the initramfs simply couldn't find or mount it automatically. This happens when the initramfs was built before a driver was added, or when the root device path changed (e.g., after adding a new disk that shifted `/dev/sda` to `/dev/sdb`).

After getting back into the running system (either from a live disk or by completing the manual mount and chrooting), rebuild the initramfs:

```bash
# On Debian/Ubuntu — rebuild all
sudo update-initramfs -u -k all

# On RHEL/Fedora — rebuild all
sudo dracut --regenerate-all --force

# Regenerate GRUB config in case kernel parameters also changed
# Debian/Ubuntu:
sudo update-grub

# RHEL/Fedora:
sudo grub2-mkconfig -o /boot/grub2/grub.cfg
```

Now check the boot timing on the repaired system to verify everything is clean:

```bash
systemd-analyze blame | head -20
```

```text
          3.182s plymouth-quit-wait.service
          2.841s NetworkManager-wait-online.service
          1.203s dev-sda1.device
           891ms systemd-udev-settle.service
           743ms accounts-daemon.service
           612ms snapd.service
...
```

What you're looking for in `blame` output: any unit taking more than 5 seconds is worth investigating. `NetworkManager-wait-online.service` regularly tops the list on servers without a reliable network — if the machine doesn't need network at boot, you can safely disable it with `systemctl disable NetworkManager-wait-online.service`.

To see the dependency chain that determined the total boot time:

```bash
systemd-analyze critical-chain graphical.target
```

```text
The time when unit became active or started is printed after the "@" character.
The time the unit took to start is printed after the "+" character.

graphical.target @8.421s
└─multi-user.target @8.420s
  └─network-online.target @8.419s
    └─NetworkManager-wait-online.service @5.577s +2.841s
      └─NetworkManager.service @4.103s +1.473s
        └─dbus.service @3.891s +211ms
```

This tells you exactly which unit is on the critical path and how long each link in the chain took. The `@` is when it became active; the `+` is how long it took to start. A unit that takes a long time but is not on the critical chain does not affect total boot time — only the chain matters.

## ⚠️ Gotchas

### Editing `/etc/default/grub` Without Running the Generator

The most common mistake new sysadmins make. You edit `/etc/default/grub`, add a kernel parameter, reboot — and nothing changed. The kernel is still running with the old parameters.

```bash
# You edited /etc/default/grub and added "nomodeset" to GRUB_CMDLINE_LINUX_DEFAULT
# You reboot expecting it to take effect
# It doesn't. Why?
cat /proc/cmdline
```

```text
BOOT_IMAGE=/boot/vmlinuz-6.8.0-47-generic root=UUID=abc123 ro quiet splash
```

`/etc/default/grub` is input to the generator. `grub.cfg` is what GRUB actually reads. Until you run the generator, `grub.cfg` is unchanged.

```bash
# Debian/Ubuntu — always run this after editing /etc/default/grub
sudo update-grub

# RHEL/Fedora/Rocky — same result, different command
sudo grub2-mkconfig -o /boot/grub2/grub.cfg
```

After running the generator, verify your change landed in `grub.cfg`:

```bash
grep "nomodeset" /boot/grub/grub.cfg          # Debian/Ubuntu
grep "nomodeset" /boot/grub2/grub.cfg         # RHEL/Fedora
```

The rule: `/etc/default/grub` is never read by GRUB. It is only read by `update-grub`/`grub2-mkconfig`. A change there is inert until the generator runs.

### initramfs Out of Sync After a Kernel Update

When a kernel package update installs a new vmlinuz, the packaging scripts are supposed to run `update-initramfs` or `dracut` automatically. Sometimes they don't — due to disk space issues, a failed hook, or a manual kernel installation. The next boot drops to the dreaded `(initramfs)` prompt.

```bash
# What you see on the console instead of a login prompt:
# Begin: Loading essential drivers ... done.
# Begin: Running /scripts/init-premount ... done.
# Begin: Mounting root file system ... Begin: Running /scripts/local-top ...
# ALERT! UUID=abc123 does not exist. Dropping to a shell!
# (initramfs)
```

The kernel was updated but the initramfs still references the old kernel version's modules. The fix, after booting from the GRUB menu into an older working kernel or a live environment:

```bash
# Rebuild for all kernels — never skip --all, you may have multiple installed
sudo update-initramfs -u -k all       # Debian/Ubuntu
sudo dracut --regenerate-all --force  # RHEL/Fedora
```

The rule: after any manual kernel installation or after any `update-initramfs`/`dracut` failure, verify that `/boot/initrd.img-<version>` or `/boot/initramfs-<version>.img` exists and has a recent timestamp matching the kernel binary.

### `nomodeset` Gives You Working Boot But Broken Graphics

`nomodeset` is the standard advice for fixing a system that hangs at the kernel's GPU initialization. It works — but it disables Kernel Mode Setting entirely. Your display will run at a low resolution through the VESA or EFI framebuffer driver, with no hardware acceleration, no high refresh rates, and no HDMI audio.

```bash
# With nomodeset, you'll see something like this from lspci:
lspci -k | grep -A2 VGA
```

```text
00:02.0 VGA compatible controller: Intel UHD Graphics 770
        Subsystem: ...
        Kernel driver in use: efifb          # ← efifb means nomodeset is active
        Kernel modules: i915
```

`efifb` instead of `i915` (Intel), `amdgpu`, or `nvidia` means you are running the fallback framebuffer with no KMS. This is correct for getting an unbootable system booted, but it is not a permanent fix. The real fix is to identify why KMS is failing — usually a missing firmware file, a GPU that needs a newer driver, or a conflict with an early KMS handoff.

```bash
# After removing nomodeset and getting proper drivers, verify:
lspci -k | grep -A2 VGA
```

```text
00:02.0 VGA compatible controller: Intel UHD Graphics 770
        Kernel driver in use: i915           # ← correct
```

The rule: `nomodeset` is a diagnostic tool and a recovery step, not a permanent kernel parameter. If you need it permanently, your real problem is an unsupported or misconfigured GPU driver.

### `update-grub` vs `grub-mkconfig` vs `grub2-mkconfig` — They're All the Same Thing

On Debian/Ubuntu, `update-grub` is a wrapper script that calls `grub-mkconfig -o /boot/grub/grub.cfg`. On RHEL/Fedora/Rocky, the command is `grub2-mkconfig`. On Arch, it is `grub-mkconfig`. They all produce the same result — a regenerated `grub.cfg`.

If you are following a tutorial and the command doesn't exist, check the table in Distro Notes below. Never copy a `grub.cfg` from one system to another — the UUIDs, kernel versions, and paths will be wrong.

## Distro Notes

| Distro Family | `grub.cfg` location (BIOS) | `grub.cfg` location (UEFI) | Generator command | initramfs tool |
|---|---|---|---|---|
| Debian / Ubuntu | `/boot/grub/grub.cfg` | `/boot/efi/EFI/debian\|ubuntu/grub.cfg` | `update-grub` | `update-initramfs -u` |
| RHEL / CentOS / Rocky | `/boot/grub2/grub.cfg` | `/boot/efi/EFI/redhat/grub.cfg` | `grub2-mkconfig -o /boot/grub2/grub.cfg` | `dracut --force` |
| Fedora | `/boot/grub2/grub.cfg` | `/boot/efi/EFI/fedora/grub.cfg` | `grub2-mkconfig -o /boot/grub2/grub.cfg` | `dracut --force` |
| Arch | `/boot/grub/grub.cfg` | `/boot/grub/grub.cfg` | `grub-mkconfig -o /boot/grub/grub.cfg` | `mkinitcpio -P` |
| Alpine (containers) | Usually no GRUB (PV/HVM) | N/A | N/A | `mkinitfs` |

Note: On UEFI RHEL systems, `/etc/grub2-efi.cfg` is a symlink to the actual location in the ESP. On Arch, the GRUB config path is the same regardless of firmware type because Arch uses a unified `/boot` layout.

## Alternatives & Comparisons

| Bootloader | When to Use | Notes |
|---|---|---|
| GRUB2 | Default choice on every major distro | Supports BIOS, UEFI, Secure Boot, encrypted /boot, multiboot. Current release: 2.14 |
| systemd-boot | UEFI-only, single-distro, simple setups | Minimal configuration, fast, but no support for BIOS or encrypted /boot without extra steps |
| LILO | Legacy — only encountered on very old systems | No longer maintained; cannot boot from GPT disks |
| rEFInd | Multi-OS setups, Mac dual boot | Auto-discovers kernels; useful for complex EFI environments but adds complexity on servers |

For any new server or desktop installation today, GRUB2 is the correct choice unless you have a specific reason to use `systemd-boot`. `systemd-boot` is worth considering for container hosts and CI systems where boot simplicity matters and BIOS compatibility is irrelevant.

## Trade-offs

**Where this knowledge shines:**

- When a system won't boot, understanding the four-stage sequence tells you exactly where in the chain the failure is — before you even see a prompt
- Kernel parameter changes at the GRUB menu give you a surgical recovery path without needing a live disk
- `systemd-analyze blame` and `critical-chain` make boot time optimization concrete and evidence-based rather than guesswork

**Where this costs you:**

- GRUB2's configuration is split across `/etc/default/grub`, `/etc/grub.d/` scripts, and the generated `grub.cfg` — the indirection is intentional but confusing until you internalize the generator model
- initramfs rebuilds take time (seconds to minutes depending on distro) and are easy to forget after manual kernel operations
- UEFI Secure Boot adds an entire certificate trust chain; a shim or key mismatch at firmware level is difficult to debug without physical access to the machine or an IPMI/iLO console

**Performance note:** On modern UEFI systems, firmware POST is the dominant boot time component for many systems — often 3–5 seconds before the OS even starts. Disabling unused firmware features (network PXE boot scan, legacy option ROMs) can reduce this significantly.

## When to Use / When to Avoid

✅ **Edit `/etc/default/grub` when:**
- You need to add a permanent kernel parameter (e.g., `intel_iommu=on` for VFIO, `mitigations=off` for benchmarking)
- You want to change the default boot entry or timeout for a multi-kernel system
- You're configuring a headless server to redirect console output to a serial port (`console=ttyS0,115200`)

❌ **Never directly edit `grub.cfg` because:**
- It is overwritten by `update-grub`/`grub2-mkconfig` on every kernel update
- Syntax errors in `grub.cfg` produce an unbootable system with no fallback
- The correct place for custom entries is `/etc/grub.d/40_custom`

✅ **Use the GRUB `e` key (one-time parameter edit) when:**
- The system fails to boot and you need to test a parameter change before committing it
- You need `systemd.unit=rescue.target` to enter rescue mode without modifying the permanent config
- You're debugging a graphics issue and want to test `nomodeset` without making it permanent

✅ **Rebuild initramfs when:**
- You have manually installed a kernel (not through the package manager)
- You have added a kernel module that must be present at early boot (e.g., a storage controller)
- You changed initramfs hooks or `dracut` modules
- The system dropped to an `(initramfs)` prompt and you've diagnosed a stale image as the cause

✅ **Use `systemd-analyze blame` when:**
- A system's boot time has increased and you need to identify the responsible unit
- You're hardening a server and want to find services that are slow to start but not needed at boot

⚠️ **Common mistakes:**
- Editing `/etc/default/grub` and rebooting without running the generator — nothing changes
- Running `update-initramfs` or `dracut` without `--force` on RHEL/Fedora when an initramfs already exists — dracut will refuse to overwrite without `--force`
- Assuming `nomodeset` is a fix rather than a workaround — it disables KMS permanently for that boot entry

## Veteran Perspective

The boot process is one of those topics where everyone understands the high-level summary after five minutes, and then the first time something breaks they discover they understood nothing at all.

The most common mistake I see from engineers who "know" the boot process is that they conflate `grub.cfg` with `/etc/default/grub`. They add a parameter, the system reboots, the parameter isn't there, and they conclude GRUB is broken. GRUB isn't broken — they haven't learned the generator model. The generator model is a clean design: your intent lives in `/etc/default/grub`, the generated artifact lives in `grub.cfg`, and the tool that translates between them is `update-grub`. Every other tool in Linux works this way (think `nginx -t && nginx -s reload`). The pattern is the same. Once you see it as a pattern, you stop being confused by it.

The intermediate trap is the initramfs. Kernel updates through the package manager handle initramfs regeneration automatically — and so you go months or years without ever thinking about it. Then one day you install a kernel manually, or a disk fills up during an update and the initramfs hook silently fails, or you add a new storage controller. The next boot drops to an `(initramfs)` prompt. If you've never debugged that prompt before, it looks like complete disaster. It isn't. The filesystem is fine. The kernel is fine. The initramfs just can't find the root device because it was built before the driver existed. Three commands fix it.

The deeper design principle: the boot process is a series of escalating capability levels. The firmware can't read Linux filesystems. GRUB can read them but can't run a full OS. The kernel can run a full OS but needs a temporary environment to bootstrap. Systemd can manage a full system but only after all the prerequisites are in place. Each layer's constraint defines the next layer's job. When you understand why each stage exists, the handoffs between them make perfect sense — and when something breaks, you can immediately identify which stage failed and what capability it was missing.

On Secure Boot: in 2026 this is not optional anymore on enterprise hardware. More firmware vendors are shipping with Secure Boot enforced by default and no easy toggle. The Microsoft certificate expiration in June 2026 has caught several organizations off-guard — they have systems with unpatched shim packages that will have trouble enrolling new keys after the cutoff. If you're managing a fleet of Linux machines, audit your shim versions now: `rpm -q shim` on RHEL-family, `dpkg -l shim-signed` on Debian-family. The expiration affects new key enrollment, not existing booting — but the implication is clear: update or be unable to recover from a future secure boot configuration issue.

The tool most people underuse is `systemd-analyze critical-chain`. `blame` lists slow units but doesn't tell you which ones matter. A unit that takes 10 seconds but runs in parallel with everything else adds zero seconds to total boot time. Only units on the critical chain matter for overall boot latency. Learn to read the chain output — the `@` is when it became active, the `+` is how long it took. A production server should have a `graphical.target` or `multi-user.target` activation time under 10 seconds from kernel start. If it's higher, `critical-chain` will tell you exactly what's on the hot path.

## Try It Yourself

1. Run `systemd-analyze` on your current system. Note the four timing segments: firmware, loader, kernel, userspace. Which is largest on your machine?

2. Run `systemd-analyze blame | head -10`. Identify the slowest unit. Look it up with `systemctl status <unit-name>` — does it need to run at boot, or could it be started later or disabled?

3. Find the GRUB config on your system. Run `cat /proc/cmdline` to see the actual parameters the kernel booted with, then trace them back to where they were set (is it in `GRUB_CMDLINE_LINUX_DEFAULT` in `/etc/default/grub`? Or were they set somewhere else?).

4. Reboot your system (or a VM). At the GRUB menu, press `e` to edit the boot entry. Add `systemd.log_level=debug` to the end of the `linux` line and boot. Watch the verbose output. Press Ctrl+Alt+F2 once the system boots to get a console if the display is flooded. Then check `journalctl -b | head -100` to see the full systemd startup log.

5. **Challenge:** On a VM, after booting normally, run `systemd-analyze plot > /tmp/boot.svg`. Transfer the SVG to your host and open it in a browser. Identify the three longest parallel chains in the boot graph. Then look at the critical chain with `systemd-analyze critical-chain` and confirm whether those long chains appear on the critical path. Write down one unit you would disable or delay to improve boot time, and explain why it is safe to do so.

## Summary

- The boot process has four stages: firmware (BIOS/UEFI) → bootloader (GRUB2) → kernel + initramfs → systemd (PID 1). Each stage has exactly one job: complete its setup and hand off to the next.
- Never edit `grub.cfg` directly. Edit `/etc/default/grub`, then run `update-grub` (Debian/Ubuntu) or `grub2-mkconfig` (RHEL/Fedora) to regenerate it. Changes to `/etc/default/grub` have no effect until the generator runs.
- The initramfs is a temporary RAM filesystem that exists solely to mount the real root. If it goes out of sync with the kernel — after a manual kernel install or a failed update hook — the system drops to an `(initramfs)` prompt. Fix it with `update-initramfs -u -k all` or `dracut --regenerate-all --force`.
- `systemd-analyze blame` ranks slow units; `systemd-analyze critical-chain` shows what actually determined total boot time. Only units on the critical chain affect overall boot latency.
- UEFI Secure Boot requires a chain of signed binaries: firmware → shim → GRUB → kernel. In 2026, the Microsoft 2011 signing certificate is expiring — verify your shim and GRUB packages are up to date on any production system with Secure Boot enabled.
