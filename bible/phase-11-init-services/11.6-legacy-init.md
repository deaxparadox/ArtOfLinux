## The Problem This Solves

You're debugging a production system. You `ssh` into a RHEL 6 box — a legacy application server that never got migrated. There's no `systemctl`. There's no `journalctl`. You type `systemctl status httpd` and the shell returns `command not found`.

Now what?

Or you're looking at an Alpine Linux container in a Kubernetes cluster. Someone on the ops team is saying "it runs OpenRC, not systemd." You've never touched OpenRC. You don't know its commands. You don't know where its service scripts live. You're working blind.

Or you're reading a five-year-old runbook and it tells you to run `chkconfig httpd on` to enable a service at boot. Is that still valid? Does it do what you think? On a modern system, does it even work?

Legacy init systems aren't going away. They live in embedded devices, old enterprise systems that never got upgraded, containers running minimal distributions, and everywhere BSD and Gentoo are deployed. You don't need to master these systems. You need to recognize them, navigate them with confidence, and understand why systemd replaced them so you can explain it to someone who thinks the old way was fine.

## Mental Model

Think of booting a computer like opening a restaurant for the day.

SysV init is the manager who has a laminated checklist on a clipboard. Every morning, they go through it in order: turn on the lights (step 1), start the espresso machine (step 2), put out the chairs (step 3). Step 4 doesn't start until step 3 is done. If the espresso machine breaks mid-shift and turns off, nothing restarts it — the manager finishes the checklist and goes home. The coffee stays broken until someone notices.

systemd is the manager with a whiteboard and a team. Everything that can happen in parallel does. Dependencies are tracked. If the espresso machine breaks, a monitor notices and restarts it. The manager is always watching.

OpenRC is a different manager: more experienced than the clipboard approach, supports parallel startup, tracks dependencies — but still uses shell scripts and still expects a human to notice when things die.

runit and s6 are the security specialists: tiny, paranoid, focused entirely on one thing — ensuring every service has a supervisor watching it 24/7 and restarting it immediately if it dies.

## Theory

### SysV Init

SysV init (System V init) was the original init system for Unix System V and became the default on most Linux distributions through the 1990s and 2000s. It is process ID 1 — the first userspace process the kernel starts after boot.

The central concept is the **runlevel**: a numeric state (0–6) that defines which services should be running. The mapping of runlevels to system states was never standardized across distributions, which is the source of constant confusion:

| Runlevel | Meaning (Linux convention — not portable) |
|---|---|
| 0 | Halt — shuts the system down |
| 1 | Single-user mode — recovery, no network, root only |
| 2 | Multi-user without NFS (Debian/Ubuntu uses this as the default desktop runlevel) |
| 3 | Multi-user with networking, no GUI (RHEL/Fedora convention for servers) |
| 4 | Unused / user-defined |
| 5 | Multi-user with GUI (RHEL/Fedora convention for desktop) |
| 6 | Reboot |

Runlevel 3 on a RHEL system means "server mode, no GUI." Runlevel 3 on a Debian system historically meant something different. This non-portability is one of the reasons systemd replaced runlevels with named targets (`multi-user.target`, `graphical.target`) that mean the same thing everywhere.

Services in SysV init are shell scripts stored in `/etc/init.d/`. Each script is required to accept at least four arguments: `start`, `stop`, `restart`, and `status`. The script does the work manually — fork a process, write a PID file, redirect logs. There is no structured output, no standard way to check if the service is alive, and no automatic restart if it dies.

To decide which services run at which runlevel, the system uses symbolic links in `/etc/rc[N].d/` directories. A link named `S20httpd` in `/etc/rc3.d/` means "start httpd at runlevel 3, ordering number 20." A link named `K80httpd` means "kill httpd when leaving that runlevel, ordering number 80." The ordering number is the only dependency management SysV provides — it is sequence, not a real dependency graph.

### The Boot Sequence Problem

SysV starts services strictly sequentially. Service B does not start until service A has finished starting. If your system has 40 services, each taking 2 seconds to initialize, that's 80 seconds of serial initialization time — even if 38 of those services have no dependency on each other and could run simultaneously.

On modern hardware with SSDs and many CPU cores, this is brutally wasteful. Systems that booted in 60–90 seconds under SysV boot in 5–10 seconds under systemd. The difference is parallelism, and SysV was structurally incapable of it.

### Upstart

Upstart was Ubuntu's answer to SysV, introduced around 2006. It replaced sequential startup with an event-driven model: instead of "start service B after service A finishes," Upstart said "start service B when event X occurs." This enabled some parallelism and added the ability to restart crashed services.

Upstart was the default init system on Ubuntu from version 6.10 through 14.10. It never achieved wide adoption outside Ubuntu and was replaced by systemd in Ubuntu 15.04 (2015). RHEL 6 also shipped Upstart. You may still encounter Upstart configuration files (`/etc/init/*.conf`) on very old Ubuntu or RHEL 6 systems. For most purposes, knowing Upstart existed is enough — you are unlikely to need to write new Upstart jobs.

### Why systemd Replaced Both

systemd solved four problems simultaneously that SysV and Upstart could not:

1. **Parallel startup with a real dependency graph.** systemd reads all unit files before starting anything, builds a dependency tree, and starts everything that can be started simultaneously. Services that genuinely depend on each other start in the right order. Services that don't share dependencies start in parallel.

2. **Socket activation.** systemd can hold a socket open and queue connections before the service that owns it has finished starting. Dependent services see the socket as available immediately and don't need to retry-connect. This eliminates one entire class of startup ordering problems.

3. **Automatic restart.** `Restart=on-failure` in a unit file tells systemd to restart the service if it exits with a non-zero status. This requires one line. In SysV, achieving this required a separate supervisor process.

4. **Integrated logging.** `journald` collects stdout, stderr, and kernel messages from every service in one structured, queryable store. SysV services had to handle their own logging — most redirected to files in `/var/log/`, inconsistently named and formatted.

### OpenRC

OpenRC is not SysV init. This is a common confusion worth being direct about.

OpenRC is a dependency-based init system that runs on top of a separate PID 1 process — typically BusyBox init on Alpine Linux, or sysvinit on Gentoo. OpenRC manages services and handles boot/shutdown sequencing. PID 1 itself is a separate, simpler process.

OpenRC uses shell scripts like SysV (stored in `/etc/init.d/`), but adds real dependency declarations at the top of each script. It supports parallel startup when enabled. It tracks service state in `/run/openrc/`. The commands are `rc-service` (to control services) and `rc-update` (to manage which services run at which runlevels).

OpenRC is actively maintained and ships with cgroup v2 support. Its primary distributions are Gentoo and Alpine Linux. It is a real, modern init system — not a relic.

The key distinction from systemd: OpenRC does not provide logging integration, socket activation, or timers. It manages services. Full stop.

### runit and s6

runit and s6 are supervision systems. They represent a different design philosophy from both SysV and systemd: each service gets a permanent supervisor process whose only job is to watch that service and restart it if it dies. This is called a **supervision tree**.

A supervision tree means: the init system never "finishes" starting a service and walks away. It maintains a parent-child relationship with every supervised process indefinitely. If the service exits for any reason, the supervisor restarts it within milliseconds — no manual intervention, no alerting, no health checks. The supervisor is always watching.

runit is used as the default init system on Void Linux and is a popular choice for managing processes inside Docker containers. s6 is architecturally similar to runit but more modular; it is used by distributions like Artix Linux (as part of s6-rc) and is increasingly chosen for container process supervision.

Neither runit nor s6 is a general-purpose init system in the way systemd is. They do not provide logging integration, network management, or hardware event handling. On a desktop or server system, they typically work alongside other tools that handle those responsibilities.

## How It Works (Mechanics)

### SysV Boot Sequence

When the kernel hands off to SysV init (PID 1), the sequence is:

```text
# Conceptual flow — not a real command
kernel starts /sbin/init (SysV) as PID 1
→ init reads /etc/inittab to determine default runlevel
→ init runs /etc/rc.sysinit (RHEL) or /etc/rcS.d/ scripts (Debian) — early system setup
→ init runs /etc/rc<N>.d/ scripts in alphabetical order for the target runlevel
    → scripts starting with S are started, in numeric order
    → scripts starting with K are stopped (when leaving a runlevel)
→ init runs /sbin/mingetty on each configured TTY — the login prompt
```

Each `S` script runs, waits for completion, and returns before the next one starts. This is the sequential bottleneck.

### What an Init Script Looks Like

A SysV init script is a shell script that tests its first argument and branches:

```bash
#!/bin/bash
# /etc/init.d/myservice — simplified SysV init script
# chkconfig: 2345 20 80
# description: My example service

PIDFILE=/var/run/myservice.pid
DAEMON=/usr/bin/myservice

case "$1" in
    start)
        echo -n "Starting myservice: "
        # start the daemon, write PID to file manually
        start-stop-daemon --start --pidfile $PIDFILE \
            --make-pidfile --background --exec $DAEMON
        echo "done"
        ;;
    stop)
        echo -n "Stopping myservice: "
        start-stop-daemon --stop --pidfile $PIDFILE
        rm -f $PIDFILE
        echo "done"
        ;;
    restart)
        $0 stop
        $0 start
        ;;
    status)
        if [ -f $PIDFILE ] && kill -0 $(cat $PIDFILE) 2>/dev/null; then
            echo "myservice is running (PID $(cat $PIDFILE))"
        else
            echo "myservice is not running"
        fi
        ;;
    *)
        echo "Usage: $0 {start|stop|restart|status}"
        exit 1
        ;;
esac
```

Notice what this script does not do: it does not restart the service if it crashes. It does not track whether the service is healthy. It does not integrate with a central logging system. If `myservice` exits at 3am, the PID file still exists but `kill -0` on a non-existent PID will fail silently until someone runs `status` manually.

### How `chkconfig` and `update-rc.d` Work

The symlinks in `/etc/rc[N].d/` are what actually control which services run at which runlevel. `chkconfig` (RHEL) and `update-rc.d` (Debian) are tools that manage those symlinks so you don't have to create them by hand.

```bash
# What chkconfig on/off does under the hood:
# chkconfig httpd on → creates /etc/rc2.d/S15httpd -> /etc/init.d/httpd
#                                /etc/rc3.d/S15httpd -> /etc/init.d/httpd
#                                /etc/rc4.d/S15httpd -> /etc/init.d/httpd
#                                /etc/rc5.d/S15httpd -> /etc/init.d/httpd
# chkconfig httpd off → removes those symlinks
ls -la /etc/rc3.d/
```

```text
lrwxrwxrwx 1 root root 16 Jan 10 2019 S15httpd -> ../init.d/httpd
lrwxrwxrwx 1 root root 16 Jan 10 2019 S22mysql -> ../init.d/mysql
lrwxrwxrwx 1 root root 14 Jan 10 2019 S85sshd -> ../init.d/sshd
```

The number after S/K is the ordering hint. `S15httpd` starts before `S22mysql` and both start before `S85sshd`. That ordering is not a dependency declaration — it is a human guess that becomes stale the moment anything changes in the service landscape.

### How OpenRC Manages Services

OpenRC service scripts live in `/etc/init.d/` (same location as SysV) but they use an OpenRC-specific shell library (`/lib/rc/sh/openrc-run.sh`) that provides structured hooks:

```bash
#!/sbin/openrc-run
# /etc/init.d/myservice — OpenRC service script

description="My example service"

depend() {
    # declare real dependencies — OpenRC enforces these
    need net        # requires network before starting
    after logger    # run after logger if it's enabled
}

start() {
    ebegin "Starting myservice"
    start-stop-daemon --start --exec /usr/bin/myservice \
        --pidfile /run/myservice.pid --make-pidfile --background
    eend $?
}

stop() {
    ebegin "Stopping myservice"
    start-stop-daemon --stop --pidfile /run/myservice.pid
    eend $?
}
```

The `depend()` function is the key difference from SysV. OpenRC reads these dependency declarations and builds a partial order before starting services. `need` means "this service cannot start without that one." `after` means "if that service is enabled, start this one after it."

### The runit Service Directory

runit's model is entirely different from both SysV and OpenRC. A runit service is a directory, not a script. The directory must contain an executable file named `run`:

```text
/etc/sv/myservice/
├── run          ← executable: starts the service (run in foreground, never daemonize)
└── log/
    └── run      ← executable: handles logging for this service
```

The `run` script is critical: it must not daemonize. It runs the service in the foreground. If the service forks to background and the `run` script exits, runit sees it as a service crash and immediately restarts it. This is intentional — runit's supervision model requires the service process to be a child of the supervisor.

```bash
#!/bin/sh
# /etc/sv/myservice/run
exec /usr/bin/myservice --foreground
```

The `exec` is intentional. It replaces the shell process with the service process. The supervisor watches the actual service process directly, not a shell wrapper.

On a modern systemd system, you almost certainly have `service` available. Run this to see it working and observe that it calls `systemctl` underneath:

```bash
# service is a compatibility wrapper on systemd systems
# Use strace to see what it actually calls
which service
```

```text
/usr/sbin/service
```

```bash
# Check what service is — on systemd systems it's a script, not a binary
file /usr/sbin/service
```

```text
/usr/sbin/service: POSIX shell script, ASCII text executable
```

```bash
# Open the script and look at the first few lines
head -20 /usr/sbin/service
```

```text
#!/bin/sh
# This is a script that provides compatibility between the SysV init
# and systemd.
# The script is part of the initscripts package.
...
```

```bash
# Run service with strace to see it redirect to systemctl
strace -f service ssh status 2>&1 | grep -E 'execve|systemctl' | head -5
```

```text
execve("/usr/sbin/service", ["service", "ssh", "status"], 0x7fff...) = 0
[pid 12345] execve("/usr/bin/systemctl", ["systemctl", "status", "ssh.service"], ...) = 0
```

The `service ssh status` call becomes `systemctl status ssh.service`. The wrapper exists for backwards compatibility with scripts and runbooks written before systemd. It works — but it hides what's actually happening. If you're on a systemd system, use `systemctl` directly. You get better output, more control, and you know exactly what you're doing.

If you're on an Alpine Linux system with OpenRC, try the OpenRC equivalents:

```bash
# On Alpine — list all services and their current state
rc-status
```

```text
Runlevel: default
 sshd                          [  started  ]
 crond                         [  started  ]
 networking                    [ stopped   ]
```

Now try this on any system to confirm which init system you're on:

```bash
# Check PID 1 to identify the init system
ls -la /proc/1/exe
```

```text
lrwxrwxrwx 1 root root 0 Jun 15 10:00 /proc/1/exe -> /usr/lib/systemd/systemd
```

If you see `systemd`, you're on systemd. If you see `/sbin/init` pointing at a BusyBox or OpenRC binary, you're not.

## Real-World Example

You're handed access to a legacy RHEL 6 server. The application team says "the myapp service keeps stopping and we don't know why." This is a SysV init system. Here's how you navigate it.

```bash
# First: confirm you're on SysV and which runlevel you're in
runlevel
```

```text
N 3
```

```bash
# N means the previous runlevel (N = none, just booted)
# 3 means current runlevel is 3 (multi-user with networking, no GUI)

# List services in runlevel 3
chkconfig --list | grep ":3:on"
```

```text
httpd           0:off   1:off   2:off   3:on    4:on    5:on    6:off
myapp           0:off   1:off   2:off   3:on    4:on    5:on    6:off
sshd            0:off   1:off   2:on    3:on    4:on    5:on    6:off
```

```bash
# Check if myapp is currently running
service myapp status
```

```text
myapp is stopped
```

```bash
# The init script should tell us where it logs — check the script
grep -E 'log|LOG|LOGFILE' /etc/init.d/myapp
```

```text
LOGFILE=/var/log/myapp/myapp.log
```

```bash
# Look at recent log entries
tail -50 /var/log/myapp/myapp.log
```

```text
[2024-03-15 14:22:01] Starting myapp...
[2024-03-15 14:22:03] Connected to database at 10.0.1.15
[2024-03-15 14:22:05] Listening on :8080
[2024-03-15 14:23:41] ERROR: database connection lost: connection refused
[2024-03-15 14:23:41] Fatal error — exiting
```

The service crashed because the database went away. Under SysV init, when the service exits, nobody restarts it. It stays down until a human runs `service myapp start`. There is no `Restart=on-failure` equivalent. The only solution on SysV is to either add a cron job that monitors and restarts the service, or to add a wrapper loop in the init script itself.

```bash
# Start the service manually — check if the database is back first
ping -c 1 10.0.1.15
service myapp start
service myapp status
```

```text
myapp (pid 8823) is running...
```

```bash
# Make sure it's enabled to survive reboots — check both runlevels 3 and 5
chkconfig myapp on    # enables for runlevels 2, 3, 4, 5 by default
chkconfig --list myapp
```

```text
myapp           0:off   1:off   2:on    3:on    4:on    5:on    6:off
```

Now compare this with what the same workflow looks like under OpenRC on Alpine:

```bash
# On Alpine Linux — check service status
rc-service myapp status

# Start it
rc-service myapp start

# Enable at boot (add to default runlevel)
rc-update add myapp default

# Verify
rc-update show | grep myapp
```

```text
     myapp |      default
```

The commands are different but the mental model is the same: status, start, enable at boot, verify. What's different is that OpenRC has real dependency tracking — if `myapp` depends on a database service, OpenRC will enforce that ordering at boot.

## ⚠️ Gotchas

### `service nginx restart` Works on systemd — But It's a Trap

You're on a systemd system. You run `service nginx restart`. It works. The nginx process restarts. You feel fine.

```bash
service nginx restart
```

```text
 * Restarting nginx nginx                                                [ OK ]
```

The problem is what this hides. That command called `systemctl restart nginx.service` internally — but you didn't see the systemd output. If the restart failed, the `service` wrapper may show `[ FAILED ]` without showing you the underlying systemd error message. It also hides whether the unit has `Restart=on-failure` set, what the current unit state is, or whether there's a dependency issue.

```bash
# Use systemctl directly — you get real, structured output
systemctl restart nginx
systemctl status nginx
```

```text
● nginx.service - A high performance web server and a reverse proxy server
     Loaded: loaded (/lib/systemd/system/nginx.service; enabled; vendor preset: enabled)
     Active: active (running) since Mon 2024-03-15 14:30:01 UTC; 2s ago
    Process: 9021 ExecStartPre=/usr/sbin/nginx -t (code=exited, status=0/SUCCESS)
   Main PID: 9022 (nginx)
```

The `service` command exists so old scripts and runbooks don't break. Write new work using `systemctl` directly. The `service` wrapper is a backward compatibility shim, not a feature to use in 2024.

### SysV Services Don't Restart Automatically When They Crash

This is the most dangerous SysV misunderstanding in production. You set up an init script, confirm the service starts on boot, and call it done. The service crashes at 3am. It stays down. Nobody notices until users complain.

```bash
# On a SysV system — start the service
service myapp start
# Simulate a crash
kill -9 $(cat /var/run/myapp.pid)
# Check service status one minute later
service myapp status
```

```text
myapp is stopped
```

There is no equivalent to `Restart=on-failure`. The service is down and nothing will bring it back except a human or a separate monitoring tool. If you're maintaining a SysV system and you need automatic restart, you need to either add a monitoring cron job or a wrapper in the init script:

```bash
# In the init script start() function — a restart loop hack
# This is ugly but it's what SysV requires
while true; do
    /usr/bin/myapp
    echo "$(date): myapp exited, restarting in 5s" >> /var/log/myapp/myapp.log
    sleep 5
done &
```

This is one of the main reasons systemd exists. The SysV answer to "how do I auto-restart a crashed service" is "write your own supervisor" or "add a cron job." systemd's answer is one line in a unit file.

### Runlevel Numbers Are Not Portable — Don't Hardcode Them

If you're writing a chkconfig header for an init script, the numbers in the header matter:

```bash
# What the chkconfig header looks like in /etc/init.d/myapp
# chkconfig: 2345 20 80
```

On RHEL/Fedora: runlevel 3 is "server mode (no GUI)" and 5 is "desktop mode." On Debian/Ubuntu: runlevel 2 is the default multi-user mode and there is historically no differentiation between 2, 3, 4, and 5.

A script written for RHEL with `# chkconfig: 2345 20 80` will not behave the same on Debian. If you hardcode `chkconfig --level 3`, that means nothing on a Debian system using SysV. This non-portability is why systemd replaced runlevels with named targets (`multi-user.target`) that explicitly describe the state they represent, regardless of distribution.

### OpenRC Is Not SysV — Don't Treat Them Identically

Both OpenRC and SysV use `/etc/init.d/` for service scripts. Both use runlevels. It looks like the same thing.

```bash
# On Alpine (OpenRC), this fails
chkconfig myservice on
```

```text
-ash: chkconfig: not found
```

```bash
# OpenRC uses its own command
rc-update add myservice default
```

OpenRC has `rc-service` and `rc-update`. SysV systems have `service` and `chkconfig` (RHEL) or `update-rc.d` (Debian). The concepts overlap but the commands are different, the script format is different (OpenRC uses `openrc-run` functions, SysV uses raw `case` statements), and the dependency system is different.

The shared use of `/etc/init.d/` is the source of the confusion. Seeing that directory does not tell you which init system you're on.

## Distro Notes

| Distro Family | Init System | Service Commands | Notes |
|---|---|---|---|
| RHEL 6 / CentOS 6 | SysV + Upstart | `service`, `chkconfig` | Still encountered on legacy enterprise systems; no longer receives updates |
| RHEL 7–10 / Fedora / Rocky | systemd | `systemctl`, with `service` as wrapper | `service` and `chkconfig` call `systemctl` internally |
| Debian / Ubuntu (modern) | systemd | `systemctl`, with `service` as wrapper | `service` wrapper still ships for backward compat; `update-rc.d` still present for legacy scripts |
| Alpine Linux | OpenRC (on top of BusyBox init) | `rc-service`, `rc-update`, `rc-status` | Not SysV; actively maintained; default for Alpine containers |
| Gentoo | OpenRC (default) or systemd (user choice) | OpenRC: `rc-service`/`rc-update`; or `systemctl` | Only major distro that ships both and lets the user choose |
| Void Linux | runit | `sv`, `vsv` | `sv up <service>`, `sv down <service>`, `sv status <service>` |
| Artix Linux | runit, OpenRC, s6, or dinit (user choice at install) | Varies by chosen init | Multiple init systems officially supported |
| Embedded Linux | Often BusyBox init or SysV | Varies; often custom scripts | Many embedded devices never moved to systemd; custom init scripts are common |

## Alternatives & Comparisons

| Init System | Architecture | Auto-restart? | Parallel startup? | Actively used? |
|---|---|---|---|---|
| SysV init | Sequential, shell scripts, runlevels | No | No | Legacy only; RHEL 6, embedded |
| Upstart | Event-driven, job configs | Yes | Partial | Effectively dead; Ubuntu pre-2015, RHEL 6 |
| systemd | Dependency graph, unit files, socket activation | Yes (`Restart=`) | Yes | Dominant on desktop/server Linux |
| OpenRC | Dependency-based, shell scripts | Optional (needs stsupervision) | Yes (with parallel setting) | Active; Gentoo, Alpine |
| runit | Supervision tree, service directories | Yes (inherent) | Yes | Active; Void Linux, containers |
| s6 | Supervision tree, modular | Yes (inherent) | Yes | Active; Artix, container environments |

On a modern server or desktop, use whatever the distribution ships — which is almost certainly systemd. You don't choose an init system for a general-purpose Linux install the way you'd choose a text editor; the distribution made that choice, and fighting it creates more problems than it solves.

If you're designing a minimal container image where systemd's overhead is unwanted (it needs D-Bus, cgroup hierarchies, and a socket directory just to function properly), runit or s6 are the right tools. A supervision tree in a container that manages one or two services is elegant and lightweight. systemd in a container is a constant fight against the container runtime.

If you're on Gentoo and want something between SysV's simplicity and systemd's complexity, OpenRC is mature and well-documented.

## Trade-offs

**Where SysV / OpenRC shine:**
- SysV is simple enough to audit completely — the entire init system is shell scripts you can read start to finish
- OpenRC on Alpine gives you a minimal, auditable init layer with no D-Bus requirement, suitable for containers and embedded systems
- SysV and OpenRC have zero impact on system design when you're in a container that doesn't need PID 1 to manage multiple services

**Where SysV / OpenRC cost you:**
- SysV has no automatic service restart — a crashed service stays down indefinitely; this is not a minor issue, it is a production reliability failure
- SysV's sequential boot is a real performance penalty on systems with many services and modern SSDs
- SysV init scripts must handle their own logging, PID tracking, and restart logic — every implementation is slightly different and every one has bugs
- OpenRC is not universally supported; tooling, documentation, and community support are a fraction of what systemd has
- runit and s6 require learning an entirely different mental model (supervision trees vs. service scripts); the simplicity is real but the on-ramp is steep

**Performance note:** On a system with 30+ services, systemd's parallel startup is measurably faster than SysV's sequential approach — real-world boot time differences of 30–60 seconds on server hardware with rotating disks are documented. On modern NVMe SSDs, the difference narrows but remains meaningful.

## When to Use / When to Avoid

✅ **Use `service` (the wrapper) when:**
- You're writing a quick one-liner or following an older runbook and you just need a service restarted
- You're scripting something that needs to work on both SysV and systemd hosts (rare, but exists in old enterprise environments)

✅ **Use OpenRC when:**
- You're on Alpine or Gentoo — that's the init system; learn its commands
- You're building a minimal container and want simple, auditable service management without systemd

✅ **Use runit or s6 when:**
- You're building a Docker base image and need to run multiple processes with supervision
- You're on Void Linux — runit is the default; learn `sv`

❌ **Avoid writing new SysV init scripts when:**
- The target system is running systemd — write a unit file instead; it's two minutes of work and you get `Restart=on-failure` for free
- You need automatic service restart — SysV cannot provide this without a separate supervisor

❌ **Avoid using `service` as your primary tool on systemd systems when:**
- You're diagnosing a problem — the wrapper hides systemd's output, including failure reasons
- You're scripting production automation — use `systemctl` explicitly so the behavior is transparent

⚠️ **Common mistakes:**
- Assuming `chkconfig` or `update-rc.d` works on Alpine or Void — those tools are SysV-specific
- Grepping for `/etc/init.d/` to identify the init system — OpenRC also uses that directory
- Running SysV services in production without any external monitoring — if it crashes, it stays down
- Using `service` interactively on systemd systems and then wondering why you can't see the service's recent log output (use `journalctl -u <service>` instead)

## Veteran Perspective

The SysV init debate feels historical until you're on a RHEL 6 system at 2am trying to figure out why a critical service stopped and left no useful output anywhere. SysV's logging story is "the service wrote to a log file if it wanted to." Half the time, the service didn't want to.

The thing newcomers misunderstand about the init system wars is the framing. The argument was never really about whether shell scripts are bad. SysV works — in the same way that a single-lane dirt road works. The problem was that the computing environment changed: SSDs reduced I/O wait, multi-core CPUs made parallelism valuable, and production services became complex enough that "if a service crashes, someone will notice" stopped being acceptable. SysV was built for a world where a server booted twice a year and ran five services. That world is gone.

The thing intermediate engineers miss: `service myapp restart` and `systemctl restart myapp` are not the same command. On a systemd system the first calls the second, which means if systemctl returns an error, the service wrapper may not faithfully propagate it. More importantly, if you use `service` to restart something and then `journalctl -u myapp` doesn't show you what you expected, it's because the abstraction layer swallowed the context. Debug with `systemctl` and check logs with `journalctl`. Use `service` only when talking to systems that predate systemd.

The subtler production mistake I've seen repeatedly: teams migrate from RHEL 6 to RHEL 7+ and keep their monitoring scripts using `service <name> status` because "it still works." It does work — until it doesn't, because the exit codes from the `service` wrapper are slightly different from `systemctl`, and monitoring systems that rely on exact exit codes start misreporting service health. The `service` wrapper is a shim, not a contract.

One more thing that matters if you ever touch Alpine containers in production: OpenRC's parallel startup doesn't enable by default. Alpine boots services sequentially unless you add `rc_parallel="YES"` to `/etc/rc.conf`. A container that takes twelve seconds to be "ready" when it should take two usually has this missing. It's one line.

## Try It Yourself

1. Check which init system is running on your machine: examine PID 1 with `ls -la /proc/1/exe` or `stat /proc/1/exe`. Then confirm with `ps -p 1 -o comm=`.

2. On a systemd system, run `service ssh status` (or `service sshd status` on RHEL-family). Then run `systemctl status ssh` (or `sshd`). Compare the output — note what the `service` wrapper omits.

3. Look at a real SysV init script: on any Linux system with SysV compatibility installed, read `/etc/init.d/skeleton` (Debian) or any script in `/etc/init.d/` that ships with the OS. Find the `case "$1"` block and trace the `start` code path. Note where it writes a PID file and whether it handles a pre-existing PID file.

4. **On an Alpine Linux VM or container:** run `rc-status` to see the current runlevel and service states. Run `rc-update show` to see which runlevel each service belongs to. Try `rc-service crond stop` and `rc-service crond start`. Compare the commands with their systemctl equivalents.

5. **Challenge:** Set up a Docker container using `phusion/baseimage` (which uses runit) or build your own Alpine container with two services managed by OpenRC. Write a startup script, add it to the appropriate runlevel or supervision directory, and verify that if you `kill` the process, the supervisor restarts it automatically. *(Requires Docker and a VM; killing processes and watching them restart is the whole point.)*

## Summary

- SysV init starts services sequentially using shell scripts in `/etc/init.d/` and symlinks in `/etc/rc[N].d/`; it has no automatic service restart and no real dependency management — if a service crashes, it stays down until a human acts.
- `service <name> <action>` on a modern systemd system is a compatibility wrapper that calls `systemctl` internally — it works, but use `systemctl` directly for diagnostics, automation, and anything where the output matters.
- OpenRC is not SysV: it is a dependency-based init system used by Gentoo and Alpine Linux with real dependency declarations, parallel startup support, and active maintenance; its commands are `rc-service` and `rc-update`.
- runit and s6 are supervision-tree init systems used by Void Linux and in containers — every service gets a persistent supervisor that restarts it immediately on crash, making "automatic restart" structural rather than configured.
- Runlevel numbers are not portable across distributions; systemd replaced them with named targets (`multi-user.target`) that mean the same thing everywhere.
