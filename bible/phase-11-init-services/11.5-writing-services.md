# [11.5] — Writing systemd Service Units

## The Problem This Solves

Before systemd, running a custom daemon meant writing an init script in bash — typically a 150-line mess in `/etc/init.d/` that had to re-implement process tracking, pid files, start/stop/restart/status logic, and dependency ordering from scratch. Every script was slightly different. Half of them had race conditions. The "status" command on many SysV scripts returned exit code 0 whether the service was running or not. And when a daemon crashed, the only way to know was to check the log.

None of that is hyperbole. Most real `/etc/init.d/` scripts were copied from an example, modified without understanding, and had subtle bugs around pid file handling and cleanup that would cause double-starts or orphaned processes. When you needed to add a dependency — "don't start until PostgreSQL is ready" — you had to modify every dependent script and hope the ordering was honored.

The question a new sysadmin faces is: I have an application. I want it to start on boot, restart if it crashes, run as a non-root user, and have its logs captured. How?

systemd unit files answer that question cleanly. A `.service` file is a declarative description of what your daemon is and how it should behave. You do not write startup logic — you describe what you want, and systemd implements it.

---

## Mental Model

A systemd service file is a rental agreement for a piece of the server. The landlord (systemd) owns the property (the system's boot sequence, process table, and resource limits). You, the tenant, fill out a form describing what you need: which apartment (target), which utilities (dependencies), what the lease covers (restart policy), and what you're allowed to do (sandboxing). You don't write code to manage your tenancy — you fill out the form correctly and systemd handles enforcement.

If the form has a typo or a missing clause, the landlord doesn't reject it — the apartment either doesn't get opened correctly, or it opens in a way you didn't expect. The burden of correctness is on the form.

## Theory

A systemd service unit is a plain text file with an `.service` suffix, divided into three sections: `[Unit]`, `[Service]`, and `[Install]`. Each section controls a different aspect of the service's lifecycle.

**`[Unit]`** describes the unit itself and its relationship to other units. It contains human-readable metadata (description, documentation URL) and dependency declarations (`Requires=`, `After=`, `Wants=`). Dependency declarations are what tell systemd which order to respect during boot and shutdown.

**`[Service]`** describes the runtime behavior: what binary to run, what type of service it is, how to detect that it started, how to restart it, what user to run as, and what sandbox restrictions to apply. This is where most of the configuration lives.

**`[Install]`** describes what happens when the unit is enabled or disabled via `systemctl enable`. It is not used at runtime — it only controls which symlinks `systemctl enable` creates. If `[Install]` is missing or `WantedBy=` is wrong, `systemctl enable` will succeed silently but the service will never start on boot.

### Service Types

The `Type=` directive tells systemd how to determine whether a service has finished starting. This is more important than it sounds: if systemd gets it wrong, dependent services may start before the daemon is ready.

**`Type=simple`** (default) — systemd considers the service started the moment `fork()` returns successfully after launching the `ExecStart=` process. It does not wait for `exec()` to succeed. This means that if your binary doesn't exist or has wrong permissions, systemd will mark the service as started even as the exec fails. The failure only becomes visible a moment later when the process dies. This is the most common type but also the most misleading.

**`Type=exec`** — added in systemd 240 (December 2018). Like `simple`, but systemd waits for both `fork()` and `exec()` to complete before marking the service as started. If the binary doesn't exist, the service fails immediately and cleanly rather than appearing to start and then failing. On any system running systemd 240 or newer — which is most systems as of 2024 — prefer `Type=exec` over `Type=simple` for any service that doesn't use `sd_notify`.

**`Type=forking`** — systemd expects the `ExecStart=` process to double-fork and exit (the classic Unix daemon pattern). The parent exits; the child is the actual daemon. systemd tracks the daemon via a PID file specified in `PIDFile=`. Avoid this for new services — it requires correct PID file management, which is an entire category of bugs waiting to happen.

**`Type=oneshot`** — used for services that run once and exit. Common for scripts that set up system state at boot. By default, systemd waits for the process to finish before proceeding. Combine with `RemainAfterExit=yes` if you want the service to appear as "active" after the script finishes.

**`Type=notify`** — the daemon calls `sd_notify(3)` to explicitly tell systemd when it's ready. This is the most correct type for complex daemons that have a real "ready" state (e.g., they've bound a port, connected to a database, or loaded their configuration). systemd will not start dependent services until it receives the `READY=1` notification. If you're writing a new daemon and it has any initialization work before it's usable, implementing `sd_notify` and using `Type=notify` is the right answer.

### Restart Behavior

`Restart=on-failure` tells systemd to restart the service if the process exits with a non-zero exit code, is killed by a signal, or times out. It does not restart the service if it exits cleanly (exit code 0). For a long-running daemon that should always be up, `Restart=on-failure` is the correct choice.

`RestartSec=` controls the delay between the crash and the restart attempt. The default is 100ms — almost no delay. Setting `RestartSec=5` introduces a 5-second pause, which prevents a tight crash loop from hammering the system or generating a wall of log spam. In practice, `RestartSec=5` is a reasonable default for most services.

For production deployments where you need to cap the number of restart attempts to prevent infinite loops on persistent failures, use `StartLimitBurst=` and `StartLimitIntervalSec=`. A common production configuration: allow 3 restarts within 60 seconds before giving up.

### Dependencies

`After=network-online.target` means this service starts after the network is online — not just after the network is configured (that's `network.target`). Use `network-online.target` for services that actually need to make outbound connections at startup. The difference matters: `network.target` is reached when the network interfaces are configured; `network-online.target` is reached when at least one interface reports it's fully up.

`Wants=` is a soft dependency: if the wanted unit fails to start, this unit starts anyway. `Requires=` is hard: if the required unit fails, this unit fails too. Default to `Wants=` unless a failure in the dependency genuinely makes your service non-functional.

## How It Works (Mechanics)

When you run `systemctl start myapp.service`, systemd looks up the unit file in `/etc/systemd/system/` (or `/lib/systemd/system/` for vendor-installed units). It parses the file and builds an activation job.

For a `Type=exec` service, the activation sequence is:

```text
# Conceptual execution trace — not real code
systemd reads unit file
  → resolves dependencies (checks After=, Requires=, Wants=)
  → applies sandboxing namespaces (PrivateTmp, ProtectSystem, etc.)
  → sets up cgroup for the service (for resource tracking and SIGKILL on stop)
  → forks a child process
    → child: drops to User=/Group= if specified
    → child: sets WorkingDirectory=
    → child: loads environment from Environment= and EnvironmentFile=
    → child: calls execve() with ExecStart= binary
  → if Type=exec: waits for execve() to complete
  → if Type=notify: waits for sd_notify(READY=1) from the process
  → marks service as active
  → starts any units that have After=myapp.service
```

The cgroup assignment is important: it means systemd tracks every process spawned by the service, not just the PID of the direct child. When you run `systemctl stop`, systemd sends SIGTERM to all processes in the cgroup, waits `TimeoutStopSec` seconds, then sends SIGKILL to any survivors. There is no way for a child process to escape cleanup by double-forking — the cgroup contains it regardless.

Drop-in overrides work by placing `.conf` files in a directory named `unit.d/` adjacent to the unit file. For vendor-supplied units in `/lib/systemd/system/`, the override directory is `/etc/systemd/system/myapp.service.d/`. systemd merges all files in that directory into the unit definition, with later files overriding earlier ones. This lets you modify a vendor-provided unit without touching the original — so package upgrades can safely update the base unit without clobbering your customizations.

## Basic Example

This is a complete, minimal service file for a hypothetical application. Save it to `/etc/systemd/system/myapp.service`, then reload and start:

```bash
# Create the unit file
sudo tee /etc/systemd/system/myapp.service > /dev/null << 'EOF'
[Unit]
Description=My Application
After=network.target

[Service]
Type=exec
ExecStart=/usr/local/bin/myapp
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

# Tell systemd to re-read unit files from disk
sudo systemctl daemon-reload

# Enable (creates symlink so it starts on boot) and start
sudo systemctl enable --now myapp.service
```

```text
Created symlink /etc/systemd/system/multi-user.target.wants/myapp.service → /etc/systemd/system/myapp.service.
```

```bash
# Check the service status
sudo systemctl status myapp.service
```

```text
● myapp.service - My Application
     Loaded: loaded (/etc/systemd/system/myapp.service; enabled; vendor preset: enabled)
     Active: active (running) since Mon 2024-01-15 10:23:04 UTC; 5s ago
   Main PID: 12345 (myapp)
      Tasks: 1 (limit: 4915)
     Memory: 3.2M
        CPU: 12ms
     CGroup: /system.slice/myapp.service
             └─12345 /usr/local/bin/myapp
```

The `Loaded:` line shows the file path, whether it's enabled, and the vendor preset. The `Active:` line is what you check first — if it says `active (running)`, the process is alive. The `CGroup:` block shows every process systemd is tracking for this service.

Try this yourself with a real binary — replace `/usr/local/bin/myapp` with `/usr/bin/sleep` and use `ExecStart=/usr/bin/sleep 3600` to create a long-running dummy service. Run `systemctl status`, then `systemctl stop`, then watch the status change. If you see `activating` briefly before `active`, that's normal — that's the moment between fork and exec.

## Real-World Example

This is a production-grade service file for a Node.js web application running as a dedicated non-root user, with environment configuration loaded from a separate file, sandbox restrictions applied, and a sensible restart policy.

First, the setup — create the system user and place the binary:

```bash
# Create a dedicated system user with no login shell and no home directory
sudo useradd --system --no-create-home --shell /bin/false nodeapp

# Place the application binary (in a real deployment, this would be your CI/CD step)
sudo mkdir -p /opt/nodeapp
sudo chown nodeapp:nodeapp /opt/nodeapp

# Create an environment file with secrets (never put secrets in the unit file itself)
sudo mkdir -p /etc/nodeapp
sudo tee /etc/nodeapp/env > /dev/null << 'EOF'
NODE_ENV=production
PORT=3000
DATABASE_URL=postgres://nodeapp:secret@localhost/appdb
EOF
sudo chown root:nodeapp /etc/nodeapp/env
sudo chmod 640 /etc/nodeapp/env   # owner root, group nodeapp, no world access
```

Now the unit file:

```ini
[Unit]
Description=Node.js Application Server
Documentation=https://github.com/myorg/nodeapp
After=network-online.target postgresql.service
Wants=network-online.target postgresql.service

[Service]
Type=exec
User=nodeapp
Group=nodeapp
WorkingDirectory=/opt/nodeapp

# Load environment variables from file; the leading dash makes it optional
# (service won't fail if the file is missing — useful during initial setup)
EnvironmentFile=-/etc/nodeapp/env

# Absolute path required — ExecStart=node will fail with "not found"
ExecStart=/usr/bin/node /opt/nodeapp/server.js
ExecReload=/bin/kill -HUP $MAINPID

# Restart on crash, but not if we stopped it cleanly (exit code 0)
Restart=on-failure
RestartSec=5
StartLimitBurst=3
StartLimitIntervalSec=60

# Sandbox directives — harden the service against exploits
PrivateTmp=yes            # give the service its own /tmp, invisible to other services
ProtectSystem=strict      # mount the filesystem read-only (except /proc, /dev, /run)
ProtectHome=yes           # make /home, /root, and /run/user inaccessible
NoNewPrivileges=yes       # prevent setuid escalation even if the binary supports it
ReadWritePaths=/opt/nodeapp/data  # exception: this path stays writable

# Standard output/error goes to the journal automatically
StandardOutput=journal
StandardError=journal
SyslogIdentifier=nodeapp

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now nodeapp.service
sudo systemctl status nodeapp.service
```

```text
● nodeapp.service - Node.js Application Server
     Loaded: loaded (/etc/systemd/system/nodeapp.service; enabled; vendor preset: enabled)
     Active: active (running) since Mon 2024-01-15 10:30:00 UTC; 3s ago
   Main PID: 13211 (node)
      Tasks: 7 (limit: 4915)
     Memory: 52.1M
        CPU: 0.342s
     CGroup: /system.slice/nodeapp.service
             └─13211 /usr/bin/node /opt/nodeapp/server.js
```

**Walkthrough of the key decisions:**

`After=network-online.target postgresql.service` plus `Wants=` — the application needs both the network and PostgreSQL before it can serve requests. `Wants=` rather than `Requires=` means that if PostgreSQL isn't installed or fails to start, this service still attempts to start rather than failing immediately. If your application truly cannot function without PostgreSQL, change `Wants=` to `Requires=`.

`EnvironmentFile=-/etc/nodeapp/env` — the leading dash is critical. Without it, a missing file is a startup failure. With it, a missing file is silently ignored and the service starts with only the environment variables systemd provides. The dash prefix means "this is optional."

`ExecReload=/bin/kill -HUP $MAINPID` — this wires up `systemctl reload` to send SIGHUP to the main process, which by convention causes a graceful configuration reload. `$MAINPID` is a special variable systemd expands to the PID of the main service process.

`ProtectSystem=strict` — this is the most aggressive level. It mounts the entire filesystem as read-only, with exceptions for `/proc`, `/dev`, and `/run`. This means the service can read system files but cannot write to `/etc`, `/usr`, `/var`, or anything else outside of what you explicitly allow with `ReadWritePaths=`. If your application tries to write a file outside those paths, it gets `EROFS` (read-only filesystem), not a permission error.

`StartLimitBurst=3` and `StartLimitIntervalSec=60` — this caps restart attempts at 3 within 60 seconds. If the service crashes 3 times in a minute, systemd stops trying and marks it as `failed`. This prevents a service with a persistent configuration error from consuming all system resources in a restart loop. To reset it manually: `systemctl reset-failed nodeapp.service`.

### Drop-in Override Example

Suppose a vendor package installs a service to `/lib/systemd/system/nginx.service` and you need to change `LimitNOFILE` and add an environment variable without modifying the package-managed file:

```bash
# This command opens an editor for the override file and creates the directory
sudo systemctl edit nginx.service
```

```ini
# Write only the directives you want to change — not the entire file
[Service]
LimitNOFILE=65536
Environment="EXTRA_OPTS=--with-debug"
```

```bash
# Verify the merge looks right
sudo systemctl cat nginx.service
```

```text
# /lib/systemd/system/nginx.service
[Unit]
Description=A high performance web server and a reverse proxy server
...

# /etc/systemd/system/nginx.service.d/override.conf
[Service]
LimitNOFILE=65536
Environment="EXTRA_OPTS=--with-debug"
```

`systemctl cat` shows the unit as systemd sees it after merging: the base file first, then each override file. The merged result is what actually runs. You do not need to copy the entire unit file into the override — only the directives you want to change.

### Timer Unit Example

A systemd timer replaces cron for system-level periodic tasks. It consists of two unit files: a `.service` (the work to do) and a `.timer` (the schedule).

```ini
# /etc/systemd/system/backup.service
[Unit]
Description=Backup Application Data
After=network-online.target

[Service]
Type=oneshot
User=backup
ExecStart=/usr/local/bin/backup.sh
StandardOutput=journal
StandardError=journal
```

```ini
# /etc/systemd/system/backup.timer
[Unit]
Description=Daily Backup Timer
Requires=backup.service

[Timer]
# Run at 02:30 every day
OnCalendar=*-*-* 02:30:00
# If the system was off at 02:30, run the missed job when it boots
Persistent=true
Unit=backup.service

[Install]
WantedBy=timers.target
```

```bash
# Enable and start the timer (not the service — the timer manages the service)
sudo systemctl enable --now backup.timer

# Verify the timer is scheduled
systemctl list-timers backup.timer
```

```text
NEXT                        LEFT     LAST                        PASSED  UNIT           ACTIVATES
Tue 2024-01-16 02:30:00 UTC 14h left Mon 2024-01-15 02:30:01 UTC 9h ago  backup.timer   backup.service
```

`Persistent=true` is what separates systemd timers from cron for servers that are not always on. If the machine was powered off at 2:30am, the backup runs immediately on next boot rather than being silently skipped until the next scheduled time.

### Path Unit Example

A path unit watches the filesystem and activates a service when something changes. Useful for processing files that are dropped into a directory:

```ini
# /etc/systemd/system/process-uploads.path
[Unit]
Description=Watch for new uploads

[Path]
# Activate when this path exists (useful for one-shot triggers)
PathExists=/var/spool/uploads/trigger.flag
# Or activate whenever any file in a directory changes:
# PathChanged=/var/spool/uploads/incoming/

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl enable --now process-uploads.path
```

When `/var/spool/uploads/trigger.flag` appears, systemd automatically starts `process-uploads.service`. When the service finishes, the path unit goes back to watching. Use `PathChanged=` to trigger on any file modification in a directory.

## ⚠️ Gotchas

### `Type=simple` Hides exec() Failures

You set up a service, run `systemctl start`, and it reports success. Then a moment later, `systemctl status` shows it as failed. The timeline looks like: started → immediately failed.

```bash
sudo systemctl start myapp.service
sudo systemctl status myapp.service
```

```text
● myapp.service - My Application
     Active: failed (Result: start-limit-hit)

Jan 15 10:00:01 host systemd[1]: myapp.service: Main process exited, code=exited, status=203/EXEC
Jan 15 10:00:01 host systemd[1]: myapp.service: Failed with result 'exit-code'.
```

Status 203/EXEC means `execve()` failed — the binary doesn't exist, has wrong permissions, or has a missing shared library. With `Type=simple`, systemd marks the service started immediately after `fork()`, then discovers the exec failure when the child process dies. The service appears to start, then fails.

With `Type=exec`, systemd waits for `exec()` to complete and the failure is reported immediately and cleanly:

```text
Jan 15 10:00:01 host systemd[1]: myapp.service: Failed to execute /usr/local/bin/myapp: No such file or directory
Jan 15 10:00:01 host systemd[1]: myapp.service: Failed at step EXEC spawning /usr/local/bin/myapp: No such file or directory
```

If you're on systemd 240 or newer (check with `systemctl --version`), default to `Type=exec`. The feedback loop is faster and the error message is clearer.

### `ExecStart=` Requires an Absolute Path

This is one of the most common mistakes writing a first service file:

```ini
# This will fail — relative path
[Service]
ExecStart=myapp --config /etc/myapp.conf
```

```text
myapp.service: Failed to locate executable myapp: No such file or directory
```

`ExecStart=` does not search `$PATH`. The path must be absolute. systemd does not run in a user shell context — there is no shell, no `$PATH` lookup, and no current directory unless you set `WorkingDirectory=`:

```ini
# Correct
[Service]
ExecStart=/usr/local/bin/myapp --config /etc/myapp.conf
```

If you're not sure where the binary lives: `which myapp` or `command -v myapp`.

### Missing `[Install]` Means `enable` Does Nothing

You write a unit file, run `systemctl enable myapp.service`, and it reports success. The server reboots, and the service never starts.

```bash
sudo systemctl enable myapp.service
```

```text
# If [Install] section is missing or WantedBy= is empty:
# (no output — the command succeeds silently without creating any symlinks)
```

```bash
# Check whether the symlink was actually created
systemctl is-enabled myapp.service
```

```text
static
```

`static` means the unit can be used as a dependency but has no `[Install]` section — `enable` has nothing to act on. The service will never automatically start on boot unless something else pulls it in.

The fix: add the `[Install]` section with `WantedBy=multi-user.target` (or the correct target for your service's startup context — `graphical.target` for GUI-dependent services, `timers.target` for timers):

```ini
[Install]
WantedBy=multi-user.target
```

Then reload and enable:

```bash
sudo systemctl daemon-reload
sudo systemctl enable myapp.service
systemctl is-enabled myapp.service
```

```text
enabled
```

### `EnvironmentFile=` Fails If the File Is Missing

If you specify `EnvironmentFile=/etc/myapp/env` without the dash prefix, the service will fail to start if the file doesn't exist — even if the environment variables in it are all optional:

```text
myapp.service: Failed to load environment files: No such file or directory
myapp.service: Failed with result 'resources'.
```

Prefix with `-` to make the file optional:

```ini
[Service]
EnvironmentFile=-/etc/myapp/env
```

The dash means "if this file doesn't exist, that's fine — continue." This is especially useful during first deployment, when the environment file might not be created until after the package is installed.

### `ProtectSystem=strict` Breaks Services That Write to Standard Paths

`ProtectSystem=strict` is aggressive — it makes the entire filesystem read-only except `/proc`, `/dev`, and `/run`. If your service writes log files to `/var/log/myapp/`, writes a pid file to `/var/run/`, or creates a socket in `/tmp/`, those writes will fail silently or with a permission error.

The fix is to explicitly grant write access to the paths your service needs:

```ini
[Service]
ProtectSystem=strict
ReadWritePaths=/var/log/myapp /var/lib/myapp   # grant write access to specific paths
RuntimeDirectory=myapp                          # creates /run/myapp, writable, cleaned on stop
StateDirectory=myapp                            # creates /var/lib/myapp, writable, persisted
LogsDirectory=myapp                             # creates /var/log/myapp, writable
```

`RuntimeDirectory=`, `StateDirectory=`, and `LogsDirectory=` are systemd-managed directories that sidestep the need to grant raw `ReadWritePaths=` access. They're created with the correct ownership automatically, and `RuntimeDirectory=` is cleaned up when the service stops. Prefer these over raw `ReadWritePaths=` wherever possible.

### `WantedBy=` Is Not the Same as Starting the Service

`WantedBy=multi-user.target` in `[Install]` does exactly one thing: when you run `systemctl enable`, it creates a symlink in `/etc/systemd/system/multi-user.target.wants/`. That symlink is what causes the service to be activated when `multi-user.target` is reached during boot.

`systemctl enable` does not start the service right now. Use `--now` or run `systemctl start` separately:

```bash
sudo systemctl enable --now myapp.service   # enable AND start
# vs
sudo systemctl enable myapp.service         # enable for next boot only
```

## Distro Notes

| Distro Family | Notes |
|---|---|
| Debian / Ubuntu | Unit files land in `/lib/systemd/system/` for packages; systemd 232+ shipped since Ubuntu 16.04 / Debian Stretch; `Type=exec` available from Ubuntu 20.04 / Debian Buster (systemd 241+) |
| RHEL / Fedora / CentOS / Rocky | Unit files in `/usr/lib/systemd/system/`; RHEL 9 / Fedora 37+ ship systemd 251+; `ProtectSystem=strict` may require SELinux policy adjustments on RHEL |
| Arch | Rolling release; typically ships a very recent systemd; `/usr/lib/systemd/system/` for packages, `/etc/systemd/system/` for local overrides |
| Alpine (containers) | Uses OpenRC by default, not systemd; if a container runs systemd as PID 1, it requires a specially configured image — Alpine does not support this out of the box |

On RHEL/CentOS with SELinux enforcing, adding `ProtectSystem=strict` or `PrivateTmp=yes` can trigger AVC denials if the service accesses paths that SELinux labels as system-managed. Check `ausearch -m avc -ts recent` and use `audit2allow` to generate policy if needed — but understand the policy before applying it.

## Alternatives & Comparisons

| Approach | When to Use | Notes |
|---|---|---|
| systemd unit file | Any daemon on a modern Linux system (2015+) | The standard; supports all features discussed in this topic |
| SysV init script (`/etc/init.d/`) | Legacy systems only; maintaining existing scripts | Still supported on systemd systems via compatibility layer; no new scripts should be written this way |
| Supervisor (`supervisord`) | Running daemons under a non-root user in environments where systemd is unavailable (some containers) | Python-based; per-process configuration; lacks native systemd integration |
| Docker / container | Services that need full isolation or multi-process environments | Often combined with systemd (`Type=notify` for `podman` or `docker` managed by systemd) |
| cron | Simple periodic tasks that don't need dependency management | For anything that needs `After=`, `Persistent=`, or journal logging, a systemd timer is strictly better |

For any daemon running on a Linux system with systemd — which is essentially all modern Linux servers — write a systemd unit file. The argument for `supervisord` only holds in environments where systemd isn't available or isn't PID 1 (some containers). For periodic tasks, systemd timers handle missed-run recovery that cron cannot.

## Trade-offs

**Where this shines:**

- Dependency ordering, restart policy, resource limits, and sandboxing are all declarative — no code to write, no race conditions to introduce
- cgroup-based process tracking means every child process the daemon spawns is captured; `systemctl stop` cleans up the entire tree, not just the main PID
- Drop-in overrides let package maintainers and sysadmins coexist: packages update their base unit files; your overrides in `.d/` directories survive upgrades untouched
- Sandboxing directives like `PrivateTmp=` and `ProtectSystem=strict` confine a compromised daemon to a limited view of the filesystem with zero code changes to the daemon itself

**Where this costs you:**

- Unit file syntax has real subtleties — the `Type=` selection in particular has non-obvious implications for when dependent services start; getting it wrong produces silent startup ordering failures, not error messages
- `ProtectSystem=strict` and the full sandbox suite require careful mapping of which paths the service actually writes to; overly aggressive sandboxing can cause confusing write failures
- systemd is Linux-only; if your application or deployment scripts need to support macOS, BSD, or containers without systemd, a unit file is not portable

**Performance note:** `PrivateTmp=yes` and namespace-based sandboxing directives create new Linux namespaces per service. Namespace creation has measurable overhead on services that start and stop very frequently (e.g., `Type=oneshot` services run hundreds of times per second). For services that start once at boot and run continuously, this overhead is immeasurable.

## When to Use / When to Avoid

✅ **Use this when:**

- You have any long-running daemon that needs to survive crashes, start on boot, or run as a specific user — write a unit file; there is no better mechanism on a systemd system
- You need a service to start after another specific service — `After=` handles this reliably in ways cron and manual scripts cannot
- You want to harden a third-party application without modifying its code — add `PrivateTmp=yes`, `NoNewPrivileges=yes`, and `ProtectSystem=strict` in a drop-in override
- You need reliable periodic task execution that handles missed runs — use a systemd timer with `Persistent=true` instead of cron

❌ **Avoid this when:**

- You need to run a service as PID 1 inside a container — systemd is not designed to be PID 1 in a typical container; use a dedicated container init like `tini` instead
- The task is a user-level daemon that should only run when a specific user is logged in — use `systemctl --user` and user units in `~/.config/systemd/user/` instead of system units

⚠️ **Common mistakes:**

- Writing `ExecStart=/usr/bin/bash -c 'myapp --config /etc/myapp.conf'` to wrap the command in a shell — this makes the shell PID 1 of the service, not the actual application, which breaks `ExecReload=` and makes process tracking unreliable; use the binary directly
- Setting `User=root` explicitly — if your application genuinely needs root, leave `User=` out; but audit whether it actually needs root or just needs a specific capability; use `AmbientCapabilities=` to grant individual capabilities instead of full root
- Using `After=postgresql.service` without verifying PostgreSQL is actually ready — `After=` means "start after the unit is active" but doesn't mean the database inside PostgreSQL has finished recovery and is accepting connections; for true readiness, implement a readiness probe in the application or use a wrapper script that polls the database

## Veteran Perspective

The single thing that has caused the most service startup failures I've debugged is getting `Type=` wrong, and not in the obvious direction. People think the failure mode is "service doesn't start" — but the real failure mode is "service appears to start, and the dependent service gets started, and the dependent service fails in a confusing way fifteen seconds later because the upstream wasn't actually ready."

If you have `Type=simple` and your daemon has any initialization time — loading a model, connecting to a database, binding a socket — all the downstream services start while your daemon is mid-initialization. They probe it, it's not ready, they fail, they get restarted, maybe they succeed, maybe the whole dependency chain enters a cascade of restarts. This is the source of those morning incidents where the server "took a long time to come up after rebooting." The answer is almost always a missing `Type=notify` and missing `sd_notify` integration.

The fix is either: add `sd_notify` support to your daemon and use `Type=notify`, or use `Type=exec` plus application-level retry logic in your clients. The worst answer is `Type=simple` plus a `sleep 5` in `ExecStartPost=` — I have seen this in production unit files and it is a fragile lie.

On sandboxing: the gap between "I added `PrivateTmp=yes`" and "I actually checked what the service can access" is enormous. `systemd-analyze security myapp.service` gives you a security score and a breakdown of which directives are and aren't set. Run it on your unit file. You'll see that most default services score around 7–9 out of 10 on the exposure scale — meaning they're largely unsandboxed. Adding just `PrivateTmp=yes`, `ProtectSystem=strict`, `NoNewPrivileges=yes`, and `ProtectHome=yes` typically drops a service from 9 to 4. Four lines of config. That's a meaningful reduction in blast radius if the service is compromised.

The `WantedBy=` gotcha has burned every sysadmin I know at least once. You `systemctl enable` the service. You see "Created symlink..." and assume all is well. The server reboots. The service doesn't start. You spend twenty minutes checking the service logic before someone points out there's no `[Install]` section. The lesson: always verify with `systemctl is-enabled myapp.service` after enabling. `enabled` means good. `static`, `disabled`, or `generated` means something is wrong with your `[Install]` section.

Drop-in overrides are underused. The pattern of copying a package-managed unit file from `/lib/systemd/system/` to `/etc/systemd/system/` and modifying it in place is a maintenance trap — the package will try to update `/lib/systemd/system/` on the next upgrade and your copy in `/etc/` will mask the update silently. Use `systemctl edit` to create proper overrides. That's what it exists for.

## Try It Yourself

1. Check the systemd version on your system with `systemctl --version`. Confirm whether `Type=exec` is available (requires systemd 240+). Look at what version shipped with your distribution.

2. Create a unit file that wraps `sleep 3600` — just enough to have a real process. Give it `User=nobody`, `PrivateTmp=yes`, and `NoNewPrivileges=yes`. Start it, run `systemctl status`, and inspect the cgroup tree shown in the output. Then stop it and verify the process is gone.

3. Take the service you created in step 2 and run `systemd-analyze security <yourservice>.service`. Look at the security score. Add `ProtectSystem=strict`, `ProtectHome=yes`, and `ProtectKernelTunables=yes`. Re-run the analysis. Note how the score changes.

4. Create a drop-in override for an existing vendor-managed service (e.g., `ssh.service` or `cron.service`) using `sudo systemctl edit`. Add `Environment="MY_VAR=hello"`. Run `systemctl cat` on the service and verify the override appears in the merged output. Run `systemctl daemon-reload` and confirm the override applies.

5. **Challenge:** Write a complete systemd timer that runs a backup script every day at 3am, handles the case where the system was off at 3am (using `Persistent=true`), runs as a dedicated `backup` system user, logs to the journal, and uses `ProtectSystem=strict` with appropriate `ReadWritePaths=` for the backup destination. Enable only the `.timer` unit (not the `.service`), verify it appears in `systemctl list-timers`, and manually trigger it with `systemctl start backup.service` to confirm it runs correctly before waiting for the scheduled time. Do this in a VM.

## Summary

- A service file has three sections: `[Unit]` (metadata and dependencies), `[Service]` (runtime behavior), and `[Install]` (what `systemctl enable` creates) — `[Install]` is not used at runtime, only by `enable`/`disable`
- `Type=exec` (available since systemd 240) is a strict improvement over `Type=simple` for most services: it waits for `execve()` to succeed before marking the service started, giving immediate and accurate failure feedback instead of a delayed crash
- `ExecStart=` requires an absolute path — no shell, no `$PATH` lookup; prefix `EnvironmentFile=` with `-` to make it optional and prevent startup failures when the file doesn't exist yet
- `PrivateTmp=yes`, `ProtectSystem=strict`, `ProtectHome=yes`, and `NoNewPrivileges=yes` together meaningfully sandbox a service against filesystem access and privilege escalation with four lines of config; run `systemd-analyze security <unit>` to audit the score
- Use `systemctl edit unit.service` — not a direct copy of the vendor file — to write drop-in overrides; this keeps package upgrades clean and your customizations intact
