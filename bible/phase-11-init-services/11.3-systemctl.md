## The Problem This Solves

Before systemd, every Linux distribution had its own init system — SysV init, Upstart, OpenRC — and each had its own tools and conventions for starting, stopping, and enabling services. On SysV systems, you used `service nginx start` and `chkconfig nginx on` as two completely separate commands. On Upstart, you used `start nginx` and `update-rc.d`. On OpenRC, different again. If you moved between distros, you were relearning the same concepts in a different dialect every time.

Worse, the old tools gave you almost no visibility. `service nginx status` would tell you "running" or "not running" — and nothing else. Which process ID? How long has it been up? Did it fail and restart? How many times? What did it print to stderr before dying? You had no idea without digging through log files manually.

Enabling a service at boot on SysV meant creating or modifying runlevel symlinks in `/etc/rc*.d/` by hand, or using `chkconfig` and hoping it understood your service's LSB header. The relationship between "start now" and "start on boot" was completely separate, managed by completely separate tools.

systemctl replaces all of that. One tool controls every aspect of service lifecycle — start, stop, enable, disable, introspect. It gives you rich status information, direct log access, and a consistent interface across every systemd-based distribution. That's the whole fleet today: Debian, Ubuntu, Fedora, RHEL, Arch, and everything downstream.

## Mental Model

Think of systemctl as the remote control for a smart home system.

The smart home hub (systemd) is always running in the background, managing every device in the house — lights, locks, thermostat, everything. The remote control (systemctl) is how you talk to the hub. You press "turn on the porch light" (start) and the hub handles the details. You set a schedule (enable) so the porch light comes on automatically at sunset every day. You can query the hub to see the current state of every device in the house (list-units). And if a device is malfunctioning, the hub's history tells you exactly when it failed and what it tried to do before giving up (status).

The key insight: the remote control is not the hub. systemctl sends instructions and queries to a running systemd process. It doesn't do the work itself — it talks to something that does. This matters when you think about why `daemon-reload` is needed: the hub has its own internal picture of the world, and you have to explicitly tell it when that picture needs to be refreshed.

## Theory

### What systemctl Is

systemctl is the primary interface for interacting with the systemd system and service manager. When you run any systemctl command, it communicates with PID 1 (the running systemd process) over D-Bus. systemctl itself is not the service manager — it's a client that talks to the service manager.

This distinction matters. systemd is always running, maintaining state about every unit it knows about. systemctl is just a messenger. If systemd is stuck or unresponsive, systemctl commands will hang or fail — not because systemctl is broken, but because the process it's talking to isn't answering.

### The Two Orthogonal States

Every service has two completely independent states, and conflating them is the most common source of confusion.

**Runtime state** — is the service running right now?
- `active (running)` — the process is alive
- `inactive (dead)` — not running
- `failed` — it exited with an error or was killed by a signal

**Boot-time state** — will the service start automatically on the next boot?
- `enabled` — a symlink exists in the appropriate `.wants/` or `.requires/` directory that will cause systemd to start this unit during boot
- `disabled` — no such symlink exists; the unit will not start unless explicitly started
- `masked` — a symlink to `/dev/null` exists; the unit cannot start under any circumstances
- `static` — the unit has no `[Install]` section; it cannot be enabled but may be started as a dependency
- `linked` — enabled via a symlink to a unit file outside the standard unit path

These two states are independent. A service can be running now but disabled at boot (you started it manually, it won't survive a reboot). A service can be enabled at boot but not currently running (it's set to start next boot, but hasn't been started yet in this session).

### enable Does Not start — It Creates Symlinks

`systemctl enable` does one thing: it creates symlinks. When you enable a service, systemd reads the `[Install]` section of the unit file, finds the `WantedBy=` directive, and creates a symlink in the corresponding `.wants/` directory. For a service with `WantedBy=multi-user.target`, enabling creates:

```text
/etc/systemd/system/multi-user.target.wants/nginx.service -> /usr/lib/systemd/system/nginx.service
```

That symlink is what causes the service to start on the next boot. The service is not started now. The running system is unaffected. Only the symlink was created.

`systemctl disable` removes that symlink. Nothing else. The service continues running if it was already running.

### mask Is a Hard Veto

`systemctl mask` creates a different kind of symlink — one pointing to `/dev/null`:

```text
/etc/systemd/system/nginx.service -> /dev/null
```

When systemd tries to load a unit and finds that its unit file is a symlink to `/dev/null`, it marks the unit as `masked` and refuses to start it under any circumstances. It cannot be started manually. It cannot be started as a dependency. Even `systemctl start` will fail with an explicit error. This is the correct tool when you need an absolute guarantee that a service will never run — for security reasons, for compliance reasons, or because you've installed a replacement service and the original must never accidentally activate.

### daemon-reload Keeps systemd's In-Memory State Fresh

systemd reads unit files when it starts and caches the result in memory. If you add a new unit file, modify an existing one, or remove one, systemd does not automatically notice. The changes are invisible to it until you run `systemctl daemon-reload`.

`daemon-reload` tells the systemd process to re-read all unit files from disk, rebuild its dependency tree, and update its internal state. It does not restart any services. It does not affect currently running processes in any way. It purely refreshes systemd's knowledge of what unit files exist and what they contain.

Forgetting this step after editing a unit file is one of the most common causes of "why isn't my change taking effect?" — you edited the file, but systemd is still using the version it read at boot.

### User Sessions

systemd runs two separate instances on a system. The **system instance** runs as PID 1 as root and manages system-wide services. The **user instance** (`systemd --user`) runs as each logged-in user and manages that user's personal services. These instances are completely separate. Commands without `--user` talk to the system instance. Commands with `--user` talk to the user's own instance.

User unit files live in `~/.config/systemd/user/`. The user instance starts when the user logs in and (by default) stops when all of their sessions end.

## How It Works (Mechanics)

When you run `systemctl start nginx.service`, the following happens:

1. systemctl opens a connection to systemd via D-Bus (the socket at `/run/systemd/private/`).
2. systemctl sends a `StartUnit` method call to the `org.freedesktop.systemd1` D-Bus service.
3. systemd receives the request, looks up the `nginx.service` unit in its in-memory unit table, and checks the unit's current state.
4. If the unit is masked (its file is a symlink to `/dev/null`), systemd immediately returns an error.
5. If the unit is inactive or failed, systemd transitions it to `activating` state and executes the `ExecStart=` command defined in the unit file.
6. systemd sets up the cgroup for the process, applies any resource limits, sets the environment, and forks the process.
7. Once the process is running (or has signaled readiness, for `Type=notify` or `Type=dbus` services), systemd transitions the unit to `active`.
8. systemctl receives the response and exits, printing the result.

For `systemctl enable`, the mechanics are entirely different — no D-Bus communication with the running systemd process is needed for the symlink creation. systemctl reads the unit file, parses the `[Install]` section, and creates the symlinks directly in the filesystem. It does tell systemd to reload afterward in some versions, but the core operation is just filesystem manipulation.

For `systemctl daemon-reload`, systemctl sends a `Reload` D-Bus method call, and PID 1 re-reads all unit files from disk. The kernel's inotify facility can detect unit file changes, and newer systemd versions will warn you when you need to run `daemon-reload` (you'll see a hint in `systemctl status` output if the on-disk file differs from the in-memory version).

```bash
# See whether systemd knows a unit file has changed on disk
systemctl show nginx.service --property=NeedDaemonReload
```

```text
NeedDaemonReload=yes
```

When you see `NeedDaemonReload=yes`, run `daemon-reload` before making any further changes to that unit's behavior.

## Basic Example

```bash
# Check whether SSH is running and enabled
systemctl status sshd
```

```text
● sshd.service - OpenSSH server daemon
     Loaded: loaded (/usr/lib/systemd/system/sshd.service; enabled; preset: enabled)
     Active: active (running) since Mon 2025-03-10 14:22:07 UTC; 2 days ago
   Main PID: 1047 (sshd)
      Tasks: 1 (limit: 4611)
     Memory: 5.6M (peak: 6.1M)
        CPU: 47ms
     CGroup: /system.slice/sshd.service
             └─1047 "sshd: /usr/sbin/sshd -D [listener] 0 of 10-100 startups"
```

```bash
# Stop the service
sudo systemctl stop sshd

# Confirm it stopped
systemctl is-active sshd
```

```text
inactive
```

```bash
# Start it again
sudo systemctl start sshd

# Enable and start in a single command
# --now: start immediately in addition to creating the boot-time symlink
sudo systemctl enable --now sshd
```

```text
Created symlink /etc/systemd/system/multi-user.target.wants/sshd.service → /usr/lib/systemd/system/sshd.service.
```

Read the `systemctl status` output carefully. The second line — `Loaded:` — tells you two things in parentheses: where the unit file came from, whether it's `enabled` or `disabled`, and the preset. The third line — `Active:` — is the runtime state, and it tells you how long the service has been in that state. These two lines together show you the full picture at a glance.

Try this yourself: run `systemctl status sshd` (or substitute `ssh` on Debian/Ubuntu — it's the same unit, named differently). If `sshd` is not installed on your system, try `systemctl status cron` or `systemctl status NetworkManager`. You're looking for the `Loaded:` and `Active:` lines.

## Real-World Example

This is the full workflow you'd use when deploying a new service on a production system — creating the unit file, testing it, enabling it for boot, and confirming it survives a restart.

```bash
# Write a minimal custom service unit file
sudo tee /etc/systemd/system/myapp.service << 'EOF'
[Unit]
Description=My Production Application
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=myapp
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/bin/myapp --config /etc/myapp/config.toml
Restart=on-failure
RestartSec=5s
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
EOF

# Tell systemd the new file exists
# This is mandatory after creating or editing any unit file
sudo systemctl daemon-reload

# Test the service — start it without enabling at boot
sudo systemctl start myapp.service

# Check the outcome — look for Active: active (running)
systemctl status myapp.service
```

```text
● myapp.service - My Production Application
     Loaded: loaded (/etc/systemd/system/myapp.service; disabled; preset: disabled)
     Active: active (running) since Mon 2025-03-10 15:45:02 UTC; 8s ago
   Main PID: 24817 (myapp)
      Tasks: 4 (limit: 4611)
     Memory: 18.2M
        CPU: 123ms
     CGroup: /system.slice/myapp.service
             └─24817 /opt/myapp/bin/myapp --config /etc/myapp/config.toml
```

```bash
# If the service works correctly, enable it for boot
# and start it in one command (it's already running, so this is safe —
# enable --now will start it if it isn't running, and skip start if it is)
sudo systemctl enable --now myapp.service
```

```text
Created symlink /etc/systemd/system/multi-user.target.wants/myapp.service → /etc/systemd/system/myapp.service.
```

```bash
# Verify both states are correct
systemctl is-active myapp.service   # should say: active
systemctl is-enabled myapp.service  # should say: enabled
```

```text
active
enabled
```

```bash
# Later: check all failed units across the entire system
# This is the first command to run when something is wrong
systemctl --failed
```

```text
  UNIT              LOAD   ACTIVE SUB    DESCRIPTION
● broken.service    loaded failed failed Broken Service

LOAD   = Reflects whether the unit definition was properly loaded.
ACTIVE = The high-level unit activation state, i.e. generalization of SUB.
SUB    = The low-level unit activation state, values depend on unit type.
1 loaded units listed.
```

```bash
# Now modify the unit to add an environment variable
sudo systemctl edit myapp.service
```

This opens your editor with `/etc/systemd/system/myapp.service.d/override.conf` — a drop-in override file. You don't edit the original unit file. You write only the sections you want to override:

```ini
[Service]
Environment="APP_ENV=production"
Environment="LOG_LEVEL=info"
```

Save and exit. systemctl automatically runs `daemon-reload` for you when you use `systemctl edit`. Then apply the change:

```bash
sudo systemctl restart myapp.service
```

Notice the pattern: `systemctl edit` + automatic daemon-reload + explicit restart. The drop-in at `/etc/systemd/system/myapp.service.d/override.conf` survives package updates that might overwrite the original unit file in `/usr/lib/systemd/system/`. Your customizations are safe.

## ⚠️ Gotchas

### enable Doesn't Start the Service — And disable Doesn't Stop It

The expectation: you run `systemctl enable nginx` and nginx starts. It doesn't.

```bash
sudo systemctl enable nginx
```

```text
Created symlink /etc/systemd/system/multi-user.target.wants/nginx.service → /usr/lib/systemd/system/nginx.service.
```

```bash
systemctl is-active nginx
```

```text
inactive
```

`enable` creates the boot-time symlink. That's it. The service is not running. If you need it running right now, you must also run `systemctl start nginx` — or use `enable --now` to do both in one command.

The same trap runs in reverse: `systemctl disable nginx` removes the boot-time symlink, but nginx continues running until the next reboot or until you explicitly stop it. If you're trying to take a service out of commission immediately, `disable` alone is not enough.

```bash
# The correct pattern when you want to start a service and have it survive reboots:
sudo systemctl enable --now nginx

# The correct pattern when you want to stop a service and prevent it from coming back:
sudo systemctl disable --now nginx
```

The `--now` flag was added in systemd 238. It is available on any modern distribution.

### Forgetting daemon-reload After Editing a Unit File

You edit `/etc/systemd/system/myapp.service`, then restart the service. Nothing changes. You edit it again. Still nothing. systemd is not reading your file — it's using the version it loaded into memory, which was the last version it knew about before your edits.

```bash
sudo nano /etc/systemd/system/myapp.service
# make changes, save
sudo systemctl restart myapp.service
# change has no effect
```

The fix is always the same:

```bash
sudo systemctl daemon-reload
sudo systemctl restart myapp.service
```

`daemon-reload` first, then restart. In that order, every time.

systemd will actually warn you about this. If you run `systemctl status myapp.service` after editing a unit file without reloading, you'll see:

```text
Warning: The unit file, source configuration file or drop-ins of myapp.service changed on disk. Run 'systemctl daemon-reload' to reload units.
```

Read the warnings. systemd is telling you exactly what to do.

### mask Is Permanent and Easy to Forget

You mask a service to prevent it from running — maybe a legacy service your company's security policy requires be disabled. Time passes. Someone else tries to enable the service and can't figure out why it won't start.

```bash
sudo systemctl start postfix
```

```text
Failed to start postfix.service: Unit postfix.service is masked.
```

A masked service fails with a specific error. The fix is to unmask before enabling or starting:

```bash
sudo systemctl unmask postfix
sudo systemctl enable --now postfix
```

The trap is forgetting you masked something. There is no obvious indication in normal operation — you have to specifically check:

```bash
systemctl is-enabled postfix
```

```text
masked
```

Use `systemctl list-unit-files --state=masked` to audit all masked units on a system you've inherited or that has been managed by multiple people.

### restart vs. reload — Unnecessary Downtime

You change nginx's configuration and run `sudo systemctl restart nginx`. It works, but you just dropped every active connection to your web server while it restarted. If nginx supports `reload`, there was no reason to do that.

`restart` terminates the process and starts a new one. For a web server under active load, that means connections are dropped. `reload` sends a signal (usually SIGHUP) to the existing process, which re-reads its configuration file and applies the changes while staying running.

```bash
# This drops active connections:
sudo systemctl restart nginx

# This doesn't — new workers start with the new config, old workers finish their requests:
sudo systemctl reload nginx
```

Check whether a service supports reload before restarting it:

```bash
systemctl cat nginx.service | grep ExecReload
```

```text
ExecReload=/bin/sh -c "/usr/sbin/nginx -s reload"
```

If `ExecReload=` is present, the service supports reload. Use it instead of restart when you're changing configuration. Use `try-reload-or-restart` when you're unsure — it sends a reload if supported, and falls back to restart if not.

### --user Talks to a Different systemd

systemctl without `--user` talks to the system instance (PID 1). systemctl with `--user` talks to the user's personal systemd instance. If you mix them up, you'll see confusing "unit not found" errors.

```bash
# This fails — my custom user service is not in the system instance:
systemctl status my-pomodoro.service
```

```text
Unit my-pomodoro.service could not be found.
```

```bash
# This works — it talks to your user instance:
systemctl --user status my-pomodoro.service
```

```text
● my-pomodoro.service - Pomodoro Timer
     Loaded: loaded (/home/user/.config/systemd/user/my-pomodoro.service; enabled; preset: enabled)
     Active: active (running) ...
```

User units live in `~/.config/systemd/user/`. The user instance has its own enable/disable state, its own daemon-reload requirement, and its own journal. Everything works the same way, just scoped to the user.

## Distro Notes

| Distro Family | Behavior |
|---|---|
| Debian / Ubuntu | SSH service is named `ssh.service`, not `sshd.service`; both `ssh` and `sshd` work as aliases in recent versions |
| RHEL / Fedora / Rocky | SSH is `sshd.service`; SELinux can prevent services from starting even when systemd state is correct — check `journalctl -xe` for AVC denials |
| Arch | systemd is the only supported init system; no SysV compatibility layer; unit files in `/usr/lib/systemd/system/` |
| Ubuntu (pre-15.04) | Upstart was the default; `systemctl` was not available; if you're on modern Ubuntu (15.04+) you have systemd |

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `service <name> <action>` | Legacy SysV compatibility layer | On systemd systems this usually calls systemctl underneath; use systemctl directly — the indirection hides information |
| `systemctl` | All service management on systemd systems | The correct tool for any modern Linux system |
| `initctl` | Upstart systems only | Ubuntu pre-15.04, very old systems; not relevant on modern distributions |
| `rc-service` / `rc-update` | OpenRC systems | Alpine Linux, Gentoo; completely separate init system with its own tooling |

On any system running systemd — which is every major distribution — `systemctl` is the tool. `service` and `chkconfig` are compatibility wrappers that exist for muscle memory, not because they offer anything systemctl doesn't. Use systemctl directly: you get better output, better error messages, and you're working with the actual API rather than a wrapper.

## Trade-offs

**Where this shines:**

- Rich introspection — `systemctl status` gives you PID, cgroup, memory, CPU, recent log lines, and state transitions all in one view. The old `service status` gave you "running" or nothing.
- Atomic enable+start with `--now` eliminates the classic two-command pattern and the bugs it caused when people forgot the second command.
- Drop-in overrides via `systemctl edit` let you customize vendor-provided unit files without touching the original — your overrides survive package updates.
- Consistent interface across the entire unit ecosystem — services, sockets, timers, mounts, and devices all respond to the same subcommands.

**Where this costs you:**

- systemctl requires D-Bus and a running systemd. If systemd itself is broken or PID 1 is hung, systemctl is useless. In a recovery scenario (booting to a rescue shell), you're managing processes manually or with `kill`.
- The two-state model (runtime vs. boot-time) genuinely confuses newcomers and causes operational mistakes. Experienced sysadmins have shipped outages by enabling without starting or disabling without stopping.
- Masked units create invisible blockers. A system you've inherited may have things masked that you don't know about, and the failure mode is cryptic to anyone who doesn't know to check.
- `--user` and system-scope units are completely separate namespaces. Code that runs fine with `--user` will fail silently if deployed without it, and vice versa.

## When to Use / When to Avoid

✅ **Use `systemctl start` / `stop` when:**
- You need to change the running state of a service right now without affecting boot behavior
- Testing a new service before committing to enabling it permanently

✅ **Use `systemctl enable --now` when:**
- Deploying a new service that should run immediately and survive reboots — this is the correct default for production deployments

✅ **Use `systemctl reload` when:**
- You've changed a configuration file and the service supports `ExecReload=` — it avoids dropping connections and is faster than restart

✅ **Use `systemctl mask` when:**
- A service must never run, under any circumstances — security requirement, replacement service installed, compliance mandate

✅ **Use `systemctl daemon-reload` when:**
- After creating a new unit file
- After editing any existing unit file (including drop-ins)
- After removing a unit file

✅ **Use `systemctl --failed` when:**
- Something is wrong and you don't know where to start — this is the first command in any service debugging session

❌ **Avoid `service <name> start` for day-to-day work:**
- It calls systemctl underneath on modern systems, but hides the output formatting and gives you less information. Use systemctl directly.

❌ **Avoid `systemctl restart` for config changes on services that support reload:**
- Check for `ExecReload=` first. Unnecessary restarts drop active connections and waste the zero-downtime capability systemd provides.

❌ **Avoid editing unit files in `/usr/lib/systemd/system/` directly:**
- Those files are owned by your package manager. The next package update will overwrite your changes. Use `systemctl edit` to create drop-in overrides in `/etc/systemd/system/<unit>.d/` instead.

⚠️ **Common mistakes:**
- Running `enable` without checking whether the service is actually running afterward
- Running `disable` and assuming the service has stopped
- Editing a unit file and forgetting `daemon-reload` before restarting
- Using `restart` on a web server or database without checking whether `reload` would do the job without downtime
- Running `systemctl` without `--user` when trying to manage a user-session service

## Veteran Perspective

The enable/start confusion has caused more production incidents than I can count, and the pattern is always the same: someone enables a service as part of a deployment, assumes it's running, and moves on. The service isn't running. Nobody notices until 3am when someone tries to reach it and can't. The fix is `enable --now` as your default when deploying a service. Never `enable` without `--now` unless you specifically need the service to not start until reboot. There is almost never a reason to do that on purpose.

Here's what the man page won't tell you about `daemon-reload`: it's not optional, it's not a suggestion, and the window between "edit a unit file" and "run daemon-reload" is a booby trap. I've watched engineers spend thirty minutes debugging why their service change wasn't taking effect, editing the file again and again, each time running `systemctl restart` directly after — getting angrier at every iteration. The edit was correct every time. systemd was just reading a stale version. The habit to build is: edit, daemon-reload, restart. In that exact order. Write it on a sticky note until it's automatic.

The distinction between `disable` and `mask` is one of the most important things to understand on a system you're inheriting. `disable` is reversible and obvious — the service just isn't in any `.wants/` directory. `mask` is a hard veto that leaves no footprint in the normal command output. If you're handed a system where a service "just won't start no matter what," the first thing to check is `systemctl is-enabled <service>`. If it says `masked`, you have your answer. The second thing to check is whether it was masked intentionally — don't unmask without understanding why it was masked in the first place.

The `reload` vs. `restart` question is a production discipline question, not a technical one. Both work. The difference is whether you care about the users currently connected to your service. For a web server handling live traffic, `restart` is a forced outage. `reload` is invisible. nginx, Apache, HAProxy, PostgreSQL, and most major daemons support reload. The habit of checking for `ExecReload=` before reaching for `restart` is the mark of someone who has had to explain dropped connections to a business stakeholder.

On `--user` services: they're under-used and under-understood. If you have a process that should run as a specific user, doesn't need system-level privileges, and shouldn't outlive that user's session, a user service is exactly the right tool. Cron jobs that run as user processes are a common antipattern — they're hard to monitor, hard to restart on failure, and produce logs that go nowhere. A user systemd timer does the same thing with proper logging, automatic restart on failure, and `systemctl --user status` for free.

I've watched teams mask a conflicting service during an incident — `systemctl mask apache2` — ship a fix, declare victory, and close the ticket. Six months later, a new engineer tries to enable a feature that depends on apache2, spends an afternoon debugging, and eventually finds a symlink pointing to `/dev/null`. The lesson: if you mask something operationally, write it down. Create a ticket. Leave a comment in your infrastructure-as-code. The system will not remind you. `systemctl list-unit-files --state=masked` is the audit command; run it periodically on any system with multiple administrators.

## Try It Yourself

1. Run `systemctl status sshd` (or `ssh` on Debian/Ubuntu). Read every line of output. Find the `Loaded:` line and identify whether the service is enabled or disabled. Find the `Active:` line and note how long it has been running. The answer to "is this service healthy?" lives in those two lines.

2. Run `systemctl list-units --type=service --state=running` to see every service currently active on your system. Pick one you don't recognize and run `systemctl cat <unit>` to see its unit file. Read the `[Unit]`, `[Service]`, and `[Install]` sections and try to understand what each directive does.

3. Run `systemctl list-unit-files --state=enabled` to see everything configured to start at boot. Cross-reference with `systemctl list-units --type=service --state=active` to see whether there's anything enabled that isn't currently running — that's a service that's configured to start but failed or was stopped manually.

4. Create a minimal user service. Write a file at `~/.config/systemd/user/hello.service` with this content:
   ```ini
   [Unit]
   Description=Hello World Service
   
   [Service]
   Type=oneshot
   ExecStart=/bin/echo "Hello from a systemd user service"
   RemainAfterExit=yes
   
   [Install]
   WantedBy=default.target
   ```
   Then run `systemctl --user daemon-reload`, `systemctl --user start hello.service`, and `systemctl --user status hello.service`. Notice that it runs as your user, not root, and the logs appear in your user journal.

5. **Challenge (VM recommended):** Deploy a real service from scratch. Install `nginx` (or any web server), create a drop-in override using `sudo systemctl edit nginx` that adds `LimitNOFILE=65536` (raises the open file descriptor limit — a common production tuning), verify the override took effect with `systemctl show nginx --property=LimitNOFILE`, then test that a config reload works without dropping connections by running `while true; do curl -s -o /dev/null http://localhost; done` in one terminal while running `sudo systemctl reload nginx` in another. Confirm no errors in the curl loop during the reload.

## Summary

- `systemctl enable` creates boot-time symlinks; it does not start the service. `systemctl start` starts the service without touching boot configuration. Use `enable --now` to do both at once — this is the correct default when deploying a service.
- `systemctl disable` removes boot-time symlinks but does not stop a running service. `systemctl mask` creates a symlink to `/dev/null` and is an absolute veto — a masked unit cannot start under any circumstances, even as a dependency.
- After creating or editing any unit file, run `systemctl daemon-reload` before restarting the service. systemd reads unit files at startup and caches them; edits are invisible until you tell it to reload.
- `reload` sends a signal to the running process to re-read its configuration without stopping; `restart` kills and respawns it. Use `reload` when the service supports it (`ExecReload=` is present) — it avoids dropping connections.
- `systemctl --user` talks to the user's personal systemd instance, not the system-wide PID 1. User unit files live in `~/.config/systemd/user/`. Everything works identically, scoped to the user.
