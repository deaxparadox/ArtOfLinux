# 14.3 — Linux Namespaces

> Namespaces are the kernel mechanism that lets one Linux host look like many independent systems simultaneously — they are the reason containers exist.

## The Problem This Solves

Before namespaces, if you ran two services on one Linux machine, they shared everything: the same process tree, the same hostname, the same network interfaces, the same filesystem root, the same IPC objects. If one service crashed PID 1, the machine went down. If one service changed the hostname, every other service saw the new name. If two services both wanted to listen on port 80, one of them lost.

The traditional answer was: use separate physical machines. Or virtual machines — full kernel overhead, minutes to start, gigabytes of RAM per instance, even if all you needed was an isolated process tree.

The problem wasn't that isolation was impossible. The problem was that isolation was binary: either you ran on bare metal and shared everything, or you ran in a VM and reproduced the entire kernel and hardware stack. There was no middle ground — a way to say "this process should see a different hostname than that process, but they should still share the same kernel."

Namespaces are that middle ground. They let the kernel give different processes a different *view* of a resource without copying the resource itself. Two processes can each believe they are PID 1 in their own process tree, running on a machine called "webserver-prod", listening on port 80 — all on the same kernel, with no hypervisor layer in between. This is what makes containers fast: there is no virtualized hardware, no second kernel to boot. The isolation comes from the kernel remapping what each process is allowed to see.

## Mental Model

Imagine a large open-plan office with hundreds of employees. Everyone can see everyone else, hear everyone's phone calls, walk into any meeting room, and read any whiteboard. It's chaos for isolation.

Now imagine the building gets divided into private offices with opaque walls. Each office has its own door number starting at 1, its own phone extension system, its own whiteboard. The employees in office A cannot see what's on office B's whiteboard, and both offices have a "Room 1" conference room that is actually different rooms. The building is still the same building. The heating system, the electrical wiring, the roof — all shared. Only the *view* each office has of the internal resources is different.

Namespaces are those walls. The kernel is the building. Processes inside a namespace believe they own a whole machine. From outside, the host kernel sees all processes and all resources simultaneously.

## Theory

A namespace is a kernel object that wraps one specific category of system resource and provides an isolated instance of it. When a process belongs to a namespace, every operation on that resource category goes through the namespace — the process sees only what the namespace exposes, not the full system-wide view.

Linux currently has eight namespace types. Each was added independently as the kernel team identified new resources that needed isolation:

**PID namespace** — isolates the process ID space. Each PID namespace has its own numbering starting at 1. The first process started in a new PID namespace becomes PID 1 within it; from the host, that same process has a different PID in the global tree. PID namespaces nest: a process in a child PID namespace is visible to parent namespaces under its host PID, but invisible to sibling namespaces. If PID 1 inside a namespace dies, the kernel sends SIGKILL to every other process in that namespace.

**Network namespace (net)** — isolates the entire networking stack: network interfaces, routing tables, firewall rules (iptables/nftables), port bindings, and socket state. Two processes in different network namespaces can both bind to port 80 without conflict. The host can bridge or route between namespaces using virtual ethernet pairs (veth).

**Mount namespace (mnt)** — isolates the filesystem view. Processes in different mount namespaces can have different filesystems mounted at the same path, or see entirely different filesystem trees. The mount namespace is what lets a container have its own `/proc`, its own `/tmp`, even its own root filesystem — without touching the host filesystem.

**UTS namespace** — isolates two values: the hostname and the NIS domain name (the structures returned by `uname(2)`). The name "UTS" comes from `struct utsname` in the kernel. This is the cheapest namespace to create: it costs almost nothing and lets each container report a different hostname to applications that call `gethostname()`.

**IPC namespace** — isolates System V IPC objects (message queues, semaphores, shared memory segments) and POSIX message queues. Without IPC namespace isolation, any process on the system can write to any shared memory segment if it knows the key. With isolation, IPC objects in one namespace are completely invisible to processes in another.

**User namespace** — isolates user and group ID spaces. A process inside a user namespace can be mapped to UID 0 (root) inside the namespace while running as an unprivileged user (e.g., UID 1000) on the host. This makes unprivileged containers possible: a user who cannot normally call `chroot` or create new namespaces can do both inside a user namespace where they are effectively root, with privileges that are real within the namespace but carry no weight outside it. User namespaces are the most security-sensitive type.

**Cgroup namespace** — isolates the process's view of its position in the cgroup hierarchy. Without this, a containerized process could read `/proc/self/cgroup` and discover its actual position in the host cgroup tree, leaking information about the host's container orchestration structure.

**Time namespace** — added in Linux 5.6. Isolates two monotonic clocks: `CLOCK_MONOTONIC` and `CLOCK_BOOTTIME`. Processes in a time namespace can see different offsets applied to these clocks, making it possible to restore a checkpointed container with a consistent view of elapsed time without the clocks jumping forward relative to the container's internal state.

Every process belongs to exactly one instance of each namespace type. When a process is created with `fork()` or `clone()`, it inherits its parent's namespaces by default. Creating a new namespace requires either the `CAP_SYS_ADMIN` capability on the host, or an unshared user namespace (which grants a limited form of privilege isolation for unprivileged users).

The critical invariant: namespaces are reference-counted kernel objects. A namespace persists as long as at least one process belongs to it, *or* as long as a file descriptor holds a reference to it (via a bind mount of `/proc/PID/ns/TYPE`). When the last reference goes away, the namespace is destroyed.

## How It Works (Mechanics)

### Creating a Namespace: The System Call Layer

Three system calls manipulate namespaces:

- `clone(flags)` — creates a new process *and* optionally new namespaces in one call. The `flags` argument is an OR of `CLONE_NEW*` constants. This is what container runtimes use internally.
- `unshare(flags)` — creates new namespaces for the *calling* process without forking. The process moves into the new namespace immediately.
- `setns(fd, nstype)` — moves the calling process into an *existing* namespace, identified by a file descriptor pointing to a `/proc/PID/ns/TYPE` file.

The `CLONE_NEW*` constants correspond directly to namespace types:

```text
CLONE_NEWPID    → PID namespace
CLONE_NEWNET    → network namespace
CLONE_NEWNS     → mount namespace  (the original; "NS" predates the naming convention)
CLONE_NEWUTS    → UTS namespace
CLONE_NEWIPC    → IPC namespace
CLONE_NEWUSER   → user namespace
CLONE_NEWCGROUP → cgroup namespace
CLONE_NEWTIME   → time namespace
```

### The /proc/PID/ns/ Directory

Every namespace a process belongs to is exposed as a symbolic link under `/proc/PID/ns/`:

```text
/proc/1234/ns/
├── cgroup -> cgroup:[4026531835]
├── ipc    -> ipc:[4026531839]
├── mnt    -> mnt:[4026531840]
├── net    -> net:[4026531969]
├── pid    -> pid:[4026531836]
├── pid_for_children -> pid:[4026531836]
├── time   -> time:[4026531834]
├── time_for_children -> time:[4026531834]
├── user   -> user:[4026531837]
└── uts    -> uts:[4026531838]
```

The number in brackets is the namespace inode number — a unique identifier for that namespace instance. Two processes that show the same inode number for a given namespace type are in the same namespace. This is how you verify isolation: if `ls -la /proc/PID_A/ns/pid` and `ls -la /proc/PID_B/ns/pid` show different inodes, the processes are in different PID namespaces.

The `pid_for_children` and `time_for_children` entries are a subtle detail: the kernel separates "the namespace this process currently operates in" from "the namespace that new children will be born into." This matters for `unshare(CLONE_NEWPID)` — the calling process itself stays in its old PID namespace, but its children are born into the new one.

### How PID Namespace Nesting Works

```text
Host PID namespace
├── systemd (PID 1)
├── sshd    (PID 892)
└── bash    (PID 1543)  ← starts a new PID namespace
    └── Child PID namespace
        ├── bash  (PID 1 inside NS, PID 1543 on host — same process, two identities)
        └── ps    (PID 2 inside NS, PID 1601 on host)
```

A process in the child namespace cannot see PIDs 1 (systemd) or 892 (sshd) from the host. But the host can see PID 1543 and 1601. The isolation is one-directional: parent sees child, child cannot see parent.

### User Namespace UID Mapping

When a user namespace is created, it starts with an empty UID/GID map — no user identity is valid until the mapping is written. The mapping is written to `/proc/PID/uid_map` and `/proc/PID/gid_map` by a process outside the namespace with appropriate privilege:

```text
# Format: ID-inside-NS  ID-outside-NS  length
0  1000  1
```

This single line maps UID 0 inside the namespace to UID 1000 on the host, for a range of 1 UID. From inside the namespace, the process appears to be root. From outside, the kernel actually enforces operations as UID 1000, which has no special privileges. Any file that process creates on the host filesystem will be owned by UID 1000.

The kernel enforces strict rules on who can write to `uid_map`: you can write it only once, only before the process has done anything that would make the mapping inconsistent, and the host UIDs referenced must be within ranges that `/etc/subuid` has allocated for the unprivileged user creating the namespace.

### Time Namespace Offsets

The time namespace works differently from other namespaces — it isn't inherited by `unshare()` directly. The new namespace is created via `unshare(CLONE_NEWTIME)`, but the calling process continues in its old time namespace. Only processes subsequently created by that process join the new time namespace (hence `time_for_children` in `/proc/PID/ns/`). The clock offsets are written to `/proc/PID/timens_offsets` before any process enters the namespace.

## Basic Example

```bash
# Create a new UTS namespace and change the hostname inside it
# --uts creates a new UTS namespace
sudo unshare --uts bash
```

Once inside the new shell, verify and change the hostname:

```bash
# Check the hostname — it inherited from the host
hostname

# Change it inside the namespace
hostname container-01

# Verify the change is visible inside
hostname
```

```text
yourhost
container-01
```

Now open a second terminal and check the host hostname:

```bash
# On the host, the hostname is unchanged
hostname
```

```text
yourhost
```

The two `hostname` commands return different values — different processes, different UTS namespaces, same kernel. Now try it yourself. The key thing to notice: as soon as you exit the `unshare bash` shell, the UTS namespace is destroyed because no process is holding a reference to it anymore. The hostname change vanishes with it.

## Real-World Example

This example builds a minimal isolated environment by hand — isolated PID tree, hostname, and network stack — the way container runtimes do it internally.

```bash
# Create a new shell with isolated PID, UTS, and network namespaces
# --pid       new PID namespace
# --fork      run bash as a child so it becomes PID 1 in the new NS
# --mount-proc remounts /proc to reflect the new PID namespace's process tree
# --uts       new UTS namespace
# --net       new network namespace
sudo unshare --pid --fork --mount-proc --uts --net bash
```

Inside the new shell, verify the isolation:

```bash
# Check the process tree — only two processes are visible
ps aux
```

```text
USER       PID %CPU %MEM    VSZ   RSS TTY      STAT START   TIME COMMAND
root         1  0.0  0.0   4628  3820 pts/1    S    10:31   0:00 bash
root         7  0.0  0.0   8908  3400 pts/1    R+   10:31   0:00 ps aux
```

```bash
# The host's hundreds of processes are invisible
# Set a unique hostname for this environment
hostname isolated-env

# Check networking — only loopback exists; no eth0, no host interfaces
ip link show
```

```text
1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN mode DEFAULT group default qlen 1000
    link/loopback 00:00:00:00:00:00 brd 00:00:00:00:00:00
```

```bash
# From the host (in another terminal), confirm the new namespaces are visible
# and have different inode numbers from the host namespaces
ls -la /proc/$(pgrep -n bash)/ns/pid /proc/1/ns/pid
```

```text
lrwxrwxrwx 1 root root 0 Jun 15 10:31 /proc/4821/ns/pid -> pid:[4026532345]
lrwxrwxrwx 1 root root 0 Jun 15 10:31 /proc/1/ns/pid    -> pid:[4026531836]
```

The different inode numbers — `4026532345` vs `4026531836` — confirm these are separate PID namespace instances.

Now enter the existing namespace from outside using `nsenter`. In a second terminal:

```bash
# Enter the network namespace of the isolated shell
# -t     target PID to read namespaces from
# --net  enter only the network namespace
# --pid  enter only the PID namespace
sudo nsenter -t $(pgrep -n -P $(pgrep unshare)) --net --uts bash
```

```bash
# You're now inside the same network and UTS namespaces
hostname
ip link show
```

```text
isolated-env
1: lo: <LOOPBACK> mtu 65536 qdisc noop state DOWN ...
```

This is how container debugging tools work. `docker exec`, `kubectl exec`, and `nsenter` are all doing the same thing: finding the PID of the container's init process, opening the namespace file descriptors under `/proc/PID/ns/`, and calling `setns()` to move into those namespaces before executing the requested command.

## ⚠️ Gotchas

### `--pid` Without `--fork` Does Not Give You PID 1

The most common mistake when experimenting with PID namespaces. You might try:

```bash
# Looks like it creates a PID namespace for this shell
sudo unshare --pid bash
```

```bash
# Check what PID we are inside
echo $$
```

```text
1
```

The shell *reports* PID 1, but look at what happens when you run anything:

```bash
ps aux
```

```text
Error, do this: mount -t proc proc /proc
```

Or worse — if `/proc` is not remounted — you see the host's full process tree and your shell showing some high PID, not 1. The confusion: `unshare --pid` creates the new namespace, but the *calling shell* itself (the parent) remains in the old namespace. The shell started as the child is what gets PID 1 in the new namespace — but that child is just the `bash` you specified, and it didn't get a fresh `/proc`.

The correct invocation requires both `--fork` (so `unshare` forks a child that becomes PID 1) and `--mount-proc` (so `/proc` reflects the new namespace):

```bash
# Correct: the child bash is PID 1, and /proc shows only the new namespace
sudo unshare --pid --fork --mount-proc bash
```

```bash
echo $$
ps aux
```

```text
1
USER       PID %CPU %MEM ...
root         1  0.0  0.0  bash
root         8  0.0  0.0  ps aux
```

The rule: `--pid` creates the namespace; `--fork` makes the next process the PID 1 of that namespace; `--mount-proc` makes `/proc` tell the truth about what's inside.

---

### Mount Namespace Changes Don't Propagate by Default

You create a new mount namespace, mount something inside it, exit, and expect the mount to be gone from the host. It is. But you might also expect that mounts you make on the *host* are visible inside your namespace — and whether they are depends on mount propagation, which is not what `unshare --mount` alone gives you.

```bash
# Create a new mount namespace
sudo unshare --mount bash

# Create a tmpfs mount inside the namespace
mount -t tmpfs tmpfs /mnt/test
df -h /mnt/test
```

```text
Filesystem      Size  Used Avail Use% Mounted on
tmpfs           7.7G     0  7.7G   0% /mnt/test
```

```bash
# On the host in another terminal — the mount is NOT visible
df -h /mnt/test
```

```text
df: /mnt/test: No such file or directory
```

This is correct and expected. The isolation works in both directions for private mounts. The subtlety is that by default on modern systems, new mount namespaces are created as a copy of the parent's mount table, but the propagation type of each mount point in the copy matches the parent's — and systemd sets most mount points to `shared` by default. A shared mount point propagates events in both directions.

If you need strict isolation — where nothing you do inside the namespace escapes, and nothing the host mounts is visible inside — you need to change propagation to private:

```bash
sudo unshare --mount bash

# Disable propagation from this namespace to the host
mount --make-rprivate /

# Now mounts inside are truly isolated
mount -t tmpfs tmpfs /mnt/test
```

If you just run `unshare --mount` without `--make-rprivate /`, mounts at shared propagation points will still appear on the host. This has caused confused sysadmins to wonder why their "isolated" mounts were leaking out.

---

### User Namespaces Expand the Attack Surface

User namespaces let an unprivileged user create a namespace where they appear as root. This is powerful and intentional — it's how rootless containers (Podman, rootless Docker) work. But it also means an unprivileged user can now exercise syscalls and kernel codepaths that were previously gated behind `CAP_SYS_ADMIN`. The kernel's user namespace implementation has historically been a source of privilege escalation bugs.

For this reason, many distributions ship with unprivileged user namespace creation disabled:

```bash
# Check whether unprivileged user namespace creation is allowed
sysctl kernel.unprivileged_userns_clone
```

```text
kernel.unprivileged_userns_clone = 0
```

A value of `0` means only root can create user namespaces. On Ubuntu 24.04 and later, this is restricted further by the `apparmor_restrict_unprivileged_userns` AppArmor policy. On Debian stable, the default is `0`. On Fedora and Arch, it's typically `1` (allowed).

If you need rootless containers on a system where this is disabled, you'll need to either enable it system-wide (accepting the risk) or use a tool like `newuidmap`/`newgidmap` with proper SUID configuration. Do not enable it blindly on a multi-tenant server without understanding what workloads are running.

---

### Time Namespace Only Offsets Two Clocks — `CLOCK_REALTIME` Is Not Isolated

The time namespace name implies it isolates time in general. It does not.

`CLOCK_REALTIME` — the wall clock that `date` and most user-space applications use — is **not namespaced**. If a process inside a time namespace calls `clock_gettime(CLOCK_REALTIME)`, it gets the same value as the host. Only `CLOCK_MONOTONIC` and `CLOCK_BOOTTIME` receive per-namespace offsets.

This means time namespaces are not a way to give a container a fake date. They exist specifically for checkpoint/restore use cases (CRIU) where a container that was frozen and resumed should see consistent monotonic time relative to its own start, not a jump forward corresponding to the time it was paused.

If you need to run a process with a fake wall clock time, use `libfaketime` or `faketime` wrappers — they intercept `gettimeofday()` in userspace. Time namespaces won't help you.

---

### Entering a PID Namespace with `nsenter` Doesn't Change Your PID Namespace

This one is subtle. When you call `nsenter --pid -t <PID>`, the processes you subsequently execute run in the target PID namespace. But `nsenter` itself — the process that called `setns()` — does not move into the new PID namespace. The calling process's PID namespace is fixed at the point the process was created and cannot be changed after the fact. What changes is `pid_for_children`: any child process spawned after `setns(CLONE_NEWPID)` will be born into the new PID namespace.

This is why `nsenter` always execs a new process (like `bash` or `sh`) rather than running in place — the exec'd child is what actually lives in the target namespace.

## Alternatives & Comparisons

| Approach | What It Provides | Trade-off |
|---|---|---|
| Linux namespaces | Kernel-native isolation, per-resource granularity, zero overhead | Complexity; each namespace type has its own rules and gotchas |
| Full VMs (KVM, VMware) | Complete isolation including separate kernel | High overhead; minutes to start; gigabytes of RAM; separate kernel to patch |
| chroot | Filesystem root isolation only | Not a security boundary; PID, network, users are fully shared; trivially escapable |
| seccomp | Syscall filtering | Not isolation — it restricts what a process can call, not what it can see |

Namespaces and seccomp are complementary, not alternatives. A properly isolated container uses both: namespaces to constrain visibility, seccomp to constrain behavior. `chroot` alone is not a security boundary and should never be treated as one.

## Trade-offs

**Where this shines:**

- Process isolation with near-zero performance overhead — there is no hypervisor, no hardware virtualization; namespace lookups are fast kernel operations
- Granular isolation — you can isolate just the network namespace and nothing else, or mix and match as your use case demands
- Composability — container runtimes combine all namespace types with cgroups (resource limits) and seccomp (syscall filtering) to build a full isolation model from these primitives
- Unprivileged containers — user namespaces allow containers to run without any host privileges, which is the correct direction for multi-tenant environments

**Where this costs you:**

- Complexity — each namespace type has independent semantics, propagation rules, and gotchas; debugging a misconfigured namespace is hard
- Security surface — user namespaces in particular have historically introduced kernel vulnerabilities by exposing kernel codepaths to unprivileged users; every new namespace type adds code that can be reached without root
- `/proc` fragmentation — if you create PID namespaces without remounting `/proc`, the process filesystem shows inconsistent data; tooling that reads `/proc` (monitoring agents, `ps`, `lsof`) can behave unpredictably inside improperly constructed namespaces
- Namespace leaks — forgetting to clean up bind-mounted namespace file descriptors can keep namespaces alive after all processes exit, wasting kernel memory and potentially causing confusing behavior

**Performance note:** Namespace creation via `clone(CLONE_NEW*)` adds roughly 1-5ms to process startup depending on how many namespace types are requested and the system's kernel version. This is negligible for long-running containers but measurable for workloads that fork thousands of short-lived isolated processes per second.

## When to Use / When to Avoid

✅ **Use namespaces when:**

- Building or debugging container runtimes — understanding which namespace type provides which isolation is essential to reasoning about container security
- Running multiple services on one host that would otherwise conflict — different hostnames, port bindings, or filesystem layouts
- Creating isolated test environments without the overhead of a VM
- Implementing checkpoint/restore workflows where time namespace offsets keep monotonic clocks consistent
- Debugging a running container by entering its namespaces with `nsenter` — you can join a container's network namespace without needing `docker exec`

❌ **Avoid namespaces (or be careful) when:**

- Using user namespaces on a hardened multi-tenant server — the attack surface expansion may not be acceptable; reach for rootful containers with seccomp and AppArmor instead
- Treating `chroot` plus a mount namespace as a security boundary — it isn't one; network and PID are still shared unless you also create those namespaces
- Creating time namespaces expecting to fake `CLOCK_REALTIME` — use `libfaketime` instead
- Using `unshare --pid` without `--fork` — you will not get a usable PID namespace; see the gotcha above

⚠️ **Common mistakes:**

- Confusing `pid_for_children` with the process's current PID namespace — they can differ after `unshare(CLONE_NEWPID)` until an exec happens
- Forgetting that a bind mount of `/proc/PID/ns/TYPE` keeps the namespace alive after the process dies — always clean up bind mounts when done
- Assuming `nsenter --net` also changes the PID namespace — each namespace type must be explicitly specified; specifying one does not imply the others
- Using the inode number as a persistent identifier across reboots — it's only unique within a single boot session

## Veteran Perspective

The thing most people get wrong about namespaces is thinking of them as a binary — "the container is isolated" or "it isn't." Namespaces are per-resource. You can have a container with a fully isolated network stack but a shared IPC namespace. Whether that's the right choice depends on your threat model.

When I see container escape reports, the first question I ask is: which namespaces were actually in use? A shocking number of "containers" in production use a mount namespace and maybe a UTS namespace, but share the host PID namespace — which means a process inside can see every PID on the host and send signals to them. That's not a container. That's a process with a different root filesystem.

The `--privileged` flag in Docker is the single most misused feature in container tooling. It disables most namespace protections and grants nearly all capabilities. I have been in incident reviews where a compromised application was running privileged because "we weren't sure if it needed it and it worked." Never run privileged unless you can state specifically which capability or namespace access the workload requires and why.

User namespaces deserve respect. The engineering behind rootless containers is genuinely clever — mapping UIDs so that a process believes it owns resources it doesn't actually own at the kernel level is a beautiful solution to the problem of running containers without root. But the implementation has a history of bugs. CVE-2022-0492, CVE-2023-32233, and several others were exploitable specifically because user namespaces gave attackers a path to kernel codepaths that were otherwise unreachable. On a server where you don't need rootless containers, turning off `kernel.unprivileged_userns_clone` is the right call.

The time namespace is the one most engineers don't know exists. It was added in 5.6 primarily for CRIU (Checkpoint/Restore In Userspace) — the tool that lets you freeze a running container, save its state to disk, and resume it later. Without time namespace support, resuming a checkpointed container caused monotonic clocks to jump forward by however long the container was frozen, breaking any application that compared two monotonic timestamps across the checkpoint boundary. Knowing the time namespace exists saves you from the trap of thinking you can use it to spoof the wall clock.

In a technical interview, if someone asks you "how do containers work?", the wrong answer is "Docker." The right answer starts with namespaces and cgroups, explains what each provides, and mentions that a container runtime is just a program that calls `clone()` with the right flags, sets up a cgroup, and execs into a pivot-rooted filesystem. That answer demonstrates you understand the kernel, not the vendor tooling.

## Try It Yourself

1. Run `ls -la /proc/self/ns/` and note the inode numbers for each namespace type. Then run `ls -la /proc/1/ns/`. Which namespaces do you share with PID 1?

2. Create a new UTS namespace with `sudo unshare --uts bash`. Change the hostname to anything. Open a second terminal and confirm the host hostname is unchanged. Exit the namespace and verify the changed hostname is gone.

3. Run `sudo unshare --pid --fork --mount-proc bash` and then `ps aux`. Note that you are PID 1. Now run `kill 1` from inside the namespace and observe what happens. (`kill 1` sends SIGTERM to PID 1 — inside a PID namespace, this kills the namespace's init process.)

4. Find a running Docker container on your machine (or start one with `docker run -d nginx`). Get its main process PID with `docker inspect --format '{{.State.Pid}}' <container>`. Use `sudo nsenter -t <PID> --net bash` to enter its network namespace and run `ip addr` — you'll see the container's network interfaces without needing `docker exec`.

5. **Challenge (VM recommended):** Create a complete isolated environment by hand: a new user namespace with UID 0 mapped to your unprivileged user, a new PID namespace, a new mount namespace, and a new network namespace — all without `sudo`. Write the uid_map and gid_map to establish the mapping, remount `/proc` inside, and verify you appear as root inside without having root on the host. This is the foundation of rootless containers.

## Summary

- Linux has eight namespace types — PID, net, mnt, UTS, IPC, user, cgroup, time — each isolating a different category of system resource; processes inherit their parent's namespaces and can only join a different namespace via `clone()`, `unshare()`, or `setns()`
- A namespace persists as long as at least one process belongs to it *or* a file descriptor references its entry in `/proc/PID/ns/TYPE`; bind-mounting that file is how you keep a namespace alive after all its processes exit
- `unshare --pid` without `--fork` and `--mount-proc` does not produce a usable PID namespace — the fork is required so the child becomes PID 1, and the proc remount is required so `/proc` reflects the new namespace
- User namespaces enable unprivileged containers by remapping UIDs so UID 0 inside the namespace maps to a non-root UID on the host; many distributions disable unprivileged user namespace creation (`kernel.unprivileged_userns_clone=0`) because the feature has a history of kernel privilege escalation vulnerabilities
- The time namespace (Linux 5.6+) offsets only `CLOCK_MONOTONIC` and `CLOCK_BOOTTIME` — it does not affect `CLOCK_REALTIME`; it exists for checkpoint/restore consistency, not wall-clock fakery
