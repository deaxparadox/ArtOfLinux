## The Problem This Solves

You have a service degrading in production. Response times are spiking every few minutes. CPU looks normal. Memory looks fine. The logs say nothing — the application is running, it's just slow.

What you need to know: which system calls is it making when it slows down? Is it hammering the filesystem? Waiting on locks? Making unexpected network connections? Making thousands of redundant `stat()` calls on a hot path?

The obvious tool is `strace`. But strace works by intercepting every syscall through `ptrace()` — stopping the process on every kernel entry and exit, waking the tracer, reading registers, formatting output, then releasing the process. On a process making 100,000 syscalls per second, strace can slow it down by **10x to 100x**. In some benchmarks, traced processes run 173 times slower. You cannot attach strace to a production service without making the incident worse.

The older alternative is to instrument the kernel itself — modify the kernel source, add your measurement code, recompile, reboot. That takes hours and requires root access to a running production machine that you don't want to reboot.

For decades, there was no good answer. You either ran blind or made things worse.

eBPF — extended Berkeley Packet Filter — is the answer that the Linux kernel community built. It lets you load a small program directly into the kernel at runtime. The program attaches to the event you care about — a syscall, a function entry, a network packet, a hardware counter. It runs when that event fires. It collects exactly the data you asked for. It hands that data to your userspace tool. And it does all of this with overhead typically under 1% — safe to run in production, forever, on every machine you own.

## Mental Model

Think of the Linux kernel as a live concert venue. Your applications are performers on stage. The venue has strict security — no one gets backstage without authorization, and unauthorized modifications to the stage can bring the whole show down.

eBPF is like hiring an official venue auditor. The auditor gets a special backstage pass, but before they're let in, security reviews their badge and their toolkit: *can you break anything? can you access areas you're not supposed to?* If the audit passes, the auditor goes backstage and watches — taking notes at every door, timing how long each performer waits, counting who enters which room. The auditor hands you a report in real time.

The critical detail: the auditor is inside the venue. They don't have to wait outside and guess based on what performers report. They see the raw, unfiltered truth. But they also can't break anything — security checked their toolkit before they went in.

That security check is the eBPF verifier. The backstage pass is the kernel's program loader. The report is your eBPF map.

## Theory

### What eBPF Is

eBPF is a kernel subsystem that allows loading sandboxed programs into the kernel at runtime, without modifying the kernel source or loading a kernel module. These programs are written in a restricted subset of C, compiled to eBPF bytecode, verified for safety by the kernel, JIT-compiled to native machine code, and then attached to hook points throughout the kernel.

The name "extended Berkeley Packet Filter" is historical. The original BPF was a 1992 mechanism for filtering network packets in the kernel (used by `tcpdump`). Extended BPF, added to the kernel around version 3.18 and substantially expanded through 4.x and 5.x, turned that packet filter into a general-purpose in-kernel execution environment.

Today, eBPF programs can observe and interact with nearly every part of the kernel.

### eBPF Program Types

Each eBPF program has a type that determines where it can attach and what data it can access. The major types are:

**kprobe / kretprobe** — attach to any kernel function entry or return. You can observe the function's arguments on entry and its return value on exit. Used heavily for kernel performance analysis.

**uprobe / uretprobe** — attach to any userspace function in any process, including library functions. You can trace calls to `malloc`, `SSL_write`, or any function in any binary without modifying it.

**tracepoint** — attach to stable, named trace points that kernel developers have deliberately placed in the code. Unlike kprobes (which can attach to any function, including internal ones that may change between kernel versions), tracepoints have a stable ABI. Preferred when a suitable tracepoint exists.

**XDP (eXpress Data Path)** — attaches at the network driver level, before the kernel networking stack processes the packet. The earliest possible hook point for network packets — useful for packet filtering, DDoS mitigation, and load balancing at wire speed.

**TC (Traffic Control)** — attaches to the TC layer of the networking stack, after the driver but before routing. Can inspect and modify packets in both directions.

**Socket filter** — attaches to a socket and filters which packets are delivered to that socket. The original BPF use case.

**LSM (Linux Security Module)** — attaches to security hook points throughout the kernel. Used to implement custom security policies without writing a full LSM module. Requires kernel 5.7+.

**Perf events** — attaches to hardware performance counters, software counters, or sampling events. Used for CPU profiling and performance analysis.

### eBPF Maps

eBPF programs run in kernel space, but your analysis happens in userspace. The bridge between them is the **eBPF map** — a kernel data structure that both the eBPF program and your userspace tool can read and write.

Maps are typed key-value stores. The kernel provides many types:

- `BPF_MAP_TYPE_HASH` — a hash table, fast lookups by key
- `BPF_MAP_TYPE_ARRAY` — array indexed by integer, often used for aggregated counters
- `BPF_MAP_TYPE_PERF_EVENT_ARRAY` — a ring buffer for streaming individual events to userspace
- `BPF_MAP_TYPE_HISTOGRAM` — used in bpftrace's `hist()` to maintain in-kernel histograms

The pattern is: your eBPF program fires on every event, updates a counter or histogram in a map, and then returns. Your userspace tool reads the map periodically and displays the aggregated results. This aggregation-in-kernel is why eBPF has such low overhead — most events never cross the kernel/userspace boundary as individual records. Only summaries do.

### The eBPF Verifier

Before any eBPF program is loaded, the kernel passes it through the **verifier** — a static analysis engine that determines whether the program is safe to run.

The verifier performs an abstract interpretation of the program: it simulates every possible execution path, tracking the type and value range of every register and stack slot at every instruction. It rejects programs that:

- Access memory out of bounds (it knows the size of every map value and every packet buffer)
- Follow invalid pointers or dereference null
- Could run forever (unbounded loops — programs must always terminate)
- Call arbitrary kernel functions (only a whitelisted set of BPF helper functions may be called)
- Leak kernel memory to userspace
- Access the wrong data for their program type

If any path violates any rule, the program is rejected before it ever executes a single instruction. This is the guarantee that makes eBPF safe: a loaded eBPF program has been formally verified to be safe by the kernel, regardless of who wrote it or what it does.

### CO-RE: Compile Once, Run Everywhere

One of the hardest problems with kernel programs is portability. The kernel's internal data structures — `struct task_struct`, `struct sk_buff`, etc. — have different field offsets across kernel versions. A program compiled for kernel 5.15 might read the wrong bytes when run on kernel 6.1.

**CO-RE (Compile Once, Run Everywhere)** solves this. It combines three pieces:

**BTF (BPF Type Format)** — a compact representation of the kernel's type information, similar to DWARF debug info but much smaller. Modern kernels ship with BTF enabled (`CONFIG_DEBUG_INFO_BTF=y`). You can verify this with `ls /sys/kernel/btf/vmlinux`.

**libbpf** — the userspace loading library that reads the BTF from both the compiled eBPF program and the running kernel, and **relocates** field accesses at load time to use the correct offsets for the current kernel.

**Clang compiler extensions** — annotations in eBPF C code (like `__builtin_preserve_access_index`) that tell the compiler to emit relocatable field accesses rather than hard-coded offsets.

With CO-RE, you compile your eBPF program once and ship it as a static binary. The same binary runs correctly on any kernel that has BTF enabled — typically kernel 5.2+ — without recompilation. This is the modern way to write production eBPF tools.

## How It Works (Mechanics)

Here is the complete lifecycle of an eBPF program, from writing to running:

```text
# Conceptual flow — not a real command sequence
1. Developer writes eBPF C code
         ↓
2. Clang compiles to eBPF bytecode (.o file)
         ↓
3. Userspace loader (libbpf / bpftrace / BCC) opens the .o file
         ↓
4. Loader calls bpf(BPF_PROG_LOAD, ...) syscall
         ↓
5. Kernel runs the verifier — full static analysis
         ↓ (rejected → EACCES + verifier log)
         ↓ (accepted → JIT compile to native code)
         ↓
6. Loader attaches program to hook (kprobe, tracepoint, XDP, etc.)
         ↓
7. Program runs on every matching event — updates maps
         ↓
8. Userspace tool reads maps → displays results
```

The `bpf()` syscall is the entry point for all eBPF operations: loading programs, creating maps, attaching programs to hooks, and reading map data. You can trace eBPF activity itself with `strace -e bpf <command>`.

When the verifier accepts a program, the kernel's JIT compiler converts the eBPF bytecode to native x86-64 (or ARM64, or other) machine code. The compiled native code is stored in kernel memory and called directly when the hook fires. This means the per-event cost of an eBPF program is a native function call — nanoseconds, not microseconds.

For kprobe attachment, the kernel writes a trap instruction (`int3` on x86) at the probe point in the kernel function. When that instruction executes, it triggers a trap handler which invokes the eBPF program. The program runs, updates its maps, and returns. The kernel resumes executing the traced function from the next instruction.

For XDP, the attachment is at the network driver's receive path. When a packet arrives at the NIC, the driver calls the attached XDP program *before* allocating an `sk_buff` (the kernel's packet structure). This is why XDP is so fast — the kernel hasn't done any of its normal packet processing yet. The XDP program returns a verdict: `XDP_PASS` (continue to normal stack), `XDP_DROP` (discard), `XDP_TX` (retransmit on the same interface), or `XDP_REDIRECT` (forward to another interface).

## Basic Example

The fastest way to experience eBPF is through `bpftrace`, which provides a high-level language for one-liner tracing.

```bash
# Install bpftrace (requires root to run)
# Debian/Ubuntu: apt install bpftrace
# RHEL/Fedora:   dnf install bpftrace

# Trace every file open system call across the entire system
# Shows: process name, PID, and the filename being opened
sudo bpftrace -e 'tracepoint:syscalls:sys_enter_openat { printf("%-16s %-6d %s\n", comm, pid, str(args.filename)); }'
```

```text
nginx            1234   /var/www/html/index.html
sshd             891    /etc/ssh/sshd_config
systemd-resolve  612    /run/systemd/resolve/stub-resolv.conf
python3          4521   /usr/lib/python3/dist-packages/apt/__init__.py
nginx            1234   /var/log/nginx/access.log
...
```

You're watching every `openat()` syscall on the entire machine in real time — zero modification to any running process, no restarts, no recompilation. Press Ctrl-C to stop. The overhead is negligible.

Now try a histogram — tracking how many bytes are returned per `read()` call:

```bash
# Print a histogram of vfs_read() return values (bytes read per call)
# @bytes = accumulates values into a power-of-2 histogram
sudo bpftrace -e 'kretprobe:vfs_read { @bytes = hist(retval); }'
```

```text
^C

@bytes:
[0]                  847 |@@@@@@@@@@                                          |
[1]                 4321 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@|
[2, 4)              1203 |@@@@@@@@@@@@@@@                                     |
[4, 8)               891 |@@@@@@@@@@@                                         |
[8, 16)              234 |@@                                                  |
[16, 32)             189 |@@                                                  |
...
[4K, 8K)             421 |@@@@@                                               |
[8K, 16K)            103 |@                                                   |
```

The histogram is computed entirely inside the kernel, in the map. Only the final summary crosses the kernel/userspace boundary when you press Ctrl-C.

Now try this yourself. Run the `openat` one-liner in one terminal, then in another terminal run `ls /etc/` or start a process. Watch your trace terminal light up. Then modify the program to filter by a specific process name: change `printf(...)` to `if (comm == "nginx") { printf(...); }` and re-run. You're writing kernel instrumentation in a one-liner.

## Real-World Example

### Diagnosing Slow Process Startup with BCC Tools

A Python service is taking 4 seconds to start instead of the usual 0.5 seconds. The service has no debug logging during startup. Here's how to diagnose it with BCC tools.

```bash
# Install BCC tools
# Debian/Ubuntu: apt install bpfcc-tools
# RHEL/Fedora:   dnf install bcc-tools
# Tools are installed as opensnoop-bpfcc, execsnoop-bpfcc, etc. on Debian
# On RHEL they're installed directly as opensnoop, execsnoop, etc.

# First: watch which files it opens during startup
# Run this BEFORE starting the service, then start the service in another terminal
sudo opensnoop -p $(pgrep -f "myservice") 2>/dev/null
# Or trace by name:
sudo opensnoop -n python3
```

```text
PID    COMM               FD ERR PATH
4821   python3             3   0 /usr/lib/python3.11/importlib/_bootstrap.py
4821   python3             3   0 /usr/lib/python3.11/importlib/_bootstrap_external.py
4821   python3             3   0 /usr/lib/python3.11/zipimport.py
...
4821   python3            -1   2 /etc/ssl/certs/ca-bundle.crt          ← ENOENT
4821   python3            -1   2 /etc/pki/tls/certs/ca-bundle.crt      ← ENOENT
4821   python3            -1   2 /usr/share/ssl-certs/ca-bundle.crt    ← ENOENT
4821   python3             3   0 /etc/ssl/certs/ca-certificates.crt    ← found after 8 tries
...
```

The `ERR` column shows error codes. `2` is `ENOENT` — file not found. The service is searching for an SSL certificate bundle by trying eight different paths before finding the right one. Each miss involves a `stat()` and an `openat()` returning ENOENT — but you wouldn't see this without eBPF.

Now look at the disk I/O latency to determine whether the filesystem is the bottleneck:

```bash
# Histogram of block I/O latency in microseconds during the startup
# -m flag: output in milliseconds
sudo biolatency -m 1 5   # 1-second intervals, 5 samples
```

```text
     msecs               : count     distribution
         0 -> 1          : 8421     |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@|
         2 -> 3          : 231      |@                                       |
         4 -> 7          : 18       |                                        |
         8 -> 15         : 3        |                                        |
        16 -> 31         : 1        |                                        |
```

Most I/O is under 1ms, which is healthy. The slowdown isn't disk latency — it's the wasted syscall attempts on the wrong certificate paths.

Now use `tcpconnect` to verify the service isn't making unexpected network calls during startup:

```bash
# Show every TCP connection made by the service during startup
sudo tcpconnect -p 4821
```

```text
PID    COMM         IP SADDR            DADDR            DPORT
4821   python3      4  10.0.0.5         169.254.169.254  80    ← AWS metadata endpoint
```

There it is: the service is making an HTTP request to the AWS metadata endpoint (`169.254.169.254`) during startup — presumably to fetch credentials. That call is timing out for 3 seconds before falling back to a local config file. Without eBPF, you would never have seen this from outside the process.

The fix: set the `AWS_DEFAULT_REGION` and credential environment variables to skip the metadata fetch on this non-AWS server. Startup time: 0.5 seconds.

### Building a Custom Latency Probe with bpftrace

Once you've identified a slow function, you can measure it precisely. Suppose `vfs_write` latency is suspect:

```bash
# Measure latency of vfs_write() per process, print histogram on exit
sudo bpftrace -e '
kprobe:vfs_write   { @start[tid] = nsecs; }
kretprobe:vfs_write
/@start[tid]/
{
  @latency_us = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'
```

```text
^C

@latency_us:
[0]                    3 |                                                    |
[1]                 8921 |@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@|
[2, 4)              2341 |@@@@@@@@@@@@@                                       |
[4, 8)               189 |@                                                   |
...
[1M, 2M)               2 |                                                    |   ← outliers > 1 second
```

Those two outliers above 1 second — that's your latency spike. The kprobe recorded entry time per-thread in `@start[tid]`, the kretprobe computed the delta and added it to a histogram. All computation in the kernel. The 2-second call latency is now visible, actionable, and quantified.

## ⚠️ Gotchas

### Unbounded Loops Are Rejected by the Verifier

The verifier must guarantee termination. If you write a loop that the verifier cannot prove will exit, the program is rejected at load time.

```c
// This will NOT load — verifier cannot prove the loop terminates
for (int i = 0; ; i++) {
    // do something
    if (map_lookup(&data, &i)) break;
}
```

```text
libbpf: load bpf program failed: Permission denied
libbpf: -- BEGIN DUMP LOG ---
0: R1=ctx(off=0,imm=0) R10=fp0
...
back-edge from insn 12 to 7   ← verifier detected unbounded loop
```

Use bounded loops with an explicit maximum count, which the verifier can check statically:

```c
// This works — verifier can prove this exits after at most 100 iterations
for (int i = 0; i < 100; i++) {
    // do something
    if (map_lookup(&data, &i)) break;
}
```

Alternatively, use **tail calls** — a BPF mechanism where one eBPF program can call another, up to a chain depth of 33 — to express iteration that exceeds a single program's instruction limit.

### bpftrace Requires Root and Sees Everything

`bpftrace` runs as root, and when it attaches a `uprobe` or reads `/proc/PID/mem`, it has full visibility into every process on the machine. In a multi-tenant environment — a shared server where multiple teams' processes run — this is a significant security consideration.

```bash
# This shows the arguments to SSL_write() across all processes — including passwords
sudo bpftrace -e 'uprobe:/usr/lib/x86_64-linux-gnu/libssl.so.3:SSL_write { printf("%s\n", str(arg1)); }'
```

The eBPF verifier ensures the program is safe from the kernel's perspective. It does not restrict what data an authorized root user can observe. Running bpftrace in production is a privileged operation. Treat access to bpftrace like access to `strace` or `gdb` on production — controlled, audited, and scoped to the specific investigation.

### BCC Compiles at Runtime and Needs Kernel Headers

BCC tools work differently from bpftrace and libbpf/CO-RE programs. BCC compiles eBPF C code to bytecode at the moment you run the tool, using Clang/LLVM and the running kernel's headers. If the kernel headers are missing, BCC fails with a cryptic error.

```bash
sudo opensnoop
```

```text
modprobe: ERROR: could not find module by name='bpf'
Unable to load BPF program
```

Or, more commonly on newer systems:

```text
/virtual/main.c:6:10: fatal error: 'linux/types.h' file not found
```

The fix is always the same: install the kernel headers that match your running kernel.

```bash
# Debian/Ubuntu
sudo apt install linux-headers-$(uname -r)

# RHEL/Fedora/Rocky
sudo dnf install kernel-devel-$(uname -r)

# Verify they're in place
ls /usr/src/linux-headers-$(uname -r)/include/linux/types.h
```

This is one reason the community is moving toward CO-RE-based tools: they compile to a single binary that ships with its type information baked in via BTF, and don't require kernel headers on the target machine.

### Kernel Version Requirements Are Not Uniform

Different eBPF features require different minimum kernel versions. Hitting a missing feature produces confusing errors.

```text
libbpf: failed to load object 'myprobe.bpf.o'
libbpf: kernel rejected BPF program with error -22 (EINVAL)
```

Key thresholds:

- **Kernel 4.18+** — core eBPF program types (kprobe, tracepoint, XDP) work reliably
- **Kernel 5.2+** — BTF is available in-kernel (`/sys/kernel/btf/vmlinux` exists); CO-RE becomes viable
- **Kernel 5.7+** — LSM BPF hooks (`BPF_PROG_TYPE_LSM`) are available
- **Kernel 5.8+** — `BPF_MAP_TYPE_RINGBUF` (the modern replacement for perf event arrays)
- **Kernel 5.13+** — `bpf_loop()` helper; bounded iteration without explicit loops

Before deploying an eBPF-based tool, check the kernel version on your target systems and verify the tool's stated requirements:

```bash
uname -r
# If using BCC: also verify BTF support
ls /sys/kernel/btf/vmlinux && echo "BTF present" || echo "BTF missing — CO-RE unavailable"
```

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `bpftrace` | Interactive one-liner tracing, custom probes, investigative work | High-level language; compiled and loaded on the fly; requires root; kernel 4.9+ for basic use, 5.2+ for CO-RE |
| BCC tools (`opensnoop`, `execsnoop`, `biolatency`, `tcpconnect`, etc.) | Immediate answers with pre-built tools; 70+ ready-to-run scripts | Compiles at runtime; requires kernel headers; use the right tool name for your distro |
| `libbpf` + CO-RE | Writing portable eBPF tools that ship as static binaries; production agents | The modern way; requires BTF on target kernel (5.2+); steeper initial learning curve |
| `perf` | CPU profiling, hardware counter analysis, flame graphs | Ships with the kernel; does not require eBPF; limited to profiling use cases |
| `strace` | Debugging individual processes in development; when overhead is acceptable | 10–100x overhead; never in production on hot paths |
| `ftrace` | Low-level kernel function tracing without installing additional tools | Built into the kernel; accessible via `/sys/kernel/debug/tracing`; less ergonomic than bpftrace |
| Cilium | Kubernetes cluster networking, network policy, service mesh | Not a debugging tool — a full CNI plugin; uses eBPF for packet routing and policy enforcement; replaced Flannel and Calico as the most-used CNI in production in 2025 |

The right starting point for most engineers is `bpftrace` for investigation and BCC tools for common questions. When you need a portable agent — something that runs continuously on all your hosts — `libbpf` + CO-RE is where you end up.

## Trade-offs

**Where this shines:**

- Production observability with negligible overhead — eBPF programs aggregate data in-kernel; only summaries cross to userspace. Always-on profiling at roughly 1% CPU overhead across an entire cluster is achievable.
- Universal hook points — any kernel function, any userspace function, any network packet, any hardware event. Nothing is hidden from an eBPF program if you have the appropriate kernel version and permissions.
- No instrumentation required in the target — you can trace a third-party binary, a Go runtime, an OpenSSL library, without modifying or restarting it.
- The verifier provides strong safety guarantees — a loaded eBPF program cannot crash the kernel, access arbitrary memory, or run forever. This is a formal property, not a convention.

**Where this costs you:**

- Steep learning curve to go beyond one-liners — writing production libbpf programs requires understanding the verifier's rules, BTF relocations, map types, and the helper function API. This is genuinely complex.
- Root (or `CAP_BPF`) required — there is no safe way to load eBPF programs as a non-privileged user in most production configurations. This limits use in multi-tenant environments.
- Verifier complexity — the verifier rejects programs it cannot prove safe, and verifier error messages can be extremely opaque. A valid-looking program may be rejected because the verifier cannot track a value range precisely enough, even though the program would be safe at runtime.
- Kernel version fragmentation — the eBPF subsystem evolves rapidly. A program using kernel 5.8 features cannot be used on a 5.4 kernel. Managing this across a heterogeneous fleet requires careful testing.
- BTF requirement for CO-RE — systems running kernels below 5.2, or with `CONFIG_DEBUG_INFO_BTF` disabled, cannot run CO-RE programs. Custom/embedded kernels frequently have this disabled.

**Performance note:** The overhead of a well-written eBPF program is typically measured in tens of nanoseconds per event — a native function call. XDP programs have demonstrated the ability to drop malicious packets at rates exceeding 14 million packets per second on commodity hardware, at far lower CPU cost than iptables, which must process each packet through the full networking stack.

## When to Use / When to Avoid

✅ **Use this when:**

- You need production observability on a running system without acceptable overhead for strace or a restart for application-level instrumentation.
- You're building a continuous monitoring agent — syscall auditing, network flow logging, latency percentile tracking — that must run always-on.
- You're debugging a third-party binary or a library (e.g., tracing OpenSSL writes to see plaintext before encryption) without source access.
- You're implementing network-layer filtering or load balancing that must handle millions of packets per second — XDP is the right layer.
- You need portable, shipmable eBPF tools that work across kernel versions without recompilation — use libbpf + CO-RE.
- You're running Kubernetes and evaluating CNI options — Cilium's eBPF-based networking outperforms iptables-based approaches at scale.

❌ **Avoid this when:**

- The kernel is below 4.18 — basic eBPF works but is too limited for serious use; treat older kernels as eBPF-unavailable.
- You need to modify kernel behavior in complex ways — eBPF can filter and redirect, but for deep kernel modifications, a kernel module or upstream patch is the right tool.
- The problem can be solved with a one-time `strace` run on a development machine — not every diagnostic needs eBPF; strace is still the right choice for development debugging.
- You're working inside a minimal container (Alpine, distroless) without host kernel access — eBPF runs in the kernel, not in the container; it belongs on the host or in a privileged DaemonSet.

⚠️ **Common mistakes:**

- Running `bpftrace` without realizing it's tracing the whole system, not just the target process — always add a filter: `if (pid == target_pid)` or use the `-p` flag where available.
- Using BCC tools without installing kernel headers and being confused by the resulting build errors — always install `linux-headers-$(uname -r)` before using BCC.
- Forgetting to call `delete(@start[tid])` in kretprobe handlers — map entries for threads that never hit the return probe accumulate and eventually exhaust map space.
- Writing `args->filename` instead of `args.filename` in bpftrace — the syntax changed in bpftrace 0.19; older one-liners from blog posts may use the old form.

## Veteran Perspective

Most engineers discover eBPF the wrong way: they hear it described as "you can run programs in the kernel" and immediately think kernel modules. That framing is wrong and leads to frustration. eBPF programs are not general-purpose kernel code. They are **constrained probes**. They cannot call arbitrary kernel functions. They cannot block. They cannot allocate memory freely. The verifier will reject anything it can't prove safe. Once you accept that the constraint is the point — not a limitation to work around — eBPF becomes a powerful and reliable tool rather than a frustrating one.

The overhead comparison with strace is real but often misunderstood. strace has 10–100x overhead because it uses `ptrace`, which stops the traced process on every syscall — entry and exit — to let the tracer read and format the data. The overhead scales linearly with syscall rate. eBPF's overhead is almost entirely in the eBPF program's execution per event, which is nanoseconds of native JIT-compiled code. These are fundamentally different architectures, not just different implementations of the same idea.

The verifier is where most intermediate users hit a wall. The error messages are notoriously bad. `libbpf: -- BEGIN DUMP LOG --` followed by 200 lines of register state is not helpful when you're trying to debug why a valid-looking program was rejected. The practical answer is: keep programs small, split complex logic into multiple programs, avoid dynamic map keys that the verifier can't track, and don't trust that a program which worked on kernel X will work on kernel Y. The verifier is not stable across kernel versions — it gets stricter and more permissive at different points. Test on your actual target kernels.

Cilium is worth understanding even if you're not directly writing eBPF. In 2025, Cilium surpassed Flannel and Calico to become the most widely adopted CNI plugin in production Kubernetes clusters. Almost every large-scale Kubernetes deployment is running eBPF at the networking layer, whether the engineers on the team know it or not. If you're troubleshooting Kubernetes networking, Cilium's `hubble` CLI — built on eBPF — gives you real-time flow visibility that iptables-based solutions can't match.

The deeper point about eBPF is that it's not just a debugging tool. It's a programmable platform inside the kernel. Cilium is a network stack. Falco is a security monitoring system. Pixie is an observability platform. These are not toys — they're infrastructure that major companies run in production at petabyte scale. The entry point is a `bpftrace` one-liner. The destination, if you keep going, is writing the infrastructure that everyone else's applications run on top of.

This is also the right place to say something about where the Linux journey leads. eBPF is not the end — it's a window into the part of the kernel that was previously invisible. Every one-liner you write reveals something about how Linux actually works: how the scheduler decides which process runs next, how the networking stack routes packets, how filesystems serve reads. The tools in this curriculum — from `ls` in Phase 1 to eBPF in Phase 14 — are all interfaces to the same underlying system. The system is worth understanding. You've put in the work to get here. Don't stop.

## Try It Yourself

1. Run the `openat` bpftrace one-liner from the Basic Example section. In another terminal, run `cat /etc/hostname`. Watch the trace output. Identify the line that shows `cat` opening `/etc/hostname`. Requires: `sudo`, `bpftrace` installed.

2. Use `execsnoop` (from BCC tools) to watch which new processes are spawned on your system over a 30-second window. Run it, then in another terminal run `apt list --installed` or `ls /etc`. Read the output: what processes does `apt` or `ls` spawn? Requires: `sudo`, `bcc-tools` or `bpfcc-tools` installed.

3. Combine bpftrace with a filter: modify the openat one-liner to show only file opens performed by `sshd`. Hint: change the program body to `if (comm == "sshd") { printf(...); }`. Then SSH into the machine in another terminal and watch what `sshd` opens during login.

4. Use `biolatency` to measure your disk I/O latency distribution over 10 seconds: `sudo biolatency 1 10`. Then create disk I/O by running `find / -type f > /dev/null` in another terminal. Compare the latency distribution before and during the `find`. What changed in the histogram?

5. **Challenge:** Write a bpftrace script (not a one-liner) that tracks the top 10 processes by number of `openat()` syscalls over a 30-second window, and prints them sorted by count when you press Ctrl-C. Hint: use a `@count[comm]` map and print it in the `END` probe. You'll need to research bpftrace's `print()` and map sorting. Test in a VM. This is the kind of thing you'd write during a real incident to find a process that's hammering the filesystem.

## Summary

- eBPF lets you load verified, sandboxed programs into the Linux kernel at runtime — attached to syscalls, functions, network packets, or hardware events — with overhead typically under 1% vs. strace's 10–100x slowdown.
- The **eBPF verifier** statically checks every possible execution path before a program loads, guaranteeing it cannot crash the kernel, access invalid memory, or run indefinitely — this is a formal safety property, not a convention.
- **eBPF maps** are the shared data structures between kernel programs and userspace tools; programs aggregate data in-kernel and only summaries cross the kernel/userspace boundary, which is why eBPF overhead is so low.
- **CO-RE (Compile Once, Run Everywhere)** uses BTF type information to make eBPF programs portable across kernel versions — compile once, ship as a binary, run correctly on any kernel with BTF support (5.2+).
- The practical toolchain is: `bpftrace` for investigation and one-liners, BCC tools (`opensnoop`, `execsnoop`, `biolatency`, `tcpconnect`) for common pre-built questions, and `libbpf` + CO-RE for shipping production agents; Cilium is what eBPF looks like at the Kubernetes networking layer.
