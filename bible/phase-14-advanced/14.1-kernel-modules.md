# [14.1] — Kernel Modules

> The Linux kernel is not a monolith you rebuild every time you add a driver — it is a core with a loading dock, and kernel modules are the cargo: discrete pieces of kernel code that can arrive, do their job, and leave without the system ever stopping.

## The Problem This Solves

Early UNIX kernels were compiled as a single binary. If you needed support for a new piece of hardware — a SCSI controller, a new filesystem, a network protocol — you modified the kernel source, recompiled the entire kernel, rebooted, and hoped the new binary worked. On production servers, this was the procedure. Every driver change, every filesystem addition, every experimental protocol meant a full rebuild and a reboot.

The second problem was bloat. A kernel compiled with support for every conceivable driver and filesystem was enormous. Most of it sat idle in memory, wasting resources for hardware that wasn't present and filesystems that weren't mounted. But removing a driver meant recompiling — so administrators erred toward including everything just in case.

The third problem was deployment. Linux distributions needed to ship a single kernel binary that could work on hardware they'd never seen. Compile in too little, and the installer fails on unusual hardware. Compile in everything, and the kernel is a 200 MB blob that still might miss something.

Loadable kernel modules solved all three. A distribution ships a lean core kernel and a library of `.ko` files — compiled module objects — for thousands of drivers. On first boot, the system detects hardware and loads only the modules that match. Modules can be added after deployment without a reboot. The kernel stays small in memory. And driver updates don't require a full kernel rebuild.

The consequence you need to understand: modules run in kernel space. There is no isolation, no memory protection, no process boundary between a module and the rest of the kernel. A bug in a user program crashes that program. A bug in a kernel module crashes the entire system.

## Mental Model

Picture the kernel as an aircraft carrier already at sea. The ship's core — engines, navigation, command systems — is built in and cannot be changed while underway. But the flight deck has a series of standardized attachment points. Aircraft can land, lock in, and operate as part of the carrier's mission. When a mission is over, the aircraft detaches and the deck is clear again.

Kernel modules are the aircraft. They attach to standardized hooks in the kernel (`module_init`, `module_exit`, the device model, the VFS layer), operate with the full authority and resources of the carrier, and can be detached cleanly when no longer needed. The ship never stops. The attachment and detachment happen while everything is running.

The critical part of the analogy: an aircraft that malfunctions mid-flight doesn't just crash itself — it can damage the deck and take down the carrier. That's why you test kernel code in a VM before you ever load it on a real system.

## Theory

A kernel module is a compiled ELF object file with the extension `.ko` (kernel object). It contains compiled C code, metadata, and relocation information that allows the kernel to link it into the running kernel image at load time.

Every module must declare a license with `MODULE_LICENSE()`. This is not a formality. The kernel uses this to determine whether to taint itself. A tainted kernel is one that has loaded code under a non-GPL-compatible license — when a tainted kernel oopses or panics, kernel developers are entitled to ignore the bug report because the non-GPL code may be the cause. Shipping a module under `"GPL"` grants users the rights the GPL requires and keeps the kernel untainted.

Modules interact with the rest of the kernel through the kernel's exported symbol table. When a module references a function like `printk()` or a data structure defined in the kernel core, the kernel's module loader resolves that symbol at load time. If a symbol is not exported (via `EXPORT_SYMBOL()` or `EXPORT_SYMBOL_GPL()`), a module cannot use it. This is the enforcement mechanism for the kernel's internal API.

The module system tracks three things for every loaded module:

**Name.** The module's identifier within the kernel — shown in `lsmod`, used by `modprobe` and `rmmod`.

**Size.** How much kernel memory the module occupies.

**Use count.** How many other modules or kernel subsystems hold a reference to this module. A module with a use count greater than zero cannot be unloaded — the kernel refuses because unloading it would leave dangling references that would crash the system on next access.

Dependencies are tracked in the module's metadata and in a file called `modules.dep`, generated by `depmod`. When you load a module that depends on another, the kernel — through `modprobe` — resolves the full dependency chain and loads every required module in the correct order.

The kernel distinguishes three sources of modules:

**In-tree modules** ship with the kernel source itself, compiled against the exact kernel they run on.

**Out-of-tree modules** live outside the kernel source tree and must be compiled separately, against the kernel headers for the running kernel. Every NVIDIA proprietary driver, every VirtualBox kernel component, every hardware driver that isn't accepted upstream is an out-of-tree module.

**DKMS-managed modules** are out-of-tree modules whose source is registered with the DKMS framework. When a new kernel is installed, DKMS automatically recompiles them for the new version.

The kernel versions its internal ABI through a `vermagic` string embedded in every module. When you try to load a `.ko`, the kernel checks that the module's `vermagic` matches the running kernel exactly — same version, same configuration flags. A mismatch is a hard failure. This is why you cannot take a `.ko` compiled for kernel `6.5.0-15-generic` and load it on `6.5.0-17-generic`.

## How It Works (Mechanics)

### Loading a module: what the kernel actually does

When you run `modprobe e1000e` (the Intel gigabit NIC driver), the sequence is:

1. `modprobe` reads `/lib/modules/$(uname -r)/modules.dep` to build the dependency graph for `e1000e`.
2. For each dependency not yet loaded, `modprobe` calls `insmod` on the dependency's `.ko` file first, in dependency order.
3. For the target module itself, `modprobe` reads the `.ko` file and passes it to the kernel via the `init_module()` or `finit_module()` system call.
4. The kernel's module loader verifies the `vermagic` string. Mismatch → fail immediately.
5. The loader allocates a `struct module` in kernel memory, copies the module's sections into kernel memory, and performs ELF relocation — resolving every symbol the module references against the kernel's live symbol table.
6. If relocation succeeds, the kernel calls the module's `init` function (registered via `module_init()`).
7. The `init` function does whatever setup the driver needs: registering a device, hooking into the VFS, allocating IRQs. If it returns 0, the module is now live. If it returns a negative error code, the kernel unwinds and frees the memory.

```text
# Conceptual flow — not a real command
modprobe e1000e
  → read modules.dep
  → resolve dependencies
  → for each dep: finit_module() syscall
      → verify vermagic
      → ELF relocation
      → call module_init()
  → finit_module() for e1000e itself
      → verify vermagic
      → ELF relocation
      → call e1000e_init_module()
        → pci_register_driver()
        → return 0  ← module is live
```

### Unloading a module

`rmmod` issues a `delete_module()` system call. The kernel checks the module's use count. If it's greater than zero, the syscall fails with `EBUSY`. If it's zero, the kernel calls the module's `exit` function (registered via `module_exit()`), which must undo everything `init` did — deregister the device, free IRQs, release memory. After `exit` returns, the kernel frees the module's memory.

### lsmod and the module table

`lsmod` reads `/proc/modules`, which the kernel maintains as a live view of loaded modules. Each line contains the module name, size in bytes, use count, and the names of modules that depend on it (the "used by" list).

```text
# /proc/modules format — each column's meaning:
# name      size    use-count    used-by-list
# e1000e    245760  0            -
# ptp       28672   1            e1000e
```

The `lsmod` output reformats this: the "Used By" column shows which modules reference this one — the reverse of the dependency direction. If `ptp` shows `e1000e` in its "Used By" column, it means `e1000e` depends on `ptp` (not the other way around). This trips everyone up the first time.

### Module parameters

Modules can export parameters — kernel-space variables that can be set at load time or, if the module allows it, changed at runtime through `/sys/module/<name>/parameters/`.

```bash
# Load the usbcore module with a specific parameter
modprobe usbcore autosuspend=2
```

The parameter is declared in the module source with `module_param()`. At runtime:

```bash
# Read the current value of a parameter for a loaded module
cat /sys/module/usbcore/parameters/autosuspend
```

```text
2
```

## Real-World Example

A common scenario on any server: a new piece of hardware ships with an out-of-tree driver, or a kernel update breaks a DKMS-managed module and you need to diagnose why modules aren't loading at boot.

### Diagnosing a module that won't load

```bash
# Check if the module is loaded at all
lsmod | grep -i nouveau

# If nothing prints, check what the kernel said when it tried
journalctl -k | grep -i nouveau

# Check if the module is blacklisted
grep -r nouveau /etc/modprobe.d/

# Force-probe it manually and watch for errors
sudo modprobe nouveau 2>&1
```

The `journalctl -k` output (kernel ring buffer via the journal) tells you exactly why a module failed — `vermagic` mismatch, missing dependency, `init` returned an error. `dmesg` is the lower-level alternative if journald isn't running.

### Blacklisting a module to prevent loading at boot

The most common use of `/etc/modprobe.d/` is blacklisting. A typical scenario: the open-source `nouveau` driver conflicts with the proprietary NVIDIA driver, and you need `nouveau` permanently disabled.

```bash
# Create a blacklist file — filename can be anything ending in .conf
sudo tee /etc/modprobe.d/blacklist-nouveau.conf << 'EOF'
blacklist nouveau
options nouveau modeset=0
EOF
```

```bash
# Rebuild initramfs so the blacklist takes effect at boot
# On Debian/Ubuntu:
sudo update-initramfs -u

# On RHEL/Fedora/Rocky:
sudo dracut --force
```

The `options nouveau modeset=0` line is belt-and-suspenders: even if something else loads `nouveau` despite the blacklist, it disables modesetting. Without the initramfs rebuild, the blacklist file exists on disk but the initramfs — which loads modules before your root filesystem is mounted — doesn't know about it. Blacklisting without rebuilding the initramfs has caused more "why is nouveau still loading" questions than I can count.

### Writing, compiling, and loading a minimal out-of-tree module

> ⚠️ **WARNING:** Kernel code runs with full kernel privileges. A bug — a null pointer dereference, an integer overflow, writing past an array bound — does not crash a process. It crashes the entire system, potentially corrupting filesystems in the process. Write kernel modules in a VM only. Never develop kernel code on a system with data you care about.

Create a working directory and the two required files:

```bash
mkdir -p ~/kernel-module-lab && cd ~/kernel-module-lab
```

```c
// hello.c — minimal kernel module
#include <linux/init.h>
#include <linux/module.h>
#include <linux/kernel.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("Minimal hello-world kernel module");
MODULE_VERSION("0.1");

static int __init hello_init(void)
{
    printk(KERN_INFO "hello: module loaded\n");
    return 0;  /* 0 = success; negative errno = failure, module won't load */
}

static void __exit hello_exit(void)
{
    printk(KERN_INFO "hello: module unloaded\n");
}

module_init(hello_init);
module_exit(hello_exit);
```

```makefile
# Makefile — out-of-tree module build
# 'obj-m' tells kbuild this is a module (not built-in)
obj-m += hello.o

# KDIR points to the build infrastructure for the running kernel
KDIR := /lib/modules/$(shell uname -r)/build

all:
	make -C $(KDIR) M=$(PWD) modules

clean:
	make -C $(KDIR) M=$(PWD) clean
```

Note the indentation in the Makefile: those are hard tabs, not spaces. Make requires tabs. Using spaces produces `missing separator` errors that are famously confusing.

```bash
# Install the kernel headers needed for out-of-tree builds
# Debian/Ubuntu:
sudo apt install linux-headers-$(uname -r)

# RHEL/Fedora/Rocky:
sudo dnf install kernel-devel-$(uname -r)

# Build the module
make
```

```text
make -C /lib/modules/6.8.0-45-generic/build M=/home/user/kernel-module-lab modules
make[1]: Entering directory '/usr/src/linux-headers-6.8.0-45-generic'
  CC [M]  /home/user/kernel-module-lab/hello.o
  MODPOST /home/user/kernel-module-lab/Module.symvers
  CC [M]  /home/user/kernel-module-lab/hello.mod.o
  LD [M]  /home/user/kernel-module-lab/hello.ko
make[1]: Leaving directory '/usr/src/linux-headers-6.8.0-45-generic'
```

```bash
# Load the module
sudo insmod hello.ko

# Verify it's loaded
lsmod | grep hello

# Check the kernel log for the printk output
sudo dmesg | tail -3

# Unload it
sudo rmmod hello

# Confirm the exit message
sudo dmesg | tail -3
```

```text
[  892.441203] hello: module loaded
[  897.112044] hello: module unloaded
```

The `insmod` command takes a path to a `.ko` file directly. It does not resolve dependencies. For a production module with dependencies, you'd use `modprobe` instead — but for development and testing, `insmod ./hello.ko` is the right tool because you're explicitly controlling which file gets loaded.

### Managing an out-of-tree module with DKMS

The problem with manually compiled modules: the next kernel update breaks them. You reboot into the new kernel, the module is gone, and whatever depended on it stops working. DKMS solves this by rebuilding registered modules automatically on every kernel install.

```bash
# Install dkms
sudo apt install dkms          # Debian/Ubuntu
sudo dnf install dkms          # RHEL/Fedora/Rocky

# Place your module source where DKMS expects it
sudo mkdir -p /usr/src/hello-0.1
sudo cp hello.c /usr/src/hello-0.1/
sudo cp Makefile /usr/src/hello-0.1/
```

```bash
# Create the DKMS configuration file
sudo tee /usr/src/hello-0.1/dkms.conf << 'EOF'
PACKAGE_NAME="hello"
PACKAGE_VERSION="0.1"
BUILT_MODULE_NAME[0]="hello"
DEST_MODULE_LOCATION[0]="/updates"
AUTOINSTALL="yes"
EOF
```

```bash
# Register the module with DKMS
sudo dkms add -m hello -v 0.1

# Build it for the current kernel
sudo dkms build -m hello -v 0.1

# Install it
sudo dkms install -m hello -v 0.1

# Verify DKMS status
dkms status
```

```text
hello/0.1, 6.8.0-45-generic, x86_64: installed
```

From this point forward, when `apt` or `dnf` installs a new kernel, a post-install hook triggers DKMS to rebuild `hello` for the new kernel version before the system reboots into it.

## ⚠️ Gotchas

### `rmmod` Fails With "Module is in use" — Find What's Blocking It

You want to unload a module. `rmmod` refuses.

```bash
sudo rmmod e1000e
```

```text
rmmod: ERROR: Module e1000e is in use
```

The natural response is to check `lsmod` for the use count:

```bash
lsmod | grep e1000e
```

```text
e1000e                245760  0
```

Use count is zero. Still fails. What's happening? The module might be in use by a network interface that hasn't been released. Or the module might be in use by something that doesn't appear in the `lsmod` use count in the expected way.

The right diagnostic is to look at the "Used by" column for the full dependency chain, and to bring down any interfaces using the driver before unloading:

```bash
# Find the interface this driver manages
ip link show | grep -i "link/ether"

# Bring down the interface first
sudo ip link set eth0 down

# Now try rmmod again — or better, use modprobe -r
sudo modprobe -r e1000e
```

`modprobe -r` is the correct tool for unloading: it removes the named module AND all modules that were pulled in as dependencies and are no longer needed. `rmmod` only removes the named module. `modprobe -r` is what you want in practice.

### Blacklisting Doesn't Unload an Already-Loaded Module

You add a file to `/etc/modprobe.d/`, rebuild the initramfs, and expect the module to disappear. It doesn't.

```bash
# You added this:
# blacklist nouveau

# But the module is still here:
lsmod | grep nouveau
```

```text
nouveau              2936832  3
```

Blacklisting tells the kernel not to load a module in the future. It has no effect on modules already loaded in the current session. To unload a module that's currently running, you must explicitly remove it:

```bash
# Attempt to unload — will fail if the display server is using it
sudo modprobe -r nouveau
```

If the module is in use by a running process (like a display server using the GPU), you must stop that process first, unload the module, and then the blacklist takes effect on the next boot.

### A Module Built for the Wrong Kernel Will Silently Refuse to Load

You compiled a module on a system running kernel `6.5.0-15-generic` and copied the `.ko` to a machine running `6.5.0-17-generic`.

```bash
sudo insmod hello.ko
```

```text
insmod: ERROR: could not insert module hello.ko: Invalid module format
```

"Invalid module format" almost always means a `vermagic` mismatch. The module's embedded kernel version string does not match the running kernel. Confirm:

```bash
# Check what kernel the module was built for
modinfo hello.ko | grep vermagic

# Check the running kernel
uname -r
```

```text
vermagic:       6.5.0-15-generic SMP preempt mod_unload
6.5.0-17-generic
```

The fix is to recompile the module on the target machine (or against its kernel headers). There is no workaround. The `vermagic` check is intentional — an ABI mismatch between a module and the kernel it's loaded into is a guaranteed crash.

### `module_param()` Types Must Match Exactly

A common module development mistake: declaring a parameter as one type and passing a different type at load time.

```bash
# Your module has: module_param(debug_level, int, 0644)
# This is fine:
sudo modprobe mymodule debug_level=2

# This is not:
sudo modprobe mymodule debug_level=true
```

```text
modprobe: ERROR: could not insert 'mymodule': Invalid argument
```

The kernel parses parameter values strictly. `bool` parameters accept `0`/`1` or `Y`/`N`. `int` parameters accept decimal integers. Passing the wrong format returns `EINVAL` and the module never loads. Check `modinfo mymodule` for the `parm:` fields — they show both the parameter name and its type.

### `kernel.modules_disabled = 1` Is a One-Way Door

A hardened system might have `kernel.modules_disabled` set to 1 in `/etc/sysctl.conf` or set via `sysctl -w` after boot.

```bash
sudo sysctl kernel.modules_disabled=1

# Now try to load anything:
sudo modprobe hello
```

```text
modprobe: ERROR: could not insert 'hello': Operation not permitted
```

The expected behavior is that setting it back to 0 re-enables loading:

```bash
sudo sysctl kernel.modules_disabled=0
```

```text
sysctl: setting key "kernel.modules_disabled": Operation not permitted
```

Once `kernel.modules_disabled` is set to 1, it cannot be set back to 0 without a reboot. The kernel enforces this with a write-once semantic on the underlying variable. This is by design — the security property is that once you lock module loading, nothing (not even root) can unlock it without a reboot.

The implication: if you set this in `/etc/sysctl.conf` and later need to load a new driver, you need a reboot. Plan accordingly, and never set it in an automated script unless you are certain all required modules are already loaded.

## Alternatives & Comparisons

| Tool | When to Use | Notes |
|---|---|---|
| `modprobe` | Loading or unloading modules in any real scenario | Resolves dependencies, reads `/etc/modprobe.d/` options, uses `modules.dep` |
| `insmod` | Loading a specific `.ko` file directly during development | No dependency resolution; takes a file path, not a module name |
| `rmmod` | Removing a specific single module | Fails if the module has dependents — must unload those first manually |
| `modprobe -r` | Removing a module and its now-unused dependencies | The right tool in production; cleaner than chaining `rmmod` calls |
| `depmod` | Rebuilding the module dependency database | Run after installing a new `.ko` to `/lib/modules/` manually |

Use `modprobe` for everything in production. `insmod` is for development workflows where you're explicitly loading a freshly compiled `.ko` from a local path. `rmmod` is technically correct but `modprobe -r` is more complete. `depmod` you'll rarely run manually — package managers call it for you.

## Trade-offs

**Where this shines:**

- Hardware support without a kernel rebuild — a new NIC driver or filesystem can be loaded on a running server without a reboot
- Memory efficiency — modules for unused hardware are not resident in memory; only loaded modules consume kernel memory
- Hot-fix capability in some situations — a driver bug fixed in a module can be deployed by unloading the old module and loading the new one, without rebooting (when the module supports it)
- Distribution scalability — a single kernel binary ships with thousands of optional `.ko` files, covering hardware the distribution vendor never tested directly

**Where this costs you:**

- Security surface — every loaded module is kernel code with full privileges; a compromised or malicious module has unrestricted access to the entire system, all memory, all devices
- Stability risk from out-of-tree modules — in-kernel modules go through code review and are tested against the kernel they ship with; out-of-tree modules have no such guarantee, and the interface they rely on can change between kernel versions
- DKMS brittleness — DKMS works reliably when kernel headers are installed before the kernel itself; in automated update pipelines, ordering matters, and a failed DKMS build means a system that boots without the module
- The `vermagic` wall — you cannot share compiled `.ko` files across kernel versions; every deployment of an out-of-tree module requires a build step

**Performance note:** Module load time is typically measured in milliseconds for most drivers. The primary cost is the one-time `insmod`/`modprobe` call. Once loaded, a module's code is in kernel memory and has no per-call overhead compared to built-in kernel code.

## When to Use / When to Avoid

✅ **Use `modprobe` (and dependency management) when:**

- Loading or unloading any module on a production system
- Diagnosing why a device isn't recognized at boot
- Configuring module parameters via `/etc/modprobe.d/` options files
- Implementing blacklists for conflicting drivers

✅ **Use DKMS when:**

- An out-of-tree driver must survive kernel updates (GPU drivers, VPN kernel components, hardware vendor drivers)
- You're packaging a driver for distribution to users who will update their kernels
- A hardware vendor provides a DKMS package — use it instead of manual compilation

✅ **Use `kernel.modules_disabled = 1` when:**

- Hardening a server after all required modules are loaded (typically set late in the boot process)
- Meeting a security compliance requirement that mandates locking module loading post-boot

❌ **Avoid writing kernel modules when:**

- A user-space solution exists — eBPF, FUSE, netfilter user-space helpers, and UIO (Userspace I/O) can solve many problems without kernel code
- You don't have a test environment (VM) to develop and validate in before touching a real system

❌ **Avoid `insmod` on production systems:**

- `insmod` takes a file path and skips dependency resolution and options in `/etc/modprobe.d/`; on a production system, you almost always want the full `modprobe` behavior

⚠️ **Common mistakes:**

- Blacklisting a module and expecting it to be immediately unloaded — blacklist is future-only; current session requires `rmmod` or `modprobe -r`
- Forgetting to rebuild the initramfs after adding a blacklist — the change has no effect at boot without it
- Using `modprobe -r` without checking whether the module's device is in active use — if a filesystem is mounted or a network interface is up, unloading will fail or, worse, leave the system in a degraded state

## Veteran Perspective

The mistake newcomers make is thinking of kernel modules as "plugins" — something like a browser extension that runs in a sandbox. They don't. A kernel module is not sandboxed from the kernel in any way. The moment you call `module_init()`, your code runs with every privilege the kernel has: access to every physical memory address, every I/O port, every interrupt vector. A single erroneous pointer dereference locks up the machine. There is no exception handler that catches your mistake and kills just your module.

The intermediate mistake is underestimating the stability boundary. The Linux kernel has no documented, stable internal API for module authors. The functions and structures you use today may be renamed, restructured, or removed in the next kernel release. This is deliberate — Linus Torvalds has been explicit about it for twenty years. In-tree modules follow the kernel's evolution. Out-of-tree modules don't, and the DKMS build failures you see after kernel updates are usually the kernel's internal API changing underneath a module that wasn't updated to match.

DKMS is a genuinely good solution for the "out-of-tree module survives kernel update" problem, but it only works if the module source is still compatible with the new kernel's headers. DKMS builds, it doesn't fix. A DKMS module that fails to build after a kernel update means a system that boots without that driver — which, if the driver is for the root disk's storage controller, means a system that doesn't boot. The lesson: never put a storage controller driver in DKMS on a production system without a plan for that failure mode.

The `vermagic` check trips up everyone who tries to copy a compiled `.ko` from one machine to another or from a CI pipeline to a server. The check exists for a reason: the kernel's internal structures have different sizes and layouts depending on build-time configuration flags. A module compiled against `CONFIG_PREEMPT_DYNAMIC` has different expectations about kernel data structures than one compiled without it. Loading the wrong module would produce silent memory corruption, not a visible error. The `vermagic` check converts that into a hard failure.

In interview settings, kernel module questions are often a proxy for whether you understand the kernel/user boundary. The question "why can't you share a `.ko` between two servers running the same kernel version?" is designed to see if you know about `vermagic` and build-time configuration differences. The follow-up "how would you handle driver updates across a fleet of servers?" is where DKMS enters the conversation.

One incident worth describing: a team ran a production service on a server with a DKMS-managed module for a custom PCIe device. Automated unattended-upgrades ran overnight, installed a new kernel, DKMS attempted to rebuild, the build failed because a kernel header changed, and the server rebooted into the new kernel without the module. The PCIe device became inaccessible. The service failed. The monitoring alert fired. The on-call engineer spent two hours diagnosing why the device had "disappeared" before finding the DKMS build log. The fix: pin the kernel version in `/etc/apt/preferences.d/` on servers with DKMS-critical drivers, and run a post-upgrade check in CI that validates DKMS status before allowing automated reboots.

## Try It Yourself

1. Run `lsmod` and find the module with the highest use count. Then run `modinfo` on it to see its description, license, and dependencies. What is it? Why does so much depend on it?

2. Run `modinfo ext4` (or `modinfo xfs` depending on your filesystem). Find the `parm:` fields. What parameters does the module accept? What do they do?

3. Write the `hello.c` module and `Makefile` from the Real-World Example section above, build it in a VM, load it with `insmod`, confirm the message in `dmesg`, and unload it. This is the most important exercise in this topic — do it in a VM, not on your main machine.

4. After loading and unloading the hello module, try to unload it a second time. What error do you get? Now try to load it with a typo in the path: `insmod ./helo.ko`. What error do you get? These are the two most common `insmod` errors — learn what they look like.

5. **Challenge (VM only):** Add a `module_param()` to the hello module that accepts an integer called `repeat_count`. In `hello_init()`, print the "module loaded" message `repeat_count` times using a loop. Load the module with `insmod hello.ko repeat_count=5` and verify the output in `dmesg`. Then expose the parameter as writable via sysfs (`module_param(repeat_count, int, 0644)`) and verify it appears under `/sys/module/hello/parameters/`. Changing it at runtime won't retrigger the loop (the init function already ran), but understanding why is the point.

## Summary

- A kernel module is compiled kernel code (`.ko` file) that can be loaded and unloaded into a running kernel without rebooting; modules run in kernel space with full privileges and no sandbox.
- `modprobe` is the right tool for loading and unloading in all production scenarios — it resolves dependencies and reads `/etc/modprobe.d/` options; use `insmod` only during development when you're loading a specific file directly.
- The `vermagic` string embedded in every `.ko` must match the running kernel exactly — version, configuration flags, and all — or the kernel refuses to load it; compiled modules cannot be shared across kernel versions.
- Blacklisting a module in `/etc/modprobe.d/` prevents future loading but does not unload a currently running module; you must also `modprobe -r` the running module and rebuild the initramfs for the change to take effect at boot.
- DKMS manages out-of-tree modules across kernel updates by automatically recompiling them on kernel install; it solves the rebuild problem but not the compatibility problem — if the kernel's internal API changes, the build fails and the module is absent on the next boot.
