## The Problem This Solves

Imagine you're deploying three web services on a single server. All three are Python applications. Service A needs Python 3.8 and a specific version of `cryptography`. Service B needs Python 3.11 and a different version of the same library. Service C ships its own vendored dependencies that conflict with both. On a bare system, this is unsolvable. You can't install two conflicting versions of the same library system-wide. Virtualenvs help with Python, but they don't isolate system libraries, kernel interfaces, or filesystem paths. Service A can read Service B's secrets if they both run as the same user. A crash in Service C's native extension can pollute the process space of the host.

The old answer was virtual machines — give each service its own OS. That works, but a VM carries its own kernel, its own boot sequence, its own memory budget. Running fifty VMs to isolate fifty microservices costs you fifty kernels worth of RAM and fifty boot cycles.

The container answer is different: you don't need a separate kernel. You need a process that *believes* it has its own kernel. Linux already has the tools to make a process believe that. Namespaces give it its own view of PIDs, filesystems, network interfaces, and hostname. cgroups limit how much CPU and memory it can consume. overlayfs gives it its own writable filesystem without duplicating the entire underlying image. And `pivot_root` or `chroot` changes what it calls `/`.

Put those four things together and you have a container. Docker, containerd, Podman — they are all orchestration layers on top of these four Linux kernel features. Understanding them is how you debug containers that won't start, understand CVEs that escape container boundaries, and reason clearly about what "container security" actually means in practice.

## Mental Model

Think of a theatre stage. The stagehands, director, and crew see the whole building — the loading dock, the dressing rooms, the rigging above. But when an actor steps onto the stage and the curtain comes down behind them, they see only the set. The walls are painted canvas, the doors don't go anywhere, the horizon is a backdrop — but to the actor, it's a castle. They can't see the backstage, and backstage can't see them (unless someone opens the curtain).

A container is that curtain. The host kernel is the entire building. The container process sees only the set it was given — its own filesystem, its own process list, its own network. It can't look backstage. The kernel knows everything, but the process inside only knows what the curtain reveals. When the curtain is well-constructed, the actor can't find the edge. That construction is what this topic is about.

## Theory

### What a container actually is

A container is not an object the kernel knows about. There is no `container` syscall, no container type in `/proc`, no kernel structure named `container_t`. A container is a *description* applied to a regular Linux process (or process tree) that happens to have:

- One or more **namespaces** isolating its view of PIDs, filesystems, network, IPC, UTS, and users
- A **cgroup** limiting its resource consumption
- A **root filesystem** (rootfs) that is different from the host's, typically provided by an overlayfs mount
- An alternate root established via `chroot` or `pivot_root`

The combination of these four things creates the experience of isolation. Remove any one of them and the boundary weakens or disappears entirely.

### chroot — the original attempt

`chroot` is a syscall (and command) that changes the root directory (`/`) for a process and its children. A process inside a chroot jail resolves all absolute paths relative to the new root. `/etc/passwd` inside the jail resolves to `/jail/etc/passwd` on the host. The process cannot `cd /` and walk up to the real `/etc/passwd` — the path resolution stops at the chroot boundary.

This sounds like isolation. It isn't. `chroot` changes *pathname resolution* for a single process. It does not change the mount namespace, does not affect open file descriptors obtained before the chroot call, does not affect network access, and critically — does not restrict a root process from escaping. A process with `CAP_SYS_CHROOT` can call `chroot()` again to move the root somewhere else, including back to a parent directory it knows about from before it was jailed. The classic chroot escape is a textbook exercise in four syscalls.

The man page for `chroot(2)` says it bluntly: "This call changes an ingredient in the pathname resolution process and does nothing else." That's not security. That's filesystem scoping.

### pivot_root — the real isolation

`pivot_root` is a different syscall. Instead of relabeling what `/` means for the current process's pathname resolution, it *replaces the root mount of the current mount namespace* with a new filesystem, moving the old root to a specified directory under the new root.

The difference matters for two reasons.

First, `pivot_root` operates at the mount namespace level. If the process has its own mount namespace (via `CLONE_NEWNS`), then pivoting the root affects the entire namespace view, not just that one process. The old root isn't reachable by walking up the directory tree — it's a separate mount, accessible only at the path you designated.

Second, after pivoting, you unmount the old root entirely. Once that unmount succeeds, the process in the container's mount namespace has no reference whatsoever to the host's filesystem. There is no `/proc/self/root` trick, no `openat(AT_FDCWD, "/../../../etc/passwd", ...)` escape. The host filesystem simply does not exist in the mount namespace.

Having `CAP_SYS_ADMIN` inside an unprivileged user namespace does not allow escaping a properly constructed `pivot_root` + namespace container. The capability is scoped to that user namespace, not to the parent. This is the fundamental security upgrade over `chroot`.

### overlayfs — layered filesystems

Container images are distributed as layers. An Ubuntu 22.04 base image is one layer. Installing nginx on top of it adds a second layer — only the files changed by that installation, as a tar archive of the diff. Your application code adds a third layer. This means you don't ship 200MB of Ubuntu with every container; the base layer is shared across thousands of containers on the same host.

overlayfs is the Linux filesystem that assembles these layers at mount time. It takes a stack of read-only `lowerdirs` and one writable `upperdir`, and presents a unified `merged` view. Reads fall through the stack from the top downward until a file is found. Writes go to `upperdir` only — the lower layers are never modified. Deletes are implemented via *whiteout files*: a special file placed in `upperdir` that masks a file in `lowerdir` from appearing in `merged`.

The `workdir` is an implementation detail that overlayfs requires for atomic operations during writes. It must be on the same filesystem as `upperdir` — not just on the same disk, but in the same filesystem mount. If you put `upperdir` on ext4 and `workdir` on tmpfs, the mount fails with `EXDEV`.

### OCI image format

The Open Container Initiative standardized the image format so that any OCI-compliant runtime can run any OCI-compliant image. An OCI image stored on disk is a directory containing:

- `oci-layout` — a JSON file identifying the OCI layout version
- `index.json` — the top-level manifest index, pointing to one or more image manifests by content-addressable digest
- `blobs/sha256/` — a directory of content-addressed blobs: each blob is either a layer tar archive or a JSON config or manifest file

The **manifest** (`manifest.json` in runc parlance, a blob in OCI layout) lists the config blob's digest and the ordered list of layer blobs. The **config** describes the image's metadata: environment variables, entrypoint, command, working directory, and the `rootfs.diff_ids` — the digests of the uncompressed layers in order, used to verify layer integrity.

Each **layer** is a tar archive representing a filesystem diff. Files added or modified appear as normal entries. Files deleted appear as whiteout entries: a zero-byte file named `.wh.<filename>` in the same directory as the deleted file, or `.wh..wh..opq` to indicate an opaque directory (replace the entire directory from lower layers).

### runc and containerd

`runc` is the reference OCI runtime. Given a *bundle* — a directory containing `rootfs/` and `config.json` — `runc` calls the right sequence of Linux syscalls to create namespaces, set up mounts, configure cgroups, and execute the container's init process. `runc` is a thin tool. It doesn't pull images, doesn't manage container lifecycles beyond running them, doesn't expose a daemon socket.

`containerd` sits above `runc`. It manages image pulls, unpacking layers into overlayfs snapshots, creating bundles, calling `runc`, watching containers, and collecting exit status. Docker's engine calls `containerd`; Kubernetes's container runtime interface (CRI) speaks to `containerd` (or `cri-o`) directly. When your Docker container doesn't start, the error often lives in `containerd`'s logs. When the container starts but misbehaves, the fault is usually in the bundle that `runc` received.

## How It Works (Mechanics)

### The sequence the kernel executes

When `runc` starts a container, the approximate sequence of kernel operations is:

```text
# Conceptual flow — not a real command sequence
1. runc calls clone(CLONE_NEWPID | CLONE_NEWNS | CLONE_NEWNET | CLONE_NEWUTS | CLONE_NEWIPC | CLONE_NEWUSER)
   → kernel creates new PID, mount, network, UTS, IPC, and user namespaces for the child
2. In the child: set up UID/GID mappings (writes /proc/self/uid_map, gid_map)
3. In the child: mount overlayfs at a temp path
   → kernel assembles lowerdir stack + upperdir + workdir → merged view
4. In the child: bind-mount /dev, /proc (new procfs), /sys into the merged rootfs
5. In the child: call pivot_root(new_root, old_root_under_new)
   → kernel swaps the mount namespace's root mount
6. In the child: unmount old_root (now at a path like /old-root inside the new root)
   → old host filesystem disappears from the namespace entirely
7. In the child: apply cgroup limits (write to /sys/fs/cgroup/...)
8. In the child: drop capabilities, apply seccomp filter
9. In the child: exec() the container's entrypoint (e.g. /bin/sh)
```

Steps 3–6 must happen in exactly this order. Mount the new filesystem before pivoting. Set up `/proc` as a new procfs mount inside the container before the old proc disappears. If you mount `/proc` after `pivot_root` but forget to mount it at all, the container's init process will not be able to resolve `/proc/self` — some programs fail silently, others crash immediately.

### overlayfs mount internals

When the kernel assembles an overlayfs mount, it keeps an internal list of lower layer dentries. On a file lookup in `merged`:

1. The kernel checks `upperdir` first. If the file exists there, that version is returned.
2. If not found in `upperdir`, the kernel walks down the `lowerdir` stack from top to bottom.
3. If a whiteout is found anywhere in the stack, the file is masked — it does not appear in `merged`.
4. Copy-on-write: if a process opens a file from `lowerdir` for writing, the kernel copies it to `upperdir` first, then returns the `upperdir` copy for writing. The original in `lowerdir` is never modified.

This is why the first write to a large file in a container can be slow: the entire file is copied up to `upperdir` before the write proceeds.

### What pivot_root does to the mount table

```bash
# Illustrative — trace what pivot_root does to /proc/mounts
# Before pivot_root (inside a new mount namespace):
# /dev/sda1 / ext4 rw 0 0
# overlay /new-root overlay rw,lowerdir=...,upperdir=...,workdir=... 0 0

# After pivot_root("/new-root", "/new-root/old-root") and umount("/old-root"):
# overlay / overlay rw,lowerdir=...,upperdir=...,workdir=... 0 0
# tmpfs /proc proc rw 0 0
# (no entry for /dev/sda1 — it was on the old root, which was unmounted)
```

The host's block device mounts vanish from the namespace entirely. A process inside the container that tries to mount `/dev/sda1` directly would fail — not because it lacks permission, but because from within the namespace, `/dev/sda1` as a path on the original ext4 filesystem does not exist.

## Basic Example

The minimum viable container: an Alpine minirootfs, isolated with `unshare`, `chroot`, and a new mount for `/proc`.

```bash
# Download Alpine minirootfs (about 3MB)
wget -q https://dl-cdn.alpinelinux.org/alpine/v3.19/releases/x86_64/alpine-minirootfs-3.19.1-x86_64.tar.gz

# Create rootfs directory and extract
mkdir -p /tmp/mycontainer/rootfs
tar -xzf alpine-minirootfs-3.19.1-x86_64.tar.gz -C /tmp/mycontainer/rootfs

# Enter new namespaces and chroot into the minirootfs
# --fork: required for PID namespace to take effect
# --pid: new PID namespace (we'll be PID 1)
# --mount: new mount namespace so our /proc mount doesn't affect the host
# --uts: new hostname namespace
sudo unshare --fork --pid --mount --uts chroot /tmp/mycontainer/rootfs /bin/sh
```

Once inside, you need to mount `/proc` manually — it isn't there by default:

```bash
# Inside the container shell:
mount -t proc proc /proc
ps aux
```

```text
PID   USER     TIME  COMMAND
    1 root      0:00 /bin/sh
    2 root      0:00 ps aux
```

The shell is PID 1. The host's process list is completely invisible. The hostname is whatever the host had (you can change it with `hostname newname` — that change is scoped to the UTS namespace and won't affect the host).

Try it: run `ps aux` on the host in another terminal. You will see the `unshare` and `chroot` processes. From inside the container, they don't exist. That's the PID namespace working.

If your `ps` output shows hundreds of processes instead of just two, the `/proc` mount didn't take effect — make sure you ran `mount -t proc proc /proc` inside the chroot, not before it.

## Real-World Example

A complete container-from-scratch using `pivot_root` and overlayfs, the way an actual runtime would do it. This is closer to what `runc` executes under the hood.

> ⚠️ **WARNING:** The commands below create new mount namespaces and perform `pivot_root`. Run these in a VM or a disposable environment. A mistake in mount namespace setup can lock you out of `/proc` or `/sys` on the host if run incorrectly outside a new namespace. All commands require `sudo`.

```bash
# --- Step 1: Build the overlay filesystem ---
# We need: a lower (read-only image), upper (writable), work (same fs as upper), merged

mkdir -p /tmp/ctr/{lower,upper,work,merged,bundle/rootfs,bundle/old-root}

# Populate lower with Alpine minirootfs (reuse from basic example)
tar -xzf alpine-minirootfs-3.19.1-x86_64.tar.gz -C /tmp/ctr/lower

# Mount overlayfs: lower=image, upper=writable layer, work=internal scratch
sudo mount -t overlay overlay \
  -o lowerdir=/tmp/ctr/lower,upperdir=/tmp/ctr/upper,workdir=/tmp/ctr/work \
  /tmp/ctr/merged
```

```bash
# --- Step 2: Prepare the rootfs for pivot_root ---
# pivot_root requires: new_root is a mount point, and old_root is under new_root

# Bind-mount the merged overlay to bundle/rootfs so it's a proper mountpoint
sudo mount --bind /tmp/ctr/merged /tmp/ctr/bundle/rootfs

# Mount a fresh /proc inside the new root (must do BEFORE pivot_root, inside new mount ns)
sudo mount -t proc proc /tmp/ctr/bundle/rootfs/proc
```

```bash
# --- Step 3: Enter namespaces and run pivot_root ---
# We use a small script so unshare can exec it after namespace creation

cat > /tmp/ctr/init.sh << 'EOF'
#!/bin/sh
# This runs inside the new mount namespace after unshare forks

# pivot_root: swap "/" to /bundle/rootfs, put old root at /bundle/rootfs/old-root
cd /tmp/ctr/bundle/rootfs
pivot_root . old-root

# Update PATH for Alpine
export PATH=/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin:/sbin:/bin

# Unmount the old root — after this, the host filesystem is gone from this namespace
umount -l /old-root
rmdir /old-root

# Start an interactive shell as PID 1
exec /bin/sh
EOF
chmod +x /tmp/ctr/init.sh

# Enter new PID, mount, UTS, and IPC namespaces; run our init script
sudo unshare --fork --pid --mount --uts --ipc /tmp/ctr/init.sh
```

```text
/ # ps aux
PID   USER     TIME  COMMAND
    1 root      0:00 /bin/sh
    2 root      0:00 ps aux
/ # ls /
bin    dev    etc    home   lib    media  mnt    opt    proc   root   run    sbin   srv    sys    tmp    usr    var
/ # cat /etc/alpine-release
3.19.1
/ # ls /old-root
ls: /old-root: No such file or directory
```

The last line tells the whole story. `/old-root` doesn't exist because we unmounted it. The host filesystem is not reachable. The container is running Alpine with PID 1 being a shell, in a new PID namespace, on a writable overlayfs that layered the image into `lower` and keeps changes in `upper`.

Walk through what each step accomplished:

**Step 1** creates the overlayfs. The Alpine rootfs is in `lower` and will never be modified. Any file the container writes goes to `upper`. If you exit and re-enter, the `upper` directory still has those writes — the lower layer is clean for reuse.

**Step 2** prepares for `pivot_root`. The requirement is that the new root must already be a mount point before you call `pivot_root`. Bind-mounting it to `bundle/rootfs` satisfies this. Mounting `/proc` here, before the pivot, means the container will have a working `/proc` immediately after the switch — without an extra `mount` call inside.

**Step 3** is where the namespace and root switch happen. `unshare --fork --pid` creates a new PID namespace; the `--fork` is critical — without it, `unshare` itself becomes PID 1 and doesn't exec the script. Inside the script, `pivot_root . old-root` swaps the mount namespace's root. The unmount of `/old-root` seals the container — that path no longer resolves to anything.

After the session, clean up:

```bash
# After exiting the container shell, clean up mounts (run on the host)
sudo umount /tmp/ctr/merged
sudo rm -rf /tmp/ctr
```

## runc and OCI Bundles

The above is the manual version of what `runc` does. With `runc` installed:

```bash
# Install runc (Ubuntu/Debian)
sudo apt install runc

# Create an OCI bundle
mkdir -p /tmp/oci-bundle/rootfs
tar -xzf alpine-minirootfs-3.19.1-x86_64.tar.gz -C /tmp/oci-bundle/rootfs

# Generate a default config.json — this is the OCI runtime spec for the container
cd /tmp/oci-bundle
runc spec
```

`runc spec` creates a `config.json` that looks like:

```text
{
  "ociVersion": "1.0.2",
  "process": {
    "terminal": true,
    "user": { "uid": 0, "gid": 0 },
    "args": [ "sh" ],
    "env": [ "PATH=/usr/local/sbin:...", "TERM=xterm" ],
    "cwd": "/"
  },
  "root": { "path": "rootfs", "readonly": true },
  "mounts": [
    { "destination": "/proc", "type": "proc", "source": "proc" },
    ...
  ],
  "linux": {
    "namespaces": [
      { "type": "pid" }, { "type": "network" }, { "type": "ipc" },
      { "type": "uts" }, { "type": "mount" }
    ]
  }
}
```

Edit `config.json` to customize the container — change `args` to run a different command, add environment variables, adjust namespace configuration, set resource limits in the `linux.resources` key. Then run it:

```bash
sudo runc run mycontainer
```

`runc` reads `config.json`, calls `clone()` with the specified namespaces, mounts `/proc` and other pseudo-filesystems, applies cgroup limits, and executes `sh`. No Docker daemon, no containerd. Just the kernel and a JSON spec.

## ⚠️ Gotchas

### chroot Alone Is Not Container Isolation

The most common misconception. People set up a chroot and believe they have a container. They don't.

```bash
# This looks like container isolation
sudo chroot /tmp/mycontainer/rootfs /bin/sh
```

A root process inside that chroot can escape it:

```bash
# Inside the chroot, as root — classic escape
mkdir -p /tmp/escape
# The trick: chroot to a subdirectory, then use relative paths to walk up past the original chroot
# This is a well-documented attack; the point is that chroot() does not limit a root process
```

```text
# After the escape, the process is back at the real /
```

The underlying rule: `chroot` changes pathname resolution for one process. A process with `CAP_SYS_CHROOT` can call `chroot()` again to move its root elsewhere. Combined with `open()` keeping file descriptors across chroot boundaries, a sufficiently privileged process can always escape. `pivot_root` plus a new mount namespace plus `umount` of the old root is the correct construction. With no path to the old root and no `CAP_SYS_ADMIN` scoped outside the user namespace, there is nothing to escape to.

### overlayfs workdir Must Be on the Same Filesystem as upperdir

This one produces a confusing error and the mount will silently fail or return `EXDEV`.

```bash
# Wrong: workdir on tmpfs, upperdir on ext4
sudo mount -t overlay overlay \
  -o lowerdir=/lower,upperdir=/mnt/data/upper,workdir=/tmp/work \
  /merged
```

```text
mount: /merged: failed to mount overlay filesystem at /merged.
```

Or, depending on kernel version, you get `EXDEV` in `dmesg`:

```text
overlayfs: workdir and upperdir must reside under the same mount
```

The rule is not "same disk" or "same device" — it is "same filesystem mount." The kernel uses the workdir to stage atomic rename operations during copy-up. A rename across filesystem boundaries (`EXDEV`) is impossible, so the kernel enforces that workdir and upperdir share a superblock. Fix it by putting both on the same mount:

```bash
# Correct: workdir and upperdir are directories within the same ext4 mount
mkdir -p /mnt/data/{upper,work}
sudo mount -t overlay overlay \
  -o lowerdir=/lower,upperdir=/mnt/data/upper,workdir=/mnt/data/work \
  /merged
```

### PID Namespace + /proc: Order of Operations

Creating a PID namespace without properly mounting `/proc` inside the new root causes subtle failures — programs that query `/proc/self/status` will see the host's proc, which lists the host's PID, not the container's PID 1.

```bash
# Wrong order: entering the PID namespace and then trying to mount /proc after pivot_root
# has already happened without having set up /proc first
sudo unshare --fork --pid chroot /rootfs /bin/sh
# Inside: mount -t proc proc /proc   ← too late if pivot_root already ran; /proc sees wrong ns
```

The correct order is:

1. Create the new mount namespace (`CLONE_NEWNS`)
2. Mount the new rootfs (overlayfs or bind mount)
3. Mount `/proc` as `procfs` inside the new rootfs, targeting the new rootfs path
4. Call `pivot_root` to swap the root
5. Unmount old root
6. The `/proc` mount is now in the correct mount namespace and correct location

The key insight: mount namespace isolation means that mounts done before `CLONE_NEWNS` are inherited (as a copy) by the child. Mounts done after `CLONE_NEWNS` in the child namespace are private. You want `/proc` mounted after the namespace split and inside the new rootfs so it reflects the container's PID namespace, not the host's.

### runc spec Generates a Root-Mode Bundle

By default, `runc spec` creates a config that runs as UID 0 (root). For production use in rootless containers, you need user namespace mappings.

```bash
# Generates a rootless config (sets up user namespace UID/GID mappings)
runc spec --rootless
```

Without `--rootless`, running `runc run` as a non-root user will fail because setting up mount namespaces and calling `pivot_root` requires elevated privileges in the initial user namespace. The `--rootless` flag adds a user namespace block to `config.json` with appropriate `uidMappings` and `gidMappings`, allowing an unprivileged user to run the container via user namespace privilege escalation.

## Alternatives & Comparisons

| Tool / Approach | When to Use | Notes |
|---|---|---|
| `chroot` | Legacy process sandboxing, build environments where security is not the concern | Not a security boundary; fine for chrooted `bind` DNS or build chroots; never use for untrusted workloads |
| `pivot_root` + namespaces (manual) | Learning how containers work; writing a container runtime | This is what runc does; manual use is for education and custom runtimes |
| `runc` | OCI-compliant container execution; what Docker/containerd call internally | The reference implementation; stable, minimal, audited |
| `crun` | Lightweight runc replacement (written in C, not Go) | Lower startup overhead; default in Podman; OCI-compliant |
| `gVisor` (runsc) | Untrusted workloads where syscall filtering is not enough | Implements a user-space kernel; much stronger isolation; significant performance cost |
| `Kata Containers` | Full VM-level isolation with container UX | Uses a lightweight VM per container; heavier than runc but stronger than namespaces |
| `Podman` | Rootless containers without a daemon | Drop-in Docker replacement; calls runc/crun; no background daemon required |

For day-to-day production use: `containerd` + `runc` (via Docker or Kubernetes CRI) is the mainstream path. For rootless development: `Podman`. For truly untrusted multi-tenant workloads: `gVisor` or Kata. Never use bare `chroot` for anything that touches untrusted input.

## Trade-offs

**Where this shines:**

- Process-level isolation with near-zero overhead — a container starts in milliseconds, not the seconds of a VM
- Filesystem layering via overlayfs means a 500MB image is shared between 100 containers on the same host; only their write layers (`upperdir`) are per-container
- Namespaces + cgroups is the Linux kernel's native model — no hypervisor, no hardware virtualization required
- OCI standardization means images built by Docker run on containerd, Podman, Kubernetes, and any OCI-compliant runtime without modification

**Where this costs you:**

- The isolation boundary is only as strong as the namespace + capability configuration. A misconfigured container running as root with `CAP_SYS_ADMIN` and a writable host `/proc` mount is essentially running on the bare host
- overlayfs copy-up latency: the first write to any large file in `lowerdir` incurs a full file copy to `upperdir`. Heavy write workloads (databases) should use bind mounts or volumes, not overlayfs
- Debugging failures requires understanding the full stack — kernel namespaces, mount setup, cgroup hierarchy, OCI spec — because errors can surface at any layer and the error messages are often terse
- Shared kernel means a kernel vulnerability can affect all containers on the host simultaneously; in a VM, a kernel exploit in one guest doesn't directly threaten others

**Performance note:** overlayfs read performance is close to native for workloads that don't trigger copy-up. The copy-up penalty only occurs on the first write to a file that lives in `lowerdir`. Subsequent writes to that file go directly to `upperdir` with no additional overhead.

## When to Use / When to Avoid

✅ **Use this when:**
- You need to understand why a container won't start — knowing the `pivot_root` + overlayfs + namespace sequence tells you exactly where to look in `dmesg` and `runc` logs
- You're writing tooling that manages containers — Kubernetes operators, CI runners, build systems — and need to reason about what the runtime does
- You're evaluating a container escape CVE — the vulnerability is almost always a violation of one of: capability scoping, namespace isolation, or mount namespace integrity
- You need rootless containers in a CI environment where Docker daemon isn't available — `runc spec --rootless` + user namespaces is the answer

❌ **Avoid this when:**
- You need strong multi-tenant isolation for untrusted code — use `gVisor` or Kata Containers; namespace isolation alone is not sufficient when the attacker controls the workload
- You're running stateful workloads like databases — mount a volume (bind mount to host path) rather than relying on the container's `upperdir`; overlay write amplification will hurt you
- You're tempted to use `chroot` as a security boundary — it isn't one; if the process inside has root, it can escape

⚠️ **Common mistakes:**
- Forgetting that `--pid` in `unshare` requires `--fork` — without `--fork`, the namespace is created but the calling process doesn't become PID 1; the shell will see a confusing PID
- Mounting `/proc` before entering the mount namespace — you end up with the host's procfs inside the container's rootfs, leaking host PID information
- Leaving the old root mounted after `pivot_root` — the host filesystem is still accessible at `/old-root` inside the container until you explicitly unmount it
- Building overlayfs stacks with more than about 500 lower layers — the kernel has a practical limit and performance degrades with deep stacks; real runtimes flatten or merge layers aggressively

## Veteran Perspective

The thing that surprises most people who come from a "containers are just Docker" background is how thin the abstraction really is. You can replicate the core of what Docker does in about thirty lines of shell and two commands: `unshare` and `chroot`. The Docker daemon, containerd, image registries, layer caching — all of it is infrastructure built on top of five syscalls: `clone`, `mount`, `pivot_root`, `umount`, and `exec`. Understanding this is not academic. It changes how you debug.

The intermediate-level mistake I see most often is treating namespace isolation as equivalent to security isolation. Namespaces are a *visibility* boundary, not a *permission* boundary. A process inside a container can still exhaust host resources if cgroups aren't applied. A container running as root with `--privileged` (which grants `CAP_SYS_ADMIN` and mounts the host's device tree) is not isolated at all — it's just a fancy `chroot` with extra steps. The security model of containers requires all four components: namespaces, cgroups, capability reduction, and a seccomp filter. Any one of them missing and your "container" is a liability.

The `chroot` vs `pivot_root` distinction is not just trivia — it comes up in CVEs. Container escape vulnerabilities from the 2019 `runc` exploit (CVE-2019-5736) and various overlayfs privilege escalations (`CVE-2021-3493` on Ubuntu) all exploit gaps between what container runtimes assume is isolated and what Linux actually enforces. Knowing the mechanics means you can read a CVE advisory and immediately understand whether your configuration is affected, rather than waiting for a vendor patch note to tell you what to do.

The overlayfs `workdir` / `upperdir` co-location requirement is the kind of thing that bites you in staging when your CI runner has `/tmp` on a separate tmpfs from where your container storage lives. The error message — or the silent mount failure — is opaque. Knowing the rule means you fix it in thirty seconds instead of spending an hour reading kernel source.

For interviews at companies doing serious infrastructure work: understanding `pivot_root`, the OCI spec, and why `runc spec` produces what it does will separate you from everyone who knows Docker commands but can't explain why a container restart with a new image leaves the previous `upperdir` layer behind (answer: the old container had its own `upperdir` and overlayfs writes are not shared; the new container gets a fresh `upperdir`).

The deeper design principle here is that Linux does not have a container abstraction — it has composable isolation primitives. That composability is both the power and the risk. You can build exactly the container you need. You can also build a container-shaped sieve that leaks everything. The kernel does what you tell it. These notes exist so you tell it the right thing.

## Try It Yourself

1. Run the basic example from the Basic Example section. Inside the container, run `hostname` — then, without exiting, open a new terminal on the host and check `hostname` there. They should match. Now run `hostname container1` inside the container (you'll need `sudo unshare --uts` to change this), exit, and confirm the host hostname is unchanged.

2. After running the basic example, exit the container and inspect `/tmp/mycontainer/rootfs/etc/hostname`. It won't exist — Alpine's minirootfs is minimal. Now create a file inside the container at `/tmp/test.txt`. Exit and check whether that file exists on the host filesystem at `/tmp/mycontainer/rootfs/tmp/test.txt`. Explain what you find.

3. Build the overlayfs real-world example. While the container is running, open a second terminal on the host and run `ls /tmp/ctr/upper` — you should see no changes yet. Inside the container, run `echo hello > /tmp/hostwrite.txt`. Check `/tmp/ctr/upper` again on the host. Where did the file land? Why does it appear there and not in `/tmp/ctr/lower`?

4. Install `runc` and run `runc spec` in an empty directory. Read the generated `config.json`. Find the `namespaces` array and note which namespaces are listed. Add a cgroup memory limit of 64MB by adding a `resources` block under the `linux` key. Verify your JSON is valid with `python3 -m json.tool config.json`. *(Requires `runc` installed; run in a VM.)*

5. **Challenge:** Build a minimal OCI bundle from scratch without using Docker. Download the Alpine minirootfs, create the `rootfs/` directory, run `runc spec`, modify `config.json` to run a Python HTTP server (`python3 -m http.server 8080`) instead of `sh`, and start it with `runc run webserver`. From the host, try `curl localhost:8080`. You'll need to figure out why it doesn't work and fix the network namespace configuration in `config.json` — hint: the container has its own network namespace with no routing configured. *(Requires a VM with `runc` and `python3` in the rootfs.)*

## Summary

- A container is a regular Linux process with namespaces (visibility isolation), cgroups (resource limits), an overlayfs root (layered filesystem), and `pivot_root` (root mount replacement) — the kernel has no native "container" concept
- `chroot` changes pathname resolution for one process and is not a security boundary; a process with `CAP_SYS_CHROOT` can escape it in four syscalls; `pivot_root` replaces the mount namespace's root and, combined with unmounting the old root, is escape-proof from within the namespace
- overlayfs assembles a stack of read-only `lowerdirs` and one writable `upperdir` into a `merged` view; writes go to `upperdir` only (copy-on-write); `workdir` must be on the same filesystem mount as `upperdir` or the mount fails
- An OCI image is a stack of tar-archive layer diffs plus a JSON config and manifest; `runc` takes a bundle (rootfs + `config.json`) and executes the container using the OCI runtime spec
- `runc spec` generates a base `config.json` for a bundle; editing it directly is how you customize namespaces, capabilities, mounts, and resource limits without a higher-level tool
