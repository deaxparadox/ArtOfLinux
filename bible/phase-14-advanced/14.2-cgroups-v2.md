# [14.2] — Control Groups v2 (cgroups v2)

## The Problem This Solves

Imagine you're running twenty services on a single machine — a web server, a database, a background job processor, and seventeen other things. A developer pushes a bug: one of those jobs enters an infinite loop and starts allocating memory as fast as the kernel will give it. Within minutes, the OOM killer starts firing. It kills your database. The box becomes unstable. Every other service on it goes down.

You had set `ulimit -v` on that process. It didn't matter — `ulimit` is per-process, not per-group-of-processes. The job processor forked worker children, and each child had its own independent limit. The sum of all their memory usage was unbounded.

Now imagine a container runtime trying to guarantee that a tenant's container gets exactly 2 CPU cores and no more than 512 MB of RAM, regardless of what code runs inside it. `ulimit` can't do this. The limits don't compose. There's no way to say "the sum of all processes in this group is bounded."

Before cgroups, the only answer was to put each service on its own machine. That's expensive. Cgroups are how Linux solved the multi-tenant resource isolation problem without hardware separation. They are also, directly, how Docker, Kubernetes, and systemd enforce resource limits today.

## Mental Model

Think of cgroups as a budget system for an organization.

The organization (the kernel) has a fixed pool of resources: CPU time, memory, disk bandwidth. It divides that pool into departments (cgroups). Each department has a budget — a hard ceiling it cannot exceed and a target share that determines what it gets when resources are contested. When a department creates sub-teams, those sub-teams draw from their department's budget, not from the organization's unlimited pool. Every person hired into a sub-team counts against their department's headcount cap.

The organization doesn't care how a department uses its budget internally. It only cares that the total spend never exceeds what was allocated. A department that goes over its memory budget gets its people (processes) slowed down or, if they can't be slowed, removed (OOM killed).

The whole tree of departments and sub-teams maps exactly to the cgroup hierarchy under `/sys/fs/cgroup/`.

## Theory

### What cgroups are

A cgroup (control group) is a kernel mechanism for organizing processes into a named hierarchy and applying resource accounting and enforcement to each node in that hierarchy. Every process on a Linux system belongs to exactly one cgroup. The kernel tracks each process's membership and enforces the limits of that cgroup against the combined resource usage of all processes within it.

There are two things cgroups do that `ulimit` cannot:

**Grouping.** A cgroup collects a process and all its descendants (unless they are explicitly moved). The limit applies to the sum — not to each individual process. If a cgroup's memory limit is 512 MB, then every process inside it, combined, cannot exceed 512 MB.

**Accounting.** The kernel charges resource usage to the cgroup, not just to individual processes. You can see exactly how much CPU, memory, and I/O a service has consumed in aggregate. This is where container metrics come from.

### cgroups v1 vs cgroups v2

**v1** was the original design. Each resource type had its own independent hierarchy — a tree for CPU, a separate tree for memory, another for block I/O, and so on. A process could be in `/sys/fs/cgroup/cpu/webserver/` for CPU purposes and simultaneously in `/sys/fs/cgroup/memory/tier-a/` for memory purposes. These trees were completely independent.

This flexibility became a problem. Because the trees were independent, cross-controller coordination was impossible. You could not write a coherent policy that said "this group of processes gets 2 CPUs and 512 MB of memory together." The memory controller didn't know anything about the CPU controller's tree. Controllers accumulated inconsistencies in how they handled edge cases because each was designed in isolation.

**v2** — introduced in Linux 4.5, stabilized by 5.2 — replaced the multiple independent hierarchies with a single unified tree. Every process is in one place. Every controller applies to the same node. The kernel can now reason about a group's total resource envelope in a coordinated way. This is what made proper container isolation tractable.

The single tree lives entirely under `/sys/fs/cgroup/`. There is no `/sys/fs/cgroup/memory/` separate from `/sys/fs/cgroup/cpu/` anymore. One cgroup, all controllers together.

### The unified hierarchy

The tree under `/sys/fs/cgroup/` mirrors the systemd unit hierarchy on modern systems. Every systemd service lives in its own cgroup. Every user session lives in its own slice. The path encodes the containment relationship:

```text
/sys/fs/cgroup/
├── system.slice/
│   ├── nginx.service/
│   ├── postgresql.service/
│   └── sshd.service/
├── user.slice/
│   └── user-1000.slice/
│       └── session-3.scope/
└── init.scope/
```

Each directory is a cgroup. Each cgroup contains files — the interface files that let you read accounting data and write limits.

### Controllers

A controller is a kernel subsystem that enforces one category of resource constraint. In v2, controllers are enabled per-node in the tree by writing to the `cgroup.subtree_control` file. A controller only applies to a cgroup if it has been enabled at every level from root down to that cgroup.

The major controllers in v2:

- **cpu** — CPU time allocation: bandwidth limits (`cpu.max`) and relative scheduling weight (`cpu.weight`)
- **memory** — Memory usage limits, both hard (`memory.max`) and soft (`memory.high`)
- **io** — Block I/O bandwidth and IOPS limits (`io.max`)
- **pids** — Maximum number of processes/threads in the group (`pids.max`); the primary defense against fork bombs
- **cpuset** — Pins the cgroup to specific CPU cores and NUMA nodes

### The no-internal-process constraint

This is the invariant that confuses people most. In v2, a non-leaf cgroup — one that has child cgroups — cannot also contain processes directly. A cgroup can either have children or have processes, not both.

This rule exists to enforce clean delegation semantics. If a cgroup could hold both its own processes and child cgroups, it would be ambiguous what "the limit of this cgroup" means for its own processes versus its children. The rule eliminates that ambiguity.

In practice: you create a cgroup for a group of related services, then create child cgroups for each service. The parent cgroup is an organizational node with no processes in it. The leaf cgroups hold actual processes.

The root cgroup (`/sys/fs/cgroup/`) is exempt from this rule. systemd runs as PID 1 in the root cgroup and creates the entire hierarchy below it.

## How It Works (Mechanics)

### The filesystem interface

cgroups v2 exposes its entire interface through a virtual filesystem mounted at `/sys/fs/cgroup/`. Creating a cgroup is `mkdir`. Configuring a limit is `echo value > /sys/fs/cgroup/mygroup/memory.max`. Reading accounting data is `cat /sys/fs/cgroup/mygroup/memory.current`. Moving a process into a cgroup is writing its PID to `cgroup.procs`.

This is the Unix philosophy applied at the kernel level: the interface is the filesystem. No ioctl. No custom syscall. Just files.

### Key interface files in every cgroup

Every cgroup directory contains these files automatically:

```text
cgroup.controllers       — which controllers are available here (read-only)
cgroup.subtree_control   — which controllers are enabled for children (write to activate)
cgroup.procs             — PIDs of processes in this cgroup (write to move a process here)
cgroup.events            — event notifications (populated, frozen state)
cgroup.type              — "domain", "threaded", or "domain invalid"
```

When a controller is enabled, additional files appear:

```text
cpu.max           — CPU bandwidth ceiling: "quota period" in microseconds
cpu.weight        — Relative CPU scheduling weight (1–10000, default 100)
cpu.stat          — Accumulated CPU usage statistics

memory.max        — Hard memory ceiling; OOM kill if exceeded
memory.high       — Soft memory ceiling; throttle/reclaim if exceeded
memory.current    — Current memory usage (read-only accounting)
memory.stat       — Detailed breakdown of memory usage

io.max            — Block I/O bandwidth and IOPS limits
pids.max          — Maximum number of processes allowed
pids.current      — Current process count (read-only)
```

### How a limit is enforced: the CPU controller

When `cpu.max` is set to `50000 100000`, the kernel's Completely Fair Scheduler (CFS) bandwidth controller allocates 50,000 microseconds (50 ms) of CPU time to this cgroup per 100,000 microsecond (100 ms) period. If processes in the cgroup exhaust that 50 ms quota before the period ends, all processes in the cgroup are throttled — they are removed from the run queue and cannot run again until the next period begins. This is a hard pause, not a gradual slowdown.

`cpu.weight` works differently. It's a relative share used only during contention. If two cgroups both want CPU and one has weight 200 while the other has weight 100, the first gets roughly twice the CPU time. When there's no contention — when nobody else wants the CPU — a cgroup with weight 1 can use 100% of the machine. `cpu.weight` is fair scheduling, not a cap.

### How the memory controller works

The memory controller tracks the combined memory usage of all processes in a cgroup and compares it against two thresholds in order:

1. **`memory.high`** — When usage crosses this, the kernel aggressively tries to reclaim memory from the cgroup: it drops caches, writes dirty pages to disk, and slows the processes by making them wait for reclaim to complete. The process is not killed. This is throttle-under-pressure.

2. **`memory.max`** — When usage crosses this and cannot be reduced by reclaim, the kernel invokes the cgroup-level OOM killer. It selects a process within the cgroup to kill. Only processes inside the cgroup are at risk — other cgroups are not affected.

The practical design: set `memory.high` at, say, 400 MB and `memory.max` at 512 MB. A well-behaved process stays under 400 MB and never feels any pressure. A misbehaving process crossing 400 MB gets throttled — it slows down instead of crashing. If it keeps growing past 512 MB despite throttling, it gets killed. This two-level approach prevents unexpected kills while still providing a hard ceiling.

### Enabling controllers: the delegation chain

A controller must be enabled at every level of the hierarchy from root down to the cgroup where you want to use it. You enable it by writing to `cgroup.subtree_control` of the parent:

```bash
# Enable the cpu and memory controllers for children of this cgroup
echo "+cpu +memory" > /sys/fs/cgroup/myslice/cgroup.subtree_control
```

If you skip a level — enable in the root but not in an intermediate cgroup — child cgroups below that intermediate node will not have the controller available. The kernel enforces this delegation chain strictly.

### systemd's role

systemd is the cgroup manager on modern Linux. When a service unit starts, systemd creates a cgroup for it under the appropriate slice, moves the service's main process into it, and optionally configures limits based on the unit's `CPUQuota=`, `MemoryMax=`, and similar directives. When the service stops, systemd moves all processes out and removes the cgroup.

Every process on a systemd system lives in a cgroup managed by systemd. You don't need to create any cgroups yourself for normal service management — they already exist.

## Basic Example

```bash
# Create a new cgroup
sudo mkdir /sys/fs/cgroup/demo

# Enable the memory controller for this cgroup's children
# (requires the memory controller to be in the parent's subtree_control)
sudo cat /sys/fs/cgroup/cgroup.subtree_control
```

```text
cpuset cpu io memory hugetlb pids rdma misc
```

```bash
# Set a 100 MB memory limit on our new cgroup
echo "104857600" | sudo tee /sys/fs/cgroup/demo/memory.max

# Move the current shell into the cgroup
echo $$ | sudo tee /sys/fs/cgroup/demo/cgroup.procs
```

```text
12345
```

```bash
# Verify the shell is now in the cgroup
cat /proc/$$/cgroup
```

```text
0::/demo
```

```bash
# Read memory accounting for the cgroup
cat /sys/fs/cgroup/demo/memory.current
```

```text
2457600
```

Look at the `memory.current` output — that's roughly 2.4 MB for the shell process. The limit is 100 MB. If processes in this cgroup grow past 100 MB, the kernel will invoke the OOM killer against them.

The `cat /proc/$$/cgroup` output shows the unified hierarchy path. The single entry starting with `0::` is the v2 format — one line, one hierarchy. On a system running v1 controllers alongside v2, you'd see multiple lines here; on a pure v2 system like a modern Ubuntu or Fedora, you see exactly one.

Try this yourself in a VM. Move a shell into the cgroup, then try `stress --vm 1 --vm-bytes 200M` and watch what happens when it crosses the `memory.max` threshold. (Install `stress` first: `apt install stress` or `dnf install stress`.)

## Real-World Example

This scenario: you're on-call and a batch job is hammering the database server with unbounded I/O and CPU. You need to throttle it without killing it or restarting it.

```bash
# Find the batch job's PID
pgrep -a batch_processor
```

```text
47821 /usr/bin/python3 /opt/batch/processor.py --mode full
```

```bash
# Create a cgroup for it
sudo mkdir /sys/fs/cgroup/throttled_batch

# Enable cpu and io controllers (they must be in the parent's subtree_control)
# Check what's available at the root level first
cat /sys/fs/cgroup/cgroup.subtree_control
```

```text
cpuset cpu io memory hugetlb pids rdma misc
```

```bash
# Allow cpu and io to propagate into our new cgroup
echo "+cpu +io" | sudo tee /sys/fs/cgroup/throttled_batch/cgroup.subtree_control

# Limit to 25% of one CPU: 25,000 microseconds out of every 100,000
echo "25000 100000" | sudo tee /sys/fs/cgroup/throttled_batch/cpu.max

# Find the block device major:minor for the disk the job hammers
# (use lsblk to find it — here it's 8:0 for /dev/sda)
lsblk -o NAME,MAJ:MIN /dev/sda
```

```text
NAME MAJ:MIN
sda    8:0
```

```bash
# Limit block I/O to 10 MB/s read and write
echo "8:0 rbps=10485760 wbps=10485760" | sudo tee /sys/fs/cgroup/throttled_batch/io.max

# Move the process into the cgroup — all threads move atomically
echo 47821 | sudo tee /sys/fs/cgroup/throttled_batch/cgroup.procs

# Verify it landed there
cat /sys/fs/cgroup/throttled_batch/cgroup.procs
```

```text
47821
```

```bash
# Watch CPU throttling in real time
watch -n 1 cat /sys/fs/cgroup/throttled_batch/cpu.stat
```

```text
usage_usec 12483910
user_usec 9823740
system_usec 2660170
nr_periods 143
nr_throttled 89
throttled_usec 8934210
```

The `nr_throttled` and `throttled_usec` fields tell the story: 89 out of 143 scheduling periods were throttled. The process is being held to its 25% quota. It's still running — still making progress — but it's no longer dominating the scheduler.

The `io.max` format is `major:minor rbps=<bytes> wbps=<bytes>`. You can also limit IOPS with `riops=` and `wiops=`. These limits are enforced by the block layer: when the cgroup exceeds its I/O budget, reads and writes block until the next accounting period.

This approach is the right one for incident response: you constrain the offender without taking the service down. If this is a recurring problem, the right fix is to put the batch job in its own systemd scope with appropriate `CPUQuota=` and `IOWriteBandwidthMax=` settings in the unit file.

## ⚠️ Gotchas

### Writing to cgroup.procs Moves the Entire Process, Not Just a Thread

A learner might assume that writing a thread ID to `cgroup.procs` moves only that thread.

```bash
# Looking at a multi-threaded process
ps -T -p 47821
```

```text
  PID  SPID TTY          TIME CMD
47821 47821 ?        00:01:23 processor
47821 47822 ?        00:00:45 processor
47821 47823 ?        00:00:38 processor
```

```bash
# Writing the main PID to cgroup.procs
echo 47821 | sudo tee /sys/fs/cgroup/throttled_batch/cgroup.procs
```

All three threads — 47821, 47822, and 47823 — move into the new cgroup simultaneously. You cannot put individual threads of the same process into different cgroups using `cgroup.procs`. The kernel moves the entire process group atomically.

This is intentional. cgroups v2 limits apply to a process's total resource usage. Splitting threads across cgroups would make the accounting incoherent — you'd be charging half a process's memory to two different budgets. The unified design choice is that a process is the atomic unit of cgroup membership.

If you genuinely need per-thread control, look into threaded cgroups (set via `cgroup.type`), which are a specialized feature for specific workloads like thread pools.

### memory.max Kills Without Warning; memory.high Throttles First

The mistake: setting only `memory.max` thinking it will "slow down" a process when it gets close to the limit.

```bash
# This configuration has no buffer zone
echo "536870912" | sudo tee /sys/fs/cgroup/myapp/memory.max  # 512 MB hard limit
# memory.high is unset — defaults to "max" (no soft limit)
```

When the process hits 512 MB, the kernel invokes the OOM killer immediately. There is no warning. There is no gradual throttle. The process gets killed. If this is a production database, that's an outage.

The correct approach is to set `memory.high` below `memory.max` to create a buffer zone:

```bash
# 400 MB soft limit — throttle and reclaim before we're in danger
echo "419430400" | sudo tee /sys/fs/cgroup/myapp/memory.high

# 512 MB hard limit — OOM kill only if throttling couldn't help
echo "536870912" | sudo tee /sys/fs/cgroup/myapp/memory.max
```

Now: the process crosses 400 MB, the kernel backs off the pressure before the cliff. If the workload is genuinely large, it slows down. If it's a leak, the slowdown is observable in your metrics and you can act before the kill. The 112 MB gap is intentional — it gives reclaim room to work.

### Deleting a cgroup While It Still Has Processes Fails Silently With EBUSY

```bash
# Process is still in the cgroup
cat /sys/fs/cgroup/demo/cgroup.procs
```

```text
12345
```

```bash
# This will fail
sudo rmdir /sys/fs/cgroup/demo
```

```text
rmdir: failed to remove '/sys/fs/cgroup/demo': Device or resource busy
```

The `rmdir` fails with EBUSY. The cgroup will not be deleted until it contains no processes and no child cgroups. Move all processes to another cgroup first, then remove it:

```bash
# Move the shell back to the root cgroup
echo 12345 | sudo tee /sys/fs/cgroup/cgroup.procs

# Now the cgroup is empty and can be removed
sudo rmdir /sys/fs/cgroup/demo
```

The kernel enforces this because a cgroup with processes is a live accounting entity. Deleting it while processes are inside would orphan those processes from resource tracking. The EBUSY error is the kernel saying: you haven't finished the cleanup.

### cgroups v1 and v2 Cannot Both Manage the Same Controller

On a system where systemd has taken control of the unified v2 hierarchy, attempting to mount a v1 hierarchy for the same controller will fail or produce undefined behavior.

```bash
# This will fail on a system where systemd owns v2
sudo mount -t cgroup -o cpu cgroup /mnt/cgroup-cpu
```

```text
mount: /mnt/cgroup-cpu: special device cgroup does not exist.
```

Modern systemd (v248+) mounts the unified cgroup2 filesystem and uses it exclusively. It does not leave any v1 controllers available. If you need to verify which mode your system is in:

```bash
mount | grep cgroup
```

```text
cgroup2 on /sys/fs/cgroup type cgroup2 (rw,nosuid,nodev,noexec,relatime,nsdelegate,memory_recursiveprot)
```

A single `cgroup2` mount with no separate `cgroup` mounts — that's a pure v2 system. Don't try to layer v1 on top. If you're working with older container runtimes that require v1, you need to tell systemd to run in unified=no mode at boot (via kernel parameter), but this gives up all the v2 benefits.

## Alternatives & Comparisons

| Mechanism | Scope | What it limits | When to use |
|---|---|---|---|
| `ulimit` / `setrlimit` | Single process | File descriptors, stack size, virtual memory, open files | Per-process limits on a single service; legacy compatibility |
| cgroups v1 | Process group, per-controller trees | CPU, memory, I/O, network (separate hierarchies) | Legacy systems, older container runtimes requiring v1; avoid on new systems |
| cgroups v2 | Process group, unified hierarchy | All resources together, coordinated | Everything on modern Linux (kernel 5.2+); what containers actually use |
| systemd unit directives | Service managed by systemd | All cgroup v2 resources via readable unit file syntax | The right abstraction for service-level limits — use this instead of writing to `/sys/fs/cgroup/` manually |

The honest recommendation: if you're managing resource limits for services on a modern Linux system, use systemd unit directives (`CPUQuota=`, `MemoryMax=`, `MemoryHigh=`, `IOWriteBandwidthMax=`). They write to cgroup v2 on your behalf, they survive reboots without a service file change, and they're visible in `systemctl show myservice`. Writing directly to `/sys/fs/cgroup/` is valuable for understanding the mechanism and for one-off incident response. It is not how you do this in production day to day.

## Trade-offs

**Where this shines:**
- Coordinated multi-resource limits on a group of processes — the fundamental thing neither `ulimit` nor any per-process mechanism can do
- Container resource isolation with accounting precision — this is where Kubernetes resource requests and limits ultimately land
- Incident response: constraining a runaway process without killing it, in seconds, without a restart
- Visibility: `/sys/fs/cgroup/<name>/memory.current`, `cpu.stat`, `io.stat` give you exact accounting per service

**Where this costs you:**
- Complexity: the delegation chain (enabling controllers at every level from root down) is a non-obvious prerequisite that will bite you when it's silently missing
- The no-internal-process constraint adds structural discipline that forces you to think carefully about your cgroup hierarchy before writing processes into it
- Interaction with container runtimes: Docker, containerd, and Kubernetes all manage their own cgroup hierarchies; if you also manage cgroups manually on the same host, you can create conflicts
- Debugging throttled processes looks like a performance problem, not a configuration problem — `cpu.stat`'s `nr_throttled` counter is not surfaced in standard monitoring without explicit instrumentation

**Performance note:** The cgroup accounting overhead is real but small — typically under 1% for CPU and memory controllers on workloads that don't have extremely high fork/exit rates. The `pids` controller and `cpuset` controller add negligible overhead. The `io` controller adds measurable latency on high-IOPS workloads; benchmark before applying to latency-sensitive storage.

## When to Use / When to Avoid

✅ **Use this when:**
- Running multiple services on one machine and you need guaranteed isolation — any production multi-tenant system
- Deploying containers — this is what the runtime is doing; understanding it lets you diagnose limits that kubernetes or docker have set
- A service is misbehaving and you need to throttle it without downtime — write `cpu.max` and `memory.high` directly; it takes effect in seconds
- You're building a service that forks workers and want to enforce a PID ceiling to prevent fork bombs — `pids.max` is the right tool

❌ **Avoid this when:**
- You need per-process limits on a single binary — `ulimit` / `pam_limits` is simpler and sufficient
- You're on a system with kernel older than 5.2 — you'll hit missing controller features; check with `uname -r` first
- You're manually managing cgroups on a host that a container runtime already owns — you will fight the runtime; configure through the runtime's own API instead

⚠️ **Common mistakes:**
- Forgetting to enable a controller in `cgroup.subtree_control` of the parent before trying to use it in a child — the child's interface files simply won't exist
- Setting `memory.max` without `memory.high` and being surprised when a process gets OOM-killed instead of throttled
- Expecting `rmdir` to work on a non-empty cgroup — always verify with `cat cgroup.procs` first
- Writing limits that don't survive a reboot — direct writes to `/sys/fs/cgroup/` are volatile; use systemd unit directives for persistent limits

## Veteran Perspective

The surface-level mistake with cgroups is treating it as a "set it and forget it" safety net. People write `memory.max` in a unit file, deploy, and assume the service can never cause a memory problem. Then at 2am the service starts leaking, hits `memory.max`, gets OOM-killed, and systemd restarts it, and it OOM-kills again thirty seconds later, and you have a restart loop that looks like a crash loop because nobody set `memory.high` to give the system room to signal a problem before the kill.

The intermediate mistake is not understanding the accounting model. `memory.max` in cgroup v2 includes the process's page cache — the files it has read from disk that the kernel cached in RAM on its behalf. A process that reads large files may look like it's "using" 800 MB when most of that is clean page cache the kernel can evict instantly. If you set `memory.max` to 512 MB and the service reads log files, it may start getting OOM-killed because of page cache you didn't account for. Use `memory.stat` to understand the breakdown: `anon` (actual heap/stack), `file` (page cache), and `shmem` before drawing conclusions from `memory.current`.

The deeper design principle is that cgroups v2 was built to solve a coordination problem, not just a limiting problem. The reason the unified hierarchy exists — the reason v1 was replaced — is that resource management decisions require visibility across resources simultaneously. A scheduler that only knows about CPU cannot make a good eviction decision when the system is under memory pressure too. The unified hierarchy lets the kernel see the full picture of what a cgroup is consuming. This is why Kubernetes uses cgroups v2 for quality-of-service classes: it can make better scheduling decisions when it sees the complete resource envelope of a pod.

In interviews: if you're asked how Docker limits container memory and you say "cgroups," that's correct but incomplete. The better answer names v2, explains `memory.max` vs `memory.high`, mentions that the limit applies to the entire process tree including the container's init process, and notes that `docker stats` is reading directly from `memory.current`. That answer shows you've actually worked with the mechanism, not just heard the word.

Here's the war story: a team had a Kubernetes cluster with nodes occasionally going NotReady. The symptom was kernel OOM kills of kubelet itself. Nobody had set node-level cgroup limits, but the issue turned out to be a namespace that had set container memory limits in bytes that looked right (512Mi) but whose pods had init containers that each ran a migration script. The init containers counted against the pod's memory cgroup. The migration scripts loaded entire database snapshots into memory. Each ran sequentially, but they had run before in a different version where the limits weren't enforced. With cgroup v2 enforcement on the upgraded nodes, every deploy triggered an OOM kill of the migration container, the pod went into CrashLoopBackOff, and the node control plane got destabilized. Fix: increase the init container `memory.limit` separately from the main container, because they're separate containers in the pod's cgroup hierarchy. The takeaway: cgroup limits apply per-container, not per-pod by default in Kubernetes, and init containers have their own limits.

## Try It Yourself

1. On your machine, run `systemd-cgls` and find where your shell's process lives in the cgroup hierarchy. Then run `cat /proc/$$/cgroup` and compare the output. They should agree.

2. Create a cgroup at `/sys/fs/cgroup/testgroup`, write your shell's PID into `cgroup.procs`, then run `cat /proc/$$/cgroup` again to confirm the move. Move yourself back to the root cgroup when you're done.

3. Set `memory.high` to 50 MB on a test cgroup, move a shell into it, and then run `stress --vm 1 --vm-bytes 80M --vm-keep` (install `stress` if needed). Watch `cat /sys/fs/cgroup/testgroup/memory.current` while it runs. Does the process slow down? Does it get killed? Why or why not?

4. Run `systemd-cgtop` and find the top three cgroups by memory usage on your system. Then use `systemctl show <service>` for one of them to see what memory limits systemd has configured. Are there any limits at all? What would you set for a production service?

5. **Challenge (VM required):** Write a systemd drop-in for an existing service (pick something benign like `cron` or `rsyslog`) that sets `CPUQuota=10%`, `MemoryHigh=64M`, and `MemoryMax=128M`. Verify with `systemctl cat <service>` that the drop-in was loaded. Verify with `systemd-cgls` and `cat /sys/fs/cgroup/system.slice/<service>.service/cpu.max` that the limits are actually in the cgroup interface files. Then remove the drop-in and confirm the limits are gone after a `daemon-reload` and service restart.

---

## Summary

- A cgroup groups processes and enforces resource limits (CPU, memory, I/O, PIDs) on their combined usage — not per-process but per-group; this is the fundamental thing `ulimit` cannot do.
- cgroups v2 replaced v1's multiple independent per-controller hierarchies with a single unified tree at `/sys/fs/cgroup/`; every modern distro (Fedora 31+, Ubuntu 21.10+, Debian 11+, RHEL 9+) uses v2 by default.
- `memory.high` throttles without killing; `memory.max` invokes the OOM killer — always set `memory.high` below `memory.max` to create a buffer zone and avoid surprise kills.
- `cpu.max` is a hard bandwidth ceiling (`quota period` in microseconds); `cpu.weight` is a relative share that only matters during contention — they are independent mechanisms that can be combined.
- systemd owns the cgroup hierarchy on modern Linux; use unit file directives (`CPUQuota=`, `MemoryMax=`, `MemoryHigh=`) for persistent limits rather than writing directly to `/sys/fs/cgroup/`.
