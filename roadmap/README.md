# Linux Learning Roadmap

A structured path from zero to veteran. Each phase builds on the last, but you
can jump to any topic — the roadmap is a map, not a cage.

For the complete reference on every topic, see the [Bible](../bible/README.md).

---

## How to Progress

- **Don't just read — type it.** Linux is a physical skill as much as a mental one; your fingers need to know the commands, not just your eyes.
- **Break things in a VM.** You will not truly understand a dangerous command until you've run it in a safe environment and watched what happens.
- **Don't skip Phase 2.** The filesystem is the foundation everything else builds on — a weak mental model here becomes confusion in every later phase.
- **A weak phase becomes a wall two phases later.** If something in Phase 5 doesn't click, the problem is usually in Phase 4 or 3 — go back.
- **The checkpoints are not optional.** Reading is not learning. Each checkpoint forces you to combine multiple concepts into something that either works or doesn't.
- **Use the bible, not a search engine.** When you need depth on a topic, the bible has the full picture — a search result has a fragment.

## Phase 1 — Getting Comfortable with Linux

> The goal of this phase is to open a terminal on any Linux system and navigate it without hesitation.

**What you'll be able to do after this phase:**
- Navigate the filesystem from the command line without thinking about it
- Read any file on the system using the right tool for its size and type
- Find help for any command without leaving the terminal
- Explain the difference between a kernel, a shell, a terminal, and a distribution
- Customize your shell prompt and understand what fires on startup

**Topics in learning order:**

```
1.1  What Is Linux?
     └─ Unix (1969) → GNU Project → Linux kernel: how they became one OS
     └─ Kernel vs OS vs distribution — three things most people conflate
     └─ Why the same kernel runs your phone, a supercomputer, and a web server

1.2  The Terminal
     └─ TTY vs terminal emulator vs shell — three separate things, often confused
     └─ Login vs non-login vs interactive shells — which startup files load and why
     └─ Keyboard shortcuts that save hours: Ctrl+R, Ctrl+A, Ctrl+E, Ctrl+W

1.3  Navigating the Filesystem
     └─ Absolute vs relative paths — the confusion that trips up every beginner
     └─ ls -la decoded column by column — before you can read permissions, read the output
     └─ cd -, cd ~, pushd/popd — navigation shortcuts worth learning early

1.4  Reading and Writing Files
     └─ cat vs less vs head/tail — choosing the right tool for the file size
     └─ vim survival: insert mode, normal mode, how to exit (yes, this is real)
     └─ The -rw-r--r-- string you'll see constantly — preview of Phase 3

1.5  Getting Help
     └─ man sections 1, 2, 5, 8 — most people only know section 1 exists
     └─ apropos — finding a command when you don't know its name
     └─ tldr as a complement to man pages, not a replacement
```

**Phase 1 checkpoint:** On a fresh Linux VM, find the five largest files under `/var` using only command-line tools, read the man page for two commands you haven't used before, and write a summary of what each does into a file called `~/notes.txt` — no file manager, no browser.

## Phase 2 — The Linux Filesystem

> The goal of this phase is to understand why files live where they live, and what "everything is a file" actually means at the kernel level.

**What you'll be able to do after this phase:**
- Explain the purpose of every top-level directory in the FHS without looking it up
- Distinguish between regular files, symlinks, device files, sockets, and pipes by their `ls` output
- Explain what an inode is, what it stores, and why you can run out of inodes before disk space
- Mount and unmount filesystems, add a persistent mount to fstab, and verify it survives reboot
- Use `/proc` to read live kernel data about running processes

**Topics in learning order:**

```
2.1  Filesystem Hierarchy Standard (FHS)
     └─ /proc and /sys are not real directories — generated by kernel at runtime
     └─ /var vs /tmp — what persists across reboots and what doesn't
     └─ The /usr merge — why /bin is now a symlink to /usr/bin on modern systems

2.2  Everything Is a File
     └─ /dev/null, /dev/zero, /dev/urandom — the most useful "files" that aren't files
     └─ Unix domain sockets — how processes talk to each other via the filesystem
     └─ Why this abstraction makes pipes, redirection, and devices interchangeable

2.3  File Types
     └─ The first character in ls -l: d, -, l, b, c, s, p — what each means
     └─ Hard links vs symlinks — the same data vs a pointer to a path
     └─ Dangling symlinks — when the target disappears and the pointer stays

2.4  Inodes
     └─ An inode stores everything about a file except its name
     └─ Why deleting a hard-linked file doesn't free disk space until all links are gone
     └─ df -i — inode exhaustion looks identical to disk-full errors until you check

2.5  Mounting
     └─ The mount tree: everything in Linux hangs off one root /
     └─ fstab fields decoded: device, mountpoint, fstype, options, dump, pass
     └─ tmpfs is RAM — it disappears on reboot, and most people don't realize /run is tmpfs
```

**Phase 2 checkpoint:** Write a one-page filesystem tour: show the inode number of three different file types using `stat`, find a process's open file descriptors in `/proc/PID/fd/`, create a tmpfs mount at `/mnt/test` with a 100MB limit, add it to `/etc/fstab`, and verify it appears in `df -h` after remounting.

## Phase 3 — File Operations & Permissions

> The goal of this phase is to manage files and permissions with full confidence — knowing what every bit means, not just what the command does.

**What you'll be able to do after this phase:**
- Set permissions correctly for any scenario: shared directories, web roots, scripts, secrets
- Explain exactly what read, write, and execute mean on both files and directories
- Use `find` to audit a system for dangerous permission configurations
- Configure ACLs for scenarios that basic user/group/other can't handle
- Recognize setuid, setgid, and sticky bit in the wild and explain why they're there

**Topics in learning order:**

```
3.1  Creating, Copying, Moving, Deleting
     └─ cp -a vs cp -r — preserving permissions, timestamps, and symlinks
     └─ mv is atomic within a filesystem but not across — the rename() syscall
     └─ find -exec vs xargs — when each is faster and why

3.2  Permissions
     └─ Execute on a directory means traverse, not run — the most common misconception
     └─ Octal notation decoded: 755, 644, 700, 600 — what each is used for and why
     └─ Why chmod 777 on anything is almost always wrong

3.3  Ownership
     └─ A file has exactly one owner and one group — not multiple of either
     └─ chown user:group vs chown user — whether the group changes
     └─ Why root can chown to anyone but regular users can only chown to themselves

3.4  Special Permissions
     └─ setuid on /usr/bin/passwd — why it's there and why it's controlled carefully
     └─ setgid on a directory — new files inherit the directory's group, not the creator's
     └─ Sticky bit on /tmp — why you can't delete other users' files there

3.5  Access Control Lists (ACL)
     └─ When the basic model breaks: a file needs two different groups with different access
     └─ getfacl / setfacl syntax — it's not obvious the first time
     └─ Default ACLs — inherited automatically by new files created inside a directory
```

**Phase 3 checkpoint:** Set up a shared project directory: create a group `devteam`, a directory `/srv/project` where any member can create and edit files but only the owner of each file can delete it, a subdirectory `logs/` where any member can append but not overwrite, and a script `deploy.sh` that executes as a specific non-root user via setuid — audit the result with `find` and `getfacl`.

## Phase 4 — Text Processing & The Unix Philosophy

> The goal of this phase is to use the terminal as a data processing engine — composing small tools into pipelines that solve real problems.

**What you'll be able to do after this phase:**
- Build pipelines that process log files, extract fields, filter, sort, and count without writing a script
- Use grep with full regular expression syntax to find anything in text
- Edit files in-place with sed and extract structured data with awk
- Redirect stdin, stdout, and stderr independently and understand what each means
- Explain the Unix philosophy and why it produces more composable tools than monolithic ones

**Topics in learning order:**

```
4.1  The Unix Philosophy
     └─ Do one thing well — why small tools beat monolithic programs for composability
     └─ Text as the universal interface — why everything outputs text and what that enables
     └─ The pipeline mental model: data flows left to right through transformations

4.2  grep and Regular Expressions
     └─ BRE vs ERE vs PCRE — three different regex dialects, one command family
     └─ -A, -B, -C context flags — the ones that save you in real log investigation
     └─ Character classes [[:alpha:]] — portable regex that survives locale changes

4.3  sed — Stream Editor
     └─ The s/pattern/replacement/flags substitution — the command you'll use 90% of the time
     └─ -i in-place editing — and why always using -i.bak is a good habit
     └─ Address ranges — applying edits only between two line patterns

4.4  awk — Data Extraction and Reporting
     └─ $1, $2, $NF — field splitting and why -F changes everything
     └─ BEGIN and END blocks — setup and teardown logic
     └─ Associative arrays — counting and aggregating without a database

4.5  The Classic Toolkit
     └─ sort -k for multi-field sorting — the flag most people don't know
     └─ uniq requires sorted input — the gotcha that produces wrong counts silently
     └─ xargs -P for parallel execution — turning a list into concurrent jobs

4.6  Redirects and Pipes
     └─ 2>&1 vs &> — two ways, one meaning, and which one works in which shell
     └─ tee — splitting output to a file and stdout simultaneously
     └─ Process substitution <() — using command output as a file argument
```

**Phase 4 checkpoint:** Write a single pipeline — no scripts, no loops, no intermediate files — that reads `/var/log/syslog` (or equivalent), extracts the top 10 most frequent source hostnames or services with their counts, formats the output as a right-aligned table, and writes the result sorted descending to `~/log-report.txt`.

## Phase 5 — Shell & Scripting

> The goal of this phase is to write shell scripts that automate real tasks, handle errors correctly, and don't break in ways that are hard to debug.

**What you'll be able to do after this phase:**
- Write scripts that fail loudly on errors instead of silently continuing
- Handle signals, clean up on exit, and pass arguments correctly
- Debug a broken script using tracing and know which tool to reach for
- Explain the quoting rules that cause subtle bugs in 90% of shell scripts
- Use shellcheck as a mandatory part of your writing process

**Topics in learning order:**

```
5.1  Shell Basics
     └─ bash vs sh vs dash — why #!/bin/bash and #!/bin/sh are not the same
     └─ Interactive vs non-interactive shells — why your script doesn't have your aliases
     └─ The PATH: how the shell finds commands and what happens when it doesn't

5.2  Variables, Quoting, and Expansion
     └─ Unquoted variables cause word splitting — the source of half of all shell bugs
     └─ ${VAR:-default} and family — safe variable expansion with fallbacks
     └─ $(cmd) vs backticks — use $() always; nesting backticks is unreadable

5.3  Control Flow
     └─ [[ ]] vs [ ] — bash extended test vs POSIX test and why it matters
     └─ for f in *.txt vs for f in $(ls *.txt) — the globbing vs subshell difference
     └─ case patterns — faster than if-elif chains for matching strings

5.4  Functions, Arguments, Exit Codes
     └─ local variables — without local, your functions pollute the script's namespace
     └─ $? is reset by every command — capture it immediately or it's gone
     └─ return vs exit — functions return, scripts exit; misusing these breaks pipelines

5.5  Writing Safe Scripts
     └─ set -euo pipefail — what each flag does and why all three belong together
     └─ Quoting every variable: "$var" not $var — the rule with no exceptions
     └─ shellcheck: treat its warnings as errors, not suggestions

5.6  Debugging Scripts
     └─ bash -x script.sh vs set -x inline — full trace vs targeted trace
     └─ bash -n for syntax checking without execution
     └─ trap ERR — catching errors as they happen, not after the fact
```

**Phase 5 checkpoint:** Write a script that watches a directory for new `.log` files (using a loop and `inotifywait` or `stat`), appends each new filename with a timestamp to `~/watcher.log`, sends `SIGUSR1` to a process named in `$1` when a `.trigger` file appears, and exits cleanly on `SIGTERM` with a cleanup message — `set -euo pipefail` throughout, zero shellcheck warnings.

## Phase 6 — Processes & System Resources

> The goal of this phase is to see exactly what your system is doing — identify any process, understand its state, and control it with the right tool.

**What you'll be able to do after this phase:**
- Explain the difference between a process, a thread, a zombie, and an orphan
- Find any process by name, PID, user, or open file
- Send the right signal for the right situation — not just SIGKILL for everything
- Read `/proc/PID/` to answer questions about a running process without attaching a debugger
- Use job control to manage background tasks from the shell

**Topics in learning order:**

```
6.1  The Process Model
     └─ fork() creates a copy; exec() replaces it — the two-step that launches everything
     └─ Zombie processes: finished but parent hasn't called wait() — what they cost you
     └─ PID 1 adopts orphans — why systemd is the parent of most of your processes

6.2  Viewing Processes
     └─ ps aux vs ps -ef — BSD vs SysV syntax producing nearly the same output
     └─ Process states: R, S, D, Z, T — D is uninterruptible sleep and it matters
     └─ pstree -p — visualizing parent-child relationships; useful for debugging forks

6.3  Signals
     └─ SIGTERM vs SIGKILL — always try SIGTERM first; SIGKILL leaves no cleanup
     └─ SIGHUP to reload config — why daemons use it instead of restart
     └─ kill -l — there are 34 signals; you need to know about 8 of them

6.4  Job Control
     └─ Ctrl+Z suspends (SIGTSTP), not kills — bg resumes it in the background
     └─ nohup vs disown — keeping jobs alive after you log out
     └─ wait $pid — the right way to collect background job exit codes in scripts

6.5  Resource Limits
     └─ ulimit -n — open file limit; the first thing to check on "too many open files"
     └─ /proc/PID/limits — current limits for a running process
     └─ cgroups overview — the kernel mechanism systemd uses under the hood

6.6  /proc Filesystem
     └─ /proc/PID/cmdline, environ, cwd, exe, fd/ — a complete picture of any process
     └─ /proc/meminfo and /proc/cpuinfo — system-wide stats without any external tools
     └─ /proc/PID/maps — virtual memory layout; useful for diagnosing memory issues
```

**Phase 6 checkpoint:** Without installing any tools, answer these questions about your running system using only `/proc` and standard commands: which process has the most open file descriptors, what is the full environment of PID 1, and which process is holding a specific named file open — document every command used and what in the output gave you the answer.

## Phase 7 — Users, Groups & Authentication

> The goal of this phase is to manage users, privileges, and SSH access on a Linux server — knowing exactly what each configuration line does and why.

**What you'll be able to do after this phase:**
- Create users and groups and explain every field in `/etc/passwd` and `/etc/shadow`
- Configure sudo to grant specific commands to specific users without giving full root
- Set up SSH key authentication and harden `sshd_config` against common attacks
- Explain how PAM sits between a service and the actual authentication mechanism
- Lock, expire, and audit user accounts on a production system

**Topics in learning order:**

```
7.1  User and Group Management
     └─ /etc/passwd fields decoded: username, x, UID, GID, GECOS, home, shell
     └─ useradd vs adduser — the difference on Debian/Ubuntu systems
     └─ id, who, w, last — the four commands for answering "who is on this system"

7.2  The Auth Files
     └─ /etc/shadow: why passwords aren't in /etc/passwd and what the hash fields mean
     └─ chage — password aging; the tool most admins forget exists until an audit
     └─ Locked accounts ($!) vs expired accounts — two different states, two different fixes

7.3  Privilege Escalation
     └─ sudo vs su — prefer sudo; it logs, it limits, it doesn't need the root password
     └─ visudo and NEVER editing /etc/sudoers directly — syntax errors lock you out
     └─ NOPASSWD: when it's justified and when it's a security hole

7.4  PAM
     └─ PAM sits between the service (sshd, sudo, login) and the actual auth mechanism
     └─ required vs requisite vs sufficient — how control flags combine
     └─ pam_limits.so — how ulimit settings get applied at login

7.5  SSH
     └─ Ed25519 keys over RSA in 2024 — smaller, faster, and not vulnerable to bad RNG
     └─ ~/.ssh/config Host blocks — the feature that makes SSH actually usable at scale
     └─ ProxyJump — SSH through a bastion host in one command
```

**Phase 7 checkpoint:** On a fresh VM, configure it for a team: create a user `deploy` in a group `deployers`, write a sudoers rule granting `deploy` the ability to run `systemctl restart nginx` only (no password prompt), set up SSH key auth for `deploy`, disable root login and password auth in `sshd_config`, add a `fail2ban` jail for SSH — verify each step cannot be bypassed before moving to the next.

## Phase 8 — Package Management

> The goal of this phase is to install, manage, and build software on any Linux distribution — including when no package exists.

**What you'll be able to do after this phase:**
- Manage packages on Debian/Ubuntu and RHEL/Fedora systems with confidence
- Pin a package to a specific version and explain why that's sometimes necessary
- Compile software from source, track what was installed, and remove it cleanly
- Explain the difference between a package manager, a package, and a repository
- Describe why containers changed software distribution and what problem they solved

**Topics in learning order:**

```
8.1  apt and dpkg (Debian/Ubuntu)
     └─ dpkg is the low level; apt is the dependency resolver on top
     └─ apt install vs apt-get install — they're almost the same; apt is for humans
     └─ apt-mark hold — pinning a package; what breaks if you forget to do this

8.2  dnf and rpm (RHEL/Fedora/CentOS)
     └─ rpm -qf /path/to/file — which package owns this file; invaluable for debugging
     └─ dnf history undo — rolling back a transaction; the feature apt doesn't have
     └─ EPEL — what it is and why you almost always need it on RHEL systems

8.3  pacman (Arch)
     └─ pacman -Syu — the full system upgrade in one command
     └─ AUR and PKGBUILD — user-maintained packages and the trust model they require
     └─ pacman -Qdt — orphaned packages; the cleanup step most users skip

8.4  Compiling from Source
     └─ ./configure --prefix=/usr/local — why you set the prefix and what it changes
     └─ make -j$(nproc) — parallel builds; the flag that makes compilation tolerable
     └─ checkinstall — turning a make install into a trackable package

8.5  Containers as an Alternative
     └─ Why OCI images solved "it works on my machine" at the distribution level
     └─ The image layer model — and why layer order matters for build efficiency
     └─ Static binaries — the extreme version of self-contained software distribution
```

**Phase 8 checkpoint:** Install `curl` via the package manager, hold it at its current version, then download and compile the latest `curl` from source into `/usr/local` — both versions should coexist, you should be able to switch between them with `PATH` manipulation, and you should be able to uninstall the source-compiled version cleanly.

## Phase 9 — Networking

> The goal of this phase is to diagnose and configure Linux networking from the command line — no GUI, no guessing, just tools and understanding.

**What you'll be able to do after this phase:**
- Read and configure network interfaces, routes, and DNS resolution on any Linux system
- Diagnose a network problem methodically — from physical layer to application layer
- Write firewall rules that allow what you intend and block what you don't
- Set up SSH tunnels for port forwarding and SOCKS proxying
- Use `curl` to interact with any HTTP API from the command line

**Topics in learning order:**

```
9.1  Network Fundamentals
     └─ ip addr, ip link, ip route — the three commands that replaced ifconfig, route
     └─ CIDR notation: 192.168.1.0/24 decoded — address, mask, broadcast, host range
     └─ DNS resolution order: /etc/nsswitch.conf → /etc/hosts → resolver

9.2  Diagnostic Tools
     └─ ss -tulpn — the single command that shows every listening service and its PID
     └─ dig +trace — following DNS resolution from root servers down
     └─ mtr — combining ping and traceroute; the right tool for intermittent packet loss

9.3  Network Configuration Files
     └─ /etc/resolv.conf and why it gets overwritten by NetworkManager/systemd-resolved
     └─ /etc/nsswitch.conf — the file that decides which resolver is consulted first
     └─ NetworkManager vs systemd-networkd — which manages your interfaces and how to tell

9.4  Firewalls
     └─ Netfilter is the kernel framework; iptables/nftables are userspace tools on top
     └─ nftables vs iptables — nftables is the future; iptables-nft is the bridge
     └─ ufw / firewalld — distro wrappers that hide complexity (and sometimes the gotchas)

9.5  SSH Advanced
     └─ -L localport:remotehost:remoteport — local forwarding decoded
     └─ -R remoteport:localhost:localport — reverse tunnel for reaching NAT'd machines
     └─ ControlMaster connection reuse — one TCP connection for many SSH sessions

9.6  HTTP from the Command Line
     └─ curl -v shows the full request/response cycle — use it when debugging APIs
     └─ curl -w "%{time_total}" — measuring latency and TTFB from the CLI
     └─ wget for downloading; curl for everything else

9.7  Network Namespaces
     └─ ip netns — creating isolated network stacks; the foundation of containers
     └─ veth pairs — virtual ethernet cables connecting namespaces
     └─ ip link add br0 type bridge — software bridge; how VMs share a physical NIC
```

**Phase 9 checkpoint:** On a two-VM setup: configure a firewall on VM1 allowing SSH only from VM2's IP (all other SSH blocked), set up a local SSH tunnel from VM2 forwarding port 8080 to port 80 on VM1, start a simple HTTP server on VM1's port 80, and use `curl` through the tunnel to fetch a file — document every command with a one-line explanation of what it does.

## Phase 10 — Storage & Filesystems

> The goal of this phase is to manage disks, partitions, and filesystems without fear — and without losing data.

**What you'll be able to do after this phase:**
- Partition a disk, create a filesystem, and mount it persistently without looking anything up
- Set up an LVM volume that spans multiple disks and resize it online
- Choose the right filesystem for a given workload and explain the trade-offs
- Run a backup that you can actually restore from
- Interpret `df` and `du` output and find where disk space actually went

**Topics in learning order:**

```
10.1  Block Devices and Partitioning
      └─ /dev/sda vs /dev/sda1 — disk vs partition; the most basic distinction
      └─ MBR vs GPT — 4-partition limit, 2TB limit, and why GPT is always the right choice now
      └─ lsblk — the cleaner way to see disk topology before touching anything

10.2  Filesystems
      └─ ext4 is safe and boring; xfs is fast for large files; btrfs has snapshots — know when each matters
      └─ mkfs destroys data instantly and silently — always lsblk before mkfs
      └─ fsck runs only on unmounted filesystems — attempting it on a live fs causes corruption

10.3  LVM
      └─ PV → VG → LV: physical volume, volume group, logical volume — the three layers
      └─ lvextend + resize2fs — the two-step that extends a live ext4 volume online
      └─ LVM snapshots — consistent backups of a running database without downtime

10.4  RAID
      └─ RAID is not a backup — it protects against disk failure, not accidental deletion
      └─ RAID 5 rebuild time on large disks can take days — the URE risk during rebuild
      └─ mdadm --detail /dev/md0 — the one command to always check first

10.5  Disk Usage
      └─ df shows filesystem-level usage; du shows directory-level usage — different questions
      └─ du -sh /* | sort -h — finding the largest directories at the root level
      └─ Deleted files still consuming space — lsof | grep deleted reveals them

10.6  Backups
      └─ rsync --link-dest — space-efficient incremental backups using hard links
      └─ tar -czf vs rsync — archiving vs mirroring: different tools for different goals
      └─ Test your restore before you need it — a backup you haven't tested isn't a backup
```

**Phase 10 checkpoint:** On a VM with two spare disks: create a volume group spanning both, carve out a 5GB logical volume, format it as ext4, mount it persistently via `/etc/fstab` using UUID (not device name), verify it survives reboot, then extend it by 3GB online without unmounting — verify the new size with `df -h` and `lvdisplay`.

## Phase 11 — Init Systems & Service Management

> The goal of this phase is to control the full service lifecycle on a Linux system — from boot to shutdown — and write your own systemd units.

**What you'll be able to do after this phase:**
- Explain every step of the Linux boot process from power-on to login prompt
- Start, stop, enable, disable, mask, and override any systemd service
- Read and write systemd unit files for services, timers, and paths
- Query and filter the journal to find exactly what you need
- Diagnose a service that fails to start by reading its logs and status

**Topics in learning order:**

```
11.1  The Boot Process
      └─ BIOS/UEFI → GRUB → kernel → initramfs → PID 1 — the handoff chain
      └─ initramfs purpose: mounts the real root before the real root is available
      └─ systemd-analyze blame — finding what's slowing down boot time

11.2  systemd Architecture
      └─ Units: service, socket, timer, mount, target — each type serves a different purpose
      └─ Wants vs Requires — soft vs hard dependencies and why the distinction matters
      └─ Targets replace runlevels — multi-user.target is "the system is up"

11.3  systemctl
      └─ enable creates symlinks; start runs now — they are independent actions
      └─ mask prevents a unit from ever starting, even as a dependency — stronger than disable
      └─ systemctl --failed — the first thing to check after a reboot you didn't expect

11.4  journalctl
      └─ journalctl -u nginx.service -f — follow a specific unit's logs
      └─ journalctl --since "1 hour ago" -p err — errors in the last hour
      └─ Persistent journal requires Storage=persistent in journald.conf — not the default everywhere

11.5  Writing a systemd Service
      └─ Type=simple vs Type=exec vs Type=forking — choosing wrong breaks restart logic
      └─ Restart=on-failure with RestartSec= — the combination that keeps services alive
      └─ Sandboxing directives: PrivateTmp=, ProtectSystem=, NoNewPrivileges= — free hardening

11.6  Legacy Context
      └─ SysV init scripts: /etc/init.d/, sequential startup, no dependency declarations
      └─ Why systemd won: parallel startup, dependency graph, socket activation, logging
      └─ service and chkconfig still work on systemd — they're wrappers, not the real tool
```

**Phase 11 checkpoint:** Write a complete systemd setup: a service that runs a shell script as a non-root user, restarts on failure with a 5-second delay, logs a startup message to the journal, and a timer that triggers it every 10 minutes — create a drop-in override that adds `MemoryMax=100M` without editing the original unit file, and verify the whole setup with `systemctl status` and `journalctl`.

## Phase 12 — Performance & Observability

> The goal of this phase is to diagnose a slow or misbehaving Linux system using only command-line tools — and know which tool answers which question.

**What you'll be able to do after this phase:**
- Identify whether a performance problem is CPU, memory, I/O, or network
- Attach to a running process with `strace` and read what it's doing
- Interpret load averages correctly — including when a high average is not a problem
- Find which process has a specific file locked or a specific port open
- Profile a program's system call behavior without modifying its source

**Topics in learning order:**

```
12.1  CPU Analysis
      └─ vmstat 1 — the one-second snapshot that shows run queue, swap, and CPU split
      └─ %wa (iowait) — CPU time waiting for disk; often mistaken for a CPU problem
      └─ perf stat — hardware counter data; instructions per cycle reveals real efficiency

12.2  Memory Analysis
      └─ "available" not "free" — the column in free -h that actually matters
      └─ OOM killer log in dmesg/journalctl — what it killed, why, and what the score was
      └─ /proc/meminfo: Dirty and Writeback — unflushed writes; high values mean I/O pressure

12.3  I/O Analysis
      └─ iostat -x 1 — %util and await per device; near 100% util means a saturated disk
      └─ iotop -o — only showing processes currently doing I/O
      └─ Dirty page writeback tuning — vm.dirty_ratio and when to touch it

12.4  Load Averages
      └─ Load average counts runnable + uninterruptible-sleep processes — not just CPU
      └─ A load of 4.0 on a 4-core machine is 100% — on an 8-core machine it's 50%
      └─ Sustained high load with low CPU% points to I/O wait — check iostat next

12.5  strace and ltrace
      └─ strace -p PID — attaching to a running process without restarting it
      └─ strace -e trace=open,read — filtering to only the syscalls you care about
      └─ strace overhead is real — it slows the traced process 2-10x; don't use on production lightly

12.6  lsof
      └─ lsof -i :8080 — which process is using this port; works where ss doesn't show the pid
      └─ lsof +D /mnt/data — why you can't unmount a filesystem (something has it open)
      └─ lsof | grep deleted — finding the process keeping a deleted file's space allocated
```

**Phase 12 checkpoint:** Without installing any new tools, produce a performance report for a running system: identify the top CPU consumer and its state, find where memory is going (is the page cache large or is a process leaking?), find the process generating the most disk I/O, run `strace -c` on a command of your choice for 10 seconds and report which syscall dominates — document every command and what the output told you.

## Phase 13 — Security

> The goal of this phase is to harden a Linux server against common attack vectors — and audit one that may already be misconfigured.

**What you'll be able to do after this phase:**
- Audit a system's file permissions for dangerous configurations using `find`
- Configure SELinux or AppArmor and diagnose denials without disabling them
- Apply kernel hardening via `sysctl` and explain what each setting does
- Read `auditd` logs and set up rules to detect specific suspicious actions
- Produce a hardening checklist and verify each item before signing off

**Topics in learning order:**

```
13.1  Principle of Least Privilege
      └─ Each service runs as its own user — not root, not nobody
      └─ Capabilities: cap_net_bind_service lets a non-root process bind port 80
      └─ The minimal footprint: if a service doesn't need it, it shouldn't have it

13.2  File Permissions Audit
      └─ find / -perm -4000 2>/dev/null — all setuid binaries; audit this list on every new system
      └─ find / -perm -0002 -type f — world-writable files; any of these is a risk
      └─ chattr +i on /etc/passwd — immutable flag; even root can't edit without removing it first

13.3  Mandatory Access Control
      └─ SELinux contexts: every file and process has a label; denials are explicit not implicit
      └─ audit2allow is powerful and dangerous — read what it generates before applying it
      └─ AppArmor profiles: path-based, easier to write, less granular than SELinux

13.4  Kernel Hardening
      └─ net.ipv4.tcp_syncookies=1 — SYN flood protection; should be on everywhere
      └─ kernel.randomize_va_space=2 — ASLR; make sure it's 2, not 1 or 0
      └─ net.ipv4.ip_forward=0 — unless this machine is a router, it should be off

13.5  Audit Logging
      └─ auditctl -w /etc/passwd -p wa — watch a file for writes and attribute changes
      └─ ausearch -f /etc/passwd — finding all accesses to a specific file
      └─ last and lastb — successful and failed logins; know your baseline

13.6  Attack Surface and Hardening Checklist
      └─ SSH: key-only, no root, MaxAuthTries 3, ClientAliveInterval
      └─ Services: disable everything not in use; ss -tulpn is your audit command
      └─ Updates: unattended-upgrades / dnf-automatic for security patches
```

**Phase 13 checkpoint:** Harden a fresh VM against a documented checklist: SSH (key-only, no root login, brute-force protection), firewall (only ports 22 and 80 open, all else dropped), permissions audit (find and fix all world-writable files and unexpected setuid binaries), at least 5 sysctl hardening settings with justification, and one auditd rule watching `/etc/sudoers` — verify every item with the appropriate command and record the output.

## Phase 14 — Advanced Topics

> The goal of this phase is to understand the kernel internals that power containers, observability tools, and modern Linux infrastructure — not just use them.

**What you'll be able to do after this phase:**
- Load, unload, and inspect kernel modules — and write a minimal one
- Explain how cgroups v2 limits resources per process group
- Describe all seven namespace types and what each isolates
- Build a container from scratch using only kernel primitives — no Docker
- Write a `bpftrace` one-liner to observe kernel behavior live

**Topics in learning order:**

```
14.1  Kernel Modules
      └─ lsmod, modinfo, modprobe, rmmod — the four commands for module lifecycle
      └─ /etc/modprobe.d/ for options and blacklisting — the right way to disable a driver
      └─ Hello-world .ko — module_init(), module_exit(), printk() — kernel code is C

14.2  cgroups v2
      └─ The unified hierarchy: one tree, all controllers — different from cgroups v1
      └─ memory.max vs memory.high — hard limit (OOM) vs soft limit (throttle)
      └─ systemd puts every service in its own cgroup — systemd-cgls shows the tree

14.3  Namespaces
      └─ Seven types: pid, net, mnt, uts, ipc, user, time — each isolates one aspect
      └─ unshare --pid --fork --mount-proc — running a process in its own PID namespace
      └─ nsenter -t PID --net --pid — entering a running container's namespaces

14.4  Containers from Scratch
      └─ chroot changes root but doesn't isolate — containers need namespaces too
      └─ pivot_root is the correct way to change root; chroot alone is not secure isolation
      └─ overlayfs: lowerdir (image) + upperdir (writes) + merged (the view) — how image layers work

14.5  eBPF
      └─ eBPF programs run in the kernel but are verified safe before loading
      └─ bpftrace -e 'tracepoint:syscalls:sys_enter_open { printf("%s\n", comm); }' — every process opening a file
      └─ BCC tools: opensnoop, execsnoop, biolatency — production-safe observability
```

**Phase 14 checkpoint:** Build a container from scratch on a Linux VM — no Docker, no containerd: use `unshare` to create isolated PID and network namespaces, use a minimal Alpine rootfs with `pivot_root` to set it as the container root, give it a network interface using a `veth` pair and a bridge, and verify from inside the container that it has its own PID 1 and its own network stack — document every kernel primitive used and what it does.
